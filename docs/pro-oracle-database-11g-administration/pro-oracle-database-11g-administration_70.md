在前面的命令中，你正在检查操作系统上的所有文件，试图定位一个文件名以“alert”开头、以“.log”结尾的文件。执行全局搜索时，当 `find` 命令尝试访问它没有读取权限的文件时，将会抛出错误。`2>/dev/null` 非常有用，因为它将所有错误消息重定向到 `/dev/null`（位桶）文件。结果是你不会看到大量的错误消息，只有当 `find` 成功找到与字符串模式匹配的文件时，你才会看到输出。

当尝试查找 `alert.log` 文件时，你可能会发现不止一个文件。当经验不足的 DBA 将 `alert.log` 从标准位置移动到非标准位置，并且将旧文件散落在整个操作系统中时，就会发生这种情况。

当你认为找到了正确的 `alert.log` 文件时，检查其上的日期以确保它是最近更新的。有时你找到了一个 `alert.log`，但结果发现它不是当前文件。你可以使用 `ls` 命令的长格式列表来检查最后更新时间：

```
$ ls -altr alert_<SID>.log
```

一旦找到了正确的 `alert.log` 文件，检查其中最近的条目以查找错误，或者在与 `alert.log` 文件相同的目录中查找跟踪文件：

```
$ ls -altr *.trc
```

如果其中任何跟踪文件已经存在好几天了，考虑移动或删除它们。

## 删除文件

不用说，删除文件时要非常小心。试图解决问题时，最糟糕的事情就是让情况变得更糟。意外删除一个关键文件可能是灾难性的。对于你识别为候选删除的任何文件，考虑移动（而不是删除）它们。如果你有一个有可用空间的挂载点，将文件移动到那里，保留几天，然后再删除它们。

如果你已经确定了可以删除的文件，那么在实际删除它们之前，先列出你将删除的文件。至少，我会在删除任何文件之前执行此操作：

```
$ ls -altr <file_name>
```

查看 `ls` 命令返回的结果后，删除文件。此示例使用 Linux/Unix 的 `rm` 命令永久删除文件：

```
$ rm <file_name>
```

你也可以根据文件的年龄来删除文件。例如，假设你确定任何超过两天的跟踪文件都可以安全删除。通常，`find` 命令与 `rm` 命令结合使用来完成此任务。在删除文件之前，首先列出 `find` 命令的结果：

```
$ find . -type f -mtime +2 -name "*.trc"
```

如果你对文件列表满意，则添加 `rm` 命令来删除它们：

```
$ find . -type f -mtime +2 -name "*.trc" | xargs rm
```

在前面的代码行中，`find` 命令的结果通过管道传递给 `xargs` 命令，该命令为 `find` 命令找到的每个文件执行 `rm` 命令。这是一种基于文件年龄删除文件的高效方法。但是，务必非常确定你知道哪些文件将被删除。

另一个有时会占用大量空间的文件是 `listener.log` 文件。由于此文件被监听器进程持续写入，你不能简单地删除它。如果你需要保留此文件的内容，那么首先将其复制到有可用磁盘空间的备份位置，然后截断该文件。在此示例中，`listener.log` 文件首先被复制到 `/ora01/backups`，然后通过以下方式截断该文件：

```
$ cp listener.log /ora01/backups
```

接下来，使用 `cat` 命令将 `listener.log` 的内容替换为 `/dev/null` 文件（该文件包含零字节）：

```
$ cat /dev/null > listener.log
```

## 检查告警日志

在处理数据库问题时，`alert.log` 应该是你检查相关错误信息的首要文件之一。你可以使用操作系统工具或 ADRCI 实用程序来查看 `alert.log` 文件和相应的跟踪文件。

### 通过操作系统工具查看告警日志

导航到包含 `alert.log` 的目录后，你可以通过查看文件的末尾（最下面的部分）来查看最新的消息（换句话说，最新的消息写入文件的末尾）。要查看最后 50 行，请使用 `tail` 命令：

```
$ tail -50 alert_<SID>.log
```

你可以使用 `-f` 开关持续查看最新的条目：

```
$ tail -f alert_<SID>.log
```

你也可以直接用操作系统编辑器（如 `vi`）打开 `alert.log`：

```
$ vi alert_<SID>.log
```
