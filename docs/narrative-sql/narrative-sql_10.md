# 3. 使用 WHERE 过滤事实

`WHERE` 子句是 SQL 的一个关键特性。它指定了选择、更新或删除行所应满足的条件。由于此语句，从查询返回的记录会根据某些条件是否为真来进行过滤。这使你能够将搜索细化到较大数据集内的一个数据子集，从而只对符合特定条件的数据条目执行操作。

## WHERE 简介

如前所述，SQL 语句是构建查询的第一步。要求 SQL 执行特定任务的过程始于创建语句。如前所述，SQL 中的四个基本语句是 `SELECT`、`UPDATE`、`DELETE` 和 `INSERT`。`子句` 是 SQL 语句的一个组成部分，用于指定条件或修改语句的操作。可以使用子句来过滤、排序或分组数据。子句指定了检索或操作什么数据以及应如何处理这些数据。本章将讨论常用子句之一——`WHERE`。`WHERE` 子句就像一个过滤器，允许你仅从表中选择符合特定条件的特定记录。这对于缩小大型数据集范围以找到所需信息非常有用。

### WHERE 在数据叙事中的重要性

在有效的数据叙事过程中，`WHERE` 子句在基于数据的特定部分构建叙述方面起着基础性作用。该子句使叙事者能够聚焦于揭示模式的数据，并创建更有针对性、更具吸引力和信息量的叙述。

### WHERE 子句的剖析

在 SQL 语句中，`WHERE` 子句通常位于 `FROM` 子句之后。它可以包含多个通过逻辑运算符（如 `AND`、`OR` 和 `NOT`）组合的条件。`WHERE` 子句中的每个条件还可以使用函数和子查询来进一步细化过滤条件。这允许根据特定的叙事需求定制复杂和动态的数据检索策略。基本上，它具有以下语法：

```
SELECT column1, column2, ...
FROM table_name
WHERE condition
```

其中 `column1, column2, ...` 是你想要检索的列或字段，`table_name` 是从中检索数据的表的名称，`condition` 是行包含在结果集中的标准。此条件可以包括等于 (`=`)、不等于 (`<>` 或 `!=`)、大于 (`>`)、小于 (`<`) 等比较。表 3-1 总结了 `WHERE` 子句中最常用的比较运算符。它们对于通过比较列值与常量或列与列本身来基于特定条件过滤数据至关重要。

表 3-1

WHERE 子句的 SQL 比较运算符

| 运算符 | 描述 | 使用示例 |
| --- | --- | --- |
| `=` | 等于 | `WHERE age = 30` |
| `!=` | 不等于 | `WHERE age != 30` |
| `<>` | 不等于 | `WHERE age <> 30` |
| `>` | 大于 | `WHERE age > 30` |
| `<` | 小于 | `WHERE age < 30` |
| `>=` | 大于或等于 | `WHERE age >= 30` |
| `<=` | 小于或等于 | `WHERE age <= 30` |
| `BETWEEN` | 在包含范围内 | `WHERE age BETWEEN 25 AND 35` |
| `LIKE` | 搜索模式 | `WHERE name LIKE 'J%'` |
| `IN` | 匹配列表中的任意值 | `WHERE age IN (20, 30, 40)` |
| `IS NULL` | 当列为 `NULL` 时匹配 | `WHERE name IS NULL` |

## 子查询筛选
子查询允许你创建高度特定的筛选条件。子查询可以根据子查询的结果动态生成筛选条件来处理复杂筛选。子查询使你能够从多个角度探索数据。为了保持主查询的整洁和易于理解，你可以将复杂的筛选逻辑封装在子查询中。——例如，在查找销售额高于去年的产品时。平均值可以由子查询计算出来，然后使用该值进行筛选。

```sql
SELECT product_id, sales_volume, year
FROM sales_data
WHERE sales_volume > (
    SELECT AVG(sales_volume)
    FROM sales_data
    WHERE year = 2023
) AND year = 2024
```

本质上，此查询首先计算上一年的平均销售额，然后筛选本年度的销售数据，仅显示那些销售额超过该平均值的产品。假设 `sales_data` 表包含诸如 `product_id`（每个产品的唯一标识符）、`sales_volume`（特定年份每个产品的销售数量）和 `year`（销售数据的年份）等列，这个嵌套查询可以找到今年所有销售额高于去年平均值的产品。

## SQL 中使用 WHERE 子句的常见错误及避免方法

本节介绍了与使用`WHERE`子句相关的常见错误。使用`WHERE`子句进行查询时，需要仔细注意数据类型、空值和大小写敏感性。



### 数据类型问题

在 SQL 中，根据不同的数据类型过滤数据是一项常见任务。如果数据类型处理不当，`WHERE` 子句可能会导致错误或意外结果。

例如，假设一家公司希望根据员工的入职日期来筛选员工。员工数据存储在一个名为 `Employees` 的表中，其中的 `StartDate` 列是字符串类型而非日期类型。下面是导致错误的错误查询示例：

```
SELECT * FROM Employees WHERE StartDate = '2023-05-01'
```

比较日期的字符串表示可能会产生问题，特别是当日期的格式不同时。例如，`'2023-5-1'` 和 `'2023-05-01'` 会被视为不同的值，从而导致结果遗漏。

解决方案是在过滤之前将 `StartDate` 列转换为正确的日期类型：

```
SELECT * FROM Employees WHERE CAST(StartDate AS DATE) = '2023-05-01'
```

将 `StartDate` 转换为日期类型可确保比较准确且一致，防止潜在的不匹配情况发生。

> 注意
>
> SQL 中的 `CAST()` 函数用于将一个值从一种数据类型转换为另一种数据类型。在处理可能以一种格式存储但需要另一种格式处理或比较的数据时，这特别有用。`CAST()` 函数有助于确保数据类型在操作中正确匹配，防止错误并保证结果的准确性。`CAST()` 函数的基本语法如下：
>
> ```
> CAST(expression AS target_data_type)
> ```
>
> `expression` 是要转换的值或列，而 `target_data_type` 是您希望将表达式转换成的数据类型。

数值比较也存在问题。假设 `Salary` 列存储为字符串。

```
SELECT * FROM Employees WHERE Salary > '50000'
```

由于字符串比较的工作方式与数值比较不同，此查询可能无法返回正确的结果。一种解决方案是在过滤之前将 `Salary` 列转换为数值类型。

```
SELECT * FROM Employees WHERE CAST(Salary AS DECIMAL) > 50000
```

数据类型转换确保了数值比较能够正确执行，从而产生预期的结果。

> 注意
>
> `DECIMAL` 是 SQL 中用于存储定点数的数据类型，具有保证的精度和小数位数。与浮点数（例如 `FLOAT`）可能因内部表示而损失精度不同，`DECIMAL` 提供了十进制值的精确表示。十进制数据类型非常适合存储财务数据、测量值或任何精确十进制值至关重要的场景。`DECIMAL` 通常在括号内定义两个参数：精度（`p`）和小数位数（`s`）。
>
> *   精度（`p`）表示数字可以容纳的总位数，包括小数点前后的位数。
> *   小数位数（`s`）表示小数点右侧允许的位数。
>
> 例如，`DECIMAL(5,2)` 指定精度为五位数，允许的数字如 `123.45` 和 `-98.76`，它们在小数点后有两位数字。

### 条件中的逻辑错误

在 `WHERE` 子句中，逻辑错误会显著影响查询的性能。常见的逻辑错误包括 `AND` 和 `OR` 运算符的错误使用，以及对运算符优先级的误解。`AND` 和 `OR` 运算符用于组合条件以简化数据过滤。

考虑一个名为 `Members` 的表，它包含以下列：`MemberID`、`FirstName`、`LastName`、`MembershipType`（例如 `Standard`、`Premium`）、`MonthlyFee`、`JoinDate` 和 `Active`（指示会员资格是否当前有效）。

健身房的经理想要识别那些要么拥有高级会员资格，要么每月支付超过 40 美元的会员，但他们只想查看当前会员资格有效的会员。

```
SELECT *
FROM Members
WHERE MembershipType = 'Premium' OR MonthlyFee > 40 AND Active = TRUE
```

这里存在一个由于对运算符优先级理解错误而导致的逻辑错误。SQL 会先计算 `AND`，然后再计算 `OR`。因此，该查询被解释为：

```
SELECT *
FROM Members
WHERE (MembershipType = 'Premium') OR (MonthlyFee > 40 AND Active = TRUE)
```

这将返回所有高级会员，无论他们是否有效，以及任何月费超过 40 美元且有效的会员。由于包含了无效的高级会员，这个结果并不符合预期。

```
SELECT *
FROM Members
WHERE (MembershipType = 'Premium' OR MonthlyFee > 40) AND Active = TRUE
```

添加括号确保了遵循预期的逻辑。该查询现在筛选出要么是高级会员，要么每月支付超过 40 美元的会员，并同时检查他们是否有效。通过这种方法，获得了准确且预期的结果，同时过滤掉了无效的会员。

> 注意
>
> 编写 SQL 查询时可能发生两种类型的错误：语法错误和语义错误。
>
> *   **语法错误**：这些是 SQL 语句形式或结构中的错误。它们很容易检测到，因为 SQL 引擎会抛出指示问题的错误消息。例如，缺少逗号、关键字使用不正确或括号不匹配。
> *   **语义错误**：这些是逻辑错误，查询语法正确但未产生预期结果。语义错误很难检测，因为 SQL 引擎在执行查询时不会有任何错误消息，但返回的数据不正确或具有误导性。
>
> 特别是语义错误可能特别成问题，因为它们可能被忽视，导致不正确的数据分析和错误的决策。语法错误导致查询无法运行，而语义错误则导致有效的 SQL 查询产生不正确或意外的结果。

### NULL 值处理

在 `WHERE` 语句中处理 `NULL` 值是另一个常见的错误来源。`NULL` 表示未知或缺失的数据，涉及 `NULL` 的比较可能会产生意外结果。

在 SQL 中，`NULL` 不等于任何东西，甚至不等于另一个 `NULL`。因此，需要特殊处理来正确过滤 `NULL` 值。

考虑一个包含以下列的 `Employees` 表：`EmployeeID`、`FirstName`、`LastName`、`Department` 和 `PhoneNumber`（可能包含 `NULL` 值）。现在希望分析数据，筛选出没有电话号码的员工，即那些在申请表中未注明电话号码的员工。然而，此查询包含一个逻辑错误。

```
SELECT *
FROM Employees
WHERE PhoneNumber = NULL
```

此查询不返回任何结果，因为 `NULL` 不等于任何东西，包括另一个 `NULL`。比较 `PhoneNumber = NULL` 总是为假。以下查询包含正确的逻辑。

```
SELECT *
FROM Employees
WHERE PhoneNumber IS NULL
```

使用 `IS NULL` 可以正确识别 `PhoneNumber` 列为 `NULL` 的行。此查询返回所有未注明电话号码的员工。

> 注意
>
> SQL 中的 `NULL` 值表示缺失、未知或不适用的数据。它是一个空白字段，表示特定字段中不存在值。`NULL` 不等同于零或空字符串。它是一个表示值缺失的独特标记。`NULL` 不能使用标准比较运算符（例如 `=` 和 `!=`）直接比较。而是使用专门的运算符，如 `IS NULL` 和 `IS NOT NULL`。任何涉及 `NULL` 的算术运算或连接操作都会导致 `NULL`。SQL 提供了检查 `NULL` 值的特定语法：
>
> 检查 `NULL`：
>
> ```
> SELECT * FROM Employees WHERE PhoneNumber IS NULL
> ```
>
> 检查非 `NULL`：
>
> ```
> SELECT * FROM Employees WHERE PhoneNumber IS NOT NULL
> ```


### 大小写敏感性

一个常见的错误是假设字符串比较时不区分大小写。SQL 在不同数据库中的行为可能因大小写敏感性设置而异。PostgreSQL 默认是大小写敏感的。标识符（如表名、列名和数据值）会根据其大小写被区别对待。例如，`SELECT * FROM Users` 不会返回名为 `users` 的表的结果。

假设你需要查找一个姓氏为 `smith` 的用户。根据数据库的标记设置，以下查询可能会漏掉如 `'Smith'` 或 `'SMITH'` 这样的条目。

```sql
SELECT * FROM Employees WHERE LastName = 'smith'
```

解决方案是使用函数或调整排序规则设置来确保不区分大小写的比较：

```sql
SELECT * FROM Employees WHERE LOWER(LastName) = 'smith'
```

通过将 `LastName` 转换为小写，该查询能正确识别 `'smith'` 的所有变体。

注意

`UPPER` 和 `LOWER` 函数遵循相似的语法。需要牢记的是，结果的排序顺序可能会受到使用 `UPPER` 或 `LOWER` 的影响。由于大写字母通常在小写字母之前排序，排序后的顺序可能与原始的大小写敏感顺序不同。

## 总结

作为提升数据查询技能的一部分，本章探讨了 `WHERE` 子句的用法。通过理解 `WHERE` 子句及其相关函数，你可以编写更高效、更准确的 SQL 查询。

### 关键点

*   `WHERE`：根据指定条件筛选记录。
*   SQL 中的日期：处理日期和时间数据类型，允许进行精确的基于日期的查询。
*   `BETWEEN`：选择给定范围内的值，包括端点。
*   SQL 模式匹配：使用 `LIKE` 在文本字段中搜索模式。
*   `CAST`：将值从一种数据类型转换为另一种，以确保正确的比较和操作。
*   在 SQL 中使用 `WHERE` 时的常见错误：
    *   `数据类型`：确保数据类型之间的兼容性以避免错误。
    *   `NULL 值`：正确处理 `NULL` 值，因为它们表示未知或缺失的数据，可能会影响查询结果。
    *   `逻辑错误`：通过理解运算符优先级并使用括号，避免在组合使用 `AND` 和 `OR` 条件时出错。
    *   `大小写敏感性`：SQL 可能区分大小写，这可能会影响基于文本的条件和比较。

### 核心要点

*   `WHERE 子句`：对于根据指定条件筛选记录和实现精确数据检索至关重要。
*   SQL 中的日期：正确处理日期和时间数据类型对于进行精确的基于日期的查询和比较至关重要。
*   `BETWEEN 操作符`：用于选择特定范围内的值（包括端点），简化了范围查询。
*   SQL 模式匹配：`LIKE` 操作符允许使用模式搜索文本字段，从而更轻松地找到符合特定条件的记录。
*   `CAST 函数`：将值从一种数据类型转换为另一种，确保在 SQL 查询中进行正确的比较和操作。
