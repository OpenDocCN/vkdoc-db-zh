# 列名

我们之前讨论过什么构成有效的名称。通常，它们不应干扰 SQL 语言，并应避免使用空格或连字符等特殊字符，除非你想为此付出双引号的代价。

列名也应该是唯一的：显然，你不能有两个同名的列。

*没有特殊名称*。例如，将某物命名为 `id` 并不会使其成为主键。这取决于开发者来想出一个合适的名称。也没有特殊的位置。你可以按你喜欢的任何顺序排列你的列，但同样，这取决于开发者来想出一个合适的顺序。

许多开发者忘记（或不在乎）谁将使用数据库，因此你有时会得到非常晦涩或令人困惑的表名和列名。说真的，这没有借口。一个好的名称应该易于使用。

### 数据类型

在 SQL 中，有三种主要数据类型以及类型之间的一些变体。以下是其中一些示例：

| 数字 | 日期 | 字符串 |
| --- | --- | --- |
| `integer`, `decimal()`, `float` | `date`, `datetime` | `char()`, `varchar()` |

可能还有其他数据类型，例如 **Boolean**（真/假值）或 **BLOB**（二进制数据）。

还有进一步的变体。例如，你可以决定一个整数是范围有限的短整数，还是范围更大的长整数。你可以决定一个字符串的长度，或者一个小数的精度。以下是示例数据库中使用的一些类型：

| 类型 | 描述 |
| --- | --- |
| `INT` | 又名 `INTEGER`：一个整数，在此情况下范围在 -2,147,483,648 到 2,147,483,647 之间 |
| `DECIMAL(4,1)` | 又名 `NUMERIC`：这是一个具有固定小数位数的小数。其表示法可能令人困惑：它表示最多四位数字，*包括*一位固定小数位。还有具有可变精度的 `FLOAT` |
| `VARCHAR(60)` | 一个最多可延伸至 60 个字符长的字符串 |
| `CHAR(4)` | 一个固定为四个字符的字符串。如果你输入少于四个字符，它将用空格填充，这就是为什么应忽略尾部空格 |
| `BOOLEAN` | 一个值，要么是 `true` 要么是 `false`，以数学逻辑先驱之一乔治·布尔命名 |
| `DATE` | 不含时间的日期 |
| `TIMESTAMP` | 又名 `DATETIME`：一个包含时间（精确到秒的一小部分）的日期 |

注意

*   原则上，`VARCHAR` 更节省空间，而 `CHAR` 处理效率更高（DBMS 无需计算字符串长度，因为它已经知道）；实际上，现代 DBMS 无论如何都相当高效，因此这种区别不那么重要。
*   一些 DBMS 不支持布尔数据类型，因此只能用整数 `1` 或 `0` 来凑合。一些 DBMS 还支持一种称为 `BIT` 的非常短的整数。
*   注意邮政编码和电话号码是字符串，即使它们仅限于数字。这是因为它们并不是真正的数字：它们的数值没有意义；一个值并不真正大于或小于另一个值，甚至它们的顺序也无关紧要。

选择哪种数据类型部分取决于最正确的类型（出生日期*必须*是日期），部分取决于了解你在现实生活中可能遇到的情况（60 个字符长对于电子邮件地址是否足够？）。



### 主键

主键是表中每一行的唯一标识符。它不需要有特定的名称，也不一定是第一列。但如果你养成良好的设计习惯，处理起来会更容易。

原则上，任何具有唯一值的列（例如客户的电子邮件地址）都有资格作为主键。然而，一个好的主键应具备以下特点：

- 主键必须保证唯一。
- 主键永远不能是 `NULL`。
- 主键不太可能改变。

正是最后一个特点让电子邮件地址变得有点棘手。客户有可能更改他们的电子邮件地址，虽然数据库可以处理这种情况，但这确实会降低其可靠性。

这引出了下一点。通常，主键有两种类型：

- 自然主键是代表真实数据的主键。例如，客户的电子邮件地址或一个国家的唯一两字母或三字母代码。
- 代理主键没有内在价值——它是一个任意的代码，不提供关于数据的任何其他信息。

有些自然主键，如国家代码，永远不会改变，因此可能是良好的候选。但许多自然主键会改变，因此通常依赖于一个单独的代理键，它没有理由改变。

许多数据库开发人员为他们的主键使用各种名称，但它们通常都包含 `id` 的某种变体。实际上，`id` 经常被用作主键的宽松同义词。

真正定义主键的是在定义中添加 `PRIMARY KEY`。

一个表中只能有一个主键。然而，主键可以是复合的：由两列组合而成，而不是通常的一列。

有时，主键在 `CREATE TABLE` 语句的末尾定义：

```
CREATE TABLE something (
id INT,
--  等等
PRIMARY KEY (id)
);
```

通常不需要以这种方式定义主键，但这是定义复合主键的唯一方式。

SQL 允许你在表定义之后定义一个新的主键（只要还没有主键）。它还允许你向现有列添加主键，只要该列没有 `NULL` 值并且现有值是唯一的。

主键不一定是整数。例如，它可以是一个精心构造的字符串，只要保证它是唯一的。

然而，如果你觉得图省事，主键可以是一个简单的自动序列号。显然，许多数据库开发人员都喜欢省事，因为许多数据库使用自动序列号作为主键。

自动序列号最初并不在标准 SQL 中，因此每个数据库管理系统（DBMS）都自行设计了各自的实现方式。以下是一些变体：

| DBMS           | 自动序列实现方式                 |
|----------------|----------------------------------|
| PostgreSQL     | `GENERATED BY DEFAULT AS IDENTITY` |
| Oracle         | `GENERATED BY DEFAULT AS IDENTITY` |
| Microsoft SQL  | `IDENTITY(1,1)`                 |
| MySQL          | `AUTO_INCREMENT`                |
| SQLite         | `AUTOINCREMENT`                 |

SQL 标准是 `GENERATED BY DEFAULT AS IDENTITY`，PostgreSQL 和 Oracle 使用的就是这个。旧版本的 PostgreSQL 使用 `SERIAL`，而旧版本的 Oracle 则根本没有自动序列功能，只能凑合使用触发器（一种自动化机制）。

### 约束

正如我们提到的，约束是任何限制有效值构成的规则。一般来说，如果你试图输入一个违反约束的值，DBMS 会引发错误并拒绝该操作。

DBMS 并不关心数据在现实中是否正确，甚至是否合理。如果你在 `familyname`（姓氏）字段中输入你的出生地，那是你自己的事。DBMS 只关心数据是否有效；即，数据是否通过了约束检查。

第一个约束是数据类型本身。如果你将列定义为数字，那么你就不能在其中放入字符串。

对于 SQLite，请忽略上面这句话^(¹⁰)以及下面这句话。它允许在任何列中存放任何类型的数据。当你定义一列时，你设置的是一个“亲和性”，这是一种偏好，但它并不强制你遵守。

数据类型还包括额外的约束，例如字符串的长度或数字的精度。

除了数据类型之外，你还可以通过额外的约束来更好地控制数据。这些约束包括：

- `NOT NULL`：声明该列不能为空。
- `UNIQUE`：不允许列中出现重复值。
- `DEFAULT`：如果未指定值，则自动为新行提供一个值。
- `CHECK`：一条自定义规则。
- 此外，除了数据类型，你还可以通过指定该值必须匹配另一个表中的值来获得更多控制。这被称为外键。

在这里，我们将更详细地研究这些约束。

### NOT NULL

你可能遇到的第二个约束是 `NOT NULL`。它的意思如其字面所示：数据不能是 `NULL`；换句话说，始终需要一个值。

一个可能包含值的列有时被称为可为空的列。

一个列可为空，并不意味着你实际上不需要这个值。例如，除非你知道客户的地址，否则无法向其发货。允许为空仅仅意味着你可以稍后收集这些信息，而不必在添加行时立即提供。

带有 `NOT NULL` 约束的列看起来像这样：

```
familyname VARCHAR(24) NOT NULL
```

`NOT NULL` 约束可以与这里讨论的其他约束组合使用。

### UNIQUE

没有理由两个客户不能有相同的出生日期，但你不希望他们有完全相同的电子邮件地址。SQL 既不知道也不关心这意味着什么，所以你需要自己告诉 DBMS。

将一列标记为 `UNIQUE` 会使 DBMS 确保新的电子邮件地址不能与另一个客户的相同。它不关心谁先输入，后输入的那个会失败。

唯一约束的列不一定必须有值，除非它同时是 `NOT NULL`。只有实际值才会被比较是否唯一，而 `NULL` 值都会被忽略。

除了微软的产品。微软有一个怪癖，它确实会匹配 `NULL`，并且出人意料地会将两个 `NULL` 声明为相同，因此违反了唯一约束。值得庆幸的是，有一个解决方法，但这确实使得在可为空的列上添加唯一约束变得有点棘手。

带有 UNIQUE 约束的列看起来像这样：

```
title VARCHAR(24) CONSTRAINT uq_minipaintings_title UNIQUE
```

或者更简单些：

```
title VARCHAR(24) UNIQUE
```

第一种形式允许你为唯一约束命名。数据库中的所有对象，包括约束，都有一个唯一的名称。如果你不关心名称是什么，可以使用第二种形式，让 DBMS 编造一个；不过，这个名字可能不太好看。

约束的名称可以用于例如将来某个时候需要删除它的时候。

### DEFAULT

严格来说，这被认为是一种约束，尽管你可能会怀疑这是否正确。

有时，你可以假设一个值，除非有相反的值。例如，你可以假设一幅画的数量应该是 `1`，除非输入了另一个值。

如果你想实现一个默认值，你可以这样写：

```
quantity NOT NULL DEFAULT 1
```

这是两个不同的约束：第一个 `NOT NULL` 禁止 `NULL`，而第二个 `DEFAULT 1` 将在没有其他值时自动填入值 `1`。

在示例数据库中，没有这样做，这就是为什么允许出现一些 `NULL`，而你不得不使用 `coalesce()` 来强制赋值。

有时，默认值只是为了方便。有时，它可以作为一个安全的后备选项。




### CHECK

前面的约束相当宽泛，涉及 SQL 数据的一些基本属性。有时，你需要一个可以被简单地视为特定于你环境的约束。这有时被称为业务规则。

`CHECK`约束是一种通用的杂项约束，它允许你使用 SQL 能理解的任何内容。它的形式如下：

```
CHECK (condition)
```

前面的条件看起来完全像一个`WHERE`子句。本质上，它的意思是，如果你正在寻找匹配项（`WHERE …`），那么这一行就应该被包括在内。有了这个约束，所有值（当然`NULL`除外，它不是一个值）都应该匹配。

例如：

```
-- 匹配 4 位数字：
CHECK (postcode SIMILAR TO [0-9][0-9][0-9][0-9])
-- 出生日期不晚于（最晚等于）死亡日期：
CHECK (born<=died)
```

约束可以应用于单个列，如前面的第一个示例，但也可以应用于多个列，例如当你想比较它们时。

### 外键

外键也是一种约束。在这种情况下，它要求一个表中的值必须匹配另一个表中的值。它的形式如下：

```
column type REFERENCES table(column)
```

每列都必须有一个数据类型，并且被引用的列必须具有完全相同的类型。你会注意到前面的形式实际上没有使用术语“foreign key”。这是简短形式，还有一个使用该术语的较长形式。

请注意，外键是在引用其他表的表中定义的——从“子”表到“父”表——而不是相反。

正如你从`paintings`表中看到的，外键列不一定需要有值。与所有约束一样，外键要求只适用于实际值，而不适用于`NULL`。

如果你*也*要求一个值，你需要同时设置`NOT NULL`。你将在`sales`表中看到这一点，其中的`customerid`外键*必须*有一个有效值；以及在`saleitems`表中，`saleid`和`paintingid`列都必须有一个值；拥有一个不知道客户是谁的销售，或者一个不知道画作是什么的销售项目，都没有任何意义。另一方面，拥有一幅艺术家未知的画作是可以的。

与所有约束一样，SQL 不允许你包含无效数据。问题是，如果另一个表中的某一行被删除了，会发生什么？

例如，如果你试图删除一个艺术家会发生什么？如果没有画作引用这个艺术家，那么我们可以继续。另一方面，如果有一些画作，那么删除将导致画作引用一个无效（不存在）的艺术家，这将违反约束。

DBMS 永远不会允许无效数据，因此有三种可能的处理方式：

*   首先，它可以简单地禁止删除，直到你首先删除那些麻烦的画作。这是默认行为。
*   其次，它可以允许删除，但随后自动将`painting.artistid`外键设置为`NULL`。
*   第三，它可以允许删除，但随后自动删除那些违规的画作。这被称为`cascading`删除。

第一种选项是默认的，如果你想要其他选项，则需要指定它们。设置`NULL`是一个相对无痛的选项，所以如果你想要的话，这里可以考虑。

从另一个表中删除相关行是非常激烈的，并且可能是一个坏主意，尤其是在这种情况下，特别是因为画作也可能影响销售项目。

然而，对于销售和销售项目的情况就不同了。在这里，你可能会决定删除一笔销售，例如当销售尚未完成（尚未付款）时。在这种情况下，同时删除相关的销售项目是有意义的。

要启用级联删除，我们附加

```
ON DELETE CASCADE
```

当然，这又是另一个问题了，即你首先是否应该删除数据。
