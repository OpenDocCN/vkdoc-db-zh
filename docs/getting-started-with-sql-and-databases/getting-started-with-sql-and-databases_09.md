# 3. 数据过滤

`DBMS` 的主要作用之一是存储数据。另一个主要作用是检索数据。要检索数据，你使用 `SELECT` 语句。然而，你通常不想要所有数据，因此你会希望过滤结果。

在本章中，我们将探讨如何使用 `WHERE` 子句过滤你的请求。在此过程中，我们将涵盖重要主题，例如

*   `WHERE` 子句如何工作
*   处理缺失值
*   处理不同的数据类型：数字、字符串和日期
*   组合过滤器
*   使用“通配符”

如果你有百万行数据，你通常不希望得到百万行的结果。稍后，我们将研究如何汇总大量数据。现在，我们将看看如何获取*部分*数据。


## WHERE 子句

`WHERE` 子句用于将结果限制为符合特定条件的行。例如，如果你想将顾客列表限制为身高较矮的顾客，可以使用

```sql
SELECT *
FROM customers
WHERE height<170;
```

你应该会看到类似这样的结果：

| id | 姓名 | 名字 | … | 身高 | … | 注册日期 |
| --- | --- | --- | --- | --- | --- | --- |
| 186 | Gunn | Ray | … | 163.8 | … | 2021-11-15 |
| 523 | Sights | Seymour | … | 167.3 | … | 2022-07-11 |
| 351 | Tate | Dick | … | 167.8 | … | 2022-03-13 |
| 422 | Why | Wanda | … | 163.2 | … | 2022-05-05 |
| 121 | Ting | Lil | … | 162.8 | … | 2021-10-06 |
| 545 | Knife | Jack | … | 167.4 | … | 2022-07-24 |
| ~ 116 行 ~ |

`WHERE` 子句后面跟着一个**断言**——一个结果为真或假的测试。在这个例子中，测试是 `height<170`，只有满足这个条件为真的行才会被返回。

对于像这样简单的比较，你会用到在学校里学到的经典运算符。后面会有一个它们的表格。

你可以通过使用 `NOT` 运算符来反转断言：

```sql
SELECT *
FROM customers
WHERE NOT height<170;
```

这将只返回身高*不*小于 170 的行。

| ~ 132 行 ~`

在许多情况下，`NOT` 有另一种表达方式：

```sql
SELECT *
FROM customers
WHERE height >= 170;    --  或 height>=170
```

`>=` 周围的空格不是必需的；它们只是为了更清晰。

类似地，你可以测试精确匹配：

```sql
SELECT *
FROM customers
WHERE height=170;
```

结果不会有很多：

```sql
SELECT *
FROM customers

SELECT *
FROM customers
```

对于像前面例子这样简单的算术表达式，你不会经常看到 `NOT` 形式的用法。然而，对于其他一些表达式，它更自然。

### 与列无关的断言

`WHERE` 子句中的断言根本不必涉及表中的任何列。例如，以下都是可接受的：

```sql
SELECT *
FROM customers
WHERE 1=1;  --  所有行
SELECT *
FROM customers
WHERE 1=0;  --  无行
```

在第一个例子中，断言 `1=1` 对所有行显然为真，所以所有行都被返回。断言 `1=0` 从不成立，所以没有行被返回。

这些查询显然是无意义的，你可能永远不需要运行它们。然而，会有其他一些情况，你可能想测试断言本身，而不是获取实际数据。

### 全部与无

`WHERE` 子句的结果总是有效的，即使它不是你想要的结果。例如：

```sql
SELECT * FROM customers WHERE id>0; --  所有行
SELECT * FROM customers WHERE id<0; --  无行
```

一个断言可能导致返回所有行。这可能是一个无意义的断言，但它并非没有意义：下次，情况可能会不同。

一个断言可能导致根本没有行返回。同样，下次情况也可能不同。查询不存在的东西并不是错误；但如果你接着*假定*会有更多结果，那*可能*就是错误了。

请注意，在这种情况下，`id` 必须为正数并没有技术上的原因。这只是一个非常普遍的惯例。

### 处理 NULL

`WHERE` 子句返回断言为**真**的行。如果一个断言不为真，它*不一定*是**假**。另一种可能是它是未知的。

如果你计算前面例子的结果数量，你会发现身高小于 170 的顾客数量，加上身高不小于 170 的顾客数量，小于顾客总数。这是因为*部分*顾客没有记录身高值。

在 SQL 中，`NULL` 代表一个缺失的值。为了方便起见（也可能会造成困惑），它通常用单词 **NULL** 显示，通常带有不同的颜色或背景色。然而，实际上那里空无一物。

值可能缺失的原因有很多：

*   该信息不适用，例如当一个人尚未去世时的死亡日期。
*   该信息不可用，例如未提供的出生日期；你不能说这个人还没出生。
*   该信息被视为无关紧要，例如额外的送货说明。
*   不是上述原因，但我们就是不在意，所以没有输入。

不幸的是，SQL 不区分这些不同的原因，因此在解释如何处理缺失值时需要小心；从技术上讲，对于 `NULL`，你只能说数据缺失，尽管有时你可以从上下文中*推断*一些信息。

例如，死亡日期的缺失*本身*并不意味着一个人还没有去世。你需要更多的数据来确定。

有时，人们会提到“NULL 值”，但这在术语上是矛盾的。这很重要，因为 `NULL` *在涉及值比较时总是被跳过*。这就是为什么 `height<170` 的行与 `NOT height<170` 的行组合起来并不等于总行数：两个断言都只测试值，从不测试 `NULL`。在本例中，我们将直接使用 `NULL` 作为名词；有些人更喜欢“NULL 标记”这个表述。

你可以尝试用以下方法查找 `NULL`：

```sql
SELECT *
FROM customers
WHERE height=NULL;  --  不起作用
```

但它不会起作用。更准确地说，它会执行，但你不会得到任何结果。这是因为相等比较 `=` 只比较两个*值*。

这完全说得通。如果你不知道身高，那么你就无法说它与另一个值相比如何。如果两个身高都是 `NULL`，你不能说它们相等，因为你根本不知道它们是什么。

你可以在下面这个简单的例子中看到 `NULL` 比较：

```sql
SELECT *
FROM customers
WHERE NULL=NULL;
```

不会有结果，因为比较 `NULL` 总是为假。

#### 有意忽略 NULL

如果 `NULL=NULL` 总是为假，为什么 SQL 还允许比较 `NULL` 呢？更现实的情况是，你可能会比较两个列，而这两个列都可能包含 `NULL`。例如：

```sql
SELECT *
FROM artists
WHERE born=died;
```

不应该有任何结果，但你可能会得到几行：

| id | 姓名 | 名字 | 别名 | 出生年份 | 去世年份 | 国籍 |
| --- | --- | --- | --- | --- | --- | --- |
| 348 | Daumier | Honoré | daumier | 1879 | 1879 | French |
| 370 | Vermeer | Johannes | vermeer | 1675 | 1675 | Dutch |
| ~ 2 行 ~`

没有哪个在出生年份去世的人成为著名艺术家，所以任何匹配都表明有错误。然而，只有当*两个*值都存在且相同时，匹配才有意义。

在这里，你肯定希望为了比较而忽略 `NULL`。


### 查找 NULL

如何查找缺失值？SQL 提供了一个特殊的表达式 `IS NULL` 来查找 `NULL`：

```sql
SELECT *
FROM customers
WHERE height IS NULL;       --  身高缺失
```

这将返回所有身高缺失的记录：

| id | familyname | givenname | … | height | … | registered |
| --- | --- | --- | --- | --- | --- | --- |
| 474 | Free | Judy | … |   | … | 2022-06-12 |
| 377 | Money | Xavier | … |   | … | 2022-04-02 |
| 321 | King | May | … |   | … | 2022-02-11 |
| 46 | Ering | Hank | … |   | … | 2021-08-15 |
| 350 | Bea | May | … |   | … | 2022-03-12 |
| 500 | Mentary | Rudi | … |   | … | 2022-06-25 |
| ~ 56 行 ~ |

你也可以查找所有存在的身高值：

```sql
SELECT *
FROM customers
WHERE height IS NOT NULL;   --  身高已存在
```

这将返回其余的记录：

| id | familyname | givenname | email | … | registered |
| --- | --- | --- | --- | --- | --- |
| 186 | Gunn | Ray | ray.gunn186@example.net | … | 2021-11-15 |
| 144 | King | Ray | ray.king144@example.net | … | 2021-10-18 |
| 179 | Inkling | Ivan | ivan.inkling179@example.com | … | 2021-11-08 |
| 475 | Blood | Drew | drew.blood475@example.net | … | 2022-06-13 |
| 523 | Sights | Seymour | seymour.sights523@example.net | … | 2022-07-11 |
| 341 | Idate | Val | val.idate341@example.com | … | 2022-03-03 |
| ~ 248 行 ~ |

在第一个例子中，`IS NULL` 用于筛选缺失值。在第二个例子中，`IS NOT NULL` 用于筛选存在的值。你会发现，第二个例子的结果数量与将 `height<170` 和 `NOT height<170` 的结果相加是一样的。

你可以通过合并这些查询的结果来验证：

```sql
SELECT * FROM customers WHERE height<170
UNION
SELECT * FROM customers WHERE NOT height<170;
```

`UNION` 子句将两个或多个 `SELECT` 语句的结果合并，只要列匹配即可。稍后你会看到更多关于 `UNION` 的内容。

说到 `NOT`，SQL 也接受另一种表达式：

```sql
SELECT *
FROM customers
WHERE NOT height IS NULL;
```

这个表达式在英语中读起来不太自然，也许如果你的名字叫尤达大师，会觉得更自然些。不过，它确实有一个小优点，就是遵循了 `NOT` 在其他用法中的相同模式，即紧跟在 `WHERE` 关键字之后。这样，你的筛选条件要么是 `WHERE`，要么是 `WHERE NOT`（没有“可能”）。

### 数字

在前面的例子中，`height` 是一个数字，所以比较很简单。数字遵循所谓的 **金发姑娘三分法**：比较两个数字时，结果要么是太小，要么是太大，要么是正好。

要比较两个数字，你可以使用表 3-1 中所示的比较操作符。

表 3-1

比较操作符

| 操作符 | 含义 | 否定形式 |
| --- | --- | --- |
| `a = b` | 值相等 | `NOT a = b` ; `a <> b` |
| `a < b` | 小于 | `NOT a < b` ; `a >= b` |
| `a > b` | 大于 | `NOT a > b` ; `a <= b` |
| `a <= b` | 小于或等于（“不高于”） | `NOT a <= b` ; `a > b` |
| `a >= b` | 大于或等于（“不低于”） | `NOT a >= b` ; `a < b` |

注意 `NOT a < b` 与 `a > b` 不同，因为必须考虑相等的情况。

正如你稍后将看到的，这些操作符也适用于非数字比较，但需要赋予更恰当的含义。

#### 离散值 vs. 连续值

比较以下两个语句：

```sql
SELECT *
FROM artists
WHERE born >= 1700;
SELECT *
FROM artists
WHERE born > 1699;
```

它们当然会给你相同的结果，因为 `1700` 紧接在 `1699` 之后。

```sql
SELECT *
```

| --- | --- | --- | --- | --- | --- | --- |
| 144 | King | Ray | … | 176.8 | … | 2021-10-18 |
| 179 | Inkling | Ivan | … | 170.3 | … | 2021-11-08 |
| 475 | Blood | Drew | … | 171.0 | … | 2022-06-13 |
| 341 | Idate | Val | … | 177.1 | … | 2022-03-03 |
| 588 | Skies | Grace | … | 171.5 | … | 2022-08-13 |
| 326 | Todeath | Boris | … | 186.3 | … | 2022-02-15 |

这很可能会得到非常不同的结果，因为 `170cm` 并不紧接在 `169cm` 之后。与出生年份不同，以厘米为单位的身高可以涉及小数部分。

我们说出生年份的值是 **离散的**：它们是整数值，但没有中间值。另一方面，身高的值是 **连续的**：存在中间值。^(⁴)

对于离散值，你可以在前面两个例子之间做出选择，因为一个离散值肯定在下一个之前。然而，对于连续值，你没有这种选择。

有时，连续值可以被简化为离散值。例如，时间是连续的，但你可以将其简化为离散的日期。货币价值可能是连续的，但你可以通过忽略小数部分（例如，只考虑美元而不考虑美分）使其离散化。
