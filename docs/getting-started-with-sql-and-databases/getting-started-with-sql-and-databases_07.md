# 1. 从 SQL 开始

如果你对数据库一般而言是新手，特别是对 SQL 不太熟悉，一开始可能会有点令人畏惧。在本章中，我们将通过查看如何从一个简单的表中获取数据来初步体验 SQL。

原则上，你可以选择任何你想要的方式来管理你的数据，有些人确实只是在电子表格甚至文字处理器中这样做。但对于重要的、未组织的数据来说，这种方式不够充分，因此我们依赖于更结构化的东西。

组织数据没有唯一的方式，也并非所有数据都能以相同方式组织。然而，大多数时候，有一种流行的数据组织方式，SQL 语言就是为此开发的。管理数据库的软件通常被称为数据库管理系统（Database Management System），每次都说这么长有点麻烦，所以我们将简称其为 `DBMS`。

一个 SQL 数据库是一个或多个 `表` 的集合。每个表是一种类型数据的集合。示例数据库中的一些表有：

*   客户
*   绘画作品
*   艺术家

我们稍后会更仔细地查看数据库的结构，但目前重要的是数据没有混杂在一起。例如，`客户` 表包含了关于客户的所有信息，没有其他内容。

每个表包含 `行` 和 `列`。一行是数据的一个实例。例如，`客户` 表中的每一行代表一个客户。一列是行的一个详细信息。例如，`客户` 表有单独的列用于客户的电子邮件地址、电话号码等。

当然，数据库远不止这些，第 2 章将更深入地探讨这些概念。

在本章中，我们将使用 `SELECT` 语句（这是获取数据的基本命令）来探索一个表——即 `客户` 表——的内容。在此过程中，你将看到结果如何被过滤、重新计算和排序。

我们这里介绍的内容都将在后面的章节中更详细地讲解，所以对于本章的内容，你可以采取一种相对轻松的方式学习。

如果你运行下面的示例代码，你的结果可能不完全相同。这是因为示例数据是随机生成的，可能与书中使用的数据不同。你可能还会发现不同的 `DBMS` 在呈现未排序数据的方式上存在差异。

## 基本 SELECT 语句

要从表中读取数据，你使用 `SELECT`：
```sql
SELECT * FROM customers;
```
你将得到类似表 1-1 的结果。
表 1-1
*结果*
| id | 姓氏 | 名字 | 电子邮件 |   | 注册日期 |
| --- | --- | --- | --- | --- | --- |
| 474 | Free | Judy | judy.free474@example.net | … | 2022-06-12 |
| 186 | Gunn | Ray | ray.gunn186@example.net | … | 2021-11-15 |
| 144 | King | Ray | ray.king144@example.net | … | 2021-10-18 |
| 179 | Inkling | Ivan | ivan.inkling179@example.com | … | 2021-11-08 |
| 475 | Blood | Drew | drew.blood475@example.net | … | 2022-06-13 |
| 523 | Sights | Seymour | seymour.sights523@example.net | … | 2022-07-11 |
| ~ 约 304 行 ~ |

这被称为 `SELECT` 语句，它将从 `客户` 表中获取所有行。

语句通常由两个或更多 `子句` 组成。在本例中，它们是 `SELECT` 子句和 `FROM` 子句。

注意

*   `SELECT` 并不意味着 *显示*，尽管数据库客户端不知道还能对结果做什么。其他软件可能只是获取数据以便进一步处理。
*   `*` 并不意味着所有 *行*。它是表中所有 *列* 的简写。
*   我们将要做的大部分工作都会涉及 `SELECT` 语句。

#### 大小写敏感性

SQL 语言对大小写不敏感，意味着你可以用大写或小写输入语句：

*   传统上使用 `大写` 来突出显示关键字（`SELECT` 和 `FROM`），但当你有语法高亮时，这就不那么重要了。
*   传统上表名和列名也使用小写输入，但大多数 `DBMS` 并不太在意这一点。

```sql
select * from customers;
```
本书将对关键字使用 `大写`，但你不必这样做。

### 空格

虽然一个简单的语句很容易放在一行内，但你可以根据需要添加任意多的换行符和额外的空格或制表符：
```sql
SELECT
*
FROM customers;
```
最重要的是保持代码的可读性，并使用空格来帮助管理语句。

随着本书的进展，会有更多关于布局的建议。然而，这些只是建议，因为 SQL 在最少空格的情况下也能正常工作。

### 子句顺序

SQL 最初提议的名称是 `SEQUEL`，结构化*英语*查询语言。其想法是语法会类似于英语。

这导致了一个语法上的怪癖。例如，如果你说：
```
从冰箱里拿牛奶
```
你*首先*去冰箱，*然后*拿牛奶。也就是说，`从` 在 `拿` 之前执行。

同样，在 `SELECT` 语句中，`FROM` 子句在 `SELECT` 子句*之前*执行。

然而，你不能那样写语句：

*   你必须写成 `SELECT … FROM … ;`。
*   它的意思是 `从 … 选择 …`。

稍后，你将看到额外的子句以及它们应该放在哪里。

在这个简单的例子中，子句顺序的事实并不重要。然而，稍后子句顺序将解释为什么一些更复杂的例子不能像你预期的那样工作。

### 分号 (;

SQL 标准要求每条语句的末尾有一个分号 `;`。你可以将语句分成多行，但分号标记了语句的最终结束。

*   大多数 `DBMS` 会允许你在只有单条语句时忽略分号。然而，在多条语句之间，你至少需要一个分号。
*   Microsoft SQL 也允许你在多条语句时省略分号，除非它们在同一行，但即使是 Microsoft 也不认为这是一个好主意（参见 `https://docs.microsoft.com/en-us/sql/t-sql/language-elements/transact-sql-syntax-conventions-transact-sql#transact-sql-syntax-conventions-transact-sql`）。

我们建议你*始终*使用分号，即使是单条语句或对于 Microsoft SQL。这样，你可以确保你的代码不易出错。

## 选择特定列

星号字符 `*` 是选择表中所有列的简写。你可以指定一个或多个列来代替：
```sql
SELECT id, givenname, familyname
FROM customers;
```
这会给你表 1-2。
表 1-2
*结果*
| id | 名字 | 姓氏 |
| --- | --- | --- |
| 474 | Judy | Free |
| 186 | Ray | Gunn |
| 144 | Ray | King |
| 179 | Ivan | Inkling |
| 475 | Drew | Blood |
| 523 | Seymour | Sights |
| ~ 约 304 行 ~ |

这选择了三列；它仍然选择了所有行。

列列表由逗号分隔：

*   列不必按原始顺序排列。
*   你可以跳过任何你想要的列。
*   注意，在本例中，`名字` 和 `姓氏` 的顺序被调换了，并且电子邮件列被省略了。
*   逗号后面的空格是可选的：如果你认为它使代码更易读，就包含它。
*   逗号是一个分隔符：*不要* 在最后一列之后再放一个逗号，因为 SQL 会期待另一列。

这是一个常见的错误：
```sql
SELECT id, givenname, familyname, -- 多余的逗号
FROM customers;
```
始终指定列名是一个好习惯，即使你要选择所有列。



### 列顺序

默认的列顺序（即使用 `SELECT *` 时看到的顺序）是在表创建时定义的。即使你有权限，可能也无法更改它。

在 SQL 中，没有所谓“正确”的列顺序，以不同顺序选取列也不会带来性能差异。也就是说，不存在`首选的`列顺序，因此你可以根据自己的需求（无论是为了展示还是为了输入到另一个应用）来选择最合适的顺序。

### 布局

随着列列表的增长，更有创意地布局列是有意义的：

```sql
SELECT
id,
givenname, familyname
FROM customers;
```

如前所述，实际的间距并不重要，因此前面的示例使用间距来使语句更具可读性。

*   列列表是垂直的，而非内联的。
*   列列表从 `SELECT` 命令缩进，以表明它们属于同一个子句。
*   `givenname` 和 `familyname` 在同一行，以表明它们在概念上是相关的。

如果你记得使用 Tab 键，你会发现布局更容易维护。

同时，如前所述，你可以使用任何喜欢的间距；只需确保语句尽可能可读。

### 使用 SELECT *

在实际使用中，即使你真的需要所有列，使用 `SELECT *` 也被认为是**不良实践**；务必明确指定所有列名。这是因为

*   你无法控制结果的列顺序。
*   基础表结构的更改可能导致下次查询结果不同。

然而，在本书中，你会看到 `SELECT *` 被频繁使用：

*   `SELECT *` 是探索表的好方法。
*   许多示例将专注于新子句，因此实际选择的列无关紧要。

只需记住，当认真使用 SQL 时，你应该始终列出实际的列名。

### 计算列

选取的列不必是原始列。它们可以从一个或多个列`派生`出来。除此之外，这意味着表永远不需要保留一个值的变体，因为需要时总可以重新计算。

例如：

```sql
SELECT
id, givenname, familyname,
height,         --  高度，单位为厘米
height/2.54     --  高度，单位为英寸
FROM customers;
```

表 1-3 显示了结果。

表 1-3
*结果*

| 474 | Judy | Free |   |   |
| --- | --- | --- | --- | --- |
| 186 | Ray | Gunn | 163.8 | 64.488… |
| 144 | Ray | King | 176.8 | 69.606… |
| 179 | Ivan | Inkling | 170.3 | 67.047… |
| 475 | Drew | Blood | 171.0 | 67.323… |
| 523 | Seymour | Sights | 167.3 | 65.866… |
| ~ 约 304 行 ~ |

在 `customers` 表中，高度以厘米为单位测量。对于那些偏好传统度量单位的人，你可以通过除以 `2.54` 将其转换为英寸。^(¹)

设计一个同时包含厘米和英寸的列将是一个错误。表绝不应该包含本质上与另一个列相同的、经过伪装的列。如你所见，你总是可以重新计算出另一个值。

### 注释

在复杂的脚本中，包含关于代码功能的注释是很有用的。`注释`是任何会被 SQL 忽略、但旨在供人类阅读的文本。

你已经在前面的示例中看到了一些注释。标准的注释是位于 `--` 字符之后、直到行尾的文本：

```sql
SELECT
id, givenname, familyname,
height/2.54 as inches   -- 1in = 2.54cm
FROM customers;
```

前面的注释用于解释*为什么*我们要除以 `2.54`。

严格来说，`--` 后面必须跟一个`空格`。然而，大多数（但不是全部）DBMS 也允许使用 `制表符` 代替空格，并且有些（但不是全部）DBMS 甚至不要求空白字符：

```sql
-- 这是一个标准注释（使用空格）
--  这使用了制表符，可能有效，但不一定（使用制表符）
--这也可能有效
```

你很快就会发现哪些变体对你的 DBMS 有效。通常，注释会以不同的颜色高亮显示。

### 块注释

`大多数` DBMS 也支持一种非官方的块注释：

```sql
/* 块注释 */
```

这种风格也被称为 C 风格注释，因为它在 C 编程语言中被使用。

块注释以 `/*` 组合开始，以反向的 `*/` 组合结束。它可以跨越多行或仅占一行的一部分。

```sql
/*  这是一个介绍性的 SELECT 语句
本书的其余部分将进行更详细的介绍 */
SELECT
id,
/* 名称: */ givenname, familyname
FROM customers;
```

通常，你应该避免使用非标准的 SQL 特性，因为你永远不知道未来会怎样。然而，这一特性被广泛支持，以至于你可以将其视为仅仅是非官方补充的一个缺失功能。

### 注释的用途

由于 SQL 完全忽略注释文本，你可以随意书写，即使它是胡言乱语。然而，以下是注释的常见用途：

*   解释代码中不显而易见的内容
*   在复杂脚本中充当章节标题
*   临时禁用部分代码

以下是一个展示注释不同用途的示例：

```sql
/*  SQL 示例器
================================================
这是一个介绍性的 SELECT 语句
本书的其余部分将进行更详细的介绍
================================================ */
SELECT
id,
--  email,
givenname, familyname,
height/2.54 as inches   -- 2.54 厘米 = 1 英寸
FROM customers;
```

在前面的示例中，`email` 列被禁用，`inches` 列得到了解释，整个脚本前面有一个标题注释块。实际的查询也进行了缩进，以确保良好的可读性。

通常，如果你想禁用代码，直接删除它即可。使用注释来代替删除被称为**注释掉代码**。你可能会注释掉代码的原因包括：

*   测试或故障排除
*   将其保留在那里作为一个选项，留待进一步商议
*   将其用作其他代码的替代方案

至于解释性代码，**不要**过度注释。只解释不显而易见的部分。说得太多就像喊“狼来了”的男孩。通常，其他人只会选择忽略。


## 筛选行

通常，你并不想要表中的所有行，而只需要其中的一部分。`WHERE` 子句用于决定选择哪些行：

```sql
SELECT
id,
givenname,familyname,
height/2.54 AS inches
FROM customers
WHERE state='NSW';
```

这一次，你会得到表 1-5 中的内容。

表 1-5
*结果*

| id | 名字 | 姓氏 | 英寸 |
| --- | --- | --- | --- |
| 474 | Judy | Free | [NULL] |
| 144 | Ray | King | 69.606… |
| 341 | Val | Idate | 69.724… |
| 351 | Dick | Tate | 66.063… |
| 429 | Tom | Morrow | 61.772… |
| 234 | Nat | Ering | 67.638… |
| ~ 67 行 ~ |

表达式 `state='NSW'` 称为一个**断言**，其结果要么是 `true`，要么是 `false`。`WHERE` 子句只选择那些断言为 `true` 的行。

注意 `NSW` 周围的单引号 `' … '`。在 SQL 中，文本值被称为**字符串**，并用单引号括起来。不要使用双引号 `" … "`，因为大多数数据库管理系统（DBMS）会以不同方式解释双引号。另外，请注意字符串是大写的，这与 `customers` 表中的数据相匹配。在一些 DBMS 中，你也可以使用小写，但在另一些中则不行。

你将在本书后面学到更多关于字符串的知识。

### 子句顺序

`WHERE` 子句是在 `FROM` *之后*、`SELECT` *之前* 进行求值的：

```sql
SELECT …
FROM …
--  SELECT 在此处处理！
WHERE … ;
```

用英语来说，这表示：

1.  从表开始。
2.  筛选一些行。
3.  选择一些列。

然而，请记住，你必须按照前面的顺序编写 SQL。

### 分号的放置

在开发代码时，很容易犯以下错误：

```sql
SELECT *
FROM customers;
WHERE state='NSW'   --  哎呀
```

这是因为你在前一个版本的结尾正确地使用了分号分隔，然后只是在其后添加了一个新的子句。在开发代码时，将分号放在单独一行可能会有所帮助：

```sql
SELECT *
FROM customers
WHERE state='NSW'
;
```

这样在添加额外的子句时会更容易。当你完成所有操作后，你总是可以整理分号的位置。

## 对结果排序

从数学上讲，表是行的一个**集合**。除其他外，这意味着行的顺序是无关紧要的。

一些 DBMS 会按照数据添加的相同顺序输出结果。一些 DBMS 会以看似随机的顺序输出结果，这取决于数据在内部的管理方式。

SQL 标准特意*不*告诉 DBMS 如何完成其工作，唯一的保证是行的顺序*不*被保证，也就是说，除非你强制指定。

`ORDER BY` 子句将结果按指定的顺序排列：

```sql
SELECT
id,
givenname, familyname,
height/2.54 as inches
FROM customers
WHERE state='NSW'
ORDER BY familyname, givenname;
```

结果将显示在表 1-6 中。

表 1-6
*结果*

| id | 名字 | 姓氏 | 英寸 |
| --- | --- | --- | --- |
| 44 | Helen | Back | 67.913… |
| 162 | Ginger | Beer | 70.039… |
| 99 | Minnie | Bus | 65.315… |
| 270 | Mary | Christmas | 65.118… |
| 487 | Horace | Cope | 68.622… |
| 419 | Barbie | Cue | 62.520… |
| ~ 67 行 ~ |

在这个例子中，你按 `familyname` 排序结果，并且在姓氏相同时，按 `givenname` 排序。

你可以按一列或多列排序，可以是升序或降序。

严格来说，结果不再是一个集合，因为集合是无序的。在某些情况下，一旦使用了 `ORDER BY` 子句，你可能就无法再进行任何进一步的处理。

你将在后面学到更多关于 `ORDER BY` 子句的知识。

### 子句顺序

`ORDER BY` 无论是书写还是求值都是最后的：

```sql
SELECT …
FROM …
WHERE …
--  SELECT 在此处处理
ORDER BY … ;
```

用英语来说，这表示：

1.  从表开始。
2.  筛选一些行。
3.  选择一些列。
4.  最后，对结果排序。

然而，请记住，你仍然必须按照前面的顺序编写 SQL。

| SA | Richmond |
| ~ 79 行 ~ |

在这里，你将获得 `state` 和 `town` 的不同*组合*。在结果集中，去重的不是 `state` 也不是 `town`——而是它们的组合。可以说我们现在拥有了 `state`/`town` 的分组。

同样，你会看到 `NULL` 作为一个单独的分组。在这组数据中，没有不带城镇的州，反之亦然，这就是为什么只有一个包含 `NULL` 的分组。

## 总结

以下是我们一直在开发的 SQL 示例：

```sql
/*  SQL 示例
================================================
这是一个介绍性的 SELECT 语句
本书的其余部分将更详细地介绍
================================================ */
SELECT
id,
--  email,
givenname, familyname,
height/2.54 as inches   -- 2.54 cm = 1 英寸
FROM customers
WHERE state='NSW'
ORDER BY familyname,givenname;
```

这说明了 SQL `SELECT` 语句的主要部分，以及注释和布局的使用。

基本的 SELECT 语句是：

```sql
SELECT columns
FROM table;
```

注意 `SELECT` 是在 `FROM` *之后* 进行求值的。

### 编写 SQL

SQL 是一种简单的语言，有一些规则和一些关于可读性的建议。

*   SQL 对使用额外空格的要求比较宽松。你应该使用足够的空格，以使你的 SQL 更具可读性。
*   每个 SQL 语句以分号 (;) 结尾。
*   SQL 语言对大小写不敏感，列名也是如此。表名可能对大小写敏感，这取决于操作系统。

请记住，语言的某些部分是灵活的，但仍然必须遵循严格的语法。

### Columns

`SELECT`语句将从表中选择一列或多列数据。

*   你可以按任意顺序选择列。
*   `SELECT *`表达式用于选择*所有*列。
*   列可以是计算得出的。
*   计算列应使用别名命名；非计算列也可以使用别名。

请记住，在编写良好的 SQL 语句中，你不应该使用`SELECT *`来选择列。然而，在本书中，我们将使用它来专注于新子句的介绍。

### Comments

注释是提供给阅读者看的附加文本，会被 SQL 忽略。

*   SQL 有标准的单行注释：`-- etc`
*   大多数 DBMS 还支持非标准的块注释：`/* … */`
*   注释可用于解释某些内容或充当章节标题。它们也可以用于禁用部分代码，就像你在进行故障排除或测试时可能做的那样。

请记住，要谨慎使用注释，仅在它们确实能告诉读者需要知道的信息时才使用。

### Filtering Data

可以使用`WHERE`子句来过滤行。

*   有时，过滤结果可能没有任何行；这并非错误。
*   在过滤字符串时，值可能区分大小写，也可能不区分，这取决于 DBMS 和实际的数据库。

`WHERE`子句可用于返回已知是唯一的单行，或行的一个子集。偶尔，你可能得到没有任何匹配条件的行。

### Row Order

SQL 表是无序的行集合。

*   行的顺序是无关紧要的，并且可能出乎意料。
*   你可以使用`ORDER BY`子句对结果进行排序。

从技术上讲，一旦你使用了`ORDER BY`，结果就不再是一个真正的集合。通常，这无关紧要，但有些操作在`ORDER BY`之后无法执行。

### Clause Order

到目前为止介绍的四个主要子句是按以下顺序编写的：

```sql
SELECT columns
FROM table
WHERE conditions
--  SELECT 在此处被求值
ORDER BY columns
```

`SELECT`子句是在`ORDER BY`子句之前最后被求值的。
