# 内存优化列存储索引的权衡与优化策略

基于磁盘的执行计划建议使用覆盖索引以避免导致总计 1383 次读取的聚集索引扫描。未使用列存储索引的内存优化表使用索引查找来返回结果，而使用列存储索引的内存优化表则使用列存储索引扫描来返回相同的结果。

在这三个示例表中，列存储索引扫描是成本最低且速度最快的选项。尽管如此，维护内存优化列存储索引的成本并非微不足道。除了已经介绍过的限制之外，这种结构还受到多种限制：

*   不能使用`ALTER TABLE`在内存优化列存储索引上添加列、创建索引、更改数据类型或进行表初始创建后常见的许多其他 DDL 更改。
*   列存储索引出于必要性会被复制到磁盘，以帮助在服务器重启后加速数据库恢复。
*   与基于磁盘的表一样，混合聚集列存储和非聚集行存储索引会消除 SQL Server 自动应用 Vertipaq 优化的能力。
*   不允许在内存优化列存储索引上使用计算列。
*   由于内存优化列存储索引只允许作为聚集索引，因此不能对其应用筛选器。
*   索引维护极其有限。重建仅可用于调整哈希索引桶计数（如果需要），并且不允许重新组织操作。如果需要，可以使用`Sys.sp_memory_optimized_cs_migration`来处理增量存储的内容。
*   不允许在内存优化列存储索引上使用列存储归档压缩。

与内存优化列存储索引相关的注意事项清单意义重大，清楚地表明这不是一个可以在未经仔细考虑的情况下使用的特性。放置列存储索引的理想场景如下：

*   具有高并发性的 OLTP 表。
*   被确定为迁移到内存优化表的异常候选的 OLTP 表。
*   目标表上存在频繁的分析查询，导致大量扫描和不可接受的资源消耗。
*   OLTP 和 OLAP 操作无法拆分到独立的数据源中。

即使一个表满足这些条件，也务必进行彻底测试，并能够量化内存优化列存储索引的性能优势是否大于其限制和缺点。这是一个独特的架构特性，只适用于一小部分高流量、高并发的 OLTP 表。更多情况下，其他替代方案会更可行，例如：

*   具有聚集行存储索引和非聚集列存储索引的基于磁盘的表。
*   带有额外覆盖非聚集索引的内存优化表。
*   独立的 OLTP/OLAP 数据处理/存储系统。

### 优化策略

一般来说，优化应该从解决最常见用例的方案开始。只有在以下一种或多种情况成立时，才应调整和实现额外的复杂层：

*   其他解决方案未能提供足够的性能提升。
*   该情况可以归类为规则的例外或边缘情况。
*   新策略的测试和质量保证被证明能带来显著收益（在性能、资源节省或两者兼有）。

没有任何一组架构挑战存在放之四海而皆准的解决方案，因此在应用本章（或任何地方）介绍的技术时保持谨慎非常重要。保持简单，只有在简单方案失效时，才应考虑不那么简单的解决方案。
