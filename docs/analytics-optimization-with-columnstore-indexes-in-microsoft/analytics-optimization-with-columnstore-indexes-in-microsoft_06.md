# 在事务型数据库的 **OLTP** 表中存储 **OLAP** 数据

将分析数据存储在 **OLTP**（联机事务处理）表中很可能是一个陷阱。**OLTP** 表旨在处理频繁写入和读取小范围行的工作负载。在这些场景中，通常需要从这些行中检索许多列。这些表的设计是为了管理并发性，其理念是给定的事务只涉及少数行。

**OLTP** 表与分析数据的要求是相悖的。分析数据通常涉及针对许多行的聚合查询，但通常不需要返回所有列。

对于小型表（少于几百万行），**OLTP** 表足以应对分析数据。对于包含数百万或数十亿行的大型表，这些表在提供跨大范围的数据服务时会遇到困难。与任何其他分析解决方案相比，它们可能会消耗更多空间，并导致对其数据的查询速度更慢。这是因为它们是针对小范围查找（少数行，多列）进行优化的，其架构并非为应对分析所需的大规模行数和聚合而设计。

将 **OLTP** 表保留用于小型分析数据集，在这些场景下扫描表的大部分（或全部）数据不会造成过高的开销。或者，它们可用于查找表或其他小型维度表。例如前面清单 **1-3** 中的 `LogSource` 表使用 `SMALLINT` 来编号其行。其主键最多只能包含 65,535 个值（-32,767 到 32,767），因此该表注定是相对较小的。它足够紧凑，如果需要，对文本列建立索引也不会带来过高的开销。
