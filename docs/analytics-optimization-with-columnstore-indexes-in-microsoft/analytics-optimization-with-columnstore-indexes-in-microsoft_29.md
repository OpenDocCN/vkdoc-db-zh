# 压缩延迟

事务性数据通常在短时间内被频繁写入和读取，随后其数据访问频率会随着时间的推移而越来越低。新数据可被视为热数据，而稍旧的数据可被视为暖数据，更旧的数据则可被视为冷数据。尽管表本质上是事务性的，但绝大多数数据访问只涉及热数据。

例如，考虑一个在线订单跟踪系统。以下是订单创建、处理和完成所涉及的部分步骤摘要，以及关于数据访问的说明：

1.  订单被提交以购买一些很酷的潮品。数据被写入数据库。
2.  订单被处理。数据库中的数据被更新。
3.  商品被拣出并打包。数据库中的数据被更新。
4.  商品发货。数据库中的数据被更新。
5.  商品被追踪直至送达。每次物流更新时，数据库中的数据都被更新。
6.  商品送达。数据库中的数据被更新。
7.  用户对订单流程和商品留下反馈。
8.  用户未来通过订单历史列表以只读方式访问此数据。
9.  最终，用户很少/不再访问此数据。

此数据流清晰地表明了从热数据（步骤#1–6）到暖数据（步骤#7–8）再到变为冷数据（步骤#9）的演进过程。分析查询将最频繁地处理大段的历史数据，这在列表步骤#6 之后变得更为明显。追踪特定订单无疑是事务性需求，主要适用于步骤#1–5，因为这些步骤与想要知道订单位置和预计到达时间的人最相关。

列存储索引在越接近列表底部时性能最佳，而在列表顶部的频繁更新对于压缩行组来说将是负担。每次`DELETE`操作都会在行组中将一行标记为已删除并浪费空间。每次`UPDATE`操作将删除一行，并在增量存储中新插入一行。

为了应对这一挑战，可以在列存储索引上使用压缩延迟。此功能将数据从增量存储移动到压缩段的时间延迟一个预设的时间段。该时间段基于数据预计保持为“热”状态的时长。在压缩被延迟的这段时间内，无需担心`DELETE`或`UPDATE`操作对压缩行组的影响。相反，对未压缩的增量存储数据的读取速度将比数据最终被压缩后要慢。然而，对于一个大表来说，热数据的百分比将很小，而暖数据或冷数据将构成表的大部分。同样重要的是，聚集行存储索引将服务于事务性查询，使得这些进程可以避开增量存储中的数据。

考虑清单 12-3 中`Sales.Orders`表上的列存储索引示例。如果通常有一个 10 分钟的时间段，数据在此期间被频繁更新，之后数据开始冷却并减少修改，那么可以将压缩延迟添加到索引创建语句中，如清单 12-4 所示。

```
CREATE NONCLUSTERED COLUMNSTORE INDEX NCCI_Orders
ON Sales.Orders (OrderDate, CustomerID, IsUndersupplyBackordered, SalespersonPersonID)
WITH (COMPRESSION_DELAY = 10 MINUTES);
```

该索引最初的表现将与之前完全相同。区别在于数据写入增量存储时。任何插入增量存储的行在写入压缩行组之前将至少保留 10 分钟。如果没有压缩延迟，元组移动程序会定期将行从增量存储移动到压缩行组。有了此功能，它现在将至少等待 10 分钟再执行此操作。
