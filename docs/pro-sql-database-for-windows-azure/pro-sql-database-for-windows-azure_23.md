# 第 3 章

### 安全性

与大多数公司的其他系统相比，数据库环境可能是安全性方面最薄弱的环节，但银行等少数行业除外。原因是数据库被认为完全位于内部网络边界内，而内部网络被认为是安全的，通常无法从互联网直接访问。

随着`SQL 数据库`和大多数数据库即服务解决方案的出现，对数据库安全性的关注上升到了首位，主要有两个原因：您不再直接控制您的数据，并且数据可以直接从互联网访问。因此，充分利用`SQL 数据库`的所有功能并了解其局限性变得尤为重要。

## 概述

在深入探讨`SQL 数据库`的具体细节之前，让我们先看一个通用的安全框架，以评估数据库即服务可能对您产生的影响。以下讨论基于保密性、完整性和可用性（`CIA`）所封装的基本安全原则。这被称为`CIA 三元组`，是最被广泛接受的安全分类形式之一。`SQL 数据库`与传统的`SQL Server`安装相比，有其不同的优缺点，因此回顾`CIA 三元组`的每个领域，以了解如何应对其劣势并利用其优势，至关重要。

### 保密性

*保密性*是确保数据只能由授权用户访问的能力。它旨在保护您的数据免受窥探或无意泄露，通过使用多种技术实现，包括：

- **加密**。创建*密文*（加密信息），该密文可通过共享密钥或证书解密。
- **哈希处理**。生成无法解密的密文（通常用于密码存储）。
- **访问控制**。基于上下文信息控制对数据的访问。
- **身份验证**。控制谁可以访问数据库以及用户可以访问数据库中的哪些对象。
- **防火墙**。使用技术将网络连接限制在已知机器的列表内。

`SQL 数据库`提供了一些新功能，例如防火墙（如前所述）；然而，它目前尚不支持原生数据加密（例如透明数据加密 [`TDE`] 和字段级加密），这使得对其他保密技术的强调更为重要。

另一方面，`SQL Server`不提供原生防火墙（尽管可以使用`Windows`原生防火墙或购买第三方数据库防火墙），但它提供了强大的加密功能。最后，`SQL Server`和`SQL 数据库`都提供了哈希功能。

由于`SQL 数据库`不提供原生加密，您的代码需要完成所有繁重的工作。别担心！在本章中，您将看到如何使用`C#`实现哈希和加密，以及如何将密文存储在`SQL 数据库`（或就此而言，`SQL Server`）中。

### 完整性

数据*完整性*是指确保信息只能由授权用户修改，并在必要时可以撤销损坏的目标。数据的完整性可能以多种方式受到威胁，例如恶意的`SQL 注入`攻击，或无意中在表上执行`TRUNCATE`语句，从而擦除所有记录。

一般来说，您可以在数据库中实施以下完整性措施：

- **授权**。控制谁可以更改什么数据。
- **备份**。创建事务一致的数据库快照，以便从中恢复数据。
- **基于角色的访问**。为公司中的不同角色（例如开发人员和支持人员）提供最低限度的访问权限。
- **审计**。跟踪数据库访问和数据更改，为取证分析提供审计跟踪。

从完整性角度来看，`SQL 数据库`尚未提供与`SQL Server`相同的功能。
