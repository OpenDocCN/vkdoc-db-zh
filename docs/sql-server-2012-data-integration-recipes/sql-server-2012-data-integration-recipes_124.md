# 为了更好地理解数据剖析的必要性

为了更好地理解数据剖析的必要性，请考虑 DBA 或开发人员经常面临的 ETL 场景。一个文件或某个数据的链接出现在你的电子邮件中，你需要“集成它”。可能有一些（有限的）文档，而且文档和数据都可能——也可能根本不可信。对数据进行剖析，进行属性和关系分析，可以为你节省大量时间，因为它使你能够：

*   **发现数据中的异常和矛盾**。
*   **向提供数据的人提出更准确、更深入的问题**。
*   **向你公司或部门的人（他们确信自己刚刚给了你一根解决所有数据需求的魔杖）解释数据可以如何被实际使用，以及它如何（或不能）集成到你的目标数据库和数据仓库中**。

作为 ETL 过程一部分的数据剖析可能因以下任何原因而至关重要：

*   维护数据质量指标的跟踪记录。
*   在数据质量超出定义的阈值时向你发出警报。
*   使你能够开发更精细的数据质量分析，其中质量的渐变（相对于简单的通过或失败）可以发展。这些分析可以在加载例程期间或之后作为警报发送。

一旦你有了这些分析，那么下一个问题应该是“这为什么重要？”嗯，答案将取决于你的具体数据集。如果所讨论的列包含对后续处理至关重要的数据（例如客户 ID），那么你可能希望为`NULL`设置一个阈值（绝对值或百分比），超过该阈值就停止处理。或者，如果此列中的`NULL`将导致 ETL 例程后期的进程失败，那么单个`NULL`可能就足以使数据流无效。关键在于，现在获取数据可能只需要几秒钟。在经过五六个小时的处理后才发现错误，真是浪费时间。
