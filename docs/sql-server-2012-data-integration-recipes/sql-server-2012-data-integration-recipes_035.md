# 使用 OPENROWSET 和 Schema.ini 导入文本文件

在以下情况下，您应使用此方法：
*   当您有一个结构一致的分隔符 CSV 或文本文件时。
*   当源数据文件不是特别大时——否则，您最好使用 `BULKINSERT`，如配方 2-10 所述。
*   当已启用即席查询时（参见配方 1-4）。
*   当源文件结构清晰（所有列都有逗号分隔的标题）时，您应该能够使用类似下面所示的代码片段将源数据返回到 SQL Server。

因为 `OPENROWSET` 被设计为 T-SQL 编程的组成部分，用于读取——并加载——分隔符数据文件到 SQL Server，所以您可能期望它允许使用 T-SQL 变量来定义其各种参数。不幸的是，情况并非如此。使用 T-SQL 变量的唯一方法是结合 `OPENROWSET` 使用动态 SQL。这意味着整个命令将是一个字符串变量，其他 T-SQL 变量在需要的地方插入。这可能很快变得极其笨拙且难以调试（记住所有那些带引号的字符串）。然而，如果您记住 `OPENROWSET` 本质上是为即席数据请求设计的，那么您可以将这个限制视为对良好编程实践的一种推动。在这些示例中，我使用了 `SELECT...INTO`。没有理由阻止您使用 `INSERT INTO...SELECT` 代替。

## 提示、技巧和陷阱

*   源数据文件必须完美无瑕（或至少被 Microsoft 文本驱动程序视为无错）才能正常工作。
*   可以使用 T-SQL 如配方 1-4 所述，对列名进行别名设置，并对数据进行子集筛选。
*   如果您使用的 `WHERE` 子句涉及文本值，则需要用双引号将文本括起来。
*   第一行中的列标题是可选的，但如果没有列名，文本驱动程序将使用第一行作为列标题。要解决这个问题，请参阅如何使用 Schema.ini 文件，如配方 2-6 所述。
*   如果需要，`SELECT` 语句可以包含 `ORDER BY`。
*   必须启用即席查询，如配方 1-4 所述。
*   ADODB 提供程序 MSDASQL 可以正确读取 CSV 文件，这与配方 2-9 中描述的 BCP 实用程序不同。
*   对于需要指定分隔符和分隔符的复杂源数据文件，您需要使用 `BULK INSERT`，如配方 2-10 所述。
*   如果第一行中有任何 `NULL` 字段，您将收到以下错误消息：
    ```
    Msg 492, Level 16, State 1, Line 1 (更多信息已省略)
    ```
*   通过 `OPENQUERY` 和 `OPENROWSET` 获得的结果集中不允许有重复的列名。在这种情况下，您必须手动向源文本文件添加列名（对于千兆字节大小的文件，这可能很困难甚至无法实现），或者使用模式信息文件，如配方 2-6 所述。
*   由于 MSDASQL 驱动程序和 ACE 驱动程序都有 64 位版本，因此在 64 位环境中两者都可以工作。
*   我假设您将在操作系统服务器上运行此类导入——可能使用远程桌面连接。根据我的经验，Vista 会导致 MSDASQL 出现权限问题。这些问题的快速解决方案是以管理员身份运行 SSMS。
