# 工作原理

SSMA 是一款功能非常强大的工具，但其帮助你将 Access 数据库升级到 `SQL Server` 的方式也非常灵活。正如我们在简明示例中所见，这些方式包括：根据 `SSMA` 中的元数据修改 `SQL Server` 表、迁移单表数据、生成单表或多表报告、为项目应用自定义数据类型映射、使用 `SSMS` 为目标表编写 `T-SQL` 脚本，以及查找 Access 数据库。

正如你所见，`SSMA` 创建的 `SQL Server` 表本质上是经过 `SQL Server` 数据类型适配后的源 Access 表。然而，你**并非必须**接受 `SSMA` 建议的数据类型或字段长度，完全可以随心所欲地进行修改。请注意，执行此类操作将导致旧版本的表被删除并重新创建，所有数据都需要重新加载。`SSMA` 根据久经考验的映射方案转换 Access 数据。但是，如果你愿意，可以为整个项目修改此映射。

`SSMA` 最有价值的功能之一是其能够提前预警潜在问题。你可以选择留意、忽略——甚至质疑该报告的明智性——但提前了解数据迁移可能在何处失败通常很有帮助。

存在一些已知的迁移问题；不过，其中大多数不适用于简单的数据迁移，仅在进行完整的 `SQL Server` 后端（搭配 Access 前端）升级时才会成为问题。

有趣的是，许多潜在问题与使用 Access 升级向导时遇到的问题相同。因此，我建议你首先查看前面在配方 1-10 中给出的提示——特别是那些关于如何为无故障升级准备 Access 数据库的有效方法。

你还可以通过研究几个经典问题领域（如表 1-5 所示）来节省时间和避免麻烦。

表 1-5. SSMA 问题领域

| 问题 | 解决方案 |
| --- | --- |
| Access 对象名称可能是 `SQL Server` 关键字。 | Access 和 `SQL Server` 有不同的保留关键字，因此你需要仔细检查 Access 是否使用了某个保留字，从而导致 `SQL Server` 报错。 |
| 字段和表名 | Access 字段和表名可以包含通常认为对 `SQL Server` 不可接受的字符。你应该修改 `SSMA` 生成的 `T-SQL`，然后用它来创建目标表。 |
| 日期 | `SQL Server` 的 `DATETIME` 类型仅接受 1753 年 1 月 1 日 至 9999 年 12 月 31 日 的日期范围。然而，Access 接受 1000 年 1 月 1 日 至 9999 年 12 月 31 日 的日期范围。请将 Access 数据字段映射到 `SQL Server 2012` 中的 `DATETIME2` 字段。 |

请记住，如果你面临一个可能棘手的 Access 数据库需要加载到 `SQL Server`，创建该数据库的副本并着手在复制的 Access 数据库中解决任何问题可能会更容易——这可能包括移除索引和约束、重命名表和字段等。根据我的经验，这通常是快速将源数据导入 `SQL Server` 的最快方法。

#### 提示、技巧与陷阱

*   你不能更改字段名称，因为 `SSMS` 的一个主要功能是将数据和元数据迁移到 `SQL Server`，同时将 Access 保留为数据接口。更改字段名称会使接口无法工作，因此——不能更改字段名称。
*   如果你确实必须更改字段名称，并希望将 `SSMA` 用作元数据脚本工具，那么你必须使用 `SSMA` 生成的 `T-SQL` 脚本，并在 `SSMS` 中运行它们之前进行调整。是的，你必须将每一个脚本单独复制并粘贴到 `SSMS` 中。
*   虽然你必须记住点击 `Apply` 来完成对一个表的所有修改，但你只需与数据库同步一次。只需记得右键单击 `SQL Server` 元数据窗口中的数据库名称来执行此操作。
*   如果你要复制数据的目标表不为空，你会看到一个警告对话框。如果继续操作，该表将被截断。
*   每个报告都保存在项目目录下的名为 "report" 的子目录中。
*   你也可以添加和移除类型映射——但在真实场景中我不建议这样做。
*   为 `Default Project Settings` 执行此操作将更改所有未来 `SSMA` 项目的数据类型映射。
*   要创建没有索引或主键的表，只需选择 `CREATE` 语句的代码，不要复制 `ALTER TABLE` 语句。
*   `SSMA` 无法迁移使用工作组保护的数据库。你必须在 `SSMS` 使用前移除工作组保护。我建议你查阅 Microsoft 网站以获取最佳操作方法。

#### 总结

本章包含了许多将 Office 数据导入 `SQL Server` 的方法。其中一些可能看起来相似、重复，或者简直令人费解。因此，为了让你有一个更清晰的概览，表 1-6 展示了我对各种方法及其优缺点的总结。

表 1-6. 本章建议的技术

| 技术 | 优点 | 缺点 |
| --- | --- | --- |
| `OPENROWSET` | 使用传递查询。 | 设置可能比较棘手。 |
| `OPENDATASOURCE` | 允许使用 `T-SQL` 操作源数据。 | 设置可能比较棘手。 |
| 链接服务器 | 允许使用 `T-SQL` 操作源数据。 | 设置可能困难。实践中通常非常慢。 |
| 导入和导出向导 | 易于使用。可以生成 `SSIS` 包。 | 转换选项有限。 |
| `SSIS` | 完整的数据转换、修改和操作。支持增量数据加载。 | 实现时间较长。无法一次性创建所有目标对象。 |
| `SSMA` | 迭代式开发。选项广泛。可以一次性创建所有目标对象。 | 转换选项有限。不支持增量数据加载。不支持部分数据选择。 |
| Access 升级向导 | 易于使用。可以一次性创建所有目标对象。 | 速度非常慢。转换选项有限。不支持增量数据加载。 |

那么，哪种解决方案最适合特定的挑战取决于许多因素，例如你开发解决方案的可用时间、对弹性和日志记录的需求——或者仅仅取决于需求的性质。当你的需求只是从 Excel 一次性加载少量记录时，你可能不会尝试去开发一个惊天动地的 `SSIS` 包。相反，自动化导入/导出向导是不可能的，因此你不会想用它来进行日常数据加载。

当你需要执行定期数据加载时，数据加载技术的选择可能会变得更棘手。基于 `T-SQL` 的解决方案之一（`OPENROWSET`、`OPENDATASOURCE` 或链接服务器）可以大大简化部署。在这种情况下，你很可能只是将一个存储过程复制到生产服务器和/或执行一些简单的 `T-SQL` 代码片段。然而，如果你希望能够跟踪数据加载的所有细节，那么毫无疑问，`SSIS` 及其多种多样的日志记录可能性（在本书最后一章中阐述）是应选择的工具。

因此，“视情况而定”可能是“我应该采取哪种方法？”这个问题的答案。从本章要记住的主要一点是，有多种工具可用，你需要了解它们的可能性和局限性。这样，你就可以在特定情况下选择并应用最合适的技术，从而为你特定的挑战获得最佳解决方案。
