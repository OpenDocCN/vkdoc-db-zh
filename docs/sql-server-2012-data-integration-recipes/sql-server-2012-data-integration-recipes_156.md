# 批量插入与索引优化注意事项

*   ***必须指定 `TABLOCK` 提示***。这是因为如果未指定 `TABLOCK`，并发线程会相互阻塞。
*   ***并行化最适合用于非重叠的数据源***——即那些任何“关键”字段在每个源文件中都完全分离的文件。
    如果你正在使用 `BULK INSERT`、BCP 或 `INSERT ... SELECT * FROM OPENROWSET(BULK...)`，并且表为空且具有聚集索引（同时数据文件中的数据按照聚集索引键列的相同方式排序），你还必须执行以下操作：
    *   在聚集索引已就位的情况下批量导入数据。
    *   指定 `ORDER` 提示以及 `TABLOCK` 提示。
    如果目标表为空，这应该比先删除聚集索引、加载数据，然后再重新生成聚集索引更快，因为不需要排序步骤。

#### 提示、技巧与陷阱

*   创建索引可能相当耗时，并且（特别是创建聚集索引时）可能消耗大量系统资源。因此显而易见的方法是尝试并行运行索引操作。这在 SSIS 中可以通过使用包含一系列执行 SQL 任务的序列容器轻松处理，每个任务处理一个或多个索引操作。我唯一的建议是不要过度这样做，当然也不要设置太多并行任务，因为磁盘子系统很可能在你获得任何更短的处理时间之前就达到饱和。我的建议是尝试一两个并行任务，一次扩展一个，然后计时结果。实际上，在并行重建或重组索引时，我也建议在单个进程中处理单个表的所有索引，不要使用多个线程重新处理同一个表的索引。
*   如果你删除了二级索引，可以尝试通过从不同的客户端运行每个二级索引的创建来并行重新创建它们。然而，这可能很棘手。
*   如果你导入的 ***少量*** 新数据相对于现有数据量而言，删除并重建索引可能适得其反。重建索引所需的时间可能比在批量操作期间节省的时间更长。
*   你可以通过使用以下命令来避免锁升级：
    ```sql
    ALTER TABLE CarSales SET (LOCK_ESCALATION = DISABLE)
    ```
*   如果启用了跟踪标志 610，对具有索引的表进行插入通常是日志最小化操作。具有聚集索引的表支持多个批量加载流并发插入数据，前提是这些流包含非重叠数据。如果检测到数据重叠，数据流将会阻塞——但不会死锁。
*   对于具有索引的表，不应使用 `BATCHSIZE = 0`——请设置一个批大小，因为批量导入操作使用大批次大小往往表现更好。唯一的注意事项是，如果目标上有多个索引，由于排序要求，这可能会增加内存压力。此外，在并行加载中（未使用 `TABLOCK` 提示），大的批大小可能导致阻塞。

#### 总结

我希望本章让你深入了解了用于减少 ETL 过程加载数据时间的众多技术。对效率的探索集中在高效使用 SQL Server 批量插入 API 上，进而学习以最佳方式管理目标表索引和约束，以确保应用了日志最小化。

然而，优化技术如同需要克服的不同挑战和需要解决的问题一样多。我只能建议你在面临严格的 SLA 时退后一步，全面评估问题。始终 **aim for** 批量加载，尝试确保日志最小化到位，并使用你能找到的最高效的驱动程序/提供程序。将源数据缩小到 **essential**，并避免任何多余的列。

如果你确定不会截断源数据，甚至可以缩小数据类型。通常，在源数据库中排序数据，并要求源文件预先排序（然后向 SSIS、BCP 或 BULK INSERT 指明这一点）更有效。

最重要的是，不要 **anything as a given**——测试，再测试，反复测试。

## ETL 索引策略

在定制 ETL 过程时，值得仔细思考你的索引策略。本章展示了以下方法：

*   禁用除聚集索引外的所有索引。然后在目标表和/或暂存表上重建所有索引。
*   删除所有索引（聚集、非聚集和任何其他索引）。然后，加载数据，最后重新创建所有索引——以正确的顺序进行，避免浪费时间和服务器资源。

其他潜在策略包括：

*   禁用除聚集索引外的所有索引。然后删除任何聚集索引并加载数据。最后，创建任何聚集索引并重建其他索引。
*   禁用除聚集索引外的所有索引。然后删除任何聚集索引并加载数据。接着，创建任何聚集索引。可能在更晚的时候，重建对 ETL 包优化运行至关重要的任何其他索引，留下用户索引以离线方式重建。
