# 工作原理

好吧，最终，您尝试了所有其他方法，并得出结论：`ODBC`是唯一剩余的解决方案。是的，它可能很慢（至少通过`ADO.NET`提供程序运行时），而且对某些数据类型可能很笨拙，但归根结底，这可能是唯一的出路。有趣的是，微软已表明`ODBC`是未来，因此这种连接被降级为“局外人”地位的时代显然已经过去。

由于特定的`ODBC`连接（如用于`MySQL`和其他关系型数据源）在其他地方有详细描述，本指南是对如何创建和使用`ODBC`连接管理器的一个简明通用概述。由于每个`ODBC`驱动程序都有其特定的配置选项，您必须了解如何设置每一个；因此请准备好深入研究源文档和您能找到的任何其他有用参考资料。这意味着出于必要，我在解释`ODBC`驱动程序的配置技术时必须保持非常通用。

我想大多数时候您会使用`ODBC 管理工具`创建`DSN`。

在 32 位或 64 位系统上，该工具位于`%SystemRoot%\system32\odbcad32.exe`。诚然，这是一个令人惊讶的命名约定。如果您在 64 位机器上需要 32 位版本，则需要查找`%SystemRoot%\Windows\SysWOW64\odbcad32.exe`。

我倾向于创建系统`DSN`而不是用户`DSN`。原因如下：

*   因为它们不特定于运行该进程的用户，所以减少了一个潜在的故障源。
*   可能更重要的是，因为用户`DSN`没有注册表重定向，并且`ODBC 管理工具`同时显示 32 位和 64 位`DSN`，您最终可能尝试使用一个在您所用环境中无法工作的`DSN`。

同样值得注意的是，在 64 位服务器上从控制面板运行`odbcad32.exe`将会运行 64 位版本的`ODBC`连接管理器。

#### 提示、技巧与陷阱

*   如果您在`SSIS`中创建基于连接字符串的`DSN`，`ODBC 管理工具`无论如何都会在磁盘上创建一个`.dsn`文件。如果您不想将其存储在磁盘上，必须手动删除此文件。
*   `ODBC 管理工具`显示 32 位和 64 位驱动程序的用户`DSN`。然而，这并不意味着 64 位应用程序会使用 32 位驱动程序，因为用户`DSN`没有注册表重定向。
*   系统`DSN`提供重定向。因此，32 位驱动程序和 64 位驱动程序的系统`DSN`是分开保存的。64 位`ODBC 管理工具`不显示由 32 位`ODBC 管理工具`创建的系统`DSN`。同样，32 位`ODBC 管理工具`也不显示由 64 位`ODBC 管理工具`创建的系统`DSN`。
*   因此，在 64 位系统上，建议您尽可能同时安装 32 位和 64 位版本的驱动程序。
*   在`SSIS 2005`和`2008`中，不要尝试通过在连接管理器窗格中右键单击，然后选择`新建连接`，再选择`ODBC`来设置`ODBC`连接管理器。这将不起作用，因为它不是“托管”连接！
