# 第 15 章：日志记录与审核

设计、开发和实施复杂的数据集成系统始终是对你的知识和创造力的考验。然而，当系统似乎准备投入生产时，挑战并不会停止。由于没有 ETL 系统可以做到完美，任何工业级的 **process** 都需要能够优雅地处理错误。不仅如此，它还必须能够告诉你发生了错误，并且能够决定是否可以继续进行数据加载。它还必须能够在进程运行时跟踪其进度，并主动返回你需要的所有信息和指标，以便在进程运行期间和（更可能）运行后监控进程。这意味着收集以下数据：

*   运行了什么，以及何时运行的。
*   什么成功了，以及何时成功的。
*   什么失败了，包括何时以及为何失败，大多数时候这包括错误源、错误消息、错误代码和错误的严重性。

同样重要的是在进程的每个重要阶段保存指标（计数器）——用于错误行和成功处理的行。

除此之外，你还需要另一个基本指标——进程所有重要部分的耗时。想要这些数据有三个主要原因：

*   如果进程失败，尽快发现并纠正错误。
*   在进程运行后，获得一个易于访问的高层次概览，并关注任何潜在的警告信号。
*   构建进程期间收集的 **essential** 指标随时间变化的比较。这些比较将让你能够看到——在积累的数据的冷酷清晰光线下——进程的哪些部分花费了更多时间或资源。因此，你可以主动解决任何潜在问题。

然而，在许多情况下，仅仅将进程从头到尾运行是不够的。然后你需要验证结果，并对于最终的数据集，生成以下（以及其他）计数器：

*   新记录
*   已更新记录
*   关键最终数据集中的 `NULLS`
*   核心数据的计数器

以及以下任何或所有内容：

*   在暂存表中添加、删除或更新的记录数计数器，以便与过程计数器进行交叉引用，并确保没有遗漏或多余的内容。
*   元素比较（如果你从外部供应商获取数据，他们可能会发送有关发送的数据量和类型的信息，你应该将其与实际收到的数据进行比较）。
*   关键数据项的单元测试（暂用此术语），以验证可接受的数据计数水平以及 **essential** 领域计数器和百分比。
