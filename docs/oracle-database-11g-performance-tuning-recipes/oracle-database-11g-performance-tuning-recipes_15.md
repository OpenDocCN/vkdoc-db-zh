# 第 7 章

### 6-10. 终止资源密集型进程

### 问题

你已经识别出一个消耗大量系统资源的进程（参见配方 6-9），并确定它是一个需要被终止的失控 SQL 语句。

### 解决方案

有三种基本方法可以终止 SQL 进程：

*   如果你能访问正在运行 SQL 语句的终端，可以按下 `Ctrl+C` 尝试终止进程。
*   确定会话 ID 和序列号，然后使用 SQL 的 `ALTER SYSTEM KILL SESSION` 语句。
*   确定操作系统进程 ID，并使用 `kill` 实用程序来停止进程。

如果你碰巧能访问运行资源密集型 SQL 语句的终端，可以尝试按下 `Ctrl+C` 来终止进程。通常你无法访问终端，必须使用 SQL 语句或操作系统命令来终止进程。

#### 使用 SQL 杀死会话

如果它是一个 Oracle 进程，并且你已经确定了 `SID` 和 `SERIAL#`（参见配方 6-9），你可以在 SQL*Plus 中终止进程。以下是通用语法：

```sql
alter system kill session 'integer1, integer2 [,integer3]' [immediate];
```

在上面的语法语句中，`integer1` 是 `SID` 列的值，`integer2` 是 `SERIAL#` 列的值（来自 `V$SESSION`）。在 RAC 环境中，你可以选择为 `integer3` 指定实例 ID 的值。实例 ID 可以从 `GV$SESSION` 视图中检索。

以下是一个终止 `SID` 为 1177、`SERIAL#` 为 38583 的进程的示例：

```sql
SQL> alter system kill session '1177,38583';
```

如果成功，你应该会看到此输出：

```
System altered.
```

当你杀死一个会话时，这会将会话标记为已终止，回滚活动事务（在会话内），并释放任何锁（由会话持有）。会话将保持在终止状态，直到所有依赖事务被回滚。如果回滚事务需要一分钟或更长时间，Oracle 会将该会话报告为“标记为要终止”，并将控制权返回给 SQL 提示符。如果你指定 `IMMEDIATE`（可选），Oracle 将回滚任何活动事务并立即将控制权返回给你。

#### 使用操作系统 kill 命令

如果你能访问数据库服务器，并且拥有可以终止 Oracle 进程的操作系统帐户权限（例如 `oracle` 操作系统帐户），你也可以直接使用 `kill` 命令终止进程。

例如，假设你运行了 `ps` 命令，并已经完成相关工作，确定有一个已运行数小时并需要被终止的 SQL 语句。`kill` 命令直接终止操作系统进程。在此示例中，进程 ID 6254 被终止：

```bash
$ kill -9 6254
```

### 工作原理

有时你会发现自己处于需要终止挂起的 SQL 进程、数据库作业或消耗大量资源的 SQL 语句的情况。例如，你可能有一个测试服务器，其中一个作业已经运行了几个小时，消耗了大量服务器资源，需要被停止，以便其他作业能够及时继续处理。

手动杀死 SQL 语句将导致事务被回滚。因此请谨慎操作。确保你正在终止正确的进程。如果你错误地终止了关键进程，这显然会对应用程序和相关数据产生不利影响。
