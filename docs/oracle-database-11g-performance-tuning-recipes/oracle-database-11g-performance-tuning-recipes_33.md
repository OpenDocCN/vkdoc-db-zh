# 工作原理

当您使用 DBCA 创建数据库时，自动优化器统计信息收集默认是启用的。如果您已禁用自动统计信息收集，可以通过执行“解决方案”部分所示的过程来重新启用它。一旦启用自动统计信息收集，数据库会在统计信息变得过时时收集它们——数据库根据对表和索引所做的更改来判断这一点。自动化统计信息收集消除了您自己收集统计信息所涉及的所有工作。

当您启用自动优化器统计信息收集时，“auto optimizer stats collection”任务会调用 `DBMS_STATS.GATHER_DATABASE_STATS_JOB_PROC` 过程。该过程与 `DBMS_STATS.GATHER_DATABASE_STATS` 过程几乎相同——最大的区别在于数据库仅在您指定的维护窗口内收集统计信息。如果您未指定维护窗口，数据库将使用默认的维护窗口，该窗口在每个晚上 10 点至早上 6 点以及周末全天开放。“auto optimizer stats collection”作业首先为最需要新统计信息的对象收集统计信息。因此，自动统计信息收集作业将首先为没有任何统计信息的对象，或经历了大量修改（通常是大约 10%的行）的对象收集统计信息。这样，如果统计信息收集作业在维护窗口结束前未能完成，数据库可以确保最过时的统计信息对应的对象一定会被刷新。

您可以查询 `DBA_OPTSTAT_OPERATIONS` 视图以了解自动统计信息收集作业的开始和结束时间，如下所示：

```sql
SQL> select operation,target,start_time,end_time from dba_optstat_operations
2*   where operation='gather_database_stats(auto)';

OPERATION                      START_TIME                       END_TIME
---------------------          ----------------------------     ----------------------------
gather_database_stats          26-APR-11 10.00.02.970000 PM     26-APR-11 10.03.11.671000 PM
…
SQL>
```

![提示](img/square.jpg) 提示：自动统计信息收集对于数据日常变化程度适中的 OLTP 数据库效果非常好，但对于大多数执行夜间 ETL 作业数据加载的数据仓库则不然。

`DBA_TAB_MODIFICATIONS` 视图存储了关于表插入、删除和更新的信息。默认情况下，`GATHER_DATABASE_STATS` 过程的 `OPTIONS` 参数被设置为 `GATHER AUTO`。这意味着一旦启用自动统计信息收集，数据库将为所有行受到 `insert`、`delete` 和 `update` 操作影响超过 10%的表收集统计信息。

数据库的自动统计信息收集对大多数 OLTP 数据库效果良好。然而，在数据仓库环境中，您可能会遇到问题，因为自动统计信息收集作业在夜间维护窗口运行。如果您的 ETL 或 ELT 作业在自动作业已收集完某张表的统计信息之后才将数据加载到该表中，您最终可能得到该表不具代表性的统计信息。在数据仓库环境中，在加载过程完成后立即手动收集统计信息并禁用默认的自动统计信息收集作业是一个好主意。

请注意，自动统计作业除了收集所有模式对象的统计信息外，还收集字典统计信息（针对 SYS 和 SYSTEM 模式）。
