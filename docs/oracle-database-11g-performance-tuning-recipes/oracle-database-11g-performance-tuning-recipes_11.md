# 工作原理

当 Oracle 服务器处理缓冲区高速缓存中的数据块时，这些更改会首先以重做日志条目的形式写入重做日志，然后才被写入磁盘。重做日志条目使数据库能够通过各种操作（如 `INSERT`、`UPDATE` 和 `DELETE`）以及 DDL 操作来重做或重建数据库更改。因此，Oracle 重做日志对于任何数据库恢复都至关重要，因为正是这些重做日志条目使数据库能够应用从过去某个时间点开始对数据库所做的所有更改。然而，更改的数据不会直接进入重做日志；Oracle 首先将更改写入一个名为重做日志缓冲区的内存区域。你可以使用 `LOG_BUFFER` 参数来配置这个内存缓冲区的大小。Oracle 日志写入器（LGWR）进程将重做日志缓冲区条目写入活动的重做日志文件（或文件组）。每当缓冲区填充三分之一时，或者当数据库写入器请求 LGWR 写入重做日志文件时，LGWR 会将缓冲区的内容刷新到磁盘。此外，在每次服务器进程执行 `COMMIT` 或 `ROLLBACK` 时，LGWR 进程都会将缓冲区的内容写入磁盘上的重做日志文件。

重做日志缓冲区是一个可重用的缓存，因此当条目被写出到重做日志文件时，用户进程会将新的条目复制到重做日志缓冲区中。虽然 LGWR 通常工作得足够快，使得缓冲区中有空间，但更大的缓冲区总是有更多的空间容纳新条目。由于增大 `LOG_BUFFER` 大小没有任何成本，因此可以将其设置为高于该参数建议的最大值 1MB。

如果你的数据库正在处理大型更新，即使没有 `COMMIT` 语句，LGWR 也必须频繁地将重做日志缓冲区刷新到重做日志文件中，以保持缓冲区填充不超过三分之一。在这种情况下，增加重做日志缓冲区的大小是一个可以接受的解决方案，并允许 LGWR 跟上大量进入重做日志缓冲区的条目。如果你认为 LGWR 进程的性能不够快，这也在某些方面抵消了缓慢的 I/O 系统的影响。有几种方法可以减轻重做日志缓冲区的压力：你可以对所有批处理作业批量执行 `COMMIT` 操作，并在可能的情况下（例如在常规数据加载期间）指定 `NOLOGGING` 选项。当你在数据加载期间指定 `NOLOGGING` 选项时，Oracle 不需要使用重做日志文件，因此也会绕过重做日志缓冲区。

调整 `LOG_BUFFER` 参数的大小相当容易。只需执行以下语句来获取当前的“重做日志空间请求比率”：
```sql
SQL> select round(t.value/s.value,5) "Redo Log Space Request Ratio"
     from v$sysstat s, v$sysstat t
     where s.name = 'redo log space requests'
     and t.name = 'redo entries'
```
重做日志空间请求比率只是总重做日志空间请求与重做条目的比率。你也可以查询 `V$SYSSTAT` 视图以查找统计信息 `redo buffer allocation retries` 的值。此统计信息显示进程等待重做日志缓冲区空间的次数：
```sql
SQL> select name,value from V$SYSSTAT
     where name= 'redo buffer allocation retries';
```
在一段时间内多次执行此 SQL 查询。如果“重做缓冲区分配重试”统计信息的值在此时间段内稳定上升，则表明重做日志缓冲区正面临空间压力，因此进程正在等待将其重做日志条目写入重做日志缓冲区。如果你持续看到这种情况，就必须增加重做日志缓冲区的大小。

## 第四章：监控系统性能

监控系统和数据库性能是一项复杂的任务，管理性能可能涉及许多方面，包括内存、磁盘、CPU、数据库对象和数据库用户会话——这仅仅是个开始。本章重点介绍如何使用 Oracle 的自动工作负载存储库（AWR）来收集中有关数据库中发生活动的数据，并帮助将这些原始数据转换为有用的信息，以评估特定时间段内数据库的性能。通常，当数据库中出现性能问题时，很容易知道性能问题何时发生，因为在此给定时间段内数据库活动“缓慢”。知道这个时间段是使用 AWR 信息执行分析的起点。

当你创建 Oracle 数据库时，AWR 会自动创建，并自动收集数据库中活动的统计数据。其中一些数据是实时或接近实时的，有些数据则代表这些活动的历史统计信息。关于活动会话的最新数据存储在性能统计信息存储库的活动会话历史（ASH）组件中。更早的历史数据快照通常称为 AWR 快照。

AWR 进程默认从 Oracle 的动态性能视图捕获每小时快照，并将其存储在 AWR 中。这使得 DBA 能够查看任何时间段内的数据库活动，无论是单个小时的时间段，还是 AWR 中存储的所有活动。例如，如果你有一段时间数据库性能较差，你可以生成一个 AWR 报告，该报告将仅提供该特定时间段内的数据库活动统计信息。

ASH 组件旨在为 DBA 提供更实时的会话信息视图，这些信息不会捕获在 AWR 快照中。ASH 存储库中存储的会话信息是每秒从动态性能视图采样一次的数据。

Oracle 多年来通过其前身 UTLBSTAT/UTLESTAT 和 Statspack 在数据库中拥有类似的信息，但直到 Oracle10g 推出 AWR 后，报告数据才会自动生成或保存。现在，这些信息可以更轻松地监控数据库性能，无论是分析数据库中当前活动的实时数据，还是分析历史数据（根据数据库中 AWR 的配置，这些数据可能已有数天、数周甚至数月之久）。

### 4-1. 实施自动工作负载存储库（AWR）

#### 问题

你希望为调优目的在数据库中存储历史数据库性能统计信息。

#### 解决方案

通过在数据库中实施和使用自动工作负载存储库（AWR），Oracle 将基于间隔的历史统计信息存储在你的数据库中以供将来参考。此信息可用于查看给定时间段内数据库中发生的情况。默认情况下，应启用数据库中的 AWR。需要验证的关键初始化参数是 `STATISTICS_LEVEL` 参数：
```sql
SQL> show parameter statistics_level

NAME                                 TYPE        VALUE
------------------------------------ ----------- ------------------------------
statistics_level                     string      TYPICAL
```
此参数可设置为 `BASIC`、`TYPICAL`（默认值）和 `ALL`。只要参数设置为 `TYPICAL` 或 `ALL`，就会为 AWR 收集统计信息。如果参数设置为 `BASIC`，你只需修改参数即可开始为数据库收集 AWR 统计信息：
```sql
alter system set statistics_level=TYPICAL scope=both;
```
