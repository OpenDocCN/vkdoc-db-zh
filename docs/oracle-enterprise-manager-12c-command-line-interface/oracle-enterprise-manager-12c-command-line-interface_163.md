# 验证命令行中是否传入了 SID
if [ ${#1} -eq 0 ]; then
   if tty -s; then
      read thisSID?"请提供数据库名称："
  else
      echo "命令行未提供数据库名称"
      ExitCleanly
   fi
   read thisSID?"请再次输入要监控的数据库名称："
else
   thisSID=`print $@ | awk '{print$NF}' | tr '[A-Z]' '[a-z]'`
fi

BO_NAME=scripted_blackout_${thisSID}

CleanUpFiles
EndBlackout
ExitCleanly
```

## 第 2 节：EM CLI 与 Veritas Cluster Server

Veritas Cluster Server (VCS)¹ 是一种健壮的故障切换机制，部署为 RAC One-Node 的替代方案。通过 VCS，关键资源（如数据库实例）会被识别和监控。如果检测到故障，VCS 将卸载包含二进制文件和数据库文件的驱动器，在集群的另一个节点上重新挂载它们，并重启服务。

OEM 代理必须安装在集群内每个主机的静态驱动器上。它们从不随 VCS 资源组进行故障转移，因为它们严格与单一主机关联。这给 VCS 故障转移带来了一个问题。当数据库、其 Oracle 主目录及其所有文件移动到另一台主机时，你如何跟踪目标？

![Image](img/sq.jpg) **提示** 许多人认为在 VCS 上将代理安装在虚拟主机名上是一种最佳实践。详情请参阅“如何在虚拟主机名 / 虚拟 IP 地址上安装 12c 代理？”（文档 ID 1469995.1）。

当目标因 VCS 而重新定位时，EM CLI 的 `relocate_targets` 动词会将目标跟踪的责任转移给另一个代理。本节将描述此过程的工作原理，然后展示 shell 脚本本身。

### 程序逻辑

此脚本从环境中收集数据，并决定哪些目标在移动，哪些代理受到影响。该脚本的可扩展性和灵活性源于对三个系统的查询：VCS 集群、OEM 存储库和 EM 代理。请注意，为数据库和监听器目标在代理之间转移责任的语法略有不同。我们将为生成实际执行工作的 EM CLI 参数文件生成一组单独的工作文件。

### Veritas Cluster Server

VCS 集群被组织成组，这些组向下细分到各个数据库和监听器目标。集群中的主机处于最高层级。VCS 心跳监控活动节点上的一个或多个关键资源，并在必要时将组的所有成员² 从当前 ONLINE 节点重新定位到当前 OFFLINE 的另一个节点。我们将利用这一知识来确定需要管理的主机和代理名称。

由于 EM 代理只能监控运行于同一主机（或 Oracle 集群）上的目标，我们需要仔细确定哪些目标与正在故障转移的 VCS 组相关。这些数据库和监听器目标是唯一有资格通过 EM CLI 重新定位的目标。我们将使用第二组 VCS 查询结果来决定哪些 EM 目标需要重新定位。

例如，假设有两个 Oracle 安装，每个都有自己的 Oracle 基础文件系统。当然，每个基础目录或主目录包含支持其自身目标所需的所有文件，并且可能有许多数据库和监听器使用这些主目录。它们都是同一个故障转移组的成员。假设这些组名为 QUAL 和 TEST。

*   QUAL 组有一个名为 `alexis_q` 的数据库和一个名为 `LSNRQ` 的监听器。
*   TEST 组有三个数据库 `alexis_t1`、`alexis_t2` 和 `alexis_t3`，以及一个共享的监听器 `LSNRT`。
*   `Alexis_q` 数据库被确定为 QUAL 集群组的关键资源。
*   TEST 集群组中的所有三个数据库都被确定为该组中的“关键资源”。

如果 VCS 没有收到针对 `alexis_q` 的心跳响应，QUAL 集群组中的所有资源将故障转移到另一个节点。

如果 TEST 节点中的任何一个数据库变得无响应，该节点上的所有内容，包括健康的数据库，都将由 VCS 在另一个节点上重新启动。

VCS 重新定位通常只需要一两分钟，在此期间资源组的健康成员（共享二进制文件的其他数据库及任何相关的监听器）被关闭，文件系统被卸载然后重新挂载，最后所有资源被启动。无需人工干预。

### Oracle Enterprise Manager

两台主机上的数据库和监听器可能在 Enterprise Manager 中被忽略、删除或根本未被发现，因此在 OEM 中无需更改任何内容。我们将检查 EM 存储库和本地代理，以决定哪些目标需要重新定位。³

在此示例中，只有数据库 `alexis_q` 和 `alexis_t1` 在 OEM 中配置了监控。当然，EM 代理知道所有数据库，但 OEM 监控和指标收集仅适用于前两个及其各自的监听器。就 OEM 而言，我们可以完全忽略其他两个数据库。重新定位完成后，新节点上的代理将发现 `alexis_t2` 和 `alexis_t3` 作为其自身主机发现的一部分。另一个节点上的代理也会定期搜索新目标，因此它基本上会忘记未被监控的数据库。EM CLI 命令将转移其关于 `alexis_q` 和 `alexis_t1` 的历史记录、指标收集和首选凭据的知识。毕竟，知识转移才是此操作的目的。
