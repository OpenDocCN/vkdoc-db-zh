# 7. 内容至关重要

## 妥善的内务管理

推荐的最佳实践是保持内务管理的条理性——当然是以会计师的方式。记录谁在什么时候做了什么。

为所有对象类型跟踪以下信息是很好的做法：

*   谁创建了这条记录？
*   记录是何时创建的？
*   谁最后一次更改了这条记录？
*   最后一次更改的日期和时间是什么时候？
*   这条记录是如何加载的？
*   记录是何时加载的？
*   记录的源系统是什么？
*   版本控制日期是什么（起始和结束）？
*   以及更多……

如果此类元数据在数据源中可用，我相信很多人会受益于在 API 中提供它们。其中一些信息可以在解析器层面添加。

## 标量数据类型

我们何时需要关注数据类型？

如果你还没有考虑过，现在是思考*API 层面*数据类型的时候了。GraphQL 的简单标量类型是预先定义好的。只需为意外情况做好准备。你需要考虑需要高精度的大型数字的精度和表示形式。对于日期和时间戳，你必须定义自己的自定义类型。如果有高级业务需求，你需要了解这一点。请参阅下一节关于处理时间的内容。

物理数据类型不太可能完全相同，但希望它们是兼容的（并且可以在解析函数中映射）。类型转换也是如此。

## 呈现日期和时间

日期和时间是数据库中的一些棘手挑战，在 GraphQL 中也是如此。GraphQL 是一个相对较新的框架，并且仍在发展。在撰写本文时，`Date`可以被实现为自定义标量类型（`DIY`）。GitHub 上有许多项目正在这个领域开发优秀成果。例如，GraphQL ISO Date^(²⁶)，网址为 [`https://github.com/excitement-engineer/graphql-iso-date`](https://github.com/excitement-engineer/graphql-iso-date)。

如你所知，日期和时间遵循一些为时区管理制定的、稍显不规则的国际规则。它们并非系统性的，无法轻易从字段值中推导出来。

时区确实难以处理。日历稍微容易一些，但仍然受制于许多国家和文化的差异。如果你拿一个普通的美国二月份日历，你会注意到有两个在美国以外不适用的特殊日子。情人节正逐渐进入欧洲，但没有总统日。

如果你的数据模型需要支持全球运营，你可能需要为每个事物至少使用四个变量：

*   本地日期（纯日期）
*   本地时间（不涉及日期）
*   全球日期（纯日期）
*   全球时间（例如，设置为 GMT）

关于在解析器中解决日期和时间问题的更多细节，请参见相关章节。
