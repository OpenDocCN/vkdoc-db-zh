# 9. 从图到树

### API 级别的结构设计

模式定义位于服务器端。你在应用侧看到的是模式设计所暴露的结果。

GraphQL 列表修饰符是生成结果树子分支的一种简单方法。在我们的`Email`示例中，我们有很多列表。例如，标记消息的关键字：

```
keywords: [Keyword] @relation(name: "Tags")
```

列表可以处理空值，这在 GraphQL 中有语法上的可视化表示：

*   数据可以为空：`[Keyword]`
*   数据不能为空：`[Keyword!]`
*   列表不能为空：`[Keyword!]!`

另一个在复杂情况下有用的 GraphQL 构造是 *联合类型*。

还记得关于联系人拥有接口与不拥有联系人作为类型的设计挑战吗？联合类型可以帮助构建一个合并的联系人列表（发起者和接收者）：

```
Union Contacts = Originator | Destination
```

然而，由于贡献的对象类型可能不完全兼容，你可能需要在查询中使用内联片段：

```
1   {
2     SearchResult {
3       ... on Originator {
4         origin_date
5       }
6       ... on Destination {
7         received_date
8       }
8     }
9   }
```

有关更多信息，请参阅位于 [`http://graphql.org/learn/`](http://graphql.org/learn) 的 GraphQL 文档^(³³)。

在本书的背景下，你必须记住，我们正在为 GraphQL API 中共享根查询（每个模式）的结果树构建一个范围。

这个数据模型是一个有向图，可以使用图技术（但不是 GraphQL）沿任意方向遍历。GraphQL 必须为每一次`查询`（`query`）产生一个树。

请注意，图 9-2 中所示的也是一个属性图——现在只是模式图的一个“Banzai”版本，而且还是扭曲的。

这很大程度上取决于你的根查询（`root query`）是什么样子。它决定了可能查询的视角。一个模式（`schema`）应该服务于特定应用程序的需求。

你可以在根查询中拥有多个字段。你也可以在配置中拥有多个模式。模式缝合（`schema-stitching`）的可能性使你能够合并模式，本质上是将根字段组合到合并后的根查询中。更多信息请参阅 GraphQL 文档。

如果某个特定模式中的查询都应该始于发起者（`originator`）级别，因为这是应用程序所需要的，那么我们可以根据这种模式构建 API 查询结果树，如图 9-2 所示。
![`../images/471428_1_En_9_Chapter/471428_1_En_9_Fig2_HTML.jpg`](img/471428_1_En_9_Fig2_HTML.jpg)
**图 9-2**
将发起者作为查询的根

这意味着，例如，地址级别或关键字级别的属性将会冗余地出现在树中，因为它们必须被反规范化（`denormalized`）到较低级别，或者，替代性地，放入一个 GraphQL 列表（`list`）结构。

属性图与 GraphQL 开发者高度相关。它们极大地简化了对当前数据的分析，并有助于组织生成的 API 模式和查询结构。

完成这些之后，我们也已经处理了：

*   正确地暴露所暴露数据（查询结果）中固有的关系结构。
*   处理多对多（`M:M`）关系的遍历以生成结果树（包括模式和查询结果）。

这种可视化是一种直观易懂（相当不错）的业务和应用程序术语表示，可以与业务人员讨论。

获得查询范围概览的一个简单方法是旋转和扭动应用程序图数据模型，使其以自上而下、从左到右的方式可视化结果树范围（参见图 9-3）。

派生出的树以关键字为根，将逐个关键字地传递所有被标记了该关键字的消息。对于每条这样的消息，将列出所有发起者及其各自的地址和所有目的地。

导航图数据模型也可以通过视觉上“套出”单个子树（层次结构）来简化。这将让你感觉到可以构造哪些结果树。

换句话说，反规范化和复制是你的好朋友。对于 GraphQL 而言，数据存在于 API 中，即在运行时存在于缓存中。

### 注意

GraphQL 的强大优势之一正是基于此：在分布式应用程序的缓存中，拥有一个精心自定义并可用的树状结构，可以带来更好的性能和更少的阻塞。

因此，暴露冗余数据没有问题，对于某些物理数据存储来说，这是必要的。显然，基于树的反规范化结果可能包含以下内容：

*   `数据`（`Data`）：选定的叶节点数据（模式设计中的选定属性）。
*   `导航字段`（`Navigation fields`）：中间和顶层节点属性，包括标识键以及分类层次结构的中间级别等。这是为了方便之故。

重要的是要包含所有必要的中间和顶层键。否则，你将无法唯一标识结果中的每个级别。

将已经反规范化的结构（例如在 NoSQL 存储中）映射到 GraphQL，在大多数情况下通常应该是可行的且相对容易实现的，但改变序列化顺序并不是胆怯者能做的。

你也可以考虑使用重复的列组。在 1:M 关系中，有时重复字段不会太多，因此可以很好地工作，并且受到 GraphQL 列表的支持。

一般来说，将图表示的解决方案数据模型映射为树表示的查询结果是非常直接的。你的可视化将帮助你擅长于此，一旦你的模式图变得很大，你会欣赏这一点。

生成正确的结果树是解析器（`resolver`）需求的重要组成部分，这正是我们下一章要讨论的内容。
