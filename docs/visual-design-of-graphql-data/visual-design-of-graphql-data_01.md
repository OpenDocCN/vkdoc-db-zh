# 引言

*您了解 `GraphQL` 的基础知识，但仍不确定如何在 `GraphQL` 中正确处理业务内容和 API 结构？*

`GraphQL` 对于应用程序（以及人们）来说确实是一个有吸引力的数据 API。

`GraphQL` 模式对于 `GraphQL` API 的成功至关重要。大多数开发项目涉及多个利益相关者。当然有开发者，还有业务专家以及将要使用 API 内容的应用所有者。这意味着模式不仅是一个范围契约，也是 API 所涵盖上下文的结构和含义的权威来源。在许多其他上下文中，复杂结构和语义必须在具有不同背景的多人之间进行有效沟通。而诀窍往往归结为：*使用良好的可视化！*

本书的主要主张是图可视化：`GraphQL` 模式的结构和含义必须被可视化，本书将向您展示如何做到。由于模式是一个 `数据图`，包含以有向图组织的、相互关联的概念网络，因此日益流行的 `属性图范式` 非常适合于可视化 `GraphQL` 结构和语义。

本书的第二个主题是质量。`GraphQL` API 可用于多种场景，可能包括遗留数据和/或外部来源的数据。在所有情况下都必须确保质量。这既在定义层面（业务术语等），也在数据内容层面（以业务友好的格式有意义地呈现数据）。本书总结了此类质量保证措施最重要的前 10 个关注领域。

在本书的此版本中，添加了以下内容：

*   跟进最新版本的 `GraphQL` 规范（小幅调整）
*   简要解释“模式拼接”等及其对开发风格的影响
*   讨论解析器要求，特别是与遗留 SQL 数据相关的要求
*   解释 `Neo4j`-`GraphQL` 集成，并探讨将 `GraphQL` 应用于现有和新的 `Neo4j` 图数据库。

通过这种方式，用 `GraphQL` API 为旧数据库和新数据库提供前端所面临的（数据和元数据质量）挑战是本书的第三个主题。

本书包含基于真实数据发现和统一经验教训的简单指南。这有助于开发者和架构师在最终的 API 设计中获得良好的质量。而可视化技术则有助于制作关于 API 设计结构的、令人信服的可视化沟通。

在模式质量上投入时间意味着开发者依据清晰的定义工作，这进而带来更高的生产力和结构良好的应用程序。
