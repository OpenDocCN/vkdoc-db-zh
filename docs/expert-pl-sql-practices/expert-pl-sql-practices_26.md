# PL/SCOPE

PL/SCOPE 是 PL/SQL 编译器的一项功能，它为数据字典扩展提供数据。该功能获取编译时收集的信息（编译器本就需要的信息）并将其放入一个视图（通常有三个：`USER_*`、`ALL_*` 和 `DBA_*`）中，名为 `USER_IDENTIFIERS`。数据字典提供有关在 PL/SQL 代码中使用的变量和其他标识符的信息。除了显示所有标识符的名称外，它还能识别变量是什么（数据类型以及变量与参数等的区别）、它们如何被使用（声明、引用、赋值等）以及它们在何处被使用（声明、赋值等的上下文）。

![images](img/square.jpg) `注意` PL/Scope 是版本 11g 及以上的一个特性。如果您仍在使用低于 11.0 的版本，您将需要升级才能同时利用 PL/Scope 和 `DBMS_HPROF`（动态分析将在后面讨论）。如果您使用的是早期版本，您可以从前面讨论的字典视图中获取部分此类信息。我建议升级，因为在 11g 中您将受益于除代码分析之外的许多其他功能。

您代码中使用的每个标识符（这对任何语言都适用，而不仅仅是 PL/SQL）都有一个上下文。这个上下文是使用分析中最重要的部分。一个变量可能在包（`DECLARATION`）内的一个过程（`DEFINITION`）中定义。值的赋值可能在定义时发生，也可能在后面的代码主体中作为赋值步骤发生。对于变量或迭代器，您将能够看到代码中任何未被引用（`REFERENCEd`）的变量。对于过程或函数，您可以发现它是否曾被调用（`CALLed`）。接下来我将更详细地展示这方面的示例。

## 什么是标识符？

标识符是您可以在代码中使用的任何东西。您可以使用一个过程（您调用它）。您可以使用一个变量（您可以给它赋值或从中读取值）。您可以使用数据类型（看起来很奇怪，但当您声明变量、常量、迭代器等时就会用到它们）。您可以使用触发器、同义词、形式参数，甚至异常。标识符基本上是您可以在代码中使用或引用（嵌入式 SQL 除外）的任何东西。

以下是您在使用 PL/SQL 时会遇到的不同类型的标识符：

*   使用的数据类型和子类型（包括集合）
*   游标和 REF 游标
*   过程
*   函数
*   包和包体
*   触发器
*   对象类型和类型体
*   参数
*   函数返回值
*   变量、迭代器、常量
*   异常
*   同义词（不会解析为任何实际对象，没有验证）

每个在 `USER_IDENTIFIERS` 视图中都有一个唯一的签名列。这个签名是由 Oracle 提供的十六进制字符串——它在数据库中是唯一的。如果您有一个名为 `v_my_id` 的标识符在两个不同的包或独立过程中声明，那么签名在这些程序单元中将是唯一的。如果您在同一个程序单元内的多个作用域中标识了同一个变量，每个变量都将有自己的签名。包内重载的过程也会有两个不同的签名。

## 标识符的作用域

PL/Scope 通过理解上下文，可以提供变量的作用域，因此得名 PL/Scope。这种能力在大多数情况下并非至关重要，但在某些情况下会有所帮助。一个例子是，当您在两个作用域中定义了同一个变量却没有意识到时。您可能认为您正在为较高作用域的变量赋值，但实际上是在为局部作用域的变量赋值。PL/Scope 显示的赋值无法告诉您您原本想做什么，但至少您会看到同一个调用栈中在多个作用域声明了相同的变量（不良实践警示）。您可以通过始终引用变量的命名作用域来最小化这种危险，但我发现这会使代码更难以维护而不是更容易。我只是尽量不重用名称。

同样地，PL/Scope 通过理解上下文，可以提供过程代码的调用栈。动态代码分析（稍后将介绍）的用途之一是验证代码覆盖率。代码覆盖率是在回归测试中实际执行所有代码。PL/Scope 可以给您一个静态视图，至少显示每个程序单元被引用。如果您有一个定义了但从未被调用的过程，这不一定是坏事（它可能从外部上下文被调用），但它确实提供了可以审查的额外信息。

说到外部上下文，PL/Scope 确实会显示在外部上下文中被调用（`CALLed`）或引用（`REFERENCEd`）的标识符。只要外部上下文是用 PL/Scope 开启编译的，您就可以连接到该外部引用。例如，当您声明一个 `NUMBER` 类型的变量时，它会对外部的 `DATATYPE_NUMBER`（在 `DBMS_STANDARD`（一个 Oracle 标准包）中声明的子类型）进行外部引用。这将是一个外部上下文（从当前程序单元的角度看是外部的）。

如果一个程序单元中的一个过程调用了另一个程序单元中的一个过程，并且两个过程都使用了 PL/Scope，您就可以看到完整的调用栈。您无法跟踪栈进入外部 C 过程或 Java 单元，因为它们不受 PL/Scope 覆盖。相反，您会看到调用（`CALL`），而栈将在此处终止（嗯，不是终止；当外部引用返回时，您会重新拾起栈）。
