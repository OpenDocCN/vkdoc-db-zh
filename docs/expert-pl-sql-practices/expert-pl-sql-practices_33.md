# 标准包-本地断言

PL/SQL 中的大多数实质性编程都发生在包内部，因此决定如何在包的上下文中强制执行契约是一个重要的考量。我开发了一个简单、标准化的包本地断言实现，用于我所有的 PL/SQL 包中。仅使用这个相对简单的结构和相关的编码规范，就可以在定义和严格执行 PL/SQL 软件契约方面取得显著进展。

首先需要标准化的是关于契约强制执行的机制和信号含义。所采用的机制是使用`RAISE_APPLICATION_ERROR`内置过程引发一个特定的异常（称为`ASSERTFAIL`）。代码清单 8-3 中的 `DBC_example` 包规范展示了如何声明 `ASSERTFAIL` 异常，并使用 `PRAGMA` 编译器指令将其关联到一个特定的异常编号。

### 代码清单 8-3. 标准化的包规范模板
```sql
CREATE OR REPLACE PACKAGE DBC_example
AS
  ----------------------------------------------------------
  -- SLPA 声明（标准化的本地包断言）
  ----------------------------------------------------------
  ASSERTFAIL     EXCEPTION;
  ASSERTFAIL_C   CONSTANT INTEGER := -20999;
  PRAGMA EXCEPTION_INIT(ASSERTFAIL, -20999);
  ----------------------------------------------------------
END DBC_example;
/
```

应用程序作用域内的所有包都应标准化使用一个单一的错误编号作为 `ASSERTFAIL` 异常，以避免混淆。这样，`ASSERTFAIL` 在整个应用程序中就具有了共同的含义。这一点很重要，因为清晰的契约强制要求对契约违规有明确且普遍的解释，这正是 `ASSERTFAIL` 的全部目的。最后，`ASSERTFAIL` 的异常编号 (`-20999`) 也被分配给一个包常量，以便在断言过程内部引用。类似地，所有包都应以如 代码清单 8-4 所示的标准化实现开头，该清单取自 `DBC_example` 包的主体。

### 代码清单 8-4. 标准化的包主体示例
```sql
CREATE OR REPLACE PACKAGE BODY DBC_example
AS
  -- 用于 assertfail 消息的包名
  PKGNAME_C      CONSTANT VARCHAR2(20) := $$PLSQL_UNIT;  -- 注意：10g 构造
  -- 前置声明 assert，以便所有模块实现都能引用它
  PROCEDURE assert (condition_IN IN BOOLEAN
                   ,msg_IN       IN VARCHAR2 := NULL
                   ,module_IN    IN VARCHAR2 := NULL);
  ----------------------------------------------------------
  -- 标准的本地包断言过程
  ----------------------------------------------------------
  PROCEDURE assert (condition_IN IN BOOLEAN
                   ,msg_IN       IN VARCHAR2 := NULL
                   ,module_IN    IN VARCHAR2 := NULL)
IS
    l_assertmsg VARCHAR2(2048) := 'ASSERTFAIL:'||SUBSTR(PKGNAME_C,1,30)||'.'; -- 断言消息
  BEGIN
    -- 测试被断言的条件
    IF NOT NVL(condition_IN,FALSE)   -- 输入为空时视为失败
    THEN
      -- 完成断言消息的初始化
      l_assertmsg := l_assertmsg || SUBSTR(NVL(module_IN,'?MODULE?')||':'||msg_IN,1,2046);
      -- 抛出标准化的异常
      RAISE_APPLICATION_ERROR (ASSERTFAIL_C, l_assertmsg, FALSE); -- FALSE
    END IF;
  END assert;

BEGIN
  assert(TRUE); -- 包初始化标记
END DBC_example;
/
```

代码清单 8-4 中的断言与之前在代码清单 8-2 中作为公共过程呈现的断言非常相似，略有不同的是，它使用了 Oracle 10g 查询指令 `$$PLSQL_UNIT` 来标识包，而不是依赖于单独声明（并且可能被错误初始化）的常量。对 `RAISE_APPLICATION_ERROR` 的调用会将长消息截短至恰好适合 2048 个字符，其中为包名留出最多 30 个字符的空间。在实践中，挑战此限制的消息不一定那么有帮助，但使用子字符串有助于代码的正确性和健壮性。

代码清单 8-4 并不代表为了在所有包中提供本地断言（独立于其他过程）而需要付出太多代码代价。我更倾向于包在初始时能够完全独立编译的理念（并且甚至因此有点反感 `RAISE_APPLICATION_ERROR`）。

因此，我个人的偏好是，所有 PL/SQL 包都使用像这样的标准规范和主体模板开始，以便在包内部和跨包采用标准化的契约强制用法。

或者，你也可以采用一个简单、标准化的 `PUBLIC` 断言过程，如代码清单 8-2 所示，并让所有契约强制都依赖于它。

最后，我可以建议你创建一个真正精妙的共享包断言实现，带有花哨的调用栈处理来辅助错误诊断，但我不推荐这样做。本章的目的是教育和说服你在代码中以简单的方式强制执行契约，而不是用花哨的断言机制来让人印象深刻。

### 本地还是全局断言？

熟悉传统 PL/SQL 智慧的精明读者很可能会质疑我推荐将标准化断言结构嵌入所有包，而不是创建一个使用独立过程或专用断言包的单一共享机制。

老实说，我更喜欢包本地断言的原因，与其说是技术性的，不如说是情感和政治上的，但我还是会尝试说明理由。

首先，对我来说，采取面向契约的方法不仅仅是采用一项新技术。它更像是一种宗教皈依，因此可能会在未皈依者中遇到强烈的抵制。因此，在就部署和使用通用机制达成一致方面，可能会遇到相当大的政治阻力，而且宗教战争从来都是赢不了的。在本地包的安全范围内私下进行崇拜要容易得多，至少在群众皈依之前是如此。

其次，我喜欢假装并使用 assert，就好像它是 PL/SQL 语言原语一样，因此总是使用包限定的引用会不断提醒我事实并非如此。到处都拥有相同实现的本地断言有助于维持我的这种错觉。

第三，另一个个人偏好是总是从没有任何外部依赖项开始新包（尽管我开发的几乎没有几个包真正保持独立）。

最后，让每个包都本地声明 `ASSERTFAIL` 异常有一个微小的技术优势，那就是异常在包之间是可以区分的。所以，例如，如果某个代码块调用来自两个不同包的模块，它可以区分是哪个 `ASSERTFAIL` 被引发。然而，这很少有影响，因为 `ASSERTFAIL` 实际上永远不应该被处理，而应允许其一路向外传播。

所以，情况就是这样：偏好并推荐包本地断言机制而非共享全局结构的四个真正不充分的理由。但正如已经提到的，尽管推理薄弱，偏好却很强烈。

幸运的是，Oracle 11g 引入了优化的 PL/SQL 编译器，这实际上为我的推荐提供了一个新的、相当有力的性能论据，即代码内联。本章末尾简要讨论了该主题及其支持包本地断言的含义。

### 使用 ASSERT 实现契约

在决定标准化断言机制并将其嵌入每个程序包后，现在的问题是如何将其用于实现契约？为此需要一点规范；不过，根据我的经验，前期在编码上投入的额外精力，会因确保了一大类重要错误要么不存在、要么会被代码本身立即检测到而获得远超投入的回报。

Listing 8-5 中的流程展示了如何在代码中直接使用标准化的 `assert` 来实现良好的前置条件和糟糕的后置条件。前置条件是输入参数不为 `NULL`，这无法以声明方式强制实现，并且往往是一个非常重要的要求，具体原因将在后文讨论。相比之下，后置条件则说明了在代码中使用 `assert` 时遵循调用规范的价值。该断言未能利用 `module_IN` 或 `msg_IN` 参数，因此在调试相关问题时几乎无法提供有效信息。

## Listing 8-5. 正确与错误使用 Assert 的示例过程

```
PACKAGE BODY DBC_example
…
  ----------------------------------------------------------
  -- 测试/示例过程和函数
  ----------------------------------------------------------
  PROCEDURE proc01 (p1 varchar2)
  IS
    -- 用于 assert 调用的本地模块名称
    l_module VARCHAR2(30) := 'PROC01';
  BEGIN
    -- 强制非空输入前置条件
    assert(p1 IS NOT NULL,'p1 NOT NULL', l_module);

    -- 过程逻辑
    null;

    -- 一个非常糟糕的后置条件！
    assert(FALSE,'Boo hoo on you');  -- 未传递 module 参数的 assert 失败
  END proc01;
```

`Proc01` 接受一个输入参数 `p1`，并通过以下代码行断言契约前置条件，即它必须为 `NOT NULL`：

```
assert(p1 IS NOT NULL,'p1 NOT NULL',l_module);
```

请注意，该前置条件是在模块主代码块的第一行进行测试的。请记住，在软件契约下，如果前置条件未满足，模块没有义务计算输出，因此立即测试它们可以让模块尽早失败，避免不必要地浪费计算周期。

另请注意通过参数 `msg_IN` 和 `module_IN` 向 `assert` 传递额外信息所采用的约定。对于第一个参数，消息仅仅是构成待断言契约条件的布尔表达式的字符串字面量。对于第二个参数，模块声明一个局部变量 `l_module`，并用其自身名称初始化它，该名称也会传递给 `assert`。

除了针对 `NULL` 输入断言契约前置条件外，`proc01` 还实现了一些有缺陷的说明性逻辑：它断言了一个最终的后置条件，该条件总是会抛出 `ASSERTFAIL` 异常，但没有传递任何有用的诊断信息以包含在错误消息中。

通过使用 `NULL` 输入值调用 `proc01` 进行测试，会从 SQL*Plus 中输出以下结果：

```
BEGIN DBC_example.proc01(null); END;
*
ERROR at line 1:
ORA-20999: ASSERTFAIL:DBC_EXAMPLE.PROC01:p1 NOT NULL
ORA-06512: at "APPL01.DBC_EXAMPLE", line 104
ORA-06512: at "APPL01.DBC_EXAMPLE", line 25
ORA-06512: at line 1
```

乍一看，这些结果似乎并不那么令人印象深刻，但对我来说，它们是纯金。例如，第一个 `ORA-20999` 错误表明：

*   代码中存在契约违规错误，由 `ASSERTFAIL` 指示。
*   该错误在 `DBC_EXAMPLE` 包的 `PROC01` 模块中被捕捉到。
*   契约违规是输入变量 `p1` 未能通过 `NOT NULL` 检查。
*   调用堆栈表明 `DBC_EXAMPLE` 包中第 25 行的断言检查失败。

一个错误已被确定无疑地识别出来，捕捉到错误的代码位置已知，错误的性质也已揭示。回想一下你上次试图追踪涉及意外运行时数据值的错误的经历，然后告诉我这难道不酷毙了吗？

阅读调用堆栈并不直观，需要一些实现知识。从 `ORA-20999` 错误开始，这是本地 `assert` 中调用 `RAISE_APPLICATION_ERROR` 的地方，堆栈中的下一个条目必定是 `assert` 过程本身，因此堆栈中的第三个条目是调用 `assert` 的代码位置。这一点很重要，因为堆栈中的下一个条目是调用 `assert` 的模块的调用者。当断言是前置条件时，表明这个调用者违反了契约条件，因此调用代码存在错误，错误位置在指示的行（在此简单示例中是第 1 行）暴露出来。执行 `proc01` 的第二次测试，这次传递了非 `NULL` 输入以满足前置条件，但在糟糕的断言上出错，得到以下输出：

```
BEGIN DBC_example.proc01('VALUE'); END;
*
ERROR at line 1:
ORA-20999: ASSERTFAIL:DBC_EXAMPLE.?MODULE?:Boo hoo on you
ORA-06512: at "APPL01.DBC_EXAMPLE", line 104
ORA-06512: at "APPL01.DBC_EXAMPLE", line 29
ORA-06512: at line 1
```

第二个 `ASSERTFAIL` 有助于说明为 `assert` 的模块名称和消息参数传递信息值所需的额外规范的价值。在这种情况下，仍然知道存在契约违规错误以及错误发生在哪个包中，但缺乏额外的诊断信息明显是不利的。

我强烈建议你花时间，在调用 `assert` 时始终包含模块名称和失败条件的信息性描述。许多模块可能有多个输入参数，其中每个参数都可能受一个甚至多个契约前置条件约束。如果不进一步考虑模块的计算目标，这看起来会像是大量的额外输入。相信我，第一次 `ASSERTFAIL` 异常在运行时跳出，并准确告知你先前未发现的错误的位置和内容时，你会非常高兴自己投入了这些精力。

### 一项额外的改进

如前所述，前置条件违规表明调用模块中存在错误，而后置条件违规则是被调用模块中的错误。因此，对于确定契约违规错误的责任归属，了解 `ASSERTFAIL` 是来自前置条件还是后置条件断言至关重要。

Listing 8-6 中的改进是由我的朋友 Steven Feuerstein 提出的，作为区分前置和后置条件违规的一种方法。

## Listing 8-6. 区分前置与后置条件

```
-- 用于前置条件的 assert
  PROCEDURE assertpre (condition_IN IN BOOLEAN
                       ,msg_IN      IN VARCHAR2 := NULL
                       ,module_IN    IN VARCHAR2 := NULL)
  IS
  BEGIN
     assert (condition_IN,'Pre:'||msg_IN,module_IN);
  END assertpre;
-- 用于后置条件的 assert
  PROCEDURE assertpost(condition_IN IN BOOLEAN
                       ,msg_IN       IN VARCHAR2 := NULL
                       ,module_IN    IN VARCHAR2 := NULL)
  IS
  BEGIN
     assert (condition_IN,'Post:'||msg_IN,module_IN);
  END assertpost;
```

这个想法很简单：使用单独命名的过程来断言前置和后置条件，这些过程调用原始的 `assert`，但在消息中添加“Pre:”或“Post:”。这些额外的例程不会增加太多代码负担，并且显著改进了错误诊断，因此我已将它们添加到标准化的包模板中。当然，必须在正确的上下文中调用它们以避免混淆：切勿使用 `assertpost` 检查前置条件，也切勿使用 `assertpre` 检查后置条件。

### 面向契约的函数原型

PL/SQL 函数提供了额外的机会，可以将更多的契约规范引入代码中。请将清单 8-7 中的函数视为所有 PL/SQL 函数的原型。

***清单 8-7.** PL/SQL 函数原型*

```sql
FUNCTION  func01 (p1 varchar2) RETURN NUMBER
  IS
    l_return NUMBER;
    l_module VARCHAR2(30) := 'FUNC01';
  BEGIN
    -- 前置条件
    assertpre(p1 IS NOT NULL,'p1 NOT NULL',l_module);

    -- 函数逻辑
    null;

    -- 后置条件检查与返回
    assertpost(l_return IS NOT NULL, 'RETURN NOT NULL',l_module);

    RETURN l_return;
  END func01;
```

关于此函数，请注意以下关键点：

*   它通过一个使用标准化名称（当然还有恰当类型）声明的局部变量来返回结果。
*   它在开始处使用模块标识符和消息约定来断言前置条件。
*   它仅在代码的最后一行使用 `RETURN` 语句退出。
*   它在 `RETURN` 语句之前立即使用标准约定来断言后置条件。

由于函数总是有一个返回值，因此可以并且应该将其编码为只有一个返回此值给调用模块的出口点。遵循这一约定，使得后置条件检查可以精确地在 `RETURN` 语句之前的一个地方指定。

使用一个标准名称的局部变量专用于 `RETURN` 语句，这使得我能够知道函数逻辑何时正在操作返回值。那些在 `RETURN` 语句中直接使用复杂表达式计算输出的“聪明”函数，看起来可能很优雅，但它们削弱了通过遵循我的简单约定所能提供的对后置条件断言的控制级别。

遵守函数原型的编码约定对代码可靠性大有裨益，因为函数可以在明确定义的入口和出口点强制执行契约条款。具有多个可能出口点的函数（或过程）可能导致需要在多个地方重复后置条件断言，这本身在契约条款需要更改时就可能成为错误的来源。

### 示例：测试奇数和偶数整数

到目前为止的例子都是简单的演示，用于调用前置和后置条件断言，而没有任何实际的计算需求。本节将提出一对具有实际计算需求的函数，尽管需求非常小，也许不太实用。

想象一个应用程序，其逻辑关键取决于特定数据值是奇数还是偶数整数。在此类应用程序中，拥有符合以下契约协议的模块可能是有用的：

*   一个函数，接受一个非 NULL 整数作为输入，如果该整数是奇数则返回布尔值 `TRUE`，否则返回 `FALSE`。
*   一个函数，类似地接受一个非 NULL 整数，当输入整数是偶数时返回 `TRUE`，否则返回 `FALSE`。

这两个函数显然是相关的，因为对于任何给定的整数，它们应该返回完全相反的结果，因为整数只能是偶数或奇数，并且必须是两者之一。清单 8-8 中的代码使用了前面介绍的函数原型和编码约定来满足第一个需求。

***清单 8-8.** 用于测试数字是否为奇数的函数*

```sql
FUNCTION  odd (p1 number) RETURN BOOLEAN
  IS
    l_return BOOLEAN;
    l_module VARCHAR2(30) := 'ODD';
  BEGIN
    -- 前置条件
    assertpre(p1 IS NOT NULL,'p1 NOT NULL',l_module);
    assertpre(p1 = TRUNC(p1), 'p1 INTEGER',l_module);

    -- 函数逻辑：如果 p1 是奇数则返回 TRUE
    l_return := MOD(p1,2)=1;
    -- 后置条件检查与返回
    assertpost(l_return IS NOT NULL, 'RETURN NOT NULL',l_module);

    RETURN l_return;
  END odd;
```

清单 8-8 中的函数几乎与清单 8-7 中的函数原型完全相同。清单 8-8 中的函数分别在函数的开始和结束处断言前置和后置条件，在代码的最后一行返回计算结果。

该函数断言了两个前置条件：输入非 NULL，并且它还是一个整数。聪明的读者可能会观察到，第二个断言隐含了第一个断言，因为如果 `p1` 的输入值是 `NULL`，那么这第二个断言也会失败，因此第一个断言在某种意义上可能是多余的。在这种情况下，可能诱人只断言第二个条件；然而，这代表了一种捷径，它模糊了实际上有两个独立前置条件的事实：输入必须非 NULL 并且它必须是一个整数值的数字。采用面向契约编码方式的部分动机是希望在代码本身中对契约要求非常明确，因此即使复合表达式隐含了这些简单断言，也应在复合断言之前加上对其组成部分的任何简单断言。这说明了代码中断言的一个重要附加好处：它们既记录又强制执行软件契约。

`odd` 函数的计算需求仅用一行代码就满足了，但函数本身包含了五行可执行代码。我们功能需求的最简实现可以由清单 8-9 中的函数满足。

***清单 8-9.** 测试奇数的最简实现*

```sql
FUNCTION  odd2 (p1 number) RETURN BOOLEAN
  IS
  BEGIN
    RETURN MOD(p1,2)=1;
  END odd2;
```


虽然列表 8-9 中的实现满足了基本的计算要求，即当`p1`为奇数时返回`TRUE`，但它并未强制执行契约要求，即只接受非`NULL`的整数并且只返回`TRUE`或`FALSE`。如果传入`p1`为`NULL`，该函数将返回`NULL`作为输出，尽管它声称返回的是布尔值。这迫使调用模块要么自行对输入值断言非`NULL`，要么使用`NVL`或其他特殊情况结构来处理`NULL`输出。因此，为确保正确性所需的额外代码并未消失，而是被转移到了调用应用程序中，所有这些应用程序都必须正确实现它。

同样，该最小实现并未检查`p1`是否为整数，而`MOD`函数会欣然接受其参数的非整数值并返回结果，因此函数在此情况下不会失败；然而，一个依赖于整数奇偶性的系统不太可能满足于接受非整数输入并判断其奇偶性的例程。

现在，考虑列表 8-10 中互补函数`even`的实现，该函数用于计算输入整数是否为偶数。

**列表 8-10.** 一个测试偶数性的函数

```sql
FUNCTION  even (p1 number) RETURN BOOLEAN
  IS
    l_return BOOLEAN;
    l_module VARCHAR2(30) := 'EVEN';
  BEGIN
    -- 前置条件
    assertpre(p1 IS NOT NULL,'p1 NOT NULL',l_module);
    assertpre(p1 = TRUNC(p1), 'p1 INTEGER',l_module);

    -- 函数逻辑：如果 p1 是偶数则返回 TRUE
    l_return := p1/2 = TRUNC(p1/2);

    -- 后置条件检查和返回
    assertpost(l_return IS NOT NULL, 'RETURN NOT NULL',l_module);
    assertpost(NOT(l_return AND odd(p1)),'NOT(EVEN AND ODD)',l_module);
    assertpost(l_return OR odd(p1),'EVEN OR ODD',l_module);

    RETURN l_return;
  END even;
```

列表 8-10 的函数看起来与`odd`函数（列表 8-8）非常相似，因为它紧密遵循了面向契约的函数原型，但仍有一些重要区别值得注意。一方面，该函数逻辑本可以轻松地使用取模逻辑，就像`odd`函数那样，使用`l_return := MOD(p1,2)=0`，但它却使用了简单的除以 2 和`TRUNC`函数：

```sql
l_return := p1/2 = TRUNC(p1/2);
```

这引出了关于`even`函数最值得注意的一个特性，即对后置条件的广泛断言。首先是简单的期望，即`l_return`非`NULL`，随后是这些逻辑谜题：

```sql
assertpost(NOT(l_return AND odd(p1)),'NOT(EVEN AND ODD)',l_module);
assertpost(l_return OR odd(p1),'EVEN OR ODD',l_module);
```

这两个复杂的断言实际上通过互补的`odd`函数来测试`even`函数即将返回值的正确性。第一个断言这两个函数`even`和`odd`不能同时对`p1`做出相同判断，如果一个为`FALSE`，则另一个必为`TRUE`，反之亦然。第二个断言对于任何给定的`p1`，它们中至少有一个必须为`FALSE`。

这些断言通过将计算结果与采用不同算法策略且计算上独立的函数的结果进行比对，来测试函数结果的正确性。这是极少数可以将计算结果的正确性作为后置条件进行断言的案例之一。

当然，没有什么是完全免费的，在这种情况下，断言复杂的后置条件实际上相当于将计算逻辑执行了三次——一次在`even`函数本身中，另外两次在调用`odd`函数进行后置条件断言时。与`odd`函数的紧密耦合也意味着，如果该函数存在错误，该错误也会延伸到`even`函数中。

### 有用的契约模式

在我自己的代码中，我认识到并欣赏几种有用的契约模式，将在以下小节中描述。采用这些经过充分理解和商定的模式，可以提升代码的正确性和清晰度，因为某些契约根据模式被理解为适用，无需逐模块进行规范或分析。

#### 非空输入 / 非空输出

在 PL/SQL 上下文中，最基本和最有用的软件契约也许是那些消除了围绕`NULL`值处理的问题性和通常模棱两可情况的契约。大多数算法期望基于实际数据进行计算，并且可能需要在数据缺失或为`NULL`值时提供特殊情况或替代的计算结果。这可能会给 API 引入不必要的歧义或复杂性，更好的做法是在可能的情况下简单地禁止`NULL`值。

找出由`NULL`值导致的错误可能非常困难，特别是当计算因此并未显式失败时。计算机程序逻辑本质上是二元的，而`NULL`是一个不受欢迎的第三者。

因此，我最喜欢的契约之一可以简单地描述（从被调用模块的角度看）如下：如果你同意不传给我`NULL`输入值，你肯定将收到非`NULL`的输出值作为回报。只要能够实施，这种契约协议几乎普遍有价值。在实践中，大部分契约执行是验证输入前置条件，而这通常意味着只接受非`NULL`值。当然，模块随后也必须遵守并尽可能强制执行其在后置条件下的义务，即只生成非`NULL`输出。这种契约模式还具有内在的公平性：双方承担相同的义务，并获得相似的确定性收益。

#### 函数返回非空

这种模式是前一种模式的特化，它认识到在模块必须向调用者返回输出时，使用函数而不是过程具有显著优势。基于函数的编程模型有助于实施有纪律的契约执行，如前面讨论的函数原型所示。此外，可靠地返回非`NULL`值的函数可以在调用代码中引用，以代替常量或变量。例如，想象一个名为`my_function`的函数，它接受一个输入参数`p1`，并保证返回一个非`NULL`的数字作为输出。这使我能够安全地编写如下代码：

```sql
IF my_function(p1=>'X') > my_function(p1=>'Y')
THEN do_something;
END IF;
```

这里的意图是清晰、明确的，并且可以直接从代码中读懂：比较`my_function`在参数值'X'和'Y'上的结果，如果前者大于后者则采取行动。

现在假设`my_function`不保证输出非`NULL`值。那么任何调用它的代码都必须自行决定在此情况下该怎么做，例如：

```sql
IF NVL(my_function(p1=>'X'), 0) > NVL(my_function(p1=>'Y'), 1)
THEN do_something;
END IF;
```

这段代码由于引入了`NVL`函数调用对的干扰性语法噪音而变得难以阅读。此外，意图远不那么明显，特别是考虑到当输入值为'X'时`my_function`的`NULL`输出被转换为 0，而当输入值为'Y'时转换为 1。为什么会这样？这是此代码中的错误，还是`my_function`内在目的的某种神秘后果？当函数契约薄弱且调用者必须适应模棱两可的输出时，就会发生这种情况。
