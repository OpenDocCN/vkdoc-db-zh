# 第 3 章

## PL/SQL 与并行处理

作者：`Dominic Delmolino`

想象一下，如果你去杂货店，一次只允许一个人进去。其他所有人都必须在外面排长队等待，直到每个人完成购物才能进入商店。更糟糕的是，想象一下你在店内只允许购买一件商品。如果你需要多件商品，你必须为每一件商品重新到店外排队。

这种一次只进行一项活动的例子被称为 `串行处理`，虽然有可能加快这个活动，比如给每个购物者穿上轮滑鞋让他们在店里移动得更快，或者允许他们带购物车进去挑选多件商品，但它仍然受限于每次只让一个人购物的设计。

谢天谢地，大多数商店都使用 `并行处理`——允许同时发生多个活动的能力。通过允许多人同时购物，商店在每一天能处理更多的订单。只要每个购物者购买的是不同的商品，他们的活动就是独特且独立的——这使它们非常适合并行处理。事实上，只有少数几次需要将这个活动串行化——通常是当两个或更多人想要购买货架上的同一件商品时。这时，购物者需要通过协商谁得到这件商品来 `协调` 他们的活动。

并行处理活动在自然界和社中很常见；我们都见过昆虫大军和鱼群将庞大的任务分而治之。随着收集和存储信息的成本持续降低，大型数据处理任务变得越来越普遍。随着公司开始积累海量数据，他们转向并行处理选项，作为在多个处理器和机器上分析和应对数据的方式。一种名为 `MapReduce` 的并行编程技术作为一种通用构建可并行运行程序的方法而广受欢迎。在本章中，你将探索这些分而治之的技术，并使用 PL/SQL 将它们应用于数据库活动。在本章结束时，你将能够使用 PL/SQL 和并行管道化函数编写你自己的并行 `MapReduce` 程序。但首先，让我们深入了解并行处理的历史。

### 为何需要并行处理？

一言以蔽之：速度。我们所有人都希望事情进展得更快，工作完成得更迅速，能够按需获得答案。尽管计算机的速度越来越快，其电路数量成倍增加，时钟速度不断提高，片上能力也越来越多，我们仍然不满足。随着我们遇到单个处理器速度提升的物理限制，我们开始寻找其他加速程序的方法。

如果计算机能自动找出将问题分解成更小部分并在这些更小的部分上并行运行程序的方法，那就太好了。自动识别并行性已经有了大量的研究，但事实证明，计算机很难理解哪里可以应用并行机会。人们曾尝试编写编译器，将高级编程语言转换为 CPU 的显式并行指令流，但在这方面并没有太大成功，这使得我们需要自己手动设计并行处理策略。

好消息是，如果实施得当，并行处理技术除了简单的加速之外，还为我们提供了若干好处，包括：

*   更好的系统利用率，因为多处理器系统内的所有处理器都可以应用于一个任务。
*   消除了“全有或全无”的过程，在部分子任务失败时允许部分完成。
*   能够在固定时间内处理越来越大的数据集，而不是简单地将固定数据集加速到更短的时间内完成。

事实证明，这些好处往往值得设计和实现并行程序所需的开销和艰苦工作。
