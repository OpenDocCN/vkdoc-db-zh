# PL/SQL 与并行处理

## 你可以引用尚未存在于数据库中的对象

如果你为每个月都有一个独立的生成过程（即年份和月份是过程名称的一部分），那么创建一个通用包装器比编写一个巨大的 CASE 语句更容易。以下是这样一个包装器的示例：

```sql
CREATE PROCEDURE p_runMonthly(i_dt date)
is
    v_sql_tx VARCHAR2(256):='BEGIN p_run_'||to_char(i_dt,'YYYYMM')||'; END;';
BEGIN
    EXECUTE IMMEDIATE v_sql_tx;
END;
```

甚至更好的是，当你必须向现有列表中添加另一个表时，这个通用调用器不需要任何维护。当然，有人可能会质疑在统一的命名模式上构建逻辑的安全性，但我见过太多这种类型的实现了。每次创建一个新对象都会给相关人员带来巨大的压力，除非所有引用都被包装进动态 SQL。

## 让你的代码与失效的依赖项隔离开来

有时，存在依赖关系并非好事，原因有很多：

*   `代码生成器`：如果你将对生成模块的调用包装进动态 SQL，你就可以刷新这些模块而不会使所有依赖项失效。在使用 JDBC 连接数据库时，这可能是至关重要的，因为它们（与 Net8 调用相比）的容错性要低得多。
*   `远程对象`：因为你的维护周期可能与你项目中涉及的其他团队的维护周期不同，所以有可能当你通过数据库链接引用对象时，它们是不可访问或无效的。在这种情况下，唯一安全的方法是将所有远程调用包装进动态 SQL。
*   `逻辑循环`：尽管非常困难，但你可能会创建一个对象引用链自身循环的情况。在这种情况下，唯一的解决方案是让其中一个步骤成为一个动态 SQL 模块。

### 总结

总的来说，本章涵盖的材料表明，你不应该害怕偏离服务器端应用开发中那些久经考验的方法。当然，仍然会存在合理的学习曲线和隐藏的陷阱，但应用更高级技术（如动态 SQL）所带来的收益是巨大的，包括：

*   开发人员可以实现更好的代码灵活性和可持续性。
*   架构师可以构建需要更高级别抽象的解决方案，包括代码库和代码生成器。
*   管理员可以获得对现有系统更详细的控制，并提高代码的可管理性。

所有这些积极影响最终都指向一个结论：有可能“更好、更快、更便宜”地构建现代 IT 项目（现在我保证在台下也会重复同样的故事！）。你只需要拥有合适的工具，并接受这样一个事实：除非你处理未知，否则它将违背你的意愿反过来处理你。
