# 方案的优化使用

一个应用程序的理想方案数量取决于架构。对于手写代码，每个方案包含 20-100 个包可能是一个理想的平均大小。包太少，模块数量以及接口的大小和数量可能会过多，这违反了上述模块化规则，并导致授权和结构更改的管理开销。包太多，则可理解的模块化单元又太大。

方案有几个附加特性，在设计时应予以考虑。它们对于部署第三方组件很有用，因为它们的命名空间可以防止名称冲突。方案可用于强制实施安全性，因为在所有情况下（包括动态 SQL）都会由数据库检查安全性。除了前面描述的对象权限外，方案还可以用于将关键操作集中到一个具有许多系统权限的小型、“微内核”式方案中。此外，方案具有各种属性（例如，表空间上的配额），并且是启用版本控制的单元。
