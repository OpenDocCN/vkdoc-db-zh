# 触发器执行顺序的重要性

## 问题再现

考虑以下触发器 `tr_pay_follow_up`：

```sql
create or replace trigger tr_pay_follow_up
before update
on payments
for each row
begin
        dbms_output.put_line ('This is tr_pay_follow_up');
        if (
                (:new.risk_rating = 'HIGH' and :new.pay_mode = 'C')
                or (:new.risk_rating = 'MEDIUM' and :new.pay_mode = 'K')
                or (substr(:new.credit_card_no,2,5) = '23456')
        ) then
                :new.follow_up := 'Y';
        else
                :new.follow_up := 'N';
        end if;
end;
/
```

我特意放置了 `dbms_output` 行来显示触发器的执行情况。创建这两个触发器后，对表执行更新操作。

```sql
update payments
set credit_card_no = '1234567890123456',
    amount = 100000,
    pay_mode = 'K';
```

输出如下：

```
This is tr_pay_follow_up
This is tr_pay_risk_rating
1 row updated.
```

之后，从表中查询数据。

```sql
SQL> select * from payments;
```

```
    PAY_ID CREDIT_CARD_NO   AMOUNT P RISK_R F
    ------ ---------------- ------ - ------ -
         1 1234567890123456 100000 K HIGH   N
```

检查上述输出，您会注意到付款金额超过了 10,000——这是 `RISK_RATING` 为 `HIGH` 的条件。触发器计算了该条件并正确设置了值。付款方式是现金（`PAY_MODE` 设置为 `K`），金额是 100,000，信用卡号也匹配了模式“23456”——这些都是 `FOLLOW_UP` 应为“Y”的条件；但结果却是“N”。难道触发器没有触发吗？不，它触发了，您在执行更新语句后的输出中看到了“`This is tr_pay_follow_up`”。如果触发器触发了，为什么没有正确设置值呢？

答案在于触发器的执行顺序。请注意执行更新语句后的输出。第一个触发的是 `tr_pay_follow_up`；而不是 `tr_pay_risk_rating`。当触发器触发时，所有列的值都满足 `follow_up` 的条件，但 `RISK_RATING` 是 `NULL`。因此，触发器没有找到满足 `follow-up` 条件的行。当另一个触发器触发后，它正确设置了 `RISK_RATING` 列，但此时前一个触发器已经执行完毕，不会再次执行。然而，如果触发器的执行顺序相反（风险评级触发器在前，跟进触发器在后），情况就会如您所预期的那样。

这是一个很好的例子，说明了触发器执行顺序对应用程序逻辑至关重要。显然，当前示例会被视为应用程序代码中的一个错误。更糟糕的是，这个场景无法随意重现；有时执行顺序符合预期，有时则相反。如果您有大量的此类触发器，出现错误的可能性会更高。

您可以通过将所有代码放在单个更新前行触发器中来控制执行代码，从而解决此问题。您可以为每个触发器逻辑创建单独的过程，并在单个更新前行触发器中调用它们。这可能有效，但并不建议，因为您只有一个触发器。如果您想禁用其中一个触发器的功能怎么办？如果有多个触发器，您只需禁用该触发器，并在需要时重新启用。

## 解决方案：指定执行顺序

在 Oracle Database 11g Release 1 中，有一个优雅的解决方案。您可以指定触发器的执行顺序。

```sql
create or replace trigger tr_pay_follow_up
before update
on payments
for each row
follows tr_pay_risk_rating
begin
        dbms_output.put_line ('This is tr_pay_follow_up');
        if (
                (:new.risk_rating = 'HIGH' and :new.pay_mode = 'C')
                or (:new.risk_rating = 'MEDIUM' and :new.pay_mode = 'K')
                or (substr(:new.credit_card_no,2,5) = '23456')
        ) then
                :new.follow_up := 'Y';
        else
                :new.follow_up := 'N';
        end if;
end;
/
```

注意子句 `follows tr_pay_risk_rating`，它指定该触发器应在风险评级触发器之后执行。现在您可以保证触发器的执行顺序：风险评级触发器先触发，然后是跟进触发器。使用这个新设置后，如果您重新执行前面的示例，将看到以下结果：

```
… perform the same update as before …
This is tr_pay_risk_rating
This is tr_pay_follow_up
1 row updated.
```

```sql
SQL> select * from payments;
```

```
    PAY_ID CREDIT_CARD_NO   AMOUNT P RISK_R F
    ------ ---------------- ------ - ------ -
         1 1234567890123456 100000 C HIGH   Y
1 row selected.
```

现在的执行顺序是 `tr_pay_risk_rating` 然后是 `tr_pay_follow_up`——这是有保证的。第一个触发器正确设置了列值，这使得第二个触发器能够正确计算跟进标志。此功能现在允许您构建模块化代码，并可以随意开启和关闭。



### 初始创建时禁用触发器

谈到触发器，还有一个问题甚至会困扰最有经验和最谨慎的开发者。在许多组织中，诸如新触发器代码之类的应用程序变更会在一个计划好且已批准的“变更窗口”内部署，这个窗口通常是影响应用程序最小的时间段，并且所有相关组件会一起部署。假设你正在为一个表创建一个新触发器（而不是修改现有的触发器）。应用程序变更已完美协调，你创建了新的触发器，如下所示：

```sql
create or replace trigger tr_u_r_test1
before update on test1
for each row
declare
    l_col1 number;
begin
    select col1 into l_col1 from test3 where col1 = :new.col1;
    if (l_col1 > 10000) then
        :new.col2 := :new.col2 * 2;
    end if;
end;
/
```

然后它失败并显示此消息：

```
Warning: Trigger created with compilation errors.
```

```
SQL> show error
Errors for TRIGGER TR_U_R_TEST1:
LINE/COL ERROR
-------- -----------------------------------------------------------------
4/2      PL/SQL: SQL Statement ignored
6/7      PL/SQL: ORA-00942: table or view does not exist
```

令你震惊的是，你发现遗漏了一个小小的先决条件——用户未被授予在表 `TEST3` 上的 `select` 权限，因此触发器主体无效。然而，由于它是启用状态，更新语句仍会触发该触发器。你可以通过检查触发器状态来确认这一点。

```sql
select status from user_triggers where trigger_name = 'TR_U_R_TEST1';
```

```
STATUS
--------
ENABLED
```

触发器处于启用状态这一事实造成了一个非常严重的情况。因为这是一个更新触发器，并且它是启用的，所以更新语句将在该表上失败。以下是一个示例：

```
SQL> update test1 set col2 = 2;
update test1 set col2 = 2
       *
ERROR at line 1:
ORA-04098: trigger 'ARUP.TR_U_R_TEST1' is invalid and failed re-validation
```

由于触发器已启用，它将为每一行被更新的行触发，但所有执行都会失败，因为代码无效。这种执行失败会导致应用程序中断。你可以通过删除触发器或禁用它来立即修复中断。

```sql
drop trigger tr_u_r_test1;
alter trigger tr_u_r_test1 disable;
```

然而，在一个繁忙的系统中，你可能无法获得锁来实施这两种解决方案中的任何一种，导致中断持续更长时间。为了解决这个问题，Oracle 11g 提供了一个更好的选项；你可以最初创建禁用的触发器，如下所示。注意 `disable` 子句。

```sql
create or replace trigger tr_u_r_test1
before update on test1
for each row
disable
declare
    l_col1 number;
begin
    select col1 into l_col1 from test3 where col1 = :new.col1;
    if (l_col1 > 10000) then
        :new.col2 := :new.col2 * 2;
    end if;
end;
```

在此触发器创建后，如果你检查其状态，它将处于禁用状态。如果触发器由于缺少对象或权限而无法编译，它不会引起任何应用程序问题，因为它不会被启用。如果触发器未能编译，你会立即知道，并能够通过添加引用的对象或授予必要的权限来纠正问题。这样，在启用触发器之前，你就拥有了所有必要的先决条件。你也可以以稍有不同的方式使用此功能；你可以在变更窗口之前创建触发器，并在窗口期间使用 `alter trigger … enable` 命令来启用它。

### 总结

由于父对象的更改而导致的依赖对象验证是不可避免的。在本章中，你学习了减少失效次数的不同技术。让我们总结一下开发失效机会更少、依赖链更短的应用程序所需了解的关键点。

*   依赖链显示了其他人如何引用对象。更改一个对象可能使依赖对象的有效性受到质疑。为了解决这种可能性，Oracle 可能会将它们标记为无效，强制稍后重新解析。
*   当对象变为无效时，可以手动重新编译它们以使其有效。当某些会话调用它们时，对象将在运行时被强制重新编译。
*   解析需要解析锁和 CPU 周期，因此会影响性能。
*   在 RAC 数据库中，失效和重新验证必须在所有实例中发生。这导致实例间消息传递和锁定增加，加剧了性能问题。
*   包提供不同类型的依赖。包体（而不是规范）依赖于在代码中引用的对象。因此，当引用的对象被更改时，规范不会失效（除非规范也引用了被更改的对象）。包的依赖项显示规范作为引用，而不是包体。因此，虽然对引用对象的更改可能会使包体失效，但包规范可能保持有效，不会导致下游失效。
*   使用包而不是独立的存储过程和函数，以缩短依赖链并减少失效。
*   向包中添加新的存储过程和函数时，始终添加到末尾，而不是中间或顶部。
*   考虑在包内使用视图而不是表，这将把表的变更与包隔离开来，减少失效。
*   更改同义词时，使用 `CREATE OR REPLACE synonym` 语法，而不是 `drop/create`。
*   不要使用类似 `select *` 或 `insert into <Table> values (…)` 的结构。明确使用列名以减少隐式列的使用，并降低失效的可能性。
*   避免引用整个行的声明，例如 `TableName%rowtype`。而是使用 `TableName.ColumnName%type`。
*   为不同类型的代码使用多个触发器。通过 `FOLLOWS` 子句建立依赖关系和执行顺序。
*   始终创建禁用状态的触发器，并且仅在它编译良好后才启用它。
*   当更改一个可能正在使用的对象时，与其关闭应用程序来获取锁，不如使用 `ddl_lock_timeout` 参数让 alter 语句等待合适的时间。

所有这些建议都只是建议，而不是需要盲目遵循的教条。你必须根据你的特定场景仔细考虑它们，并在实际实施前权衡利弊。在本章中，你了解了依赖关系如何工作、失效如何发生以及如何避免它们，从而使你的应用程序具有高可用性，并通过减少不必要的工作（如强制解析）来提高应用程序性能。
