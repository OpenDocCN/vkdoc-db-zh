## 前言

当我为《数据库设计入门》*Beginning Database Design*的第一版撰写前言时，我曾表达过希望这本书能成为一本广受欢迎的经典著作。我觉得它理应如此。作为一名技术审校者，我非常欣赏克莱尔（Clare）的技巧，她将一个通常被呈现得枯燥乏味的学科，变成了一本生动有趣的书，也欣赏她剖析思维过程的能力，这让你能够从功能需求出发，设计出能够保持数据一致性、支持增长并承载负荷的数据库。《数据库设计入门》并没有像许多书籍那样，用故作高深的术语阐述一些近乎神圣的规则。它解释了目标、常见错误、错误的原因以及你应该采取的替代方案。它揭示了规则背后的逻辑，而这一切都浓缩在一本篇幅简短、极具可读性的书中。

五年过去了，看到我的希望已经成真，《数据库设计入门》已成为这一重要主题（数据库无处不在，数据库设计是任何严肃的软件开发人员核心知识体系的一部分）下的领先著作之一，这让我深感欣慰。本版保留了第一版所有成功的特质，包括克莱尔清晰的文笔和幽默感，如果页数有所增加，那主要是为了加入一些练习，让读者可以测试自己的理解程度，并将其解决方案与提供的答案进行比较。再次担任技术审校者，我有幸见证了克莱尔对她的书所做的细微改进——其实已没有太多需要改进的地方——她在这里澄清了一个句子，在那里改进了一个例子。圣埃克苏佩里（Saint-Exupéry）——《小王子》*The Litte Prince*的作者——有一句名言：完美不是无可增添，而是无可删减。我确信克莱尔会同意我的看法，这句为飞机工程而写的话，同样适用于数据库设计。我也觉得这本书已无删减空间。

### 引言

每个人都在保存数据。大型组织花费数百万来管理其薪资、客户和交易数据。出错的代价是严重的：企业可能倒闭，股东和客户蒙受损失，对许多组织（航空公司、卫生委员会、能源公司）来说，甚至可以说个人安全也可能受到威胁。然后还有诉讼。成功设计、安装和维护如此大型数据库的问题，是许多关于数据管理和软件工程的书籍的主题。然而，大型组织内部以及小型企业、俱乐部和私人机构中也使用着许多小型数据库。当这些数据库出现问题时，虽然不会登上报纸头版；但其成本，通常是隐性的，可能同样严重。

我们在哪里能找到这些较小的电子数据库？体育俱乐部会有会员信息和比赛结果；小型企业可能维护自己的客户数据。在大型组织内部，也会有许多小型项目来维护那些不易或不便由大型系统级数据库管理的数据信息。研究人员可能保存自己的实验和调查结果；小组需要管理自己的值班表或跟踪设备；部门可能保存自己的详细账目，只向组织的财务软件提交摘要。

这些小型数据库大多由最终用户建立。他们的主要工作并非计算机专业人员。他们通常是科学家、管理人员、技术人员、会计师或教师，其中许多人在使用电子表格或数据库软件方面只有有限的技能。

由此产生的数据库往往不尽如人意。人们花费时间和精力在诸如 Microsoft Access 之类的数据库产品中建立几个表格，或者在诸如 Excel 之类的产品中设置电子表格。还花了更多时间收集和输入数据。但总是在（通常在短时间内）生成一个看起来相当简单的报告或查询时出现问题。这通常是因为表格的设置方式使得所需结果非常难以（甚至不可能）实现。

## 弄错问题

一个不能满足期望的数据库会变成一种代价高昂的练习，而且不止一个方面。我们显然有花费在建立一个不尽人意的应用程序上的时间和精力成本。然而，更严重的问题是无法充分利用宝贵的数据。对于研究数据尤其如此。科学和社会研究人员可能花费大量资金和多年时间来设计实验、聘请助手以及收集和分析数据，但往往很少考虑将其存储在设计得当的数据库中。不幸的是，一些相当简单的设计错误可能意味着大量潜在信息的丢失。眼前的目标也许能满足，但数据的未预期用途可能受到严重影响。来年的资助机会也就失去了。
