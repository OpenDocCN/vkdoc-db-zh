# 第 11 章 事件与错误处理

*我们已为任何可能或不可能发生的未预见事件做好了准备。*
——丹·奎尔，美国前副总统

在构建企业级 ETL 解决方案时，你的任务之一是使其准备好处理未预见的事件——正如丹·奎尔精辟指出的那样，那些可能发生也可能不发生的事件。部分规划以基础设施规划的形式出现（如硬件规模确定以及网络和存储容量规划），但一个健壮的 ETL 系统要求你也关注于构建错误处理。尽管你的大部分注意力将集中在可以预见的问题上，但未预见的事件可能造成最大的困扰。本章涵盖 SSIS 事件和事件处理，以及错误处理和恢复。

### SSIS 事件

如果你编写过.NET Windows 窗体程序或用于动态网页的 JavaScript 函数，你可能已经熟悉了*事件*的概念。事件是为响应特定活动而触发的。

在 GUI 驱动的.NET 应用程序中，你可能会捕获诸如鼠标点击和按键之类的事件。在像 SSIS 这样并非由 GUI 驱动的应用程序中，事件不依赖于外部用户界面交互，而是由数据处理活动驱动。

SSIS 中的事件有两种类型：带有事件处理器的事件，和*日志事件*（后者仅记录信息，没有事件处理器）。带有事件处理器的事件让你有机会运行自己定制的子程序来响应事件。在 SSIS 中，事件处理器仅仅是包含一个控制流的容器，该控制流会在响应事件时触发。你可以向此控制流添加任何任务，尽管最常用的任务往往是`脚本任务`、`执行 SQL 任务`和`数据流任务`。

SSIS 将事件纳入你设计的每个 ETL 包中。默认情况下，SSIS 会在任务、容器和包级别引发几种类型的标准事件。内置的 SSIS 事件及其触发级别列于表 11-1 中。
