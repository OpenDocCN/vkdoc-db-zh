## 网络

Azure 在网络方面的一个根本性差异在于，由于引入了负载均衡器、防火墙、虚拟化等多层架构，其延迟可能比传统的本地环境更高。优化连接 SQL Server 的应用程序网络性能的黄金法则之一是减少网络往返次数。如果无法做到这一点，建议将应用程序层整合到一台适当规格的虚拟机中。这可以确保网络调用无需离开虚拟机。

如果您有从本地环境“向上”迁移到云端的应用程序或系统，最好确定应用程序对数据库调用的网络特征以及其通信频繁程度。在无法按需更改以控制通信频繁程度和网络往返次数的情况下，您可以考虑使用 Azure 快速通道，它可以在 Azure 数据中心与本地基础设施或共置环境之间创建私有连接。根据您选择的层级和地理位置，带宽最高可达 10Gbps。

注意

微软在其内部应用程序的混合部署中使用快速通道配置，并已观察到这些环境中的性能得到显著提升，相比之下，未使用快速通道时则不然。

`Ctrl+C` 和 `Ctrl+V` 是深受全球 IT 专业人士喜爱的强大功能，但有时也会成为性能的祸根。使用复制粘贴功能或任何其他方式，在虚拟机与本地环境之间移动任何类型的数据，都会消耗您的带宽。因此，当您看到开发人员在本地机器上运行 Management Studio，连接到 Azure VM 上托管的 SQL Server 实例，并对拥有百万行数据的表执行 `SELECT *` 查询时，就该紧张了！始终建议在 Azure 中设置一个（或多个）供开发人员用于开发/测试工作的客户端工作站。如果需要将数据移入或移出 Azure 区域或本地环境，出于性能和成本考虑，最好在移动前对数据进行压缩。

## 存储

性能的下一个方面是存储布局的配置。虚拟机的 Azure 存储账户和计算资源应位于同一数据中心内，以减少传输延迟。如果未能做到这一点，性能将呈指数级下降。这类似于将床和床垫分别放在两个房间。床垫的功能可能得以发挥，但只有将两者组合在一起，才能真正实现其全部价值！

建议对托管在实例上的实例和数据库使用 SQL Server 的高可用性和灾难恢复功能，因为无法保证跨多个磁盘的写入顺序是一致的。这主要是为什么推荐用于托管 SQL Server 数据库文件的数据磁盘使用本地冗余存储（`LRS`）选项。需要记住的另一个方面是，`LRS` 存储的入口/出口带宽是其同类配置（如地域冗余存储/区域冗余存储）的两倍。

第一个自动化检查是使用虚拟机属性，比较计算资源和存储的位置，确保它们位于同一位置。实现此自动化的步骤是：检索附加到虚拟机的数据磁盘的存储配置文件，并确定是否为它们启用了复制。清单 7-2 中的 PowerShell 脚本假设资源组中的存储账户与部署在同一资源组中的 SQL Server 实例相关联。它会检查是否启用了存储复制以及存储和计算是否位于同一位置。
