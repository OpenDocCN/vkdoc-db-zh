# 9. ODA 的商业价值

摘要

2011 年 9 月，Oracle 宣布推出 Oracle 数据库一体机，作为一款创新的、针对数据库优化的入门级集成系统。该发布承诺了易于部署、按需增长付费的许可模式以及单一供应商支持。

当一项新技术被引入，并且商业兴趣逐渐浓厚时，公司内部总会有人问：“这比我们现在的方式是一个更好的解决方案吗？”。回答这个问题并克服变革阻力的第一步是调查该新技术的商业案例。Oracle 数据库一体机已成为 Oracle 集成系统产品线中非常成功的产品。本章概述了 ODA 提供的商业优势，您可以利用这些信息来考虑部署 ODA 解决方案的商业案例，并评估任何潜在的技术风险。


## 业务挑战

许多公司在应对日益扩大的业务压力范围时面临着相同的挑战。IT 部门被要求变得敏捷，并快速提供基础设施，甚至是在所有业务需求尚未完全明确之前。项目时间线正变得更短且分为多个阶段。虽然项目在初期阶段可能会延迟，但这并不意味着项目的截止日期必然会被更改。相反，项目的后期阶段可能需要以更短的时间线来弥补延误。数据库部门通常处于基础设施供应流程的末端。数据库管理员（DBA）被要求快速部署新的数据库基础设施，并尽早向 IT 客户提供业务价值。如果当轮到 DBA 开始工作时项目已经延迟，那么实际上准备时间会降至零。遵循收集需求、订购硬件、并让多个团队通过大量交接来构建数据库基础设施的旧有周期性做法，已完全无法满足现代业务的时间要求。

许多公司面临着来自固定预算的成本压力。新的外部公共云产品正在竞争它们的基础设施预算。多年来，一直存在向商用硬件解决方案缓慢转移以降低成本的趋势。尽管与硬件相关的成本因商用解决方案而下降，但构建和支持数据库基础设施解决方案的成本仍然很高。

公司常常面临拥有高端技能人才短缺的挑战。总体而言，IT 人员、DBA 和其他员工并不短缺。然而，“在任何经济环境中都难以找到优秀人才”这一说法确实持续存在。公司总是在寻找能够解决任何技术问题、快速完成工作并迅速建立专业领域知识的人才。

新技术的快速引入给高端技能人才的可用性带来了额外压力。这些新技术的快速引入可能要求支持人员在没有太多指导的情况下迅速掌握知识。支持人员必须对当前的挑战做出快速反应，任务必须迅速完成。几乎没有犯错或发生导致项目延误的问题的余地。常见问题包括：

*   如今员工的工作负荷确实已经饱和。没有时间进行“在职培训”（OJT）和研究。一切都必须被快速启动。
*   项目的按小时计费咨询成本可能非常高，加剧了成本压力。这增加了快速部署和提高效率的压力。

在审计和行业法规管理着公司为保护其系统而必须采取行动的时代，安全合规性变得至关重要。在数据库服务器和数据库均为定制构建的环境中，要实现整个应用和数据库组合的合规性非常困难。

## ODA 登场

Oracle 开发了`Oracle 数据库一体机`（`Oracle Database Appliance`），即`ODA`，旨在为上一节提到的业务挑战提供解决方案。如果`ODA`有一个核心主题，那就是“**傻瓜式基础设施部署**”。虽然有些人可能会对这个说法感到意外，但它实际上是为了证明`ODA`部署的简单性和优雅性。`ODA`的其他主题包括使用一键式流程部署的省时与便捷、`RAC`部署的简易性以及管理的便捷性。查看`ODA`设置海报后，您就会明白 Oracle 正在提供一种新的业务开展方式。

当前的`X3-2 ODA`型号是一个**集成系统**（engineered system）。集成系统是指设计为作为一个单一、集成的打包解决方案协同运行的硬件和软件。虽然公司可以构建自己的商用硬件解决方案，但很少有公司有能力构建自己的集成系统。大多数公司根本没有资源或商业效益来构建自己的集成系统，更不用说构建一个为高可用性 Oracle 数据库优化的系统了。

`X3-2 ODA`预打包了两台服务器，共计 32 个 2.9Gz `Intel E5-2690` CPU 核心。一台`ODA`包含一个存储单元，其中有 20 个 900G 数据驱动器，总原始存储为 18T。可以通过添加第二个存储扩展机架（额外提供 18T 原始存储）进行扩展，总计 36T 原始存储。`ODA`部署时采用双份或三份镜像（可在安装时配置），在单个存储单元上，使用三份镜像时可用存储约为 6T，使用双份镜像时约为 9T。应参考`ODA 入门指南`，根据所选部署选项了解镜像存储选项和磁盘组的确切容量规格。当前的`X3-2`服务器每台配备 256G 内存。包含四个 SSD 驱动器，总存储为 800G，用于联机重做日志。

**注意**

`ODA`的容量必将随着底层 Oracle 服务器硬件产品线的更新换代而变化。几乎在硬件型号发布的同时，Oracle 就开始开发下一代替代产品。

第一代`ODA`是一个固定的、独立的一体化设备。对于`X3-2`第二代型号，服务器和存储单元被模块化，以便产品线能够利用 Oracle 的产品路线图，跟随 Intel 产品线的发展发布新硬件。

`ODA`提供了来自单一供应商的集成系统，具有单一的问题上报点。该设备的管理主要是自包含的，并与 Oracle 全面的`Oracle Enterprise Manager`（`OEM`）管理和监控解决方案集成。

`ODA`已被各种规模的大量客户群购买和部署。许多财富 500 强公司报告称，从资源规模角度来看，它们的大部分数据库都可以部署在`ODA`上。未来可以期待容量更大、性能更高的型号。

`ODA`上也可使用虚拟化功能，以支持使用`Oracle VM 虚拟化管理器`（`OVM`）模板快速部署数据库和应用。已经发布了`WebLogic`的模板，Oracle 也已宣布将提供额外模板。

在可预见的未来，`ODA`将在 Oracle 集成系统产品阵容中保持其地位。2013 年 8 月，Oracle 扩展了其集成系统产品阵容，推出了一款名为`Oracle 虚拟化计算一体机`（`Oracle Virtualized Compute Appliance`），即`OVCA`的新产品。该系统提供了一个通用的`Oracle VM`（`OVM`）供应解决方案，而`ODA`则提供了较低成本的入门点，并且其虚拟化已针对运行数据库进行了优化。


## 快速、简单的数据库基础设施部署

ODA 提供的首要业务优势是快速便捷的数据库基础设施和数据库实例部署。部署 ODA 无需单独规划设备、配置存储区域网络 (`SAN`) 基础设施、部署 `RAC` 私有互连网络、安装操作系统 (`OS`)，也无需安装 `RAC`、`ASM`、网格基础设施、Oracle 数据库软件，或指派项目经理来协调所有这些工作。

部署 ODA 只需按照设置海报和部署文档中概述的步骤操作。服务器接入网络并构建部署配置文件后，构建 ODA 的时间略少于一小时。成果是一个双节点 `RAC` 集群，包括文件系统、`ASM` 存储、网格和数据库主目录的配置，一个本地的 `OEM` 数据库控制台，以及一个初始数据库。您的公司可以在不到一天的时间内在 `RAC` 集群上运行应用。

如果部署时间对您的公司至关重要，那么 ODA 将满足在完成服务器布线和其他网络依赖项后数天内，就拥有完全部署好的高可用性数据库和虚拟化应用解决方案的需求。普通的 DBA 只需阅读文档即可部署 ODA。如果您的公司正在部署更大规模的 ODA 产品组合，那么为团队增加额外的技能可能很有意义。

在部署了我们的第一台 ODA 之后，我们统计了在其他自己搭建的平台上构建数据库所涉及的预部署、安装和部署后步骤的数量。在不创建数据库的情况下，涵盖数据库服务器所有基础步骤的数量在 103 到 107 步之间。而在 ODA 上完成相同结果并部署一个 `RAC` 集群的等效步骤数量，仅二十几步。不用说，我们估计构建和设置时间的缩减在本质上是几何级数的。每家公司都应该自行统计使用当前构建流程来构建数据库服务器和高可用性数据库所需的步骤数量以及团队间的交接次数。

我们在部署 ODA 数据库基础设施的成本方面也经历了类似的降低。每家公司都有其定义组织角色和职责的方法与规则。可以说，如果这符合贵公司对角色和职责的看法，您可以在没有 `SAN` 团队和系统管理员参与的情况下构建 ODA。部署 ODA 所需的网络工作量也有所减少。

我们与一些公司讨论了构建 ODA 与他们目前做法所需资源的差异，许多公司报告说，这种范式转变可能会引发一些有趣的内部讨论，因为他们正将战略转向工程化系统。角色和职责的转变可以提前解决，也可以随着公司对平台更加熟悉并获得更多经验而逐渐改变。

这就引出了问题：“部署 ODA 需要多长时间？”。要回答这个问题，您必须意识到 ODA 与其他任何工程化系统一样存在依赖项。您仍然需要订购 ODA 并安排发货。ODA 需要相当简单的网络布线，并且您仍然需要完成任何 `DNS` 和防火墙请求。然而，其中大部分工作可以在 ODA 到达之前并行完成。通过仔细规划，ODA 可以非常快速地部署，实际的构建工作耗时不到一天。

作者第一次使用 ODA 是在一个需要高可用性和非常快速部署的项目中。我们需要订购设备并在两周内让四组 `RAC` 集群投入运行。当时，ODA 平台还很新，但我们决定冒险一试，因为这是我们最佳的选择。我们订购了四台 ODA 并要求立即发货。ODA 在周一到达，我们在周二进行了上架和布线。一些 Oracle 的人员在周三上午飞抵，看看是否能提供帮助，因为这个平台非常新。我们前往数据中心，在周三下午并行构建了所有四台 ODA——Oracle 的人员在一旁观察。

过程进展得非常顺利，以至于我们决定取出一台 ODA，在整个 DBA 部门的午餐时间面前，现场演示从裸机搭建一个全新、可运行的生产环境 ODA `RAC` 集群。我们在大家吃完午餐时正好完成了部署。人们明白他们正在目睹一种构建数据库服务器的新范式。在那一天之前，仅凭几名 DBA 在一个下午并行构建四个 `RAC` 集群是不可想象的。表 9-1 列出了许多由于部署 ODA 而不再需要关心的工作。

那么接下来发生了什么？如今，被应用团队或项目经理问到“我们能把这个数据库放在 ODA 上吗？”并不少见。当您得到这类问题时，说明您处于一个很好的位置。

ODA 安装过程消除了不可预测、浪费时间的设置问题和交接，这些问题会导致额外的协调、依赖和延迟。部署时间以天计算，而不是周或月。ODA 在满足项目需求挑战方面提供了很大的灵活性，这是许多数据库和基础设施部门必须应对的。

ODA 可以快速预部署，以避免在需要之前购买容量。它们是支持基础设施即服务 (`IaaS`)、数据库即服务 (`DBaaS`) 或基于云的服务以支持公司 IT 战略的，您大多数 Oracle 数据库的完美平台。当您被问到：“您是如何在 `DBaaS` 服务中如此快速高效地提供数据库的？”——那么您就处于一个更好的位置了。ODA 并非能满足每个项目需求。`IOPS` 数量存在限制，计算节点目前无法扩展到两个以上。然而，您可能会发现 ODA 能满足您大多数数据库的需求。


## 轻松搞定 RAC

构建稳定且满足高可用性要求的 RAC 集群需要大量的前期规划和工作，以及安装前后的环境验证检查。除了 RAC 安装手册外，还需要分析其他文档：

*   [RAC 通用文档 (MOS 810394.1)](https://support.oracle.com) 和任何特定平台的入门套件
*   [RAC FAQ (MOS 220970.1)](https://support.oracle.com)
*   [Oracle 的支持说明 (MOS 1344678.1)](https://support.oracle.com)，提供了稳定集群的步骤，并附有平台特定的扩展
*   [RAC 信息中心说明 (MOS 1452965.2)](https://support.oracle.com)
*   您特定平台的 RAC 设置建议
*   RAC 已知问题支持文档

此外，您还需要查阅任何能提供关于设置 RAC 以支持其他 Oracle 产品（如 eBusiness Suite 和 ATG Web Commerce）建议的文档。

如果您要构建自己的 RAC 集群，还需要学习额外的工具。Oracle 构建了各种工具来帮助验证 RAC 环境，例如：

*   `RACcheck` (MOS 1268927.1)
*   `Cluster Verify` (MOS 316817.1)
*   用于数据库检查的 `RDA` (MOS 314422.1 和 250262.1)

当然，还有所有操作系统特定的设置、软件包要求和检查。在传统平台上完成所有这些努力是非常值得的。幸运的话，所有这些内容会被记录一次，并且会有人努力跟上所有的文档更新和 `RACcheck` 结果，以保持“自己动手”文档的更新。然而，这样做需要大量的时间、合适的人员、部门间大量的交接，以及大量的后续质量保证工作以确保工作正确完成。

对于大多数公司，尤其是中小型公司来说，记录和分发复杂的技术信息并培训人员理解这些信息是一个挑战。增加挑战的还有实施所有这些要求所需的多团队间的所有交接。有人可能忘记将互联流量路由到冗余的、专用于 RAC 的交换机（这些交换机也很昂贵），或者忘记在服务器和交换机上都配置巨型帧。还可能存在与支持 IO 多路径的外部供应商软件相关的额外问题。由于所有这些场景都非常真实，Oracle 发现有必要投资建设广泛的 RAC 诊断设施。

Oracle 正在通过扩展 `RACcheck` 和其他工具来捕获设置问题，帮助管理这种复杂性。然而，结论是，Oracle RAC 保证组和支持人员如今非常忙碌是有真实原因的。一体化系统和 ODAs 将正确部署 RAC 的大量工作从您的肩膀上卸下。

ODA 通过使 RAC 配置自包含，为所有这些挑战提供了一个解决方案。没有外部互联需求或需要在标准 ODA 部署过程之外进行 RAC 配置更改的说明。如果您担心您的公司是否有能力成功完成构建真正高可用性 RAC 集群所需的所有工程工作，那么 ODA 可能有助于提供一个解决方案。成功部署 RAC 集群所需的专业知识大大减少。这对各种规模的公司都是一个主要优势。ODA 将成功部署双节点 RAC 集群的能力扩展到大众，这也是 ODA 成为“**轻松搞定 RAC**”解决方案的部分原因。

在 ODA 安装过程中，可以部署 RAC One、双节点 RAC 集群或单实例数据库。ODA 在几小时内根据 Oracle 的最佳实践部署 RAC 集群。Oracle 在设备内置了物理 RAC 互联。季度自动补丁流程简化了为 `grid/clusterware` 和 Oracle Homes 打上最新 PSU 补丁和错误修复的过程。这些季度补丁与操作系统和固件更改一起作为一个完整的单元进行测试，使 ODA 成为维护持续 RAC 稳定性的绝佳解决方案。

> 注意：作者曾与多家小型公司讨论过非 ODA 系统上的 RAC 安装。许多中小型公司根本没有资源自己执行此类安装。这包括研究所有 RAC 要求的技术资源，以及为互联流量购买额外设备（如专用 RAC 交换机）的成本。

虽然 ODA 在部署和支持 RAC 集群方面表现出色，但对于任何平台上的 RAC 部署团队来说，理解良好的 RAC 设计最佳实践以消除节点间的争用仍然是需要了解的。此外，双节点集群要求一个服务器能够在节点切换期间处理整个处理负载。用 `Data Guard` 补充 RAC 可以缓解故障转移容量方面的担忧。

### 轻松实现高可用性

ODA 通过易于安装的`RAC`和`RAC One`部署来提供高可用性。除了在单个设备上配置高可用性外，ODA 还便于部署`Data Guard`以扩展高可用性解决方案。这些部署可以跨越数据中心，以提供地理层面的保护。Oracle 通过`OEM`、一键式 ODA `Data Guard`部署脚本，以及专注于在 ODA 上部署`Data Guard`的补充文档，来协助`Data Guard`的部署过程。除了简化的`Data Guard`部署外，ODA 通过更高水平的稳定性支持高可用性，因为它是一个经过整体测试的打包解决方案。

ODA 补丁的网格和数据库部分可以以滚动方式部署。在撰写本文时，ODA 补丁的基础设施部分要求两个节点上的集群件完全关闭，但这在未来可能会改变。`RAC One`为生产环境和非生产环境实例提供高可用性能力，这些实例设计为仅在两个 ODA 节点中的一个上运行，但可以故障转移到第二个 ODA 节点。

ODA 通过大大降低部署`RAC`解决方案所需的专业知识水平，快速启动高可用性实施。这包括无需执行`RAC`互联设计、无需研究所有`RAC`要求和最佳实践、无需配置`ASM`、无需消除所有单点故障，以及一个精心设计的`RAC`集群的所有其他方面。

高可用性也通过部署经过标准测试的配置以及选择为所有`ASM`磁盘范围进行双倍或三倍镜像来实现。丢失一个磁盘不会导致系统停机。ODA 操作系统还实现了 Oracle Linux 坚不可摧内核（`UEK`）。ODA 由一个专门的 Oracle 工程系统支持团队提供支持，该团队对该平台有深入的了解。

大多数硬件组件都是冗余的。一个 ODA 有两个服务器节点和冗余电源。系统各组件之间的系统互联是冗余的，在当前的 X3-2 型号中通过`SAS`电缆实现。

在 ODA 上更换磁盘很容易执行。只需拨动存储单元前面的开关，拔出驱动器，滑入新驱动器，然后将开关拨回原位。

ODA 减少了实现高可用性所需的活动部件数量。在大多数情况下，更少的活动部件和更少的交接意味着更高的可用性。

有几种高可用性部署模式：
*   **双活（Active-Active）**。这种模式通过部署在 ODA 上的`RAC`实现，提供跨两个节点的负载均衡处理能力以及在两个服务器之间故障转移的能力。
*   **主备（Active-Passive）**。这种模式通过`RAC One`、`RAC`和/或`Data Guard`实现。处理被配置为在单个节点上运行。在 ODA 内部，由`RAC One`或`RAC`支持故障转移。跨 ODA 的故障转移，包括跨地理距离故障转移的能力，由`Data Guard`处理。

实现高可用性需要额外的设计，例如`RAC`服务，以及如果`Data Guard`是解决方案一部分的话，还需要基于角色的服务。你的高可用性实现的有效性将与投入解决方案的设计和测试量成正比。

### 成本与价值主张

ODA 不符合传统的“自建还是购买”采购模式。公司当然可以选择构建自己的基于通用硬件的基础设施。然而，典型的公司没有资源来构建自己的工程系统。无论如何，请查看 Oracle 关于“自建还是购买”ODA 的视频。你会在主要的 Oracle ODA 产品页面和 Oracle 的 YouTube 频道上找到一些视频，详细说明了部署的简便性以及你为何需要认真考虑购买一个更好的、预打包解决方案的价值主张。有些视频带有一定的幽默元素，因此非常值得观看。内容是动态的，因此我们没有包含任何网络链接，因为它们可能会更改。不过，找到它们是相当容易的。

从硬件角度来看，购买 ODA 时你会获得大量资源。对于当前的 X3-2 型号，这包括两个服务器，每个服务器有 256G 内存，总共 32 个核心，以及大量存储。当前的存储量是 18T 原始磁盘，可以部署为大约 9T 可用的双倍镜像，或 6T 可用的三倍镜像。通过存储扩展机架，存储可以再次加倍到 36T 原始存储。ODA 入门手册根据所选的部署选项详细说明了确切可用的存储量。

然而，ODA 的秘诀不在于硬件。ODA 的秘诀在于软件。其成分包括在数小时内完成`RAC`部署、对所有层级（`BIOS`、`固件`、`OS`、`ILOM`、`网格`、`数据库`）的自动化修补，从而使系统保持最新、稳定和合规。ODA 硬件——带有标准的、嵌入式的 Oracle 硬件支持设施，并辅以大量增值管理软件——提供了极佳的价值主张。优势包括以下：
*   ODA 附带一个嵌入式验证工具包（`oakcli`——Oracle 应用套件命令行界面）。`oakcli`包含一套完整的诊断和管理实用程序，以维护 ODA“保持简单”的管理主题。
*   ODA 允许你在硬件上限制已部署的许可和资源，采用“按成长付费”模式，而不是预先支付所有容量和许可的费用。额外的容量通过申请密钥来启用。
*   预打包的软件和部署模型提供了在整个数据库组合中部署标准化数据库平台的能力。ODA 是解决传统手工构建服务器和数据库模式的方案，该模式导致数据中心充满了“图腾柱”，没有两个图腾柱看起来是一样的。

毫无疑问，ODA 标准化部署模式降低了管理和行政成本。这部分是由于构建步骤的减少，以及需要参与的人员和团队数量的减少，同时也减少了团队之间的所有交接。ODA 部署后，系统的统一性质和平台特定的自动化降低了持续的管理成本。持续的管理节省来自于：支持一个 ODA 所需的团队数量减少、平台的稳定性、对所有设备组件的自动化修补、更少的活动部件、服务器和组件的冗余、应用套件自动化，以及包括`ASR`和`ILOM`在内的嵌入式支持设施。

快速设置和标准化模型是`DBaaS`或云服务的绝佳构建块。在 ODA 上创建新的单实例、`RAC One`和`RAC`数据库的过程已缩短至 15 分钟。ODA 是工程系统，可以由任何规模的公司部署。ODA 通常作为 Exadata 的低成本低生命周期环境，同时也是那些希望长期投资于 Exadata 的公司的切入点和初始成功故事。



### Oracle ODA 平台概述

Oracle 正在对该平台进行持续投入。新功能按季度发布，且软件变更向后兼容之前的硬件世代。ODA 最初是面向中小企业的解决方案，但随后大型企业也开始采购它。这有助于推动 ODA 产品获得更大的发展动力。由于 ODA 是单一供应商产品，消除了供应商间交接的问题，所有组件作为一个整体的测试都在 Oracle 内部完成。从故障排除的角度看，客户只需联系一个供应商来处理所有支持问题。当发起支持请求时，它会由专门设计的系统支持团队处理。

快速部署基础设施或预部署基础设施的能力，通过让项目快速启动以实现业务收益，从而带来整体成本节约。ODA 随附足够的资源，可作为整合平台并支持您的内部共享服务计划。ODA 也支持标准的 Oracle 资源管理工具，例如实例锁定和数据库资源管理，以支持整合工作。

最终，公司需要决定通过自行开发解决方案与购买像 ODA 这样的预打包解决方案相比，愿意承担多少工作和开销。ODA 的一个关键好处是，它让公司能够将时间花在核心业务功能上，而非日常支持任务上。

关于总体拥有成本（TCO）数字呢？Oracle 会公布数字——就像所有供应商为其竞品所做的那样。Oracle 的 ODA TCO 研究目前可在 Oracle 的主要 ODA 产品页面上找到。Oracle 的数字对于支持您的初始采购可能有帮助。然而，最佳的 TCO 数字是您随着对该平台的深入了解而为您的公司量身定制的数字。

### Oracle 硬件

ODA 利用标准的 Oracle（原 Sun）硬件管理工具来管理设备并增强其抵御常规故障的能力。这些管理工具包括 `ILOM` 和自动服务请求（`ASR`）工具。正如您之前在本书中所见，`ILOM` 允许您远程管理硬件，例如将其关机和开机，并运行一整套诊断程序。`ILOM` 的功能非常全面，在撰写本文时，Oracle 已发布了八本独立的手册来涵盖所有功能。同样，`ASR` 软件提供“电话-home”能力，以自动发出警报并将硬件故障消息上传到 Oracle 以启动服务请求流程。

ODA 硬件既是模块化的也是弹性的。服务器是模块化的，由两个独立的单元通过线缆连接而成。同样，存储也是模块化的。电源是冗余且可现场更换的。磁盘驱动器是可热插拔的。

Oracle 硬件由现场服务部门支持，该部门将修复遇到的任何硬件故障。支持现场服务部门的是 Oracle 支持组织，该组织将通过协助诊断以确定问题原因来响应服务请求，然后派遣现场服务小组来解决问题。

### 安全与合规性

ODA 的第一个安全特性是在季度 ODA 版本中包含季度 `PSU` 错误修复和安全补丁。季度 ODA 补丁记录在 `MOS` 注释 `888888.1` 中。Oracle 在发布季度 `PSU` 补丁集后大约两到四周发布季度 ODA 版本。Oracle 采用这种方法将最新的安全补丁包含在每个 ODA 版本中，并继续将补丁集的所有组件作为一个完整单元进行测试的流程。一键补丁过程的简便性是确保 ODA 定期打补丁以保持系统合规性的关键推动因素。保持系统安装最新的安全补丁是 `PCI`（信用卡行业）和其他安全合规性认证的要求。

ODA 的第二个安全实施特性是在开发发布过程中的安全扫描。Oracle 使用来自第三方供应商的安全扫描软件独立执行这些检查。扫描结果用于升级部署在 ODA 上的 `Oracle Linux` 软件包版本，以保持系统合规性。

第三个锁定 ODA 的安全机制是 ODA 特定的、由 Oracle 提供的 `STIG` 脚本（`MOS` 注释 `1456609.1` 和 `1461102.1`）。`STIG` 脚本是美国国防部安全技术实施指南（`STIG`）流程的一部分。运行 `STIG` 脚本是一个两步过程。第一步，脚本可以在检查模式下运行以查找安全违规。第二步，“修复”过程允许纠正这些被标记的违规行为。

通过查看代码，可以发现一些 `STIG` 安全检查包括以下列表。执行的安全检查列表未在 `MOS` 注释中详细说明。

*   类别 1：这是国防部的说法，意为“你最好修复这个问题。”
    *   `grub` 的密码是否启用？
    *   `sendmail decode` 命令是否在 `/etc/aliases` 中被注释掉了？
    *   特权账户 `halt` 是否存在？
    *   `Ctrl-Alt-Del` 组合键是否可用于关闭系统？
    *   `RealVNC` rpm 是否安装在服务器上？
    *   内核中是否找到了对 USB 设备的支持？
*   类别 2：这是国防部的说法，意为“记录这些问题并制定计划来修复。”
    *   单用户模式启动是否无需密码即可启用？
    *   `pam-tally` 账户配置和登录失败管理工具是否配置为在连续三次登录失败后锁定账户？
    *   系统是否禁止使用过去的密码？
    *   密码强度有多安全？密码是否少于八个字符？
    *   是否配置了延迟，使用户在登录失败后再次尝试登录前需要等待？
    *   密码是否必须每 `60` 天至少更改一次？
    *   密码是否可以在 `24` 小时内更改超过一次？
    *   `cron` 访问是否受控？
    *   是否可以通过 `ssh` 直接以 `root` 身份登录系统？
    *   `tcp dump` rpm 是否安装在系统上？
    *   所有文件系统的权限是否正确？
    *   是否存在任何不必要的账户？
    *   `sendmail help` 命令是否启用？
*   类别 3：这是国防部的说法，意为“记录风险并决定如何处理。”
    *   `UNIX` `man` 页面权限是否正确？
    *   `sendmail` 版本是否被隐藏？

`STIG` 脚本中还有一些额外的检查项，即使是您经验丰富的系统管理员也可能需要查阅和弄清楚。`STIG` 脚本的目的是检查您的安全设置实践，而不是 ODA 的开箱即用安全实现。搜索“我的 Oracle 支持（`MOS`）”只找到了针对 ODA 和 `Exadatas` 的 `STIG` 实施文档。这个安全实施检查脚本是 ODA 的另一个增值主张。

您选择实施哪些 `STIG` 脚本修复将取决于您公司的安全标准。并非每家公司的安全要求都与美国国防部相同。然而，安全是某些公司选择谨慎行事的领域。至少，`STIG` 流程执行了一个 ODA 特定的国防部安全分析。

Oracle 还发布了 `Oracle Linux` 安全手册，`Oracle Linux` 团队也发布了额外的博客，介绍锁定系统的步骤。虽然将这些步骤直接转换到 ODA 上并不总是容易的，但它们确实是指导您系统安全锁定的宝贵指南。虽然全面覆盖服务器安全实施超出了本章的范围，但可以采取额外的步骤来锁定您的 ODA 系统，例如通过跳板机限制对 ODA 的访问。




### 虚拟化技术

2013 年 3 月，Oracle 在软件版本 2.5 中发布了对 ODA 上 Oracle 虚拟化解决方案的支持。此后，Oracle 在每个版本中都持续增强 ODA 的虚拟化能力。在 ODA 上实施虚拟化是使用 ODA 的一个主要业务场景。Oracle 在 ODA 上实施的虚拟机（VM）目前是独一无二的，因为其虚拟化已针对数据库进行了优化。预计虚拟化将继续成为 Oracle 的一个重要发展方向，包括增强 ODA 上的功能。

Oracle 通过在`oakcli`中实施 VM 管理，简化了 ODA 虚拟机的管理，包括克隆、启动和停止 VM 的功能。`oakcli`已得到增强，可以导入模板。支持 ODA 上的虚拟化无需单独的 VM 管理器。

ODA 平台的“按需扩展能力”意味着 ODA 上可以提供额外的容量来运行非数据库应用和其他基础设施支持软件。Oracle 为打包的 ODA 应用解决方案创造了一个短语——“一体化解决方案”。Oracle 正在将其软件移植到 ODA 专用的 VM 模板。在撰写本文时，一个 WebLogic 模板已经为 ODA 发布，并且已宣布了一个 JD Edwards 模板。预计还会有更多针对其他 Oracle 软件产品的模板。Oracle 的合作伙伴也已加入进来，开始开发他们自己的应用“一体化解决方案”。

最大化硬件利用率并快速部署应用的商业价值主张是巨大的。各种规模的公司通常需要部署大型咨询公司、系统集成商和咨询人员来部署复杂的软件。通过 VM 模板在 ODA 上部署应用，有望通过快速部署标准化镜像来显著节省成本。Oracle 已声明未来将发布更多的 ODA VM 模板。

### ODA 技术解决方案

ODA 是一种技术上的“一体化解决方案”。该解决方案包含许多您应该了解的主要组件。我们将在本节介绍这些内容。

Oracle Appliance Kit（`oakcli`）是一个内置的、多功能的设备管理工具。`oakcli`命令参考可以在 MOS 1417879.1、最新的 ODA 版本发行说明以及 ODA 入门指南中找到。ODA 发行说明和入门指南通常比 MOS 说明更新。新的`oakcli`功能每季度发布一次，因此您需要主动查看季度发行说明以了解最新功能。

以下是您可以使用`oakcli`执行的一些操作：

*   ODA 管理
    *   部署 ODA
    *   为 ODA 打补丁
    *   大约 15 分钟内创建一个新数据库
    *   从当前 ODA 版本创建新的数据库软件主目录
    *   删除数据库和数据库主目录
    *   运行完整的系统检查
    *   运行磁盘校准
    *   配置和测试 ASR（自动服务请求）
    *   显示硬件详细信息和版本
    *   显示软件和固件版本
    *   显示磁盘、磁盘组和其他存储详细信息
    *   清理已应用的补丁
    *   购买额外的 Oracle 许可证时激活核心
*   诊断
    *   运行完整的系统检查
    *   定位磁盘驱动器并点亮其 LED 指示灯以指示需要更换的具体磁盘
    *   运行磁盘诊断
    *   运行`ODAchk`运行状况检查
    *   收集并打包诊断信息
*   虚拟化
    *   创建或克隆 ODA 虚拟机
    *   配置 ODA 虚拟机上的资源
    *   启动和停止 ODA_Base（虚拟化 ODA 上的数据库分区）或其他 ODA 虚拟机
    *   显示 ODA VM 模板的信息
    *   部署/导入 Oracle VM 模板
    *   为 ODA 虚拟机创建 CPU 池
    *   显示一个或所有 ODA 上虚拟机的详细信息

    *   打开 VM 控制台以管理 ODA VM

Oracle 还提供额外的 ODA 诊断实用程序作为 ODA 软件产品集的一部分。主要实用程序是`ODAchk`（MOS 1485630.1）。这是`RACcheck`的 ODA 专用版本。Oracle 建议您部署最新版本的`ODAchk`，因此在`RACcheck`和 Exadata 的`Exachk`工具更新后，您必须检查 MOS 说明以获取新版本。

ODA 附带标准的`top`、`vmstat`和`sar`服务器资源监控实用程序。ODA 还附带详细的服务器资源收集实用程序`OSWatcher`（MOS 301137.1 和 461053.1）。ODA RAC 软件还包含两个适用于自安装平台的标准 RAC 实用程序：

*   集群运行状况监视器（MOS 736752.1 和 1328466.1）— 细粒度资源监控
*   跟踪文件分析器实用程序（MOS 1513912.1）— 实时事件捕获

ODA 附带一组专为性能和在 ODA 上整合数据库而设计的数据库模板。数据库模板的大小调整记录在入门指南中。模板的大小范围从超小型、小型、中型、大型、超大型到超超大型。除了配置内存、重做日志大小和初始数据库大小外，这些模板还决定了可以在 ODA 上部署的数据库数量。

以下是一些需要了解的其他软件实用程序和工具：

*   ODA 附带两个允许启动基于 GUI 的工具会话的实用程序。它们是 VNC 服务器和 ILOM 远程控制台的 StartX。
*   服务器软件包括前面讨论的 ILOM 和 ASR 设施。这些服务器管理实用程序是 Oracle 相对于通用商用硬件的额外硬件增值点。
*   ODA 预部署了一个名为`Logwatch`的实用程序，用于监控文件系统空间以及通过`ssh`进行的身份验证失败和登录。它是自动计划的。
*   ODA 可以运行其他主动支持工具（MOS 1459344.1）。一个很好的例子是`oratop`（1500864.1），它生成一个显示整体数据库和数据库服务器活动的单面板。

ODA 具有独特的“最坏情况”恢复能力。当一切丢失，服务器操作系统磁盘配置失效且数据库无法启动时，只要问题根源不是重大硬件故障，ODA 提供了使用裸机恢复来重建设备的能力。使用此方法，ODA 可以在大约两小时内重建。当然，您将需要数据库的备份以及任何需要恢复以使 ODA 进入可操作状态的额外对象，例如导出文件和用于 RAC 服务的 OCR。

最后，ODA 附带 Oracle Secure Backup 产品的许可证，用于备份 ODA。
