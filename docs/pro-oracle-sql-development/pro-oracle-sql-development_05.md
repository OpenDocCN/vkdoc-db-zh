# 3. 通过测试增强信心与知识

测试不仅仅是简单地证明程序正确。我们必须不断挑战自我以让自己变得更好。同样，我们也必须挑战我们的程序和想法。测试是我们将科学引入技艺的方式，用关于世界如何运作的真实知识取代猜测和迷思。构建好的测试可能需要大量的时间和精力，而 Oracle 数据库为构建可复现的测试用例带来了独特的挑战。幸运的是，Oracle 提供了许多工具来帮助我们测试，例如 SQL、PL/SQL、数据字典、动态性能视图等等。

## 通过自动化测试建立信心

自动化测试需要大量投入，但几乎总是值得付出这些额外努力的。在我们决定采用自动化测试之后，必须决定如何处理数据以及如何构建自动化测试框架。

### 更快地修复错误

编程是困难的，充满了模糊的需求、意外的副作用和退化问题。完全的自动化测试显然会提高我们程序的质量。自动化测试令人惊讶之处在于，它最重要的好处是微妙的，并且与质量只有间接的关系。

减少修复错误的时间是第一个明显、直接的好处。测试是痛苦的，我们中很少有人喜欢做这件事。我们总是找借口逃避它，而且有大量的认知偏见帮助我们忽略测试。手动测试比自动化测试更容易——如果只做一次的话。我们自欺欺人地认为，“这部分我们只需要测试一次；等代码能运行后，我们就不会再改它了，所以没有机会再出错。”或者也许测试是别人的工作，我们认为自己不必担心。

当我们对自身超人编程能力的假设不可避免地被证明是错误时，需要付出巨大的代价。在刚制造错误后立即自己修复，会容易几个数量级。上下文在脑海中还很新鲜，也没有沟通成本。不需要与测试人员讨论错误，不需要在跟踪系统中记录缺陷，也不需要多次向管理层解释问题。构建一个自动化测试系统符合我们自身的利益，即使存在独立的测试团队。

### 获得信心并避免偏见

自动化测试让我们对程序充满信心。它不仅能切实帮助我们晚上睡得更好，还让我们能够自由地进行更多更改。

关于小改动引发意外副作用的可怕故事比比皆是。为了避免这些噩梦，许多程序员只做尽可能小的改动。但为了节省昂贵的手动测试时间，程序员需要将改动打包在一起。采用手动测试时，每次发布都是一种妥协。要么我们因节省昂贵的测试而承担错误的风险，要么我们的代码因为避免改动而无法随时间改进。

自动化测试彻底改变了我们发布软件的方式。当运行整个测试套件变得毫不费力时，我们就不必太担心进行改动。每次发布的成本降低了，我们可以更频繁地发布。在拥有自动化测试之前，甚至不要去想“敏捷”这个词。

有了自动化测试，我们可以随心所欲地更改任何内容、在任何时间、出于任何原因。只要测试通过并完全覆盖了所有东西，就放手去做吧。那个私有函数没有被任何地方调用了？删掉它。希望那个变量有个更好的名字？重命名它。就是有一个包，你单纯不喜欢它的风格？改掉它。持续、小量的重构会随时间产生复利效应，并偿还我们的技术债务。

自动化测试也帮助我们避免偏见。拥有一个结构化、标准化的评估程序体系，让我们更难对自己撒谎。当截止日期迫近、我们工作到深夜时，很容易就会眯起眼睛说结果看起来“足够好了”。总是有可能在结果上撒谎，但当我们面对一个巨大的“失败”消息和整个系统停止运行时，就很难忽视我们代码的低劣质量。

### 测试驱动开发

通过测试驱动开发，我们可以将测试从软件开发生命周期中的“二等公民”提升到一个突出位置。在测试驱动开发中，自动化测试是在我们开始添加功能或修复错误之前创建的。测试用例可以成为需求的主要形式。当测试通过时，需求就得到了满足，我们的工作也就完成了。

测试驱动开发也帮助我们避免偏见。测试就是标准，最好在我们开始做事之前就定义好目标。开始一个令人兴奋的新项目却不能立即开始编码，这可能会令人恼火。但请记住，到项目结束时我们会感到疲惫。如果把测试留到最后，我们直到对项目感到厌倦、只想让它结束时，才会定义成功的标准。

然而，自动化测试并非总是合适的。有时我们只是需要实验，不知道代码会带我们走向何方。有时任务是一次性的，或者不那么重要。有时，比如对于大批量数据加载，数据过于“脏”且定义不清。如果我们对项目还不够了解，就不想创建一个虚假的标准。自动化测试应该是一个实用的工具，而不是一个狂热的理想。



### 创建有用的测试数据

应用程序单元测试与数据库编程单元测试的最大区别在于对数据的关注点。不要简单地导入尽可能多的数据，而应专注于创建适量且相关的数据。

即使是为处理 PB 级数据而设计的关系型数据库，**少即是多**。导入海量数据用于测试是很诱人的。创建良好的样本数据很困难，而其他环境（甚至可能是生产环境）中往往已有预生成的数据。请不要屈服于这种诱惑。

在开发中使用生产数据是在自找麻烦。生产数据或许能带来眼前的便利，但日后可能演变成灾难。组织的数据安全规则最终会更新。一个组织今天允许在低级环境中使用生产数据，明天就加以禁止的情况并不少见。清理这些数据，或对其进行去标识化处理，可能非常痛苦。

无论如何，生产数据并没有那么有用。生产数据可能只代表了用户迄今生成的内容，而不一定是用户可能生成的极限。不要仅从生产数据中推导需求。

在创建样本数据时，每一字节都应有其意义。加载生产数据最大的问题之一是它**令人应接不暇**。对于测试数据，**质量远比数量重要**。没有人能理解一吉字节的数据，因此我们不应将如此多的数据加载到我们的系统中。我们的系统已经足够复杂了，不必再通过加载不必要的数据来进一步混淆自己。最好花时间只创建必要的数据。

生成数据的难度只取决于业务逻辑。不要费心寻找程序来帮助生成虚假数据。没有商业程序能理解我们定制的系统。一旦我们理解了业务规则以及确切需要生成什么数据，有几个简单的 Oracle SQL 模式就能让工作变得轻松。

生成虚假数据最简单的方法是使用`LEVEL`技巧：
```sql
--生成三行。
select level a from dual connect by level <= 3;
A
```

要加载特定数据，可以从`DUAL`进行`SELECT`，并用`UNION ALL`集合运算符组合语句：
```sql
--生成特定数据。
select 1 a, 2 b from dual union all
select 0 a, 0 b from dual;
A   B
-   -
1   2
0   0
```

### 创建大型测试数据

虽然准确性和易用性是测试数据最重要的特征，但有时我们需要为性能测试使用大型数据集。大型复杂系统理想情况下应至少有两个数据集：一个用于自动单元测试的小数据集，以及一个用于较罕见的性能测试的大型数据集。我们需要在将程序迁移到生产环境之前，确保它们能够进行扩展。

结合并调整上一节学到的技巧，几乎足以生成大量测试数据。`LEVEL`技巧存在我们需要规避的内存限制。超过约一百万行后，level 技巧会产生错误：
```sql
--引发：ORA-30009: CONNECT BY 操作内存不足
select count(*) from
(
select level from dual connect by level <= 999999999
);
```

当发生上述错误时，我们只需将加载过程分解为多个步骤：创建表，用`LEVEL`加载初始数据，然后反复将表插入自身以使表大小加倍。当存在唯一键或外键时，创建数据可能会变得棘手。但通常当我们加载大量数据时，是用于性能测试，数据本身没有太多意义：
```sql
--生成大量数据。
create table test1(a number);
insert into test1 select level from dual connect by level <= 100000;
insert into test1 select * from test1;
insert into test1 select * from test1;
insert into test1 select * from test1;
...
```

对于极大规模的数据加载，还有更多技巧可以提供帮助，例如创建表时使用`NOLOGGING`选项，以及使用`INSERT /*+ APPEND */`来启用更快的直接路径写入。加载大型数据存在风险，直接路径插入将在后续章节讨论。加载相关数据更多地是业务逻辑挑战而非技术挑战。前面介绍的简单技巧足以让我们开始。

我们可能希望采用**混合方法**：对于频繁的单元测试，使用精心制作的数据；对于较少进行的用户验收测试和性能测试，使用生产数据的副本。虽然频繁运行单元测试能增加我们对变更的信心，但仍有一些有风险的变更，我们希望能在接近生产环境的环境中运行它们以获得额外的安心。对于某些性能测试，由于查询执行计划主要依赖于表和列的统计信息，拥有一个完美匹配生产环境数据分布的环境会很有帮助。如果只需要定期将生产数据克隆到一个数据库中，使用生产数据带来的不便就不那么糟糕了。但我们肯定不希望将生产克隆作为测试数据的唯一来源。

### 清理测试数据

创建测试数据的一部分工作是在完成时将其移除。如果我们不小心，测试数据可能会污染环境并导致问题。自动化单元测试框架通常有一个`TearDown`过程来处理测试后的清理工作。清理测试结果并不总是轻而易举；向一个表中插入数据可能导致结果分散在整个模式中。

与其试图逐块移除测试数据，不如重置整个模式可能更容易。在私有的数据库开发环境中，这就像删除并重新创建模式一样简单。在共享数据库上，如果我们没有重建环境的脚本，仍然有技术可以帮助我们更快地进行测试。我们可以使用闪回、虚拟机、多租户等技术来重建环境。



### 如何构建自动化测试

如果您尚未开始使用自动化单元测试，那么入门其实非常简单。市面上有各种不同的测试框架可供选择，甚至自己编写一个简单的测试程序也完全可以。不必被众多选择弄得不知所措——无论选择哪个程序，都比什么都不做要好。

`JUnit` 是最流行的单元测试框架，尽管它是为 `Java` 设计的。该程序属于 `xUnit` 家族，几乎每种编程语言都有类似的程序。单元测试在 `Oracle` 中不如在 `Java` 中那么普遍，但仍然有几种选择。目前最流行的选项是 `utPLSQL`。

安装和使用 `utPLSQL` 非常简单：下载一个免费开源程序，解压缩，然后运行一个 `SQL*Plus` 脚本即可。详细步骤可以在该程序的代码仓库 [`https://github.com/utPLSQL/utPLSQL`](https://github.com/utPLSQL/utPLSQL) 中找到。

本章仅涵盖该程序的一小部分内容。重点并非重现一个 `utPLSQL` 教程，而是向您展示入门有多么简单。不要被所有选项吓倒。即使我们只使用其 5% 的功能，该程序也足以重塑我们的整个开发流程。

`utPLSQL` 要求我们在代码中添加“注解”。注解通过包规范中格式化的注释来配置单元测试。测试通过“期望”来执行，即一条将实际值与预期值进行比较的语句。测试通过调用一个预构建的包来执行，其输出可以多种格式查看。

例如，假设我们想要测试太空数据库中的数据。开始时，我们可能想检查两张主要表 `LAUNCH` 和 `SATELLITE` 中的数据行数。首先，我们创建一个包规范来存放注解。这些注解定义了一个名为“Space”的单一测试套件，以及两个测试：

```
create or replace package space_test as
-- %suite(Space)
-- %test(Check number of rows in launch)
procedure test_launch_count;
-- %test(Check number of rows in satellite)
procedure test_satellite_count;
end;
/
```

接下来，我们需要实现规范并调用期望。下面的包主体展示了两个过程。每个过程都计算行数，将计数保存到一个变量中，然后将该变量与硬编码的预期数字进行比较。实际上有 70,535 次发射和 43,113 颗卫星。但为了使结果更有趣，让我们为卫星数量设置一个错误的预期值：

```
create or replace package body space_test as
procedure test_launch_count is
v_count number;
begin
select count(*) into v_count from space.launch;
ut.expect(v_count).to_(equal(70535));
end;
procedure test_satellite_count is
v_count number;
begin
select count(*) into v_count from space.satellite;
ut.expect(v_count).to_(equal(9999));
end;
end;
/
```

以下是运行单元测试的几种方法之一：

```
begin
ut3.ut.run();
end;
/
```

以下是运行的 `DBMS_OUTPUT`。输出包含程序名称和两个测试的结果。其中一个测试成功，没有显示任何信息。但有一个期望失败了，输出解释了失败的原因：

```
Space
Check number of rows in launch [.004 sec]
Check number of rows in satellite [.005 sec] (FAILED - 1)
Failures:
1) test_satellite_count
Actual: 43113 (number) was expected to equal: 9999 (number)
at "JHELLER.SPACE_TEST.TEST_SATELLITE_COUNT", line 13 ut.expect(v_count).to_(equal(9999));
Finished in .013 seconds
2 tests, 1 failed, 0 errored, 0 disabled, 0 warning(s)
```

就是这样。在不到一小时内，用仅仅几行代码，就构建了一个功能完备的自动化测试套件。自动化测试就是可以如此简单。

还有许多其他功能可供使用。我们可以生成更漂亮的报告，检查代码覆盖率以确保所有代码行都被测试过，在持续集成系统中使用自动化测试等等。但这些功能并非总是必要，所以不要因为它们而不敢使用单元测试。

既然我们已经开始编写单元测试，接下来该何去何从？首先，像对待普通代码一样对待这些测试包。测试包可能会变得混乱，充满重复的样板代码，并且为每个测试包含复杂的设置。如果单元测试代码量大于其余代码，也不要惊讶。为了帮助管理单元测试，请将代码拆分成逻辑上的包和文件，并赋予它们良好的名称和注释。

标准的手动测试建议同样适用于自动化单元测试。尽量测试每一行代码，理想情况下是每一条路径。当存在循环时，为边界条件创建测试——差一错误通常发生在循环的开始或结束处，而不是中间。有时一个测试应该专注于一件小事，这样当测试失败时，我们就确切知道要修复什么。其他时候，一个测试应该涵盖很多内容，以确保我们尽可能多地发现问题。无论采用何种风格，我们必须让失败显而易见；单一失败就应该中断构建并停止所有后续流程。

## 用最小、完整且可验证的示例构建知识

构建非正式的测试用例很困难但很重要。创建一个最小、完整且可验证的测试用例需要时间和纪律。我们必须确保测试用例易于与他人分享，并且要警惕 XY 问题。

### 为什么要花这么多时间构建可复现的测试用例？

开发者需要帮助，需要提出很多问题。尽管我们在幼儿园听过相反的说法，但确实存在很多糟糕的问题。提出编程问题的最佳方式是附带一个可复现的测试用例。

构建测试用例的第一个好处是，我们很可能会自己解决问题。重新陈述问题并想象向别人解释它，将有助于我们从不同的角度看问题。

奥卡姆剃刀原理指出，最简单的答案通常就是正确的。对于 `Oracle` `SQL` 问题，最简单的答案通常是以下之一：我们忘了提交、我们打错了字，或者我们实际得到的报错信息并非我们声称的那样。当人们用这些简单的答案回应我们的问题时，这令人沮丧却又很常见，所以不如提前回应他们的疑虑。在人们愿意投入时间解决我们的复杂问题之前，我们必须证明我们遇到的并非简单问题。

一个好的测试用例也是向自己证明某事并帮助我们避免偏见的绝佳方式。当我们全力以赴想让某件事正常工作或运行得更快时，可能只会看到我们想看到的东西。构建具有可复现、清晰、客观结果的测试，使得自我欺骗变得更加困难。

可复现的测试用例对于应对潜在的对抗性情况也是必要的。许多人有动机不相信我们，他们会想，“*我们*的软件是完美的——那种故障或行为是不可能的。”这些回应可能会让我们生气，但我们不应该往心里去。无益的回应通常源于有偏见的人，他们下意识地知道，如果我们的投诉毫无根据，他们就可以早点下班回家。或者他们的管理层只关心开启和关闭的工单数量；让工单消失和解决真正问题一样算作业绩。有时我们需要玩一些愚蠢的游戏才能获得我们付费的支持。

支持工程师让工单消失的最简单方法就是索要更多信息。如果他们要求足够的信息，最终我们会放弃并让他们关闭工单。创建一个好的测试用例将避免许多此类游戏。

“最小、完整且可验证的示例”这个短语源于 Stack Overflow，并随着时间的推移而发展，旨在帮助人们提出有用的问题。您可以在 [`https://stackoverflow.com/help/mcve`](https://stackoverflow.com/help/mcve) 阅读完整描述。以下部分在 `Oracle` 环境下解释了该短语的不同组成部分。




### 极简

省略不必要的代码。

我甚至想用这一句话就写完这一整节。可惜，简化并非易事，我们需要付出努力才能让事情看起来简单。

编程是困难的，而缺陷可能迅速增多。测试案例中的代码越多，引入无关问题的机会就越大，人们关注错误重点、吹毛求疵于无关错误的机会也越大。大段的代码需要更多时间去理解，愿意查看它的人也就越少。如果我们面对的是一个纯技术问题，就应尽可能排除业务逻辑。我们必须将聚光灯聚焦在尽可能少的代码上。（但我们必须按代码行数和变量数来衡量代码，而非字符数；不要为了减小文件大小而使用单字母变量名，也不要忘记添加有用的注释。）

同样的规则也适用于模式对象和数据。一列够用，就不要创建两列。一行够用，就不要插入两行数据。

棘手之处在于，我们的问题可能发生在很大的上下文环境中。一个问题可能发生在涉及几十张表、读取数十亿行数据的查询中。要移除尽可能多的代码和数据，并仍能重现缺陷，可能需要花费数小时。将大查询对半切割、检查是否失败并重复此过程，可能会耗费很长时间。最终，我们会得到一个能放进小型 SQL 工作表的问题。

最小化代码是耗时且乏味的。但这个过程让我们能精确定位确切的问题，比争论代码和数据更容易，并且避免了上传没人想看的大文件。

### 完整

我们必须创建一个可运行且自包含的测试案例。然后，我们必须在一个全新的数据库上编写代码，以确保测试案例没有缺失任何意外的依赖项。别人不会为了帮我们解决问题而去安装其他产品，所以确保一切都在一个地方。

以下是一个完整测试案例的示例。该代码创建对象、添加数据、收集优化器统计信息，并运行问题查询：

```sql
--创建表。
create table simple_table(simple_column number);
--添加示例数据。
insert into simple_table select 1 from dual;
--如果是性能问题，则收集统计信息。
begin
dbms_stats.gather_table_stats(user, 'simple_table');
end;
/
--这是失败或运行缓慢的查询。
select *
from simple_table;
```

前面的代码是理想情况，但我们整个模式是现实。我们需要在中间找到平衡点。有时，从前面的代码开始，添加真实代码直到出现问题会有所帮助。其他时候，从真实模式开始，逐步删减直到代码能正常工作会更有帮助。

除了代码，我们还应提供完整的版本号和版本。版本和版本对于发现特定缺陷很重要，它们也限制了可用的故障排除和解决方法。平台通常无关紧要，但包含它也无妨。例如：前面的代码在 Oracle 19.3.0.0.0 企业版和 64 位 Windows 10 上有效。

收集系统参数也很重要。参数有数百个之多，很难知道哪些是重要的。我们可以仅查找非默认参数来缩小搜索范围，使用以下查询：

```sql
select name, value
from v$parameter
where isdefault = 'FALSE'
order by name;
```

但即使前面的查询也不总是足够好。例如，在某些处理器上，`CPU_COUNT` 的默认值是错误的，并可能导致性能问题。而且许多参数可以在会话级别更改，无论是作为登录脚本的一部分，还是通过触发器。遗憾的是，无法获取所有非默认会话参数的列表。

要让一个测试案例真正完整是不可能的——如果你想从头创建一个测试案例，你必须先发明宇宙。但稍加努力，我们就能在 99% 的情况下获得足够的信息。至少，我们可复现的测试案例向他人表明了我们是认真的，这使得其他开发人员更有可能尝试示例代码并调查问题。

### 可验证

没人能验证一个“不工作”的代码。我们需要为我们的代码提供客观的度量标准，例如完整的错误消息或详细的性能预期。

我们都讨厌错误消息，希望它们消失。但我们必须抵制快速 dismiss（置之不理）和忽略错误消息的冲动。默认情况下，Oracle 错误会生成对象名称、错误号、错误消息和行号。如果错误通过多个层级传播，所有层级的错误消息都将包含在内。默认情况下，Oracle 包含了解决大多数问题所需的所有信息。但只有当我们记下*完整*的错误消息时，它才有帮助。开发人员会不遗余力地抛出有用的异常消息。不要忽视这些消息并说“出了个错误”，从而浪费了他们的工作。

`SQL*Plus` 不擅长交互式查询和编程。`SQL*Plus` 的最大优势在于它简单、通用、纯文本的 Oracle 接口。如果我们在 `SQL*Plus` 中能完成某件事，那么任何人都可以复制文本、运行代码并自行验证结果。

像 Oracle SQL Developer 这样的程序对于大多数数据库工作来说好用得多，但对于创建测试案例来说却没那么有用。如果我们发送截图，其他开发人员就不容易提取代码。如果我们只发送在 SQL Developer 中使用的文本，其他开发人员可能会缺失我们如何运行代码的信息——存在具有不同行为的不同窗口类型。而且，在 SQL Developer 中，输出和错误消息并不总是易于复制。

`SQL*Plus` 看起来很丑，但至少是我们都认可的那种丑。通过一些简单的技巧，我们可以创建一个 `SQL*Plus` 环境，清晰地展示我们运行了什么、在哪里运行的以及运行了多长时间：

```sql
SQL> set sqlprompt "_user'@'_connect_identifier> "
JHELLER@orcl9> set timing on
JHELLER@orcl9> select count(*) from dba_objects;
COUNT(*)

Elapsed: 00:00:00.21
JHELLER@orcl9>
```

更改 SQL 提示符可以清楚地显示是哪个用户运行了代码以及在哪个数据库上运行。人们很少会粗鲁到直接问：“你确定你在正确的数据库上吗？”但在他们心里，他们确实在想这个问题，所以我们不妨让他们安心。另一方面，有时数据库名称是敏感的，我们需要隐藏它。对于文本输出，查找和替换名称很简单，而截图往往会包含无意的细节，甚至可能来自其他程序。

`SET TIMING ON` 命令显示了每条语句的实际运行时间。而且语句本身很容易复制和粘贴。只需付出很少的努力，我们就可以消除测试案例的许多不确定性。

对于性能调优来说，包含精确的时间预期至关重要。一个处理单行数据的 OLTP 系统可能期望结果在不到一秒内返回。一个处理十亿行数据的数据仓库可能期望结果在不到一周内返回。我们必须提供比“它很慢”更多的信息。解释我们的代码花了多少时间、我们认为它应该花多少时间以及为什么，会很有帮助。列出我们的性能假设也有帮助，因为有时我们的预期是不切实际的。也许我们认为应该不到一秒就完成的过程，在我们当前的数据结构、算法和硬件下，根本不可能在不到一秒内完成。




## 共享测试

创建可复现的测试用例对我们自身的学习有益，但大多数时候我们需要与他人共享这些测试用例。如果我们想要为世界知识添砖加瓦，就需要发布测试的所有细节，以便他人能够证明我们是错误的。如果没有一个完全可复现的测试用例，我们所做的就称不上是科学。

组织喜欢将所有测试用例据为己有，但很少有必要隐藏这些信息，尤其是在敏捷环境中——客户应该被赋予查看所有测试的权限。如果幸运的话，客户会在开发过程早期就发现问题或缺失的假设。

为了优化共享，请尽可能使用文本。在 IDE 中创建测试用例更容易，但值得在`SQL*Plus`中重建最终版本，以确保一切都可以被复制和粘贴。如果我们必须使用截图，应该花时间重新排列窗口使其尽可能小，并编辑图片以仅包含相关细节。

理想情况下，我们测试用例的受众拥有一个私有的 Oracle 数据库，具有完全权限以创建无限数量的对象。如果我们的受众没有这样的环境，现在有越来越多用于构建和共享代码片段的“fiddle”网站。一个流行的数据库网站是[`https://dbfiddle.uk`](https://dbfiddle.uk)。该网站允许任何人创建一个模式，查询该模式，并通过一个简单的 URL 与他人共享该模式。Oracle 也有一个类似的网站：[`https://livesql.oracle.com`](https://livesql.oracle.com)。Oracle 的网站功能比其他网站更多，但运行代码片段需要一个账户。

## 避免 XY 问题

在发送测试用例的最终版本之前，我们需要退一步，尝试避免 XY 问题。编程往往会引导我们走上一条幽深的小径。我们想要做 X，并且认为 Y 是解决方案。当 Y 不奏效时，我们会针对 Y 提问，而我们真正的问题其实在于 X。

XY 问题的一个常见例子是，人们会问：“如何将逗号分隔的列表拆分成不同的行？”这是一个重要的问题，有时确实有必要这样做。但拆分字符串意味着有人没有遵循关系模型最基本的规则。提问的开发者以及将来某天取代他们的人，试图使用一个破损的数据模型时，将会遇到很多麻烦。正确的答案是“别那么做”。这不是人们想听到的答案，但却是他们需要听到的答案。

我们陷入这个 XY 陷阱，是因为编程偶尔需要关注微小的细节，而我们看不到全局。摆脱陷阱的方法是偶尔停下来，多问自己几次“为什么？”

当我们发现自己为做某件怪事而寻求帮助时，解释为什么我们需要用这种怪异的方式去做，这很重要。如果我们解释清楚我们并非愚蠢；是业务规则或技术限制愚蠢，那么我们更有可能获得同情和帮助。如果幸运的话，有人会打断我们并说：“你的前提错了——那个限制不再适用于版本 A 了，你可以改用功能 B。”

## Oracle 性能测试

为 Oracle 性能问题构建可复现的测试用例尤其具有挑战性。Oracle 性能测试用例往往充满了海森堡 bug（heisenbugs）——即当别人观察时就会消失的错误或行为。由于 Oracle SQL 是一种声明式语言，Oracle 在如何执行操作方面有很大的回旋余地，包括改变主意的能力。第四部分涵盖了高级性能调优概念，例如使用采样和剖析来深入研究我们的 PL/SQL 程序以找到慢速 SQL，然后使用提示和监控来深入研究我们的 SQL 以找到慢速操作。在深入这些细节之前，性能测试的第一步是学习如何准确测量单条语句的整体性能。

实际运行时间是性能测试的核心度量单位。如果某样东西运行得更快，说明它使用的资源更少，对用户和系统都更好。不必太担心测量 CPU、I/O、锁、互斥量等。

但是，只测量运行时间这条规则也有重要的例外。例如，我们不希望创建一个并行解决方案，它虽然将运行时间减少了幾秒，却将 CPU 利用率从 1%提高到了 99%。运行时间并不总是与资源消耗相关。一些 Oracle 开发人员发现，主动使用像`RUNSTATS`这样的程序来监控资源消耗很有帮助。

性能测试中最令人沮丧的部分是运行时间不一致。Oracle 数据库会随着时间的推移而预热并自我调优，因此查询第二次或第三次执行比第一次执行显著快的情况相当常见。

缓存是导致性能测试不一致的最大因素。当我们第一次运行查询时，Oracle 从磁盘读取表和索引块，并将这些块存储在内存中。第二次运行查询时，数据库可以通过快得多的内存访问来读取这些块。根据我们的环境，我们可能希望在“热”或“冷”状态下测试系统。在数据仓库上，我们的大表不太可能被缓存，因此测试冷系统更好。在 OLTP 系统上，常用的表应该在内存中，因此测试热系统更好。我们可以使用命令`ALTER SYSTEM FLUSH BUFFER_CACHE`来清除 Oracle 的内存。但是，该命令不会清除操作系统缓存，而且我们可能不想在共享数据库上运行它，以免影响其他用户。

间歇性的系统活动也可能破坏我们周全的测试用例。这在共享数据库上问题更严重，但即使是我们的笔记本电脑也可能突然开始运行防病毒扫描而变慢。与其测量困难的系统活动，我们可以多次运行测试，并在竞争方案之间交替进行。在不同选项之间交替会使系统突然变快或变慢的情况更加明显。我们可能还想去掉最小和最大的运行时间。

执行计划也会随时间变化。如果我们等待足够长的时间，系统很可能会自动收集缺失的统计信息。Oracle 优化器也能够从错误中学习，并可能在第二次或第三次运行时使用统计信息反馈、自适应游标共享和 SQL 计划指令。（这些功能在第四部分有简要讨论。）仔细检查执行计划的`Note`部分，看看是否应用了任何奇怪的技巧。为了给测试创建一个干净的起点，有时需要运行`ALTER SYSTEM FLUSH SHARED_POOL`并重新收集统计信息。但与清除缓冲区缓存一样，如果在共享数据库上运行，我们需要小心执行这些命令。

## Oracle 侦探工具包

E. F. Codd 的关系数据库第一规则是，数据库必须能够完全通过关系接口进行自我管理。在 Oracle 数据库中，有大量关于数据库的信息可用，并且这些信息可以通过数据库本身访问。当我们的测试不可避免地需要深入了解我们的程序和模式内部时，我们可以使用数据字典、动态性能视图、Oracle 包和函数以及其他 Oracle 工具。



## 数据字典视图

大约有 1000 个数据字典视图，它们提供了关于 Oracle 数据库内部几乎所有内容的元数据。数据字典视图有四种不同的前缀：`USER_`、`ALL_`、`DBA_`和`CDB_`。`USER_`视图提供当前用户的信息，`ALL_`视图提供我们有权访问的所有用户的信息，`DBA_`视图提供所有用户的信息，而`CDB_`视图则提供所有容器中所有用户的信息。

数据字典具有很好的向后兼容性。这意味着我们可以为版本 X 编写一个脚本，并可以放心地期望该脚本在版本 X+1 中也能工作。列名几乎从不改变，除非在视图末尾添加了新列。这种向后兼容性的缺点是数据字典包含了一些已弃用的功能。字典中的`LONG`列很难处理。目前仍然没有简单的方法可以对`LONG`值使用 SQL 函数，因此搜索这些列需要一些额外的步骤。

例如，查找所有默认值为零的列本应很简单：
```sql
SQL> select table_name, column_name, data_default
2  from dba_tab_columns
3  where to_char(data_default) = '0'
4  order by 1,2,3;
where to_char(data_default) = '0'
*
ERROR at line 3:
ORA-00932: inconsistent datatypes: expected CHAR got LONG
```
不幸的是，我们需要增加一个额外的步骤才能使前面的查询工作。首先，我们必须转换并保存数据，然后才能过滤结果：
```sql
create table convert_tab_columns as
select table_name, column_name, to_lob(data_default) data_default
from dba_tab_columns
where data_default is not null;
select table_name, column_name, to_char(data_default) data_default
from convert_tab_columns
where to_char(data_default) = '0'
and rownum = 1
order by 1,2,3;
TABLE_NAME             COLUMN_NAME   DATA_DEFAULT
--------------------   -----------   ------------
CDB_LOCAL_ADMINAUTH$   LCOUNT        0
```
数据字典的另一个问题是，有些列名现在已经没有意义了。在 Oracle 的一个古老版本中，权限只授予表，视图名称`DBA_TAB_PRIVS`是有意义的。但不幸的是，该视图现在包含了所有对象类型的权限。

还有一些古老的、已弃用的数据字典视图应该避免使用。这些视图出于向后兼容性而保留，但包含的数据量远不及取代它们的新视图。请避免使用`CAT`、`COL`、`COLS`、`TAB`和`TABS`。请注意，与许多其他数据库不同，Oracle 不包含`INFORMATION_SCHEMA`。

撇开这些小问题，数据字典拥有丰富的信息。我们不需要记住这 1000 个视图，也不需要在办公室墙上贴海报。但我们可能希望搜索并收藏我们所用 Oracle 版本的《数据库参考》。以下列出了最重要的数据字典视图，其中一些视图的含义不言自明：

1.  `*_ADVISOR_*`：自动建议程序的状态和结果，随着新的自治数据库功能的推出，这些视图可能变得越来越重要。
2.  `*_AUDIT_TRAIL`：审计数据取决于配置，但通常包含有关登录历史的有用信息。
3.  `*_AUTOTASK`：用于收集统计信息、自动调优等的特殊系统作业。
4.  `*_CONSTRAINTS`：唯一、检查、主键、外键约束。
5.  `DUAL`：一个只有一行的表——当我们想生成数据时很有用。
6.  `*_ERRORS`：导致对象无效的代码中的错误。
7.  `*_HIST_*`：用于自动工作量存储库的视图，用于性能调优（但需要诊断包许可）。
8.  `*_INDEXES`，`DBA_IND_COLUMNS`。
9.  `*_JOBS/SCHEDULER`：计划作业。
10. `*_OBJECTS`：所有模式对象——用于查找事物很有用。
11. `*_RESUMABLE`：等待资源的会话。
12. `*_ROLE_PRIVS/SYS_PRIVS/TAB_PRIVS`：角色、系统和对象权限。
13. `*_SOURCE`：除视图外所有代码对象的源代码。
14. `*_SYNONYMS`。
15. `*_TABLES`，`TAB_COLUMNS`。
16. `*_USERS`。
17. `*_VIEWS`：包含视图的源代码（不包含在`*_SOURCE`中）。

## 动态性能视图

超过 850 个动态性能视图包含有关 Oracle 数据库当前状态的信息。这些视图更多是关于 Oracle 本身而非我们的数据，因此它们往往比数据字典更隐晦，但同样有用。动态性能视图有两种类型：`V$`和`GV$`。唯一的区别是，`V$`视图包含关于我们当前所连接实例的数据，而`GV$`包含所有实例的数据。如果我们连接的不是真正应用集群数据库，那么这两种视图几乎是相同的。

与数据字典类似，动态性能视图也具有良好的向后兼容性。完整列表可以在《数据库参考》中找到。以下列出了最常见的视图，并对其中含义不言自明的视图进行了解释：

1.  `V$ASM*`：自动存储管理数据
2.  `V$DATABASE`：来自控制文件的数据库信息
3.  `V$INSTANCE`：数据库的当前状态
4.  `V$PARAMETER`：初始化参数
5.  `V$PGASTAT`，`V$PGA*`：程序全局区（PGA）数据（会话内存，用于排序和哈希等操作）
6.  `V$PROCESS`：活动进程，可以与操作系统进程关联
7.  `V$PX*`：并行进程
8.  `V$SESSION`：当前会话及其来源
9.  `V$SESSMETRIC`，`V$SESSTAT`：会话指标和统计信息，用于衡量资源消耗和性能调优很有用
10. `V$SGASTAT`，`V$SGA*`：系统全局区（SGA）数据（系统内存，用于缓存数据和执行计划等）
11. `V$SQL*`：关于 SQL 语句及其执行计划的数据
12. `V$SYSMETRIC*`，`V$SYSSTAT`：系统指标和统计信息，用于衡量总资源消耗很有用
13. `V$VERSION`：数据库的精确版本，位于`BANNER_FULL`列

动态性能视图`V$PARAMETER`尤其重要，该视图中的每一行都可能至关重要。它有超过 400 行，每一行在《数据库参考》中都有详尽的文档说明。很难说哪些参数最关键；即使是一个错误的参数也可能以奇怪的方式破坏数据库。虽然管理员需要了解大多数参数，但以下参数最可能引起 SQL 开发人员的兴趣：

1.  `compatible`：数据文件的兼容版本可能会限制可用功能。一旦我们更改此参数，就无法回退。
2.  `cpu_count`：应该是处理器的数量，但在某些平台上，它被错误地设置为支持的线程数；这个数字对于并行处理很重要。
3.  `cursor_sharing`：如果我们的应用程序无法使用绑定变量，将参数从默认的`EXACT`更改为`FORCE`可以缓解一些性能问题。（绑定变量在第 14 章讨论。）
4.  `ddl_lock_timeout`：允许 DDL 语句在对象被锁定时等待，而不是立即失败。
5.  `memory_target/pga_aggregate_target/sga_target`：Oracle 内存设置极其复杂；在修改这些设置之前，请仔细阅读相关手册几遍。
6.  `nls*`：国家语言支持（NLS）参数控制数据的存储、比较、排序和格式化方式——无论是按二进制值还是根据特定语言规则。不要仅仅依赖数据库格式设置——它们经常被会话设置覆盖。
7.  `optimizer_index_caching/optimizer_index_cost_adj/db_file_multiblock_read_count`：经常被滥用的参数——除非你想自找麻烦，否则不要设置这些参数。
8.  `parallel*`：控制并行处理；在更改这些参数之前，请仔细思考并阅读并行处理指南。
9.  `processes`，`sessions`：限制 Oracle 操作系统进程和连接的数量。这些数字经常需要增加。



我们不应更改前述参数，除非我们自认比 Oracle 的工程师更聪明。当然，我们确实有很多时候比 Oracle 工程师更了解情况，例如当我们对自身环境有特定认知时。比如，只有我们知道数据库需要处理多少会话。如果我们有一个必须支持一千个连接的数据库，那么就有必要增加 `SESSIONS` 和 `PROCESSES` 参数，尽管这样做会略微增加内存需求。

另一方面，我们并不知道我们的索引性能比 Oracle 预期性能快多少或慢多少；修改 `OPTIMIZER_INDEX_COST_ADJ` 几乎肯定是个坏主意。虽然有时修改这个参数确实能通过将执行计划从全表扫描改为索引扫描来提升特定查询的性能。但我们不应该为了修复一条语句的问题而调整整个系统。解决方案的规模应该与问题的规模相匹配，而 Oracle 允许我们按会话或按查询设置该参数。我之所以将这个参数列在前面，不是因为我们有定制它的需要，而是因为当其他人错误地修改它后，我们经常需要将其改回默认值。

### 用于检查数据库的关系型工具

数据库中还散布着许多其他工具，可用于调查数据库。

`VSIZE` 是一个有用的函数，它返回值所占的字节数。乍一听这似乎是个微不足道的特性，但令人惊讶的是，许多架构决策都取决于数据类型的大小。有一个愚蠢但持续存在的迷思，认为 Oracle 总是分配尽可能多的空间。只需调用几次 `VSIZE` 就能驱散这个迷思。`VSIZE` 还表明，小数字所占的空间可能比我们预期的要多：

```
--值的字节大小：
select
vsize(0) zero_size,
vsize(1) one_size,
vsize(10) ten_size,
vsize(date '2000-01-01') date_size,
vsize(cast('a' as varchar2(4000))) string_size
from dual;
ZERO_SIZE  ONE_SIZE  TEN_SIZE  DATE_SIZE  STRING_SIZE
---------  --------  --------  ---------  -----------
1         2         2          7            1
```

`DUMP` 函数显示值的内部内容。在处理非 ASCII 字符时，这个函数非常有用。例如，想象一个包含链接的表。当我们查询该表时，一切看起来都正常：

```
select * from random_links;
URL

google.com
```

等一下——让我们深入探究一下。那个字符串看起来像“google.com”，但数据库里的值真的是“google.com”吗？看看下面的输出，你能否发现问题所在：

```
select dump(url,16) from random_links;
DUMP(URL,16)

Typ=1 Len=11: 67,6f,c3,b6,67,6c,65,2e,63,6f,6d
```

你可能首先注意到的是 `Len=11`。长度是 11 字节，但“google.com”只有十个字符。如果你仔细观察十六进制值，会发现第二个“o”并不是真正的“o”，而是“ö”。这是一个带分音符（即上方有两个点）的“o”。

这种技巧被称为视觉欺骗。它是故意难以察觉的，如果没有像 `DUMP` 这样的函数，我们可能永远发现不了问题。如果你好奇，这个恶意字符串是这样创建的：

```
insert into random_links values('go'||unistr('\00f6')||'gle.com');
```

理论上，我们可以使用数据字典重建任何对象。例如，与表相关的所有特性都可以在数据字典视图中找到，如 `ALL_TABLES`、`ALL_TAB_COLUMNS`、`ALL_CONSTRAINTS` 等。但实际上，要重建对象的完整 DDL（数据定义语言）是困难的，尤其是对于那些晦涩的特性。函数 `DBMS_METADATA.GET_DDL` 可以让我们的工作轻松很多，并且它有很多用于调整格式的选项。例如，以下代码用于查找表 `LAUNCH` 背后的语句：

```
select dbms_metadata.get_ddl(
object_type => 'TABLE',
name        => 'LAUNCH',
schema      => 'SPACE') ddl
from dual;
DDL

CREATE TABLE "SPACE"."LAUNCH"
(    "LAUNCH_ID" NUMBER,
"LAUNCH_TAG" VARCHAR2(15 CHAR),
"LAUNCH_DATE" DATE,
...
```

但如果我们发现自己扮演起模式侦探的角色，那我们就已经输了。正如第 2 章所讨论的，找到模式脚本的最佳方式是查看受版本控制的纯文本文件。前面查询的完整输出，虽然在技术上是正确的，但存在一些格式问题，并包含大量无价值的信息，而我们的安装脚本则全是有效信息，没有噪音。我们本不应忍受这些无意义的字节。

`SYS_CONTEXT` 函数可以检索有关当前会话上下文的数据。我们可以创建自定义上下文，但最常见的上下文是默认的 `USERENV`。这是一种无需查看多个表就能获取当前会话相关信息的便捷方式。例如，以下代码用于获取我们用于连接数据库的主机名：

```
select sys_context('userenv', 'host') from dual;
```

对于代码分析，PL/Scope 在需要理解代码的任务中非常有用。PL/Scope 可以查找特定的依赖关系、未使用的变量等。而 `DBMS_DEBUG` 包可以帮助我们在代码执行时逐行单步调试。这些是有趣的工具，但通常只有在与 IDE（集成开发环境）结合使用时才有帮助。

### 用于检查数据库的非关系型工具

Oracle 还包含许多非关系型工具来帮助检查数据库。（性能调优是一个大主题，将在第四部分介绍。）一个程序可以与数据库协作，甚至可能在数据库程序内部运行，但它不一定是关系型的。这种区分很重要，因为非关系型工具不能像关系型工具那样适用于多种场景。

例如，SQL*Plus 中的 `DESC` 命令是一种快速获取表信息的便捷方式：

```
SQL> desc space.launch
Name                  Null?    Type
--------------------- -------- -----------------
LAUNCH_ID             NOT NULL NUMBER
LAUNCH_TAG                     VARCHAR2(15 CHAR)
LAUNCH_DATE                    DATE
...
```

虽然 `DESC` 命令很有用，但我们可能要避免过度依赖它，因为它仅在 SQL*Plus 中可用。尽管几乎所有开发者都能使用 SQL*Plus，但大多数开发者使用的是图形化的 IDE。使用数据字典查询，或调用 `DBMS_METADATA.GET_DDL`，可能需要多做一些工作，但为了确保我们构建纯粹的关系型解决方案，这是值得的。

传统的导出和数据泵导出程序，即 `EXP` 和 `EXPDP`，也可以用来查找有用的数据和元数据。未文档化的 `ORADEBUG` 命令可以输出单个数据块，帮助我们精确查看数据在磁盘上的存储方式。Oracle 跟踪可以帮助我们理解优化器的决策，并确切看到后台为支持我们的查询正在运行什么。还有一整套工具通常只对数据库管理员有用：监听器控制、服务器控制等等。

Oracle 是闭源的，但如果我们熟悉所有这些元数据工具，我们就能弄清楚到底发生了什么。了解各种各样的数据库工具是有益的，因为不同的人有不同的风格和偏好来调查数据库问题。例如，我通常在 PL/SQL Developer 中工作，使用 Ctrl+右键单击来描述表。但当我与更喜欢 SQL*Plus 的 DBA 合作时，我会使用 `DESC` 命令向他们展示表元数据。


## 总结

当我们开始编程时，应对自己的程序和想法保持怀疑态度。最终，我们必须对自己的程序和知识建立信心，但这种信心应源于自动化单元测试和简洁的测试用例。幸运的是，Oracle 提供了丰富的视图和工具来帮助我们建立这种信心。但靠我们自己能做的终究有限，最终我们会需要他人的帮助。下一章将帮助我们找到可靠的 Oracle 知识来源。
