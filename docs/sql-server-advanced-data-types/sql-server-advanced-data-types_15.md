## SQL Server 数据类型与 XML 基础

### INT 数据类型与存储优化

SQL Server 中最常用但也最常被误用的数据类型是 `INT`。当开发者需要存储整数时，`INT` 通常是默认选择。然而，开发者往往只需要存储较小的值。`INT` 使用 4 字节数据，而如果 `SMALLINT` 足够，则只需 2 字节存储空间。对于小型查找表（少于 255 行），可以使用 `TINYINT` 作为主键，每行仅使用 1 字节存储空间。

你可能会想，如今存储相对便宜，这里差 2 或 3 字节真的有影响吗？要回答这个问题，不应仅从存储角度考虑，还应从内存使用和应用性能角度思考。

假设你有一个包含 10 亿行的事实表，需要与五个维度表进行连接，每个维度表包含 30,000 行。如果在这些维度表中使用 `INT` 数据类型作为主键，该列在每个维度表中将是 117KB，但事实表中对应的外键列将达到 3.72GB！现在将其乘以 5（覆盖每个维度表），总计达到 18.6GB。这就是 SQL Server 在执行查询连接表之前，可能需要从磁盘读取到内存，然后再溢出到 `TempDB` 的数据量。如果我们改用 `SMALLINT` 数据类型，SQL Server 只需处理 9.3GB 的数据。

### 数据类型一致性的重要性

选择数据类型时的另一个考虑是，当相同数据存储在多个列中时，确保使用相同的数据类型。例如，假设你有一个 ETL 过程，从 OLTP 数据库抽取数据到数据仓库。你可能会通过运行查询（基于日期等列进行连接或过滤）来填充数据仓库。如果列之间的数据类型不一致——例如，一个表中日期存储为 `DATE` 类型，而另一个列中存储为 `DATETIME2`——那么就会发生隐式转换操作。这些操作会对性能产生负面影响，甚至会阻止使用最优的索引。

**提示**：如果需要进行数据类型转换，我通常建议手动使用 `CAST` 或 `CONVERT` 函数来执行，而不是允许隐式转换。这可以在调试时使代码更易于理解。

## 总结

SQL Server 开箱即用地提供了大量数据类型，可以满足大多数开发场景的需求。数值可以存储为整数，或具有精确或近似的小数精度。字符数据可以使用 Unicode 或非 Unicode 字符集存储。SQL Server 还可以存储日期、时间或两者。你还可以选择记录时区信息。

可以使用 `CAST` 或 `CONVERT` 函数在不同的数据类型之间转换值。使用 `CONVERT` 函数时，可以通过样式参数提供额外的翻译信息。

在使用 SQL Server 开发时，使用正确的数据类型至关重要。选择数据类型时，开发者应使用能够存储所有必需值的、限制最严格的数据类型。如果错误地使用了范围更广的数据类型，那么随着数据库规模的扩大，可能会对内存利用率和性能产生严重影响。

## 理解 XML

XML（可扩展标记语言）是一种类似于 HTML 的标记语言，设计目的是用于存储和传输数据。与 HTML 类似，XML 由标签组成。然而，与 HTML 不同，这些标签不是预定义的，而是由文档作者定义。XML 文档具有树状结构，以根节点开始，并包含子节点（也称为子元素）。每个元素可以包含数据，也可以包含 1 到 n 个属性。

## 理解 XML

每个属性都可以包含描述该元素的数据。例如，假设你需要将销售订单的详细信息以 XML 格式存储。很自然地，可以认为每个销售订单应存储在文档中一个独立的元素里。但是，销售订单的属性，如订单日期、客户 ID、产品 ID、数量和价格，该如何处理呢？这些信息既可以作为销售订单元素的子元素来存储，也可以作为该元素的属性来存储。对于何时应使用子元素或属性来描述元素的属性，并没有固定的规则。这个选择权在文档作者手中。
