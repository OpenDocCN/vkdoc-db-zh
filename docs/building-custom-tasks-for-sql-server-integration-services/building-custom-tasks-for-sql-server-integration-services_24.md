# 21. 测试任务

在前面的章节中，我们为本书第二版构建的“执行目录包任务”版本编写了功能代码。现在是时候测试“执行目录包任务”了。


## 手动测试用例

虽然存在能够自动化测试生成、执行和报告中重复工作的测试框架，但如果你的工作是日复一日地开发软件，你迟早会接触到测试框架。你偶尔会遇到某个测试需求，而所选测试框架对此要么无能为力，要么显得“笨拙”。在这种情况下，你将需要构建一个手动测试。

所有良好的测试都由一系列场景、预期答案或结果（断言）以及对实际结果的测量组成。场景与断言的组合被称为一个 `用例`。一个用例可能包含多个断言。当我们执行测试时，我们会逐一处理每个用例，测量结果并记录测量值。

Andy 的软件测试格言

软件测试是一项细致且要求高的工作。在本章中，我们只测试部分（而非全部）用例。如果我们测试所有用例，本章将长达 150 页，而我的编辑（理所当然地）会指出这是一章内容，而非一份测试计划。

## 用例

在本节中，我们将探讨针对“执行目录包任务”的以下用例：

1.  `任务存在性`
2.  `任务验证`
3.  `任务编辑器` – `常规`

前几个用例涉及目视检查——简单地查看 `执行目录包任务` 是否出现在 `SSIS 工具箱` 中，如果是，则在 `SSIS 工具箱` 中选择 `执行目录包任务` 并验证工具箱描述。

这些测试使用手动测试文档格式进行记录。作者见过类似的手动测试文档以 Excel 电子表格形式呈现，但更倾向于使用 Word 文档来记录手动测试。
