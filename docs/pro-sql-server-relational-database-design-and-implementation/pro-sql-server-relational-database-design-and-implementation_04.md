# 2. 需求简介

> 忙碌并不总是意味着真正的工作。所有工作的目的都是生产或成就，要实现这些目标，就必须有远见、系统、计划、智慧和真诚的目标，以及汗水。看起来在做，并不等于真正在做。——托马斯·爱迪生，美国发明家和商人

如果有什么比在没有完全理解成功要求的情况下做一件简单任务更糟糕的话，那就是在同样的情况下做一件复杂的任务。这种事每天都在发生，计算机项目在创建时往往对最终目标只有一个模糊的想法。有时这很有趣，并能引导你走向有趣的东西……有时，但当该项目的目标是向一个知道自己想要什么的客户交付价值时，几乎从不如此。因此，任何参与软件项目的人，都必须在开始设计任何软件之前，花时间弄清楚客户想要什么。如果你非常幸运，这个人可能不是你，因为捕获需求比后续章节中的任何任务都要困难得多。

然而，在本章的剩余部分，我们将假设自己戴上了业务分析师的帽子，需要在开始设计数据库之前收集需求。我们将保持非常简单，只涵盖能应付过去的最小必要量。

启动计算机项目时，通常做的第一件事是采访用户，并问类似“你们到底想要什么？”的问题，然后倾听，同时记住用户通常不是技术专家。有些人是。还有一些人完全相信自己是，但实际上并不是。所以一定要让很多人参与进来，并在尝试解决问题之前真正花时间理解客户的问题。我本人历史上的一个主要问题是，我会试图快速跳过理解客户需求这一步，直接进入数据库内部设计。

问题在于用户不考虑数据库；他们考虑的是用户界面（UI）、报表以及他们希望事物呈现的外观和感觉。当然，用户为 UI 或报表格式指定的很多内容最终都会反映在数据库设计中；你的职责是确保需求中包含足够的信息，以便在不太多考虑数据将如何显示、处理或使用的情况下设计存储。在这个过程中，数据有其自身的本质，必须遵守，否则你将发现自己会与你构造的结构陷入斗争。在本章中，我们将探讨一些你想要获取的基本数据类型以及可以查找的位置，以确保你从业务分析师那里获得了正确类型的需求来开始数据库设计过程。

当然，如果你是新手，你可能会想这听起来像是很多文档工作，而不是很多设计（和编码）。无论你怎么看，规划每个项目都是如此。如果你幸运，你会遇到负责需求收集的分析师，这样你就可以专注于设计和编写软件。然而，确保有人收集需求的重要性再怎么强调也不为过。在一个软件项目中（实际上是任何项目，但让我们专注于软件项目），以下几个阶段是常见的：

*   需求收集：记录系统将是什么样子，并确定项目成功的标准。
*   设计：将需求转化为实施计划。
*   实施：编写软件代码。
*   测试：验证软件是否按预期工作。
*   维护：进行更改以解决测试中未发现的问题。


项目在需求收集之后的每个阶段，都依赖于需求来确保目标得以实现。需求如同路线图，为你指明到达目的地的方向和目标。若不先明确需求就尝试构建数据库，就如同没有地图的旅行。旅途或许有趣，但你可能会发现自己**本该在阿尔伯克基左转**，结果非但未能在皮斯莫海滩晒羽毛，反而不得不与雪人搏斗。缺乏合理的需求，很大比例的项目将无法满足用户需求。一个需要考虑的非常实际的讨论是：多少需求才算足够？在软件开发的早期，这些阶段是针对整个项目依次进行的，即你收集所有可能需要的需求，然后在开始编码前设计整个软件项目，等等。这种项目安排方法被称为**“瀑布方法”**，因为一个步骤的产出会流入下一个步骤。瀑布方法常被嘲笑为普遍糟糕（当然也很酷不起来），因为每个阶段都必须全部完成才能进入下一阶段。越来越多的团队正在使用**“敏捷”**项目开发方法，但阶段与瀑布方法相同，主要区别在于迭代和时间分配。在进行设计前，你仍然需要知道要做什么；在进行编码前，也仍然需要先完成设计。

本章我想阐明的一个重要观点很简单：无论你是否愿意，这些阶段都将被执行。我曾参与过一些项目，在项目启动时几乎同时开始了实现工作。最终，我们不得不回头收集需求，以弄清楚用户为何对我们的产出不满意。而当我们直接从需求收集跳到实现时，情况更是一团糟，因为每个程序员都各行其是，最终系统中的每个数据库、每个对象、每个接口都看起来完全不同。这是一场混乱，很可能直到今天仍在从中艰难跋涉。

本书确实主要是关于设计和实现的（尽管第 13 章有一节关于数据质量和测试）。然而，本章之后，我将假设需求已经完成，设计阶段已正式开始。关于软件需求收集和文档记录过程的书籍已有很多，因此我甚至不打算尝试深入举例说明需求。相反，我会快速列出我在需求中所寻找的要点。正如作家盖莱特·伯雷斯曾谈论艺术时所说，“我对艺术一无所知，但我自己喜欢什么我知道”，需求方面的情况也确实非常相似。无论如何，需求应当被捕获，通常你可以通过几个关键标准来区分需求的好坏：

*   需求通常应包含极少关于如何解决问题的**技术细节**；它们应只包含问题的定义和成功标准。例如，一个好的需求文档可能会说“职员们必须心算所有的加法，这很慢且容易出错。为了项目成功，我们希望以避免错误的方式完成数学计算。”一个差的需求文档则会将最后一句替换为“...我们希望使用德州仪器 TI3243 计算器完成数学计算。”计算器可能是一个解决方案，而且可能是个很好的方案，但这个决定应留给技术人员，以避免过度限制最终解决方案。
*   使用的语言应尽可能**具体**。例如，考虑这样的陈述：“我们只为新招聘的数据库管理员支付每年 20,000 美元的年薪，且第一次加薪在六个月后。”如果这是实际需求，公司将永远无法招聘到合格的数据库管理员——永远不能。如果你在软件中照此实现了该需求，那么当公司第一次想打破这个规则时（比如保罗·尼尔森这样的人才出现了），那个用户会诅咒你的名字，雇佣保罗担任首席执行官，仅仅是个虚职，六个月后，再将他的职位更改为数据库管理员。（用户总会找到完成工作的方法！）如果需求写得足够具体，它会说“我们通常只...”，其实施方式将大不相同。
*   需求应便于客户**阅读和验证**。纯粹而简单，使用用户能理解的语言，而不是他们仅仅一带而过的技术术语，这样他们直到软件未能满足需求时才会意识到你错了。简单的图表和图片也是很好的沟通工具。

在我看来，你如何记录需求并不重要，重要的是它们**被写下来**。写下来。写下来。希望如果你忘了本章我说的其他内容，你会记住这一点。如果你已婚或有父母，你可能犯过这样的错误：“是的 ______，我保证会为你完成那件事”，然后很快就忘记了具体说了什么，最终导致争吵。“是的，你确实说过你想要屏幕是蓝色的！”你对你的客户说。此时，你无异于在说他们是骗子或愚蠢的，这可不是好的商业做法。另一方面，如果你转发了他们同意将屏幕设为蓝色的文档，那么承认错误的责任就在他们那边了。

最后，在软件创建过程的其余部分，我们将如何使用书面需求？在设计阶段，需求是你塑造软件的指南。技术细节由你（或公司标准）决定：是两张表还是三张表，是存储过程还是即席访问，是`C#`还是`VB`？但最终输出应能通过将设计与需求进行比较来验证。当需要进行整体系统测试时，你将把需求作为成功的标准。之后，你将用需求来测试你的设计和代码，以确保你满足了它们。

在本章中，我将介绍需求收集过程中的两个特定部分：

*   **记录需求**：我将简要介绍在整个项目过程中，关于记录需求你会关注哪些方面。
*   **寻找需求**：在这里，我将讨论寻找信息的地方以及挖掘这些信息的一些技巧。

需求不是项目中微不足道的部分，而且绝对不应该被省略，但像任何事情一样，它们也可能被过度执行。本章将给你一点建议，告诉你该从哪里寻找，或者，如果你是那个不必负责收集需求的幸运程序员，该确保哪些方面已被审视。编程的悲哀现实是，如果你创建的系统很糟糕，是因为你得到的需求很差劲，那么需要重写的不会是收集需求的人。


## 记录需求

如果你曾去过一个没人和你说同样语言的地方，你就会体会到那种仅仅因为沟通不畅而产生的孤立感。周围人说的话你都听不懂，无论你问多少次洗手间在哪里，换来的都只是茫然的眼神。这与智力无关；只因你们语言不通。这听起来显而易见，但你不能指望另一个国家的全体国民都完美学会你的语言，只为了满足你的需求。更好的办法是你去学习他们的语言。即使两人说同一种基础语言，也常常会有让人困惑的方言和措辞差异。而这正是我们常常对用户所抱有的期待。

信息技术专业人士和我们的客户之间也往往存在这类沟通问题，因为我们这些技术人员常常与客户说着不同的“方言”，甚至不同的“语言”。客户倾向于用他们行业的语言来思考，而我们则习惯用计算机解决方案的思路来思考。当你作为用户时，很可能也有和他们相同的感受。例如，想想 SQL Server 的工具。我们这些关系型程序员很难向工具设计者传达我们在 SQL Server 工具中想要什么。他们为大多数任务做得足够好，但显然，他们与用户并不完全在同一个频道上。

在分析过程中，你应该尽早养成一个习惯：尽可能多地、合理地记录、记录、再记录你所获取的信息。有时人们会休假，带走了他们脑海中大量的工作知识，而他们有时休完假就不回来了（你懂的，因为他们可能得到了一份在迪士尼乐园担任丛林巡航船长的工作！）。这个情景有多种变体，并非都令人愉快地细思。没有文档记录，你很快就会面临丢失重要细节的风险。千万不要试图把所有事情都记在脑子里，因为即使是记忆力最好的人，也往往会忘记项目的细节（尤其是如果他们最终留在了那个“魔法王国”）。

以下是一些有用的建议，当你开始记录用户需求时可以参考：

*   尝试维护一套文档，用于共享系统需求和规格信息。需要考虑的重要文档包括设计会议记录、描述口头变更请求的文档，以及对所有规格（如功能、技术、测试等）的签收确认。
*   除了正式文档，及时更新并确保设计团队成员充分知情也很重要。为所有信息建立并维护一个公共知识库，并保持其更新。
*   在任何你添加了用户未提供或未明确同意的信息之处，务必加以注明。
*   尽早确定项目范围，并尽力使需求范围与之保持一致。这将防止项目变得过于庞大或多样化，以至于无法在合理的时间和预算内完成。在过程的早期就厘清那些影响预算的变更，特别是会增加预算的变更，可以避免未来的不快。
*   要简洁，但也要全面。一份文档可以是 1000 页，也可以是 10 页，但表达的内容可以相同。关键的检验标准是：用户、架构师和程序员能否使用它并实现最终目标？

一旦你生成了文档，一个关键步骤紧随其后：确保客户同意你对其目标的描述版本。在进行整个系统设计过程中，客户无疑会在实体、数据点、业务规则、用户界面、颜色等几乎所有方面改变主意，而你必须为此做好准备。无论客户想要或需要什么，都是你必须努力去实现的，因为他们最终控制着项目。不幸的是，这常常意味着需要通过项目经理这样的第三方进行沟通，并且需要有足够的灵活性来应对任何提出的变更，无论大小。这种设置起初听起来很棒，因为你以为项目经理会为你翻译，并站在质量和正确性的一边，有时确实如此。但更多时候，经理会把客户的需求误解翻译成一些相当奇怪的东西，然后坚称那就是客户的愿望。“我需要所有数据在一个屏幕上”会被翻译成“我需要所有数据在一张表里”。最好的情况是，经理能分清谁是技术人员、谁有业务需求。如果你从事的是一份典型的工作，那么更接近现实的恐怕是最坏的情况。

除了与客户交谈之外，收集尽可能多的笔记、打印件、截图、便携式硬盘等载有电子表格、数据库备份、Word 文档、电子邮件、手写笔记等资料的载体，这些对于当前问题的任何现有解决方案都至关重要。这些数据在发现数据元素、屏幕、报告以及其他需要设计到你应用程序中的要素时会非常有用。通常，你会在客户的“文物”中找到在构建数据模型时极为宝贵的信息。

> **提示**
> 在设计和实施的整个过程中，你无疑会发现对原始需求的变更。务必持续更新你的文档，因为世界上写得最好、格式最精美的文档，如果过时了，也就毫无用处。


## 收集需求

### 收集需求的方法

收集需求可能是一项艰巨的任务，因此从小处着手，并持续添加，直到你和你的客户都同意你已经充分理解了问题，可以继续推进。最初并不需要收集一个大型系统所有领域的每一个需求；系统可以被分解成若干部分，通常称为主题区域。主题区域的大小取决于团队的需求和所使用的开发方法。例如，像 `Scrum` 这样的敏捷方法，会将所有事情分解成小单元，以便频繁地进行设计、编码、测试和发布（例如，每两周一次）；而像瀑布式方法则会要求你先设计整个系统，然后再开始编码。无论你采用什么方法，最终的结果至少应该看起来是相似的（允许技术和公司需求发生变化）。重要的是，所有的开发方法都会告诉你一件事：在编码之前，先设计好你要编码的内容。

对于收集需求，有很多工具和方法论可用于记录流程、业务规则和数据库结构。统一建模语言（`UML`）是一个可能的选择；微软解决方案框架（采用 `UML`）和 Rational Unified Process 是其他选择。`IDEF` 系列方法中也有几种用于业务流程建模的模型类型；我们将在 第 3 章 介绍它们的数据建模技术。我将采用实体-关系（`E-R`）建模方法 `IDEF1X` 来为数据库建模。对于项目中非数据库结构部分，我将不会介绍任何其他建模语言，而是使用一种简单的手动电子表格方法，这无疑是记录需求的最常用方法——即使是在中型组织中，在文档工具上花钱也可能比教你的宠物半蜜蜂 `Eric` 在玩 `Words with Friends`（“再次嗡嗡叫？”）时做出好的单词选择还要困难。

### 详细规格说明的重要性

无论使用什么工具来记录需求，数据库设计过程的需求都是一样的。需要获取规格说明，以引导你作为数据库设计师发现以下所有内容：

*   实体和关系
*   属性和域
*   可以在数据库中强制执行的业务规则
*   需要使用数据库的过程

没有这些规格说明，你要么需要不断回去询问客户一大堆问题（他们有时每被问两次就会给出三种不同的答案，这会锻炼你的辨别能力），要么就开始做猜测。虽然猜错几次对于学习如何不做事情来说是很好的教育，但这肯定不是高效工作的办法（除非你碰巧是了不起的 `Kreskin`，能 `99.9%` 地猜对，尽管我很确定那是个把戏，他也做了他的需求收集工作）。

作为实施数据库系统过程的主要部分，数据架构师的目标将是生成数据库的图形化模型，我们将在后续章节中介绍这一点。

**提示**

在项目的早期阶段，先弄清楚“是什么”和“为什么”；然后再处理“怎么做”。一旦你了解了需要构建内容的细节，构建它的过程就会相当自然地进行，你甚至可能将现有的模式应用到解决方案中。

模糊性可能导致后续过程中不必要的讨论、争论甚至诉讼。因此，请确保你的客户理解你将为他们做什么，并使用能够被清晰理解但又足够具体以描述你在信息收集过程中学到的内容的语言。

### 数据库架构师的活动

在整个发现过程中，将会收集和生成一些构件，这些构件将在整个实施过程中用作参考资料。构件是任何对设计很重要的文档，例如访谈记录、电子邮件、示例文档等。在本节中，我将讨论一些你作为数据库架构师需要非常关注的主要活动类型：

*   采访客户
*   提出正确的问题
*   与现有系统和原型协作

这绝不是寻找和获取文档来源的详尽列表；事实上，还差得远。目标只是为了让你开动脑筋，思考从客户那里获取哪些信息，从而让你的工作更轻松。


### 客户访谈

设计数据存储的人员（通常称为 `数据架构师`）可能永远不会见到用户，更不用说参与正式访谈了。项目经理、业务分析师和/或系统架构师可能会提供所有所需信息。有些项目可能只涉及一位 `数据架构师`，或者由一个人身兼数职，比整个第四机动部队演习时承担的角色还多。这两种方式我都经历过：我曾参与早期的设计会议，也根据文档工作过。与你合作的人越优秀，后一种选择就越有利。在本节中，我将快速介绍客户访谈的基础知识，因为在几乎任何项目中，你最终都需要对客户进行一定程度的访谈。

客户访谈通常是项目真正启动的地方。这是自由、非结构化信息流开始的起点。然而，这也是沟通鸿沟开始的地方。许多客户通常以视觉方式思考——考虑表单、网页，或许还有简单的用户界面。用户也倾向于仅从自己的视角思考。例如，他们可能用 `错误` 这个词来表示流程为何没有按预期运行。这些错误状况可能不仅是实际的错误，也可能是用户做出的选择。因此，像 `计划内维护` 这样的值也可能被归类为错误状况。这很大程度上需要帽子上绣着“分析师”字样的人去分析用户实际想要什么。

因此，工作就是平衡客户感知到的想要和需求与他们真实的需求：一个结构合理、能很好地置于用户界面背后并捕捉他们真正目标的数据库，具体来说，就是让他们的业务生活更轻松、更有利可图的信息。更改表单以添加新的文本框、标签或其他元素是一项相对简单的任务，这会给用户造成错觉，认为创建整个应用程序是件容易的事。如果你想要证明，那就犯个愚蠢的错误：演示一个外观精致、使用非硬编码值的原型应用程序，让客户以为它真的能运行。客户可能会对你如此迅速地搭建出某些东西印象深刻，并期望你快完成了。他们很少会理解，隐藏在“引擎盖下”的东西——即数据库以及其他业务逻辑层和界面对象层——才是所有主要工作发生的地方。

> **提示**
>
> 虽然在过程中，视觉元素是寻找用户所需数据线索的好地方，但你要小心，不要让数据库设计过分围绕某个特定的界面。数据的结构应由数据的含义决定，而不是由其呈现方式决定。呈现更多是界面设计的任务，而非数据库设计的任务。

与用户进行头脑风暴会议也能一次收集大量信息，取得很好效果，只要与会人数不要太多（如果你的会议需要现场提供午餐的餐饮服务，那你们做不出什么重大决策）。这里的关键是要确保有人主持会议，防止 `强势人物` 压制其他人、只发表自己的意见（如果你自己就是那个 `强势人物`，那就更糟了！）。认真对待每位受访者的发言，因为每个人都可能持有不同但有价值的视角。有时（好吧，通常是这样）最好的信息并非来自高管，而是来自实际工作的人。不要假设第一个人就能代表其他人，即使他们都在同一个项目上工作，或者即使这个人是经理（甚至是主要公司的总裁或所有者，尽管有时需要极大的技巧来走好这根钢丝）。

在许多情况下，当主导型人物无法被约束，或者胆小的人无法被鼓励参与时，应采用一对一的面谈，让所有客户畅所欲言，不受意志坚强（尽管有时并非头脑更强）的同事不合时宜的打断。请注意，声音最大、最大胆的人可能没有最好的想法，而那个坐在后面一言不发的安静的人，可能掌握着整个项目的关键。确保至少考虑每个人的意见。

本书这部分内容是我以最谦卑的态度写就的，因为我在设计流程的这一部分犯的错误比其他任何部分都多（而且像所有经验丰富的人一样，我在软件工程的各个层面都犯过相当多的错误）。客户访谈是我所遇到的流程中最困难的部分之一。对于经验丰富的分析师来说，这似乎不是一个合适的话题，但即使是我们中最优秀的人也需要被提醒：操之过急、欺压客户、在他们告诉你之前就告诉他们想要什么，甚至未能管理好用户的期望，都可能导致一个开发良好的系统走向失败。如果你的地基不稳，最终产品很可能也不稳固。

### 提出正确的问题

在粉刷任何房屋的内部之前，粉刷公司的代表会问他们每位客户一系列问题（使用什么颜色？粉刷哪些房间？儿童房？用于写写擦擦？）。几乎任何计算机软件项目也是如此。在接下来的章节中，有一些问题对于系统开发中的数据库设计方面至关重要。显然，这不是一个详尽的清单，但它肯定足以让你开始，所以至少，你不会有一天坐在房间里却不知道该说什么。

#### 需要什么数据？

如果 `数据架构师` 是项目团队的一员，那么系统显然需要一些数据。大多数用户在较高层面上知道他们希望从系统中看到什么数据。例如，如果他们在会计部门，他们希望看到按某某分组汇总的美元和美分。在你的流程中的某个时间点，区分什么是必需的数据和什么只是“有了更好”的数据将非常重要。这显然是一个非常直接的问题，客户可能根本不知道他们需要什么，但许多用户已经在使用数据，无论是在现有系统上（他们要么讨厌这些系统，并很高兴你为他们替换掉；要么会因为你改变现状而讨厌你），或者是在他们自 `VisiCalc` 以来一直使用的电子表格系统中。


### 数据将如何使用？

了解客户计划如何使用系统中的数据确实是一项重要的信息。这样你不仅能理解将要尝试建模的流程，还能开始清晰地描绘出需要存储的数据类型。

例如，想象一下你被要求为一个牙科诊所创建一个联系人数据库。你可能需要了解以下内容：

*   联系人姓名是否仅用于打电话，就像一个快速电话簿？
*   客户是否会向联系人列表的成员发送电子邮件或发布信息？为此是否需要将姓名细分为不同的组？
*   客户是否会使用这些姓名来征求邮件回复，例如预约提醒？
*   记录家庭成员信息是否重要？他们是否希望在重要的日子向相关人员寄送卡片？

在设计流程的早期阶段，使用方式的问题可能看起来像是越界了，在某种程度上，你也可能是对的。但从宏观上看，使用信息绝对是有用的。了解数据可能被使用的流程类型很重要，但它可能出现在哪个屏幕上则相对不那么重要。例如，以地址为例。如果只是偶尔使用才捕获地址，你可能只需要给用户一个单行的字符串来输入整个地址。但如果你的业务涉及邮寄，你可能需要按照邮政局的精确规格来格式化它，这样就不必支付和普通人类一样的邮费。

#### 哪些规则管理数据的使用？

几乎你想要存储的每一条数据，都会有规则来管理其存储、使用和访问方式。这些规则将为你将要创建的模型提供大量指导。以我们之前的联系人为例，你可能会发现以下规则：

*   每个联系人必须有一个有效的电子邮件地址。
*   每个联系人必须有一个有效的街道地址。
*   客户使用邮件例程检查每个电子邮件地址，在该例程成功执行之前，该联系人不是有效联系人。
*   联系人必须根据他们遇到的问题类型进行细分。
*   只有特定用户才能访问联系人的电子邮件地址。

在流程早期阶段收集规则时，仔细处理措辞很重要。很多时候，你得到的规则在写下来时看起来相当直白，但实际情况往往并非如此简单。在假设规则为真之前，与分析师（可能还需要直接与客户）确认你所审查的规则是至关重要的。

举一个恰当的例子，什么是“有效”的电子邮件地址？嗯，就是与该联系人准确对应的电子邮件地址。当然，但到底如何验证呢？事实是，在许多系统中你并不验证。通常，它的实现意味着该字符串符合电子邮件地址的格式，即在其他字符之间有一个&符号，并且在多个字母数字值之间有一个点（`.`）（例如 `%@%.%`，加上 A 和 Z、0 和 9 之间的所有字符、下划线等），但具体含义完全取决于解释。另一方面，在其他类型的系统中，你实际上要求用户从电子邮件中获取一些信息来验证它确实是一个有效的电子邮件地址，并且输入数据的人拥有它的权限。这很大程度上取决于系统的需求，但这个问题用完全相同的英文词语可以轻松提出。

真正的问题出现在你过于严格地解释规则，最终产品变得不可接受，因为你对数据施加了客户不想要的过度限制性规则，或者你遗漏了客户真正需要的规则。我曾经犯过一个大错，这导致一个系统在早期上线的几周内陷入困境。客户似乎希望严格执行的规则，根据其客户的需求，有时需要在个案基础上被覆盖。不幸的是，我们的程序没有让用户覆盖这些规则，而且他们在用户验收测试中从未尝试模拟这种情况，因此只能咬牙切齿、夜不能寐地修复这个问题。

有些规则可能存在另一个问题：客户想要这个规则，但实现它不可行或不切实际。例如，客户可能要求网站的所有注册访客都必须输入有效的手机号码，但能确定访客会提供这些数据吗？而到底什么是有效的手机号码？你能仅通过格式来验证它，还是必须通过呼叫或向电话公司查询来验证该号码？如果用户提供的是固定电话呢？在这个流程阶段，可实施性的考虑是有限的。将来必须有人执行这个规则，这个问题会在流程后期得到解决。



## 报告涵盖哪些数据？

报告往往是设计过程中最容易被遗忘的部分之一，但对客户而言，它几乎无疑是项目中最重要的环节。通常，让计算机系统产生价值的关键在于其能够详细报告系统中所有活动的能力——例如，组织不同部门的生产力如何，销售人员的效能如何（比如在第一个星期三之后的第二个星期二，因为……嗯，只是因为每个企业都会做一些在外人看来古怪的事情，甚至有些内部人员也这么觉得）。所有这些问题构成了计算机系统存在意义的“为什么”，即使对非常小的公司来说也值得为此部署系统。

许多新手开发者会将报告的设计与实现留到最后一刻（这是我在过去多年中犯过不止一次的错误，也因此吃过很多苦头）。对用户而言，报告是将数据转化为信息的地方，是重要决策的基础，可能成就或毁掉一家公司。需要注意的是，这不是工具的问题。`Reporting Services`、`Power BI` 等工具都能让任务变得更轻松。问题在于知道要捕获什么数据。如果你需要知道冰柜过去五天的温度，那么初步设计的一部分就是以时间序列方式捕获至少五天的温度数据。如果你不知道这个需求，可能只会存储当前冰柜温度的单个数值，从而无法满足报告需求。这本是个容易解决的问题，但往往在项目后期会出现许多类似的小麻烦，最终导致实现效果不尽如人意。第一次就做对，显然比事后补救要好得多。

回到联系人示例，客户希望在报告中看到什么名称？以下几种形式值得考虑：

*   名，姓
*   名，中间名，姓
*   姓，名
*   昵称

尽早敲定此类问题很重要，无论它们此刻在你看来多么微不足道或愚蠢。它们对客户很重要，而你应该始终记住，客户是付费方。坦白说，报告最重要的规则是：你无法报告没有捕获的数据。

从数据库设计的角度看，报告内容极其重要，因为它通常有助于发现其他方式未考虑到的数据需求。暂时不必关心报告的美观性，因为那可能会让人忍不住直接开始编码，而偏离了建模。

提示

不要忽视任何现有报告，因为新系统中可能有对应的报告。虽然很少直接照搬旧报告，但其内容往往包含客户在表达需求时可能从未想到的数据。当前生产系统中可能有数百份报告，在新系统中，除非能整合许多报告，否则数量几乎肯定会上升。具体的报告架构将在第 14 章中更详细地介绍。

## 数据目前在哪里？

偶尔有机会创建一个完全没有现有数据的新数据库是件好事。这会让工作变得轻松愉快。不幸的是，随着年复一年，找到一个全新系统来实施的可能性，比布朗队赢得超级碗还要低（无意冒犯布朗队球迷，但事实如此）。唯一可能的例外是构建一个交钥匙方式销售给最终用户的产品（那样的话，现有数据是他们的问题，或者如果你购买他们的系统，就成了你的问题）。在我参与过的几乎每一个系统中，我都在创建某个其他系统的更好版本，因此我们必须考虑转换对最终用户重要的现有数据。（其中有几个是全新的系统。这些经历非常美好，原因有很多：我们不仅不需要处理数据转换，也不需要处理现有流程和代码。）

每个组织都不同。有些将数据集中存储在一个地方，而有些则分散在许多（非常多）地方。数据很少已经存在于一个你可以轻松访问的结构良好的数据库中。如果真是那样，客户为什么还要来找你呢？客户的数据通常存在于以下各种地方：

*   大型机或遗留服务器：数百万行活跃的`COBOL`代码仍在许多公司运行。
*   电子表格：电子表格是查看、筛选和分析数据的绝佳工具，但绝不是维护复杂数据的合适场所。大多数用户知道如何将电子表格用作数据库，但不幸的是，他们并不擅长确保数据的完整性，因此这些数据无疑会让你头疼不已。
*   桌面数据库，如`Microsoft Access`：桌面数据库是很好的工具，易于部署和使用。然而，这种易用性往往意味着这些数据库是由非技术人员构建和维护的，设计不佳，当数据库需要扩容或修改时可能会引发许多问题。
*   文件柜：即使在二十一世纪的今天，许多公司仍然很少或根本不使用计算机（除了玩纸牌游戏），而是堆积大量纸质文件。你的项目可能只是用基于计算机的系统替换一个文件柜，或者提供一个简单的数据库来记录现有纸质文档的物理位置。

你需要纳入正在设计的`SQL Server`数据库的数据，将来自这些以及你从客户那里发现的其他奇怪而美妙的来源（现实常常比小说更离奇）。更糟糕的是，电子表格、文件柜和设计不良的计算机化数据库并不强制执行数据完整性（而且通常桌面数据库、大型机应用程序甚至现有的`SQL Server`数据库也不一定做得那么完美），因此要随时准备处理脏数据，在存入你崭新的数据库之前必须进行清理。

## 数据是否需要与其他系统集成？

一旦你大致了解了客户重要数据的位置，就可以开始确定你新的`SQL Server`解决方案中的数据，将如何与那些保持原格式的数据进行交互。这可能包括构建通往大型机的复杂网关连接、连接到其他`SQL Server`、`Oracle`数据库或`Hadoop`系统的链接服务器连接，甚至链接到电子表格。在设计阶段，对此问题不能做太多假设。仅了解你需要处理的架构，对后续工作也可能有所帮助。

提示

永远不要期望你将要转换或集成的数据具有任何质量。太多项目启动时对所需工作量的预估不足，而数据清理一直是其中最难估准的部分。理解数据库中已有的内容已经够难了，如果数据质量很差，你的工作难度将成倍增加。


## 这些数据价值几何？

对数据进行价值判断至关重要。在某些情况下，数据具有显著的货币价值。例如，稍后在第 4 章中将介绍的牙科诊所案例中，价值在于记录了对患者进行的治疗、向患者及其保险公司收取的费用。若没有这些文档记录，要找出这些数据并最终为完成的工作获得报酬，可能需要耗费数小时甚至数天。这类数据具有特定的货币价值，因为收款越快，产生的利息就越多，意味着利润更高。如果客户通过流程优化，将付款周转时间从一个月缩短至一周，这可能意味着多赚不少钱。

另一方面，仅仅因为现有数据唾手可得，并不意味着它就应该被纳入新数据库。需要告知客户所有可用数据，并向其提供将其迁移到新数据库的成本估算。迁移遗留数据的费用可能很高，客户应有机会做出决策，以便将资金用于更重要的用途。

## 谁会使用这些数据？

“谁将使用数据？”这个问题，可能在需求收集的早期阶段，并不会立刻让你觉得这是一种需要考虑的数据类型。在设计界面时，通常实际按下按钮的是谁，可能对按钮设计影响不大（除非需要考虑残障人士的使用）。然而，“谁”这个问题的答案，可以着手回答几种不同类型的问题：

*   **安全性**：“谁将使用数据？”可以从两个方面理解。首先，这些人是唯一关心这些数据的人。其次，这些人是唯一有特权使用这些数据的人。后者将要求你为使用设置权限边界。有趣的是，若加入像《健康保险流通与责任法案》(HIPAA) 或《萨班斯-奥克斯利法案》(SOX) 这类隐私法，或是全球范围内其他上百条善意但往往惩罚数据库管理员多于违规者的法律，问题就更复杂了。
*   **结构**：如果多个用户群体需要相同的数据，但需求却大相径庭，这可能在后续过程中导致不同的设计结果。
*   **并发**：支持一个并发用户系统的考量，与支持十个、一百个、一千个乃至更多并发用户的系统是不同的。用户数量不应改变我们的概念或逻辑设计，但它肯定会改变我们设计物理层的方式。关于并发，本书直到很后面（第 11 章）才会详细讨论，但此刻你正在提问，并很可能需要指定未来的硬件/软件，所以现在就了解起来也无妨。

这个关于谁会使用数据的选择，与你在需求收集过程中已解答的所有其他问题息息相关。当然，这些问题仅仅是信息收集过程的开始，在能够开始构建数据库之前，还有大量工作要做，所以你还得按捺住性子再等一阵。

## 是否在替换现有系统？

如果你正在为一个现有的数据库系统编写新版本，那么对现有系统的访问将是福也是祸。显然，你能收集到的关于先前系统及其数据结构的信息越多越好。所有的界面屏、数据模型、对象模型、用户文档等等，对设计过程都很重要。

然而，除非你只是对现有系统进行修订，否则通常旧数据库系统仅作为完整性参考点，而非初始蓝图。在大多数情况下，你将要替换的现有系统会有许多需要修复而非模仿的问题。如果被替换的系统毫无问题，客户为什么要替换它？也许只是为了迁移到更新的技术，但没有人会仅仅为了找乐子而替换一个运转正常的系统。替换现有系统最糟糕的部分是什么？测试。旧系统显示订单数量：100；新系统显示：102。哪个是对的？无论新代码测试得多么完善，总得有人证明新系统是正确的。

使现有系统变得复杂的另一个因素是遗留数据。如果你有 20 年的历史数据，可能基于一个专有的、基于分层文件的主机应用程序，常常会出现两个问题。第一：新系统“需要”像旧系统一样工作。第二：20 年的数据通常带来了海量的结构问题。因此，需求、设计和实施计划的一部分需要解决如何将旧数据导入的问题。

早期设计过程中可能也存在原型。原型可以作为有用的工具，用来沟通如何通过计算机解决现实世界的问题，或者当你试图重新设计当前流程的管理方式时。它们的作用是作为一个概念验证——一个与设计团队和最终用户共同充实项目关键要素的机会，项目的成败将取决于这些要素。

## 利用其他类型的文档

除了访谈和现有系统，你还可以从其他来源寻找数据规则和与设计项目相关的其他信息。通常，项目经理会获取这些文档；有时，它们可能无法提供给你，你只能依据他人对文档内容的描述。在这些情况下，我发现最好将你的理解书面化，并确保明确是谁对你无法查阅的文档含义做出了何种说明。一如既往，以下列表绝非详尽，但应能启发你思考可以从哪里获取你正创建或替换系统的现有文档。

### 早期项目文档

如果你在一家为其他公司开发软件的公司工作，你会发现项目早期通常会创建一些用于征求成本和可能解决方案的文档，例如：

*   **报价请求 (RFQ)**：一份包含相当成熟规格说明的文档，由组织发出以确定解决方案的成本。
*   **提案请求 (RFP)**：针对不太成熟的想法，组织希望看到潜在解决方案并了解其大致成本。

这些文档都包含了有价值的信息，可以帮助你设计解决方案，因为你可以了解在你介入之前客户的需求。当然，情况会变，最终解决方案并不总是与最初的要求一致，但一份 RFP 或 RFQ 的副本应被添加到你将在后续流程中使用的信息堆中。尽管这些文档通常包含对问题和期望解决方案的粗略信息，但你可以用它们来确认最初想要数据库系统的原因，并更准确地把握要在其中存储的数据类型。

无论如何，如果你能获得这些文档的副本，你将能看到客户的思考模式以及客户为何希望开发这样一个系统。

### 合同或客户工单

获取合同副本看似是收集设计信息的一种相当激进的方法，具体取决于你所在的组织类型。坦率地说，在公司架构中，你可能需要突破层层管理，才能让他们理解你为何需要查看合同。合同本身可能因其使用的语言而难以阅读（类似于一种简明版本的编程语言，并夹杂着故意的模糊性，以便日后给律师们留下争论的余地）。然而，只要勤勉地过滤掉法律术语，你就能发现本质上相当于系统基本需求集的内容——通常这些是你必须严格满足、否则可能无法获得报酬的要求。更有趣的是，你可能会发现一些已承诺给客户、但实施团队却从未听说过的内容。

合同之所以如此有吸引力，原因很简单。一般来说，它是你将要瞄准的目标。无论客户说了什么，或者现有系统是什么样的，如果合同规定你交付某种水上交通工具，而你因为下层客户改变主意却未修改合同，最终交付了一辆一级方程式赛车，那么你可能无法获得报酬，因为你的项目被视为失败（当然是比喻性的说法，也许他们会允许你保留那辆一级方程式赛车？）。

### 服务等级协议

合同中一个对设计过程同样重要的部分是规定的服务等级。这可能包括每分钟的网页访问量、数据库中的行数等等。所有这些都需要进行测量、存储、测试等。当进入测试和优化阶段时，了解目标服务等级可能具有巨大价值。你可能还会发现一些需要存储的数据，用于验证是否达到了服务等级。

### 审计计划

别忘了审计。构建系统时，你必须考虑系统未来是否可能被审计，以及由谁审计。政府机构、ISO 9000 认证客户以及其他受标准组织监督的客户，很可能有严格的审计要求。其他客户也可能有财务审计流程。尤其需要关注的是各种隐私政策、儿童数据限制、信用卡加密规则等等。所有这些不仅要求你遵守监管机构制定的规则，还要求你记录运营的某些部分。这些审计计划可能包含可用于设计过程的有价值信息。

### 原型

原型有点像是一个活的文档，创建它是为了让用户能感受软件可能如何为他们工作。原型是极好的沟通工具，但它们侧重于视觉呈现，而非内部结构。原型真正的问题在于，如果为原型创建了数据库，那么这个数据库通常没什么实际价值。因此，当开始数据库设计时，你可能会被要求接手一个仓促开发的原型数据库，并“让它能用”，或者更糟的是，“稍作润色”。实际上，你可能会继承一个结构混乱、组织不清的原型，而你的任务将是立即将其转化为生产数据库（通俗地说，就是“昨天就该完成”）。

有时可能需要你提醒客户，将原型仅视为交互式图片，用于让客户尝试某个概念，通常是为了让你们的公司签下合同。作为数据架构师，你必须尽最大努力，只将原型代码用作工作文档，以指导你自己的设计。原型有助于确保你不会遗漏用户所需的任何关键信息——例如姓名字段、搜索操作，甚至是一个按钮（这可能暗示着某个数据元素）——但它们可能完全无法告诉你关于最终数据库设计的任何信息。

### 遵循最佳实践

在处理和收集需求时，遵循以下一些最佳实践列表可能很有用：

*   **保持勤勉：** 仔细检查所有内容，确保所表达的内容合理。在进入下一步之前，务必尽可能多地理解约束系统的业务规则。早期阶段的错误可能会在后续阶段像滚雪球一样越来越大。
*   **记录文档：** 文档的格式并非那么重要，重要的是尽可能多地记录客户的需求。确保文档所有相关方都能理解，并且对后续的实施工作有用。
*   **保持沟通：** 与客户保持持续沟通对于确保设计不偏离正轨至关重要。危险在于，如果你开始误解客户的需求，那么之后的每个决定都可能是错误的。尽可能多地与客户进行面对面交流。

## 总结

在本章中，我简要介绍了一些关于文档编制和需求收集的基础知识。这是创建软件过程中最重要的部分之一，因为它是后续一切工作的基础。如果基础牢固，后续流程才有成功的可能。如果基础草率，那么构建起来的系统其余部分很可能也是如此。此过程的目的是尽可能多地获取关于客户希望从其系统中获得什么的信息。作为数据架构师，这些信息可能会交付给你，或者至少大部分是。无论哪种方式，目标都是理解用户的需求。

一旦你从用户那里获取了尽可能多的文档，真正的工作就开始了。贯穿所有这些文档，目标是为你做好准备，进入下一步：创建数据模型，该模型将以非常正式的方式记录以下所有内容：

*   实体和关系
*   属性和域
*   可在数据库中强制执行的业务规则
*   需要使用数据库的流程

由此，将产生一个概念数据模型，该模型具有许多将在实际实施的数据库中存在的特性。在接下来的章节中，数据库设计肯定会从这个概念模型发生变化，但它将共享许多相同的特性。
