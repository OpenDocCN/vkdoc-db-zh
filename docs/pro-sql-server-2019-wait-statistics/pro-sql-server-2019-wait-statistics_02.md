# 引言

性能是许多数据库实施中的一个热点问题。许多企业在其数据库负载增加或规模扩大时，都会遇到与性能相关的问题。存在多种方法可在各个层面提高 SQL Server 的性能。其中许多性能优化方法被认为是最佳实践，例如执行索引维护以确保碎片化的索引不会拖慢您的查询，或更新统计信息以便 SQL Server 数据库引擎能拥有正确的信息来生成良好的执行计划。除了这些数据库维护方法外，您也可以选择更深入地钻研特定的查询性能故障排查，例如通过确保用成本较低的运算符替换成本高昂的运算符来优化查询。当然，总还有“大锤”式的方法，即用更新、性能更好的硬件替换当前硬件，希望这能解决您遇到的性能问题。

无论您选择哪种方法来优化或排查 SQL Server 性能，总涉及两个常见资源：时间和金钱。理想情况下，我们在致力于提高性能时，希望花费尽可能少的时间和金钱。知道应将时间和金钱集中于何处至关重要。如果您能发现性能问题的根源并在该层面解决它，就可以节省大量本会花在分析表象上的时间和金钱。

在某种程度上，我们可以将对性能问题核心的探寻比作医疗检查。医生并非只是开出各种药物直到某种药物实际见效，而是始终试图找到问题的根源，以便能对症下药，开出最适合该特定病症且不引起副作用的药物。同样的方法也适用于 SQL Server。不审视问题的真正根源就实施所有可能的解决方案，很可能无法解决真正的根本问题（除非您运气特别好），甚至可能使情况恶化。

这正是等待统计信息能提供帮助的地方。等待统计信息是在 SQL Server 数据库引擎执行查询的核心位置生成和维护的，为了解是什么拖慢了您的查询提供了宝贵的见解。在最新版的 SQL Server（SQL Server 2017）中，有 921 种不同类型的等待统计信息，并且随着新功能的引入或现有功能的修改和扩展，每个版本中这个数字都在增长。这是一大量可免费获取以帮助您进行故障排查的信息！

本书是我帮助您理解 SQL Server 等待统计信息的尝试。它将详细阐述等待统计信息是如何生成的，以及您如何利用这些信息来优化或排查您的 SQL Server 安装的性能。我还将描述特定的等待统计信息，并就如何自行解决问题为您提供指导。就本书而言，我个人认为过程比到达目的地更重要。因此，我花了更多时间描述和解释导致特定等待类型发生的原因，而不是写下所有可能降低其等待时间的方法。如果您理解了为什么会产生某种等待类型，以及它与 SQL Server 的哪个部分相关，您将很快发现自己的方法来降低它们的等待时间。

遗憾的是，由于不同等待统计信息的数量庞大，不可能描述和讨论全部。为此，我不得不选择一部分等待统计信息纳入本书。我的选择方法是从许多不同的 SQL Server 安装中收集等待统计信息数据，并挑选出最常见或对性能破坏性最大的那些，最终选定了 45 种不同的（或分组的）等待类型。
