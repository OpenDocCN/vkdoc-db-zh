# 1. 数据库驱动型 Web 开发基础

数据库最早由北美罗克韦尔公司的软件工程师在 20 世纪 70 年代中期创建。他们当时使用静态文件来记录他们正在生产的工作。他们开始注意到工作正在被重复，因为像设计阿波罗时代火箭这样的项目会产生数十万个不同的零件。每个零件都被赋予一个静态文件。在关系型数据库投入使用之前，跟踪这数十万个零件是一场噩梦。有了它，他们能够发现重复的工作并将其消除。

本章将从回顾数据库驱动型网站的基础知识开始，然后转向网络托管的优缺点，深入探讨可用的选项。你将看到现代数据库如何融入使用三种不同服务器（Perl、Apache 和 MySQL 服务器）来生成网页的过程。这三种服务器正是我们将在本书中学习协同使用的。继续阅读吧，这将充满乐趣！

## 数据库驱动方法的演变

在互联网的早期，数据库并未与网站一起使用。所有页面都是静态 HTML 页面。在今天看来，这些网站浏览起来会相当乏味，因为我们已经习惯了现代数据库驱动型网站。

世界逐渐认识到数据库是更高效的发展方向，最终导致了我们数据库驱动的世界。从银行到社交网络，我们所有的世俗服务现在都使用某种形式的数据库。

值得注意的是互联网。数据库被广泛用于提供我们每天浏览的页面。它们无缝集成到网页中，并改变了我们生活的世界。数据库将长期存在，因此了解一些关于它们的知识至关重要。

由于数据库在服务器上运行，必须有一种访问这些数据库的方法。现代计算中有许多不同的语言，因此也有很多不同的方法。我们将重点放在 MySQL 数据库与 Perl 脚本语言之间的交互上。

Perl 在与数据库交互方面具有强大的能力。它是一种已经存在了几十年的强大语言。这是一项成熟的技术，最适合处理文本。Perl 代表“实用提取与报告语言”，最初用于操作常见静态文本文件内的文本。由于我们在 Web 开发中使用的语言都是基于文本的，因此处理大型文本文档（例如 HTML 标记）时，Perl 是理想的选择。

结合数据库，Perl 是一个强大且有能力的引擎，可以驱动现代网站。Perl 通常在数据库运行的同一服务器或至少同一网络上运行。这允许快速高效地交互由大块文本数据组成的内容，例如当今互联网上每个页面中使用的 HTML 代码。

数据库正在不断发展，以满足更新、更快、更强大网站的需求。一个网站使用多个数据库来提供其网页服务是很常见的。这种可扩展能力是当今网站的基石。随着网站的增长，需要更多的数据库来运行网站。将你的网站数据从一两个数据库分配到，比如说，十几个数据库，可以让你在沉重的用户负载下，及时地从网站的不同部分访问这些数据库。

## 如何识别数据库驱动型网站

大多数人只是理所当然地认为他们的网页会加载预期的信息。作为一名开发者，你需要知道如何识别一个数据库驱动型网站。有几种方法可以做到这一点。我们将探讨识别它们的不同方法，并总体上了解使用了什么数据库、如何使用以及为什么使用。

### 网站地址（URL）

识别数据库驱动型网站的第一个，也许也是最简单的方法，是浏览器地址栏中的 URL。如果你查看一个静态 HTML 页面，你会看到一个以 `.html` 文件扩展名结尾的文件名地址。检查以下示例：

`http://www.domain.com/index.html`

你可以清楚地看到网址的组成部分：地址以标准的 `http://www.domain.com` 开头，以名为 `index.html` 的文件结尾。

大多数网站使用 `index.html` 作为其网站的默认起点。它是 Apache Web 服务器视为任何给定网站基础文件的主页，尽管完全有可能使用不同的文件名作为你网站的起点。配置 Web 服务器将在本书的后续章节中介绍。

`.html` 扩展名告诉你这是一个静态 HTML 页面。地址栏后面没有任何内容，因此可以很有把握地推测该网站根本没有使用数据库。

当你使用 Perl 呈现数据库驱动的页面时，文件扩展名是 `.cgi` 或 `.pl`。如果使用了数据库，文件扩展名后会显示一个问号。问号后面的内容对每个网站都是唯一的。检查以下 URL：

`http://www.domain.com/thisfile.cgi?id=1234`

你可以看到 URL 的基本组成部分，包括一个名为 `thisfile` 的 Perl 脚本，扩展名为 `.cgi`。然后是一个问号，后面跟着 `id=1234` 语句。这是一个很好的指标，表明正在使用数据库。问号后面可能有许多不同的语句，每个语句之间用 `&` 符号分隔，如下所示：

`http://www.domain.com/thisfile.cgi?id=1234&id2=5678`

你现在可以看到 URL 显示了两个不同的语句，称为参数。这些参数表明参数内的两组数据正在被传递给服务器。虽然这些参数也可能不使用数据库，但超过一个参数却未在某个环节使用数据库的情况是非常罕见的。



## 不同的内容区域

通过点击几个链接，你也能看出网站在使用数据库，因为你会发现一个页面看起来与前一个非常相似，只是主内容区域显示的内容不同。这是因为数据库驱动网站通常使用模板来呈现数据。我所说的模板，是指一套用于呈现页面顶部、底部和侧边栏的标记（例如 `HTML`）。页面中唯一变化的部分，比如说，是页面的中心区域。这就是由数据库生成的内容。

另一个识别数据库驱动网站的简便方法是注意广告横幅和按钮。如果它们随每个页面变化，甚至在你刷新页面时广告也会变化，那么在某种程度上，这一定使用了数据库才能实现。

当使用 `Perl` 和数据库时，你会注意到浏览器会弹出一个对话框，询问你是否要“重新发送表单数据”。这是因为你刚刚向数据库提交了数据，并点击了浏览器的“后退”按钮。刚才提交信息的 `Perl` 脚本又运行了一次，重复创建了你刚刚在数据库中创建的条目。

对于那些不与数据库交互的 `Perl` 脚本，你不会看到“重新发送表单数据”的对话框。随着我们学习本书的章节，你会发现这种情况经常发生。解决此问题的少数方法之一是在 `Perl` 脚本中提供一个易于识别的链接，该链接可以将你带回上一页，而无需重新发送表单数据，从而避免重复的数据库条目。

重新发送表单数据可能对你的网站有害，因为它会在你的数据库中复制数据。虽然重新运行的 `Perl` 脚本可能像计算页面点击量一样简单，但也可能像向服务器上传一打图片那样复杂——你会看到那 12 张图片在你的服务器上被复制了。

## 添加搜索功能

判断一个网站是否使用数据库的一个可靠方法是它是否提供搜索功能。虽然在不使用数据库的情况下搜索一个网站是可能的，但这在服务器资源方面代价高昂，并不常用。搜索数十甚至数百个静态的 `HTML` 文件会耗费过多的时间和服务器资源。

然而，通过数据库搜索内容正是数据库被创造出来的目的。`MySQL` 非常适合做这个，因为它是一个关系型数据库。在本章的另一个小节中，我们将进一步探讨“关系型数据库”这个术语。

搜索功能在大多数数据库驱动的网站上都很常见。`MySQL` 与 `Perl` 相结合，可以为你的网站集成一些非常强大的搜索功能。在本书后面的章节中，我们将进行的项目就包含一个搜索功能，它能根据用户在搜索框中输入的一个词或短语来查找用户和帖子。

## 用户名和密码注意事项

判断你是否在使用数据库的一个明确且简单的方法是，你是否必须使用用户名和密码登录。虽然存在一种不使用数据库而使用用户名和密码的方法，但这种方法并不涉及真正的数据库。

在提供用户名和密码时，脚本会告诉数据库首先查找用户名。一旦找到用户名，就检索密码。检索后，如果存档的密码与提供的密码相符，用户就成功登录了。从那时起，数据库最常用于向用户呈现页面，而页面内容大多基于该用户个人信息的某个方面。

## 使用 `.htaccess` 文件保护目录

不使用传统数据库而使用用户名和密码的唯一方法是使用一种较旧的用户识别方法。这种旧方法就是使用 `.htaccess` 文件。该文件内包含一个名称/值对，其中存储了用户名和密码。`.htaccess` 文件只是一个包含命令和数据的文本文件。

一旦用户名和密码通过验证，服务器上 `.htaccess` 文件所在目录的直接下级目录就可以被该特定用户访问。此检查由 `Apache` Web 服务器完成，而非 `Perl` 或数据库。

虽然这在某些情况下效果很好，但你最多只能使用约 100 个用户。达到 100 个用户的限制后，验证用户名和密码将耗费过多时间。这是因为必须将提供的用户名和密码与简单 `.htaccess` 文件中的 100 个不同的名称/值对逐一进行比较验证。

虽然简单用户验证操作是 `.htaccess` 文件的基本功能，但应该知道 `Apache` Web 服务器允许你在其中包含特殊的命令，以限制或更改该 `.htaccess` 文件所涵盖的每个页面的条件。

如果你是一名网络管理员，在安装和配置 `Apache` Web 服务器时，你需要在一个名为 `httpd.conf` 的配置文件中进行一系列更改。该文件包含了 `Apache` Web 服务器成功提供网页所需的所有信息。大多数这些命令，例如用于允许用户访问的命令，都可以放入 `.htaccess` 文件中。这个便捷的功能正是 `.htaccess` 文件至今仍在互联网上使用的原因。

## 何时何地使用数据库

如果你有向用户展示某个特定网页的需求，并且该网页的主要看点是可变且流动的内容——例如一个根据每个用户而异的用户个人资料页面——那么你通常会使用数据库。

使用数据库让你能够灵活地向用户呈现动态内容。所谓动态，指的是通常根据用户操作而变化的数据。生成当前用户正在查看的数据的用户可能不是当前用户——由于一个网站上可能有数千甚至数百万用户，查看不同用户生成的数据是很常见的。

如果你看到需要采用诸如显示单个用户信息、搜索功能等特性，或者需要大量使用 `HTML` 表单，那么你就应该使用数据库。

在互联网上使用数据库时，你将能够轻松地看出你访问的每个网站中数据库的使用位置和方式。你会了解到数据库驱动网站所能实现的灵活性和强大功能。通过使用像 `HTML` 这样的简单标记代码，你将看到数据库可以扩展可能的创作范围。

## Web 托管基础知识

假设你有了一个网站的想法，并希望将其变为现实。你首先需要考虑的事情之一就是选择哪家网络托管公司。

简而言之，网络托管有三种选择——共享计划、`VPS`（虚拟专用服务器）计划和独立服务器计划——每种都有其自身的优缺点。在本章剩余部分，我们将逐一探讨。



## 共享服务器

共享服务器计划正如你所想——你与许多不同的网站共享你的服务器。这个计划非常适合在网站开发的初期阶段使用。你通常会从共享计划开始，将你的网站开发成一个完整的产品。这是最便宜的选项，同时仍能提供良好的功能，但如果你计划拥有大量用户，这通常不是长久之计。需要注意的是，你的共享服务器有两种寻址方式：基于名称或基于 IP。基于名称的服务器地址是我们人类通常理解的方式，例如 [`www.domain.com`](http://www.domain.com)。另一种命名约定，基于 IP，是一种使用数字“八位组”的寻址数字形式。IP 代表 Internet Protocol（互联网协议），即四组由三个数字组成、用句点分隔的数字：

`127.34.56.124`

如前例所示的 IP 地址是一个有效的 IP 地址。它包含四个用点分隔的数字。这些数字是互联网上的基础寻址系统。为了将这些数字与我们人类能理解的单词匹配起来，我们需要使用 DNS 服务器。DNS 代表域名服务器。

共享托管计划通常具备你将网站从无到有发展成功能齐全的互联网产品所需的所有功能。然而，在你签订合同并为服务器上的空间付费之前，有几件事你必须了解。

网络托管公司在使用共享服务器计划时，只允许你的网站使用一定数量的资源。资源包括磁盘空间、处理器使用百分比以及通常在 30 天内测量的带宽使用限制。

有些计划限制了你能够创建和使用的 MySQL 数据库数量。你应该询问你的网络托管商，你的 MySQL 数据库是在你正在使用的同一台服务器上，还是通过本地网络上专门运行 MySQL 数据库的服务器访问。这一点很重要，因为为了使用数据库，你必须知道在数据库位于专用服务器上时使用网址作为位置，或者在数据库与网站其余部分存储在同一台服务器上时使用“localhost”。这种寻址方式将反映在每个使用数据库的 Perl 脚本中，因此了解这一点很重要。

由于本书的主要脚本语言是 Perl，询问网络托管公司是否允许你安装更多不属于标准 Perl 发行版的 Perl 模块（例如 `DBI.pm`）非常重要。在本书的后续章节中，你还需要使用 `size.pm` 和 `resize.pm`。`size.pm` 和 `resize.pm` 是用于返回图像大小 (`size.pm`) 和调整图像大小 (`resize.pm`) 的 Perl 模块。关于这两个 Perl 模块的更多内容将在后面的章节中介绍。

你还应该询问你的网络托管商 Perl 可执行文件的位置。这条信息是每个 Perl 脚本最顶部的一行代码，被称为“shebang”。如果脚本找不到 Perl 可执行文件，它将无法运行。Perl 最常见的位置如下：

```
#!/usr/bin/perl
```

我遇到过一些网络托管公司，它们将每个最新版本的 Perl 都列为 shebang，让你可以选择使用哪个版本。有些公司使用不寻常的 shebang。请查看网络托管商网站上的常见问题解答或支持部分以获取 Perl 的位置。

我曾遇到过一家不提供 Perl 作为脚本语言的网络托管公司。他们的借口是 Perl 不可扩展，这与事实相去甚远。Perl 是无限可扩展的——你只需要知道如何做到。通过使用子域名和文件，你可以将一个极其复杂的网站分散到整个服务器阵列上。

## VPS 服务器

虚拟专用服务器计划在服务器能力和可定制性方面比共享计划更进一步。VPS 计划保证了处理器使用百分比、内存（RAM）和硬盘空间。它在不产生专用服务器较高成本的情况下，高度模拟了专用服务器。大多数计划在每个 30 天周期内分配的带宽也比共享服务器计划多。

VPS 计划模拟了专用服务器的功能。它们比共享计划更昂贵，但额外的费用通常是值得的。当你的网站处于最终测试阶段时，可以使用 VPS 服务器。再次提醒，请注意你的 VPS 服务器上允许和禁止的内容。VPS 服务器是那些需要专用服务器功能但又不想承担额外成本的低流量、低带宽网站的良好解决方案。

## 专用服务器

专用计划是三种常见计划类型中最昂贵的，也是功能最丰富的，因为通常只有一个网站位于单台服务器上。一些网络托管公司允许你在每台专用服务器上拥有多个网站。

你通常会在网站完成并开始有用户访问后使用专用服务器。它们是网络托管公司提供的最昂贵的计划，因为只有你的网站使用该服务器。当你的流量开始导致服务器运行缓慢，或者你在共享或 VPS 服务器上的硬盘空间不足时，可以使用专用服务器。

专用服务器通常以功能和性能的滑动比例提供。如果你打算租用专用服务器，请确保它具有硬盘冗余功能，例如 `RAID 1`。`RAID` 代表独立磁盘冗余阵列。

最常见的两种 `RAID` 类型是 `RAID 1` 和 `RAID 5`。`RAID 1` 涉及两个相互镜像数据的独立硬盘驱动器。只涉及两个硬盘驱动器，其理念是如果一个硬盘驱动器发生故障，数据在另一个镜像磁盘上仍然可用。这是最常见的 `RAID` 形式。

`RAID 5` 被称为磁盘条带化。最多可以有七个磁盘参与存储你的数据。如果一个磁盘丢失，其余正常工作的硬盘驱动器上仍保留足够的信息来重建已故障并被更换的硬盘驱动器上的数据。

## 电子邮件选项

你应该向网络托管商咨询的另一项基本服务是 Linux 的 `sendmail` 功能。虽然大多数人都熟悉 Windows，但它在互联网上很少用作服务器。这就是 Linux 的用武之地。Linux 是用于提供你每天看到的网页的主流服务器。

为了通过脚本发送电子邮件，你需要使用 `sendmail`。`sendmail` 是一个非常基础的电子邮件服务器，已使用了几十年。通过它，你可以根据脚本操作动态发送电子邮件。

你还需要询问关于在线电子邮件客户端的情况。有许多电子邮件客户端可供选择，它们的功能都大致相同。然而，你要找的是一个稳定且经过时间考验的客户端。最常见的电子邮件客户端是 Horde，它运行良好。它是可用的最古老、开发最完善的在线电子邮件客户端之一。这是一项成熟的技术。通过它，你将能够从任何具有互联网服务的计算机在线接收和发送电子邮件。你收到的电子邮件存储在服务器上，因此只要你记住了服务器的地址，就可以从任何地方访问。



## 安全交易

一个需要询问的重要事项是，您是否能在收集支付信息或敏感信息的网站部分使用 SSL（安全套接层）。许多网络托管公司仅允许在独立服务器或`VPS`服务器上启用 SSL。

您可以通过查看浏览器的地址栏来判断一个页面是否受到 SSL 保护。地址栏开头不是`http://`，协议应为`https://`。这表明正在使用数字证书，您的数据处于安全保护之下，免受窥探。有些网站并不将 SSL 的使用局限于特定部分——他们使用 SSL 来保护整个网站的数据。虽然这样做没问题，但通常没有必要。

## Plesk 还是 cPanel？

大多数网络托管公司会提供两种不同的网络应用程序来管理您的服务器，即`Plesk`或`cPanel`。它们的功能大致相同，只有少许差异。它们看起来略有不同，但实现的目标一致。选择使用哪种服务器管理应用程序取决于您。有些人更喜欢`Plesk`，有些人则偏好`cPanel`。使用哪种应用程序完全由您决定，因为大多数网络托管公司都会提供这两种最常见服务器管理器的选择。

## 硬盘空间和数据传输限制

另一个值得关注的问题是分配给您的硬盘空间量。`MySQL`数据库会占用大量空间，因此您应该选择至少有 5GB 可用硬盘空间的共享主机方案。

如果您计划上传文件，请确保有足够的空间来维持网站运行。如果您使用的是共享主机方案，请选择提供硬盘空间最多的方案。当您使用的空间达到租用空间的三分之二时，就该考虑升级您的主机解决方案了。您的服务器管理器（`Plesk`或`cPanel`）会以易于理解的方式告诉您已使用的空间量和剩余空间量。

大多数网络托管公司对您在 30 天内可使用的数据传输量有限制。请务必询问您账户的带宽限制，以及如果您的数据传输需求超出带宽配额需要支付的额外费用。

## 转售商

大多数网络托管公司允许通过第三方转售其服务。这些转售商通常只提供非常低价的共享网络托管服务，例如每月 3.99 美元。他们纯粹是共享服务提供商，通常不提供独立服务器或`VPS`方案，因此您的升级选择有限。

虽然这些转售商在您网站生命周期的初期阶段尚可使用，但如果您已经收集了用户数据并不得不迁移到独立服务器或`VPS`服务器，他们可能会带来一些问题。您需要找到一种方法，将用户数据从转售商的共享服务器迁移到您主要网络托管公司的独立服务器或`VPS`服务器。

使用非主要服务提供商的网络托管公司，有时也会导致对共享服务器进行某些更改时出现漫长的延迟。您的技术支持请求需要先经过转售商，然后转到主要托管公司寻求解决方案，之后再返回给转售商，转售商再将请求结果告知您。这个过程可能耗时很长，导致您不得不坐着等待结果。以上就是关于共享服务器的内容；接下来让我们继续探讨`VPS`服务器。

## 安装 Perl 模块

在互联网上使用 Perl 时，您最终会需要安装更多的 Perl 模块来完成某些任务。如前一节所述，您可以使用`Plesk`或`cPanel`来完成此操作。两种方法都适用，并分别使用`CPAN.bat`或`PPM`。深入探讨`CPAN.bat`和`PPM`超出了本书的范围。

## 文件传输协议（FTP）客户端

FTP 客户端是允许您使用 FTP 远程访问服务器上文件的程序。借助它们，您可以在服务器上移动文件、创建目录，并在创建和调试时更新您的 cgi 脚本。

有许多不同的 FTP 客户端可供选择。到目前为止，我发现最好的是一款名为`FileZilla`的免费软件应用程序。它有一个连接管理器，允许您保存要连接的不同网站列表，并提供将您的网站打造成一个完整互联网产品所需的所有功能。

### 文件权限

您服务器上的每个文件都分配有一组权限。有些是自动分配的，有些则需要您手动分配。Perl cgi 脚本一旦上传，会缺乏用户通过浏览器运行它们所需的权限。

要分配适当的权限，请查看您的 FTP 客户端。应该有一种简单的方法来更改相关文件或文件的权限。

Linux 的`chmod`命令用于更改文件的权限。在`FileZilla`中，只需选中您需要更改权限的文件或文件。右键单击选中的文件或文件，然后选择`权限`。将打开一个包含九个复选选项的对话框。还有一个字段，您可以根据 Linux 的权限体系直接输入正确的权限代码。要让用户在您的服务器上运行 cgi 脚本，请在此字段中输入`755`（不带引号）。您会看到对话框会自动更新相应的字段。点击`确定`，您就更改了文件的权限，允许用户通过其浏览器在您的`cgi-bin`目录中执行脚本。

## 使用 printenv.pl 获取您的路径语句

关于服务器设置，您首先需要了解的事情之一就是路径语句。路径语句中包含了从您账户的根目录到您将使用的工作目录（例如将存放您所有网站 cgi 脚本的`cgi-bin`目录）的目录层级结构。

要获取您的路径语句，您需要运行一个虽小但功能强大的脚本，名为`printenv.pl`。只需上传该文件并将其权限调整为`chmod 755`。在浏览器中运行该脚本，您就能找到路径语句。以下是这个小脚本中使用的代码：

```
print "Content-type: text/plain; charset=iso-8859-1\n\n";
foreach $var (sort(keys(%ENV))) {
$val = $ENV{$var};
$val =~ s|\n|\\n|g;
$val =~ s|"|\\"|g;
print "${var}=\"${val}\"\n";
}
```

这段小代码告诉您服务器环境中的所有信息，包括许多资源的路径语句，这些资源在本书后续章节中会派上用场。

服务器上的路径语句是指向可直接从操作系统或相关服务器访问的资源的指针。例如，在 Windows 中设置的路径语句可能会影响 Perl 的 shebang（解释器路径）。如果您将 Apache 安装到`c:\Apache`，您 Perl 脚本中的 shebang 将如下所示：

```
C:\Apache\bin\perl
```

如您所见，这个位置允许服务器找到`perl`，但您可能需要给出`perl.exe`可执行文件的完整路径，如下所示：

```
C:\Apache\bin\Perl\perl.exe
```

提供`perl.exe`的完整路径是一个安全隐患。仅当您不想从控制面板设置操作系统的路径语句时才使用它。
