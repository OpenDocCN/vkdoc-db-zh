## 前言

你即将阅读的这本书，是关于使用 `SQLTXPLAIN`（也可简称为 `SQLT`）进行 SQL 调优的主题。你可能对这个 `SQLT` 工具的功能略知一二，但对其背后的故事却知之甚少。这篇前言提供了一些背景信息，讲述了这个工具是如何演变成今天的样子。希望你能喜欢这幕后一瞥，了解多年来我研究你即将学习的这个工具时，那些逐渐汇聚起来的动机和事件。

在 1990 年代末，我和当时的许多顾问一样，是个“空中飞人”。我是一名甲骨文“现场支持工程师”，在制造和 SQL 调优方面有相当不错的专业知识，并且选择加入了短期任务团队。（那时我们有两个选择：短期或长期任务！）。我过去常常为了这些任务在各个站点之间频繁出差。

对于像我这样愿意到现场处理影响大型制造企业的 Oracle ERP“大火”的人来说，制造和 SQL 调优是一个很好的组合。生活很紧张，但也很棒！经过几周的任务后，我开始注意到一些模式：当试图诊断性能不佳的 SQL 语句时，需要检查的方面很多，而 `trace/tkprof` 并不能给我提供快速有效诊断 SQL 调优问题所需的一切信息！通过我的任务，我开发了一套灵活的脚本。这就是传奇的 `coe_xplain.sql` 的诞生过程。

当时的经理卡尔·达达伊支持我的努力，实际上还鼓励我建立自己的工具集，以便在一周的时间内交付结果。于是，我几乎在每一个性能任务中都使用 `coe_xplain.sql`，而且总是能得到很好的结果。随着时间的推移，我很高兴自己开发了这个小工具，因为我对 `tkprof` 的大多数改进建议至今仍悬而未决。

有一天，正当我愉快地使用我的个人脚本 `coe_xplain.sql` 以及我开发的其他一些脚本时，当时合作的一家公司里一位才华横溢的 DBA 问我，他是否可以保留我的工具集以备后用。我对于留下我的“玩具”的担忧源于经验：我知道，你编写的任何代码都会伴随你一生，这既是奖励也是诅咒。如果你的代码只有几个小错误，那还好。否则，你会感觉被困扰和追捕几十年！

尽管如此，我还是决定分享我的 `coe_xplain.sql` 和其他小工具，并附带说明用户需“按原样”接受它们。一年后，同一位 DBA 问我是否可以在 `Metalink`（现在称为 `MyOracle Support`）上发布我的 `coe_xplain.sql` 脚本，这样如果我决定继续更新，他就能持续获取新版本。这对我来说是一个转折点。在 2001 年将一个工具发布在 `Metalink` 上，对我来说只意味着一件事：我愿意做出妥协，尽可能让这个工具免于错误，并且我必须对功能增强请求做出回应，即使这意味着要拒绝所有的请求。我当时就知道（正如我现在所知），当我说“可以”而且想说“可以”的时候，我总是很难说“不”。

因此，在 `coe_xplain.sql` 发布在 `Metalink` 上后，我很快就开始收到一些类似这样的问题：“你能在这个原本不错的脚本里加上这个小小的功能吗？”

2002 年底，`coe_xplain.sql` 脚本已经变得很大，我决定是时候真正将其升级为 PL/SQL 了。这意味着要用一个 PL/SQL 包重写所有功能，而不是将几个 SQL 语句嵌入一个大型脚本中。那时我仍然是一个名为“卓越中心”团队的一员。这就是为什么 `coe_xplain.sql` 有“coe_”这个前缀。如今，甲骨文内部有许多团队共享“CoE”这个名字，因此我觉得它的意义有些被稀释了。（类似的情况也发生在“BDE”团队，意思是“Bug 诊断与升级”。这就是为什么我的一些脚本过去和现在仍有“bde_”前缀的原因。）

我决定是时候将 `coe_xplain.sql` 升级为更健壮的东西了。我不再想以我工作的团队名字来命名我的脚本。于是在 2002 年，在第二个主要版本中，这个 `coe_xplain.sql` 工具变成了 `SQLTXPLAIN`，我将其发布在 `Metalink`（`MyOracle Support`）的注释 `215187.1` 下，这至今仍是它的位置。`SQLTXPLAIN` 这个名字大致来源于“SQL Tuning and Explain Plan”。我曾在网上搜索过，没有找到任何关于 `SQLTXPLAIN` 这个名字的参考。我试图避免与其他工具名、产品或公司名称冲突，截至目前，至少在这个命名方面，我成功了！

`SQLTXPLAIN` 很快被甲骨文支持部门内部的许多 Oracle ERP 团队采用，并逐渐扩展到甲骨文开发部门。那时我收到的大多数 `SQLT` 增强请求都是针对电子商务套件（`EBS`）的，因此 `SQLTXPLAIN` 主要变成了一个诊断 `EBS` 内性能不佳的 SQL 语句的工具。从 2002 年一直到 2006 年，这个工具都安装在应用程序模式内，即 `EBS` 的 `APPS` 模式中。因此，`SQLTXPLAIN` 曾经有很强的 `EBS` 依赖关系。它创建在应用程序模式内，在那里创建和删除对象。但别担心！它现在已经不这么工作了。

2002 年到 2006 年那几年，我在甲骨文的日子很艰难。我忙于开发内部应用程序，同时还在攻读计算机科学硕士学位，因此我面临着双重挑战。更不用说我和妻子还要抚养四个青少年了！因此，`SQLTXPLAIN` 在 2007 年之前实际上被冻结了。

我之前没提，现在说一下：`SQLTXPLAIN` 从来不是甲骨文的官方项目。直到 2007 年，`SQLTXPLAIN` 都是我周末的宠物项目（一个很大的项目！）。我几乎所有个人的空闲时间都用于开发和维护这个工具。它是我的宝贝，至今仍是。我非常幸运，我的妻子凯莉·桑塔纳也在甲骨文工作，在我忙碌的这些年里她给予了我极大的支持。如果没有她的耐心和理解，我可能早就不得不放弃 `SQLTXPLAIN`，并将其从 `Metalink` 上下架了！

2006 年底，我调到了甲骨文内部的服务器技术卓越中心（`ST CoE`）。我之前在`应用卓越中心`，现在进入了`ST CoE`。那是“另一边”，有人称之为“黑暗面”，或者干脆就是“另一边”。我带去了我所有的 `EBS` 专业知识和我所有的工具。史蒂夫·弗兰奇，我在甲骨文遇到过的最好的经理之一，批准我继续维护我的工具，因此我可以在正常工作时间内处理 `SQLTXPLAIN`。这对我来说是一个巨大的激励，因此我在甲骨文的新家里非常开心。在那之前，我的大部分脚本实际上都是在夜间和周末工作的。当你的爱好和你的工作如此顺畅地融为一体时，问题就来了：而且，你还有在家工作的特权。所以今天我会说我几乎全天候、每周七天都在照看 `SQLTXPLAIN`。但当然，我仍然要睡觉和吃饭！
