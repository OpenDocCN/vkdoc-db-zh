## 运行运行状况检查

SQL 运行状况检查脚本不是 SQLT。它是一个完全独立的实用程序，从不同的 Metalink 注释中下载，并以不同的方式使用。那么为什么它会在本书中被提及？当无法安装 SQLT 时，SQLHC 被认为是一个备用选项。这种情况并不经常发生，但一旦发生，通常是由于站点限制。这有很多原因：

*   不允许在生产环境安装任何东西（安全原因）。
*   不允许在生产环境安装任何东西（担心影响生产）。
*   安装和测试过程耗时太长，而你遇到了生产问题。
*   不信任 SQLT。
*   SQLT 太复杂。
*   没有足够的空间安装 SQLT。

首先我要说明的是，SQLT 占用空间很小，并且已经在许多不同的系统上经过测试。商业原因可能妨碍及时安装新软件包：例如，如果必须在开发和 QA 环境中进行测试，然后才能到达生产环境。如果你有这些规定，你应该遵守这些业务规则或本地 IT 标准，并考虑改用运行状况检查脚本。然而，我希望我已经展示了 SQLT 是一个值得信赖的工具，能为用户带来诸多好处。如果你遇到禁止使用 SQLT 的情况，你仍然可以选择次优方案，即对目标 SQL 运行 SQL 运行状况检查脚本。

## 什么是 SQL 运行状况检查？

SQL 运行状况检查 (SQLHC) 脚本可从 Oracle Metalink 注释 1366133.1 免费下载。它是一个包含三个简单 SQL 脚本 (`sqldx.sql`、`sqlhc.sql` 和 `sqlhcxec.sql`) 的 zip 文件，你可以在目标系统上以 sys 身份运行。直到最近，`sqlhc.sql` 还是唯一的脚本，因此我们将主要讨论该脚本，并在本章后面介绍其他脚本。所有这些脚本都不需要安装：它们只是 SQL 脚本，使用起来非常简单。这些脚本是为满足某些站点禁止安装 SQLT（它有一个模式、模式对象、表和索引、包和函数）的要求而开发的。由于像 `sqlhc.sql` 这样的脚本只是一个简单的脚本，更多的站点会接受它进行安装，因此更多的站点可以利用 SQLHC 提供的建议。SQLHC 是从 SQLT 发展而来的（它提供了 SQLT 功能的一个子集），因此关于 SQLHC 能做什么的讨论涵盖了本书已经涉及的许多主题。

以下是 SQL 运行状况检查报告涵盖的部分内容：


## sqlhc.sql 脚本

*   一份观测报告，类似于 SQLT，但由于对数据库的访问权限有限而受到限制。
*   SQL 文本。
*   表信息摘要，包括行数和统计采样大小。
*   每个表的列数和直方图。
*   每个表的所有索引信息摘要。
*   该 SQL 所使用的 SQL 计划基线和概要文件的描述。
*   游标共享的状态及共享的原因。
*   带有关联统计信息的 SQL 计划历史。
*   执行计划历史。
*   活动会话历史信息。
*   列统计信息和直方图类型。
*   系统参数及其值，以及默认或非默认性质。
*   所有执行计划的详细描述。
*   该 SQL 的 SQL 监视器报告。

上述功能由 SQLHC 的主脚本 `sqlhc.sql` 生成。实际上，早期版本的 SQLHC 仅包含这一个脚本。后来的版本还包含了 `sqldx.sql` 和 `sqlhcxec.sql`。`sqldx` 例程是一个出色的工具，用于生成不依赖于 SQLT 的测试用例。我们将在“`sqldx.sql` 脚本”一节中详细讨论。我们也将介绍 `sqlhcxec`。现在让我们来看看主脚本 `sqlhc.sql`。任何调优工作的第一步都应该是查找明显错误的地方。这被称为检查清单：如果缺少一个明显的索引，为何还要进行漫长的调优过程呢？

在数据库或任何特定 SQL 上可能出错的事情清单非常长。健康检查脚本和 SQLT 内置了这些错误检查。SQLHC 执行大约一百项检查（SQLT 的检查数量是其两到三倍）。通过这些内置检查，你可以在开始任何调优工作之前发现并修复这些问题。在 SQLT 中，这些错误会在主报告的“观测”部分指出（参见第 1 章）。在 SQLHC 中，报告的第一页 HTML 文件中也有一个观测部分。如果你的 SQL 中只有少数几个在投入生产前因为一些“明显”错误被捕捉到，那么下载并使用 SQL 健康检查脚本绝对是值得你花时间的。

## sqlhc.sql 产生什么

`sqlhc.sql` 脚本是一个单独的 SQL 脚本，它生成一个 zip 文件，其中包含四个 HTML 文件、三个 zip 文件和一个跟踪文件。如果你对引入系统的每条新 SQL 只做一件事——运行 `sqlhc.sql`，那么健康检查脚本的努力就是值得的。这些 HTML 文件本身就是报告，非常值得仔细阅读。`sqlhc.sql` 脚本易于使用，仅需两个参数（我们将在本章后面的“运行 SQLHC 脚本”一节中看一个运行示例）。

请记住，这些文件可以按任何顺序查看，且彼此之间不相互依赖。如果你有一个特定的区域想要检查，可以直接查看相应的文件，而无需查看中间的文件。然而，如果你想彻底检查所有内容，主要 HTML 文件已为你方便地编号。示例文件名如下所示：

*   `sqlhcxec_20130103_205245_81s67vj4pjqm8_1_health_check.html`
*   `sqlhcxec_20130103_205245_81s67vj4pjqm8_2_diagnostics.html`
*   `sqlhcxec_20130103_205245_81s67vj4pjqm8_3_execution_plans.html`
*   `sqlhcxec_20130103_205245_81s67vj4pjqm8_4_sql_detail.html`

我们将在下一节查看 `sqlhc.sql` 生成的所有文件。

本节描述主要报告文件。提到的前四个文件是 `sqlhc.sql` 创建的 HTML 页面。它们都包含 SQL 文本，因此可以确保你查看的是正确的文件，并按从 1 到 4 的顺序查看。

*   第一个 HTML 报告，即主健康检查脚本，为你提供了观测和 SQL 通用检查的良好概述。它涵盖观测以及表和索引摘要。应仔细审查观测部分以查找任何问题。
*   第二个 HTML 报告更详细，涵盖了许多杂项领域，但最重要的是它涵盖了当前统计信息（在“当前 SQL 统计信息”部分）以及实例和系统参数。它还涵盖了游标共享信息和执行计划摘要。历史信息也包括在内，涉及表、列、索引、实例和系统参数。此报告还涵盖了 SQL 计划基线和概要文件，以及 11g 的新特性 SQL_Patches。
*   如果你认为执行计划存在问题或执行计划意外更改，第三个报告很有用。它显示了当前和历史的执行计划。
*   第四个报告将为你提供一些计时信息，并显示你应该集中调优努力的地方。

其他文件包括：

*   一个包含每个执行计划的 SQL 监视器执行详细信息的 zip 文件。
*   用于调试优化器问题的 10046 和 10053 跟踪文件。
*   一个包含 SQL 执行结果的文本文件。这是为了检查我们运行的是否是正确的 SQL，以及结果是否符合预期。
*   一个包含运行日志和一些辅助 SQL 的 zip 文件。
*   最后，一个 SQLDX zip 文件，其中包含 CSV 格式的所有已发现信息。这将允许使用电子表格进行进一步的后续处理。

这里有许多有用的文件，但此输出最有用的部分是四个 HTML 文件和 SQL 监视器输出。在运行一个示例脚本后，我将展示一些示例。你可以看到，虽然我们没有运行 SQLT，但我们仍然有大量的信息可供使用。

## 运行 SQLHC 脚本

在我开始示例之前，我已经登录到 Metalink（或 My Oracle Support）并下载了构成 SQLHC 脚本文件的 zip 文件。该文件名为 `sqlhc.zip`。它包含 `sqldx.sql`、`sqlhc.sql` 和 `sqlhcxec.sql`。这个简单的 zip 文件就是获取下面示例中列出的信息所需的全部。考虑到无尽的 HTML 报告、历史信息收集和执行计划报告，你为什么不想对每个部署的 SQL 运行此脚本作为检查呢？
