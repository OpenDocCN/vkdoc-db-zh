# Oracle 数据卫士与 SQLT 工具概述

数据卫士是 Oracle 公司开发的一项技术，旨在应对"如果……会怎样"的场景。如果我的数据中心因洪水完全被摧毁会怎样？如果存放我数据库的建筑物被火灾烧毁会怎样？传统上，这个问题通过及时的备份来解决：备份被运送到场外，存储在安全地点，然后可以在约定的时间范围内访问，并在另一个站点（推测不在灾区）恢复到备用的硬件上。这种策略存在许多困难：执行备份、将备份运输到安全地点、从安全地点获取正确的备份、使用备份和恢复程序重建系统，最后为需要使用系统的员工提供访问权限。最后也是最重要的，你需要定期测试这些程序，否则当灾难来临时，你可能拥有所有数据和设备，却不知道如何将它们组合成一个可工作的系统。当然，这个恢复过程可能耗费相当长的时间。你肯定已经测试过你的恢复程序了吧？我曾合作过的一个站点对此非常重视，在测试当天，他们会四处走动，给人和设备贴上红色贴纸，并告诉这些人他们不能参加此次测试（推测是在"外星人绑架你的数据中心"的场景中被外星人绑架了）。测试这些场景很困难；不足为奇的是，许多站点没有做足够的测试，只是交叉手指并祈求好运。

数据卫士使保护数据站点变得简单得多。前面提到的所有复杂情况都有可能被消除。使数据卫士物理备用数据库工作的技术在概念上非常简单。跟踪数据库中每个更改的归档日志（或重做日志）通过源系统上的特殊进程传输到备用系统上的特殊进程，在那里这些更改被逐块应用到源数据库的副本上。这个称为传播的过程可以在备用数据库上应用数据。数据可以与主数据库同步应用，也可以落后于主数据库一个预设的量。你甚至可以拥有多个备用数据库，全部从主数据库收集更改。借助数据卫士活动备用数据库，你甚至可以让故障转移（即主数据库发生故障，备用数据库必须接管的过程）以最小的混乱发生。

数据卫士是灾难恢复技术的一次巨大飞跃。完整的覆盖范围远远超出了本书的范围；但是，对该工具有一个基本的、高层次的了解是有用的。

可以说，Oracle 数据卫士物理备用数据库允许数据中心一直在线。没有停机时间。当然，你仍然需要备份来处理那些讨厌的系统性错误（你删除了错误的数据）。数据卫士使你的测试场景简单得多。现在你只需要执行一次切换（一个受控的计算机角色切换，与故障转移相对），一旦切换完成，你就可以立即开始测试。

在 11g 第 1 版中，Oracle 改变了数据卫士，使得物理备用数据库不仅可以用于故障转移或切换操作，还可以用于只读操作。通常，这是为了允许在备用数据库上运行报告。许多站点发现这对他们的运营是一个巨大的好处。现在，他们可以将昂贵的报告操作（就资源使用而言）转移到备用数据库上，并为主数据库腾出资源用于在线事务处理（OLTP）。

## SQLTXTRSBY

一旦报告操作转移到物理备用（和逻辑备用）数据库上，我们可能再次遇到性能不佳的 SQL。在读/写数据库上出现的性能问题正是 SQLT 可以帮助我们解决的那类问题。唯一的问题是，`SQLTXTRACT` 和 `SQLTXECUTE` 需要数据库的读/写访问权限（以在 SQLT 存储库中存储数据以及安装包和过程）。如果我们甚至无法将性能数据存储在出现性能问题的数据库上，SQLT 又如何能帮助我们呢？这就是 `SQLTXTRSBY` 发挥作用的地方。

它提供了一些特殊的过程来处理这些特殊情况。随着数据卫士（和其他只读数据库）变得越来越流行，上述性能问题在数据卫士实例上出现得越来越多。为了处理这些问题，`SQLTXTRACT` 被调整为无需本地存储即可工作。这是通过创建特殊的例程实现的，这些例程从一个读/写数据库运行，并通过数据库链接延伸到只读数据库，收集所需信息，整理后呈现在读写数据库上。让我们更详细地看看 `SQLTXTRACT` 的一些限制，然后讨论使用 `SQLTXTRSBY`。

## SQLTXTRACT 限制

我总是把 `SQLTXTRACT`（别名 `XTRACT`）和 `SQLTXTRSBY`（别名 `XTRSBY`）想象成一个超级英雄和他（或她）的搭档。超级英雄 `XTRACT` 似乎无所不能，但每个超级英雄都有他们的阿喀琉斯之踵，在这里就是无法在只读数据库上工作。`XTRSBY` 拥有主英雄所不具备的特殊技能。在这种情况下，`XTRSBY` 可以进入只读的数据卫士系统并提取信息，而 `XTRACT` 则不能。物理备用数据库不能容忍写操作。不能有两个系统 potentially 更新相同的数据（当然，除非你有多主复制，但那是另一本书的内容了），除非主系统有办法知道故障转移系统在做什么，而数据卫士不允许这样做。这就是为什么物理备用数据库以只读模式打开。如果你碰巧进入一个只读数据库并尝试插入一些数据，你会得到如下错误消息：

```
SQL> insert into test1 values (1);
insert into test1 values (1)
            *
ERROR at line 1:
ORA-00604: error occurred at recursive SQL level 1
ORA-16000: database open for read-only access
```

我们在上面的例子中看到，向只读数据库插入数据是不可能的。错误消息甚至 somewhat 说明了发生了什么。`XTRACT` 需要保存一些数据来进行分析，因此这一步（在本地数据库中保存数据）对 `XTRACT` 行不通。`XTRSBY` 通过将所需数据通过数据库链接拉取到一个读/写数据库并将其本地存储，而不是存储在只读数据库上，从而解决了这个问题。

`XTRACT` 还依赖于创建一个用户来安全地存储其对象。`XTRACT` 无法在只读数据库上执行此操作。

```
SQL> connect / as sysdba
Connected.
SQL> create user sqltxplain_read_only identified by orcle;
create user sqltxplain_read_only identified by orcle
            *
ERROR at line 1:
ORA-00604: error occurred at recursive SQL level 1
ORA-16000: database open for read-only access
```
