# APEX 5.0 简介

欢迎来到 Oracle Application Express (APEX) 的精彩世界。你即将学习如何使用一个能彻底革新你对构建基于 Oracle 的 Web 系统之思考和方式的工具。它确实为我带来了这样的改变。

在 APEX 出现之前，为驻留在 Oracle 数据库中的数据开发完全交互式、基于 Web 的系统，几乎总是意味着要学习一门新的、通常很复杂的语言，比如 Java、.NET 或 PHP，然后想办法将你选择的语言与数据无缝集成。这往往还意味着，需要整合那些已经以 `PL/SQL` 程序单元形式编码的业务规则。

在这种情况下，可能要花费数月甚至数年的时间，才能对你选择的语言足够熟练，从而开始编写一个功能性的系统。如果你像许多开发者一样，就会因为花了过多时间在看似相对简单的任务上而感到沮丧。

别担心！冗长复杂的 Web 开发平台的时代可能已离你而去。

## 什么是 APEX？

APEX 是一个 100% 基于浏览器的快速应用程序开发（RAD）工具，它能帮助你快速、以相对较少的编程工作量，创建丰富、交互式的、基于 Oracle 的 Web 应用程序。

市场上有许多 RAD 开发工具和平台。如果你处理的是驻留在 Oracle 数据库中的数据，有几点让 APEX 独树一帜，从而成为一个更具吸引力的开发平台。首先，最重要的是，APEX 构建并使用 `SQL` 和 `PL/SQL` 作为其核心语言。对于那些已经使用 Oracle 数据库的人来说，这是一个巨大的优势，因为这意味着你可以立即运用已有的知识。即使你没有 Oracle 背景，但若将要使用 Oracle 数据库，你需要学习其特定的 `SQL` 方言，并且很可能在某个时候需要使用 `PL/SQL` 过程式语言。

当从已经将大量业务逻辑编码到存储的 `PL/SQL` 程序单元中的、基于 Oracle 的系统进行迁移时，`PL/SQL` 程序单元会变得更加有益。在这种情况下，你几乎可以立即利用那些逻辑，只需付出很少的努力或对现有代码进行很少的改动。

另一个巨大的优势是，APEX 是一个声明式工具，它提供了功能丰富、旨在让你的工作更轻松的核心功能。由于 APEX 处理了所有基于 Web 应用程序通用的许多底层功能，你可以专注于你的应用程序特有的逻辑。

你所需完成的大部分工作，都可以使用作为 `APEX 应用程序构建器` 一部分提供的众多内置向导之一来完成。这些向导引导你完成定义应用程序功能的流程，然后将该信息存储为元数据。向导完成后，你可以编辑和增强其功能，甚至用你自定义的 `SQL` 和 `PL/SQL` 例程替换它。当你熟练掌握 APEX 后，你甚至可能会完全绕过向导，直接生成更复杂的定义。

在本书的学习过程中，你可能会发现自己需要掌握其他一些工具，但实际上，你完全可以仅使用你的 Web 浏览器和 APEX 所提供的功能，来开发一个非常丰富的应用程序。

## APEX 简史

APEX 已经存在了相当长的时间——可能比大多数人知道的还要久。APEX 的第一个公开版本（当时称为 `HTML DB`）发布于 2004 年，但其历史可以追溯到更早。

### 远古历史

APEX 的根源在于一项已存在相当长时间的技术。事实上，APEX 在底层用来生成发送到浏览器的 `HTML` 的 `PL/SQL Web Toolkit` 的部分组件，最早可以追溯到 1994 年。

在那个时期，你实际上可以手工用 `PL/SQL` 编写 Web 应用程序，不幸的是我们很多人确实这么做过。这不仅需要精通 `PL/SQL` 和 `HTML`，还需要圣人般的耐心和倔驴般的决心。最终的产物并不美观，而且以今天的标准来看绝对不安全，但它是可用的，尽管功能有限。

此后不久，Oracle 推出了 `PL/SQL` 服务器页面（PSPs）。这涉及先编写静态 `HTML` 并包含特殊的 Oracle 标记以指示动态数据的位置。当你将输出调整到所需样式后，通过一个名为 `LOADPSP` 的程序运行它。该程序将原始的 `HTML` 和特殊的 Oracle 标记转换成一个 `PL/SQL` 过程，该过程再次使用 `PL/SQL Web Toolkit` 来输出 `HTML`，包括你请求的动态数据。在当时，这是一个巨大的飞跃。我曾在一家公司工作，使用 `PSP` 技术构建了整个框架，并在多个客户那里部署了它。

最终，在 1997 年，`WebDB` 登场了。作为现在称为 APEX 的真正鼻祖，`WebDB` 是革命性的，因为它是一个 100% 基于 Web 的工具，允许开发者设计 Web 应用程序。尽管当时 Java 似乎正在接管世界，但它完全用 `PL/SQL` 编写。开发者可以将 `WebDB` 指向他们的数据库，并生成用于创建表单、报表、图表和日历的代码。它没有会话状态管理，也没有模板；一旦代码生成，你就无法通过该工具回溯修改。

`WebDB` 使得许多希望加入基于 Web 浪潮的公司得以实现目标，而无需花费大量时间和精力重新培训员工。作为其成功的一个证明，我知道有好几家公司目前仍在生产环境中运行着 `WebDB` 系统。

不幸的是，`WebDB` 的日子是有限的。因为它生成代码（如果你不喜欢它生成的代码，那也没办法），在它被 Oracle 的 `Portal` 产品吸收时，已经开始失宠。然而，它的创造者 Mike Hichwa 并没有忘记 `WebDB` 曾展现过的伟大曙光。


### 近期历史

大约在 1999 年，Oracle 公司首席执行官拉里·埃里森交给软件开发副总裁迈克·希瓦一项任务，为 Oracle 公司创建一个内部的日历和日程安排系统。最初的方案是使用 WebDB 生成初始代码，然后从此处开始手动编写所有更改。然而，迈克将此视为一个机会，可以将 WebDB 完全重写成更有用的东西。因此，在乔尔·卡尔曼和汤姆·凯特的帮助下，Oracle Flows 诞生了。

基于内部日历和日程安排系统的成功，该团队获准继续推进，将 Oracle Flows 打造成一个产品。2001 年，使用当时称为 Flow Builder 的工具，迈克和他的团队开始为各种客户实施系统，包括一个他们成功取代了一个严重偏离正轨的 Java 开发项目的情况。

到 2003 年，该团队已经证明了该工具的强大能力，并获准将其作为产品发布。HTML DB 1.5 作为 Oracle 10gR1 的一个免费选项向公众发布。

此后，陆续推出了多个版本，每个版本都提供了改进的功能和特性。以下是版本及其一些较显著功能的简要列表：

*   HTML DB 1.6（2004 年）引入了主题、主从表单、页面组、页面锁定以及一些多语言功能。
*   HTML DB 2.0（2005 年）引入了 SQL 工作坊、图形化查询构建器、数据库对象浏览器和会话状态保护。
*   APEX 2.2（2006 年）引入了打包应用程序、APEX 字典视图和访问控制向导。
*   APEX 3.0（2007 年）引入了与 BI Publisher 集成的 PDF 打印、从 Microsoft Access 迁移的功能，以及页面和区域缓存。
*   APEX 3.1（2008 年）引入了交互式报表、仅运行时安装能力以及改进的安全性。
*   APEX 3.2（2009 年）引入了用于 Oracle Forms 系统的迁移助手以及各种安全增强功能。
*   APEX 4.0（2010 年）是一次巨大的飞跃，引入了动态操作和插件——分别是声明式引入服务端逻辑和扩展核心 APEX 环境的方式。同时还引入了新的团队开发模块。
*   APEX 4.1（2011 年）包含了新的面向用户的数据上传功能、增强的错误处理能力以及对表格化表单大幅改进的支持。
*   APEX 4.2（2012 年）最初引入了一些新主题以及对调试 API 的增强，但在其两年多的生命周期中，补丁版本引入了诸如 HTML 5 图表和更深层次安全增强等变化。

### APEX 5.0 与未来

于是我们迎来了 APEX 5.0 的发布。虽然 4.0 至 4.2 版本引入的变化无疑改变了 APEX 开发的格局，但 5.0 版本引入的变化已将 APEX 提升到了一个可以正当地与许多流行的基于桌面的开发环境相媲美的高度。

APEX 5.0 最初的重点是通过为开发人员提供一个大幅增强的开发环境，使开发丰富的交互式 Web 应用程序变得更加容易。然而，开发团队引入了如此多的新功能——实际上是解决问题的新方法——以至于很难不选择 APEX 作为基于 Oracle 应用程序的首选开发平台。

APEX 新的**页面设计器集成开发环境**完全改变了开发人员与页面设计交互的方式。它仿照许多流行的桌面 IDE，让开发人员现在可以在一个页面上同时处理项目、布局、属性和操作。引入了新的拖放式页面布局界面，允许开发人员轻松定位区域和项目。分组编辑允许开发人员一次性编辑多个项目的属性。新的页面设计器唯一的缺点是，由于其布局特性，你可能会发现自己需要更多的屏幕空间。然而，随着宽屏显示器变得普及，这对大多数人来说应该不是问题。

除了新的页面设计器 IDE 之外，APEX 5.0 最令人兴奋的新功能之一是**通用主题**。这个新的应用程序用户界面摒弃了过去复杂的模板，使开发人员无需了解 HTML、CSS 或 APEX 模板设计的复杂细节，就能构建更现代、响应式和一致的应用程序。

新的通用主题允许你通过所谓的**主题样式**——一种添加到基础 CSS 上的层叠样式表来调整许多属性。这可以通过新的**主题滚轮**工具完成，让你能够直观地修改主题。通用主题还允许你通过使用**模板选项**轻松自定义页面上项目的显示方式。

经过相当长一段时间的规划，**灵活工作区认证**终于由 APEX 团队实现；这允许 APEX 管理员定义 APEX 本身将如何认证开发人员。与 APEX 应用程序类似，工作区现在可以针对单点登录服务器、LDAP 等进行认证。

**交互式报表**不再局限于每页一个，这使你摆脱了自其诞生以来就困扰它们的限制。交互式报表还获得了一些新功能。添加了一个**数据透视视图**，允许最终用户选择列并提供用于透视报表的函数。这是以前只能通过大量手动编码或创建/使用插件才能实现的功能。当使用新的通用主题时，报表列头现在可以定义为在用户向下滚动页面时保持固定位置。

引入了对**对话框页面类型**的原生支持，从而允许任何页面以正常方式显示或作为弹出对话框显示。页面可以定义为“模态”或“非模态”。模态页面不允许最终用户与底层页面交互，而非模态页面则允许此类交互。

引入了新的**jQuery Mobile 和平板主题**，并利用了最新 jQuery Mobile 库的较新功能。面板、弹出窗口和对话框（以及其他功能）现在都在移动界面中可用。

改进的图表引擎为大数据集提供了增强的性能。为视障人士添加了无障碍访问改进。添加了一个新的`APEX_AUTHORIZATION`程序包，以帮助管理应用程序内的授权。等等，不一而足。

### 入门所需

本书的目标是让你开始使用 APEX，以一种能够让你朝着精通该产品的目标成长的方式引导你。首先，你需要三样东西：对一个 APEX 实例的访问权限、一个网络浏览器以及一份 SQL Developer。
