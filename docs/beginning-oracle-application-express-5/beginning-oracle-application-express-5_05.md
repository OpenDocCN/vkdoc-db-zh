# 3. 识别问题与设计解决方案

每个计算机系统都是（或至少应该是）解决某种问题的结果。虽然“Hello World”应用程序很棒，但我坚信，学习任何技术的最佳方式是将其应用于一个真实的问题，看看它实际是如何工作的。

我在整本书中都遵循这一原则。本章将讨论大多数组织中一个非常普遍的、可以通过技术解决的问题。你还将探讨在设计基于 Web 的系统时，特别是使用`APEX`时，需要考虑的一些要素。

## 识别系统需求

几乎每家公司，无论规模大小，在某个时候都需要实施某种帮助台系统。无论是用于跟踪员工问题和疑问的内部系统，还是用于跟踪商业软件或硬件客户问题的外部系统，帮助台系统的基本原理都相当标准。

大多数帮助台系统都是由问题单或简称为“工单”的概念驱动的。这个术语源于计算机出现之前的日子：大多数问题是通过电话报告的，故障排查员使用纸质工单来记录通话。那张纸质工单上包含的信息有问题的描述、遇到问题的人的姓名、问题记录的时间等等。然后，在故障排查和（希望如此）解决问题的过程中，工程师们记录下每一步的操作，并包含他们在此过程中收集到的任何问题文档。如今，如果看到一个没有计算机化的帮助台系统，那将是非常令人惊讶的，即使它只是一个带有注释和状态的问题电子表格。

在本章中，你将使用`APEX`来构建帮助台系统。在深入之前，你需要清晰地理解你试图解决的问题。至少，你需要审视当前的系统。
