# 临时表

Oracle 提供了两种类型的临时表：全局临时表和私有临时表。全局临时表是 Oracle 长期支持的一项功能。它们是持久的数据库对象，存储在磁盘上，并且对所有会话可见。之所以称为临时表，是因为其中的数据仅在事务或会话期间存在。

从 Oracle 18c 开始，您可以创建私有临时表。与全局临时表不同，私有临时表仅存在于内存中，并且仅对创建它的会话可见。您可以将私有临时表定义为在事务或会话持续时间后存在，之后私有临时表会被自动删除。

**注意**

如果您使用过其他数据库技术（如 SQL Server 或 MySQL），私有临时表的行为更符合您在这些环境中的使用习惯。

在以下章节中，我将首先描述全局临时表的工作原理，然后再介绍更新的私有临时表技术。

## 全局临时表

全局临时表用于在事务或会话期间保存中间结果集。临时表中的数据仅对当前会话可见——即使当前会话提交（`COMMIT`）了数据，任何其他会话也看不到其他会话的数据。多用户并发性对于临时表也不是问题，因为一个会话永远不会通过使用临时表来阻塞另一个会话。即使我们“锁定”临时表，也不会阻止其他会话使用它们自己的临时表。

**注意**

正如我们在第 9 章中所观察到的，临时表生成的重做日志比常规表少得多。然而，由于临时表会为其包含的数据生成撤销信息，它们仍会生成一定数量的重做日志。`UPDATE` 和 `DELETE` 操作生成的重做日志最多；`INSERT` 和 `SELECT` 操作生成的重做日志最少。我们还在第 9 章中看到，从 Oracle 12c 开始，可以将临时表配置为几乎不生成重做日志；这是通过将 `TEMP_UNDO_ENABLED` 参数设置为 `TRUE` 来实现的。

临时表将从当前登录用户的临时表空间分配存储；或者，如果从定义者权限的存储过程中访问，则使用该存储过程所有者的临时表空间。全局临时表实际上只是表本身的一个模板。创建临时表的操作不涉及存储分配；不会分配 `INITIAL` 区，就像为常规的堆组织表所做的那样（除非启用了延迟段功能）。相反，在运行时，当会话首次向临时表中放入数据时，将为该会话创建一个临时段。由于每个会话获得自己的临时段（而不仅仅是现有段的一个区），因此每个用户可能在不同的表空间中为其临时表分配空间。例如，`USER1` 可能将其临时表空间设置为 `TEMP1`，因此他们的临时表将从该空间分配。`USER2` 可能将 `TEMP2` 作为其临时表空间，他们的临时表将在那里分配。

Oracle 的全局临时表在某些方面与其他关系数据库中的临时表相似，主要区别在于它们是*静态*定义的。您为每个数据库创建一次，而不是为数据库中的每个存储过程创建一次。它们始终存在——它们将作为对象存在于数据字典中，但它们总是显示为空，直到您的会话将数据放入其中。它们是静态定义的这一事实允许您创建引用临时表的视图、创建使用静态 SQL 引用它们的存储过程等等。

临时表可以是*基于会话*的（数据在表中跨 `COMMIT` 持久存在，但断开/重连后不保留）。它们也可以是*基于事务*的（数据在 `COMMIT` 后消失）。下面是一个展示两者行为的示例。我使用 `SCOTT.EMP` 表作为模板：

```
$ sqlplus eoda/foo@PDB1
SQL> create global temporary table temp_table_session
on commit preserve rows
as
select * from scott.emp where 1=0;
Table created.
```

`ON COMMIT PRESERVE ROWS` 子句使其成为基于会话的临时表。行将保留在该表中，直到我的会话断开连接或我通过 `DELETE` 或 `TRUNCATE` 物理删除它们。只有我的会话能看到这些行；在我提交（`COMMIT`）后，任何其他会话也永远看不到我的行。

```
SQL> create global temporary table temp_table_transaction
on commit delete rows
as
select * from scott.emp where 1=0;
Table created.
```

`ON COMMIT DELETE ROWS` 使其成为基于事务的临时表。当我的会话提交时，这些行会消失。行的消失方式是简单地释放分配给我的表的临时区——自动清除临时表不涉及任何开销。
