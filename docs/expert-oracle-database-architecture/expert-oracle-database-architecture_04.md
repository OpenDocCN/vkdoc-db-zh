# 使用绑定变量与 SQL 注入安全

## 性能影响与锁机制

`runstats` 工具将生成一份报告，显示锁存器利用率和统计信息的差异。在此，我要求 `runstats` 打印出差异大于 9500 的任何内容。你可以看到，在第一种使用绑定变量的方法中，我们`硬解析`了 3 次，而未使用绑定变量时，我们硬解析了 10,000 次（每次插入一次）。但这种硬解析的差异只是冰山一角。你可以看到，在非绑定变量方法中，我们使用的"锁存器"数量比使用绑定变量时多了一个数量级。这个差异可能会引出一个问题："什么是锁存器？"

让我们来回答这个问题。锁存器（Latch）是 Oracle 用于序列化访问共享数据结构的一种锁。共享池就是一个例子；它是系统全局区（SGA）中的一个大型共享数据结构，Oracle 在此存储已解析、已编译的 SQL。当您修改此共享结构中的任何内容时，必须注意一次只允许一个进程进入。（如果两个进程或线程试图同时更新相同的内存数据结构，后果将非常严重——会导致数据损坏。）因此，Oracle 采用了一种`锁存`机制，一种轻量级的锁定方法来序列化访问。不要被"轻量级"这个词所迷惑。锁存器是序列化设备，一次只允许一个进程访问（内存结构）。硬解析实现使用的锁存器是一些最常用的锁存器。这些包括用于共享池和库高速缓存的锁存器。这些都是人们频繁竞争的"重要"锁存器。所有这一切意味着，随着我们增加同时尝试硬解析语句的用户数量，我们的性能会随着时间的推移而逐渐恶化。解析的人越多，等待锁存共享池的人就越多，队列就越长，等待时间也就越长。

## 使用 Java 编译类比

不使用绑定变量执行 SQL 语句，非常类似于在每次方法调用之前都编译一个子例程。想象一下，将 Java 源代码发送给您的客户，在调用类中的方法之前，他们必须先调用 Java 编译器、编译该类、运行该方法，然后丢弃字节码。下次他们想执行相同的方法时，他们会做同样的事情：编译它、运行它、然后丢弃它。您永远不会考虑在您的应用程序中这样做；您也不应该在数据库中考虑这样做。

## 安全风险：SQL 注入

不使用绑定变量的另一个影响，对于采用字符串拼接的开发人员来说，是安全——具体来说是所谓的`SQL 注入`。如果您不熟悉这个术语，我鼓励您暂时放下这本书，使用您选择的搜索引擎，查找 SQL 注入。在我撰写本版时，它返回了超过五百万条搜索结果。SQL 注入问题有据可查。

> **注意**
>
> SQL 注入是一个安全漏洞，开发者接受最终用户的输入，并将该输入拼接到查询中，然后编译并执行该查询。实际上，开发者接受了来自最终用户的 SQL 代码片段，然后编译并执行这些片段。这种方法允许最终用户可能修改 SQL 语句，使其做一些应用程序开发者从未打算做的事情。这几乎就像留下一个打开的终端，其中登录并连接了一个具有 SYSDBA 权限的 SQL*Plus 会话。您简直是在乞求某人过来输入一些命令，编译它，然后执行它。结果可能是灾难性的。

事实上，如果您不使用绑定变量，如果您使用前面显示的 `PROC2` 中的字符串拼接技术，您的代码就容易受到 SQL 注入攻击，必须仔细审查。并且它应该由实际上不喜欢编写该代码的开发人员来审查——因为代码必须受到批判性和客观性的审查。如果审查者是代码作者的同事，或者更糟，是朋友或下属，那么审查将不会像应有的那样严格。开发出的不使用绑定变量的代码必须受到怀疑——它应该是不使用绑定变量的特殊情况，而不是常态。

## 演示示例：SQL 注入是如何工作的

为了演示 SQL 注入有多么阴险，我展示这个小例子。首先，我将在我的可插拔数据库中创建一个具有 DBA 角色的用户：

```sql
$ sqlplus system/foo@PDB1
SQL> create user pwd_mgr identified by pwd_mgr_pwd;
User created.
SQL> grant unlimited tablespace to pwd_mgr;
SQL> grant dba to pwd_mgr with admin option;
Grant succeeded.
```

> **注意**
>
> 如果在您的 PDB 中无法授予 DBA 角色，则改为将 `PDB_DBA` 角色授予 `pwd_mgr` 用户。

接下来，连接到该用户并创建一个小过程：

```sql
SQL> conn pwd_mgr/pwd_mgr_pwd@PDB1
SQL> create or replace procedure inj( p_date in date )
as
    l_username   all_users.username%type;
    c            sys_refcursor;
    l_query      varchar2(4000);
begin
    l_query := '
    select username
    from all_users
    where created = ''' ||p_date ||'''';
    dbms_output.put_line( l_query );
    open c for l_query;
    for i in 1 .. 5
    loop
        fetch c into l_username;
        exit when c%notfound;
        dbms_output.put_line( l_username || '.....' );
    end loop;
    close c;
end;
/
Procedure created.
```

> **注意**
>
> 此代码最多只打印五条记录。它是为在"空"模式中执行而开发的。具有许多现有表的模式可能导致与接下来显示的结果不同的各种效果。一种效果可能是您看不到我试图在示例中向您展示的表——那是因为我们只打印五条记录。另一种可能是数值或值错误——那可能是由于表名很长。这些事实都不会使示例失效；它们都可以被想要窃取您数据的人绕过。

现在，我认识的大多数开发人员看到这段代码都会说它是安全的，不会受到 SQL 注入。他们会这么说，因为例程的输入必须是一个 Oracle `DATE` 变量，一个 7 字节的二进制格式，表示世纪、年、月、日、小时、分钟和秒。`DATE` 变量不可能改变我的 SQL 语句的含义。事实证明，他们大错特错了。这段代码可以被"注入"——在运行时轻松修改——由任何知道如何做的人完成（而且，显然，有人知道如何做！）。如果您按照开发者"期望"的方式执行该过程，您可能会看到如下结果：

```sql
SQL> set serverout on
SQL> exec inj( sysdate )
select *
from all_users
where created = '05-FEB-21'
PL/SQL procedure successfully completed.
```

此结果显示 SQL 语句被安全地构造——正如预期的那样。那么，有人如何以恶意的方式使用这个例程呢？嗯，假设您的项目中有另一个开发者——邪恶的开发者。开发者有权执行该过程，以查看今天在数据库中创建的用户，但他们无权访问拥有此过程的模式中的任何其他表。现在，他们不知道这个模式中存在哪些表——安全团队认为"通过隐藏实现安全"是好的——所以他们不允许任何人发布表名。因此，他们不知道存在以下特定的表：

```sql
SQL> create table user_pw
    ( uname varchar2(30) primary key,
      pw    varchar2(30));
Table created.
SQL> insert into user_pw   ( uname, pw ) values ( 'TKYTE', 'TOP SECRET' );
1 row created.
SQL> commit;
Commit complete.
```
