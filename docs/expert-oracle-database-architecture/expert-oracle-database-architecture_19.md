# 多块大小

你可以在同一个数据库中使用多种数据库块大小。你可以混合使用"默认"块大小（即你最初创建数据库时使用的块大小；`SYSTEM`和所有`TEMPORARY`表空间所用的大小）和最多其他四种块大小。每种唯一的块大小都必须有其自己的缓冲区缓存区域。默认池（default pool）、保留池（keep pool）和回收池（recycle pool）只会缓存默认大小的块。为了在数据库中使用非默认块大小，你需要配置一个缓冲池来容纳它们。

在这个示例中，我的默认块大小是 8KB。我将尝试创建一个块大小为 16KB 的表空间：

```
$ sqlplus / as sysdba
SQL> create tablespace ts_16k datafile '/tmp/ts_16k.dbf'  size 5m blocksize 16k;
create tablespace ts_16k
*
ERROR at line 1:
ORA-29339: tablespace block size 16384 does not match configured block sizes
SQL> show parameter 16k
NAME                                 TYPE        VALUE
------------------------------------ ----------- --------------------------
db_16k_cache_size                    big integer 0
```

目前，由于我还没有配置 16KB 缓存，我无法创建这样的表空间。我现在可以采取几种措施来纠正这种情况。我可以设置`DB_16K_CACHE_SIZE`参数并重启数据库。我可以缩小其他 SGA 组件之一，以便为现有 SGA 中的 16KB 缓存腾出空间。或者，如果`SGA_MAX_SIZE`参数大于我当前的 SGA 大小，我也许可以直接分配一个 16KB 缓存。

注意

在数据库启动并运行时，调整各种 SGA 组件的大小是可能的。如果你希望 SGA 的大小能够"增长"到超出其初始分配值，你必须将`SGA_MAX_SIZE`参数设置为某个大于已分配 SGA 的值。例如，如果启动后你的 SGA 大小为 800MB，而你想为缓冲区缓存额外增加 200MB，那么你必须事先将`SGA_MAX_SIZE`设置为 1GB 或更大，以允许这种增长。

在这个示例中，由于我使用的是 ASMM（自动共享内存管理），并且不希望手动设置其他缓存，我将设置`DB_16K_CACHE_SIZE`并重启：

```
SQL> alter system set db_16k_cache_size = 16m scope=spfile;
SQL> startup force
SQL> show parameter 16k
NAME                                 TYPE        VALUE
------------------------------------ ----------- --------------------------
db_16k_cache_size                    big integer 16M
```

所以，现在我已经建立了另一个缓冲区缓存：一个用于缓存所有大小为 16KB 的块。默认池将占用剩余的缓冲区缓存空间，这可以通过查询`V$SGASTAT`看到。这两个缓冲区缓存是互斥的；如果一个"满了"，它不能使用另一个的空间。这为 DBA 提供了对内存使用的非常精细的控制，但这是有代价的。代价就是复杂性和管理。这些多块大小最初并非作为性能或调优功能而设计（如果你需要多个缓存，你已经有了`default`、`keep`和`recycle`池），而是为了支持可传输表空间——将格式化数据文件从一个数据库取出并传输或附加到另一个数据库的能力。它们是为了将使用 8KB 块大小的事务处理系统中的数据文件，传输到使用 16KB 或 32KB 块大小的数据仓库而实现的。

然而，多块大小在测试理论方面确实能起到很好的作用。如果你想看看你的数据库在使用不同块大小时如何运作——例如，如果使用 4KB 块而不是 8KB 块，某个特定表将消耗多少空间——你现在可以轻松地进行测试，而无需创建一个全新的数据库实例。

你也可以将多块大小用作针对特定段集合的非常聚焦的调优工具，通过为它们提供自己的私有缓冲池。或者，在混合系统中，事务处理用户可以使用一组数据，而报告/仓库用户可以查询另一组独立的数据。事务处理数据将受益于更小的块大小，因为块上的争用更少（每个块的数据/行更少，意味着通常更少的人会同时争抢同一个块），并且缓冲区缓存利用率更高（用户只将他们感兴趣的数据——单行或小行集——读入缓存）。基于事务处理数据的报告/仓库数据，则可能受益于更大的块大小，部分原因是块开销更小（整体存储占用更少）以及可能更大的逻辑 I/O 尺寸。并且由于报告/仓库数据没有相同的更新争用问题，每个块有更多行这一事实不是问题而是优势。此外，事务处理用户实际上拥有了自己的缓冲区缓存；他们不必担心报告查询会占用他们的缓存。

但一般来说，默认池、保留池和回收池应该足以对块缓冲区缓存进行微调，而多块大小将主要用于在数据库之间传输数据，或许也用于混合报告/事务处理系统。


### 共享池

共享池是 SGA 中最关键的内存区域之一，尤其关乎性能与可扩展性。共享池过小可能严重损害性能，甚至导致系统看似挂起。共享池过大也可能产生同样的效果。而使用不当的共享池同样会带来灾难。

共享池究竟是什么？它是 Oracle 缓存众多“程序”数据片段的地方。当我们解析一条查询时，其解析后的表示形式会被缓存在此。在执行完整查询解析工作之前，Oracle 会先搜索共享池，查看该工作是否已完成。您运行的 PL/SQL 代码也缓存在共享池中，因此下次运行时，Oracle 无需再次从磁盘读取。PL/SQL 代码不仅在此缓存，也在此共享。如果您有 1000 个会话都执行相同的代码，则只会加载一份代码副本并在所有会话间共享。Oracle 将系统参数存储在共享池中。数据字典缓存（关于数据库对象的缓存信息）也存储于此。简而言之，几乎所有重要数据都存储在共享池中。

共享池的特点是包含大量小（通常为 4KB 或更小）的内存块。请注意，4KB 并非硬性限制。偶尔也会有超过此大小的分配，但总体目标是使用小内存块，以防止因分配大小从极小到极大剧烈变化而导致的碎片化。共享池中的内存基于 LRU（最近最少使用）机制管理。在这方面，它与缓冲区缓存类似——如果不使用，就会被淘汰。一个名为 `DBMS_SHARED_POOL` 的内置包可用于改变此行为——强制将对象固定在共享池中。您可以使用此过程在数据库启动时加载常用的过程和包，使其不会因老化而被清除。但通常情况下，如果一段时间内共享池中的某块内存未被重用，它将面临老化清除。即使是可能相当庞大的 PL/SQL 代码，也通过分页机制管理，因此当您执行一个非常大的包中的代码时，只有需要的代码才会以小块形式加载到共享池中。如果您在很长一段时间内未使用它，并且共享池已满且需要空间给其他对象时，它将被老化清除。

破坏 Oracle 共享池的最简单方法是不使用绑定变量。正如您在第 1 章所见，不使用绑定变量可能使系统陷入困境，原因有二：

*   系统消耗大量 CPU 时间来解析查询。
*   由于查询从未被重用，系统需消耗大量资源来管理共享池中的对象。

如果提交给 Oracle 的每条查询都是唯一的（因为硬编码了唯一值），那么共享池的概念就基本被破坏了。共享池的设计初衷是让查询计划被反复使用。如果每条查询都是全新的、前所未见的，那么缓存只会增加开销。共享池反而成了性能的阻碍。许多人试图用一种常见但错误的技术来解决此问题，即向共享池添加更多空间，这通常只会使情况比之前更糟。随着共享池不可避免地再次填满，它所带来的开销甚至比更小的共享池还要大，原因很简单：管理一个大的、满的共享池比管理一个小的、满的共享池需要更多的工作。

解决此问题的唯一真正方法是使用共享 SQL 来重用查询。之前在第 1 章中，我们简要了解了 `CURSOR_SHARING` 参数，它可以在该领域充当临时的“拐杖”。然而，真正解决此问题的唯一方法是首先使用可重用的 SQL。即使在超大规模的系统上，我发现通常最多也只有 10,000 到 20,000 条唯一的 SQL 语句。大多数系统只执行几百条唯一的查询。

下面的真实示例展示了如果共享池使用不当，情况会变得多糟糕。我曾被要求处理一个系统，其标准操作程序是每晚关闭数据库，清除 SGA，然后重新启动。这样做的原因是系统在白天会出现问题，CPU 完全饱和，如果让数据库运行超过一天，性能确实开始下降。他们在一个 1.1GB 的 SGA 中使用了 1GB 的共享池。这是真实的：0.1GB 专用于块缓冲区缓存和其他元素，而 1GB 专用于缓存那些永远不会再次执行的唯一查询。他们进行冷启动的原因是，如果让系统运行超过一天，共享池中的空闲内存就会耗尽。此时，老化结构（尤其是从如此大的结构中）清除的开销会压垮系统，导致性能严重下降（尽管性能本来也不怎么样，因为他们还要管理 1GB 的共享池）。此外，使用该系统的人员不断想要为机器添加越来越多的 CPU，因为硬解析 SQL 非常消耗 CPU。通过修正应用程序并让其使用绑定变量，不仅物理机需求下降了（他们后来拥有的 CPU 能力是所需的数倍），而且分配给各个池的内存也反转了过来。共享池不再是 1GB，而是分配了不到 100MB——并且在连续运行数周内，他们从未用完所有空间。

注意

对于可插拔数据库，`SHARED_POOL_SIZE` 是可选的。如果为可插拔数据库设置了此参数，则它会设置该可插拔数据库共享池的最小大小。当此参数在可插拔数据库级别未设置时，则该可插拔数据库可使用的共享池量没有限制，仅受容器数据库共享池最大大小的约束。

`SHARED_POOL_SIZE` 通常是共享池中 `SUM(BYTES)` 报告的最大贡献者，但它并非唯一贡献者。例如，参数 `CONTROL_FILES` 为共享池的“杂项”部分贡献了每个文件 264 字节。不幸的是，`V$SGASTAT` 中的“共享池”和参数 `SHARED_POOL_SIZE` 的命名方式如此，因为该参数确实贡献了共享池的大小，但它不是唯一的贡献者。假设您使用的是手动共享内存管理（即您自行设置了 `SHARED_POOL_SIZE` 参数），您应该看到两者之间存在一一对应的关系：

```
SQL> select sum(bytes)/1024/1024 mbytes from v$sgastat where pool = 'shared pool';
MBYTES

SQL> show parameter shared_pool_size;
NAME                                 TYPE        VALUE
------------------------------------ ----------- --------------------------
shared_pool_size                     big integer 128M
```

注意

在此示例中，我使用的是手动共享内存管理！


## 大型池

大型池之所以如此命名，并非因为其结构“庞大”（尽管其物理尺寸可能确实很大），而是因为它用于分配那些共享池无法处理的大块内存。大型池是一种回收式内存空间，而共享池则更像保留缓冲池——如果某些内容被频繁使用，你就会将其保留在缓存中。

在大型池中分配的内存以**堆**的方式管理，其方式类似于 C 语言通过`malloc()`和`free()`管理内存。一旦你“释放”了一块内存，它就可以被其他进程使用。在共享池中，实际上并没有“释放”内存块的概念。你会分配内存，使用它，然后停止使用。过一段时间，如果需要重用那块内存，Oracle 会将你的内存块**老化换出**。仅使用共享池的问题在于，一种尺寸并不总是适合所有情况。

大型池专门被以下功能使用：

*   *共享服务器连接*，用于在 SGA 中分配 UGA 区域
*   *语句的并行执行*，用于分配进程间消息缓冲区，这些缓冲区用于协调并行查询服务器
*   *备份*，在某些情况下用于 RMAN 磁盘 I/O 缓冲区

如你所见，这些内存分配都不应该在一个设计用于管理小内存块的 LRU 缓冲池中管理。例如，对于共享服务器连接内存，一旦会话注销，这部分内存就永远不会被重用，因此应该立即返回给池。此外，共享服务器 UGA 的内存分配往往“很大”。将共享服务器内存放入共享池会导致它分裂成大小不一的碎片，而且，你会发现那些永远不会被重用的大块内存会老化换出那些可能被重用的内存。这迫使数据库稍后做更多的工作来重建那个内存结构。

并行查询消息缓冲区也是如此，因为它们不是 LRU 可管理的。它们被分配，在使用完成前无法被释放。一旦它们传递了消息，就不再需要，应该立即释放。对于备份缓冲区，这一点更加突出——它们很大，一旦 Oracle 使用完毕，它们就应该直接“消失”。

使用共享服务器连接时，大型池并非强制要求，但强烈建议配置。如果你没有大型池并且使用了共享服务器连接，那么分配将来自共享池。这肯定会在一段时间内导致性能下降，应该避免。

如果设置了`SGA_TARGET`，但没有为`LARGE_POOL_SIZE`设置值，则默认为零（意味着由 Oracle 数据库内部决定）。如果设置了`LARGE_POOL_SIZE`，则该值将用作内存池的最小值。

## Java 池

添加 Java 池是为了支持在数据库中运行 Java。如果你用 Java 编写了一个存储过程，Oracle 在处理该代码时将使用这块内存。参数`JAVA_POOL_SIZE`用于为所有会话特定的 Java 代码和数据固定分配给 Java 池的内存量。

Java 池的使用方式取决于 Oracle 服务器的运行模式。在专用服务器模式下，Java 池包含每个 Java 类的共享部分，这部分实际上是每个会话使用的。这些基本上是只读部分（执行向量、方法等），每个类大约 4KB 到 8KB。因此，在专用服务器模式下，Java 池所需的总内存相当适中，可以根据你将使用的 Java 类数量来确定。请注意，在专用服务器模式下，没有会话状态存储在 SGA 中，因为此信息存储在 UGA 中，并且，如你所记得的，在专用服务器模式下，UGA 包含在 PGA 中。

当使用共享服务器连接连接到 Oracle 时，Java 池包含以下两部分：

*   每个 Java 类的共享部分。
*   部分用于每个会话状态的 UGA，这部分从 SGA 内的 Java 池中分配。剩余的 UGA 将照常位于共享池中，或者，如果配置了大型池，则将位于大型池中。

如果设置了`SGA_TARGET`而没有设置`JAVA_POOL_SIZE`，则默认为零（意味着由 Oracle 内部决定）。如果指定了`JAVA_POOL_SIZE`，则该值表示内存池的最小值。

## 流池

流池是 Oracle 产品使用的 SGA 结构，这些产品包括 Oracle GoldenGate、XStream、Oracle Streams、Oracle 高级队列和 Oracle 数据泵。

流池的大小通过设置`STREAMS_POOL_SIZE`参数来控制。如果`SGA_TARGET`是一个非零值，那么自动 SGA 内存管理将使用`STREAMS_POOL_SIZE`的设置作为流池的最小值。如果`SGA_TARGET`和`STREAMS_POOL_SIZE`都设置为零，那么流池最多使用共享池的百分之十。

使用流池的产品将缓冲队列消息。这些功能使用内存队列，而不是使用永久的基于磁盘的队列及其伴随的开销。如果这些队列被填满，它们最终会溢出到磁盘。如果使用内存队列的 Oracle 实例由于某种原因（实例故障（软件崩溃）、断电或其他原因）失败，这些内存队列将从重做日志中重建。

流池仅在那些使用需要此内存区域空间的功能（如 GoldenGate、Streams 等）的系统中才重要。在这些环境中，应设置`STREAMS_POOL_SIZE`以避免“窃取”共享池的百分之十。

## 内存池

数据库**内存中**特性包含一个高度优化的**内存中列存储**，它与现有的行格式缓冲缓存协同工作。内存中列存储的主要目的是提高分析操作所执行的列式数据访问的性能。你可以将其视为为表中的每一列都有一个常规索引（用于分析）。但是，它与常规索引不同，不需要日志记录或任何对数据库的写入。如果你的应用程序执行大量的分析，那么实施此内存特性可能会让你受益。

**内存中**特性需要额外的 Oracle 许可证才能使用。然而，从 Oracle 21c 开始，当以有限的“基础级别”方式实施时，此功能无需额外费用即可使用。通过将`INMEMORY_FORCE`设置为`BASE_LEVEL`来实例化基础级别内存中功能。此基础级别允许你在不购买内存中选件的情况下试验内存中功能。启用基础级别时，内存中列存储对于一个容器数据库限制为 16GB。此外，在基础级别中，内存中对象的压缩级别自动设置为`QUERY LOW`。*Oracle 数据库许可信息用户手册*详细说明了哪些内存中功能可用于不同的版本和服务。

由于内存中是一个额外付费的选件，并且有大量功能可用，我将指引你参考*Oracle 数据库内存中指南*（截至本书撰写时为 281 页）以获取有关此功能的更多详细信息。描述所有功能和实现细节超出了本书的范围。

注意

可以在可插拔数据库中设置`INMEMORY_SIZE`，以限制该可插拔数据库的内存中列存储的最大大小。可插拔数据库的值之和不必等于容器数据库的设置（可插拔数据库设置的总和可以大于容器数据库设置）。


### 系统全局区（SGA）内存管理

SGA 是一组共享内存结构，包含 Oracle 实例的数据和控制结构。当多个用户并发访问实例内存时，数据可以在用户之间共享。这就是为什么 SGA 有时被称为共享内存区。管理 SGA 内存有三种方式：

*   `自动内存管理 (AMM)`：设置 `MEMORY_TARGET` 参数。通过设置 `MEMORY_TARGET` 参数，您允许实例调整 SGA 和 PGA 内存区域的大小。
*   `自动共享（或 SGA）内存管理 (ASMM)`：设置 `SGA_TARGET` 参数。通过设置 `SGA_TARGET` 参数，您允许实例调整各个 SGA 组件的大小。
*   `手动共享内存管理`：设置所有必要的池和缓存参数。

在本节深入之前，我想先说明，调整 SGA 内存可能会让人困惑。之所以困惑，是因为 Oracle 使用了非常相似但含义截然不同的缩写词，例如 AMM 和 ASMM。同样令人困惑的是，在 Oracle 关于调整 SGA 的文档中，您会发现相互矛盾且互斥的建议。例如，看看 Oracle 对 AMM 的建议：

*   *Oracle 性能调优指南* 指出，Oracle 强烈建议使用 AMM。
*   *Oracle 数据库管理指南* 也建议您启用 AMM。
*   *Oracle 数据库管理指南* 同时指出，使用 AMM 仅适用于 SGA 和 PGA 内存大小小于 4GB 的数据库。
*   Oracle 支持文档 `749851.1` 指出，使用 HugePages 的 Linux 环境与 AMM 不兼容。这一点很有趣，因为 Oracle 的大部分客户都在启用了 HugePages 的 Linux 服务器上运行数据库。

那么，该用哪种呢？在理想环境下，每个人都应该使用 AMM，对吧？只需设置一个参数（`MEMORY_TARGET`）就完成了。然而，在现实世界中，事情并非如此非黑即白。有时，您环境的某些方面（例如，使用 Linux 操作系统）会引导您选择某个方向。尽管如此，内存管理可以归结为以下简单建议：

*   对于大型数据库和/或 Linux 环境中的数据库，请使用 ASMM 来管理您的 SGA，并使用自动 PGA 内存管理来设置您的 PGA。在此模式下，您将 `MEMORY_TARGET` 设置为零，然后将 `SGA_TARGET` 和 `PGA_AGGREGATE_TARGET` 设置为非零值。
*   对于小型数据库和/或非 Linux 环境中的数据库，请使用 AMM。对于此方法，您将 `MEMORY_TARGET` 设置为非零值，并将 `SGA_TARGET` 和 `PGA_AGGREGATE_TARGET` 设置为零。然后，Oracle 将根据需要调整内存池的大小。在此模式下，您可以选择将 `SGA_TARGET` 和 `PGA_AGGREGATE_TARGET` 设置为非零值，Oracle 将使用这些设置作为分配给这两个值的最小内存量。
*   不要对您的 SGA 和 PGA 使用过时的手动内存管理方法（除非您有非常充分的理由，例如您运行的是旧版 Oracle 且没有其他内存管理选项可用）。

如果您处于 Linux 环境中，那么在大多数情况下（Linux 环境），您应该使用 ASMM。我想在本节开头指出这一点，以便明确您应该使用什么进行内存管理。说完这些，接下来我将讨论每种 SGA 管理技术。

#### 自动内存管理

AMM 有点像一站式解决您所有内存设置的方案。DBA 现在可以通过设置一个内存参数——`MEMORY_TARGET`——来轻松应对。这个 `MEMORY_TARGET` 代表了组合的 SGA 和 PGA 分配应该努力保持在内的总内存量（请记住，PGA 内存可能有些难以控制）。数据库将根据工作负载历史记录动态确定合适的 SGA 大小和 PGA 大小。随着时间的推移，随着数据库中执行的工作负载发生变化，分配给 SGA 和 PGA 的内存也会发生变化。例如，如果您白天是繁重的 OLTP（在线事务处理），晚上是繁重的批处理，您可能会发现白天的 SGA 远大于 PGA，而晚上的 SGA 远小于 PGA。这反映了这两种应用程序类型不同的内存需求。

**注意：** 如果您在使用 HugePages 的 Linux 环境中运行 Oracle 数据库，请阅读 Oracle 支持文档 ID `749851.1`。该说明指出，使用 AMM 与 Linux HugePages 不兼容。使用 Linux HugePages 是 Oracle 数据库的标准做法，因为它可以提高性能。

DBA 可以通过设置 `SGA_TARGET` 和 `PGA_AGGREGATE_TARGET` 为每个内存区域设置下限，或者通过将其值设置为该下限来为 SGA 中的每个池设置下限。如果您使用的是存储的参数文件，数据库将记住池以及 SGA 和 PGA 的最佳设置。例如，在我的一个测试系统上，我设置了

```
SQL> alter system set memory_target = 756m scope=spfile;
SQL> alter system set sga_target = 300m scope=spfile;
SQL> alter system set pga_aggregate_target=300 scope=spfile;
```

接下来，我重启（停止/启动）实例以使参数生效：

```
SQL> startup force;
```

检查 spfile，它当前具有以下设置：

```
SQL> create pfile='/tmp/pfile' from spfile;
File created.
SQL> !cat /tmp/pfile;

CDB.__inmemory_ext_roarea=0
CDB.__inmemory_ext_rwarea=0
CDB.__java_pool_size=0
CDB.__large_pool_size=4194304
CDB.__oracle_base='/opt/oracle'#ORACLE_BASE set from environment

CDB.__sga_target=314572800
CDB.__shared_io_pool_size=8388608
CDB.__shared_pool_size=213909504

```

如您所见，双下划线参数包括了 `__sga_target` 和 `__pga_aggregate_target` 设置以及各个池。这些值是基于 spfile 中的内存参数设置以及观察到的服务器工作负载推导出来的。通过这种方式，Oracle 会记住您上次最佳的 SGA/PGA 设置，并在下次启动时使用它们。

**注意：** 最后这个功能，即存储池的推荐值，仅在您使用 spfile 作为初始化文件（而不是基于文本的 `init.ora` 文件）时才有效。


## 自动共享内存管理 (ASMM)

在近期所有的 DBA 经验中（包括 Oracle 云中的数据库），这是管理 SGA 内存最常见的方法。在 ASMM 下，用于调整自动调优组件总大小的主要参数是 `SGA_TARGET`。该参数可以在数据库运行时动态调整，最大可达 `SGA_MAX_SIZE` 参数的设置值。`SGA_MAX_SIZE` 参数默认与 `SGA_TARGET` 相等，因此，如果您计划增加 `SGA_TARGET`，必须在启动数据库实例之前将 `SGA_MAX_SIZE` 设置得更大。如果当前正在使用 `MEMORY_TARGET`，请先将其设为零，然后将 `SGA_TARGET` 设置为一个非零值，例如：

```
SQL> alter system set memory_target=0 scope=spfile;
SQL> alter system set sga_target = 1600m scope=spfile;
```

设置 `SGA_TARGET` 后，我将重新启动数据库并验证设置：

```
SQL> startup force;
SQL> show parameter sga_target
NAME                                 TYPE        VALUE
------------------------------------ ----------- --------------------------
sga_target                           big integer 1600M
```

如果设置了 `SGA_TARGET`，则 Oracle 会自动调整以下内存组件的大小：

*   缓冲区缓存 (`DB_CACHE_SIZE`)
*   共享池 (`SHARED_POOL_SIZE`)
*   大池 (`LARGE_POOL_SIZE`)
*   Java 池 (`JAVA_POOL_SIZE`)
*   流池 (`STREAMS_POOL_SIZE`)
*   数据传输缓存 (`DATA_TRANSFER_CACHE_SIZE`)

如果您将这些参数设置为非零值（同时设置了 `SGA_TARGET`），则这些值将作为对应内存区域的最小值。通常，如果您知道使用数据库的应用程序需要较大的最小值才能正常运行，才会为这些参数设置非零值。

数据库将使用 `SGA_TARGET` 值减去任何其他手动调整大小的组件（如 `DB_KEEP_CACHE_SIZE`、`DB_RECYCLE_CACHE_SIZE` 等）的大小，并用该内存量来调整默认缓冲池、共享池、大池和 Java 池的大小。在运行时，实例将根据需要在这四个内存区域之间动态分配和重新分配内存。当共享池内存不足时，实例可以选择将缓冲区缓存缩小若干兆字节（一个粒度大小），并相应增加共享池，而不是向用户返回 ORA-04031“无法为共享内存分配 N 字节”错误。

> 注意
> 要使用 ASMM，参数 `STATISTICS_LEVEL` 必须设置为 `TYPICAL`（默认值）或 `ALL`。如果未启用统计信息收集，数据库将没有做出必要大小调整决策所需的历史信息。

随着时间的推移，随着实例的内存需求被确定，各种 SGA 组件的大小会变得或多或少固定下来。数据库还会在数据库启动和关闭之间记住这四个组件的大小，这样就不必每次从头开始为您的实例计算合适的大小。它通过四个双下划线参数来实现这一点：`__db_cache_size`、`__java_pool_size`、`__large_pool_size` 和 `__shared_pool_size`。在正常或立即关闭期间，数据库会将这些值记录到 spfile 中，并在启动时使用它们来设置每个区域的默认大小。我们可以通过以下方式查看这些参数：

```
SQL> create pfile='/tmp/tmp.ora' from spfile;
SQL> !cat /tmp/tmp.ora
```

以下是输出片段：

```

CDB.__inmemory_ext_roarea=0
CDB.__inmemory_ext_rwarea=0
CDB.__java_pool_size=16777216
CDB.__large_pool_size=33554432
CDB.__oracle_base='/opt/oracle'#ORACLE_BASE set from environment

```

此外，如果您知道希望为这五个区域中的某个区域使用特定的最小值，可以在设置 `SGA_TARGET` 的同时设置该参数。实例将使用您的设置作为下限，即该特定区域可能的最小尺寸。

> 注意
> `SGA_TARGET` 参数对于可插拔数据库是可选的。当为可插拔数据库设置此参数时，它指定了该可插拔数据库在任何时候可以使用的最大 SGA。如果在可插拔数据库级别未设置此参数，则它使用的 SGA 大小没有限制，但受根容器数据库 SGA 大小的约束。

## 手动共享内存管理

手动共享内存管理应该很少使用；因此，我不打算详细讨论此技术。这主要是因为使用前面描述的方法（由 Oracle 为您管理 SGA 内存）更为简单。但是，如果您需要对 SGA 内存的自动调优区域进行一定程度的控制，则可以将 `MEMORY_TARGET` 和 `SGA_TARGET` 设置为零。当 `MEMORY_TARGET` 设置为零时，会禁用 AMM；当 `SGA_TARGET` 设置为零时，会禁用 ASMM。

当禁用自动内存管理技术时，您可以通过指定以下内存参数的大小来手动设置 SGA 的大小：`DB_CACHE_SIZE`、`SHARED_POOL_SIZE`、`LARGE_POOL_SIZE`、`JAVA_POOL_SIZE` 和 `STREAMS_POOL_SIZE`。这些参数中的每一个都有一个默认值，如果您没有显式设置它们，Oracle 将使用该默认值。例如，`DB_CACHE_SIZE` 将被设置为 48MB 或 4MB 乘以 CPU 数量（取两者中较大的一个）。

请注意，即使您已关闭此功能，Oracle 仍可能执行一定程度的自动 SGA 内存重新分配。Oracle 这样做是为了防止进程在共享池中空间不足。

> 提示
> 有关使用手动共享内存管理时自动调整 SGA 大小的更多详细信息，请参阅 Oracle 支持说明 1269139.1。
