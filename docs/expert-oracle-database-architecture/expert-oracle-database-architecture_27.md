# 7. 并发与多版本控制

开发多用户、数据库驱动应用程序的关键挑战之一是最大化并发访问，同时确保每个用户能够以一致的方式读取和修改数据。在本章中，我们将详细探讨 Oracle 如何实现*多版本读取一致性*，以及这对您——开发人员和 DBA——意味着什么。我还将引入一个新术语，*写入一致性*，并用它来描述 Oracle 如何在具有读取一致性的读取环境中工作，以及在混合读取和写入环境中如何工作。


## 什么是并发控制？

并发控制（Concurrency controls）是数据库提供的一系列功能，允许多人同时访问和修改数据。正如前两章所述，锁（lock）是 Oracle 用来调节对共享数据库资源的并发访问并防止并发数据库事务之间相互干扰的核心机制之一。简而言之，Oracle 使用各种锁，包括以下类型：

*   `TX`（事务）锁：这些锁在数据修改事务期间被获取。
*   `TM`（DML 队列）和 `DDL` 锁：这些锁用于确保在修改对象内容时（`TM` 锁）或修改对象本身时（`DDL` 锁），对象的结构不会被更改。
*   `Latches` 和 `mutexes`：这些是 Oracle 用来协调对其共享数据结构访问的内部锁。在本章中，我们会将两者统称为 `latches`，尽管根据 Oracle 版本的不同，它们在你的操作系统上可能通过 `mutex` 来实现。

在每种情况下，获取锁的开销都很小。`TX` 事务锁在性能和基数方面都具有极高的可扩展性。`TM` 和 `DDL` 锁会在可能的情况下以限制性最小的模式应用。`Latches` 和 `enqueues` 都非常轻量且快速（`enqueues` 稍重一些，但功能更丰富）。问题只出现在设计不佳的应用程序中，它们会持有锁的时间**超过必要**，从而导致数据库阻塞。如果你设计良好，Oracle 的锁定机制将允许构建可扩展、高度并发的应用程序。

> 注意
> 我用了“超过必要”这个说法。这并不意味着你应该尽快提交（结束你的事务）。事务应该恰好是它们所需的长度——不要更长。也就是说，你的事务是你的工作单元；它是全有或全无的。你应该在工作单元完成时提交，而不是之前——也不要晚于该提交的时候！

但 Oracle 对并发的支持远不止高效的锁定。它实现了一个**多版本**（multiversioning）架构，提供了对数据的受控且高度并发的访问。多版本描述了 Oracle 能够同时物化数据的多个版本的能力，也是 Oracle 提供数据读一致视图（即，相对于某个时间点的一致结果）的机制。多版本一个相当令人愉快的副作用是，数据的读取者永远不会被数据的写入者阻塞。换句话说，写操作不会阻塞读操作。这是 Oracle 与其他数据库之间的根本区别之一。在 Oracle 中，只读信息的查询永远不会被阻塞；它永远不会与另一个会话发生死锁，也永远不会得到一个在数据库中不存在的答案。

> 注意
> 在分布式**两阶段提交**（two-phase commit）的处理过程中，有一小段时间 Oracle 会阻止对信息的读访问。由于这种处理相对罕见且是例外情况（该问题仅影响那些在准备阶段和提交阶段之间启动、并试图在提交到达之前读取数据的查询），我将不在此详细讨论。

Oracle 用于读一致性的多版本模型默认在**语句级别**（对每个查询）应用，也可以在**事务级别**应用。这意味着提交给数据库的每个 SQL 语句至少能看到数据库的一个读一致视图——如果你希望这个读一致视图达到事务级别（一组 SQL 语句），你也可以做到，正如我们将在本章的“SERIALIZABLE”部分中看到的那样。

数据库中事务的基本目的是将数据库从一个一致状态带到下一个一致状态。ISO SQL 标准定义了各种**事务隔离级别**（transaction isolation levels），这些级别定义了一个事务对另一个事务所做更改的**敏感程度**（sensitive）。敏感度越高，你的应用程序执行的事务之间数据库必须提供的隔离程度就越高。在下一节中，我们将看看 Oracle 如何通过其多版本架构以及绝对最小化的锁定来支持每一个定义的隔离级别。

## 事务隔离级别

ANSI/ISO SQL 标准定义了四种事务隔离级别，相同的事务场景可能产生不同的结果。也就是说，以相同的方式、使用相同的输入执行相同的工作，可能会根据你的隔离级别得到不同的答案。这些隔离级别是根据三个“现象”来定义的，这些现象在给定的隔离级别下是否被允许：

*   **脏读**（Dirty read）：这个术语的含义与其字面意思一样糟糕。它允许你读取未提交的，或**脏**（dirty）数据。你可以通过打开他人正在写入的操作系统文件并读取碰巧在那里的任何数据来达到这种效果。这会损害数据完整性，违反外键，并忽略唯一约束。
*   **不可重复读**（Nonrepeatable read）：这简单地意味着，如果你在时间 `T1` 读取了一行，并尝试在时间 `T2` 重新读取该行，该行可能已更改、可能已消失、可能已被更新等等。
*   **幻像读**（Phantom read）：这意味着如果你在时间 `T1` 执行了一个查询，并在时间 `T2` 重新执行它，可能已有额外的行被添加到数据库中，这将影响你的结果。这与不可重复读的不同之处在于，对于幻像读，你已经读取的数据并未被更改，而是有*更多*的数据满足了你的查询条件。

> 注意
> ANSI/ISO SQL 标准定义的是**事务级别**的特性，而不仅仅是单独的语句级别特性。在接下来的内容中，我们将检查事务级别的隔离，而不仅仅是语句级别的隔离。

SQL 隔离级别是根据是否允许上述每种现象来定义的。我注意到一个有趣的现象，SQL 标准并不强制特定的锁定方案或规定特定的行为，而是通过这些现象来描述这些隔离级别，从而允许存在许多不同的锁定/并发机制（见表 7-1）。

表 7-1
ANSI 隔离级别

| 隔离级别 | 脏读 | 不可重复读 | 幻像读 |
| --- | --- | --- | --- |
| `READ UNCOMMITTED` | 允许 | 允许 | 允许 |
| `READ COMMITTED` | – | 允许 | 允许 |
| `REPEATABLE READ` | – | – | 允许 |
| `SERIALIZABLE` | – | – | – |

Oracle 明确支持标准中定义的 `READ COMMITTED` 和 `SERIALIZABLE` 隔离级别。然而，这并未揭示全部情况。SQL 标准试图建立隔离级别，以允许在每个级别执行的查询具有不同程度的一致性。`REPEATABLE READ` 是 SQL 标准声称能保证查询结果读一致性的隔离级别。根据其定义，`READ COMMITTED` 不会给你一致的结果，而 `READ UNCOMMITTED` 是用于获取非阻塞读取的级别。

然而，在 Oracle 中，`READ COMMITTED` 具有实现读一致查询所需的所有属性。在许多其他数据库中，`READ COMMITTED` 查询可能且将会返回那些在数据库的任何时间点都从未存在过的答案。此外，Oracle 也支持 `READ UNCOMMITTED` 的**精神**。提供脏读的目的是为了提供非阻塞读取，使得查询不会被相同数据的更新阻塞，也不会阻塞这些更新。然而，Oracle 不需要脏读来实现这一目标，也不支持它们。脏读是其他数据库为了提供非阻塞读取而必须使用的一种实现方式。
