# 回滚段实际上太小了

场景是这样的：你有一个系统，其中的事务都很小。因此，只需要分配很少的回滚段空间。例如，假设以下情况属实：

*   每个事务平均产生 8KB 的回滚数据。
*   你平均每秒执行五笔这样的事务（即每秒 40KB 回滚数据，每分钟 2400KB）。
*   你有一个产生 1MB 回滚数据的事务，平均每分钟发生一次。总的来说，你每分钟产生约 3.5MB 的回滚数据。
*   你的系统配置了极其少量的回滚空间。

对于处理这些事务而言，这个数据库的回滚空间是绰绰有余的。回滚段大约平均每三到四分钟就会回绕并重用空间。如果你是基于执行数据修改的事务来调整回滚段大小的，那你的做法是正确的。

然而，在同一个环境中，你可能还有一些报表需求。其中一些查询需要运行很长时间——可能需要五分钟。问题就出在这里。如果这些查询需要运行五分钟*并且*需要看到查询开始时存在的数据视图，那么你遇到 `ORA-01555` 错误的概率就非常高。由于你的回滚段会在查询执行期间回绕，你知道自查询开始以来生成的一些回滚信息已经消失——它被覆盖了。如果你访问到一个在你开始查询前不久被修改过的数据块，该块的回滚信息将丢失，你就会收到 `ORA-01555` 错误。

这里有一个小例子。假设我们有一个表，其中包含块 1, 2, 3, …, 1,000,000。表 9-2 展示了一系列可能发生的时间。

## 长时间运行查询的时间线

| 时间 (分钟:秒) | 操作 |
| --- | --- |
| 0:00 | 我们的查询开始。 |
| 0:01 | 另一个会话 `UPDATE` 了块 1,000,000。该操作的回滚信息被记录到某个回滚段中。 |
| 0:01 | 这个 `UPDATE` 会话执行了 `COMMIT`。它生成的回滚数据仍然存在，但现在如果需要空间，它就可能被覆盖。 |
| 1:00 | 我们的查询仍在继续。当前读到块 200,000。 |
| 1:01 | 系统活动频繁。到目前为止我们已经生成了略多于 14MB 的回滚数据。 |
| 3:00 | 我们的查询仍在进行。当前大约在块 600,000 处。 |
| 4:00 | 我们的回滚段开始回绕，并重用在 0:00 查询开始时处于活动状态的空间。具体来说，我们刚刚重用了在 0:01 更新块 1,000,000 时所使用的回滚段空间。 |
| 5:00 | 我们的查询终于到达了块 1,000,000。它发现该块自查询开始以来已被修改。它去回滚段中查找该块的回滚信息，试图获取一致读。此时，它发现所需的信息已不存在。`ORA-01555` 错误被抛出，查询失败。 |

这就是问题产生的全部条件。如果你的回滚段大小使得它们在查询执行期间很有可能被重用，并且你的查询访问的数据很可能被修改，那么你反复遇到 `ORA-01555` 错误的可能性就非常高。如果出现这种情况，你必须将 `UNDO_RETENTION` 参数设置得更高，让 Oracle 自行计算需要保留多少回滚数据（这是推荐的方法；比你自己尝试计算完美的回滚段大小要容易得多），或者重新调整回滚段大小使其更大（或者数量更多）。你需要配置足够的回滚空间，使其持续时间能覆盖你的长时间运行查询。系统当初是为修改数据的事务调整大小的——你忘记了为系统的其他组件调整大小。

有两种方法来管理系统中的回滚：

### 自动回滚管理
在此方法中，通过 `UNDO_RETENTION` 参数告知 Oracle 需要保留回滚数据多长时间。Oracle 将根据并发工作负载决定创建多少个回滚段以及每个回滚段应该多大。数据库甚至可以在运行时在单个回滚段之间重新分配区，以满足 DBA 设置的 `UNDO_RETENTION` 目标。这是推荐的回滚管理方法。

### 手动回滚管理
不要使用这种方法。这是 Oracle 早期版本遗留下来的方式。

使用手动回滚管理时，DBA 必须手动设置回滚段的大小和数量。不要使用手动方法。了解这个主题的知识主要是为了在团队中给资深 DBA 留下深刻印象时有用。

在自动回滚管理下，从 `ORA-01555` 的角度看，事情就简单多了。DBA 首先创建一个可自动扩展的回滚表空间，然后告诉数据库运行时间最长的查询（或闪回操作）需要多长时间，并将该值设置在 `UNDO_RETENTION` 参数中。Oracle 将尝试至少保留回滚数据达到该时长。如果分配了足够的空间用于增长，Oracle 将扩展回滚段而不是回绕——以努力遵守 `UNDO_RETENTION` 周期。这与手动管理的回滚形成鲜明对比，后者会在可能的情况下尽快回绕并重用回滚空间。主要就是出于这个原因——对 `UNDO_RETENTION` 参数的支持——我强烈建议尽可能使用 `自动回滚管理`。这个单一参数大大降低了 `ORA-01555` 错误的可能性（当它设置得当时）。

如果你有一个固定大小的回滚表空间，那么 Oracle 会忽略 `UNDO_RETENTION` 参数。在这种配置下，你可以使用诸如“回滚指导”之类的工具来帮助你合理调整回滚表空间的大小。可以通过 Enterprise Manager 或 `DBMS_ADVISOR` PL/SQL 包访问回滚指导。调整回滚段大小时，最重要的两个考虑因素是运行时间最长的查询的长度以及你为闪回操作所需的最长间隔。

我有点过于深入 DBA 的角色了，所以我们接下来会讨论下一个案例。重要的是你要理解，这种情况下的 `ORA-01555` 错误是由于系统没有根据你的工作负载正确调整大小造成的。唯一的解决方案就是根据你的工作负载正确调整大小。这不是你的错，但既然你遇到了它，那就是你的问题。这就像在查询过程中临时空间用完一样。你要么为系统配置足够的临时空间，要么重写查询，使它们使用不需要临时空间的执行计划。

为了演示这个效应，我们可以设置一个小而有点人为的测试。我们将创建一个非常小的回滚表空间，并使用一个会话产生许多小事务，几乎可以保证它会多次回绕并重用其分配的空间——无论 `UNDO_RETENTION` 设置如何，因为我们不允许回滚表空间增长。使用这个回滚段的会话将修改一个表 `T`。它将对 `T` 进行全表扫描，并从“头”到“尾”读取它。在另一个会话中，我们将执行一个通过索引读取表 `T` 的查询。这样，它将以某种随机的方式读取表：它会读第 1 行，然后是第 1000 行，接着是第 500 行，然后是第 20,001 行，依此类推。通过这种方式，我们往往会非常随机地访问数据块，并且可能在查询处理过程中多次访问。在这种情况下，遇到 `ORA-01555` 错误的概率几乎是百分之百。

因此，在一个会话中，我们首先连接到可插拔数据库（而不是根容器），并在其中创建一个新的回滚表空间：

```sql
$ sqlplus eoda/foo@PDB1
SQL> create undo tablespace undo_small
datafile '/tmp/undo.dbf' size 2m
autoextend off;
Tablespace created.
SQL> alter system set undo_tablespace = undo_small;
System altered.
```


## ORA-01555

现在，我们将设置要查询和修改的表 `T`。注意，我们在这个表中是随机排序数据的。`CREATE TABLE AS SELECT` 倾向于将从查询中获取的行按顺序放入数据块中。我们只是把行打乱，这样它们就不会被人为地按任何顺序排序，从而随机化它们的分布：

```
SQL> drop table t purge;
Table dropped.
SQL> create table t
as
select *
from all_objects
order by dbms_random.random;
Table created.
SQL> alter table t add constraint t_pk primary key(object_id);
Table altered.
SQL> exec dbms_stats.gather_table_stats( user, 'T', cascade=> true );
PL/SQL procedure successfully completed.
```

现在我们准备好进行修改了：

```
SQL> begin
for x in ( select rowid rid from t )
loop
update t set object_name = lower(object_name) where rowid = x.rid;
commit;
end loop;
end;
/
```

当这个 PL/SQL 代码块运行时，我们将在另一个会话中运行一个查询。那个查询将读取表 `T` 并处理每条记录。它将在获取下一条记录之前花费大约 1/100 秒来处理每条记录（使用 `DBMS_LOCK.SLEEP(0.01)` 模拟）。我们将在查询中使用 `FIRST_ROWS` 提示，让它使用我们创建的索引，通过索引按 `OBJECT_ID` 排序从表中读取行。由于数据是随机插入到表中的，我们往往会相当随机地查询表中的数据块。这个块只会运行几秒钟就会失败：

```
SQL> declare
cursor c is
select /*+ first_rows */ object_name
from t
order by object_id;
l_object_name t.object_name%type;
l_rowcnt      number := 0;
begin
open c;
loop
fetch c into l_object_name;
exit when c%notfound;
dbms_lock.sleep( 0.01 );
l_rowcnt := l_rowcnt+1;
end loop;
close c;
exception
when others then
dbms_output.put_line( 'rows fetched = ' || l_rowcnt );
raise;
end;
/
rows fetched = 191
declare
*
ERROR at line 1:
ORA-01555: snapshot too old: rollback segment number 39 with name
"_SYSSMU39_2997928315$" too small
```

如你所见，它在处理了少量记录后就失败了，出现了 `ORA-01555: snapshot too old` 错误。为了纠正这个问题，我们要确保完成两件事：

*   数据库中的 `UNDO_RETENTION` 设置至少要足够长，以便这个读取过程能够完成。这将允许数据库增长撤销表空间，以保存足够的撤销数据供我们完成操作。
*   允许撤销表空间增长，或者你手动为其分配更多的磁盘空间。

对于这个例子，我确定我的长时间运行过程大约需要 720 秒完成（我的表中有大约 72,000 条记录，所以每行 0.01 秒，总共 720 秒）。我的 `UNDO_RETENTION` 设置为 900（单位是秒，所以撤销保留时间约为 15 分钟）。我修改了撤销表空间的数据文件，允许它一次增长 1MB，最大到 2GB：

```
SQL> alter database
datafile '/tmp/undo.dbf'
autoextend on
next 1m
maxsize 2048m;
Database altered.
```

当我再次同时运行这些过程时，两者都运行完成了。这次撤销表空间的数据文件增长了，因为它被允许增长，而且我设置的撤销保留策略也要求这样做。

```
SQL> select bytes/1024/1024
from dba_data_files
where tablespace_name = 'UNDO_SMALL';
BYTES/1024/1024
```

所以，我们没有收到错误，而是成功完成了，并且撤销增长到足以满足我们的需求。确实，在这个例子中，出现错误纯粹是因为我们通过索引读取表 `T` 并在整个表中进行了随机读取。如果我们当时快速进行了全表扫描，很可能*在这种特定情况下*不会收到 `ORA-01555` 错误。这是因为 `SELECT` 和 `UPDATE` 都会对 `T` 进行全表扫描，并且 `SELECT` 在扫描过程中很可能可以领先于 `UPDATE`（`SELECT` 只需要读取，但 `UPDATE` 必须读取、更新，因此可能更慢）。通过进行随机读取，我们增加了 `SELECT` 需要读取一个数据块的可能性，而这个数据块是 `UPDATE` 在很久之前修改并提交的。这正好展示了 `ORA-01555` 错误有些阴险的特性。它的出现取决于并发会话如何访问和操作底层表。

### 延迟块清除 (Delayed Block Cleanout)

`ORA-01555` 错误的这个原因很难完全消除，但它也很少发生，因为其发生的条件并不经常出现。我们已经讨论过块清除机制，但总结一下，它是这样一个过程：在一个块被修改后，下一个访问该块的会话可能必须检查上次修改该块的事务是否仍然处于活动状态。一旦该过程确定事务不再活动，它就会清除该块，以便下一个访问它的会话不必再经历相同的过程。为了清除块，Oracle 会确定先前事务使用的撤销段（从块的头中），然后确定撤销头是否指示事务已提交，以及（如果已提交）是何时提交的。这种确认通过两种方式之一完成。一种方式是，Oracle 可以确定事务是在很久以前提交的，即使其事务槽在撤销段事务表中已被覆盖。另一种方式是，`COMMIT` SCN 仍然在撤销段的事务表中，这意味着事务是不久前提交的，其事务槽尚未被覆盖。

要因延迟块清除而收到 `ORA-01555` 错误，必须满足以下所有条件：

*   进行了一次修改并 `COMMIT` 提交，并且块没有被自动清除（例如，该事务修改的块数超过了 SGA 块缓冲区缓存的百分之十）。
*   这些块没有被另一个会话接触，并且在我们的不幸查询（稍后显示）访问到它之前也不会被接触。
*   一个长时间运行的查询开始。这个查询最终将读取一些来自先前的那些块。该查询开始于 SCN `t1`，即它必须回滚数据以达到读一致性的读一致 SCN。当我们开始时，修改事务的事务条目仍在撤销段事务表中。
*   在查询期间，系统中进行了许多提交。这些事务不会触及有问题的块（如果它们触及了，就不会有即将发生的问题，因为它们会清除旧事务——解决了清除问题）。
*   由于大量的 `COMMIT` 操作，撤销段中的事务表循环并重用了槽。最重要的是，原始修改事务的事务条目被循环覆盖和重用。此外，系统重用了撤销段区，从而阻止了对撤销段头块本身进行一致读。
*   另外，由于大量的提交，现在记录在撤销段中的最低 SCN 已经超过了 `t1`（它高于查询的读一致 SCN）。

当我们的查询访问到在它开始之前已被修改并提交的块时，它就遇到了麻烦。通常，它会前往该块指向的撤销段，并找到修改它的事务的状态（换句话说，它会找到该事务的 `COMMIT` SCN）。如果 `COMMIT` SCN 小于 `t1`，我们的查询就可以使用这个块。如果 `COMMIT` SCN 大于 `t1`，我们的查询就必须回滚那个块。然而，问题在于，在这种情况下，我们的查询无法确定该块的 `COMMIT` SCN 是大于还是小于 `t1`。它不确定是否可以使用那个块映像。于是就导致了 `ORA-01555` 错误。
