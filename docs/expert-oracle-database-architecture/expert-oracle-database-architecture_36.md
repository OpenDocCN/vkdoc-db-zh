# 分布式事务

Oracle 的一个非常出色的特性是能够透明地处理分布式事务。我可以在单个事务范围内更新许多不同数据库中的数据。当我提交时，要么更新在所有实例中都提交，要么一个都不提交（它们都会被回滚）。我不需要额外的代码来实现这一点；我只需执行 "commit"。

## 数据库链接与访问

Oracle 中分布式事务的关键是 `database link`（数据库链接）。`database link` 是一个数据库对象，它描述了如何从你的实例登录到另一个实例。然而，本节的目的不是涵盖 `database link` 命令的语法（这在 *Oracle Database SQL Language Reference* 手册中有完整记录），而是让你了解它的存在。一旦你设置好了 `database link`，访问远程对象就很简单了：

```sql
select * from T@another_database;
```

这将从由数据库链接 `ANOTHER_DATABASE` 定义的数据库实例中的表 `T` 进行选择。通常，你会通过为其创建视图或同义词来"隐藏" `T` 是远程表的事实。例如，我可以执行以下命令，然后像访问本地表一样访问 `T`：

```sql
create synonym T for T@another_database;
```

## 执行分布式事务

现在，我已经设置好了这个 `database link`，并且可以读取一些表了，我也能够修改它们（当然，前提是我拥有适当的权限）。执行分布式事务现在与本地事务没有什么不同。我只需这样做：

```sql
update local_table set x = 5;
update remote_table@another_database set y = 10;
commit;
```

就是这样。Oracle 将在两个数据库中都提交，或者都不提交。它使用两阶段提交协议（2PC）来实现这一点。2PC 是一种分布式协议，允许影响多个不同数据库的修改被原子性地提交。它试图在提交之前尽可能缩小分布式故障的窗口。

## 两阶段提交（2PC）协议

在涉及多个数据库的 2PC 中，其中一个数据库——通常是客户端最初登录的那个——将充当分布式事务的**协调者**。这个站点将询问其他站点是否准备好提交。实际上，这个站点会去找其他站点，要求它们做好提交准备。其他每个站点都会将其"准备就绪状态"报告为 `YES` 或 `NO`。如果任何一个站点投票 `NO`，则整个事务回滚。如果所有站点都投票 `YES`，则站点协调者向每个站点广播一条消息，使提交永久生效。

这限制了可能发生严重错误的窗口期。在 2PC 的"投票"阶段之前，任何分布式错误都会导致所有站点回滚。事务的结果是确定无疑的。在提交或回滚的命令发出之后，分布式事务的结果再次变得确定无疑。只有在协调者收集投票的那个非常短暂的窗口期内，如果发生故障，结果才可能是不确定的。

## 处理不确定的事务

假设，例如，我们有三个站点参与事务，其中站点 1 是协调者。站点 1 已要求站点 2 准备提交，站点 2 已照做。然后站点 1 要求站点 3 准备提交，站点 3 也照做了。此时，只有站点 1 知道事务的结果，它现在负责将结果广播给其他站点。如果此时发生错误——网络故障、站点 1 断电，无论什么情况——站点 2 和 3 将被挂起。它们将拥有一个所谓的 `in-doubt distributed transaction`（不确定的分布式事务）。2PC 协议试图尽可能缩小错误窗口，但无法完全消除它。站点 2 和 3 必须保持该事务打开，等待来自站点 1 的结果通知。

`RECO` 后台进程的功能就是解决这个问题。这也是带 `FORCE` 选项的 `COMMIT` 和 `ROLLBACK` 发挥作用的地方。如果问题的原因是站点 1、2 和 3 之间的网络故障，那么站点 2 和 3 的 DBA 实际上可以致电站点 1 的 DBA，询问结果，然后酌情手动应用提交或回滚。

## 分布式事务的限制

在分布式事务中可以做的事情有一些限制，但不是很多，而且这些限制是合理的（至少对我来说似乎合理）。主要限制如下：

*   你不能通过 `database link` 发出 `COMMIT`。也就是说，你不能执行 `COMMIT@remote_site`。你只能从事务发起的站点提交。

*   你不能通过 `database link` 执行 DDL。这是前一个问题的直接结果。DDL 会提交事务。你只能从事务发起的站点提交；因此，你不能通过 `database link` 执行 DDL。

*   你不能通过 `database link` 发出 `SAVEPOINT`。简而言之，你不能通过 `database link` 发出任何事务控制语句。所有事务控制都继承自最初打开 `database link` 的会话；在分布式事务的各个实例中，你不能有不同的事务控制。

缺乏通过 `database link` 进行事务控制是合理的，因为只有发起站点才拥有参与事务的所有方的列表。在我们的三站点配置中，如果站点 2 尝试提交，它将无法知道站点 3 也参与了。在 Oracle 中，只有站点 1 可以发出提交命令。此时，允许站点 1 将分布式事务控制的责任委托给另一个站点。

我们可以通过设置站点的 `COMMIT_POINT_STRENGTH`（一个参数）来影响哪个站点将成为实际的提交站点。`COMMIT_POINT_STRENGTH` 将一个相对重要性级别与分布式事务中的服务器关联起来。服务器越重要（数据需要越高的可用性），它协调分布式事务的可能性就越大。如果你需要在生产机和测试机之间执行分布式事务，你可能希望这样做。由于事务协调者对事务结果*从不*存在疑问，所以最好让生产机协调分布式事务。你不太关心测试机是否有一些未完成的事务和锁定的资源。但你肯定关心生产机是否这样。

## 在分布式环境中执行 DDL

无法通过 `database link` 执行 DDL 实际上并没有那么糟糕。首先，DDL 很少见。你在安装或升级时做一次。生产系统不执行 DDL（嗯，它们*不应该*执行 DDL）。其次，有一种方法可以通过 `database link` 执行 DDL，某种意义上，使用作业队列工具 `DBMS_SCHEDULER`。你不必尝试通过链接执行 DDL，而是使用链接调度一个远程作业，在你提交后立即执行。这样，作业在远程机器上运行，不是分布式事务，并且可以执行 DDL。事实上，Oracle 复制服务执行分布式 DDL 进行模式复制就是使用这种方法。

## 自治事务

自治事务允许你创建一个“事务中的事务”，该事务可以独立于其父事务提交或回滚更改。它们使你能够暂停当前正在执行的事务，启动一个新事务，执行一些工作，然后提交或回滚——所有这些都不会影响当前正在执行的事务状态。自治事务为 PL/SQL 中的事务控制提供了一种新方法，可用于：

*   顶层匿名块
*   局部（过程中的过程）、独立或打包的函数和过程
*   对象类型的方法
*   数据库触发器

在我们探讨自治事务如何工作之前，我想强调一点：这种类型的事务功能强大，因此如果使用不当，也是危险的工具。真正需要自治事务的情况确实非常罕见。对于任何使用了它们的代码，我都会持高度怀疑态度——这类代码需要接受额外审查。使用它们时，很容易无意中在系统中引入逻辑数据完整性问题。在接下来的章节中，我们将在了解其工作原理之后，讨论何时可以安全地使用它们。

### 自治事务如何工作

演示自治事务的行为及其后果的最佳方式是通过示例。我们将创建一个简单的表来保存一条消息：

```
SQL> create table t ( msg varchar2(25) );
Table created.
```

接下来，我们将创建两个过程，每个过程都简单地将其名称 `INSERT` 到消息表并提交。然而，其中一个过程是常规过程，另一个则被编码为自治事务。我们将使用这些对象来展示在不同情况下哪些工作会持久化（被提交）在数据库中。

首先是 `AUTONOMOUS_INSERT` 过程：

```
$ sqlplus eoda/foo@PDB1
SQL> create or replace procedure Autonomous_Insert
as
pragma autonomous_transaction;
begin
insert into t values ( 'Autonomous Insert' );
commit;
end;
/
Procedure created.
```

注意这里使用了编译指示 `AUTONOMOUS_TRANSACTION`。这个指令告诉数据库，该过程在执行时，应作为一个新的自治事务执行，独立于其父事务。

注意

*编译指示*（pragma）只是一个编译器指令，一种指示编译器执行某些编译选项的方法。还有其他可用的编译指示。请参考 *Oracle Database PL/SQL Language Reference* 手册；你可以在其索引中找到它们的列表。

然后是“常规的” `NONAUTONOMOUS_INSERT` 过程：

```
SQL> create or replace procedure NonAutonomous_Insert
as
begin
insert into t values ( 'NonAutonomous Insert' );
commit;
end;
/
Procedure created.
```

现在，让我们观察 *非自治* 事务在 PL/SQL 匿名块中的行为：

```
SQL> begin
insert into t values ( 'Anonymous Block' );
NonAutonomous_Insert;
rollback;
end;
/
PL/SQL procedure successfully completed.
SQL> select * from t;
MSG
--------------------------------------------------------------------------------
Anonymous Block
NonAutonomous Insert
```

如你所见，匿名块执行的工作（它的 `INSERT`）被 `NONAUTONOMOUS_INSERT` 过程 *提交* 了。两行数据都被提交了，因此 `ROLLBACK` 命令没有可以回滚的内容。将此与自治事务过程的行为进行对比：

```
SQL> delete from t;
2 rows deleted.
SQL> commit;
Commit complete.
SQL> begin
insert into t values ( 'Anonymous Block' );
Autonomous_Insert;
rollback;
end;
/
PL/SQL procedure successfully completed.
SQL> select * from t;
MSG
--------------------------------------------------------------------------------
Autonomous Insert
```

在这里，只有在自治事务中执行并提交的工作才持久化。匿名块中的 `INSERT` 被第 4 行的 `ROLLBACK` 语句回滚了。自治事务过程的 `COMMIT` 对匿名块中启动的父事务没有影响。简而言之，这概括了自治事务的本质及其作用。

总结一下，如果你在“常规”过程中执行 `COMMIT`，它不仅会使该过程自己的工作持久化，还会使该会话中所有未提交的工作持久化。然而，在具有自治事务的过程中执行 `COMMIT`，只会使该过程自身的工作持久化。

### 何时使用自治事务

Oracle 数据库在内部支持自治事务已有相当长的时间。我们经常以递归 SQL 的形式看到它们。例如，当从序列中选择时，可能会执行一个递归事务，以便你能够立即在 `SYS.SEQ$` 表中递增序列。为支持你的序列而对 `SYS.SEQ$` 表的更新会立即提交并对其他事务可见，但你的事务尚未提交。此外，如果你回滚你的事务，序列的递增仍然保留；它不会随你的事务一起回滚，因为它已经被提交了。空间管理、审计和其他内部操作也以类似的递归方式执行。

现在这个功能已开放供所有人使用。然而，我发现自治事务在现实世界中的合法用途 *非常有限*。我一次又一次地看到它们被用作解决诸如触发器中的变异表约束等问题的变通方法。然而，这几乎总是导致数据完整性问题，因为变异表的原因是尝试读取触发器正在触发的表。嗯，通过使用自治事务，你可以查询该表，但你现在查询该表时无法看到你自己的更改（这正是变异表约束最初试图阻止的；该表正处于修改过程中，因此查询结果会不一致）。你基于该触发器中的查询所做出的任何决策都将是可疑的——你当时读取的是“旧”数据。

自治事务一个潜在的有效用途是在自定义审计中，但我强调“潜在有效”这几个字。在数据库中审计信息有比通过自定义编写触发器更高效的方法。例如，你可以使用 `DBMS_FGA` 包或直接使用 `AUDIT` 命令本身。

应用程序开发人员经常向我提出的一个问题是：“我如何才能以一种即使我的 PL/SQL 例程的工作被回滚也能持久化的方式，在我的 PL/SQL 例程中记录错误？”之前，我们描述了 PL/SQL 语句是 *原子性* 的——它们要么完全成功，要么完全失败。如果我们在 PL/SQL 例程中记录错误，默认情况下，我们记录的错误信息会在 Oracle 回滚我们的语句时一起回滚。自治事务允许我们改变这种行为，使得我们的错误日志信息即使在其余部分工作被回滚时也能持久保存。

让我们首先设置一个要使用的简单错误日志表；我们将记录错误的时间戳、错误消息和 PL/SQL 错误堆栈（用于精确定位错误来源）：

```
$ sqlplus eoda/foo@PDB1
SQL> create table error_log
( ts   timestamp,
err1 clob,
err2 clob )
/
Table created.
```

现在我们需要将错误记录到该表的 PL/SQL 例程。我们可以使用这个小例子：

```
SQL> create or replace
procedure log_error
( p_err1 in varchar2, p_err2 in varchar2 )
as
pragma autonomous_transaction;
begin
insert into error_log( ts, err1, err2 )
values ( systimestamp, p_err1, p_err2 );
commit;
end;
/
Procedure created.
```

这个例程的“魔力”在于第 5 行，我们使用了 `pragma autonomous_transaction` 指令来通知 PL/SQL，我们希望这个子程序启动一个新的事务，在其中执行一些工作并提交——而不影响当前正在处理的任何其他事务。第 9 行的 `COMMIT` 只能影响这个 `LOG_ERROR` 过程执行的 SQL。

现在让我们来测试一下。为了让它更有趣，我们将创建几个相互调用的过程：


```
SQL> create table t ( x int check (x>0) );
表已创建。
SQL> create or replace procedure p1( p_n in number )
as
begin
-- 此处为某些代码
insert into t (x) values ( p_n );
end;
/
过程已创建。
SQL> create or replace procedure p2( p_n in number )
as
begin
-- 代码
-- 代码
p1(p_n);
end;
/
过程已创建。
```

然后我们将从一个匿名块调用这些例程：

```
SQL> begin
p2( 1 );
p2( 2 );
p2( -1);
exception
when others
then
log_error( sqlerrm, dbms_utility.format_error_backtrace );
RAISE;
end;
/
begin
*
第 1 行出现错误:
ORA-02290: 违反检查约束条件 (EODA.SYS_C007751)
ORA-06512: 在第 9 行
ORA-06512: 在 "EODA.P1", 第 5 行
ORA-06512: 在 "EODA.P2", 第 6 行
ORA-06512: 在第 4 行
```

现在，我们可以看到代码执行失败了（你希望那个错误被返回，因此第 9 行的 `RAISE`）。我们可以验证 Oracle 撤销了我们的操作（我们知道对过程 `P2` 的前两次调用是成功的；值 1 和 2 已成功插入到表 `T` 中）：

```
SQL> select * from t;
未选中任何行
```

但我们也能够验证我们的错误日志信息已经持久化，并且实际上已经被提交：

```
SQL> rollback;
回退完成。
SQL> set long 1000000
SQL> select * from error_log;
TS

ERR1

ERR2

2021 年 5 月 30 日 下午 02.59.41.969061
ORA-02290: 违反检查约束条件 (EODA.SYS_C0013043)
ORA-06512: 在 "EODA.P1", 第 5 行
ORA-06512: 在 "EODA.P2", 第 6 行
ORA-06512: 在第 4 行
```

根据我的经验，这是自治事务唯一真正有效的用途——以可以独立于父事务提交的方式记录错误或信息性消息。

## 总结

在本章中，我们探讨了 Oracle 中事务管理的诸多方面。事务是将数据库与文件系统区分开来的主要特性之一。理解它们的工作原理以及如何使用它们，对于在任何数据库中正确实现应用程序都是必要的。理解 Oracle 中所有语句都是原子性的（包括其副作用），并且这种原子性被扩展到了存储过程，这一点至关重要。我们看到了在 PL/SQL 块中 `WHEN OTHERS` 异常处理程序的位置如何能极大地影响数据库中发生的变化。作为数据库开发人员，深入理解事务的工作原理至关重要。

我们审视了完整性约束（唯一键、检查约束等）与 Oracle 中事务之间有些复杂的交互。我们讨论了 Oracle 通常如何在语句执行后立即处理完整性约束，但如果我们希望，也可以将此约束验证推迟到事务结束时进行。当被修改的表彼此都相互依赖时，这个特性对于实现复杂的多表更新至关重要——级联更新就是一个例子。

我们接着探讨了人们从那些“支持”而非“促进”使用事务的数据库工作中容易沾染的一些不良事务习惯。我们审视了事务的基本规则：它们应尽可能短，但也要根据需要尽可能长。*数据完整性驱动事务大小*——这是从本章中要带走的一个关键概念。唯一应该驱动你事务大小的是支配你系统的业务规则。不是回滚空间，不是锁——是业务规则。

我们介绍了分布式事务，以及它们与单数据库事务的不同之处。我们探讨了分布式事务中施加给我们的限制，并讨论了它们存在的原因。在构建分布式系统之前，你需要理解这些限制。在单实例中有效的方法在分布式数据库中可能行不通。

本章最后探讨了自治事务，涵盖了它们是什么，更重要的是，何时应该以及何时不应该使用它们。我想再次强调，自治事务在现实世界中的合法用例极为罕见。如果你发现它成了你经常使用的一个特性，你需要非常仔细地审视原因。
