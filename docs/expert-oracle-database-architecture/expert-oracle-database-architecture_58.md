# 临时表的属性和限制

临时表可以具有永久表的许多属性。它们可以有触发器、检查约束、索引等。永久表具有而临时表不支持的功能包括：

*   它们不能有引用完整性约束。它们既不能成为外键的*目标*，也不能在它们身上定义外键。

*   它们不能有 `NESTED TABLE` 类型的列。

*   它们不能是 IOTs（索引组织表）。

*   它们不能属于任何类型的集群。

*   它们不能被分区。

*   它们不能通过 `ANALYZE` table 命令生成统计信息。

在任何数据库中使用临时表的一个缺点是，优化器通常没有关于它的真实统计信息。对于 Oracle 的*基于成本的优化器*（`cost-based optimizer`，`CBO`）而言，有效的统计信息对于优化器的成功（或失败）至关重要。在没有统计信息的情况下，优化器将猜测数据的分布、数据量以及索引的选择性。当这些猜测错误时，为大量使用临时表的查询生成的查询计划可能不是最优的。在许多情况下，正确的解决方案是根本不使用临时表，而是使用一个 `INLINE VIEW`（关于 `INLINE VIEW` 的示例，请参考刚刚运行的最后一条 `SELECT` 语句——它包含两个内联视图）来代替它。这样，Oracle 就可以访问表的所有相关统计信息，并得出最优的计划。

我经常发现人们使用临时表是因为他们在其他数据库中学到，在单个查询中连接太多表是件坏事。对于 Oracle 开发来说，这种做法必须摒弃。与其试图智胜优化器并将一个本该是单个的查询拆分成三四个查询，将它们的子结果存入临时表然后再合并这些临时表，你不如直接编写一个能回答原始问题的单个查询。在单个查询中引用多个表是可以的；Oracle 不需要为此目的使用临时表这个“拐杖”。
