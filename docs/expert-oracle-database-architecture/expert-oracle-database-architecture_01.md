# 开发成功的 Oracle 应用程序

我大部分时间都在与 Oracle 数据库软件打交道，更具体地说，是与使用该软件的人们合作。在过去的十几年里，我参与了许多项目——有成功的，也有彻底失败的——如果要将我的经验概括为几条 broad 的陈述，它们会是：

*   围绕数据库构建的应用程序——依赖于数据库——其成功与否取决于它如何使用数据库。作为推论，所有应用程序都是围绕数据库构建的；我想不出任何一个有用的应用程序不将数据持久存储在某个地方。

*   应用程序来了又走。然而，*数据* 永存。关键不在于构建应用程序；而在于这些应用程序之下的数据。

*   开发团队的核心必须有一批精通数据库的编码人员，他们负责确保数据库逻辑健全，并且系统从第一天起就构建为高性能。事后调优——在部署之后调优——意味着你当初就没有那样构建它。

这些陈述可能看起来显而易见得惊人，但根据我的经验，太多人把数据库当作一个黑盒子来对待——好像是他们不需要了解的东西。也许他们有一个结构化查询语言（`SQL`）生成器，可以让他们免于学习 `SQL` 的辛苦。

### 注意

当我在本书中提到“`SQL`”时，指的是一个结构化查询语言工具，例如 Oracle 的 `SQL*Plus` 实用程序。我之所以指出这一点，是因为有时微软的 `SQL Server` 也被称为“`SQL`”，尤其是在广泛使用微软工具的机构中。

也许他们认为自己可以像使用平面文件一样使用它，并进行“键控读取”。无论他们假设什么，我可以告诉你，这种思路几乎肯定是被误导了；你根本不可能逃避对数据库的理解。本章将讨论 *为什么* 你需要了解数据库，特别是为什么你需要理解：

*   数据库架构、其工作原理以及它的形态
*   并发控制是什么以及它们对你的意义
*   如何从第一天起就调优你的应用程序
*   有些东西在数据库中是如何实现的，这不一定与你认为它们应该被实现的方式相同
*   你的数据库已经提供了哪些功能，以及为什么通常使用已提供的功能比自己构建更好
*   为什么你可能需要的不仅仅是 `SQL` 的肤浅知识
*   数据库管理员（`DBA`）和开发人员是同一个团队，而不是处处试图智胜对方的敌对阵营

现在，这看起来像是开始之前需要学习的一长串东西，但请考虑一下这个类比：如果你正在一个全新的操作系统（`OS`）上开发一个高度可扩展的企业应用程序，你首先会做什么？希望你的回答是：“弄清楚这个新 `OS` 是如何工作的，程序将如何在上面运行，等等。”如果那不是你的答案，你很可能会失败。

例如，考虑 `Windows` 与 `UNIX/Linux`。如果你是一个长期的 `Windows` 程序员，被要求在 `UNIX/Linux` 平台上开发一个新应用程序，你将必须重新学习一些东西。内存管理方式不同。构建服务器进程的方式也有显著差异——在 `Windows` 下，你会开发一个单一进程，一个包含许多线程的单一可执行文件。在 `UNIX/Linux` 下，你不会开发一个独立的单一可执行文件；你会有许多进程协同工作。诚然，`Windows` 和 `UNIX/Linux` 都是操作系统。它们都向开发者提供许多相同的服务——文件管理、内存管理、进程管理、安全性等等。然而，它们在架构上非常不同——你在 `Windows` 环境中学到的许多东西并不适用于 `UNIX/Linux`（反之亦然，为了公平起见）。你必须“忘掉旧知识”才能成功。你的数据库环境也是如此。

对于在操作系统上原生运行的应用程序是如此，对于将在数据库上运行的应用程序也是如此：理解该数据库对你的成功至关重要。如果你不理解你的特定数据库做什么或它是如何做的，你的应用程序将会失败。如果你假设因为你的应用程序在 `SQL Server` 上运行良好，它就必然会在 `Oracle` 上运行良好，那么你的应用程序很可能会失败。而且，公平地说，反之亦然——一个可扩展的、开发良好的 `Oracle` 应用程序，不经过重大的架构修改，不一定能在 `SQL Server` 上运行。正如 `Windows` 和 `UNIX/Linux` 都是操作系统但根本不同一样，`Oracle` 和 `SQL Server`（这里几乎可以指出任何数据库）都是数据库，但也根本不同。


## 我的方法

在开始之前，我认为有必要让你了解我解决问题的方法。我倾向于采用以数据库为中心的方法来处理问题。如果能在数据库里完成，我就会在数据库里做。这有几个原因——首要原因是我知道，如果我在数据库中构建功能，我就可以在任何地方`部署`它。据我所知，没有哪个流行且商业上可行的服务器操作系统不支持`Oracle`——从 Windows 到数十种 UNIX/Linux 系统——都可以使用完全相同的`Oracle`软件和选项。我经常在我的笔记本电脑上，在虚拟机中运行 UNIX/Linux 或 Windows 系统下的`Oracle 21c、19c 和 12c`来构建和测试解决方案。然后，我可以在运行相同数据库软件但操作系统的不同服务器上部署它们。当我不得不在数据库之外实现一个功能时，我发现很难将其部署到我想要的任何地方。`Java`语言对许多人具有吸引力的主要特性之一——即他们的程序总是在相同的虚拟环境（`Java`虚拟机`JVM`）中编译，因此具有高度的可移植性——正是这个特性让数据库对我具有吸引力。数据库就是`我的`虚拟机。它是`我的`虚拟操作系统。

所以，我尽量在数据库里完成所有事情。如果我的需求超出了数据库环境所能提供的范围，我就会在数据库外用`Java`来实现。这样，几乎所有的操作系统复杂性都会对我隐藏。我仍然需要了解`我的`“虚拟机”（`Oracle`，偶尔是`JVM`）是如何工作的——你需要知道你使用的工具——但它们反过来会为我处理在特定操作系统上如何最好地完成事情。

因此，仅仅了解这一个“虚拟操作系统”的复杂性，就能让你构建出在许多操作系统上都能良好运行和扩展的应用程序。我并不是说你可以对底层的操作系统完全无知，而是说，作为一个构建数据库应用程序的软件开发者，你可以与之隔离开来，并且不必处理它的许多细节。你的`DBA`，负责运行`Oracle`软件，会比你更了解操作系统（如果他们不了解，请换一个新的`DBA`！）。如果你开发客户端-服务器软件，并且你的大部分代码在数据库之外，也在`VM`（Java 虚拟机可能是最流行的`VM`）之外，那么你当然又得关心你的操作系统了。

对于开发数据库软件，我有一个非常简单的箴言，多年来始终如一：

*   如果可能，你应该在一个`SQL`语句中完成。信不信由你，这几乎是总是可能的。随着时间推移，这一点甚至更正确。`SQL`是一门极其强大的语言。
*   如果你无法在一个`SQL`语句中完成，就在`PL/SQL`中完成——但使用尽可能少的`PL/SQL`！遵循那句老话：“代码越多 = 错误越多，代码越少 = 错误越少。”
*   如果你无法在`PL/SQL`中完成，就使用`C`外部过程。这通常是在需要原始速度或使用用`C`编写的第三方`API`时的方法。
*   如果你无法在`C`外部例程中完成，你可能需要认真思考一下为什么你必须做这件事。

在本书中，你将看到上述理念的实现。我们将使用`PL/SQL`——以及`PL/SQL`中的对象类型——来完成`SQL`本身无法完成或无法高效完成的事情。`PL/SQL`已经存在了很长时间——截至 2022 年，它已经经历了超过 34 年的调优；事实上，早在`Oracle 10g`中，`PL/SQL`编译器本身就被重写，首次成为了一个优化编译器。你会发现没有其他语言与`SQL`如此紧密地耦合，也没有任何语言在交互方面针对`SQL`进行了如此优化。在`PL/SQL`中使用`SQL`是一件非常自然的事情——而在从 Visual Basic 到`Java`的几乎所有其他语言中，使用`SQL`都可能感觉笨拙。它从未让人感觉“自然”——它不是语言本身的延伸。当`PL/SQL`力不从心时——在当今的数据库版本下这极其罕见——我们将使用`Java`。偶尔，我们会用`C`做一些事情，但通常只在`C`是唯一选择，或需要`C`提供的原始速度时。通常，这最后一个原因会随着`Java`的本地编译——将`Java`字节码转换为你平台上特定于操作系统的对象代码的能力——而消失。这使得`Java`在许多情况下能像`C`一样快地运行。

## 黑盒方法

我有一个想法，它源自亲身经验（意思是我自己犯过这个错误），即为什么以数据库为后端的软件开发工作经常失败。需要明确的是，我这里包括那些可能没有被记录为失败，但因为需要进行重大重写、重新架构或调优工作，而比原计划花费更长时间推出和部署的项目。就我个人而言，我认为这些延迟的项目就是失败：它们大多数本可以按计划（甚至更快）完成。

失败最常见的原因是缺乏对数据库的实用知识——对所使用的基本工具缺乏基本的理解。黑盒方法涉及一种有意识的决定，以保护开发人员免受数据库的影响。实际上，他们被鼓励不去了解关于它的任何事情！在许多情况下，他们被阻止去利用它。采用这种方法的原因似乎与`FUD`（恐惧、不确定和怀疑）有关。开发人员听说数据库“很难”，`SQL`、事务和数据完整性“很难”。解决方案是：不要让任何人做任何困难的事情。他们把数据库当作一个黑盒，并让某个软件工具生成所有代码。他们试图用多层保护将自己隔离起来，这样他们就不必接触这个“困难”的数据库。

这是我从未能够理解的一种数据库开发方法，部分原因在于，对我来说，学习`Java`和`C`比学习数据库背后的概念要困难得多。我现在对`Java`和`C`相当熟练，但让我能够熟练使用它们所需的实践经验，远比让我能够熟练使用数据库要多得多。对于数据库，你需要了解它的工作原理，但你不必知道所有内部细节。在用`C`或`Java/J2EE`编程时，你确实需要知道所有内部细节——而这些都是`庞大`的语言。

如果你正在构建一个数据库应用程序，`最重要的软件是数据库。` 一个成功的开发团队会认识到这一点，并希望他们的成员了解它，专注于它。很多次，我参与的项目情况几乎正好相反。一个典型的情况如下：

*   开发人员接受了完全培训，学习他们用于构建前端的`GUI`工具或语言（如`Java`）。在许多情况下，他们接受了数周甚至数月的培训。
*   团队没有接受过任何`Oracle`培训，也没有任何`Oracle`经验。大多数人没有任何数据库经验。他们还会被要求做到“数据库独立”——这是一项由于多种原因而无法实现的指令（来自管理层的命令或通过理论的学术指导学到的）。最明显的一个原因是，他们对数据库是什么或数据库能做什么了解得不够多，甚至无法找到它们之间的最低共同点。
*   开发人员遇到了大量的性能问题、数据完整性问题、挂起问题等等（但界面非常漂亮）。

由于这些不可避免的性能问题，我现在被请来帮助解决困难（过去，作为一个学习中的开发人员，我有时就是造成这类问题的原因）。


即使到了今天，我仍经常发现数据库应用程序的开发者们没有花时间阅读文档。在我的网站 `asktom.oracle.com` 上，我时常会收到类似“……的语法是什么”的问题，还附带一句“我们没有文档，所以请直接告诉我们”。我拒绝直接回答许多这类问题，而是将他们指向任何人在世界任何角落都可以免费访问的在线文档。在过去的 15 年里，像“我们没有文档”或“我们无法访问资源”这样的借口已经站不住脚了。像 [`www.oracle.com/technical-resources`](http://www.oracle.com/technical-resources)（Oracle 技术资源，前身为 Oracle 技术网络）这样的网站，让不把一整套文档随时备在手边变得不可原谅！如今，每个人都能访问所有文档；他们只需要去阅读，或者——更简单地——谷歌一下。

开发*数据库应用程序*的开发者应该被隔绝于数据库之外，这个想法本身就让我感到惊讶，但这种态度依然存在。许多人仍然坚持认为开发者没有时间接受数据库培训，而且基本上，他们不应该了解数据库的任何事情。为什么？嗯，我不止一次听到这样的说法：“但 `Oracle` 是世界上最具可扩展性的数据库，我的员工不需要学习它，它自己就能工作。”这倒是真的；`Oracle` 是世界上最具可扩展性的数据库。然而，我可以用 `Oracle` 很容易地写出性能不佳（无法扩展）的代码——如果说更容易的话——就像我可以用 `Oracle` 写出良好、可扩展的代码一样。你可以把 `Oracle` 替换成任何软件，情况也是如此。这是一个事实：写性能差的应用程序比写性能好的应用程序要容易。如果你不知道自己在做什么，在世界上最具可扩展性的数据库中构建一个单用户系统有时会过于简单。数据库是一种工具，任何工具的不当使用都可能导致灾难。你会拿一个胡桃夹子当锤子来砸核桃吗？你可以这么做，但这不是该工具的正当用途，结果会一团糟（而且很可能弄伤手指）。对你的数据库保持无知，也可能导致类似的效果。

我曾被叫去参与一个陷入困境的项目。开发者们遇到了严重的性能问题——看起来他们的系统正在串行化许多事务，也就是说——不是许多人并发工作，而是每个人都在排一个非常长的队伍，等待前面所有人完成。应用架构师向我介绍了他们系统的架构——经典的三层方法。他们让 Web 浏览器与运行 Java 服务器页面（`JSPs`）的中间层应用服务器通信。`JSPs` 又会利用另一层——企业 JavaBeans（`EJBs`）——来执行所有的 `SQL`。`EJBs` 中的 `SQL` 是由第三方工具生成的，并且是以数据库无关的方式完成的。

现在，在这个系统中，诊断任何问题都非常困难，因为没有代码被插桩或可追踪。插桩代码是一门精细的艺术，它使得开发的每两行代码中就有一行是某种调试代码——这样当你面临性能、容量甚至逻辑问题时，你可以准确追踪到问题所在。在这个案例中，我们只能将问题定位在浏览器和数据库之间的某个地方——换句话说，整个系统都值得怀疑。`Oracle` 数据库有很强的插桩，但应用程序需要能够在适当的点开启和关闭插桩——而这个系统并没有这样设计。

因此，我们面临的挑战是在细节不多的情况下尝试诊断性能问题，只能从数据库本身搜集信息。幸运的是，在这个案例中相对容易。当熟悉 `Oracle` `V$` 表（`V$` 表是 Oracle 向我们展示其插桩和统计信息的一种方式）的人查看这些表时，很明显主要的争用围绕着一张表——一种队列表。应用程序会将记录放入这张表，而另一组进程则会从这张表中取出记录并进行处理。深入挖掘后，我们发现这张表的某个列上有一个位图索引（关于位图索引的更多信息，请参阅第 11 章）。理由是这个列，即“已处理”标志列，只有两个值——`Y` 和 `N`。当记录被插入时，它们的值是 `N`，表示未处理。当其他进程读取并处理记录时，它们会将 `N` 更新为 `Y` 以表示处理完成。开发者需要快速找到值为 `N` 的记录，因此知道他们想在那个列上建立索引。他们曾在某处读到，位图索引适用于低基数列——只有少数几个不同值的列——所以这看起来很合适。（去吧，用谷歌搜索*何时使用位图索引；*低基数会一遍又一遍地出现。幸运的是，今天也有许多文章反驳这个过于简单的概念。）

但是，那个位图索引正是他们所有问题的根源。在位图索引中，一个单一的键条目指向许多行，成百上千行。如果你更新一个位图索引键（从而锁定它），该键指向的数百条记录实际上也被锁定了。因此，当有人插入一个值为 `N` 的新记录时，会锁定位图索引中的 `N` 条目，实际上也锁定了其他数百条值为 `N` 的记录。与此同时，试图读取此表并处理记录的进程将被阻止将一些 `N` 记录修改为 `Y`（已处理）记录，因为为了将其列值从 `N` 更新为 `Y`，它需要锁定同一个位图索引键。事实上，其他只是试图向此表插入新记录的会话也会被阻塞，因为它们会尝试锁定同一个位图键条目。简而言之，开发者创建了一个在任何时候最多只允许一个人进行插入或更新的表！我们可以用一个简单的场景轻松地看到这一点。

在这里，我将使用数据库中的一个自治事务，在单个会话中实现两个并发事务。自治事务启动一个与会话中已建立的任何事务都分离且不同的“子事务”。自治事务的行为就好像它处于一个完全不同的会话中——从任何意义上讲，父事务都被挂起了。自治事务可能会被父事务阻塞（正如我们将看到的），而且，自治事务无法看到父事务未提交的修改。例如，连接到我的可插拔数据库 `PDB1`：



```
$ sqlplus eoda/foo@PDB1
SQL> create table t(processed_flag varchar2(1));
表已创建。
SQL> create bitmap index t_idx on t(processed_flag);
索引已创建。
SQL> insert into t values ( 'N' );
已创建 1 行。
SQL> decl are
pragma autonomous_transaction;
begin
insert into t values ( 'N' );
commit;
end;
/
declare
*
第 1 行出现错误:
ORA-00060: 等待资源时检测到死锁
ORA-06512: 在第 4 行
```
