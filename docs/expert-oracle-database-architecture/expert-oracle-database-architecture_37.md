# 9. 重做日志与撤销

本章将向你介绍 Oracle 数据库中两个最重要的数据片段：重做日志和撤销。*重做日志* 是 Oracle 为了在发生故障时能够“重放”你的事务而记录在在线（和归档）重做日志文件中的信息。*撤销* 是 Oracle 记录在撤销段中以便撤销或回滚你的事务的信息。

在本章中，我们将讨论诸如重做日志和撤销（回滚）是如何产生的，以及它们如何融入事务等主题。我们还将讨论 `COMMIT` 和 `ROLLBACK` 语句的性能影响。我将介绍 Oracle 中这些机制的伪代码以及实际发生情况的概念性解释。我不会涵盖关于哪些文件用哪些数据字节更新的每一个内部细节。实际发生的情况稍微复杂一些，但对其工作流程有良好的理解是很有价值的，并且将帮助你理解你的操作所带来的影响。

一次又一次，我收到关于重做日志和撤销的确切数据细节的问题。人们似乎想要一个非常详细的规范，精确地了解里面到底有什么。我从不回答这些问题。相反，我专注于重做日志和撤销的意图，即它们背后的概念。我专注于重做日志和撤销的使用——而不是数据位和字节。我自己并不“转储”重做日志文件或撤销段。我确实使用提供的工具，比如 LogMiner 来读取重做日志，以及 flashback 事务历史来读取撤销，但这些都是以人类可读的格式向我呈现信息。因此，在本章中，我们不会深入探讨内部细节，而是构建一个坚实的基础。



## 什么是重做日志？

重做日志文件对 Oracle 数据库至关重要。它们是数据库的事务日志。Oracle 维护两种类型的重做日志文件：`联机`日志和`归档`日志。它们用于恢复目的；其主要用途是在发生实例或介质故障时发挥作用。

如果数据库服务器意外断电导致实例故障，Oracle 将使用联机重做日志将系统精确恢复到断电前已提交的状态。如果磁盘驱动器发生故障（介质故障），Oracle 将同时使用归档重做日志和联机重做日志，将该驱动器上数据的备份恢复到正确的时间点。此外，如果您“意外地”截断了某个表或删除了某些关键信息并提交了操作，您还可以使用联机和归档重做日志文件，将受影响数据的备份还原并恢复到“意外”发生前的那一刻。

我需要指出，现代版本的 Oracle 还提供了闪回技术。这使我们可以执行闪回查询（查询过去某个时间点的数据）、取消删除数据库表、将表恢复到之前某个时间点的状态等等。因此，我们需要执行常规恢复（使用数据库备份和归档重做日志）的情况已经减少。然而，能够执行恢复仍然是数据库管理员最重要的工作。

> **注意** 数据库还原与恢复是数据库管理员不容有失的唯一要务。

归档重做日志文件其实就是完整的旧联机重做日志文件的副本。随着系统写满日志文件，Oracle 归档器（`ARCn`）进程会将联机重做日志文件复制到另一个位置，并且可选地将其他几个副本也存放到本地和远程位置。这些归档重做日志文件用于在因磁盘驱动器损坏或其他物理故障导致故障时执行介质恢复。Oracle 可以获取这些归档重做日志文件，并将它们应用到数据文件的备份上，使其与数据库的其他部分保持同步。它们是数据库的事务历史记录。

每个 Oracle 数据库至少有两个联机重做日志组，每个组至少包含一个成员（重做日志文件）。这些联机重做日志组由日志写入器（`LGWR`）后台进程以循环方式写入。Oracle 会写入第 1 组的日志文件，当写完第 1 组的文件末尾时，它会切换到第 2 组日志文件并开始写入该组。当第 2 组日志文件写满后，它会切换回第 1 组日志文件（假设您只有两个重做日志文件组；如果您有三个，Oracle 自然会继续写入第三组）。

重做日志，或称事务日志，是构成数据库的主要特性之一。它们可能是数据库最重要的恢复结构，尽管没有其他部分（如撤销段、分布式事务恢复等）也无法运作。它们是数据库区别于常规文件系统的主要组成部分之一。联机重做日志使我们能够有效地从断电故障中恢复——这种故障可能发生在 Oracle 数据库写入器（`DBWR`）后台进程正在向磁盘写入的过程中。归档重做日志则让我们能够从介质故障中恢复，例如硬盘损坏或人为错误导致数据丢失的情况。没有重做日志，数据库提供的保护不会比文件系统更多。

关于重做日志，我还想提及一点。在 Oracle RAC 环境中，通常有两个或更多实例。RAC 配置共享一组公共的数据文件（意味着每个实例都针对同一组公共数据文件进行事务处理）。然而，参与 RAC 集群的每个实例都有自己的内存结构和后台进程（例如，日志写入器和归档器）。此外，每个实例都有自己的重做流（或常称为重做线程）。相应地，每个实例也将拥有自己的撤销段。这一点很重要，因为您可能会遇到需要排查与重做相关的性能问题，而准确识别是哪个或哪些实例可能存在重做瓶颈问题至关重要。


## 什么是撤销？

从概念上讲，撤销与重做相反。当你修改数据时，数据库会生成撤销信息，以便能将数据恢复到修改前的状态。这样做可能是为了支持多版本机制，或者因为正在执行的事务或语句因任何原因失败，又或者当我们使用 `ROLLBACK` 语句请求回滚时。重做用于在发生故障时重放事务——以恢复事务；而撤销则用于逆转一条语句或一组语句的影响。与重做不同，撤销存储在数据库内部的一组特殊的段中，称为撤销段。

注意

“回滚段”和“撤销段”被视为同义词。

一个常见的误解是，撤销用于将数据库 `物理上` 恢复到执行语句或事务之前的状态，但事实并非如此。数据库是 `逻辑上` 恢复到之前的状态——任何更改都在逻辑上被撤销——但数据结构，即数据库块本身，在回滚后可能已经不同。原因在于，在任何多用户系统中，都会存在几十、几百或几千个并发事务。数据库的主要功能之一是协调对其数据的并发访问。我们的事务所修改的块，通常也正被许多其他事务修改。因此，我们不能简单地将一个块完全恢复到事务开始时的状态——那可能会撤销其他人的工作！

例如，假设我们的事务执行了一条 `INSERT` 语句，导致分配了一个新的区（即，导致表增长）。我们的 `INSERT` 会使我们获得一个新块，将其格式化以供使用，并在其中放入一些数据。此时，其他某个事务可能会来向这个块中插入数据。如果我们回滚事务，显然我们不能取消格式化并释放这个块。因此，当 Oracle 回滚时，它实际上做的是在逻辑上执行与我们最初操作相反的操作。对于每一个 `INSERT`，Oracle 会执行一个 `DELETE`。对于每一个 `DELETE`，Oracle 会执行一个 `INSERT`。对于每一个 `UPDATE`，Oracle 会执行一个“反向 `UPDATE`”，或者一个将行恢复到修改前状态的 `UPDATE`。

注意

对于直接路径操作，这种撤销生成方式不成立，因为它们能够绕过表的撤销生成。我们稍后将更详细地讨论这些操作。

我们如何看到这个过程呢？最简单的方法大概是遵循以下步骤：

1.  创建一个空表。
2.  全表扫描该表，并观察执行读取所执行的 I/O 量。
3.  向表中插入许多行数据（不提交）。
4.  回滚该工作并将其撤销。
5.  第二次全表扫描该表，并观察所执行的 I/O 量。

那么，让我们创建一个空表：

```
$ sqlplus eoda/foo@PDB1
SQL> create table t as select * from all_objects where 1=0;
表已创建。
```

现在我们将查询它，在 SQL*Plus 中启用 `AUTOTRACE` 来测量 I/O。

注意

在此示例中，我们将每次全表扫描两次。目的是在每种情况下仅测量第二次扫描时执行的 I/O。这样可以避免优化器在可能发生的任何解析和优化期间执行的额外 I/O。

最初全表扫描该表消耗 *零* I/O：

```
SQL> select * from t;
未选定行
SQL> set autotrace traceonly statistics
SQL> select * from t;
未选定行
Statistics
----------------------------------------------------------
          0  recursive calls
          0  db block gets
          0  consistent gets
          0  physical reads
SQL> set autotrace off
```

现在，起初这可能会让您惊讶——对表的 I/O 为 *零*。这是由于延迟段创建造成的。

注意

延迟段创建功能仅在 Oracle 企业版中可用。此功能默认启用。您可以在创建表时覆盖此默认行为。

接下来，我们将向表中添加大量数据。我们将使其“增长”，然后全部回滚：

```
SQL> insert into t select * from all_objects;
已创建 72516 行。
SQL> rollback;
回退完成。
```

现在，如果我们再次查询该表，会发现这次读取表消耗了相当多的 I/O：

```
SQL> select * from t;
未选定行
SQL> set autotrace traceonly statistics
SQL> select * from t;
未选定行
Statistics
----------------------------------------------------------
          0  recursive calls
          0  db block gets
       1451  consistent gets
          0  physical reads
SQL> set autotrace off
```

我们的 `INSERT` 导致添加到表高水位线下的那些块仍然存在——已格式化，但是空的。我们的全表扫描必须读取它们以查看它们是否包含任何行。此外，我们第一次运行查询时，观察到 *零* I/O。那是由于使用延迟段创建的默认表创建模式造成的。当我们执行 `CREATE TABLE` 时，没有分配任何存储空间，甚至连一个区都没有分配。段创建被推迟到执行 `INSERT` 时，而当我们回滚时，该段仍然存在。您可以通过一个更小的例子轻松看到这一点；虽然默认是启用的，但这次我将显式请求延迟段创建：

```
SQL> drop table t purge;
表已删除。
SQL> create table t ( x int )  segment creation deferred;
表已创建。
SQL> select extent_id, bytes, blocks
  2  from user_extents
  3  where segment_name = 'T'
  4  order by extent_id;
未选定行
SQL> insert into t(x) values (1);
已创建 1 行。
SQL> rollback;
回退完成。
SQL> select extent_id, bytes, blocks
  2  from user_extents
  3  where segment_name = 'T'
  4  order by extent_id;
EXTENT_ID      BYTES     BLOCKS
---------- ---------- ----------
         0      65536          8
```

如您所见，表最初创建后没有分配存储空间——该表没有使用任何区。在执行 `INSERT` 后立即执行 `ROLLBACK`，我们可以看到 `INSERT` 分配了存储空间——但 `ROLLBACK` 并没有“释放”它。这两件事共同表明——该段确实由 `INSERT` 创建，但并未被 `ROLLBACK` “撤销创建”，并且由 `INSERT` 创建的新格式化块在第二次扫描时被扫描了——回滚是一个逻辑上的“将数据库恢复到之前状态”的操作。数据库不会完全恢复原样，只是逻辑上相同。

## 重做与撤销如何协同工作

现在让我们来看看重做和撤销在各种场景下如何协同工作。例如，我们将讨论在处理 `INSERT` 时关于重做和撤销生成的情况，以及 Oracle 如何在不同时间点发生故障时使用这些信息。

一个值得注意的有趣点是，存储在撤销表空间或撤销段中的撤销信息也受到重做的保护。换句话说，撤销数据就像表数据或索引数据一样对待——对撤销的更改会生成一些重做，这些重做会被记录（到日志缓冲区，然后到重做日志文件）。为什么是这样，在我们讨论系统崩溃时会发生什么之后，就会变得清晰。撤销数据被添加到撤销段中，并缓存在缓冲区高速缓存中，就像任何其他数据一样。

### 示例 INSERT-UPDATE-DELETE-COMMIT 场景

对于这个示例，假设我们创建了一个带索引的表，如下所示：

```
create table t(x int, y int);
create index ti on t(x);
```

然后我们将研究执行以下语句时可能发生的情况：

```
insert into t (x,y) values  (1,1);
update t set x = x+1 where x = 1;
delete from t where x = 2;
```

我们将跟踪这个事务的不同路径，并找出以下问题的答案：

*   在处理这些语句的不同时间点，如果系统发生故障会发生什么？
*   如果缓冲区高速缓存填满了会发生什么？
*   如果我们在任何时刻 `ROLLBACK` 会发生什么？
*   如果我们成功并 `COMMIT` 会发生什么？
