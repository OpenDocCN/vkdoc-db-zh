# 获取一致的答案

如果我们采用 `REPEATABLE READ` 隔离级别，那么给定查询的结果必须相对于某个时间点保持一致。大多数数据库（Oracle 除外）通过使用行级共享读锁来实现可重复读。共享读锁可防止其他事务修改我们已读取的数据。当然，这会降低并发性。Oracle 则选择了并发性更高的多版本模型来提供读一致的答案。

在 Oracle 中，通过多版本机制，我们得到的答案与查询开始执行的时间点是一致的。而在其他使用共享读锁的数据库中，我们得到的答案与查询完成的时间点是一致的——也就是说，当我们最终能拿到答案时（这一点稍后会详细说明）。

在一个使用共享读锁来提供可重复读的系统中，我们会观察到随着查询处理数据，表中的行会被锁定。因此，使用前面的例子，当我们的查询读取 `ACCOUNTS` 表时，它会在每一行上留下共享读锁，如表 7-5 所示。

表 7-5
非 Oracle 数据库中使用 READ REPEATABLE 隔离级别的时间线 1

| 时间 | 查询 | 账户转账事务 |
| --- | --- | --- |
| T1 | 读取第 1 行。当前累计金额=$500.00。第 1 行上有一个共享读锁。 | -- |
| T2 | 读取第 2 行。当前累计金额=$740.25。第 2 行上有一个共享读锁。 | -- |
| T3 | -- | 尝试更新第 1 行但被阻塞。事务挂起，直到能够获取到排他锁。 |

表 7-5 显示我们现在得到了正确的答案，但代价是物理上阻塞了一个事务，并且两个事务是顺序执行的。这是使用共享读锁来获得一致性答案的副作用之一：*数据的读取者会阻塞数据的写入者*。除此之外，在这些系统中，数据的写入者也会阻塞数据的读取者。想象一下现实生活中的自动取款机（ATM）如果以这种方式工作会怎样。

所以，你可以看到共享读锁如何抑制并发性，但它们还可能导致虚假错误的发生。在表 7-6 中，我们从原始表开始，但这次的目标是从账户 987 向账户 123 转账 $50.00。

我们刚刚遇到了经典的死锁情况。我们的查询持有了更新事务所需的资源，反之亦然。我们的查询与我们的更新事务发生了死锁。其中一个事务将被选为牺牲品并被终止。我们花费了大量时间和资源，结果却在最后失败并被回滚。这是共享读锁的第二个副作用：*数据的读取者和写入者能够并且经常会发生死锁*。

在 Oracle 中，我们实现了语句级读一致性，而不会出现读阻塞写或死锁的情况。Oracle 从不——*绝不*——使用共享读锁。Oracle 选择了实现难度更高但并发性无限的多版本方案。

## 丢失的更新：另一个可移植性问题

在采用共享读锁的数据库中，`REPEATABLE READ` 的一个常见用途可能是防止丢失更新。

注意
丢失更新的检测和解决方法在前一章中已讨论。

如果我们在一个*采用共享读锁*（而非多版本机制）的数据库中启用了 `REPEATABLE READ`，则丢失更新错误不会发生。在这些数据库中丢失更新不会发生的原因是，仅仅选择数据这一操作*就会在数据上留下锁，一旦我们的事务读取了数据，该数据就不能被任何其他事务修改。* 现在，如果你的应用程序假设 `REPEATABLE READ` 意味着“丢失更新不会发生”，那么当你将应用程序迁移到不使用共享读锁作为底层并发控制机制的数据库时，你将会遭遇痛苦的意外。

注意
在无状态环境（例如基于 Web 的应用程序）中，即使是在 `REPEATABLE READ` 隔离级别下，丢失更新也很可能是一个值得关注的问题。这是因为单个数据库会话通过连接池被许多客户端使用，并且锁不会跨调用保持。`REPEATABLE READ` 隔离级别仅在有状态环境（例如客户端-服务器应用程序所观察到的）中才能防止丢失更新。

虽然这听起来不错，但你必须记住，在读取数据时将共享读锁留在所有数据上，当然会严重限制并发读取和修改。因此，虽然在这些数据库中这个隔离级别提供了丢失更新的预防，但它是通过移除执行并发操作的能力来实现这一点的！你不可能两全其美。



### 可串行化（SERIALIZABLE）

这通常被认为是最严格的事务隔离级别，但它提供了最高程度的隔离。`SERIALIZABLE`事务在其运行环境中，仿佛没有其他用户在修改数据库中的数据。我们读取的任何行在重新读取时都保证是相同的，并且任何查询在事务生命周期内都保证返回相同的结果。例如，如果我们执行以下操作，即使我们刚刚休眠了 24 小时（或者可能会得到一个`ORA-01555, snapshot too old`错误，这将在下一章讨论），从`T`返回的答案也将是相同的：
```
SQL> select * from T;
SQL> begin dbms_lock.sleep( 60*60*24 ); end;
SQL> select * from T;
```

隔离级别`SERIALIZABLE`确保这两个查询总是返回相同的结果。无论查询运行了多久，其他事务所做的副作用（更改）对这个查询都是不可见的。

在 Oracle 中，`SERIALIZABLE`事务的实现方式，是将通常我们在语句级别获得的读一致性扩展到了事务级别。

> **注意**
> 如前所述，在 Oracle 中还有一个用`READ ONLY`表示的隔离级别。它具有`SERIALIZABLE`隔离级别的所有特性，但禁止修改。需要注意的是，`SYS`用户（或使用`SYSDBA`权限连接的用户）*不能*拥有`READ ONLY`或`SERIALIZABLE`事务。`SYS`在这方面是特殊的。

结果不是相对于语句开始时间保持一致，而是在您开始事务时就已经被预定了。换句话说，Oracle 使用回滚段来重建我们的事务开始时存在的数据，而不仅仅是语句开始时的数据。

这是一个相当深刻的想法：在你提问之前，数据库就已经知道你可能问的任何问题的答案了。

#### 可能的错误：ORA-08177
这种程度的隔离是有代价的，这个代价就是可能出现的以下错误：
```
ERROR at line 1:
ORA-08177: can't serialize access for this transaction
```
每当您尝试更新自事务开始以来已更改的行时，就会收到此消息。

> **注意**
> Oracle 尝试在行级别纯粹地实现这一点，但即使您感兴趣的行尚未被修改，您也可能收到`ORA-08177`错误。`ORA-08177`错误可能由于包含您所在行的块上的某些其他行被修改而发生。

Oracle 对串行化采取了一种乐观的方法——它赌您的事务想要更新的数据不会被任何其他事务更新。这通常是实际情况，而且通常赌注会得到回报，尤其是在快速事务的 OLTP 类系统中。如果在您的事务期间没有其他人更新您的数据，这种在其他系统中通常会降低并发性的隔离级别，将提供与没有`SERIALIZABLE`事务时相同的并发程度。这样做的缺点是，如果赌注没有得到回报，您可能会得到`ORA-08177`错误。然而，仔细想想，这个风险是值得的。如果您使用`SERIALIZABLE`事务，您不应该期望与其他事务更新相同的信息。如果您确实需要这样做，您应该使用第[1]章中描述的`SELECT ... FOR UPDATE`，这将会串行化访问。

#### 何时使用 SERIALIZABLE
因此，如果您满足以下条件，使用`SERIALIZABLE`隔离级别将是可行且有效的：
*   没有其他人修改相同数据的可能性很高
*   需要事务级别的读一致性
*   将进行短事务（以帮助使第一个要点成为现实）

Oracle 发现这种方法具有足够的可扩展性来运行其所有的 TPC-C 测试（一种行业标准的 OLTP 基准测试；详见[`www.tpc.org`](http://www.tpc.org)）。在许多其他实现中，您会发现这是通过共享读锁及其相应的死锁和阻塞来实现的。在 Oracle 中，我们不会得到任何阻塞，但如果其他会话也更改了我们想要更改的数据，我们将得到`ORA-08177`错误。然而，我们遇到这种错误的频率，不会像在其他系统中遇到死锁和阻塞那样高。

但是——总有一个“但是”——您必须注意理解这些不同的隔离级别及其含义。请记住，当隔离设置为`SERIALIZABLE`时，在您提交之前，您将看不到事务开始后数据库中所做的任何更改。试图强制执行自身数据完整性约束的应用程序在这方面必须格外小心。使用`SERIALIZABLE`，您将看不到未提交的更改；此外，您也看不到事务开始后所做的已提交更改。

#### 澄清一个误解
最后一点，要注意`SERIALIZABLE`*并不*意味着用户执行的所有事务都会表现得好像它们是一个接一个以串行方式执行的一样。它并不意味着存在某种事务的串行排序会导致相同的结果。SQL 标准先前描述的现象并不能使这一点发生。这最后一点是一个经常被误解的概念，一个小的演示将澄清它。下表代表两个会话随时间执行工作。数据库表`A`和`B`开始为空，创建如下：
```
SQL> create table a ( x int );
SQL> create table b ( x int );
```
现在，我们有表[7-7]所示的一系列事件。

表 7-7
可串行化事务示例
| 时间 | 会话 1 执行 | 会话 2 执行 |
| --- | --- | --- |
| T1 | `SQL> alter session set isolation_level=serializable;` | -- |
| T2 | -- | `SQL> alter session set isolation_level=serializable;` |
| T3 | `SQL> insert into a select count(*) from b;` | -- |
| T4 | -- | `SQL> insert into b select count(*) from a;` |
| T5 | `SQL> commit;` | -- |
| T6 | -- | `SQL> commit;` |

现在，当这一切都完成后，表`A`和`B`将各有一行值为 0 的记录。如果存在某种事务的串行排序，我们不可能让两个表都包含值 0。如果会话 1 在会话 2 之前*完整执行*，那么表`B`将有一行值为 1 的记录。如果会话 2 在会话 1 之前*完整执行*，那么表`A`将有一行值为 1 的记录。然而，按此处执行的方式，两个表都将包含值为*0*的行。它们只是执行得好像在当时是数据库中唯一的事务。无论会话 1 查询表`B`多少次，也无论会话 2 的提交状态如何，计数都将是在时间 T1 提交到数据库中的计数。同样，无论会话 2 查询表`A`多少次，计数都将与时间 T2 时相同。



### 只读

`READ ONLY` 事务与 `SERIALIZABLE` 事务非常相似，唯一的区别是它们不允许修改，因此不受 `ORA-08177` 错误的影响。`READ ONLY` 事务旨在支持报表需求，确保报表内容相对于单一时间点是一致的。在其他系统中，你可能会使用 `REPEATABLE READ` 并承受共享读锁带来的相关影响。而在 Oracle 中，你将使用 `READ ONLY` 事务。在此模式下，你使用 50 条 `SELECT` 语句来收集数据生成报表，其输出将相对于单一时间点——即事务开始的时间——保持一致。你将能够在不锁住任何数据的情况下完成此操作。

此目标通过使用与单条语句相同的多版本机制来实现。数据会根据需要从回滚段中重建，并以报表开始时存在的状态呈现给你。然而，`READ ONLY` 事务并非没有问题。在 `SERIALIZABLE` 事务中你可能会看到 `ORA-08177` 错误，而在 `READ ONLY` 事务中，你预期会看到 `ORA-01555 snapshot too old`（快照过旧）错误。这会在其他人正在积极修改你所读取信息的系统上发生。对这些信息的更改（撤销）被记录在回滚段中。但回滚段是以循环方式使用的，类似于重做日志。报表运行的时间越长，你需要的、用于重建数据的某些撤销信息可能已不复存在的几率就越大。回滚段可能已经回绕，而你所需的部分可能已被其他事务重用。此时，你将收到 `ORA-01555` 错误并必须重新开始。

解决这个棘手问题的唯一方法是为你的系统正确调整回滚表空间的大小。我一次又一次地看到人们试图通过设置尽可能小的回滚表空间来节省几兆字节的磁盘空间（“为什么要为我不真正需要的东西‘浪费’空间？”是他们的想法）。问题在于回滚表空间是数据库工作方式的关键组件，除非它大小合适，否则你将遇到此错误。在使用 Oracle 6、7、8、9、10、11 和 12 的多年里，我可以说我从未在测试或开发系统之外遇到过 `ORA-01555` 错误。在这种情况下，你知道你没有正确调整回滚表空间的大小，并且你会修复它。我们将在下一章重新讨论这个问题。

## 多版本读一致性的含义

到目前为止，我们已经看到多版本控制如何为我们提供非阻塞读取，并且我已强调这是一件好事：提供一致（正确）的答案同时具有高并发性。这能有什么问题？嗯，除非你理解它的存在及其含义，否则你很可能在错误地执行某些事务。但它会以其他方式影响我们吗？答案绝对是肯定的。我们将在接下来的章节中深入探讨具体细节。

### 一种常见的数据仓库技术失败案例

我见过人们采用的一种常见的数据仓库技术如下：

1.  他们使用触发器来维护源表中的一个 `LAST_UPDATED` 列，非常类似于前一章“乐观锁定”部分描述的方法。
2.  为了初始填充数据仓库表，他们通过在源系统上选择 `SYSDATE` 来记住当前时间。例如，假设现在正好是上午 9:00。
3.  然后，他们从事务系统中拉取所有行——一个完整的 `SELECT * FROM TABLE`——以初始填充数据仓库。
4.  为了刷新数据仓库，他们再次记住当前时间。例如，假设一个小时过去了——源系统上现在是上午 10:00。他们会记住这个事实。然后，他们拉取自上午 9:00（他们开始第一次拉取前的时刻）以来所有更改的记录，并将它们合并进去。

注意

由于时钟的粒度，此技术可能在连续两次刷新中*拉取*同一条记录。这是不可避免的。`MERGE` 操作不会受此影响（即，在数据仓库中更新现有记录或插入新记录）。

他们相信现在数据仓库中包含了自他们进行初始拉取以来修改的所有记录。他们可能确实拥有所有记录，但也同样可能没有。此技术在某些其他数据库上有效——那些采用锁系统的数据库，其中读取会被写入阻塞，反之亦然。但在具有非阻塞读取的系统中，此逻辑是有缺陷的。

要看到此示例中的缺陷，我们只需假设在上午 9:00 至少有一个未提交的打开事务。在上午 8:59:30，它更新了我们即将复制的表中的某一行。在上午 9:00，当我们开始拉取数据并读取该表中的数据时，我们不会看到对该行的修改；我们将看到它最后提交的版本。如果在查询中遇到它时它被锁定，我们将绕过锁进行读取。如果在我们读到它时它已经提交，由于读一致性允许我们只读取在语句开始时已提交在数据库中的数据，我们仍然会绕过它读取。我们不会在上午 9:00 的初始拉取中读取该行的新版本，也不会在上午 10:00 的刷新中读取已修改的行。原因是什么？上午 10:00 的刷新只会拉取该日上午 9:00 之后修改的记录，但此记录是在上午 8:59:30 修改的。我们永远无法拉取这条已更改的记录。

在许多其他读取会被写入阻塞并实现已提交但不一致读取的数据库中，此刷新过程将完美运行。如果在上午 9:00 我们进行初始数据拉取时遇到该行并且它被锁定，我们将被阻塞并等待它，然后读取已提交的版本。如果它没有被锁定，我们将直接读取当时存在的任何已提交数据。

那么，这是否意味着前面的逻辑完全不能使用？不，这意味着我们需要以稍微不同的方式获取“当前”时间。我们需要查询 `V$TRANSACTION` 并找出当前时间与该视图 `START_TIME` 列中记录的时间中最早的一个。我们需要拉取自最旧事务开始时间（或者如果没有活动事务则为当前 `SYSDATE` 值）以来所有更改的记录：

```
SQL> select nvl( min(to_date(start_time,'mm/dd/rr hh24:mi:ss')),sysdate)
from v$transaction;
```

注意

无论 `V$TRANSACTION` 中是否存在任何数据，前面的查询都有效。也就是说，即使 `V$TRANSACTION` 为空（因为当前没有事务），此查询也会返回一条记录。带有聚合函数且没有 `WHERE` 子句的查询总是返回*至少*一行和*至多*一行。

在此示例中，那将是修改行的事务开始时的上午 8:59:30。当我们在上午 10:00 去刷新数据时，我们拉取自该时间以来发生的所有更改；当我们将这些合并到数据仓库中时，我们将拥有我们需要的一切。



### 关于热门表上超出预期的 I/O 的解释

当你面临一个在生产环境（负载沉重时）的查询，其使用的 I/O 远多于在测试或开发系统中观察到的，且你无法解释原因时，理解读一致性和多版本机制就变得至关重要。你检查查询执行的 I/O，发现它高得离谱——远超可能的范围。你在测试环境上恢复了生产实例，发现 I/O 大幅下降。但在生产环境中，I/O 仍然非常高（但似乎波动：有时高，有时低，有时中等）。原因在于，正如我们将看到的，在测试系统中，你是独立运行的，无需撤销其他事务的更改。然而，在生产环境中，当你读取一个特定的数据块时，你可能需要撤销（回滚）许多事务的更改，而每次回滚都可能涉及检索 undo 数据并应用它的 I/O 操作。

这很可能是一个针对正在发生大量并发修改的表的查询；你看到的是对 undo 段的读取，这是 Oracle 为了将数据块恢复到你查询开始时的状态所做的工作。你可以在单个会话中轻松地观察到这种效果，以理解发生了什么。我们从一个很小的表开始：

```sql
$ sqlplus eoda/foo@PDB1
SQL> drop table t purge;
SQL> create table t ( x int );
表已创建。
SQL> insert into t values ( 1 );
已创建 1 行。
SQL> exec dbms_stats.gather_table_stats( user, 'T' );
PL/SQL 过程已成功完成。
SQL> select * from t;
X
----------
         1
```

现在，我们将我们的会话设置为使用 `SERIALIZABLE` 隔离级别，这样无论在我们的会话中运行查询多少次，结果都将“截至”该事务开始的时间：

```sql
SQL> alter session set isolation_level=serializable;
会话已更改。
```

现在，我们查询这个小表并观察执行的 I/O 量：

```sql
SQL> set autotrace on statistics
SQL> select * from t;
X
----------
         1

统计信息
----------------------------------------------------------
          0  递归调用
          0  db 块获得
          7  一致读
          ...
```

所以，该查询为了完成进行了七次 I/O（`一致读`）。在*另一个会话*中，我们将反复修改这个表：

```sql
SQL> begin
  2  for i in 1 .. 10000
  3  loop
  4      update t set x = x+1;
  5      commit;
  6  end loop;
  7  end;
  8  /
PL/SQL 过程已成功完成。
```

然后回到我们的 `SERIALIZABLE` 会话，我们重新运行相同的查询：

```sql
SQL> select * from t;
X
----------
     10001

统计信息
----------------------------------------------------------
          0  递归调用
          0  db 块获得
      10004  一致读
          ...
```

那次它做了 10,004 次 I/O——一个显著的差异。那么，这些 I/O 从何而来？那是 Oracle 在回滚对该数据库块所做的更改。当我们运行第二个查询时，Oracle 知道该查询检索和处理的所有块都必须是“截至”事务开始时间的状态。当我们到达缓冲区缓存时，我们发现缓存中的块“太新”了——另一个会话已经修改了它大约 10,000 次。我们的查询无法看到那些更改，于是它开始遍历 undo 信息并撤销了最后一次更改。它发现这个回滚后的块仍然太新，于是再次回滚该块。它反复这样做，直到最终找到在我们的事务开始时数据库中已提交的那个块版本。那个块我们才可以使用——也确实使用了。

> **注意**
>
> 有趣的是，如果你重新运行 `SELECT * FROM T`，你可能会看到 I/O 再次下降到 7 左右；它不会是 10,004。原因是什么？Oracle 有能力在缓冲区缓存中存储同一个块的多个版本。当你为那个执行了 10,004 次 I/O 的查询回滚对此块的更改时，你将该版本留在了缓存中，随后执行的查询就能够访问它。

那么，我们是否只在使用 `SERIALIZABLE` 隔离级别时才遇到这个问题？不，完全不是。考虑一个运行五分钟的查询。在查询运行的五分钟内，它从缓冲区缓存中检索块。每次从缓冲区缓存中检索一个块时，它都会执行这个检查：“这个块是否太新？如果是，就回滚它。” 请记住，查询运行的时间越长，它需要的块随时间被修改的可能性就越大。

现在，数据库预期会发生这种检查（即，检查块是否“太新”以及随后的更改回滚），正因为这个原因，缓冲区缓存实际上可能在内存中包含同一个块的多个版本。这样，你所需的版本很可能就在那里，随时可用，而无需使用 undo 信息来物化。可以使用类似以下的查询来查看这些块：

```sql
SQL> select file#, block#, count(*)
  2  from v$bh
  3  group by file#, block#
  4  having count(*) > 3
  5  order by 3;
```

通常，你在任何时候在缓存中找到的一个块的版本不会超过大约六个，但这些版本可以被任何需要它们的查询使用。

通常，正是这些小型的*热*表会遇到由于读一致性导致的 I/O 膨胀问题。其他经常受此问题影响的查询是针对易变表的长时间运行的查询。它们运行的时间越长，运行时间就越长，因为随着时间的推移，它们可能需要执行更多工作才能从缓冲区缓存中检索一个块。

## 写一致性

到目前为止，我们研究的是读一致性：Oracle 使用 undo 信息来提供非阻塞查询和一致（正确）读取的能力。我们理解，当 Oracle 从缓冲区缓存中为查询读取块时，它会确保块的版本“足够旧”，可以被该查询看到。

但这引出了以下问题：写入/修改呢？当你运行如下所示的 `UPDATE` 语句，并且在该语句运行时，有人更新了它尚未读取的一行，将 `Y=5` 改为 `Y=6` 并提交，会发生什么？

```sql
Update t Set x = 2 Where y = 5;
```

也就是说，当你的 `UPDATE` 开始时，某行值为 `Y=5`。当你的 `UPDATE` 使用一致读读取表时，它看到该行在 `UPDATE` 开始时是 `Y=5`。但是，`Y` 的当前值现在是 6（不再是 5 了），在更新 `X` 的值之前，Oracle 会检查 `Y` 是否仍然是 5。现在会发生什么？更新会如何受到此影响？

显然，我们不能修改块的旧版本；当我们去修改一行时，我们必须修改该块的当前版本。此外，Oracle 不能简单地跳过这一行，因为那将是一个不一致的读取且不可预测。我们将发现，在这种情况下，Oracle 将从头重新启动写入修改。


### 一致性读取与当前读取

Oracle 在处理修改语句时确实会执行两种类型的块获取操作。它会执行：

*   `一致性读取`：在“查找”要修改的行时执行
*   `当前读取`：在获取数据块以实际更新目标行时执行

我们可以通过`TKPROF`轻松观察到这一点。考虑这个小型单行示例，它读取并更新了之前表`T`中的单行：

```
SQL> exec dbms_monitor.session_trace_enable
PL/SQL procedure successfully completed.
SQL> select * from t;
X

SQL> update t t1 set x = x+1;
1 row updated.
SQL> update t t2 set x = x+1;
1 row updated.
```

在退出 SQL*Plus 之前，显示刚生成的跟踪文件的名称和位置：

```
SQL>  select i.value || '/' || d.instance_name || '_ora_' || a.spid || '.trc' trace
from v$process a, v$session b, v$diag_info i , v$instance d
where a.addr = b.paddr
and b.audsid = userenv('sessionid')
and i.name='Diag Trace';
TRACE

/opt/oracle/diag/rdbms/cdb/CDB/trace/CDB_ora_9000.trc
```

接下来，我们导航到跟踪文件目录，定位刚生成的跟踪文件，并使用`TKPROF`工具处理它：

```
$ cd /opt/oracle/diag/rdbms/cdb/CDB/trace
$ tkprof CDB_ora_9000.trc sys=no output=myfile.txt
```

运行`TKPROF`并检查结果（在`myfile.txt`中）后，我们会看到类似以下内容（注意我从此报告中移除了`ELAPSED`、`CPU`和`DISK`列）：

```
select * from t
call     count   query    current        rows
------- ------  ------ ----------  ----------
Parse        1       0          0           0
Execute      1       0          0           0
Fetch        2       7          0           1
------- ------  ------ ----------  ----------
total        4       7          0           1
update t t1 set x = x+1
call     count   query    current        rows
------- ------  ------ ----------  ----------
Parse        1       0          0           0
Execute      1       7          3           1
Fetch        0       0          0           0
------- ------  ------ ----------  ----------
total        2       7          3           1
update t t2 set x = x+1
call     count   query    current        rows
------- ------  ------ ----------  ----------
Parse        1       0          0           0
Execute      1       7          1           1
Fetch        0       0          0           0
------- ------  ------ ----------  ----------
total        2       7          1           1
```

因此，仅仅在一次常规查询中，我们就产生了七次`查询（一致性）模式获取`。在第一次`UPDATE`期间，除了相同的七次 I/O（更新的搜索组件涉及查找更新开始时表中存在的所有行，在本例中如此）之外，还产生了三次`当前模式获取`。执行当前模式获取是为了检索`表块`（它此刻存在的状态，即包含目标行的那个块）、获取一个`回滚段块`以启动我们的事务，以及一个`回滚块`。第二次更新恰好有一次当前模式获取；由于我们不必再次执行回滚工作，我们只需要对包含要更新的行的数据块执行一次当前获取。当前模式获取的出现本身就表明发生了某种修改操作。Oracle 在使用新信息修改数据块之前，必须获取其最新的副本。

#### 读一致性如何影响修改操作

那么，读一致性如何影响修改操作呢？想象一下，您正在针对某个数据库表执行以下`UPDATE`语句：

```
Update t Set x = x+1 Where y = 5;
```

我们理解`WHERE Y=5`部分，即查询的读一致阶段，将使用一致性读取（`TKPROF`报告中的查询模式获取）来处理。它将看到的是该语句开始执行时表中已提交的、满足`WHERE Y=5`条件的记录集合（假设隔离级别为`READ COMMITTED`；如果隔离级别是`SERIALIZABLE`，则看到的将是事务开始时存在的、满足`WHERE Y=5`条件的记录集合）。这意味着，如果该`UPDATE`语句从开始到结束需要五分钟处理，并且在此期间有人向表中添加并提交了一条`Y`列值为五的新记录，那么该`UPDATE`将看不到这条新记录，因为一致性读取不会看到它。这是预期且正常的行为。但问题是，如果两个会话按顺序执行以下语句，会发生什么：

```
Update t Set y = 10 Where y = 5;
Update t Set x = x+1 Where y = 5;
```

表 7-8 展示了时间线。

表 7-8 更新顺序

| 时间 | 会话 1 | 会话 2 | 说明 |
| --- | --- | --- | --- |
| T1 | `Update t Set y=10 Where y=5;` | | 这会更新符合该条件的一行。 |
| T2 | | `Update t Set x=x+1 Where y=5;` | 使用一致性读取，这将找到会话 1 修改的记录，但由于会话 1 已锁定该行，它将无法更新。会话 2 将阻塞并等待此行。 |
| T3 | `Commit;` | | 这会释放会话 2；会话 2 解除阻塞。它最终可以对该行所在的数据块执行当前读取，该行在会话 1 开始更新时 Y 值等于 5。当前读取将显示 Y 现在等于 10，不再是 5 了。 |

因此，在您开始`UPDATE`时`Y=5`的记录，现在不再是`Y=5`了。`UPDATE`的一致性读取部分说，“你想更新这条记录，因为我们开始时`Y`是 5”，但数据块的当前版本让您想，“哦，不，我不能更新这行了，因为`Y`不再是 5 了。那样做是错误的。”

如果我们此时跳过这条记录并忽略它，那么我们将得到一个非确定性的更新。这相当于将数据的一致性和完整性抛之脑后。更新的结果（有多少行以及哪些行被修改）将取决于表中各行被访问的顺序以及碰巧正在发生的其他活动。您可以在两个不同的数据库中使用完全相同的行集合，每个数据库以完全相同的混合方式运行这些事务，但可能观察到不同的结果，仅仅因为行在磁盘上的位置不同。

在这种情况下，Oracle 会选择`重启`更新。当发现您开始时`Y=5`的行现在包含值`Y=10`时，Oracle 会悄悄回滚您的更新（仅回滚更新部分，而非事务的其他部分）并重新开始，假设您使用的是`READ COMMITTED`隔离级别。如果您使用的是`SERIALIZABLE`隔离级别，那么此时您将收到`ORA-08177: can't serialize access for this transaction`错误。在`READ COMMITTED`模式下，事务回滚您的更新后，数据库将重新启动更新（即，更改更新“参照”的时间点），并且它不会再次直接更新数据，而是进入`SELECT FOR UPDATE`模式，尝试为您的会话锁定所有`WHERE Y=5`的行。完成此操作后，它将针对该锁定的数据集运行`UPDATE`，从而确保这次能够完成而无需重启。

但是，让我们继续沿着“但是会发生什么……”的思路思考下去，如果在重启更新并进入`SELECT FOR UPDATE`模式（该模式具有与更新相同的一致性读取和当前块获取操作）后，当您去获取某行的当前版本时，发现该行在您开始`SELECT FOR UPDATE`时是`Y=5`，但现在是`Y=11`，会发生什么？那个`SELECT FOR UPDATE`将会重启，然后循环重新开始。

这引出了几个有趣的问题。我们能观察到这个过程吗？我们能看到这实际发生吗？如果可以，那又怎样？这对我们开发者意味着什么？我们现在将依次探讨这些问题。


### 观察重启

### 观察重启

要观察重启，其实比你想象的要容易。实际上，我们将使用一个简单的单行表来观察它。这是我们用来测试的表：

```sql
SQL> create table t ( x int, y int );
Table created.
SQL> insert into t values ( 1, 1 );
1 row created.
SQL> commit;
Commit complete.
```

为了观察重启，我们只需要一个触发器来打印一些信息。我们将使用一个 `BEFORE UPDATE FOR EACH ROW` 触发器，在更新时打印出行更新前后的镜像：

```sql
SQL> create or replace trigger t_bufer
before update on t for each row
begin
dbms_output.put_line
( 'old.x = ' || :old.x ||
', old.y = ' || :old.y );
dbms_output.put_line
( 'new.x = ' || :new.x ||
', new.y = ' || :new.y );
end;
/
Trigger created.
```

现在，我们来更新那一行：

```sql
SQL> set serveroutput on
SQL> update t set x = x+1;
old.x = 1, old.y = 1
new.x = 2, new.y = 1
1 row updated.
```

到目前为止，一切都如我们所料：触发器触发了一次，我们看到了旧值和新值。请注意，我们尚未提交——那一行仍然被锁定。在另一个会话中，我们将执行这个更新：

```sql
SQL> set serveroutput on
SQL> update t set x = x+1 where x > 0;
```

这当然会立即阻塞，因为第一个会话锁定了该行。如果我们现在回到第一个会话并提交，在第二个会话中将看到此输出（为清晰起见重复了更新操作）：

```sql
SQL> update t set x = x+1 where x > 0;
old.x = 1, old.y = 1
new.x = 2, new.y = 1
old.x = 2, old.y = 1
new.x = 3, new.y = 1
1 row updated.
```

如你所见，该行触发器在这里看到了该行的两个版本。该行触发器被触发了两次：一次是行的原始版本以及我们希望将其修改成的版本，另一次是实际被更新后的最终行。由于这是一个 `BEFORE FOR EACH ROW` 触发器，Oracle 看到了记录的一致性读版本以及我们希望对其所做的修改。然而，Oracle 是在 `BEFORE FOR EACH ROW` 触发器触发后，以当前模式检索数据块来实际执行更新的。它等待此触发器触发后才以当前模式获取数据块，因为触发器可以修改 `:NEW` 值。因此，Oracle 无法在此触发器执行前修改数据块，而触发器可能需要很长时间才能执行。因为一次只有一个会话可以当前模式持有数据块，Oracle 需要限制我们在该模式下持有它的时间。

在此触发器触发后，Oracle 以当前模式检索数据块，并注意到用于定位此行的列 `X` 已被修改。因为 `X` 用于定位此记录且 `X` 被修改了，数据库决定重启我们的查询。注意，将 `X` 从 1 更新为 2 并未使该行超出范围；我们仍要通过此 `UPDATE` 语句更新它。相反，事实是 `X` 用于定位行，且 `X` 的一致性读值（此处为 1）与 `X` 的当前模式读值（2）不同。现在，重启后，触发器将 `X=2`（由其他会话修改后）视为 `:OLD` 值，将 `X=3` 视为 `:NEW` 值。

所以，这显示了这些重启确实会发生。需要一个触发器才能在操作中看到它们；否则，它们通常是**不可检测的**。这并不意味着你无法看到其他症状——例如一个大型 `UPDATE` 语句在更新了许多行后回滚工作，然后发现一行导致其重启——只是很难明确地说“此症状是由重启引起的”。

一个有趣的观察是，即使语句本身不需要，触发器本身也可能导致重启发生。通常，`UPDATE` 或 `DELETE` 语句的 `WHERE` 子句中引用的列用于确定修改是否需要重启。Oracle 将使用这些列执行一致性读，并且在以当前模式检索数据块时，如果检测到其中任何列已更改，它将重启语句。通常，行中的其他列不会被检查。例如，让我们简单地重新运行前面的例子，并在两个会话中都使用 `WHERE Y>0` 来查找行；我们在第一个会话（被阻塞的那个）中看到的输出将是：

```sql
SQL> update t set x = x+1 where y > 0;
old.x = 1, old.y = 1
new.x = 2, new.y = 1
old.x = 2, old.y = 1
new.x = 3, new.y = 1
1 row updated.
```

那么，为什么当 Oracle 在查看 `Y` 值时触发了触发器两次？它检查了整行吗？从输出可以看出，更新实际上被重启了，触发器再次触发了两次，即使我们是在搜索 `Y>0` 且根本没有修改 `Y`。但是，如果我们重新创建触发器，让它只打印出触发的事实，而不是引用 `:OLD` 和 `:NEW` 值，如下所示，然后再次进入第二个会话运行更新，我们观察到它被阻塞了（当然）：

```sql
SQL> create or replace trigger t_bufer
before update on t for each row
begin
dbms_output.put_line( 'fired' );
end;
/
Trigger created.
SQL> update t set x = x+1;
fired
1 row updated.
```

提交阻塞会话后，我们将看到：

```sql
SQL> update t set x = x+1 where y > 0;
fired
1 row updated.
```

这次触发器只触发了一次，而不是两次。因此，在触发器中引用的 `:NEW` 和 `:OLD` 列值，也会被 Oracle 用于进行重启检查。当我们在触发器中引用 `:NEW.X` 和 `:OLD.X` 时，`X` 的一致性读值和当前读值被比较并发现不同。于是发生了重启。当我们从触发器中移除对该列的引用时，就没有重启了。

所以规则是：用于在 `WHERE` 子句中查找行的列集合，加上在行触发器中引用的列，将被比较。该行的一致性读版本将与该行的当前读版本进行比较；如果其中任何值不同，修改将会重启。

### 注意

你可以利用这一点信息来进一步理解为什么使用 `AFTER FOR EACH ROW` 触发器比使用 `BEFORE FOR EACH ROW` 更高效。`AFTER` 触发器不会有同样的效果——到那时，我们已经以当前模式检索了数据块。

这就引出了“我们为什么要在乎？”这个问题。



### 为什么重启对我们很重要？

首先浮现在脑海的应该是“我们的触发器触发了两次！”。我们有一个表，上面定义了一个`BEFORE FOR EACH ROW`触发器。我们更新了一行数据，但触发器却执行了两次。

想想这可能带来的影响。如果你的触发器执行了任何非事务性的操作，这可能是一个相当严重的问题。例如，考虑一个发送邮件的触发器，邮件内容是“这是数据之前的样子。它已被修改为现在的样子。”如果你直接从触发器中发送邮件（例如，使用`UTL_MAIL`），那么用户将收到两封邮件，其中一封报告的是从未实际发生的更新。

你在触发器中所做的任何非事务性操作都会受到重启的影响。考虑以下影响：

*   考虑一个维护某些 PL/SQL 全局变量的触发器，例如用于记录已处理行数的变量。当一条语句重启并回滚时，对 PL/SQL 变量的修改不会回滚。

*   任何以`UTL_`开头的函数（`UTL_FILE`、`UTL_HTTP`、`UTL_SMTP`等等）都应该被认为容易受到语句重启的影响。当语句重启时，`UTL_FILE`不会*回写*到它正在写入的文件中。

*   任何属于自治事务的触发器都必须被怀疑。当语句重启并回滚时，自治事务无法回滚。

所有这些后果都必须谨慎处理，并确信它们可能会为每一行触发多次，或者为一个最终不会被该语句更新的行触发。

你应该关心潜在重启的第二个原因是性能相关。我们使用了一个单行例子，但如果你开始一个大型批处理更新，并在处理了前 100,000 条记录后重启，会发生什么？它将回滚这 100,000 行更改，然后以`SELECT FOR UPDATE`模式重启，并在之后再次进行这 100,000 行更改。

你可能会注意到，在添加了那个简单的审计跟踪触发器（读取`:NEW`和`:OLD`值的那个）之后，性能变得比你能解释的差得多，即使除了新触发器之外什么都没有改变。可能是你正在重启过去从未使用过的查询。或者，添加一个只是偶尔更新单行的小程序，可能导致一个过去一小时就能运行完的批处理作业，因为从未发生过的重启而突然需要运行好几个小时。

这不是 Oracle 的新特性——自从引入读一致性的 4.0 版本以来，它就一直存在于数据库中。我自己直到 2003 年夏天才完全意识到它是如何工作的，在我发现它所隐含的意义之后，我就能解答许多过去“这怎么可能发生？”的问题了。这让我几乎完全放弃了在触发器中使用自治事务，也让我重新思考了一些应用程序的实现方式。例如，我永远不会直接从触发器发送邮件；相反，我总是使用`DBMS_JOB`或类似的东西，在我的事务提交后发送邮件。这使得邮件的发送是*事务性*的；也就是说，如果导致触发器触发并发送邮件的语句被重启，它执行的回滚也会回滚`DBMS_JOB`的请求。我在触发器中做的几乎所有非事务性操作都被修改为在事后通过作业完成，从而使它们都具有事务一致性。

## 总结

在本章中，我们涵盖了很多材料，其中一些可能并不总是显而易见的。然而，理解这些问题至关重要。例如，如果你没有意识到语句级重启，你可能无法弄清楚某些特定的情况是如何发生的。也就是说，你将无法解释你日常经验观察到的一些现象。事实上，如果你没有意识到重启，你可能会错误地怀疑实际的错误是由于环境或最终用户错误造成的。这将是那些无法重现的问题之一，因为观察到它需要许多事情按特定顺序发生。

我们了解了 SQL 标准中定义的隔离级别的含义，以及 Oracle 如何实现它们；有时，我们将 Oracle 的实现与其他数据库进行了对比。我们看到，在其他实现中（即采用读锁来提供一致数据的实现），并发性和一致性之间存在巨大的权衡。为了获得高度的并发数据访问，你将不得不降低对一致性结果的需求。为了获得一致、正确的答案，你将不得不忍受并发性的降低。在 Oracle 中，由于其多版本机制，情况并非如此。
