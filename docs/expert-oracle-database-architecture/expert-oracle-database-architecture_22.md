# 数据库基准测试与 Oracle 锁机制观察

基准测试是在 `informix`、`Sybase SQL Server` 和 `Oracle` 之间进行的。`Oracle` 率先出场。他们的技术人员来到现场，通读了基准测试规范，然后开始着手设置。我注意到的第一件事是，来自 `Oracle` 的技术人员准备使用一个数据库表来记录他们的计时数据，尽管我们将有数十个连接在执行操作，每个连接都会频繁地需要在这个日志表中插入和更新数据。不仅如此，他们还打算在基准测试期间`读取`这个日志表！出于善意，我把其中一个 `Oracle` 技术人员拉到一边，问他是不是疯了。他们为什么要故意在系统中引入另一个争用点？基准测试的所有进程难道不会因为对这个单一表的操作而全部串行化吗？当其他人正在大量修改这个表时，他们试图从中读取，不会把基准测试搞堵塞吗？他们为什么想引入所有这些需要额外管理的锁？我有几十个类似“你们怎么会考虑那样做？”的问题。当时，`Oracle` 的技术人员觉得我有点傻。直到我打开一个 `Sybase SQL Server` 或 `informix` 的窗口，向他们展示了两个人向一个表插入数据，或者当有人试图查询一个表而其他人正在插入行时的效果（查询每秒返回零行）。`Oracle` 的处理方式与几乎其他所有数据库的处理方式之间的差异是巨大的——简直是天壤之别。

不用说，`informix` 和 `SQL Server` 的技术人员在他们的尝试中，都不太热衷于在基准测试期间使用数据库日志表的方法。他们更倾向于将计时数据记录到操作系统的平面文件中。`Oracle` 的人离开时，对如何与 `Sybase SQL Server` 和 `informix` 竞争有了更清晰的认识：只需问观众“你们当前的数据库在数据被锁定时每秒返回多少行？”然后就此展开。

这个故事的寓意有两点。首先，`所有数据库在根本上都是不同的`。其次，当你为一个新的数据库平台设计应用程序时，你绝不能对该数据库的工作方式做出任何假设。`你必须像从未使用过数据库一样去对待每一个新数据库。` 你在一种数据库中会做的事情，在另一种数据库中要么不必要，要么根本就行不通。

在 `Oracle` 中，你会了解到：

*   事务是数据库的核心。它们是件好事。
*   你应该将提交推迟到正确的时刻。你不应该为了减轻系统压力而快速提交，因为拥有长或大的事务并不会给系统带来压力。规则是 `必须提交时才提交，不要提前`。你的事务应该只根据你的业务逻辑来决定其大小。（一个有趣的旁注：就在今天早上，我刚刚在 `http://asktom.oracle.com` 上写下了这个提交规则——“必须提交时才提交，不要提前”——可能已经是第一百万次了。有些事情永远不会改变。）
*   你应该在需要的时间内持有数据上的锁。它们是供你使用的工具，而不是需要避免的东西。锁不是一种稀缺资源。相反，你也应该只在需要的时间内持有数据上的锁。锁可能并不稀缺，但它们可能会阻止其他会话修改信息。
*   在 `Oracle` 中，行级锁没有额外开销——`完全没有`。无论你有一个行锁还是一百万个行锁，用于锁定此信息的资源数量都是相同的。当然，修改一百万行比修改一行要做更多的工作，但锁定一百万行所需的资源与锁定一行相同；它是一个固定常量。
*   你绝不应该为了“对系统更好”而升级锁（例如，使用表锁代替行锁）。在 `Oracle` 中，这对系统并不会更好——它不会节省任何资源。有些时候需要使用表锁，例如在批处理过程中，当你知道将更新整个表并且不希望其他会话锁定行时。但你使用表锁并非是为了通过避免分配行锁来让系统更轻松；在这种情况下，你使用表锁是为了确保你能获得批处理程序所需的所有资源。
*   并发性和一致性可以同时实现。你可以每次都快速且正确地获得它。数据的读取者*不会*被数据的写入者阻塞。数据的写入者*不会*被数据的读取者阻塞。这是 `Oracle` 与大多数其他关系数据库之间的根本区别之一。

随着我们涵盖本章及下一章的其余组件，我将强化这些要点。

## 锁定问题

在我们讨论 `Oracle` 使用的各种类型的锁之前，先看一下一些锁定问题是有益的，其中许多问题源于设计糟糕的应用程序，这些程序没有正确使用（或根本没有使用）数据库的锁定机制。



## 更新丢失

更新丢失是一个经典的数据库问题。实际上，它是所有多用户计算机环境中都会出现的问题。简单来说，当以下事件按此处呈现的顺序发生时，就会发生更新丢失：

1.  会话 1 中的一个事务将一行数据检索（查询）到本地内存中，并将其显示给终端用户 User1。
2.  会话 2 中的另一个事务检索同一行数据，但将其显示给不同的终端用户 User2。
3.  User1 使用应用程序修改该行，并让应用程序更新数据库并提交。会话 1 的事务现已完成。
4.  User2 也修改了该行，并让应用程序更新数据库并提交。会话 2 的事务现已完成。

这个过程被称为更新丢失，因为在步骤 3 中所做的所有更改都将丢失。例如，考虑一个允许用户更改地址、工作电话号码等的员工更新屏幕。应用程序本身非常简单：一个用于生成员工列表的小型搜索屏幕，然后能够深入了解每位员工的详细信息。这应该是小菜一碟。于是，我们编写应用程序时不使用任何锁定，只使用简单的`SELECT`和`UPDATE`命令。

然后，一个终端用户（User1）导航到详细信息屏幕，更改屏幕上的地址，点击“保存”，并收到更新成功的确认。好的，但当 User1 第二天检查记录以发送税表时，旧地址仍然在列。这怎么可能发生？不幸的是，这太容易发生了。在这种情况下，另一个终端用户（User2）在 User1 之后查询了同一条记录——在 User1 读取数据之后，但在 User1 修改数据之前。然后，在 User2 查询数据之后，User1 执行了他们的更新，收到了确认，甚至重新查询以亲眼看到更改。然而，User2 随后更新了工作电话号码字段并点击了保存，**全然不知**他们刚刚用旧数据覆盖了 User1 对地址字段所做的更改！*在这种情况下*会发生这种事的原因是，应用程序开发人员编写的程序在更新某个特定字段时，会刷新该记录的所有字段（仅仅因为更新所有列比弄清楚到底哪些列发生了更改并只更新那些列更容易）。

请注意，要发生这种情况，User1 和 User2 甚至不需要在同一时间处理该记录。他们只需要在*大约*相同的时间处理该记录。

当几乎没有或没有数据库培训的 GUI 程序员被分配编写数据库应用程序的任务时，我一次又一次地看到这个数据库问题出现。他们掌握了`SELECT`、`INSERT`、`UPDATE`和`DELETE`的工作知识，然后就开始编写应用程序。当最终的应用程序表现出刚才描述的行为时，它会完全摧毁用户对它的信心，特别是因为它看起来如此随机、如此偶发，并且在受控环境中完全无法复现（导致开发人员相信这一定是用户错误）。

许多工具，如 Oracle Forms 和 APEX（Application Express，我们用来创建 AskTom 网站的工具），通过确保记录在您查询时保持不变，并在您对其进行任何更改之前锁定它（称为*乐观锁*），来透明地保护您免受此行为的影响；但许多其他工具（如手写的 Visual Basic 或 Java 程序）则不会。那些保护您的工具在幕后所做的，或者开发人员自己必须做的，是使用两种锁定策略之一：*悲观锁*或*乐观锁*。

## 悲观锁

悲观锁方法将在用户修改屏幕上的值之前立即启用。例如，一旦用户表明其打算对他们已选择并显示在屏幕上的特定行执行更新（比如通过点击屏幕上的按钮），就会放置一个行锁。该行锁将*持续存在*，直到应用程序将用户的修改应用到数据库中的行并提交为止。

悲观锁仅在有状态或连接的环境中才有用——也就是说，在这种环境中，您的应用程序与数据库保持持续连接，并且在您的事务生命周期内，您是该连接的唯一使用者。这是 20 世纪 90 年代早中期客户端/服务器应用程序的普遍做法。每个应用程序都会获得一个直接连接，该连接仅由该应用程序实例使用。这种以有状态方式连接的方法已变得不太常见（尽管并未消失），尤其是在 20 世纪 90 年代中后期应用程序服务器出现之后。

假设您使用的是有状态连接，您可能有一个应用程序查询数据而不锁定任何内容：

```
$ sqlplus scott/tiger@PDB1
SQL> select empno, ename, sal from emp where deptno = 10;
EMPNO ENAME             SAL
---------- ---------- ----------
7782 CLARK            2450
7839 KING             5000
7934 MILLER           1300
```

最终，用户选择一行他们想要更新的记录。比方说在这种情况下，他们选择更新`MILLER`行。我们的应用程序将在那时（在用户对屏幕进行任何更改之前，但在行从数据库中取出一段时间后），绑定用户选择的值，以便我们可以查询数据库并确保数据尚未更改。在 SQL*Plus 中，要模拟应用程序将进行的绑定调用，我们可以执行以下操作：

```
SQL> variable empno number
SQL> variable ename varchar2(20)
SQL> variable sal number
SQL> exec :empno := 7934; :ename := 'MILLER'; :sal := 1300;
PL/SQL procedure successfully completed.
```

现在，除了简单地查询值并验证它们没有被更改之外，我们还将使用`FOR UPDATE NOWAIT`锁定该行。应用程序将执行以下查询：

```
SQL> select empno, ename, sal
from emp
where empno = :empno
and decode( ename, :ename, 1 ) = 1
and decode( sal, :sal, 1 ) = 1
for update nowait;
EMPNO ENAME             SAL
---------- ---------- ----------
7934 MILLER           1300
```

注意
我们为什么使用`decode( column, :bind_variable, 1 ) = 1`？它只是表达`where (column = :bind_variable OR (column is NULL and :bind_variable is NULL)`的一种简写方式。你可以编写任一种方式；在这种情况下，`decode()`只是更紧凑，并且由于在 SQL 中`NULL = NULL`既不为真（也不为假！），如果任一列允许`NULL`值，则两种方式之一将是必需的。

应用程序从屏幕上的数据为绑定变量提供值（本例中为`7934`、`MILLER`和`1300`），并从数据库中重新查询这同一行，这次锁定该行以防止其他会话更新；因此，这种方法被称为*悲观*锁。我们在尝试更新之前锁定该行，因为我们怀疑——*我们很悲观*——否则该行将不会保持不变。

由于所有表*应该*都有一个主键（前面的`SELECT`最多会检索一条记录，因为它包含了主键`EMPNO`），并且主键应该是不可变的（我们永远不应该更新它们），我们将从该语句中得到三种结果之一：

*   如果底层数据没有更改，我们将取回我们的`MILLER`行，并且该行将被锁定，防止其他人更新（但可以读取）。
*   如果另一个用户正在修改该行，我们将得到一个`ORA-00054 resource busy`错误。我们必须等待另一个用户完成操作。


## 悲观锁定与乐观锁定

如果在我们选择数据到我们表示要更新之间的时间里，有人已经更改了这一行，那么我们将查不到任何数据。这意味着我们屏幕上显示的数据是过时的。为了避免之前描述的丢失更新场景，应用程序需要在允许最终用户修改之前*重新查询*并锁定数据。采用悲观锁定后，当用户 2 试图更新电话字段时，应用程序现在会识别出地址字段已被更改，并将*重新查询*数据。因此，用户 2 不会用该字段中的旧数据覆盖用户 1 的更改。

一旦我们成功锁定了该行，应用程序将绑定新值，执行更新并提交更改：

```sql
SQL> update emp
set ename = :ename, sal = :sal
where empno = :empno;
1 row updated.
SQL> commit;
Commit complete.
```

我们现在非常安全地更改了那一行。我们不可能覆盖其他人的更改，因为我们已经验证了数据在我们最初读取它和锁定它之间没有发生变化——我们的验证确保在我们操作之前没有其他人更改它，而我们的锁确保在我们处理它时没有其他人能够更改它。

### 乐观锁定

第二种方法，称为*乐观*锁定，将所有锁定延迟到执行更新之前的时刻。换句话说，我们将在不获取锁的情况下修改屏幕上的信息。我们*乐观*地认为数据不会被其他用户更改；因此，我们等到最后一刻再确认我们是否正确。

这种锁定方法适用于所有环境，但它确实会增加执行更新的用户失败的概率。也就是说，当该用户去更新他们的行时，他们发现数据已被修改，必须重新开始。

乐观锁定的一种流行实现方式是在应用程序中保留旧值和新值，并在更新数据时使用如下更新语句：

```sql
Update table
Set column1 = :new_column1, column2 = :new_column2, ....
Where primary_key = :primary_key
And decode( column1, :old_column1, 1 ) = 1
And decode( column2, :old_column2, 1 ) = 1
...
```

在这里，我们乐观地认为数据没有被更改。在这种情况下，如果我们的更新影响了一行，那我们很幸运；数据在我们读取它和提交更新之间没有变化。如果我们更新了零行，我们就失败了；其他人更改了数据，现在我们必须决定如何继续在应用程序中操作。我们是否应该让最终用户在查询行的新值后重新输入事务（可能会导致用户沮丧，因为行有可能再次更改）？我们是否应该通过基于业务规则执行更新冲突解决来尝试合并两次更新的值（这需要大量代码）？

前面提到的`UPDATE`语句实际上会避免丢失更新，但它确实有可能被阻塞，在等待另一个会话对该行的`UPDATE`完成时挂起。如果您的所有应用程序都使用乐观锁定，那么直接使用`UPDATE`通常是没问题的，因为更新被应用和提交时，行被锁定的时间非常短。但是，如果您的某些应用程序使用悲观锁定（这将相对较长时间地持有行锁），或者如果有任何应用程序（例如批处理进程）可能长时间锁定行（超过一两秒被认为是长的），那么您应该考虑使用`SELECT FOR UPDATE NOWAIT`来验证行未被更改，并在`UPDATE`之前立即锁定它，以避免被另一个会话阻塞。

实现乐观并发控制的方法有很多种。我们已经讨论过一种，即应用程序将在其自身存储行的所有前镜像。在接下来的部分中，我们将探讨另外两种，即：

*   使用由数据库触发器或应用程序代码维护的特殊列来告诉我们记录的“版本”
*   使用基于原始数据计算的校验和或哈希值

### 使用版本列实现乐观锁

这是一个简单的实现方法，涉及为您希望防止更新丢失的每个数据库表添加一个单独的列。此列通常为 `NUMBER` 或 `DATE/TIMESTAMP` 类型。它通常通过表上的行触发器来维护，该触发器负责在每次修改行时递增 `NUMBER` 列或更新 `DATE/TIMESTAMP` 列。

### 注意

我说它是通过行触发器维护的。然而，我并没有说这是维护它的最佳或正确方式。我个人更倾向于让 `UPDATE` 语句本身来维护此列，而不是通过触发器，因为除非绝对必要（如同此例），否则应避免使用触发器。关于我为何避免使用触发器的背景，请参阅我在 *Oracle Magazine* 上发表的“触发器之困”一文，可在 Oracle 技术网络上找到：[`https://blogs.oracle.com/oraclemagazine/post/the-trouble-with-triggers`](https://blogs.oracle.com/oraclemagazine/post/the-trouble-with-triggers)。

想要实现乐观并发控制的应用程序只需保存此额外列的值，而无需保存其他所有列的前镜像。应用程序只需验证在请求更新时，数据库中此列的值是否与最初读取的值匹配。如果这些值相同，则说明该行未被更新过。

让我们看一个使用 `SCOTT.DEPT` 表的副本实现乐观锁的例子。我们可以使用以下数据定义语言（DDL）来创建表：

```sql
$ sqlplus eoda/foo@PDB1
SQL> create table dept
( deptno     number(2),
dname      varchar2(14),
loc        varchar2(13),
last_mod   timestamp with time zone
default systimestamp
not null,
constraint dept_pk primary key(deptno) );
Table created.
```

然后，我们使用 `INSERT` 语句将 `DEPT` 数据的副本插入此表：

```sql
SQL> insert into dept( deptno, dname, loc )
select deptno, dname, loc  from scott.dept;
4 rows created.
SQL> commit;
Commit complete.
```

上述代码重新创建了 `DEPT` 表，但增加了一个 `LAST_MOD` 列，该列使用了 `TIMESTAMP WITH TIME ZONE` 数据类型。我们将此列定义为 `NOT NULL` 以便必须填充它，其默认值为当前系统时间。

这种 `TIMESTAMP` 数据类型在 Oracle 中具有最高的精度，通常精确到微秒（百万分之一秒）。对于涉及用户思考时间的应用程序，`TIMESTAMP` 的这种精度级别是绰绰有余的，因为数据库检索一行数据，以及人类查看、修改它并向数据库发回更新的过程，几乎不可能在几分之一秒内完成。两人在同一瞬间读取并修改同一行的可能性确实非常小。

接下来，我们需要一种方法来维护此值。我们有两个选择：要么应用程序通过在更新记录时将其值设置为 `SYSTIMESTAMP` 来维护 `LAST_MOD` 列，要么使用触发器/存储过程来维护它。让应用程序维护 `LAST_MOD` 绝对比基于触发器的方法性能更好，因为触发器会在 Oracle 已完成的处理之上增加额外的处理。然而，这确实意味着您依赖所有应用程序在修改此表的任何地方都一致地维护 `LAST_MOD`。因此，如果每个应用程序都负责维护此字段，它就需要一致地验证 `LAST_MOD` 列是否未被更改，并将该列设置为当前的 `SYSTIMESTAMP`。例如，如果一个应用程序查询 `DEPTNO=10` 的行：

```sql
SQL> variable deptno   number
SQL> variable dname    varchar2(14)
SQL> variable loc      varchar2(13)
SQL> variable last_mod varchar2(50)
SQL>
SQL> begin
:deptno := 10;
select dname, loc, to_char( last_mod, 'DD-MON-YYYY HH.MI.SSXFF AM TZR' )
into :dname,:loc,:last_mod
from dept
where deptno = :deptno;
end;
/
PL/SQL procedure successfully completed.
```

我们可以看到其当前值是

```sql
SQL> select :deptno dno, :dname dname, :loc loc, :last_mod lm  from dual;
DNO DNAME        LOC        LM
------- ------------ ---------- ----------------------------------------
10 ACCOUNTING   NEW YORK   15-APR-2014 07.04.01.147094 PM -06:00
```

我们将使用下面的更新语句来修改信息。最后一行执行了非常重要的检查，以确保时间戳没有改变，并使用内置函数 `TO_TIMESTAMP_TZ`（`tz` 是时区的缩写）将我们在 `SELECT` 语句中保存的字符串转换回正确的数据类型。此外，`UPDATE` 语句的第 3 行在发现行被更新时，将 `LAST_MOD` 列更新为当前时间：

```sql
SQL> update dept
set dname = initcap(:dname),
last_mod = systimestamp
where deptno = :deptno
and last_mod = to_timestamp_tz(:last_mod, 'DD-MON-YYYY HH.MI.SSXFF AM TZR' );
1 row updated.
```

如您所见，更新了一行，正是我们关心的那行。我们通过主键 (`DEPTNO`) 更新了该行，并验证了在首次读取它和执行更新之间的时间段内，`LAST_MOD` 列没有被任何其他会话修改过。如果我们尝试再次更新同一条记录，使用相同的逻辑但不获取新的 `LAST_MOD` 值，我们将观察到以下情况：

```sql
SQL> update dept
set dname = upper(:dname),
last_mod = systimestamp
where deptno = :deptno
and last_mod = to_timestamp_tz(:last_mod, 'DD-MON-YYYY HH.MI.SSXFF AM TZR' );
0 rows updated.
```

注意，这次报告 `0 rows updated`，因为 `LAST_MOD` 上的谓词条件未满足。虽然 `DEPTNO 10` 仍然存在，但在我们希望更新时，该值已不再与查询该行时的时间戳值匹配。因此，应用程序知道数据库中的数据已被更改——基于没有行被修改这一事实——并且它现在必须决定如何处理这种情况。

您不会依赖每个应用程序来维护此字段，原因有很多。其一，它增加了应用程序的代码量，而且这些代码必须在修改此表的任何地方重复并正确实现。在一个大型应用程序中，这可能在很多地方出现。此外，未来开发的每个应用程序也必须遵守这些规则。应用程序代码中有很多可能遗漏的地方，从而导致此字段未被正确使用。因此，如果应用程序代码本身不负责维护这个 `LAST_MOD` 字段，那么我认为应用程序也不应该负责检查这个 `LAST_MOD` 字段（如果它能进行检查，它当然也能执行更新）。所以，在这种情况下，我建议*将更新逻辑封装在存储过程中*，根本不允许应用程序直接更新表。如果它不值得信任去维护此字段中的值，那么它也不值得信任去正确地检查它。因此，存储过程将把我们在之前的更新中使用的绑定变量作为输入，并执行完全相同的更新。在检测到更新了零行时，存储过程可以向客户端引发一个异常，以告知客户端更新实际上失败了。


## 使用校验和进行乐观锁定

另一种实现方式是使用触发器来维护`LAST_MOD`字段，但对于如此简单的功能，我的建议是避免使用触发器，而让 DML 语句来处理它。触发器会引入可测量的开销，并且在当前场景下是不必要的。此外，触发器无法确认行是否未被修改（它只能为`LAST_MOD`提供值，无法在更新时检查它）；因此，应用程序必须痛苦地意识到这个列以及如何正确使用它。所以，触发器本身并不足够。

## 使用校验和进行乐观锁定

这种方法与之前的版本列方法非常相似，但它使用基础数据本身来计算一个“虚拟”版本列。在展示如何使用其中一个提供的包之前，我将引用*Oracle 数据库 PL/SQL 包和类型参考*手册来帮助解释校验和或哈希函数背后的目标和概念：

> *单向哈希函数接收一个可变长度的输入字符串（数据），并将其转换为一个固定长度（通常更小）的输出字符串，称为哈希值。哈希值作为输入数据的唯一标识符（类似于指纹）。您可以使用哈希值来验证数据是否已被更改。*

> *请注意，单向哈希函数是一种不易逆转的哈希函数。从输入数据计算哈希值很容易，但生成哈希到特定值的数据却很困难。*

我们可以像使用版本列一样使用这些哈希或校验和。我们只需将从数据库读取数据时获得的哈希或校验和值与修改数据前获得的值进行比较。如果在我们读取数据之后、更新数据之前，有人修改了行的值，那么哈希或校验和几乎肯定会不同。

计算哈希或校验和的方法有很多。我将列出几种并在本节演示其中一种。所有这些方法都基于数据库提供的功能：

*   `DBMS_CRYPTO.HASH`：此方法能够计算安全哈希算法 1（SHA-1）或 MD4/MD5 消息摘要。建议使用 SHA-1 算法。
*   `DBMS_SQLHASH.GETHASH`：此方法支持 SHA-1、MD4 和 MD5 哈希算法。作为`SYSDBA`特权用户，您必须先将此包的执行权限授予用户，然后他们才能访问它。此包在*Oracle 数据库安全指南*中有文档说明。
*   `ORA_HASH`：此方法是一个内置的 SQL 函数，它接受一个`VARCHAR2`值作为输入，并（可选地）接受另一个控制返回值的输入对。返回值是一个数字——默认情况下是 0 到 4294967295 之间的数字。
*   `STANDARD_HASH`：这是一个内置的 SQL 函数，它使用标准哈希算法（如`SHA1`（默认）、`SHA256`、`SHA384`、`SHA512`和`MD5`）对表达式计算哈希值。返回值是一个`RAW`数据类型。

**注意**

许多编程语言都提供了一系列哈希和校验和函数，因此数据库之外可能还有其他可用的方法。也就是说，如果使用内置的数据库功能，您将提高未来（针对新语言、新方法）的可移植性。

以下示例展示了如何使用`ORA_HASH`内置函数来计算这些哈希/校验和。该技术也适用于其他列出的方法；逻辑不会有太大不同，但您调用的 API 会有所不同。首先，我们移除上一个示例中使用的列：

```sql
$ sqlplus eoda/foo@PDB1
SQL> alter table dept drop column last_mod;
Table altered.
```

然后让我们的应用程序查询并显示部门 10 的信息。请注意，在查询信息时，我们使用`ORA_HASH`内置函数计算哈希。这是我们保留在应用程序中的版本信息。以下是我们查询和显示的代码：

```sql
SQL> variable deptno number
SQL> variable dname varchar2(14)
SQL> variable loc varchar2(13)
SQL> variable hash number
SQL> begin
select deptno, dname, loc,
ora_hash( dname || '/' || loc ) hash
into :deptno, :dname, :loc, :hash
from dept
where deptno = 10;
end;
/
PL/SQL procedure successfully completed.
SQL> select :deptno, :dname, :loc, :hash from dual;
:DEPTNO :DNAME     :LOC            :HASH
---------- ---------- ---------- ----------
10 Accounting NEW YORK   2721972020
```

如您所见，哈希只是某个数字。这是我们更新前想要使用的值。要更新该行，我们需要在数据库中锁定当前存在的该行，然后将该行的哈希值与我们从数据库读取数据时计算的哈希值进行比较。执行此操作的逻辑可能如下所示：

```sql
SQL> exec :dname := lower(:dname);
PL/SQL procedure successfully completed.
SQL> update dept
set dname = :dname
where deptno = :deptno
and ora_hash( dname || '/' || loc ) = :hash
/
1 row updated.
SQL> select dept.*,
ora_hash( dname || '/' || loc ) hash
from dept
where deptno = :deptno;
DEPTNO DNAME      LOC              HASH
---------- ---------- ---------- ----------
10 accounting NEW YORK   2818855829
```

在更新后重新查询数据并再次计算哈希后，我们可以看到哈希值不同了。如果在我们之前有人修改了该行，我们的哈希值将无法匹配。我们可以通过使用第一次读取时的*旧*哈希值再次尝试更新来观察这一点：

```sql
SQL> update dept
set dname = :dname
where deptno = :deptno
and ora_hash( dname || '/' || loc ) = :hash
/
0 rows updated.
```

如您所见，由于我们的哈希值与数据库中的当前数据不匹配，因此更新了零行。

为了使这种基于哈希的方法正常工作，我们必须确保每个应用程序在计算哈希时使用相同的方法；具体来说，它们必须按`dname`、“`/`”、`loc`的顺序连接。为了使这种方法通用化，我建议向表添加一个虚拟列或使用视图来添加一列，从而将函数对应用程序本身隐藏。添加列如下所示：

```sql
SQL> alter table dept
add hash as
( ora_hash(dname || '/' || loc ) );
Table altered.
SQL> select *  from dept where deptno = :deptno;
DEPTNO DNAME      LOC              HASH
---------- ---------- ---------- ----------
10 accounting NEW YORK   2818855829
```

添加的列是一个虚拟列，因此不会产生存储开销。该值不是在磁盘上计算和存储的，而是在从数据库检索数据时计算的。

此示例展示了如何使用哈希或校验和实现乐观锁定。您应该记住，计算哈希或校验和是某种程度上 CPU 密集型的操作；它的计算成本很高。在 CPU 带宽是稀缺资源的系统上，您必须考虑这一事实。然而，这种方法对网络更友好，因为通过网络传输相对较小的哈希值，而不是传输行的前后映像（逐列比较），将消耗少得多的网络资源。



### 乐观锁还是悲观锁？

那么，哪种方法最好呢？根据我的经验，悲观锁在 Oracle 中表现非常好（但在其他数据库中可能没那么好），并且相比乐观锁有许多优势。然而，它需要与数据库建立有状态的连接，比如客户端/服务器连接。这是因为锁不会跨连接保持。仅这一事实就使得悲观锁在当今许多情况下变得不切实际。在过去，对于客户端/服务器应用程序和几十或几百个用户来说，这曾经是我的首选也是唯一的选择。然而今天，对于大多数应用程序，我会推荐采用**乐观并发控制**。为整个事务持续时间保持一个连接，代价实在太高了。

在可用的方法中，我使用哪一种？我倾向于使用带有时间戳列的版本列方法。从长远来看，它为我提供了额外的更新信息。此外，它比哈希或校验和的计算开销更小，而且在处理 `LONG`、`LONG RAW`、`CLOB`、`BLOB` 和其他非常大的列时，不会遇到哈希或校验和可能遇到的问题（`LONG` 和 `LONG RAW` 已过时；我在这里提到它们只是因为它们仍然在 Oracle 数据字典中频繁使用）。

如果我必须为一个仍在使用悲观锁定方案（例如，该表同时在客户端/服务器应用程序和 Web 上被访问）的表添加乐观并发控制，我会选择 `ORA_HASH` 方法。原因是现有的遗留应用程序可能不欢迎出现一个新列。即使我们采取额外步骤来隐藏这个额外的列，应用程序也可能因必要的触发器带来的开销而受到影响。在这方面，`ORA_HASH` 技术将是**非侵入式且轻量级的**。哈希/校验和方法可以做到非常独立于数据库，特别是当我们在数据库外部计算哈希或校验和时。然而，通过在中间层而非数据库中执行计算，我们将在 CPU 使用率和网络传输方面招致更高的资源使用惩罚。

### 阻塞

`阻塞` 发生在一个会话持有一个资源锁，而另一个会话正在请求该资源时。结果，请求会话将被阻塞——它会挂起，直到持有会话放弃被锁定的资源。在几乎所有情况下，阻塞都是可以避免的。事实上，如果你确实在交互式应用程序中发现你的会话被阻塞了，那么你很可能也一直遭受着丢失更新错误，或许自己还没意识到。也就是说，你的应用程序逻辑存在缺陷，而这正是阻塞的起因。

在数据库中会造成阻塞的五种常见 DML 语句是 `INSERT`、`UPDATE`、`DELETE`、`MERGE` 和 `SELECT FOR UPDATE`。解决被阻塞的 `SELECT FOR UPDATE` 很简单：只需添加 `NOWAIT` 子句，它就不会再阻塞。相反，你的应用程序会向最终用户返回一条消息，提示该行已被锁定。有趣的情况是剩下的四种 DML 语句。我们将逐一查看它们，看看它们为什么不应该阻塞，以及如果发生阻塞该如何纠正。

#### 被阻塞的插入操作

有几种情况 `INSERT` 会阻塞。最常见的场景是，当你的表上设置了主键或唯一约束，并且两个会话试图插入具有相同值的行时。其中一个会话将被阻塞，直到另一个会话要么提交（在这种情况下，被阻塞的会话将收到关于重复值的错误），要么回滚（在这种情况下，被阻塞的会话成功插入）。另一种情况涉及通过引用完整性约束链接的表。如果子表所依赖的父行正在被创建或删除，那么对子表的 `INSERT` 可能会被阻塞。

被阻塞的 `INSERT` 通常发生在允许最终用户生成主键/唯一列值的应用程序中。这种情况最容易通过使用序列或 `SYS_GUID()` 内置函数来生成主键/唯一列值来避免。序列/`SYS_GUID()` 被设计为在多用户环境中生成唯一键的高度并发的方法。万一你不能使用这两者，并且必须允许最终用户生成可能重复的键，你可以使用以下技术，该技术通过使用内置的 `DBMS_LOCK` 包实现手动锁来避免该问题。

> **注意**
>
> 以下示例演示了如何防止会话因主键或唯一约束而在插入语句上发生阻塞。需要强调的是，这里演示的修复方法应被视为一种短期解决方案，同时应检查应用程序架构本身。这种方法会增加明显的开销，不应轻易实施。一个设计良好的应用程序不会遇到此问题（例如，在并发环境中，你不会有持续数小时的事务）。这应该被视为最后的手段，绝对不是你希望为了“以防万一”而对应用程序中的每个表都做的事情。

对于插入操作，没有现有的行可供选择和锁定；无法防止其他人插入具有相同值的行，从而阻塞我们的会话并导致无限期等待。这就是 `DBMS_LOCK` 发挥作用的地方。为了演示这种技术，我们将创建一个带有主键的表和一个触发器，该触发器将防止两个（或更多）会话同时插入相同的值。触发器将使用 `DBMS_UTILITY.GET_HASH_VALUE` 将主键哈希到 0 到 1,073,741,823 之间的一个数字（Oracle 允许我们使用的锁 ID 号范围）。在这个例子中，我选择了大小为 1024 的哈希表，意味着我们将把主键哈希到 1024 个不同的锁 ID 之一。然后，我们将使用 `DBMS_LOCK.REQUEST` 基于该 ID 分配一个排他锁。一次只有一个会话能够做到这一点，所以如果其他人试图向我们的表中插入具有相同主键的记录，该人的锁请求将失败（并引发 `resource busy` 错误）：

> **注意**
>
> 要成功编译此触发器，必须直接向您的模式授予 `DBMS_LOCK` 的执行权限。执行 `DBMS_LOCK` 的权限可能无法通过角色获得。

```
$ sqlplus scott/tiger@PDB1
SQL> create table demo ( x int primary key );
Table created.
SQL> create or replace trigger demo_bifer
before insert on demo
for each row
declare
l_lock_id   number;
resource_busy   exception;
pragma exception_init( resource_busy, -54 );
begin
l_lock_id :=
dbms_utility.get_hash_value( to_char( :new.x ), 0, 1024 );
if ( dbms_lock.request
(  id                => l_lock_id,
lockmode          => dbms_lock.x_mode,
timeout           => 0,
release_on_commit => TRUE ) not in (0,4) )
then
raise resource_busy;
end if;
end;
/
Trigger created.
SQL> insert into demo(x) values (1);
1 row created.
```


## 死锁

*死锁*发生当你有两个会话，每个会话都持有对方想要的资源时。例如，如果我的数据库中有两个表 `A` 和 `B`，每个表中都只有一行数据，我可以轻松演示一个死锁。我只需要打开两个会话（例如，两个 `SQL*Plus` 会话）。在会话 `A` 中，我更新表 `A`。在会话 `B` 中，我更新表 `B`。现在，如果我在会话 `B` 中尝试更新表 `A`，我将被阻塞。因为会话 `A` 已经锁定了这一行。这不是死锁；这仅仅是阻塞。我尚未造成死锁，因为有机会会话 `A` 会提交或回滚，然后会话 `B` 在那时继续执行。

如果我回到会话 `A` 并尝试更新表 `B`，我就会造成死锁。两个会话中的一个将被选为受害者，其语句将被回滚。例如，会话 `B` 更新表 `A` 的尝试可能会被回滚，并出现类似以下的错误：

```
update a set x = x+1
*
ERROR at line 1:
ORA-00060: deadlock detected while waiting for resource
```

会话 `A` 更新表 `B` 的尝试将保持阻塞状态——`Oracle` 不会回滚整个事务。只有导致死锁的其中一条语句被回滚。会话 `B` 仍然锁定了表 `B` 中的行，而会话 `A` 正在耐心地等待该行变得可用。在收到死锁消息后，会话 `B` 必须决定是提交在表 `B` 上未完成的工作、回滚它，还是走另一条路径并在稍后提交。一旦该会话提交或回滚，另一个被阻塞的会话就会像什么都没发生过一样继续执行。

`Oracle` 认为死锁非常罕见且不寻常，以至于每次发生时都会在服务器上创建一个跟踪文件。跟踪文件的内容大致如下：

```
DEADLOCK DETECTED ( ORA-00060 )
See Note 60.1 at My Oracle Support for Troubleshooting ORA-60 Errors
[Transaction Deadlock]
The following deadlock is not an ORACLE error. It is a
deadlock due to user error in the design of an application
or from issuing incorrect ad-hoc SQL. The following
information may aid in determining the deadlock:
...
```

显然，`Oracle` 认为这些应用程序死锁是应用程序自身引发的错误，而且在大多数情况下，`Oracle` 是对的。与许多其他 `RDBMS` 不同，死锁在 `Oracle` 中非常罕见，几乎可以认为不存在。通常，你必须人为地构造条件才能引发一个。

根据我的经验，`Oracle` 数据库中死锁的首要原因是**未索引的外键**。（第二个原因是针对可能并发更新的表上的位图索引，我们将在第 11 章介绍。）在三种场景下，`Oracle` 会在修改父表后对子表放置一个全表锁：

*   如果你更新了父表的主键（如果你遵循关系数据库的规则——主键应该是不可变的，这种情况就非常罕见），在没有外键索引的情况下，子表将被锁定。
*   如果你删除了父表的一行，在没有外键索引的情况下，整个子表也将被锁定。

在 `Oracle 9*i*` 及以上版本中，这些全表锁是短期存在的，意味着它们只需要在 `DML` 操作期间持有，而不是整个事务期间。即便如此，它们能够也确实会导致严重的锁问题。作为第一点的演示，如果我们设置如下一对表，暂时还不会发生异常：

```
$ sqlplus eoda/foo@PDB1
SQL> create table p ( x int primary key );
Table created.
SQL> create table c ( x references p );
Table created.
SQL> insert into p values ( 1 );
1 row created.
SQL> insert into p values ( 2 );
1 row created.
SQL> commit;
Commit complete.
SQL> insert into c values ( 2 );
1 row created.
```

现在，为了在一个会话中演示我们捕获到这个阻塞的 `INSERT` 问题，我们将使用一个 `AUTONOMOUS_TRANSACTION`，使得下一段代码的执行看起来像是在另一个 `SQL*Plus` 会话中进行的。实际上，如果你使用另一个会话，行为将是一样的。开始演示：

```
SQL> declare
pragma autonomous_transaction;
begin
insert into demo(x) values (1);
commit;
end;
/
declare
*
ERROR at line 1:
ORA-00054: resource busy and acquire with NOWAIT specified or timeout expired
ORA-06512: at "SCOTT.DEMO_BIFER", line 14
ORA-04088: error during execution of trigger 'SCOTT.DEMO_BIFER'
ORA-06512: at line 4
```

这里的核心概念是，将受触发器保护的表的给定主键值放入一个字符串中。然后我们可以使用 `DBMS_UTILITY.GET_HASH_VALUE` 为该字符串生成一个基本唯一的哈希值。只要我们使用的哈希表大小小于 1,073,741,823，我们就可以使用 `DBMS_LOCK` 独占锁定该值。

哈希处理后，我们获取该值并使用 `DBMS_LOCK` 请求以超时时间 `ZERO` 独占锁定该锁 ID（如果其他人已锁定该值，此操作会立即返回）。如果我们超时或因任何原因失败，我们就抛出 `ORA-54 Resource Busy` 错误。否则，我们什么都不做——可以插入，我们不会阻塞。在提交我们的事务时，所有的锁，包括此 `DBMS_LOCK` 调用分配的锁，都将被释放。

当然，如果你的表的主键是 `INTEGER` 并且你预计键值不会超过十亿，你可以跳过哈希步骤，直接使用该数字作为锁 ID。

你需要调整哈希表的大小（本例中为 1024）以避免因不同的字符串哈希到相同数字而产生人为的“资源忙”消息。哈希表的大小将取决于应用程序（数据）特性，并且也会受到并发插入数量的影响。你还可以在触发器中添加一个标志，允许人们打开或关闭此检查。例如，如果我打算插入数百或数千条记录，我可能就不希望启用此检查。

### 被阻塞的合并、更新和删除

在一个交互式应用程序中——即从数据库查询一些数据，允许最终用户操作它，然后将其放回数据库——一个被阻塞的 `UPDATE` 或 `DELETE` 表明你的代码中可能存在**丢失更新**问题。（如果你的代码中出现了这种情况，我会称之为一个缺陷。）你正尝试更新一个其他人已经在更新的行（换句话说，一个其他人已经锁定的行）。你可以通过使用 `SELECT FOR UPDATE NOWAIT` 查询来避免阻塞问题，以便：

*   验证自你查询出数据以来数据未更改（防止丢失更新）
*   锁定该行（防止 `UPDATE` 或 `DELETE` 被阻塞）

如前所述，无论你采用哪种锁定方法，你都可以这样做。悲观锁定和乐观锁定都可以采用 `SELECT FOR UPDATE NOWAIT` 查询来验证行是否未更改。悲观锁定会在用户表示其想要修改数据的那一刻就使用该 `SELECT FOR UPDATE NOWAIT` 语句。乐观锁定则会在更新数据库中的数据之前立即使用该语句。这不仅会解决应用程序中的阻塞问题，还会纠正数据完整性问题。

由于 `MERGE` 本质上就是一个 `INSERT` 和 `UPDATE`（并且在较新版本的 `Oracle` 中，使用增强的 `MERGE` 语法，它还包含 `DELETE`），因此你需要同时使用这两种技术。
