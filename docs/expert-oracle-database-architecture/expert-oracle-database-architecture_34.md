# 大型事务与锁策略

现在，为了给这个讨论提供一个反面观点，回想一下我们在上一章讨论的写一致性概念，以及一个 `UPDATE` 语句如何可能需要重启。如果前面的 `UPDATE` 语句是针对行的一个子集执行的（它有一个 `WHERE` 子句，并且其他用户正在修改这个 `UPDATE` 的 `WHERE` 子句中使用的列），那么要么需要使用一系列较小的事务而不是一个大事务，要么在执行批量更新之前锁定表。这里的目标是减少发生重启的机会。

如果我们要 `UPDATE` 表中绝大多数行，这会引导我们使用 `LOCK TABLE` 命令。然而，根据我的经验，这种大规模的批量更新或批量删除（唯一真正会面临重启风险的语句类型）都是在隔离状态下完成的。那种大型的一次性批量更新或旧数据清除，通常不是在高活动期间进行的。确实，数据清除根本不应该受此影响，因为你通常会使用某个日期字段来定位要清除的信息，而其他应用程序不会修改这些数据。
