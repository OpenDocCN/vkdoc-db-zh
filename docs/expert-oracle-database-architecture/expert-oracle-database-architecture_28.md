# Oracle 事务隔离级别与多版本并发控制

除了四个标准的 SQL 隔离级别外，Oracle 还提供了另一个级别，即`READ ONLY`。`READ ONLY`事务等同于在 SQL 中无法执行任何修改的`REPEATABLE READ`或`SERIALIZABLE`事务。使用`READ ONLY`隔离级别的事务只看到那些*在事务开始时*已提交的更改，但在此模式下不允许插入、更新和删除（其他会话可以更新数据，但`READ ONLY`事务本身不能）。使用此模式，你可以实现`REPEATABLE READ`和`SERIALIZABLE`级别的隔离。

现在，让我们继续详细讨论多版本并发控制和读一致性如何融入隔离方案，以及不支持多版本并发控制的数据库如何实现相同的结果。这些信息对于使用过其他数据库并认为自己理解隔离级别工作原理的人很有指导意义。有趣的是，一个旨在消除数据库之间差异的标准，即 ANSI/ISO SQL 标准，实际上允许了差异的存在。该标准虽然非常详细，但可以用非常不同的方式来实现。

### READ UNCOMMITTED

`READ UNCOMMITTED`隔离级别允许脏读。Oracle 不使用脏读，甚至不允许脏读。`READ UNCOMMITTED`隔离级别的基本目标是提供一个基于标准的定义，以支持非阻塞读。正如我们所看到的，Oracle 默认就提供了非阻塞读。在数据库中，你很难让一个`SELECT`查询阻塞（如前所述，分布式事务是一个特例）。每一个查询，无论是`SELECT`、`INSERT`、`UPDATE`、`MERGE`还是`DELETE`，都以读一致的方式执行。将`UPDATE`语句称为查询可能看起来有点奇怪，但它确实是。`UPDATE`语句有两个组成部分：由`WHERE`子句定义的读组件和由`SET`子句定义的写组件。`UPDATE`语句读写数据库；所有 DML 语句都有这个能力。使用`VALUES`子句的单行`INSERT`是唯一的例外，因为此类语句没有读组件，只有写组件。

在第 1 章中，通过一个简单的单表查询演示了 Oracle 获取读一致性的方法，该查询检索了在游标打开后被删除的行。现在，我们将探索一个真实世界的例子，看看 Oracle 使用多版本并发控制时会发生什么，以及其他许多数据库中会发生什么。

让我们从一个基本的表和查询开始：

```
SQL> create table accounts
( account_number number primary key,
account_balance number not null);
SQL> select sum(account_balance) from accounts;
```

在查询开始之前，假设我们有表 7-2 所示的数据。

表 7-2 修改前的 ACCOUNTS 表

| 行 | 账户号码 | 账户余额 |
| --- | --- | --- |
| 1 | 123 | $500.00 |
| 2 | 456 | $240.25 |
| … | … | … |
| 342,023 | 987 | $100.00 |

现在，我们的`select`语句开始执行，并读取第 1 行、第 2 行，依此类推。

> 注意
>
> 在这个例子中，我并不是说磁盘上的行有任何物理排序。表中实际上没有第一行、第二行或最后一行。只有一组行。我们这里假设第 1 行的意思是“我们碰巧读到的第一行”，第 2 行是“我们碰巧读到的第二行”，依此类推。

在我们查询执行过程中的某个时刻，一个事务将$400.00 从账户 123 转移到账户 987。该事务执行了两个更新操作但并未提交。此时表看起来如表 7-3 所示。

表 7-3 修改期间的 ACCOUNTS 表

| 行 | 账户号码 | 账户余额 | 锁定？ |
| --- | --- | --- | --- |
| 1 | 123 | ($500.00) 变为 $100.00 | X |
| 2 | 456 | $240.25 | -- |
| … | … | … | -- |
| 342,023 | 987 | ($100.00) 变为 $500.00 | X |

因此，这两行被锁定了。如果有人试图更新它们，该用户将被阻塞。到目前为止，我们在所有数据库中看到的行为大致相同。区别将出现在查询遇到被锁定的数据时。

当我们执行的查询到达表底部包含被锁定行（第 342,023 行）的数据块时，它会注意到其中的数据自查询开始执行以来已经改变。为了提供一致（正确）的答案，Oracle 此时会创建包含此行的数据块的副本，*该副本是查询开始时该块存在的状态*。也就是说，它将读取$100.00 的值，即查询开始时存在的值。实际上，Oracle 绕过了修改后的数据；它从回滚段（也称为*撤销段*）中读取并重建数据。无需等待事务提交，就能返回一个一致且正确的答案。

现在，一个允许脏读的数据库会直接返回它在读取时看到的账户 987 的值，在这个例子中是$500.00。该查询将把转移的$400 计算两次。因此，它不仅返回了错误的答案，而且返回了一个在表中任何已提交时间点都从未存在过的总数。在多用户数据库中，脏读可能是一个危险的功能，就我个人而言，我从未见过它的用处。假设该交易不是转移，而实际上只是向账户 987 存入$400.00。脏读会计算这$400 并得到“正确”的答案，不是吗？但是，假设未提交的交易被回滚了。我们刚刚计算了一个从未实际存在于数据库中的$400。

这里的要点是，脏读不是一个特性；相反，它是一个负担。在 Oracle 中，根本不需要它。你获得了脏读的所有优点（无阻塞），却没有任何不正确的结果。



### 已提交读 (`READ COMMITTED`)

`READ COMMITTED` 隔离级别规定，事务只能读取数据库中已提交的数据。**不存在脏读**。但可能存在**不可重复读**（即，在同一事务中重新读取同一行可能返回不同结果）和**幻读**（即，事务执行过程中，之前不可见的新插入并已提交的行对查询变得可见）。`READ COMMITTED` 可能是全球数据库应用中最常用的隔离级别，它是 Oracle 数据库的默认模式；很少看到使用其他隔离级别。

然而，实现 `READ COMMITTED` 隔离并不像听起来那么简单明了。如果你查看表 7-1，它看起来很直接。显然，根据之前的规则，在任何使用 `READ COMMITTED` 隔离级别的数据库中执行的查询，其行为方式应该是相同的，不是吗？***并非如此。*** 如果你在一条语句中查询多行数据，那么在几乎所有其他数据库中，`READ COMMITTED` 隔离的效果可能糟糕得如同脏读，具体取决于实现方式。

在 Oracle 中，利用多版本机制和读一致性查询，我们从 `ACCOUNTS` 查询得到的结果与 `READ UNCOMMITTED` 示例中相同。Oracle 会重构数据，使其看起来如同查询开始时那样，返回查询开始时数据库中存在的答案。

现在，让我们看看我们之前的示例在其他数据库的 `READ COMMITTED` 模式下可能如何工作——你可能会对答案感到惊讶。我们从前面表格描述的时点继续我们的示例：

*   我们正处于表格处理的中途。我们已经读取并汇总了前 N 行。
*   另一个事务已将 $400.00 从账户 123 转移到账户 987。
*   该事务尚未提交，因此包含账户 123 和 987 信息的行被锁定。

我们知道当 Oracle 处理到账户 987 时会发生什么——它会绕过被修改的数据读取，查明其值应为 $100.00，然后完成查询。表 7-4 展示了另一个运行在某种默认 `READ COMMITTED` 模式的数据库可能如何得出答案。

表 7-4

非 Oracle 数据库使用 READ COMMITTED 隔离级别的时间线

| 时间 | 查询 | 账户转账事务 |
| --- | --- | --- |
| T1 | 读取第 1 行，账户 123，值=$500。当前总和=$500.00。 | -- |
| T2 | 读取第 2 行，账户 456，值=$240.25。当前总和=$740.25。 | -- |
| T3 | -- | 更新第 1 行（账户 123）并对第 1 行施加排他锁，阻止其他更新和读取。第 1 行原为 $500.00，现在为 $100.00。 |
| T4 | 读取第 N 行。总和 = … | -- |
| T5 | -- | 更新第 342,023 行（账户 987）并对该行施加排他锁。该行原为 $100；现在为 $500.00。 |
| T6 | 尝试读取第 342,023 行，账户 987。发现该行已被锁定。此会话将阻塞并等待该行的数据块变为可用。*此查询上的所有处理停止。* | -- |
| T7 | -- | 提交事务。 |
| T8 | 读取第 342,023 行，账户 987，看到 $500.00，并呈现一个最终答案，其中 $400.00 被重复计算。 | -- |

首先要注意的是，这个其他数据库在处理到账户 987 时，会阻塞我们的查询。此会话必须等待该行，直到持有排他锁的事务提交。这就是为什么许多人养成了每条语句都提交的坏习惯，而不是处理由将数据库从一个一致状态转移到下一个一致状态所需的所有语句组成、结构良好的事务。*在大多数其他数据库中，更新会干扰读取。* 在此场景中，真正糟糕的消息是，我们让最终用户等待一个***错误***的答案。我们仍然收到了一个答案，这个答案在数据库已提交状态的任何时间点都从未存在过，如同脏读一样，但这次我们让用户等待了一个错误的答案。在下一节中，我们将探讨这些其他数据库需要做什么才能实现读一致、正确的结果。

这里的重要教训是，在相同、看似安全的隔离级别下运行的各种数据库，在完全相同的情况下可能并且将会返回非常不同的答案。重要的是要理解，在 Oracle 中，非阻塞读取并非以牺牲正确答案为代价。有时，你可以两者兼得。

### 可重复读 (`REPEATABLE READ`)

`REPEATABLE READ` 的目标是提供一个能给出一致、正确答案并防止丢失更新的隔离级别。我们将分别查看两者的示例，了解在 Oracle 中实现这些目标需要做什么，并考察其他系统中发生的情况。
