# 外键是否应该被索引？

关于外键是否应该被索引的问题经常出现。我们在第 6 章讨论死锁时曾触及过这个话题。在那时，我指出未索引的外键是我所遇到的导致死锁的最大的单一原因，这是因为对父表主键的更新或父记录的删除将在子表上放置一个表锁（在语句完成之前，不允许对子表进行任何修改）。这会锁定比它应该锁定的更多的行，并降低并发性。

我经常在人们使用工具生成 SQL 来修改表时看到这种情况。该工具生成一个 `UPDATE` 语句，更新表中的每一列，无论该值是否被修改。这实际上更新了主键（即使他们从未更改该值）。例如，Oracle Forms 默认会这样做，除非你告诉它只将修改过的列发送到数据库。

除了可能遇到的表锁问题外，未索引的外键在以下情况下同样糟糕：

*   当你使用了 `ON DELETE CASCADE` 但没有为子表建立索引时。例如，`EMP` 是 `DEPT` 的子表。`DELETE FROM DEPT WHERE DEPTNO = 10` 应该级联到 `EMP`。如果 `EMP` 中的 `DEPTNO` 没有索引，你将得到对 `EMP` 的全表扫描。这种全扫描通常是不受欢迎的，而且如果你从父表中删除多行，子表将为每一个被删除的父行扫描一次。

*   当你从父表查询到子表时。再次考虑 `EMP`/`DEPT` 的例子。在 `DEPTNO` 的上下文中查询 `EMP` 表是非常常见的。如果你经常执行以下查询来生成报告或其他内容，你会发现没有索引会减慢查询速度：

    ```
    select *
    from dept, emp
    where emp.deptno = dept.deptno
    and dept.dname = :X;
    ```

这与我在第 10 章中为嵌套表的 `NESTED_COLUMN_ID` 建立索引所给出的论据相同。嵌套表隐藏的 `NESTED_COLUMN_ID` 无非就是一个外键。

那么，什么时候你*不需要*为外键建立索引呢？一般来说，在满足以下条件时：

*   你*不*从父表中删除记录。

*   你*不*更新父表的唯一/主键值，无论是有意的还是无意的（通过工具）。

*   你*不*从父表连接到子表，或者更一般地说，外键列不支持访问子表的重要路径，并且你不使用它们在谓词中选择该表的数据（例如从 `DEPT` 到 `EMP`）。

如果你满足所有三个标准，可以放心地跳过索引——它不需要，并且会减慢子表上的 DML 操作。如果你做了三者中的任何一项，请注意其后果。

另外，请注意，如果你认为子表正因未索引的外键而被锁定，并且你想证明这一点（或者只是想防止这种情况发生），你可以执行以下命令：

```
ALTER TABLE <table_name> DISABLE TABLE LOCK;
```

现在，任何会导致表锁的父表 `UPDATE` 或 `DELETE` 操作都将收到以下错误：

```
ERROR at line 1:
ORA-00069: cannot acquire lock -- table locks disabled for <table_name>
```

这对于追踪那些你认为不应该发生的代码（没有对父主键的 `UPDATE` 或 `DELETE`）很有用，因为最终用户会立即将此错误报告给你。
