# 10. 数据库表

在本章中，我们将讨论各种类型的数据库表，并介绍何时可能希望使用每种类型（即，何时一种表类型比另一种更合适）。我们将集中讨论表的物理存储特性：数据如何组织和存储。

曾几何时，实际上只有一种类型的表：普通表。它的管理方式就像一堆东西的管理（其定义出现在下一节）。随着时间的推移，Oracle 增加了更复杂的表类型。现在，除了堆组织表之外，还有索引聚簇表（三种类型）、索引组织表、嵌套表、临时表、外部表和对象表。每种表类型都有不同的特性，使其适用于不同的应用领域。

## 表的类型

在深入细节之前，我们将定义每种表类型。Oracle 中有九种主要的表类型，如下所示：

*   **堆组织表**：这些是普通的、标准的数据库表。数据以类似堆的方式管理。随着数据的添加，将使用在段中找到的第一个可以容纳该数据的可用空间。随着数据从表中移除，它允许空间可供后续的 `INSERT` 和 `UPDATE` 操作重新使用。这就是“堆”这个名称的由来，它指的是这种类型的表。堆是一堆空间，并且它以一种有些随机的方式被使用。

*   **索引组织表**：这些表存储在索引结构中。这为行本身强加了物理顺序。在堆中，数据被塞进任何可以容纳的地方，而在索引组织表（IOT）中，数据根据主键按排序顺序存储。

*   **索引聚簇表**：聚簇是一个或多个表的组，物理存储在相同的数据库块上，所有共享相同聚簇键值的行在物理上彼此靠近存储。这种结构实现了两个目标。首先，许多表可以物理地连接存储在一起。通常，你会期望在一个数据库块上只找到一个表的数据，但使用聚簇表，来自多个表的数据可能存储在同一个块上。其次，所有包含相同聚簇键值的数据（例如 `DEPTNO = 10`）将在物理上存储在一起。数据围绕着聚簇键值聚集。聚簇键是使用 B*树索引构建的。索引聚簇表的优点是，当访问经常在聚簇键上连接的表时，可以减少磁盘 I/O 并提高查询性能。

*   **哈希聚簇表**：这些表类似于索引聚簇表，但不是使用 B*树索引来通过聚簇键定位数据，哈希聚簇将键哈希到聚簇中，以找到数据应该所在的数据库块。在哈希聚簇中，数据就是索引（打个比方）。这些表适用于通过键上的相等比较频繁读取的数据。

*   **排序哈希聚簇表**：这种表类型结合了哈希聚簇表和索引组织表的某些方面。概念如下：你有一个键值，行将根据它进行哈希（例如，`CUSTOMER_ID`），然后是一系列与该键相关的、按排序顺序到达的记录（基于时间戳的记录），并按该排序顺序处理。例如，客户在你的订单录入系统中下订单，这些订单以先进先出（FIFO）的方式被检索和处理。在这样的系统中，排序哈希聚簇可能是适合你的数据结构。

*   **嵌套表**：这些是 Oracle 对象关系扩展的一部分。它们简单来说是系统生成和维护的、处于父/子关系中的子表。它们的工作方式与 `SCOTT` 模式中的 `EMP` 和 `DEPT` 表类似，其中 `EMP` 表是嵌套表。`EMP` 被认为是 `DEPT` 表的子表，因为 `EMP` 表有一个指向 `DEPT` 的外键——`DEPTNO`。主要区别在于，它们不是独立的堆组织表。

*   **临时表**：这些表在事务或会话的生命周期内存储临时数据。这些表根据需要从当前用户的临时表空间分配临时区。每个会话只会看到该会话分配的区；它永远不会看到在任何其他会话中创建的任何数据。临时表允许你临时持久化数据，好处是比常规堆组织表生成少得多的重做（并且从 Oracle 12*c* 开始也生成更少的撤销）。关于临时表的重做和撤销行为的完整讨论，请参见第 9 章。



## 表类型

*   **对象表**：这些表基于对象类型创建。它们具有非对象表所没有的特殊属性，例如为每一行生成一个系统级的 `REF`。对象表实际上是堆表、索引组织表和临时表的特例，并且它们的结构中也可能包含嵌套表作为其一部分。

*   **区块链表**：区块链表是不可变的（仅可插入）表。随着行被插入，它们被组织成多个链。每行（除了链中的第一行）都链接到前一行。这些行是防篡改的，意味着不能被移除或更新。每一行都使用一个加密哈希值进行签名，该哈希值基于新插入数据的哈希值和链中前一行的哈希值。这使得公司能够维护一个关于当前行和任何历史交易的防篡改审计跟踪。

*   **外部表**：这些表中的数据并不存储在数据库本身中；相反，它们驻留在数据库外部的普通操作系统文件中（文件中的列数据通常由分隔符或位置界定）。外部表使你能够像查询数据库内部的普通堆组织表一样，查询驻留在数据库外部的文件。它们是将数据导入数据库的最有用的手段（它们是非常强大的数据加载工具）。此外，外部表具有卸载功能；这提供了一种在 Oracle 数据库之间移动数据而无需使用数据库链接的简便方法。我们将在第 15 章详细探讨外部表。

以下是一些关于表的通用信息，无论其类型如何：

*   一个表最多可以有 1000 列，但我建议不要设计包含最大列数的表，除非有迫切需要。当列数远少于 1000 时，表的效率最高。Oracle 在内部会将超过 254 列的行存储在相互指向的独立行片段中，必须重新组装这些片段才能生成完整的行映像。

*   一个表可以有几乎无限数量的行，但你会遇到其他限制来阻止这种情况发生。例如，通常一个表空间最多可以有 1022 个文件（尽管有 `BIGFILE` 表空间可以让你超越这些文件大小限制）。假设你有一个典型的表空间，并且使用的是 32GB 大小的文件——也就是说，总大小为 32,704GB。这将是 2,143,289,344 个块，每个块大小为 16KB。你可能能够在一个块中容纳 160 行，每行 80 到 100 字节。这将给你带来 342,926,295,040 行。然而，如果你对表进行分区，你可以轻松地将这个数字乘以很多倍。例如，考虑一个有 1024 个哈希分区的表——那将是 1024 * 342,926,295,040 行。虽然存在限制，但在你甚至接近一个表中拥有 351,156,526,120,960 行之前，你就会遇到其他实际限制。

*   一个表可以拥有的索引数量与列的排列组合数一样多（以及这些列上的函数的排列组合，以及你能想到的任何唯一表达式的排列组合）。随着基于函数的索引的出现，理论上你可以创建的索引数量变得无限！然而，实际的限制，例如整体性能（你添加的每个索引都会给对该表的 `INSERT` 操作增加开销），将限制你实际创建和维护的索引数量。

*   即使在单个数据库内，你可以拥有的表数量也没有限制。但同样，实际的限制会将这个数字保持在合理的范围内。你不会拥有数百万个表（因为这么多表不切实际地创建和管理），但你可能拥有数千个表。

### 术语

在本节中，我们将介绍与表相关的各种存储参数和术语。并非所有参数都用于每种表类型。例如，`PCTUSED` 参数在 IOT 的上下文中没有意义（原因将在第 11 章变得显而易见）。我们将作为每种表类型讨论的一部分来涵盖相关参数。目标是介绍并定义这些术语。在适当的情况下，关于使用特定参数的更多信息将在后续章节中介绍。
