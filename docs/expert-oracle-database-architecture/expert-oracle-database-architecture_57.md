# Oracle 与 SQL Server/Sybase 的临时表使用差异

如果你有在 SQL Server 和/或 Sybase 中使用临时表的经验，那么对你来说主要的考虑是：不要执行 `SELECT X, Y, Z INTO #TEMP FROM SOME_TABLE` 来动态创建和填充临时表，你应该：

*   像创建永久表一样，在应用安装时一次性创建好你所有的全局临时表。

*   在你的存储过程中，只需简单地使用 `INSERT INTO TEMP (X,Y,Z) SELECT X,Y,Z FROM SOME_TABLE`。

为了强调这一点，这里的目标不是在运行时在存储过程中创建表。这不是在 Oracle 中使用临时表的正确方式。DDL 是一个昂贵的操作；你应该避免在运行时执行它。应用的临时表应该在应用安装期间创建——*永远不*要在运行时创建。如果你试图在 PL/SQL 运行时动态创建全局临时表（或任何表），你将遇到以下陷阱：

*   你将在运行时执行 DDL。DDL 极其昂贵；它涉及数百条递归 SQL 语句。DDL 涉及大量的序列化操作（一次一个，排队进行）。

*   你将不得不在 PL/SQL 中使用动态 SQL 来使用这些表。你将失去所有静态、编译时 SQL 的好处。这是一个巨大的损失。

*   你将永远无法同时运行两个存储过程的副本。由于两个存储过程实例都会试图删除并创建同一个临时表，它们会相互冲突（在这种场景下，你可以在每次创建过程时动态生成一个唯一的临时表名，但这会引入复杂性和潜在的排错难题）。

*   你的表最终可能会一直存在——也就是说，你的代码不会正确地删除它们。由于一个不可预见的错误（一次断电就足够了），你的过程可能无法完成。电力恢复后，你的表仍然会在那里。你将不得不时不时地手动清理对象。

简而言之，在 PL/SQL 运行时创建表没有任何好的理由，只有*不*在 PL/SQL 运行时创建表的理由。
