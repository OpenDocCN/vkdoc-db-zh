# 不完整的检查点与重做日志大小

当我们填满日志文件 1 并切换到日志文件 2 时，Oracle 会启动一个检查点。此时，`DBWn` 进程开始将受日志文件组 1 保护的所有脏块写入磁盘。在 `DBWn` 刷新完受该日志文件保护的所有块之前，Oracle 无法重用（覆盖）它。如果我们在 `DBWn` 完成检查点之前尝试重用它，我们的数据库 `ALERT` 日志中就会出现如下消息：

```
...
Thread 1 cannot allocate new log, sequence 16
Checkpoint not complete
Current log# 3 seq# 15 mem# 0: /opt/oracle/oradata/CDB/redo03.log...
```

因此，当此消息出现时，数据库中的处理会暂停，同时 `DBWn` 会急于完成其检查点。Oracle 在那一刻会将所有可能的处理能力都赋予 `DBWn`，希望它能更快完成。

在一个调整良好的数据库实例中，你永远不想看到这条消息。如果你确实看到了，那么你很清楚，你为最终用户引入了人为的、不必要的等待。这总是可以避免的。目标（通常这是针对 DBA，而非开发人员）是分配足够多的联机重做日志文件，以便在日志切换引发的检查点完成之前，你永远不会尝试重用日志文件。如果你经常看到此消息，则意味着 DBA 没有为应用程序分配足够的联机重做日志，或者 `DBWn` 需要调整以更高效地工作。

不同的应用程序会产生不同数量的重做日志。决策支持系统（DSS，仅查询）或数据仓库（DW）系统自然会比 OLTP（事务处理）系统产生的日常联机重做日志少得多。在数据库中对二进制大对象（BLOB）进行大量图像处理的系统可能比简单的订单录入系统产生多得多的重做。一个有 100 个用户的订单录入系统产生的重做量可能只有 1000 个用户的十分之一。因此，对于你的重做日志没有“正确”的大小，尽管你确实需要确保它们对于你独特的工作负载足够大。

在设置联机重做日志的大小和数量时，你必须考虑许多事情。其中许多超出了本书的范围，但我会列出一些让你有个概念：

*   **高峰工作负载**：你希望系统不必等待检查点不完整消息，不会在高峰处理期间出现瓶颈。你应该根据你的高峰处理来调整重做日志的大小，而不是根据平均每小时的吞吐量。如果你每天生成 24GB 的日志，但其中 10GB 是在上午 9 点到 11 点之间生成的，你会希望重做日志足够大，以支撑你度过那两个小时的高峰期。按平均每小时 1GB 来调整大小可能就不够了。

*   **许多用户修改相同的块**：在这里，你可能希望有较大的重做日志文件。由于每个人都在修改相同的块，你希望在将它们写出到磁盘之前尽可能多地更新它们。每个日志切换都会触发检查点，因此你希望不频繁地切换日志。然而，这可能会影响你的恢复时间。

*   **平均恢复时间**：如果你必须确保恢复花费尽可能少的时间，你可能会倾向于使用较小的重做日志文件，即使上一点成立。处理一两个小的重做日志文件比处理一个巨大的文件在恢复时花费的时间更少。整个系统日常运行可能比绝对最佳情况慢（由于过多的检查点），但恢复所花费的时间会更短。还有其他数据库参数也可以用来减少这个恢复时间，作为使用小重做日志文件的替代方案。

## 归档重做日志

Oracle 数据库可以以两种模式之一运行：`ARCHIVELOG` 模式和 `NOARCHIVELOG` 模式。这两种模式之间的区别在于，当 Oracle 要去重用一个重做日志文件时会发生什么。“我们会保留该重做日志的副本，还是让 Oracle 直接覆盖它，永远丢失它？”这是一个需要回答的重要问题。除非你保留这个文件，否则你无法从备份中将数据恢复到那个时间点。

假设你每周六备份一次。现在，周五下午，在你本周生成了数百个重做日志之后，你的硬盘故障了。如果你没有运行在 `ARCHIVELOG` 模式下，你现在唯一的选择如下：

*   删除与故障磁盘关联的表空间。任何在该磁盘上有文件的表空间都必须被删除，包括该表空间的内容。如果 `SYSTEM` 表空间（Oracle 的数据字典）或其他重要的系统相关表空间（如你的 `UNDO` 表空间）受到影响，你就不能这样做。你将不得不使用下一个选项。

*   恢复上周六的数据，并丢失你本周所做的所有工作。

这两个选项都不太吸引人。两者都意味着你会丢失数据。另一方面，如果你一直在 `ARCHIVELOG` 模式下运行，你只需找到另一个磁盘，并将受影响的文件从上周六的备份中恢复到它上面。然后，你将应用归档重做日志，并最终应用联机重做日志到它们（实际上是以快进模式重放一周的事务）。你什么都不会丢失。数据被恢复到故障发生时的状态。

人们经常告诉我，他们的生产系统不需要 `ARCHIVELOG` 模式。我还没遇到过任何人在这种说法上是正确的。我相信，除非系统处于 `ARCHIVELOG` 模式，否则它就不是一个生产系统。不在 `ARCHIVELOG` 模式下的数据库，总有一天会丢失数据。这是不可避免的；如果你的数据库不在 `ARCHIVELOG` 模式下，*你将会丢失数据*（不是可能，而是*将会*）。

“我们正在使用 RAID-5，所以我们完全受到保护”是一个常见的借口。我见过这样的情况：由于制造错误，RAID 组中的所有磁盘几乎在同一时间全部冻结。我还见过硬件控制器将损坏引入数据文件的情况，因此人们用 RAID 设备安全地保护了损坏的数据。RAID 也不能保护你免受操作员错误的影响，这是数据丢失最常见的原因之一。RAID 并不意味着数据是安全的，它可能更可用，可能*更安全*，但仅仅存储在 RAID 设备上的数据总有一天会丢失；这只是时间问题。

“如果我们有硬件或操作员错误发生前的备份，并且归档日志没有受到影响，我们本可以恢复。”底线是，对于数据有价值的系统，没有任何借口不运行在 `ARCHIVELOG` 模式下。性能不是借口；正确配置的归档增加很少甚至没有开销。这一点，加上一个丢失数据的快速系统是无用的这一事实，意味着即使归档增加 100%的开销，你仍然需要这样做。如果一个功能可以移除而不丢失任何重要的东西，那么它就是开销；开销就像蛋糕上的糖霜。保护你的数据并确保你不丢失数据不是开销——这是 DBA 的主要工作！

只有测试或*可能*开发系统应该运行在 `NOARCHIVELOG` 模式下。大多数开发系统应该运行在 `ARCHIVELOG` 模式下，原因有两个：

*   这是你将在生产中处理数据的方式；你希望开发的行为和反应与你的生产系统一致。

*   在许多情况下，开发人员会从数据字典中提取他们的代码，修改它，然后将其编译回数据库。开发数据库保存着当前版本的代码。如果开发数据库在下午遭受磁盘故障，你整个早上编译和重新编译的所有代码会发生什么？它丢失了。
