# 数据库性能优化与块清除机制

## 重做日志与检查点优化

*   **重新创建更大尺寸的日志文件**：这将延长填满联机重做日志到需要重用它之间的时间。如果重做日志文件的使用是“突发性的”，同样适用于 `Archival required` 消息。如果你有一段时间会生成大量日志（夜间加载、批处理过程），随后是一段相对平静的时期，那么拥有更大的联机重做日志可以在平静时期为 `ARCn` 进程争取足够的时间来追上。这种方法的利弊与前述添加更多文件的方法完全相同。此外，它可能会将检查点推迟到更晚发生，因为检查点至少在每次日志切换时发生，而现在日志切换的间隔将变得更长。

*   **使检查点发生得更频繁、更连续**：使用更小的块缓冲区缓存（并不完全可取）或各种参数设置，例如 `FAST_START_MTTR_TARGET`、`LOG_CHECKPOINT_INTERVAL` 和 `LOG_CHECKPOINT_TIMEOUT`。这将迫使 `DBWn` 更频繁地将脏块写出。这种方法的好处是故障恢复时间会减少。联机重做日志中需要应用的工作量总会更少。缺点是，如果块被频繁修改，它们可能会更频繁地写入磁盘。缓冲区缓存将无法达到它应有的效果，并且可能会破坏下一节将要讨论的块清除机制。

你采取的方法将取决于你的具体情况。这是必须在数据库级别、考虑整个实例情况下来解决的问题。

## 块清除

在本节中，我们将讨论**块清除**，即移除我们修改过的数据库块上的“锁”相关信息。当我们讨论下一节臭名昭著的 `ORA-01555: snapshot too old` 错误时，理解这个概念很重要。

如果你回顾一下第 6 章我们关于数据锁及其管理方式的讨论。我描述了它们实际上是数据的属性，存储在块头中。这样做的一个副作用是，下次访问该块时，我们可能需要对其进行清除——换句话说，移除事务信息。这个操作会生成重做，并且如果块还不是脏的，会导致该块变脏，这意味着一个简单的 `SELECT` *可能生成重做*，并可能导致在下一个检查点时大量块被写入磁盘。然而，在大多数正常情况下，这不会发生。如果你的事务大多是中小型事务（OLTP），或者你有一个在加载操作后执行直接路径加载或使用 `DBMS_STATS` 来分析表的数据仓库，你会发现块通常已经被为你清除了。如果你回顾本章前面“COMMIT 做什么？”一节，`COMMIT` 时处理的步骤之一是，如果修改的块仍在 SGA 中且可访问（没有其他人在修改它们），则重新访问这些块并清除它们。此活动称为**提交清除**，它是清除我们修改块上事务信息的活动。最优情况下，我们的 `COMMIT` 可以清除块，以便后续的 `SELECT`（读取）无需再进行清除。只有对该块的 `UPDATE` 才会真正清除我们残留的事务信息，并且由于 `UPDATE` 已经生成重做，清除操作就不再显眼。

我们可以通过理解提交清除的工作原理来强制清除*不*发生，从而观察其副作用。在与我们的事务相关的提交列表中，Oracle 会记录我们修改过的块列表。每个这样的列表长度是 20 个块，Oracle 会根据需要分配任意多个这样的列表——但有一个上限。如果我们修改的块的总和超过块缓冲区缓存大小的 10%，Oracle 将停止分配新列表。例如，如果我们的缓冲区缓存设置为缓存 3000 个块，Oracle 将维护最多 300 个块（3000 的 10%）的列表。在 `COMMIT` 时，Oracle 将处理这 20 个块指针的列表，如果块仍然可用，它将执行快速清除。因此，只要我们修改的块数量不超过缓存中块数量的 10%*并且*我们的块仍在缓存中且对我们可用，Oracle 就会在 `COMMIT` 时清除它们。否则，它就直接跳过它们（即不清除它们）。

理解这一点后，我们可以设置人为条件来观察清除如何工作。我将 `DB_CACHE_SIZE` 设置为较低的 16MB，这足以容纳 2048 个 8KB 的块（我的块大小是 8KB）。接下来，我创建一个表，使得一行正好适合一个块——我永远不会在一个块中放两行。然后我用 10,000 行填充这个表并 `COMMIT`。我们知道 10,000 个块远远超过 2048 的 10%，所以数据库将无法在提交时清除所有这些脏块——其中大多数甚至可能已经不在缓冲区缓存中了。我将测量到目前为止生成的重做量，运行一个将访问每个块的 `SELECT`，然后测量该 `SELECT` 生成的重做量。
