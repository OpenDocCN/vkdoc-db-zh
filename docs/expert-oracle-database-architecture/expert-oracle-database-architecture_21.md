# 6. 锁定与栓锁

开发多用户、数据库驱动应用程序的一个关键挑战是最大化并发访问，同时确保每个用户都能以一致的方式读取和修改数据。实现这一点的锁定机制是任何数据库的关键特性，而 Oracle 在提供这些机制方面表现出色。然而，Oracle 对这些功能的实现是特定于 Oracle 的——就像 SQL Server 的实现是特定于 SQL Server 一样——这取决于你，作为应用程序开发人员，要确保当你的应用程序执行数据操作时，它正确地使用了这些机制。如果你未能做到这一点，你的应用程序将表现出意外的行为，并且不可避免地，你的数据完整性将受到损害（如第 1 章所演示）。

在本章中，我们将详细探讨 Oracle 如何锁定数据（例如，`表中的行`）和共享数据结构（例如，在 `SGA` 中的结构）。我们将研究 Oracle 锁定数据的粒度以及这对作为开发人员的你意味着什么。在适当的时候，我会将 Oracle 的锁定方案与其他流行实现进行对比，主要是为了破除行级锁定增加开销的迷思；实际上，只有当实现本身增加开销时，它才会增加开销。在下一章中，我们将继续这个讨论，研究 Oracle 的多版本技术以及锁定策略如何与之交互。

## 什么是锁？

`锁` 是用于调节并发访问共享资源的机制。请注意我使用了术语“共享资源”而不是“数据库行”。确实，Oracle 在行级锁定表数据，但它也在许多其他级别使用锁，以提供对各种资源的并发访问。例如，当一个 `存储过程` 正在执行时，该过程本身会被锁定在一种模式中，允许其他人执行它，但不会允许另一个用户以任何方式更改该 `存储过程` 的那个实例。在数据库中，锁用于允许并发访问这些共享资源，同时提供数据完整性和一致性。

在单用户数据库中，锁不是必需的。根据定义，只有一个用户在修改信息。然而，当多个用户访问和修改数据或数据结构时，拥有一个机制来防止对同一信息的并发修改至关重要。这正是锁定的意义所在。

理解这一点非常重要：数据库中实现锁定的方式与数据库的种类一样多。仅仅因为你有某个特定关系数据库管理系统（`RDBMS`）锁定模型的经验，并不意味着你了解关于锁定的一切。例如，在我深入接触 Oracle 之前，我使用过其他数据库，包括 Sybase、Microsoft SQL Server 和 Informix。所有这三个数据库都提供了用于并发控制的锁定机制，但在每个数据库中实现锁定的方式存在深刻而根本的差异。

为了说明这一点，我将概述我从 Sybase SQL Server 开发人员，到 Informix 用户，最终成为 Oracle 开发人员的历程。这是很多年前的事了，SQL Server 的粉丝们现在可能会告诉我：“但是我们现在有行级锁定！” 确实如此：SQL Server 现在可能使用行级锁定，但其实现方式与 Oracle 中的实现方式完全不同。这是苹果和橘子的比较，而这正是关键点。

作为一名 SQL Server 程序员，我几乎从不考虑多个用户并发向一个表插入数据的可能性。在那个数据库中，这种情况很少发生。当时，SQL Server 仅提供页级锁定，而且由于所有数据往往都插入到非聚簇表的最后一页，两个用户的并发插入根本不会发生。

> **注意**
> SQL Server 聚簇表（具有聚簇索引的表）在某些方面与 Oracle 集群相似，但非常不同。SQL Server 过去只支持页（块）级锁定；如果每一行插入都去到表的“末尾”，那么在那个数据库中永远不会有并发插入或并发事务。SQL Server 中的聚簇索引用于将行插入到整个表中，按聚簇键排序，从而提高了该数据库的并发性。

同样的问影响了并发更新（因为在 SQL Server 中，一个 `` `UPDATE` `` 实际上是一个 `` `DELETE` `` 后跟一个 `` `INSERT` ``）。也许这就是为什么 SQL Server 默认在执行每一条语句后立即提交或回滚，以牺牲事务完整性为代价来尝试获得更高的并发性。

因此，在大多数情况下，使用页级锁定，多个用户无法同时修改同一个表。雪上加霜的是，当一个表正在被修改时，许多查询实际上也被该表阻塞了。如果我尝试查询一个表，而需要的一个页被一个更新锁定了，我就会等待（并且等待，再等待）。锁定机制非常糟糕，以至于支持持续超过一秒的事务是致命的——整个数据库看起来就像冻结了一样。因此，我养成了很多坏习惯。我了解到事务是“坏的”，你应该快速提交，永远不要持有数据锁。并发是以一致性为代价的。你要么想做对，要么想快。我开始相信你无法两者兼得。

当我转向 Informix 时，情况好了一些，但好得不多。只要我记住创建一个启用了行级锁定的表，那么我实际上可以让两个人同时向该表插入数据。不幸的是，这种并发性代价高昂。Informix 实现中的行级锁在时间和内存方面都很昂贵。获取和释放它们需要时间，而且每个锁都消耗实际内存。此外，系统可用的锁总数必须在启动数据库之前计算好。如果你超过了那个数字，那你就只能自认倒霉了。因此，大多数表无论如何都是用页级锁定创建的，而且和 SQL Server 一样，行级和页级锁都会立即阻止查询。结果，我发现我再次想要尽可能快地提交。我使用 SQL Server 时养成的坏习惯只是得到了强化，此外，我还学会了将锁视为一种非常稀缺的资源——值得觊觎的东西。我了解到你应该手动将锁从行级升级到表级，试图避免获取太多锁而导致系统崩溃，而我确实让它崩溃了——很多次。

当我开始使用 Oracle 时，我并没有真正费心去阅读手册来了解锁定在这个特定数据库中是如何工作的。毕竟，我已经使用数据库相当长一段时间了，并且被认为是这个领域的某种专家（除了 Sybase、SQL Server 和 Informix，我还使用过 Ingress、DB2、Gupta SQLBase 和各种其他数据库）。我陷入了这样一个陷阱：相信我知道事情 `应该` 如何运作，所以我认为它们 `当然` 会以那种方式运作。`我大错特错了。`

正是在一次基准测试中，我发现了自己错得有多离谱。在这些数据库的早期，供应商为非常大的采购项目进行基准测试以看看谁能最快、最简单、功能最齐全地完成工作是很常见的。
