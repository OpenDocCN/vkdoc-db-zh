# 读一致性与非阻塞读

让我们看看多版本机制带来的影响：读一致性查询和非阻塞读。如果你不熟悉多版本机制，下面的代码可能会让你感到惊讶。为了简单起见，假设我们要读取的表在每个数据库块中存储一行（数据库中最小的存储单元），并且在这个例子中我们正在对表进行全表扫描。

我们要查询的表是一个简单的 `ACCOUNTS` 表。它存储了银行账户的余额。它的结构非常简单：

```sql
SQL> create table accounts
( account_number number primary key,
account_balance number);
```

在现实中，`ACCOUNTS` 表会有数十万行，但为了简单起见，我们只考虑一个有四行的表，如表 1-2 所示。（我们将在第 7 章更详细地讨论这个例子。）

**表 1-2**
账户表内容

| 行 | 账户号码 | 账户余额 |
| --- | --- | --- |
| 1 | 123 | $500.00 |
| 2 | 234 | $250.00 |
| 3 | 345 | $400.00 |
| 4 | 456 | $100.00 |

我们想运行一个日终报告，告诉我们银行里有多少钱。这是一个极其简单的查询：

```sql
SQL> select sum(account_balance) from accounts;
```

当然，在这个例子中，答案显而易见：$1250。但是，如果我们读了第 1 行，而在我们读第 2 行和第 3 行时，一个自动柜员机（ATM）生成了针对此表的事务，将$400 从账户 123 转移到账户 456，会发生什么？我们的查询在第 4 行计入了$500，结果得出$1650，不是吗？嗯，当然，这是需要避免的，因为这会是一个错误——在任何时间点，账户余额列中都不存在这个金额的总和。读一致性就是 Oracle 避免这种情况发生的方式。Oracle 的方法与大多数其他数据库不同，你需要理解它是如何工作的。

在许多其他数据库中，如果你想获得一个“一致”且“正确”的查询答案，你要么必须在计算总和时锁定整个表，要么必须在读取行时锁定它们。这可以防止其他人在你获取结果的过程中更改数据。如果你提前锁定了表，你将得到查询开始时数据库中的答案。如果你在读取数据时锁定数据（通常称为 *共享读锁*，它阻止更新，但不阻止其他读取器访问数据），你将得到查询结束时数据库中的答案。这两种方法都在很大程度上抑制了并发性。表锁会在你的查询持续期间阻止对整个表进行任何更新（对于一个只有四行的表，这可能只是一个非常短的时间，但对于有数十万行的表，可能需要几分钟）。而“边读边锁”的方法会阻止对你已经读取和处理过的数据进行更新，并且实际上可能导致你的查询和其他更新之间的死锁。

之前我说过，如果不理解多版本的概念，你就无法充分利用 Oracle。下面就是一个原因。Oracle 使用多版本机制来获取查询开始时存在的数据，并且查询将在 **不锁定任何东西** 的情况下进行（虽然我们的账户转账事务更新了第 1 行和第 4 行，这些行会对其他写入者锁定，但不会对其他读取器，比如我们的 `SELECT SUM...` 查询锁定）。事实上，Oracle 没有“共享读锁”（其他数据库中常见的一种锁类型）——它不需要。所有可以消除的抑制并发性的因素都已被移除。

我见过一些真实的案例，一个不理解 Oracle 多版本能力的开发者编写的报表会将整个系统锁得死死的。原因在于：开发者希望从他的查询中获得读一致（即正确）的结果。在他工作过的其他所有数据库中，这都要求锁定表或使用 `SELECT ... WITH HOLDLOCK`（一种 SQL Server 机制，用于在读取过程中以共享模式锁定行）。因此，开发者要么在运行报表前锁定表，要么使用 `SELECT ... FOR UPDATE`（这是他找到的与 `WITH HOLDLOCK` 最接近的替代方案）。这会导致系统基本上停止处理事务——这是不必要的。

那么，Oracle 是如何在不锁定任何数据的情况下——换句话说，在不降低并发性的情况下——在读取过程中获得正确、一致的答案（$1250）的呢？秘密在于 Oracle 使用的事务机制。每当你修改数据时，Oracle 会在两个不同的位置创建条目（大多数其他数据库会将两个条目放在同一个位置；对它们来说，撤销和重做只是“事务数据”）。一个条目进入重做日志，Oracle 在其中存储足够的信息来 **重做** 或“前滚”事务。对于插入操作，这将是插入的行。对于删除操作，它在概念上是一条删除文件 X、块 Y、行槽 Z 中行的消息。等等。另一个条目是一个 **撤销** 条目，写入一个撤销段。如果你的事务失败并需要撤销，Oracle 将从撤销段中读取“之前”映像并恢复数据。除了使用这个撤销段数据来撤销事务外，Oracle 还用它来在读取块时撤销对块的更改——将块恢复到你的查询开始时的状态。这使你能够直接读取穿过锁，并且无需自己锁定任何数据即可获得一致、正确的答案。

因此，就我们的例子而言，Oracle 得到答案的过程如表 1-3 所示。

**表 1-3**
多版本机制实战

| 时间 | 查询 | 账户转账事务 |
| --- | --- | --- |
| T1 | 读取第 1 行；余额 = $500；当前总和 = $500。 | |
| T2 | | 更新第 1 行；在第 1 行上放置排他锁，阻止其他更新（但不阻止读取）。第 1 行现在有$100。 |
| T3 | 读取第 2 行；余额 = $250；当前总和 = $750。 | |
| T4 | 读取第 3 行；余额 = $400；当前总和 = $1150。 | |
| T5 | | 更新第 4 行；在第 4 行上放置排他锁，阻止其他更新（但不阻止读取）。第 4 行现在有$500。 |
| T6 | 读取第 4 行；发现第 4 行已被修改。它实际上会回滚该块，使其看起来与时间 = T1 时一样。查询将从该块中读取值 $100。 | |
| T7 | | 提交事务。 |
| T8 | 给出答案 $1250。 | |

在时间 T6，Oracle 实际上“读穿”了我们的事务在第 4 行上放置的锁。这就是非阻塞读如何实现的：Oracle 只检查数据是否已更改；它不关心数据当前是否被锁定（这暗示数据可能已更改）。Oracle 只是简单地从撤销段中检索旧值，然后继续处理下一个数据块。

这是多版本机制的又一个清晰展示。同一信息的多个版本，都处于不同的时间点，在数据库中都是可用的。Oracle 能够使用这些不同时间点的数据快照来为我们提供读一致性查询和非阻塞读。

这种数据的读一致性视图总是在 SQL 语句级别执行。任何单个 SQL 语句的结果都相对于它们开始的时间点保持一致。这种特性使得像下面这样的插入操作能产生可预测的数据集：

```sql
for x in (select * from t)
loop
insert into t values (x.username, x.user_id, x.created);
end loop;
```


当查询开始执行时，`SELECT * FROM T`的结果是确定的。该`SELECT`语句不会看到由`INSERT`生成的任何新数据。想象一下，如果它能看到的话——这条语句可能会成为一个永无止境的循环。如果随着`INSERT`在表`T`中生成更多行，`SELECT`能够“看到”这些新插入的行，那么前面的代码将会产生未知数量的行。如果表`T`最初有 10 行，那么当我们结束时，`T`中可能最终会有 20、21、23 行或无限多行。这将是完全不可预测的。所有语句都提供了这种一致读取，因此像下面这样的`INSERT`也是可预测的：

```
insert into t select * from t;
```

`INSERT`语句将会获得表`T`的一个读一致视图。它不会看到自己刚刚插入的行；相反，它只会插入在`SELECT`开始时已存在的行。一些数据库甚至不允许像前面那样的递归语句，因为它们无法判断实际上可能插入多少行。

因此，如果你习惯于其他数据库在查询一致性和并发方面的处理方式，或者你从未需要处理这些概念（即你没有真正的数据库经验），你现在就能明白理解 Oracle 如何运作对你来说将是多么重要。为了最大化 Oracle 的潜力，并编写出正确的代码，你*需要*理解这些问题在 Oracle 中的表现——而不是它们在其他数据库中是如何实现的。

## 数据库独立性？

至此，你或许能看出我本节要讨论的方向。我已经提到了其他数据库，以及特性在每个数据库中的实现方式如何不同。除了某些只读应用程序之外，我的观点是：构建一个完全数据库独立且高度可扩展的应用程序是极其困难的——事实上，除非你确切且详细地知道每个数据库如何运作，否则几乎是不可能的。而且，如果你真的详细知道每个数据库如何运作，你就会明白数据库独立性并不是你真正想要达成的目标（这是一个非常循环的论证！）。

为了说明，让我们重新审视最初的资源调度器示例（在添加`FOR UPDATE`子句之前）。假设这个应用程序是在一个与 Oracle 的锁/并发模型完全不同的数据库上开发的。我在这里要说明的是，如果你将应用程序从一个数据库迁移到另一个，你将不得不验证它在这些不同环境中是否仍能正确工作，并且在迁移过程中需要对它进行实质性的修改！

让我们假设我们最初将资源调度器应用程序部署在一个采用阻塞式读取（读取被写入阻塞）的数据库中。同时，假设业务规则是通过数据库触发器实现的（在`INSERT`发生之后，但在事务提交之前，我们会验证该时间段内表中是否只有我们的行存在）。在一个阻塞式读取系统中，由于这个新插入的数据，对这个表的插入操作确实会串行化。第一个人会插入对“房间 A”在周五下午 2:00 到 3:00 的请求，然后运行一个查询来查找冲突。下一个人会尝试插入一个冲突的请求，并在查找冲突时被阻塞（等待新插入的数据变为可读状态）。在那个阻塞式读取的数据库中，我们的应用程序看起来行为良好，尽管如果我们双方都插入了行然后试图读取对方的数据，它同样很容易*死锁*（第 6 章关于锁的内容会涵盖这个概念）。我们对资源分配冲突的检查会一个接一个地发生，永远不会并发进行。

如果我们把这个应用程序迁移到 Oracle，并简单地假设它的行为方式会相同，那我们会大吃一惊。在 Oracle 上，它采用行级锁并提供非阻塞读取，因此它看起来表现不佳。正如我们之前看到的，我们必须使用`FOR UPDATE`子句来串行化访问。没有这个子句，两个用户可能为同一资源安排相同的时间段。这是不理解我们所使用的数据库在多用户环境中如何运作的直接后果。

当应用程序从数据库 A 迁移到数据库 B 时，我多次遇到类似的问题。当一个在数据库 A 中运行无瑕的应用程序在数据库 B 中无法工作或以明显怪异的方式工作时，人们的第一反应是数据库 B 是“坏数据库”。简单的事实是，数据库 B 只是运作方式*不同*。没有哪个数据库是错的或坏的；它们只是不同。了解并理解它们的工作方式将极大地帮助你处理这些问题。将应用程序从 Oracle 迁移到 SQL Server 会暴露 SQL Server 的阻塞式读取和死锁问题——反之亦然。

例如，我曾被请去帮助将一些 Transact-SQL（SQL Server 的存储过程语言）转换为 PL/SQL。进行转换的开发人员抱怨说 Oracle 中的 SQL 查询返回了“错误的”答案。查询看起来像这样：

```
declare
  l_some_variable   varchar2(25);
begin
  if ( some_condition )
  then
    l_some_variable := f( ... );
  end if;
  for x in ( select * from T where x = l_some_variable )
  loop
    ...
```



这里的目标是找出表 `T` 中所有满足以下条件的行：如果某个条件不满足，则 `x` 为 `NULL`；如果某个条件满足，则 `x` 等于一个特定值。

问题在于，在 Oracle 中，当 `L_SOME_VARIABLE` 未被设置为一个特定值（即保持为 `NULL`）时，此查询不会返回任何数据。而在 Sybase 或 SQL Server 中，情况并非如此——查询会找到那些 `x` 被设置为 `NULL` 值的行。我几乎在每次从 Sybase 或 SQL Server 向 Oracle 迁移时都会遇到这个问题。SQL 应该在三值逻辑下运行，而 Oracle 实现 `NULL` 比较的方式符合 ANSI SQL 的要求（其中 `NULL` 表示一种未知状态，而非一个值）。根据这些规则，将 `x` 与 `NULL` 进行比较既不真也不假——它实际上是*未知*的。以下代码段说明了我的意思：

```
SQL> select * from dual where null=null;
no rows selected
SQL> select * from dual where null  null;
no rows selected
SQL> select * from dual where null is null;
D
-
X
```

第一次看到这个可能会令人困惑。它证明了在 Oracle 中，`NULL` 既不等于 `NULL`，也不不等于 `NULL`。SQL Server 默认情况下不这样做：在 SQL Server 和 Sybase 中，`NULL` 等于 `NULL`（默认情况下；在当前版本的 SQL Server 中，默认行为可能已修改为遵循 ANSI 标准）。没有哪个数据库的处理方式是*错误*的——只是*不同*而已。实际上，所有这些数据库都符合 ANSI 标准（符合 ANSI 标准并不意味着你支持 100% 的标准，远非如此；详情请参见下一节“标准的影响”），但它们的工作方式仍然不同。存在着一些歧义、向后兼容性问题等等需要克服。例如，SQL Server 支持 ANSI 的 `NULL` 比较方法，只是不是默认方式（如果默认就这样做，会破坏成千上万个基于该数据库构建的现有遗留应用程序）。

在这种情况下，一个解决方案是将查询改写为这样：

```
select *
from t
where ( x = l_some_variable OR (x is null and l_some_variable is NULL ))
```

然而，这又导致了另一个问题。在 SQL Server 中，此查询会使用 `x` 上的索引。在 Oracle 中可能并非如此，因为 B*树索引（关于索引技术的更多内容见第 11 章）不会索引完全为 `NULL` 的条目。因此，如果你需要查找 `NULL` 值，B*树索引并不总是有用的。

注意
只要 Oracle B*树索引中至少有一列被定义为 NOT NULL，表中的所有行实际上都会出现在索引中，并且 `x` 为 null 的谓词可以并且将会使用索引来检索行。

在这种情况下，为了最小化对代码的影响，我们给 `x` 分配了一个它在实际中永远不可能取到的值。这里，根据定义，`x` 是一个正数，所以我们选择了数字 `-1`。于是，查询变成了：

```
select * from t where nvl(x,-1) = nvl(l_some_variable,-1)
```

并且我们创建了一个基于函数的索引：

```
create index t_idx on t( nvl(x,-1) );
```

通过最小的改动，我们达到了相同的结果。从这个例子中需要认识到的重要点是：

*   数据库是不同的。使用一个数据库的经验在某种程度上可以转移到另一个数据库，但你必须准备好面对一些*根本性*的差异以及一些非常细微的差异。
*   细微的差异（如对 `NULL` 的处理）可能产生与根本性差异（如并发控制机制）一样大的影响。
*   了解数据库、它的工作原理以及其功能的实现方式，是克服这些问题的唯一途径。

开发人员经常问我（通常一天不止一次）如何在数据库中完成某些特定操作，例如，“我如何在存储过程中创建临时表？”我不会直接回答这类问题。相反，我会反问：“*为什么*你想这样做？”很多时候，得到的回答是：“在 SQL Server 中我们在存储过程中创建临时表，我们需要在 Oracle 中也这样做。”这正是我预料中的回答。那么，我的回应就很简单了：“你并不是真的想在 Oracle 的存储过程中创建临时表——你只是认为你想这样做。”事实上，在 Oracle 中这样做是非常糟糕的事情。如果你在 Oracle 的存储过程中创建表，你会发现：

*   执行 DDL 会抑制可扩展性。
*   不断执行 DDL 并不快。
*   执行 DDL 会提交你的事务。
*   为了访问这个表，你将不得不在所有存储过程中使用动态 SQL——无法使用静态 SQL（因为该表在编译时不存在）。
*   PL/SQL 中的动态 SQL 不如静态 SQL 快，也不如其优化得好。

关键在于，你并不想像在 SQL Server 中那样完全照搬地做这件事（如果你在 Oracle 中根本还需要临时表的话）。你应该按照 Oracle 中最佳的方式来做事情。就像如果你反过来从 Oracle 迁移到 SQL Server，你不会想创建一个供所有用户共享的单一表来存放临时数据（Oracle 就是这样做的）。那会限制其他数据库的可扩展性和并发性。所有数据库并非生而平等；它们都非常不同。

这并不是说你不能在 Oracle 中使用临时表。你可以，你很可能也会用到。只是*在 Oracle 中使用它们的方式与你在 SQL Server 中使用的方式不同*（反之亦然）。



## 标准的影响

如果所有数据库都符合 `SQL99` 标准，那么它们必然是相同的。至少这通常是假设。在本节中，我将破除这一迷思。

`SQL99` 是数据库的 `ANSI/ISO` 标准。它是 `SQL92` `ANSI/ISO` 标准的后继者，而后者又取代了 `SQL89` `ANSI/ISO` 标准。它现在已被 `SQL2003`、`SQL2008` 和 `SQL2011` 等标准更新所取代。该标准定义了一种语言 (`SQL`) 和行为（事务、隔离级别等），用以说明数据库将如何运作。您知道许多市售数据库至少在一定程度上符合 `SQL99` 标准吗？您是否也知道，这对于查询和应用程序的可移植性而言，意义甚微？

从 `SQL92` 标准开始，标准分为四个级别：

*   `入门级`：这是大多数供应商已遵从的级别。它是对前代标准 `SQL89` 的小幅增强。*没有任何数据库供应商获得过更高级别的认证*，实际上，曾负责认证 `SQL` 符合性的机构——美国国家标准与技术研究院 (`NIST`)——甚至已不再进行认证。我是 1993 年使 `Oracle 7.0` 获得 `NIST` 认证、符合 `SQL92` 入门级标准的团队成员之一。符合入门级标准的数据库，其功能集是 `Oracle 7.0` 能力的一个子集。

*   `过渡级`：就功能集而言，此级别大约介于入门级和中间级之间。

*   `中间级`：此级别增加了许多功能，包括（此列表绝非详尽无遗）
    *   动态 `SQL`
    *   用于参照完整性的级联 `DELETE`
    *   `DATE` 和 `TIME` 数据类型
    *   域
    *   变长字符串
    *   `CASE` 表达式
    *   数据类型间的 `CAST` 函数

*   `完全级`：此级别增加了以下规定（同样，此列表并非详尽无遗）
    *   连接管理
    *   `BIT` 字符串数据类型
    *   可延迟完整性约束
    *   `FROM` 子句中的派生表
    *   `CHECK` 约束子句中的子查询
    *   临时表

入门级标准不包括外连接、新的内连接语法等特性。过渡级确实指定了外连接语法和内连接语法。中间级增加了更多特性，而完全级当然包含了 `SQL92` 的全部。大多数关于 `SQL92` 的书籍并未区分各个级别，这导致了混淆。它们展示的是理论上实现了 `SQL92` 完全级的数据库会是什么样子。这使得你无法拿起一本 `SQL92` 书籍，就将书中内容应用到任意一个 `SQL92` 数据库上。关键在于，入门级的 `SQL92` 能提供的很有限，而如果你使用了中间级或更高级别的任何特性，你就有无法移植应用程序的风险。

`SQL99` 仅定义了两个符合性级别：`核心级` 和 `增强级`。它试图远超传统的 `SQL`，并引入了对象关系构造（数组、集合等）。它涵盖了 `SQL MM`（多媒体）类型、对象关系类型等。没有供应商在认证其数据库“符合” `SQL99` 核心级或增强级，实际上，据我所知，没有供应商声称其产品完全符合其中任何一个级别。

您不必害怕使用供应商特定的功能——毕竟，您为此支付了高昂的费用。每个数据库都有其独特的技巧箱，您总能找到一种方法在每个数据库中执行给定的操作。为您当前的数据库使用最合适的方案，并在迁移到其他数据库时重新实现组件。使用良好的编程技术来将自己与这些变化隔离开来。编写 `OS` 可移植应用程序的人——例如 `Oracle` 内核开发人员——也采用了相同的技术。

## 确保您能够适应

目标是充分利用您可用的设施，但要确保您能够根据具体情况更改实现。打个比方，`Oracle` 是一个可移植的应用程序。它可以在许多操作系统上运行。然而在 `Windows` 上，它以 `Windows` 的方式运行：使用线程和其他 `Windows` 特有的设施。相比之下，在 `UNIX/Linux` 上，`Oracle` 作为多进程服务器运行，使用独立的进程来完成在 `Windows` 上由线程完成的工作——这就是 `UNIX/Linux` 的方式。“核心 `Oracle`”功能在两个平台上都可用，但其底层实现方式却大相径庭。您那些必须在多个数据库上运行的数据库应用程序也将是相同的。

例如，许多数据库应用程序的一个常见功能是为每一行生成唯一键。当您插入行时，系统应自动为您生成一个键。`Oracle` 为此实现了名为 `SEQUENCE` 的数据库对象。`SYS_GUID()` 是另一个提供唯一键的函数。`Informix` 有一个 `SERIAL` 数据类型。`Sybase` 和 `SQL Server` 有一个 `IDENTITY` 类型。每个数据库都有实现此功能的方法。然而，这些方法是不同的，无论是在操作方式还是可能的结果上。因此，对于有见识的开发者来说，有两条路可以走：

*   开发一种完全独立于数据库的方法来生成唯一键。
*   适应不同的实现，并在为每个数据库实现键时使用不同的技术。

> **注意**
> `Oracle` 有一个 `IDENTITY` 类型列。在底层，它创建了一个序列并将您的列默认值设置为该值——使其工作方式非常类似于 `SQL Server` 的 `IDENTITY` 类型或 `MySQL` 的 `AUTO_INCREMENT` 类型。

第一种方法的理论优势在于，从一种数据库迁移到另一种数据库时无需更改任何内容。我称其为“理论”优势，因为这种实现的缺点如此巨大，以至于使其完全不可行。要开发一个完全独立于数据库的过程，您必须创建一个这样的表：

```
$ sqlplus eoda/foo@PDB1
SQL> create table id_table
( id_name  varchar2(30) primary key,
id_value number );
Table created.
SQL> insert into id_table values ( 'MY_KEY', 0 );
1 row created.
SQL> commit;
Commit complete.
```

然后，为了获得一个新键，您必须执行以下代码：

```
SQL> update id_table
set id_value = id_value+1
where id_name = 'MY_KEY';
1 row updated.
SQL> select id_value
from id_table
where id_name = 'MY_KEY';
ID_VALUE

SQL> exit;
```

看起来很简单，但结果（注意是复数）如下：

*   一次只能有一个用户处理一个事务行。您需要更新该行以递增计数器，这将导致您的程序在该操作上串行化。最理想的情况是，一次只能有一个人生成此键的新值。
*   在 `Oracle` 中（其他数据库的行为可能不同），在 `SERIALIZABLE` 隔离级别下，并发执行此操作时，除第一个尝试的用户外，所有用户都将收到错误 “`ORA-08177: can’t serialize access for this transaction`”。

例如，使用可序列化事务（这在 `J2EE` 环境中更为常见，许多工具自动将其用作默认隔离模式，而开发者通常并不知晓），您将观察到以下行为：

```
$ sqlplus eoda/foo@PDB1
SQL> set transaction isolation level serializable;
Transaction set.
SQL> update id_table  set id_value = id_value+1  where id_name = 'MY_KEY';
1 row updated.
SQL> select id_value
from id_table
where id_name = 'MY_KEY';
ID_VALUE

```

现在（保持之前的会话连接），我们将转到另一个 `SQL*Plus` 会话并执行相同的操作，即并发请求一个唯一 `ID`：

```
-- 会话 2
SQL> set transaction isolation level serializable;
Transaction set.
SQL> update id_table
set id_value = id_value+1
where id_name = 'MY_KEY';
```



此时，此处将发生阻塞，因为一次只能有一个事务更新该行。这展示了第一种可能的结果——我们将阻塞并等待该行。但因为我们使用的是 Oracle 的`SERIALIZABLE`隔离级别，我们将在提交第一个会话的事务后观察到以下行为：

```
-- Session 1
SQL> commit;
Commit complete.
```

第二个会话将立即显示以下错误：

```
-- Session 2
SQL> update id_table
set id_value = id_value+1
where id_name = 'MY_KEY';
update id_table
*
ERROR at line 1:
ORA-08177: can't serialize access for this transaction
```

无论之前的提交语句顺序如何，该错误都会发生。只要你的事务试图修改自其开始以来被其他会话修改过的任何记录，就会触发此错误。

因此，那段号称数据库无关的逻辑实际上根本**不是**数据库无关的。根据隔离级别的不同，它甚至可能在单个数据库中也无法可靠地运行！有时我们会阻塞等待；有时我们会收到错误信息。说最终用户在两种情况下（等很久或等很久后得到一个错误）都会感到不快，这算是很温和的说法了。

这个问题的严重性在于，我们的实际事务远比上述示例要庞大得多。示例中的`UPDATE`和`SELECT`语句只是构成该事务的众多潜在语句中的两条。我们还没有将刚刚生成的对应主键的行插入到表中，也还没有完成该事务所需的其他任何工作。这种串行化将成为扩展性的一大限制因素。试想一下，如果这种技术用于处理订单的网站，并且这是我们生成订单号的方式，那将带来怎样的影响。将无法实现多用户并发，我们将被迫顺序处理一切。

解决这个问题的正确方法是为每个数据库使用最优的代码。在 Oracle 中，有如下两种方法（假设需要生成主键的表是`T`）：

```
$ sqlplus eoda/foo@PDB1
SQL> create sequence s;
Sequence created.
SQL> create table t
( x          number default s.nextval
constraint t_pk primary key,
other_data varchar2(20));
Table created.
```

或者，你可以使用`IDENTITY`属性而跳过序列的显式创建：

```
SQL> create table t
( x          number generated as identity
constraint t_pk primary key,
other_data varchar2(20));
Table created.
```

注意

`IDENTITY`列必须是数值数据类型。

但请注意，它并非真正跳过了序列生成——序列是由数据库自动生成的。如果你无法使用自动生成的序列，则可以使用以下逻辑来获得自动生成的代理主键：

```
SQL> create table t
( pk number primary key,
other_data varchar2(20));
Table created.
SQL> create sequence t_seq;
Sequence created.
SQL> create trigger t before insert on t
for each row
begin
:new.pk := t_seq.nextval;
end;
/
Trigger created.
```

这将自动地、透明地为插入的每一行分配一个唯一的键。一个更注重性能的方法是简单地：

```
Insert into t ( pk, .... ) values ( t_seq.NEXTVAL, .... );
```

也就是说，完全跳过触发器的开销（这*绝对*是我的首选方法）。你可以使用其他数据库的类型达到相同的效果。`CREATE TABLE`语法会有所不同，但最终结果是一样的。在这里，我们不遗余力地利用每个数据库的特性来生成一个*非阻塞*的、高并发的唯一键，并且没有对应用程序代码引入任何实际更改——在这个案例中，所有逻辑都包含在 DDL 中。

#### 分层编程

一旦你理解了每个数据库*都会以不同的方式实现特性*，另一个为了可移植性而进行防御性编程的例子是，在必要时对你的数据库访问进行分层。假设你使用`JDBC`进行编程。如果你只使用纯 SQL 的`SELECT`、`INSERT`、`UPDATE`和`DELETE`语句，你可能不需要抽象层。只要你将使用的构造限制在你打算支持的每个数据库都支持的范围内，并且已验证它们的行为完全一致（记住关于`NULL= NULL`的讨论！），你完全可以在应用程序中直接编写 SQL。然而，这意味着你将使用性能不佳的 SQL——而且你脑子里似乎需要掌握比我认识的任何人都要多的关于更多数据库的知识（毕竟，这是判断某个操作是否有可能在所有数据库上表现一致的唯一方法！）。另一种更具可移植性且能提供更好性能的方法是使用存储过程来返回结果集。你会发现每个供应商的数据库都可以从存储过程返回结果集，但返回的方式各不相同。你必须为不同数据库编写的实际存储过程源代码是不同的。

你在这里的两个选择是：要么不使用存储过程返回结果集，要么为不同的数据库实现不同的代码。我肯定会遵循为不同供应商编写不同代码的方法，并大量使用存储过程。这看起来似乎会增加在不同数据库上实现的时间。然而，你会发现使用这种方法在多个数据库上实现实际上更容易。与其试图找到在*所有*数据库上都能完美运行的 SQL（也许在某些数据库上表现更好），你将为该数据库实现运行效果最佳的 SQL。你可以在应用程序本身之外完成这项工作，这为你在调整应用程序时提供了更大的灵活性。你可以在数据库中修复一个性能不佳的查询，并立即部署该修复，而无需给应用程序打补丁。此外，通过这种方法，你可以自由地利用供应商对 SQL 的扩展。例如，Oracle 支持广泛的 SQL 扩展，如分析函数、SQL 模型子句、行模式匹配等。在 Oracle 中，你可以自由使用这些 SQL 扩展，因为它们位于应用程序“之外”（即，隐藏在数据库中）。在其他数据库中，你可能会使用它们提供的任何特性来达到相同的结果。你已经为这些功能付费了，不妨使用它们。
