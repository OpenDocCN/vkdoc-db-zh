# 简化问题解决之道

解决任何问题总有两种方式：简单的方法和困难的方法。一次又一次，我看到人们选择了困难的方法。这并非总是有意为之，更多时候是出于无知。他们从未想到数据库竟然能做“那种事”。而我则相反，我总是假设数据库无所不能，只有当我发现它确实无法完成某件事时，才会选择困难的方式（自己动手编写）。

例如，我经常被问到：“如何确保终端用户在数据库中只有一个会话？”（其实我本可以用其他数百个例子。）这必定是许多应用的需求，但我从未参与过这类项目——我找不到这样限制用户的充分理由。然而，人们就是想这么做，而当他们付诸实践时，通常都会选择困难的方式。例如，他们会设置一个由操作系统运行的批处理作业，来检查 `V$SESSION` 表，并强制终止那些拥有多个会话的用户的连接。或者，他们会创建自己的表，让用户登录时在应用中插入一行记录，登出时再删除。这种实现方式总是会导致大量的服务台求助电话，因为当应用崩溃时，删除记录的操作永远不会执行。我见过许多其他“有创意”的实现方式，但没有一个比下面这个方法更简单：

```
$ sqlplus eoda/foo@PDB1
SQL> create profile one_session limit sessions_per_user 1;
Profile created.
SQL> alter user scott profile one_session;
User altered.
现在我们尝试两次连接到 SCOTT 用户；第二次尝试应该会失败：
SQL> connect scott/tiger@PDB1
Connected.
SQL> host sqlplus scott/tiger@PDB1
ERROR:
ORA-02391: exceeded simultaneous SESSIONS_PER_USER limit
```

就是这样——现在，任何拥有 `ONE_SESSION` 配置文件的用户都只能登录一次。当我提出这个解决方案时，通常能听到手拍额头的声响，紧接着是那句“我从来不知道它还能这样做”。花时间熟悉你手头工具的能力，可以在开发过程中为你节省大量的时间和精力。

同样的“保持简单”原则也适用于更宏观的架构层面。我强烈建议人们在采用非常复杂的实现方案之前，要深思熟虑。系统中的可变部分越多，可能出错的地方就越多，而在一个过于复杂的架构中精确追踪错误发生的位置绝非易事。使用无数层架构来实现可能真的很“酷”，但如果一个简单的存储过程就能更好、更快、更省资源地完成任务，那复杂架构就不是正确的选择。

我曾见过一些项目，应用开发已经进行了好几个月，却依然看不到尽头。开发者们使用的是最新最好的技术和语言，但开发进度并不快。这原本并不是一个庞大的应用——也许问题恰恰就在于此。如果你在建一个狗窝（一个小木工活），你不会动用重型机械。你会使用一些小型电动工具，但根本用不上那些“大家伙”。另一方面，如果你在建造一栋公寓楼，你会雇佣数百人来参与这个项目；你会使用大型机器——你会用完全不同的工具来解决这个问题。应用开发也是如此。没有单一的“完美架构”，没有单一的“完美语言”，也没有唯一的“完美方法”。

例如，为了构建我的网站，我使用了 APEX（Application Express）。我的网站是一个规模较小的应用；只有一两个开发者参与开发。它大概有 20 个屏幕。对于这个实现来说，PL/SQL 和 APEX 是正确的选择——它不需要几十号人用 Java 编码、制作 EJB、使用 Hibernate 等等。这是一个简单的问题，用简单的方式解决了。当今确实有一些复杂、大规模、巨型的应用（我们大多购买现成的：如 HR 系统、ERP 系统等），但更多的是成千上万的小型应用。我们需要为工作选择合适的方法和工具。

我永远会选择那个能完全解决问题、且最简单的架构，而不是复杂的那个。回报可能是巨大的。每种技术都有其适用的位置。并非每个问题都是钉子，所以我们不能只用工具箱里的锤子。


## 开放性

我经常看到人们因为另一个原因而采取困难的方式行事，这再次涉及到一个理念：我们应当不惜一切代价追求开放性和数据库独立性。开发者希望避免使用封闭、专有的数据库特性——即便是像存储过程或序列这样简单的特性——因为这样做会将他们锁定在某个数据库系统中。那么，让我提出这样一个观点：当你开发一个读写应用程序的那一刻，你已经在某种程度上被锁定了。一旦开始运行查询和数据修改，你就会发现数据库之间存在细微（有时甚至不那么细微）的差异。例如，在某个数据库中，你可能会发现`SELECT COUNT(*) FROM T`会与一个简单的两行数据更新操作发生死锁。而在 Oracle 中，你会发现`SELECT COUNT(*)`永远不会阻塞正在更新被统计数据的写入者。你可能遇到过这样的情况：由于数据库锁定模型的副作用，某个业务规则似乎在其中一个数据库中被强制执行，但在另一个数据库中却未被执行。你会发现，在完全相同的混合事务下，不同数据库生成的报告会得出不同的答案，这完全是因为基本实现的差异。你会发现，能够直接从一个数据库迁移到另一个数据库的应用程序非常罕见。SQL 的解释（例如，`NULL=NULL`的例子）和处理方式上的差异将始终存在。

在一个项目中，开发者们正在使用 Visual Basic、ActiveX 控件、IIS 服务器和 Oracle 数据库构建一个基于 Web 的产品。我被告知，开发团队表示担忧，因为业务逻辑是用`PL/SQL`编写的，导致产品变得依赖于数据库，并且被问道：“我们该如何纠正这个问题？”

我对这个问题感到有点惊讶。查看他们选择的技术列表时，我无法理解依赖数据库为何是一件“坏事”：

*   开发者选择了一种语言，这使他们锁定在单一供应商提供的单一操作系统上（他们本可以选择 Java）。
*   他们选择了一种组件技术，这使他们锁定在单一操作系统和供应商上（他们本可以选择 J2EE）。
*   他们选择了一个 Web 服务器，这使他们锁定在单一供应商和单一平台上（为什么不选择更开放的技术呢？）。

他们所做的其他每一项技术选择都将他们锁定在非常具体的配置中——事实上，唯一在操作系统方面为他们提供了一些选择的技术就是数据库。

尽管如此（他们选择那些技术肯定有充分的理由），我们仍然看到一群开发者做出有意识的决定，不去使用其架构中一个关键组件的功能，并且是以开放性的名义这样做。我相信，你应该仔细挑选技术，然后尽可能充分地利用它们。你为这些技术付出了高昂的代价——充分挖掘它们的潜力难道不是最符合你利益的吗？我不得不假设他们期待充分利用其他技术的全部潜力，那么为什么数据库是个例外？考虑到数据库对他们的成功至关重要，这个问题甚至更难回答。

如果我们从开放性的角度来审视这个论点，可以稍微改变一下视角。你将所有数据放入数据库中。数据库是一个非常开放的工具。它支持通过大量开放系统协议和访问机制进行数据访问。到目前为止听起来很棒，堪称世界上最开放的东西。

然后，你将所有应用逻辑，更重要的是，将你的安全机制都放在数据库之外。也许是在访问数据的 Bean 中。也许是在访问数据的 JSP 中。也许是在你的 Visual Basic 代码中。也许是在 Hibernate 生成的代码中。最终结果是，你刚刚关闭了你的数据库——你让它变得“不开放”了。人们无法再接入现有技术来利用这些数据；他们必须使用你的访问方法（或者干脆绕过安全机制）。这在今天看来似乎很好，但你必须记住，今天的尖端技术是昨天的概念，并将成为明天陈旧疲惫的技术。在关系型世界（很可能大多数对象实现也是如此）中，历经三十多年依然屹立不倒的是数据库本身。数据的前端几乎每年都在变化，而当它们变化时，那些将所有安全机制都内置于自身、而非置于数据库中的应用程序，就成了未来发展的障碍和绊脚石。

Oracle 数据库提供了一个称为细粒度访问控制（`FGAC`）的功能。简而言之，这项技术允许开发者在数据库中嵌入过程，这些过程可以在查询提交给数据库时对其进行修改。这种查询修改用于限制客户端将接收或修改的行。该过程可以查看是谁在执行查询、何时执行查询、是哪个应用程序请求数据、他们是从哪个终端执行查询等等，并可以据此适当地限制数据访问。通过`FGAC`，我们可以强制执行安全策略，例如：

*   由某类用户在正常工作时间之外执行的任何查询都返回零条记录。
*   任何数据都可以返回到安全设施内的终端，但只有非敏感信息可以返回到远程客户端终端。

基本上，`FGAC`允许我们将访问控制定位在数据库中，紧挨着数据。用户是从 Bean、JSP、使用`ODBC`的 Visual Basic 应用程序还是`SQL*Plus`来访问数据变得不再重要——相同的安全协议都将被强制执行。你已经为即将到来的下一项技术做好了充分准备。

现在我问你，哪种实现更“开放”？是那种使得所有对数据的访问只能通过调用 Visual Basic 代码和 ActiveX 控件（如果你愿意，可以用 Java 代替 Visual Basic，用 EJB 代替 ActiveX——我并非在针对某项特定技术，而是在评论一种实现）来完成的方案，还是那种允许任何能与数据库对话的东西进行访问，通过诸如`SSL`、`HTTP`和 Oracle Net（以及其他）等多样化的协议，或者使用`ODBC`、`JDBC`、`OCI`等 API 来实现的方案？我还没有见过哪个即席查询工具会去“查询”你的 Visual Basic 代码。但我知道有几十种工具可以执行 SQL。

人们完全有自由做出追求数据库独立性和完全开放的决定，很多人也尝试这样做，但我相信这是个错误的决定。无论你使用什么数据库，都应该充分利用它，尽可能地榨干该产品的每一项功能。无论如何，你会在调优阶段（这似乎总是发生在部署之后不久）发现自己这样做。令人惊讶的是，仅仅通过利用数据库软件的能力就能让应用程序运行速度提升五倍时，数据库独立性的要求可以多么迅速地被抛弃。


## 如何让它运行得更快？

标题中的这个问题是我经常被问到的。每个人都在寻找那个 `fast = true` 的开关，认为“数据库调优”就意味着你去调整数据库。事实上，根据我的经验，超过 80%（经常是 100%）的性能提升，都体现在应用设计和实现层面，而非数据库层面。在调整运行于数据库之上的应用程序之前，你无法有效地调优数据库。

随着时间的推移，我们可以在数据库层面设置一些开关，以帮助减轻粗劣编程错误的冲击。例如，Oracle 有一个参数 `CURSOR_SHARING=FORCE`。可以说，这个特性实现了一个 `自动绑定器`。它会静默地将写成 `SELECT * FROM EMP WHERE EMPNO = 1234` 的查询，为我们重写为 `SELECT * FROM EMP WHERE EMPNO = :x`。这 `可以` 显著减少硬解析的数量，并减少我们在“架构”部分讨论过的库闩锁等待——`但是`（总是有个“但是”）它可能带来一些副作用。使用游标共享时，一个常见的副作用是这样的：

```
SQL> select /* TAG */ substr( username, 1, 1 )
from all_users au1
where rownum = 1;
S
-
S
SQL> alter session set cursor_sharing=force;
Session altered.
SQL> select /* TAG */ substr( username, 1, 1 )
from all_users au2
where rownum = 1;
SUBSTR(USERNAME,1,1)

S
```

那里发生了什么？为什么第二个查询（可以说是一样的查询）由 `SQL*Plus` 报告的列突然变得这么大？如果我们看看游标共享设置为我们做了什么，这一点（以及其他一些东西）就会变得显而易见：

```
SQL> select sql_text from v$sql where sql_text like 'select /* TAG */ %';
SQL_TEXT

select /* TAG */ substr( username, 1, 1 )
from all_users au1
where rownum = 1
select /* TAG */ substr( username, :"SYS_B_0", :"SYS_B_1" )
from all_users au2
where rownum = :"SYS_B_2"
```

游标共享从查询中移除了信息。它找到了 `每一个` 字面量，包括我们正在使用的 `substr` 常量。它将它们从查询中移除，并用绑定变量替换。SQL 引擎不再知道该列是长度为 1 的 `substr` 结果——它的长度是不确定的。此外，你可以看到 `where rownum = 1` 现在也被绑定了。这看起来是个好主意；然而，优化器刚刚被移除了一些重要信息。它不再知道“这个查询将检索单行”；它现在认为“这个查询将返回前 N 行，而 N 可以是任何数字”。这会对生成的查询计划产生负面影响。

此外，我已经表明，虽然 `CURSOR_SHARING=FORCE` 比解析和优化大量独特的查询运行得快得多（请参阅前面关于绑定变量的部分），但我也发现它比开发者自行进行绑定的查询要慢。这并非源于游标共享代码的任何低效，而是源于程序本身的低效。在许多情况下，不使用绑定变量的应用程序在解析和重用游标方面也效率低下。因为应用程序认为每个查询都是唯一的（它将它们构建为唯一的语句），它永远不会使用同一个游标超过一次。事实是，如果程序员一开始就使用了绑定变量，他们本可以解析一次查询并重用多次。正是这种解析的开销降低了整体的潜在性能。

### 注意
同样重要的是要指出，`CURSOR_SHARING=FORCE` 不会修复 SQL 注入漏洞。绑定发生在你的最终用户重写查询之后；SQL 注入已经发生了。`CURSOR_SHARING=FORCE` 并不会让你比以前更安全。只有通过使用绑定变量本身，开发者才能实现一个防 SQL 注入的应用程序。

基本上，重要的是要记住，简单地开启 `CURSOR_SHARING = FORCE` 并不一定能解决你的问题。它很可能会引入新的问题。`CURSOR_SHARING` 在某些情况下是一个非常有用的工具，但它不是万能药。一个开发良好的应用程序永远不需要它。从长远来看，在适当的地方使用绑定变量，在需要时使用常量，才是正确的方法。

### 注意
没有万能药，一个都没有。如果有，它们就会成为默认行为，你也就永远听不到它们了。

即使数据库层面确实可以设置一些开关（而且它们确实少之又少），与并发问题以及执行糟糕的查询（由于查询写得不好或数据结构不佳）相关的问题，是无法通过一个开关来修复的。这些情况需要重写（并且经常需要重新架构）。移动数据文件、调整参数以及其他数据库层面的开关，通常对应用程序的整体性能影响甚微。这绝对达不到你为了使得应用程序可接受而需要实现的两倍、三倍……`n` 倍的性能提升。你的应用程序有多少次只是慢了百分之十？慢百分之十，没人会抱怨。慢上五倍，人们就会生气了。我重申：通过移动数据文件，你无法获得五倍的性能提升。你只有通过修复应用程序，或许通过让它执行显著更少的 I/O，才能实现大幅度的性能提升。

### 注意
这只是想说明事物是如何随时间变化的。我经常写道，通过移动数据文件，你无法获得五倍的性能提升。但是随着 Oracle Exadata 等硬件解决方案的出现，你实际上可以通过简单地移动数据文件，将响应时间（返回数据所需的时间）减少五倍、十倍、五十倍甚至更多。但这更像是一个“我们彻底改变了硬件架构”的故事，而不是“我们重组了一些存储”。此外，让应用程序在 Exadata 上仅仅运行快五倍或十倍对我来说会是令人失望的——我会希望它“快”五十倍或更多——这需要重新思考应用程序的实现方式。

性能是设计、构建和测试工作贯穿整个开发阶段都必须持续关注的事情。它绝不应该是事后才考虑的东西。我常常惊讶于人们经常等到应用程序已经交付给客户、部署到位并实际运行之后，才开始对其进行调优。我见过一些实现，其中应用程序交付时除了主键之外，没有任何其他索引。查询从未经过调优或压力测试。应用程序从未在超过少量用户的情况下进行过试用。调优被认为是产品安装的一部分。对我来说，这是一种不可接受的方式。你的最终用户应该从第一天起就得到一个响应迅速、全面调优的系统。会有足够多的“产品问题”需要处理，而不应该让糟糕的性能成为用户的首个体验。用户期望新应用程序有一些错误，但至少不要让用户在痛苦的长时间等待中看着那些错误出现在屏幕上。



### DBA 与开发者的关系

毋庸置疑，最成功的系统建立在数据库管理员（DBA）与应用程序开发者之间的共生关系之上。在本节中，我将从开发者的视角，探讨开发者与 DBA 之间的分工（假设每个严肃的开发项目都配备有 DBA 团队）。

作为开发者，你未必需要知道如何安装和配置软件。这应该是 DBA，或许还有系统管理员（SA）的职责。设置 Oracle Net、启动监听器、配置共享服务器、启用连接池、安装数据库、创建数据库等等——这些功能我都归入 DBA/SA 的职责范畴。

一般来说，开发者无需知道如何调优操作系统。我自己通常将这项任务留给系统的 SA。作为数据库应用软件的开发者，你需要熟练使用你所选择的操作系统，但不应期望自己还需要去调优它。

DBA 最大的单一职责是数据库的恢复与还原。注意，我说的不是“备份”，而是“还原与恢复”，并且我认为这是 DBA 的唯一职责。理解回滚和重做机制如何工作——是的，这是开发者必须了解的。而知道如何执行表空间时间点恢复，开发者则可以略过。知道可以做到这一点可能有用，但真正需要执行它——不，这不是你的事。

在数据库实例级别进行调优，并确定最佳的 `PGA_AGGREGATE_TARGET` 值——这通常是 DBA 的工作（而且数据库非常愿意并能够协助他们确定正确的数值）。在特殊情况下，开发者可能需要为会话更改某些设置，但在数据库级别，这是 DBA 的责任。一个典型的数据库支持的不仅仅是单个开发者应用。只有支持所有应用的 DBA 才能做出正确的决策。

分配空间和管理文件是 DBA 的工作。开发者提供他们对空间需求的预估（他们认为自己需要多少），但其余的事情由 DBA/SA 处理。

基本上，开发者不需要知道如何运行数据库。他们需要知道如何在数据库中运行。开发者和 DBA 就像是在处理同一个谜题的不同部分。当你的查询消耗了太多资源时，DBA 会来找你；而当你无法想出如何让系统运行得更快时（那时才进行实例调优，在应用完全调优之后），你会去找 DBA。

这一切因环境而异，但我认为其中确实存在分工。一个好的开发者通常是一个很糟糕的 DBA，反之亦然。在我看来，他们是两种不同的技能组合、两种不同的思维模式和两种不同的个性。

## 总结

在本章中，我们通过一些实例探讨了为何你需要了解数据库。我所举的例子并非孤立事件——它们日复一日地发生着。我观察到此类问题不断循环往复。

让我们快速回顾一下要点。如果你正在使用 Oracle 进行开发：

*   你需要理解 Oracle 架构。你不必精通到能重写服务器的程度，但应该足够了解，以便清楚使用特定功能所带来的影响。
*   你需要理解锁和并发控制，并且要知道每个数据库实现这些功能的方式都*不同*。如果不了解，你的数据库就会给出“错误”的答案，你将遇到严重的争用问题，导致性能低下。
*   不要将数据库视为一个无需理解的黑盒。数据库是大多数应用中最关键的部分。试图忽视它将是致命的。
*   不要重复发明轮子。我见过不止一个开发团队因此陷入困境，不仅在技术层面，也在人际关系层面，原因在于他们不了解 Oracle 免费提供的功能。当有人指出他们花了过去几个月时间实现的特性其实是数据库一直以来的核心功能时，这种情况就会发生。请阅读*《Oracle 数据库概念》*指南——新特性指南——这些随软件免费提供的文档！
*   尽可能简单地解决问题，尽可能多地使用 Oracle 的内置功能。你为它付出了高昂的代价。
*   软件项目来了又去，编程语言和框架也是如此。我们开发者被期望在几周，也许是几个月内让系统上线运行，然后转向下一个问题。如果你一遍又一遍地重新发明轮子，你将永远无法跟上疯狂的开发节奏。正如你不会在 Java 中自己构建哈希表类一样——因为它自带了——你应该使用你手头可用的数据库功能。当然，做到这一点的第一步，就是了解你手头到底有什么。请继续阅读。

基于最后一点，软件项目和编程语言可能会更迭——但*数据*是永恒的。我们构建使用数据的应用程序，而这些数据将被许多应用程序随时间推移而使用。重点不在于应用程序，而在于数据。使用允许数据被使用和重用的技术和实现方式。如果你将数据库仅仅当作一个数据桶，使得所有对数据的访问都必须通过你的应用程序，那你就本末倒置了。你无法对你的应用程序进行“即席查询”。你无法在旧应用程序之上构建新的应用程序。但是，如果你使用了数据库，你会发现随着时间的推移，添加新的应用程序、报告或其他任何东西都会容易得多。
