# 6. 硬件使用

在考虑如何设计查询时，你已经了解了基于集合的设计的要素。但在使用 SQL Server 时，还有其他需要考虑的事项。你所使用的是一个超越其各个组成部分的系统。SQL Server 有许多方面：存储的数据、访问数据的过程，以及如何访问数据的流程。SQL Server 不仅仅是软件，它还涉及该软件如何与你的计算机或服务器的硬件进行交互。

在使用硬件方面，SQL Server 利用了多个不同的部件。有些部件可能看起来很直接，而另一些起初可能令人惊讶。SQL Server 被称为关系数据库管理系统。作为关系数据库管理系统的一部分，SQL Server 的目的是实现数据的高效存储和检索。这意味着，运行 SQL Server 的系统上，最重要的硬件部件之一是内存。谈到数据的长期存储，硬盘驱动器就变得最为关键。在本章中，我将它们称为 `storage`，因为存储不仅限于硬盘。然而，还有其他一些不太明显的元素，例如 CPU。

在本章中，你将审视一些编码模式和反模式，这些模式可能导致服务器资源被有效或无效地分配。对于反模式，你将探讨替代方法，以实现在不使这些有限的硬件资源负担过重的情况下的相同结果。

## T-SQL 设计中的内存考量

当你初次开始在 SQL Server 中处理数据时，可能会惊讶地发现内存是最重要的因素之一。这在很大程度上与内存在 SQL Server 内部的用途有关。SQL Server 旨在利用内存。有一个称为 `caching` 的过程，它将信息保留在内存中。对于 SQL Server，有两种缓存类型非常重要。第一种与执行计划的缓存有关。这是 SQL Server 保存有关将来如何再次执行存储过程或即席查询的指令的地方。其目的是利用 SQL Server 已经计算出的关于执行特定查询的最佳方式的知识。第二种与查询设计相关的缓存几乎可能更为重要。它涉及信息如何存储在内存中。这被称为 `缓冲池`。

在本节中，我谈到数据存储在缓冲池中。如果我们需要的数据不在缓冲池中，会发生什么？这可能是 SQL Server 中事物变得有点复杂的地方。SQL Server 只与缓冲池内部的数据进行交互。因此，如果需要的数据不在缓冲池中，第一步是确定哪些数据使用频率较低。然后，你将这些数据从缓冲池中移除。这些第一步已经为我们提供了关于内存使用的线索。如果有某些东西不必要地填满了缓冲池，你现在就知道关键数据可能会被清除出缓冲池，以便为这个新数据腾出空间。

真正的关键在于限制 SQL Server 必须处理的数据量。在编写查询时，这可以通过你的连接逻辑来处理；也可以通过你的 `where` 子句条件来处理。你需要确保访问尽可能少的数据。访问所需的最少量的数据，可以让你在使用 T-SQL 时最小化内存的使用。列表 6-1 提供了一个查询，其中 SELECT 语句没有包含所有列。

```sql
SELECT cus.FirstName,
cus.LastName,
prd.ProductName
FROM dbo.Customer cus
INNER JOIN dbo.CustomerOrder ord
ON cus.CustomerID = ord.CustomerID
INNER JOIN dbo.OrderDetail dtl
ON ord.CustomerOrderID = dtl.CustomerOrderID
INNER JOIN dbo.Product prd
ON dtl.ProductID = prd.ProductID
WHERE prd.ProductName LIKE 'Tel%'
AND cus.LastName LIKE 'G%'
AND ord.IsActive = 1;
```

内存的速度和多功能性是 SQL Server 的重要组成部分。另一个因素与内存成本相对于存储成本的关系有关。这种成本差异凸显了有效使用内存的重要性。有许多不同类型的东西存储在内存中。SQL Server 用来访问满足查询标准所需数据的过程，作为执行计划保存在内存中。为查询结果检索的数据也存储在内存中。你还可以让数据库对象存在于内存中，例如表变量或临时表。你经常听到 `OLTP` 这个术语。它真正的含义是，你希望有一个能够执行大量写入和更新活动的系统。这些类型的活动在我们的应用程序中频繁发生。当提及写入和更新活动时，例如在食谱数据库中，每次访问食谱、每次对食谱进行更改，或每个新配料添加到系统中，都属于此类活动。




### 内存管理与性能优化

当系统中存储大量数据时，用户通常只需要处理其中很小的一个子集。关键技能在于判断哪些数据应保持随时可用。这将提升添加新条目、更新现有条目或访问已有条目的速度。此时，对硬件的更好理解能提供帮助。

我们通常将内存视为服务器或计算机上的`RAM`。然而，还有其他类型的内存可用。内存是`SQL Server`用于快速访问数据最常见的硬件。

虽然执行计划和数据都可以保存在缓存中，但`SQL Server`会执行某些计算来确定为计划缓存或缓冲池分配多少内存。此外，复杂的计算用于决定在计划缓存或缓冲池中保留某物多久。真正的目标是确保不经常访问的存储过程或查询从计划缓存中清除，以免占用不必要的空间。同样，对于缓冲池，目标是只保留正在被主动使用的数据，并清除应用程序不再需要的数据。在这两种情况下，项目只有在缓存已满且需要为新的执行计划或数据腾出空间时才会被清除。

另一种解决方案是缓冲池扩展。缓冲池仅存在于`RAM`上，而缓冲池扩展允许你将缓冲池扩展到使用`RAM`和固态驱动器（`SSDs`）。然而，并非缓冲池中的所有数据页都可以驻留在`SSD`上。只有缓冲池中与写入存储的数据页匹配的页才能位于`SSD`上。这意味着对该页的所有更改都已写入磁盘。当更改被写入磁盘时，此过程称为“固化”。在更改写入磁盘之前，这些页称为“脏页”。在页写入磁盘后，它们称为“干净页”。考虑到`SSDs`可能比购买额外的`RAM`更具成本效益，这有助于减少当`SQL Server`因业务应用程序访问数据缺乏可用内存而频繁将页面移出缓冲池时发生的`I/O`瓶颈。

关于计划缓存，有一些需要记住的特定注意事项。默认情况下，`SQL Server`针对处理存储过程和预处理语句进行了优化。这种对存储过程的优化也是使用计划缓存时的一个因素。在理想情况下，你的计划缓存中会有大量的存储过程，而即席查询很少。然而，这并非总是适用于每个业务。你的公司可能使用`ORM`（对象关系映射）或像`LINQ`这样的技术在应用程序代码中编写即席查询。虽然它们便于开发人员编写，但它们给`SQL Server`带来了额外开销。通常，`ORM`软件编写的`T-SQL`代码功能可行，但在性能上可能不如开发人员编写的查询优化。此外，`ORM`软件还倾向于创建可能填满计划缓存的即席查询。当计划缓存充满即席查询时，它可能在缓存中创建同一查询的多个版本，浪费计划缓存中的查询分配，并导致其他有价值的计划被清除。如果这些查询没有转换为预处理语句，这种情况尤其可能发生。

如果存储过程因应用程序而被清除出计划缓存，那么当下次调用该存储过程时，`SQL Server`将需要重新计算其执行计划。这将需要额外的`CPU`成本来生成新的执行计划。有各种方法可以处理计划缓存中存在大量即席查询的情况。然而，这超出了本书的范围。

我在与`SQL Server`交互时遇到的另一个问题与缓冲池以及缓冲池中保存的数据有关。我曾不止一次听说，在处理非常大的表时，你将永远陷入不得不处理访问大量数据的困境。当表设计、索引、查询或上述任何组合未经过优化时，这种确切的情况就可能发生。然而，有方法可以预防。

在使用内存和`SQL Server`时，要考虑的主要方面是当你编写查询时，它们必须返回正确的结果，并在应用程序可接受的时间内完成。然而，根据整体表设计或查询的编写方式，这些查询访问的数据量可能远远超过必要。起初这可能看起来不是一个重大问题。当数据表是新的且未充满数据时，通常不会是问题。清单`6-2`中的查询用于在应用程序中返回一些数据。

```sql
SELECT cus.FirstName,
       cus.LastName,
       cus.Country,
       ord.OrderNumber,
       prd.ProductName,
       dtl.QuantitySold,
       prd.ProductPrice
FROM dbo.Customer cus
INNER JOIN dbo.CustomerOrder ord
    ON cus.CustomerID = ord.CustomerID
INNER JOIN dbo.OrderDetail dtl
    ON ord.CustomerOrderID = dtl.CustomerOrderID
INNER JOIN dbo.Product prd
    ON dtl.ProductID = prd.ProductID
WHERE prd.ProductName LIKE 'Tel%'
  AND cus.LastName LIKE 'G%'
  AND ord.IsActive = 1;
```

如你所见，一个最初看似性能良好的查询，随着应用程序成熟并获取更多数据，可能开始出现性能问题。

在这些情况下，应用程序频繁访问的数据可能正被从缓冲池中清除。当存储过程必须筛选大量数据时，你通常可以通过查看与该查询关联的逻辑读取看到这一点。你可能正在处理一个只返回五个结果的查询；然而，它必须经过两百万次逻辑读取才能返回结果。这表明你可能正在使用比预期更多的缓冲池。




这类行为可能导致缓冲池中的大量数据被清空。SQL Server 中有一个名为 `page life expectancy`（页生存期）的值，用于显示数据页在缓冲池中停留的平均时间。我习惯将 `page life expectancy` 作为一个指标，用以判断是否有查询在 SQL Server 中读取的数据量可能远大于结果集返回的数据量。这可能导致这些查询本身使用了超出其实际需要的内存。当我处理与应用程序配合使用的查询时，它们通常一次不涉及超过十条记录。即使某个查询用于某种形式的报表，一次通常也不会超过几千条记录。即便我一次查看 20 万条记录，我仍然不希望为了获取这些结果而进行数千万次的逻辑读取。我可以添加一些索引来帮助最小化查询的逻辑读取次数。清单 6-3 展示了添加的索引。

```sql
CREATE NONCLUSTERED INDEX IX_Customer_CustomerNameName
ON dbo.Customer(LastName ASC, FirstName ASC);
CREATE NONCLUSTERED INDEX IX_CustomerOrder_CustomerID_IsActive
ON dbo.CustomerOrder(CustomerID ASC, IsActive ASC);
CREATE NONCLUSTERED INDEX IX_OrderDetail_CustomerOrderID_ProductID
ON dbo.OrderDetail(CustomerOrderID ASC, ProductID ASC);
```

然而，你需要谨慎地为每一个独特的场景都创建索引。为每一个慢查询都创建索引可能会因为维护每个索引所需的磁盘活动增加而对性能产生负面影响。此外，SQL Server 在创建索引时确实会分配内存以便对数据进行排序。使用索引可以成为确保你只访问所需数据的强有力方式。你还需要理解数据以及相关的 `T-SQL`，以便在不给 SQL Server 增加过重负担的情况下有效地创建索引。

有一些查询可以用来确定存储过程使用了多少逻辑读取。如前所述，逻辑读取是查询是否影响内存的一个指标。如果存储过程存在于计划缓存中，你可以找出哪些存储过程的逻辑读取数更高。清单 6-4 展示了查找存储过程及其逻辑读取数的查询。

```sql
USE OutdoorRecreation;
GO
SELECT db.[name] AS name_database,
obj.[name] AS name_stored_procedure,
sts.execution_count,
sts.total_logical_reads
FROM sys.objects obj
INNER JOIN sys.dm_exec_procedure_stats sts
ON obj.[object_id] = sts.[object_id]
LEFT JOIN sys.databases db
ON sts.database_id = db.database_id;
```
清单 6-4
查找计划缓存中存储过程的逻辑读取数

这个查询会显示当前 `OutdoorRecreation` 数据库计划缓存中的存储过程。它包含了该存储过程已被执行的次数以及所有执行产生的逻辑读取总数。要分析每个存储过程的平均逻辑读取数，你需要用 `total_logical_reads` 除以 `execution_count`。一旦你识别出逻辑读取数最高的查询，你可能需要回顾第 8 章，学习如何优化这些查询的逻辑读取。

尽管内存可以被认为是 SQL Server 可用的最强大、最有益的硬件，但它也有相关的金钱成本。内存可以弥补尚未优化的 `T-SQL`。然而，通常会达到一个临界点，即表变得非常大而查询又未完全优化。那时，很容易倾向于通过增加更多内存来解决性能问题。在许多情况下，这只是针对长期问题的短期解决方案。除此之外，购买额外内存相关的成本并不总是企业能够负担的选项。防止或解决此问题的一个方法是查看系统中的存储过程和预准备语句，并找出那些可以通过减少相关查询的逻辑读取而得到改进的查询。
