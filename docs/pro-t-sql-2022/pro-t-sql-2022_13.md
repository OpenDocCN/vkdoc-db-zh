# 考虑存储与 T-SQL 设计

我已经介绍了如何与你计划频繁访问的数据进行交互；你还需要考虑大部分数据是如何存储的。在处理存储时，重要的是考虑哪些活动会导致你必须访问磁盘上的数据。这些正是你希望最小化或改进的活动类型。在某些情况下，当我看到大量的磁盘 I/O 时，这可能表明 T-SQL 设计的某些其他部分需要改进。这可能只是症状，而非根源。

SQL Server 使用索引来组织数据，并使其能够轻松快速地检索。对于大于几个数据页的表，SQL Server 会尝试使用索引来遍历各个数据页，并快速找到所需数据。然而，在某些情况下，必要的索引可能不存在。这导致 SQL Server 必须读取更多的数据页，有时甚至是表中的所有数据，以确定哪些行是满足查询需求所必需的。

一旦数据被判定为非优先事项，SQL Server 便可以访问磁盘并获取其所需的数据。这些数据将被存储在缓冲池中，直到未来的某个时刻它被认为不再需要。就硬件速度而言，这个过程本身已经引入了延迟。如果数据不在缓冲池中，还需要额外的时间来确定哪些记录需要从缓冲池中移除。将这些记录从缓冲池中移除并访问磁盘，会带来从磁盘检索数据的额外时间成本。

如果这些数据仅用于每四到六小时运行一次的作业，那么大部分数据可能被判定为对所有其他进程无关紧要。这就导致了从缓冲池中移除的数据出现高频率更替。新数据可以从磁盘获取并保存到缓冲池中。然后，这些新数据可能会从缓冲池中移除，并被从磁盘检索到的更新数据所取代。

除了处理大量数据所带来的这种开销之外，还有其他与磁盘或存储交互的活动。当新数据被写入系统时，该数据将被存储在事务日志中。数据将持续添加到事务日志中，直到日志被清除。当事务提交且信息被写入磁盘时，日志可能会被刷写。在系统中添加、更新或删除大量数据的情况下，这可能导致大量的磁盘活动以修改表，并且由于日志增长以及插入操作中保存的数据（用于回滚），磁盘存储空间也会增加。

了解数据是如何存储的，以及 SQL Server 如何通过索引和存储在磁盘上的其他数据页轻松找到这些数据，这很重要。当信息被添加到索引或索引需要更新时，这些操作也会产生存储成本。这不仅涉及存储数据所需的时间，还包括访问磁盘进行这些添加或更改所涉及的时间。

在使用 T-SQL 时，你需要考虑你的查询如何影响数据存储。正如你在上一节所见，这可以意味着多件事情。你需要确保在给定时间内，你没有访问、插入、更新或删除超出你实际需要的数据量。但这并不是编写 T-SQL 时唯一需要考虑的因素。你可以做一些事情来监控与查询相关的存储使用情况。其中大部分涉及查看物理读取次数、预读读取次数或与查询相关的 I/O 成本。

如果你已经确认你正在处理与你想要交互的数据量最少，那么可能无法再降低与你的 T-SQL 查询相关的存储成本。在某些情况下，就像内存一样，发现巨大的存储使用量可能指向其他问题。这些问题可能包括数据如何保存在磁盘上以及索引是如何设计的等。

### 索引和页拆分对存储的影响

在某些情况下，你可能会发现频繁的索引更新导致了巨大的存储成本。这可能是由于索引需要根据其设计方式不断重新组织。如果在 `dbo.CustomerOrder` 表的 `ShipDate` 上创建了一个非聚集索引，那么当 `ShipDate` 被填充或更改时，就需要更新该索引。当这种情况发生时，`ShipDate` 可能需要移动到不同的页。如果该页已满，SQL Server 将创建一个新页。这个新页可能会从旧页转移一些数据过来，而新数据将被添加到该页中。同时还会创建一个指针，以便 SQL Server 知道在逻辑序列中去哪里查找下一个数据页。当这种情况发生时，就称为 `索引碎片`。这可能导致使用此索引的查询性能不佳，因为索引信息没有按顺序存储。虽然这在 SSD 上问题较小，但索引碎片仍然可能导致索引占用比正确组织时更多的空间。

在涉及页拆分的情况下，你也可能看到与此活动相关的显著存储成本。你的表的排序方式或索引的设计方式可能导致大量的页拆分。例如，我可以在 `dbo.CustomerOrder` 表中使用 `ShipDate` 作为聚集索引来创建表。如果应用程序没有为每个订单填充发货日期，或者如果发货日期被更改，那么数据记录可能会被移动到不同的页。在某些情况下，该记录可能被移动到一个已经满了的页。如果是这样，该页将会拆分，以便将记录添加到有可用空间的数据页中。你可能会看到维护这些索引相关的磁盘活动增加。

### 存储引擎的作用

关于存储成本如何运作的更多信息，了解 SQL Server 存储引擎如何操作可能会有所帮助。在处理 T-SQL 查询时，有一个方面更频繁地依赖于存储引擎。这与访问当前不在缓冲池中的数据有关。

存储引擎具有几个不同的功能。这些功能涉及访问不在缓冲池中的数据。存储引擎的另一个功能与需要锁时如何处理数据有关。存储引擎的第三个功能涉及与应用程序使用的 T-SQL 运行无关的命令。

存储引擎在访问数据时可能会遇到性能瓶颈。这些瓶颈反过来会导致访问存储数据时出现性能问题。一个例子是当查询访问数据表中的行或与索引交互时。在其他时候，存储引擎可能需要处理与为缓冲池检索数据相关的页分配。如果你的公司决定实施行版本控制，这可能会导致存储性能问题。虽然这超出了本书的范围，但如果你正在使用行版本控制并且遇到与存储性能相关的问题，你可能需要进行额外的研究以确认行版本控制不是原因。



当访问数据行或索引时，SQL Server 将使用存储引擎。被访问的数据可能存储在 `buffer pool` 中。由存储引擎负责跟踪这些行在 `B-Tree` 中的位置。`B-tree` 是一种自平衡的二叉搜索树，为数据存储创建层级结构。这种特定的结构设计允许多个分支，使得其子项（称为 `nodes` 或 `levels`）比典型的二叉搜索树更"扁平"。`B-Tree` 是专门为处理数据库等大量数据而设计的。在 SQL Server 中管理、添加、更新或删除索引中的条目时，也会发生相同的过程。这在存储引擎中统称为 `row and index operations maintenance`。

在编写查询以访问数据时，SQL Server 使用一个称为 `read` 的指标（发音与动词 "to read" 相同）来跟踪查询需要访问的数据。如果数据已在内存中，则访问该数据的行为称为 `logical read`。有时，访问数据可能需要 SQL Server 访问磁盘或存储上的数据页。数据页是 SQL Server 存储的最小数据单元，大小为 8 kb。如果从磁盘访问数据，则该数据称为 `physical read`。如果数据页被移至 `buffer pool`，该操作称为 `read-ahead read`。无论哪种情况，SQL Server 都需要访问磁盘以获取请求的记录。当你编写一个查询，并且它需要检索不存在于 `buffer pool` 中的记录时，SQL Server 将需要访问磁盘上的那些记录。这对 SQL Server 来说可能是一个非常正常的过程。但是，如果查询需要读取的记录数量多于查询结果所需的数量，则可能导致磁盘活动超出实际需求。监控从磁盘检索数据所花费时间的一种方法是使用 `latch waits`。这表示 SQL Server 访问不存在于 `buffer pool` 中的数据所花费的时间量。

除了 SQL Server 在存储引擎内管理数据访问外，存储引擎还负责管理与数据锁定相关的活动。这类活动可能受到服务器、应用程序或事务上配置的 `isolation level` 的影响。`isolation level` 决定了 SQL Server 将如何处理事务间的锁和行版本控制。在许多情况下，`isolation level` 的配置由不同于负责编写 `T-SQL` 代码的团队来管理。存储引擎还负责处理 SQL Server 中的锁。

与存储相关的其他可能导致性能问题的因素涉及查询设计的其他方面，这些方面可能与硬盘驱动器交互，例如需要将大量活动记录到事务日志的数据库操作。这可能包括插入、更新或删除大量数据。SQL Server 设计为及时将这些更改记录到事务日志。然而，这些修改也需要记录到磁盘上。根据碎片情况或其他使用硬盘的进程，这可能会导致额外的性能问题。

`tempdb` 是 SQL Server 的另一部分，可能会产生与存储相关的显著开销。可能存在一些查询导致 `tempdb` 内部发生大量活动。如果你正在创建大型临时表，就可能发生这种情况，因为这些信息可能存储在 `tempdb` 中。也可能是 SQL Server 未能正确估算 `execution plan`。这通常是由于过时的 `statistics`（即表中数据的直方图）造成的。如果这些 `statistics` 不正确，SQL Server 可能不会分配足够的资源来执行查询。如果 `tempdb` 与数据或日志文件放在同一驱动器上，可能会出现磁盘延迟问题。为避免此问题，通常建议将 `tempdb` 移动到与数据和日志文件不同的独立驱动器上。在这种情况下，`tempdb` 上的任何存储开销应该只影响同样依赖 `tempdb` 的其他查询。

编写 `T-SQL` 查询从根本上与 SQL Server 如何与已存储的数据交互相关联。这种交互的某些方面是可以控制的，例如确保你访问尽可能少的数据。而其他因素，例如 SQL Server 如何访问数据或管理锁，可能超出你的控制范围。无论如何，了解所有这些部分如何结合并可能影响你的查询性能至关重要。



## 考虑 CPU 与 T-SQL 设计

SQL Server 不仅与内存和存储交互；它也与服务器 CPU 交互。这是在设计查询时需要考虑的最终环节之一。当涉及到 SQL Server 的许可时，CPU 可能是一笔显著的成本。部分原因可能在于不理解 SQL Server 所需的最佳 CPU。SQL Server 有其许可限制。如果不遵守这些限制，你可能会购买 SQL 永远无法识别或使用的 CPU 核心。由于大多数人按核心付费，因此确保每个核心尽可能强大是有意义的。你甚至可能通过节省 SQL 许可费用，来抵消更高性能 CPU 所带来的额外成本。此外，还有一个问题是在你的 T-SQL 代码中执行计算。

如前所述，有时大量使用内存会导致 I/O 问题。同样，当存在大量 I/O 活动时，这也可能导致 CPU 问题。在整体 T-SQL 设计中考虑内存时，有两种类型的缓存。在那一节中，我主要关注了缓冲区缓存或缓冲池。在本节中，我将完整循环一圈，讨论执行计划的缓存如何导致 CPU 性能问题。

当你运行一个查询时，SQL Server 必须查看所有涉及的表、这些表上的所有索引、所有的联接谓词和 WHERE 子句，以弄清楚哪种表、索引和联接类型的组合能够带来可接受且准确的查询性能。这个计算的结果被称为 `` `查询执行计划` ``。

虽然有时查询很简单，这个过程快速而容易，但其他时候它可能资源消耗大得多。特别是对于第二种情况，我们非常希望 SQL Server 能够重用该计划，而不是在执行类似查询时必须重新计算它。

为了让 SQL Server 知道如何执行查询，它必须利用 CPU。这被视为一种 CPU 成本。期望是这种成本不应该导致性能问题，因为频繁执行的预 pared 语句和存储过程的计划已经保存在缓存中，不需要以任何显著的频率重新创建这些执行计划。

这就引出了是什么导致 SQL Server 判断需要一个新的执行计划。这直接关系于理解 SQL Server 在创建执行计划时做了什么。创建执行计划的一部分是生成一个值，允许 SQL Server 快速找到先前创建的执行计划。这个值被称为`计划哈希`。目标是在必要时创建执行计划，并尽可能多地重用这些执行计划。所有这些计划都存储在`计划缓存`中。只有看起来相同的查询才具有相同的`计划哈希`，并会使用相同的执行计划。虽然你可能有几个看起来相似或返回相同结果的查询，但 SQL Server 可能不会给它们相同的`计划哈希`。如果 SQL Server 在`计划缓存`中没有识别出现有的计划，它将需要创建一个新的执行计划。这种对`计划哈希`的依赖有助于确定是否需要创建新的执行计划。

这导致我们依赖于使用参数。这些参数可以存在于存储过程或预 pared 语句中。如果正在执行的查询与`计划缓存`中的某个查询具有相同的`计划哈希`，SQL Server 将不会重新创建新的执行计划。这个创建执行计划的过程被称为`` `编译` ``。这可能是你的查询消耗的 CPU 资源超出预期的众多原因之一。

SQL Server 也可能判断需要为某个查询创建新的执行计划，即使执行缓存中已经存在相关的`计划哈希`。在 SQL Server 2016 中引入的，数据库表的统计信息是根据表中行数的动态阈值来更新的。如果统计信息更新了，SQL Server 可能会选择生成一个新的执行计划。这可能是频繁数据更改的结果，也可能是参数嗅探的结果。有关参数嗅探的更多信息，请参见第 2 章和第 8 章。无论哪种方式，它都会导致 SQL Server 识别为`` `重新编译` ``的东西。就像查询`编译`有相关成本一样，`重新编译`在 CPU 上也有相关成本。

如果查询正在处理大量 I/O，这可能会蔓延导致 CPU 性能问题。虽然一个查询可能访问了比必要更多的数据，但这可能并非由于整体的 T-SQL 查询设计。它可能是由于过时的统计信息。虽然管理统计信息的概念超出了本书的范围，但你需要意识到统计信息对执行计划的影响。统计信息让 SQL Server 能够快速了解特定数据表或索引内的数据整体分布情况。其概念是，正确的统计信息有助于 SQL Server 对如何创建执行计划做出最佳猜测。如果这些值过时或陈旧，SQL Server 可能会决定一个执行计划，而这个计划如果 SQL Server 拥有最新的统计信息本可以更高效。由于统计信息过时，估计的读取次数和实际的读取次数之间可能存在显著差异。这可能会对 CPU 产生性能影响。你可以看到统计信息已经更新，估计和实际返回的行数保持相对一致。

还有一种可能是，SQL Server 必须遍比最初预期更多的硬盘或存储。这可能是由于涉及碎片的情况。在 CPU 活动高于正常水平的情况下，它可能是由于索引碎片造成的。这意味着 SQL Server 可能只估计了检索数据所需的一定工作量，因为它假设索引没有碎片。然而，由于索引存在碎片，SQL Server 必须做潜在比预期更多的工作来访问请求的数据。

另一个在你的 T-SQL 查询设计中可能发生的常见问题涉及数据类型。在 SQL Server 中有许多不同的地方你需要定义数据类型。它们可以在表、临时表、参数、存储过程、变量中定义，甚至可以通过你的应用程序代码指定数据类型。如第 1 章所讨论的，SQL Server 可以在后台将一种数据类型转换为另一种；这个过程被称为`` `隐式转换` ``。虽然 SQL Server 可以处理`隐式转换`并有一个结构化的过程来执行，但强制 SQL Server 执行`隐式转换`也可能带来相关的 CPU 成本。代码清单 6-5 是`隐式转换`的一个例子。

```sql
DECLARE @IsActive NVARCHAR(1);
SET @IsActive = 1;
SELECT CustomerID,
       FirstName,
       LastName,
       [Address],
       City,
       PostalCode,
       IsActive,
       DateCreated,
       DateModified,
       DateDisabled
FROM dbo.Customer
WHERE IsActive = @IsActive;
```
