# 8. 为 T-SQL 优化数据库

在本书的这一部分，我介绍了编写 `T-SQL` 代码的许多方面。我首先介绍了在查询中使用基于集合的设计。在设计查询时，设计能有效利用硬件的查询也很重要。检查查询性能的一种方法是使用执行计划。一旦你获得了想要改进的查询的执行计划，就可以开始专注于如何优化该查询。在优化 `T-SQL` 方面，有一些可用的选项。

当你开始对查询进行性能调优时，有许多选项可供选择。过去，一个选项是手动识别性能不佳的查询并进行改进。在识别需要优化的查询时，有许多不同的因素需要考虑。从 `SQL Server 2017` 开始，附加功能允许 SQL Server 自动改进执行 `T-SQL` 某些方面的性能。随着 `SQL Server 2019` 的发布，有了更多的增强功能，可以在不改变底层 `T-SQL` 代码的情况下，帮助查询运行得更好、更快。本章将展示 SQL Server 的最新变化，包括 `SQL Server 2022`，以及如何利用它们在不重写 `T-SQL` 的情况下提升代码性能。


## 使用查询存储

在 SQL Server 中执行查询时，一个关键优势是使用执行计划。在第[7]章中，我简要讨论了内存如何在 SQL Server 中使用，而在第[8]章中，我解释了执行计划如何保存在它们自己的计划缓存中。执行计划的一个挑战发生在执行计划从计划缓存中被清除时。生成新的执行计划不仅有关联成本，而且还存在新的执行计划可能比原始计划性能更差的风险。

从 SQL Server 2016 开始，有了一项关于如何管理执行计划的新功能。除了仅在计划缓存中提供执行计划外，还可以选择保存有关已生成执行计划的历史信息。最大的优势在于与计划本身一起保存的性能信息。这个新功能称为**查询存储**。要使用查询存储，必须按清单[8-1]所示启用它。

```sql
ALTER DATABASE Menu
SET QUERY_STORE (OPERATION_MODE = READ_WRITE);
-- 清单 8-1
-- 启用查询存储的查询
```

启用查询存储后，SQL Server 可以随时间跟踪执行计划。SQL Server 2022 还引入了为可用性组中的副本编译查询存储数据的功能。这可能非常有用，特别是如果您使用只读副本来卸载报告任务。除了跟踪执行计划外，查询存储还记录执行统计信息。从 SQL Server 2017 和 Azure SQL Database 开始，查询存储还跟踪有关特定查询执行的等待统计信息。您可以使用查询存储来识别一个被编译或优化成不同计划的查询。您还可以使用查询存储来强制优化器一致地选择特定性能更好的计划。

在 SQL Server 2016 中，您必须手动管理查询存储以确定您的 T-SQL 代码是否有更好的执行计划。如果您在查询存储中找到一个会受益于某个执行计划的查询，您需要手动为该查询强制一个计划。这导致了随时间推移的额外维护，因为随着数据库中数据的变化，您需要返回并撤销强制的计划。

### 优化逻辑读取

您可能见过一些看起来运行良好的 T-SQL 代码。代码及时执行并返回正确的结果。然而，当您查看某个 SQL Server 监控工具时，您可能会注意到一些意外行为，表明您的系统上存在性能问题。在某些情况下，您可能会发现一些查询正在读取未包含在结果集中的内存中的页面。读取数据页可以识别为逻辑读取。您应该查看 T-SQL 代码的执行计划，以找出可能导致此差异的原因。这涉及查看正在执行的 T-SQL 代码，并尝试确定可以采取哪些措施来提高性能。

优化此查询以减少读取量的优势在于，这应该会减少作为此查询执行的一部分而读入内存的数据页数量。换句话说，读取比查询结果所需更多数据的缺点是，应用程序更频繁使用的页面可能会过早地从内存中清除。不幸的是，我们通常只在发现应用程序运行缓慢或崩溃时才意识到性能问题。此时，很难诊断出导致问题的具体查询。如果您无法访问监控软件，这种情况可能会变得更加难以排查。当我最初开始调查性能问题时，我会检查 SQL Server 代理的作业历史记录。有时您会发现一个与生产问题同时运行的进程。虽然这并不能确认 SQL Server 代理作业是原因，但它可能值得调查。

对于这些情况，最好是知道导致性能问题的确切存储过程或预处理语句。除了第三方软件、查询存储或先前设置的扩展事件之外，您可能只有很少的可用选项。此外，我更喜欢在发生中断之前识别可能导致性能问题的查询。如果我无法访问第三方工具，我会使用 DMV 在查询成为问题之前找到它们。我经常将前述任何操作与性能监视器结合使用，以帮助随时间跟踪 SQL Server 并确认我的性能调整是否有效。

如果您没有第三方工具或者您想更熟悉 DMV 以找到需要性能调整的 T-SQL，您可以查询 `sys.dm_exec_query_stats`。这个 DMV 跟踪已执行的 T-SQL 代码的统计信息，并且仍然是计划缓存的一部分。这将仅返回尚未从计划缓存中清除的查询的结果。如果您认为此 DMV 缺少您期望看到的一些存储过程，您可能需要对您的计划缓存进行一些研究，并确定缓存是否被即席查询填满。您需要将此数据与 `sys.dm_exec_sql_text` 连接，以查找与统计信息关联的 T-SQL 代码。

在尝试查找具有高逻辑读取的查询时，您希望根据各种条件查看 T-SQL 代码。查看平均逻辑读取通常是我寻找要性能调整的查询时的主要目标。我更喜欢这个指标，因为它通常能识别在服务器上频繁运行并且读取量可能超出我预期的 T-SQL 代码。如果有一个在服务器上频繁运行的进程，我不期望平均读取次数会很高，因为一个频繁的进程意味着 T-SQL 代码与应用程序直接相关。还有一些查询运行频率不高，但运行时使用大量逻辑读取。根据我的经验，如果总逻辑读取量很高而总执行次数很低，我会将这些查询的优先级排在那些具有更高平均逻辑读取值和大量执行次数的 T-SQL 代码之后。
