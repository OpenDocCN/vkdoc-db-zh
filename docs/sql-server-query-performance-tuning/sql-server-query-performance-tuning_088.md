# 第 9 章 ■ 索引分析

## 索引的其他特性

索引的其他属性可能对性能产生正面或负面影响。这里将探讨其中的一些行为。

### 不同的列排序顺序

SQL Server 支持为复合索引的不同列指定不同的排序顺序。

假设你需要一个索引，其第一列按升序排序，第二列按降序排序，以消除可能非常耗时的排序操作。你可以按如下方式实现：

```sql
CREATE NONCLUSTERED INDEX i1 ON t1(c1 ASC, c2 DESC);
```

### 计算列上的索引

你可以在计算列上创建索引，只要为该计算列定义的表达式符合某些限制，例如仅引用包含该计算列的表中的列，并且它是确定性的。

### BIT 数据类型列上的索引

SQL Server 允许你在 `BIT` 数据类型的列上创建索引。单独在 `BIT` 数据类型列上创建索引的能力优势不大，因为这样的列通常只有两个唯一值，除非在极少数情况下，绝大多数数据是一个值，只有少数行是另一个值。如前所述，这种低选择性（唯一值数量）的列通常不是索引的良好候选者。

然而，当你考虑覆盖索引时，这个特性就显得很重要了。因为覆盖索引要求包含索引中的所有列，能够将 `BIT` 数据类型列添加到索引键中，使得覆盖索引在需要时可以包含这样的列（在通常属于 `INCLUDE` 子句的列之外）。

### CREATE INDEX 语句作为查询处理

`CREATE INDEX` 操作被集成到查询处理器中。优化器可以在创建索引时利用现有索引来减少扫描成本和排序。

以 `Person.Address` 表为例。存在一个包含多个列的非聚集索引：`AddressLine1`、`AddressLine2`、`City`、`StateProvinceId` 和 `PostalCode`。如果你需要针对 `City` 列运行查询，并且已有这个索引，你会得到对该索引的扫描。现在像这样创建一个新索引：

```sql
CREATE INDEX IX_Test ON Person.Address(City);
```

你可以在图 9-17 中看到，优化器选择扫描现有索引，而不是扫描整张表来创建新索引，因为新索引所需的列已经包含在其他非聚集索引中。

### 并行索引创建

SQL Server 支持为 `CREATE INDEX` 语句使用并行计划，就像支持其他 SQL 查询一样。在多处理器机器上，索引创建不会局限于单个处理器，而是会受益于多个处理器。你可以使用 SQL Server 的 `max degree of parallelism` 配置参数来控制在 `CREATE INDEX` 语句中使用的处理器数量。通过执行 `sp_configure` 存储过程可以看到此参数的默认值为 0。

```sql
EXEC sp_configure 'max degree of parallelism';
```

默认值 0 意味着 SQL Server 可以使用系统中所有可用的 CPU 来并行执行 T-SQL 语句。在具有四个处理器的系统上，可以通过执行 `sp_configure` 将最大并行度设置为 2。

```sql
EXEC sp_configure 'max degree of parallelism', 2;
RECONFIGURE WITH OVERRIDE;
```

这允许 SQL Server 最多使用两个 CPU 来并行执行 T-SQL 语句。此配置设置立即生效，无需重启服务器。

查询提示 `MAXDOP` 可用于 `CREATE INDEX` 语句。同时请注意，并行 `CREATE INDEX` 功能仅在 SQL Server 企业版中可用。

### 联机索引创建

索引的默认创建是作为脱机操作执行的。这意味着在表上放置排他锁，在创建索引期间限制用户访问。可以以联机操作的方式创建索引。这允许用户在索引创建期间继续访问数据。代价是增加创建索引所需的时间和资源。从 SQL Server 2012 开始引入，包含 `varchar(MAX)`、`nvarchar(MAX)` 和 `varbinary(MAX)` 的索引实际上可以在线重建。联机索引操作仅在 SQL Server 企业版中可用。

### 考虑数据库引擎优化顾问

一种简单的索引方法是使用 SQL Server 提供的数据库引擎优化顾问工具。这是一个基于使用情况的工具，它查看特定的工作负载，并与查询优化器协作以确定各种索引组合的相关成本。基于该工具的分析，你可以适当地添加或删除索引。
