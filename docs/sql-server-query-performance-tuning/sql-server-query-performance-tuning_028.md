# 第 1 章 SQL 查询性能调优

## 过度的阻塞和死锁

由于 SQL Server 完全遵循原子性、一致性、隔离性和持久性（ACID）原则，数据库引擎确保并发事务所做的修改彼此正确隔离。默认情况下，一个事务看到的数据要么是另一个并发事务修改之前的状态，要么是另一个事务完成之后的状态——它不会看到中间状态。

由于这种隔离性，当多个事务试图以不兼容的方式并发访问同一资源时，数据库中就会发生 `阻塞`。两个进程无法同时更新同一数据。此外，由于 SQL Server 中的所有更新都建立在数据页（大小为 8KB 的行集合）之上，你可以看到，即使两个进程并未更新同一行，也可能发生阻塞。就确保正确的数据存储和检索而言，阻塞是件好事，但在错误的地方出现过多阻塞会拖慢速度。

与阻塞相关但实际是另一个问题的是，当两个资源试图升级或扩展锁资源并相互冲突时，就会发生 `死锁`。查询引擎会确定回滚代价最小的进程，并选择其作为 `死锁牺牲品`。这要求重新提交数据库请求以成功执行。尽管许多人认为死锁是一个结构性问题，但它实际上是一个根本性的性能问题。查询的执行时间会受到其面临的阻塞和死锁（如果有的话）数量的负面影响。

为了实现多用户数据库应用程序的可扩展性能，正确控制查询的隔离级别和事务范围以最小化阻塞和死锁至关重要；否则，即使硬件资源可能未被充分利用，查询的执行时间也会显著增加。我将在第 20 和 21 章深入探讨这个问题。

## 非集合操作

Transact-SQL 是一种基于集合的脚本语言，这意味着它操作的是数据集。这迫使你从列的角度而非行的角度思考问题。非集合式的思维会导致过度使用游标和循环，而不是探索更高效的连接和子查询。T-SQL 语言提供了丰富的机制来操作数据集。为了让性能出色，你需要利用这些机制，而不是在代码中强制采用逐行处理的方法，后者会扼杀性能。关于如何做到这一点的例子贯穿全书；同时，我将在第 18 章讨论 T-SQL 最佳实践，在第 22 章讨论游标。

## 不恰当的数据库设计

数据库应进行充分规范化，以提高数据检索性能并减少阻塞。

例如，如果你有一个规范不足的数据库，客户和订单信息在同一张表中，那么客户信息将在该客户的所有订单行中重复。这种每行信息的重复会增加获取客户所有订单所需的页面读取次数。同时，处理客户订单的数据写入器将锁定所有包含客户信息的行，从而可能阻塞所有试图访问客户资料的其他数据写入器/数据读取器。

数据库的过度规范化可能与规范化不足一样糟糕。过度规范化增加了检索数据所需的连接数量和复杂性。过度规范化的数据库包含大量表，但每个表的列数很少。过度规范化并不是我经常遇到的问题，但当我遇到时，它会对性能产生严重影响。更常见的情况是处理结构规范化不足或不当的问题。

查询中连接过多也可能是因为数据库实体未清晰分区，或者查询服务于一组复杂的需求，而通过创建新的存储过程可能能更好地满足这些需求。
