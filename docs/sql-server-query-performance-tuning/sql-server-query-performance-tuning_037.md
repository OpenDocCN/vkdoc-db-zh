# 第二章：内存性能分析

SQL Server 2014 引入了一种新的表类型，称为*内存*表。它将表的存储从磁盘转移到内存中，从而显著提升了性能。然而，并非所有的表或工作负载都能从这一新功能中受益。你需要密切关注内存表的常规查询性能指标，并利用特定的 DMV（动态管理视图）来监控内存表。我将在第 23 章详细讨论这一切。如果你的工作负载与内存表配合不佳，或者系统内存不足，你可能需要将那些内存表移回磁盘存储。

### 增加系统内存

SQL Server 的内存需求取决于 SQL 活动所处理的数据总量。它与数据库的大小或传入的 SQL 查询数量没有直接关联。例如，一个内存密集型的查询如果对两个小表进行无任何过滤条件以缩小结果集的交叉连接，就可能给系统内存带来巨大压力。

最简单快速的解决方法之一就是购买并安装更多内存以直接增加系统内存。然而，找出消耗物理内存的原因仍然很重要，因为如果应用程序工作负载是极端内存密集型的，你很快就会受到系统可访问最大内存量的限制。要识别哪些查询使用了更多内存，可以查询 `sys.dm_exec_query_memory_grants` DMV 并收集有关查询及其 I/O 使用情况的指标。不过，当你使用 `JOIN` 或 `ORDER BY` 语句针对此 DMV 运行查询时请务必小心；如果你的系统已经处于内存压力之下，这些操作可能导致你的查询本身需要分配内存授权。

### 从 32 位处理器更改为 64 位处理器

将物理服务器从 32 位处理器切换到 64 位处理器（并随之升级 Windows Server 软件）会从根本上改变 SQL Server 的内存管理能力。SQL Server 的内存限制从 3GB 提升到高达 8TB，具体取决于操作系统版本和特定的处理器类型。

在 SQL Server 2012 之前，可以通过使用地址窗口扩展（Address Windowing Extensions）向 SQL Server 实例添加最多 64GB 的数据缓存。这些功能在 SQL Server 2012 中被移除，因此 32 位的 SQL Server 实例被限制为只能访问 3GB 内存。由于此限制，只有小型系统才应运行 SQL Server 2014 的 32 位版本。

### 压缩数据

数据压缩在存储和检索信息方面有许多显著的优点。它还有一个许多人没有意识到的好处：当压缩信息存储在内存中时，它会保持压缩状态。这意味着可以用更少的系统内存移动更多的信息，从而提高你的整体内存吞吐量。

这一切确实会带来一定的 CPU 开销，因此你需要密切关注，以确保你不是在简单地转移压力。有时，由于数据的性质，你可能看不到太大的压缩效果。

### 启用 3GB 进程地址空间

标准的 32 位地址最多可以映射 4GB 的内存。因此，32 位 Windows 操作系统进程的标准地址空间被限制为 4GB。在这 4GB 的进程空间中，默认情况下，上部 2GB 预留给操作系统，下部 2GB 供应用程序使用。如果你在 32 位操作系统的 `boot.ini` 文件中指定 `/3GB` 开关，操作系统只保留 1GB 的地址空间，应用程序可以访问最多 3GB。这也被称为*4GB 调优*（4GT）。此目的不需要新的 API。

因此，在一个具有 4GB 物理内存和默认 Windows 配置的机器上，你会发现可用内存大约为 2GB 或更多。为了让 SQL Server 使用最多 3GB 的可用内存，你可以在 `boot.ini` 文件中添加 `/3GB` 开关，如下所示：

```
[boot loader]
timeout=30
default=multi(0)disk(0)rdisk(0)partition(1)\WINNT
[operating systems]
multi(0)disk(0)rdisk(0)partition(1)\WINNT=
"Microsoft Windows Server 2012 R2 Advanced Server"
/fastdetect /3GB
```

如后续章节所述，对于物理内存超过 16GB 的系统，或需要更高内核内存量的系统，不应使用 `/3GB` 开关。

64 位系统上的 SQL Server 2014 在 x64 平台上可以支持高达 8TB。将生产系统，特别是企业级生产系统，放在 32 位架构上已经没有太大意义了。

### 处理碎片

虽然存储碎片听起来可能不像是一个性能问题，但由于 SQL Server 从磁盘检索信息到内存的方式，访问的是信息页。如果你有高水平的碎片，这将直接转化为你的内存管理问题，因为你必须将从磁盘检索到的页面按原样存储到内存中，包括空白空间。因此，碎片不仅影响存储，也会影响内存。

## 小结

在本章中，你了解了性能监视器和 DMO（动态管理对象）。你探索了在 SQL Server 中收集内存和内存行为指标的不同方法。理解内存的行为方式将帮助你理解系统的性能表现。你还看到了除简单地购买更多内存之外，解决内存问题的多种可能方案。SQL Server 将会利用你能提供的所有内存，所以请管理好这项资源。
