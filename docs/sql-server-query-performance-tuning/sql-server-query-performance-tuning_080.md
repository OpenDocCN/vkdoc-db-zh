# 第 8 章 ■ 索引架构与行为

聚集索引建立在主键上，虽然对该表的大多数访问可能都是通过该键进行的，这使得索引很有用，但在当前这个实例中，聚集索引的性能并未达到你需要的水平。虽然你可以将索引的定义扩展到包含查询中的所有其他列，但它们对于聚集索引的功能并非必需，并且会干扰主键的操作。相反，你可以使用 `INCLUDE` 操作来将其中定义的列存储在索引的叶级。它们不会以任何方式影响索引的键结构，但通过牺牲一些额外的磁盘空间，可以提供使非聚集索引成为覆盖索引（稍后详细介绍）的能力。在这个实例中，创建一个不同的索引是合适的。

```sql
CREATE NONCLUSTERED INDEX ixTest
ON Sales.CreditCard (ExpMonth, ExpYear)
INCLUDE (CardNumber);
```

现在，当再次运行查询时，结果如下：

表 'CreditCard'。扫描计数 1，逻辑读取 32 次

CPU 时间 = 0 毫秒，elapsed time = 166 毫秒。

在这种情况下，`SELECT` 语句不包含任何需要从非聚集索引页跳转到表的数据页的列，而这通常是导致对于大型结果集和/或排序输出而言，非聚集索引比聚集索引开销更大的原因。这种非聚集索引被称为 ``覆盖索引``。

测试完成后清理索引。

```sql
DROP INDEX Sales.CreditCard.ixTest;
```

## 本章小结

在本章中，你学习了索引是减少查询逻辑读取次数和磁盘 I/O 的有效方法。尽管索引可能会给操作查询增加开销，但即使是像 `UPDATE` 和 `DELETE` 这样的操作查询也能从索引中受益。

要为特定查询决定索引键列，需要评估查询的 `WHERE` 子句和联接条件。诸如列选择性、宽度、数据类型和列顺序等因素对于决定索引键中的列非常重要。由于索引主要用于检索少量行，因此索引列的选择性应该非常高。需要注意的是，非聚集索引包含聚集索引键的值作为其行定位器，因为这种行为极大地影响了索引类型的选择。
