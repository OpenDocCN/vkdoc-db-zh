上一章展示了如何收集查询性能指标。本章将展示如何利用这些指标来识别运行时间长或频繁调用的查询。然后，我将介绍 Management Studio 内置的工具，以便你能够理解给定查询的性能表现。我还将花大量时间讨论如何使用执行计划，这是深入了解查询优化器决策的最佳途径。

在本章中，我将涵盖以下主题：

## 高开销查询

既然你已经看到了两种收集查询性能指标的不同方法，现在让我们看看这些数据代表什么：高开销查询本身。当 SQL Server 的性能变差时，很可能正在发生几件事。

*   首先，某些查询对系统资源造成高压。这些查询会影响整个系统的性能，因为服务器无法足够快地服务其他 SQL 查询。
*   此外，高开销查询会阻塞所有请求相同数据库资源的其他查询，进一步降低那些查询的性能。优化高开销查询不仅可以提高它们自身的性能，还可以通过减少数据库阻塞和对 SQL Server 资源的压力来提高其他查询的性能。
*   最后，一个本身开销不大的查询可能每分钟被调用数千次，仅仅由于次优代码的简单累积，就可能导致主要的资源瓶颈。

要开始确定你需要花时间处理哪些查询，你将使用我到目前为止谈到的资源。例如，假设查询在缓存中，你将能够使用 `DMO` 来整合有意义的数据以确定开销最高的查询。或者，因为你已经使用 `Extended Events` 捕获了查询，你可以访问该数据作为识别开销最高查询的手段。
