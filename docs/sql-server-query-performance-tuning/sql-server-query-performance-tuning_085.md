### 索引视图

SQL Server 中的数据库视图类似于虚拟表，但它只是一个 `SELECT` 语句的输出。你使用 `CREATE VIEW` 语句创建视图，然后可以像查询表一样对视图编写查询。视图本身不存储任何数据——只存储与之关联的 `SELECT` 语句。每次查询视图时，定义该视图的 `SELECT` 语句都会被发送到优化器。

通过在视图上创建唯一的聚集索引，数据库视图可以在磁盘上具体化。这样的视图被称为 `索引视图` 或 `物化视图`。在视图上创建唯一聚集索引后，视图的结果集会立即具体化，并持久化存储在数据库的物理存储中，从而省去了在查询执行时进行昂贵操作的开销。视图具体化后，可以在该索引视图上创建多个非聚集索引。实际上，这将视图（再次说明，它只是一个查询）变成了一个具有定义存储的实际表。

#### 优势

你可以通过以下方式使用索引视图来提升查询性能：

*   聚合可以预计算并存储在索引视图中，以最小化查询执行期间的昂贵计算。
*   表可以预先连接，并且结果数据集可以被具体化。
*   连接或聚合的组合可以被具体化。

#### 开销

索引视图可能会对 `OLTP` 数据库产生重大开销。索引视图的一些开销如下：

*   基础表中的任何更改都必须通过执行视图的 `SELECT` 语句反映到索引视图中。
*   对定义了索引视图的基础表的任何更改，都可能引发该索引视图的一个或多个非聚集索引的更改。如果更新了聚集键，则聚集索引也必须更改。
*   索引视图增加了数据库持续维护的开销。
*   数据库需要额外的存储空间。

创建索引视图的限制包括：

*   视图上的第一个索引必须是唯一的聚集索引。
*   只有在创建了唯一的聚集索引之后，才能在索引视图上创建非聚集索引。
*   视图定义必须是 `确定性` 的——也就是说，对于给定的查询，它只能返回一种可能的结果。（SQL Server 联机丛书中提供了确定性和非确定性函数的列表。）
*   索引视图必须只引用同一数据库中的基础表，而不能引用其他视图。
*   索引视图可以包含 `float` 列。但是，此类列不能包含在聚集索引键中。
*   索引视图必须与视图中引用的表进行架构绑定，以防止表架构的修改（这通常是一个主要问题）。
*   视图定义在语法上有若干限制。（SQL Server 联机丛书中提供了视图定义的语法限制列表。）
