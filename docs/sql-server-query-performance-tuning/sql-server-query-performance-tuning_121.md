# 第 13 章 ■ 索引碎片

将图 13-16 中的查询结果与图 13-15 中的早期结果进行比较。你可以看到内部和外部碎片都得到了有效减少。以下是输出分析：

*   **内部碎片**：该表有 42 行，平均行大小（2,019.38 字节）允许每页最多存储四行。如果行被高度整理以将内部碎片降至最低，那么表中应有 11 个数据页（或聚集索引中的叶级页）。你可以从上述输出中观察到以下信息：
    *   **叶级（或数据）页的数量**：`pagecount = 11`
    *   **页面中的信息量**：`avg_page_space_used_in_percent = 95.33 percent`
*   **外部碎片**：至少需要两个区来容纳这 11 个页。要实现最低程度的外部碎片，两个区之间不应有任何间隔，并且所有页都应按照索引的逻辑顺序物理排列。上述输出显示了顺序错误的页数 = `avg_fragmentation_in_percent = 36.6 percent`。虽然这可能不是完美的碎片水平，但考虑到索引的大小，超过 20%是可以接受的。区的数量较少且彼此对齐，访问速度会更快。

在 SQL Server 2005 及更高版本中重建索引也会压缩大型对象（LOB）页。你可以通过设置 `LOB_COMPACTION = OFF` 来选择不压缩。如果你不担心存储空间，但关心索引重组花费的时间，关闭此选项可能是可取的。

当你在创建索引时使用 `PAD_INDEX` 设置，它决定了索引中间级页面上要保留多少可用空间，这可以帮助你处理页拆分。在索引重建时会考虑到这一点，除非你另有指定，否则新页面将被设置回你最初在索引创建时确定的值。我几乎从未见过这在大多数系统上产生重大影响。你需要在你的系统上进行测试以确定它是否有帮助。

如果你没有另行指定，默认行为是对所有分区中的所有索引进行碎片整理。如果你想控制该过程，你只需指定要重建哪个分区。

如前所述，`ALTER INDEX REBUILD` 技术能有效减少碎片。你也可以用一条语句重建表的所有索引。

```
ALTER INDEX ALL ON dbo.Test1 REBUILD;
```

尽管这是最有效的碎片整理技术，但它确实有一些开销和限制：

*   **阻塞**：与前面两种索引重建技术类似，`ALTER INDEX REBUILD` 会在系统中引入阻塞。它会阻塞所有试图访问该表（或表上任何索引）的其他查询。它也可能被那些查询阻塞。
*   **事务回滚**：由于 `ALTER INDEX REBUILD` 在操作上是完全原子性的，如果在完成之前被停止，那么在此之前执行的所有碎片整理操作都将丢失。

你可以使用 `ONLINE` 关键字运行 `ALTER INDEX REBUILD`，这将减少锁定机制，但会增加重建索引所需的时间。

## 执行 `ALTER INDEX REORGANIZE` 语句

`ALTER INDEX REORGANIZE` 无需重建索引即可减少索引碎片。它通过按索引键的逻辑顺序重新排列索引的现有叶级页来减少外部碎片。它压缩页面内的行，减少内部碎片，并丢弃产生的空页。这种技术不使用任何新页面进行碎片整理。

为了避免与 `ALTER INDEX REBUILD` 相关的阻塞开销，这种技术使用了一种非原子的在线方法。在执行其各个步骤时，它会请求少量锁且锁定时间很短。每完成一个步骤，它就会释放锁并进入下一步。当尝试访问一个页面时，如果发现该页面正在被使用，它会跳过该页面且不再返回。这允许其他查询与 `ALTER INDEX REORGANIZE` 操作同时在表上运行。此外，如果此操作被中途停止，那么在此之前执行的所有碎片整理步骤都会被保留。

由于 `ALTER INDEX REORGANIZE` 不使用任何新页面来重新排序索引，并且会跳过被锁定的页面，因此这种方法提供的碎片整理程度通常低于 `ALTER INDEX REBUILD`。为了观察 `ALTER INDEX REORGANIZE` 与 `ALTER INDEX REBUILD` 相比的相对有效性，请重建上一节中用于 `ALTER INDEX REBUILD` 的测试表。

现在，要减少聚集索引的碎片，使用 `ALTER INDEX REORGANIZE` 如下：

```
ALTER INDEX i1 ON dbo.Test1 REORGANIZE;
```
