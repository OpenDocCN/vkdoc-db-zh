# SQL Server 统计信息与索引碎片管理

## 统计信息管理

### 异步更新统计信息

在大多数情况下，等待统计信息在查询计划生成前进行更新（这是默认行为）是完全可以接受的。只有在极少数情况下，当统计信息更新或由此更新引起的执行计划重编译的代价过高（甚至高于使用过时统计信息的代价）时，你才可以考虑启用统计信息的异步更新。但需要明白，这可能会导致那些本可受益于更新统计信息的存储过程，在下次运行前性能表现较差。请记住——必须先启用统计信息的自动更新功能，才能启用异步更新。

### 收集统计信息的采样量

通常建议使用默认的采样率。这个比率是由一个高效算法根据数据量和修改次数决定的。尽管默认采样率在大多数情况下是最佳选择，但如果你发现某个特定查询的统计信息不准确，可以使用`FULLSCAN`选项手动更新它们。你也可以选择使用`SAMPLE number`来设置特定的采样百分比。这个数字既可以是一个百分比，也可以是一个固定的行数。

如果需要频繁执行此操作，可以添加一个 SQL Server 作业来处理。出于性能考虑，请确保该 SQL 作业安排在非高峰时段运行。要识别默认采样率并非最佳的情况，应在排查数据库性能问题时，分析高成本查询的统计信息有效性。请记住，`FULLSCAN`代价高昂，因此只应在你确定能真正受益的表或索引上运行。

## 小结

如本章所述，SQL Server 基于成本的优化器需要准确的统计信息，以了解用于筛选和联接条件中的列的信息，从而确定高效的处理策略。索引键的统计信息总是在创建索引时生成，并且默认情况下，SQL Server 也会随着数据变化保持索引列和非索引列的统计信息更新。这使得它能够根据当前的数据分布情况确定最佳处理策略。

尽管你可以禁用自动创建统计信息和自动更新统计信息功能，但建议保持这些功能*开启（on）*，因为它们对优化器带来的好处几乎总是超过其开销成本。对于一个高成本的查询，应分析统计信息，以确保自动统计信息维护能兑现其承诺。最好的消息是，只要稍加留意，你就可以高枕无忧，因为自动统计信息在大多数时候都能很好地完成工作。如果使用手动统计信息维护程序，则可以利用 SQL Server 作业将其自动化。

即使拥有适当的索引和统计信息，一个严重碎片化的数据库也会导致数据检索成本增加。在下一章中，你将了解索引中的碎片如何影响查询性能，并学习如何分析和解决碎片问题。
