# SQL 查询性能调优

## 服务器配置
尽管对硬件、操作系统和 `SQL Server` 实例进行适当配置对于数据库应用获得最佳性能至关重要，但这些领域已经高度标准化，通常只需花费有限的时间即可完成针对性能的正确配置。而另一方面，诸如查询设计和索引策略等应用设计问题，则是您的代码和数据集所独有的。因此，数据访问层通常比硬件、操作系统或 `SQL Server` 配置有更多的优化空间。图 1-3 展示了一项针对 346 名数据专业人士的调查结果（经 Paul Randal 许可）：[`bit.ly/1gRANRy`](http://bit.ly/1gRANRy)。

`图 1-3. 性能问题的根本原因`

如您所见，前两个问题是 `T-SQL` 代码和糟糕的索引。前六大问题中有四个都直接与 `T-SQL`、索引、代码和数据结构相关。我的经验与其他受访者一致。您可以通过首先关注数据访问领域（包括逻辑/物理数据库设计、查询设计和索引设计）来获得数据库应用性能的最大提升。

诚然，如果您专注于硬件配置和升级，可能会获得令人满意的性能提升。然而，应用程序发送的一个糟糕的 `SQL` 查询会耗尽所有可用的硬件资源，无论您拥有多少资源。因此，糟糕的应用设计会使硬件升级的需求变得非常高，甚至超出您的成本限制。在存在繁重 `SQL` 工作负载的情况下，专注于硬件配置和升级通常只能带来较低的投资回报。

## SQL 查询性能调优
您应该从两个层面分析应用程序对 `SQL Server` 数据库造成的压力：

`高层级`：分析数据库应用对各个硬件资源造成的压力以及 `SQL Server` 安装的整体行为。对此的最佳度量是各种等待状态。此信息可以通过两种方式帮助您。首先，它帮助您确定 `SQL Server` 应用中性能不佳需要集中关注的领域。其次，它帮助您识别较高层级是否存在配置不当的问题。然后，您可以决定在无法使用 `Performance Monitor` 工具调整应用程序的情况下（如第 2 章所述）可能需要升级哪种硬件资源。

`低层级`：识别应用程序内部的确切元凶——换句话说，就是那些在较高层级造成大部分可见压力的 `SQL` 查询。这可以使用 `Extended Events` 工具和各种 `动态管理视图` 来完成，如第 6 章所述。

## SQL Server 性能杀手
现在让我们考虑可能降低 `SQL Server` 性能的主要问题领域。通过预先了解 `SQL Server` 中主要的性能杀手，您将能够将调优工作集中在可能的原因上。

一旦您优化了硬件、操作系统和 `SQL Server` 设置，`SQL Server` 中的主要性能杀手如下，按大致顺序排列（最严重的排在前面）：

*   索引不足
*   统计信息不准确
*   查询设计不当
*   执行计划生成不佳
*   过度的阻塞和死锁
*   非基于集合的操作，通常是 `T-SQL` 游标
*   数据库设计不合理
*   碎片过多
*   执行计划不可重用
*   查询频繁重新编译
*   游标使用不当
*   数据库事务日志配置不当
*   `tempdb` 的过度使用或配置不当

让我们快速浏览一下每个问题。

### 索引不足
索引不足通常是 `SQL Server` 中最大的性能杀手之一。如果查询缺乏适当的索引，`SQL Server` 在执行查询时必须检索和处理更多的数据。这会导致对磁盘、内存和 `CPU` 产生巨大压力，从而显著增加查询执行时间。增加的查询执行时间进而可能导致 `SQL Server` 中出现过度的阻塞和死锁。您将在第 8-12 章学习如何确定索引策略和解决索引问题。

通常，索引被认为是数据库管理员 (`DBA`) 的职责。然而，`DBA` 无法主动定义如何使用索引，因为索引的使用是由开发人员编写的数据库查询和存储过程决定的。因此，定义索引必须是共同的责任，因为开发人员通常更了解要检索的数据，而 `DBA` 则更了解索引的工作原理。不了解查询而创建的索引作用甚微。

`注意`：由于不了解查询而创建的索引作用甚微，数据库开发人员需要至少像熟悉 `T-SQL` 那样理解索引。

### 统计信息不准确
`SQL Server` 严重依赖基于成本的优化，因此准确的数据分布统计信息对于有效使用索引至关重要。没有准确的统计信息，`SQL Server` 内置的查询优化器就无法准确估计查询影响的行数。由于从表中检索的数据量对于决定如何优化查询执行非常重要，如果数据分布统计信息维护不准确，查询优化器的效果就会大打折扣。统计信息可能会因未更新而过时。您还可能看到数据以倾斜方式分布的问题损害了统计信息。在自动递增的列（如 `identity` 或日期时间）上的统计信息，可能会随着新数据的添加而过时。您将在第 12 章学习如何分析统计信息。

### 查询设计不当
索引的有效性在很大程度上取决于您编写 `SQL` 查询的方式。从表中检索过多的行，或者指定的筛选条件返回了比所需更大的结果集，都会使索引失效。为了提高性能，您必须确保编写的 `SQL` 查询能够充分利用新的或现有的索引。未能编写高性价比的 `SQL` 查询可能会阻止 `SQL Server` 选择合适的索引，从而增加查询执行时间和数据库阻塞。第 20 章涵盖了如何编写有效的查询。

查询设计不仅涵盖单个查询，还包括通常用于实现数据库功能（例如队列读取器和写入器之间的队列管理）的一组查询。即使设计中使用的单个查询性能良好，数据库的整体性能也可能非常差。解决这种瓶颈需要对 `SQL Server` 不同的特性有广泛的了解，这些特性会影响数据库功能的性能。您将在本书中看到如何使用 `SQL` 查询设计有效的数据库功能。

### 执行计划生成不佳



允许 SQL Server 建立高效的存储过程并反复重用该过程而不必重新编译的相同机制，在某些情况下可能会适得其反。一个糟糕的执行计划可能是一个真正的**性能杀手**。当一个称为 `参数探测` 的过程出现问题时，常常会导致不准确且性能低下的计划。参数探测是源于查询优化器所使用机制的一个过程，该优化器基于统计信息中采样或特定的值来确定最佳计划。理解统计信息和参数如何结合以创建执行计划，以及你可以采取哪些措施来控制它们，这一点非常重要。统计信息在第 12 章中介绍，执行计划分析在第 14 和 15 章中介绍。我额外添加了第 16 章，专门讨论不良的参数探测问题以及如何最好地处理它。
