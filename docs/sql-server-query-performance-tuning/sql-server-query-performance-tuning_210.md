# 第 26 章 ■ SQL Server 优化清单

### 避免对存储过程名称使用 `sp_` 前缀

作为规则，不要为用户存储过程使用 `sp_` 前缀，因为 SQL Server 会假定以 `sp_` 开头的存储过程是系统存储过程，而它们应该位于 `master` 数据库中。使用 `sp` 或 `usp` 作为用户存储过程的前缀非常普遍。这既不是主要的性能问题，也不是大麻烦，但何必自找麻烦呢？`sp_` 前缀带来的性能影响在第 19 章的“小心命名存储过程”一节中有详细解释。完全去掉前缀是个不错的方法。您有足够的空间使用描述性的对象名称，无需使用那些不能增加查询功能定义的奇怪缩写。

### 最小化触发器的使用



触发器为在数据库内自动化行为提供了一种极具吸引力的方法。由于触发器在数据被其他进程（无论这些进程是什么）操纵时触发，因此可被用于确保某些功能在数据变更时自动执行。然而，同样的功能也使其具有危险性，因为对于正在系统上工作的开发人员或数据库管理员（DBA）来说，它们并非立即可见。在设计查询和排查性能问题时，必须考虑到触发器的存在。由于它们带有一种隐性的成本，因此需要仔细权衡是否使用触发器。在使用触发器之前，请确保解决当前问题的唯一方法就是触发器。如果你确实使用了触发器，请在尽可能多的地方记录这一事实，以确保其他开发人员和 DBA 能考虑到触发器的存在。

### 考虑将表放入内存存储

尽管新的内存存储机制存在大量限制，但其性能提升非常显著。如果你拥有一个高吞吐量的 OLTP 系统，并且观察到大量的 I/O 争用，尤其是在门锁（latches）方面，那么内存存储是一个可行的选择。你可能还想探索将内存存储用于表变量，以帮助提升其性能。如果你的数据不需要持久化，你甚至可以使用 `SCHEMA_ONLY` 持久性选项在内存中创建表。所有这些方法都能带来显著的性能收益。

但请记住，你必须有足够的内存来支持这些选项。这里没有什么魔法。你是在通过投入大量的内存（也就是金钱）来解决问题。此外，要实现这一点，你还必须运行 SQL Server 的企业版。

## 配置设置

以下是服务器和数据库配置设置的清单，这些设置对数据库性能有重大影响：
- 内存配置选项
- `并行的成本阈值`
- `最大并行度`
- `即席工作负载优化`
- `已阻塞进程阈值`
- 数据库文件布局
- 数据库压缩

我将在后续章节中更详细地介绍这些设置。

### 内存配置选项

如第 2 章“SQL Server 内存管理”部分所述，强烈建议将 `最大服务器内存` 设置配置为一个由系统配置确定的非默认值。SQL Server 的这些内存配置在第 2 章的“内存瓶颈分析”和“内存瓶颈解决方案”部分有详细说明。

### 并行的成本阈值

在具有多处理器的系统上，可以并行执行查询。并行的默认值为 5。这代表优化器估计该查询需要执行五秒的成本。在大多数情况下，我发现这个值设置得太低；换句话说，设置更高的并行阈值会带来更好的性能。在你的系统上进行测试将帮助你确定合适的值。为这个值提供建议可能有点危险，但我还是要这么做。我会从测试值 35 开始，看看后续效果如何。

### 最大并行度

当系统有多个可用处理器时，默认情况下 SQL Server 会在并行执行期间使用所有处理器。

为了更好地控制机器负载，你可能会发现限制并行执行使用的处理器数量是有用的。此外，你可能需要设置处理器关联性，以便为操作系统以及与 SQL Server 一起运行的其他服务保留特定的处理器。完全禁用并行可能会使 OLTP 系统受益。首先尝试增加并行的成本阈值，因为即使在 OLTP 系统中，也有一些查询会从并行执行中获益。你也可以探索使用资源调控器来控制某些工作负载的可能性。

### 即席工作负载优化

如果你的系统的主要调用是以即席或动态 SQL 的形式发出，而不是通过定义良好的存储过程或参数化查询（例如你在一些对象关系映射（ORM）软件实现中可能看到的那样），那么启用 `即席工作负载优化` 设置将减少过程缓存的消耗，因为对于初始查询调用，会创建计划存根，而不是完整的执行计划。

这将在第 17 章中详细讨论。

### 已阻塞进程阈值

`已阻塞进程阈值` 设置以秒为单位定义触发已阻塞进程报告的时间点。当查询运行并超过此阈值时，报告就会被触发。同时也会触发一个警报，可用于发送电子邮件或短信。测试单个系统以确定应将其设置为何值。你可以使用 SQL Profiler 定义的跟踪中的事件来监控此设置。

### 数据库文件布局

为方便参考，以下是布局数据库文件时应考虑的最佳实践：
- 将用户数据库的数据文件和事务日志文件放在不同的磁盘上。这允许事务日志磁头顺序推进，而不会因数据文件常用的非顺序 I/O 而随机移动。
- 将事务日志放在专用磁盘上也能增强数据保护。如果数据库磁盘发生故障，你将能够通过备份事务日志来保存直至故障发生点的已完成事务。在恢复过程中使用这个最后的事务日志备份，你将能够将数据库恢复到故障发生点。这被称为 `时间点恢复`。
- 避免为事务日志使用 `RAID 5`，因为对于每个写入请求，`RAID 5` 磁盘阵列产生的磁盘 I/O 次数是 `RAID 1` 或 `10` 的两倍。
- 你可以为数据文件选择 `RAID 5`，因为即使在繁重的 OLTP 系统中，读取请求的数量通常是写入请求的七到八倍。此外，对于读取请求，`RAID 5` 的性能与具有相同总磁盘数的 `RAID 1` 和 `RAID 10` 相似。
- 考虑转向更现代的磁盘子系统，如 `SSD` 或 `FusionIO`。

有关数据库文件布局和 RAID 子系统的详细理解，请参阅第 3 章的“磁盘瓶颈解决方案”部分。

### 数据库压缩

自 2008 年起，SQL Server 的企业版和开发版就提供了数据压缩功能。由于更多数据存储在单个页面上，这可以在空间使用和性能方面带来巨大收益。这些收益的代价是增加了系统 CPU 和内存的开销；然而，收益通常远远超过成本。在实施压缩时请考虑到这一点。

## 数据库管理

供你参考，以下是与性能相关的数据库管理活动的简要清单，你应将其作为管理数据库服务器过程的一部分定期执行：
- 保持统计信息最新。
- 将索引碎片保持在最低水平。
- 避免使用 `自动关闭` 或 `自动收缩` 等自动数据库功能。
- 最小化 SQL 跟踪的开销。
