# 第 1 章 SQL 查询性能调优

## 过度碎片化

在分析数据检索操作时，通常可以假设数据是按索引指示的、有序的方式组织的。然而，如果包含数据的页以无序方式碎片化，或者由于频繁的页拆分导致它们包含的数据量很少，那么数据检索操作所需的读取操作次数将远高于正常情况。碎片化导致的读取操作次数增加会损害查询性能。在第 13 章，你将学习如何分析和消除碎片化。

### 不可重用的执行计划

为了以高效的方式执行查询，SQL Server 的查询优化器会花费相当多的 CPU 周期来创建一个具有成本效益的执行计划。好消息是该计划会被缓存在内存中，因此一旦创建，你就可以重用它。

然而，如果设计的计划无法插入参数值，那么 SQL Server 每次使用不同值重新提交相同的查询时，都会创建一个新的执行计划。因此，为了获得更好的性能，以有助于 SQL Server 缓存和重用执行计划的形式提交 SQL 查询极其重要。我还将讨论诸如计划冻结、强制查询计划以及使用 `即席工作负载优化` 等主题。你将在第 15 章详细了解如何提高执行计划的可重用性。

## 频繁的查询重编译

确保执行计划可重用（独立于查询中使用的值）的标准方法之一是使用存储过程或参数化查询。使用存储过程来执行一组 SQL 查询允许 SQL Server 创建参数化的执行计划。

`参数化执行计划` 独立于存储过程或参数化查询执行期间提供的参数值，因此具有很高的可重用性。频繁的查询重编译会增加 CPU 压力和查询执行时间。我将在第 15 章详细讨论存储过程以及语句重编译的各种原因和解决方案。

## 游标的不当使用



相较于常规的基于集合的 SQL 查询，偏好基于游标（逐行）的结果集——正如 Jeff Moden 精妙命名的“RBAR”（Row By Agonizing Row，逐行痛苦处理，发音为“ree-bar”）——会给 SQL Server 带来大量额外开销。应尽可能使用基于集合的查询。但若不得不使用游标，则务必选用高效的游标类型，例如仅快速转发型。过度使用低效游标会加大 SQL Server 资源压力，导致系统性能下降。如果你必须使用游标，第 22 章将讨论如何正确操作游标。

## 数据库事务日志配置不当

未能遵循通用建议来配置数据库事务日志，会对基于在线事务处理（OLTP）的 SQL Server 数据库性能产生负面影响。为了实现最佳性能，SQL Server 高度依赖于高效访问数据库日志。第 3 章涵盖了如何正确配置数据库事务日志的某些方面。
