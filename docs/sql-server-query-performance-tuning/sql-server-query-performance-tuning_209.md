# 第 26 章 ■ SQL Server 优化清单

尽管索引的性能方面已在第 8、11 和 12 章中详细解释，但这里我将重申一份简短的建议清单，以便于您参考：

-   为索引选择窄列。
-   确保候选列中数据的选择性非常高（即该列必须具有大量唯一值）。
-   优先选择整数数据类型（或整数数据类型的变体）的列。同时，避免在 `VARCHAR` 等字符串数据类型的列上创建索引。
-   在多列索引中，考虑将选择性更高的列列在前面。
-   在索引中使用 `INCLUDE` 列表，作为一种在不更改索引键结构的情况下使索引覆盖查询的方法。通过将列添加到键中（而非 `INCLUDE` 列表），您可以避免昂贵的书签查找操作。
-   在决定对哪些列创建索引时，要特别注意查询的 `WHERE` 子句、`JOIN` 条件列和 `HAVING` 子句。这些可以作为进入表的入口点，特别是当 `WHERE` 子句中的条件对某个列使用高选择性的值或常量进行数据过滤时。这样的子句可以使该列成为索引的主要候选。
-   在选择索引类型（聚集或非聚集）时，请牢记聚集索引和非聚集索引类型的优缺点。

设计聚集索引时要格外小心，因为表上的每个非聚集索引都依赖于聚集索引。因此，在设计和实现聚集索引时，请遵循以下建议：

-   尽可能保持聚集索引狭窄。您不希望因为拥有一个宽的聚集索引而加宽所有非聚集索引。
-   首先创建聚集索引，然后在表上创建非聚集索引。
-   如果需要，在 `CREATE INDEX` 命令中使用 `DROP_EXISTING = {ON|OFF}` 选项以单一步骤重建聚集索引。您不希望重建表上所有非聚集索引两次：一次在删除聚集索引时，另一次在重新创建聚集索引时。
-   不要在频繁更新的列上创建聚集索引。如果这样做，表上的非聚集索引将因为需要与聚集索引键值保持同步而产生额外负载。
-   在适用的情况下，例如当您需要跨大型数据集进行聚合时，考虑使用列存储索引。

为了跟踪您已创建的索引并确定需要创建的其他索引，您应该利用 SQL Server 2014 提供的动态管理视图。通过定期（比如每周一次）检查 `sys.dm_db_index_usage_stats` 中的数据，您可以确定哪些索引实际上正在被使用，哪些是冗余的。那些没有为您的查询做出贡献以帮助您提升性能的索引，只会消耗系统资源。它们不仅需要更多的磁盘空间，而且在表中数据发生变化时，还需要额外的 I/O 来维护索引内的数据。另一方面，查询 `sys.dm_db_missing_indexes_details` 将显示系统认为缺失的潜在索引，甚至会建议 `INCLUDE` 列。您可以访问 DMV `sys.dm_db_missing_indexes_groups_stats` 来查看有关可能受益于特定索引组的查询被调用次数的汇总信息。请记住，要彻底测试这些建议，不要假定它们一定是正确的。所有这些都只是建议。

将这些技巧结合起来，可以为您提供一个长期维护系统中索引的最佳方法。
