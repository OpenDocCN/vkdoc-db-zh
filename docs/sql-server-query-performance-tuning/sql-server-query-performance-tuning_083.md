# 第 9 章 ■ 索引分析

### 索引连接

*索引连接*是索引交集的一种变体，其中覆盖索引技术被应用于索引交集。

如果没有单个索引能覆盖查询，但多个索引组合起来可以覆盖查询，那么 SQL Server 可以使用索引连接来完全满足查询，而无需访问基表。

让我们看看这种索引技术的实际应用。对“索引交集”一节中的查询稍作修改，如下所示：

```sql
SELECT soh.SalesPersonID,
       soh.OrderDate
FROM Sales.SalesOrderHeader AS soh
```

在这种情况下，你可以在 `OrderDate` 列上创建一个新的非聚集索引。

```sql
CREATE NONCLUSTERED INDEX IX_Test
ON Sales.SalesOrderHeader (OrderDate);
```

再次运行你的 `SELECT` 命令。

如你所见，SQL Server 将两个非聚集索引都作为索引 seek（而非扫描）加以利用，然后采用一种交集算法来获取两个子集的索引交集。接着，它从结果数据集中执行键查找，以检索索引中未包含的其余数据。但是，该计划的复杂性表明性能可能更差。检查统计信息 I/O 和时间后，你会发现实际上你确实获得了显著的性能提升：

```
表 'SalesOrderHeader'。扫描计数 2，逻辑读取 10 次
CPU 时间 = 0 毫秒，耗时 = 31 毫秒。
```

读取次数从 689 次下降到了 10 次，尽管该计划在表内使用了三个不同的访问点。执行时间也减少了。你还可以看到计划中发生了额外的操作，例如排序和键查找，通过进一步调整索引，也许能够消除这些操作。然而，值得注意的是，由于你通过 `SELECT *` 命令返回所有列，因此无法通过使用 `INCLUDE` 列来消除键查找，所以你可能还需要调整查询。

为了提高查询性能，SQL Server 可以对表使用多个索引。因此，与其创建宽索引键，不如考虑创建多个窄索引。SQL Server 将能够在需要时一起使用它们，而在不需要时，查询也能从窄索引中受益。在创建覆盖索引时，请确定索引的宽度是否可接受，以及使用包含列是否足以完成任务。如果不行，则识别出现有的非聚集索引中是否包含了覆盖索引所需的大多数列。你可能已经拥有两个现有的非聚集索引，它们共同服务于覆盖索引所需的所有列。如果可能，适当调整现有非聚集索引的列顺序，使优化器能够考虑这两个非聚集索引之间的索引交集。

有时，可能由于以下原因，你不得不创建一个单独的非聚集索引：

• 不允许重新排列现有索引中的列顺序。

• 覆盖索引所需的一些列可能未包含在现有的非聚集索引中。

• 两个现有非聚集索引中的列总数可能多于覆盖索引所需的列数。

在这种情况下，你可以在剩余的列上创建一个非聚集索引。如果新索引与一个现有非聚集索引的组合列顺序满足覆盖索引的要求，优化器将能够使用索引交集。在确定新索引的列及其顺序时，尝试通过关注其他查询来最大化其效益。

删除为测试创建的索引。

```sql
DROP INDEX Sales.SalesOrderHeader.IX_Test;
```
