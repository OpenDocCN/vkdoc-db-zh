## 索引开销

索引带来的性能优势确实伴随着成本。带有索引的表除了表的数据页面外，还需要额外的存储和内存空间来存放索引页面。数据操作查询（`INSERT`、`UPDATE` 和 `DELETE` 语句，或 CRUD 的创建、读取、更新、删除部分）可能需要更长的时间，并且对于不断变化的表，维护索引需要更多的处理时间。这是因为，与 `SELECT` 语句不同，数据操作查询会修改表的数据内容。如果 `INSERT` 语句向表中添加一行，那么它还必须在索引结构中添加一行。如果索引是聚集索引，则开销更大，因为必须按照正确的顺序将该行添加到数据页面本身中，这可能需要将其他数据行重新定位到新条目位置之下。`UPDATE` 和 `DELETE` 数据操作查询以类似方式更改索引页面。

在设计索引时，您将从两个不同的角度进行操作：一是现有生产系统，您需要衡量索引的整体影响；二是战术方法，通常只关注索引的即时利益，在初始设计系统时使用。当您必须处理现有系统时，您应确保索引的性能优势超过处理资源上的额外成本。您可以通过使用扩展事件（在第 3 章中解释）进行整体工作负载优化（在第 25 章中解释）来实现这一点。当您只关注索引的即时利益时，SQL Server 提供了一系列动态管理视图来提供有关索引性能的详细信息，例如 `sys.dm_db_index_operational_stats` 或 `sys.dm_db_index_usage_stats`。视图 `sys.dm_db_index_operational_stats` 显示正在使用的索引上的低级别活动，例如锁和 I/O。视图 `sys.dm_db_index_usage_stats` 返回随时间推移对索引发生的各种索引操作的统计计数。这两者都将在第 20 章讨论阻塞时更广泛地使用。

■ **注意** 在本书中，我使用 `STATISTICS IO` 和 `STATISTICS TIME` 来测量正在运行的查询。您可以将 `SET` 命令添加到代码中，也可以更改查询窗口的连接设置。我建议只更改连接设置。
