## 性能调优流程

性能调优流程包括识别性能瓶颈、对已识别的问题进行优先级排序、排查其原因、应用不同的解决方案、量化性能改进，然后不断重复整个流程。你需要具备一点创造力，因为大多数时候，没有单一的“银弹”可以提升性能。挑战在于缩小可能原因的范围，并评估不同解决方案的效果。在迭代调优过程中，你甚至可以撤销之前的修改。

### 核心流程

在调优过程中，你必须检查可能影响基于 SQL Server 的应用程序性能的各种硬件和软件因素。在进行性能分析时，你应该自问以下一般性问题：

-   同一服务器上是否运行着其他资源密集型应用程序？
-   硬件子系统的能力是否足以承受最大工作负载？
-   SQL Server 配置是否正确？
-   共享环境（无论是虚拟机还是平台）是否有足够的资源？或者我正在处理那里的配置问题，甚至是来自外部力量的资源争用？
-   SQL Server 与数据库应用程序之间的数据库连接是否高效？
-   数据库设计是否支持最快的数据检索（以及用于可更新数据库的修改）？
-   由 SQL 查询组成的用户工作负载是否经过优化以减少 SQL Server 上的负载？
-   哪些进程导致系统变慢？这反映在各种等待状态、性能计数器和动态管理对象的测量中。
-   工作负载是否支持所需的并发级别？

如果这些因素中的任何一个配置不当，那么整个系统性能可能会受到影响。让我们简要探讨一下这些因素。

在同一服务器上运行另一个资源密集型应用程序会限制 SQL Server 可用的资源。即使是作为服务运行的应用程序也可能消耗大量系统资源，从而限制 SQL Server 可用的资源。例如，应用程序可能被配置为以比 SQL Server 更高的优先级使用处理器。`优先级`是分配给资源的权重，促使处理器在执行时给予其更高的优先权。要确定进程的优先级，请按照以下步骤操作：

1.  启动 Windows 任务管理器。
2.  选择 查看 > 选择列。
3.  选中 "基本优先级" 复选框。
4.  单击 "确定" 按钮。

这些步骤会将 "基本优先级" 列添加到进程列表中。随后，你将能够确定 SQL Server 进程 (`sqlservr.exe`) 默认以 "正常" 优先级运行，而 Windows 任务管理器进程 (`taskmgr.exe`) 则以 "高" 优先级运行。因此，为了允许 SQL Server 最大化利用可用资源，你应该查找在 SQL Server 计算机上运行的所有非必要应用程序/服务，并确保它们不是资源消耗大户。

不正确地配置硬件可能会阻止 SQL Server 从可用资源中获得最大收益。需要考虑的主要硬件资源是处理器、内存、磁盘和网络。如果某个特定硬件资源的容量较小，那么它可能很快成为 SQL Server 的性能瓶颈。

虽然我不涉及硬件选择，但作为查询调优的一部分，你确实需要了解你所拥有的硬件可能在何处以及如何导致性能瓶颈。第 2、3 和 4 章详细介绍了其中一些硬件瓶颈。

你还应该查看 SQL Server 的配置，因为正确的配置对于优化应用程序至关重要。有一长串定义 SQL Server 安装通用行为的配置项。这些配置可以使用系统存储过程 `sys.configurations` 查看和修改。

其中许多配置也可以通过 SQL Server Management Studio 以交互方式管理。

由于 SQL Server 配置适用于整个 SQL Server 安装，因此通常首选标准配置。好消息是，通常你无需修改这些配置中的大多数；默认设置在大多数情况下效果最佳。事实上，一般建议是将大多数 SQL Server 配置保持为默认值。我在本书中会详细讨论配置参数，并对更改其中一些参数提出一些建议。数据库选项的情况也类似。model 数据库上的默认设置对大多数系统来说是足够的。你可能需要调整自动增长设置，使其不同于默认值，但许多其他属性，如自动关闭或自动收缩，应保持关闭状态，而其他属性，如自动创建统计信息，在大多数情况下应保持开启。
