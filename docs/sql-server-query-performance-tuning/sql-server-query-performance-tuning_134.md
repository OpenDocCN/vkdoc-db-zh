# 第 15 章 ■ 执行计划缓存行为

为提高计划缓存的可重用性，请使用 `sp_executesql` 将动态 SQL 字符串作为参数化查询执行。

```sql
DECLARE @n NVARCHAR(3) = '776',
    @sql NVARCHAR(MAX),
    @paramdef NVARCHAR(6);

SET @sql = 'SELECT * FROM Sales.SalesOrderDetail sod '
    + 'JOIN Sales.SalesOrderHeader soh '
    + 'ON sod.SalesOrderID=soh.SalesOrderID ' + 'WHERE sod.ProductID=@1' ;

SET @paramdef = N'@1 INT';

--使用系统存储过程 sp_executesql 执行动态查询
EXECUTE sp_executesql
    @sql,
    @paramdef,
    @1 = @n;
```

使用 `sp_executesql` 作为显式参数化查询执行，会为查询生成参数化计划，从而增加执行计划的可重用性。

### 谨慎地将查询的可变部分参数化

在将查询的可变部分转换为参数时要小心。某些变量的取值范围可能差异极大，导致适用于某一范围值的执行计划可能不适用于其他值。这可能导致糟糕的参数嗅探（在第 16 章中介绍）。

### 不要允许查询中对象的隐式解析

SQL Server 允许在不同架构下创建具有相同名称的多个数据库对象。例如，表 `t1` 可以使用两种不同的架构（`u1` 和 `u2`）创建，它们各自独立。大多数系统中的默认所有者是 `dbo`（数据库所有者）。如果用户 `u1` 执行以下查询，SQL Server 会首先尝试查找表 `t1` 是否存在于用户 `u1` 的默认架构中。

```sql
SELECT * FROM tl WHERE cl = 1;
```

如果没有找到，则会尝试查找表 `t1` 是否存在于 `dbo` 用户下。这种隐式解析允许用户 `u1` 在另一架构下创建另一个表 `t1` 实例，并（使用相同的应用程序代码）临时访问它，而不影响其他用户。

在生产数据库上，我建议使用架构所有者并避免隐式解析。如果不这样做，在生产服务器上使用隐式解析会增加以下开销：

-   需要更多时间来识别对象。
-   会降低计划缓存可重用性的效率。

## 小结

SQL Server 基于成本的查询优化器不是根据查询的确切语法，而是通过评估使用不同处理策略执行查询的成本来决定有效的执行计划。使用不同处理策略的成本评估在多个优化阶段完成，以避免在优化查询上花费过多时间。然后，执行计划会被缓存，以便在重新执行相同查询时节省执行计划生成的成本。为提高缓存计划的可重用性，SQL Server 支持在重新运行查询、可变部分取不同值时重用执行计划的不同技术。

使用存储过程通常是提高执行计划可重用性的最佳技术。SQL Server 为存储过程生成参数化执行计划，以便在使用相同或不同参数值重新运行存储过程时可以重用现有计划。但是，如果存储过程的现有执行计划失效，则无法在不重新编译的情况下重用该计划，从而降低了计划缓存可重用性的效率。
