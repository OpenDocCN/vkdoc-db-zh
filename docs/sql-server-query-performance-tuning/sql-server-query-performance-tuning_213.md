# 第 26 章 ■ SQL Server 优化检查清单

#### 有效缓存执行计划

你必须确保查询的 `执行计划` 不仅被缓存，而且被频繁重用。通过采用以下最佳实践来实现：

• 避免以非参数化的即席查询方式执行查询。相反，对查询的可变部分进行参数化，并使用存储过程或 `sp_executesql` 系统存储过程提交参数化查询。

• 如果必须使用大量即席查询，请启用 `即席工作负载优化` 选项，该选项将在第一次调用查询时创建计划存根而不是完整计划。这大大减少了使用的 `过程缓存` 量。

• 在执行相同参数化查询的每个连接中使用相同的环境设置（如 `ANSI_NULLS`）。这一点很重要，因为查询的 `执行计划` 取决于连接的环境设置。

• 如前面“显式定义对象的所有者”一节所述，在查询中访问对象时显式限定其所有者。

前面这些关于计划缓存的方面在第 16 章中有详细解释。

#### 最小化执行计划的重新编译

为了最小化为查询生成 `执行计划` 的成本，你必须确保缓存中的计划不会因你可控的原因而失效或重新编译。以下推荐的最佳实践可最小化存储过程计划的重新编译：

• 不要在你的 `存储过程` 中交错使用 `DDL` 和 `DML` 语句。你应该将所有 `DDL` 语句放在 `存储过程` 的顶部。

• 在 `存储过程` 中，避免使用在 `存储过程` 外部创建的 `临时表`。



• 对于小型数据集，优先使用表变量而非临时表。

• 不要在存储过程中更改 ANSI SET 选项。

• 如果实在无法避免重编译，那么识别导致重编译的存储过程语句，并通过 `sp_execute_sql` 系统存储过程来执行它。

存储过程重编译的原因及推荐解决方案在第 17 章中有详细解释。

### 采用数据库事务的最佳实践

你为并发设计的查询越有效，查询就能越快完成而不会相互阻塞。在设计查询中的事务时，请考虑以下建议：

• 将事务的范围尽可能缩短。在事务中，仅包含那些为了数据一致性必须一起提交的语句。
