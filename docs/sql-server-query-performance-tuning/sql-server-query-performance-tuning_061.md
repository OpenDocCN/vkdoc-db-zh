# 第 6 章 ■ 查询性能指标

使用脚本创建会话使您能够在大量服务器上实现自动化。使用脚本启动和停止会话意味着您可以通过计划事件（例如通过 SQL Agent）来控制它们。

在第 18 章中，您将学习如何在捕获 SQL 工作负载活动以进行长时间分析时，控制会话的计划。

■ **注意** 本节中所示定义的会话所捕获的时间以微秒为单位存储，而非毫秒。如果未考虑到单位之间的这种差异，可能会引起混淆。您必须基于微秒进行筛选。

## 扩展事件建议

扩展事件在信息收集方式上是一个颠覆性的改变，以至于过去使用跟踪事件时出现的许多问题领域已在很大程度上被消除。您大大减少了对严格限制收集的事件数量或返回字段数量的担忧。但是，正如前面提到的，过度加载收集的事件仍然可能对系统产生负面影响。仍然有几个特定领域需要注意。

*   适当设置最大文件大小。
*   避免使用调试事件。
*   避免使用 `No_Event_Loss`。

我将在以下各节中更详细地讨论这些内容。

### 适当设置最大文件大小

文件的默认值为 1GB。考虑到使用扩展事件可以收集的信息量，这实际上非常小。建议将此数字设置得更高一些，大约在 50GB 到 100GB 之间，以确保您有足够的空间来捕获信息，并且不会因为缓冲区填满而等待文件子系统为您创建文件。这可能导致事件丢失。但这确实取决于您的系统。如果您对预期的输出级别有很好的把握，请根据您的具体环境设置更合适的文件大小。

### 避免调试事件

扩展事件不仅为您提供了一种观察 SQL Server 行为及其内部机制的机制，其能力远超跟踪事件，而且 Microsoft 将其用作 SQL Server 故障排除的一部分。许多事件与调试 SQL Server 相关。这些事件在向导中默认不可用，但您可以通过 T-SQL 命令访问它们，并且有一种方法可以在“会话编辑器”窗口中通过通道选择来启用它们。

除非有 Microsoft 的直接指导，否则请不要使用它们。它们可能会发生变化，并且仅供 Microsoft 内部使用。如果您确实觉得需要实验，则需要密切关注任何包含中断操作的事件。这意味着如果事件触发，它将导致 SQL Server 在触发事件的确切代码行处停止。这意味着您的服务器将完全离线并处于未知状态。如果在生产系统中执行此操作，可能会导致重大中断。它可能导致数据丢失和数据库损坏。
