# 第 2 章 ■ 内存性能分析

### 动态管理对象



要即时获取以前仅在性能监视器中可用的大量数据快照，SQL Server 通过一组动态管理视图（DMV）和动态管理函数（DMF）在内部提供了一些相同的数据以及大量不同的信息，它们被统称为*动态管理对象*（DMO）。这些是捕获系统当前性能快照的极其有用的机制。我将在本书中介绍多个 DMO，但现在我将重点介绍几个对监控服务器性能和建立基线最重要的对象。

`sys.dm_os_performance_counters` 视图在查询中显示 SQL Server 计数器，使你能够立即运用 T-SQL 的全部强大功能来处理数据。例如，这个简单的查询将返回 `Logins/sec`（每秒登录次数）的当前值：

```sql
SELECT
    dopc.cntr_value,
    dopc.cntr_type
FROM sys.dm_os_performance_counters AS dopc
WHERE dopc.object_name = 'SQLServer:General Statistics'
    AND dopc.counter_name = 'Logins/sec';
```

对于我的测试服务器，此查询返回的值为 `200`。对于你的服务器，如果你有命名实例，则需要在 `object_name` 比较中替换为相应的服务器名称。值得注意的是 `cntr_type` 列。此列告诉你正在读取的是什么类型的计数器（微软在 http://bit.ly/1mmcRaN 有相关文档）。例如，前面的计数器返回值 `272696576`，这意味着此计数器是一个平均值。还有表示瞬时快照、自服务器启动以来的累积值等其他类型。了解度量指标代表什么是理解这些指标的重要一环。

有大量 DMO 可用于收集有关服务器的信息。我在这里再介绍一个你会发现自己经常访问的对象，`sys.dm_os_wait_stats`。此 DMV 显示自 SQL Server 上次启动或计数器重置以来，SQL Server 中线程等待各种资源的累积视图。等待时间是在工作完成后记录的，因此这些数字并不反映任何活动线程。识别系统中发生的等待类型是开始识别瓶颈来源的最简单机制之一。你可以按多种方式对数据进行排序；第一个示例使用这个简单的查询查看当前计数最长的等待：

```sql
SELECT TOP (10) dows.*
FROM sys.dm_os_wait_stats AS dows
ORDER BY dows.wait_time_ms DESC;
```

你不仅可以看到特定等待累积的总时间，还可以看到其发生的次数以及某事物必须等待的最长时间。从这里，你可以确定等待类型并开始进行故障排除。最常见的等待类型之一是 I/O。如果你的十大等待类型中出现 `ASYNC_IO_COMPLETION`、`IO_COMPLETION`、`LOGMGR`、`WRITELOG` 或 `PAGEIOLATCH`，你可能遇到了 I/O 争用，现在你便知道从何着手了。关于等待类型的更详细分析以及如何在 SQL Server 中将其用作监控工具，请阅读微软白皮书《SQL Server 2005 Waits and Queues》（http://bit.ly/1e1I38f）。虽然它是为 SQL Server 2005 编写的，但很大程度上仍适用于更新版本的 SQL Server。你总是可以通过 MSDN 支持（http://bit.ly/1hBzLrZ）直接从微软获取关于更冷门等待类型的信息。最后，说到等待类型，Bob Ward 的知识库（收集于 http://bit.ly/1afzfjC）是必读资料，尽管目前未再维护。

## 硬件资源瓶颈

通常，SQL Server 数据库性能受以下硬件资源的压力影响：

*   内存
*   磁盘 I/O
*   处理器
*   网络

超出硬件资源容量的压力会形成瓶颈。要解决系统的整体性能问题，你需要识别这些瓶颈，因为它们限制了整体系统性能。

### 识别瓶颈

资源瓶颈之间通常存在关联。例如，处理器瓶颈可能是过度分页（内存瓶颈）或磁盘速度慢（磁盘瓶颈）的症状。如果系统内存不足，导致过度分页，并且磁盘速度慢，那么最终结果之一将是处理器利用率过高，因为处理器必须花费大量 CPU 周期在内存中换入换出页面，并管理由此产生的大量 I/O 请求。更换更快的处理器可能有点帮助，但这并非最佳的整体解决方案。在这种情况下，增加内存是更合适的解决方案，因为它将减轻对磁盘和处理器的压力。事实上，升级磁盘可能比升级处理器是更好的解决方案。如果可能的话，减少工作负载也会有帮助，当然，调整查询以确保最大效率也是一个选择。

定位瓶颈的最佳方法之一是识别正在等待其他资源完成其操作的资源。你可以使用性能监视器计数器或 DMO（如 `sys.dm_os_wait_stats`）来收集该信息。由资源服务的请求的响应时间包括请求在资源队列中必须等待的时间以及执行请求所花费的时间，因此最终用户的响应时间与系统中的排队量成正比。

识别瓶颈的另一种方法是参考系统的响应时间和容量。例如，到磁盘的吞吐量通常应该接近供应商所建议的磁盘处理能力。因此，测量诸如 `disk sec/transfer`（磁盘传输秒数）之类的信息将表明磁盘是否因负载过重而变慢。

并非所有资源都有显示排队级别的特定计数器，但大多数资源都有表示该资源过度占用的某些计数器。例如，内存没有这样的计数器，但大量的硬页错误代表物理内存的过度占用（硬页错误将在本章后面的“Pages/Sec 和 Page Faults/Sec”部分解释）。其他资源，如处理器和磁盘，有特定的计数器来指示排队级别。例如，计数器 `Page Life Expectancy`（页面寿命）表示页面在缓冲池中未被引用的情况下将停留多长时间。这表明 SQL Server 管理其内存的能力如何，因为较长的寿命意味着缓冲区中的数据片段将保持可用，等待下一次引用。然而，较短的寿命意味着 SQL Server 正在快速地将页面移入移出缓冲区，可能表明存在内存瓶颈。

你很快就会看到在分析每种瓶颈类型时应使用哪些计数器。

### 瓶颈解决方案

一旦识别出瓶颈，你可以通过两种方式解决它们。

*   你可以增加资源容量。
*   你可以减少到达该资源的请求速率。

增加容量通常需要额外的资源，如内存、磁盘、处理器或网络适配器。

你可以通过更有选择地处理资源请求来减少到达速率。例如，当遇到磁盘子系统瓶颈时，你可以选择增加磁盘子系统的容量，或者减少 I/O 请求数量。
