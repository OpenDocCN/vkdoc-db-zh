# 第 9 章 ■ 索引分析

在创建大量覆盖索引之前，请考虑 SQL Server 如何通过索引交叉，高效且自动地为查询动态创建覆盖索引。

覆盖索引在物理上按顺序组织所有索引列的数据。因此，从磁盘 I/O 的角度来看，一个不使用包含列的覆盖索引，对于完全由覆盖索引中的列满足的所有查询而言，就变成了一个聚集索引。如果查询的结果集需要排序输出，那么覆盖索引可以用来在物理上维持与结果集要求相同顺序的列数据——然后它可以像聚集索引一样用于排序输出。如前一个示例所示，对于请求一定范围行和/或排序输出的查询，覆盖索引可以提供比聚集索引更好的性能。包含列不是键的一部分，因此无法像索引的键列那样提供相同的排序优势。

## 建议

为了利用覆盖索引的优势，请注意 `SELECT` 语句中的列列表，只移动你需要的数据。对于覆盖索引，尽可能使用较少的列以保持索引键尺寸较小也是一个好主意。在合适的地方，使用 `INCLUDE` 语句添加列。由于覆盖索引包含了查询中使用的所有列，它往往非常宽，从而增加了覆盖索引的维护成本。你必须在覆盖索引带来的性能提升与其维护成本之间取得平衡。如果索引中所有列的字节数与该表中单个数据行的字节数相比很小，并且你确定利用覆盖索引的查询将频繁执行，那么使用覆盖索引可能是有益的。

**提示：** 覆盖索引也有助于解决阻塞和死锁问题，你将在第 19 章和第 20 章中看到。

### 索引交叉

如果一个表有多个索引，那么 SQL Server 可以使用多个索引来执行查询。SQL Server 可以利用多个索引，根据每个索引选择小的数据子集，然后执行这两个子集的交叉（即，只返回满足所有条件的行）。SQL Server 可以利用表上的多个索引，然后采用连接算法来获取两个子集之间的索引交叉。

在下面的 `SELECT` 语句中，对于 `WHERE` 子句的列，该表在 `SalesPersonID` 列上有一个非聚集索引，但在 `OrderDate` 列上没有索引：

```sql
--SELECT * 在此查询中是故意使用的
SELECT soh.*
FROM Sales.SalesOrderHeader AS soh
WHERE soh.SalesPersonID = 276
AND soh.OrderDate BETWEEN '4/1/2005' AND '7/1/2005';
```

如你所见，优化器没有使用 `SalesPersonID` 列上的非聚集索引。由于也需要 `OrderDate` 列的值，优化器选择了聚集索引来获取所有引用列的值。检索此数据的 I/O 如下：

`表 'SalesOrderHeader'。扫描计数 1，逻辑读取 689`

`CPU 时间 = 16 毫秒，耗时 = 47 毫秒。`

为了提高查询的性能，可以将 `OrderDate` 列添加到 `SalesPersonId` 列上的非聚集索引中，或者将其定义为同一索引的包含列。但在这种实际场景中，修改现有索引时可能需要考虑以下几点：

• 出于各种原因，可能不允许修改现有索引。
• 现有的非聚集索引键可能已经相当宽。
• 使用现有索引的其他查询的成本将受到修改的影响。
