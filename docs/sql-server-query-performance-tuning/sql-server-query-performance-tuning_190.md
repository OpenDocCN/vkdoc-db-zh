# 第 24 章 ■ 数据库性能测试

知道如何识别性能问题以及如何修复它们是很好的技能。但问题在于，你需要能够证明你所做的改进是真正的改进。虽然你能够也应该在调整查询或添加索引前后捕获性能指标，但确保你看到的是可衡量的改进的最佳方法是让你所做的更改发挥作用。测试不仅仅是简单地运行几次查询，然后将其投入生产系统并祈求好运。你需要一种系统的方法，使用在现实环境中针对你的系统运行的全套查询来验证性能改进。`SQL Server` 从 2012 版本开始，通过其 `Distributed Replay` 工具提供了这样的机制。

`Distributed Replay` 与来自 `SQL Profiler` 及其创建的跟踪事件的信息一起工作。

跟踪事件捕获信息的方式与 `Extended Events` 工具有些相似，但跟踪事件是一种较旧（且功能较弱）的在系统中捕获事件的机制。在 `SQL Server` 2012 版本发布之前，你可以使用 `SQL Server` 的 `Profiler` 工具通过服务器端跟踪重放捕获的事件。这虽然可行，但过程极其有限。例如，该工具只能在单台机器上运行，并且它处理重放机制的方式——一个以串行方式运行的单线程过程，而不是现实中发生的情况。

微软为 `SQL Server` 添加了能够从多台机器以并行方式运行的功能。在微软提供一种通过 `Extended Events` 输出使用 `Distributed Replay` 的机制之前，你仍然需要使用跟踪事件来进行部分性能测试。

`Distributed Replay` 并不是一个被广泛采用的工具。大多数人干脆跳过了实施可重复测试的想法。其他人可能会使用一些提供更多功能的第三方工具。我强烈建议你进行某种形式的测试，以确保你的调整工作正在对你的系统产生可准确衡量的积极影响。

本章涵盖以下主题：

• 数据库测试的概念

• 如何创建服务器端跟踪

• 使用 `Distributed Replay` 进行数据库测试

## 数据库性能测试

数据库性能和负载测试的通用方法相当简单。你需要捕获生产系统在正常负载下的调用，然后能够在测试系统上反复重放该负载。

这使你能够直接测量由代码或结构更改引起的性能变化。

不幸的是，在现实世界中实现这一点并不像听起来那么简单。

首先，你不能简单地捕获查询的记录。相反，你必须首先确保你可以将生产数据库恢复到测试系统上的某个时间点。具体来说，你需要能够恢复到你在系统上开始记录事务的确切时刻，因为如果你恢复到任何其他时间点，你可能会拥有不同的数据甚至不同的结构。这将导致重放机制生成错误而不是有用的信息。这意味着，首先，你必须有一个处于 `Full Recovery` 模式的数据库，以便你可以进行定期的完整备份以及日志备份，从而恢复到测试将开始的特定时间点。

一旦你建立了恢复到适当时间点的能力，你将需要配置你的查询捕获机制——在本例中，是由 `Profiler` 生成的服务器端跟踪定义。重放机制将准确定义你需要捕获哪些事件。你需要设置你的捕获过程，使其对你的系统影响尽可能小。

接下来，你必须处理跟踪捕获的大量数据。根据你的系统规模，你可能在短时间内有大量的事务。所有这些数据都必须存储和管理，并且会产生许多文件。

你可以将此过程设置在单台机器上；然而，为了真正看到好处，你将需要设置多台机器来支持 `Distributed Replay` 工具的重放能力。这意味着你需要将这些机器作为你测试过程的一部分。不幸的是，除了 `Enterprise` 版本外，所有版本都只能使用单个客户端，因此在设置测试环境时需要考虑到这一点。

当你将所有这些不同的部分都准备就绪后，就可以开始测试了。当然，这引出了一个新问题：你到底在用数据库测试做什么？

### 可重复的过程

如第 1 章所述，性能调优你的系统是一个迭代过程，你可能需要经历多次才能将性能提升到你所需的位置并保持在那里。因为业务会随时间变化，你的数据分布、应用程序、数据结构以及所有支持它的代码也会随之变化。鉴于这一切，你能为测试做的最重要的事情之一就是创建一个可以反复运行的过程。

你需要创建一个可重复测试过程的主要原因是你不能总是确保本书前面章节中概述的方法在所有情况下都有效。这种疑虑意味着你需要能够验证你所做的更改是否带来了性能的积极改进。

如果没有，你需要能够撤销任何你所做的更改，进行一组新的更改，然后重复测试，迭代地重复这个过程。你可能会发现你需要重复整个调优周期，直到你达到这一轮的目标。

由于这个过程的迭代性质，你绝对需要专注于尽可能地自动化它。这就是 `Distributed Replay` 工具发挥作用的地方。
