# SQL Server 查询性能调优指南

## 书籍介绍与核心观点

尽管微软多年来持续开发 SQL Server，加上众多数据专业人士的辛勤工作，你可能以为基本的性能问题早已成为历史，但事实并非如此。性能常常是人们开发系统时最后才考虑的事情。不幸的是，这意味着当系统投入生产后，它通常会变成最大的问题。你不能仅仅依赖一通告诉你“服务器 Z 上数据库 Y 的存储过程 X 运行**很慢**”的电话。你需要有机制来自行发现这些信息。你也无法基于笼统的“慢”这个词来工作。和什么相比是慢的？上周？上个月？在你的开发系统中运行的方式？而一旦你确认某个东西确实运行缓慢，你需要找出原因。它需要索引吗？它有索引但没有被使用吗？是 CPU、磁盘、内存、用户数量还是数据量的问题？既然你已经识别出问题和原因，你就必须采取行动。怎么做？重写查询？修改 WHERE 子句？当你开始进行性能调优时，向你涌来的问题是无穷无尽的。

本书为你提供了回答这些问题所需的工具。我将向你展示如何设置机制来收集服务器上 SQL Server 实例和数据库的性能指标。我将介绍收集单个`T-SQL`调用数据的更具战术性的方法。在此过程中，我将讨论索引结构、选择和维护；如何最好地编写`T-SQL`代码；如何测试该代码；以及一大堆其他主题。我写这本书的目标之一，是使用类似于你在现实世界中会看到的查询类型的示例来交付所有这些内容。介绍的工具和方法大多可通过 SQL Server 标准版获得，尽管有些仅适用于 SQL Server 企业版。在你可能遇到它们时，都会特别指出。书中几乎所有的调优建议都直接适用于 Microsoft Azure SQL Database（`MASD`），也适用于更“接地气”的 SQL Server 2014。书中大部分调优建议也立即适用于在虚拟机（`VMs`）中运行的服务器。对于`MASD`和`VMs`，主要区别在于你可以收集哪些性能指标以及对这些指标的信任程度。与代码和结构相关的性能解决方案在这两种环境中都立即适用。

有人可能会认为，从 SQL Server 2008 到 2012 再到 2014，许多查询调优的基础知识并没有发生根本性的变化。因此，可能不立即清楚为何需要一本新的、更新的版本。在这些不同的 SQL Server 版本中发生变化的是查询运行的位置、可用于理解系统的指标以及可用于调优查询的工具。本书此次更新的重点，除了添加关于 SQL Server 2014 中可用新功能的信息外，还在于阐明在其他那些环境中可用的信息和响应类型。我借此机会对章节进行了完全的重组和重命名。引入了几个新章节，这使我既能扩展书中材料的范围，又使其更易于理解。

主要的一点是学会如何回答将要向你提出的各种各样的问题。本书为你提供了这样做的工具，并以一种有条不紊的方式回答这些问题，从而消除了当今性能优化中如此常见的许多猜测工作。性能问题并不可怕。有了正确的工具，你可以以一种赢得同行和客户尊重的冷静与可靠来处理性能问题。这将直接促成你和他们的成功。

## 引言

### 本书目标读者

这本书适用于几乎所有对系统性能负责的人。数据库管理员当然是目标读者，因为他们负责设置系统、创建基础设施并长期监控它。开发人员也是，因为还有谁会生成所有那些结构良好且高性能的`T-SQL`代码呢？数据库开发人员，比任何人都更是目标受众，如果仅仅因为这正是我所做的工作的话。

任何有能力编写`T-SQL`、设计表、实现索引或在 SQL Server 系统上操纵服务器设置的人，都将或多或少需要这些信息。

### 本书结构

本书的目的是尽可能多地使用“看起来真实”的查询。为此，我需要一个“真实”的数据库。我本可以创建一个并迫使每个人去寻找下载链接。相反，我选择使用由微软创建的示例数据库，名为`AdventureWorks2012`。这可以通过 CodePlex ([www.codeplex.com/MSFTDBProdSamples](http://www.codeplex.com/MSFTDBProdSamples))获得。我建议手边保留一份还原副本，并在阅读了书中几个主题后重置你的示例数据库。微软会不时更新这些数据库，因此你可能会看到不同的数据集或与本书所列内容不同的某些查询行为。我选择`AdventureWorks2012`并非因为它代表了完美的数据库设计，而是因为它存在一些设计缺陷和数据分布问题，使其比某些完美的测试用例更准确地反映了现实世界。

在某种程度上，本书建立在前面章节所呈现的知识之上。然而，大多数章节在特定主题内呈现的信息是独特的，因此你可以跳进跳出特定的章节。不过，按顺序从第 1 章阅读到第 26 章，你仍然会获得最大的收益。

*   第 1 章“SQL 查询性能调优”，介绍了性能调优的迭代过程。你将初步了解建立性能基线、识别瓶颈、解决问题和量化改进。
*   第 2 章“内存性能分析”，使用性能监视器指标和动态管理对象作为收集系统上内存信息的机制，开始这一过程。
*   第 3 章“磁盘性能分析”，继续探索系统瓶颈，专门介绍如何收集磁盘性能指标。你将再次使用性能监视器和动态管理对象，并增加一些额外的`T-SQL`查询。
*   第 4 章“CPU 性能分析”，以 CPU 结束系统瓶颈的讨论。我还将介绍一些网络监控，尽管这在 SQL Server 中是一个相当罕见的问题，而且通常 DBA 或开发人员对此无能为力。使用的工具与前面章节相同。
*   第 5 章“创建基线”，利用前三章的信息来定义基线。基线代表系统中的一个已知点，你可以通过比较它来了解系统性能随时间的变化情况。
*   第 6 章“查询性能指标”，定义了“深入观察”并查看系统上正在运行何种查询的最佳方法。它详细介绍了新的扩展事件（`Extended Events`）工具。本章首次确定了用于监控查询的几个最有用的动态管理视图和函数。
