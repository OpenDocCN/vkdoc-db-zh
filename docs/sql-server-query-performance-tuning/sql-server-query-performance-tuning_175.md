# 第 21 章 ■ 死锁的原因与解决方案

你还可以通过避免对其中一个争用资源发起锁请求来解决死锁。当资源仅用于读取数据时，可以采用此方法。修改资源时，系统总会获取该资源的排他锁（X 锁）以维护资源的一致性；因此，在死锁情况下，识别那些只读的资源访问，并尽可能通过使用脏读功能来避免它们对应的锁请求。你可以使用以下技术来避免对争用资源的锁请求：

*   实现行版本控制。
*   降低隔离级别。
*   使用锁提示。

#### 实现行版本控制

与其试图通过更严格的锁定方案来阻止对资源的访问，不如通过 `READ_COMMITTED_SNAPSHOT` 隔离级别或 `SNAPSHOT` 隔离级别来实现行版本控制。行版本控制隔离级别用于减少阻塞，如第 20 章所述。由于它们减少了阻塞（这是死锁的根本原因），因此也有助于解决死锁问题。通过以下 T-SQL 引入 `READ_COMMITTED_SNAPSHOT`，你可以在 `tempdb` 中获得行的版本，从而有可能消除前述死锁场景中由锁引起的争用：

```sql
ALTER DATABASE AdventureWorks2012
SET READ_COMMITTED_SNAPSHOT ON;
```

这将允许任何必要的读取操作而不会引起锁争用，因为读取操作针对的是数据的另一个版本。行版本控制会带来开销，尤其是在 `tempdb` 中，并且在从多个资源（而不仅仅是查询中使用的表或索引）整合数据时。但是，增加的 `tempdb` 开销与减少死锁、提高并发性所带来的好处之间的权衡可能是值得的。

#### 降低隔离级别

有时，`SELECT` 语句请求的（S）锁会导致循环阻塞的形成。你可以通过将包含 `SELECT` 语句的事务的隔离级别降低到 `READ UNCOMMITTED` 来避免这种类型的循环阻塞。这将允许 `SELECT` 语句在不请求（S）锁的情况下读取数据，从而避免循环阻塞。然而，读取未提交数据会带来一个严重问题，即向客户端返回错误数据。你需要在走投无路的情况下，才考虑将此作为消除死锁的方法。

同时，检查连接是否将自己设置为 `SERIALIZABLE`。有时在线连接字符串生成器会包含此选项，开发者可能会完全无意中使用它。`MSDTC` 默认使用可序列化隔离级别，但可以更改。

#### 使用锁提示

我绝对不推荐这种方法。但是，你可以使用以下锁提示来解决前述技术中呈现的死锁：

*   `NOLOCK`
*   `READUNCOMMITTED`

与 `READ UNCOMMITTED` 隔离级别类似，`NOLOCK` 或 `READUNCOMMITTED` 锁提示将避免给定会话请求的（S）锁，从而防止循环阻塞的形成。

锁提示的作用是在查询级别，并且仅限于应用它的表（及其索引）。

`NOLOCK` 和 `READUNCOMMITTED` 锁提示仅允许在 `SELECT` 语句以及 `INSERT`、`DELETE` 和 `UPDATE` 语句的数据选择部分中使用。

最小化锁争用的解决技术会带来脏读的副作用，这可能并非在所有事务中都可接受。脏读可能涉及由于页面拆分和页面重排而导致的行缺失或额外行。因此，仅在数据质量要求不高的情况下使用这些解决技术。

## 小结

如本章所述，死锁是进程间冲突阻塞的结果，并通过错误号 `1205` 报告给应用程序。你可以通过使用各种资源收集死锁信息来分析死锁的原因，但扩展事件 `Xml_deadlock_report` 可能是最好的选择。

你可以使用多种技术来避免死锁；适用哪种技术取决于参与会话执行的查询类型、在相关资源上持有和请求的锁，以及管理所需隔离级别的业务规则。通常，你可以通过重新配置索引和事务隔离级别来解决死锁。然而，有时你可能需要重新设计应用程序或在发生死锁时自动重新执行事务。请记住，从本质上讲，死锁是一个性能问题，任何能够使查询运行更快的措施都有助于缓解（即使不能完全消除）查询中的死锁。
