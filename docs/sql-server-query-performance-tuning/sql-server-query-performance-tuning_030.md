# 第 1 章 ■ SQL 查询性能调优

## tempdb 使用过度或配置不当

任何 SQL Server 实例只有一个 `tempdb`。由于临时存储（例如涉及用户对象的操作，如临时表和表变量）、系统对象（如游标或用于联接的哈希表）以及包括排序和行版本控制在内的操作都使用 `tempdb` 数据库，因此 `tempdb` 很容易成为性能瓶颈。

所有这些选项以及其他因素都会导致 `tempdb` 内出现空间、I/O 和争用问题。第 3 章将介绍一些有助于解决此问题的配置选项，其他章节则会根据所处理问题的性质介绍其他相关选项。

## 小结

在这个介绍性章节中，你已经了解到 SQL Server 性能调优是一个迭代过程，包括识别性能瓶颈、排查其原因、应用不同的解决方案、量化性能改进，然后重复这些步骤，直到达到所需的性能水平。

为了辅助这个过程，你应该创建一个系统基线，以便与你的修改进行比较。在整个性能调优过程中，你需要客观看待你想要执行的调优工作量——你总是可以让查询运行得再快一点，但付出的努力是否值得其成本？最后，由于性能取决于用户活动模式和数据，因此你必须定期重新评估数据库服务器的性能。

为了从 SQL Server 数据库系统中获得最佳性能，理解数据库应用程序对服务器造成的压力至关重要。在接下来的两章中，我将讨论如何分析这些压力，既从较高的系统层面，也从较低的 SQL Server 活动层面。然后，我会展示如何将两者结合起来。

在本书的其余部分，你将深入探讨本章前面提到的最大的 SQL Server 性能杀手。你将了解到，如果使用不当，这些单独因素会如何影响性能，以及如何解决或避免这些陷阱。
