内存与磁盘 I/O 密切相关。即使你认为遇到的问题是直接与内存相关的，你也应该收集 I/O 指标，以便理解系统在这两种资源之间的交互行为。

现在，我将引导你了解这些计数器，以便更好地理解其可能用途。

### 可用字节数
`可用字节数`计数器表示系统中可用的物理内存。你也可以查看 `可用千字节数` 和 `可用兆字节数`，它们提供相同的数据但粒度较粗。为了获得良好性能，此计数器的值不应过低。如果 SQL Server 配置为动态内存使用，则此值将由调用 Windows API 控制，该 API 决定何时以及释放多少内存。如果此值长时间处于极低状态，并且 SQL Server 的内存使用量没有变化，则表明服务器正承受严重的内存压力。

### 页面数/秒 和 页面故障/秒
要理解 `页面数/秒` 和 `页面故障/秒` 计数器的重要性，你首先需要了解页面故障。

*页面故障* 发生在一个进程需要的代码或数据不在其*工作集*（其在物理内存中的空间）中时。

它可能导致软页面故障或硬页面故障。如果故障页面在物理内存的其他地方找到，则称为*软页面故障*。当进程需要的代码或数据不在其工作集或物理内存的其他地方，必须从磁盘检索时，就会发生*硬页面故障*。

磁盘访问的速度对于机械硬盘来说是毫秒级，对于固态硬盘（SSD）可能低至 0.1 毫秒，而内存访问的速度是纳秒级。磁盘访问与内存访问之间的这种巨大速度差异，使得硬页面故障的影响与软页面故障相比非常显著。

`页面数/秒` 计数器表示每秒为解决硬页面故障而从磁盘读取或写入磁盘的页面数。`页面故障/秒` 性能计数器表示系统处理的每秒总页面故障数——软页面故障加上硬页面故障。这些主要是负载的度量指标，并非性能问题的直接指标。

由 `页面数/秒` 指示的硬页面故障不应持续高于正常水平。对于什么数值表示问题，没有硬性规定，因为这些数字会因系统配置（内存量、类型以及磁盘访问速度）的不同而有很大差异。

如果 `页面数/秒` 计数器值很高，你可以将其拆分为 `输入页面数/秒` 和 `输出页面数/秒` 来分析。

- *输入页面数/秒*：应用程序只会等待输入页面，而不会等待输出页面。
- *输出页面数/秒*：页面输出会给系统带来压力，但应用程序通常不会直接感受到这种压力。页面输出通常由应用程序需要写回磁盘的脏页表示。`输出页面数/秒` 只有在磁盘负载成为问题时才需要关注。

此外，在 `页面数/秒` 值很高的情况下，可以检查 `进程:页面故障/秒` 以找出是哪个进程导致了过多的分页。`进程` 对象是提供系统上运行进程性能数据的系统组件，每个进程由其对应的实例名称表示。

例如，SQL Server 进程由 `Process` 对象的 `sqlservr` 实例表示。此计数器的高数值通常意义不大，除非 `页面数/秒` 也很高。`页面故障/秒` 的值在正常应用行为中可能变化很大，从每秒 0 到 1,000 都是可接受的。因此，建立基线对于确定预期正常行为至关重要。

### 页面文件使用率百分比 和 页面文件使用率百分比
Windows 系统中的内存并非全部是物理机的物理内存。Windows 会将那些未立即激活的内存交换到页面文件中，从而在物理内存空间中移入移出。这些计数器用于了解在你的系统上这种交换发生的频率。作为系统性能的一般度量，这些计数器仅适用于 Windows 操作系统，而不直接适用于 SQL Server。然而，虚拟内存不足的影响会波及 SQL Server。收集这些计数器是为了了解 SQL Server 面临的内存压力是内部的还是外部的。如果是外部内存压力，你需要进入 Windows 操作系统来确定可能的问题所在。

### 缓冲区缓存命中率
*缓冲区缓存* 是数据页被读入的缓冲页池，通常是 SQL Server 内存池中最大的部分。此计数器的值应尽可能高，特别是对于 OLTP 系统，这类系统通常具有相当规范的数据访问模式，这与仓库或报表系统不同。对于大多数生产服务器，此计数器的值达到 99% 或更高是极为常见的。`缓冲区缓存命中率` 值低表明很少有请求可以由缓冲区缓存满足，其余请求由磁盘满足。

当发生这种情况时，要么 SQL Server 仍在预热阶段，要么缓冲区缓存的内存需求超过了可用于其增长的最大可用内存。如果缓存命中率持续较低，你可能需要考虑为系统增加更多内存，或者通过使用良好的索引和其他查询调优机制来减少内存需求，除非你处理的是包含大量即席查询的报表系统。在处理报表系统时，持续观察到缓存命中率极低是可能的。

这使得缓冲区缓存命中率成为理解系统行为方面的有趣指标，但它本身并不足以表明潜在的性能问题。虽然这个数字代表了系统内一种有趣的行为，但它不是精确问题的良好度量标准，而是展示了一种行为类型。
