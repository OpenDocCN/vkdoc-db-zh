# 磁盘性能分析与瓶颈解决

## I/O 等待与瓶颈识别

当你查看各种 I/O 锁存操作时，会发现它们是导致等待发生的原因。与 `sys.dm_io_virtual_status` 类似，你无法从这个动态管理对象（DMO）中获得具体的查询信息，但它确实可以识别 I/O 是否存在瓶颈。像许多性能计数器一样，你不能只看单个数值。你需要将当前值与基线值进行比较，才能判断当前的状况。

前面展示的 `WHERE` 子句使用了 `PAGEIOLATCH%`，但你还应该查找与其他 I/O 进程相关的等待，例如 `WRITELOG`、`LOGBUFFER` 和 `ASYNC_IO_COMPLETION`。

运行此查询后，你将获得已发生等待的次数以及总等待时间的汇总。你还会得到这些等待的最大值，从而了解最长的等待时间是多久，因为有可能单个等待就导致了大部分的等待时间。

## 解决磁盘瓶颈

一些常见的解决磁盘瓶颈的方法如下：

- 优化应用程序工作负载
- 使用更快的 I/O 路径
- 使用 RAID 阵列
- 使用 SAN 系统
- 使用固态硬盘（SSD）
- 正确对齐磁盘
- 使用带电池备份的控制器缓存
- 增加系统内存
- 创建多个文件和文件组
- 将日志文件移动到独立的物理驱动器
- 使用分区表

接下来，我将逐一介绍这些解决方案。

### 优化应用程序工作负载

我无法再强调优化应用程序工作负载在解决性能问题时的重要性。

具有最高读写次数的查询将会导致大量的磁盘 I/O。本书后续部分将更详细地介绍优化这些查询的策略。

### 使用更快的 I/O 路径

最高效的解决方案之一，也是你在任何可能的时候都应该采用的，就是使用具有更快磁盘传输速度的驱动器、控制器和其他架构。然而，你不应该仅仅在未经进一步调查的情况下升级磁盘驱动器；你需要找出导致磁盘压力的原因。

### 使用 RAID 阵列

获得磁盘 I/O 并行性的一种方法是创建一个单一的驱动器池来服务所有 SQL Server 数据库文件（事务日志文件除外）。这个池可以是一个单独的 RAID 阵列，在 Windows Server 2012 R2 中表现为一个物理磁盘驱动器。驱动器池的有效性取决于 RAID 磁盘的配置。

在所有可用的 RAID 配置中，最常用的 RAID 配置如下（也如图 3-1 所示）：

- **RAID 0**：带区化，无容错
- **RAID 1**：镜像
- **RAID 5**：带奇偶校验的带区化
- **RAID 1+0**：带镜像的带区化

### RAID 0

由于此 RAID 配置没有容错能力，你只能在数据可靠性不是问题的场景下使用它。阵列中任何磁盘的故障都会导致磁盘子系统中的数据完全丢失。因此，你不应将其用于构成数据库的任何数据文件或事务日志文件，可能的例外是名为 `tempdb` 的系统临时数据库。RAID 0 中每个磁盘的 I/O 数由以下等式表示：

```
每个磁盘的 I/O 数 = (读取次数 + 写入次数) / 阵列中的磁盘数
```

在此等式中，`Reads` 是对磁盘子系统的读取请求数，`Writes` 是对磁盘子系统的写入请求数。

### RAID 1

RAID 1 通过将数据磁盘镜像到另一个独立的磁盘上，为关键数据提供高容错能力。它可用于完整数据能够容纳在一个磁盘上的场景。用户数据库的事务日志文件、操作系统文件以及 SQL Server 系统数据库（`master` 和 `msdb`）通常足够小，适合使用 RAID 1。

RAID 1 中每个磁盘的 I/O 数由以下等式表示：

```
每个磁盘的 I/O 数 = (读取次数 + 2 * 写入次数) / 2
```

### RAID 5

RAID 5 在许多情况下是一个可接受的选择。它通过有效地仅使用一个额外磁盘来保存其他磁盘数据的计算奇偶校验信息，提供合理的容错能力，如图 3-1 所示。当 RAID 5 配置中出现磁盘故障时，I/O 性能会变得非常糟糕，尽管系统在故障驱动器下运行时仍然可用。

任何写入操作占总磁盘请求超过 10% 的数据都不是 RAID 5 的良好候选对象。

因此，在只读卷或磁盘写入百分比较低的卷上使用 RAID 5。

RAID 5 中每个磁盘的 I/O 数由以下等式表示：

```
每个磁盘的 I/O 数 = (读取次数 + 4 * 写入次数) / 阵列中的磁盘数
```

如这个等式所示，对 RAID 5 磁盘子系统的写入操作被放大了四倍。对于每个传入的写入请求，磁盘子系统上有以下四个对应的 I/O 请求：

- 一次读取 I/O，从要修改内容的数据磁盘读取现有数据
- 一次读取 I/O，从相应的奇偶校验磁盘读取现有奇偶校验信息
- 一次写入 I/O，将新数据写入要修改内容的数据磁盘
- 一次写入 I/O，将新的奇偶校验信息写入相应的奇偶校验磁盘

因此，每个写入请求的四个 I/O 包含两次读取 I/O 和两次写入 I/O。

在 OLTP 数据库中，所有数据修改都会作为数据库事务的一部分立即写入事务日志文件，但数据文件本身的内容与事务日志文件内容的同步是以批处理操作异步进行的。此操作由 SQL Server 的内部进程（称为 `checkpoint process`）管理。此操作的频率可以通过使用 SQL Server 的 `recovery interval (min)` 配置参数来控制。请记住，检查点的时机可以通过 SQL Server 2012 中引入的间接检查点来控制。

由于高事务性 OLTP 数据库中事务日志文件的连续写入操作，将事务日志文件放在 RAID 5 阵列上会降低阵列的性能。虽然在可能的情况下不应将事务日志文件放在 RAID 5 阵列上，但数据文件可以放在 RAID 5 上，因为对数据文件的写入操作是间歇性的，并且为了提高写入操作的效率而分批进行。

### RAID 6

RAID 6 是在 RAID 5 基础上增加的一层。在 RAID 5 的存储中添加了一个额外的奇偶校验块。这不会对读取产生任何负面影响。这意味着对于读取，其性能与 RAID 5 相同。对于额外的写入操作会有额外的开销，但并不算大。添加这个额外的奇偶校验块是因为如今的 RAID 阵列变得如此之大，数据丢失是不可避免的。这个额外的奇偶校验块充当了检查机制，以更好地确保你的数据安全。
