# 第 8 章 ■ 索引架构与行为

#### 何时使用聚集索引

在某些情况下，使用聚集索引是有益的。我将在接下来的章节中讨论这些情况。

##### 直接访问数据

由于所有数据都存储在聚集索引的叶级页中，因此任何时候访问聚集数据时，数据都是立即可用的。聚集索引的一个用途是支持最常用的数据访问路径。任何对聚集索引的访问都不需要额外的读取来检索数据，这意味着针对聚集索引的查找或扫描操作不需要额外的读取来获取数据。这可能是 Microsoft 默认将主键设为聚集索引的另一个原因。由于主键通常是访问表中数据最可能的方式，因此它很适合用作聚集索引。

请记住，主键作为聚集索引是默认行为，但不一定是访问数据最常见的路径。这可能是通过外键约束、表中的备用键或其他列实现的。在规划和设计聚集索引时，要同时考虑存储和访问，这样应该就没问题了。

只有当你访问表中相当一部分数据时，聚集索引作为数据的主要路径才效果良好。另一方面，如果你只访问数据的小子集，使用非聚集覆盖索引可能更合适。此外，你还需要考虑定义数据访问路径的列数和类型。由于聚集索引的键成为非聚集索引的指针，过宽的聚集键会严重影响非聚集索引的性能和存储。

##### 检索预排序数据

当数据检索需要排序时，聚集索引特别高效（覆盖非聚集索引对此也有用）。如果你在可能需要排序的列（或多列）上创建聚集索引，那么行将按该顺序物理存储，从而消除了检索数据后进行排序的开销。

让我们通过实例来看看这一点。创建一个测试表，如下所示：

```sql
IF (SELECT OBJECT_ID('od')) IS NOT NULL
    DROP TABLE dbo.od ;
GO

SELECT pod.*
INTO dbo.od
FROM Purchasing.PurchaseOrderDetail AS pod;
```

新表 `od` 仅包含数据创建而成。它没有任何索引。你可以通过执行以下命令来验证表上的索引，该命令不返回任何内容：

```sql
EXEC sp_helpindex 'dbo.od';
```

为了理解聚集索引的用途，获取一个按特定列排序的大范围行集。

```sql
SELECT od.*
FROM dbo.od
WHERE od.ProductID BETWEEN 500 AND 510
ORDER BY od.ProductID;
```

你可以从 `STATISTICS IO` 输出中获取执行此查询（在没有任何索引的情况下）的成本。

```
Table 'od'. Scan count 1, logical reads 78
CPU time = 15 ms, elapsed time = 388 ms.
```

为了改进此查询的性能，你应该在 `WHERE` 子句的列上创建索引。此查询既需要一个行范围，也需要排序的输出。此查询的结果集要求符合聚集索引的建议。因此，按如下所示创建聚集索引，并重新检查查询成本。

```sql
CREATE CLUSTERED INDEX i1 ON od(ProductID);
```

当你再次运行查询时，查询（带有聚集索引）的结果成本如下：

```
Table 'od'. Scan count 1, logical reads 8
CPU time = 16 ms, elapsed time = 373 ms.
```

创建聚集索引减少了逻辑读取次数，因此应该有助于提升查询性能。

另一方面，如果你在候选列上创建非聚集索引（而不是聚集索引），则查询性能可能会受到不利影响。让我们验证一下在这种情况下非聚集索引的影响。

```sql
DROP INDEX od.i1;
CREATE NONCLUSTERED INDEX i1 on dbo.od(ProductID);
```

查询（带有非聚集索引）的结果成本如下：

```
Table 'od'. Scan count 1, logical reads 78
CPU time = 0 ms, elapsed time = 434 ms.
```

非聚集索引甚至没有在最终的执行计划中被直接使用。相反，你会得到一个表扫描，但在此新计划中，对数据进行排序的估计成本与原始表扫描不同，因为索引为优化器提供了额外的选择性来估计成本，即使索引未被使用。完成后请删除测试表。

```sql
DROP TABLE dbo.od;
```

#### 聚集索引的不良设计实践

在某些情况下，最好不要使用聚集索引。我将在接下来的章节中讨论这些情况。

##### 经常更新的列

如果聚集索引列经常更新，这将导致所有非聚集索引的行定位器相应地更新，从而显著增加相关操作查询的成本。这还会影响数据库并发性，因为在更新期间会阻塞引用同一表部分和非聚集索引的所有其他查询。因此，避免在高度可更新的列上创建聚集索引。

■ **注意** 第 20 章更深入地介绍了阻塞。

为了理解仅修改聚集键列的 `UPDATE` 语句的成本是如何因表上的非聚集索引而增加的，请考虑以下示例。`Sales.SpecialOfferProduct` 表在主键上有一个复合聚集索引，该主键也是来自两个不同表的外键；这是一个经典的多对多连接。在此示例中，我使用以下语句更新了两个列中的一个（注意使用事务来保持测试数据完整）：

```sql
BEGIN TRAN
SET STATISTICS IO ON;

UPDATE Sales.SpecialOfferProduct
SET ProductID = 345
WHERE SpecialOfferID = 1
  AND ProductID = 720;

SET STATISTICS IO OFF;
ROLLBACK TRAN
```

`STATISTICS IO` 输出显示了所需的读取次数。

```
Table 'Product'. Scan count 0, logical reads 2
Table 'SalesOrderDetail'. Scan count 1, logical reads 1246
Table 'SpecialOfferProduct'. Scan count 0, logical reads 15
```

如果你向表中添加一个非聚集索引，你会看到读取次数增加，如下所示：

```sql
CREATE NONCLUSTERED INDEX ixTest
ON Sales.SpecialOfferProduct (ModifiedDate);
```

当你再次运行相同的查询时，`SpecialOfferProduct` 表的 `STATISTICS IO` 输出发生了变化。

```
Table 'Product'. Scan count 0, logical reads 2
Table 'SalesOrderDetail'. Scan count 1, logical reads 1246
Table 'SpecialOfferProduct'. Scan count 0, logical reads 19
```

由于聚集索引更新而导致的读取次数随着非聚集索引的增加而增加。请确保删除该索引。

```sql
DROP INDEX Sales.SpecialOfferProduct.ixTest;
```

##### 宽键

由于所有非聚集索引都将其聚集键作为行定位器，出于性能原因，你应该避免在非常宽的列（或多列）或太多列上创建聚集索引。如前一节所述，聚集索引应尽可能窄。

### 非聚集索引

这也意味着在插入每一行时，获取唯一值会增加一点处理开销。基于所有这些原因，在可能的情况下使聚集索引具有唯一性是有意义的。这就是为什么主键的默认行为是使其成为聚集索引的一个重要原因。

### 非聚集索引

非聚集索引不会影响数据在表页中的顺序，因为非聚集索引的叶级页与表的数据页是分离的。需要通过一个指针（行定位器）来从非聚集索引中的索引行导航到数据行，无论该数据是存储在聚集索引还是堆中。正如你在前面的“聚集索引”一节中所了解到的，行定位器的结构取决于数据页是存储在堆中还是聚集索引中。对于堆，行定位器是指向数据行 `RID` 的指针；对于有聚集索引的表，行定位器是聚集索引键。

#### 非聚集索引维护

非聚集索引的行定位器值在聚集索引行发生物理重定位时，仍然保持相同的聚集索引值。

在作为堆的表中（即没有聚集索引），为了优化此维护成本，SQL Server 在页拆分后，会向旧数据页添加一个指向新数据页的指针，而不是更新所有相关非聚集索引的行定位器。虽然这降低了非聚集索引的维护成本，但它增加了从非聚集索引行导航到堆内数据行的成本，因为在旧数据页和新数据页之间增加了一个额外的链接。因此，使用聚集索引作为行定位器可以减少与非聚集索引相关的这种开销。

#### 定义查找操作

当查询请求的列不属于优化器选择的非聚集索引的一部分时，就需要进行查找。

当查询针对聚集索引时，这可能是一个键查找；当针对堆执行时，则是一个 `RID` 查找。过去，这些查找的通用术语源自旧的定义名称，即 `书签查找`。由于自 SQL Server 2000 以后人们在执行计划中不再看到该短语，因此该术语的使用越来越少。现在你只需将其称为查找，然后定义其类型、键或 `RID`。查找通过遵循索引行中的行定位器值，从表中获取相应的数据行，这需要在数据页上进行一次逻辑读取，以及在索引页上的一次逻辑读取，以及一个联接操作将数据组合到一个共同的输出中。但是，如果查询所需的所有列在索引本身中都可用，则无需访问数据页。这被称为 `覆盖索引`。

这些查找是为什么大型结果集更适合使用聚集索引的原因。聚集索引不需要查找，因为聚集索引的叶级页和数据页是相同的。

#### 非聚集索引建议

由于一个表只能有一个聚集索引，你可以利用多个非聚集索引的灵活性来帮助提高性能。我将在以下章节中解释决定是否使用非聚集索引的因素。

#### 何时使用非聚集索引

当你只想从大型表中检索单独的少量行和列时，非聚集索引最有用。随着要检索的列数增加，拥有 `覆盖索引` 的可能性会降低。然后，如果你还要检索大量行，则任何查找的开销成本都会按比例上升。要从表中检索单独的少量行，索引列应具有高选择性。

此外，会有一些索引需求不适合聚集索引，如“聚集索引”部分所述。

*   频繁可更新的列
*   宽键

在这些情况下，你可以使用非聚集索引，因为与聚集索引不同，它不会影响表中的其他索引。在频繁可更新的列上创建非聚集索引的成本不如在该列上创建聚集索引高。对非聚集索引的 `UPDATE` 操作仅限于基表和非聚集索引本身。它不会影响表上的任何其他非聚集索引。类似地，在宽列（或列集）上的非聚集索引不会增加任何其他索引的大小，而聚集索引则会。但是，即使在高度可更新的列或宽列（或列集）上创建非聚集索引时，也要保持谨慎，因为正如本章前面所述，这会增加操作查询的成本。

■ **提示** 非聚集索引也有助于解决阻塞和死锁问题。我将在第 20 和 21 章更深入地介绍这一点。

#### 何时不使用非聚集索引

非聚集索引不适用于检索大量行的查询。此类查询更适合使用聚集索引，后者不需要单独的查找来检索数据行。因为除了在非聚集索引页上的逻辑读取外，查找还需要额外的逻辑读取来访问数据页，所以对于大量行（例如在需要一个接一个查找的循环联接中），使用非聚集索引的查询成本会显著增加。SQL Server 查询优化器会考虑此成本，并可能在检索大型结果集时放弃使用非聚集索引。

如果你的要求是从表中检索大型结果集，那么在筛选条件（或联接条件）列上拥有非聚集索引可能没有用，除非你使用一种特殊类型的非聚集索引，称为 `覆盖索引`。我将在第 9 章详细介绍这种索引类型。

### 聚集索引与非聚集索引

在选择聚集索引和非聚集索引之间的主要考虑因素如下：

*   要检索的行数
*   数据排序要求
*   索引键宽度
*   列更新频率
*   查找成本
*   任何磁盘热点

#### 聚集索引相对于非聚集索引的优势

在决定为无索引的表选择索引类型时，聚集索引通常是首选。

因为索引页和数据页是相同的，所以聚集索引不必像非覆盖非聚集索引那样从索引行跳转到基行。

为了理解在这些情况下，即使是检索少量行，聚集索引如何能优于非聚集索引，可以创建一个测试表，其中一列具有高选择性。

```sql
IF (SELECT OBJECT_ID('dbo.Test1')) IS NOT NULL
    DROP TABLE dbo.Test1;
GO

CREATE TABLE dbo.Test1 (C1 INT, C2 INT);

WITH Nums
AS (SELECT TOP (10000)
        ROW_NUMBER() OVER (ORDER BY (SELECT 1)) AS n
    FROM master.sys.all_columns AS ac1
    CROSS JOIN master.sys.all_columns AS ac2
)
INSERT INTO dbo.Test1 (C1, C2)
SELECT n, n
FROM Nums;
```

以下 `SELECT` 语句仅从表中获取 10,000 行中的 1 行：

```sql
SELECT t.C1, t.C2
FROM dbo.Test1 AS t
WHERE C1 = 1000;
```

这将产生如图 8-21 所示的图形化执行计划，以及 `SET STATISTICS IO` 和 `STATISTICS TIME` 的输出如下：

```
Table 'Test1'. Scan count 1, logical reads 39
CPU time = 0 ms, elapsed time = 1 ms.
```

考虑到前面 `SELECT` 语句检索的结果集很小，在 `c1` 上创建非聚集索引可能是一个不错的选择。

```sql
CREATE NONCLUSTERED INDEX incl ON dbo.Test1(C1);
```
