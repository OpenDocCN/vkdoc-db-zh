# 第 26 章 ■ SQL Server 优化检查清单

### 保持统计信息最新

数据库统计信息对性能的影响在第 12 章中有详细解释；不过，这个简短的清单将作为保持统计信息最新的快速参考：

-   允许 SQL Server 使用配置参数 `AUT0_CREATE_STATISTICS` 和 `AUTO_UPDATE_STATISTICS` 的默认设置，自动维护表中数据分布的统计信息。
-   作为一种主动措施，您可以在系统需要且支持的情况下，以编程方式定期更新每个数据库对象的统计信息。这种做法可以在自动更新统计信息功能未能提供满意结果时，部分保护您的数据库免受过时统计信息的影响。在第 12 章中，我将说明如何设置一个 SQL Server 作业来以编程方式定期更新统计信息。
-   请记住，您还可以异步方式更新统计信息。这减少了更新统计信息时对其的争用；因此，如果您的系统访问量相当稳定，可以使用此方法更频繁地更新统计信息。

**注意** 请确保在索引碎片整理作业完成之前安排统计信息更新作业，如本章后面所述。

### 保持最低限度的索引碎片

以下最佳实践将帮助您保持最低限度的索引碎片：

-   在非高峰时段定期对数据库进行碎片整理。
-   定期确定索引的碎片级别；然后，根据碎片情况，要么重建索引，要么执行第 13 章中概述的碎片整理查询来整理索引碎片。
-   请记住，非常小的表根本不需要进行碎片整理。
-   对于超大型数据库，整理索引碎片时可能适用不同的规则。
-   如果您的索引仅用于单次查找操作，那么碎片不会影响性能。

### 避免使用 AUTO_CLOSE 或 AUTO_SHRINK 等数据库函数

`AUTO_CLOSE` 会在最后一个用户连接关闭时干净地关闭数据库并释放其所有资源。这意味着缓存中的所有数据和查询都会被自动清空。当下一个连接到来时，不仅数据库需要重启，所有数据也必须重新加载到缓存中。此外，存储过程和其他查询也需要重新编译。这对大多数数据库系统来说是一项极其昂贵的操作。请将 `AUT0_CL0SE` 保留为默认的 OFF 设置。

`AUTO_SHRINK` 会定期收缩数据库的大小。它可以收缩数据文件，并且在简单恢复模式下，也可以收缩日志文件。在此过程中，它可能会阻塞其他进程，严重减慢系统速度。通常，在启用了 `AUTO_SHRINK` 的系统上，文件增长也被设置为自动发生，因此当数据或日志文件需要增长时，您的系统会再次变慢。此外，您会看到物理文件存储在操作系统级别变得碎片化，严重影响性能。请将数据库大小设置为合适的大小，并监控其增长需求。如果必须自动增长，请按物理增量增长，而不是按百分比增长。

## 数据库备份

数据库备份是一个广泛的话题，在这本查询优化书籍中无法得到应有的重视。尽管如此，我建议当涉及到数据库性能时，请关注数据库备份过程的以下几个方面：

-   差异备份和事务日志备份频率
-   备份分布
-   备份压缩

接下来的章节将详细阐述这些建议。

### 差异备份和事务日志备份频率

对于 OLTP 数据库，定期备份数据库是强制性的，以便在发生故障时，可以将数据库还原到不同的服务器上。对于大型数据库，完整的数据库备份通常需要很长时间，因此不能频繁执行完整备份。因此，完整备份在较长的时间间隔执行，而在两个连续完整备份之间，安排更频繁的差异备份和事务日志备份。通过设置频繁的差异备份和事务日志备份，如果数据库完全故障，可以将数据库恢复到某个时间点。

差异备份可用于减少完整备份的开销，因为它只备份自上次完整备份以来更改的数据。因为这可能快得多，所以对生产系统造成的减速也更少。每种情况都是独特的，因此您需要找到最适合您的方法。作为一般规则，我建议每周进行一次完整备份，然后每天进行差异备份。之后，您可以确定事务日志备份的需求。

频繁备份事务日志会给服务器增加少量开销，尤其是在高峰时段。对于大多数企业来说，可接受的数据丢失量（以时间衡量）通常优先于节省日志磁盘空间或提供理想的数据库性能。因此，在安排事务日志备份时，必须考虑到可接受的数据丢失量，而不是随意将备份计划设置为较短的时间间隔。

### 备份计划分布

当需要备份多个数据库时，必须确保所有完整备份不会安排在同一时间，以免硬件资源同时受到冲击。如果备份过程涉及将数据库备份到中央 SAN 磁盘阵列，那么来自所有数据库服务器的完整备份必须分布在备份时间窗口内，以便中央备份基础设施不会同时受到过多备份请求的冲击。同时用大量备份请求淹没中央基础设施，会迫使基础设施的组件花费大量资源来管理过多的请求。这种资源管理不善会显著增加备份持续时间，导致完整备份在高峰时段继续进行，从而影响用户请求的性能。

为了最大限度地减少完整备份过程对数据库性能的影响，您必须首先确定可以安排完整备份的非高峰时段，然后将完整备份分布在该非高峰时段内，如下所示：

1.  确定必须备份的数据库数量。
2.  根据数据库对业务的重要性进行优先级排序。
3.  确定可以安排完整数据库备份的非高峰时段。
4.  按以下公式计算两个连续完整备份之间的时间间隔：时间间隔 = （总备份时间窗口） / （完整备份数量）。
5.  按照数据库优先级顺序安排完整备份，第一个备份从备份窗口开始时间启动，后续备份按照上述方程计算的时间间隔均匀分布。

这种完整备份的均匀分布将确保备份基础设施不会同时受到过多备份请求的冲击，从而减少完整备份对数据库性能的影响。
