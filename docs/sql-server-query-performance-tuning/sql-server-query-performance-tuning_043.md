# 第四章：CPU 性能分析

本章通过讨论 CPU、网络和通用 SQL Server 指标，结束本书对系统的探讨。CPU 是系统的工作引擎，它保持一切运行。收集和传输数据、维护系统以及排序访问所需的所有不同计算都是由 CPU 执行的。在 CPU 上遇到瓶颈可能是一个难以摆脱的过程。与内存（您有时可以轻松安装更多）或磁盘（您有时可以轻松添加更多或升级）不同，CPU 是您所运行系统的一个组成部分，并且通常只能通过购买更新的机器来升级。因此，您需要密切关注 CPU 使用率。

网络很少成为 SQL Server 的主要瓶颈，但同样值得密切关注。最后，有一些 SQL Server 内部进程您需要收集指标。本章涵盖以下主题：

-   如何收集处理器指标
-   通过 T-SQL 查询可用的其他指标
-   解决处理器瓶颈的方法

## 处理器瓶颈分析

SQL Server 大量使用任何可用的处理器资源。您可以使用表 4-1 中的性能监视器计数器来分析处理器资源上的压力。

**表 4-1. 用于分析 CPU 压力的性能监视器计数器**

| 对象(实例[,实例 N]) 计数器 | 描述 | 值 |
| :--- | :--- | :--- |
| Processor(_Total)% Processor Time | 处理器处于忙状态的时间百分比 | 平均值 < 80%，但需与基线比较 |
| Processor(_Total)% Privileged Time | 处理器在特权模式下花费的时间百分比 | 平均值 < 10%，但需与基线比较 |
| System Processor Queue Length | 处理器上未完成的请求数量 | 平均值 < 2，但需与基线比较 |
| System Context Switches/sec | 处理器从一个线程切换到另一个线程的速率 | 平均值 < 5,000，但需与基线比较 |
| SQL Server:SQL Statistics Batch Requests/sec | 每秒收到的 SQL 命令批处理数 | 根据您的标准工作负载 |
| SQL Server:SQL Statistics SQL Compilations/sec | SQL 被编译的次数 | 根据您的标准工作负载 |
| SQL Server:SQL Statistics SQL Recompilations/sec | 重编译次数 | 根据您的标准工作负载 |

让我们更详细地讨论这些计数器。

### % Processor Time

`% Processor Time` 不应持续过高（大于 80%）。任何持续超过 90% 的处理器时间的影响实际上与 100% 相同。如果 `% Processor Time` 持续较高，而磁盘和网络计数器值较低，您的首要任务必须是减轻处理器的压力。请记住，这里的数字只是建议；人们可以基于有效理由不同意这些数字。将它们作为评估系统的起点，而不是具体的建议。

例如，如果 `% Processor Time` 是 85%，并且通过监控 I/O 计数器您发现磁盘使用过多，那么很可能大部分处理器时间都花在了管理磁盘活动上。这将反映在处理器的 `% Privileged Time` 计数器中，如下一节所述。在这种情况下，首先优化磁盘瓶颈将是有利的。此外，请记住，磁盘瓶颈反过来可能是由于内存瓶颈造成的，如本章前面所述。

您可以跟踪机器上所有处理器的聚合处理器时间，也可以单独跟踪特定处理器的利用率百分比。这允许在 SQL Server 运行在四处理器机器的三个处理器上的情况下隔离数据收集。请记住，您可能会看到一个处理器满负荷运行，而另一个处理器负载很小。在这种情况下，平均值无法反映现实情况。将平均值仅作为一个指标，而将个体值作为系统实际负载和处理的更准确度量。

### % Privileged Time

Windows 服务器上的处理在两种模式下完成：*用户模式*和*特权*（或*内核*）模式。所有系统级别的活动，包括磁盘访问，都在特权模式下完成。如果您在专用的 SQL Server 系统上发现 `% Privileged Time` 达到 20% 到 25% 或更高，那么系统可能正在进行大量的外部处理。它可能是 I/O、筛选器驱动程序（如加密服务）、有缺陷的 I/O 组件，甚至是过时的驱动程序。专用 SQL Server 系统上的 `% Privileged Time` 计数器最多应为 5% 到 10%，但请使用您的基线来确定您系统上哪些行为看起来是正常的。

### Processor Queue Length

`Processor Queue Length` 是处理器队列中的线程数量。（即使在具有多个处理器的计算机上，也只有一个处理器队列。）与磁盘计数器不同，`Processor Queue Length` 计数器不读取已经在运行的线程。在 CPU 利用率较低的系统上，`Processor Queue Length` 计数器通常为 0 或 1。

持续的 `Processor Queue Length` 计数器大于 2 通常表示处理器拥塞。由于存在多个处理器，您可能需要考虑处理处理器队列长度的调度程序数量。超过调度程序数量（通常与处理器 1:1 对应）两倍的处理器队列长度也可能表明存在处理器瓶颈。虽然较高的 `% Processor Time` 计数器表示处理器繁忙，但持续较高的 `Processor Queue Length` 计数器是一个更确定的指标。如果超过了建议值，这通常表示有比当前处理器数量在最优方式下能够服务的更多的线程准备就绪。
