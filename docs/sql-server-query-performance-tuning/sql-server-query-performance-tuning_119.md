# 第 13 章 ■ 索引碎片

## 碎片解决方案

你可以通过重新排列索引行和页面，使它们的物理顺序和逻辑顺序匹配来解决索引碎片问题。为了减少外部碎片，你可以物理地重新排序索引的叶级页面，使其遵循索引的逻辑顺序。你可以通过以下技术实现这一点：

- 删除并重新创建索引
- 使用 `DROP_EXISTING = ON` 子句重新创建索引
- 对索引执行 `ALTER INDEX REBUILD` 语句
- 对索引执行 `ALTER INDEX REORGANIZE` 语句

### 删除并重新创建索引

消除索引碎片的最简单方法之一显然是先删除索引，然后重新创建它。

删除并重新创建索引可以最大程度地减少碎片，因为它允许 SQL Server 为索引使用全新的页面，并使用现有数据适当地填充它们。这避免了内部和外部碎片。不幸的是，这种方法有许多严重的缺点。

- **阻塞**：这种碎片整理技术会给系统增加大量开销，并导致阻塞。删除和重新创建索引会阻塞对该表（或表上任何其他索引）的所有其他请求。它也可能被针对该表的其他请求阻塞。
- **索引缺失**：在索引被删除，并且可能被阻塞并等待重新创建期间，针对该表的查询将无法使用该索引。这可能导致该索引原本打算解决的性能不佳问题。
- **非聚集索引**：如果被删除的索引是聚集索引，那么在聚集索引被删除后，表上所有的非聚集索引都必须重建。然后在聚集索引重新创建后，它们又必须再次重建。这导致了进一步的阻塞以及其他问题，例如语句重新编译（详见第 17 章）。
- **唯一约束**：用于定义主键或唯一约束的索引不能使用 `DROP INDEX` 语句删除。此外，唯一约束和主键都可以被外键约束引用。在删除主键之前，必须首先删除所有引用该主键的外键。尽管这是可行的，但这是一种有风险且耗时的索引碎片整理方法。

可以使用 `ONLINE` 选项来删除聚集索引，这意味着在删除过程中索引仍然可读，但这只能让你避免前述的阻塞问题。基于所有这些原因，删除并重新创建索引不推荐用于生产数据库，尤其是在非高峰时段之外的任何时候。

### 使用 DROP_EXISTING 子句重新创建索引

为了避免在重建聚集索引时两次重建非聚集索引的开销，请使用 `CREATE INDEX` 语句的 `DROP_EXISTING` 子句。这可以在一个原子步骤中重新创建聚集索引，避免重新创建非聚集索引，因为行定位器使用的聚集索引键值保持不变。

要使用 `DROP_EXISTING` 子句在一个原子步骤中重建聚集索引键，请执行如下 `CREATE INDEX` 语句：

```
CREATE UNIQUE CLUSTERED INDEX FirstIndex
ON dbo.Test1(C1)
WITH (DROP_EXISTING = ON);
```

你可以在聚集索引和非聚集索引上使用 `DROP_EXISTING` 子句，甚至可以将非聚集索引转换为聚集索引。但是，不能用它将聚集索引转换为非聚集索引。

这种碎片整理技术的缺点如下：

- **阻塞**：类似于 `DROP` 和 `CREATE` 方法，此技术也会导致并面临来自访问该表（或表上任何索引）的其他查询的阻塞。
- **带约束的索引**：与第一种方法不同，带有 `DROP_EXISTING` 的 `CREATE INDEX` 语句可用于重新创建带约束的索引。如果该约束是主键，或者唯一约束与外键关联，那么在 `CREATE` 语句中不包含 `UNIQUE` 关键字将导致如下错误：

  **注意** `msg 1907, Level 16, State 1, Line 1` 无法重新创建索引 'pK_name'。新的索引定义与现有索引强制实施的约束不匹配。

- **具有多个碎片化索引的表**：随着表数据碎片化，索引通常也会变得碎片化。如果使用这种碎片整理技术，则必须识别表上的所有索引并分别重建它们。

你可以通过使用 `ALTER INDEX REBUILD` 来避免与此技术相关的最后两个限制，如下所述。

### 执行 ALTER INDEX REBUILD 语句

`ALTER INDEX REBUILD` 在一个原子步骤中重建索引，就像带有 `DROP_EXISTING` 子句的 `CREATE INDEX` 一样。

由于 `ALTER INDEX REBUILD` 也会物理地重建索引，它允许 SQL Server 分配全新的页面，以将内部和外部碎片降至最低。但与带有 `DROP_EXISTING` 子句的 `CREATE INDEX` 不同，它允许动态重建一个索引（支持 `PRIMARY KEY` 或 `UNIQUE` 约束），而无需删除和重新创建约束。
