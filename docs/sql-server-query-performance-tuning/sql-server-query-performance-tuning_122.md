# 第 13 章 ■ 索引碎片

从输出中可以看出，如上一节所示，`ALTER INDEX REORGANIZE` 在减少碎片方面不如 `ALTER INDEX REBUILD` 有效。对于高度碎片化的索引，`ALTER INDEX REORGANIZE` 操作可能比重建索引花费更长的时间。此外，如果一个索引跨越多个文件，`ALTER INDEX REORGANIZE` 不会在文件之间迁移页面。然而，使用 `ALTER INDEX REORGANIZE` 的主要好处是它允许其他查询同时访问表（或索引）。

表 13-1 总结了这四种碎片整理技术的特性。

表 13-1. 碎片整理技术的特性

| 特性/问题 | 删除并创建索引 | 使用 `DROP_EXISTING` 创建索引 | `ALTER INDEX REBUILD` | `ALTER INDEX REORGANIZE` |
| :--- | :--- | :--- | :--- | :--- |
| 当聚集索引碎片化时重建非聚集索引 | 两次 | 不需要 | 不需要 | 不需要 |
| 缺失索引 | 是 | 否 | 否 | 否 |
| 对带约束的索引进行碎片整理 | 非常复杂 | 中等复杂 | 简单 | 简单 |
| 同时对多个索引进行碎片整理 | 否 | 否 | 是 | 是 |
| 与其他进程的并发性 | 低 | 低 | 中等，取决于并发用户活动 | 高 |
| 中途取消且无事务 | 危险 | 进度丢失 | 进度丢失 | 进度保留 |
| 碎片整理程度 | 高 | 高 | 高 | 中等到低 |
| 应用新的填充因子 | 是 | 是 | 是 | 否 |
| 更新统计信息 | 是 | 是 | 是 | 否 |
