# 第 23 章 ■ 内存优化 OLTP 表与存储过程

遗憾的是，如果您尝试按当前定义运行此查询，将收到以下错误：`错误号 10775，级别 16，状态 1，过程 AddressDetails，行 5530`

`对象 'dbo.CountryRegion' 不是内存优化表，无法从本地编译存储过程访问。`

虽然您可以查询内存中表与标准表的混合组合，但只能针对内存中表创建本地编译存储过程。我将使用前述方法将 `dbo.CountryRegion` 表加载到内存中，然后再次运行脚本。此次它将成功编译。如果您随后像之前一样使用 `@City = 'Walla Walla'` 执行查询，其执行时间在 `SSMS` 中甚至不会显示出来。

您必须通过扩展事件捕获该事件，如图 23-9 所示。

**图 23-9.** 显示本地编译存储过程执行时间的扩展事件

那里的执行时间单位不是毫秒而是微秒。因此，查询执行时间已从原始运行的 `154ms` 减少到内存中运行的 `15ms`，最终降至小于 1/2 毫秒。

这是一个相当显著的性能提升。

但是，存在限制条件。如前所述，您必须仅引用内存中的表。

分配给存储过程的参数值不能接受 `NULL` 值。如果您选择将参数设置为 `NOT NULL`，则必须同时提供初始值。否则，所有参数都是必需的。您必须强制实施与基础表的架构绑定。最后，存储过程需要存在于一个 `ATOMIC` 块中。原子块要求事务中的所有语句都成功，否则事务内的所有语句都将被回滚。

关于本地编译存储过程还有另外几个有趣之处。您只能检索估算的执行计划，而无法获取实际执行计划。如果您在 `SSMS` 中开启实际执行计划然后执行查询，不会显示任何内容。但是，如果您请求估算执行计划，则可以检索到一个。图 23-10 显示了先前创建的存储过程的估算计划。

**图 23-10.** 本地编译存储过程的估算执行计划

**图 23-11.** 两种不同执行计划中的 SELECT 操作符属性

您期望看到的许多信息都消失了，因为本地编译存储过程的运行方式与其他查询不同。使用执行计划来确定这些查询的行为在此处与标准查询中一样有价值，但其内部机制将会不同。

## 建议

虽然内存中表和本地编译存储过程可以在 `SQL Server` 内带来性能的显著提升，但您仍然需要评估在您的情况下是否值得使用它们。这些新对象行为上的限制意味着它们并非在所有情况下都适用。此外，由于对硬件的要求以及需要企业级 `SQL Server` 安装，许多人将无法实现这些新对象及其行为。要确定您的工作负载是否适合使用这些新对象，您可以采取若干措施。
