# SQL Server 高级索引：列存储与特殊索引类型

## 列存储索引的使用与优势

在部署了 `非聚集列存储索引` 后，优化器现在便有了使用该索引来满足前述查询的选项。就像优化器可用的所有其他索引一样，`列存储索引` 也有关联的成本，因此对于针对表的任何给定查询，它可能被选用，也可能不被选用来满足要求。在本例中，如果你重新运行最初的聚合查询，可以看到优化器判定使用 `列存储索引` 相关的成本对该查询是有利的。执行计划现在如图 9-15 所示。

**图 9-15.** `列存储索引` 被使用，而非 `聚集索引`
如你所见，查询的基本操作是相同的，但扫描的是 `列存储索引` 而非 `聚集索引`。真正的差异体现在查询的读取次数和执行时间上。


检索数据所需读取次数的大幅减少以及速度的些许提升，都是能够引用按列而非按行索引的信息的结果。前述查询以所谓的 *行模式* 运行。`列存储索引` 有两种操作模式：`批处理模式` 和 `行模式`。两者之中，`批处理模式` 要快得多。它利用了一种特殊的数据访问方法，允许以批次方式拉取大量数据，因此无需解压。`批处理模式` 处理在使用多处理器机器和具有大缓存的现代处理器时，还具有额外的优势。要查看 `列存储` 是以 `批处理` 还是 `行模式` 运行，你只需检查 `列存储` 操作的属性即可。例如，在所述查询中，图 9-16 显示了来自 `列存储扫描` 操作符的属性表。

**图 9-16.** `实际执行模式`
联机丛书中记录了一组有限的操作会导致 `批处理模式` 处理，但在具有足够处理器的系统上处理这些操作时，你将看到又一次显著的性能提升。

`列存储索引` 不要求你以特定顺序排列列，这与 `聚集` 和 `非聚集索引` 不同。同样，与这些其他索引不同，你应该将多个列放入一个 `列存储索引` 中，以便跨这些列获得益处。换句话说，如果你预计将来需要查询某个列，应主动将其添加到 `列存储索引` 定义中。但是，如果你从 `列存储索引` 中检索大量列，可能会看到一些性能下降。

你也可以看到 `聚集列存储索引` 的实际应用。由于不能有约束，你只需用此脚本重新创建表：

```sql
SELECT *
INTO dbo.TransactionHistoryArchive
FROM Production.TransactionHistoryArchive;

CREATE INDEX ClusteredColumnstoreTest
ON dbo.TransactionHistoryArchive
(TransactionID);
```

从那里，就有可能创建一个 `聚集列存储索引`，有效地将表从传统的 `聚集表` 转换为 `列存储表`，因为与常规 `聚集索引` 类似，`聚集列存储索引` 存储数据。语法如下：

```sql
CREATE CLUSTERED COLUMNSTORE INDEX ClusteredColumnstoreTest
ON dbo.TransactionHistoryArchive
WITH (DROP_EXISTING = ON);
```

这将生成一个现在存储为 `聚集列存储` 的表。如果你修改之前的聚合查询以针对此新表运行，并将查询指向此表，你将看到与使用 `非聚集列存储索引` 时相同的执行计划和性能。但是，与你之前创建的索引不同，这个新结构可以被更新。

虽然使用 `列存储索引` 的限制有些严格，特别是对于 `非聚集列存储索引`，但对于能在这些限制内操作的结构来说，其好处是显而易见的。

请务必清理任何剩余的对象。

```sql
DROP TABLE dbo.TransactionHistoryArchive;
DROP INDEX Production.TransactionHistoryArchive.ix_csTest;
```

### 特殊索引类型

随着 Microsoft 向 `SQL Server` 引入特殊数据类型和存储机制，为这些特殊存储类型建立索引的方法也随之发展。详细解释所有这些特殊索引类型的每一个细节超出了本书的范围。在以下部分中，我将介绍每种索引类型的基本概念，以便为在调整查询时使用它们提供可能性。

#### 全文索引

你可以通过在 `VARCHAR`、`NVARCHAR`、`CHAR` 和 `NCHAR` 字段中使用 `MAX` 值，在 `SQL Server` 中存储大量文本。针对这些大字段的普通 `聚集` 或 `非聚集索引` 将是不可行的，因为单个值可能远远超过索引中的页大小。因此，一种不同的文本索引机制是使用 `全文引擎`，该引擎必须正在运行才能处理 `全文索引`。你也可以在 `VARBINARY` 数据上构建 `全文索引`。

你需要在表上有一个唯一的列。最佳性能候选者是整数：`INT` 或 `BIGINT`。然后，此列与单词一起使用，以标识它属于表中的哪一行，以及在字段内的位置。`SQL Server` 允许对 `全文索引` 进行增量更改，包括 `更改跟踪` 或基于时间的更改，也允许完全重建。

`SQL Server 2012` 引入了另一种处理文本的方法，称为 `语义搜索`。它使用文档中的短语来识别存储在数据库内的不同文本集之间的关系。

#### 空间索引

`SQL Server 2008` 中引入了存储 `空间数据` 的功能。这些数据可以是 `geometry` 类型，也可以是非常复杂的 `geographical` 类型，字面意义上标识了地球上的一个点。至少可以说，为这类数据建立索引是复杂的。`SQL Server` 将这些索引存储在扁平的 `B 树` 中，类似于常规索引，区别在于它也是一个由四个网格连接在一起的层次结构。每个网格可以被赋予 `低`、`中` 或 `高` 的密度，以概述每个网格的大小。存在一些机制来支持 `空间数据类型` 的索引，使得不同类型的查询，例如查找一个对象何时在另一个对象的边界内或附近，都能受益于索引固有的性能提升。

`空间索引` 只能针对 `geometry` 或 `geography` 类型的列创建。它必须在基表上，表上不能有索引视图，并且表必须有一个 `主键`。在任何给定的表列上，最多可以创建 249 个 `空间索引`。不同的索引用于定义不同类型的索引行为。更多信息请参阅 Alastair Aitchison 所著的 *Beginning Spatial with SQL Server 2008* (Apress, 2009)。

#### XML 索引

`XML` 作为数据类型在 `SQL Server 2005` 中引入，可以存储为良构的 `XML 数据` 而非文本。可以使用 `SQL Server` 支持的 `XQuery 语言` 对此数据进行查询。为了增强性能能力，定义了一组特殊的索引。一个 `XML 列` 可以有一个 `主索引` 和若干 `辅助索引`。`主 XML 索引` 将 `XML 数据` 的属性、特性和元素分解并存储为一个内部表。表上必须有一个 `主键`，并且该 `主键` 必须是 `聚集的` 才能创建 `XML 索引`。创建 `XML 索引` 后，就可以创建 `辅助索引`。这些索引有 `PATH`、`VALUE` 和 `PROPERTY` 类型，具体取决于你如何查询 `XML`。更多详情，请查阅 Michael Coles 所著的 *Pro SQL Server 2008 XML* (Apress, 2008)。
