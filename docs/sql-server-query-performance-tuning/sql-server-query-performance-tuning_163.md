# SQL Server 事务与锁：ACID 属性

在 SQL Server 中，一个数据库查询可以作为一个独立的工作逻辑单元执行，也可以参与一个更大的工作逻辑单元。一个更大的工作逻辑单元可以使用 `BEGIN TRANSACTION` 语句以及 `COMMIT` 和/或 `ROLLBACK` 语句来定义。每个工作逻辑单元必须遵循一组称为 `ACID` 属性的四个属性：

*   `原子性`
*   `一致性`
*   `隔离性`
*   `持久性`

我将在接下来的章节中介绍这些属性，因为理解事务的工作原理是理解锁的基础。

## `原子性`

一个工作逻辑单元必须是 `原子性` 的。也就是说，工作逻辑单元的所有操作要么全部完成，要么不保留任何效果。为了理解工作逻辑单元的原子性，请看以下示例：

```sql
USE AdventureWorks2012;
GO

IF (SELECT OBJECT_ID('dbo.ProductTest')) IS NOT NULL
    DROP TABLE dbo.ProductTest;
GO

CREATE TABLE dbo.ProductTest (
    ProductID INT CONSTRAINT ValueEqualsOne CHECK (ProductID = 1));
GO

--所有 ProductID 作为一个逻辑工作单元添加到 t1
INSERT INTO dbo.ProductTest
SELECT p.ProductID
FROM Production.Product AS p;
GO

SELECT *
FROM dbo.ProductTest; --返回 0 行
```

SQL Server 将前面的 `INSERT` 语句视为一个工作逻辑单元。`dbo.ProductTest` 表的 `ProductID` 列上的 `CHECK` 约束只允许值为 1。尽管 `Production.Product` 表中的 `ProductID` 列以值 1 开始，但它也包含其他值。因此，`INSERT` 语句根本不会向 `dbo.ProductTest` 表添加任何记录，并且由于 `CHECK` 约束会引发错误。这种原子性由 SQL Server 自动确保。

到目前为止，一切正常。但是，在更大的工作逻辑单元的情况下，你应该注意 SQL Server 的一个有趣行为。假设前面的插入任务由多个 `INSERT` 语句组成。它们可以组合形成一个更大的工作逻辑单元，如下所示：

```sql
BEGIN TRAN
    --开始：逻辑工作单元
    --第一个：
    INSERT INTO dbo.ProductTest
    SELECT p.ProductID
    FROM Production.Product AS p;
    --第二个：
    INSERT INTO dbo.ProductTest
    VALUES (1);
COMMIT --结束：逻辑工作单元
GO
```

在前面脚本中已创建 `dbo.ProductTest` 表的情况下，`BEGIN TRAN` 和 `COMMIT` 语句对定义了一个工作逻辑单元，表明事务内的所有语句本质上都应该是原子的。然而，SQL Server 的默认行为并不确保用户定义事务范围内的某个语句失败时会撤消先前语句的效果。在前面的事务中，第一个 `INSERT` 语句将如前所述失败，而第二个 `INSERT` 语句完全没有问题。SQL Server 的默认行为允许第二个 `INSERT` 语句执行，即使第一个 `INSERT` 语句失败。如下的 `SELECT` 语句将返回第二个 `INSERT` 语句插入的行：

```sql
SELECT *
FROM dbo.ProductTest; --返回一行，其中 t1.c1 = 1
```

可以通过以下两种方式确保用户定义事务的原子性：

*   `SET XACT_ABORT ON`
*   显式回滚

让我们快速了解一下这些方法。

### `SET XACT_ABORT ON`

你可以使用 `SET XACT_ABORT ON` 语句修改前面部分中 `INSERT` 任务的原子性：

```sql
SET XACT_ABORT ON;
GO

BEGIN TRAN
    --开始：逻辑工作单元
    --第一个：
    INSERT INTO dbo.ProductTest
    SELECT p.ProductID
    FROM Production.Product AS p;
    --第二个：
    INSERT INTO dbo.ProductTest
    VALUES (1);
COMMIT
    --结束：逻辑工作单元
GO

SET XACT_ABORT OFF;
GO
```

`SET XACT_ABORT` 语句指定当事务中的语句失败时，SQL Server 是否应自动回滚并中止整个事务。第一个 `INSERT` 语句的失败将自动挂起整个事务，因此第二个 `INSERT` 语句将不会被执行。`SET XACT_ABORT` 的作用域是连接级别，在重新配置或关闭连接之前一直有效。默认情况下，`SET XACT_ABORT` 为 `OFF`。

### 显式回滚

你还可以使用 SQL Server 中的 `TRY/CATCH` 错误捕获机制来管理用户定义事务的原子性。如果 `TRY` 代码块中的语句产生错误，则 `CATCH` 代码块将处理该错误。如果发生错误并且激活了 `CATCH` 块，则可以回滚用户定义事务的全部工作，并阻止进一步语句的执行，如下所示：

```sql
BEGIN TRY
    BEGIN TRAN
        --开始：逻辑工作单元
        --第一个：
        INSERT INTO dbo.ProductTest
        SELECT p.ProductID
        FROM Production.Product AS p;
        --第二个：
        INSERT INTO dbo.ProductTest
            (ProductID)
        VALUES (1);
    COMMIT --结束：逻辑工作单元
END TRY
BEGIN CATCH
    ROLLBACK
    PRINT 'An error occurred'
    RETURN
END CATCH
```

`ROLLBACK` 语句回滚事务中在该点之前执行的所有操作。有关如何在基于 SQL Server 的应用程序中实现错误处理的详细说明，请参阅 MSDN 库文章“在 Transact-SQL 中使用 TRY . . . CATCH”（[`bit.ly/PNlAHF`](http://bit.ly/PNlAHF)）或介绍性文章“SQL Server 错误处理工作台”（[`bit.ly/1nC3VBt`](http://bit.ly/1nC3VBt)）。

由于原子性属性要求工作逻辑单元的所有操作要么全部完成，要么不保留任何效果，因此 SQL Server 通过授予事务对受影响资源的独占访问权，将其工作与其他事务的工作 `隔离`。这意味着事务可以在需要时安全地回滚其所有操作的效果。授予事务在受影响资源上的独占权会阻塞在此期间试图访问这些资源的所有其他事务（或数据库请求）。因此，虽然原子性是为了维护数据完整性所必需的，但它也带来了锁这种不良的副作用。

## `一致性`

一个工作逻辑单元应该导致数据库状态从一个 `一致` 状态迁移到另一个状态。在事务结束时，数据库状态应完全一致。SQL Server 始终确保数据库的内部状态正确有效，方法是自动将受影响数据库资源的所有约束作为事务的一部分应用。SQL Server 确保内部结构（如数据和索引布局）的状态在事务之后是正确的。例如，当修改表的数据时，SQL Server 会自动识别表上的所有索引、约束和其他依赖对象，并作为事务的一部分将必要的修改应用于所有依赖的数据库对象。

业务规则所需的数据逻辑一致性应由数据库开发人员确保。业务规则可能要求将更改应用于多个表。数据库开发人员应相应地定义一个工作逻辑单元，以确保满足业务规则的所有标准。SQL Server 提供了不同的事务管理功能，数据库开发人员可以使用这些功能来确保数据的逻辑一致性。


## 一致性（ACID 特性）

正如刚才所解释的，维护一致的逻辑状态需要按照业务规则使用事务来定义逻辑工作单元。此外，为了维护一致的物理状态，SQL Server 会识别并处理作为逻辑工作单元一部分的依赖数据库对象。逻辑工作单元的原子性特性会在此期间阻止所有其他试图访问受影响对象的事务（或数据库请求）。

因此，尽管为了维护数据库有效的逻辑和物理状态需要一致性，但它也引入了阻塞这一不良副作用。

### 隔离性

在多用户环境中，可以同时执行多个事务。这些并发事务应彼此隔离，以便一个事务所做的中间更改不会影响其他事务的数据一致性。事务所需的 *隔离* 程度可能有所不同。SQL Server 提供了不同的事务隔离级别特性，以实现事务所需的隔离程度。

■ **注意**：本章后面的“隔离级别”部分将解释事务隔离级别。

事务在数据库资源上操作的隔离要求可能会阻止其他事务访问该资源。在多用户数据库环境中，通常同时执行多个事务。必须保护进行中事务所做的数据修改免受其他事务修改的影响。例如，假设一个事务正在修改表中的几行数据。在此期间，为了维护数据库一致性，您必须确保其他事务不会修改或删除相同的行。SQL Server 通过适当阻塞，从事务逻辑上隔离其活动，从而允许多个事务同时执行而不会相互破坏工作。

隔离性导致的过度阻塞会对数据库应用程序的可伸缩性产生不利影响。一个事务可能会无意中长时间阻塞其他事务，从而损害数据库并发性。由于 SQL Server 使用锁管理隔离性，因此理解 SQL Server 的锁定架构非常重要。这有助于您分析阻塞场景并实施解决方案。

■ **注意**：本章后面的“捕获阻塞信息”部分将解释数据库锁的基本原理。

### 持久性

一旦事务完成，事务所做的更改就应该是 *持久* 的。即使在事务完成后立即切断机器电源，事务内所有操作的效果也应保留。SQL Server 通过跟踪事务日志中正在修改的数据的前像和后像来确保持久性。事务完成后，SQL Server 确保事务所做的所有更改都被保留——即使 SQL Server、操作系统或硬件发生故障（日志磁盘除外）。在重新启动期间，SQL Server 运行其数据库恢复功能，该功能从事务日志中识别已完成事务的待处理更改，并将它们应用到数据库资源。此数据库功能称为 *前滚*。

恢复间隔时间取决于重新启动期间需要应用到数据库资源的待处理更改数量。为了减少恢复间隔时间，SQL Server 会间歇性地应用运行中事务所做的中间更改，这是由 `recovery interval` 选项配置的。`recovery interval` 选项可以使用 `spconfigure` 语句进行配置。间歇性应用中间更改的过程称为 *检查点* 过程。在重新启动期间，恢复过程识别所有未提交的更改，并使用事务日志中数据的前像将它们从数据库资源中移除。

持久性属性并非大多数阻塞的直接原因，因为它不需要事务的操作与其他事务的操作隔离。但在间接方式上，它增加了阻塞的持续时间。由于持久性属性要求将正在修改的数据的前像和后像保存到磁盘上的事务日志，因此它增加了事务和阻塞的持续时间。

SQL Server 2014 引入了一项功能，可以通过修改给定数据库的持久性行为来减少延迟（等待查询提交并写入日志的时间）。现在可以使用延迟持久性。这意味着当一个事务完成时，它会立即向应用程序报告为成功的事务，从而减少延迟。

但此时对日志的写入尚未发生。这也可能允许在系统仍在将所有输出写入事务日志的同时完成更多事务。虽然这可能会增加系统内的表观速度，并可能减少事务日志 I/O 上的争用，但这本质上是一个危险的选择。这是一个难以提出的建议。Microsoft 建议了三种可能使其具有吸引力的情况。

*   *您不关心可能丢失部分数据*：由于您可能需要从日志备份恢复到某个时间点，通过选择将数据库置于延迟持久性模式，在必须进行恢复时您可能会丢失一些数据。
*   *日志写入期间存在高度争用*：如果在事务写入日志时看到大量等待，延迟持久性可能是一个可行的解决方案。但是，如前所述，您也需要容忍数据丢失。
*   *您正经历较高的整体资源争用*：服务器上的大量资源争用归根结底是由于锁保持时间过长。如果您看到大量争用，并且看到日志写入时间长或日志上也存在争用，并且您对数据丢失有较高的容忍度，这可能是一种可行的方法来帮助减少系统的争用。

换句话说，我建议仅当您满足所有这些标准时，才使用延迟持久性，其中第一个标准最为重要。

■ **注意**：在四个 ACID 属性中，隔离属性（也用于确保原子性和一致性）是 SQL Server 数据库中阻塞的主要原因。在 SQL Server 中，隔离是使用锁实现的，如下一节所述。

## 锁

当会话执行查询时，SQL Server 确定需要访问的数据库资源；如果需要，锁管理器会将会话授予不同类型的锁。如果另一个会话已被授予锁，则该查询会被阻塞；但是，为了提供事务隔离和并发性，SQL Server 使用不同的锁粒度，如下节所述。

### 锁粒度
