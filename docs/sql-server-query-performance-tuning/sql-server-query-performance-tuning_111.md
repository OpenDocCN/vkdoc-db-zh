# 第 12 章：统计信息、数据分布与基数

#### 解决统计信息过时问题

有时，过时或不正确的统计信息可能比缺失的统计信息更具破坏性。基于旧的统计信息或对已更改数据的部分扫描，优化器可能决定采用某种特定的索引策略，而这可能与当前的数据分布极不匹配。不幸的是，执行计划不会像对缺失统计信息那样，对过时或不正确的统计信息显示同样明显的警告。然而，有一个名为 `inaccurate_cardinality_estimate` 的事件。这是一个调试事件，这意味着它的使用在生产系统上可能有些问题。我强烈建议谨慎使用，仅在适当过滤且仅用于短时间的情况下使用，但我需要指出这一点。

识别过时统计信息更传统也更安全的方法是，检查优化器估算的影响行数与实际影响行数的接近程度。

以下示例展示了如何识别并解决过时的统计信息问题。图 12-35 显示了由 `DBCC SHOW_STATISTICS` 提供的列 `C1` 上非聚集索引键的统计信息。

```sql
DBCC SHOW_STATISTICS (Test1, iFirstIndex);
```

图 12-35. 索引 `FirstIndex` 上的统计信息

这些结果显示列 `C1` 的密度值为 0.5。现在考虑以下 `SELECT` 语句：

```sql
SELECT *
FROM dbo.Test1
WHERE C1 = 51;
```

由于表中的总行数当前为 10,002，筛选条件 `C1 = 51` 的匹配行数可估算为 5,001 (=0.5 * 10,002)。这个估算行数 (5,001) 与此列值的实际匹配行数相差甚远。表中实际只包含一行 `C1 = 51` 的数据。
