## 阻塞基础知识

在理想世界中，每个 SQL 查询都能并发执行，不会被其他查询阻塞。

然而，在现实世界中，查询*确实*会相互阻塞，类似于在十字路口，一辆车通过绿色交通信号灯时会阻塞其他等待通过十字路口的车辆。在 SQL Server 中，这种交通管理以 `锁管理器` 的形式实现，它控制对数据库资源的并发访问以维护数据一致性。对数据库资源的并发访问在多个数据库连接之间受到控制。

我想在继续之前确保事情是清晰的。数据库中使用了三个听起来相同且相互关联但含义不同的术语。这些术语经常被混淆，人们也经常误用它们。这些术语是 `锁`、`阻塞` 和 `死锁`。锁是 SQL Server 管理多个会话过程的组成部分。当一个会话需要访问某段数据时，会对其放置某种类型的锁。这与阻塞不同，阻塞是指一个会话或线程需要访问某段数据，但必须等待另一个会话的锁被释放。最后，死锁是指两个会话或线程形成了有时被称为 `致命拥抱` 的情况。它们都在等待对方释放锁。死锁也可以被称为永久性阻塞情况，但它是即使等待任何时间段也不会自行解决的那种。

死锁将在第 21 章更详细地介绍。因此，锁可能导致阻塞，而锁和阻塞在死锁中都起作用，但这是三个截然不同的概念。请理解这些术语之间的区别并正确使用它们。这将有助于你理解系统、进行故障排除以及与其他数据库管理员和开发人员交流。

在 SQL Server 中，数据库连接由会话 ID 标识。连接可能来自一个或多个应用程序，以及这些应用程序上的一个或多个用户；对 SQL Server 而言，每个连接都被视为一个独立的会话。两个会话在同一时间访问同一段数据时发生的阻塞，是 SQL Server 中的自然现象。每当两个会话试图以冲突的方式访问一个共同的数据库资源时，`锁管理器` 会确保第二个会话等待，直到第一个会话完成其工作，这与系统内的事务管理相协调。例如，一个会话可能正在修改表记录，而另一个会话试图删除该记录。由于这两个数据访问请求是不兼容的，第二个会话将被阻塞，直到第一个会话完成其任务。

另一方面，如果两个会话试图同时读取一个表，两个请求都会被允许执行而不会阻塞，因为这些数据访问请求彼此兼容。

通常，阻塞对会话的影响很小，不会明显影响其性能。然而，有时由于查询和/或事务设计不佳（或者运气不好），阻塞会显著影响查询性能。在数据库应用程序中，应尽一切努力最小化阻塞，从而增加可以同时使用数据库的用户数量。
