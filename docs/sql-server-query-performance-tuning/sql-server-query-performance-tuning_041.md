# 第三章 ■ 磁盘性能分析

如果表 `HumanResources.JobCandidate` 和 `Person.BusinessEntity` 被分别放置在各自包含一个文件的独立文件组中，那么磁盘就可以通过多个 I/O 路径进行读取，从而提升性能。

出于性能和恢复的目的，建议如果要使用多个文件组，则主文件组应仅用于系统对象，而次文件组应仅用于用户对象。这种方法提高了从损坏中恢复的能力。如果主数据文件和日志文件完好无损，数据库的可恢复性会更高。仅将主文件组用于系统对象，并将所有用户相关对象存储在一个或多个次文件组上。

将数据库分散到多个文件中（即使在同一驱动器上），可以轻松地将数据库文件移动到独立的驱动器上，从而使未来的磁盘升级更容易。例如，要将用户数据库文件 (`AdventureWorks2012_2.ndf`) 移动到新的磁盘子系统 (`F:`)，您可以遵循以下步骤：

1.  分离用户数据库，如下所示：
    ```sql
    USE master;
    GO
    EXEC sp_detach_db 'AdventureWorks2012';
    GO
    ```
2.  将数据文件 `AdventureWorks2012_2.ndf` 复制到新磁盘子系统上的文件夹 `F:\Data\` 中。
3.  通过引用适当位置的文件重新附加用户数据库，如下所示：
    ```sql
    USE master;
    GO
    sp_attach_db 'AdventureWorks2012R2',
        'R:\DATA\AdventureWorks2012.mdf',
        'F:\DATA\AdventureWorks2012_2.ndf',
        'S:\LOG\AdventureWorks2012.ldf';
    GO
    ```

### 将日志文件移动到独立的物理磁盘

只要可能，SQL Server 事务日志文件应始终位于与其他所有 SQL Server 数据库文件分离的独立硬盘驱动器上。与数据文件所需的非顺序（或随机）I/O 不同，事务日志活动主要由顺序写入 I/O 组成。将事务日志活动与其他非顺序磁盘 I/O 活动分离可以带来 I/O 性能的提升，因为它使包含日志文件的硬盘驱动器能够专注于顺序 I/O。但是请记住，事务日志也存在随机读取，并且数据读写也可以像事务日志一样是顺序的。只是事务日志写入具有强烈的顺序性倾向。

从硬盘访问数据所需时间的主要部分花费在磁盘主轴头的物理移动以定位数据上。一旦数据被定位，数据就会以电子方式读取，这比主轴头的物理移动要快得多。如果在日志磁盘上仅进行顺序 I/O 操作，则日志磁盘的主轴头可以用最少的物理移动写入日志磁盘。然而，如果同一个磁盘也用于数据文件，主轴头在写入日志文件之前必须移动到正确的位置。这增加了写入日志文件所需的时间，从而损害了性能。

即使使用 SSD 磁盘，将数据与事务日志隔离也意味着工作将被分配到多个位置，从而提高性能。

此外，对于拥有多个 OLTP 数据库的 SQL Server，事务日志文件应在不同的物理驱动器上彼此物理分离以提高性能。此要求的例外情况是只读数据库或数据库更改很少的数据库。由于对只读数据库不进行联机更改，因此不会对日志文件执行写入操作。因此，对于只读数据库，不需要将日志文件放在单独的磁盘上。

作为一个经验法则，您应尽可能尝试将具有最高 I/O 的文件与其他高 I/O 文件隔离。这将减少磁盘上的争用并可能提高性能。要识别那些使用最多 I/O 的文件，请参考 `sys.dm_io_virtual_file_stats`。
