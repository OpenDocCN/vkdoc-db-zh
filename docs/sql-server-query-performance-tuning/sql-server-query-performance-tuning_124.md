# 第 13 章 ■ 索引碎片

### 碎片整理与分区

如果你拥有海量数据库，一种有效的数据管理标准机制是将其拆分为分区。虽然分区在某些罕见情况下可能有助于提升性能，但其首要目的还是为了管理数据。然而，索引和分区的一个问题是，如果你重建索引，在重建期间它将不可用。这意味着对于海量索引上的分区，你可能会在重建过程中使大部分数据处于离线状态。SQL Server 2012 引入了在线重建的能力。如果你有一个分区索引，它看起来会像这样：

```sql
ALTER INDEX i1 ON dbo.Test1
REBUILD PARTITION = ALL
WITH (ONLINE = ON);
```

这可以重建整个分区，并以在线操作的方式进行，意味着在重建过程中，索引在很大程度上保持可用。但是，对于某些分区来说，这是一项庞大的任务，可能会导致服务器负载过高，并且需要更多的 `tempdb` 存储空间。SQL Server 2014 引入了新功能，允许你指定单个分区。

```sql
ALTER INDEX i1 ON dbo.Test1
REBUILD PARTITION = 1
WITH (ONLINE = ON);
```

这减少了重建操作的开销，同时在重建期间仍保持索引基本可用。我需要强调是“基本”在线，因为在重建过程中仍然会发生一定程度的锁和争用。这并非完全无开销的操作，但比起替代方案已经有了巨大的改进。

谈到分区索引重建操作涉及的锁，SQL Server 2014 还引入了另一项新功能。你可以通过调整 `REBUILD` 命令来修改重建操作期间使用的锁优先级。

```sql
ALTER INDEX i1 ON dbo.Test1
REBUILD PARTITION = 1
WITH (ONLINE = ON (WAIT_AT_LOW_PRIORITY (MAXDURATION = 20, ABORT_AFTER_WAIT = SELF)));
```

这样做的作用是设置重建操作愿意等待的时长（以分钟为单位）。然后，它允许你决定为了给索引重建让路而中止哪些进程。你可以让它中止自身或造成阻塞的进程。最有趣的是，等待进程被设置为低优先级，因此它不会占用大量系统资源，并且任何新传入的事务都不会被这个进程阻塞。

结合这两项新功能，使用 SQL Server 2014 管理分区中的索引变得容易得多。

## 填充因子的重要性

你还可以通过在页面内压缩更多行来减少内部碎片，从而减少页面内的空闲空间。索引叶页面内可以进行的最大压缩量由 `fill factor` 控制，你将在下文看到。

索引的内部碎片是通过在索引的叶页面中获取更多行来减少的。在叶页面中获得更多行减少了索引所需的总页面数，进而降低了检索一系列索引行所需的磁盘 I/O 和逻辑读取。另一方面，如果索引键值具有高度的事务性，那么完全使用的索引页面会导致页拆分。因此，对于一个事务性表，需要在最大化页面中的行数和避免页拆分之间取得良好平衡。

SQL Server 允许你使用 `fill factor` 来控制索引叶页面内的空闲空间量。如果你知道表上会有大量的 `INSERT` 查询或对索引键列的 `UPDATE` 查询，那么你可以使用 `fill factor` 预先在索引叶页面中添加空闲空间，以最小化页拆分。如果表是只读的，你可以使用高 `fill factor` 创建索引以减少索引页面的数量。

默认的 `fill factor` 为 0，这意味着叶页面被填满到 100%，尽管在 B 树结构的分支节点中会保留一些空闲空间。索引的 `fill factor` 仅在创建索引时应用。

随着键的插入和更新，索引中的行密度最终会在一个狭窄的范围内趋于稳定。

正如你在前一章关于 `UPDATE` 和 `INSERT` 导致的页拆分的章节中看到的，当发生页拆分时，通常原始页面的一半会被移动到一个新页面，无论创建索引时使用了何种 `fill factor`，这种情况都会发生。

为了理解 `fill factor` 的重要性，我们使用一个有 24 行的小型测试表。

```sql
IF (SELECT OBJECT_ID('dbo.Test1')) IS NOT NULL
    DROP TABLE dbo.Test1;
GO

CREATE TABLE dbo.Test1 (C1 INT, C2 CHAR(999));

WITH Nums
AS (SELECT 1 AS n
    UNION ALL
    SELECT n + 1
    FROM Nums
    WHERE n < 24
)
INSERT INTO dbo.Test1 (C1, C2)
SELECT n * 100, 'a'
FROM Nums;
```

通过使用默认 `fill factor` 创建聚集索引来增加叶（或数据）页面中的最大行数。

```sql
CREATE CLUSTERED INDEX FillIndex ON Test1(C1);
```

由于平均行大小为 1,010 字节，一个聚集索引叶页面（或表数据页面）最多可包含 8 行。因此，24 行至少需要 3 个叶页面。你可以在图 13-18 所示的 `sys.dm_db_index_physical_stats` 输出中确认这一点。

**图 13-18.** 填充因子设置为默认值 0

请注意 `avg_page_space_used_in_percent` 是 100%，因为默认的 `fill factor` 允许页面中压缩最大行数。由于一个页面无法包含部分行来完全填满页面，因此即使使用默认的 `fill factor`，`avg_page_space_used_in_percent` 通常也会略低于 100%。

为了减少由 `INSERT` 和 `UPDATE` 操作引起的初始页拆分频率，通过使用如下所示的 `fill factor` 重新创建聚集索引，在叶（或数据）页面中创建一些空闲空间：

```sql
ALTER INDEX FillIndex ON dbo.Test1 REBUILD
WITH (FILLFACTOR = 75);
```

因为每个页面的总空间可容纳 8 行，75% 的 `fill factor` 将允许每页 6 行。因此，对于 24 行，叶页面的数量应增加到 4 个，如图 13-19 所示的 `sys.dm_db_index_physical_stats` 输出。

**图 13-19.** 填充因子设置为 75

请注意 `avg_page_space_used_in_percent` 大约为 75%，这正是由 `fill factor` 设置的。这允许在每个页面中再插入两行而不会导致页拆分。你可以通过向第一组 6 行（`C1 = 100 - 600`，包含在第一页中）添加两行来确认这一点。

```sql
INSERT INTO dbo.Test1
VALUES (110, 'a'), --第 25 行
      (120, 'a');  --第 26 行
```

图 13-20 显示了当前的碎片情况。

当处理大型数据库及其关联的索引时，可能有必要使用分区将表和索引拆分到不同的磁盘上。随着分区内数据的变化，分区上的索引也可能变得碎片化。在处理分区索引时，你需要确定是希望作为 `ALTER INDEX` 命令的一部分，对一个、部分或所有分区执行 `REORGANIZE` 还是 `REBUILD`。分区索引无法在线重建。请记住，执行任何影响所有分区的操作都可能是一项代价高昂的操作。

如果指定了索引上的压缩（即使是在分区索引上），你必须确保在执行 `ALTER INDEX` 操作时将其压缩设置为之前的值；如果不这样做，压缩设置将会丢失，你将不得不重新创建索引。这对于非聚集索引尤其重要，因为它们不会从表继承压缩设置。


***图 13-20.** 新增记录后的碎片化情况*

## 第 13 章 ■ 索引碎片化

请注意，尽管索引的填充因子为 75%，但 `平均页面密度（完整）` 已降至 67.49%，其计算方式如下：

`平均页面密度（完整）` = 每页平均行数 / 每页最大行数 = (27 / 5) / 8 = 67.5%

从前面的例子可以看出，填充因子在索引创建时被应用。但之后，随着数据被修改，它就不再有意义。无论填充因子如何设置，每当发生页拆分时，原始页面的行都会分布到两个页面之间，`avg_page_space_used_in_percent` 会相应调整。因此，如果你使用了非默认的填充因子，应确保定期重新应用填充因子以保持其效果。

你可以通过重建索引，或使用 `ALTER INDEX REORGANIZE` 或 `ALTER INDEX REBUILD` 命令（如前所示）来重新应用填充因子。`ALTER INDEX REORGANIZE` 命令会考虑索引创建时指定的填充因子。`ALTER INDEX REBUILD` 命令也会考虑原始填充因子，但如果需要，它允许指定一个新的填充因子。

如果没有对填充因子进行定期维护，无论是默认还是非默认的填充因子设置，索引（或表）的 `avg_page_space_used_in_percent` 最终都会稳定在一个狭窄的范围内。因此，在大多数情况下，无需手动维护填充因子，默认填充因子通常就足够好了。

在决定填充因子时，你还应考虑最后一个方面。即使对于繁重的 OLTP 应用程序，数据库读取次数通常也比写入次数多 5 到 10 倍。指定一个非默认的填充因子可能会降低读取性能，其降低程度与填充因子设置成反比，因为它将键值分散到更广的区域。在数据库级别设置填充因子之前，请使用性能监视器比较 `SQL Server:Buffer Manager:Page reads/sec`（SQL Server:缓冲区管理器:页面读取/秒）计数器与 `SQL Server:Buffer Manager:Page writes/sec`（SQL Server:缓冲区管理器:页面写入/秒）计数器，仅当写入量占读取量的相当大比例（超过 30%）时才使用填充因子选项。此外，优化器在构建执行计划进行选择时会考虑页面数量。如果你的页面数量较多，可能会在计划中看到一些糟糕的选择，导致性能不佳。

## 自动维护

在事务量巨大的数据库中，表和索引会随着时间的推移而产生碎片。因此，为了提高性能，你应该定期检查表和索引的碎片情况，并对碎片化程度高的对象进行碎片整理。你可能还需要考虑工作负载情况，并根据负载以及索引的碎片化级别来决定何时对索引进行碎片整理。你可以按照以下步骤对数据库进行此类分析：

1.  确定当前数据库中需要分析碎片的所有用户表。
2.  确定每个用户表和索引的碎片情况。
3.  根据以下注意事项确定需要碎片整理的用户表和索引：
    *   碎片化程度高，即 `avg_fragmentation_in_percent`（平均碎片百分比）大于 20%
