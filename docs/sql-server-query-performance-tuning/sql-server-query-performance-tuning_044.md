# 第 4 章 ■ CPU 性能分析

`Context Switches/sec`（每秒上下文切换）计数器监控计算机上所有处理器从一个线程切换到另一个线程的综合速率。当运行中的线程自愿 relinquishes 处理器、被更高优先级的就绪线程抢占，或者在用户模式与特权模式之间切换以使用执行体或子系统服务时，就会发生上下文切换。它是计算机上所有处理器中运行的所有线程的`Thread:Context Switches/sec`（线程：每秒上下文切换）之和，以切换次数进行度量。

每个处理器每秒 5,000 次`Context Switches/sec`属于优秀到良好的范围。高数值很大程度上取决于你的 CPU 速度，因此需要随时间测量性能，并将此数字与你的基线进行比较，以了解何时可能出现偏离。

### 批处理请求/秒

`Batch Requests/sec`（每秒批处理请求）为你提供了一个很好的指标，显示了系统上承受的负载量，这与处理器上承受的负载有直接关联。因为你的系统上可能看到大量低成本查询，或者少量高成本查询，所以你不能孤立地看这个数字，而必须参考本节中定义的其他计数器；每秒 10,000 个请求可以被认为是繁忙的系统。更大的数值可能值得关注，但这完全取决于你的系统正常情况是怎样的。了解什么值在你自己的系统中具有意义的最佳方法是建立一个基线，然后从那里开始监控。请记住，这里的高数字不一定需要担心。如果你的所有其他资源都控制得当，并且你持续维持着较高的`batch requests/sec`，这仅仅意味着你的服务器很忙。

### SQL 编译/秒

`SQL Compilations/sec`（每秒 SQL 编译）计数器在其聚合中同时显示了批处理编译和语句重编译。

当服务器首次启动（或在故障转移或任何其他启动类型事件之后），这个数字可能非常高，但它会随着时间的推移趋于稳定。一旦稳定后，与基线度量相比，编译出现显著或持续的峰值就值得关注了，并且肯定会在处理器上表现为问题，因为查询编译是一项昂贵的操作。如果你正在使用某种对象关系映射引擎，例如`nHibernate`或`Entity Framework`，高编译次数可能是正常的，尽管成本同样高昂。第 14 章详细介绍了 SQL 编译。

### SQL 重编译/秒

`SQL Recompilations/sec`（每秒 SQL 重编译）是对批处理和语句重编译的度量。大量的重编译可能导致处理器压力。因为语句重编译是此计数的一部分，所以它可能比 SQL Server 2005 之前的版本高得多。第 17 章详细介绍了查询重编译。

## 其他测量 CPU 性能的工具

你也可以使用 DMO 来捕获有关 CPU 的信息。这些 DMO 中的信息需要通过运行查询来捕获，然后将其作为基线度量的一部分保留下来。

### `Sys.dm_os_wait_stats`

等待统计信息是了解系统是否存在瓶颈的好方法。不过，你不能简单地认为大于`x`就是一个坏数字。你需要随时间收集指标，以了解你的系统上什么是正常的。偏离正常值的部分才值得关注。针对此 DMO 查询信号等待时间可以指示 CPU 瓶颈。

### `Sys.dm_os_workers` 和 `Sys.dm_os_schedulers`

这些 DMO 显示 Windows 操作系统内的工作线程和调度器线程。定期针对这些运行查询将使你能够获取处于可运行状态的进程数量。这是处理器负载的一个极佳指示。

## 处理器瓶颈解决方案

一些常见的处理器瓶颈解决方案如下：

- 优化应用程序工作负载



• 消除或减少过度的编译/重编译

• 使用更多或更快的处理器

• 不运行不必要的软件

让我们依次考虑这些解决方案。

### 优化应用程序工作负载

要识别处理器密集型的查询，请使用扩展事件会话（我将在下一章讨论）捕获所有 SQL 查询，然后按 CPU 列对输出进行分组。消耗 CPU 时间最多的查询对 CPU 压力的贡献最大。随后，您应该分析并优化这些查询，以减轻 CPU 的压力。通常，导致 CPU 压力的原因并非查询内的大量计算，而是逻辑 I/O 中的争用。解决 I/O 问题通常也能帮助您解决 CPU 问题。您也可以直接查询 `sys.dm_exec_query_stats` 或 `sys.dm_exec_procedure_stats` 动态管理视图，以实时查看即时问题。最后，通过使用查询哈希和查询计划哈希，您可以识别并调整常见的查询或常见的执行计划（这将在第 14 章详细讨论）。本书的大部分后续章节都与优化应用程序工作负载有关。

### 消除过度的编译/重编译

一定数量的查询编译和重编译是不可避免的，特别是如前所述，当使用 ORM 工具时。只有当这些情况在持续期间内大量发生时，才意味着存在问题。同样值得注意的是它们之间的比例。编译数量高而重编译数量低，意味着系统中查询重用较少（查询重用在第 9 章详细讨论）。大量的重编译将导致处理器使用率过高。解决重编译的方法在第 17 章介绍。

### 使用更多或更快的处理器

最容易的解决方案之一，也是您大多数时候会采用的方案，是增加系统处理能力。然而，由于处理器升级成本高昂，您应首先尽可能优化 CPU 密集型操作。

可以通过提升单个处理器的性能或增加处理器数量来提高系统的处理能力。当 `% Processor Time` 计数器高而 `Processor Queue Length` 计数器低时，提升单个处理器的性能是合理的。当 `% Processor Time` 计数器和 `Processor Queue Length` 计数器都高时，则应考虑增加处理器数量。增加处理器数量允许系统同时执行更多请求。

### 不运行不必要的软件

公司政策通常要求在服务器上安装病毒检查软件。服务器上也可能运行其他产品。在可能的情况下，同一台服务器上不应运行与 SQL Server 无关的不必要软件。与维护 Windows Server 或 SQL Server 无关的外部应用程序最好放置在不同的机器上。

## 网络瓶颈分析

在 SQL Server OLTP 生产环境中，很少有性能问题是由网络问题导致的。您在 OLTP 环境中遇到的大多数网络问题实际上是硬件或驱动程序限制，或者是交换机或路由器的问题。其中大多数问题最好使用网络监视器工具进行诊断。但是，性能监视器也提供了收集网络活动数据的对象，如表 4-2 所示。

---|---|---|---
`Network Interface(网卡)` | `Bytes Total/sec` | 网卡上字节传输的速率 | 平均值 < 网卡容量的 50%，但需与基线比较
