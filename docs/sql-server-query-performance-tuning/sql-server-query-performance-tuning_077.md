# 第 8 章 ■ 索引架构与行为

为了理解宽聚集索引对非聚集索引的影响，将聚集索引列 `C2` 的数据类型从 `INT` 修改为 `CHAR(500)`。

```sql
DROP INDEX dbo.Test1.iClustered;
ALTER TABLE dbo.Test1 ALTER COLUMN C2 CHAR(500);
CREATE CLUSTERED INDEX iClustered ON dbo.Test1(C2);
```

再次运行针对 `sys.dm_db_index_physical_stats` 的查询，将返回 `图 8-20` 中的结果。

`图 8-20. 宽索引的索引页数量`

你可以看到，宽聚集索引增加了非聚集索引行的宽度。由于非聚集索引行宽度很大，一个 8KB 的索引页无法容纳所有的索引行。相反，需要两个索引页来存储全部 20 个索引行。对于大型表而言，由于大的聚集索引键大小而导致的非聚集索引尺寸膨胀，会显著增加非聚集索引的页数。

因此，大的聚集索引键大小不仅影响其自身的宽度，还会加宽表上所有的非聚集索引。这会增加表上所有索引的索引页数量，从而增加索引所需的逻辑读取和磁盘 I/O。

##### 一步重建聚集索引

由于非聚集索引依赖于聚集索引，使用独立的 `DROP INDEX` 和 `CREATE INDEX` 语句重建聚集索引会导致所有非聚集索引被重建两次。为避免此情况，请使用 `CREATE INDEX` 语句的 `DROP_EXISTING` 子句，以单个原子步骤重建聚集索引。

类似地，你也可以对非聚集索引使用 `DROP_EXISTING` 子句。

值得注意的是，在 SQL Server 2005 及更高版本中，当你直接重建聚集索引时，不会看到非聚集索引也被重建。

##### 尽可能使聚集索引唯一

因为聚集索引用于存储数据，你必须能够找到每一行。虽然聚集索引在其定义和存储层面不一定是唯一的，但如果键值不唯一，SQL Server 将无法找到这些行，除非有一种方法可以使聚集索引唯一地标识每个独立数据行的位置。因此，SQL Server 会向非唯一的聚集索引添加一个值以使其唯一。这个值称为 `唯一化值`。正如前面提到的，它会增加你的聚集索引以及所有非聚集索引的大小。
