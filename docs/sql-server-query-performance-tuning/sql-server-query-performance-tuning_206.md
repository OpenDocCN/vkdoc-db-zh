# 面向性能的数据库设计考量

数据库设计是一个宽泛的主题，无法在一本查询调优书籍的简短章节中详尽阐述；尽管如此，我建议您关注以下几个设计方面，以确保从早期阶段就重视数据库性能：

*   平衡欠规范化与过度规范化
*   利用实体完整性约束
*   利用域和参照完整性约束
*   采用索引设计的最佳实践
*   避免在存储过程名称中使用 `sp_` 前缀
*   最小化触发器的使用
*   考虑将表放入内存存储

### 平衡欠规范化与过度规范化

在设计数据库时，您有两个极端的选项：

*   将完整数据保存在单个扁平表中，几乎没有规范化。
*   通过将每个属性拆分成自己的表，从而以细粒度的表保存数据，允许每个属性保存无限数量的多个值。

请别误会。有些场景非常适合使用 NoSQL 风格的数据库，采用键值对机制进行存储甚至检索。但是，关系型存储仍然有其必要性。合理的规范化能提升数据库性能。拥有大量列的宽表通常是欠规范化数据库的特征。`欠规范化`会导致数据过度重复，可能引发错误结果，并常常损害查询性能。例如，在一个订单系统中，您可以将客户资料和该客户的所有订单保存在单个表中，如表 26-1 所示。

**表 26-1. 原始客户表**

| CustID | Name | Address | Phone | OrderDt | ShippingAddress |
| :--- | :--- | :--- | :--- | :--- | :--- |
| Liu Hong | Boise, ID, USA | 123-456-7890 | 08-Jul-04 | Boise, ID, USA |
| Liu Hong | Boise, ID, USA | 123-456-7890 | 10-Jul-04 | Austin, TX, USA |

将客户资料和订单信息一起保存在单个表中，会导致客户资料在该客户的每个订单中重复出现，使得表中的行非常宽。结果，一个数据页中可以保存的客户资料数量就更少了。对于一个只关心一系列客户资料（而非其订单信息）的查询，与将客户资料单独保存在一张表的设计相比，需要读取更多的数据页。此外，由于所有数据都在一个大表中，您会看到更多的锁和并发问题，因为更多人会更频繁地访问同一页或同一行中的相同数据。这一点尤其明显，因为每页存储的数据行会更少（因为行更宽）。为了避免欠规范化的性能影响，您必须将具有 1:N 关系的两个逻辑实体（例如，客户资料和订单）规范化到单独的表中，如表 26-2 和表 26-3 所示。

**表 26-2. 新的客户表**

| CustID | Name | Address | Phone |
| :--- | :--- | :--- | :--- |
| Liu Hong | Boise, ID, USA | 123-456-7890 |

**表 26-3. 订单表**

| CustID | OrderDt | ShippingAddress |
| :--- | :--- | :--- |
| 08-Jul-04 | Boise, ID, USA |
| 10-Jul-04 | Austin, TX, USA |

是的，这些表还有进一步规范化的可能；然而，这取决于您和您的业务部门来决定是否需要这样做。

同样，过度规范化对查询性能也不利。`过度规范化`会导致在过多狭窄的表之间进行过多的连接。对一个表中基数的错误估计可能会在连接时严重影响大量其他表。尽管 20 个表的连接可以运行得很好，而 2 个表的连接也可能出问题，但一个好的经验法则是：当连接条件涉及的表超过 8 到 12 个时，应更仔细地检查该查询。这并非意味着低于这个数量就是好的，高于就是坏的；但是，这个连接数量应该作为一个评估的标志。为了从数据库中获取任何有用的内容，数据库开发人员必须在 SQL 查询中连接大量的表。例如，如果您为客户的姓名、地址和电话号码分别创建了单独的表，那么您将不得不连接至少三个表来检索客户信息。如果数据（例如，客户姓名和地址）具有一对一关系，并且通常被查询一起访问，那么将数据规范化到单独的表中反而会损害查询性能。

### 利用实体完整性约束

`数据完整性`对于确保数据库中的数据质量至关重要。数据完整性的一个基本组成部分是`实体完整性`，它将一行定义为特定表中的唯一实体；也就是说，表中的每一行都必须是唯一可标识的。用作表中唯一行标识符的列（或多列）必须表示为该表的主键。

有时，一个表可能包含一个额外的列（或多列），它们也可以用来唯一标识表中的某一行。例如，一个 `Employee` 表可能有 `EmployeeID` 和 `SocialSecurityNumber` 列。

`EmployeeID` 列作为唯一的行标识符，可以定义为 `主键`。同样，`SocialSecurityNumber` 列可以定义为 `候选键`。在 SQL Server 中，可以使用唯一约束来定义候选键，它们本质上是主键的“小兄弟”。实际上，唯一约束和主键约束在后台都使用唯一索引。

值得注意的是，关于使用自然键（例如前面示例中的 `SocialSecurityNumber` 列）还是代理键（例如 `EmployeeID` 列），存在着真诚的分歧。

我见过两种设计都取得了成功，但每种方法都有其优缺点。与其建议选其一，我不如给您几个同时使用它们的理由，以及与每种方法相关的一些成本，从而避免陷入教条式的争论。标识列通常是 `INT` 或 `BIGINT` 类型，这使得它更窄，更容易建立索引，从而提高性能。此外，将主键的值与任何业务知识分离，在某些圈子里被认为是良好的设计。这种方法的一个缺点是，这些数字有时会被赋予业务含义，而这绝不应该发生。另一件需要记住的事情是，您必须为候选键创建唯一约束，以防止在不应该存在重复的情况下创建多行。这增加了您必须存储和维护的信息量。自然键提供了清晰、人类可读、具有真实业务含义的主键。它们往往是更宽的字段——有时非常宽——使得它们在索引内的效率较低。此外，有时数据可能会发生变化，这会在您的数据库和企业中产生深远的连锁影响。

我只想重申，两种方法都可以很好地工作，并且每种都提供了大量的调优机会。只要应用和维护得当，任何一种方法都能保护您数据的完整性。
