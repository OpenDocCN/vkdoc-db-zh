# 第三章：磁盘性能分析

磁盘和磁盘子系统（包括控制器、连接器和管理软件）是任何计算系统中最慢的部分之一。多年来，内存变得越来越快，CPU 也是如此。但磁盘，除了我们最近看到的一些如固态硬盘（SSD）等技术的显著改进外，变化并不大；磁盘仍然是大多数系统中最慢的部分之一。

这意味着你将希望能够监控你的磁盘以了解其行为。在本章中，你将探讨以下领域：

*   使用系统计数器收集磁盘性能指标
*   使用其他机制收集磁盘行为
*   解决磁盘性能问题

## 磁盘瓶颈分析

SQL Server 可能有苛刻的 I/O 要求，而且由于磁盘速度相对于内存和处理器速度慢得多，磁盘资源的争用会显著降低 SQL Server 性能。分析和解决任何磁盘资源瓶颈都可以显著提高 SQL Server 性能。

`PhysicalDisk`计数器代表物理磁盘上的活动。`LogicalDisk`计数器代表在物理磁盘上创建的逻辑子单元（或分区）。如果在一个物理磁盘上创建两个分区，比如 R:和 S:，那么可以使用逻辑磁盘计数器监控各个逻辑磁盘的磁盘活动。然而，因为磁盘瓶颈最终发生在物理磁盘上，而不是逻辑磁盘上，所以通常更推荐使用`PhysicalDisk`计数器。

请注意，对于硬件独立磁盘冗余阵列（RAID）子系统（更多关于 RAID 的信息，请参阅“使用 RAID 阵列”部分），计数器将整个阵列视为单个物理磁盘。例如，即使 RAID 配置中有十个磁盘，操作系统也会将它们表示为一个物理磁盘，因此您将只有一个针对该 RAID 子系统的`PhysicalDisk`计数器集。同样适用于存储区域网络（SAN）磁盘（具体信息请参阅“使用 SAN 系统”部分）。因此，前表中显示的一些数字可能比系统实际支持的要低（或高）很多。

将这些数字视为监控磁盘的一般准则，并根据技术不断发展、硬件改进可能导致性能差异的事实进行调整。我们正越来越多地使用固态硬盘甚至 SSD 阵列，这使得磁盘 I/O 操作的速度提高了几个数量级。在我们尚未转向 SSD 的领域，我们正在利用 iSCSI 接口。在使用这类硬件时，请记住这些数字更符合传统磁盘驱动器的实际情况，而这些驱动器正在迅速被淘汰。

`% 磁盘时间`
`% 磁盘时间`计数器监控磁盘忙于读/写活动的时间百分比。这是一个很好的负载指标，但不是性能问题的具体指标。请将此信息记录为基本基线的一部分，以便与数值进行比较，从而了解磁盘访问何时发生剧烈变化。

`当前磁盘队列长度`
`当前磁盘队列长度`是在收集性能数据时磁盘子系统上未完成的请求数量。它包括在快照时刻正在处理的请求。一个磁盘子系统只有一个磁盘队列。在包括 RAID、SAN 和其他类型阵列的现代系统中，可能有大量的磁盘和控制器来协助信息的传输。所有这些硬件使得测量磁盘队列长度不如以前重要，但这个度量作为系统负载的指标仍然有用。您需要知道队列长度何时发生剧烈变化，因为这可能是 I/O 问题的一个信号。

但是，与过去不同，现在无法提供一个您可以比较自己系统的固定值。相反，您需要计划从各个系统捕获此信息，并将其用作随时间比较的参考点。

`磁盘传输次数/秒`
`磁盘传输次数/秒`监控磁盘上读和写操作的速率。如今典型的硬盘驱动器对于顺序 I/O（IOPS）每秒可以执行大约 180 次磁盘传输，对于随机 I/O 每秒可以执行大约 100 次磁盘传输。在随机 I/O 的情况下，`磁盘传输次数/秒`较低，因为涉及更多的磁盘臂和磁头移动。OLTP 工作负载（主要进行单例操作、小操作和随机访问的工作负载）通常受限于每秒磁盘传输次数。因此，对于 OLTP 工作负载，您受到限制更多是因为一个磁盘每秒只能执行 100 次磁盘传输，而不是其 1000MB 每秒的吞吐量规格。

■ **注意** 一个 SSD 的 IOPS 范围可以从大约 5,000 到某些高端 SSD 系统的 500,000。您对`磁盘传输次数/秒`的监控需要相应地调整比例。

由于磁盘固有的缓慢，建议尽可能保持每秒磁盘传输次数较低。

`磁盘字节数/秒`
`磁盘字节数/秒`计数器监控在读或写操作期间每秒从/向磁盘传输的字节数。一个以 7200RPM 旋转的典型磁盘每秒可以传输大约 1000MB。通常，OLTP 应用程序不受磁盘子系统传输容量的限制，因为 OLTP 应用程序在单个数据库请求中访问少量数据。如果数据传输量超过磁盘子系统的容量，那么磁盘子系统上就会开始积压，正如`磁盘队列长度`计数器所反映的那样。

同样，对于 SSD 访问，这些数字可能要高得多，因为它主要受限于驱动器到主机接口引起的延迟。

`平均磁盘秒数/读取` 和 `平均磁盘秒数/写入`
`平均磁盘秒数/读取` 和 `平均磁盘秒数/写入` 跟踪从磁盘读取或写入磁盘平均所需的时间（以毫秒为单位）。了解磁盘处理接收到的写入和读取操作的效率如何，可以有力地指示问题所在。如果向/从磁盘移动数据需要超过大约 10 毫秒，您可能需要检查硬件和配置，以确保一切正常工作。

为了使事务日志表现良好，您需要获得更好的响应时间。

**额外的 I/O 监控工具**
与所有其他工具一样，您需要将从性能监视器收集的信息与其他来源可用的数据进行补充。关于 I/O 和磁盘问题的真正有用的信息都在 DMO（动态管理对象）中。

`Sys.dm_io_virtual_file_stats`
这是一个返回有关构成数据库的文件信息的函数。您可以像下面这样调用它：

```sql
SELECT *
FROM sys.dm_io_virtual_file_stats(DB_ID('AdventureWorks2012'), 2) AS divfs;
```

它返回有关文件的几个有趣的信息列。最有趣的是停滞数据，即用户等待不同 I/O 操作的时间。首先，`io_stall_read_ms`表示用户等待读取的时间量（以毫秒为单位）。然后是`io_stall_write_ms`，它显示在该数据库文件上写操作必须等待的时间量。您还可以查看通用数字`io_stall`，它表示文件上所有 I/O 等待。为了使这些数字有意义，您还需要获取另一个值`sample_ms`，它显示了测量的时间量。您可以将此值与其他值进行比较，以了解 I/O 问题在多大程度上阻碍了您的系统。此外，您可以将其缩小到特定文件，从而知道是什么减慢了日志或特定数据文件的速度。这是确定是否存在 I/O 瓶颈的极其有用的度量。但它在识别具体瓶颈方面帮助不大。

`Sys.dm_os_wait_stats`
这是一个通常有用的 DMO，它显示有关系统上等待的汇总信息。要确定是否存在 I/O 瓶颈，您可以利用这个 DMO 进行查询，如下所示：

```sql
SELECT *
FROM sys.dm_os_wait_stats AS dows
WHERE wait_type LIKE 'PAGEIOLATCH%';
```
