# 3. 关系理论与查询优化

## 关系操作与等价规则

既然我们已经介绍了主要的关系操作，你可能会觉得少了点什么：连接操作在哪里？我们知道连接操作至关重要。答案其实显而易见：一个连接操作可以表示为一个笛卡尔积后接一个过滤操作。从关系理论的角度来看，连接操作是冗余的。这是*声明式语言*工作方式的完美示例；形式化定义是寻找连接结果的一种方式（但不是唯一方式）。如果我们计算两个关系的笛卡尔积，然后应用一个过滤器，我们将得到期望的结果。但希望没有数据库引擎会在较大的数据集上采用这种方法；这可能真的需要花费数年时间！在第 3 章中，我们将讨论如何比基于形式化定义的直接计算更高效地实现连接。

关系操作还包括分组、并集、交集和差集。

我们进行优化所需要的最后一个关系理论部分是*等价规则*。所有关系操作都满足若干等价规则，包括：

*   **交换律** – `JOIN(R,S) = JOIN(S,R)`
    交换律意味着两个关系的顺序并不重要。如果我们有两个关系 R 和 S，那么 `R JOIN S` 将产生与 `S JOIN R` 相同的结果。

*   **结合律** – `JOIN(R, JOIN(S,T)) = JOIN(JOIN(R,S), T)`
    结合律意味着，如果我们有三个关系 R、S 和 T，我们可以选择先执行 `R JOIN S`，然后将结果与 `T` 连接；或者我们可以先执行 `S JOIN T`，然后将第一次连接的结果与 `R` 连接，两种情况下的结果将是等价的。

*   **分配律** – `JOIN(R, UNION(S,T)) = UNION(JOIN(R,S), JOIN(R,T))`
    分配律意味着，如果我们将一个关系与另外两个关系的并集进行连接，其结果将等同于分别执行两次连接（`R JOIN S` 和 `R JOIN T`），然后对结果取并集。

前面列出的等价规则只是数十条规则中的几个例子。为什么了解这些规则很重要？为了效率，可能需要以与列出顺序不同的顺序执行操作。后续章节中将有很多此类转换的示例。等价性确保一个查询可以用几种不同的表达式表示，这为优化器提供了动力。

## 逻辑操作

表示 SQL 查询所需的逻辑操作集合包括所有关系操作，但其语义不同。如前所述，SQL 的 `project` 操作不会删除重复项。因此包含了一个额外的删除重复项的操作。

还需要其他额外操作来表示在关系理论中无法表达的 SQL 结构，例如，左连接、右连接和全外连接产生的结果不是一个关系（但仍然是一个 SQL 表）。

许多等价规则对逻辑操作也有效。对于任何相对复杂的查询，优化器可以从海量的表达式中选择最佳的一个。更多关于关系理论的信息可以在章末注释的参考资料中找到。

## 查询作为表达式：集合式思维

编写声明式查询对人类来说并非易事。人们更熟悉的是动作，而不是规则或条件。采用集合式思维^(²) 会更容易：我们可以思考对表的操作，而不是对单个对象（或行）的操作。

前面提到的所有逻辑操作都可以用 SQL 轻松表达。这些操作接受表作为参数，无论是存储在数据库中的表，还是先前操作结果输出的表。

一个作为 SQL 查询编写的 PostgreSQL 表达式将由优化器处理，并很可能被另一个使用前面讨论的等价规则的等价表达式所替换。

由于任何关系操作的结果都是一个关系，它可以直接传递给下一个关系操作，而无需中间存储。一些数据库开发人员选择创建临时表来存储中间结果，但这种做法会产生不必要的计算开销并阻碍优化器。

用更理论化的语言来说，上一段指出优化器产生高效执行计划的能力取决于两个因素：

*   丰富的等价规则集提供了大量等价表达式的空间。
*   关系操作不产生副作用，例如临时表——也就是说，产生的唯一东西就是操作的结果。

## 操作与算法

为了使查询可执行，逻辑操作必须被替换为物理操作（也称为算法）。在 PostgreSQL 中，这种替换由查询规划器执行，查询的整体执行时间取决于选择了哪些算法以及选择得是否正确。

当我们从逻辑层转向物理层时，数学关系被转换为存储在数据库中的表，我们需要确定从表中检索数据的方法。任何存储的数据都必须使用下一章讨论的*数据访问算法*之一来提取。通常，数据访问算法与消耗其结果的操作相结合。

更复杂的逻辑操作，如连接、并集和分组，可以用几种替代算法来实现。有时，一个复杂的逻辑操作会被替换为多个物理操作。

这些算法在第 3 章中有详细讨论。

## 总结

数据库引擎通过将 SQL 查询解析为逻辑计划、转换结果、选择算法来实现逻辑计划，最后执行所选算法，来解释 SQL 查询。数据库引擎使用的逻辑操作基于从关系理论衍生的操作，理解这些对于像数据库一样思考至关重要。
