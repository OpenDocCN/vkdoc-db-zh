# 15. 更复杂的过滤与搜索

前面的章节讨论了几种在 PostgreSQL 中使用索引支持过滤和搜索的方法。为什么还需要更多？为什么之前没有介绍这些索引？先前关于这个话题的讨论集中在最常见的索引上，即几乎所有应用程序都需要的那些。然而，有些数据类型无法用 `B-tree` 等索引来高效支持。


## 全文搜索

前面章节讨论的所有内容都适用于结构化数据，并且迄今为止考虑的所有查询都是布尔型的。也就是说，一行数据要么需要用于计算结果，要么不需要；计算出的行要么属于输出，要么不属于。不存在中间状态。SQL 是一种用于处理结构化数据的强大语言，非常适用于这类分析。

在本节中，我们将探讨非结构化数据。最常见的非结构化数据例子是用自然语言写成的文本。这类文本通常被称为 `document`（文档）。与结构化数据不同，文档搜索总是不精确的，因为我们通常感兴趣的是文档的含义，而这种含义并未在文档内容中精确表达。然而，搜索标准必须在查询中精确表达。欢迎来到不确定的世界！

文档搜索有几种不同的模型；PostgreSQL 中实现的模型称为布尔模型。请注意，现代互联网搜索引擎使用更复杂的模型。

在布尔搜索模型中，一个文档被视为一个 `terms`（词项）列表。每个词项通常对应文档中的一个单词，但会使用特定的语言工具进行转换。这种转换是为了提高搜索质量所必需的。例如，我们期望单词 “word” 和 “words” 应映射到同一个词项。这些转换并非易事：“a leaf” 和 “leaves” 是同一个单词的不同形式；然而，“to leave” 则是不同的。而且这不仅仅是词形变化：含义取决于上下文。例如，在描述网络协议的文档中，“host” 和 “computer” 具有相同的含义，但在与组织会议相关的文档中，这些词是不同的。

在 PostgreSQL 中，定义这些转换的语言规则被封装在一个 `configuration`（配置）中。有几种针对不同语言的预定义配置可用，也可以定义额外的配置。其中一个预定义配置不依赖于任何语言。

这种语言处理的输出表示为 `ts_vector` 类型的值。`ts_vector` 的值是词项列表，与任何语言甚至文本无关。一个 `ts_vector` 可以从任何值列表构建。

为什么文本搜索被称为 `full text search`（全文搜索）？在黑暗时代（20 世纪 70 年代），硬盘容量很小，词项列表是从标题或摘要中构建的。因此，全文意味着文档中的所有单词都被视为词项的来源。

类似地，文档搜索的查询表示为 `ts_query` 类型的值。这些值由查询的文本表示构造而成，该表示可以包含单词以及逻辑运算符 `AND`、`OR` 和 `NOT`。一个简单的查询仅由单词组成（单词之间隐含着 `AND`）。

查询中的一个词项（单词）如果存在于文档中则产生 `true`，否则产生 `false`。词项的布尔值使用查询中包含的运算符组合成一个布尔值。对于仅包含词项的简单查询，如果查询中的所有词项都存在于文档对应的 `ts_vector` 中，则该文档匹配。

匹配运算符 `@@` 如果文档满足查询则返回 `true`，否则返回 `false`。它可以用在 `SELECT` 语句的 `WHERE` 子句中，或任何需要布尔表达式的地方。

布尔搜索产生明确的结果：文档要么匹配查询，要么不匹配。这仍然不确定吗？是的，仍然不确定。当文档转换为 `ts_vector` 时，一些信息会丢失；当查询转换为 `ts_query` 时，一些信息同样会丢失。

PostgreSQL 的全文搜索功能可以在没有任何索引的情况下工作。PostgreSQL 也提供了可以加速文本搜索的特殊索引类型。我们将在本章后面讨论这些索引类型。

## 多维和空间搜索

前面章节考虑的过滤条件使用了标量属性。某些类型的索引，例如复合索引，可能包含多个属性，但属性的顺序至关重要：如果未指定第一个属性的值，索引就毫无用处。

对于某些应用程序和数据类型来说，属性的固定顺序是不理想的。这些应用程序要求对多个属性进行对称处理，不偏向任何特定属性。典型的例子是平面或三维空间中具有坐标属性的对象。这类数据被统称为空间数据。

更重要的是，空间数据通常需要不同类型的查询。最常见的查询是：
*   *范围查询* – 查找位于空间中某指定点一定距离内或更近的所有对象。
*   *最近邻查询* – 查找最接近指定点的 k 个对象。
即使使用多个索引，这些查询也无法通过一维索引来支持。

当然，这类搜索不仅限于空间。坐标可以带有时间戳值，甚至是离散域中的值。

PostgreSQL 提供了适合空间数据的索引类型。这些索引将在下一节简要讨论。

## PostgreSQL 中的广义索引类型

`CREATE INDEX` 操作符在创建索引时包含一个可选的索引类型。到目前为止，前面的例子中尚未演示这一点，因为之前创建的所有索引都是 B-tree 索引，而 B-tree 是索引类型的默认值。其他可能的值（在撰写本文时）包括 `hash`、`GIST`、`spgist`、`GIN`、`BRIN` 以及用户定义类型（如果已安装）。

本节将更详细地讨论其中一些类型。

### GIST 索引

一些应用程序使用多属性对象，例如在表面或平面上具有坐标的点。之前我们讨论了包含多个属性的复合索引。然而，复合索引不是对称的：属性是有序的。相比之下，点的坐标应该被对称处理。

GIST 是一个索引结构家族，其中的每一个都支持特定的数据类型，并且可以配置为实现几种不同的基于树的索引结构。具体来说，它为空间数据实现了称为 R 树 的索引结构。PostgreSQL 发行版包含了对 R 树 和其他几种索引的支持；更多类型可以作为扩展安装。

作为 `GIST` 索引类型实现的 R 树 正是此处所需：它为空间对象（如点和矩形）建立索引。形状更复杂的对象在索引中使用矩形边界框表示。搜索条件用矩形表示。所有与搜索矩形相交的对象都作为搜索结果返回。

当然，包含在 `GIST` 索引中的属性不一定是坐标。这些属性也可以表示时间或其他有序数据类型。

### 全文搜索的索引

PostgreSQL 提供了两种可用于支持文本搜索的索引类型。首先考虑 `GIN` 索引，其中 `GIN` 代表广义倒排索引（Generalized Inverted Index）。

就索引而言，一个文档被视为一个词条（或词元）列表，例如前面讨论过的 `ts_vector` 数据类型的值。

对于至少出现在一个文档中的每个词条，倒排索引包含一个包含该词条的文档列表。因此，整体结构是对称的：一个文档有一个词条列表，而一个词条有一个文档列表。这种对称性解释了为什么这种索引类型被称为“倒排”。

倒排索引可以高效地支持文本搜索。例如，要查找包含查询中指定的所有词条的文档，PostgreSQL 会扫描这些词条的所有文档列表，并只保留出现在查询中所有词条列表中的文档。这些列表是有序的，因此只需对列表进行一次遍历就足以生成结果集。

`GIN` 索引可以作为函数索引创建，其表达式将被索引的文档转换为 `ts_vector`，或者 `ts_vector` 的值可以作为单独的列存储。前者的优点是占用空间较少，后者的优点是索引不依赖于配置（因为配置仅用于计算 `ts_vector` 的值）。如果存储了 `ts_vector` 的值，索引可以引用用不同自然语言编写并使用不同配置转换为 `ts_vector` 的文档。通常，预计算的 `ts_vector` 性能会更好，但这始终取决于用于生成向量的函数和系统配置。

`GIN` 的结构并非源自自然语言或与之相关；如前所述，它将文档视为列表。因此，它可以处理文档以外的数据——实际上，任何包含多个值的属性类型都可以，例如数组。`GIN` 索引会找到所有包含查询中指定的所有值的多值属性的行，就像它找到包含指定词条的所有文档一样。

文档（`ts_vector` 类型的值）也可以用 `GIST` 进行索引。要构建这样的索引，`ts_vector` 的值会被转换为固定长度的位图。为了构造位图，出现在任何被索引文档中的每个词条都会被哈希处理，以获取一个代表其在位图中位置的数字。每个词条都是一个相同长度的位图，其中只有一位等于 1（代表该词条），所有其他位都设置为零。一个文档的位图是该文档中所有代表词条的位图的按位逻辑 `OR` 运算结果。因此，文档位图在出现在该文档中的词条对应的所有位置上都有一位等于 1。查询位图以类似方式构造。

基于此索引的搜索基于以下事实：如果文档的位图在查询位图有 1 的每个位置上都包含 1，则该文档满足查询。

不同的词条可能被哈希到相同的位置。因此，`GIST` 索引可能返回与查询无关的文档；通常需要重新检查 `ts_vector` 值，但 PostgreSQL 可以自动识别这一点。

随着不同词条数量的增加，错误匹配的数量也会增加。因此，`GIST` 索引对于不同词条总数较小的文档集合是高效的。这在自然语言文本中并不常见，所以 `GIN` 索引在这种情况下通常更高效。用于文本搜索的 `GIST` 索引在特殊情况下仍然有用。

### 索引非常大的表

任何索引都会占用一定空间，大表上的索引可能非常大。是否有可能减小索引的大小？

数据库教科书区分了稠密索引和稀疏索引。到目前为止介绍的所有索引都是稠密的；也就是说，它们包含被索引列（或列）的所有值。稀疏索引只包含所有值的一小部分，但仍然减少了查找被索引属性任何值所需的读取次数。这与不加速搜索未包含在索引中的值的条件索引不同。

一些数据库系统总是按照（代理）主键的顺序存储表行。主键上的索引然后可以为每个表块只包含一个值，因此它可以是一个稀疏索引。更高级的数据库系统支持基于其他属性值对表进行排序，这些值不一定需要是唯一的。这种表组织方式也为稀疏索引提供了条件。有时这些索引被称为聚簇索引，因为具有被索引列相同值的行被放置在一起。

PostgreSQL 没有提供任何显式控制行顺序的方法。然而，在许多情况下，行是自然排序的。例如，记录特定类型事件的行很可能会附加到表中，并且会自然地按照到达时间戳排序，这使得稀疏索引成为可能。

PostgreSQL 中实现的稀疏索引的泛化形式称为 `BRIN`（块范围索引）。创建了 `BRIN` 索引的表被视为一系列块范围，每个范围由固定数量的相邻块组成。对于每个范围，`BRIN` 索引条目包含该块范围中包含的列值的摘要。例如，摘要可能包含事件日志表中时间戳列的最小值和最大值。

要查找被索引属性的任何值，只需找到适当的块范围（使用索引），然后扫描该范围内的所有块即可。

摘要方法的结构取决于被索引列的类型。对于区间，摘要可能是一个包含块范围中所有区间的区间。对于空间数据，摘要可以是一个包含块范围中所有框的边界框。

如果列值未排序或行在表中未排序，扫描 `BRIN` 索引将返回多个需要扫描的块范围。

摘要操作是昂贵的。因此，PostgreSQL 为 `BRIN` 索引维护提供了多种选择：`BRIN` 索引可以通过触发器自动更新；或者，延迟的摘要可以自动与 vacuum 操作一起完成，也可以手动启动。
