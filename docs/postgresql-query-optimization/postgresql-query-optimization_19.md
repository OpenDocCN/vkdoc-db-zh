# 第 17 章 结论

## 等等——还有更多内容！

在本书中，我们始终坚持数据库优化不仅限于优化单个查询；单个查询并非凭空而来。尽管如此，前几节描述的优化算法只是指导如何优化单个查询——或者更准确地说，是如何从一开始就以正确的方式编写查询。然而，我们还介绍了一些其他技术。

以下是其他需要考虑的事项：

*   **参数化**——很可能你正在优化的查询是参数化的，也就是说，如果对 `flight_id` 有条件筛选，它不会总是 `flight_id=1234`，而可能是任意 `flight_id`。正如我们在第 5 章讨论的，根据具体的筛选值，最具限制性的条件可能会有所不同（例如，“已取消”的航班状态将比大多数其他条件更具限制性）。

*   **动态 SQL**——在后一种情况下，正确的方法是使用动态 SQL，当筛选条件本身变化时也是如此。

*   **函数**——如第 12 章所述，函数本身并不能提高性能，并且可能显著降低执行时间。然而，如果需要动态 SQL，则很难避免使用函数。

*   **数据库设计变更**——在处理查询时，你可能会觉得需要进行一些 DDL 变更，从创建新索引到更改表结构。你需要与 DBA 和系统架构师合作，以确定哪些变更可以实施。

*   **与应用程序的交互**——如果你的查询由应用程序执行，查询性能可能很好，但整体应用程序性能可能不佳。如果你和你的团队选择使用 NORM 或其他类似方法，你将需要与应用程序开发人员合作，确定哪些属于业务逻辑，哪些属于数据库逻辑，如第 11 章和第 13 章所述。

## 总结

本章提供了一个分步指南，帮助你掌握如何从一开始就正确编写查询。我们鼓励你在下一个项目中尝试并遵循这些步骤。

如同所有美好的事物一样，本书也即将结束。

在引言中，我们分享到，我们撰写这本书是因为我们觉得我们*不能不*写它。无数次，我们遇到这个问题：“你能推荐一本适合 `PostgreSQL` 入门的书吗？”

在大多数情况下，提出这个问题的人并非数据库开发新手。`PostgreSQL` 尚未成为大多数教育机构的首选数据库，一个典型的“刚接触 `PostgreSQL`”的开发人员已经知道如何编写语法正确的 `SELECT` 语句，但对 `PostgreSQL` 的细节不熟悉。这不仅包括任何细微的语言差异，更重要的是，还包括数据存储方式和查询处理方式的差异。

是的，当然，文档总是可用的，但除非你确切知道要找什么，否则并不总是容易找到所需内容。其他资源包括许多关于各种主题的优秀教程，以及领先的 `PostgreSQL` 专家的博客。然而，大多数时候，它们专注于特定主题，展示了 `PostgreSQL` 的许多优秀特性，但并不一定指明它们在全局中的确切位置。

当然，本书也没有呈现全貌——`PostgreSQL` 功能丰富，我们并未试图提供详尽的覆盖。相反，我们从不同的角度来探讨 `PostgreSQL`：我们演示如何让这些优秀的功能发挥作用。

话虽如此，我们希望本书能成为数据库开发人员开始探索 `PostgreSQL` 时的首选书籍。我们也希望那些已经使用 `PostgreSQL` 一段时间的人也能从中找到一些有用的信息，也许是一些他们以前未曾使用过的技巧。

我们的目标是为你提供一个可以用来应对数据库开发挑战的结构，以及一个在“如果……”情况下可以咨询的资源。我们希望以本书为指南，能让你更容易地在 `PostgreSQL` 文档中找到更多细节。

除了文档，还有许多其他有价值的资源可用，包括

*   [PostgreSQL Wiki](https://wiki.postgresql.org/wiki/Main_Page)，其中包含数千条有用的提示、使用模式和反模式
*   [*The Art of PostgreSQL*](https://theartofpostgresql.com/)，这是一本提高你的 SQL 技能并学习基于 `PostgreSQL` 应用程序开发的各种功能的好书
*   `PostgreSQL` 专家的博客（[Planet PostgreSQL](https://planet.postgresql.org/) 聚合器是一个很好的资源）
*   [PostgreSQL IRC 聊天](https://www.postgresql.org/community/irc/)

在整本书中，我们不仅试图解释做什么，还解释其原理，因为如果你知道“为什么”，你就能够识别出类似解决方案可能适用的其他情况。理解关系理论是理解这些“为什么”的关键，这就是为什么本书以相当一部分理论开始。对于那些坚持读完理论章节的人，你的努力将得到回报。这些理论基础让你离“像数据库一样思考”更近一步，这使你能够从一开始就正确地编写查询，而不是“先写然后优化”。

此外，我们引入了 `postgres_air` 模式，现已开源，可在 [`https://github.com/hettie-d/postgres_air`](https://github.com/hettie-%25C2%25ADd/postgres_air) 获取。我们希望这个真实的数据集能作为培训、实验和演示工具以及教育资源，对大家有所帮助。

在引言中，我们的目标读者被描述为在 `PostgreSQL` 领域工作的 IT 专业人士，他们希望开发高性能和可扩展的应用程序；任何职位名称包含“数据库开发人员”或“数据库管理员”的人；或任何负责编程数据库调用的后端开发人员。我们希望本书的其中一个收获是，所有这些群体与业务所有者之间的协作是开发高性能应用程序的关键。

数据库查询并非在真空中运行：数据库是一项服务。如果一切运行良好，数据库工作是无形的；如果出现问题，则会极其引人注目。考虑到这一点，我们希望你的工作大部分时候能保持无形！

现在，请尽情享受 `PostgreSQL` 吧！
