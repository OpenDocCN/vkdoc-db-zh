# 3. 用户和策略存储

身份和访问管理系统围绕一个核心存储系统构建，该系统能够维护用户信息、角色、策略和凭据。这些存储系统也可用于存储有关网络资源、外设和应用程序的信息。大多数情况下，目录存储使用基于轻量目录访问协议（`LDAP`）的系统。其他存储方法包括数据库存储和文件存储。然而，`LDAP` 提供了一个更广泛接受的标准，并且与更广泛的应用程序兼容。

`LDAP` 定义了一种标准方法，用于访问有关身份数据、网络资源、外设，甚至加密证书和服务的信息。身份信息可以包含用户凭据、联系信息以及其他用于识别个人的相关数据。这些数据可以被电子邮件等应用程序用于电子邮件地址。或许最有用的功能是提供身份验证和角色定义的能力。应用程序可以使用目录和标准的 `LDAP` 命令来确保用户能够访问系统，并且只能访问他们被授权使用的数据。

## 用户和策略存储概述

存储在 `LDAP` 目录中的数据通常以有组织的方式构建，以便在浏览和查询信息时更易于访问。此外，这些对象符合定义的格式，包括必需和强制的属性。这些属性不仅有助于识别人或对象，还使应用程序能够查询符合特定条件的用户组。

虽然 `LDAP` 并未定义存储什么数据或其结构如何，但它定义了数据的访问方式。基于 `LDAP` 的目录的优势在于，任何平台上运行的应用程序都可以访问它们。这些应用程序可以使用 `TCP/IP`、`LDAP` 协议和 `HTTP` 来查询目录存储。因此，可以通过使用安全套接层（`SSL`）和传输层安全性（`TLS`）来提供额外的安全性。这通过提供实施安全性的标准方法，允许更快地开发应用程序。存在多种用于管理和浏览目录的查询工具。

其他身份存储方法，例如基于文件的安全性和数据库用户存储，未提供标准的数据访问方法。这会导致应用程序开发效率低下，因为应用程序必须考虑到身份存储来开发，并且开发人员必须知道如何访问必要的信息。此外，当组织从传统系统迁移到新产品或新的开发工作时，这可能会导致兼容性问题。

如前所述，用户信息（如凭据和联系数据）可以存储在 `LDAP` 目录中。数据通常以树状结构存储，使组织更容易。在大多数情况下，树的根部定义了组织或安全领域。基本可分辨名称（`DN`）可以设置为类似 `dc=mycompany,dc=com` 的形式，其中 `DC` 代表域组件。这提供了一个有效的组织标识，同时足够高层，可以根据地区、业务部门或企业定义的其他分类来细分较低层级。目录树可能类似于图 3-1。

由于 `LDAP` 并未规定数据结构或粒度，公司在很大程度上可以自由地根据当前和未来的情况，以合理的方式创建目录结构。设计决策可能是基于分布式管理、未来数据分区计划或许多其他原因。

用户和资源条目存储在这些分支中。大多数情况下，用户身份就是这样存储的。例如，用户 John Daily 可能在 mycompany 的美国分公司工作。在第一个示例中，该条目可以通过 `cn=john.daily,ou=US,ou=NorthAmerica,ou=users,dc=mycompany,dc=com` 来唯一标识。这就是所谓的 `DN`。通常使用基于名称的方法来分配 `DN`，例如 `john.daily`。然而，其他组织可能希望使用分配的标识符来创建用户账户，以简化名称更改或防止如果有第二个 John Daily 加入公司时出现重复用户账户。



如果各个条目不包含可用于识别用户的标准化信息，那么整理用户条目几乎没有用处。除了存储联系信息外，用户目录还可用于维护活动数据，例如上次登录时间或上次登录终端。这些数据可用于跟踪历史记录，在更高级的系统中，它还可通过向管理员或用户提示异常活动来识别可能的欺诈行为。与此同时，有关失败登录的信息可用于在尝试次数过多后锁定账户。为确保特定类型的每个条目都包含相同的可用数据，目录可以通过对象类进行配置。

对象类用于定义每个条目必须配置的标准属性。这有助于强制要求必填字段和可选数据，以便在查询用户时，无论返回哪个用户，客户端都能确保检索到预期的结果。可以为各种类型的条目创建对象类，包括用户、网络资源、外设，甚至会议室。对象类的定义由管理员构建，可以简单到只需一两个必填字段，也可以包含子对象类以及必填和可选属性的组合。

例如，对象类可以将一个 `person` 定义为需要 `Surname` 或 `sn`、`Common Name` 或 `cn`、`Given Name` 和 `Date of Birth`。一个额外的对象类 `organizationPerson` 可能会被定义为包含 `manager`、`department`、`telephone number` 和 `office location` 等属性。还可以创建具有不同属性配置文件的更多对象类，例如一个名为 `user` 的类，它可能要求用于身份验证的属性，如 `password` 或 `failedLoginAttempts`。目录中的条目可以使用一个或多个对象类来创建，并且可以定义目录树的整个分支，使得该分支中的所有条目都使用相同的对象类组合创建，从而确保北美组织单元内的所有用户都按照图 3-3 所示的 `Person`、`OrganizationPerson` 和 `User` 属性创建。

应用程序权限和对网络资源的访问授权可以在目录存储中定义。与用户身份类似，角色和权限通常也组织成目录结构。虽然可以单独为用户在应用程序或其他系统中授予权限，但在 `LDAP` 目录中使用组可以提供为多个用户授予相同权限的能力。它还将用户到权限的映射和处理负担转移到了 `LDAP` 目录上。

目录中的组可以由用户条目或其他组构成。存储在目录中后，应用程序可以使用标准的 `LDAP` 应用程序编程接口（`API`）命令，在授予访问请求资源之前验证用户的组成员资格。这些应用程序角色通常直接映射到应用程序内的权限。

企业内多个目录的存在增加了管理工作量和成本。当一个组织维护一个网络用户目录、可能还有两三个应用程序目录，以及其他系统的专有身份目录时，每个目录都需要进行管理以跟上不断变化的局面。这需要为每个目录投入更多的人力资源时间。除了持续管理外，还需要进行设置和配置所需的时间，以及保持所有目录以最佳水平运行所需的计算能力和网络资源。

由于管理多个目录所需的管理任务，各种身份存储变得不一致是非常常见的。在身份生命周期的设置、维护和移除过程中，确保用户账户在多个存储中创建可能变得非常繁琐。这导致了账户在一个目录中创建或删除但在其他目录中没有的可能性。因此，当新用户尝试访问他们尚未获得账户的资源时，速度会减慢。另一方面，这也带来了本应停用的用户仍然可以访问机密信息的可能性。

如前所述，多个目录引发了许多安全问题，这些问题的发生是由于难以保持所有实例同步。除此之外，它还会导致数据不一致。随着目录的管理以及用户条目在多个位置被修改，条目属性可能会变得过时。结果，应用程序可能无法访问关于用户或权限的最新信息。

### Oracle Internet Directory

Oracle Internet Directory (`OID`) 是 Oracle 基于数据库的、兼容 `LDAP` v3 的目录服务器。它不仅能够存储身份数据，还能与现有的其他数据存储进行同步。基于 Oracle 数据库，`OID` 能够利用高级功能，在存储和备份层面提供高水平的安全性。`OID` 的可扩展性支持大量用户和事务，即使在多站点环境中也是如此。高可用性是组织关注的一个关键问题。`OID` 提供了相关机制，帮助 IT 部门为其客户提供极高的可靠性。总体而言，`OID` 为组织提供了其可能需要的所有身份和策略存储选项。

作为目录服务器，`OID` 围绕 `LDAP` v3 兼容性构建。这不仅确保了 Oracle 应用程序可以引用 `OID`，而且其他具有 `LDAP` 兼容性的第三方应用程序和服务也能够利用组织的投资。与 Active Directory 等网络访问目录或特定于应用程序的目录不同，`OID` 是一个通用目录。通用目录遵循一个共同的标准，可以支持不同的应用程序。尽管 Active Directory（一种网络）是 `LDAP` 标准的实现，但它专为网络操作系统（`OS`）定制，并包含网络身份验证和授权所需的属性。它是一个更专业的工具，可能更难以适应通用目的。因此，`OID` 为公司提供了一个整合目录以使用 `OID` 的平台。


### 安全与数据隐私

如果一个组织正在考虑使用 Oracle Identity Management，那么安全必然是其首要关注点之一。从客户端/提供者层一直到数据存储甚至备份层级，都应维持高水平的安全性。在客户端应用和认证层面，OID 支持 `SSL`。`SSL` 确保数据在通过网络发送之前被加密。在此基础上，它提供了相应的机制来确认接收方是经过授权的，对数据进行解密，并确保消息的完整性。

如前所述，以安全方式传输数据可确保消息不会被第三方截获和解读。一旦请求者接收到数据或将其写入磁盘，数据隐私方面的考量要求在这些层面也必须极其谨慎地保护数据。OID 使用 Oracle 数据库作为其数据存储，因此能够利用先进的数据库功能来实现这一目标。

Oracle 透明数据加密 (`TDE`) 为 OID 提供了一种加密在数据库内写入磁盘的数据的方法。当 OID 数据写入数据库时，配置了 `TDE` 的表空间确保数据在写入数据文件时即被加密。随后，当结果集返回给客户端时，数据会被解密。如果存储设备或备份丢失或被盗，加密技术可确保身份数据保持私密和安全。此加密层发生在数据库内部。拥有正确授权并使用已授权应用程序的用户可以正常查看数据，无需任何额外要求。然而，试图绕过应用程序直接从数据库文件读取数据的未经授权人员或进程，将只能看到加密数据。所有这些对应用程序都是透明的，这意味着 OID 不需要任何额外的配置更改即可利用数据库内的此功能。图 3-4 描绘了一个 `TDE` 环境。

身份数据不仅需要防范入侵者的安全，有时还必须防范来自内部的威胁，这些威胁甚至可能包括拥有特权数据库账户的用户。不幸的是，大多数内部数据库账户（如 `sys` 或 `system`）被许多人所知晓，例如数据库管理员 (`DBAs`)。这可能导致对关键数据的未授权访问，即使访问是无意的。Oracle Database Vault 允许管理员有效地将数据库账户锁定在应用数据之外，同时仍允许这些账户执行数据库管理操作。因此，通过防止特权账户访问 OID 使用的私密身份数据，这提高了数据库层的安全性。

即使不在数据库内使用 `TDE` 或 Data Vault，OID 也开箱即用地支持对敏感数据元素进行加密。敏感属性包括 `orclpasswordattribute` 和 `orclrevpwd`。此类值在存储操作期间被加密，并在需要时被解密。如果 OID 内启用了数据隐私模式（这是默认模式），则使用 `AES256` 对这些值进行加密。

### 易用性与管理

本书前面已经讨论过，安全不应成为应用程序性能或用户生产力的瓶颈。提高安全性不应需要复杂的界面或额外的资源来维护。OID 通过使用 Oracle 的 Fusion Middleware Control 界面和 Oracle Directory Service Manager (`ODSM`) 简化了管理任务和环境管理。结合这些图形用户界面 (`GUI`) 和各种命令行选项，可以轻松地管理和监控 OID。这使得故障排除和持续维护更加简单，并使目录管理等日常操作更加直接。

`ODSM` 界面是 Oracle 基于标准 Web 的 `GUI`，可用于管理 OID、Oracle Virtual Directory (`OVD`) 和 Oracle Unified Directory (`OUD`)。该界面可被非管理员用户用作基本的 `LDAP` 浏览器，被管理用户用于管理属性、条目和目录对象，甚至用于管理配置条目。

作为基本的 `LDAP` 浏览器`ODSM` 允许非管理用户能够查看和搜索其他用户。任何经过认证的用户（即存在并输入了有效凭据的用户）都可以浏览目录。组织无需开发自定义应用程序或投资昂贵的第三方产品，即可让其用户利用 OID 搜索目录中其他用户的联系信息。需要注意的是，当非特权用户登录 `ODSM` 界面时，他或她将只能查看“数据浏览器”选项卡，而无法对目录进行任何更改。该用户的视图也将仅限于他或她被授权访问的数据。

管理用户可以将 `ODSM` 用作管理工具。当超级用户账户 `cn=orcladmin` 或 `DirectoryAdminGroup` 的成员登录时，该用户将看到比普通认证用户更多的数据选项。管理用户可以浏览相同的数据，但可以选择创建和修改用户及组条目、管理策略（如密码策略），甚至创建新的对象类和对象类型。这些用户还能够管理系统配置条目。

维护 OID 不仅仅是管理目录和跟踪 `LDAP` 条目。还需要持续监控系统性能、监控系统状态，并管理后台进程，例如目录集成平台 (`DIP`)。Fusion Middleware Control 界面是一个基于 Web 的工具，允许系统管理员执行这些操作。

如果 OID 安装在 WebLogic 域中，`WLS` 提供对控制屏幕的访问。在 Fusion Middleware Control 提供的众多功能中，它允许管理员查看性能摘要、启动和停止环境、查看端口使用情况、获取大小调整建议、设置与其他 `LDAP` 实例的同步和复制，以及管理安全。图 3-5 显示了 Fusion Middleware Control 摘要屏幕。使用此屏幕，管理员可以获得 OID 状态的高级视图。

### 目录同步

多年来，OID 一直是 Oracle 身份管理解决方案以及众多 Oracle 应用程序和融合中间件产品中应用安全的核心，这些产品在设计上都围绕 OID 进行身份验证和授权。正因如此，在那些已部署 Oracle 产品的组织中，OID 已稳固地成为事实上的安全标准。然而，很少有组织愿意从他们的网络目录访问平台（如活动目录）迁移出来。对大多数组织而言，活动目录提供了高级别的网络安全。OID 则允许进行定制，以用作应用或通用身份目录。其由 DIP 提供的同步功能，是关键能力，使组织能够保留在活动目录或其他第三方目录系统上的投资，同时利用 OID 的优势来支撑他们的应用程序。

DIP 使组织能够保持 OID 与不同目录系统的同步。可以为 DIP 创建配置文件，以连接并复制身份数据到 OID。在此复制过程中，集成配置文件允许对身份数据进行转换，以匹配 OID 环境或使用它的应用程序所需的结构和属性。DIP 可以根据组织在性能和目录规模方面的要求进行配置。或许组织拥有一个遍布全球的庞大用户群。重新组织数据结构可能是有意义的。使用 DIP，OID 中的数据结构不一定需要与活动目录中存储的结构相匹配。DIP 支持将多个身份存储同步到单个 OID 实例。由于 OID 支持存储多个上下文，组织可以轻松地组织和来自每个存储的用户。图 3-8 展示了一个示例，说明 DIP 如何将多个不同的目录源整合为一个单一的源。

![A352855_1_En_3_Fig8_HTML.jpg](img/A352855_1_En_3_Fig8_HTML.jpg)

目录同步在出现不一致之前都是一项宝贵的资源。这些不一致最常源于在 OID 中编辑用户或组属性，而真实数据源可能是另一个目录系统，例如活动目录。一个常见的属性是用户密码。用户不希望为不同的系统和应用程序管理多个密码。要求他们这样做会导致安全性削弱，因为用户倾向于将密码写下来或使用更简单的密码。在 OID 中更改一个从活动目录同步过来的用户密码，可能会导致不一致，从而可能阻止用户登录所需资源。OID 能够使用外部身份验证插件和外部密码过滤器来帮助防止这种情况发生。

外部身份验证插件允许管理员配置 OID，将凭证验证移交给该用户的源目录。这样做，OID 不会为任何从源复制的用户维护用户密码。相反，用户属性会触发 OID 将凭证传递到源进行验证。因此，有效用户必须在 OID 和源中都存在，但在 OID 中的密码更改不会影响身份验证，因为唯一有效的密码存储在源中。这样，真实数据源仍然是源目录，密码必须在源目录中更改。DIP 负责保持所有属性同步，而插件处理身份验证，从而提供了一个无需同步密码的、非常安全的环境。使用外部身份验证，如图 3-9 所示，OID 将凭证验证移交给源目录。

## Oracle 统一目录

Oracle 在通过 OID 提供强大的 LDAP 目录方面拥有悠久的历史。最近，此产品已升级为包含 OUD，这是一个完全用 Java 实现的 LDAP 存储库，它提供了许多可扩展性选项，同时还包括同步、代理和虚拟化等功能。

## 架构

OUD 是基于 Java EE 平台从头开始构建的。据 Oracle 称，这使他们能够利用使用 Java 虚拟机提供的内存管理能力和调优。对于数据存储，OUD 利用了 Oracle 的 Berkeley Database Java Edition。这个面向对象的数据库提供了一个稳定的平台，非常适合于映射目录中的条目和属性。这种架构允许更少的 I/O 延迟和与平台无关的备份及管理能力，从而在提高整体性能的同时，使整个系统对系统架构决策的依赖性大大降低。

与 OID 非常相似，OUD 能够支持高可用性架构。OUD 在设计上将高可用性作为首要任务。它提供了在多个数据中心的多个节点上进行部署的能力，这不仅提供了本地故障转移，还提供了灾难恢复能力。此外，凭借 OUD 的引用功能和透明重路由能力，中断可以立即故障转移到其他数据中心，从而提供 99.99%的正常运行时间。




### 可扩展性

OUD 架构及其支持多节点和数据中心的能力提供了极高的可用性。结合高可用性，可扩展性使企业能够根据当前需求启动足够大的目录环境，而无需为未来的需求预留空间。Oracle 声明 OUD 可以在商品服务器上扩展到数十亿用户。这是通过代理功能和将身份数据分布在部署在一系列服务器上的分区中实现的。分区的分布允许工作负载分散在多个服务器之间，从而减少了通常在数据中心仅能将负载平衡到一两个服务器的环境中所见的工作量和 I/O 操作。

为了为 OUD 的可扩展性增加最大价值，全局索引允许 OUD 将请求映射到正确的分发分区。OUD 可以利用数据分区将数据存储分散到多个节点，甚至多个数据中心。这不仅有助于分散工作负载，而且如果按区域完成，还可以用来减少网络延迟的影响。如果企业的用户遍布全国甚至全球，它很可能也在不同区域战略性地设有数据中心。利用 OUD 分区数据的能力，可以将区域用户放置在最适合他们经常工作的区域的服务器上。全局索引使 OUD 能够跟踪条目的位置。如果一个节点收到请求，它会迅速被引导到正确的节点（无论其位置）以验证用户。此功能不仅可以帮助组织降低成本，还可以提高用户生产力。

### 复制

与 OID 类似，OUD 提供了一种在多个节点之间复制数据的机制。此功能对于确保最高可用性至关重要。复制确保在身份存储库的一个节点上更新的数据被传播到集群的其他成员节点。延长的延迟或未正确复制的环境可能导致安全问题，因为用户可能能够访问他或她已被移除一段时间的资源，或者密码更改可能不会立即生效。OUD 引入了高级复制。高级复制模型使用复制服务器来同步与其他目录服务器的数据更改。通过将复制工作转移到单独的服务器，同步更改的工作从目录服务器上卸载，允许它们将资源集中用于目录查询。为确保复制周围的安全，OUD 复制服务确保客户端应用程序请求的身份数据仅在数据已复制到其他成员后才被释放。此外，通过防止敏感数据被复制到可能安全性较低的服务器，身份安全性得到提高。

### 可用性与可管理性

与 OID 一样，OUD 使用 ODSM 和 Fusion Middleware Control 进行管理。由于这些内容先前已经讨论过，此处不再赘述。

## Oracle Virtual Directory

大多数组织已经对多个身份存储进行了投资。这些包括他们的网络身份存储、特定于应用程序的身份存储，甚至可能包括特定于业务单元的 LDAP 目录。在许多情况下，各种数据存储包含相同的用户或相同数据的子集。这常常导致混淆，管理员试图管理用户组并了解各种存储的位置。将这些目录同步到中央存储库可能导致其他问题，例如增加的数据存储需求以及确保应用程序能够以预期的方式利用该存储库。引入 OVD 就是为了这些问题。OVD 允许组织将其身份目录投资整合到一个集中式存储库中，而无需复制数据。

OVD 被宣传为目录聚合工具。它为组织提供了访问正在使用的各种用户存储库的能力，而无需复制其中包含的数据。此外，OVD 提供数据转换服务，允许应用程序以期望或需要的格式接收数据。OVD 甚至可以为非 LDAP 类型的源（如基于数据库或文件的身份存储）提供 LDAP 接口。

## 架构

部署在 WLS 上，OVD 利用了许多 WLS 的优势。在 WLS 环境中，OVD 能够与凭证存储、审计和日志框架集成，并使用 Fusion Middleware Control 接口进行管理。

OVD 由四个逻辑层或组件组成，它们协同工作以提供所有服务，例如数据聚合和转换。从第一层开始，OVD 提供了允许外部应用程序连接到它的监听器。下一层由验证请求有效性并确定使用哪个适配器来满足请求的进程组成。请求被传递到一个连接机制，该机制与各种适配器一起操作以检索数据和转换数据集。这些层协同工作，向应用程序提供看似单一数据源的内容。图 3-11 显示了 OVD 的主要组件。

适配器允许 OUD 连接到基础身份存储，无论是 LDAP 还是数据库存储库。提供了用于连接到 LDAP、数据库和本地存储的适配器。此外，可以配置自定义适配器来处理各种其他存储库。

由于其架构，OVD 具有极强的容错能力。可以在单独的服务器甚至不同区域部署多个具有相同配置的 OVD 实例。无需复制数据或存储文件，也无需提供数据库高可用性选项。每个 OVD 实例都使用自己的适配器连接到后端存储库，因此任何实例都可以根据负载平衡或中断来服务请求。此外，数据保证与源存储库一样新。

除了能够跨多个服务器部署以提供自身的容错能力外，还可以配置 OVD 适配器以在源目录的多个实例之间分散负载。例如，如果有两个使用复制的 Active Directory 实例，则可以配置 OVD 适配器引用两者，从而在其中一个实例发生故障时提供自动故障转移。



### 4. Oracle 目录服务的安装与配置

Oracle 为轻量级目录访问协议（LDAP）数据存储提供了多种选择。Oracle Internet Directory (`OID`) 和 Oracle Unified Directory (`OUD`) 均提供存储功能，而 Oracle Virtual Directory (`OVD`) 则允许将多个不同的 LDAP 存储源呈现为一个单一来源。本章将介绍每个产品的安装与配置步骤。截至本文撰写时，Oracle E-Business Suite 已认证可用于 `OID`，但尚未认证 `OVD` 或 `OUD`。此处介绍所有三种产品的安装过程，作为未来参考指南。然而，在后续章节中，仅会使用 `OID`。

## 安装前任务

在开始实际的软件安装并将二进制文件部署到文件系统之前，必须注意有一些任务必须完成，以确保操作成功。应谨慎创建或完成这些任务，例如创建操作系统（OS）用户、设置内核参数以及授予适当的权限。这些事项将在本章后续部分进行介绍。

### 操作系统用户

对于大多数 Oracle 应用程序的安装，应创建操作系统用户和组来执行安装和配置任务。创建操作系统组将允许其他操作系统用户执行与应用程序环境管理相关的特定任务。在 Linux 环境中安装 Oracle 应用程序时，最常见的操作系统用户和组是 `oracle` 用户以及 `oinstall` 或 `dba` 组。

要创建必要的 `oinstall` 和 `dba` 组，请以 `root` 用户身份执行以下命令：

```
[root@clouddemolab home]# groupadd oinstall
[root@clouddemolab home]# groupadd dba
```

创建组后，创建 `oracle` 用户。

```
[root@clouddemolab home]# useradd -g oinstall -G dba oracle
```

注意

`-g` 表示用户应添加到的主组。`-G` 表示任何附加组。

要设置用户密码，请以 `root` 用户身份使用以下命令：

```
[root@clouddemolab home]# passwd oracle
```

### 聚合

`OVD` 并不存储其呈现的身份数据。相反，通过使用上一节提到的适配器，`OVD` 充当多个后端目录的代理。这降低了与复制和同步多个存储源相关的成本和管理工作量，同时利用了组织内已有的存储库。它还通过减少开发期间所花费的时间和精力来提高效率，因为无需构建新的目录或接口。

大多数公司都有几个（如果不是几十个）不同的用户存储库。这些可能包括网络或企业目录、特定业务部门的目录、电子邮件、实现其自身目录的启用 `LDAP` 的应用程序，甚至是诸如 `EBS` 或 `PeopleSoft` 之类的数据库源。在许多情况下，存储在一个源中的数据不易适配与其他应用程序协同工作，因此需要数据复制。通常，控制应用程序的各个业务部门有强制要求他们保留对其目录的控制权。无论出于何种原因必须维护多个数据存储，`OVD` 都可以整合它们并提供那个单一访问点。

为了促进身份存储信息的聚合，`OVD` 提供了转换或映射数据以匹配应用程序或后端数据存储预期格式的能力。这些映射以双向流进行。这允许将请求格式化为符合存储要求，同时返回的数据随后被转换为匹配客户端应用程序的要求。`OVD` 还可以配置为重新映射属性名称和值。

作为代理，`OVD` 系统地将每个传入请求路由到适当的适配器，使用的逻辑会检查 `DN` 模式、过滤器、属性和查询，以确定满足请求的最佳可用适配器。这减少了为获取结果而需要查询的适配器数量，优化了 `OVD` 的性能，并允许配置更复杂的环境而不影响性能。

### 访问管理

许多组织正在努力为其用户提供单点登录（`SSO`）功能。然而，如前所述，这些组织可能拥有多个用户存储库。前面提到，`OVD` 为客户端应用程序访问各种后端目录提供了一个统一的前端。访问管理系统也是如此。像 `OAM` 这样的产品通常需要一个 `LDAP` 存储库来提供身份验证和授权服务。无论后端存储库如何，提供那个 `LDAP` 前端正是 `OVD` 设计的目的。

`OVD` 为访问管理系统提供的另一个好处是一个抽象层。随着时间的推移，底层数据存储会发生变化。这些变化可能导致客户端应用程序或访问管理系统失败。通过将底层目录结构与客户端应用程序分离，`OVD` 通过确保数据持续转换或翻译成所需的确切格式来保护它们免受变更影响。

## 总结

本章介绍了构成 Oracle 目录服务的三个主要产品，它们为 Oracle 身份和访问管理提供了基础。`OID` 和 `OUD` 都提供用户管理和存储解决方案，并为客户端应用程序和服务公开 `LDAP` 接口。`OVD` 帮助组织整合多个目录的管理，并为需要身份验证服务的应用程序提供一个统一的来源。本书的其余部分将集中介绍 `OUD` 的安装，因为 Oracle 已表示该产品将是 Oracle 目录服务的发展方向。然而，当 `OVD` 提供额外功能或替代解决方案时，也会对其进行讨论。



### 操作系统配置

在安装 Oracle Fusion Middleware 基础架构和 Oracle Identity Management 软件之前，确保操作系统满足最低要求和配置至关重要。以下列出了所需的内核参数、软件包以及需要进行的文件更改。

需要设置以下内核参数：

```
kernel.sem  256  32000  100  143
kernel.shmmax 10737418240
```

要设置这些参数，请编辑位于 `/etc` 目录下的 `sysctl.conf` 文件。

```
[root@clouddemolab home]# vi /etc/sysctl.conf
```

在该文件的相应部分添加或编辑以下行：

```
# Controls the maximum number of shared memory segments, in pages
kernel.shmall = 4294967296
kernel.sem = 256 32000 100 142
kernel.shmmax = 10737418240
```

在 `sysctl.conf` 文件中设置这些值后，您必须激活这些设置并使用以下命令验证新值是否生效：

```
[root@clouddemolab home]# /sbin/sysctl –p
net.ipv4.ip_forward = 0
net.ipv4.conf.default.rp_filter = 1
net.ipv4.conf.default.accept_source_route = 0
kernel.sysrq = 0
kernel.core_uses_pid = 1
net.ipv4.tcp_syncookies = 1
net.bridge.bridge-nf-call-ip6tables = 0
net.bridge.bridge-nf-call-iptables = 0
net.bridge.bridge-nf-call-arptables = 0
kernel.msgmnb = 65536
kernel.msgmax = 65536
kernel.shmmax = 68719476736
kernel.shmall = 4294967296
kernel.sem = 256 32000 100 142
kernel.shmmax = 10737418240
```

必须将打开文件限制设置为 4096 以支持该实例。为此，请编辑 `limits.conf` 文件。

```
[root@clouddemolab home]# vi /etc/security/limits.conf
```

如果环境要安装在 Oracle Linux 或 RedHat Linux 上，您还必须在 `/etc/security/limits.d/90-nproc.conf` 中进行编辑。如果遗漏了此步骤，该文件中的值可能会覆盖 `limits.conf` 文件中的值。

在上面列出的两个文件中，确保添加或编辑以下行：

```
* soft nofile 4096
* hard nofile 65536
* soft nproc 2047
* hard nproc 16384
```

编辑此文件后，必须重启服务器以确保所有更改生效。

## 验证安装

完成 OID 的所有安装和配置任务后，验证环境非常重要。这涉及登录到各个组件并确保它们可运行。此时还建议您验证启动和停止脚本，并为管理设置受管服务器。

在上一节中，您安装了 WLS 作为 OID、OVD、DIP 和 ODSM 的基础架构。这些组件应已自动启动。检查操作系统以查看为支持 OID 而启动的进程。

打开控制台并以 oracle 用户身份登录到主机。使用之前创建的域名，使用 `ps` 命令检查从域目录运行的进程。有关此过程的视图，请参见图 4-46。

Oracle 进程管理器和通知服务器 (`OPMN`) 控制实际的 OID 实例。`OPMN` 在域配置期间安装。虽然它是 OID 的进程管理器，但它不在 WLS 内运行。它为 OID 环境提供启动和关闭功能以及监控和重启服务。在验证过程中，将使用 `opmnctl` 命令检查进程状态。

在与之前相同的控制台中，将 `ORACLE_INSTANCE` 设置为在域配置步骤中创建 OID 实例的位置。这很可能在 `MIDDLEWARE_HOME` 目录内；例如，`/home/oracle/OIDMiddleware/asinst_1`。

```
export ORACLE_INSTANCE=/home/oracle/OIDMiddleware/asinst_1
```

## 本章小结

本章介绍了 OID 的安装和初始配置。这为成功的身份管理实现奠定了基础。从这一点开始，您可以开始在 Oracle 的通用 LDAP 目录中管理应用程序用户和属性。应用程序，无论是 Oracle 的还是第三方的，都可以连接到 OID 作为 LDAP 身份存储来验证用户凭证。数据库也可以使用该身份存储进行身份验证。尽管 OID 为基于身份的系统提供了基本的 LDAP 身份验证和数据存储，但该环境仅维护对单个应用程序的单一身份验证。Oracle 访问管理器可以提供更好的用户体验，用户只需登录一次即可访问已纳入环境的应用程序。Oracle 身份管理器将提供用户自助和服务管理服务。这些附加组件依赖于一个稳固的 LDAP 实现，并将在未来的章节中介绍。
