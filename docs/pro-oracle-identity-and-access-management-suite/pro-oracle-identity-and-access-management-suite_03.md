# 2. 安装前考虑因素与先决条件

### 容量规划

要成功地在企业中实施 Oracle 身份与访问管理套件或其部分组件以提供应用安全保障，离不开对当前环境的正确分析以及对未来的适当规划。这将确保不仅能满足即时需求，并且在将来添加更多用户和应用程序时，性能也不会下降。本章节为那些处于身份管理实施规划阶段的人员提供指导。

#### Fusion Middleware

Oracle 身份与访问管理与其他`Fusion Middleware`产品和应用一样，依赖于`WebLogic Server`架构。作为`Fusion Middleware`架构的核心，确保`WLS`环境得到正确配置至关重要。这始于恰当的规划与分析。在决定如何设置环境（如内存、硬件、灾难恢复等）时，规划`WebLogic`环境涉及多个方面。下文将对此进行讨论。

性能要求是一个重要的考虑点。必须考虑应用程序用户对部署在实例上的应用将期望何种服务级别。所讨论的应用是否严重依赖于身份或其他数据、是否内存密集型、或者应用是否有特定的响应时间要求？同时还需考虑环境将要承受的预期负载。

##### 评估容量要求

`WLS`必须在满足最低要求的服务器上安装和配置。然而，满足最低要求并不总能保证部署在该环境中的应用程序能以最佳水平运行并提供系统用户所需的服务级别。在规划`WebLogic`实施时，应与应用规划人员或开发者合作，确定预期用户数量、并发连接数、期望响应时间等。此外，收集有关应用程序本身的信息非常有帮助，包括事务耗时、每秒事务数以及应用复杂度。还需考虑应用架构，例如数据库连接和外部服务访问。所有这些都将成为确定`WLS`所需硬件要求以及软件调优的有用输入。所有这些可能需要安装一个测试环境，并根据这些数字进行负载测试。这种做法也能为未来的调优工作提供一个可靠的基准。

##### 硬件

容量规划始于计算硬件需求，并确保环境达到或超过这些要求。一个很好的起点是计划用于该环境的软件所附带的文档。`WLS`最低要求一颗`1-GHz CPU`，但待安装组件的具体要求将决定整个系统的要求。例如，在一种分布式架构中，`Oracle Access Manager`和`Oracle Identity Manager`安装在一台主机上，而轻量级目录访问协议服务器安装在另一台上，则需要两台物理主机。`OAM/OIM`主机将需要六个或更多核心，而`LDAP`主机仅需要两个。

##### 内存

`WLS`需要`8 GB`可用内存（`4 GB`物理内存和`4 GB`交换空间）。这些代表了最低要求，具体数值会根据托管服务器数量及部署在其中的软件而变化。必须考虑服务器本身的内存需求。可以通过为`Linux`操作系统分配`3 GB`、为`WebLogic Admin Server`分配`3 GB`、以及为每个托管服务器分配`3 GB`（此数字将根据部署内容而变化）来确定一个起点。这将为`WLS`硬件所需的可用内存提供一个初步估算。本章后续将提供关于内存更详细的讨论。

##### 存储

尽管`WLS`不需要大量前期存储，但规划磁盘空间以允许增长是重要的。同样，必须考虑将在`WebLogic`托管服务器内部署的所有软件的磁盘空间要求。`WLS`安装大约需要`580 MB`，额外需要`570 MB`用于`coherence`。总空间需求约为`1.2 GB`。如前所述，需考虑要部署的应用程序。例如，`Oracle Internet Directory`、目录集成平台和目录服务管理器将需要额外大约`800 MB`。

##### 网络

主机上的网络负载和网络配置将对`WLS`实例的感知性能产生影响。Oracle 建议在千兆网络上安装`WLS`并使用硬件负载均衡器。

##### 集群

在物理或虚拟主机上增加 CPU 和内存将提升`WLS`环境内的性能。这些更改同样会影响部署在`受管服务器`中的应用程序的性能。然而，它们无法降低单个服务器的整体负载。增加服务器或构建集群可以带来更大的性能提升。

构建`WebLogic`集群环境包括添加一个或多个运行`WLS`的额外主机，并将应用程序部署到`受管服务器`中，或者部署在单个多处理器服务器的多个实例上。不过，性能仍然会受到物理服务器上其他资源的限制。跨多个服务器构建`WebLogic`集群环境的一个额外好处是，通过在物理服务器故障时提供故障转移支持来提高可用性。向`WLS`集群添加一个新的独立主机并对系统进行负载均衡，将会把负载分散到多个主机上。`WebLogic`将处理会话信息和集群成员监控，以确保数据的连续性。一个正确配置的`WebLogic`集群对于客户端应用程序而言，将呈现为一个单一实例。

集群的负载均衡优势在于，它能将应用程序负载均匀地分布到多个主机上。这要求`受管服务器`部署在集群内的每个节点上。需要注意的是，集群中的每个`WebLogic`实例都管理着自身与数据库及底层服务的连接。此功能进一步提升了整体环境的性能，因为每个节点都处理自己的一组应用程序请求。此外，可以在不影响用户的情况下向环境添加额外的集群成员，从而实现动态可扩展性。

集群带来的高可用性改进，使得在节点因故障丢失时能够实现故障转移冗余，因为`WebLogic`基础设施处理会话信息和应用程序状态，集群可以提供自动故障转移能力。此外，如果需要，整个`受管服务器`可以迁移到另一个节点。

总之，确保一个成功的`WebLogic`环境需要考虑系统整体将承受的基本负载。单台服务器必须至少具备`WLS`所需的最低 CPU 和内存要求，同时也必须满足待部署应用程序的要求。通过`WebLogic`集群功能，可以进一步获得性能提升和故障转移能力。这些特性将负载分散到多个主机上，每个主机处理一部分工作，并在一个或多个节点丢失时承担起额外的负载。Oracle 身份与访问管理套件的所有组件都要求一个`WebLogic`环境。本章后续将讨论有关整体环境的一般性指南和注意事项，以帮助建立一个高性能、高可用的系统。

### 企业部署拓扑

Oracle 身份与访问管理代表了 Oracle 基于最前沿 Oracle 技术构建的全面身份管理套件的最佳实践方案。它在提高用户生产力的同时，也提供了一个高度安全且易于适应的环境。为支持涵盖性能、高可用性和可扩展性在内的广泛企业需求，Oracle 身份与访问管理利用了融合中间件架构。借助融合中间件，组织可以获得多种选择，从适用于开发环境的单节点，到位于不同地理区域数据中心、用以支持负载和最大灾难恢复能力的多节点集群。本节将讨论 Oracle 建议的各种拓扑结构，包括每种结构的使用原因及其能为企业带来的益处。

#### 单节点

单节点架构描述了一种所有服务都在单个`WebLogic`节点上运行的环境。虽然可能涉及多个物理或虚拟主机，但每台主机运行不同的组件，且不提供故障转移。例如，`OID`可能只在`Host_1`上运行一个实例。`OAM`可能运行在`Host_2`上。然而，在这种情况下，所有认证请求都将通过`Host_2`上的`OAM`进行路由。`OAM`会检查`Host_1`上的`OID`。如果`Host_1`或`Host_2`中任何一台主机宕机，所有操作都将停止。这种拓扑不提供任何冗余。同时，它也不提供负载均衡。如果应用程序负载过重，单台服务器将不得不处理增加的流量。正是由于这两个原因，单节点拓扑通常只适用于开发环境。

图 2-1 描述了单节点拓扑。应用环境的每一层都由一个主机/单个`WLS`构成，包含两个`受管服务器`。如图 2-1 所示，没有冗余或负载均衡机制。这种配置是最简单的实现方式，很可能在一天之内就能配置完成。虽然这对于小型组织或不预期高负载的组织来说可能是可以接受的，但较大的组织通常需要比此架构提供的更高容错能力和性能水平。

总的来说，单节点拓扑适用于能够容忍停机时间或没有高可用性要求的环境。例如，开发团队可以使用它们来开发需要身份管理的应用程序。开发环境可以选择离线，而不影响普通用户群体。通过进行定期备份并制定良好的恢复计划，如果用户群体较小或不需要 7x24 小时可用性，较小的组织或许可以通过运行单节点环境来节省一些成本。

#### 本地高可用性

Oracle Fusion Middleware 利用 WLS 的集群功能，使组织能够为冗余和更好的性能实现集群。正如之前所讨论的，集群环境提供了可扩展性，因为可以动态地向环境中添加更多节点，以应对增加的负载或为未来任务做准备。如果组织预期负载增加或业务增长，可以添加新节点，并且负载均衡可以确保负载均匀分布在集群的所有成员中。

本地高可用性集群是位于单个数据中心或地理位置内的环境。这提供了冗余和负载均衡能力，是基础的集群环境。然而，单个数据中心的电力故障或灾难将导致服务完全中断，直到环境重新启动或恢复。需要注意的是，环境可以由两个或更多 WLS 实例组成。一个使用多个 Web 层和一个双节点应用层的典型负载均衡环境如图 2-2 所示。

Fusion Middleware 环境中的集群对象也允许实现更高的可用性。集群的节点可以在对用户几乎没有影响的情况下添加或移除。例如，如果管理员注意到某个集群成员出现 I/O 或内存问题等问题，并且有其他成员可用以承担负载，则可以将该成员离线以解决问题。借助集群提供的冗余，发生故障的服务器进程可以自动故障转移到集群的另一个成员。

自动服务器迁移在仅运行在一个节点上的应用程序失败时非常有用。在这种情况下，Fusion Middleware 允许将服务器配置为自动或手动迁移。如果 WebLogic 检测到服务故障，单例服务将自动迁移到另一个节点。这对于诸如 `JMS` 或 `JTA` 事务恢复系统之类的服务特别有价值，这些服务只能在一个服务器上运行。

本地高可用性集群可针对计划外停机提供改进的性能和冗余，并且它们还可以通过允许以滚动方式应用系统补丁使组织受益。使用集群，组织可以在不关闭整个环境的情况下应用小型补丁。可以通过关闭单个节点或指示负载均衡器将其从可用服务器中移除，然后应用补丁来逐个修补节点。在该节点上的补丁完成后，可以重新启动它并重新加入集群，然后可以继续处理下一个节点，而不会对用户群体产生不利影响。

这些单站点多节点集群非常适合需要解决性能问题并提供高可用性的组织。环境通过使用负载均衡器将请求分配到多个服务器来处理负载。Fusion Middleware 架构提供了从 Web 层到应用层的负载均衡以及从应用层到数据库层的负载均衡。这确保了最有能力处理请求的服务器来处理它们。然而，这取决于服务器进程的类型。

有状态的 Java 应用程序（如 `WebCenter` 或 `ADF`）利用 `WebLogic` 代理插件 `mod_wl_ohs` 来监控服务器状态以及会话状态副本信息的位置。如果集群中的某个节点丢失，`WebLogic` 代理插件会将请求路由到可用节点，并从副本恢复会话的状态。无状态应用程序依赖负载均衡器将请求路由到集群的可用成员。对于 `C` 语言组件等有状态应用程序，不存在会话状态信息副本。负载均衡器保持会话粘性，如果服务器丢失，将路由到下一个可用节点。但是，由于没有会话信息可用，用户将需要开始一个新的会话。

本地高可用性集群通过在多个服务器成员之间分散负载，提供了高水平的性能和可靠性。它们还使企业能够应用补丁和小规模升级，而不会对其用户群体产生不利影响。通过 Fusion Middleware 架构，从 Web 层到数据层都提供了高可用性和负载均衡。


### 灾难恢复与最高可用性

虽然本地高可用集群适用于许多组织，并能提供高水平的性能和故障转移能力，但有些组织需要跨多个地理区域分散负载并提供故障转移。这在遭遇整个数据中心损失时，可提供最高的可用性和最快的恢复速度。这种拓扑结构也可用于为地理上分散的用户群体提供更好的性能。这对组织来说是成本最高的方案，因为它需要更多的硬件和多个数据中心场地，同时也增加了复杂性和管理难度。然而，对于需要 `99.999`% 正常运行时间且有极高容量和性能要求的组织而言，考虑到它所带来的益处，这仍是终极解决方案。

对于在全球开展业务的组织而言，最高可用性是一项必要要求。这些组织无法承受其系统宕机或安全成为服务访问的瓶颈。这些公司内部的 IT 团队可能会选择将数据复制到多个数据中心。**主备环境**允许更快的恢复。如果主站点丢失，在故障转移过程中，所有流量都可被定向到备用站点。当主数据中心重新上线后，所有流量会被重定向回主站点，并重新开始复制以确保备用站点数据一致。另一种选择是在数据中心之间维护一个**双活集群**。在这种情况下，所有流量会在所有数据中心的所有节点之间进行负载均衡。如果一个数据中心丢失，负载均衡器会自动将流量导向剩余的服务器。

集群还带来了一个额外好处，即可将计算能力和任务分散到多个节点上。许多大型公司在全球拥有海量的身份数据和用户。通过使用代理服务器并将数据分布在多个地理位置，组织可以减少网络流量和查询时间。例如，在纽约设立一个数据中心，存储北美地区的身份数据；同时在东京设立另一个数据中心，处理亚洲地区的身份数据，这样就能将数据本地化，从而降低亚洲国家请求的网络延迟。由于每个区域的数据源更小，查询时间也随之减少。对于用户群体地理分布不广但拥有大量数据的组织，同样的功能可用于在多台服务器上处理数据子集，从而减少查询时间，同时通过跨多个位置复制服务器来改善恢复时间和可靠性。这种环境的一个示例见图 `2-3`。

#### 拓扑实现

到目前为止，本章主要集中在 `Fusion Middleware` 环境的通用术语以及可以采取哪些措施来提高性能和可用性上。身份与访问管理的各个组件利用这些拓扑结构为组织提供这些益处。以下将结合身份与访问管理的组件来讨论这些内容。

这些建议基于 Oracle 企业部署指南 (`EDG`)。`EDG` 代表了由 Oracle 工程师开发的、旨在提供最佳性能和可靠性的建议架构。虽然它们没有给出具体的硬件或调优要求，但提供了关于应如何安装和配置环境的总体思路。在确定组织的整体需求时，应遵循这些总体指导原则。使用这些指南也将有助于建立更标准化的环境，从而更易于添加新资源、进行故障排查，以及让新管理员更容易定位必要的组件。


##### Oracle 目录服务

Oracle 统一目录 (`OUD`) 支持多层级的负载均衡和分发。最基本的形式是简单负载均衡。这种实现方式允许代理根据服务器状态和负载来确定哪个节点应处理请求。这将在一个或多个服务器之间平均分配工作负载，从而提供故障转移和性能提升，如图 2-4 所示。

图 2-6 描绘了一个环境，其中 LDAP 服务器在`数据中心 1`内复制，并且该环境被复制到`数据中心 2`。每个数据中心内的负载均衡根据组织的需求进行配置。故障转移事件可以配置为在`数据中心 1`丢失时将请求路由到`数据中心 2`。然而，另一个选项是为饱和事件配置故障转移。如果`数据中心 1`变得不堪重负，请求可以被发送到`数据中心 2`，直到饱和度恢复到正常水平。

前面的例子提供了基于每个服务器上复制数据的数据中心内各服务器之间的负载均衡。下一个选项涉及在单个数据中心内对服务器间的分发数据进行负载均衡。此选项非常适合没有辅助数据中心，但具有非常大的身份数据需求并预期负载很重的组织。如图 2-7 所示，身份数据按字母顺序拆分到两个 LDAP 服务器，同时多个故障转移节点提供稳定性和负载均衡。

负载均衡的分发数据

在图 2-7 中，身份数据分布在两台服务器上，允许每个分区仅服务于基于查询字符串的请求。然后每个分区被复制到第二个实例以实现负载均衡。这种架构提供了极快的性能，因为请求是根据服务器包含的内容以及与其他服务器相比所承受的负载来定向到服务器的。

尽管负载均衡的分发架构提供了高水平的性能和在服务器问题发生时的一些冗余，但如果数据中心出现问题，它提供的故障转移能力有限。最高级别的可用性和性能将所有这些概念结合起来，创建了一个多数据中心负载均衡分发数据环境。图 2-8 展示了跨多个故障转移站点的分发架构。

具有数据中心间故障转移的负载均衡分发数据

#### Oracle 访问管理器

`OAM`利用`WLS`集群功能来提供高可用性和性能可扩展性。这些功能依赖于从 Web 层到数据库层的正确设置。需要注意的是，为了更高的可用性和性能，负载均衡器应在`OHS` Web 服务器之前处理所有具有 HTTP 流量会话保持性的 HTTP 请求。`OAM`负载均衡和简单故障转移如图 2-9 所示。

![`A352855_1_En_2_Fig9_HTML.jpg`](img/A352855_1_En_2_Fig9_HTML.jpg)

图 2-9.

基本 Oracle 访问管理本地高可用架构

如图 2-9 所示的 Oracle 访问管理本地高可用架构中，请求首先由负载均衡的 Web 层处理，该层由两个或多个带有`mod_wl_ohs`模块的 Oracle HTTP 服务器组成。`mod_wl_ohs`模块使用名为`mod_wl_ohs.conf`的文件进行配置，该文件标识了 WebLogic 受管服务器的位置。如果`OAM`保护请求的资源，配置在`OHS`实例上的`WebGate`组件会收集必要的凭证并将其转发给`OAM`。需要注意的是，应在负载均衡器上设置 HTTP 会话保持性，以确保后续用户请求被路由到正确的服务器。

`OAM`部署在一组`WLS`上，并使用标准域、计算机和集群功能进行集群。根据常规集群，`管理服务器`安装在两个 WebLogic 节点上。但是，它一次只在一台服务器上运行。`OAM 服务器 1`和`OAM 服务器 2`部署在`OAM 主机 1`和`OAM 主机 2`上配置的机器上。因为两台机器属于同一个 WebLogic 域，所以它们可以由运行在`OAM 主机 1`上的单个`管理服务器`实例控制。

从`OAM 服务器`实例，身份验证查询通过负载均衡器发送到后端 LDAP 服务器。如前一节所述，LDAP 服务器可以是`OVD`、`OID`或`OUD`。为了获得最佳性能和可用性，LDAP 服务器层应配置为如前一节所述的负载均衡。

虽然单个 Oracle 访问管理实例可以支持大量用户，同时仍能提供良好的性能，但多实例配置可以提供冗余以实现容错和恢复，以及负载均衡以提供更好的性能。
