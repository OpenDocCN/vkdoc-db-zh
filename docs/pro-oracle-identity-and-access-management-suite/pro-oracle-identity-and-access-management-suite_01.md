# 引言

许多组织都会为安装在其环境中的每个产品规划贯穿其产品生命周期的安全措施。大多数产品都会提供某种用户存储机制。对于那些只实施了少量产品的较小环境而言，使用这些独立的用户存储库或许是可行方案。然而，在大型组织中，这通常会导致存在多个身份存储、数据在业务部门间重复，以及管理上的麻烦。此外，使用独立的用户管理功能可能导致用户需要为其使用的所有产品维护多个用户名和密码。最终，这些用户要么会使用安全性较低的密码，更糟糕的是，可能会将用户名和密码列成清单并放在键盘底下。

解决方案是实施一个单一的身份数据源，供所有应用程序用于认证和授权。轻量目录访问协议（`LDAP`）旨在提供一种在身份存储中查找信息的标准方法。借助 `LDAP`，应用程序现在有了从外部存储对用户进行认证和授权的标准方法，只要该存储符合 `LDAP` 标准即可。业务部门现在可以实施符合 `LDAP` 标准的软件来访问中央用户存储库，不再需要用户维护多个账户。Oracle Internet Directory (`OID`) 是 Oracle 对通用 `LDAP` 目录的实现。其他 Oracle 产品，如电子商务套件、WebCenter Content 和 `OBIEE`，都设计为与 `OID` 协同工作。作为一个通用的、符合 `LDAP` 标准的目录，`OID` 也可以被其他第三方应用程序所利用。

Oracle 更进一步，推出了 Oracle Access Manager (`OAM`) 来提供单点登录功能。这样，每个应用程序不再需要单独的登录请求，而是可以配置为利用现有的浏览器令牌进行身份验证，从而免除用户多次输入凭据的麻烦。`OAM` 支持安全断言标记语言 (`SAML`)，因此可以配置为向基于云的第三方应用程序提供认证服务。这也意味着像人力资本管理这样的 Oracle 云端解决方案，可以与组织的 `OAM` 单点登录环境进行集成。

任何实施若不包含某种形式的身份生命周期管理，都是不完整的。过去，每个应用程序都负责管理自己的身份存储。这导致用户不得不为他们访问的每个应用程序创建和维护多个账户。当用户入职时，可能需要几天甚至几周时间才能获得所需的所有访问权限。相反，当用户离开组织时，其账户可能不会被及时停用，甚至完全没有停用。这带来了巨大的安全风险。Oracle Identity Management (`OIM`) 的推出，为企业级身份生命周期管理带来了新的治理水平。`OIM` 提供了一个管理用户身份数据的中央接口。它可以连接到标准的 `LDAP` 目录，或者通过使用 Oracle Virtual Directory，它可以管理来自多个存储的数据。`OIM` 提供的自动化能力可以减少用户入职所需的工作量，并确保在用户离开组织时撤销其访问权限。

Oracle 身份与访问管理套件结合了这里讨论的关键要素，为用户管理提供了一个端到端的解决方案。尽管实施起来应该很简单，但要让一切正常且高效地运行，仍需要完成各种步骤。本书旨在为在您的环境中启动并运行身份与访问管理套件提供指南。它演示了安装和配置过程，并包含一些架构方面的讨论，以帮助确定您的环境需要什么。
