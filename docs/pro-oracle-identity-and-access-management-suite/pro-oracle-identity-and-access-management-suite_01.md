# 1. Oracle 身份和访问管理套件概述

Oracle Fusion Middleware 产品部署在 WebLogic Server 架构中。WebLogic Server 提供了一个可扩展的环境，使企业能够部署和管理 Oracle 产品及 Java 应用程序，并具备访问数据库和消息服务的能力。WebLogic Server 作为应用服务器层运行。WebLogic 提供的功能包括集群、高可用性、可管理性、监控、安全性和数据库集成。

Oracle 身份和访问管理套件由多个组件组成，每个组件都服务于非常具体的用途。这些组件包括目录服务、访问管理或单点登录（SSO）、身份管理、自助服务门户，以及配置、治理和报告服务。

本章将介绍 Oracle Fusion Middleware WebLogic Server 环境，以及配置 Oracle 身份和访问管理套件所涉及的主要组件。您还将了解与本书其余部分相关的 Oracle 身份管理组件的描述。

##### WebLogic Server

Oracle 的 WebLogic Server（WLS）是一个完全符合 J2EE 标准的应用服务器，它将支持 Oracle 身份和访问管理套件组件。此环境提供了部署自定义应用程序以及 Oracle Fusion Middleware 产品（如企业内容管理和 Oracle 身份与访问管理）所需的组件。由于 WebLogic Server 是一个应用服务器，用户能够通过已部署的应用程序端口使用 Web 浏览器访问这些应用程序，或者通过充当反向代理的 HTTP 服务器进行访问。

为提供企业级服务，WebLogic Server 支持多项环境特性。

*   **编程模型：** WLS 支持 Java EE 部署环境、Web 服务支持、Java 消息服务、可扩展标记语言（XML）功能、Java 数据库连接（JDBC）连接资源及其他组件。
*   **高可用性：** 通过 WebLogic 集群将工作负载分配到多个服务器，并具备检测和管理过载情况的能力来实现支持。持久化存储和存储转发服务允许临时存储 JMS 消息并在集群中跨服务递送。
*   **安全：** WLS 提供了一个内置的轻量目录访问协议（LDAP）2.0 身份存储，可用于管理对部署在服务器上的服务的访问。除此之外，WLS 可以配置为针对许多不同的外部数据存储进行身份验证，例如 Oracle Internet Directory、Active Directory 等。此外，WLS 框架允许集成身份断言器，如 Oracle Access Manager（OAM）。
*   **诊断框架：** 提供收集和分析关于服务器上运行进程的运行时数据的能力。这可用于诊断问题并优化性能以获得更好的体验。

就 Oracle 身份和访问管理环境而言，本书将讨论如何使用以下组件。

*   **管理服务器** 为管理部署的所有组件提供了一个图形用户界面（GUI）。每个 WLS 域都有一个管理服务器，该服务器可以在任何 WebLogic 主机上运行。管理服务器管理域、集群、受管服务器、数据源、安全设置和应用程序部署的配置数据。
*   **域** 是 WLS 中服务器资源的逻辑管理单元。环境的所有方面都包含在一个域中，包括受管服务器、数据库源、消息服务、应用程序部署、机器和集群。
*   **机器** 代表托管受管服务器的物理主机。单个管理服务器可以管理多个机器。管理服务器与每台机器的节点管理器通信，以执行每个受管服务器的启动和停止操作。单个物理主机如果需要，可以配置多台机器监听不同的端口。
*   **WebLogic 集群** 由一个或多个共同工作的 WLS 实例组成，提供高可用性和横向扩展环境的能力。
*   **受管服务器** 是一个逻辑上的 WLS 构造，应用程序可以部署于此。
*   **数据源** 是在 WLS 内配置的 JDBC 连接，供部署在各个受管服务器中的应用程序使用。将数据源定向到特定的受管服务器使其仅对指定的服务器可用。
*   **安全领域** 定义了保护应用程序资源的身份验证提供程序和断言器。安全组、用户和策略都可以在 WebLogic 安全领域中定义。

##### Oracle 目录服务

Oracle 目录服务构成了 Oracle 身份和访问管理的核心。目录服务由多个选项组成，提供身份和策略存储、目录同步以及虚拟化功能，这些功能可以被企业使用的各种应用程序所利用。这些选项包括：

*   **Oracle Internet Directory (OID)：** 基于数据库、完全兼容 LDAP-V3 的身份目录。
*   **Oracle Unified Directory (OUD)：** 基于 Java、符合 LDAP-V3 标准的身份目录。
*   **Oracle Virtual Directory (OVD)：** 目录集成，支持管理多个源而无需数据复制。


## Oracle Internet Directory (Oracle 互联网目录)

`OID` 是一个完全遵循 `LDAP-V3` 标准的目录服务，它使用 `Oracle Database` 作为存储后端。这使得 `OID` 能够利用数据库的特性，例如 `Real Application Clusters`（真正应用集群）和 `Multimaster Replication`（多主复制）。结合 `OID` 集群和 `Fusion Middleware`（融合中间件）架构，它可以提供一个高可用且可扩展的环境。`Oracle Directory Services Manager`（Oracle 目录服务管理器）提供了一个标准的前端界面，用于维护 `OID` 中的用户、安全组、对象类、属性和策略。图 1-1 展示了在使用 `Real Application Clusters` 数据库环境下的 `OID` 或 `OVD` 的基本高可用环境。

`OID` 通过支持存储多个上下文（context），允许存放不同的身份数据。这使得可能存储在多个源中的数据可以在单个实例中进行管理。例如，如果企业已经实施了多个 `Active Directory Lightweight Directory Service`（Active Directory 轻型目录服务）实例来管理各种启用了 `LDAP` 的应用程序中的用户，则可以利用 `OID` 为所有这些应用程序提供单一的 `LDAP` 数据源。

通过使用 `Oracle database` 作为数据存储库，`OID` 能够利用诸如 `Transparent Data Encryption`（透明数据加密）和 `Database Vault`（数据库保险库）等特性，在操作的每个级别提供安全性。利用这些数据库特性的能力，通过让数据库来处理数据存储和备份安全，从而实现了安全职责的分离。

`OID` 提供了与企业中使用的其他目录进行同步的能力。如图 1-2 所示，`Directory Integration Platform`（目录集成平台）允许管理员为 `Active Directory`、`Sun eDirectory`、`OpenLDAP` 等目录服务创建和维护同步配置文件。这使得企业能够整合用户存储库，并使用标准化的通用 `LDAP` 目录来为应用程序提供安全性。

除了通过复制数据并转换格式以匹配 `OID` 存储需求来集成多个不同的身份存储之外，`OID` 还能够在 `OID` 节点之间执行数据复制，以提供高可用性和可扩展的性能。

对于 `OID`，组织会面临多个复制概念。完全复制涉及将整个目录传播到其他 `IOD` 节点，而不是仅将指定部分的结构发送到其他节点。对于复制的传输层，`OID` 支持 `LDAP` 复制和 `Database Advanced Replication`（数据库高级复制）。前者依赖 `LDAP` 协议将数据从一个 `OID` 实例传送到另一个实例，而 `Oracle Database Advanced Replication` 则要求数据库在数据库实例之间复制数据。最后一个概念是复制方向。`OID` 支持单主、多主和扇出复制。顾名思义，单主复制可以被认为是从主节点到所有其他节点的单向复制。多主复制允许从任何节点进行的更改被复制到其他节点。扇出复制是前两种方向的结合，其中主节点复制到其他节点，然后这些子节点可以向其他节点复制全部或部分数据。

尽管 `OID` 有悠久的历史，并且目前与其他 `Oracle Identity Management`（Oracle 身份管理）组件（如 `OIM` 和 `OAM`）以及 `Fusion Middleware` 产品和应用程序兼容，但 Oracle 已经指出目录服务的未来方向是 `OUD`。

## Oracle Unified Directory (Oracle 统一目录)

`OUD` 代表了 Oracle 发布的业界首个基于 `Java`、符合 `LDAP-V3` 和目录服务标记语言 (`DSML`) `v2` 标准的目录服务，它在一个平台中结合了存储、代理、同步和虚拟化功能。`OUD` 使用商用级硬件和灵活的部署架构，提供了高水平的性能和弹性扩展性以及高可用性。在提供这种级别的服务时，`OUD` 能够保持高水平的安全性和监控能力。

`OUD` 的架构支持全局索引，这增强了其弹性扩展性。此功能允许管理员添加服务器；`OUD` 将根据需要处理将新请求路由到新服务器和存储的过程。这消除了构建新环境和迁移大量数据的需要。此索引的另一个好处是可以按需解决规模需求。管理员不再需要确定当前的大小要求并估计未来几年的增长。条目也可以分布在多个目录存储实例上。图 1-3 显示了一个分布在多个节点上并集群以提供故障转移保护的环境。

为了在支持高可用性的同时提升高性能，`OUD` 分离了目录服务和复制服务的任务。如前所述，`OUD` 允许将服务器分布在多个数据中心。为了支持这一点，`OUD` 引入了复制服务器，如图 1-4 所示。这些是专门用于在整个环境中复制数据的服务器，将客户端请求处理任务留给目录服务器。

Oracle Unified Directory 复制

在图 1-4 所描述的 `OUD` 复制过程中，每个 `OUD` 服务器连接到一个复制服务器。然后它向该服务器发送和接收所有更改。复制服务器将接收到的更改传达给环境中的所有其他复制服务器。`OUD` 复制服务器再将这些更改传达给与它们连接的 `OUD` 服务器。一个由发起更改的 `OUD` 服务器分配的更改号标识该更改，并被复制服务器用来确保更改被传播到其他 `OUD` 服务器。此更改号也存储在持久存储中，用于向可能已与复制服务器断开连接的 `OUD` 服务器填充更改。如果更改同时发生在两个 `OUD` 服务器上并导致冲突，每个 `OUD` 服务器将重放更改，直到所有冲突解决。

复制服务器管理与 `OUD` 服务器的连接，并监听来自其他复制服务器的更改。这些更改被填充到与其连接的目录服务器。当配置 `OUD` 服务器进行复制时，这些复制服务器会自动创建。因此，它们可以在相同的 `Java` 虚拟机 (`JVM`) 或主机上运行。为了节省资源，可以配置一个 `OUD` 服务器同时执行目录和复制服务器功能。但是，对于较大的环境，Oracle 建议将这些功能分离到不同的服务器。

与 `OID` 非常相似，`Directory Integration Platform`（目录集成平台）可用于从第三方 `LDAP` 存储库复制身份数据，为应用程序认证提供中央数据存储。值得注意的是，这与前面为 `OID` 介绍的功能相同。因此，它生成的是必须维护的身份数据副本。一些组织可能不希望管理多个 `LDAP` 数据副本。这个问题由 `OVD` 来解决。



### Oracle 虚拟目录

OVD 提供了一种将多个身份存储呈现为单一来源的方法，无需进行同步和数据复制。呈现单一来源使企业应用程序能够从多个来源（如数据库、EBS、Active Directory 和 OID）访问身份数据。

聚合多个数据源使企业能够跨多个来源利用现有的身份数据，而无需复制数据或设置复杂的同步任务。由于 OVD 呈现多个数据源而无需复制底层数据，因此可以显著节省存储成本。

OVD 的一个关键能力是能够将身份数据转换为应用程序特定的视图。因此，它可以将非 LDAP 数据（如数据库或 Web 服务）以企业中现有各种应用程序所需的正确格式呈现。例如，OVD 可用于将 Siebel 客户主数据作为 LDAP 源呈现，以供其他应用程序进行身份验证。这种转换能力不仅允许多个应用程序按需查看数据，还允许组织保留对如何在其自身存储库中管理和共享身份数据的控制权。

OVD 不仅可以将多个数据存储呈现为单一来源，还支持分割配置文件的概念。例如，企业可能将所有身份数据存储在 Active Directory 中。但是，像 EBS 这样的应用程序可能需要存储在 OID 或人力资源数据库中的额外元数据进行授权。OVD 适配器将允许应用程序将这些数据视为单个条目。图 1-5 展示了使用 OVD 整合多个目录源的表示。由于此环境是虚拟的，所有操作均在无需复制的情况下完成。

OVD 构建为可扩展且高度可用，无论企业规模多大都能支持。性能扩展可以像向集群添加额外节点一样简单。这可以在中心位置或地理位置上完成。高可用性不仅在 OVD 环境中得到支持，它还支持其数据源的负载均衡和高可用性。尽管此配置允许更简化的环境且没有数据重复，但如果用户从一个源中被移除，可能会遇到数据丢失、孤立账户和组等问题。

### Oracle 身份与访问管理

大多数组织需要身份服务，例如 LDAP 存储库。有时这通过简单的网络目录（如 Microsoft Active Directory）实现。在考虑应用程序安全性时，需要更通用的目录。之前介绍的 OID、OVD 和 OUD 可用于提供此服务。随着越来越多的应用程序引入业务，用户在切换应用程序时可能会被身份验证对话框淹没。

OAM 是身份和访问管理套件的一个组件，可用于提供 SSO 功能。此功能可以大大减少用户一天中必须进行身份验证的次数，从而提高生产力。通过结合外部和第三方应用程序的联合身份验证，可以进一步构建这种减少。

通过 OAM 提供 SSO 减少了身份验证请求，因此应用程序用户保持安全的密码和账户变得很重要。账户管理成为必需，因为单个账户可能访问大量系统和数据。Oracle Identity Management 是身份管理堆栈的关键组成部分，提供用户自助服务、用户管理和高级别的可审计性。

#### Oracle 访问管理器

Oracle Fusion Middleware 应用程序和产品被设计为支持 Oracle Identity Manager (OIM) 组件，如 OID 和 OUD 以获取身份信息。此外，这些资源可以使用 OAM 进行保护。Oracle Access Manager 利用会话管理、身份上下文和风险分析等功能，向应用程序环境提供身份验证和授权、策略管理与执行以及 SSO 功能。OAM 还提供附加功能，最显著的是为采用多个 Oracle 产品的组织提供 SSO 环境，以及支持与安全断言标记语言 (SAML) 兼容的第三方产品的身份联合。

OAM 的核心是向企业应用程序提供身份验证和授权服务。它还允许在基于 Oracle Fusion Middleware 的环境中实现 SSO 功能。图 1-6 提供了 OAM 架构的高级描述。对受保护资源的访问请求由称为 WebGate 或 mod_osso 实例的过滤器拦截，并发送到 OAM 进行处理。OAM 引用可配置的身份验证和授权策略，以确定是否授予客户端访问权限。成功身份验证后，设置一个 Cookie 以确保持续访问所请求的资源以及 OAM 环境中的其他资源。

### Oracle 自适应访问管理

OAM 不仅提供认证和授权服务，还使组织能够在多个层面（如 Web、数据和应用程序层）提高安全意识和欺诈检测能力。Oracle 自适应访问管理器 (OAAM) 通过增加学习行为和检测潜在欺诈活动的能力，加强了 OAM 认证机制的安全性。OAAM 的实时和批处理风险分析组件用于使用模式、规则引擎、操作和交易分析来验证用户身份和活动的有效性。

以下是 OAAM 的一些风险分析组件：

*   设备指纹识别：此功能收集用户交易过程中所用连接设备的属性。这些属性用于识别可能的欺诈访问请求。
*   行为分析：OAAM 可以学习用户的正常行为、设备信息、位置和其他数据，以便主动检测可能的滥用或安全漏洞。
*   风险引擎分析：此功能允许对事件、用户档案、设备指纹、地理位置和其他数据进行实时分析，以确定用户交互期间的风险水平。分析轨迹可以被审计，以确保采取了适当的措施。
*   预测分析：这有助于与 Oracle Data Mining 集成，以基于历史数据提供异常检测。在测试不同模型时，也可以将此功能用于基于分析数据做出变更决策。
*   调查取证：此功能可用于为审计和欺诈调查工作提供数据集。可以使用 Oracle Business Intelligence Publisher 自定义数据，也可以使用开箱即用的模板。这些报告可以帮助安全调查人员识别实例和可能相关的欺诈访问。
*   通用风险快照：用于备份、恢复和迁移现有安全策略与配置。

面向用户、用于防范恶意软件和网络钓鱼攻击的组件增强了 OAAM 的分析功能，以提供端到端的解决方案。可以向最终用户提供虚拟认证设备、基于知识的认证问题和一次性密码机制，以防止未经授权的账户访问。

Oracle 自适应访问管理的面向用户组件包括：

*   设备指纹识别：提供了 OAAM 收集用户设备元数据的能力。这些数据可能包括 Cookie、硬件配置、地理位置和网络配置。此数据用于检测用户行为的变化。
*   基于知识的认证：通过要求用户回答安全问题，提供次要的认证形式。
*   答案逻辑：即使存在小的拼写错误，此组件也能使 OAAM 检测到基于知识的认证问题的根本正确答案。
*   一次性密码：此功能使用户能够通过短信或电子邮件接收一次性使用的密码，用于系统认证。
*   自助密码管理：用户可以使用自助服务创建和重置其凭据。

### 身份联合

随着公司转向基于云的服务或采用外部 Web 应用程序和其他跨域服务，能够使用相同凭据对这些服务进行认证的需求日益增长。身份联合通过消除传统上为存在于多个域中的应用程序（分散环境）所需的多组凭据，使组织能够简化用户访问管理。OAM 支持与 SAML、OpenID、表单填充和 OAuth 的联合。

Oracle 访问管理器身份联合的两个功能组件是服务提供者 (Service Provider) 和身份提供者 (Identity Provider)。身份提供者组件负责建立用户身份、过滤属性、断言身份信息和维护会话。服务提供者负责映射属性、链接身份并将身份信息传递给应用程序。

在登录期间，作为身份提供者，OAM 首先对用户进行认证。如果用户会话已超时，身份提供者将确定用户是否需要重新认证。最后，身份提供者组件确定合作应用程序是否需要与请求期间指定的级别或方案兼容的质询。为了支持这一点，OAM 允许配置灵活的认证机制。身份联合使用联合认证方法和 OAM 认证方案映射来控制应如何对用户进行认证质询。

在注销期间，Access Manager 身份联合服务提供者支持两种不同的流程，具体取决于注销发起的位置：来自 OAM 或来自联合合作应用程序。

在 OAM 发起的注销期间：

1.  用户通过 OAM 请求注销。
2.  OAM 结束 Access Manager 会话。
3.  OAM 指示各个 WebGate 实例删除用户的会话 Cookie。
4.  OAM 联合服务执行注销操作，通过 HTTP 重定向用户或通过简单对象访问协议发送注销请求，导致合作应用程序注销。
5.  OAM 联合服务终止联合会话。

如果注销是通过合作应用程序发起的：

1.  合作应用程序将用户重定向到 OAM 联合服务。
2.  联合服务将会话标记为注销。
3.  用户被重定向到 OAM 进行注销。
4.  OAM 指示各个 WebGate 实例删除用户的会话 Cookie。
5.  OAM 联合服务执行注销操作，通过 HTTP 重定向用户或通过简单对象访问协议发送注销请求，导致合作应用程序注销。
6.  OAM 联合服务终止联合会话。

OAM 内身份联合的服务提供者组件与身份提供者协同工作，以在联合环境中提供欺诈和风险感知。当身份提供者对用户进行认证时，可以触发服务提供者以适当的认证级别创建会话，该级别与联合认证方法映射。这些属性方法作为来自身份提供者的请求属性或映射到 OAM 会话中传入的断言属性名。

Oracle 访问管理身份联合支持多种联合技术，包括 SAML、OpenID、OAuth、社交身份和表单填充。

#### 联合身份认证

*   基于 SAML 的联合身份认证：`SAML` 是一个行业标准的开放框架，允许共享安全信息。它提供了一种在跨多个域的应用程序之间传输信息的标准方法。它还可以用于链接单个用户在多个站点的账户。`SAML` 协议提供可在多个安全框架内使用的标准安全令牌。`SAML` 还提供了一种表示安全令牌的标准方法，该令牌可以在业务流程或事务之间传递。这是通过使用 `XML` 文档来实现的。
*   基于 OpenID 的联合身份认证：`OpenID` 允许任何网站利用身份验证标准，而无需开发自己的系统。它采用单个令牌，可供多个系统使用。`OAM` 支持交换 `OpenID` 标识符以交换身份数据。这可以通过使用包含 `NameID` 和其他可选属性的令牌，或通过使用 `NameID` 格式（一个散列的用户属性）以及存储在数据存储中的生成值来完成。
*   基于 OAuth 的联合身份认证：该技术支持委托授权。它是一项行业标准，旨在透明地将一个站点上的私有数据共享给另一个站点。`OAM` 在其身份联合组件、移动和社交安全以及 `API` 网关（`APGs`）中支持 `OAuth`。通过这些组件，`OAM` 提供符合 `OAuth 2.0` 的令牌颁发、令牌验证和令牌吊销服务。

#### 移动和社交访问

随着当今组织对云和其他基于 `Web` 的应用程序的依赖日益加深，其客户和内部用户需要从包括移动设备在内的多个设备和位置进行访问。除了从移动设备访问企业资源外，许多组织还正在部署其用户群日常使用的应用程序的移动版本。Oracle 通过 `OAM` 的移动和社交访问组件解决了这一日益增长的需求。

Oracle 的移动和社交访问利用 `OAM` 的核心功能来保护应用程序。这些功能包括凭据收集器、身份验证和授权服务以及 `SSO`。移动和社交访问在设计上以安全为平台，与 `OAM` 堆栈的自适应访问管理部分集成，以提供审计、设备指纹识别、风险分析和身份验证兼容性。

移动和社交访问允许企业提供以下功能：

*   符合 `OAuth 2.0` 的授权委托。
*   基于浏览器和本机移动应用程序访问身份存储。
*   与基于云的身份服务（如 `Google` 和 `Facebook`）进行交互。
*   用于 `LDAP` 操作的 `REST` 接口，可用于用户配置文件服务。

#### API 和 Web 服务安全

`OAM` 提供了保护 `Web` 服务安全的能力，以支持行业对面向服务架构（`SOA`）日益增长的使用。如今，越来越多的组织正在暴露可以集成到其他地方使用的应用程序中的 `Web` 服务。例如，一家公司可能希望在其公司内部网门户中包含一个股票市场应用程序。这些 `Web` 服务必须受到保护，以确保只有授权访问其功能，以免被滥用。`OAM` 提供符合标准的功能，使用 `Web` 服务管理器（`WSM`）、安全令牌服务和 `APG` 组件来保护 `Web` 服务和应用程序编程接口（`APIs`）。

`WSM` 使用 `WS-Security`、`SAML` 和 `OAuth` 等协议保护 `Fusion` 中间件产品和应用程序服务。`WSM` 允许企业通过定义策略、执行安全和监控事件来适应不断变化的安全环境。`OAAM` 进一步增强了这些功能，以帮助组织识别可能的安全漏洞并分析风险，从而实时制定新的安全策略。`OAM` `WSM` 处理 `Web` 服务安全的所有方面，包括身份验证、授权、机密性和完整性，使用公钥基础设施来加密和解密传递的数据。

`APG` 用于保护部署在云中或企业内部的服务和 `API` 的访问安全。为此，`APG` 能够评估流量以查找可能的威胁，有选择地限制可能包含威胁的请求，并主动扫描内容以查找格式错误的请求和病毒。该网关还允许企业集成 `LDAP` 身份和策略存储（如 `OUD` 或 `Active Directory`）以执行身份验证和授权规则。

安全令牌服务通过提供令牌生命周期服务（获取、验证和取消）来管理服务提供商和服务消费者之间的关系。该服务支持 `Web` 服务与 `API` 网关之间的信任、`SSO` 环境，或者由安全令牌服务在客户端、令牌颁发机构和令牌消费者之间启用的身份传播。

#### 云访问门户

`OAM` 访问门户旨在简化对合作伙伴应用程序和与 `OAM` 集成的应用程序的访问。访问门户提供了一个跨平台界面，允许用户从任何设备访问 `OAM` 保护的资源。该门户使管理员能够自动将应用程序预配置到用户的账户中，并使他们能够为用户提供一个目录，供用户选择要包含在其个性化界面上的应用程序。此外，访问门户将允许用户在线更新其凭据，而无需联系管理员或服务台。

门户上可以使用所有 `OAM` 支持的身份验证类型，从而最大化组织的投资并提高用户生产力。它还能够访问联合的合作伙伴应用程序并使用表单填充身份验证注入，这允许用户存储将用于软件即服务（`SaaS`）或 `Web` 应用程序登录表单的凭据。访问门户为组织在企业内部部署 `SSO` 环境提供了框架，通过为用户提供访问其应用程序的中心位置来提高用户群的生产力。


#### Oracle 身份管理器

随着组织的发展和其用户群能够在线访问越来越多的资源，对更严格控制和直观管理工具的需求也随之增长。用户生产力不能以牺牲企业安全和治理为代价。企业必须持续了解谁拥有对什么的访问权限以及他们如何获得该访问权限，以便及时发现可能的滥用或欺诈行为。OIM 构成了身份与访问管理套件产品中的身份治理组件。

必须对用户身份生命周期进行严格控制，以确保用户在入职第一天就拥有正确的访问权限，并在他们改变角色或离开公司时根据需要撤销该访问权限。此外，一些用户可能需要访问对其职位而言不寻常的资源。许多此类权限可以基于一套既定的规则授予，或由用户选择并经经理和管理员批准。然而，需要工具并实施严格的控制以确保正确的访问。

OIM 是一个供应工具，它利用了前面讨论的身份存储和访问系统。Oracle 身份与访问管理套件的其他部分致力于在企业应用环境中存储或虚拟化身份信息并提供认证和授权服务，而 OIM 则优化了整个身份环境的供应、生命周期、审计和对账服务。这个组件完善了满足所有企业安全需求的单一身份管理平台。

为了实现提高用户生产力和效率的任务，OIM 提供了一整套功能，包括自助服务门户、供应和账户管理工作流、定制化和个性化。

#### 自助服务

OIM 的自助服务功能通过赋予个人用户管理其许多账户详情的能力来提升用户效率。这些功能包括重置和找回丢失的密码、申请新权限以及注册他们的账户。员工在丢失密码时不再需要提交服务台工单。相反，用户可以通过回答预设的验证问题来请求密码重置。需要系统新权限或额外权限的用户可以使用自助服务门户进行申请，参考一个可能的授权选项目录。OIM 还提供允许新用户在系统中注册并根据管理员定义的规则集获得必要的预设默认权限的功能。

#### 工作流

在大多数组织中，用户根据部门或职位被授予权限。围绕这些授予的规则可以在 OIM 中定义，并在创建新用户时应用。此外，当员工申请新权限时，由业务定义的复杂工作流可以处理实际授予权限之前所需的审批矩阵。这些功能确保了最高水平的可审计性，以确定谁拥有对什么的访问权限、谁以及何时授予了该权限。所有这些在调查可能的滥用或进行安全审计时都是非常有用的信息。

#### 委派管理

随着组织的成长，委派管理的需求变得明显。一个单一的 IT 组织可能没有足够的资源来处理公司的所有权限请求。OIM 提供了将管理任务分散到负责相关内容的各个业务部门的能力。通过使用高级安全配置，这也确保了委派管理只授予那些需要的领域。例如，财务管理员应该只拥有对财务组权限的访问权限，而不能授予对法务部门内容的访问权限。

#### 审计

尽管 OIM 提供了极大的灵活性，但重要的是要注意频繁的安全审计变得多么有价值。OIM 包含一系列功能，使这些审计易于管理，同时提供可立即使用的信息和服务。使用内置报告和门户可以让管理员和审计员快速浏览或详细了解安全状态。此外，可以识别孤立或异常账户，以便采取所需的操作来对账这些账户。

### 总结

至此，我们已经介绍了 Oracle 身份与访问管理的各个主要组件。这些包括融合中间件架构、目录服务、访问管理和身份管理。尽管这些产品中没有任何一个能单独提供组织可能需要的一切，但可以将它们视为完整的企业级身份管理解决方案的构建块，每个都提供其自己的一套功能。因此，短期目标可能只是为 Oracle 应用提供身份验证的组织，可以轻松地随着需求的增长扩展功能。在图 1-7 中，整体架构被呈现为一个单一单元。需要指出的是，这并不意味着所有组件都必须驻留在单个服务器上。
