# 13. 身份管理器策略管理

Oracle 身份管理器 (`OIM`) 在管理组织的身份数据方面承担了大量功能。通过提供多种工具来协助管理员、经理、服务台人员和最终用户维护一致且可审计的身份，`OIM` 已成为企业中的关键工具。`OIM` 策略，如密码策略、访问策略和审批策略，可以帮助组织在推进效率提升的同时保持严密的安全。使用 `OIM` 策略，可以配置系统以允许许多用户权限组合，包括自我注册、请求新权限、管理其他用户以及授予权限的能力。


### 访问策略

大多数组织拥有不同的用户群体。例如，经理可能有权为其员工授予权限，服务台员工可能能够解锁或重置用户密码，而用户管理员则可以创建、修改和停用用户账户。尽管 `OIM` 为各类用户提供了相当大的灵活性，但必须将此功能谨慎地分配给合适的用户，以确保整个系统的安全性。访问策略通过允许将角色分配给单个用户、用户组或符合特定条件的用户，来提供这种安全性。这些选项对于构建一个可管理的设置来说至关重要。

访问策略定义了用户在系统内拥有的权限。外部客户用户可能能够注册其账户并管理诸如密码和个人信息等属性。然而，企业可能决定不允许外部用户在未致电服务台的情况下解锁其账户。另一方面，服务台用户通常被授予更多的权限。他们的访问权限可能包括创建和停用账户的能力。组织可以设置访问策略，允许员工申请新权限，经理可以使用工作流来批准或授予这些权限。`OIM` 为企业提供了一个框架，用于设置管理身份数据的简单和复杂的规则。

### 密码策略

要求您的用户维护 `OID` 能在一定程度上提供控制密码策略的能力。`OIM` 增加了功能，可真正定制您组织的密码管理行为。

大多数组织会设置策略，要求用户密码符合一组规则。这些规则可能包括诸如最小字符数、大小写字母、特殊字符等。诸如此类的规则组合增加了密码的复杂性。密码越复杂，入侵者就越难猜测或暴力破解。

使用如图 13-8 所示的 Oracle Identity Manager 管理界面，您可以通过点击 `Policies` 并选择 `Password Policies` 来为组织创建和维护密码策略。

![A352855_1_En_13_Fig8_HTML.jpg](img/A352855_1_En_13_Fig8_HTML.jpg)

图 13-8. Identity Manager 管理界面

点击 `Manage Password Policies` 后，您将看到可用密码策略的列表。您可以创建新策略或使用默认策略。创建密码策略后，您必须将策略分配给适当的用户群体。`OIM` 能够根据要求为不同的用户组分配不同的密码规则。例如，人力资源员工可能需要维护更高水平的密码复杂性，因为他们能访问敏感的个人数据。图 13-9 所示的列表显示了为 `OIM` 用户创建的默认密码策略和一个示例密码策略。虽然您可以修改此密码策略，但建议您创建自己的策略。

图 13-9. 密码策略列表

从图 13-9 列出密码策略的地方，您可以选择一个并进行编辑，以根据自己的要求进行定制。图 13-10 显示了创建策略时可以使用的所有选项。

![A352855_1_En_13_Fig10_HTML.jpg](img/A352855_1_En_13_Fig10_HTML.jpg)

图 13-10. 密码策略配置界面

创建密码策略时，注意界面的各个部分非常重要。

*   第一部分定义了密码的基本规则。
    *   `Minimum Length`：指定密码必须达到的字符数才能通过复杂性规则。
    *   `Minimum Password Age (Days)`：此数字定义了用户在更改密码前必须使用该密码的天数。使用数字 `0` 允许在同一天更改密码。
    *   `Warn After (Days)`：这是系统向用户发送密码即将过期提醒通知前的天数。通常在到期前三到五天发送。
    *   `Disallow Past Passwords`：输入一个数字以禁止用户重复使用以前的密码。通常设置为禁止前五个密码。
    *   `Expires After (Days)`：此数字确定密码在用户被迫更改前的最大有效期。
*   第二部分 `Complex Password` 可用于使用预配置的密码复杂性规则。它包括以下规则：
    *   密码长度必须至少为六个字符。
    *   密码必须满足以下五条规则中的三条：
        *   必须包含大写字母。
        *   必须包含数字字符。
        *   必须包含小写字母。
        *   必须包含特殊字符 (`!@#$%^&*`)。
        *   必须包含 Unicode 字符。
    *   不能包含用户的登录名、名字或姓氏。

**注意**：应该指出的是，默认 `Complex Password` 选择中列出的最小字符数实际上基于界面第一部分定义的字符数。如果您在该部分输入了 `10` 个字符，那么 `Complex Password` 规则选择实际上将要求 `10` 个字符，而不是所示的六个。

此配置页面的第三部分 `Custom Policy` 允许对前面的复杂规则进行完全定制。需要注意的是，对于单个密码策略，您只能使用 `Custom Policy` 或 `Complex Password`。在这里，您可以精确定义要强制执行的规则。您可以要求五个大写字母和三个特殊字符，或更多的数字字符。此部分让您完全控制复杂性规则。但是，请记住，如果密码策略限制性太强，可能会导致用户写下密码。

页面的最后一部分是 `Challenge Options`。此部分用于配置 `OIM`，以允许用户在忘记密码或锁定账户时回答一个或多个验证问题。此选项可以通过允许用户管理自己的锁定和遗忘密码，显著减少帮助台呼叫。

勾选 `Enable Challenge Policy Support` 复选框后，配置选项将出现。在这里，您可以配置适合您组织要求的自定义问题。您还可以允许用户输入自己的问题和答案。定义问题后，您还可以选择要求用户输入唯一的答案、定义答案可能的字符数，或两者兼有。要进一步定制此系统，您可以输入任意多个问题，但允许用户选择并回答总数中定义的最小数量问题。

### 小结

在 `OIM` 中使用策略可以涵盖用户权限和密码。这允许管理员在配置安全的身份数管理系统的同时，也最小化必须执行的日常管理工作量。访问策略可用于根据职位、角色授予、组成员资格等来控制用户被允许使用的功能。此外，这些策略可用于为用户组提供细粒度的访问权限。密码策略可以通过确保用户群体使用安全、复杂的密码来帮助维护安全的系统。

## 14. Oracle Identity Manager 表单与定制

在之前的章节中，您已经看到 Oracle Identity Manager (`OIM`) 提供了多种工具，组织可以提供给其用户，使他们或多或少能够自给自足。使用这些工具可以消除帮助台呼叫，并允许更快、更高效地处理权限请求以及入职和离职任务。这些工具还通过减少处理请求所需的人数以及简化管理账户的流程步骤，提供了更高级别的安全性。这些好处对许多组织来说是一个福音，因为他们可以看到节省资金和增强安全性。然而，开箱即用，许多可用的表单可能无法满足每个组织的外观、感觉或功能需求。

Oracle 提供了多种方法来为各个组织定制 `OIM`。`OIM` 允许进行大量的定制。这些定制的范围可以从简单的徽标和颜色更改、提示的本地化，到完全自定义的表单和报告。根据组织的特定要求，定制级别会有所不同。在许多情况下，颜色更改、徽标更改和页面布局就足够了。偶尔，可能需要创建自定义表单，可以使用表单创建工具来创建。在某些情况下，需要整个自定义应用程序或完全定制，可以利用 `OIM` 应用程序编程接口 (`API`) 来创建。使用 `API` 创建自定义应用程序超出了本书的范围。但是，本章涵盖了定制和使用表单创建工具。


### 基础定制

OIM 的定制工作分为几个不同的复杂级别。在最基础的层面，OIM 允许管理员更改诸如背景颜色、徽标以及各种页面的视觉组件等内容。由于其实现方式，这些更改大多会影响所有页面。最基本的修改涉及更改级联样式表（`CSS`）等文件或替换图像文件。

`CSS` 在网页开发中已使用很长时间。这些文件允许设计者为项目一次性定义字体、颜色、背景等元素，而无需在部署中为每个页面单独跟踪它们。

图 14-1 展示了 OIM 身份自助服务主页。使用诸如 `HTTPFox` 或 `Google Chrome` 这类能够查看网页元素的浏览器工具，你将能够确定整个页面所使用的 `CSS` 文件、元素和属性。这些信息将是更改 OIM 应用程序页面的关键。

更改徽标、线条或其他视觉元素后，您可能希望添加新元素，例如提示文本或新字段。与更改视觉元素非常相似，您必须首先找到要修改的区域或区域。这可以使用右侧的结构窗格或左侧的内容区域来完成。请注意，当您在页面上选择区域或项目时，右侧将选择相应的结构元素。根据您希望添加元素的位置，应找到父容器并在那里添加它。您可以使用容器编辑器重新排序项目。图 14-9 显示了选中面板的结构窗格。在此示例中，要添加的新元素是一些帮助文本，用于指导用户更改其密码。这将被添加到包含密码字段的面板表单布局中。

至此，您已更换了徽标并向“我的信息”页面添加了一个文本字段。如果您还记得，自定义都是在单独的沙盒中完成的。到目前为止创建的更改仅在您的沙盒处于活动状态时可见，并且仅对进行编辑的用户可见。系统的其他用户仍然看到未编辑或先前已发布的标准页面。点击 `Publish Sandbox` 选项会将所做的更改合并到应用程序的主线中。此功能对于测试和更改 UI 非常有用，而不会影响当前用户。

## 总结

OIM 提供了大量表单，有助于简化人员流动率高的环境，或希望赋予用户处理请求和重置密码权限的组织。这种自动化和大量的自助服务应用程序表单可以大大减少帮助台呼叫和管理员协助的次数。但是，开箱即用的 UI 可能无法满足所有组织的要求。因此，Oracle 提供了一种易于使用的沙盒式表单编辑器，允许对活动实例进行更改，而不必担心未经测试的更改会影响用户。系统允许开发人员在沙盒中进行更改并测试这些更改。只有当沙盒实际发布时，更改才会进入实时系统。本章演示了两种常见的 UI 更改类型。
