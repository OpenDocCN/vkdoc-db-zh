# 成本模型演进

查询优化器过去基于执行 SQL 语句所需的物理读取次数来估算成本。这种方法被称为*I/O 成本模型*。其主要缺点在于单块读取和多块读取的成本被视为相同。¹ 因此，多块读取操作（如全表扫描）在成本估算上被人为地赋予了优势。在 Oracle8*i*之前，尤其是在 OLTP 系统中，初始化参数`optimizer_index_caching`和`optimizer_index_cost_adj`曾用于解决此问题（关于这两个初始化参数的详细信息，请参见第 5 章）。事实上，这些参数的默认值过去只适用于报表系统和数据仓库。从 Oracle9*i*开始，引入了一种新的成本计算方法，称为*CPU 成本模型*，以解决此缺陷。要使用 CPU 成本模型，需要向查询优化器提供关于数据库引擎运行所在系统的性能信息，称为*系统统计信息*。本质上，系统统计信息提供以下信息：

*   I/O 子系统的性能
*   CPU 的性能

尽管名为 CPU 成本模型，但它同样会考虑物理读取的成本。不过，它不仅仅基于物理读取的次数，还会考虑 I/O 子系统的性能。不要被其名称误导。

在 Oracle9*i*中，默认情况下没有可用的系统统计信息。这意味着默认使用的是 I/O 成本模型。从 Oracle Database 10*g*开始，默认总是存在一组系统统计信息。因此，默认使用的是 CPU 成本模型。实际上，从 Oracle Database 10*g*开始，使用 I/O 成本模型的唯一方法是在 SQL 语句级别指定提示`no_cpu_costing`。在所有其他情况下，查询优化器都会使用 CPU 成本模型。²

系统统计信息有两种：*非工作负载统计信息*和*工作负载统计信息*。两者的主要区别在于衡量 I/O 子系统性能的方法。前者运行合成基准测试，而后者使用应用程序基准测试。在详细讨论这两种方法的差异之前，让我们先看看系统统计信息是如何存储在数据字典中的。

## 应用程序基准测试与合成基准测试

*应用程序基准测试*，也称为*真实基准测试*，基于真实应用程序正常运行产生的负载。虽然它通常能提供关于系统真实性能的非常好的信息，但由于其性质，并非总是能够以可控的方式应用。

*合成基准测试*是由一个不执行实际工作的程序产生的负载。其主要思想是通过执行类似的操作来模拟（建模）应用程序负载。虽然它可以很容易地以可控的方式应用，但通常产生的性能数据不如应用程序基准测试。尽管如此，它对于比较不同系统可能还是有用的。
