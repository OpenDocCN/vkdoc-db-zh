# B 树索引与位图索引

简而言之，在某些情况下只能考虑 B 树索引。如果你的情况不属于那些特定场景，那么大多数时候都应该考虑使用位图索引。表 9-2 总结了在选择 B 树索引还是位图索引时需要考量的关键特性。

位图索引的使用主要受两种情况限制。首先，主键和唯一键只能使用 B 树索引。这别无选择。其次，只有 B 树索引支持行级锁，因为（B 树和位图）索引中的锁在内部是针对索引条目设置的。由于单个位图索引条目可能索引数千行数据，修改一个被位图索引的列可能会阻止数千行其他行（由同一索引条目引用的行）的并发修改，这可能极大地限制可扩展性。第三个问题只出现在 Oracle 数据库 10g 版本之前，是因为位图索引存在空间管理问题。这导致（过于）频繁的索引重建。因此，直到 Oracle9i 版本，它们只用于极少修改的列。

此外，如果你想利用多索引的高效组合或星型转换，就需要使用位图索引。

请注意，表 9-2 并未对索引的选择性或被索引列的基数提供评论。尽管许多关于位图索引的书籍和论文包含如下建议：

> 这些索引适用于低基数数据。通过压缩技术，它们可以用最少的 I/O 生成大量 rowid。组合非选择性列上的位图索引，可以高效地进行 AND 和 OR 操作，涉及大量 rowid 且 I/O 开销极小。
>
> — Oracle 数据库性能调优指南 11g 第 1 版

老实说，在我看来，这样的信息至少是误导性的。事实是，一个选择性差的 SQL 语句永远无法通过从索引获取 rowid 列表来高效执行。这是因为，在此类情况下，构建 rowid 列表所需的时间（无论是 B 树索引还是位图索引）都远远小于使用这些 rowid 访问表所需的时间。话虽如此，位图索引在处理低基数数据时确实比 B 树索引表现更好。但要小心，*更好*并不一定意味着*高效*。例如，一个*差*的产品不会因为它比一个*非常差*的产品更好就变成*好*产品。也许组合位图索引非常高效，但重申一遍，构建 rowid 列表仅仅是开始。行数据仍然需要被访问。另外，我还得提一下`OR`。如果你仔细想想，会意识到用`OR`组合多个非选择性条件只会导致选择性更差，而如果你想使用索引，目标应该是提高选择性。总之，在 B 树索引和位图索引之间做选择时，忘掉选择性和基数吧。

除了表 9-2 总结的差异外，B 树索引和位图索引在处理相同的 SQL 条件时效率并不相同。实际上，位图索引通常更强大。表 9-3 总结了两种索引应对不同类型条件的能力。接下来的章节旨在提供一些相关示例。我还将描述不同的属性和限制。所有示例都基于脚本`conditions.sql`。测试表及其索引是使用以下 SQL 语句创建的：

```sql
CREATE TABLE t (
  id NUMBER,
  d1 DATE,
  n1 NUMBER,
  n2 NUMBER,
  n3 NUMBER,
  n4 NUMBER,
  n5 NUMBER,
  n6 NUMBER,
  c1 VARCHAR2(20),
  c2 VARCHAR2(20),
  pad VARCHAR2(4000),
  CONSTRAINT t_pk PRIMARY KEY (id)
);

INSERT INTO t
SELECT rownum AS id,
       trunc(to_date('2007-01-01','yyyy-mm-dd')+rownum/27.4) AS d1,
       nullif(1+mod(rownum,19),10) AS n1,
       nullif(1+mod(rownum,113),10) AS n2,
       nullif(1+mod(rownum,61),10) AS n3,
       nullif(1+mod(rownum,19),10) AS n4,
       nullif(1+mod(rownum,113),10) AS n5,
       nullif(1+mod(rownum,61),10) AS n6,
       dbms_random.string('p',20) AS c1,
       dbms_random.string('p',20) AS c2,
       dbms_random.string('p',255) AS pad
FROM dual
CONNECT BY level <= 10000
ORDER BY dbms_random.value;

CREATE INDEX i_n1 ON t (n1);
CREATE INDEX i_n2 ON t (n2);
CREATE INDEX i_n3 ON t (n3);
CREATE INDEX i_n123 ON t (n1, n2, n3);
CREATE BITMAP INDEX i_n4 ON t (n4);
CREATE BITMAP INDEX i_n5 ON t (n5);
CREATE BITMAP INDEX i_n6 ON t (n6);
CREATE INDEX i_c1 ON t (c1);
CREATE BITMAP INDEX i_c2 ON t (c2);
```

## 等值条件与 B 树索引

对于 B 树索引，等值条件通过两种操作之一执行。第一种是 `INDEX UNIQUE SCAN`，专用于唯一索引。顾名思义，它最多返回一个 rowid。以下查询是一个示例。执行计划通过操作 2 的访问谓词确认，对列 `id` 的条件是使用索引 `t_pk` 应用的。然后，操作 1 使用从索引中提取的 rowid 访问表 `t`。这是通过 `TABLE ACCESS BY INDEX ROWID` 操作完成的。注意，这两个操作都只执行一次。

```sql
SELECT /*+ index(t) */ * FROM t WHERE id = 6
```

```
------------------------------------------------------------
| Id | Operation                   | Name | Starts | A-Rows |
------------------------------------------------------------
|  1 | TABLE ACCESS BY INDEX ROWID | T    |      1 |      1 |
|* 2 |   INDEX UNIQUE SCAN         | T_PK |      1 |      1 |
------------------------------------------------------------

    2 - access("ID"=6)
```

> **注意** 在本节中，为了强制索引扫描，提示 `index` 仅通过指定表名来使用。当以这种方式使用时，查询优化器可以自由地从可用索引中选择一个。在所有示例中，`WHERE` 子句中只存在一个谓词。因此，查询优化器总是根据该谓词中引用的列来选择索引。然而，最佳实践是明确指定应使用哪个索引。我将在下一节中给出一些示例。

第二种操作是 `INDEX RANGE SCAN`，用于非唯一索引。此操作与前一个操作的唯一区别在于它可以提取多个 rowid（示例中为 527 个），而不仅仅是一个。

```sql
SELECT /*+ index(t) */ * FROM t WHERE n1 = 6
```

```
------------------------------------------------------------
| Id | Operation                   | Name | Starts | A-Rows |
------------------------------------------------------------
|  1 | TABLE ACCESS BY INDEX ROWID | T    |      1 |    527 |
|* 2 |   INDEX RANGE SCAN         | I_N1 |      1 |    527 |
------------------------------------------------------------

    2 - access("N1"=6)
```

默认情况下，索引扫描按升序执行。因此，提示 `index` 指示查询优化器也以这种方式操作。要明确指定扫描顺序，可以使用提示 `index_asc` 和 `index_desc`。以下查询展示了如何操作。在执行计划中，降序扫描通过操作 `INDEX RANGE SCAN DESCENDING` 显示。请注意，两个操作返回的数据相同，只是顺序不同。

```sql
SELECT /*+ index_desc(t) */ * FROM t WHERE n1 = 6
```

```
--------------------------------------------------------------------
| Id | Operation                          | Name | Starts | A-Rows |
--------------------------------------------------------------------
|  1 | TABLE ACCESS BY INDEX ROWID        | T    |      1 |    527 |
|* 2 |   INDEX RANGE SCAN DESCENDING      | I_N1 |      1 |    527 |
--------------------------------------------------------------------

    2 - access("N1"=6)
        filter("N1"=6)
```

然而，从实际角度来看，为等值条件进行降序扫描是没有意义的。关于范围条件的章节将描述这种访问路径何时真正有用。

## 等值条件与位图索引

对于位图索引，等值条件通过三种操作执行。按执行顺序，第一个操作是 `BITMAP INDEX SINGLE VALUE`，它扫描索引并应用限制。顾名思义，此操作查找单个值。第二个操作 `BITMAP CONVERSION TO ROWIDS` 将从第一个操作获得的位图转换为 rowid 列表。第三个操作使用第二个操作构建的 rowid 列表访问表。注意，所有三个操作都只执行一次。

```sql
SELECT /*+ index(t i_n4) */ * FROM t WHERE n4 = 6
```

```
----------------------------------------------------------------
| Id | Operation                      | Name | Starts | A-Rows |
----------------------------------------------------------------
|  1 | TABLE ACCESS BY INDEX ROWID    | T    |      1 |    527 |
|  2 |   BITMAP CONVERSION TO ROWIDS  |      |      1 |    527 |
|* 3 |   BITMAP INDEX SINGLE VALUE    | I_N4 |      1 |      1 |
----------------------------------------------------------------

    3 - access("N4"=6)
```

> **注意** 在本节中，为了强制索引扫描，提示 `index` 同时指定了表名和索引名。换句话说，提示指定了应使用哪个索引。当以这种方式使用时，仅当存在具有该名称的索引时，提示才有效。由于索引的名称可以轻松更改（例如，使用 SQL 语句 `ALTER INDEX RENAME`），因此在实践中，具有该语法的提示也极易因错误而失效。

## IS NULL 条件与 B 树索引

对于 B 树索引，`IS NULL` 条件只能通过复合 B 树索引应用，前提是应用了多个 SQL 条件且其中至少一个不是基于 `IS NULL` 或不等式。以下查询演示了这一点。执行计划通过操作 2 的访问谓词确认，对列 `n2` 的条件是使用索引 `i_n123` 应用的。另请注意，操作 2 仅返回 5 行，而对于前面没有 `n2 IS NULL` 限制的章节中的示例，范围扫描返回了 527 行。

```sql
SELECT /*+ index(t) */ * FROM t WHERE n1 = 6 AND n2 IS NULL
```

```
--------------------------------------------------------------
| Id | Operation                   | Name   | Starts | A-Rows |
--------------------------------------------------------------
|  1 | TABLE ACCESS BY INDEX ROWID | T      |      1 |      5 |
|* 2 |   INDEX RANGE SCAN          | I_N123 |      1 |      5 |
--------------------------------------------------------------

    2 - access("N1"=6 AND "N2" IS NULL)
```

如下例所示，当为索引的引导列指定 `IS NULL` 条件时，也会使用完全相同的执行计划。

```sql
SELECT /*+ index(t) */ * FROM t WHERE n1 IS NULL AND n2 = 8
```

```
--------------------------------------------------------------
| Id | Operation                   | Name   | Starts | A-Rows |
--------------------------------------------------------------
|  1 | TABLE ACCESS BY INDEX ROWID | T      |      1 |      4 |
|* 2 |   INDEX RANGE SCAN          | I_N123 |      1 |      4 |
--------------------------------------------------------------

    2 - access("N1" IS NULL AND "N2"=8)
        filter("N2"=8)
```
