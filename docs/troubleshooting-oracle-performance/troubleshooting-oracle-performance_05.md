# 集成与验收测试

集成和验收测试的目的是验证应用程序的功能和性能要求以及稳定性。性能测试与功能测试同等重要这一点，无论怎样强调都不过分。从所有角度来看，一个性能不佳的应用程序，其糟糕程度不亚于一个无法满足功能需求的应用程序。在这两种情况下，应用程序都是无用的。然而，只有先明确定义了性能要求，才可能验证这些要求。

缺乏正式的性能要求会导致两个主要问题。首先，在集成和验收测试期间很可能不会进行严肃且有条理的压力测试。这样，应用程序就会在不确定是否能支撑预期负载的情况下投入生产。其次，在性能方面，什么是可以接受的，什么不是，这并不总是显而易见的。通常，只有极端情况（换句话说，性能非常好或非常差时）会被不同的人以相同方式评判。如果无法达成一致，随之而来的便是冗长、烦人且低效的会议。

在实践中，设计、实施并执行良好的集成和验收测试来验证应用程序的性能并非易事。要取得成功，必须应对三大主要挑战：

*   压力测试应设计成能生成具有代表性的工作负载。为此，存在两种主要方法。第一种是让真实用户执行真实工作。第二种是使用模拟用户的工具。两种方法各有利弊，其使用应视具体情况而定。在某些情况下，两种方法可以结合使用，以对应用程序的不同部分进行压力测试，或作为互补方式。
*   为了生成具有代表性的工作负载，需要具有代表性的测试数据。不仅行数和行的大小应匹配预期数量，数据分布和内容也应与真实数据匹配。例如，如果某个属性应包含城市名称，使用真实的城市名称要比使用像 `Aaaacccc` 或 `Abcdefghij` 这样的字符串好得多。这一点很重要，因为在应用程序和数据库中，肯定存在许多不同数据可能导致不同行为的情况（例如，在使用索引或对数据应用哈希函数时）。
*   测试基础设施应尽可能接近，理想情况下应与生产基础设施相同。这对于高度分布式的系统以及需要与大量其他系统协作的系统来说尤其困难。

在顺序生命周期模型中，集成和验收测试阶段发生在项目接近尾声时，如果导致性能问题的架构重大缺陷被发现得太晚，这可能是个问题。为避免此类问题，应在编码和单元测试阶段也进行压力测试。请注意，迭代生命周期模型没有此问题。事实上，在迭代生命周期模型中，每次迭代都应进行压力测试。

## 你遇到性能问题了吗？

应用程序的性能很可能迟早会受到质疑。如果像前面章节描述的那样，你已经仔细定义了性能要求，那么判断该应用程序是否确实遇到性能问题应该相当简单。如果你没有仔细定义这些要求，答案将在很大程度上取决于回答问题的人。

有趣的是，在实践中，导致对应用程序性能提出质疑的最常见场景可归为很少的几类。它们简要列举如下：

*   用户对应用程序的当前性能不满意。
*   系统监控工具提醒你，基础设施的某个组件正在经历超时或异常负载。
*   响应时间监控工具通知你，服务级别协议（SLA）未得到满足。

第二点和第三点之间的区别尤为重要。因此，在接下来的两节中，我将简要描述这些监控解决方案。之后，我将介绍一些看似需要调优但实际上完全不需要的情况。

### 系统监控

系统监控工具基于通用系统统计信息执行健康检查。它们的目的是识别出现的异常负载模式以及故障。尽管这些工具可以同时监控整个基础设施，但需要强调的是，它们仅监控单个组件（例如，主机、应用服务器、数据库或存储子系统），而不考虑它们之间的相互作用。因此，当支撑系统的基础设施单个组件出现异常时，要确定对系统响应时间的影响是困难的，对于复杂基础设施几乎是不可能的。一个例子是特定资源的高使用率。换句话说，来自系统监控工具的警报只是一个警告，表明应用程序或基础设施可能存在问题，但用户可能根本没有体验到任何性能问题（称为 `false positive`）。相反，可能存在用户正在经历性能问题，但系统监控工具无法识别的情况（称为 `false negative`）。最常见也是最简单的 `false positive` 和 `false negative` 案例出现在监控具有大量 CPU 的 SMP 系统的 CPU 负载时。假设你有一个 16 核 CPU 系统（或具有四个四核 CPU 的系统）。每当利用率约为 75%时，你可能会认为这太高了，系统是 CPU 受限的。然而，如果运行的任务数量远大于 CPU 数量，这种负载可能是非常健康的。这就是 `false positive`。反之，当你看到 CPU 利用率约为 8%时，你可能认为一切正常。但如果系统正在运行一个未并行化的单一任务，那么该任务的瓶颈很可能就是 CPU。事实上，100%的 1/16 只有 6.25%，因此，单个任务不可能消耗超过 6.25%的可用 CPU。这就是 `false negative`。

### 响应时间监控

响应时间监控工具（也称为 `application-monitoring tools`）通过由 `robots` 处理的合成事务执行健康检查。这些工具测量应用程序处理关键事务所花费的时间，如果时间超过预期的阈值，它们会发出警报。换句话说，它们像用户一样利用基础设施，并像用户一样抱怨性能不佳。由于它们从用户的角度探测应用程序，因此不仅能够检查单个组件，更重要的是能够检查整个应用程序的基础设施。因此，它们专门用于监控服务级别协议。
