## 卡里·米尔斯帕的前言

`我`认为过去十年里，Oracle 性能领域发生的最棒的事情，就是如今在书店里能买到的信息质量发生了质的飞跃。

在以往，你买到的关于 Oracle 性能的书籍看起来都大同小异。它们暗示你的 Oracle 系统不可避免地遭受着过多 I/O 的困扰（这其实`并非`必然），或者内存不足（它们声称这与 I/O 过多是同一回事，这也不正确）。它们会向你展示大量你可能会运行的 SQL 脚本，并告诉你应该调整你的 SQL。它们说，这样就能解决一切问题。

那是一个黑暗的时代。

克里斯的这本书是那个给我们带来……光明的家族谱系中的一员。黑暗与光明之间的区别，归结为一个简单的概念。这个概念大约从你十岁起，数学老师就要求你掌握：`展示你的计算过程`。

我指的不是那种“展示与讲述”——有人声称他在数百个客户现场将性能提升了数百个百分点，因此他就是专家。我指的是`展示你的计算过程`，这意味着记录一个相关的基线测量，进行一个受控实验，记录第二个相关的测量结果，然后公开透明地展示你的结果，以便你的读者能够理解，甚至在他们愿意时复现你的测试。

这非常重要。当作者们开始这样做时，Oracle 社区的听众们开始变得聪明得多。自 2000 年以来，Oracle 社区中有越来越多的人提出关于性能的明智问题并要求明智的答案。同时，那些许多人曾经深信不疑的糟糕想法，其被淹没的速度也加快了。

在这本书中，克里斯遵循了行之有效的模式。他告诉你有用的东西。但他不止于此。他向你展示`他是如何知道的`，也就是说，他向你展示`你自己如何能够发现`。他展示了他计算的过程。

这给你带来两大好处。首先，展示计算过程帮助你更深入地理解他所展示的内容，这使他的课程更容易被你记住和应用。其次，通过理解他的例子，你不仅能理解克里斯向你展示的东西，还将能够回答克里斯尚未涵盖的其他好问题……比如在本书付印之后，Oracle 的下一个版本中会发生什么。

对我来说，这本书既是一本技术参考，也是一本“说服性”参考。它包含了大量有完整记录的、我可以复用的功课。它还包含了一些雄辩的新论点，涉及我与克里斯持有相同观点和热情的几个方面。克里斯在这本书中使用的论点将帮助我说服更多人去做正确的事情。

克里斯是一个聪明、充满活力的人，他站在戴夫·恩索尔、莱克斯·德·哈恩、安乔·科尔克、史蒂夫·亚当斯、乔纳森·刘易斯、汤姆·凯特以及其他几位我认为为我们这个领域带来严谨性的英雄们的肩膀上。现在，我们也有克里斯的肩膀可以站立了。

## 乔纳森·刘易斯的前言

我开始使用 Oracle RDBMS 是在 20 多年前，大约花了三年时间，我才意识到故障排除和调优已经被赋予了一种近乎神秘的声誉。

一位开发人员曾将一个查询交给 DBA 小组，因为它运行得不好。我检查了执行计划，检查了数据模式，并指出通过给其中一个表添加索引可以消除大部分工作。开发人员的回应是：“但它不需要索引；这是一个小表。”（顺便说一句，那是在 6.0.36 版本的时代，当时“短”表的定义是“不超过四个块长”。）尽管如此，我还是创建了索引，查询运行速度提高了大约 30 倍——然后我不得不进行大量的解释。

故障排除不依赖于魔法、神秘或神话；它依赖于理解、观察和解释。正如理查德·费曼曾经说过的：“你的理论有多美无关紧要；你有多聪明也无关紧要。如果你的理论与实验不符，那它就是错的。”关于 Oracle 性能，有许多“理论”是错误的，本应在多年前就从集体记忆中删除——而克里斯蒂安·安托尼尼是帮助清除它们的人之一。

在这本书中，克里斯蒂安·安托尼尼旨在描述事情是如何真实运作的，你应该注意哪些类型的症状，以及这些症状意味着什么。最重要的是，他鼓励你在观察和分析中保持条理，并坚持相关的细节。有了这些建议，你应该能够在性能问题出现时识别出真正的问题，并以最合适的方式处理它们。

虽然这是一本可能应该从头到尾仔细阅读的书，但我认为不同的读者会以不同的方式从中受益。有些读者可能在浏览时捕捉到零星的特别见解，就像我在第四章中对`高度平衡直方图`的解释所做的那样——多年来我一直试图为这个名字找到一个直观清晰的理由，克里斯蒂安的描述突然让它变得显而易见。

有些读者可能会发现对某些功能的简短描述有助于他们理解 Oracle 为何实现该功能，并允许他们从示例中推断出与其应用相关的场景。对我而言，第五章中关于`安全视图合并`的描述就是这样一个例子。

其他读者可能会发现他们有一部分书会反复阅读，因为它涵盖了他们正在使用的某些特别重要且相关的功能的许多细节。我确信，第九章中关于`分区`的广泛讨论是许多人会反复研读的内容。

这本书内容丰富——都值得一读。谢谢你，克里斯蒂安。

## 引言

`Oracle 数据库引擎`已经发展成一个庞大的软件。这不仅意味着单个使用者已无法精通掌握其在近期版本中提供的所有功能，也意味着其中一些功能很少被用到。实际上，在大多数情况下，要高效并成功地使用`Oracle 数据库引擎`，掌握并利用有限的核心功能就足够了。这正是为何在本书中，我将仅根据我的经验，介绍那些对于解决您可能遇到的大多数与数据库相关的性能问题所必需的功能。

## 本书结构

本书分为五个部分：

> 第一部分涵盖了一些阅读后续章节所必需的基础知识。第一章“性能问题”不仅解释了为何必须在正确的时机以方法论的方式处理性能问题，还说明了理解业务需求和问题为何至关重要。第二章“关键概念”描述了数据库引擎在解析和执行`SQL 语句`时所执行的操作，并介绍了本书中常用的一些术语。
>
> 第二部分阐述了在基于`Oracle 数据库引擎`的环境中如何处理性能问题。第三章“识别性能问题”提供了一个用于识别性能问题的详细分析路线图，同时描述了可与之配合使用的若干工具和技术。
>
> 第三部分描述了负责将`SQL 语句`转化为执行计划的组件：查询优化器。第四章“系统与对象统计信息”描述了系统统计信息和对象统计信息是什么、如何收集它们，以及它们对查询优化器为何重要。第五章“配置查询优化器”涵盖了一条配置路线图，您可以用它来为查询优化器寻找一个良好的配置。第六章“执行计划”详细描述了如何获取、解读和判断执行计划的效率。第七章“SQL 优化技术”讨论了`Oracle 数据库引擎`提供的`SQL 优化技术`。
>
> 第四部分展示了`Oracle 数据库引擎`提供的用于高效执行`SQL 语句`的功能。第八章“解析”描述了`SQL 语句`如何被解析，以及如何识别、解决和规避解析问题。第九章“优化数据访问”描述了可用的数据访问方法以及如何在它们之间做出选择。第十章“优化连接”讨论了如何高效地将多个数据集连接在一起。第十一章“超越数据访问与连接优化”描述了高级优化技术，例如并行处理和物化视图。第十二章“优化物理设计”解释了为何优化数据库的物理设计很重要。
>
> 第五部分提供了本书中用作示例的文件列表。此外，还提供了一份参考文献目录，列出了我在撰写本书时所使用的资料来源。

## 目标读者

本书面向参与排查基于`Oracle 数据库引擎`的应用程序性能问题的性能分析师、应用程序开发人员和数据库管理员。

不要求具备特定的优化知识。然而，期望读者对`Oracle 数据库引擎`有实际使用知识，并能熟练使用`SQL`。本书的部分章节涵盖了特定编程语言（如`PL/SQL`、`Java`、`C#`和`C`）特有的功能。介绍这些功能仅为给使用不同编程语言的广大开发人员提供针对性信息。您可以只挑选自己使用或感兴趣的部分，跳过其他部分。

## 覆盖的版本？

本书涵盖的最重要概念与您使用的`Oracle 数据库引擎`版本无关。然而，不可避免的是，在讨论实现细节或所提供的功能时，部分信息具有版本特定性。本书明确讨论了从`Oracle9i`第 2 版到`Oracle Database 11g`第 1 版的当前可用版本。它们如下所示：

*   `Oracle9i` 第 2 版，直至版本 9.2.0.8
*   `Oracle Database 10g` 第 1 版，直至版本 10.1.0.5
*   `Oracle Database 10g` 第 2 版，直至版本 10.2.0.4
*   `Oracle Database 11g` 第 1 版，版本 11.1.0.6

如果文本没有明确指出某个功能仅适用于特定版本，则意味着它适用于所有这些版本。
