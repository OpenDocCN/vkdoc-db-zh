# 索引与条件操作

单列索引无法用于应用 `IS NULL` 条件。这是因为 `NULL` 值不存储在索引中。因此，查询优化器在此类情况下无法利用索引。即使你尝试使用 `index` 提示强制使用它，也会执行全表扫描。

```
SELECT /*+ index(t i_n1) */ * FROM t WHERE n1 IS NULL
```

```
-----------------------------------------------------
| Id | Operation          | Name  | Starts | A-Rows |
-----------------------------------------------------
|* 1 | TABLE ACCESS FULL  | T     |      1 |    526 |
-----------------------------------------------------
```

```
    1-filter("N1" IS NULL)
```

## IS NULL 条件与位图索引

对于位图索引，`IS NULL` 条件的处理方式与相等条件相同。这是因为位图索引以与任何其他值相同的方式存储 `NULL` 值。

```
SELECT /*+ index(t i_n4) */ * FROM t WHERE n4 IS NULL
```

```
--------------------------------------------------------------
| Id | Operation                  | Name   | Starts | A-Rows |
--------------------------------------------------------------
|  1 | TABLE ACCESS BY INDEX ROWID| T      |      1 |    526 |
|  2 |   BITMAP CONVERSION TO ROWIDS |      |    526 |    526 |
|* 3 | BITMAP INDEX SINGLE VALUE  | I_N4   |      1 |      1 |
--------------------------------------------------------------
```

```
    3 - access("N4" IS NULL)
```

## 范围条件与 B 树索引

对于 B 树索引，范围条件的处理方式与非唯一索引上的相等条件相同，或者换句话说，使用 `INDEX RANGE SCAN` 操作。对于范围条件，索引类型（即其唯一性）无关紧要。因为是范围扫描，总是可以返回多个 rowid。例如，下面的查询显示了一个应用于主键列的范围条件：

```
SELECT /*+ index(t (t.id)) */ * FROM t WHERE id BETWEEN 6 AND 19
```

```
-------------------------------------------------------------
| Id | Operation                  | Name  | Starts | A-Rows |
-------------------------------------------------------------
|  1 | TABLE ACCESS BY INDEX ROWID| T     |      1 |     14 |
|* 2 |   INDEX RANGE SCAN         | T_PK  |      1 |     14 |
-------------------------------------------------------------
```

```
    2 - access("ID">=6 AND "ID"<=19)
```

> **注意** 本节中使用了多个提示来强制索引扫描，通过指定表名以及索引应创建在哪些列上。与指定索引名的方法相比，这种方法的优点是提示不依赖于索引名。这为提示提供了更强的健壮性。请注意，此语法从 Oracle Database 10*g* 开始可用。

如前一节所述，默认情况下按升序执行索引扫描。这意味着当 `ORDER BY` 应用于与范围条件相同的列时，结果集已经排序。因此，不会执行显式排序。但是，当需要降序的 `ORDER BY` 时，需要执行显式排序，如下所示。排序由操作 1 `SORT ORDER BY` 执行。注意如何使用 `index_asc` 提示强制升序索引扫描。

```
SELECT /*+ index_asc(t (t.id)) */ *
FROM t
WHERE id BETWEEN 6 AND 19
ORDER BY id DESC
```

```
-------------------------------------------------------------
| Id | Operation                  | Name  | Starts | A-Rows |
-------------------------------------------------------------
|  1 | SORT ORDER BY              |       |      1 |     14 |
|  2 |   TABLE ACCESS BY INDEX ROWID| T    |      1 |     14 |
|* 3 |    INDEX RANGE SCAN        | T_PK  |      1 |     14 |
-------------------------------------------------------------
```

```
    3 - access("ID">=6 AND "ID"<=19)
```

同样的查询自然可以利用降序索引扫描来避免显式排序。这是一个例子，其中 `SORT ORDER BY` 操作不再出现在执行计划中：

```
SELECT /*+ index_desc(t (t.id)) */ *
FROM t
WHERE id BETWEEN 6 AND 19
ORDER BY id DESC
```

```
---------------------------------------------------------------------
| Id | Operation                           | Name | Starts | A-Rows |
---------------------------------------------------------------------
|  1 | TABLE ACCESS BY INDEX ROWID         | T    | 1      |     14 |
|* 2 |   INDEX RANGE SCAN DESCENDING       | T_PK | 1      |     14 |
---------------------------------------------------------------------
```

```
  2 - access("ID">=6 AND "ID"<=19)
```

## 范围条件与位图索引

对于位图索引，范围条件的处理方式与相等条件相同。唯一的区别是使用 `BITMAP INDEX RANGE SCAN` 操作代替 `BITMAP INDEX SINGLE VALUE` 操作。

```
SELECT /*+ index(t (t.n4)) */ * FROM t WHERE n4 BETWEEN 6 AND 19
```

```
-------------------------------------------------------------
| Id | Operation                   | Name | Starts | A-Rows |
-------------------------------------------------------------
|  1 | TABLE ACCESS BY INDEX ROWID | T    |      1 | 6840   |
|  2 |   BITMAP CONVERSION TO ROWIDS|      |      1 | 6840   |
|* 3 |    BITMAP INDEX RANGE SCAN  | I_N4 |      1 |   13   |
-------------------------------------------------------------
```

```
  3 - access("N4">=6 AND "N4"<=19)
```

对于位图索引，由于没有升序和降序扫描的概念，因此无法避免从而优化 `ORDER BY` 操作。

## IN 条件

`IN` 条件没有特定的访问路径。相反，在执行计划中，`INLIST ITERATOR` 操作指出部分执行计划由于 `IN` 条件而执行多次。以下三个查询显示了索引扫描本身使用的操作如何取决于索引类型。第一个是唯一索引，第二个是非唯一索引，第三个是位图索引。基本上，`IN` 条件就是一系列相等条件。请注意，与索引和表访问相关的操作对于 `IN` 列表中的每个值都会执行一次（参见 `Starts` 列）。

```
SELECT /*+ index(t t_pk) */ * FROM t WHERE id IN (6, 8, 19, 28)
```

```
--------------------------------------------------------------------
| Id | Operation                          | Name | Starts | A-Rows |
--------------------------------------------------------------------
| 1  | INLIST ITERATOR                    |      |      1 |      4 |
| 2  |   TABLE ACCESS BY INDEX ROWID      | T    |      4 |      4 |
|* 3 |    INDEX UNIQUE SCAN               | T_PK |      4 |      4 |
--------------------------------------------------------------------
```

```
  3 - access("ID"=6 OR "ID"=8 OR "ID"=19 OR "ID"=28)
```

```
SELECT /*+ index(t i_n1) */ * FROM t WHERE n1 IN (6, 8, 19, 28)
```

```
--------------------------------------------------------------------
| Id | Operation                          | Name | Starts | A-Rows |
--------------------------------------------------------------------
|  1 | INLIST ITERATOR                    |      |      1 |   1579 |
|  2 |   TABLE ACCESS BY INDEX ROWID      | T    |      4 |   1579 |
|* 3 |    INDEX RANGE SCAN                | I_N1 |      4 |   1579 |
--------------------------------------------------------------------
```

```
  3 - access("N1"=6 OR "N1"=8 OR "N1"=19 OR "N1"=28)
```

```
SELECT /*+ index(t i_n4) */ * FROM t WHERE n4 IN (6, 8, 19, 28)
```


## LIKE 条件

`-------------------------------------------------------------------`
`| Id | Operation                           | Name | Starts | A-Rows|`
`-------------------------------------------------------------------`
`|  1 |` `INLIST ITERATOR`                   |      |      1|   1579 |`
`|  2 | TABLE ACCESS BY INDEX ROWID        | T    |      `**`4**` |   1579|`
`|  3 | BITMAP CONVERSION TO ROWIDS        |      |      `**`4**` |   1579|`
`|* 4 | BITMAP INDEX SINGLE VALUE          | I_N4 |      `**`4**` |      3|`
`-------------------------------------------------------------------`

`  4 - access(("N4"=6 OR "N4"=8 OR "N4"=19 OR "N4"=28))`

只要匹配模式不以通配符（下划线`_`和百分号`%`字符）开头，`LIKE` 条件就会以与范围条件相同的方式执行。否则，就无法避免全表扫描或全索引扫描。以下示例展示了这种行为。前两个查询分别针对列 `c1` 和 `c2`，检索所有以字母 *A* 开头的行。因此，可以进行范围扫描。第三和第四个查询分别针对列 `c1` 和 `c2`，检索所有在任何位置包含字母 *A* 的行。因此，会执行全索引扫描。

`SELECT /* index(t i_c1) */ * FROM t WHERE` `c1` `LIKE '`**`A%`**`'`

`--------------------------------------------------------------------`

`--------------------------------------------------------------------`

`      filter("C1" LIKE 'A%')`

`SELECT /* index(t i_c2) */ * FROM t WHERE` `c2` `LIKE '`**`A%`**`'`

`--------------------------------------------------------------------`

`  3 - access("C2" LIKE 'A%')`
`      filter(("C2" LIKE 'A%' AND "C2" LIKE 'A%'))`

## Min/Max 函数与 B-tree 索引

为了高效执行包含 `min` 和 `max` 函数的查询，B-tree 索引提供了两种特定的操作。第一种是 `INDEX FULL SCAN (MIN/MAX)`，当查询未指定范围条件时使用。然而，尽管名称如此，它并不执行完整的索引扫描。它只是获取最右侧或最左侧的索引键。

`SELECT /*+ index(t t_pk) */` `min` `(id) FROM t`

`--------------------------------------------------------------------`
`| Id | Operation                           | Name | Starts | A-Rows |`
`--------------------------------------------------------------------`
`|  1 | SORT AGGREGATE                     |      |      1 |      1 |`
`|  2 |` `INDEX FULL SCAN (MIN/MAX)`        | T_PK |      1 |      1 |`
`--------------------------------------------------------------------`

第二种是 `INDEX RANGE SCAN (MIN/MAX)`，当查询在函数使用的同一列上指定了条件时使用：

`SELECT /*+ index(t t_pk) */` `min` `(id) FROM t WHERE id > 42`

`--------------------------------------------------------------------`
`| Id | Operation                           | Name | Starts | A-Rows |`
`--------------------------------------------------------------------`
`|  1 | SORT AGGREGATE                     |      |      1 |      1 |`
`|  2 |  FIRST ROW                         |      |      1 |      1 |`
`|* 3 |` `INDEX RANGE SCAN (MIN/MAX)`       | T_PK |      1 |      1 |`
`--------------------------------------------------------------------`

`  3 - access("ID">42)`

不幸的是，当两个函数（`min` 和 `max`）在同一查询中使用时，无法应用此优化技术。在这种情况下，会执行全索引扫描。以下查询是一个示例：

`SELECT /*+ index(t t_pk) */ min(id), max(id) FROM t`

`--------------------------------------------------------------------`
`| Id | Operation                           | Name | Starts | A-Rows|`
`--------------------------------------------------------------------`
`|  1 | SORT AGGREGATE                     |      |      1 |      1|`
`|  2 |  INDEX FULL SCAN                   | T_PK |      1 |  10000|`
`--------------------------------------------------------------------`

### Min/Max 函数与位图索引

对于位图索引，没有可用于执行 `min` 和 `max` 函数的特定操作。用于等值条件和范围条件的操作同样适用。

## 基于函数的索引

为了利用索引，需要遵循的基本规则之一是：切勿在 `WHERE` 子句中修改索引列返回的值。否则，查询优化器将无法使用在引用列上创建的索引。例如，如果在列 `c1` 上存在索引，那么类似 `upper(c1)='SELDON'` 这样的限制条件无法通过 `c1` 列上的索引高效应用。这一点显而易见，因为你只能搜索存储在索引中的值，而不是其他内容。以下示例（与本节其他示例一样）基于脚本 `fbi.sql`：

```
SQL> CREATE INDEX i_c1 ON t (c1);
```

```
SQL> SELECT * FROM t WHERE upper(c1) = 'SELDON';
```

```
-----------------------------------------
| Id | Operation               | Name  |
-----------------------------------------
|* 1 |` `TABLE ACCESS FULL`    | T     |
-----------------------------------------
```

```
  1 - filter(UPPER("C1")='SELDON')
```

换句话说，每次将索引列作为参数传递给函数或参与表达式时，都无法对该列上创建的索引进行索引范围扫描。唯一的例外是当约束条件确保索引包含必要信息时。在这个特定案例中，列 `c1` 上的两个约束为查询优化器提供了该信息：

```
SQL> ALTER TABLE t ADD CONSTRAINT t_c1_upper CHECK (c1 = upper(c1));
```

```
SQL> ALTER TABLE t MODIFY (c1 NOT NULL);
```


`SQL> SELECT * FROM t WHERE upper(c1) = 'SELDON';`

```
--------------------------------------------
| Id | Operation                    | Name |
--------------------------------------------
|  0 | SELECT STATEMENT             |      |
|  1 |  TABLE ACCESS BY INDEX ROWID | T    |
|* 2 |   INDEX RANGE SCAN           | I_C1 |
--------------------------------------------
```

`2 - access("C1"='SELDON')`
`    filter(UPPER("C1")='SELDON')`

显然，当一个限制条件具有很高的选择性时，你会希望利用索引。为此，如果无法修改 `WHERE` 子句或指定约束，你可以创建一个 `基于函数的索引`。简而言之，这是在函数返回值或表达式结果上创建的索引。

`SQL> CREATE INDEX i_c1_upper ON t (upper(c1));`

`SQL> SELECT * FROM t WHERE upper(c1) = 'SELDON';`

```
-------------------------------------------------------
| Id | Operation                           | Name       |
-------------------------------------------------------
|  1 | TABLE ACCESS BY INDEX ROWID         | T          |
|* 2 |  INDEX RANGE SCAN                   | I_C1_UPPER |
-------------------------------------------------------
```

`2 - access(UPPER("C1")='SELDON')`

***

`注意` 在 Oracle9*i* Release 2 补丁集 9.2.0.3 之前，用户需要拥有 `query rewrite` 系统权限才能创建基于函数的索引。此外，查询优化器只有在初始化参数 `query_rewrite_enabled` 设置为 `TRUE` 并且初始化参数 `query_rewrite_integrity` 设置为 `trusted` 或 `stale_tolerated` 时，才会考虑使用此类索引。根据 MetaLink 说明 "Bug 2799028" (2799028.8) 中的描述，自补丁集 9.2.0.4 起，这些要求不再必要。

***

从 Oracle Database 11*g* 开始，为了避免在索引以及可能很多条 SQL 语句中重复函数或表达式，可以基于该函数或表达式创建一个虚拟列。这样，索引可以直接在虚拟列上创建，并且代码可以透明地处理其定义。以下示例展示了如何添加、索引并使用一个对 `c1` 列应用 `upper` 函数的虚拟列：

`SQL> ALTER TABLE t ADD (c1_upper AS (upper(c1)));`

`SQL> CREATE INDEX i_c1_upper ON t (c1_upper);`

`SQL> SELECT * FROM t WHERE c1_upper = 'SELDON';`

```
------------------------------------------------------------
| Id | Operation                                  | Name       |
------------------------------------------------------------
|  1 | TABLE ACCESS BY INDEX ROWID                | T          |
|* 2 |  INDEX RANGE SCAN                          | I_C1_UPPER |
------------------------------------------------------------
```

`2 - access("C1_UPPER"='SELDON')`

正如你在第 4 章中所见，使用虚拟列的另一个优点是，`dbms_stats` 包会为它们收集对象统计信息。有了这些信息，查询优化器就能够正确估算基于虚拟列的限制条件的基数。

尽管本章中的示例都是基于 B-tree 索引的，但基于函数的位图索引同样受到支持。

### 语言排序索引

默认情况下，数据库引擎执行 `二进制比较` 来比较字符串。在这种方式下，字符是根据其二进制值进行比较的。因此，只有当每个对应字符的数字代码完全相同时，两个字符串才被认为是相等的。

自 Oracle Database 10*g* Release 2 起，数据库引擎还能够执行 `语言比较`。使用这些比较时，字符的数字代码不必完全相同也能匹配。例如，你可以指示数据库引擎将小写和大写字符视为相等，或者将带重音符号和不带重音符号的字符视为相等。

***



**注意** 在先前版本中也部分提供了语言比较功能。然而，并非所有的运算符和函数都支持它们。因此，为避免产生不一致或误导性的结果，我建议仅在 Oracle Database 10g Release 2 或更高版本中使用它们。本节将不讨论 Oracle Database 10g Release 1 及之前版本中可用的功能。

***

要管理 SQL 运算符的这种行为，可以使用初始化参数 `nls_comp`。它可以设置为以下值之一：

`binary`：使用二进制比较。这是默认值。

`linguistic`：使用语言比较。初始化参数 `nls_sort` 指定了应用于比较的语言排序序列（从而也是规则）。可接受的值是 `binary`（这对语言比较没有用）以及初始化参数 `nls_language` 接受的大多数值。

`ansi`：此值仅用于向后兼容。应使用 `linguistic` 代替。

动态初始化参数 `nls_comp` 和 `nls_sort` 可以在实例和会话级别设置。在会话级别，可以使用 SQL 语句 `ALTER SESSION` 进行设置，也可以在操作系统级别的客户端定义（例如，在 Microsoft Windows 中通过注册表项）。

例如，考虑一个存储以下数据的表（表和测试查询可在脚本 `linguistic_index.sql` 中找到）：

```sql
SQL> SELECT c1 FROM t;

C1
--------------------
Leon
Léon
LEON
LÉON
```

默认情况下，执行二进制比较。要使用语言比较，必须将初始化参数 `nls_comp` 设置为 `linguistic`，并且用于比较的语言排序序列（从而也是规则）必须通过初始化参数 `nls_sort` 指定。以下示例使用 `generic_m`，这是一个基于拉丁字符的 ISO 标准：

```sql
SQL> ALTER SESSION SET nls_comp = linguistic;

SQL> ALTER SESSION SET nls_sort = generic_m;

SQL> SELECT c1 FROM t WHERE c = 'LEON';

C1
--------------------
LEON
```

正如预期的那样，之前的设置没有发生任何特别情况。有用的功能由 `generic_m` 的两个扩展提供。第一个是 `generic_m_ci`。如下面的查询所示，使用它时，比较是*大小写不敏感*的：

```sql
SQL> ALTER SESSION SET nls_sort = generic_m_ci;

SQL> SELECT c1 FROM t WHERE c = 'LEON';

C1
--------------------
Leon
LEON
```

第二个是 `generic_m_ai`。如下面的查询所示，使用它时，比较是*大小写和音标不敏感*的：

```sql
SQL> ALTER SESSION SET nls_sort = generic_m_ai;

SQL> SELECT c1 FROM t WHERE c = 'LEON';

C1
--------------------
Leon
Léon
LEON
LÉON
```

从功能角度来看，这是极好的。通过设置两个初始化参数，您就能够控制 SQL 运算符的行为。不过，让我们看看在下面的示例中，当初始化参数 `nls_comp` 设置为 `linguistic` 时会发生什么：

```sql
SQL> CREATE INDEX i_c1 ON t (c1);

SQL> ALTER SESSION SET nls_sort = generic_m_ai;

SQL> ALTER SESSION SET nls_comp = binary;
SQL> SELECT /*+ index(t) */ * FROM t WHERE c1 = 'LEON';

--------------------------------------------
| Id | Operation                    | Name |
--------------------------------------------
|  1 | TABLE ACCESS BY INDEX ROWID  | T    |
|* 2 |   INDEX RANGE SCAN           | I_C1 |
--------------------------------------------

  2 - access("C1"='LEON')

SQL> ALTER SESSION SET nls_comp = linguistic;

SQL> SELECT /*+ index(t) */ * FROM t WHERE c1 = 'LEON';

-----------------------------------------
| Id | Operation                 | Name |
-----------------------------------------
|* 1 | TABLE ACCESS FULL         | T    |
-----------------------------------------

  1 - filter(NLSSORT("C1",'nls_sort=''GENERIC_M_AI''')=HEXTORAW('022601FE02380232'))
```

显然，当初始化参数 `nls_comp` 设置为 `linguistic` 时，索引不再被使用。原因由输出的最后一行指出。由于函数 `nlssort` 被隐式地应用于索引列 `c1`，因此无法在索引中进行查找。因此，为此目的，需要创建一个基于函数的索引来避免全表扫描。必须认识到，索引的定义必须包含与初始化参数 `nls_sort` 相同的值。因此，如果使用多种语言，则应该创建多个索引。

```sql
SQL> CREATE INDEX i_c1_linguistic ON t (nlssort(c1,'nls_sort=generic_m_ai'));

SQL> ALTER SESSION SET nls_sort = generic_m_ai;

SQL> ALTER SESSION SET nls_comp = linguistic;

SQL> SELECT /*+ index(t) */ * FROM t WHERE c1 = 'LEON';

------------------------------------------------------------------
| Id | Operation                               | Name            |
------------------------------------------------------------------
|  1 | TABLE ACCESS BY INDEX ROWID             | T               |
|* 2 |   INDEX RANGE SCAN                      | I_C1_LINGUISTIC |
------------------------------------------------------------------

  2 - access(NLSSORT("C1",'nls_sort=''GENERIC_M_AI''')=HEXTORAW('022601FE02380232'))
```

直到 Oracle Database 10g Release 2，另一个限制是，为了应用 `LIKE` 运算符，数据库引擎无法利用语言索引。换句话说，无法避免全索引扫描或全表扫描。从 Oracle Database 11g 开始，此限制不再存在。

尽管本节中的示例基于 B-tree 索引，但也支持语言位图索引。

总之，语言比较是一个强大的功能，对于 SQL 语句是透明的。然而，只有当一组适配的索引可用时，数据库引擎才能高效地应用它们。由于客户端级别的设置可能影响索引的使用，因此仔细规划其使用至关重要。

## 复合索引

到目前为止，除一个例外，我只讨论了索引键由单个列组成的索引。然而，索引可以包含多个列（B-tree 索引的限制是 32 列，位图索引是 30 列）。具有多列的索引称为*复合索引*（有时也使用术语*连接索引*或*多列索引*）。在这方面，B-tree 索引和位图索引具有完全不同的行为。因此，将分别进行讨论。

### B-tree 索引

复合索引的目的有两个。首先，它们可用于实施由多个列组成的主键或唯一键约束。其次，它们可用于实施由多个 SQL 条件组合 `AND` 构成的限制。请注意，当多个 SQL 条件组合 `OR` 时，复合索引无法高效使用！

自然，讨论如何使用复合索引来实施限制是很重要的。以下查询用于此目的：

`SELECT * FROM t WHERE n1 = 6 AND n2 = 42 AND n3 = 11`

让我们首先看看使用单列索引时会发生什么。

使用在列 `n1` 上构建的索引，从索引扫描中返回 527 个 rowid。由于索引仅存储与列 `n1` 相关的数据，只有谓词 `n1=6` 可以通过操作 2 高效地通过索引应用。另外两个谓词由操作 1 作为过滤器应用。由于操作 2 返回了大量 rowid，执行总共产生了 327 次逻辑读取。在检索单行时，这是不可接受的。



`-----------------------------------------------------------------------------`
`| Id | 操作                          | 名称    | 开始次数 | 返回行数 | 缓冲区 |`
`-----------------------------------------------------------------------------`
`|* 1 | 通过索引 ROWID 访问表           | T       |        1 |        1 |    327 |`
`|* 2 |  索引范围扫描                 | `I_N1`    |        1 |      527 |      4 |`
`-----------------------------------------------------------------------------`

```
  1 - filter(("N3"=11 AND "N2"=42))
  2 - access(`"N1"=6`)
```

在列 `n2` 上建立索引后，情况与前一个示例基本相同。唯一的改进是索引扫描返回的 rowid 更少（89 个）。因此，总的逻辑读（85 次）也大幅减少。

`---------------------------------------------------------------------`
`| Id | 操作                      | 名称 | 开始次数 | 返回行数 | 缓冲区 |`
`---------------------------------------------------------------------`
`|* 1 | 通过索引 ROWID 访问表       | T    |        1 |        1 |     85 |`
`|* 2 | 索引范围扫描              |`I_N2`  |        1 |       89 |      4 |`
`---------------------------------------------------------------------`

```
  1 - filter(("N3"=11 AND "N1"=6))
  2 - access(`"N2"=42`)
```

在列 `n3` 上建立索引后，情况仍然与前几个类似。实际上，索引扫描返回了大量的 rowid（164 个）。总的逻辑读数（141 次）仍然过高。

`---------------------------------------------------------------------`
`| Id | 操作                      | 名称 | 开始次数 | 返回行数 | 缓冲区 |`
`---------------------------------------------------------------------`
`|* 1 | 通过索引 ROWID 访问表       | T    |        1 |        1 |    141 |`
`|* 2 |  索引范围扫描             |  `I_N3` |        1 |      164 |      4 |`
`---------------------------------------------------------------------`

```
  1 - filter(("N2"=42 AND "N1"=6))
  2 - access(`"N3"=11`)
```

总之，这三个索引都无法高效地应用谓词。这三个限制条件单独来看，其选择率都太高了。这一观察结果与数据字典中存储的对象统计信息一致。实际上，每个列的不同值的数量都很低，如下列查询所示：

```sql
SQL> SELECT column_name, num_distinct
  2 FROM user_tab_columns
  3 WHERE table_name = 'T' AND column_name IN ('ID', 'N1', 'N2', 'N3');

COLUMN_NAME NUM_DISTINCT
------------ ------------
ID                 10000
N1                    18
N2                   112
N3                    60
```

在这种情况下，使用一个建立在多个列上的索引来应用各种条件会更有效。例如，下面的执行计划显示了如果在这三列上创建一个复合索引会发生什么。关键要理解的是，使用此索引时，逻辑读数较低，因为索引扫描只返回满足整个 `WHERE` 子句的行。

`---------------------------------------------------------------------`
`| Id | 操作                      | 名称 | 开始次数 | 返回行数 | 缓冲区 |`
`---------------------------------------------------------------------`
`|  1 | 通过索引 ROWID 访问表       | T    |        1 |        1 |      4 |`
`|* 2 | 索引范围扫描              |`I_N123`|        1 |        1 |      3 |`
`---------------------------------------------------------------------`

```
  2 - access(`"N1"=6 AND "N2"=42 AND "N3"=11`)
```

此时，必须认识到数据库引擎能够在并非所有建立索引的列都在 `WHERE` 子句中被引用时使用索引。基本要求是应对索引键的引导列应用条件。例如，在前一个示例中使用的索引 `i_n123`，对列 `n2` 或 `n3` 的条件是可选的。以下查询显示了一个没有 `n2` 列条件的例子：

```sql
SELECT * FROM t WHERE n1 = 6 AND n3 = 11
```

`------------------------------------------------------------------------`
`| Id | 操作                      | 名称   | 开始次数 | 返回行数 | 缓冲区 |`
`------------------------------------------------------------------------`
`|  1 | 通过索引 ROWID 访问表       | T      |        1 |        1 |      8 |`
`|* 2 |  索引范围扫描             | I_N123 |        1 |        1 |      7 |`
`------------------------------------------------------------------------`

```
  2 - access("N1"=6 AND "N3"=11)
      filter("N3"=11)
```

在某些情况下，即使在索引键的引导列上没有条件，也可以（高效地）使用索引。这种操作称为*索引跳跃扫描*。但是，只有当引导列的不同值数量非常少时，使用它才有意义，因为会对引导列的每个值执行一次独立的索引范围扫描。以下查询显示了这样一个例子。注意提示 `index_ss` 和操作 `INDEX SKIP SCAN`。

```sql
SELECT /*+ `index_ss`(t i_n123) */ * FROM t WHERE n2 = 42 AND n3 = 11
```

`------------------------------------------------------------------------`
`| Id | 操作                      | 名称   | 开始次数 | 返回行数 | 缓冲区 |`
`------------------------------------------------------------------------`
`|  1 | 通过索引 ROWID 访问表       | T      |        1 |        2 |     33 |`
`|* 2 |  `INDEX SKIP_SCAN`        | I_N123 |        1 |        2 |     31 |`
`------------------------------------------------------------------------`

```
  2 - access("N2"=42 AND "N3"=11)
      filter(("N2"=42 AND "N3"=11))
```

由于支持对“常规”索引扫描（使用操作 `INDEX SKIP SCAN DESCENDING`）进行降序索引跳跃扫描，因此可以使用 `index_ss_asc` 和 `index_ss_desc` 两个提示来控制扫描顺序。

谈到复合索引，我认为有必要提一下我在处理它们时遇到的最常见错误以及最常被问到的问题。这个错误与过度索引有关。一种误解是认为只有当索引键的所有列都在 `WHERE` 子句中使用时，数据库引擎才能利用索引。正如你刚刚在几个例子中看到的，事实并非如此。这种误解通常导致在同一张表上创建多个具有相同引导列的索引——例如，一个索引包含列 `n1, n2` 和 `n3`，另一个包含列 `n1` 和 `n3`。第二个索引通常是多余的。请注意，多余的索引是一个问题，因为它们不仅会降低修改索引数据的 SQL 语句的速度，还会不必要地浪费空间。



## 如何选择索引列的顺序以及索引压缩技术

最常见的问题是，如何选择列的顺序？例如，如果一个索引键由`n1`、`n2`和`n3`列组成，最佳顺序是什么？当所有被索引的列都出现在`WHERE`子句中时，索引的效率与列的顺序无关。因此，最佳顺序是能最大程度地增加在`WHERE`子句不完全包含所有索引列时使用该索引的机会。换句话说，应该让索引尽可能多的 SQL 语句能够使用。为确保这一点，最常（理想情况下）在`WHERE`子句中指定的列应该作为首列。当多个列的使用频率相同时，有两种相反的方法可以遵循：

*   首列应该是具有最多不同值的列。如果未来 SQL 语句的限制条件仅应用于该特定列，这将是有用的。换句话说，这最大化了查询优化器选择该索引的机会。
*   首列应该是具有最少不同值的列。这可能有助于实现更好的索引压缩比。

## 索引压缩

B 树索引和位图索引的一个重要区别在于存储索引叶块中键时使用的压缩方式。位图索引总是被压缩的，而 B 树索引仅在被请求时才压缩。

在未压缩的 B 树索引中，每个键都被完整存储。换句话说，如果多个键具有相同的值，则该值会为每个键重复存储。因此，在非唯一索引中，同一个值在同一个叶块中存储多次是很常见的。为消除这些重复出现，可以压缩索引键。可以通过使用`COMPRESS`参数（以及可选的，需要压缩的列数）来（重新）构建索引来实现这一点。例如，索引`i_n123`由三列组成：`n1`、`n2`和`n3`。使用`COMPRESS 1`，指定仅压缩列`n1`；使用`COMPRESS 2`，指定压缩列`n1`和`n2`；使用`COMPRESS 3`，指定压缩所有三列。当未指定要压缩的列数时，对于非唯一索引，所有列都会被压缩；对于唯一索引，则压缩列数减一。

由于列是从左到右压缩的，因此列应按选择性递减排序以获得最佳压缩效果。但是，只有当重新排序索引列不会阻止查询优化器使用该索引时，才应重新排序。

默认情况下，B 树索引压缩并未激活，因为它并不总是能减小索引的大小。实际上，在某些情况下，索引在压缩后可能会变得更大！因此，仅当存在真正优势时才应启用压缩。有两种方法可以检查给定索引的预期压缩率。首先，可以先构建一次未压缩的索引，然后再构建一次压缩的索引，然后比较大小。其次，可以让 SQL 语句`ANALYZE INDEX`执行分析，以找出最优的压缩列数以及使用最优压缩可以节省多少空间。以下示例显示了对索引`i_n123`进行的此类分析。请注意，分析的输出被写入表`index_stats`中。在此情况下，你会得知通过压缩两列可以节省索引当前所占空间的 17%。

```sql
SQL> ANALYZE INDEX i_n123 VALIDATE STRUCTURE;
SQL> SELECT opt_cmpr_count, opt_cmpr_pctsave FROM index_stats;
OPT_CMPR_COUNT OPT_CMPR_PCTSAVE
-------------- ----------------
             2               17
```

以下 SQL 语句不仅展示了如何实现索引`i_n123`的压缩，还展示了如何检查压缩的结果：

```sql
SQL> SELECT blocks FROM index_stats;
    BLOCKS
----------
        40
SQL> ALTER INDEX i_n123 REBUILD COMPRESS 2;
SQL> ANALYZE INDEX i_n123 VALIDATE STRUCTURE;
SQL> SELECT blocks FROM index_stats;
    BLOCKS
----------
        32
```

从性能角度来看，压缩索引的关键优势在于，由于其尺寸更小，执行索引范围扫描和索引全扫描时需要的逻辑读取更少。然而，缺点是遭受块争用的可能性增加（此主题在第 12 章中介绍）。

## 位图索引

很少创建复合位图索引。这是因为可以高效地组合多个索引来应用限制条件。为了了解位图索引的强大之处，让我们看几个查询。

第一个查询利用了三个位图索引，这些索引与`AND`组合以应用三个等式条件。请注意，提示`index_combine`强制执行此类计划。首先，操作 4 通过查找满足该列限制条件的行来扫描基于`n5`列的索引。结果位图被传递给操作 3。然后，操作 5 和 6 分别对在`n6`和`n4`列上创建的索引执行相同的扫描。一旦完成三个索引扫描，操作 3 计算三个位图集合的`AND`。最后，操作 2 将结果位图转换为 rowid 列表，然后操作 1 使用它们来访问表。

```sql
SELECT /*+ index_combine(t i_n4 i_n5 i_n6) */ *
FROM t
WHERE n4 = 6 AND n5 = 42 AND n6 = 11
```
```
-------------------------------------------------------------------------
| Id | Operation                      | Name | Starts | A-Rows | Buffers |
-------------------------------------------------------------------------
|  1 |  TABLE ACCESS BY INDEX ROWID   | T    |      1 |      1 |       7 |
|  2 |   BITMAP CONVERSION TO ROWIDS  |      |      1 |      1 |       6 |
|  3 |    BITMAP AND                  |      |      1 |      1 |       6 |
|* 4 |     BITMAP INDEX SINGLE VALUE  | I_N5 |      1 |      1 |       2 |
|* 5 |     BITMAP INDEX SINGLE VALUE  | I_N6 |      1 |      1 |       2 |
|* 6 |     BITMAP INDEX SINGLE VALUE  | I_N4 |      1 |      1 |       2 |
-------------------------------------------------------------------------
  4 - access("N5"=42)
  5 - access("N6"=11)
  6 - access("N4"=6)
```

第二个查询与第一个非常相似。唯一的区别是使用了`OR`而不是`AND`。请注意执行计划中只有操作 3 发生了变化。

```sql
SELECT /*+ index_combine(t i_n4 i_n5 i_n6) */ *
FROM t
WHERE n4 = 6 OR n5 = 42 OR n6 = 11
```
```
------------------------------------------------------------------------
| Id | Operation                     | Name | Starts | A-Rows | Buffers |
------------------------------------------------------------------------
|  1 |  TABLE ACCESS BY INDEX ROWID  | T    |      1 |    767 |     396 |
|  2 |   BITMAP CONVERSION TO ROWIDS |      |      1 |    767 |       7 |
|  3 |    BITMAP OR                  |      |      1 |      1 |       7 |
|* 4 |     BITMAP INDEX SINGLE VALUE | I_N4 |      1 |      1 |       3 |
|* 5 |     BITMAP INDEX SINGLE VALUE | I_N6 |      1 |      1 |       2 |
|* 6 |     BITMAP INDEX SINGLE VALUE | I_N5 |      1 |      1 |       2 |
------------------------------------------------------------------------
  4 - access("N4"=6)
  5 - access("N6"=11)
  6 - access("N5"=42)
```
