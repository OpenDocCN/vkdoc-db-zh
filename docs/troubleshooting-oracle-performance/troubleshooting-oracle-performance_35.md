# 执行计划列

## 基本信息（始终可用）

| **列** | **描述** |
| --- | --- |
| `Id` | 执行计划中每个操作（行）的标识符。如果数字前带有星号，表示稍后将提供该行的谓词信息。 |
| `Operation` | 要执行的操作。这也被称为行源操作。 |
| `Name` | 执行操作的对象。 |

## 查询优化器估算值

| **列** | **描述** |
| --- | --- |
| `Rows (E-Rows)` | 操作预计返回的行数。 |
| `Bytes (E-Bytes)` | 操作预计返回的数据量。 |
| `TempSpc` | 操作预计使用的临时空间量。 |
| `Cost (%CPU)` | 操作的估算成本。括号内给出的是 CPU 成本的百分比。请注意，此值在执行计划中是累积的。换句话说，父操作的成本包含了其子操作的成本。 |
| `Time` | 执行操作预计所需的时间（HH:MM:SS）。 |

## 分区处理

| **列** | **描述** |
| --- | --- |
| `Pstart` | 要访问的第一个分区的编号。如果在解析时未知，则设置为 `KEY, KEY(I), KEY(MC), KEY(OR)` 或 `KEY(SQ)`。 |
| `Pstop` | 要访问的最后一个分区的编号。如果在解析时未知，则设置为 `KEY, KEY(I), KEY(MC), KEY(OR)` 或 `KEY(SQ)`。 |

## 并行与分布式处理

| **列** | **描述** |
| --- | --- |
| `Inst` | 对于分布式处理，指操作所使用的数据库链接的名称。 |
| `TQ` | 对于并行处理，指用于从属进程间通信的表队列。 |
| `IN-OUT` | 并行或分布式操作之间的关系。 |
| `PQ Distrib` | 对于并行处理，指生产者向消费者发送数据时所使用的分发方式。 |

## 运行时统计信息*

| **列** | **描述** |
| --- | --- |
| `Starts` | 特定操作被执行的次数。 |
| `A-Rows` | 操作实际返回的行数。 |
| `A-Time` | 执行操作实际花费的时间（HH:MM:SS.FF）。 |

## I/O 统计信息*

| **列** | **描述** |
| --- | --- |
| `Buffers` | 执行期间执行的逻辑读操作次数。 |
| `Reads` | 执行期间执行的物理读操作次数。 |
| `Writes` | 执行期间执行的物理写操作次数。 |

## 内存利用统计信息

| **列** | **描述** |
| --- | --- |
| `OMem` | 最优执行所需的估计内存量。 |
| `1Mem` | 单次通过执行所需的估计内存量。 |
| `O/1/M` | 以最优/单次通过/多次通过模式执行的次数。 |
| `Used-Mem` | 上次执行期间操作使用的内存量。 |
| `Used-Tmp` | 上次执行期间操作使用的临时空间量。此值必须乘以 1,024 才能与其他内存利用列保持一致（例如，32K 表示 32MB）。 |
| `Max-Tmp` | 操作使用的最大临时空间量。此值必须乘以 1,024 才能与其他内存利用列保持一致（例如，32K 表示 32MB）。 |

*仅在启用执行统计信息时可用。

下一节显示查询块名称和对象别名：

```
Query Block Name / Object Alias (identified by operation id):
-------------------------------------------------------------

   1 - SEL$1
   2 - SEL$1 / T2@SEL$1
   3 - SEL$1 / T2@SEL$1
   4 - SEL$1 / T1@SEL$1
```

对于执行计划中的每个操作，您都可以看到它关联到哪个查询块，以及可选地，它在哪个对象上执行。当 SQL 语句像本示例中多次引用同一表时，此信息至关重要。此节从 Oracle 数据库 10*g*版本开始可用。查询块名称将与提示一起在第 7 章中更详细地讨论。

第四节仅从 Oracle 数据库 10*g*第 2 版开始可用。它显示了强制该特定执行计划所需的一组提示。这组提示称为*大纲*。第 7 章描述了如何使用存储的大纲和 SQL 计划基线来存储和利用这样的大纲。

```
Outline Data
-------------

  /*+
      BEGIN_OUTLINE_DATA
      IGNORE_OPTIM_EMBEDDED_HINTS
      OPTIMIZER_FEATURES_ENABLE('10.2.0.3')
      ALL_ROWS
      OUTLINE_LEAF(@"SEL$1")
      INDEX_RS_ASC(@"SEL$1" "T2"@"SEL$1" ("T"."ID"))
      FULL(@"SEL$1" "T1"@"SEL$1")
      LEADING(@"SEL$1" "T2"@"SEL$1" "T1"@"SEL$1")
      USE_HASH(@"SEL$1" "T1"@"SEL$1")
      END_OUTLINE_DATA
  */
```

下一节显示应用了哪些谓词。对于每个谓词，会显示其位置（行号）和应用方式（访问或过滤）。

```
Predicate Information (identified by operation id):
---------------------------------------------------

   1 - access("T1"."N"="T2"."N")
   3 - access("T2"."ID">=6 AND "T2"."ID"<=19)
   4 - filter("T1"."ID">6)
```

访问谓词用于通过利用高效的访问结构（例如，像操作 1 中那样的内存中的哈希表，或像操作 3 中那样的索引）来定位行，而过滤谓词仅在行已从存储它们的结构中提取之后才应用。请注意，此部分既包含 SQL 语句本身中存在的谓词，也可能包含由查询优化器或虚拟专用数据库生成的谓词。在上面的示例中，您有以下谓词：

*   第 1 行的哈希连接使用谓词 `"T1"."N"="T2"."N"` 来连接两个表。换句话说，访问谓词也可能显示连接条件。在此特定情况下，访问谓词用于指定将使用访问表 `t2` 返回的值来探测包含表 `t1` 数据的内存中的哈希表（哈希连接的工作原理在第 10 章中有详细描述）。
*   第 3 行的索引扫描访问索引 `t_pk` 以查找表 `t1` 的 `id` 列。在这种情况下，访问谓词显示了执行查找所依据的键。
*   在第 4 行，通过全表扫描读取表 `t3` 中的所有行。然后，当行从数据块中提取出来后，应用谓词 `"T1"."ID">6` 对它们进行过滤。

以下部分显示了执行每个操作时作为输出返回的哪些列。此功能仅从 Oracle 数据库 10*g*开始可用。示例如下：

```
Column Projection Information (identified by operation id):
-----------------------------------------------------------

   1 - (#keys=1) "T2"."N"[NUMBER,22], "T2"."ID"[NUMBER,22],
       "T2"."PAD"[VARCHAR2,1000]
   2 - "T2"."ID"[NUMBER,22], "T2"."N"[NUMBER,22], "T2"."PAD"[VARCHAR2,1000]
   3 - "T2".ROWID[ROWID,10], "T2"."ID"[NUMBER,22]
   4 - "T1"."N"[NUMBER,22]
```

在这种情况下，值得注意的是，第 2 行的表访问返回 `id, n` 和 `pad` 列，而第 4 行的全表扫描只返回 `n` 列。因此，第 2 行每行返回的估算数据量（列 `Bytes`）(7,392/14 = 528 字节) 远大于第 4 行 (25,844/994 = 26 字节)。第 12 章 提供了关于数据库引擎部分读取行的能力的更多信息。

最后，有一个部分提供关于优化阶段、环境或 SQL 语句本身的注释和警告：

```
Note
-----
dynamic sampling used for this statement
```


在此告知，查询优化器使用了动态采样来收集对象统计信息。在 Oracle9*i*第 2 版中，此部分没有标题，并且包含的信息比 Oracle Database 10*g*版本更少（例如，未给出是否使用了动态采样）。

## 函数显示

函数 `display` 返回存储在计划表中的执行计划。返回值是集合 `dbms_xplan_type_table` 的一个实例。该集合的元素是对象类型 `dbms_xplan_type` 的实例。此对象类型的唯一属性名为 `plan_table_output`，其类型为 `VARCHAR2`。³ 该函数具有以下输入参数：

*   `table_name` 指定计划表的名称。默认值为 `plan_table`。如果指定为 `NULL`，则使用默认值。
*   `statement_id` 指定在执行 SQL 语句 `EXPLAIN PLAN` 时可选提供的 SQL 语句名称。默认值为 `NULL`。如果使用默认值，则显示最近插入计划表的执行计划（前提是未指定 `filter_preds` 参数）。
*   `format` 指定输出中提供哪些信息。有基本值（`basic, typical, serial, all`, 和 `advanced`），并且为了更精细的控制，可以向它们添加额外的修饰符（`alias, bytes, cost, note, outline, parallel, partition, peeked_binds, predicate, projection, remote`, 和 `rows`）。如果要添加信息，修饰符可选择以 `+` 字符为前缀（例如，`basic +predicate`）。如果要移除信息，修饰符以 `–` 字符为前缀（例如，`typical –bytes`）。可以同时指定多个修饰符（例如，`typical +alias –bytes –cost`）。表 6-2 和 表 6-3 分别完整描述了基本值和修饰符。默认值为 `typical`。基本值 `advanced` 和修饰符仅在 Oracle Database 10*g* 第 2 版及以后版本中可用。
*   `filter_preds` 指定查询计划表时应用的限制条件。该限制是基于计划表某一列的常规 SQL 谓词（例如，`statement_id='test3'`）。默认值为 `NULL`。如果使用默认值，则显示最近插入计划表的执行计划。此参数在 Oracle Database 10*g* 第 2 版及以后版本中可用。

要使用函数 `display`，调用者只需要对该包具有 `EXECUTE` 权限，并且对计划表具有 `SELECT` 权限。

**表 6-2.** *`format` 参数接受的基本值*

| **值** | **描述** |
| --- | --- |
| `basic` | 仅显示最少的信息，基本上只有操作及其执行的对象 |
| `typical` | 显示最相关的信息，基本上包含除别名、大纲和列投影信息外的所有内容 |
| `serial` | 与 `typical` 类似，但不显示并行处理信息 |
| `all` | 显示除大纲外的所有可用信息 |
| `advanced` | 显示所有可用信息 |

**表 6-3.** *参数 `format` 接受的修饰符*

| **值** | **描述** |
| --- | --- |
| `alias` | 控制包含查询块名称和对象别名的部分的显示。 |
| `bytes` | 控制执行计划表中 `Bytes` 列的显示。 |
| `cost` | 控制执行计划表中 `Cost` 列的显示。 |
| `note` | 控制包含注释的部分的显示。 |
| `outline` | 控制包含大纲的部分的显示。 |
| `parallel` | 控制并行处理信息的显示，特别是执行计划表中的 `TQ, IN-OUT`, 和 `PQ Distrib` 列。 |
| `partition` | 控制分区信息的显示，特别是执行计划表中的 `Pstart` 和 `Pstop` 列。 |
| `peeked_binds` | 控制包含窥探到的绑定变量的部分的显示。由于当前 SQL 语句 `EXPLAIN PLAN` 的实现不执行绑定变量窥探，此部分未在之前的示例中显示。 |
| `predicate` | 控制包含过滤和访问谓词的部分的显示。 |
| `projection` | 控制包含列投影信息的部分的显示。 |
| `remote` | 控制远程执行的 SQL 语句的显示。 |
| `rows` | 控制执行计划表中 `Rows` 列的显示。 |
