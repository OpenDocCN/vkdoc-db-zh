# 查询优化器参数

以下部分描述了与查询优化器操作相关的若干参数。

## `optimizer_mode`

初始化参数 `optimizer_mode` 至关重要，因为它指定了查询优化器如何决定执行每条 SQL 语句最高效的执行计划。其关键性在于，它定义了“高效”一词的含义。它可能意味着“更快”、“使用更少资源”，或者其他什么。由于数据库处理的是数据，你通常会希望处理速度尽可能快。因此，“高效”的含义应该是以最快的方式执行 SQL 语句，同时不浪费不必要的资源。这对于总是完全执行的 SQL 语句（例如 `INSERT` 语句）来说是清晰的。而对于查询来说，则存在细微的差别。例如，应用程序没有义务获取查询返回的所有行。换句话说，查询可能只被部分执行。

让我举一个与 Oracle 无关的例子。当我用谷歌搜索“查询优化器”这个术语时，我立即得到一个包含前十条结果的页面。在同一页面上，我得知结果集中大约有 334,000 个页面，搜索耗时 0.13 秒。这是一个很好的例子，展示了处理过程被优化以尽快提供初始数据。如果我想访问其中一个页面，我会点击链接。此时，我通常对只获取前几行不感兴趣。我需要整个页面可用且格式正确，然后我才会开始阅读。在这种情况下，处理应被优化为尽可能快地提供所有数据，而不仅仅是片段。每个应用程序都属于前面描述的两类情况之一：要么是快速交付结果集的第一部分很重要，要么是快速交付整个结果集很重要。

要选择初始化参数 `optimizer_mode` 的值，你必须自问：对于查询优化器来说，是生成用于快速交付首行的执行计划更重要，还是生成用于快速交付末行的执行计划更重要：

*   如果快速交付末行很重要，你应该使用 `all_rows` 这个值。此配置通常用于报表系统、OLAP 系统、数据仓库以及缓存数据的中间层组件。
*   如果快速交付首行（或多行）很重要，你应该使用 `first_rows_n` 这个值（其中 `n` 是 1、10、100 或 1000 行）。此配置通常用于 OLTP 系统。注意，较旧的首行优化器（通过 `first_rows` 值配置）不应再使用。实际上，它仅为向后兼容而提供。

在 Oracle9*i* 及之前，默认值是 `choose`。这意味着，如果对于需要优化的 SQL 语句中引用的对象中，至少有一个对象的对象统计信息可用，则使用 `all_rows` 值；否则，使用 `rule` 值。然而，在某些情况下，即使没有对象统计信息可用，数据库引擎也会强制使用 `all_rows`。这是因为，一般来说，基于规则的优化器不支持 Oracle7 之后添加的结构（例如，分区表）。从 Oracle Database 10*g* 开始，基于规则的优化器已被弃用，因此新的默认值是 `all_rows`。因此，`choose` 和 `rule` 值不再受支持。

初始化参数 `optimizer_mode` 是动态的，可以在实例和会话级别进行更改。此外，通过以下提示（hint），可以在语句级别进行设置：

*   `all_rows`
*   `first_rows(n)`，其中 `n` 是任何自然数（大于 0）

## `optimizer_features_enable`

在每个数据库版本中，Oracle 都会在查询优化器中引入或启用新特性。如果你要升级到新的数据库版本，并希望保留查询优化器的旧行为，可以将初始化参数 `optimizer_features_enable` 设置为你要升级的数据库版本。也就是说，设置它只是一个短期的变通方法，应用程序迟早应该为新的数据库版本进行适配（优化）。此外，并非所有新特性都会被此初始化参数禁用。例如，如果你在 Oracle Database 10*g* 中将其设置为 `9.2.0`，你得到的并不是 `9.2.0` 版本的查询优化器。因此，我通常建议使用默认值，该值与数据库二进制文件版本相同。

初始化参数 `optimizer_features_enable` 的有效值是诸如 `9.2.0`、`10.1.0.5` 或 `10.2.0.3` 的数据库版本。由于文档（特别是 *Oracle Database Reference* 手册）并未针对该参数的每个补丁级别及时更新，从 Oracle Database 10*g* Release 2 开始，可以使用以下 SQL 语句生成实际支持的值：

```sql
SQL> SELECT value
  2  FROM v$parameter_valid_values
  3  WHERE name = 'optimizer_features_enable';
VALUE
---------------------
8.0.0
8.0.3
8.0.4
...
10.2.0.2
10.2.0.3
10.2.0.3.1
```

在 Oracle Database 10*g* Release 1 及之前，视图 `v$parameter_valid_values` 不存在。我知道的唯一获取最新列表的变通方法是让数据库引擎生成一条错误消息，如下例所示。你也可以将这个技巧用于其他接受预定义值的初始化参数。

```sql
SQL> ALTER SESSION SET optimizer_features_enable = 'dummy';
ERROR:
ORA-00096: invalid value dummy for parameter optimizer_features_enable, must be
from among 10.1.0.5.1, 10.1.0.5, 10.1.0.4, 10.1.0.3, 10.1.0, 9.2.0, 9.0.1, 9.0.0,
8.1.7, 8.1.6, 8.1.5, 8.1.4, 8.1.3, 8.1.0, 8.0.7, 8.0.6, 8.0.5, 8.0.4, 8.0.3, 8.0.0
```

初始化参数 `optimizer_features_enable` 是动态的，可以在实例和会话级别更改。此外，从 Oracle Database 10*g* 开始，可以使用提示 `optimizer_features_enable` 在语句级别指定一个值。以下两个示例分别展示了用于指定默认值和特定值的提示：

*   `optimizer_features_enable(default)`
*   `optimizer_features_enable('9.2.0')`

需要注意的是，初始化参数 `optimizer_features_enable` 不仅会禁用新特性，也会禁用错误修复。脚本 `bug5015557.sql` 提供了这样一个例子。如果你在 Oracle Database 10*g` 补丁集 `10.1.0.5` 或更高版本上运行它，你会看到，当参数设置为 `10.1.0.5` 时查询正确运行，但当设置为 `10.1.0.4` 时，查询会因 ORA-03113 错误而失败。

## `db_file_multiblock_read_count`

数据库引擎在多块读取（例如，全表扫描或索引快速全扫描）期间使用的最大 I/O 大小，是由初始化参数 `db_file_multiblock_read_count` 和 `db_block_size` 的值相乘决定的。因此，多块读取期间读取的最大块数，是通过最大 I/O 大小除以块大小来确定的。换句话说，对于默认块大小，初始化参数 `db_file_multiblock_read_count` 指定了读取的最大块数。这“仅仅”是一个最大值，因为至少有三种常见情况会导致多块读取的块数小于此初始化参数指定的值：

*   段头是通过单块读取读取的。
*   物理读取永远不会跨越多个区。
*   除了直接读取外，已在缓冲区缓存中的块不会从 I/O 子系统重新读取。
