# 总结

非聚集索引是内存 OLTP 引擎支持的第二类索引。它们具有相似的 SARGability 规则，与在基于磁盘的表上定义的 B-Tree 索引类似，但扫描方向与索引排序顺序相反的情况除外。

在内部，非聚集索引使用一种无锁、无闩锁的 B-Tree 变体，称为 Bw-Tree。它由内部数据页和叶数据页组成，通过映射表相互引用。叶数据页为每个独立的键值存储一行数据，并包含一个指向具有相同键值的数据行链的指针。

SQL Server 永远不会更新索引页。所有更改都通过与页面上单个 `INSERT` 和 `DELETE` 操作相对应的增量记录来引用。SQL Server 会在需要时整合长的增量链，并对数据页进行拆分和合并。所有这些过程都会创建新的数据页，并将旧页标记为待垃圾回收。

与基于磁盘的表上的索引一样，内存优化表上的索引也会引入数据修改开销。当你插入或更新数据时，SQL Server 必须维护多个索引行链。你应该避免定义过多的索引，并创建支持工作负载所需的最小索引集。

哈希索引的性能在很大程度上取决于 `bucket_count` 值。在正确的 `bucket_count` 值下，哈希索引在点查找场景中的性能会优于非聚集索引。对于数据相对静态的目录实体，它们是一个不错的选择。另一方面，当点查找不可行和/或难以估计哈希索引中的桶数量时，非聚集索引是一个好的选择。
