# 引言

2016 年对于 SQL Server 社区来说是令人欣喜的一年——我们迎来了新版 SQL Server 的发布。这是一个相当独特的版本；十多年来，新版本首次没有专注于特定的技术。在 `SQL Server 2016` 中，你可以在所有产品领域都找到增强功能，例如可编程性、高可用性、管理和商业智能（BI）。

我个人对 `In-Memory OLTP` 的所有增强感到非常兴奋。我在 `SQL Server 2014` 中就非常喜欢这项技术；然而，它当时有太多的限制。这使得它成为一项小众技术，阻碍了其广泛采用。在许多情况下，所需的系统重构成本将 `In-Memory OLTP` 的第一个版本归入了“不值得”的范畴。

我非常高兴地看到，这些限制中的大部分在 `SQL Server 2016` 中被移除了。仍然存在一些限制，但远不如第一个版本那样严重。现在，可以将系统迁移到内存中，并开始使用该技术，而无需进行大量的代码和数据库架构更改。

然而，我认为这种简便性是一把双刃剑。虽然它能显著降低采用该技术所需的时间和成本，但也可能为错误的决策和次优的实现打开大门。与任何其他技术一样，`In-Memory OLTP` 是为特定的任务集设计的，如果实现不当，可能会损害系统的性能。它也不是一种“一劳永逸”的解决方案；在实施之前必须仔细规划，并在部署后进行维护。

`In-Memory OLTP` 是一个强大的工具，它可以显著提高系统的性能。尽管如此，你需要了解其底层工作原理才能从中获得最大收益。本书的目标就是为你提供这种理解。我将解释 `In-Memory OLTP` 引擎及其组件的内部原理。我相信知识是成功实施 `In-Memory OLTP` 的基石，本书将帮助你在如何以及何时使用该技术方面做出明智的决策。

如果你阅读过我的《Pro SQL Server Internals》一书（Apress, 2016），你会注意到其中有一些熟悉的内容。然而，本书是对 `In-Memory OLTP` 更深层次的探讨，你将发现涵盖了许多新主题。你还将学习如何解决 `In-Memory OLTP` 的一些限制，以及在完全内存迁移成本效益不高的情况下，如何在现有系统中从它获益。

尽管本书涵盖的是 `SQL Server 2016` 中的 `In-Memory OLTP`，但其内容对于 `SQL Server 2017` 中的实现也应是有效的。当然，请核实该版本解除了哪些技术限制。

最后，我要感谢你选择本书并信任我。希望你能享受阅读本书的过程，就像我享受写作过程一样。
