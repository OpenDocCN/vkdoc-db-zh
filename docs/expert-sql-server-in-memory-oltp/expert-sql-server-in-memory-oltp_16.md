# 表 4-2. SELECT 语句执行时间

表 4-2 显示了在我的环境中查询的执行时间，其中针对 `dbo.HashIndex_LowBucketCount` 表的查询慢了大约 20 倍。

| dbo.HashIndex_HighBucketCount (1,048,576 个桶) | dbo.HashIndex_LowBucketCount (1,024 个桶) |
| --- | --- |
| 301 ms | 6,259 ms |

虽然你可以清楚地看到低估桶计数会降低系统性能，但高估同样也不好。首先，每个桶使用 8 个字节来存储内存指针，大量未使用的桶是对系统内存的浪费。例如，定义索引时使用 `bucket_count=100000000` 将引入 134,217,728 个桶，这将需要 128MB 的 RAM。在单个索引的范围内，这似乎不多；然而，随着索引数量的增加，这可能成为一个问题。

此外，当 SQL Server 执行索引扫描操作时，它需要扫描索引中的所有桶，额外的桶会给该过程带来一些开销。代码清单 4-4 展示了演示此类开销的查询。
