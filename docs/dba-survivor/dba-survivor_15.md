# 第二章：现在我该做什么？

那么，我们假设最初遇到的每一条警报都需要你采取行动。请注意，我并未在初始清单中包含这一点，但这只是因为响应警报应作为之前讨论任务的**次要事项**。

当你花费数小时响应某个不太关键系统的警报，而最关键系统却连续多日没有数据库备份，这说得过去吗？

但现在是时候问问别人了：“嘿！我们的警报系统是什么？” 如果可能，你应该查看不同类型的警报，看看它们是否可以被标记为**关键**或**信息性**。

你如何判断警报的严重性？你怎么知道它是否真的重要？很简单——如果它与你负责的某个数据库或客户相关，那么毫无疑问你应该调查这条警报。如果它不属于前面列出的项目，那么毫无疑问你也应该调查这条警报。换句话说，如果你不知道答案，那么答案必须是肯定的。

**提示：** 记住，如果你不知道一条警报的严重程度，那就按它非常严重来处理。

## 我应该调查那条警报吗？

既然你知道任何来源不明的警报在证明其无害之前都应被视为严重，真正的问题就变成了“下一步怎么办”？如果你团队里有其他人，那么你的第一站应该是去问他们。如果只有你自己，那就问你的经理。如果这两条路都走不通，那你还有几个其他选择。

### 检查警报系统

开始深入调查警报系统，看看能否找出为什么会生成这条警报。通过这样做，你会更多地了解警报系统，从而更容易理解将来可能收到的警报。如果你担心的警报是直接明了的事情，比如 `Database Mail` 因 `SQL Server Agent` 中作业失败而给你发送邮件，那么你的调查工作应该很简单。如果警报系统稍微复杂一些，比如 `Operations Manager`，那么可能需要花更长时间才能确定细节，但从长远来看，现在的额外努力是值得的。

在检查警报系统时，务必确认它不是将邮件发送到单个邮箱（你的邮箱），而是发送到一个**通讯组列表**。除非你计划每分每秒都保持在线并响应，否则始终使用通讯组列表来接收警报通知。

### 询问开发人员

如果你已经收集了警报所涉及服务器或数据库的详细信息，那就去问问开发人员，看他们是否对警报性质有更多了解。他们也许能阐明相关事件；可能是批量加载失败了，或者他们正在进行一些测试。最坏的情况是，这预示着某个重要问题，而他们将会非常感激你在通知其他人之前先来找他们。

但如果他们也没有更多细节呢？

### 询问服务器管理员

下一个合乎逻辑的步骤是联系服务器管理员。他们中可能有人熟悉警报系统或警报本身的性质。他们可能还配置了一些邮件规则，这对你或许也有帮助。

四处问问看；同样地，你可能正在发现比你意识到的更严重的事情，服务器团队会非常高兴你在联系……你的客户之前先来找他们。

### 询问你的客户

当所有其他方法都失败时，就去问客户自己，看看他们是否知道当前有任何问题。你可能会发现他们已经遇到了一些麻烦，他们会很高兴看到你。通过帮助他们解决任何可能的问题，你也可能发现警报的本质。如果他们目前没有任何问题，那么你现在提高他们的意识将是有益的，以便他们在以后出现任何异常时可以再次联系你。

每当一条警报出现，你都得到了一个学习新事物的机会。拥抱它，并学会爱上它，因为如果你不这样做，它很快就会成为你存在的祸根，以至于你想把你的黑莓手机（Crackberry）扔到墙上。

## 嗨，一起吃个午饭？

至此，你已经对系统进行了基本的健康检查，了解了你的客户，也知道了哪些系统最重要，那么是时候开始更多地了解在接下来几周和几个月里你将与之密切合作的人了。

**提示：** 开始一份新工作时，每个人都是陌生人。你需要学会如何结识他人，以发展互利的关系。

我本人天性非常害羞。大多数人听我这么说都不相信，但这是千真万确的。因此，我很难结识新朋友，甚至是我的同事。这让别人很难了解我，也让我很难了解他们。

反过来，这可能会使得建立牢固、成功的工作关系变得困难。如果说我这些年学到了什么，那就是与你上下链条中的人**管理关系的重要性**。

### 以吃会友

害羞或天性内向的人如何能够与他人建立和管理关系？这不是一件容易的事。事实上，有时会让人感到不知所措。但我发现了一个每个人都具备的非常普遍的特质：**每个人都要吃饭**。

这是真的。每个人都会在某个时间点吃饭。你可能会发现他们坐下来吃午饭。

你可能会发现他们早上去买杯咖啡。很可能你迟早会看到你每个同事在某个时候吃东西。而且很可能你自己也时不时地吃点东西。是什么阻止你们偶尔一起吃饭呢？为了打破僵局，在 11:45 左右去别人的办公室或隔间，介绍自己是新人，并问他们是否知道附近有什么好吃的午餐地点。

我每天早上都会走过去买杯咖啡。我并不总是自己买一杯，但小组里至少有一个人会买。那我为什么要去呢？因为我和他们共度的十分钟，给了我额外十分钟与他们在工作之外的互动时间，而如果我坐在自己的隔间里，就不会有这十分钟。

时不时地，我会发现自己和别人一起吃午饭，或者走去买午饭。在我觉得自己吃太多的日子，我会和另一群同事去健身房。换句话说，我一直努力找到以不同方式与不同群体的人互动的机会，以便我们能更好地了解彼此。这是否意味着我们总是相处融洽？绝对不是。它确实意味着我们在努力相处。

## 追踪你的进展

还记得那个清单吗？很好，因为你需要它来规划未来几周的进展。作为一名 DBA，你的很多工作都是在幕后完成的。事实上，人们常常会好奇你整天都在做什么，因为你的大部分工作最终用户根本看不到。当你试图向人们展示你所交付的一些切实成果时，你的清单将对你大有裨益。



无论你在未来几周内会见和问候多少人，除非你能向你的经理和其他人提供一些切实成果的证据，否则人们迟早会想知道你整天都在做什么。如果你的初始检查清单显示你有二十五台服务器，其中六台在 C: 盘上存放了数据和日志，另外两台根本没有备份，那么你日后报告你的二十五台服务器现在都已配置好备份且所有驱动器都已正确配置时，就会容易得多。

最终，人们记住的不是你的努力，而是最终的结果。务必妥善跟踪你的进度，以便用事实帮助人们准确理解你所交付的工作。

## 主动出击

既然你的大部分调研已经完成，你应该对你在新环境中面临的局面有了非常清晰的认识。随着时间的推移，你将在新的角色中建立自己的身份。大多数 DBA 都会归入以下两类之一：*正确先生* 或 *及时先生*。

以下哪一项最能描述你处理工作的方式，或者你希望别人如何看待你？

*及时先生*：这个人总是随时准备解决任何问题。他非常引人注目，每个人都知道他的名字，也知道他做什么。他负责解决问题。无论白天黑夜，他总是在那里。一旦出现问题——任何问题——你打电话给及时先生，他总能找到解决办法。他很少无法交付成果，每个人都称赞他的辛勤工作和努力。然而，每当他解决了一个问题，他只是做到足够好以便能转向下一个紧急任务。

*正确先生*：这个人你很少见到。出现问题时，他会探头看看，但由于问题很少，人们从不真正知道他在场。事实上，人们常常问：“那么，你在这里到底做什么？”但是，当正确先生解决一个问题时，他会彻底解决，确保永远不必再为同一个问题去碰那台服务器。然后，他会主动地对其余服务器执行相同的修复，以防它们发生类似问题，从而在问题发生前就将其解决。

你更愿意成为哪一种人？你是愿意成为那个非常显眼、终端用户因为他总是随时修复问题而喜爱的人？还是愿意成为那个人们很少见到、完全不知道你以何为生的人？

上大学时，我们总是被告知要更聪明地学习，而不是更用功。同样的原则也适用于这里：更聪明地工作，而不是更辛苦。

### `RAID` 简介

可能你需要熟悉的最重要的硬件概念是 `RAID`，它代表*廉价磁盘冗余阵列*。`RAID` 本质上是关于以不同方式将数据存储两次或更多次在不同的磁盘上。我说的有点简化了，但 `RAID` 的一个核心目标是让你在丢失一个磁盘时也能幸存下来，而不会丢失该磁盘上的任何数据。

`RAID` 的冗余部分使我们能够在丢失驱动器时幸存下来。有不同的方式提供冗余，我将在本章中介绍这些方式。

了解不同的 `RAID` 配置将在多个场合拯救你。不同的情况需要不同的 `RAID` 配置。它们可用于提高性能以及防范灾难。

### 为什么 `RAID` 如此重要？



考虑一下从全国性超级商店连锁店购买的大多数家用电脑系统。它们自带充足的内存、强大的处理能力和大量的磁盘空间。但唯独不配备备份（或外部）驱动器。您可能会问，为什么需要那种东西？用户需要它来保护重要文件，例如电子邮件、猫咪的照片以及他们所有喜爱的网页链接。

许多家庭用户不会考虑备份。他们年复一年地愉快使用，不断向硬盘中添加财务数据、照片、电子邮件收藏和 iTunes 音乐。直到有一天，他们以惨痛的方式认识到了硬盘故障。他们的硬盘坏了，电脑无法启动，所有东西都丢失了。

那么服务器呢——它们会配备额外的磁盘吗？通常，是的，订购服务器时通常会包含额外的磁盘。根据服务器型号的不同，可用的选项和磁盘数量也不同。除了需要磁盘容量外，配备额外磁盘的主要原因是为了在发生硬盘故障时，通过利用 `RAID` 来保护自己。

例如，图 3–1 展示了一种可能的 `RAID` 配置。在图 3–1 中，三个硬盘被绑定在一起，表现为一个硬盘。相同的数据被写入所有三个驱动器。因为这三个驱动器是彼此的镜像，所以我们可以损坏其中一个驱动器，仍然可以通过另外两个访问我们的数据。我们甚至可以损坏两个驱动器，只留下一个好的。

**注意：** 不要仅仅依赖 `raid` 来避免灾难。`RAID` 不是备份。当用户删除一条记录时，它会立即在所有驱动器上被删除。你仍然需要备份来从这类小问题以及服务器故障、数据中心中断和自然灾害等重大灾难中恢复。

### 用于性能的 RAID

你也可以利用 `RAID` 来提高性能。了解如何做到这一点，在尝试进行查询性能调优时会很有帮助。当你完成初始检查清单并开始收集服务器配置的基线信息时，这也会让你受益匪浅。

SQL Server 是一个非常消耗输入/输出（I/O）的系统。这意味着你能越快地从磁盘读取和写入数据，响应时间就会越快。你可以使用 `RAID` 通过利用将数据*条带化*分布在多个磁盘上的方式来增加 I/O 吞吐量。

例如，一个传统的硬盘内部有一个小磁臂，每次你想读写数据时它都会来回移动。假设你需要执行 100 次读取和 100 次写入操作，每次操作耗时 1/100 秒。在单个磁盘上需要多长时间？两秒钟。现在，假设你如图 3–2 所示配置了 `RAID`。

### 用于容错的 RAID

`RAID` 使用两种方法来提供容错能力：镜像和奇偶校验。镜像正如其名：你在两个（或更多）不同的磁盘上有两个（或更多）数据副本。图 3–1 中的场景就是一个镜像场景。

**注意：** 镜像通常被认为是两个驱动器相互镜像。然而，没有任何因素阻止你跨两个以上的驱动器进行镜像。

*奇偶校验* 是指你向其中一个磁盘添加称为*奇偶校验数据*的额外数据，这些数据可以在某个磁盘故障时用来重建你的实际数据。奇偶校验方法的优点是节省磁盘空间。图 3–3 展示了一个字节的奇偶校验数据如何保护两个字节的实际数据。

奇偶校验方法的 `RAID` 是一件好事，除非你关心性能。它会对性能产生负面影响，因为每次都要写入那个额外的奇偶校验信息，这会增加额外的开销。

### RAID 0

这个级别通常称为*磁盘条带化*，其结果通常被称为*条带集*。你的数据被分成块（或区块），并以固定顺序分布在阵列中的所有磁盘上。图 3–4 说明了这种方法。

**图 3–4.** *一个 `RAID 0` 配置示例*

`RAID 0` 会提高性能，因为所有操作都可以如前所述同时独立执行。缺点是任何一段数据只存在于一个磁盘上，因为数据被均匀地分布在所有磁盘上。一旦驱动器发生故障，该磁盘上的所有数据将永久丢失。实际上，情况更糟：失去一个驱动器，就相当于失去了阵列中的所有数据。从 `RAID 0` 阵列的驱动器故障中恢复的唯一方法是从上一次备份中恢复整个阵列。因此，`RAID 0` 不是 SQL Server 存储的好选择。值得注意的是，`RAID 0` 既不提供容错，也不提供冗余。

### RAID 1

这个级别更常被称为镜像。它就是前面图 3–1 所示的场景。你的数据只是简单地同时写入至少两个不同的位置，在单独的磁盘上创建一个或多个数据副本。一旦发生驱动器故障，阵列仍然完全可用，因为至少还有另一个驱动器副本在线且功能正常。

### RAID 5

## 你想要哪种 RAID 级别？

这当然取决于情况。你是否希望当一个驱动器故障时，你的数据仍然保持在线？大多数人都希望如此，但也许你不是。那样的话，`RAID 0` 就适合你！无需担心奇偶校验或镜像，只需将你的磁盘配置成一个大型条带集就完事了。

那性能和成本呢？大多数人也关心这些，很可能你也关心。但当涉及到 SQL Server 时，你会发现没有一个单一的 `RAID` 级别能满足你所有的需求。等等……你了解自己的需求是什么吗？嗯，让我们以最简单的方式讨论一下。

-   数据库数据文件的放置位置
-   事务日志文件的放置位置
-   tempdb 数据库文件的放置位置
-   数据库备份的放置位置

好吧，那是四个需求，但真的就这些了。你的数据文件应该放在一个驱动器或磁盘组（阵列）上，你的事务日志文件应该放在另一个单独的驱动器或磁盘组上，你的 tempdb 文件应该放在第三个组上，而你的备份应该存储在第四个组上，或者完全放在另一个服务器上（这通常是推荐的做法）。

在故障排除时，你最需要认识到的是压力来自哪里。是 tempdb 吗？如果可能，将你的 tempdb 放在 `RAID 10` 上。是事务日志吗？如果可能，将你的事务日志放在不同的 `RAID 10` 组上。如果不行，那么为你的事务日志选择 `RAID 1`。是数据文件吗？你的数据也应该放在另一组 `RAID 10` 上，但如果那不可能，那么为你的数据选择 `RAID 5`。

归根结底是：了解这些 `RAID` 级别之间的差异，以及你想使用哪些级别。这些知识在你帮助设计任何系统架构时将大有裨益。

## 为什么 RAID 5 可能适合你

首先不必担心的原因是，`SAN` 控制器具有内置缓存：作为服务器与其存储阵列之间缓冲区的内存。每当 SQL Server 向你的驱动器发出写入操作时，只要数据进入 `SAN` 控制器的缓存，这些写入就被认为是已提交的。配置正确时，该缓存比任何磁盘阵列都要快，无论是 `RAID 5` 还是 `RAID 10`。虽然你的 `SAN` 可能使用 `RAID 5`，但你的数据并非直接写入磁盘；它首先被转储到缓存中，然后再写入磁盘。

你的查询不会等待数据真正写入磁盘；当数据实际上还在缓存中时，它们就认为数据已经到位了。只有在缓存实际被填满的情况下，你才会看到性能问题，而在这种情况下，你将*处处*遇到性能问题，而不仅仅是一台服务器。
