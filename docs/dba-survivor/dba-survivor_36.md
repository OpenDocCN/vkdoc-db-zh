# 第六章：基础故障排除

大多数高级数据库管理员（DBA）都实施了某种程度的**主动监控**。这使他们能够察觉环境中的变化，从而在问题出现时知道该检查哪里、该问什么问题。但他们最初是如何知道要监控些什么的呢？

每个人都是从对环境的基本监控开始的，但随着时间的推移，你可能会发现自己在监控一些奇怪的项目。这通常源于一次过往事件，随后产生了一种本能反应——你被要求（或命令）开始监控某些特定的东西。

**注意：** 这让我想起了航空公司要求脱鞋安检的做法。这个做法是汤姆所描述的那种“反应导向型监控”的典型案例，由单一事件引发。你监控的部分内容，也将是你雇主或客户特定历史背景下的类似产物。——乔纳森·詹尼克（Jonathan Gennick）。

例如，许多公司会监控数据库备份以确保其成功运行。但很少有公司建立流程来验证数据库备份能否被及时恢复。只需发生一次你无法完成恢复的情况，你就会发现你被要求更密切地监控数据库备份。在一个具体案例中，这意味着我需要开发一个流程来验证数据库备份是否仍然存在于磁盘上，因为等待一天从磁带恢复被认为是不可接受的。

另一个需要考虑的因素是，你的监控方案需要能减轻正在发生的事件。如果某件事发生了一次，而你无法立即给出答案，你可以通过解释你将如何实施更好的监控来争取一些时间。但如果同样的事件持续发生，尽管你努力监控局势却仍无解决方案，那么你最好相信，人们对你能力的信心会逐渐减弱。

随着时间的推移，你会发现你的监控方案会随着对已发生事件和问题的反应而发展。真正重要的是最终结果，即你需要拥有能够重现事件或在事件发生时进行监视的工具。

## 常见瓶颈

`SQL Server`的问题通常分为三大类：磁盘 I/O、内存或 CPU。因此，在试图排查任何特定问题时，这三项是很好的起点。当深夜电话铃响时，你需要开始排除尽可能多的疑点。这样做将使你能够专注于找出问题的真正根源。

很可能，你得到的所有信息只是服务器“很慢”。终端用户对问题的性质知之甚少或一无所知；他们只希望问题消失。需要由你来知道在何处以及如何分析情况。因此，你应该熟悉最常见的瓶颈以及如何快速诊断是否是其中任何一个问题导致了故障。本章后面的“可用工具”一节中，我将讨论你开箱即用的各种工具。我将解释如何使用每个工具来了解有关磁盘 I/O、内存和 CPU 的更多信息。

在接下来的章节中，我将解释每一项为何重要以及它们在整体性能中扮演的角色。

### 磁盘

如果你使用数据库，那么你就需要将数据写入磁盘和从磁盘读取。你希望数据能尽快写入磁盘。你同样希望数据从磁盘读取的速度也一样快。你还希望磁盘有足够的空间来存储所有数据。所有这些都需要充分的规划和一些性能调优。不幸的是，在深夜，你不会有奢侈的时间去做其中任何一项。你需要有一种方法来快速确定磁盘 I/O 是否存在问题，无论是作为根本原因还是其他问题的症状。

除非你的数据库足够小，可以完全放入服务器内存中，否则你的性能将取决于你的磁盘 I/O。这是因为数据库引擎会不断地将数据页从内存换出到磁盘，反之亦然。`SQL Server` 还会使用 `tempdb` 来存储数据，而事务日志也需要将页面刷写到磁盘。

对于单个数据库来说，这已经产生了大量的 I/O 活动。数据、日志和 `tempdb` 都需要从磁盘读写数据。现在考虑到你的服务器上可能不止一个数据库。很容易理解磁盘 I/O 如何成为任何服务器的瓶颈。正确对齐数据文件、日志文件和 `tempdb` 对于获得良好性能至关重要。

### 内存

应用程序需要使用计算机内存才能运行，`SQL Server` 也不例外。你可能会遇到物理或虚拟性质的内存压力，在这两种情况下，压力都可能是内部或外部的。在深入此类细节之前，你需要能够判断是否确实存在内存压力。

仅仅因为 `SQL Server` 消耗内存并不意味着存在问题。

我喜欢将数据库服务器的内存配置设置为最大和最小内存相同的固定大小。在专用的数据库服务器上，我会只留下足够基础操作系统运行的内存，通常是 2-3 GB，其余的都分配给 `SQL Server`。这可能会导致一种情况：你的电话会响，电话那头的声音会说“`SQL` 消耗了大量内存，请做点什么来解决这种情况。”当然，你没有什么需要修复的，因为你就是将其配置成那样运行的！

如果 `SQL` 服务确实在满负荷或接近满负荷运行，那么你可能会开始看到与内存相关的错误消息。如果你还记得，我已经讨论过 `SQL` 如何在内存和磁盘之间交换数据页。所使用的内存区域称为 `缓冲池`。从内存读取比从磁盘读取快，因此你能在 `缓冲池` 中存储的数据越多，你的性能就越快。

但当你的服务器面临内存压力时，你的 `缓冲池` 将无法处理增加的请求，性能就会下降。这时你的电话就会响起了。

### CPU

CPU 峰值会不时发生，通常定义为可用 CPU 利用率超过 80%。CPU 峰值代表了一种矛盾；你既不想看到 CPU 被过度使用，同时你又希望使用可用的 CPU，否则它们就闲置在那里。让资源闲置可能被认为是浪费，对吧？所以，你必须确保你的 CPU 被高效使用，但又不能过度到使服务器本身负担过重。
