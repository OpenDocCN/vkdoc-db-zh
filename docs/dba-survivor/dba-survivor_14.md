# 第二章：现在我该做什么？

根据你所在公司的情况，你可能有也可能没有相关的职责。许多数据库管理员只专注于服务器实例的管理，而将操作系统的管理交给其他人。如果对你来说是这种情况，那么你需要努力去了解这个团队。服务器管理员掌握着整个公司的脉搏。他们知道哪些系统最重要，也知道在你遇到困难时去哪里找到一些额外的硬件。

## 系统管理员是你的朋友

作者：乔纳森·杰尼克

如果你和管理你服务器的系统管理员关系友好，那将大有帮助。如果一位数据库管理员能够自如地坐下来与系统管理员一起解决问题，嗯，我想没有比这更好的情况了。

系统管理员也将是构建你基础设施的人。了解你正在使用的硬件以及可能即将到位的硬件至关重要。如果你的公司计划实施一个存储区域网络，他们不会来问你如何配置它；他们会去问服务器团队的某个人。由于存储区域网络能够并且将会对数据库服务器产生巨大影响，你需要参与这些讨论。理想情况下，你越早受邀参与讨论越好，你与服务器团队相处的时间越多，就越有机会了解即将到来的变化。

你还需要经常帮助开发人员和服务器管理员相互沟通和理解。为什么会这样呢？嗯，把自己想象成一个通用翻译器。开发人员懂代码，也了解业务的各个方面；因为他们每天专注于这些事项，没有时间去上架服务器、安装路由器、更换硬盘、在存储区域网络上分配空间等等。服务器管理员做这些事情，但他们不知道如何编码或构建能像开发人员那样快速推动业务发展的应用程序。

不幸的是，除非出现问题（或者有机会相互指责），这两个群体很少交流。这就是你发挥作用的地方。开发人员可以指向你，解释当前的障碍是什么，然后你可以转身用服务器管理员能理解的语言与他们交谈。

没有你，这种情况就不会发生，挫败感会上升，最终你的公司会在硬件上花费太多金钱来解决本可以通过修改几行代码就能解决的问题。

因此，通过与这两个团队会面，你可以不断提升自己的“语言”技能，帮助维持和谐，最终还能帮助降低成本。

### 与你的客户会面

到目前为止，你几乎已经与所有可能的人会面了，除了你真正的客户。

实际上，当你是一名数据库管理员时，每个人都是你的客户，但让我们专注于应用程序和服务器的最终用户。这些人会给你提供一个与其他任何群体都截然不同的视角。

为什么与他们见面如此重要？嗯，想想这个故事。

一天，一个男人看到一则广告，租用一个家庭车库上方的卧室。这个男人外表整洁，有一份工作（没人知道是什么），而且总是待在那里。但他是个不错的家伙，房租从不拖欠，这家人也喜欢有他在身边。这家人从未注意到，这个男人总是喜欢和他们一起开车出去。目的地无关紧要；男人会问：“介意我搭个车吗？”家人就会让他上车。

每次他坐车时，都会问一两个关于车的问题，但数量不多，不至于让家人觉得奇怪。他只是以某种方式把问题融入到对话中。“那么，你觉得这里的杯架够多吗？”可能会引发关于汽车设计和功能的各种评论。

最终，男人告诉这家人他要搬出去了；他已经找到了一个新的公寓。离开之前，他终于告诉他们他的职业：他在汽车公司工作。他问那些问题是为了获得关于汽车设计的真实反馈。他们提供的反馈比任何调查所能期望的都要可靠和诚实得多；这个男人观察到的是家人实际使用产品的情况！

因此，你需要对你的最终用户做同样的事情。更多地了解他们如何使用他们的工具，他们的困扰是什么，以及哪些小事能帮助他们（从而帮助你的业务）更高效、更有生产力。去问问你的客户，你是否可以搭他们的车。你会发现的会令你惊讶。

## 那个警报严重吗？

好了，你已经收集了你的信息。你已经与各种各样的人会面了。你回到办公桌前，发现大约有 1500 封邮件在等着你。该怎么办？其中一些邮件可能是数据库警报。最好查看一下。

**提示：** 在电子邮件中设置规则来自动归档警报是件很棒的事，但前提是要仔细且正确地完成。

某个特定的警报重要吗？我不知道。它可能是。或者可能可以安全地忽略。如果它真的很重要，那么很快就会有人开始对你嚷嚷，对吧？嗯，从技术上讲，这可能是对的，但我们别让这种情况发生在你身上。当涉及到警报时，遵循一个基本规则：

*如果你不知道它们是干什么用的，那么你需要弄清楚你为什么会收到它们。*

简单易记。遵循这个规则将决定你是被积极看待还是被消极看待。考虑以下情景：一个慵懒的周日下午。你正在家里放松。你的黑莓手机收到一条你无法理解的晦涩信息。你假设其他人一定也收到了同样的信息并且知道该怎么处理，所以你明天就能知道它的意思。第二天早上上班时，有用户抱怨一个重要的数据库。报告和批量加载本应在几小时前完成，但仍在运行。用户想知道问题是什么，以及需要多久才能解决问题。你完全不知道问题是什么，但你确实记得昨天那条晦涩的信息。你做了一些研究，找出警报来自哪里，找到信息本身，开始理解它可能意味着什么，最终定位了问题。你开始让系统恢复正常运行，但到那时它基本上已经停摆了大半天了。没有其他人收到寻呼——只有你——因此所有的责任都落在你一个人的肩上。

听起来是好时光吗？并不是，这描述了我工作最初几周发生在我身上的一件事。从那天起，我就把确保我负责的任何警报系统都只提供有价值的信息作为我的个人使命。我经常听说有些公司被邮件淹没。有些是警报，有些只是信息。我总是寻求高效，这就是为什么我只想要针对可操作项目的警报。
