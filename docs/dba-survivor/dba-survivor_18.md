#### 事务复制

事务复制是指在发布者处提交的事务被复制并提交到订阅者处。这用报纸或杂志来类比就说不通了，所以请把它想象成您正在阅读本地报纸的在线版。如果有人对网站上的文章进行了修改，您下次刷新浏览器时就会看到这些更改。

何时应使用这种类型的复制？答案通常涉及具有高事务量且需要低延迟的系统，以及您的其中一个订阅者不是 Microsoft SQL Server 的情况。此外，事务复制拓扑中的订阅者通常被认为是只读的，但也可以允许订阅者对数据进行更改，然后将这些更改推送回发布者。

当您有一个非 SQL Server 的服务器，或者您需要多个订阅者时，事务复制是有意义的。如果您只有一个订阅者并且您只有 SQL Server 数据库服务器，那么您应该采用数据库镜像，如本章后面将讨论的那样。但请考虑您希望复制更改的范围。使用数据库镜像，所有事务都会被镜像并因此被复制，而使用事务复制，只有那些标记为复制的事务才会被复制。

#### 合并复制

这可能是所有复制技术中最庞大的怪兽。通过使用触发器，发布者和订阅者处的更改被合并在一起。想象一家报社需要发布晚间新闻，有十名记者在现场更新报道。现在假设两名记者正在报道同一新闻并尝试发布不同版本。哪个版本会被写入？您将如何解决冲突？

合并复制不适合意志薄弱者。当这种复制出现问题时，您最好有方法将其恢复。合并复制可能会带来大量的管理开销。那么，为什么还有人想使用它呢？

我最常见到它被用于您有与中央发布者断开连接的用户时。

您经常会看到这样的例子：销售团队成员在外拜访客户。他们的移动设备和笔记本电脑并不总是连接着，但他们可能正在收集数据。当他们连接到网络时，他们希望他们的数据能相应地更新。如果另一位销售代表尝试更新相同的数据，冲突可能（也将会）发生。

#### 快照复制

快照复制比其他两种复制形式所需的管理开销要少。顾名思义，会对数据库进行快照，并整体复制到订阅者。当您的数据不经常变化时，这非常有用。

也许您每年需要几次向销售团队更新可用产品列表。当销售代表连接到您的网络时，可以将数据的快照推送给他们，这样他们就能看到所有新产品。

在非常大的数据集上使用快照复制时要小心。如果您决定在一天中段将一个 100GB 的快照推送到 100 个订阅者，很容易引发网络问题。

如果您没有迫使您使用事务复制的多服务器平台需求，并且不想要合并复制的麻烦，那么快照复制可能是适合您工作环境的正确解决方案。虽然快照复制是最容易管理的类型，但还有一种移动数据的方法所需的工作和时间更少。

### 数据库镜像

最近，有人要求我尽可能用最简单的术语解释数据库镜像。

我能想到的唯一回答就是下面的故事：

很久很久以前，在远古时代，在时间诞生之前，人们被要求复制他们的工作。这导致历史上一些非常有趣的发明，最著名的是印刷机、复写器，以及最终在节日办公室派对上大受欢迎的现代复印机。但随着社会进步到以秒的分数而非天或周来衡量交付成果的程度，这些古老的方法已不再足够。人们面临着越来越多的需求，要同时将同一份信息传送到许多不同的地方，且不能有任何服务中断。

不久，一场大战在两只名为“集群”和“日志传送”的巨兽之间展开，它们在两个月前的一次付费观看事件中轻易击败了劣等的“SQL 复制”。集群和日志传送战斗了多年，互不相让，直到双方都伤痕累累，虚弱不堪，几乎无法站立。

就是这样。数据库镜像旨在通过结合其他工具的所有优点并摒弃所有缺点，为您提供高可用性的最佳方案。然而，有一个微小的问题使得数据库镜像并不像人们想象的那样完美：您只能获得一个镜像。

### 卡内基音乐厅

有一个老笑话，讲的是一个年轻人在纽约市问路。他拦住一位老先生问道：“去卡内基音乐厅怎么走？”那位老先生回答说：“练习、练习、再练习。” 当我担任篮球教练时，我总是告诉我的球队，要在“比赛地点”以“比赛速度”练习“比赛投篮”。换句话说，你练习的方式就是你比赛的方式，应该始终全力以赴。如果你以半速练习，你最终就会以半速比赛。

所有这些对于灾难恢复准备都适用。你需要练习并保持技能的敏锐。你上一次尝试恢复 `master` 数据库是什么时候？或者，更好的是，你*曾经*尝试过恢复 `master` 数据库吗？相信我，当灾难事件中压力山大时，你最不想做的事就是笨手笨脚地翻手册，或者因为不知道如何启动实例而用拳头砸显示器。

练习恢复 `master` 数据库。有时也尝试恢复 `msdb` 数据库。抽查你的备份文件并进行还原操作，以确保你熟悉这个过程。要特别关注你重要的系统。如果你其他事情都没做对，仅仅能够毫无问题地恢复重要系统可能就足以保住你的工作，所以请多加练习，并确保你熟悉它们可能包含的任何微妙之处。

在灾难方面，你永远无法练习得足够多。作为一名数据库管理员，你的主要职责是在灾难发生时恢复数据。因此，这是你最需要规划和最常练习的事情。

## 网络就像卫生间

当问题出现时——*问题总会出现*——人们会开始责备他们最不了解的环境部分。作为数据库管理员，你会发现人们不可避免地会因为各种问题而责备数据库服务器。根据我的经验，我看到人们因为一堆与数据库服务器毫无关系的问题而责备数据库服务器。事实是，他们知道数据库服务器存在，但他们不知道这是否是真正的问题所在，所以他们首先就把矛头指向了你。

人们最后才会想到的事情之一就是网络。你的网络一直有波动。有大量的线缆、路由器和交换机组成了你网络的骨干。有这么多需要协同工作才能让你的系统运行的活动部件，很容易理解问题会不时出现。我曾经读到一项研究，说数据库服务器是人们被要求进行故障排除的第一大基础设施，但超过 70% 的问题都直接与糟糕的应用程序代码有关。
