# 第二章：现在该做什么？

## 你已获得信息，接下来呢？

到目前为止讨论的所有内容，至少应该能让你忙到第一天午餐时间。好吧，也许你需要几天时间才能收集齐清单上的所有信息。你可能已经掌握了其中的大部分信息，也可能没有，因此收集所有资料所需的时间将取决于你所在单位特定部门的当前结构。（顺便说一句，习惯听到和说“视情况而定”吧；关于这一点后面还会详述。）不必担心，因为你可以在继续收集数据的同时，过渡到下一个阶段。

我还要建议你简化收集这些信息的方式。例如，如果你有一百台服务器，你肯定不想远程连接到每一台上去检查各自的内存设置。你需要学习如何高效地收集信息，其中很多可以使用像 `System Center Configuration Manager (SCCM)` 和 `Operations Manager` 这样的工具来收集。如果你的环境没有这些工具，那么你可能需要使用 `PowerShell`、`T-SQL` 或其他替代方案来编写基线查询脚本。

或者，你可以使用 `SQL 2008` 中一个定义的中央管理服务器，同时对多个实例运行查询。你也可以考虑使用基于策略的管理来报告并可能强制执行配置选项。无论你选择哪种方法，目标始终如一：跟踪随时间发生的变化。一旦你捕获了环境当前状态的快照，你就可以开始记录如何在出现问题时将服务器恢复到原始状态。

### 制定你的恢复计划

注意我说的是“恢复计划”而不是“备份计划”。到目前为止在你的清单中，你已经确认了数据库备份正在运行，开始抽查以确保你能从备份中恢复，并大致了解了你的重要数据库。现在是时候将所有这些整合成一个灾难恢复（DR）计划了。对于每一个被列为重要的数据库，你都应该写下灾难发生时恢复所需的具体步骤。

请务必认清这一点：如果灾难发生，你的工作将岌岌可危。如果你因为准备不足而恢复失败，那么到下周末之前，你很可能就会被调去负责“特殊项目”。避免这种情况发生的最佳方法就是练习、练习、再练习。你的业务可能每年会安排一些 DR 测试，但这不应阻止你自己更频繁地进行小规模 DR 测试。

记录每个系统以及恢复所需的所有步骤。数据库是否处于 `完整恢复` 模式？你多久进行一次事务日志备份？写下备份时间表，以便清楚恢复点在哪里。如果你的客户期望恢复到分钟级别，而你处于 `简单` 模式，那你就真的要面临灾难了。

别忘了从过去几天或几周恢复的情况。如果你的客户需要恢复两个月前的数据库备份，请确保你知道完成该工作的每一个步骤。如果你公司使用异地磁带存储服务，并且需要两天时间才能从异地取回磁带，那么你必须作为 DR 计划的一部分提前将这一事实告知你的用户。

### 与你的经理会面

在收集数据的同时，你应该与你的经理会面，讨论你的初步发现，并共同确定需要完成工作的优先级。如果你发现一台服务器没有运行任何备份，或者备份失败，那么这台服务器必须优先处理。我不在乎你哪个“最重要”的客户在第一天就冲你大喊大叫，要你修复他们写的、填满了 70GB 日志驱动器的查询，你*必须*确保自己能够从灾难中恢复。

继续与你的经理一起审查你的清单，确定哪些项目需要完成，哪些项目需要最多的资源和精力，以及哪些项目可以暂时搁置。在开始工作之前，你和你的新经理就任务及其优先级达成一致非常重要。这将对你们双方都大有裨益；它让你从现在开始有据可依。

然而，如果你和你的经理在优先级上存在分歧，请确保记录在案。当有人过来询问为什么某件具体的事情还没做时，这将对你大有帮助，你可以礼貌地解释你一直在忙于其他事情。然后，如果仍然存在分歧，你、另一个人以及你的老板可以一起重新评估你的优先级。

**提示：** 记录下来，但不要为了对抗。不要拿老板的旧邮件当头棒喝。相反，把它们当作温和的提醒，表明你正致力于完成老板定下的优先事项。你的老板会欣赏你为他的优先事项付出努力。

与你的经理会面后，你将不断受到衡量。每个人都会因为这样或那样的事情需要你。你与他人的每一次互动都将构成一种衡量形式，并通过你采取行动所产生的有形成果来验证。如果你有那份清单，并且你的经理知道这份清单，那么从第一天起，你就有具体的东西可以交付了。

想想这对你有多重要。三四周后，某个地方的某个人会问你的经理，新来的 DBA 工作情况如何。你的经理将能够迅速肯定你一直所做的出色工作（我们就假设你工作出色，而不是整天无所事事）。你将能够轻松跟踪自己的进展，最终你可以准备一份简要的总结或演示文稿，详细说明你在最初几个月的工作进展。

### 与开发人员会面

如果你公司有内部开发团队，花点时间去见见他们，认识一下。在未来的几个月里，他们将成为你成功与否的最关键因素。

了解他们正在做什么项目，目前遇到了哪些障碍，看看是否有你能提供协助的机会。

毫无疑问，开发人员会让你非常忙碌。开发人员将是那些试图利用现有技术挑战极限的人。最终结果将是为你提供大量可以利用的“培训机会”。填满了事务日志？让我来了解一下你原本想做什么。整个日志驱动器都满了？你的远程查询性能很差？你那有 27 个嵌套游标的存储过程运行时间太长？来，让我帮你处理，也许我们可以找到更好的方法来完成工作。

每个开发人员都想成为你的朋友吗？绝对不是——很难找到一个地方所有人任何时候都相处融洽。如果你遇到一个自认为比房间里所有人都懂得多的开发人员，你可以坐在那里偷笑，因为你早就知道 DBA 才是最聪明的，否则也不会整天被叫去解决他们的问题了。
