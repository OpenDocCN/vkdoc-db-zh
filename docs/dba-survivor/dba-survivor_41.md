# 第六章：基本故障排除

我常听其他资深数据库管理员说，`SQL Profiler` 不是一个用于基本故障排除的有效工具。然而，当你需要捕获 hitting 你实例的所有活动（有时称为*工作负载*）时，它却非常有用。但除非你对问题所在有个大致概念，否则要从返回的所有活动中筛选信息会让人望而却步。想想大海捞针，你就明白这个意思了。

联机丛书甚至没有将“故障排除”列为 `SQL Profiler` 的常见用途之一，但我仍然看到许多人将 `SQL Profiler` 作为他们想要用于解决性能问题的唯一工具。一系列场景可在此处找到：
`http://msdn.microsoft.com/en-us/library/ms190793.aspx`

归根结底，`SQL Profiler` 最适用于你已经有一个特定的领域需要关注的时候。否则，你就是在用霰弹枪的方法进行故障排除，这可能非常耗时。

## 磁盘 I/O

与前面提到的工具不同，没有直接的方法可以使用探查器来知道磁盘 I/O 是否是你试图排除的问题的根本原因。没有一个事件类别、事件类或数据列会写着“在此检查磁盘 I/O 问题”。

将此与等待类型进行比较，在等待类型中，你可以明确地找到一个查询是否在等待磁盘，并且可以在深夜自己决定你更喜欢哪种方法。

虽然你确实可以捕获查询的读取、写入和持续时间，但仅凭这些元素并不能告诉你磁盘 I/O 是否是问题的根本原因。它们所能做的只是指引你使用额外的工具来更深入地研究跟踪结果。例如，你可能发现一个持续时间值很高但读取和写入很低的查询。你不能因此断定你有一个潜在的磁盘 I/O 问题，因为该查询持续时间长可能仅仅是因为它被其他活动阻塞了。

### 内存

`SQL Profiler` 没有提供直接的方法来测量潜在的内存问题。没有一个事件类别、事件类或数据列会写着“在此检查内存问题”。将此与可靠性和性能监视器进行比较，在后者中你可以配置特定的计数器来捕获内存资源信息，并且当有人站在你的隔间里，要求知道还要多久才能解决他们的问题时，你可以自己决定你更喜欢使用哪种方法。

不过，你可以使用 `SQL Profiler` 来捕获查询计划。在性能事件类别中，你可以使用 `Showplan All Event` 类来捕获计划。然后你可以检查查询计划，并寻找可能消耗内存的操作符，例如 `Hash Match`。就我个人而言，我认为这就像试图用锤子来换灯泡；它不是完成手头任务的正确工具。

### CPU

与磁盘 I/O 和内存不同，你（理论上）可以使用 `SQL Profiler` 来排除 CPU 问题。当然，这里的陷阱是，在开始跟踪之前，你需要知道你有 CPU 问题。否则，你又回到了霰弹枪的方法。

所以，假设你怀疑可能存在 CPU 问题，或者你知道你可以使用 `SQL Profiler` 来寻找可能的 CPU 问题。你可以运行一个包含 `CPU` 数据列的跟踪，并针对特定的 CPU 持续时间进行筛选。之前在讨论 `SQL Profiler` 时，我列出了一个可能的阈值 `CPU > 500`，以及一些类别和类。

我认为专注于前面提到的 `TSQL` 和 `Stored Procedure` 类别是个好主意；这将允许你检查 hitting 服务器的查询，并提供足够的细节，让你能够关注可能对服务器产生不利影响的特定应用程序或数据库。

### 网站

当我孩子还小的时候，他们喜欢对几乎所有事情都问“为什么？”，即使在我已经说了“我不知道”之后。这让我和妻子在短时间内就有点厌倦了，所以我开始以特定的方式回答他们的问题。大概是这样的：

孩子们：“为什么？”
我：“我不知道。”
孩子们：“为什么？”
因为我不是什么都知道。”
孩子们：“为什么？”
因为没有人能知道所有事情。”
孩子们：“为什么？”
我：“我不知道。”

如此循环，最终他们厌倦了总是问我“为什么？”

也许他们只是过了那个阶段，但我愿意相信我帮助他们足够快地度过了那个阶段。此外，我是在教他们一个非常重要的道理：没有人知道所有事情。

你也不是。随着越来越多的功能被塞进 `SQL Server`，习惯于不要什么都知道。很快，你就会需要去寻找答案的时刻。那么，你从哪里开始你的搜索呢？

#### 搜索引擎

我的搜索引擎列表非常短。要么是 Bing，要么是 Google，仅此而已。

当你不知道答案时，这两个都是很好的起点。输入一些关键词，或者一条错误信息，然后开始寻找答案。我的一位同事称 Google 为他的“初级 DBA”。

```
http://www.google.com
http://www.bing.com
```

将搜索引擎视为 21 世纪的百科全书。它擅长让你阅读别人先前印好的页面。嗯，除了那些罕见的情况，比如搜索返回了你自己的博客网站，而你已经忘记了自己多年前写了什么。

#### 论坛

论坛擅长与其他专业人士互动。大多数论坛是断开的对话。你提交一个问题，在未来的某个时间点，它可能会也可能不会被回答。或者你在论坛中搜索与你类似的问题和情况，希望能一步步接近解决方案。

以下是我的简短论坛列表：

```
http://www.sqlservercentral.com
http://www.msdn.com
http://www.stackoverflow.com
http://www.serverfault.com
http://www.twitter.com
```

`SQLServerCentral` 是可供你参与的顶级 SQL 论坛之一。`StackOverflow` 和 `ServerFault` 在寻找和获取答案方面也相当不错。当然，你也可以直接找到源头，参与 `MSDN` 论坛。

将论坛视为聊天室，并注意你正在与之交谈的人。

其他论坛，尤其是社交网站 `Twitter`，允许你几乎实时地与他人互动。如果你连接了其他数据库专业人士，那么你很有可能在 `Twitter` 上从你的网络中获得答案，速度比任何其他类型的论坛都快。

#### 技术资源

另一组你需要随手备查的网站是技术资源，例如以下两个网站：

```
http://www.technet.com
http://www.sqlserverpedia.com
```

这些网站和链接类似于将产品手册直接掌握在你手中。在某些情况下，比如 `SQLServerPedia`，它就像一张随时可用的备忘单。
