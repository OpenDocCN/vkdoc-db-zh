# 第二章：现在我该做什么？

你应该重点验证哪些数据库的可恢复性？你的工作重心几乎可以放在任何一组或一套数据库上。这里真正的目标是让你熟悉新工作环境中的恢复流程，并验证备份是否可用。在对任何重要系统进行深入探查之前，请务必全面了解你所在机构的恢复流程。这可能会在日后为你避免一些尴尬——比如你拉响警报说某个备份无效，结果发现无效的只是你对工作流程的理解。

`提示：` 练习恢复操作的另一个绝佳理由是，当你连续熬了三个通宵后，凌晨三点接到求助电话时，你能够执行恢复操作。那时你会很疲惫，思维不会清晰。你之前的练习将成为大脑的“肌肉记忆”。熟能生巧，让你在压力和疲劳下也能做出正确的操作。

## 建立客户名单

如果你清楚自己负责哪些服务器，那么就开始询问每台服务器的客户是谁。请注意，这种调查可能会产生一份非常长的名单。对于共享系统，你可能会发现每个人都与每台服务器有部分关联。

掌握客户名单至关重要。例如，当需要重启某台服务器时，知道该联系谁来解释服务器将离线五分钟进行重启，是一件好事。在编制客户名单时，了解谁是公司高管以及他们最依赖哪些服务器，也并无坏处。

当你开始列出客户时，也应该开始询问这些客户使用的应用和系统，以及他们使用这些系统最频繁的时间段。你可能会惊讶地发现，一些人们认为相对次要的系统被全天使用，而另一些被认为最重要的系统却只每月使用一次。

建立这份名单还有一个额外好处：你获得了与客户建立关系的机会。很可能，他们会对你的主动关心感到荣幸。整理名单给了你一个联系客户的理由。不妨将其视为破冰之举——一个让双方会面的轻松借口。

`不常使用的数据库也可能至关重要`

*作者：肯·西蒙斯*

汤姆关于时间段的评论让我回想起过去管理数据仓库的日子。我们有一个数据库，一个月内有三周可以处于离线状态，而且没人会注意到。这是一个具有特定用途的关键数据库。只是其工作负载恰好导致它一个月内有三周不被使用。但在第一周，它确实被使用。在第一周，它最好能 24/7 全天候运行。在剩下的几周里，它是否在线就不那么重要了。寓意是什么？不要假设某事物的使用频率与其真正的重要性之间存在必然关联。

### 列出“最重要”的数据库

在收集重要客户名单的同时，更进一步，找出他们最重要的数据库是什么。这可以通过 (1) 直接询问他们，或 (2) 询问其他人，然后 (3) 对比这些名单来完成。你可能会非常惊讶地发现它们并不匹配。你还会更惊讶地发现，有多少人会忘记自己的某些系统，需要温和的提醒才能记起它们的重要性。作为数据库管理员，我们同等对待所有数据库，但我们承认，某些数据库确实比其他数据库更重要，尤其是在特定的时间段、周或月。

例如，你可能有一个关键任务数据仓库。公司里的每个人都会告诉你这个系统至关重要。然而，他们无法告诉你的是，它一个月只使用三天。因此，这个数据库可能离线 21 天，也不会有人提出异议。

另一个例子是交易平台。这个系统可能在每个工作日被密集使用九到十个小时。但在一天剩下的时间里，它完全不被使用。这是否意味着当这些系统不被使用时，它们就不重要了？不，不是这样。这确实意味着你正在收集关于这些系统的更多细节。如果有十七个不同的团队提到某个微小的数据库，但他们认为其重要性较低，你反而可以认为它非常重要，因为它被如此多不同的人员所触及。

这里的另一个因素是可恢复性。如果业务要求你能够将数据库恢复到某个时间点，那么你也应该将此数据库视为重要数据库，即使那位客户就坐在你对面说“它没那么重要”。我已经记不清有多少次，客户随口告诉我“这个系统真的没什么大不了的——我们需要在五分钟内恢复，而且我们整天都在用它，但你不必担心它。”是的，说得对。当然，直到灾难发生，而你必须负责快速地将一切恢复原状！

### 列出即将进行的项目和可交付成果

如果周围有人能帮你了解当前和即将进行的项目，那将有助于你知晓即将落到你头上的任务。你想尽量减少等待你的意外；了解当前规划的项目有助于你理解将为每个项目分配多少时间。并且请记住，除了项目支持和你即将开始编制的行动任务外，你还被期望维持一定水平的生产支持。

你还想知道哪些服务器将在不久的将来被停用，这样你就不会浪费时间去对那些“濒死”的服务器进行性能调优。也请留意这个角度。

### 建立环境基线

为你的环境建立基线是一项经常被忽视的功能。拥有一个记录在案的起始点的重要性再怎么强调也不为过。没有一个起始点作为参考，你将很难绘制和报告随时间推移所取得的进展。

收集你所管理服务器的基线信息的想法，简单来说就是找出你与理想状态之间存在多少偏差。你已经完成了一项基线项目——你已经评估了数据库备份。由于这个检查清单项目很快就会变得有点繁琐，请尽量专注于基础项目。

例如，在一个具有直连存储的独立数据库服务器上，你的数据文件应该与日志文件存放在不同的驱动器上。有多少台服务器拥有相同的配置？那些磁盘驱动器呢？是否有标准的 RAID 配置？（你可能会惊讶地发现，服务器在物理构建上并非总是一样的。）实例的内存设置或处理器数量如何？数据文件和日志文件有多大，平均 CPU 利用率是多少？

专注于几个关键项目来收集作为你的基线信息，并持续保存这些信息。你可以根据需要随时间推移添加额外的项目。
