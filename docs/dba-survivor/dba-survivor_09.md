# 第一章：我是如何走到这里的？

## 影子所知！

又过了一年，我发现自己来到了一家新公司，他们正在寻找一些 `PowerBuilder` 方面的帮助。我接受了这份工作，大约工作了一年左右，我意识到了两件事。第一，我需要做些 `PowerBuilder` 以外的事情，因为总会有那么一天，`PowerBuilder` 的需求会下降到我无法持续找到工作的地步。第二，我想成为一名 `DBA`。

**提示：** 在生活中找到你热爱的事物；它会让其他人生决策变得容易。

我认为 `DBA` 这个角色，不会像只懂一门特定编程语言的开发者那样容易消失。在那之前，我主要做 `PowerBuilder` 的工作，并且刚刚开始接触 `.NET`。那时，为了保持市场竞争力，每隔几年就必须学习一门新语言的想法，似乎令人生畏。但成为一名 `DBA` 意味着我将成为“幕后之人”，总是在调整各种“旋钮”，并且总能找到工作。

因此，我开始寻找机会协助处理 `DBA` 任务。当公司打算招聘一名全职 `DBA` 时，我拿着简历去找了招聘经理。她立刻告诉我，他们不会考虑雇我做这份工作，因为我没有经验。

相反，他们打算雇一位之前合作过的人。我大约失望了三十秒钟，然后经理告诉我，他们也正在考虑启动一个“初级 `DBA`”计划，以培训人员在需要时提供帮助。她想把我的名字记下来，作为有兴趣参加内部培训的人，我同意了。

## 运气、准备与机遇

如果说我从运动生涯和担任教练的日子里学到了一点什么，那就是运气可以定义为当准备遇上机遇。考虑到这一点，我开始参加那些初级 `DBA` 课程。我像一块海绵，尽可能吸收一切知识。随着时间的推移，我会请求执行一些常规任务，并得到了一些高质量的在职培训。发展到后来，当常规的 `DBA`（当时有两位）需要请一天假时，他们可以放心休假，并让人们在遇到任何问题时来找我处理。

我对于处理大量常规操作任务变得越来越得心应手。

备份、恢复、密码重置、创建登录账户，甚至一些性能调优和故障排除。我还开始参与配置必要的环境监控，帮助构建一个内部监控解决方案，以便我们能更主动地解决问题。同时，我还有我的常规日常工作，处理 `PowerBuilder`、`.NET` 和 `SQR` 报告。

然后，有一天，我所有的准备都遇到了它等待已久的机遇。两位常规 `DBA` 在日历年结束时辞职了。其中一位是合同工，他离开去其他地方签了新合同。另一位则纯粹是不想再待下去了。

他们的经理过来和我的经理谈了谈，然后他们问我是否愿意成为下一任 `DBA`。我的一位团队成员评论说，这份工作对你来说是“一个巨大的机会”，于是我走进办公室与两位经理讨论这个职位。

我告诉他们我需要一些时间考虑。嘿，我可不想显得太急切！

作者：Jonathan Gennick

“有时你会创造自己的运气。”一位朋友曾在我获得一个对职业生涯至关重要的职位后，对我说过这句话。当时我为了糊口，干了两三个月的 `COBOL` 合同工。然后客户突然终止了项目，让我们十几号人甚至更多人又回到了就业市场。

我当时的热爱是 `SQL`，所以我抓住每一个机会参与到项目的数据库端工作中。最终我“拥有”了输入/输出例程，因为我既懂 `COBOL` 又懂 `Rdb`，并且知道如何将它们结合起来。我工作认真，态度良好，并对我所做的事充满热情。回想起来，我工作做得非常出色，并且与同事们分享了我对这份工作的热情。

我的一位办公室同事——那个项目中我们三人共用一间办公室——比我自己意识到的更多地关注了我。后来他找到了我，并在我加入一家国际咨询公司过程中发挥了关键作用。正是在那里，我学习了 `Oracle`，这促使我写了几本书，又导致了职业生涯的转变（同样是由于热情和准备），最终使我得以住在上半岛，那是我长久以来的梦想之地。在我的职业生涯中，我曾有过几次“幸运”。然而，在我能想到的所有情况中，汤姆说的是对的。任何“运气”始终是工作、准备和个人成长与意外机遇相遇的结果。

## 社群

所有这些事件感觉就像是上辈子的事了。实际上像是两辈子前，因为自那以后我又有了两个孩子。在这条路上，我知道我不可能懂得关于成为一名 `DBA` 的所有事情，所以我开始接触并寻找其他的 `DBA`。我在网站上、论坛里和会议上找到了他们。现在到了这样一个程度，我会告诉别人这句话：“你得到的不仅仅是我，还有我的人脉网络。”

没过多久，我自己也开始做一些写作、帮助他人以及进行演讲。2009 年，我被任命为 `PASS` 董事会成员，同时也成为了一名 `SQL` MVP。对于一个 `PowerBuilder` 开发者来说，这还不错。

所以，这就是我最终如何成为一名 `DBA` 的故事。很多人，包括我自己，总是很好奇一个人是如何成为 `DBA` 的。在很多情况下，我们最终走上这条路是偶然的，但也有时候，我们只是为自己的机遇做好了准备。我的故事不一定独特，当然也绝非唯一。我询问了一些同事是否愿意分享他们的故事，他们的回复如下。

## 其他旅程

成为 `DBA` 有很多不同的途径。我绝不是唯一一个走了迂回路线的人。我们大多数人确实往往拥有一些技术经验或悟性，并且具备良好的组织或分析能力。但事实是，这些技能同样适用于许多其他职业。例如，酒店经理需要有条理、注重细节，并且能够理解和排除运行收银机的销售点软件的故障。

很少有学院或大学提供数据库管理作为专业。许多学校会提供计算机科学或某种信息系统专业，但没有专门培养 `DBA` 的专业。可能有一两门关于数据库设计或理论的课程，但关于灾难恢复、高可用性、服务器配置以及其他 `DBA` 需要知道的 everything else，可能就讲得不多了。因此，对于大多数人来说，成为 `DBA` 的道路不是一条直线，这是可以理解的。事实上，我们都是在某个时刻偶然进入了这个角色，然后在那里我们决定是否喜欢这份工作。

### 药剂师

在担任药剂师 16 年，期间协助配置过一些微软 Access 应用程序后，凯西决定离开她工作的医院，去找一份程序员的工作。她学习了一些 Visual Basic 4 和 HTML，创建了她的第一个 ASP 网站，并开始使用`SQL Server 6.5`。在完成了一个为期六个月的编程任务后，她被同一家公司聘为全职`DBA`。

她现在是一位已出版的作者和`SQL Server MVP`。

### 酒店经理

布伦特在担任酒店经理时，忍受着难以想象的糟糕工作时间，并应付着一些可能最暴躁的顾客，他偶然发现了 Quattro Pro 和电子表格。厌倦了长时间的工作，尤其是在节假日，他决定转行进入 IT 领域。工作时间略有改善，但现在他必须整天整夜携带寻呼机，所以实际上这算是倒退了一步。但他学到了很多关于服务器的知识，并最终接触到了数据库。如今他也是一位`SQL Server MVP`。

### 估算主管

我以前甚至不知道还有这样的职位名称，但蒂姆多年前正是从这里开始的。这份工作并不令人满意，一位朋友让他开始自学一些`T-SQL`。他抓住了跳槽的机会，尽管这意味着他将成为一名`Access`开发人员。雇佣他做开发人员的同一家公司决定让他转做公司的`SQL Server DBA`，这是他当时并不想要的机会。但十年后，他感到非常幸运能得到担任`DBA`的机会，他同样也是一位`SQL Server MVP`。

### MUMPS 程序员

乔纳森是一家大型化学品公司的`MUMPS`程序员，支持用`MUMPS`编写并在`DEC VAX/VMS`上运行的工业卫生和医疗系统。有一天，他被告知，公司已决定使用关系数据库作为后端存储来重写这些系统。

公司计划标准化使用`Rdb`，于是乔纳森在他的`VMS`命令行中输入了“`HELP RDB`”。他如饥似渴地阅读在线帮助文档。他钻研手册。他创建数据库进行实验。他迷上了`SQL`的强大功能，并迅速成为部门里处理数据库问题的首选人物。他想成为一名`DBA`。

他作为一名`MUMPS`程序员未能成为`DBA`。相反，他被解雇了。
