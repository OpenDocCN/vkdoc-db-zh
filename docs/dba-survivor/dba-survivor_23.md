如果您自己曾花时间构建和编写应用程序，您可以跳过本节，因为您已经知道开发者为何如此“需求旺盛”的所有原因。如果您来自服务器管理领域并转入数据库管理，那么请继续阅读，以便更好地了解“篱笆另一边”的生活。您可能会发现自己与开发者的共同点比想象中要多。

在我成为一名 DBA 之前，我做了六年多的程序员/分析师。在那段时间里，我参与过各种项目，有些项目备受瞩目，有些则随着时间的推移而湮没无闻。我熟悉这份工作的要求，并希望本节能让您也熟悉这些要求。

### 压力之下

开发者承受着压力。在某些情况下，压力可能很大。他们不仅要为现有产品编写新代码和进行增强开发，还要负责支持已经构建好的产品。支持问题可能在一天或一夜中的任何时候发生，但通常是在夜里。

**提示：** 生产环境支持可能导致开发者睡眠不足。如果您不熟悉睡眠不足的感觉，那就等着吧，因为 DBA 们也要分担这份负担。

当系统出现故障时，构建应用程序的开发者是第一批被叫来排查问题的人。而且他们并不总能拥有充裕的时间来修复问题，更别说正确地修复了。有时，尽管开发者希望妥善修复问题，但手动按几个按钮来让系统勉强运行可能更容易些。他们可能没有足够的时间，或者“正确地”做事可能不是优先事项。不要急于责怪开发者，优先级通常来自管理层。

因此，开发者最终可能会在半夜给您打电话，或者在白天到您的工位来，紧急请求您的帮助。而当他们真的出现在您的工位时，您可能会惊讶地发现，紧急情况的本质并不一定是生产问题。

不，他们可能遇到了开发服务器甚至测试服务器的问题。尽管它不是生产环境，但开发者遇到的问题确实可能是一个严重的问题。

看，开发者可能承受着在本周末前完成项目的压力，而开发服务器没有响应。或者他们的代码可能已部署到测试环境，但测试服务器没有响应，最终用户无法进行测试。

作为 DBA，您经常会发现自己同时处理十几个需要您关注的不同任务。但开发者的关注点通常更窄，一次只专注于一两个项目。因此，如果开发服务器出现问题，您可能会听到开发者说：“在您解决这个问题之前，我什么也做不了。”尽管这听起来可能令人惊讶，但却是千真万确的。

**注意：** 无论压力多大，或者睡眠多么不足，以不专业的方式对待同事都是不合适的。大吼大叫、提出浮夸的要求以及言语辱骂是永远不可接受的。

开发服务器本质上就是生产服务器，因为开发工作总是在向前推进。生产服务器是稳定的实体；代码应该很少更改，系统应该是集成且正常运行的，生产环境上应该极少发生变化。但开发总是在向前推进，而开发者是推动的主力。因此，您需要像响应任何生产请求一样，积极响应他们的开发请求。

### 十二个愤怒的开发者，或更多

作者：乔纳森·詹尼克

有一次，我正在排查一个仍在开发中的、运行缓慢的报表。我决定更改一个表参数，以启用该报表底层查询的并行处理。效果不错。报表完成时间缩短到原来的三分之二左右。我很满意，并将我在自己“沙盒”中做的同样更改应用到了客户的开发系统上。

然后电话就响了。

您看，我修改的那个表是整个系统的基础。它几乎是每个屏幕和报表的核心。更重要的是，我刚刚做的表参数更改导致涉及该表的每个查询都从使用 Oracle 的基于规则的优化器切换到了使用基于成本的优化器。更改是实时发生的，突然间系统中几乎所有的查询计划都被彻底搞乱了。

这只是一个开发系统，对吧？

是的，但客户的开发团队正在长时间加班、熬夜，以使新系统准备好投入生产。他们并不感到高兴。我的一个小小的更改，让一个由十几人组成的、过度劳累的团队完全停滞了。

### 业务需求

无论您从事什么业务，事情都会变化。这些变化将反映在公司维护和构建的系统中。而这些变化是由开发者构建和部署的。您应该只按吩咐去做，当有人要求成为系统管理员的固定服务器角色成员时，不要问任何问题。

**提示：** 前面段落中有一句话是假的。您能找出是哪一句吗？是最后一句——您应该始终询问为什么有人需要这样的访问权限。

您的最终用户要求很高。他们可能是坐在离您一排之遥的人，也可能是在另一个国家仓库地板上的人。如果他们的系统坏了，他们会通知您。通常这些投诉会先传到开发者那里。这意味着关于您系统发生的任何事情，您很可能是最后一个知道的。换句话说，不要为这个周末做计划了，因为某个地方的某人很可能会需要您帮忙处理某些事情。

这些业务需求将给开发者带来额外的压力，这反过来又会给您带来额外的压力。压力总是自上而下传递的。

### SQL 知识

`SQL Server`内部有`很多`功能。您是否注意到有多少组件包含`SQL`这几个字母？这里是一个简短的列表：

*   `SQL Server`
*   `SQL Server Management Studio` (`SSMS`)
*   `SQL Server Reporting Services` (`SSRS`)
*   `SQL Server Integration Services` (`SSIS`)
*   `SQL Server Analysis Services` (`SSAS`)
*   `SQL Server Notification Services` (`SSNS`)
*   `SQL Azure`



只要文件名里带有`SQL`字母，你就放心吧，迟早会有人来找你问问题。而当你回答说“不，我不会把`SSNS`的通知打包到`SSIS`里，也不会在`SSRS`里建报表，更不会把东西推到`SQL Azure`上”，你只会得到一个茫然的眼神，外加一句：“但你不是`DBA`吗？”

**提示：** 记住，没有人是万事通。

答案是肯定的。没错，我们是`DBA`，但这并不代表我们了解每一个带有`SQL`字母的产品的所有细节。这就像我问开发人员：“什么叫需要我从昨天的生产数据库恢复到测试环境，就为了拿个存储过程？你们源代码管理里没存这份存储过程吗？”他们也会是同样的反应。

### 关于 SQL 的提问
*乔纳森·杰尼克（Jonathan Gennick）*

几年前，有人找我教一门关于“`SQL`”的课。我现在记不清对方是读过我的书和文章的人，还是仅仅听说我`SQL`玩得不错的人。但我很快发现，对方想要的并不是教`SQL`这门查询语言，而是教`SQL Server`这个数据库管理系统。当然，这是两码事。相信我。我对`SQL`还算得心应手，但你肯定不想看我教`SQL Server`的课。

既然我们已经明确`SQL Server`功能繁多，你就得知道，开发人员会问你一大堆关于你从未用过的各种组件的问题。这很正常——别慌——这说明开发人员工作很尽职，正在研究完成特定任务的不同方法。

凭借`SQL Server`提供的所有功能，开发人员不仅会让你帮忙让某些东西跑起来，还会期望你能解释为什么它会以某种方式工作。很可能`SQL Server`的某个组件正以一种违背所有人预期的方式运行。当然，他们会期望你为这种行为提供解释；最好能快速找到答案，并且最好附带一个解决方案，不需要开发人员推翻几个月的编码工作。如果真的需要推翻重来，那同一个开发人员就会要求你拿出锤子，硬把方钉子敲进圆孔里。

如果出现这种情况，你应该尽力为开发人员提供几个替代解决方案。最糟糕的情况莫过于某个权宜之计的解决方案被部署到了生产环境。我见过太多次设计糟糕的系统被部署上线，仅仅因为开发人员不愿意承认，他们本可以用一种不会导致内存缓慢泄漏、每周都需要重启服务器的方式来完成工作。

通常发生这种情况时，他们会尽力耸耸肩，告诉你去找微软要补丁来修复问题。毕竟，他们没时间把事情做对，因为他们正承受着完成其他项目的压力。

## 服务级别协议

*服务级别协议（`SLA`）* 对于任何`DBA`来说都是必备的。如果你的公司已经定义了`SLA`，你必须花时间去审阅它们。了解已定义的`SLA`将有助于你确定任务的优先级。例如，如果你有两个任务要执行，一个必须在四小时内完成，另一个的期限是两天，那么你很容易决定先处理哪个。

**提示：** `SLA`有很多好处，最显著的是它们能帮助你确定当前任务的优先级。

如果你的公司没有定义明确的`SLA`，那么你必须与你的经理坐下来开始讨论。尽你所能为最常收到的请求组定义`SLA`，并讨论不同级别的环境。例如，如果最终用户需要重置生产环境的密码，这应该优先于测试环境的密码重置。很难对你收到的每一项任务都划定一条硬性界限，所以最好将请求归入特定类别。

一些类别示例可能包括：
* 密码重置
* 数据库恢复
* 数据库备份
* 创建服务器登录名
* 创建新数据库
* 性能调优

对于这些类别中的每一个，你还应考虑环境的级别。所以生产环境的请求优先于其他一切，对吧？嗯，也许吧。当然，这也取决于你的客户。

### 做一个好客户

虽然你有责任为最终用户提供优质的客户服务，但做一个好客户也有其道理。作为一名`DBA`，你从事的是服务行业；你为所有客户提供服务。而许多客户也在为他人提供服务。自然，你会期望在你的服务级别上能有一些职业礼遇。

当然，期望每个人都有这种礼遇是错误的。

即使定义了`SLA`，也无法阻止人们要求所有事情都得“昨天就完成”。开发人员会希望他们的开发服务器请求立即得到满足，根本不在乎你可能正在处理的任何生产问题，或者你需要查看昨晚的仪表板报告，或者你正在尝试进行一些必要的维护。

开发人员通常想要行动；他们想看到你在做点什么，最好是此刻就做他们需要的事情。否则，他们就会认为，当你没有专门为他们做事时，你就是闲着。当开发人员抱怨某件事耗时太长时，你就该拿实际耗时与该任务定义的`SLA`对比一下了。

**提示：** 下次你在商店或餐厅与人互动时，请记住这一点。如果你因为服务水准不合心意而感到不耐烦，在表达不满之前，先问问自己是否先做了一个好客户。

### 沟通是关键

我观察到，许多围绕任务及其完成情况的挫败感，归根结底只是一个简单的沟通问题。我数不清有多少次收到客户的电子邮件，询问某个任务的进度更新，而`DBA`明明知道该任务还需要一个多小时才能完成。那么，为什么客户会期望更早完成呢？*因为他们根本不知道这个请求应该花多长时间！*

那么如何避免这种情况呢？当然是通过沟通。让我们考虑同一个请求的两种场景。
