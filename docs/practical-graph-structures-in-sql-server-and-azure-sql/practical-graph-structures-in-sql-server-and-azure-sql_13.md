最后，在 6-3 节的表中，你可以看到，即使使用任何一种方法将 29.8 万行数据输出到临时表中并进行格式化以供查看，也并非一项繁重的任务，特别是考虑到所使用的机器（我从未用完分配给 SQL Server 的所有 RAM，因此这基本上是一个 CPU 问题，而且由于 `MAXDOP = 1`，我只使用了单个 CPU）。路径法在处理过程中已经包含了一些按我期望格式化的数据，因此通常速度更快。

我执行的最后一项测试是验证：如果我有 `NodeX` 和 `NodeY`，`NodeX` 是否是 `NodeY` 的子节点？对于所有这些测试，使用每种算法所花费的时间都可以忽略不计，甚至在六位精度下通常都显示不出来。因此，我没有将其作为表格输出，但关键在于，当你处理数据的特定切片时，无论数据集有多大，每种方法都可以非常快。

就像使用关系引擎的几乎任何查询一样，如果可以使用索引来限制处理的数据量，即使数据量巨大，也能快速工作。SQL Graph 查询也是如此。只有当涉及大量数据时，处理时间才会增加（并且并行性成为问题，至少在 SQL Server 2022 RTM 版本中是如此）。

## 总结

在这篇关于树结构的第二章中，我希望不仅能让你了解一些可用于树结构的技术，还能让你感受到，即使在一台绝非服务器级别的非常小的计算机上，也可以处理多大规模的数据集。我甚至没有深入探讨硬件利用率和带宽问题，因此当你开始在真实系统中使用这些工具并配备服务器级硬件时，效果可能因人而异。

无论如何，一个拥有近 30 万个节点的树结构绝不算小（当然也并非大得惊人），而我们能够相当快速地处理所有行，无论是使用原生的 SQL Graph 功能，还是使用替代的路径法以及构建辅助表。我承认，花费近 5 分钟来构建辅助表似乎时间很长，但根据你的需求，这可能是值得的，特别是如果你使用图进行分析目的。而且，如果你能承受重新生成辅助表的延迟，可以将它们放入暂存表中然后交换出去，而不会被用户真正注意到。
