# 1. 索引基础知识

本书的目标是通过使用索引来帮助提升 SQL 数据库性能。为此，将首先介绍索引的用途和用法。这将引出对索引中数据存储方式的讨论，重点介绍聚集索引、非聚集索引和列存储索引。我们还将探讨堆表。索引是任何数据库的关键组成部分，无论其存储在本地还是 Azure（或其他云服务）中。

本章为理解索引的逻辑设计奠定了基础。



## 为何要建立索引？

任何企业最重要的资产就是其数据。数据库的存在就是为了存储这些数据，而提供数据的一个关键环节是高效地交付数据。能够快速访问数据可以提升企业从中获取的价值。确保数据访问速度的方式就是使用有效的索引。

索引为数据赋予了顺序，而这种顺序允许数据被高效地排序、筛选和聚合。这减少了执行这些操作所需的时间和计算资源。

与索引数据相对的情况是数据存储在无序的堆中。堆就是没有索引的表。没有顺序，`SQL Server`就必须扫描整张表来返回其中的任何数据。对于较大的表，这个过程代价高昂得令人望而却步。例如，要从一个未建立索引、大小为`1 TB`的表中返回一行数据，就需要读取整整一个太字节的数据来找到被查询的那一行。

为了说明这一点，考虑一个常用来描述索引的类比——图书馆。图书馆有许多书架的书。在图书馆中，最常见的任务是找书。大多数访客要么想找一本特定的书，要么想找一本与某个非常有限的主题相关的书。

一个没有图书排序系统的图书馆对大多数访客来说毫无用处。手工一本一本地搜寻数万本书的想法将构成荒谬的工作量。图书馆使用`杜威十进分类法`系统按主题和类型来组织书籍，使用一套详细的数字来标记它们。例如，关于科学的书籍可以在`500–599`的范围内找到。数学书籍则限于`510–519`的范围，而几何学书籍具体在`516`中找到。

这个分类系统充当了图书馆的索引，使得寻找一本好的几何学书籍的访客能够快速有效地定位它。无需在图书馆里徘徊寻找关于几何学的书，也无需翻阅无关的书籍来找到它们。索引基于主题建立，并符合图书馆访客最常见的搜索模式。

当搜索的不是主题而是特定作者时会发生什么？如果需要找到关于埃德加·F·科德的书呢？一个有根据的猜测是这些书会在与数据库或计算相关的部分。然而，唯一能确定的方法就是手动搜索大量书籍，最终找到正确的那几本。

图书馆使用卡片目录或计算机来解决这个问题。这些系统允许按其他标准搜索书籍，例如作者、出版日期、语言或标题。在前面的例子中，可以搜索标题中包含"Edgar Codd"字样的书籍。这比在科学、数学和计算类别的书堆中搜寻并希望成功要高效得多。这个系统提供了`杜威十进系统`之外的二级索引，从而允许访客使用各种不同的标准来搜索书籍。

这些系统远比在图书馆里徘徊、浏览无穷无尽的书籍高效得多。反过来，一个足够小的图书馆不需要卡片目录、`杜威十进系统`或计算机。一个“小小免费图书馆”可能只包含`10`或`20`本书。在这种情况下，浏览每本书的标题花费的时间很少，比尝试使用一个复杂的系统来组织它们更有效率。

这个类比旨在不立即深入数据库结构的情况下介绍索引的概念。随着对索引的更详细讨论，图书馆的例子将被用作一种方式，将本书中更技术性的细节锚定在一个植根于日常生活的更简单的例子上。

## 主要索引类型

有多种方式来描述索引及其功能。为了简化关于索引的讨论，它们将被分为三类，本节将对此进行讨论。堆表和聚集索引描述了底层表结构中数据的存储方式。非聚集索引是建立在堆或聚集索引之上的二级索引，它按其他列组织数据，然后引用聚集索引（或堆）以检索额外信息。下面简要描述了每种索引类型及其意义。

## 堆表

正如在图书馆类比中提到的，在“小小免费图书馆”中，可用的书籍经常变化；通常只有一两排短书架的书。在这些情况下，拥有者不会花时间在`杜威十进系统`下整理书籍。相反，书籍在获取时就被放置在书架上。在这种情况下，图书馆中书籍的存储没有顺序。默认情况下，这就是`SQL Server`在表中存储数据的方式。没有明确定义的顺序系统，数据会存储在一个称为`堆`的无序结构中。

在`堆`中，添加的第一行是表中的第一条记录，第二行是表中的第二条记录，第三行是表中的第三条记录，依此类推。`堆`是一个没有在数据上定义显式顺序的表。

当创建表时没有指定任何聚集索引，其存储结构就称为`堆`。行按照添加的顺序插入到表中。需要通过扫描表中的所有行来读取行，以找到满足查询所需的行。

可以向表添加聚集索引以强制数据排序并移除其作为`堆`的状态。内存优化表不存储为堆，将在第 8 章中更详细地讨论。

> 注意
>
> 许多人不认为堆是索引。在本文的讨论背景下，它们将被视为索引，因为它们有助于确定数据的位置以及查询将如何访问数据。

### 聚集索引

在图书馆类比中，`杜威十进系统`被用来为图书馆中的所有书籍分配一个编号。这个编号允许对书籍进行排序，并确定一本书在图书馆中的位置。书的**主题**，而非其添加时间，决定了书的位置。这个系统定义了书籍在图书馆中的物理存储方式，并且是查找书籍的主要方法。在表中，提供此功能的索引称为`聚集索引`。

在`聚集索引`中，选择一个或多个列作为索引的`键列`。`键列`用于排序并确定数据在表中的存储位置。图书馆根据书籍的`杜威十进`编号将书放在书架上，而`聚集索引`则根据索引`键列`的顺序来确定表中行的位置。

用作`聚集索引`的`键列`是根据访问表记录最频繁的方法来选择的。在一个包含州和省的表中，查找表中行的最常见方法很可能是通过其缩写。在该场景中，使用州/省缩写作为`聚集索引`的`键列`将是一个明智的选择。通常，表中的主键列也将用作`聚集索引`的`键列`。这不是必需的，偶尔会有理由为主键和聚集索引使用不同的列。

与堆表一样，聚集索引决定了数据在表中的存储位置。在`聚集索引`中，非键列与`键列`一起存储。这相当于`聚集索引`决定了表本身的布局。

因为它定义了表的基本组织结构，所以任何表上一次只能有一个`聚集索引`。



### 非聚集索引

在图书馆的类比中，杜威十进制分类法是聚集索引，它根据每本书的主题为书架内的书籍定义了顺序。尽管该系统很有用，但访客经常希望根据其他标准（如作者、书名或年龄层）来搜索书籍。如果不知道书籍的主题，那么杜威十进制分类法就无法帮助定位它。

`图书馆`通过计算机和卡片目录来解决这个问题。这些提供了一个可以使用其他常见细节（如作者或书名）对书籍进行交叉引用的地方。这些解决方案类似于数据库中的`非聚集索引`。

注意

由于堆不提供数据排序，`非聚集索引`通常基于`聚集索引`创建，而不是在堆上创建。

`非聚集索引`是在现有`聚集索引`（或堆）之上创建的二级索引。在`非聚集索引`中，列根据其值被选定并排序。这些列包含对其所引用数据在堆或`聚集索引`中位置的引用。这类似于图书馆中卡片目录的工作原理。书架上的书籍顺序是静态的，但会创建额外的快捷方式，以便能够使用其他搜索标准快速定位它们。

例如，一个包含州和省并按州名排序的表，可能还包含其国家、语言或其他详细信息。根据这些其他标准搜索此表有许多合理的理由，而`非聚集索引`允许基于除州缩写或名称之外的其他详细信息快速搜索长长的地区列表。

与`聚集索引`和堆不同，一个表可以包含许多`非聚集索引`。虽然在技术上一个表最多可以定义 999 个`非聚集索引`，但要构造一个包含数百个索引的表的实际示例会很有挑战性。多个`非聚集索引`允许以多种方式搜索数据，这是`聚集索引`之外的补充。虽然一个表可以包含许多索引，但每个索引都应服务于一个合理的用例。索引需要存储和维护，因此创建过多可能会给`SQL Server`带来不必要的负担。

## 其他索引类型

在`聚集索引`、`非聚集索引`和堆的范畴内，存在着多种索引类型，每种类型都有助于搜索不同类型的数据。到目前为止，图书馆卡片目录的例子侧重于访客搜索书籍的体验，但还有许多其他场景需要不同的组织结构来解决。

考虑以下图书馆信息请求：

*   报告所有图书馆中按主题分类的书籍数量。
*   在一个图书馆的所有书籍中搜索特定文本。
*   绘制一个图书馆所有书籍的地理来源地图。

虽然非常有用，但卡片目录无法解决这些问题。`SQL Server`中也是如此，因此存在各种索引来满足不同的数据访问需求。以下是不同索引类型的简要概述。本书后续章节将详细介绍它们的功能和用法。

### 列存储索引

与图书馆书籍相关的细节，无论是存储在卡片目录还是计算机中，都是存储在一起的。单个条目包含了关于一本书你可能想了解的所有信息。当目标是查找单本书或一小部分书时，这是有效的。

当试图在更高层次上理解书籍时，这种系统就不再高效了。例如，考虑一位图书管理员想要统计图书馆中每个主题有多少本书。一本一本地查找书籍会很慢。即使是自动化过程也会很慢，因为它需要遍历所有书籍的所有数据以获取它们的主题并增加总计数。

解决这个挑战需要一种替代的数据存储方式，该方式维护一个主题列表，其中包含一些基本统计信息和指向每个主题下所有书籍的链接。这个解决方案类似于`列存储索引`，它创建了一种按列而非按行组织数据的结构。

在`列存储索引`中，行被分组在一起，每个列的数据被压缩并分开存储。在这种结构中，需要跨许多行获取少数几列数据的查询可以高效执行，因为不再需要读取每行的所有细节。`列存储索引`作为分析数据请求的解决方案而存在，有助于报告、分析和数据科学。

在本书的剩余部分，索引将在适用时分类为`行存储`或`列存储`。`行存储索引`将一行的所有细节顺序地存储在一起。`行存储索引`非常适合事务性数据处理，这在软件应用程序的日常使用中很常见。`列存储索引`是为分析性数据处理构建的，这通常出现在数据仓库、报告以及涉及数据科学和机器学习的项目中。

此类索引的好处是只需读取查询所需的列和行。在数据仓库场景中，通常只需要索引中少于 15% 的列来生成查询结果。^(¹)

`列存储索引`可以是`聚集的`或`非聚集的`。`聚集列存储索引`是无序的，包含表中的每一列。在这种情况下，`列存储索引`成为表的基本存储结构。`非聚集列存储索引`针对表中的特定列，并将允许对该索引中的任何列进行分析。它们按照底层`聚集行存储索引`的顺序进行排序。

`列存储索引`往往能非常有效地压缩。这是因为单个列的值会比跨多个不同列的值重复得多。因此，`列存储压缩`通常会看到远高于标准行或页压缩的压缩率。消耗更少的存储意味着执行查询时需要的内存更少，最终带来更快的结果。

由于其结构，`列存储索引`为数据仓库和其他分析解决方案提供了重要价值。`列存储索引`将在本书后面更详细地讨论。



## JSON 和 XML 索引

假设需要在图书馆的目录中搜索所有书籍。目录提供了书籍的层级视图。书籍有章节来概述主要部分，然后是子章节标题，为每个章节提供额外的细节。这种关系模型类似于 XML 文档的设计方式；存在节点以及它们之间的关系，这些关系定义了信息的结构。

与之前卡片目录的类比类似，遍历图书馆中的每一本书来查找包含关于埃德加·科德章节的书是低效的。同样，要找到埃德加·科德撰写的章节或包含他信息的章节，如果没有更好的方法，将是一项艰巨的任务。

解决这个问题的一种方法是列出图书馆中的每一本书，并为每本书列出其所有章节。每个章节条目中可以包含与其内容相关的标签或关键词。这提供了与卡片目录相同的优点，但针对的是一些不太标准和不太结构化的信息。一本书有多少章节，或者每个章节包含多少主题，这都不重要；这个章节列表会根据每本书实际存在的章节进行列举。

在数据库中，这正是 `XML 索引` 和对 JavaScript 对象表示法（JSON）进行索引可以实现的功能。

对于 XML 文档中的每个节点，都会在 XML 索引中创建一个条目。这些信息被持久化存储在 SQL Server 可以用来确定 XML 文档是否包含正在查询的数据的内部表中。对于 JSON 也是类似的，要索引的值通过计算列上的索引被实体化。

创建和维护 `XML 索引` 的成本可能相当高。每次更新索引时，它都需要将 XML 文档的所有节点分解（shred）到 XML 索引中。XML 文档越大，这个过程的成本就越高。然而，如果 XML 列中的数据将被频繁查询，那么创建和维护 XML 索引的成本将被在运行时无需分解所有 XML 文档所带来的好处所抵消。

## 空间索引

每个图书馆都有地图。有些地图覆盖海洋；另一些则覆盖大陆、国家、州或城市。图书馆中可以找到许多不同类型的地图，其中可能有多种类型针对同一区域。所有地图都存在两个基本挑战：

*   哪些地图重叠或包含相同的位置或信息？

*   哪些书籍在某个给定位置出版或发生？

如果没有专门构建的系统来协助，要检索所有包含明尼阿波利斯市的地图，或检索所有在明尼阿波利斯市方圆 25 英里内出版的书籍列表，将是难以回答的问题。

传统上，数据库并非为回答此类问题而构建。表代表实体，不同的实体可以通过连接（join）关联，但这些关系本质上是一维的。在物理世界中，数据通常存在于多个维度。地图是二维的，建筑物和楼层平面图是三维的。为了解决这个问题，SQL Server 提供了 `空间索引` 功能。

`空间索引` 将提供的空间信息解析为数据的四层表示。这种表示允许 SQL Server 绘制空间信息，包括几何（geometry）和地理（geography）。然后，这个数据结构可以用来确定行之间在哪里重叠以及一个点与另一个点的接近程度。

`空间索引` 存在一些限制。主要的一个是空间索引必须在具有主键的表上创建。没有主键，空间索引的创建将不会成功。此外，创建空间索引时，它们被限制使用并行处理；并且一次只能在一个表上构建一个空间索引，这影响了它们的创建速度。另外，空间索引不能在索引视图上使用。这些以及其他限制将在第 7 章中介绍。

与 `XML 索引` 类似，`空间索引` 有与其大小相关的前期和维护成本。好处在于，当需要查询空间数据时，空间索引的价值可以迅速显现。与所有索引一样，它们的主要目的是加快读取数据的速度，而成本在于创建和维护索引。`空间索引` 将在第 7 章中更深入地讨论。

## 哈希和范围索引

随着书籍进入图书馆，有时归还书籍的频率超过了将其放回书架的速度。整理书籍并将其放到指定位置需要时间。在这些时候，通常会有一位图书管理员记录归还的书籍。对于这些书籍，图书管理员通常能记住哪些书在归还队列中的位置，并且无需使用卡片目录就能拿到你想要的书。这本质上就是内存优化表使用 `哈希和范围索引` 所做的事情。唯一的区别是，通过哈希和范围索引，数百万行（或书籍）可以保留在内存中，而无需依赖基于磁盘的结构来支持它们。

`哈希索引` 允许内存优化表提供表中数据的精确点查询。用图书馆类比，索引（即图书管理员）可以确切记住每次需要时书籍在表和索引中的位置。

另一方面，`范围索引` 为内存优化表提供了高效识别项目范围的能力。例如，如果索引（或图书管理员）需要所有在上午 8:00 到 12:00 之间归还的书籍，索引将能够高效地扫描行，而不是逐行访问。

`哈希和范围索引` 只允许在内存优化表上使用。关于记住书籍位置的说法，正是哈希和范围索引与其他索引类型的不同之处。在磁盘和内存之间，聚集索引的结构相对不变。而对于哈希和范围索引，其结构是专门为快速内存访问而设计的，并仅利用磁盘来支持事务一致性以及在数据库上线时在内存中重建索引的能力。

## 全文搜索

最后一个要考虑的场景是在书籍中查找特定术语。卡片目录在提供基本的书籍信息（如作者、标题或主题）方面做得很好。然而，书籍的主题并不是你可能想用来搜索书籍的唯一关键词。许多书籍的末尾都有关键词索引，以帮助你在书中查找其他主题。当这本书完成后，会有一个索引，它将提供查找关键词并识别它出现在哪些页面（如果书中有的话）的能力。

想象一个图书馆，每本书都有一个关键词索引。如果所有这些关键词都被添加到它们自己的卡片目录中，那么访问者就能找到图书馆中包含特定关键词的每一本书。此外，他们还能准确知道在每本书中哪里引用了这个关键词。这是 `全文索引` 是什么以及如何工作的一个概述，根本区别在于 SQL Server 中的全文索引将索引和搜索大多数单词，而书中的索引只会索引一小部分对读者最相关的关键词。



## 索引的变体

至此，我们已经讨论了不同类型的索引以及它们如何帮助搜索不同类型的数据。创建索引的方式多种多样，每种方式提供的功能略有不同。有些索引不仅能对数据进行索引，还允许在底层数据上直接强制执行业务规则，同时可能提升索引性能。

### 主键

在图书馆的类比中，杜威十进分类法是一种按主题组织书籍的宝贵方法，可以快速高效地找到特定主题的书籍。然而，杜威十进分类号并非一本书所独有，许多书可能共享同一个编号。杜威十进分类号可以看作是图书馆中书籍的聚集索引。同样，书名、作者、出版日期等细节在许多书籍中也会重复出现。

为了有意义的管理图书馆，必须有一个唯一的编号来标识每一本书，且永不重复。在图书馆中，书籍通过其`ISBN`（国际标准书号）来唯一标识，使得没有两本书共享相同的`ISBN`。

在数据库中，主键唯一标识表中的一行。表中不允许有两行拥有相同的主键值。通常，主键会创建在单个列上，但在需要时也可以由多个列组成。

因为主键是为表中每一行明确定义的唯一值，所以它不能为`NULL`。此外，一张表一次只能定义一个主键。为了维持唯一性，主键必须与索引一同构建，该索引可以是聚集的或非聚集的。主键通常建立在表的聚集索引上，但在罕见的使用场景中，可能需要将主键和聚集索引放在不同的列上。更多关于此的信息包含在第 11 章中。

### 唯一索引

可能有不止一个列或列组合可以用来唯一标识表中的一行。当需要定义多种唯一标识行的方式时，可以使用唯一索引。

唯一索引与主键是不同的实体，可以定义为聚集索引或非聚集索引。一张表可以包含多个唯一索引，但它仍然只能有一个聚集索引和一个主键。

与主键类似，唯一索引受到约束，其内部值不能重复。它提供了一种唯一标识表中行的机制，可以创建在单个列或多个列上。

主键和唯一索引的一个显著区别是，唯一索引允许`NULL`值，而主键必须定义在`NOT NULL`列上。`NULL`值被视为一个离散值，因此不能像非`NULL`值那样重复出现。

### 包含列

假设有一个需求：在图书馆中找到所有 Douglas Adams 写的书，并查看每本书的页数。最简单的解决方案是逐个在卡片目录中查找每本书，定位到每本书，然后检查其页数。虽然简单，但这很繁琐，并且依赖于一个高度手动的过程来获取结果。

理想的解决方案要么是在卡片目录中包含页数信息，要么有一个单独的包含页数的书籍列表。这两种方案都能提供一个解决方案，使得无需实际定位任何书籍就能快速检索页数。在索引方面，通过`包含列`来添加索引列之外的信息。

当构建非聚集索引时，可以选择将包含列添加到索引中。这些列作为未排序的数据与索引中的键列一起存储。包含列不能包含任何已在索引初始排序列列表中使用的列。

在查询方面，允许用户通过包含列来查找排序列之外的信息。如果查询所需的一切信息都能在键列和包含列中找到，那么查询就不需要访问表的堆或聚集索引来返回结果。与卡片目录的例子类似，当每次请求类似的列列表时，包含列可以显著提高查询性能。

### 分区索引

包含大量数据的书籍可能会变得相当庞大。想想字典或莎士比亚全集。这些都是异常庞大的书籍，出版成单卷本是不切实际的。一本 20,000 页的书在物理上根本无法出版。此外，一个人阅读（或者仅仅是查阅）如此大部头的书也将是相当有挑战性的。书籍可能会大到将它们包含在单一卷中变得不切实际。最好的例子就是百科全书。

百科全书很少只包含在一本书中。书的尺寸和装订的宽度会超出任何人的处理能力。另外，在百科全书中找到所有以字母*S*开头的主题所需的时间也大大减少，因为你可以直接去 S 卷，而不需要翻阅一本巨大的书来查找它们的起始位置。

这个问题不仅限于书籍。表及其索引也可能达到某个程度，使得其大小导致使用合理的时间和资源来继续维护它们变得困难。同样地，如果一个表有数百万或数十亿行，能够扫描表的有限部分而非整个表，可以提供显著的性能提升。为了解决表上的这个问题，索引可以进行分区。

`分区`可以应用于聚集索引和非聚集索引。它允许索引沿着函数提供的值进行拆分。通过这样做，索引中的数据在物理上被分离成多个分区，而索引本身仍然是一个单一的逻辑对象。

### 过滤索引

默认情况下，非聚集索引包含其关联表中每一行的一条记录。在大多数情况下，这是理想的，它为索引提供了协助提高列中任何值的选择性的机会。为了本次讨论的目的，选择性是衡量表中满足给定过滤条件的行数的一个指标。

在某些情况下，将表中所有记录都包含在索引中并不是最佳选择。例如，最常查询的值集合可能只代表表中少数行。在这种情况下，限制索引中的行数将减少查询需要执行的工作量，从而提高查询性能。另一种情况是某个值的选择性相对于表中的行数较低。这可能是活动状态或已发货的布尔值；在这些值上建立索引不会显著提高性能，但将其过滤到仅这些记录将为查询改进提供显著的机会。

为了协助这些场景，可以对非聚集索引进行过滤以减少其包含的记录数量。在构建索引时，可以定义一个基于简单比较的条件来包含或排除记录，从而减小索引的大小。

除了上述的性能改进外，使用过滤索引还有其他好处。第一个改进是降低存储成本。由于过滤索引包含的行更少（因为过滤），索引中的数据量会更少，从而需要更少的存储空间。更少的存储使用也意味着当索引读入内存时消耗的内存更少。另一个好处是降低维护成本。较小的索引需要更少的时间和资源来进行重组和重建。



## 压缩与索引

如今的图书馆藏书量巨大。随着藏书数量的增加，会达到一个临界点，使得现有人员和资源难以管理图书馆。因此，图书馆找到了多种方法来存储书籍或其中的信息，以实现更佳的管理，而无需增加维持图书馆所需的资源。例如，书籍可以以电子方式存储。这既减少了物理书籍在图书馆中占用的空间，又简化了借阅/归还流程。

同样，索引在变得过大时，也会达到难以管理的程度。此外，访问记录所需的时间可能会增加到超出可接受的水平。SQL Server 中提供两种类型的压缩：`行级`压缩和`页级`压缩。

使用`行级`压缩时，索引在行级别压缩每条记录。启用`行级`压缩后，会对每条记录进行若干更改。行的元数据以另一种格式存储，从而减少了每列存储的信息量。记录的主要更改包括：数值数据从固定长度变为可变长度，定长字符串数据类型末尾的空格不再存储。另一项更改是，Null 或零值不需要任何存储空间。这会产生一些管理压缩数据的开销。因此，盲目地将`行级`压缩应用于表并非最佳决策，必须谨慎选择那些压缩效果良好的行作为目标。

`页级`压缩则对存储在一页上的所有行整体应用压缩，而不是单独压缩每一行。启用`页级`压缩后，会识别列中字符串值之间的相似性并进行压缩。这一点将在第 13 章详细讨论。

无论是`行级`还是`页级`压缩，都有一些重要的细节需要考虑。压缩记录会占用额外的 CPU 时间。虽然行占用的存储空间和内存（读取时）会减少，但 CPU 是管理压缩任务的主要资源，然后数据才能被存储。根据待压缩表的内容，压缩的效果会有所不同。

## 索引数据定义语言

与 SQL Server 中丰富多样的索引类型和变体类似，用于构建索引的数据定义语言 (`DDL`) 也很丰富。本节将探讨用于构建索引的 `DDL`。首先，将讨论 `CREATE` 语句及其选项，并结合本章前面讨论的概念。

为简洁起见，将不探讨索引 `DDL` 的向后兼容性功能。关于每个 SQL Server 版本中可用语法的详细信息，在 [Microsoft 的 Create Index 文档](https://docs.microsoft.com/en-us/sql/t-sql/statements/create-index-transact-sql)中有详尽记载。本书使用与 SQL Server 2022 兼容的语法编写。虽然包含的大多数 Transact-SQL (`T-SQL`) 在任何现代版本的 SQL Server 中都能运行，但在尝试将新代码发布到生产环境之前，验证和测试这些细节是有价值的。

更具体的索引实现，如 `XML`、`空间`和`全文`索引，将在本书后面的章节中探讨。

### 创建索引

索引要在数据库中存在，首先必须被创建。这是通过如代码清单 1-1 所示的 `CREATE INDEX` 语法完成的。如语法所示，前面讨论的大多数索引类型和变体都可以通过基本语法实现。

```sql
[ INCLUDE ( column_name [ ,...n ] ) ]
[ WHERE <filter_predicate> ]
[ WITH ( <relational_index_option> [ ,...n ] ) ]
[ ON { partition_scheme_name ( column_name )
| filegroup_name
| default
}
]
[ FILESTREAM_ON { filestream_filegroup_name | partition_scheme_name | "NULL" } ]
[ ; ]

```
`CLUSTERED` 和 `NONCLUSTERED` 索引之间的选择决定了索引将构建为这两种基本类型中的哪一种。排除其中任何一种类型都会默认索引为 `非聚集` 索引。
`UNIQUE` 关键字决定索引值是否必须唯一。在 `CREATE INDEX` 语法中包含它将使索引具有唯一性。创建主键索引的语法将在本章后面部分介绍。
`<object>` 选项决定索引将基于哪个基础对象构建。语法允许在表或视图上创建索引。如果需要，对象的指定可以包括数据库名称和模式名称。
指定索引对象后，索引的排序列将按照它们将被排序和使用的顺序列出。这些列被称为`键列`。每列在索引中只能出现一次。默认情况下，列在索引中按升序排序，但也可以根据需要指定降序。索引最多可以包含 32 列作为索引键的一部分，总大小不得超过 1,700 字节。在 SQL Server 2016 之前，限制为 16 列和 900 字节。

作为选项，可以在任何`非聚集`索引上指定`包含`列，它们被添加在索引的键列之后。`包含`列不进行排序；因此，在向索引添加`包含`列时无法指定排序顺序。在键列和非键列之间，索引最多可以包含 1,023 列。键列的大小限制不影响`包含`列。

如果要筛选索引，则接下来指定筛选定义。筛选条件使用 `WHERE` 子句添加到索引中。`WHERE` 子句可以使用以下任何比较运算符：`IS`、`IS NOT`、`=`、`<>`、`!=`、`>`、`>=`、`!>`、`<`、`<=` 和 `!<`。筛选索引不能使用对`计算`列、用户定义类型 (`UDT`) 列、`空间`数据类型列或 `HierarchyID` 数据类型列的比较。

创建索引时有许多可用选项。在代码清单 1-1 中，有一个用于添加索引选项的部分，由标签 `<relational_index_option>` 标记。这些索引选项控制索引的创建方式及其在各种场景中的运行方式。代码清单 1-2 提供了 `CREATE INDEX` 可用选项的 `DDL`。

### CREATE INDEX 语法选项

每个选项都允许对索引创建过程进行不同级别的控制。标有下划线的选项表示其默认值。在处理索引时，除非有特定用例需要更改它们，否则使用默认选项是值得的。表 1-1 列出了 `CREATE INDEX` 的可用选项。在后续章节中，将提供示例及其应用策略。关于 `CREATE INDEX` 语法的更多信息及其使用示例，可参考 SQL Server 的 SQL 文档。

**表 1-1: CREATE INDEX 语法选项**

| 选项名称 | 描述 |
| --- | --- |
| `FILLFACTOR` | 定义索引创建时每个数据页中应保留的空闲空间量。此选项仅在索引创建或重建时应用。 |
| `PAD_INDEX` | 指定是否应将索引的 `FILLFACTOR` 应用于索引的非叶级数据页。当需要缓解导致过度非叶级页拆分的数据操作语言（DML）操作时，使用 `PAD_INDEX` 选项。 |
| `SORT_IN_TEMPDB` | 确定是否将构建索引时产生的临时结果存储在 `tempdb` 数据库中。此选项会增加所需的空间量。 |
| `IGNORE_DUP_KEY` | 更改在对表执行插入操作时遇到重复键时的行为。启用时，违反键约束的行将失败。当默认行为被禁用时，整个插入操作将失败。 |
| `STATISTICS_NORECOMPUTE` | 指定在创建索引时是否应重新创建与该索引相关的任何统计信息。 |
| `STATISTICS_INCREMENTAL` | 指定为索引收集的统计信息是应基于整个索引创建，还是应按分区创建。 |
| `DROP_EXISTING` | 确定当表中已存在同名索引时的行为。默认情况下，当设置为 `OFF` 时，索引创建将失败。当设置为 `ON` 时，索引创建将替换现有索引。 |
| `ONLINE` | 确定在索引操作期间，表及其索引是否可用于查询和数据修改。启用时，锁被最小化，在索引创建期间持有的主要锁是意向共享锁。禁用时，锁将在操作期间阻止对索引和基础表的数据修改。在 SQL Server 2016 Service Pack 1 之前，`ONLINE` 是仅限于企业版的功能。 |
| `RESUMABLE` | 标识索引操作是否可恢复。此为 SQL Server 2019 新增功能。 |

| `ALLOW_ROW_LOCKS` | 确定是否允许在索引上使用行锁。默认情况下是允许的。 |
| `ALLOW_PAGE_LOCKS` | 确定是否允许在索引上使用页锁。默认情况下是允许的。 |
| `MAXDOP` | 覆盖索引操作期间服务器级别的最大并行度设置。该设置决定了在索引操作期间索引可利用的最大处理器数量。 |
| `DATA_COMPRESSION` | 确定在索引上使用的数据压缩类型。默认情况下，未启用压缩。通过此选项，可以指定页级或行级压缩类型。 |

### 修改索引

创建索引后，有时需要修改它。修改索引是执行索引重建或重组的方式，作为持续索引维护的一部分。此外，修改索引允许调整某些索引选项，例如压缩类型。

要修改索引，需使用 `ALTER INDEX` 语法。清单 1-4 显示了修改索引的基本语法。

```sql
ALTER INDEX { index_name | ALL }
ON <table_or_view_name>
{ REBUILD
  [ [PARTITION = ALL] [ WITH ( <rebuild_index_option> [ ,...n ] ) ]
  | [ PARTITION = partition_number [ WITH ( <rebuild_index_option> [ ,...n ] ) ] ] ]
| DISABLE
| REORGANIZE
  [ PARTITION = partition_number ]
  [ WITH ( LOB_COMPACTION = { ON | OFF } ) ]
| SET ( <index_option> [ ,...n ] )
| RESUME [WITH (<resume_index_option>[,...n])]
| PAUSE
| ABORT
} [ ; ]
```
清单 1-4: ALTER INDEX 语法

当使用 `ALTER INDEX` 语法进行索引维护时，语法中有两个选项可以使用：`REBUILD` 和 `REORGANIZE`。`REBUILD` 选项使用现有的索引结构和选项重新创建索引。它也可用于启用已禁用的索引。`REORGANIZE` 选项则对索引的叶级页面重新排序。这类似于重新洗牌，使卡片恢复顺序。这两个选项将在第 11 章中更详细地讨论。

此外，`ALTER INDEX` 语法可用于禁用索引。这通过 `ALTER INDEX` 语法下的 `DISABLE` 选项实现。禁用的索引将不会被数据库引擎使用或提供。索引被禁用后，只能通过再次使用 `REBUILD` 选项修改索引来重新启用。

除此之外，许多通过 `CREATE INDEX` 语法可用的索引选项也可在 `ALTER INDEX` 语法中使用。`ALTER INDEX` 语法可用于修改索引的压缩。它也可用于更改填充因子或填充索引设置。根据特定索引的需求，此语法可用于更改任何可用选项，尽管选项的使用方式存在一些限制。当在索引上使用 `REBUILD ALL` 分区时，所有与 `CREATE INDEX` 语法相同的可用选项都可以被修改，如清单 1-5 所示。但是，当针对单个分区进行 `REBUILD` 操作时，可用的选项列表会大大减少，如清单 1-6 所示。这是因为一些索引属性是通用的，适用于整个索引，而另一些则可以逐个分区定义。

```sql
| MAXDOP = max_degree_of_parallelism

| MAX_DURATION = <minutes> [MINUTES]

| ONLINE = { ON [ ( <low_priority_lock_wait> ) ] | OFF }
```

### 删除索引

有时索引将不再需要。这可能是由于数据库使用模式发生变化，或者某个索引与另一个索引非常相似，以至于维护两者没有必要。

要删除（或移除）索引，可以使用 `DROP INDEX` 语法。此语法包括索引名称以及构建该索引所基于的表或视图。清单 1-11 显示了删除索引的语法。从 SQL Server 2016 开始，`DROP INDEX` 语法支持 `IF EXISTS` 子句，允许在无需预先验证索引是否存在的情况下直接删除。

```
DROP INDEX [ IF EXISTS ]
index_name ON 
[ WITH (  [ ,...n ] ) ]
Listing 1-11
DROP INDEX 语法
```

除了删除索引外，根据需要还可以添加其他选项。这些选项主要适用于删除聚集索引。清单 1-12 详细列出了可用于 `DROP INDEX` 操作的选项。

```

| ONLINE = { ON | OFF }
| MOVE TO { partition_scheme_name ( column_name )
| filegroup_name
| "default"
}
[ FILESTREAM_ON { partition_scheme_name
| filestream_filegroup_name
| "default" } ]
Listing 1-12
DROP INDEX 选项
```

当聚集索引被删除时，表的基础结构将从聚集索引变为堆。构建聚集索引时，它定义了表数据的存储位置。当从聚集索引更改为堆时，必须指定新堆结构的目标位置。如果位置不是默认文件组，则需要明确指定。堆的位置可以是单个文件组，也可以由分区方案定义。此信息通过 `MOVE TO` 选项设置。除了数据位置外，还可以使用这些选项调整 `FILESTREAM` 位置。

删除索引操作的性能影响可能需要考虑。因此，`DROP INDEX` 语法中提供了选项来指定要使用的最大处理器数量，以及操作是否应联机完成。这两个选项的功能与 `CREATE INDEX` 语法中的类似。

要删除前两节中使用的索引，我们可以使用清单 1-13 中的代码。有关 `DROP INDEX` 语法及其使用示例的更多信息，请参阅 Microsoft 的在线文档。

### 索引元数据

在深入研究索引策略之前，了解 SQL Server 中可用的与索引相关的信息非常重要。索引的所有详细信息及其构建方式都存储在系统视图中。所有用户和系统数据库都包含这些目录视图，并且只返回每个查询数据库中唯一的索引。

这些视图非常有用，提供了详细的索引元数据记录，这些信息在其他地方不易获得。

#### sys.indexes

`sys.indexes` 目录视图提供数据库中每个索引的信息。对于每个表、索引或表值函数，该目录视图中都有一行。这提供了数据库中所有索引的完整清单。

`sys.indexes` 中的信息非常广泛，可以回答关于索引的许多基本问题，包括其类型、选项、筛选器等。此元数据可以指示索引是聚集索引还是非聚集索引、其填充因子、筛选器定义、是否唯一等等。

#### sys.index_columns

`sys.index_columns` 目录视图列出了每个索引中包含的列。对于作为索引一部分的每个键列和包含列，此目录视图中都有一行。除了每列的排序顺序外，还提供了键列的顺序。此视图可用于定位重复索引或确定一个索引是否在功能上与另一个索引重叠。

#### sys.index_resumable_operations

`sys.index_resumable_operations` 目录视图列出了可恢复索引重建操作的执行状态。对于每个暂停或当前正在执行的索引重建，此目录视图中都有一条记录。该视图描述了可恢复索引重建操作的 DDL，并提供了一个状态以标识其是正在运行还是已暂停。

#### sys.xml_indexes

目录视图 `sys.xml_indexes` 类似于 `sys.indexes`。此目录视图为数据库中的每个 XML 索引返回一行。该目录视图的主要区别在于它还提供了一些 XML 索引特有的信息。该视图包括 XML 索引是主 XML 索引还是辅助 XML 索引的信息。如果是辅助 XML 索引，该视图还包括辅助索引的类型。

#### sys.selective_xml_index_paths

`sys.selective_xml_index_paths` 目录视图是 `sys.indexes` 中索引的一个子集，仅包含选择性 XML 索引。每个为 XPath 创建的选择性 XML 索引在此目录视图中都有一条记录。选择性 XML 索引允许在 XML 文档的子集上创建 XML 索引，在访问 XML 数据时可以节省大量空间并提高查询性能。第 6 章将详细讨论这一点。

#### sys.selective_xml_index_namespaces

`sys.selective_xml_index_namespaces` 目录视图标识与选择性 XML 索引关联的命名空间。每个与 XML 索引关联的命名空间在此目录视图中都有一条记录，用于标识命名空间并指示其是否为默认命名空间。



#### sys.spatial_indexes

`sys.spatial_indexes`目录视图与`sys.indexes`类似。该视图为数据库中的每个空间索引返回一行。其主要区别在于提供了特定于空间索引的附加信息，例如标识该空间索引是几何索引还是地理索引。

#### sys.spatial_index_tessellations

`sys.spatial_index_tessellations`目录视图扩充了`sys.spatial_indexes`目录视图，并提供与空间索引关联的边界框和网格的详细信息。

#### sys.column_store_dictionaries

`sys.column_store_dictionaries`目录视图支持列存储索引。该视图为列存储索引中使用的每个字典返回一行。根据列的数据分布和表的分区详情，一个列可能定义有多个字典。此元数据描述了为该列构建的字典的结构和类型。

#### sys.column_store_segments

`sys.column_store_segments`目录视图通过为列存储索引中的每一列至少返回一行来支持列存储索引。列可以有多个段，每个段大约包含一百万行。此视图中的行描述了每个段的内容，例如该段是否包含空值，以及该段的最小和最大数据 ID。

#### sys.column_store_row_groups

`sys.column_store_row_groups`目录视图通过针对每个列存储行组返回一行来支持列存储段。这包括诸如行组中的行数、其状态以及在数据库中的物理位置等详细信息。

#### sys.hash_indexes

`sys.hash_indexes`目录视图类似于`sys.indexes`，但专门用于内存优化表上的哈希索引。此视图移除了一些不相关的列，并增加了`bucket_count`列。此列返回为每个哈希索引创建的存储桶数量。对于哈希索引，存储桶指的是为存储索引中的值而创建的内存位置。存储桶与索引值之间的关系详见第[8]章。

#### sys.fulltext_catalogs

`sys.fulltext_catalogs`目录视图针对数据库中的每个全文目录包含一行。

#### sys.fulltext_indexes

`sys.fulltext_indexes`目录视图针对数据库中的每个全文索引包含一行。该视图描述了该索引所属的全文目录，并提供了有关索引状态及其更新方式的详细信息。

#### sys.fulltext_index_columns

`sys.fulltext_index_columns`目录视图支持`sys.fulltext_indexes`。它针对与全文索引关联的每一列包含一行。

## Summary

本章介绍了与索引相关的诸多基础知识，涵盖了 SQL Server 中可用的各种索引类型。从堆到非聚集索引再到空间索引，每种索引都被引入，并与杜威十进分类法相关联，以此作为现实世界的类比。此示例有助于说明每种索引类型如何相互交互，以及一种索引在何种场景下能比其他索引提供价值。

本章提供了用于创建、更改和删除索引的数据定义语言（DDL）。DDL 具有丰富的选项集，可用于精细调整索引的结构，以帮助提高其在数据库中的性能。

本章还包含了记录 SQL Server 中索引的目录视图所含元数据的信息。每个目录视图都提供了有关索引结构和组成的信息。此信息可帮助研究和理解表是如何被索引的，并基于常见的查询和维护模式促进改进。

本章的细节为本书后续将要讨论的内容提供了框架。此信息为深入讨论索引提供了一个起点。本书的剩余部分将在此框架上构建，从而设计和实现有效的索引策略，以改进查询和应用程序性能。
