# 引入延迟检查机制

到目前为止，我们一直假设在事务中串行执行 DML 语句时，每个中间数据库状态都应满足所有约束；所有必需的 DI 代码都通过 DML 后触发器在语句级别立即执行。然而，有时你可能需要将 DI 代码的执行延迟到事务结束时。

###### 为何需要延迟检查？

有一类数据完整性约束在事务执行过程中总是需要被暂时违反，并在事务结束时以延迟方式重新检查。特化约束就是这样一个例子。看看我们示例数据库设计中的约束 `PSPEC1`：

```
PSPEC1(EMP,SREP) :=

/* 销售代表有目标销售额和佣金 */

{ e(EMPNO) | e∈EMP ∧ e(JOB) = 'SALESREP' } =

{ s(EMPNO) | s∈SREP }
```

因为在 `SQL` 中，你无法同时插入 `EMP` 行和 `SREP` 行，所以插入新销售代表的事务将不得不执行两次插入操作。而且，无论插入顺序如何，中间数据库状态都会违反约束 `PSPEC1`。

这个例子给了我们第一个理由，说明为何在我们的 `SQL` 世界中，你必须处理暂时的约束违反。

**DML 语句操作单一表结构**

鉴于我们设定的数据完整性约束集，某些有效的数据库状态转换很可能需要对多个表结构执行 DML 语句。因为 `数据库约束` 类别的约束其数据范围跨越多个表结构，所以极有可能出现这样的情况：第一个对某个涉及表结构执行的 DML 语句导致了数据库约束违反，而第二个在其他某个涉及表结构上执行的 DML 语句纠正了这一违反。

注意"可能性"这个词的用法。有些事务总是需要暂时违反某个特定的数据库约束，无论该事务内部的 DML 语句以何种顺序执行（前述的 `PSPEC1` 就是一例）。然而，许多数据库约束永远不需要延迟检查。通过按约束所隐含的*特定顺序*执行不同的 DML 语句，受这些约束约束的数据可以始终保持满足约束的状态。例如，子集要求就是这种情况。

只要 DML 语句按正确的顺序执行，中间数据库状态总是能满足子集要求；你将需要先向超集插入新行，然后再向子集插入。

**DML 语句以单一方式操作**

DML 语句是 `INSERT`、`UPDATE` 或 `DELETE` 语句。然而，一个表结构的有效状态转换可能需要多种类型的 DML 语句来实现，因此也可能导致需要允许表约束也存在暂时违反。

例如，考虑以下（不太现实的）表约束："销售代表的数量加上文员数量的两倍必须等于 100 或零。" 看看引入一个文员的事务。假设当前 `EMP` 表包含 100 名销售代表（这使我们的表约束为 `TRUE`）。一旦我们引入新文员，无论是通过更新销售代表的 `JOB` 字段还是插入新文员，我们总是会引入违反约束的情况，并需要第二种不同类型的 DML 语句来恢复表约束的真值。

`SQL` 语言的这个缺陷意味着，某些 `表约束` 的 DI 代码也可能需要延迟执行。我们将那些在事务内部需要暂时违反的表约束和数据库约束称为 `可延迟` 约束。
