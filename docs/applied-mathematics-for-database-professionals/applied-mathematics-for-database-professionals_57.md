# 实施 Oracle 数据库设计

## 数据完整性代码（DI 代码）

编写试图更改当前数据库状态的`wBL`代码。当它执行时，SQL `DBMS`应确保不会违反任何数据完整性约束；产生的数据库状态应满足所有数据完整性约束。

通常，你会发现嵌入在`wBL`代码中的代码，其特定目的是验证将要执行的事务不会导致违反任何相关数据完整性约束的状态。我们认为这种约束验证代码不属于`wBL`代码类，而是属于下一节讨论的*数据完整性代码*类。

**数据完整性代码（DI 代码）**

数据完整性代码是所有处理验证数据完整性约束持续有效性的*声明式*或*过程式*代码。每当`wBL`代码执行事务时，作为此类事务一部分执行的数据操作语句可能违反数据完整性约束。如果确实如此，数据完整性代码将确保此类语句失败，并且回滚其所做的更改。

**注意** 我们特意将`DI`代码视为一个独立的类，而不是业务逻辑代码类的一部分。大多数其他涉及*业务逻辑*的书籍和文章常常模糊不清`DI`代码是否被视为业务逻辑的一部分。

对于大多数数据完整性约束——实际上是除了属性约束和元组约束之外的所有约束——`DI`代码将必须执行数据检索代码（查询）来确定给定的数据操作语句是否被允许。对于多元组约束，`DI`代码总是需要执行查询（检查其他涉及的行）以验证产生的数据库状态是否仍然满足所有数据完整性约束。

例如，考虑在清单 7-26（表`universe` `tab_EMP`）中定义的“最多允许一位总裁”表约束。当一个事务试图插入一位总裁时，你应该在执行插入操作之前或之后运行一个*约束验证查询*，以根据此约束验证该插入是否被允许。

如果你决定在实际执行插入操作之前验证约束是否仍然满足，你可以执行以下约束验证查询：

```sql
select 'there is already a president'
from emp
where job='PRESIDENT'
and rownum=1
```

如果此查询返回一行，那么显然不应允许该插入继续进行；插入另一位总裁显然会创建违反此约束的数据库状态。

在另一种情况下——在插入后执行查询——你可以运行以下查询：

```sql
select count(*)
from emp
where job='PRESIDENT'
```

如果此查询返回的结果大于 1，则将违反约束，并且必须回滚该插入。

理想情况下，你希望给定所有约束的*声明*，`DBMS`能够自动推断并执行这些约束验证查询（以及一些条件代码，以阻止`DML`语句的执行或强制回滚`DML`语句）。提供此服务的`DBMS`使编写业务逻辑的开发人员的任务变得容易得多。

**注意** 如果你熟悉`Oracle`，你会知道`Oracle`的`SQL DBMS`只能为有限数量的约束类型提供此服务。

你可以将`DI`代码类细分为五个子类，每种约束类型一个：
1. 验证属性约束的代码
2. 验证元组约束的代码
3. 验证表约束的代码
4. 验证数据库约束的代码
5. 验证转换约束的代码

正如你将看到的，对于某些约束，你可以在`SQL DBMS`中以*声明式*的方式说明所需的`DI`代码（然后由`DBMS`负责维护该约束）。然而，对于大多数约束，你需要手动开发代码；也就是说，你负责实现。这意味着你需要自己设计和编写*过程式代码*来维护约束。

**注意** 有两种高级的过程式实现策略来创建`DI`代码；要么你利用*数据库触发器*，要么将`DI`代码嵌入到`wBL`代码中。我们将在本章的剩余部分更详细地讨论前者。

**用户界面代码（UI 代码）**

`UI`代码是决定业务应用程序外观和感觉的代码。它负责用户在使用应用程序时所面对的业务应用程序的前端。`UI`代码要么
- 为用户创建用户界面——业务应用程序的*外观*——并通常在用户界面中显示检索到的数据，要么
- 响应用户发起的用户界面事件，然后修改用户界面——业务应用程序的*感觉*。

创建和/或修改用户界面总是需要嵌入对业务逻辑代码的调用。这些`BL`代码将把数据返回给`UI`代码进行显示，和/或将作为用户请求事务的一部分更改数据。根据这些嵌入式`BL`代码调用的结果，`UI`代码将相应地修改用户界面。

### 声明式策略

在*声明式*策略中，您只需向`DBMS`声明数据库设计中涉及的约束。`DBMS`随后会自动确保——希望是以一种高效的方式——当违规的`DML`语句试图创建违反任何已声明约束的数据库状态时，这些语句将被回滚。

请注意，为了能够向`DBMS`声明约束，您首先需要这些约束的（某种形式的）形式化规范，以精确说明约束的*内容*。声明式策略的优势在于，您只需告诉`DBMS`约束的*内容*是什么；也就是说，您无需编写任何`DI`代码。从本质上讲，向`DBMS`声明约束就构成了`DI`代码。

一旦您声明了一个约束，`DBMS`就面临着以何种方式高效检查该约束的挑战；它的任务是生成（希望）*高效*的`DI`代码，并确保在处理事务时，这些代码在适当的时间被自动执行。

##### 注意
您将在本章中看到，上一句话中的“高效”一词是至关重要的。

Oracle 的`SQL DBMS`允许您以声明方式指定某些约束，可以作为`create table`命令的一部分，也可以在表创建后通过`alter table add constraint`命令单独指定。

从“实现属性约束”一节开始，您将找到可以声明指定的约束的完整介绍。作为预览，清单 11-1 包含一个向已创建表添加约束的示例。它展示了用于声明清单 7-26（`EMP`表的宇宙）中第二个表约束的`SQL`语法，该约束声明：

`{USERNAME}`是一个键。

### 清单 11-1. 使用`SQL`声明键
```
alter table EMP add constraint EMP_K2 unique (USERNAME);
```

通过前面的`alter table SQL`语句，您将`{USERNAME}`声明为`EMP`表结构的键。此命令允许您为约束命名（在上面的例子中，它被命名为`EMP_K2`）。如果您想删除此约束，就需要使用这个名称。

##### 注意
如果您不提供名称，`DBMS`将为您生成一个。

在本章的所有示例中，我们选择只使用`alter table add constraint`语句，以便逐步构建数据库设计的实现。您也可以选择将每个声明式添加的约束作为`create table`语句的一部分来声明。您可以在线找到作为`create table`语句一部分的、声明`UEX`数据库设计约束的`SQL`表达式，网址为：`http://www.rulegen.com/am4dp`。

### 触发式过程策略

在*触发式过程*策略中，您不向`DBMS`声明约束（*内容*）。相反，您的任务是自己编写过程化的`DI`代码（*方式*）。同时，您通过创建*触发器*来调用您的`DI`代码，从而告诉`DBMS`*何时*执行这些代码。您可以将触发器视为一段过程代码，`DBMS`将在某些事件发生时自动执行它。让我们更详细地解释一下。

##### 注意
触发器可以用于多种目的。执行`DI`代码以强制执行数据完整性只是其中之一（也是本章我们将重点关注的目的）。

每个触发器都与一个表结构相关联，并且每当事务对该表结构执行`DML`语句时，`DBMS`都会自动执行（*触发*）它。

然后，作为同一事务一部分执行的触发器代码可以检查新的数据库状态是否仍然满足所有约束。

Oracle 的`SQL DBMS`为每个表结构提供了`SQL`标准中定义的 12 种不同类型的触发器：

*   四种*插入触发器*：一种在`INSERT`语句开始执行前触发（称为*语句前插入触发器*），一种在实际插入行之前触发（*行前插入触发器*），一种在行插入后触发（*行后插入触发器*），最后一种在`INSERT`语句完成后触发（*语句后插入触发器*）。
*   四种*更新触发器*：类似于插入触发器，有四种类型的更新触发器——语句前、行前、行后和语句后。
*   四种*删除触发器*：同样的四种类型。

语句触发器（前和后）每个（触发）`DML`语句只会触发一次。

行触发器将根据`DML`语句影响的行数触发相应次数。

例如，一个`UPDATE`语句增加了员工表中所有员工（假设目前有十名）的月薪，将导致行前和行后更新触发器各自触发十次。而语句前和语句后更新触发器在此情况下各只触发一次。

行级触发器能够引用受触发`DML`语句影响的行的列值。行前和行后插入触发器可以检查新行（被插入的行）的所有列值。行前和行后删除触发器可以检查旧行（被删除的行）的所有列值。行前和行后更新触发器可以同时检查旧行的所有列值和新行的所有列值。

##### 注意
`UPDATE`语句用一组行（“新”行）替换另一组行（“旧”行）。

如果您还不熟悉 Oracle 中可用的各种表触发器类型，我们建议您参考 Oracle 手册以获取更多信息。在本章的其余部分，我们假设您熟悉它们。

对于语句级触发器来说，要引用受触发`DML`语句影响的行并非易事；它们要么看到`DML`语句开始前存在的（旧）表（前触发器），要么看到`DML`语句完成后得到的（新）表（后触发器）。您可以使语句级触发器能够看到受影响的行，但这需要您进行一些复杂的编码。在本章后面，您会发现这实际上是在触发式过程策略中开发*高效*`DI`代码的一个要求。然后我们将演示如何实现这一点。

为了说明使用触发器作为实现`DI`代码的手段，这里有一个例子。

让我们看看为`EMP`表结构的“最多允许一位总裁”表约束实现一个触发器。清单 11-2 展示了实现这样一个附加到`EMP`表结构的触发器的`SQL`语法。

### 清单 11-2. 使用`SQL`将触发器附加到表
```
create trigger EMP_AIUS
after insert or update on EMP
declare
  pl_num_presidents number;
begin
  --
  select count(*) into pl_num_presidents
  from EMP
  where job='PRESIDENT';
  --
  if pl_num_presidents > 1
  then
    raise_application_error(-20999,'Not allowed to have more than one president.');
  end if;
  --
end;
/
```

创建此触发器的方式（第二行：`after insert or update`）确保了每当`BL`代码插入或更新一名员工（此约束的*何时*）时，`DBMS`都会执行触发器的过程代码。该触发器会在这样的插入或更新执行后立即触发。因此，嵌入的代码将看到反映该插入或更新所做更改的数据库状态。


触发器主体（即 `declare` 与 `end` 之间的过程代码），如代码清单 11-2 所示，代表了这个约束的“如何实现”部分。首先执行一个查询来检索 `EMP` 表中总统的数量。然后，如果该数字大于一，触发器会通过调用 `raise_application_error()` 来产生一个错误。这个调用会引发一个 SQL 异常，进而导致触发它的 DML 语句执行失败，并使其所做的更改被撤销。在这种情况下，`DBMS` 会为你执行这条 DML 语句的回滚，这与后面描述的嵌入式过程策略相反。

请注意，让此触发器也为 `DELETE` 语句触发是低效的。当一个事务从 `EMP` 表结构中删除行时，这个特定约束不会被违反；如果在从 `EMP` 删除行之前至多有一个总统，那么在删除该行之后也仍然至多有一个总统。

在触发式过程策略中，你——`DI` 代码开发者——不仅需要考虑 `DI` 代码何时`应该运行`，还需要考虑何时 `DI` 代码的运行是`不必要的`。

这是确保 `DI` 代码被高效实现的一部分。当然，在声明式策略中，你则无需考虑这些事情。

`■注意` 代码清单 11-2 中的触发器现在会对针对 `EMP` 表结构执行的每一种 `INSERT` 和每一种 `UPDATE` 语句都进行触发。这仍然是相当低效的。例如，如果你插入一个新的职员或更新某个员工的月薪，触发器实际上根本不需要运行。在这两种情况下（与前面讨论的 `DELETE` 语句类似），触发它的 DML 语句都不可能违反该约束。

除了在效率上并非最优之外，我们还需要解决代码清单 11-2 中触发器的一个更严重的问题：在实现给定约束方面，它并不完整。仍然存在问题，我们将在“表约束实现问题”一节中更详细地讨论这些问题，这些问题涉及并发执行的事务。

## 嵌入式过程策略

最后，在实现 `DI` 代码的 `嵌入式过程` 策略中，你同样需要自己编写 `DI` 代码（“如何做”）。你不再创建调用你的 `DI` 代码的触发器，而是将 `DI` 代码嵌入到包含可能导致约束违反的 DML 语句的 `业务逻辑` 代码模块中。`DI` 代码随后作为 `业务逻辑` 代码执行的一部分运行。

这很可能是迄今为止在 `面向数据库` 应用程序中实现 `DI` 代码最常用的过程策略。

`■注意` 在这种情况下，“最常用”的策略肯定不意味着“最佳”策略。

让我们看一个例子。假设应用程序提供了一个页面，用户可以在其中输入要插入的新员工的数据。用户提供了属性 `ENAME`、`BORN`、`JOB` 和 `DEPTNO`。`BL` 代码根据与用户商定的规则确定其他员工属性（`EMPNO`、`SGRADE`、`MSAL`、`HIRED` 和 `USERNAME`）。代码清单 11-3 列出了 `UI` 代码调用来插入新员工的过程。它首先包含 `BL` 代码以确定其他员工属性，然后执行一个 `INSERT` 语句。紧接在此 `INSERT` 语句之后，`BL` 代码运行嵌入的 `DI` 代码，以确保“至多一个总统”的约束保持满足。

-- Determine username.

`INSERT`语句之后嵌入的 DI 代码与清单 11-2 中列出的触发器主体内的代码非常相似。它比`INSERT`语句触发器更高效；DI 代码仅在插入总裁时才会执行。

还要注意，在这种过程式策略中——与触发式过程式策略相反——需要由开发人员显式地强制执行语句回滚。在前面的代码中，这是通过在执行`INSERT`语句之前设置一个事务保存点，并在发现违反约束时回滚到该保存点来实现的。

嵌入式过程式策略还有另一个更严重的缺点。现在，对于一个给定的约束，你必须将 DI 代码复制到每个可能包含可能违反该约束的 DML 语句的 wBL 代码模块中。在这种策略中，一个给定约束的所有必要 DI 代码通常分散在各个 wBL 代码模块中。通常，当需要更改 WoD 应用程序的 wBL 代码时，你可能会忘记更改（添加或移除）必要的 DI 代码。更有甚者，当数据库设计的实际约束需要更改时，通常很难定位所有包含该约束的嵌入式 DI 代码的 wBL 代码模块。

`■注意` 这个缺点在某种程度上也适用于触发式过程式策略。清单 11-2 中给出的示例触发器实际上是两个触发器的简写：一个更新触发器和一个插入触发器。该约束的 DI 代码可以视为分散在两个触发器中。在更复杂的约束情况下，你将需要更多的触发器来实现 DI 代码；你可能需要多达三倍于约束中涉及的表结构数量的触发器（每个涉及的表结构一个插入、一个更新和一个删除触发器）。然而，使用触发式过程式策略，这个缺点不像在使用嵌入式过程式策略构建的 WoD 应用程序中可能出现的那么严重。

嵌入式过程式策略在并发执行事务方面存在相同的问题（我们将在“表约束实现问题”一节中解释这些问题）。

## 优选顺序

声明式策略比两个过程式策略更受青睐，这应该不足为奇。不仅声明式表述的约束免除了你开发 DI 代码的负担，它还可能优于任何自行开发的过程式实现。

别忘了，它也使你免于在 WoD 应用程序生命周期内维护此代码的负担！

`■注意` 最后一点假设 DBMS 供应商在设计和实现 DI 代码的高效*执行模型*方面已做好了研究。在“表约束实现问题”一节中，我们将向你介绍各种执行模型，从低效的到更高效的。

然而，正如你可能意识到的，在 Oracle 的 SQL DBMS 中，你只能声明性地表述有限的约束类型。请注意，这并非由于 SQL 标准固有的缺陷，而是由于该语言在 SQL DBMS 中的实现不佳所致。

对于无法声明给 DBMS 的约束，我们更倾向于遵循触发式过程式策略。以下是我们更喜欢触发式而非嵌入式过程式策略的两个主要原因：

-   与声明的约束一样，触发式过程式策略无法被规避。
-   触发式过程式策略可能创建更易管理的代码架构；DI 代码不会在大量 BL 代码中复制。DI 代码与所有 BL 代码完全分离，因此更易于管理（顺便说一下，BL 代码本身也会更易于管理）。

`■注意` 第二个原因假设触发器只包含 DI 代码。在触发器中嵌入 BL 代码通常被认为是不好的做法。

嵌入式过程式策略是所有策略中最差的；它显然可以被规避，只需使用其他应用程序（可能是 DBMS 供应商提供的一些通用应用程序）来更新数据库即可。

正如你将从“表约束实现问题”一节开始看到的，当我们开始处理表约束和数据库约束（即多元组约束）时，通过触发器为这些约束实现高效的 DI 代码远非一项简单的任务。

显然，在一个类别（属性、元组、表、数据库、转换）内实现不同约束所涉及的工作量是相同的。此外，为约束实现 DI 代码的复杂性可能随着数据范围的增加而增加。本章剩余部分将提供实现数据库设计的表结构的指南，并按范围递增的顺序，为所涉及的约束实现 DI 代码提供指南。以下是本章剩余部分的概述：

-   “实现表结构”一节涉及在 Oracle 的 SQL DBMS 中实现表结构。
-   “实现属性约束”一节讨论如何实现属性约束。
-   “实现元组约束”一节讨论如何实现元组约束。
-   “表约束实现问题”一节作为后续章节的序言。在本节中，我们将介绍在开发触发器以实现多元组约束的 DI 代码时涉及的各种问题。本节还将详细阐述不同的执行模型。
-   “实现表约束”一节讨论如何声明性地表述某些表约束，以及如何设计触发器来检查所有其他表约束。
-   “实现数据库约束”一节讨论如何声明性地表述某些数据库约束，以及如何设计触发器来检查所有其他数据库约束。
-   “实现转换约束”一节讨论一种为转换约束实现 DI 代码的方法。
-   “引入延迟检查”一节解释了对某些类型的约束延迟执行 DI 代码的必要性。
-   “RuleGen 框架”一节向你介绍 RuleGen，这是一个支持使用触发式过程式策略实现 DI 代码的框架，它处理了本章讨论的许多问题。
