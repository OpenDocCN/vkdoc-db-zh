# 在 Oracle 中实现数据库设计

至此，本书的主要焦点一直放在形式化地指定数据库设计上。当然，你想要做的不仅仅是指定一个数据库设计；你通常希望使用某个 DBMS 来 *实现* 它。在本章中，我们将注意力转向实现数据库设计。

鉴于两位作者的背景，本章特别关注在 Oracle 的 SQL DBMS 中实现数据库设计。我们假设你熟悉 PL/SQL 语言——Oracle 的过程化编程语言。我们假设诸如数据库触发器、包、过程和函数等概念都是已知的。

**注意：** 如果你在本章中遇到任何不熟悉的 PL/SQL 概念，我们建议你参考网上可用的标准 Oracle 文档 (http://tahiti.oracle.com)。

这将是一章不那么形式化的章节。在指代 SQL 构造时，我们将使用 SQL 术语（如 *行* 和 *列*）。在指代本书引入的形式化概念时，我们仍将使用形式化术语。

前几节（到“实现数据完整性代码”一节为止）确立了有关在数据库之上实现业务应用的一些概念。你还将看到，存在三种截然不同的策略来实现数据完整性约束。

接下来的章节涉及实现表结构、属性约束和元组约束（直到“实现元组约束”一节）。

“表约束实现问题”一节是一个相当大的部分。它介绍了在实现多元组约束（即表或数据库约束）时所面临的挑战。本节很大一部分将探讨各种约束验证执行模型。这些执行模型范围从效率较低的模型到更复杂（高效）的模型。

“实现表约束”和“实现数据库约束”两节分别涵盖使用“表约束实现问题”一节中引入的一种执行模型来实现表约束和数据库约束。接着探讨了转换约束的实现（“实现转换约束”一节）。

“引入延迟检查”一节涉及延迟约束验证——鉴于你在 SQL DBMS 中实现数据库设计，这是一个无法避免的棘手现象。

在本章末尾，我们将向你介绍 `RuleGen` 框架：这是一个可以帮助你在 Oracle 的 SQL DBMS 中实现数据完整性约束的框架，它应用了本章通篇解释的许多概念。

本章没有“练习”部分。

如果你熟悉不同的 SQL DBMS，那么本章讨论的大部分概念可能也适用。不过，鉴于我们的背景，我们不能完全确定。涉及事务序列化的讨论很可能是 Oracle 特有的。当然，实际的代码示例在其他 SQL DBMS 中必须以不同的方式完成。



### 第 11 章 在 Oracle 中实现数据库设计

#### 引言

迄今为止，本书的重点一直是向您展示如何正式规范地定义数据库设计。为此，我们向您介绍了一些集合论和逻辑学知识。集合论和逻辑学是必要的数学工具，使您能够专业地处理、讨论、管理和记录数据库设计。此外，正如您所看到的，数据库设计远不止是表结构；数据完整性约束是迄今为止数据库设计中最重要的一部分。

当然，您想要做的不仅仅是*规范定义*数据库设计；您通常还希望*使用*某个`DBMS`来*实现*它，并在其上构建*业务应用*。

不幸的是，我们并没有真正的关系型`DBMS`可用；我们被迫使用`SQL` `DBMS`来实现数据库设计。正如您将在本章中看到的，在`SQL` `DBMS`中实现数据库设计会面临相当多的挑战——这些挑战主要不在于实现表结构，而在于实现相关的数据完整性约束。

正如您将在本章中看到的，`SQL`为我们提供了声明性地定义数据完整性的结构；然而，并非所有这些结构都得到当前可用的`SQL` `DBMS`的支持。最值得注意的是，`CREATE ASSERTION`命令——自 1992 年以来就存在于`SQL`的官方标准中——在大多数`SQL` `DBMS`中（包括 Oracle 的）都不受支持。

在我们开始研究这些约束实现的挑战之前，我们将首先通过研究*业务应用*的通用软件架构来拓宽视野，或者，从这里开始我们更倾向于称这种应用为*窗口应用程序*。一旦我们确立了这种架构，我们将在此架构背景下讨论实现数据库设计的挑战。

**■注意** 本章将与前面几章有很大不同。它包含的数学内容要少得多，尽管您会看到，在实现数据库设计时仍然可以应用数学知识。我们认为本章为本书增添了价值，因为根据我们的经验，本章所探讨的挑战经常被忽视。

## 窗口应用程序

窗口应用程序（`WoD`）——顾名思义——是一种为用户提供数据窗口的应用程序。在`WoD`应用程序中，用户通过应用程序的各种窗口（如今通常是浏览器页面）进行导航。应用程序中的每个页面要么

• 使用户能够构建数据检索请求、执行它，然后在窗口中显示检索到的数据，要么

• 为用户提供使用已检索到的——或新输入的——数据来构建事务并执行的可能性。

由于这些应用程序的核心就是查询和操作数据，因此您应该投入相当多的精力来设计和实现底层的表结构以及支持该应用程序所需的相关完整性约束。数据库设计构成了`WoD`应用程序的基础；`WoD`应用程序的质量只能与底层数据库设计的质量一样好。

**■注意** 这实际上是我们开始撰写本书的一个主要动机。在 IT 项目中关注数据库设计能力方面，我们行业的现状相当糟糕。如今很少有 IT 专业人士接受过数据关系模型基础的教育。经常创建的数据库设计不可避免地会导致业务应用程序性能不佳，和/或难以维护。如前所述，讨论数据库设计的质量超出了本书的范围（这本身就值得另写一本书）。我们选择首先为您提供必要的工具，使您能够以清晰和专业的方式开始处理数据库设计。

所以，简而言之，`WoD`应用程序的核心就是管理业务数据。不用说，构建在数据库之上的绝大多数业务应用程序都是`WoD`应用程序。这证明有必要更仔细地审视这类应用程序的通用软件架构。

在下一节中，我们将向您介绍`WoD`应用程序代码的一种分类方案。稍后，当我们讨论数据库设计的数据完整性约束的三种实现策略时，您会看到我们将引用此代码分类方案。

## 窗口应用程序代码的分类

在本节中，我们将向您介绍一个高级的代码分类方案，它是每个`WoD`应用程序的核心。在此之前，我们需要引入两个我们将用于定义此分类方案的术语。它们是*数据检索代码*和*数据操作代码*。

*数据检索代码*是查询数据库中数据的所有代码。在我们的日常实践中，这些通常是嵌入在应用程序代码中并与`SQL` `DBMS`交互的`SQL`查询（`SELECT`表达式）。

*数据操作代码*是更改数据库中数据的所有代码。这些是嵌入在应用程序代码中、用于维护数据库中数据的`SQL`数据操作语言（`DML`）语句（`INSERT`、`UPDATE`或`DELETE`）。

**■注意** 我们将这三种`DML`语句（`INSERT`、`UPDATE`或`DELETE`）统称为*更新*语句。

引入了这两个术语后，我们现在可以介绍代码分类方案。`WoD`应用程序的所有代码都可以归入以下三个代码类之一：

• 业务逻辑代码

• 数据完整性代码

• 用户界面代码

接下来的三个小节将讨论上述三个代码类的含义。

## 业务逻辑代码（BL 代码）

业务逻辑代码可以细分为两个子类：第一，*构建并执行查询*的代码，第二，*构建并执行事务*的代码。

• *查询构建与执行代码*：这是仅包含嵌入式数据检索代码（即查询表达式）的过程式代码。该代码负责构建实际的`SQL`查询，或有条件地确定应执行哪些`SQL`查询。该代码还启动这些查询的执行，并处理`SQL` `DBMS`返回的行。我们将此类称为*读 BL 代码*（简称*rBL 代码*）。

• *事务构建与执行代码*：这是包含嵌入式数据操作代码（即`SQL` `DML`语句）的过程式代码。该代码负责构建更新语句，或有条件地确定应执行哪些更新语句。该代码还启动这些语句的执行。根据`SQL` `DBMS`给出的返回代码，该代码可能还会执行事务的提交或回滚处理。请注意，数据检索代码通常会是（事务构建的）过程式代码的一部分。我们将此类称为*写 BL 代码*（简称*wBL 代码*）。
