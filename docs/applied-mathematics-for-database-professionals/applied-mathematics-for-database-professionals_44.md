`注意` 将本质上是表约束的约束指定为数据库（多表）约束的情况并不少见。这种情况也可能在更早的阶段发生：将元组约束指定为表约束。这可以追溯到我们在第 6 章发表评论时提到“伪装的元组谓词”等术语。在本书介绍的数据库设计规范方法中，你应该始终尝试在*最早可能的阶段*指定约束：首先是属性值集，然后是元组约束，接着是表约束，最后是数据库约束。这不仅会使你的数据库设计视图清晰，也将有助于你的实施工作（在第 11 章介绍）。

以下是 `EMP` 的第三个表约束的另一种（等价的）形式化规范，可能更容易理解，并且清楚地表明不需要涉及 `DEPT` 表结构：

```
( ∀e2∈E: e2(JOB) ∈ {'PRESIDENT','MANAGER'}

⇒

( ∃e3∈E: e3(DEPTNO) = e2(DEPTNO) ∧ e3(JOB) = 'ADMIN' )

)
```

这个谓词指出，对于所有代表 `PRESIDENT` 或 `MANAGER` 的员工元组，应该存在一个代表在同一个部门工作的 `ADMIN` 的员工元组。你可能直观地感觉这个形式化规范是等价的，因此代表了相同的约束。事实上，*可以证明*它等价于 `tab_EMP` 内部此约束的形式化规范。然而，这需要开发更多关于量化的重写规则，这超出了本书的范围。

代码清单 7-27、7-28 和 7-29 定义了 `SREP`、`MEMP` 和 `TERM` 表结构的表全域。正如你所看到的，它们都有一个唯一标识约束，没有其他表约束。

###### 代码清单 7-27. *表全域 `tab_SREP`*

```
tab_SREP :=

{ S | S∈℘(tup_SREP) ∧

/* EMPNO 唯一地标识一个元组 */

( ∀s1,s2∈S: s1(EMPNO) = s2(EMPNO) ⇒ s1 = s2 )

}
```

###### 代码清单 7-28. *表全域 `tab_MEMP`*

```
tab_MEMP :=

{ M | M∈℘(tup_MEMP) ∧

/* EMPNO 唯一地标识一个元组 */

( ∀m1,m2∈M: m1(EMPNO) = m2(EMPNO) ⇒ m1 = m2 )

}
```
