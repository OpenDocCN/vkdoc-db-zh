> **注意** 这些箭头大多代表通常所说的*多对一关系*，并最终在 SQL DBMS 的实施阶段成为*外键*。然而，情况并非总是如此，你将会看到。每条箭头的确切含义将在数据库宇宙规范中给出，其中每条箭头都转化为一个数据库约束。

我们的数据库存储了公司的员工（`EMP`）和部门（`DEPT`）。部分箭头表明：
- 每位员工为一个部门工作。
- 每个部门由一位员工管理。
- 每位员工被分配到一个薪资等级（`GRD`）。

为所有薪资和/或“所属部门”变更维护员工历史记录（`HIST`）；每条历史记录描述了一位员工被分配到某个部门并享有特定薪资的时期。

我们将所有销售代表的额外信息保存在另一个表结构（`SREP`）中。我们将所有已不再为公司工作的员工（即已被终止雇佣或辞职）的额外信息保存在`TERM`中。注意，我们为已终止雇佣的员工保留`EMP`信息。我们还为所有有经理的员工（`MEMP`）保存额外信息；即，那些分配了经理的员工。

数据库还进一步保存了课程（`CRS`）、这些课程的开设班次（`OFFR`）以及这些班次的注册（`REG`）信息。更多的箭头表明：
- 一个班次必须由一位为公司工作的培训师教授。
- 一个班次对应一门已存在的课程。
- 一条注册记录将一位员工作为参与者与一个课程班次关联起来。

现在你对我们在此数据库中维护的信息有了一定了解。下一节，你将看到*数据库骨架*。如前所述，它引入了每个表结构的所有属性名称。这些名称与表结构名称一起，构成了我们数据库设计中可用的词汇表。

##### 数据库骨架

我们在数据库设计中所表示的真实世界事物的名称，包括所关注的属性名称，在所谓的**数据库骨架**中引入。我们有时将其称为*概念骨架*。正如你在第 5 章所见，数据库骨架被表示为一个集合值函数。骨架函数的定义域是表结构名称的集合。对于每个名称，该函数生成该表结构的属性名称集合；即，该表结构的标题。

示例数据库设计的数据库骨架`DB_S`在`Listing 7-1`中定义。

在`DB_S`的规范中，你可以看到嵌入的注释（`/* ... */`），用于进一步澄清为表结构和属性名称选择的缩写。

给定这个数据库骨架，你现在可以编写诸如 `DB_S(DEPT)` 的表达式，它表示 `DEPT` 表结构的属性名集合。该表达式表示集合 `{DEPTNO, DNAME, LOC, MGR}`。

通过这个表头的定义，你现在对数据库设计中每个表结构所代表的内容——它想要表示什么——有了更深一步的理解。进一步澄清表结构及其属性的*含义*的一种方法是提供*外谓词*。外谓词是一个涉及表结构所有属性的英文句子，它提供了一个关于这些属性的陈述，解释了它们相互关联的含义。以下是 `EMP` 表结构的外谓词：员工编号为 `EMPNO` 的员工，其姓名为 `ENAME`，职位为 `JOB`，出生于 `BORN`，受雇于 `HIRED`，月薪为 `MSAL` 美元，薪资等级为 `SGRADE`，分配的账户是 `USERNAME`，并且为部门编号为 `DEPTNO` 的部门工作。

它之所以被称为*外*谓词，是因为数据库管理系统无法处理这个英文句子。它是为系统的（外部）用户提供，并为所选属性名称提供解释。它被称为*谓词*，是因为你可以将这个英文句子视为被参数化的，其中参数就是嵌入的属性名称。

你可以使用当前 `EMP` 表中的元组来实例化这个外谓词。方法是将句子中出现的每个属性名称替换为给定元组中对应的属性值。这样形成的新句子可以被视为一个命题，它要么为真（TRUE），要么为假（FALSE）。由外谓词这样生成的句子是关于表所代表的真实世界的陈述。按照惯例，以此方式构建的命题被假定为真（TRUE）。这正是外谓词进一步阐明你的数据库设计含义的方式。
