##### 数据库设计

`Database Design`



数据库设计是对一组用于存储业务数据的数据结构（包括该数据的完整性要求）蓝图的非正式术语。在本书中，这些蓝图始终是关系型设计；它们将遵循数据的关系模型。

这尤其意味着，前述的数据结构是`表数据结构`；也就是说，数据通过`表`来存储（使用此后介绍的`表`的含义）。

我们通常会使用一个业务术语来指代我们正在讨论的具体数据库设计；例如，HR（人力资源）数据库设计，或教育部门数据库设计。

##### 注意
数据库设计的另一个常见含义是开发此类蓝图的过程。此过程包括需求分析、概念数据库设计、逻辑数据库设计和物理数据库设计等活动。使用这另一种含义时，你可以说本书将数据库设计视为逻辑数据库设计（过程）的最终结果。在第 3 部分（第 11 章）中，我们将探讨使用 SQL DBMS 的物理数据库设计。

#### 数据库变量
`数据库变量`是一个实现术语。它是一个变量（即，一个值的容器），能够存储`数据库设计`所描述的数据结构的实际值。在第 7 章中，我们将对数据库变量的类型——或其值集——进行数学处理。这个值集被称为`数据库宇宙`。在任何给定的时间和空间点上，一个数据库变量将持有该值集中的一个元素作为其当前值。`数据库宇宙`的元素被称为`数据库状态`。请注意，一个数据库恰好由一个`数据库变量`组成。从某种意义上说，这个变量*就是*数据库。

像每个变量一样，`数据库变量`将有一个名称，允许我们在不知道它当前持有哪个数据库状态的情况下谈论它。

##### 注意
SQL DBMS 不提供数据库变量的概念。相反，SQL DBMS 供应商已经实现了表变量的概念。一个表变量可以容纳一个表作为其值；在 SQL DBMS 中，你通过`create table`语句创建一个表变量。在这些 DBMS 中，数据库设计是通过创建与数据库设计所包含的表结构数量相等的表变量，并定义它们各自的类型（而不是只定义一个数据库变量并定义其类型）来实现的。

###### 数据库宇宙
`数据库宇宙`是一个正式术语。它是用于形式化指定`数据库设计`的`表数据结构`和完整性要求数学结构。`数据库宇宙`是`数据库设计`形式化规范的主要部分；它精确地描述了我们允许`表数据结构`存储的数据值（表）。第 7 章将建立`数据库宇宙`的形式化定义；现在只需将其视为我们允许一个给定的`数据库变量`在时间进程中持有的所有值的集合。

为了在文中便于引用，我们通常会为给定的`数据库宇宙`使用一个简短的名称（仅为符号）。

#### 数据库状态
`数据库状态`是一个正式术语。它是用于形式化指定`数据库变量`可以持有的一种值的数学结构。本章将建立`数据库状态`的形式化定义。为了在文中便于引用，我们通常会为给定的`数据库状态`使用一个简短的名称。

#### 数据库
`数据库`是以结构化方式（即符合数据库设计）存储在计算机中的实际（业务）数据的非正式术语。抽象地说，`数据库`是`数据库变量`的当前值；它是`数据库状态`的非正式术语。

#### 数据库管理系统 (DBMS)
`DBMS`是一个计算机程序，用于管理、查询和更新`数据库变量`的值。

#### 表设计
`表设计`是对一个`数据库设计`中的其中一个`表数据结构`蓝图的非正式术语。一个关系型`数据库设计`通常由多个`表设计`和一组针对数据库完整性要求的约束（见第 7 章）组成。通常讨论单个`表设计`是很方便的，因此我们将使用一个业务术语来指代我们正在讨论的具体`表设计`；例如，员工`表设计`，或部门`表设计`。

#### 表结构
`表结构`，是`表数据结构`的简称，是一个实现术语。它用于指代实现特定`表设计`的数据结构。如前所述，在`数据库设计`的实现中，你习惯于使用 SQL 表变量来实现独立的`表数据结构`。你可以将`表结构`视为一个表变量。然而，我们在本书中刻意不使用表变量这个术语；本书中开发的方法论只涉及`数据库变量`的概念。一个`表结构`可以被视为`数据库变量`的一个特定部分：仅保存所涉及的`表设计`之一的当前`表`的部分。

在第 10 章中，我们将讨论一种形式化的方法来指定`数据库变量`的更新（值变更）；我们认为只更新数据库的一个`表结构`就是对`数据库变量`的一次更新。然而，在解释我们的形式化方法时，我们将经常方便地使用`表结构`这个术语，就好像它是一个可单独寻址的（表）变量一样，我们甚至会给这样一个`表数据结构`一个名字——就像我们给表变量命名一样——使我们能够在不知道该数据结构当前持有哪个值的情况下谈论它。

##### 表
`表`是一个正式术语。它是一个数学结构，用于形式化指定一个实现特定`表设计`的`表结构`可以持有的值（或者，如果你愿意，一个表变量可以持有的值）。我们将在下一节正式确立这个术语。为了在文中便于引用，我们通常会为给定的`表`使用一个简短的名称。

#### 表集
在第 4 章中，你看到可以使用一个元组（一组有序对）来描述现实世界中的一个真命题。例如，元组 `{ (empno;100), (ename;'Smith'), (job;'MANAGER') }` 可能代表这个命题：“员工号为 100 的员工名字叫 Smith，职位是经理。”在本节中，我们将基于元组的这种用法，向你介绍一种形式化指定表的方法。

##### 表的形式化规范
一个表表示许多（零个或多个）同类型的真命题。如果你可以用一个元组来表示一个单一命题，那么显然一个表可以被视为一个`元组的集合`。

看一下列表 5-1 中名为`T1`的元组集合。

**列表 5-1.** 一个元组集合

```
T1 :=

{ { (partno;1), (name;'hammer'), (instock;22), (price;10) }

,{ (partno;2), (name;'screwdriver'), (instock;19), (price;5) }

,{ (partno;3), (name;'axe'), (instock;0), (price;30) }

,{ (partno;4), (name;'saw'), (instock;4), (price;15) }

,{ (partno;5), (name;'wrench'), (instock;7), (price;20) }

,{ (partno;6), (name;'scissors'), (instock;32), (price;5) } }
```

集合 `T1` 包含六个元组；换句话说，六个函数（回想一下，一个元组可以被视为一个函数）。所有六个函数具有相同的定义域：`{partno,name,instock,price}`。我们将所有共享相同定义域的函数集合称为一个`表`。



`T1`中的元组代表六个命题。它们可被视为以下外部谓词的实例化，例如：“零件号为`partno`的零件命名为`name`，当前库存中有`instock`件，其价格为`price`。”
