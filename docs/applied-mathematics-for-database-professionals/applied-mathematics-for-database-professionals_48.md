# 第 8 章 指定状态转移约束

在第 6 章中，你接触到了元组谓词、表谓词和数据库谓词。这些谓词处理数据的范围依次增大：单个元组内的属性值、单个表内的元组，以及单个数据库状态内的表。

“更多数据完整性谓词”一节沿着范围扩大的路径，引入了*状态转移谓词*的概念。状态转移谓词处理一对数据库状态，它们分别被称为*起始状态*和*终止状态*。

我们将使用状态转移谓词来为数据库设计定义可接受的数据库状态转移（也称为*事务*）。

在“状态转移约束”一节中，你将发现一个事务可以建模为两个数据库状态的有序对：事务开始时存在的数据库状态，以及事务结束时存在的数据库状态（即事务导致的结果状态）。

我们将定义*状态转移宇宙*的概念：它持有数据库设计所有可接受事务的集合。为了定义状态转移宇宙，你将定义代表数据库设计中*状态转移约束*的状态转移谓词。

“状态转移约束”一节以第 7 章引入的示例数据库设计的状态转移宇宙规范作为结束；它包含了该数据库设计的七个状态转移约束。示例状态转移宇宙也可在附录 A 中找到。

照例，本章末尾会有“本章小结”，随后是“练习”部分。

#### 更多数据完整性谓词

在上一章中，你接触了四类数据完整性约束：属性约束、元组约束、表约束和数据库约束。这四类约束接受或拒绝给定的数据库状态。它们可以在数据库状态的上下文中进行检查，被称为静态约束。在现实世界中，常常有要求基于静态约束以外的原因禁止某些*数据库状态转移*。这些限制状态转移的约束被称为*动态*（或*状态转移*）约束。

#### 一个简单例子

状态转移约束的一个经典说明是要求员工的工资不允许降低。正如你所见，这个要求无法在单个数据库状态的上下文中进行检查；你需要一个改变数据库状态的事务的上下文。要检查这个要求，你需要检查事务开始时存在的数据库状态（其*起始状态*）中的工资值，并检查事务结束时存在的数据库状态（其*终止状态*）中的工资值。

**注意：** 很快你会看到，我们将使用这两个数据库状态——起始状态和终止状态——来刻画一个事务。

让我们用上一章引入的数据库设计为例来演示这一点。图 8-1 显示了两个员工表 `E1` 和 `E2`。

**图 8-1.** *表 `E1` 和 `E2`*

这两个表仅在三个属性值上有所不同。

假设一个给定的事务，例如 `TX1`，起始于一个包含 `E1` 的起始状态，并产生一个包含 `E2` 的终止状态。如图 8-1 所示，`TX1` 显然降低了员工 101 的工资（从 8200 降到 7900），并改变了该员工的工资等级。`TX1` 也增加了员工 105 的工资（从 3000 增加到 4000）。事务 `TX1` 没有改变所有其他员工的工资。

**注意：** 显然，`TX1` 违反了规定工资不允许降低的状态转移约束。



现在让我们来研究如何形式化地规定一名员工的薪资不允许降低。事务 `TX1` 违反了此状态转换要求，因为在 `E1` 中的一个员工元组（比如 `e1`）和在 `E2` 中的一个员工元组（比如 `e2`）存在一种组合，它们都涉及同一名员工——即它们在 `empno` 属性上对应——并且 `e2` 中的 `msal` 值低于 `e1` 中的 `msal` 值。

假设 `dbs1` 是事务 `TX1` 的开始状态，`dbs2` 是其结束状态。注意，这意味着表达式 `dbs1(EMP)` 的计算结果为 `E1`，表达式 `dbs2(EMP)` 的计算结果为 `E2`。看一下列表 8-1 中的命题。

**列表 8-1.** 一个关于数据库状态 `dbs1` 和 `dbs2` 的命题

```
prop1 :=
(∀e1∈dbs1(EMP): (∀e2∈dbs2(EMP): e1(empno) = e2(empno) ⇒ e1(msal) ≤ e2(msal) ) )
```

该命题指出，不可能存在 `E1` 中的一个元组和 `E2` 中的一个元组的组合，使得它们在 `empno` 属性上对应，并且 `E1` 元组中的 `msal` 值大于 `E2` 元组中的 `msal` 值。这精确地反映了我们的状态转换要求。

这种类型的转换约束（在属性级别上）很常见，并且可以用另一种方式来指定：将开始状态中的表与其在结束状态中的对应版本进行连接，使得所有属性值（旧的和新的）都可用，然后限制此连接结果中元组的属性值。

**注意** 大致来说，此类转换约束可以被视为*动态*的元组连接约束。

列表 8-2 展示了一种形式化指定此要求的替代方法；它连接了开始状态 (`dbs1`) 中的 `EMP` 表与结束状态 (`dbs2`) 中的 `EMP` 表。

**列表 8-2.** 我们的状态转换要求的替代指定

```
prop2 :=
( ∀e∈((dbs1(EMP)⇓{empno,msal})◊◊{(empno;empno),(old_msal;msal)})⊗
  ((dbs2(EMP)⇓{empno,msal})◊◊{(empno;empno),(new_msal;msal)}): e(old_msal) ≤ e(new_msal) )
```

我们现在使用连接运算符 (`⊗`) 以及表投影 (`⇓`) 和属性重命名 (`◊◊`)。在此指定中，`E1` 和 `E2` 都首先投影到 `empno` 和 `msal` 属性上。然后，（`E1` 投影的）`msal` 属性被重命名为 `old_msal`，（`E2` 投影的）`msal` 属性被重命名为 `new_msal`。得到的两个表现在可以用连接运算符进行连接。此连接产生的表是一个基于 `{empno,old_msal,new_msal}` 结构的表。对于此表中的所有元组，`old_msal` 值都应小于或等于 `new_msal` 值。

#### 状态转换谓词

承接第 6 章引入的*元组*、*表*和*数据库谓词*概念，我们现在通过定义*状态转换谓词*的概念，进一步扩展数据完整性谓词可以处理的数据范围。

状态转换谓词处理的是*一对数据库状态*。这些谓词有两个参数，类型均为数据库状态；第一个参数称为开始状态，第二个参数称为结束状态。在下一节中，我们将使用状态转换谓词来为数据库设计定义可允许的状态转换。只有那些事务，其开始和结束状态能将所有状态转换谓词转换为真命题，才被认为是有效（或可允许的）事务。

注意，列表 8-1 和 8-2 中的命题也处理一对数据库状态：给定的两个数据库状态 `dbs1` 和 `dbs2`。现在看一下列表 8-3 中的状态转换谓词。这些是列表 8-1 和 8-2 中命题的“谓词版本”。

**列表 8-3.** 两个状态转换谓词

```
STP1(B,E) := ( ∀e1∈B(EMP), e2∈E(EMP): e1(empno)=e2(empno) ⇒ e1(msal) ≤ e2(msal) )
STP2(B,E) := ( ∀e∈((B(EMP)⇓{empno,msal})◊◊{(empno;empno),(old_msal;msal)})⊗
                ((E(EMP)⇓{empno,msal})◊◊{(empno;empno),(new_msal;msal)}): e(old_msal) ≤ e(new_msal) )
```

当你分别将数据库状态 `dbs1` 和 `dbs2` 作为参数 `B` 和 `E` 的值提供时，这些状态转换谓词将转换为列表 8-1 和 8-2 中列出的命题。

```
STP1(dbs1,dbs2) = prop1
STP2(dbs1,dbs2) = prop2
```

我们要求状态转换谓词不能分解为多个合取项，就像元组谓词、表谓词和数据库谓词一样。如果你能将给定的状态转换谓词 `ST(B,E)` 重写为形式为 `STa(B,E) ∧ STb(B,E)` 的表达式，那么你实际上是在处理两个状态转换谓词。

在本书的后续部分，我们还将要求状态转换谓词——比如 `ST(B,E)`——不能被重写为以下两种形式之一的谓词：`P(B)` 或 `P(E)`。在这些表达式中，`P` 代表一个数据库谓词；即一个检查单个数据库状态的谓词（在前面的情况中是 `B` 或 `E`）。这意味着我们要求一个状态转换谓词必须至少涉及开始状态中的一个表以及结束状态中的一个表。这是因为，如果谓词只涉及来自一个数据库状态（开始状态或结束状态）的表，那么该谓词将反映一个静态约束，而不是一个状态转换约束。

正如你将在下一节给出的示例中看到的，状态转换谓词通常涉及来自相同表结构的开始状态和结束状态的表。我们通常不会用两个类型为数据库状态的参数来指定状态转换谓词，而是会用两个或多个类型为表的参数来指定。为了说明这一点，列表 8-4 以这种方式重新指定了列表 8-3 中的状态转换谓词。参数 `EMPB` 和 `EMPE` 的类型是表；其意图是让参数 `EMPB` 接受在事务开始时存在的员工表状态，让参数 `EMPE` 接受在事务结束时存在的员工表状态。

**列表 8-4.** 状态转换谓词 `STP1` 和 `STP2` 的替代指定

```
STP1(EMPB,EMPE) :=
( ∀e1∈EMPB, e2∈EMPE: e1(empno) = e2(empno) ⇒ e1(msal) ≤ e2(msal) )
STP2(EMPB,EMPE) :=
( ∀e∈((EMPB⇓{empno,msal})◊◊{(empno;empno),(old_msal;msal)})⊗
      ((EMPE⇓{empno,msal})◊◊{(empno;empno),(new_msal;msal)}):
      e(old_msal) ≤ e(new_msal) )
```

在下一节中，我们将使用状态转换谓词来形式化地指定数据库设计的状态转换约束。

##### 状态转换约束

在本节中，我们将向你介绍*状态转换全域*的概念。状态转换约束将作为嵌入式状态转换谓词，在状态转换全域的形式化定义中进行指定。

##### 状态转换全域

为了形式化地指定数据库设计的状态转换约束，我们将指定该数据库设计的*所有可允许事务的集合*。你可以将一个事务建模为其开始状态和结束状态的有序对；有序对中的第一个坐标代表开始状态，第二个坐标代表结束状态。

然后，你可以将*所有可允许事务的集合*指定为此类有序对的集合。



该集合中的每一个有序对都形如 `(B;E)`，其中 `B` 和 `E` 均代表一个有效的数据库状态（即，它们是数据库设计中数据库宇宙的元素）。这个有序对集合被称为*状态转移宇宙*。如果 `(B;E)` 是状态转移宇宙的一个元素，那么允许存在将状态 `B` 转换为状态 `E` 的事务。

为了说明这一点，假设你需要为某个数据库设计指定一个状态转移宇宙，其数据库宇宙仅包含四个数据库状态。我们将其命名为 `s1` 到 `s4`。进一步假设，你只允许事务实现以下状态转移之一：

`s1` 到 `s2`

`s1` 到 `s3`

`s1` 到 `s4`

`s2` 到 `s3`

`s2` 到 `s4`

`s3` 到 `s4`

`s4` 到 `s1`

在此数据库设计中，仅允许前述七种不同的事务。你可以通过这四个数据库状态上的一个*有向图*来可视化这七个事务。

当然，对于现实世界的数据库设计，你无法用这种枚举方式来指定状态转移宇宙，因为需要枚举的事务数量巨大。你将不得不采用谓词方式来指定可接受事务的集合。

你可以通过首先构造数据库宇宙与自身的笛卡尔积，以谓词方式为给定的数据库设计形式化地指定状态转移宇宙。

该笛卡尔积包含了从一个可接受数据库状态到另一个可接受数据库状态的所有可能转移（建模为一个有序对）。

**■ 注意** 如果给定数据库设计的数据库宇宙的基数为 `n`，那么总共可以识别出 `n*n` 个不同的事务。然而，状态转移约束将不会允许所有这些事务。

通过指定状态转移谓词，你可以将这个可能转移的集合缩小为一个仅包含可接受转移的集合。清单 8-6 展示了状态转移宇宙的规范模板。

**清单 8-6.** *状态转移宇宙规范模板*

```
STU := { TX | TX∈DBU×DBU ∧
STC1(π₁(TX),π₂(TX)) ∧ STC2(π₁(TX),π₂(TX)) ∧ ... ∧ STCn(π₁(TX),π₂(TX)) }
```

在此模板中，`DBU` 代表相关的数据库宇宙，`STC1`、`STC2`、……、`STCn` 代表状态转移谓词。表达式 `π₁(TX)` 表示 `TX` 的起始状态，`π₂(TX)` 表示 `TX` 的结束状态。我们称状态转移谓词 `STC1` 到 `STCn` 为 `STU` 的*状态转移约束*。

定义状态转移宇宙 `STU` 的另一种方式如下：

```
{ (B;E) | B∈DBU ∧ E∈DBU ∧
STC1(B,E) ∧ STC2(B,E) ∧ ... ∧ STCn(B,E) }
```

此集合包含了所有有序对 `(B;E)`，其中 `B` 和 `E` 代表可接受的数据库状态。

状态 `B` 和 `E` 满足所有状态转移约束的有序对，代表了可接受的事务。

#### 完成示例数据库设计

现在，让我们通过为前一章引入的示例数据库设计指定状态转移约束，来演示所有这些概念。清单 8-7 引入了状态转移宇宙 `ST_UEX` 的形式化规范。如你所见，它建立在第 7 章指定的 `DB_UEX` 数据库宇宙之上。`ST_UEX` 的定义为我们示例数据库设计引入了状态转移约束。此定义仅通过名称指定它们。在清单 8-8 中，你将找到每个命名约束的形式化规范。

**■ 注意** 在清单 8-7 中，我们以另一种方式指定状态转移谓词：作为以表为类型的参数的谓词。

**清单 8-7.** *状态转移宇宙 ST_UEX*

```
ST_UEX := { (b;e) | b∈DB_UEX ∧ e∈DB_UEX ∧
                     ∧ STC1(b(EMP),e(EMP))
                     ∧ STC2(b(OFFR),e(OFFR))
                     ∧ STC3(b(OFFR),e(OFFR))
                     ∧ STC4(b(HIST),e(HIST))
                     ∧ STC5(b(HIST),b(EMP),e(HIST),e(EMP))
                     ∧ STC6(b(REG),e(REG))
                     ∧ STC7(b(REG),e(REG)) }
```

状态转移约束 `STC1` 涉及员工表结构。约束 `STC2` 和 `STC3` 涉及授课表结构。约束 `STC4` 涉及历史表结构。约束 `STC5` 涉及两个表结构：历史表和员工表。最后，约束 `STC6` 和 `STC7` 涉及注册表结构。

清单 8-8 提供了这七个状态转移约束的形式化定义。你将在下一节找到对每一个约束的详细阐述。

**■ 注意** 通常，状态转移约束检查的不仅仅是事务的起始和结束状态。在事务上下文中可用的其他数据项——通常由大多数 DBMS 提供——包括当前系统日期/时间（环境变量 `sysdate`）和当前登录用户的用户名（环境变量 `user`）。需要引用任何这些*事务环境变量*的需求，总是会映射到状态转移约束。静态约束本质上无法引用事务的上下文。清单 8-8 中指定的约束 `STC5` 和 `STC7` 就是这个事实的例子；它们都引用了变量 `sysdate`。

```
STC1(EMPB,EMPE) :=
/* 月薪只能增加 */
( ∀e1∈EMPB, e2∈EMPE: e1(EMPNO) = e2(EMPNO) ⇒ e1(MSAL) ≤ e2(MSAL) )

STC2(OFFRB,OFFRE) :=
/* 新授课必须以状态 SCHED 开始 */
( ∀o1∈OFFRE⇓{COURSE,STARTS} − OFFRB⇓{COURSE,STARTS}:
    ↵{ o2(STATUS) | o2∈OFFRE ∧ o2↓{COURSE,STARTS} = o1↓{COURSE,STARTS} }
    = 'SCHD' )

STC3(OFFRB,OFFRE) :=
/* 有效的授课状态转移是： */
/* SCH -> CONF, SCH -> CANC, CONF -> CANC */
( ∀o1∈OFFRB, o2∈OFFRE:
    (o1↓{COURSE,STARTS} = o2↓{COURSE,STARTS} ∧ o1(STATUS) ≠ o2(STATUS))
    ⇒
    (o1(STATUS);o2(STATUS)) ∈
    { ('SCHD';'CONF'), ('SCHD';'CANC'), ('CONF';'CANC') } )

STC4(HISTB,HISTE) :=
/* 不允许更新历史记录 */
( ∀h1∈HISTB, h2∈HISTE:
    h1↓{EMPNO,UNTIL} = h2↓{EMPNO,UNTIL}
    ⇒
    ( h1(DEPTNO) = h2(DEPTNO) ∧ h1(MSAL) = h2(MSAL) ) )

STC5(HISTB,EMPB,HISTE,EMPE) :=
/* 新历史记录必须准确反映员工更新 */
( ∀h∈(HISTE⇓{EMPNO,UNTIL} − HISTB⇓{EMPNO,UNTIL})⊗HISTE:
    h(UNTIL) = sysdate ∧
    ( ∃e1∈EMPB, e2∈EMPE:
        e1↓{EMPNO,MSAL,DEPTNO} = h↓{EMPNO,MSAL,DEPTNO} ∧
        e2(EMPNO) = h(EMPNO) ∧
        ( e2(MSAL) ≠ e1(MSAL) ∨ e2(DEPTNO) ≠ e1(DEPTNO) ) ) )

STC6(REGB,REGE) :=
/* 新注册元组必须以 EVAL = -1 开始 */
( ∀r1∈(REGE⇓{STUD,STARTS} − REGB⇓{STUD,STARTS})⊗REGE: r1(EVAL) = -1 )

STC7(REGB,REGE) :=
/* 评估的转移必须有效 */
/* 且不能在授课开始日期之前发生 */
( ∀r1∈REGB, r2∈REGE:
    (r1↓{STUD,STARTS} = r2↓{STUD,STARTS} ∧ r1(EVAL) ≠ r2(EVAL))
    ⇒
    ( ( r1(EVAL) = -1 ∧ r2(EVAL) = 0 ∧ r2(STARTS) ≤ sysdate ) ∨
      ( r1(EVAL) = 0 ∧ r2(EVAL) ∈ {1,2,3,4,5} ) ) )
```



##### 状态转换约束规范

###### 1. 约束 STC1：薪资不能降低

状态转换约束 `STC1` 展示了指定经典“薪资不能降低”要求的另一种方式。你可以直接从清单 8-4 中的 `STC1` 推导出来，方法是将存在量词重写为全称量词，然后应用德摩根定律，最后将析取式转换为蕴含式。

###### 2. 约束 STC2 与 STC3：`STATUS` 属性的转换

约束 `STC2` 和 `STC3` 处理 `OFFR` 表设计中 `STATUS` 属性允许的值转换。

- **STC2**：新插入的课程安排应始终具有状态 `'SCH'`（已安排）。
- **STC3**：允许现有课程安排的状态从 `scheduled` 转换为 `'CONF'`（已确认），或从 `scheduled` 转换为 `'CANC'`（已取消）。此外，已确认的安排也可以转换为已取消。

###### STC2 的工作方式

`STC2` 规定新安排状态值的方式如下：
1.  构造终态 `OFFR` 表在唯一标识属性（`COURSE` 和 `STARTS`）上的投影。
2.  对初态 `OFFR` 表进行相同操作。
3.  这两个投影表的差集包含了新插入安排的元组。
4.  全称量化的谓词随后从终态 `OFFR` 表中获取并选择这样一个新安排的 `STATUS` 值，并要求其等于 `'SCH'`。

###### STC3 的工作方式

`STC3` 检查初态和终态中对应 `OFFR` 元组的所有组合。组合是基于唯一标识属性进行的。它要求对于每一个这样的元组组合，要么 `STATUS` 值没有改变，要么根据前面描述的可接受值转换之一发生了改变。

清单 8-9 展示了指定约束 `STC3` 的另一种稍有不同的方式。

**清单 8-9.** `STC3` 的替代规范
```
( ∀o1∈OFFRB, o2∈OFFRE:
    ( o1↓{COURSE,STARTS} = o2↓{COURSE,STARTS} ∧ o1(STATUS) ≠ o2(STATUS) )
    ⇒
    ( ( o1(STATUS) = 'SCH' ∧ o2(STATUS) = 'CONF' ) ∨
      ( o1(STATUS) = 'SCH' ∧ o2(STATUS) = 'CANC' ) ∨
      ( o1(STATUS) = 'CONF' ∧ o2(STATUS) = 'CANC' ) ) )
```
这里我们使用了以下重写规则（留作本章末的练习）：
```
( P ⇒ (Q ∨ R) ) ⇔ ( (P ∧ ¬Q) ⇒ R )
```

###### 3. 约束 STC4：`HIST` 表的历史记录

与 `STC3` 类似，状态转换约束 `STC4` 组合了初态和终态的 `HIST` 元组，并要求 `DEPTNO` 和 `MSAL` 属性值均未改变。

###### 4. 约束 STC5：`HIST` 表的语义

约束 `STC5` 定义了 `HIST` 表设计的语义。如果一个事务更改了 `EMP` 元组的 `DEPTNO` 或 `MSAL` 属性值，那么同一事务应在 `HIST` 表中记录一个元组，记录 `DEPTNO` 和 `MSAL` 的旧值。这个新的 `HIST` 元组应将 `UNTIL` 属性设置为当前系统日期和时间。

`STC5` 对所有新插入的 `HIST` 元组进行全称量化，并要求在同一事务内，`DEPTNO` 和 `MSAL` 值应与在**初态**中找到的匹配 `EMP` 元组的 `DEPTNO` 和 `MSAL` 值相对应。当然，它也要求对于这个特定的 `EMP` 元组，**终态**反映了 `DEPTNO` 或 `MSAL` 属性值（或两者）的变化。

###### 5. `REG` 表的 `EVAL` 属性

`REG` 表设计中 `EVAL` 属性的属性值集总共包含七个元素：-1, 0, 1, 2, 3, 4, 5。其含义如下：
- `-1`：评估为时过早
- `0`：未评估或尚未评估
- `1`：非常差
- `2`：差
- `3`：一般
- `4`：好
- `5`：优秀

评估的有效值转换如图 8-4 所示。

###### 约束 STC6：新注册的初始评估值

新注册必须从评估值 -1 开始（`STC6`）。只要课程安排尚未开始，所有评估都应保持为值 -1；评估该安排为时过早。

###### 约束 STC7：评估值的转换

一旦课程安排“进行中”，培训师可以决定学生何时评估该安排。这是通过将该安排的所有注册的评估设置为值 0 来完成的。由于表宇宙 `tab_REG` 中引入的以下表约束（见清单 7-34），培训师必须在单个事务中完成此操作。

```
/* 课程安排已被评估，或评估为时过早 */
( ∀r1,r2∈R:
    ( r1↓{COURSE,STARTS} = r2↓{COURSE,STARTS} )
    ⇒
    ( ( r1(EVAL) = -1 ∧ r2(EVAL) = -1 ) ∨
      ( r1(EVAL) ≠ -1 ∧ r2(EVAL) ≠ -1 ) ) )
```

此时，注册值为 0 表示学生**尚未**评估该安排。从评估值 -1 到值 0 的转换由状态转换约束 `STC7` 蕴含式结论中的第一个析取项覆盖。该析取项也覆盖了只有在安排已开始（`r2(STARTS) ≥ sysdate`）时才能从 -1 转换为 0 这一事实。

一旦评估被设置为 0，学生可以通过将 0 更改为 1 到 5 之间的值来评估该安排。这些值转换由 `STC7` 蕴含式结论中的第二个析取项覆盖。学生也可以选择完全不进行评估，让评估保持为值 0（现在代表“未评估”）。

###### 6. 约束规范方式总结

你注意到 `STC2`、`STC5` 和 `STC6` 展示了指定新插入元组属性值限制的三种不同方式了吗？你也可以用“`STC2` 方式”或“`STC5` 方式”编写约束 `STC6`；参见清单 8-10。

**清单 8-10.** `STC6` 的替代规范
```
STC6(REGB,REGE) :=
( ∀r1∈REGE⇓{STUD,STARTS} − REGB⇓{STUD,STARTS}:
    ↵{ r2(EVAL) | r2∈e(REGE) ∧ r2↓{STUD,STARTS} = r1↓{STUD,STARTS} }
    = -1 )

STC6(REGB,REGE) :=
( ∀r∈(REGE⇓{STUD,STARTS} − REGB⇓{STUD,STARTS})⊗REGE: r(EVAL) = -1 )
```
如果你仔细研究了清单 8-8 中的规范，可能还会注意到并非所有状态转换约束都满足其 `CNF` 只有一个合取项的要求。“练习”部分的一个练习将涉及此问题。

##### 结论

我们通过重申第 7 章早些时候的一个陈述来结束本章：**信息系统的整体规范中有很大一部分实际上存在于数据库设计的规范中**。状态转换宇宙作为数据库设计规范的一个组成部分，再次证明了这一点。专门处理事务的“业务逻辑”通常可以由状态转换约束表示，因此这部分业务逻辑应该成为数据库设计规范的一部分。

#### 形式化指定查询

数据检索是数据库的关键应用。通过查询，你可以从数据库中提取与你相关的信息。在本节中，我们将演示如何形式化地指定查询。

让我们从一个简单的例子开始。我们将使用第 7 章引入的示例数据库设计。假设 `E` 代表全域 `DB_UEX` 的某个数据库状态中的员工表。以下是一个名为 `Q1` 的在 `E` 上的查询的形式化规范，它提取收入超过 6000 的培训师：

`Q1(E) := { e | e∈E ∧ e(JOB)='TRAINER' ∧ e(MSAL)>6000 }`

查询 `Q1` 返回所有表示收入超过 6000 的培训师的员工元组。

如果你感兴趣的不是完整的元组，而只是收入超过 6000 的培训师的员工号 (`EMPNO`)、姓名 (`ENAME`) 和薪水 (`MSAL`)，那么以下查询 (`Q2`) 将完成这项工作：

`Q2(E) := { e↓{EMPNO,ENAME,MSAL} | e∈E ∧ e(JOB)='TRAINER' ∧ e(MSAL)>6000 }`

**注意** `Q1` 和 `Q2` 都返回一个表作为结果。正如你将很快看到的，这在本书阐述的形式化方法中并非必须；查询也可以返回简单的值（例如，单个数值或字符串值）。然而，在实践中，我们更倾向于返回表的查询规范。

如前面的例子所示，你可以将查询视为一个函数。在这种情况下，你向该函数传递一个表作为其参数，然后它根据该参数返回另一个值。当然，查询通常涉及不止一个表。一般来说，一个查询可以被视为接受一个数据库状态（我们包含所有当前表的形式结构）作为其参数。然后，查询可以检查此数据库状态内的任何可用表，以确定其返回值。



我们同样用一个示例来演示这一点。假设 `dbs` 表示数据库全域 `DB_UEX` 的某个数据库状态。请看查询 `Q3`；它返回雇佣了总裁（如果有的话）的部门：

```
Q3(dbs) := { d | d∈dbs(DEPT) ∧

(∃e∈dbs(EMP): e(DEPTNO)=d(DEPTNO) ∧ e(JOB)='PRESIDENT') }
```

查询 `Q3` 检查状态 `dbs` 中保存的两个表；表达式 `dbs(DEPT)` 表示部门表，表达式 `dbs(EMP)` 表示员工表。因为查询的参数现在代表一个数据库状态，所以该查询可以检查该状态下任何可用的表。

正如这个例子向我们展示的，一个查询可以形式化地视为`数据库全域上的函数`。换句话说，查询是一个函数，其定义域是一组数据库状态。对于每个数据库状态，该函数返回（或者说`计算`）某个值。查询的结果可以是任何类型的值。查询 `Q3` 的结果是一个表。如果有总裁，那么结果将是一个包含一个元组的表；如果没有总裁，那么结果将是空表。数据库查询通常会生成表。事实上，对于查询语言 `SQL`，一个查询总是生成一个表。在本书介绍的形式化方法中，查询的结果也可以简单到单个数字或字符串值。这里有一个例子。查询 `Q4` 返回部门 10 的员工数量：

```
Q4(dbs) := #{ e | e∈dbs(EMP) ∧ e(DEPTNO)=10 }
```

查询 `Q4` 的结果是一个数值。这里是一个返回表的 `Q4` 的形式化规范：

```
Q4(dbs) := { { (DEPT_TEN_EMPS; #{ e | e∈dbs(EMP) ∧ e(DEPTNO)=10 }) } }
```

这个版本的 `Q4` 返回一个基于 `{DEPT_TEN_EMPS}` 的表。它包含一个元组，其中有一个属性-值对。
