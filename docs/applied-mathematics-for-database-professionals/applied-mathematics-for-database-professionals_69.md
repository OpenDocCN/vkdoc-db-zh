## 是否适用

正如你在第 11 章看到的，本书中介绍的示例数据库设计几乎不包含任何*可为空*的属性；所有表结构中除 `OFFR` 中的 `TRAINER` 和 `TERM` 中的 `COMMENTS` 外的所有属性都是强制性的（SQL 术语中的 `NOT NULL`）。在你日常工作中遇到的数据库设计中，你可能会发现更多可为空的属性，对吧？

很可能，一个属性被留作可选的主要原因是，对于某些行，可能存储在其中的值是*不适用的*。让我解释一下。可选属性通常在数据库设计中因以下三个原因之一而引入：

-   可能存储的数据值是*不适用的*。
-   数据值*尚未适用*。
-   这些值*知道更好*（如果知道），但不是强制性的。

接下来的三个小节将简要讨论这些原因。

### 不适用

不适用的数据值源于未能设计特化。这是数据库设计实现中出现可选属性的主要原因。通过不引入单独的表结构来保存仅对给定表结构中特定子集的元组适用的属性，你最终会得到可选属性。

在第 7 章的示例数据库设计中，我们专门引入了 `SREP` 和 `MEMP` 来防止 `EMP` 中出现以下可选属性：

-   `TARGET` 和 `COMM`：仅适用于（子集）销售代表
-   `MGR`：仅适用于被管理的员工；即除总裁外的所有人

通常，你会发现既没有引入 `SREP` 也没有引入 `MEMP`，并且属性 `TARGET`、`COMM` 和 `MGR` 是 `EMP` 表结构中的可为空属性。幸运的话，所有以下四个 `CHECK` 约束会伴随它们：

```
CHECK( JOB<>'SALESREP' or (TARGET is not null and COMM is not null) )
CHECK( JOB='SALESREP' or (TARGET is null and COMM is null) )
CHECK( JOB<>'PRESIDENT' or MGR is not null )
CHECK( JOB='PRESIDENT' or MGR is null )
```

前两个约束表示以下两个蕴含，指定了 `TARGET` 和 `COMM` 属性何时应持有值，何时不应持有：

-   `( JOB = 'SALESREP' ) ⇒ ( TARGET is not null and COMM is not null )`
-   `( JOB <> 'SALESREP' ) ⇒ ( TARGET is null and COMM is null )`

注意，蕴含已被改写为析取。

最后两个 `CHECK` 约束表示指定了 `MGR` 属性何时应持有值，何时不应持有的蕴含：

-   `( JOB <> 'PRESIDENT' ) ⇒ ( MGR is not null )`
-   `( JOB = 'PRESIDENT' ) ⇒ ( MGR is null )`

**注意：** 通常你会发现第二个和第四个 `CHECK` 约束被错误地省略了。

如果它们是 `EMP` 表结构的一部分，上述四个 `CHECK` 约束将实现这些特化属性的适用性要求。它们是示例数据库设计中数据库约束 `PSPEC1` 和 `PSPEC2` 的对应物。

### 尚未适用

尚未适用的数据值也源于未能设计特化。正如我们在第 7 章提到的，示例数据库设计包含两个相当棘手的变通方法。`OFFR` 表结构中的 `TRAINER` 属性和 `REG` 表结构中的 `EVAL` 属性的属性值集中包含一个特殊的 `-1` 值。在这两种情况下，该值用于表示属性值尚未适用。

显然有两种类型的课程：已分配培训师的课程和未分配培训师的课程。以下元组约束取自示例数据库全域规范，描述了何时不需要分配培训师（这里 `o` 代表一个课程元组）：

```
/* 未分配的 TRAINER 仅允许用于已取消和已计划的课程 */
o(TRAINER) = -1 ⇒ o(STATUS)∈{'CANC','SCHD'}
```

注意，此元组约束指定了已确认的课程必须有分配的培训师，而已取消或已计划的课程可以有分配的培训师。

我们可以对注册做类似的评论：一些注册包含课程的评估分数，一些则还没有。在这种情况下，没有约束精确描述注册何时持有评估，何时不持有。表全域 `tab_REG` 中指定的表约束确实指出，如果一个课程的某个注册持有评估，那么该课程的所有注册都应该持有评估。以下是该约束的规范（`R` 代表注册表）：

```
/* 课程要么被所有参与者评估，要么评估该课程为时过早 */
( ∀r1,r2∈R:
( r1↓{COURSE,STARTS} = r2↓{COURSE,STARTS} )
⇒
( ( r1(EVAL) = -1 ∧ r2(EVAL) = -1 ) ∨
( r1(EVAL) ≠ -1 ∧ r2(EVAL) ≠ -1 )
) )
```

在实现这类设计时，你经常会看到形式化规范中的特殊 `-1` 值在 SQL DBMS 中变成了 `NULL`。

然而，在正确设计的数据库中，我们应该为课程表设计和注册表设计都创建一个特化，使得每个表都持有在当前数据库设计中并非总是适用的属性（课程设计的 `TRAINER` 和注册设计的 `EVAL`）。这两个属性的属性值集不包含特殊的 `-1` 值，在实现中，这些特化属性将成为强制属性。对于课程特化，你还应该指定一个适当的数据库约束来替换当前描述何时必须分配培训师的元组约束。

### 知道更好

在数据库设计中引入可选属性的第三个原因是前一个原因的微妙变体。当可选属性因第二个原因引入时，你通常会在元组、表或数据库级别找到一个或多个数据完整性约束，精确描述了属性值何时确实适用。在没有任何此类约束的情况下，我们将可选属性的值视为“知道更好”。

在示例数据库设计的实现中，`TERM` 表结构的 `COMMENTS` 属性就是这一类别的一个例子。用户显然只是允许该属性缺少值。

### 实现指南

如果一个可选属性属于“知道更好”的类别（因此不涉及任何约束），那么该属性可能也不涉及任何业务逻辑。它可能主要是在报告中使用的属性，只是为了打印在纸上或显示在屏幕上。因此，存储在列中的 `NULL` 很少会引起下一节描述的关于使用 `NULL` 的各种问题。处理可选属性更方便，而不是创建一个单独的特化表，将该属性作为强制属性保存，这需要使用外连接来始终尝试检索该值。

**注意：** 如果你不熟悉外连接的概念，请参阅 Oracle 文档。

我们认为，不适用的数据值——由第一个原因引入的——应该始终避免。我们这样做的主要论据（除了避免下一节描述的问题）是，为这些属性引入特化有利于整个数据库设计的清晰度。当不仅仅是一个属性，而是一整组属性需要转移到一个或多个特化表时，尤其如此。

尚未适用数据值（第二个原因）的情况常常引发关于应如何处理的激烈辩论。从真正的关系角度来看，你再次需要通过创建特化并伴随相关约束来避免它们，这些约束描述了何时元组应出现在特化中。如果你决定不这样做，那么你不应既不指定也不实现描述值何时确实适用何时不适用的伴随约束。

## 三值（或更多值）逻辑

在本节中，Lex 简要探讨了 3VL。他向你介绍了 `AND`、`OR` 和 `NOT` 这三个连接词的三值真值表。他进一步研究了 3VL 中蕴含连接词的一些问题，以及 3VL 中出现的各种其他问题。

### 未知

3VL 中最关键的假设是，除了两个值 `TRUE` 和 `FALSE`，还有第三个值表示“可能”或（仍然）`UNKNOWN`。为避免误解，让我们从以下陈述开始：三值（或多值）逻辑本身没有问题。

许多数学家过去探索过这个领域；其中值得一提的是 Jan Łukasiewicz（1878–1956）。

**注意：** Łukasiewicz 还引入了*波兰表示法*，允许在不使用括号的情况下无歧义地编写表达式。这是*逆波兰表示法*（RPN）的基础，许多袖珍计算器和表达式编译器都使用它。

在 3VL 中，谓词的真值可以随时间变化，但只能从 `UNKNOWN` 变为 `TRUE` 或从 `UNKNOWN` 变为 `FALSE`。这相当微妙，因为在 3VL 中，值 `UNKNOWN` 似乎有些特殊。无论如何，这个描述太模糊了；我们需要正式的规则、定义和真值表。我们稍后会看这些细节。

3VL 是反直觉的，与经典的二值逻辑（2VL）相反。主要原因是失去了*排中律*（tertium non datur）。也就是说，如果 `P` 是未知的，那么 `¬P` 也是未知的，这反过来意味着 `P ∨ ¬P` 是未知的。考虑以下著名的亚里士多德例子。“明天将发生一场海战”既不是 `TRUE` 也不是 `FALSE`——它是未知的——但句子“要么明天发生海战，要么明天不发生海战”在现实中肯定是真的。然而，在 3VL 中它是未知的，因此具有反直觉的性质。

另一个问题源于这样一个事实：如果两个谓词是“可能的”（即它们的真值可能为 `TRUE`），那么它们的合取也是“可能的”。然而，如果其中一个是第二个的否定，这显然是错误的。Ludwik Stefan Borkowski（1914–1993）提出通过引入四值逻辑（4VL）来“修复”这个问题；这确实使前面的问题消失了，但这个解决方案不能令人满意且是临时的。

此外，3VL 并*没有*解决处理缺失信息的问题；它不允许你表示信息缺失的原因。在本节末尾，你会看到 Ted Codd 在 1990 年提出了 4VL，以区分适用和不适用的值。然而，4VL 并没有告诉你，如果你不知道某个属性值是否适用该怎么办；例如，如果佣金只适用于销售代表，但你不知道某个员工的工作，你该如何处理该员工的佣金属性？一个 `NULL` 意味着 3VL，两个 `NULL`（适用和不适用）意味着 4VL，n 个 `NULL` 意味着（n+2）值逻辑。使用完全归纳法，你可以证明你最终会得到无限多个（含义的）`NULL`，以及相应的无限多个真值。重复本段开头的部分句子：3VL 并*没有*解决处理缺失信息的问题。

更糟的是，3VL 不仅本身反直觉；它在 SQL 标准中的实现也很糟糕。在这方面，SQL 标准中可能最大的错误之一是，在 `BOOLEAN` 数据类型的属性中，`NULL` 代表逻辑值 `UNKNOWN`。`NULL` 不是用来表示你不知道值的事实吗？SQL 标准中还有更多此类错误的例子。

你必须重新审视 2VL 中的所有重言式（以及所有重写规则），以检查它们在 3VL 中是否仍然有效；你会发现几个重言式不再成立。

实际上，你甚至可以争辩说，我们应该为 3VL 中的连接词使用不同的名称和符号；毕竟，它们与 2VL 中的“相应”连接词是不同的。

### 三值逻辑的真值表

表 D-1 到 D-3 显示了 3VL 中否定、合取和析取连接词的三个标准真值表。

**表 D-1.** `NOT（否定）的三值真值表`

| P | ¬P |
| :--- | :--- |
| T | F |
| U | U |
| F | T |

如果 `P` 是 `TRUE`，则 `¬P` 仍然是 `FALSE`，反之亦然。然而，否定不再是补运算符，因为 `UNKNOWN` 的否定是 `UNKNOWN`。

**表 D-2.** `AND（合取）的三值真值表`

| P | Q | P ∧ Q |
| :--- | :--- | :--- |
| T | T | T |
| T | U | U |
| T | F | F |
| F | T | F |
| F | U | F |
| F | F | F |
| U | T | U |
| U | U | U |
| U | F | F |

如你所见，仅当 `P` 和 `Q` 都为 `TRUE` 时，`P ∧ Q` 才为 `TRUE`。如果至少有一个操作数为 `FALSE`，则结果为 `FALSE`，其余三种组合的结果为 `UNKNOWN`。

**表 D-3.** `OR（析取）的三值真值表`

| P | Q | P ∨ Q |
| :--- | :--- | :--- |
| T | T | T |
| T | U | T |
| T | F | T |
| F | T | T |
| F | U | U |
| F | F | F |
| U | T | T |
| U | U | U |
| U | F | U |

仅当 `P` 和 `Q` 都为 `FALSE` 时，`P ∨ Q` 才为 `FALSE`。如果至少有一个操作数为 `TRUE`，则结果为 `TRUE`，再次有三种结果为 `UNKNOWN` 的组合。

### 缺失的运算符

你几乎从未见过 3VL 中蕴含和等价的真值表。缺失蕴含的最可能原因是很难给出一个好的定义。事实证明，在 3VL 中，蕴含连接词不能用合取和否定，或析取和否定来定义。记住 2VL 中的以下蕴含重写规则：

`( P ⇒ Q ) ⇔ (¬P ∨ Q )`

如果你根据这个重写规则在 3VL 中定义蕴含，你会遇到这样的情况：如果 `P` 是 `UNKNOWN`，则 `P ⇒ P` 将计算为 `UNKNOWN`；显然，在任何逻辑中你都希望这是 `TRUE`。表 D-4 的最后一行显示了这个问题。

**表 D-4.** `定义三值蕴含和等价连接词的尝试`

| P | Q | P ⇒ Q | P ⇔ Q |
| :--- | :--- | :--- | :--- |
| T | T | T | T |
| T | F | F | F |
| T | U | U | U |
| F | T | T | F |
| F | F | T | T |
| F | U | T | U |
| U | T | T | U |
| U | F | U | U |
| U | U | U? | U? |

同样，如果 `P ⇒ Q` 且 `Q ⇒ P`，你希望 `P` 等价于 `Q`。这些问题都与否定不再是补运算符这一事实有关。实际上，Jan Łukasiewicz 提出了一个蕴含的真值表，其中最后一行中的两个 `U?` 值都是 `T`——因此他通过牺牲一个重言式（蕴含重写规则）“修复”了这个问题。

### 三值逻辑、重言式和重写规则

如果你想使用我们在 2VL 中确定的任何重写规则，你必须先根据 3VL 的真值表进行检查；你会发现 2VL 中的几个重言式和重写规则在 3VL 中不一定有效。列表 D-1 显示了此类重言式的两个例子。

**列表 D-1.** `在 3VL 中无效的 2VL 重言式示例`

```
P ∨ ¬P
P ⇔ P
```

### 处理三值逻辑

考虑以下两个相当明显的陈述（在 SQL 上下文中）：

-   一行出现在表中（或查询结果中）或没有出现。
-   约束被违反或没有被违反。

没有其他可能性；因此，这些都是二值问题。然而，在 3VL 中，谓词有三种可能的结果：`TRUE`、`FALSE` 或 `UNKNOWN`。不幸的是，对于这第三种可能的结果，没有直观的解释。为了让你了解不一致性，请看表 D-5，其中你看到了计算谓词（比如 `P`）在 SQL 查询的 `WHERE` 子句中与检查 SQL `CHECK` 约束的谓词时的区别。

**表 D-5.** `SQL 中的谓词评估：查询 vs. 约束`

| P(row) | `WHERE P(row)` | `CHECK(P(row))` |
| :--- | :--- | :--- |
| T | 行被接受 | 行满足约束 |
| F | 行被拒绝 | 行违反约束 |
| U | 行被拒绝 | 行满足约束 |

如你所见，对于查询中的 `WHERE` 子句，`FALSE` 和 `UNKNOWN` 导致相同的结果，而对于约束，`TRUE` 和 `UNKNOWN` 导致相同的效果。上表中的第三行显示了差异；当谓词是 `WHERE` 子句的一部分时，一行将违反谓词（因此不被检索）。然而，当该行被定义在接收该行的表上的 `CHECK` 约束评估时，该行将满足该谓词。

如果你想在符合 ISO 标准的 SQL 中控制 3VL，考虑表 D-6 中引入的三个 SQL 函数是一个好主意。它们允许你在 3VL 之上创建一个 2VL 层，从而更明确地控制三个布尔值 `TRUE`、`FALSE` 和 `UNKNOWN`。

**表 D-6.** `SQL 中的 IS {TRUE|FALSE|UNKNOWN} 运算符`

| P | `IS TRUE (P)` | `IS FALSE (P)` | `IS UNKNOWN (P)` |
| :--- | :--- | :--- | :--- |
| T | T | F | F |
| F | F | T | F |
| U | F | F | T |

注意，`UNKNOWN` 没有出现在表 D-6 的最后三列中。因此，这三个运算符使你能够将三值表达式映射到 2VL。

列表 D-2 显示了基于表 D-6 中定义的运算符的一些重写规则。

**列表 D-2.** `使用 IS 运算符在 3VL 中的一些重写规则`

```
IS TRUE ( P ) ⇔ ( P ∧ ¬ ( IS UNKNOWN (P) ) )
IS FALSE ( P ) ⇔ (¬P ∧ ¬ ( IS UNKNOWN (P) ) )
IS TRUE ( ¬P ) ⇔ ( IS FALSE (P) )
IS FALSE ( ¬P ) ⇔ ( IS TRUE (P) )
IS TRUE ( P ∧ Q ) ⇔ ( IS TRUE (P) ∧ IS TRUE (Q) )
IS TRUE ( P ∨ Q ) ⇔ ( IS TRUE (P) ∨ IS TRUE (Q) )
IS FALSE ( P ∧ Q ) ⇔ ( IS FALSE (P) ∨ IS FALSE (Q) )
IS FALSE ( P ∨ Q ) ⇔ ( IS FALSE (P) ∧ IS FALSE (Q) )
IS TRUE ( ∃x∈S: P ) ⇔ ( ∃x∈S: (IS TRUE (P) ) )
IS FALSE ( ∃x∈S: P ) ⇔ ( ∀x∈S: (IS FALSE (P) ) )
IS FALSE ( ∀x∈S: P ) ⇔ ( ∃x∈S: (IS FALSE (P) ) )
IS UNKNOWN ( ∃x∈S: P ) ⇔ ( ¬∃x∈S: ( IS TRUE (P) ) ∧ ∃y∈S: (IS UNKNOWN (P) ) )
IS UNKNOWN ( ∀x∈S: P ) ⇔ ( ¬∃x∈S: ( IS FALSE (P) ) ∧ ∃y∈S: (IS UNKNOWN (P) ) )
```

你可以使用三值真值表或使用你之前证明的 3VL 重写规则的组合来证明所有这些重言式。

### 四值逻辑

在 E. F. Codd 的 *The Relational Model for Database Management: Version 2*（Addison-Wesley, 1990）中，他提出了对第一个关系模型版本 RM/V1 的修订。早些时候，在 1979 年，他在塔斯马尼亚发表了一篇题为“Extending the Database Relational Model to Capture More Meaning”的论文，将扩展版本命名为 RM/T（T 代表塔斯马尼亚）。RM/T 的特性本应逐渐纳入 RM/V2、RM/V3 等系列版本中。

RM/V2 书中最具争议的部分是第 8 章（“缺失信息”）和第 12.4 节（“缺失信息的处理”）。在这些部分，Codd 提出了一个 4VL，试图区分信息缺失的两个最常见原因：适用和不适用，分别用两个可区分的值 `A` 和 `I` 表示。他提供的真值表如表 D-7 所示。

**表 D-7.** `NOT、OR 和 AND 的四值真值表`

| P | ¬P |
| :--- | :--- |
| T | F |
| A | A |
| I | I |
| F | T |

| | | **Q** | | |
| :--- | :--- | :--- | :--- | :--- |
| **P** | **T** | **A** | **I** | **F** |
| **T** | T | T | T | T |
| **A** | T | A | A | A |
| **I** | T | A | I | F |
| **F** | T | A | F | F |

| | | **Q** | | |
| :--- | :--- | :--- | :--- | :--- |
| **P** | **T** | **A** | **I** | **F** |
| **T** | T | A | I | F |
| **A** | A | A | I | F |
| **I** | I | I | I | F |
| **F** | F | F | F | F |

注意，最后两个真值表使用的格式与你迄今为止在本书中看到的所有其他真值表略有不同。它们被格式化为一个矩阵，其中四行代表 `P` 的四种可能真值，四列分别代表 `Q` 的四种可能值。这样做是因为析取和合取连接词是可交换的；在常规真值表格式中，这两个真值表每个都需要十六行。

注意，引入两个可区分的值 `A` 和 `I` 也意味着你必须重新审视算术运算符和字符串连接的结果；Ted Codd 提出了列表 D-3 所示的行为，其中 `x` 表示常规数字属性值，`s` 表示常规字符串值。

**列表 D-3.** `4VL 中重新审视的算术运算符和连接`

```
a + a = a
a || a = a
i + i = i
i || i = i
x + a = a
s || a = a
a + i = i
a || i = i
x + i = i
s || i = i
```

如前所述，引入两个可区分的值（`a` 和 `i`）来替换 `NULL`（以及相应的 4VL）必须被视为 Ted Codd 少数错误之一，事后看来。但事后批评总是容易的；Ted Codd 原始论文的重要性再怎么强调也不为过。
