# 事务中延迟约束的执行模型

## 在事务中处理约束违反

如果一个 DML 语句（例如`DML1`）在事务中引入了违反约束的情况，那么在该事务结束前，必须有一个后续的 DML 语句（例如`DML2`）来纠正`DML1`引入的违反。在执行`DML1`时，你可能会：

*   根本不想执行涉及的 DI 代码，而是将其安排在事务结束时执行，或者
*   让涉及的 DI 代码以某种方式执行，使其仅在检测到违反时才被安排在事务结束时*重新执行*。

在上述两种情况下，如果 DI 代码重新执行时约束仍然被违反，那么显然应该禁止该事务提交。

那么，如何安排 DI 代码在事务结束时执行呢？实际上，你无法直接实现这一点；在 Oracle 的 DBMS 中，没有办法达到此目的。一个能够让你实现此功能的本应是*提交触发器*的概念。提交触发器会在提交之前、作为系统提交过程的一部分被触发，并且它可以检查事务即将提交的最终状态。通过将 DI 代码嵌入此触发器中，你可以重新检查后续 DML 语句是否已解决所有临时的约束违反。只有在这种情况下，触发器才允许提交过程成功。不幸的是，DBMS 并未提供提交触发器的概念。

然而，有另一种方法可以让你重新执行暂时被违反的约束的 DI 代码。在本节剩余部分，我们将概述如何修改执行模型`EM6`以支持延迟检查。

## 修改执行模型以支持延迟检查

请看表 11-6。它描述了一个执行了四条 DML 语句的事务。

`DML1`涉及约束`C1`，该约束已被标识为可延迟约束。

**表 11-6.** 重新执行延迟约束`C1`的 DI 代码

| TX | 注释 |
| :--- | :--- |
| `DML1;` | 涉及约束`C1`；DI 代码触发并发现`DML1`违反了它。DI 代码允许此操作。 |
| `DML2;` | 其他约束的 DI 代码执行。`C1`的 DI 代码被重新执行；发现`C1`仍然违反。 |
| `DML3;` | 其他约束的 DI 代码执行。`C1`被重新检查；DI 代码现在发现`C1`已满足。 |
| `DML4;` | 其他约束的 DI 代码执行。不再重新检查`C1`。 |
| `COMMIT;` | |

`DML1`语句引入了对约束`C1`的违反。由于`C1`已被标识为可延迟的，其 DI 代码不会引发错误以强制回滚`DML1`。相反，它会将`C1`当前被违反的事实信息存储在此事务上下文中的某处。

当执行`DML2`时，可能涉及其他各种（不可延迟的）约束，这些约束的 DI 代码会相应地执行。在我们修改后的执行模型中，随后会检查事务的上下文，以找出某些约束当前是否被违反。如果发现这样的约束，那么修改后的执行模型现在也会重新执行此约束的 DI 代码。如果重新检查时发现约束仍然违反，则上下文保持不变。如果重新检查时发现约束已满足，则上下文将被修改以反映约束`C1`不再违反。

在上述场景中，语句`DML3`修复了约束`C1`的违反。当执行`DML4`时，执行模型现在不再为`C1`重新执行 DI 代码。

上述场景向你展示了，你可以使用后续 DML 语句触发的触发器来重新检查先前 DML 语句违反的可延迟约束。

现在你还面临一个挑战。当事务上下文仍持有表明一个或多个可延迟约束处于违反状态的信息时，能否阻止提交成功执行？答案是肯定的，你可以。



如果针对某个给定的可延迟约束的 DI 代码发现该约束被违反，它可以将此信息存储在一个会话临时表中。而如果 DI 代码发现该约束再次得到满足，它就会从会话临时表中删除相关的记录。你可以这样设置这个会话临时表：只要该表中存有记录，事务就无法成功提交。请看**清单 11-42**，它定义了这个会话临时表。

**清单 11-42.** *用于存储临时违反情况的表*

```sql
create global temporary table current_violations
(constraint_name varchar(30) not null
,constraint all_satisfied_at_commit check(0=1) initially deferred
,constraint curvio_pk primary key(constraint_name))
on commit preserve rows;
```

你看出这里使用的技巧了吗？这个会话临时表（`on commit preserve rows`）和 `all_satisfied_at_commit` 约束（`initially deferred`）的设置方式，允许 DI 代码向 `current_violations` 表中插入记录。然而，与此同时，当该表中仍存有记录时，它禁止事务成功提交，从而防止在可延迟约束被违反时事务得以提交。

正如你可能同意的那样，引入延迟执行会使执行模型变得相当复杂：

-   可延迟约束的 DI 代码现在必须适当地向 `current_violations` 表插入和删除记录。
-   你必须用过程代码扩展所有的 DI 代码，以重新检查当前在 `current_violations` 表中注册的所有约束。为了能够执行重新检查而无需复制可延迟约束的 DI 代码，你需要将当前的 DI 代码从触发器主体中移至存储过程；这使你可以从其他触发器中调用此代码。
-   你可能希望使用比迄今为止概述的更高效的执行模型。

当前，一个可延迟约束（即处于违反状态的）会在每条后续的 DML 语句上被重新检查。对于一条给定的后续 DML 语句，你可以推断出重新检查约束是否合理。例如，如果一条后续 DML 语句操作的表并未涉及该可延迟约束，那么此 DML 语句永远无法恢复该可延迟约束的违反状态。为了防止不必要的重新检查，你只应在后续 DML 语句有可能恢复违反状态时才运行重新检查。

**注意：** 事实上，你可以通过使用过渡效应来防护对可延迟约束的 DI 代码的这种重新执行。你不再需要查询过渡效应以验证一条 DML 语句是否可能违反约束，而是做相反的事情：查询过渡效应以验证一条 DML 语句是否可能恢复约束。

这就结束了对延迟检查执行模型的概述。我们以关于可延迟约束的一个重要观察来结束本节。

允许约束在事务内被暂时违反存在一个严重问题。你面临着从这些事务中执行的查询获得不正确结果的风险。例如，假设约束 `PSPEC1` 当前被违反，原因是当前事务中向 `EMP` 表结构插入了一个新的销售代表，但尚未随后在 `SREP` 表结构中进行相应的插入。现在假设你想确定销售代表的数量。当你编写数据检索语句时，你通常假设所有约束都得到满足。在这种情况下，有两种方法可以找出销售代表的数量：

```sql
select count(*)
from EMP
where JOB='SALESREP';
```

```sql
select count(*)
from SREP;
```

请注意，当 `PSPEC1` 以刚刚描述的方式被违反时，第一个 `SELECT` 表达式将返回正确结果，而第二个 `SELECT` 表达式将返回不正确的结果。实际上，在给定的中间数据库状态中，你可能会质疑销售代表的数量是否被明确定义，因为两个本应等价的查询表达式返回了不同的结果。当你允许约束被暂时违反时，获得不正确的结果是一个严重问题。

**注意：** 防止此问题的真正解决方案是在 SQL 语言中添加*多表更新*的概念。我们建议你参考 Chris Date 和 Hugh Darwen 在此主题上撰写的论文（见附录 C）。

在探讨了关于在触发式过程方法中实现 DI 代码的各种问题后，我们通过简要介绍一个可以帮助你实现 DI 代码的框架来结束本章。

## RuleGen 框架

在本章中看到了 DI 代码的各种示例后，你可以想象，随着你实现的约束数量增长，维护它们可能成为一个相当大的挑战。我们的示例数据库设计只有大约 50 个多元组约束。

真实的数据库设计通常有数百甚至超过一千个多元组约束，其中大多数无法以声明方式指定给 DBMS。

对于你实现的每一个约束，你都在一遍又一遍地重复大量代码；每个约束不同的部分是*TE 查询*、*验证查询*和*序列化代码*。如果能针对给定的约束只注册这三部分，然后由某个软件为你生成所有必需的行级和语句级触发器，那岂不是很好？

在过去几年中，作者之一 Toon Koppelaars 开发了一个名为 `RuleGen` 的框架，正是做这件事。`RuleGen` 实现了执行模型 `EM6`，包括为满足可延迟约束所需概述的增强功能。

你通过填充其存储库中的几个表在 `RuleGen` 中注册一个约束。这涉及关于约束、其涉及的表和涉及的列、TE 查询、验证查询以及序列化代码的信息。给定这些信息，`RuleGen` 将为每个涉及的表完全生成必要的行级和语句级触发器。行级触发器将维护语句级触发器所使用的过渡效应。语句级触发器将验证必要的约束。

到本书出版时，我们期望已经可以获取关于 `RuleGen` 框架的更多信息。如果你对这个框架感兴趣，你可以在 `http://www.rulegen.com/am4dp` 找到最新信息、文档和论文。我们还将在该站点上维护本书所述示例数据库宇宙中涉及的所有约束的 DI 代码的下载。


#### 章节总结

本节提供了本章内容的总结，以项目符号列表形式呈现。

*   你实现数据库设计，通常是为了在其之上构建业务应用程序。这些应用程序通常是**数据窗口**（`WoD`）应用程序。用户通过这些应用程序查询数据并进行交易。
*   一个`WoD`应用程序的所有代码可以分为三类：用户界面代码（`UI` 代码）、业务逻辑代码（`BL` 代码）和数据完整性代码（`DI` 代码）。`UI` 代码创建用户所见的用户界面，并响应用户在用户界面中发起的事件。`DI` 代码负责在用户更改数据库中的数据时，持续维护所有数据完整性约束的有效性。`BL` 代码负责组合和执行查询及交易。
*   本章的重点是如何以高效的方式实现 `DI` 代码。
*   你可以使用以下三种策略之一来实现 `DI` 代码：声明式、触发过程式或嵌入过程式。
*   你可以声明所有属性和元组约束。但只有少数表和数据库约束可以声明。
*   大多数（多行）数据完整性约束必须以过程式实现。在本章中，触发过程式策略优于嵌入过程式策略。
*   我们向你介绍了六种用于实现多元组约束的 `DI` 代码的执行模型。这些模型的效率范围很广，从相当低效（对每个 `DML` 语句都完整检查每个约束）到相当高效（以最小化方式有条件地检查约束）。
*   鉴于 Oracle 标准的 `读已提交` 隔离级别，你必须以程序方式序列化 `DI` 代码。未能这样做可能导致事务并发执行时发生约束违反。序列化过渡约束的 `DI` 代码尤其困难。
*   某些约束无法在语句级别进行验证；它们需要 `DI` 代码的延迟执行模型。扩展执行模型以适应 `DI` 代码的延迟执行并非易事。你已经看到了一个实现此目标的大纲，这涉及建立一个用于记录临时违规的中央表。
*   如果你有很多需要触发过程式实现的数据完整性约束，那么 `RuleGen` 框架可以帮助你管理所有这些约束的 `DI` 代码。

第 2 章提供了集合论的入门介绍。我们介绍了几种指定集合的方法，讨论了子集的概念，并探讨了常见的集合运算符：并集、交集和差集。本章最后处理了幂集和有序对。正如你在本书第二部分所见，集合论提供了一种优秀的语言，可以可靠地描述复杂的数据库设计、数据检索和数据操作。

第 3 章继续了第一章开始的逻辑学讨论。它向你介绍了全称量化和存在量化的关键概念，并指出了关于这两个量词的重要重写规则。其中一个重写规则表明，你可以将存在量词转换为全称量词，反之亦然——这个重写规则在本书中已多次应用。

第 4 章延续了第 2 章建立的集合论基础，并介绍了该领域的一些更多概念。你看到了如何使用函数来表示元组。我们还展示了如何使用一种特殊的函数，即集合函数，来表征现实世界中需要在数据库中表示的事物。

第 5 章和第 6 章演示了如何应用集合论和逻辑来描述数据库领域中的重要概念。我们使用这些数学学科来形式化地描述以下内容：

第 9 章和第 10 章讨论了数学在指定数据检索（查询）和数据操作（事务）领域的应用。

最后，在第 11 章，我们探讨了在知名的 `SQL` `DBMS`（`Oracle`）中实现数据库设计——特别是其所有数据完整性约束——所面临的挑战。正如你所见，对于某些类别的约束，实现是微不足道的（它们可以声明给 `DBMS`），但对于其他类别的约束，实现却是一项复杂的任务。实现高效的执行模型、序列化事务，有时还需要延迟执行约束检查，是使这项任务成为严峻挑战的几个方面。

## 第 12 章：总结与结论

本书的独特之处体现在其标题中：面向数据库专业人员的应用数学。我们已经描述了两种数学学科，它们可以作为数据库专业人员的高质量工具集。我们还描述了如何在数据完整性约束领域应用这些工具集。

数据完整性约束通过描述表结构如何表示我们所工作的现实世界，为数据库设计所捕获的数据增加了语义价值。因此，数据完整性约束规范是数据库设计规范中不可或缺的一部分。本书描述的形式化方法使我们能够精确地指定数据库设计，特别是精确地指定所有涉及的数据完整性约束。

以下引述重申了数据完整性约束的重要性：应该明确，完整性约束至关重要，因为它们控制着数据的正确性。事实上，在许多方面，完整性约束是系统中最重要的部分。

C. J. 戴特，《是什么，而非如何》（艾迪生-韦斯利出版社，2000 年）

为什么形式化的约束规范如此重要？

嗯，如果我们使用普通英语，或其某种笨拙的派生语言来表达数据完整性约束，我们不可避免地会遇到这样的问题：英语句子如何明确无误地映射到数据库设计中。不同的软件开发人员会以多种方式实现这些规范，因为他们都试图将句子——每个人都以自己的方式——转换成某种可以映射到数据库设计中的东西，然后进行编码。

任何非形式化的语言都必然是模糊的。这是无法解决的。试图用非形式化或自然语言来捕获数据完整性约束，总是无法清晰地揭示这种约束如何映射到数据库设计中，因为从非形式化的自然语言到数据库设计的形式化世界之间不存在映射。除非我们采用形式化方法，否则这个数据完整性约束问题是不可避免的。使用形式化语言，我们可以明确地描述约束如何映射到数据库设计中。

在 IT 专业领域，我们应该认识到数据库设计是受过适当教育的数据库专业人员的任务。他们的教育必须包含足够的集合论和逻辑知识，包括这些学科如何应用于数据库设计领域。本书旨在提供这种形式化的教育。

作为数据库专业人员，你如何开始在工作中应用这些知识？这里有一个你可以采用的小路线图。

1.  每当你设计下一个数据库时，首先以形式化的方式指定所涉及的数据完整性约束。将它们指定为数据完整性谓词，并采用第 7 章和第 8 章介绍的分类方案。
2.  将这种形式化方法也应用于查询，尤其是那些更复杂的查询。在形式化世界中使用重写规则，得出易于转换为 SQL 的表达式。
3.  最后，你也可以继续以形式化方式指定事务。这应该能避免在开发软件以实现“网络世界”应用程序的业务逻辑时出现的所有歧义。

你将从第 1 步获得最大的收益。它确保了数据库设计含义（语义）有文档化的唯一真相，这反过来又确保所有软件开发人员都会理解并因此以相同的方式使用该数据库设计。

一旦你形式化地指定了数据库设计，一个好的数据库设计实现就变得可能了。我们已经讨论了实现每个数据库设计重要部分的各种策略：即数据完整性约束。

以声明方式实现数据约束很容易。以过程方式实现数据约束绝非易事（如第 11 章所述）。这很耗时，并且会产生比你预期更多的复杂过程代码行（尽管其中一部分可以生成）。当前的 DBMS 技术状况，如 Oracle 的 SQL DBMS，使我们能够以健壮的方式实现数据库设计；也就是说，通过触发过程方式包含所有数据完整性约束的 DI 代码。

尽管如此，很少有数据库专业人员真正这样做。为什么？可能是因为设计完全独立于 BL 代码的 DI 代码（触发过程策略）在当前可用的 DBMS 状态下确实非常复杂。然而，这忽略了全局。未能将 DI 代码与 BL 代码完全分离，意味着无法有效地维护和管理 DI 代码，从而无法有效地管理这些约束。

我们不应低估一旦所有 DI 代码被单独实现后所能获得的收益。数据完整性约束最终变得可管理了。对于 DBMS 供应商来说，有一个有趣的机会将其产品发展为数据完整性约束引擎。

这首先取决于科学界提出更有意义的约束子类。

然后，DBMS 供应商应该为我们提供新的声明式构造，在 DBMS 级别轻松实现这些约束。

一旦有更多的声明式构造可用，不仅存在着快速构建“网络世界”应用程序的巨大潜力，而且还有潜力在维护此类应用程序的成本上实现相当大的节约。
