# 第 6 章：元组、表与数据库谓词

本节关于表谓词的内容，我们将以另一个例子作为结束，该例子引入了另外三个表谓词。代码清单 6-4 首先引入了一个建模员工的特征描述 `chr_EMP`。

**代码清单 6-4.** `特征描述 chr_EMP`

```
chr_EMP := { (empno; [100..999])
            ,(ename; varchar(20))
            ,(job; {'MANAGER','CLERK','TRAINER'} )
            ,(sal; [2000..9000])
            ,(deptno; [10..99]) }
```

对于一名员工，其员工编号 (`empno`)、姓名 (`ename`)、职位 (`job`)、薪水 (`sal`) 以及所属部门的编号 (`deptno`) 是该特征描述所捕获的相关属性。利用这个特征描述，你可以通过从 `∏(chr_EMP)` 中选择一个子集来生成一个员工表。这样的表可以作为参数传递给表谓词。代码清单 6-5 引入了三个可以通过参数 `T` 接受基于 `dom(chr_EMP)` 的表作为输入的表谓词。

**代码清单 6-5.** `更多表谓词示例`

```
P8(T) := ¬( ∃e1,e2∈T: ( (e1(job)='MANAGER' ∧ e2(job)='TRAINER') ∨
                       (e1(job)='TRAINER' ∧ e2(job)='CLERK' ) ) ∧
             e1(sal) ≤ e2(sal) )
P9(T)  := ( ∀d∈T⇓{deptno}: #{ e2| e2∈T ∧ e2(deptno)=d(deptno) } ≥ 3 )
P10(T) := ( ∀d∈T⇓{deptno}: ( Σe2∈{ e| e∈T ∧ e(deptno)=d(deptno) }: e2(sal) ) ≤ 40000 )
```

表谓词 `P8` 阐述了经理的收入高于培训师，而培训师的收入高于职员。谓词 `P9` 阐述了每个部门至少应有三名员工。谓词 `P10` 阐述了部门的薪资预算不能超过 `40000`。

图 6-2 展示了一个名为 `EMP1` 的 `∏(chr_EMP)` 子集示例。

**图 6-2.** `示例表 EMP1`

请自行验证 `EMP1` 确实是 `∏(chr_EMP)` 的一个子集，并且它满足谓词 `P8`、`P9` 和 `P10`。

**注意：** 鉴于特征描述 `chr_EMP` 中属性 `sal` 的取值集合，最后一个表谓词 `P10`（假设它可被视为一个数据完整性约束）意味着单个部门的员工数永远不可能超过 `20`，因为 `20` 乘以 `2000`（可能的最低薪资）等于 `40000`（最高的部门薪资预算）。在信息分析过程中，向最终用户提出此类*可推导的*陈述通常非常有用。它可以作为对已建立的数据完整性约束的双重检查，或者更可能的是，它会促使您获得最终用户尚未告知的更多信息。

#### 数据库谓词

现在我们继续通过进一步扩大谓词所处理数据的范围来进行探讨。在元组谓词（数据范围是一个元组）和表谓词（数据范围是一张表）之后，下一个层级是处理*多张表*的谓词。我们将这种谓词称为*数据库谓词*。一个数据库谓词有一个类型为数据库状态的参数。

为了展示几个处理数据库状态的谓词示例，我们首先引入一个部门特征描述，与之前引入的员工特征描述并列。

代码清单 6-6 定义了建模部门的特征描述 `chr_DEPT`。

**代码清单 6-6.** `特征描述 chr_DEPT`

```
chr_DEPT := { (deptno; [10..99])
             ,(dname; varchar(20))
             ,(loc; {'LOS ANGELES','SAN FRANCISCO','DENVER'} )
             ,(salbudget; [25000..50000]) }
```

对于一个部门，其部门编号 (`deptno`)、名称 (`dname`)、所在地 (`loc`) 以及薪资预算 (`salbudget`) 是该特征描述所捕获的相关属性。
