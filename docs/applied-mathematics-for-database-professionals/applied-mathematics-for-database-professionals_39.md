# 第七章：规范数据库设计

在本章中，我们将演示如何形式化地规范一个数据库设计。

形式化数据库设计规范的优势在于，不仅能避免数据库结构文档中的任何歧义，更重要的是，还能避免所有涉及的数据完整性约束的歧义。

■注意 在 IT 行业中，术语 `业务规则` 常用来指代本书所称的 `数据完整性约束`。然而，由于很少有对业务规则确切定义的清晰说明，我们对此无法确定。在本书中，我们倾向于不使用业务规则这个术语，而是使用数据完整性约束。在本章中，我们将对后者给出一个清晰的定义。

我们将通过定义数据库变量的数据类型来给出数据库设计的形式化规范。这个数据类型——本质上是一个集合——容纳了数据库变量所有可接受的数据库状态，被称为 `数据库全域`。

你将会看到，数据库全域可以以分阶段（分层）的方式构建，这同时为我们提供了一个清晰的数据完整性约束分类体系。

首先，你定义词汇表。在你想要在数据库中处理的现实世界里，有哪些事物以及这些事物的哪些方面？在这里，你为每一个被认为是必要的表结构指定一个名称，以及该表结构将拥有的属性名称。我们将引入一个示例数据库设计来演示这一点。词汇表在所谓的 `数据库骨架` 中形式化定义。进一步解释所有属性（及其关联）含义的一个好方法是，为每个表结构提供 `外在谓词`；这是一个描述所涉及属性的含义和关联的自然语言句子。

给定数据库骨架后，我们接着为每个属性定义可接受的属性值集合。这是通过对每个表结构引入一个 `特征描述` 来完成的。你在第四章已经了解了特征描述的概念。

然后，你将使用这些特征描述作为构建块，为每个表构建可接受的元组集合。这被称为 `元组全域`，并包括 `元组约束` 的形式化规范。

接着，你将使用元组全域为每个表结构构建可接受的表的集合。这个集合被称为 `表全域`，可以被视为表变量的数据类型。表全域的定义将包括相关 `表约束` 的形式化规范。

本章最后一节展示了如何将表全域汇集到可接受的数据库状态集合的定义中，这也是本章设定的目标：定义一个数据库全域。在这个阶段，你形式化地指定 `数据库（多表）约束`。

由于本章展示的示例数据库全域包含十个表结构，我们将向你介绍十个特征描述、十个元组全域和十个表全域。这加上所提供的解释性材料，使得本章篇幅相当大。然而，其中的示例数量应该能为你应用形式化理论提供一个坚实的起点，从而使你能够开始在数据库专业人员的实践中应用这种方法论。你可以在附录 A 中找到包含该设计核心要点的示例数据库设计规范版本。

在“章节小结”部分之后，有一个练习部分，主要侧重于修改或添加本章引入的示例数据库全域各层中的约束规范。

#### 记录数据库与约束

因为你在阅读本书，所以你将自己视为一名数据库专业人员。因此，规范数据库设计很可能是你工作的一部分。你可能会同意，设计数据库的过程大致包括两项主要任务：
1.  发现现实世界中那些你需要在数据库设计中引入表结构来表示的事物。这是通过与你正在设计的信息系统的用户和利益相关者进行访谈和沟通来完成的。
2.  发现将控制表结构中维护的数据的数据完整性约束。这些约束为第一步引入的表结构增添了含义，并将最终使数据库设计能够令人满意地拟合你所建模的现实。

本书第一部分引入的数学应用主要面向第二项任务；它使你能够形式化地规范数据完整性约束。我们相信，无论何时你设计一个数据库，都应该将最多的时间花在设计所涉及的数据完整性约束上。准确地——即无歧义地——记录这些数据完整性约束，可能决定了你的成败。

尽管如此，如今记录数据完整性约束最广泛的方式仍是使用自然语言，这常常会迅速陷入歧义。如果你使用纯英语来表达数据完整性约束，你不可避免地会遇到“这个英语句子如何无歧义地映射到表结构中”的问题。不同的程序员（以及用户）会以不同的方式解释这样的句子，因为他们都试图将这些句子转换为能够映射到数据库设计的东西。程序员然后会编写他们自己对约束的理解（不一定是规范制定者的本意）。

接下来的几节将证明，本书第一部分介绍的逻辑和集合论非常适合以形式化的方式捕捉包含其完整性约束的数据库设计。数据完整性约束的形式化规范确切地告诉你它们如何映射到表结构中。你不仅可以避免前面提到的歧义，而且还能获得对数据库最重要方面的清晰而专业的视角：所有涉及的数据完整性约束。

■注意 你们中的一些人可能会对接下来的例子感到惊讶，即信息系统整体规范中有多少实际上存在于数据库设计的规范中。信息系统中涉及的许多“业务逻辑”通常可以由映射到支撑信息系统的底层表结构的数据完整性约束来表示。

#### 数据库设计的内部层次

在设定了场景之后，我们现在将演示集合论和逻辑如何使你能够清晰而专业地审视数据库设计及其完整性约束。接下来的两节（非正式地）向你介绍一种看待数据库设计精髓的特殊方式。这种视角将使数据库设计的分层集合论规范成为可能。

##### 数据库的自顶向下视图




在任何给定时间点，数据库本质上都是表的集合。我们的数据库，或者更准确地说，我们的数据库变量，保存着数据库的当前状态。随着时间的推移，会发生事务，并为数据库变量分配新的数据库状态。我们需要为数据库变量指定所有`admissible database states`的集合。这个集合被称为`database universe`，实际上定义了数据库变量的数据类型。自上而下地看，对于一个涉及 n 个表结构的给定数据库设计，你可以观察到以下内容：

• 每个数据库状态都是一个可接受的 n 个表的集合（每个表结构一个），其中

• 每个表都是一个可接受的元组集合，其中

• 每个元组都是一个可接受的属性-值对集合，其中

• 每个值都是该给定属性的可接受值。

由于前面的所有层都是集合，你可以使用集合论从数学上定义它们全部。通过逻辑（通过添加嵌入式谓词），你可以精确定义每个层中“可接受”的含义；此时数据完整性约束就出现了。

那么，如何以形式化的方式指定这个称为`database universe`的集合呢？这是采用与之前介绍的层级相同的自下而上方法来完成的。首先，定义你的词汇表：在数据库中，你想处理现实世界中的哪些事物及其哪些方面？换句话说，你需要哪些表结构，每个表结构具有哪些属性？这在所谓的`database skeleton`中形式化定义。

对于在`database skeleton`中引入的每个属性，你接着定义可接受的属性值集合。你已经接触过这个；在此阶段，所有的`characterizations`（每个表结构一个）被定义。

然后，你使用`characterizations`作为构建块，为每个表结构构建（定义）`set of admissible tuples`。这涉及应用广义积运算符（参见定义 4-7）和引入元组谓词。可接受的元组集合被称为`tuple universe`。

然后，你可以使用`tuple universes`为每个表结构构建`set of admissible tables`，这被称为`table universe`。你将在本章中看到如何做到这一点；它涉及应用幂集运算符和引入表谓词。

在最后阶段，你使用先前定义的`table universes`定义可接受的数据库状态集合——`database universe`。

这种形式化定义数据库变量数据类型的方法是由荷兰数学家 Bert De Brock 与 Frans Remmen 在 20 世纪 80 年代开发的，是准确定义数据库设计（包括所有相关数据完整性约束）的一种优雅方法。参考文献`De grondslagen van semantische databases`（Academic Service，1990，荷兰语）和`Foundations of Semantic Databases`（Prentice Hall，1995）是 Bert De Brock 撰写的书籍，他在其中介绍了这种方法。

#### 约束分类模式

在这种对`database universe`的自下而上的稳固构造中，你在前面描述的每个层级上只显式允许可接受值的集合。这意味着在每个层级上，这些集合必须满足某些数据完整性约束。约束指定了哪些集合是有效的；它们限定了集合的内容。这直接引出了四类数据完整性约束：

• `Attribute constraints`：实际上，这些是你在一个`characterization`中指定的属性值集合。你可以争论“约束”一词在这里是否合适。`characterization`只是为每个属性指定属性值集合（而不进一步*约束*其中的元素）。然而，属性值集合确实*约束*了属性允许的值。

**注意**：当属性值集合在 SQL 数据库管理系统中实现时，我们将在第 11 章重新讨论这个问题。

• `Tuple constraints`：这些是你在定义`tuple universe`时指定的`tuple predicates`。`tuple predicates`约束了一个元组内不同属性值的组合。有时这些约束被称为`inter-attribute constraints`。你可以在不引用其他元组的情况下指定它们。例如，这里是 EMP（员工）表结构中属性 Job 和 Salary 之间的一个约束：

“职位为 President 的员工每月工资大于 10000 美元。”

• `Table constraints`：这些是你在定义`table universe`时指定的`table predicates`。`table predicates`约束同一表中不同元组的组合。有时这些约束被称为`inter-tuple constraints`。你可以在不引用其他表的情况下指定它们。例如：“没有员工的月工资能高于其经理”（这里我们假设 EMP 表结构中存在一个 Manager 属性，引用该员工的经理）。

• `Database constraints`：这些是你在定义`database universe`时指定的`database predicates`。`database predicates`约束不同表结构的表的组合。有时这些约束被称为`inter-table constraints`。你只能在引用不同表结构时指定它们。

例如，存在 EMP 和 DEPT 表结构之间无处不在的数据库约束：每个员工必须为一个已知的部门工作。

这四类约束接受或拒绝给定的数据库状态。它们限定数据库状态，并且通常被称为`static`或`state constraints`；它们可以在（静态）数据库状态的上下文中被检查。实际上还有一类约束。这是限制数据库状态`transitions`（基于静态约束之外的理由）的约束类别。专门限定数据库状态转换的谓词被称为`dynamic`或`state transition constraints`。我们将在第 8 章单独讨论这些。

由于前面的分类模式是由约束所处理的`scope of data`驱动的，它具有与约束实现问题密切相关的优点。当你在 SQL DBMS 中实现数据库设计时，鉴于这些系统对数据完整性约束的声明性支持较差，你将面临这些问题。这种支持的缺乏将负担加在你身上，要求你开发通常复杂的代码来强制执行约束。第 11 章将使用这里介绍的分类来研究数据完整性约束的这些实现挑战。
