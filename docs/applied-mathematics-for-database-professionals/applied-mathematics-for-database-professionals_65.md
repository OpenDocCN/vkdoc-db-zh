## 附录 A：示例数据库的形式化定义

在本附录中，我们提供了贯穿本书使用的示例数据库设计的形式化规范。

“鸟瞰概览”一节提供了示例数据库的鸟瞰概览；它展示了十个表及其关系的图示，并对这些表和关系提供了简要的非形式化描述。

然后，我们向你展示*数据库骨架* `DB_S` 的定义。在“数据库骨架 `DB_S`”一节中，你可以找到每个表涉及的属性，包括每个属性的简要描述。

本附录最大的一节是“表 universe 定义”，提供了十个*表 universe*定义。对于每个表，你将找到表 universe 规范，该规范形式化地定义了适用于该表的所有属性约束、元组约束和表约束，以及外部谓词（向用户描述表的含义）。

附录以通过指定*数据库 universe* `DB_UEX` 来定义*静态*数据库约束结束（见“数据库 universe `DB_UEX`”一节），然后通过指定*状态转移 universe* `TX_UEX` 来定义*动态*约束（见“状态转移 universe `TX_UEX`”一节）。下表总结了本附录的结构，并展示了如何从骨架开始，逐步构建到数据库和状态转移 universe 的示例数据库设计规范。

### 鸟瞰概览


图 A-1 展示了构成我们示例数据库设计的十个表（由圆角框表示）及其相互关系（由箭头表示）的示意图。

每条箭头都表示一对表之间适用的`子集要求`。这些`子集要求`表明，箭头起始表的某种投影应始终是箭头指向表的某种投影的子集。这些箭头大多代表了通常所说的`多对一关系`，并在最终的实现阶段将成为`外键约束`。

然而，正如你在第 11 章所见，情况并非总是如此。

我们将在本附录的第五节中，在数据库宇宙规范 `DB_UEX` 中给出每条箭头的确切含义。

我们的数据库保存着公司的员工 (`EMP`) 和部门 (`DEPT`)。部分箭头表示以下关系：

-   一名员工为一个部门工作。
-   一个部门由一名员工管理。
-   一名员工被分配到一个工资等级 (`GRD`)。

所有工资和/或“所属部门”变更都保留员工历史记录 (`HIST`)；每条历史记录描述了一名员工在一段时期内被分配到一个部门并享有特定工资的情况。

我们在一个单独的表 (`SREP`) 中保存所有销售代表的附加信息。我们在一个表 `TERM` 中保存那些不再为公司工作的员工（即，他们已被解雇或已辞职）的附加信息。我们还保存所有有经理的受管理员工 (`MEMP`) 的附加信息；即，那些有经理分配给他们的员工。

数据库还保存有关课程 (`CRS`)、这些课程的开设班次 (`OFFR`) 以及这些班次的注册 (`REG`) 信息。还有一些箭头表示以下关系：

-   一个班次必须由一名为公司工作的培训师来教授。
-   一个班次属于一门现有课程。
-   一条注册记录将一名员工作为一名学员记录在一个课程班次中。

## 数据库骨架 DB_S

在本节中，你将找到示例数据库骨架 `DB_S` 的规范。

`数据库骨架` 定义了我们的词汇表；对于每个表，我们引入一个表别名，并为每个表引入所涉及属性的名称。我们这里不给出表的外部谓词；你可以在本附录下一节的表空间定义中找到它们。

`数据库骨架` 是一个集合值函数；对于每个表，该函数生成该表的属性集（表头）。示例数据库的数据库骨架 `DB_S` 在清单 A-1 中定义。

### 清单 A-1. 数据库骨架定义

```
DB_S = { (EMP; -- 员工 (Employees)
          { EMPNO /* 员工编号 */
          , ENAME /* 员工姓名 */
          , JOB /* 员工职位 */
          , BORN /* 出生日期 */
          , HIRED /* 入职日期 */
          , SGRADE /* 工资等级 */
          , MSAL /* 月薪 */
          , USERNAME /* 用户名 */
          , DEPTNO } ) /* 部门编号 */
       , (SREP; -- 销售代表 (Sales Representatives)
          { EMPNO /* 员工编号 */
          , TARGET /* 销售目标 */
          , COMM } ) /* 佣金 */
       , (MEMP; -- 受管理员工 (Managed Employees)
          { EMPNO /* 员工编号 */
          , MGR } ) /* 经理：员工编号 */
       , (TERM; -- 已离职员工 (Terminated Employees)
          { EMPNO /* 员工编号 */
          , LEFT /* 离职日期 */
          , COMMENTS } ) /* 离职备注 */
       , (DEPT; -- 部门 (Departments)
          { DEPTNO /* 部门编号 */
          , DNAME /* 部门名称 */
          , LOC /* 地点 */
          , MGR } ) /* 经理：员工编号 */
       , (GRD; -- 工资等级 (Salary Grades)
          { GRADE /* 等级代码 */
          , LLIMIT /* 工资下限 */
          , ULIMIT /* 工资上限 */
          , BONUS } ) /* 年度奖金 */
       , (CRS; -- 课程 (Courses)
          { CODE /* 课程代码 */
          , DESCR /* 课程描述 */
          , CAT /* 课程类别 */
          , DUR } ) /* 课程时长（天） */
       , (OFFR; -- 课程班次 (Course Offerings)
          { COURSE /* 课程代码 */
          , STARTS /* 本次班次开始日期 */
          , STATUS /* 已安排、已确认... */
          , MAXCAP /* 最大参与容量 */
          , TRAINER /* 培训师：员工编号 */
          , LOC } ) /* 地点 */
       , (REG; -- 课程注册 (Course Registrations)
          { STUD /* 学员：员工编号 */
          , COURSE /* 课程代码 */
          , STARTS /* 课程班次开始日期 */
          , EVAL } ) /* 评估 */
       , (HIST; -- 员工历史记录 (Employee History Records)
          { EMPNO /* 员工编号 */
          , UNTIL /* 历史记录结束日期 */
          , DEPTNO /* 部门编号 */
          , MSAL } ) } /* 月薪 */
```

## 表空间定义

本节提供了示例数据库十个表空间的形式化定义。

对于每个表，你将找到四个小节：

-   表的`外部谓词`，向数据库用户描述表属性的含义。
-   `属性特征`，为每个属性附加`属性值集`。命名约定为 `chr_<表别名>`。
-   `元组空间`，为表定义`元组约束`（如果有）。命名约定为 `tup_<表别名>`。
-   `表空间`，为表定义`表约束`。命名约定为 `tab_<表别名>`。

**注意：**`元组空间`规范建立在`属性特征`规范之上；`表空间`规范又建立在`元组空间`规范之上。
