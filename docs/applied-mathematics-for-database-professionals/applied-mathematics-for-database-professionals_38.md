## 131

通过使用比清单 6-8 中使用的函数应用更复杂的函数限制，你可以同时表达涉及两个相关表多个属性的子集需求。第 7 章将介绍的数据库设计中有一个涉及多个属性的子集需求示例。

一个子集需求谓词通常涉及两个*不同*的表结构，因此代表一个数据库谓词。然而，情况并非总是如此；有时它们只涉及一个表结构。在这些情况下，它们代表一个表谓词。例如，假设表 `EMP1` 被扩展了一个 `mgr`（经理）属性，该属性保存员工的经理的员工编号。你会要求每个在 `EMP1` 中出现的经理编号也必须作为员工编号出现在 `EMP1` 中。形式上，你可以将其表达为一个子集需求：

{ t1(mgr) | t1∈EMP1 } ⊆ { t2(empno) | t2∈EMP1 }

你甚至可能要求每个在 `EMP1` 中出现的经理编号也必须作为员工编号出现在 EMP1 的某个*限制*中。也许引用的元组必须每个都有一个 `JOB` 值表明该员工确实是一名经理。形式上，你可以如下表达：

{ t1(mgr) | t1∈EMP1 } ⊆ { t2(empno) | t2∈EMP1 ∧ t2(job)='MANAGER' }

我们认为前面的表达式也是一个子集需求。

子集需求可以通过图片来可视化。你可以为子集需求"A in T1 references A in T2"画一张图，方法是绘制两个矩形（或其他形状）——一个代表 T1，一个代表 T2——并用一条从 T1 指向 T2 的有向箭头连接这两个矩形。该箭头表示 T1 表中的每个元组至少引用 T2 表中的一个元组。

图 6-6 显示了从 `EMP1` 到 `DEP2` 的子集需求。

###### 图 6-6. *从 EMP1 到 DEP2 的子集需求可视化*

请注意，子集需求的这张图并没有告诉你箭头两边涉及的属性是什么，也没有告诉你是否涉及任何属性重命名。它*仅仅是一张图*，应该始终伴随着精确说明箭头预期含义的形式谓词。

有时你知道该子集需求实际上是一个*真*子集需求。

假设你想指定这样的需求：在员工表中找到的部门编号集合是部门表中找到的部门编号集合的真子集。你可以将此数据库谓词指定如下：`P15a(D,E) := { e(deptno)| e∈E } ⊂ { d(dno)| d∈D }`

在本书中，我们将这种特殊情况也称为子集需求。另一种处理方法仍然是使用子集运算符（而不是真子集），并指定第二个数据库谓词来表达部门表中至少存在一个部门编号不出现在员工表中。以下数据库谓词 `P15b` 正是这样做的：

```
P15b(D,E) := ( ∃d∈D: ( ∀e∈E: e(deptno)≠d(deptno) ) )
```

表 `EMP1`（员工）和 `DEP2`（部门）同时符合数据库谓词 `P15a` 和 `P15b`。有时你知道该子集需求实际上是一个*相等*需求。假设你想指定每个部门都必须有员工的需求。你可以如下指定：

`P16a(D,E) := { e(deptno)| e∈E } = { d(dno)| d∈D }`

在本书中，我们将这种特殊情况也称为子集需求。同样，另一种处理方法是使用子集操作（而不是相等操作），并添加第二个完整性约束来表达每个部门至少存在一个员工。数据库谓词 `P16b` 正是表达了这一点：

```
P16b(D,E) := ( ∀d∈D: ( ∃e∈E: e(deptno)=d(dno) ) )
```

顺便问一下，你看出谓词 `P15b` 正是谓词 `P16b` 的否定吗？这是因为这两个特殊情况——真子集和相等——是互斥的。

**■注意** 子集需求的这些变体，结合子集需求所涉及表的唯一标识谓词，是实体关系图中发现的各种关系的数学对应物。

###### 特殊化谓词

看一下图 6-7 中显示的表 `TRN1`。

###### 图 6-7. *表 TRN1*

清单 6-10 中列出的关于表 `EMP1` 和 `TRN1` 的命题是 TRUE 命题。

###### 清单 6-10. *关于表 EMP1 和 TRN1 的命题*

```
P7 := {empno} 在 TRN1 中是唯一标识的
P8 := { e↓{empno}| e∈EMP1 ∧ e(job)='TRAINER' } = { t↓{empno}| t∈TRN1 }
```

**■注意** 我们已经正式定义了“*一个集合在表中是唯一标识的*”这一表述的含义。

所以 `P7` 确实是一个形式化规范。

从命题 `P7` 可以推出，表 `TRN1` 和 `EMP1` 的唯一标识属性是相同的。命题 `P8` 表明，表 `EMP1` 中培训师的员工编号等于表 `TRN1` 中找到的员工编号（粗略地说）。注意这里的*相等性*。对于在表 `EMP1` 中找到的每个培训师，你都能在表 `TRN1` 中找到一个相关的元组。反之亦然，对于 `TRN1` 中的每个元组，你都能在 `EMP1` 中找到一个培训师。

表 `TRN1` 可以被认为是保存了关于培训师（且仅关于培训师）的附加信息：他们获得认证的日期（再次粗略地说）。

这种情形在数据库设计中经常出现；对于给定表中的某个元组子集——可通过某个谓词指定（在上例中是 `e(job)='TRAINER'`）——在另一个给定表中发现了附加信息，且这两个表共享相同的唯一标识属性。在这种情况下，表 `TRN1` 被称为表 `EMP1` 的一个*特殊化*。定义 6-3 正式定义了这一概念。

**■定义 6-3：特殊化谓词** 设 `T1` 和 `T2` 是表，`A` 是 `T1` 的头部的一个给定子集，同时也是 `T2` 的头部的一个子集。设 `P` 是一个元组谓词。如果一个数据库谓词具有以下形式，则我们称其为特殊化谓词：

{ t1↓A | t1∈T1 ∧ P(t1) } = { t2↓A | t2∈T2 }

我们称“`T2` 是 `T1` 的一个特殊化”。

通常你会发现集合 `A` 在表 `T1` 和 `T2` 中都是唯一标识的。在清单 6-10 的例子中，你看到 `{empno}` 在 `EMP1` 和 `TRN1` 中都是唯一标识的。

一个给定的表可以有多个特殊化。假设你也想为经理保存附加信息；例如，经理的年度奖金。为本例假设，培训师和文员没有年度奖金，只有经理有。看一下保存经理年度奖金的表 `MAN1`（见图 6-8）；它也是 `EMP1` 的一个特殊化。

###### 图 6-8. *表 MAN1*

以下命题在表 `EMP1` 和 `MAN1` 之间成立。它符合定义 6-3 引入的形式。

{ e↓{empno}| e∈EMP1 ∧ e(job)='MANAGER' } = { t↓{empno}| t∈MAN1 }

在下一节中，我们将讨论与特殊化相关的一个概念。

###### 泛化

如果一个给定的表 `T` 有多个特殊化，例如表 `S1` 到 `Sn` (`n>1`)，使得对于 `T` 中的每个元组，在恰好一个特殊化中存在恰好一个相关的元组，那么我们说“表 `T` 是表 `S1` 到 `Sn` 的一个*泛化*”。

现在，特化表`TRN1`、`MAN1`和`CLK1`总共有七个元组。对于表`EMP1`中的每一个元组——共有七个——在表`TRN1`、`MAN1`或`CLK1`中的恰好一个表中，存在且仅存在一个对应的元组。这使得表`EMP1`成为表`TRN1`、`MAN1`和`CLK1`的一个**泛化（generalization）**。

**注意** 顺便提一下，你现在也可以指定一个从`CLK1`到`EMP1`的子集需求（涉及`CLK1`的经理属性）。这将作为本章末尾的一个练习。

在泛化表中，你经常会发现一个特定的属性，它决定了额外信息可以在哪个特化表中找到。在所使用的示例中，这个属性是`job`属性。对于表`EMP1`中`job='TRAINER'`的元组，其额外信息在特化表`TRN1`中找到。对于`job='MANAGER'`的元组，额外信息可在表`MAN1`中找到；而对于`job='CLERK'`的元组，额外信息则在表`CLK1`中找到。泛化表中的这样的属性被称为`检查属性（inspection attribute）`。

请注意，`检查属性`是一个冗余属性；也就是说，它的值可以从其他元组推导出来。对于`EMP1`中的一个给定元组，它在`TRN1`、`MAN1`或`CLK1`中存在一个对应的元组。你可以直接从该对应元组出现的位置推导出`EMP1`元组的`job`值；其值应分别为`'TRAINER'`、`'MANAGER'`或`'CLERK'`。

通常，你不应该在数据库设计中引入数据冗余；这迫使你在更新决定冗余数据的数据时，必须维护这种冗余。

或者，从数据库设计的角度来看，这迫使你引入一个约束来强制实施数据冗余。然而，在数据库设计中指定泛化属性时，`检查属性`非常有帮助（本章末尾有一个关于此的练习）。

###### 连接中的元组谓词

两个表通常通过子集需求相关联。当两个表以这种方式关联时，由于显而易见的原因，连接它们的需求会经常出现。你可以根据涉及这两个表之间子集需求的属性（本例中是`deptno`属性），连接表`EMP1`和`DEP1`中对应的元组。这种组合产生的表，其标题将是`EMP1`的标题与`DEP1`标题的并集。

图 6-10 展示了表`EMP1`和`DEP1`这种组合的结果。

###### 图 6-10. 表 `EMPDEP1`，表 `EMP1` 和 `DEP1` 的连接

表`EMPDEP1`中的每个元组被称为`连接中的元组`。`连接中的元组谓词`是处理这些连接中元组的元组谓词；被连接的元组是该元组谓词的参数。以下是关于`EMPDEP1`的一个元组谓词示例：
```
( ∀t∈EMPDEP1: t(sal)≤t(salbudget)/5 )
```
这个命题表明，`EMPDEP1`（即`EMP1`和`DEP1`的连接）中的所有元组，其`sal`属性值不超过`sabudget`属性值的 20%。在此命题中，表达式`t(sal)≤t(salbudget)/5`代表了元组谓词。如果你不以表`EMPDEP1`的形式，而是以两个被连接的表——`EMP1`和`DEP1`——的形式来表达这个命题，你将得到如下形式的命题：
```
( ∀e∈EMP1: ( ∀d∈DEP1: e(deptno)=d(deptno) ⇒ e(sal)≤d(salbudget)/5 ) )
```
它表明，任何员工的薪资都不能超过其受雇部门工资预算的五分之一。另一种正式的指定方式如下：
```
( ∀t∈EMP1⊗DEP1: t(sal)≤t(salbudget)/5 )
```
在这个命题中，表达式`t(sal)≤t(salbudget)/5`代表了一个约束连接中元组的元组谓词。这种谓词模式在数据库设计中很常见，被称为`连接中的元组谓词`。定义 6-4 正式定义了它。

**定义 6-4: 连接中的元组谓词** 设`T1`和`T2`为表，`P`为一个元组谓词。如果一个谓词具有以下形式，则它是`连接中的元组谓词`：
```
( ∀t∈T1⊗T2: P(t) )
```
我们说`P`约束了`T1`和`T2`连接中的元组。

在前面的定义中，你通常会发现表`T1`和`T2`是通过涉及连接属性的子集需求相关联的。

列表 6-11 展示了另外两个涉及表`EMP1`和`DEP1`的`连接中的元组谓词`实例。

###### 列表 6-11. 更多关于表 `EMP1` 和 `DEP1` 的连接中的元组谓词

```
P9 := ( ∀e∈EMP1: ( ∀d∈DEP1: e↓{deptno}=d↓{deptno} ⇒
        (d(loc)='LOS ANGELES'⇒ e(job)≠'MANAGER') ) )

P10 := ( ∀e∈EMP1: ( ∀d∈DEP1: e↓{deptno}=d↓{deptno} ⇒
         (d(loc)='SAN FRANCISCO'⇒ e(job)∈{'TRAINER','CLERK'}) ) )
```
`P9`指出经理不能被雇佣在洛杉矶。`P10`指出在旧金山工作的员工必须是培训师或办事员。根据图 6-10 中的示例值`EMP1`和`DEP1`，这两个命题都是`真（TRUE）`；没有经理在洛杉矶工作，并且所有在旧金山工作的员工——实际上没有这样的员工——都是培训师或办事员。

在本节中，我们定义了只涉及两个表的`连接中的元组谓词`。然而，使用连接操作符组合三个甚至更多的表通常也是有意义的。当然，这些连接中的元组也可以被元组谓词约束。以下是涉及三个表（`T1`、`T2`和`T3`）的`连接中的元组谓词`的模式：
```
( ∀t1∈T1: ( ∀t2∈T2: ( ∀t3∈T3: (t1↓A=t2↓A ∧ t2↓B=t3↓B) ⇒ P(t1∪t2∪t3) ) ) )
```
在此模式中，`A`代表表`T1`和`T2`的连接属性集，`B`代表表`T2`和`T3`的连接属性集。谓词`P`代表一个以连接中的元组为参数的谓词。在下一章中，你将看到涉及两个以上表的`连接中的元组谓词`示例。



#### 章节摘要

本节以项目符号列表的形式提供了本章的摘要。你可以在继续下一节的练习之前，使用此摘要来检查对本章引入的各种概念的理解。

-   `元组谓词`是一种带有一个类型为元组的参数的谓词。它可以基于元组所具有的属性值的组合来接受（或拒绝）元组。
-   `表谓词`是一种带有一个类型为表的参数的谓词。它可以基于表所包含的元组的组合来接受（或拒绝）表。
-   `数据库（多表）谓词`是一种带有一个类型为数据库状态的参数的谓词。它可以基于数据库状态所包含的表的组合来接受（或拒绝）数据库状态。
-   表谓词和数据库谓词有五种常见模式：`唯一标识`、`子集要求`、`特化`、`泛化`和`连接中的元组`谓词。
-   `唯一标识`谓词是如下形式的表谓词（`T`代表一个表）：`( ∀t1,t2∈T: t1↓A=t2↓A ⇒ t1=t2 )`。
    在此表达式中，`A`代表在表`T`中唯一标识元组的属性集合。
-   `子集要求`谓词是如下形式的谓词（`T1`和`T2`代表表）：`{ t1↓A| t1∈T1 } ⊆ { t2↓A| t2∈T2 }`。
    在此表达式中，`A`通常代表在表`T2`中唯一标识元组的属性集合。
-   `特化`谓词是如下形式的数据库谓词（`T1`和`T2`代表表）：`{ t1↓A| t1∈T1 ∧ P(t1) } = { t2↓A| t2∈T2 }`。
    在此表达式中，`A`代表在表`T1`和`T2`中唯一标识元组的属性集合。谓词`P`是一个指定表`T1`子集的谓词。我们称"`T2`是`T1`的特化"。`T2`被认为持有表`T1`中由谓词`P`指定的元组子集的附加信息。
-   如果给定的表`T`有多个特化，并且对于`T`中的每个元组，在其中一个特化表中恰好存在一个持有该元组附加信息的元组，那么表`T`被称为这些特化的`泛化`。
-   `连接中的元组`谓词是如下形式的谓词（`T1`和`T2`代表表）：`( ∀t1∈T1: ( ∀t2∈T2: t1↓A=t2↓A ⇒ P(t1∪t2) ) )`。
    在此表达式中，`A`代表通常参与表`T1`和`T2`之间子集要求的属性集合。谓词`P`是一个元组谓词。
