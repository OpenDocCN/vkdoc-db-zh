大多数子集要求都很直接；`PSSR1`、`PSSR5`、`PSSR6`、`PSSR9`、`PSSR10`、`PSSR11` 和 `PSSR12` 都规定了第一个参数中的每个元组都与第二个参数中的某个元组相关。

你可能会觉得 `PSSR7` 有点特别。它规定 `OFFR` 中的 `{LOC}` 引用 `DEPT` 中的 `{LOC}`。通常，对于这样的子集要求，`{LOC}` 在 `DEPT` 中应该是唯一标识符；然而，这里情况并非如此。这意味着一个 `OFFR` 元组可能与多个 `DEPT` 元组相关。

**注意** 显然，这个示例数据库设计不需要保存除地点名称之外的位置信息。如果是那样，就会引入一个单独的表结构（例如别名为 `LOC`）来保存位置信息。同时，也会引入两个子集要求：一个从 `DEPT` 到 `LOC`，另一个从 `OFFR` 到 `LOC`。

谓词 `PSSR2` 和 `PSSR3` 规定第一个参数值中的每个元组都与第二个参数值中*某个子集*中的某个元组相关；这里右侧指定的集合表示一种表限制。

谓词 `PSSR8` 规定第一个参数值中*某个子集*中的每个元组都与第二个参数值中*某个子集*中的某个元组相关；这里两侧指定的集合都表示表限制。

最后，你可能已经注意到谓词 `PSSR4` 内部使用了*真*子集操作符；这确保了并非每个员工都已终止（至少有一名员工仍在公司工作）。

谓词 `PTIJ1` 至 `PIJ15` 都限制了某些连接中允许的元组。这些连接大多只涉及两个表结构。有五个谓词（`PTIJ3`、`PTIJ9`、`PTIJ10`、`PTIJ12`、`PTIJ15`）限制了涉及两个以上表结构的连接中的元组。让我们更仔细地看看这五个。

谓词 `PTIJ3` 将一个 `SREP` 元组（变量 `s`）与其相关的 `EMP` 元组（变量 `es`）连接，继续将此 `EMP` 元组与其相关的 `MEMP` 元组（变量 `m`）连接，最后将此 `MEMP` 元组与其相关的 `EMP` 元组（变量 `em`）连接。所有在这个跨四个表的连接（注意 `EMP` 被连接了两次）中的元组都受谓词 `es(MSAL) + s(COMM)/12 < em(MSAL)` 的约束。这表示销售代表的月工资，加上其佣金的十二分之一，应小于管理该销售代表的员工的月工资。顺便说一下，由于子集要求 `PSSR3`，这位管理员工不能是销售代表——这本可能需要在小于号的右侧也加上十二分之一的佣金。

谓词 `PTIJ9` 将一个 `TERM` 元组（`t`）直接与一个 `REG` 元组（`r`）连接，并继续将该 `REG` 元组与其相关的 `CRS` 元组（`c`）连接。

**注意** 几点说明：
-   从 `TERM` 到 `REG` 的连接不是“通过 `EMP`”完成的。因为 `TERM` 是 `EMP` 的特化，你可以使用 `EMPNO` 属性将 `TERM` 直接连接到任何 `EMP` 可以连接到的其他表结构。
-   从 `REG` 到 `CRS` 的连接不是“通过 `OFFR`”完成的。根据两个子集要求 `PSSR6` 和 `PSSR10`，你可以推断出 `REG` 中的 `{CRS}` 引用了 `CRS` 中的 `{CODE}`（一个 `REG` 和 `CRS` 之间的子集要求），因此你可以按照所示方式直接将 `REG` 连接到 `CRS`。
-   此连接中的元组受谓词 `t(LEFT) ≥ r(STARTS) + c(DUR)` 约束。你是否注意到这个正式规范与嵌入的非正式注释有所不同？正式规范也涉及了 `CRS` 表结构，或者更确切地说，是它的 `DUR` 属性。
-   这里是对该约束更正确的一个非正式解释：员工不能注册在其离职当天或之后开始的课程，也不能在其离职当天或之后被终止（该员工正在注册的课程期间）。
-   现在你正走上一条在尝试非正式描述数据库约束时通常难以避免的道路：你开始从**事务**的角度描述在给定约束下哪些操作不被允许。刚才发生的就是这样：给定某个 `TERM` 元组，你不能*插入*一个 `OFFR` 元组，反之亦然，给定某个 `OFFR` 元组，你也不能插入一个 `TERM` 元组。
-   随着约束所涵盖数据范围的扩大，用自然语言非正式地陈述约束的**内容**而不与客户疏远，变得越来越困难。你常常必须深入到约束的**实现方式**中。在与最终用户的沟通中，这并非坏事；你开始用系统将如何行为（给定约束下它将不接受哪些事务）的方式进行沟通，而最终用户通常能理解这一点。你的正式规范是生成这种行为的基础（我们将在第 11 章详细讨论这一点）。例如，你可能想简要告知用户，在此约束下，某些情况下课程的时长不能被增加（更新）。

谓词 `PTIJ10` 将一个 `EMP` 元组与两个相关的 `REG` 元组（`r1` 和 `r2`）连接，继续将这两个 `REG` 元组分别与其相关的 `OFFR` 元组（`o1` 和 `o2`）连接，最后将两个 `OFFR` 元组与其相关的 `CRS` 元组（`c1` 和 `c2`）连接。此连接中的元组受以下谓词约束：
```
o1↓{COURSE,STARTS} = o2↓{COURSE,STARTS} ∨
o1(STARTS) ≥ o2(STARTS) + c2(DUR) ∨
o2(STARTS) ≥ o1(STARTS) + c1(DUR)
```
该谓词表明，如果连接不涉及相同的两个 `OFFR` 元组，那么课程 `o1` 必须在课程 `o2` 结束后开始，反之亦然。


谓词 PTIJ12 是 PTIJ9 的“培训师”版本。PTIJ12 不通过 REG（使用 STUD 属性）将 TERM 连接到 CRS，而是通过 OFFR（使用 TRAINER 属性）将 TERM 连接到 CRS，并以相同的方式限制连接结果中的元组。

谓词 PTIJ15 通过 REG 和 OFFR 将一个 EMP 元组连接到一个 CRS 元组（代表参加课程班次的学生），并继续通过 OFFR 将同一个 EMP 元组连接到另一个 CRS 元组（代表授课的培训师）。然后，它以与谓词 PTIJ10 相同的方式限制此连接中的元组。

你可能已经注意到谓词 PTIJ15 和 PTIJ13 之间存在关联；我们将在本章末尾的一个练习中处理这个问题。

谓词 PODC1 声明两个集合的交集应为空集。第一个集合包含所有已离职员工的员工编号。第二个集合包含所有管理在职（即未离职）员工的员工编号。通过指定这两个集合的交集为空，你表达了“在职员工不能由已离职员工管理”这一非正式需求。

同样，谓词 PODC2 表达了“部门由在职员工管理”这一非正式需求。

谓词 PODC3 对所有担任未取消班次的培训师的员工编号进行全称量化。对于每个这样的员工编号，它构建一个集合，包含（未取消的）OFFR 元组与其关联的 CRS 元组的连接，其中班次的培训师等于给定的员工（编号），并且班次的地点等于该员工工作部门的地点。然后，它对这个集合中所有元组的 DUR 属性值求和。以同样的方式，计算出该培训师教授的、地点与其部门地点不同的班次的（合计）持续时间。对于每位给定的培训师，前一个总和不能小于后一个总和。

当然，这不是你与用户沟通的方式。你需要坚持使用他们提供的英文句子（见代码中的嵌入注释），并核实他们在该句子中使用的术语含义。“taught by a trainer”中的动词“taught”明确意味着班次的状态必须是“已安排”或“已确认”；已取消的班次不予考虑。计算班次数量似乎涉及班次的持续时间。最后，如果一个班次的地点与其培训师所属部门的地点相同，则认为它是在“所属基地”进行的。

> **注意：** 约束 PODC3 要求 OFFR 表结构的 LOC 属性可以（有意义地）与 DEPT 表结构的 LOC 属性进行比较。约束 PSSR7 正是为了确保这一要求而引入的。

谓词 PODC4 声明，已有六名或以上学生注册的班次状态必须为“已确认”。它通过检查与给定班次关联的所有注册记录的集合的基数来实现这一点。如果该基数大于或等于六，则给定班次的状态必须为“已确认”。

谓词 PODC5 的结构与 PODC6 类似。它对所有班次进行全称量化，并要求给定班次的关联注册人数不得超过该班次的最大容量。

> **注意：** 谓词 PODC4 和 PODC5 的组合意味着，允许一个班次的最大容量小于六是毫无意义的；这样的班次永远无法达到“已确认”状态。这正是属性 MAXCAP 的属性值集从数值六开始的原因（见代码清单 7-9）。

谓词 PODC6 声明，已取消的班次不能存在注册记录。因此，每当一个班次被取消时，所有相关的注册记录都应被移除。

谓词 PODC7 构成了我们示例数据库宇宙的*点睛之笔*。它要求培训师教授的每门课程的首次班次满足特定条件。如果给定班次的培训师受雇未满一年，那么该培训师必须参加过同一课程的先前班次，并且该先前班次的培训师必须参加由该给定培训师教授的此类首次班次。

谓词 PODC7 明确包含额外的注释，以引导你理解其形式化规范。

当你尝试阅读和理解像 PODC7 这样复杂的谓词时，首先研究其自上而下的结构是很有用的。在本例中，该谓词的结构如下：



( ∀o1∈OFFR: P1(o1,OFFR) ⇒ ( P2(o1,OFFR,REG) ∨ P3(o1,EMP) ) )
谓词 `PODC7` 是对所有课程产品的全称量化，它指出：如果涉及给定课程产品和其他课程产品的某个条件 (`P1`) 为真，那么两个条件 (`P2` 和 `P3`) 的析取式应成立。条件 `P1` 指出给定课程产品是首次开设的课程产品。条件 `P2` 涉及给定课程产品、其他课程产品和注册信息；它规定了关于先前课程产品的复杂要求，该课程必须由给定课程产品的培训师讲授过。条件 `P3` 涉及给定课程产品和员工；它指出给定课程产品的培训师至少已被雇佣一年。

**注意**
约束 `PODC7` 意味着，每门课程首次开设时，必须由至少已被雇佣一年的培训师来讲授。显然，开发教材并达到讲授课程所需的技能水平至少需要一年时间。

#### 章节摘要

本节提供了本章的摘要，以项目符号列表格式呈现。它提供了本章所讨论内容的高级概述。

*   数据库设计通常使用自然语言进行记录。这些设计文档的主要部分由约束规范组成，这些约束将确保数据库设计能满意地适配现实世界。
*   如果你使用纯英文来表达数据完整性约束，你将不可避免地遇到英文句子如何明确无误地映射到表结构的问题。
*   你可以使用集合论以分层（分阶段）的方式形式化地规范数据库设计。通过形式化你的数据库设计规范，你将消除所有歧义——尤其是围绕约束的歧义。
*   在第一阶段，你为每个表结构指定一个集合函数。该集合函数引入表结构的属性名，并将相关的属性值集附加到每个属性上。这些集合函数被称为 `特征化`。
*   在第二阶段，你为每个表结构指定一组称为 `元组宇宙` 的元组。它保存该表结构可接受的元组。你可以通过获取一个特征化的广义积，并通过指定元组谓词来限制此结果，从而构造一个元组宇宙。这些元组谓词被称为 `元组约束`。
*   在第三阶段，你为每个表结构指定一组称为 `表宇宙` 的表。它保存该表结构可接受的表。你可以通过获取元组宇宙的幂集，并通过指定表谓词来限制此结果，从而构造一个表宇宙。这些表谓词被称为 `表约束`。
*   在最后阶段，你指定一组称为 `数据库宇宙` 的数据库状态。它保存被规范数据库的可接受数据库状态。你可以通过首先定义一个 `数据库特征化` 来构造数据库宇宙。数据库特征化将所有表宇宙排列成一个单一的集合函数。你可以通过获取数据库特征化的广义积，并通过指定数据库谓词来限制此结果，从而构造数据库宇宙。这些数据库谓词被称为 `数据库约束`。
*   这个形式化模型是 **你**——数据库专业人士——的数据库设计参考。用户永远不应接触这种形式化规范。你可以使用一个外部谓词，结合元组约束，来传达每个表结构属性的含义。通常，你应该通过谈论表约束和数据库约束对系统行为的影响来向用户解释它们。最后，通过使用重写规则来重写构成约束的谓词，你可以找到与用户讨论相关约束的替代性、非正式方法。
