另外三个静态约束涉及 `OFFR` 表：`PTIJ10`、`PODC4` 和 `PODC5`。然而，插入新 `CRS` 元组的事务无法违反这三个约束。它们也都涉及 `REG` 表结构。前述要点列表中的第二个特征，加上初始状态必须是有效状态（即包含在 `DB_UEX` 中）这一事实，意味着 `REG` 表中不存在任何元组。这又意味着数据库状态符合（并且在插入 `offr1` 后保持符合）`PTIJ10`、`PODC4` 和 `PODC5`。

鉴于以上特征，插入元组 `toffr1` 的事务将会成功执行。但请注意，`toffr1` 代表一个*已确认*的课程开设。我们的状态转换宇宙（第 8 章的 `ST_UEX`）的定义包含了一项状态转换约束的规范，该约束规定新课程开设必须以*已安排*状态开始（约束 `STC2`，见清单 8-8）。显然，这个事务违反了约束 `STC2`。

尝试插入 `offr1` 的事务的正确定义，例如 `Tx2`，应如下所示。

对于 `DB_UEX` 中的某个数据库状态 `dbs`，令

```
Tx2a(dbs) := { (EMP; dbs(EMP) )
             , (SREP; dbs(SREP))
             , (MEMP; dbs(MEMP))
             , (TERM; dbs(TERM))
             , (DEPT; dbs(DEPT))
             , (GRD; dbs(GRD) )
             , (CRS; dbs(CRS) )
             , (OFFR; dbs(OFFR) ∪ {toffr1} )
             , (REG; dbs(REG) )
             , (HIST; dbs(HIST)) }
```

并令

```
Tx2b(dbs) := dbs , if Tx2a(dbs)∉DB_UEX
           := Tx2a(dbs), otherwise
```

那么，

```
Tx2(dbs) := Tx2b(dbs), if (dbs,Tx2b(dbs))∈ST_UEX
          := dbs      , otherwise
```

`Tx2a` 的定义描述了插入 `toffr1` 的操作。`Tx2b` 确保该操作符合所有静态约束。`Tx2` 的定义确保该事务也符合所有状态转换约束。它通过添加一个条件来实现这一点：由事务的开始状态和结束状态组成的有序对必须是状态转换宇宙的一个元素。

通常，一个事务只修改数据库设计中涉及的少数表结构；许多表结构保持不变。指定 `Tx2a` 的另一种（且稍微简短一些）方式如下所示，它更明确地展示了这一事实：

```
Tx2a(dbs) := dbs↓{EMP,SREP,MEMP,TERM,DEPT,GRD,CRS,REG,HIST}
∪
{ (OFFR; dbs(OFFR) ∪ {toffr1}) }
```

这里我们使用函数限制（`↓`）将所有未更改的表结构添加到结束状态，并通过并集（`∪`）添加事务修改的表结构。
