`P12(T) := ( ∀d1,d2∈T: d1≠d2 ⇒ d1↓{dname,loc}≠d2↓{dname,loc} )`
表谓词 `P11` 表示不能有两个不同的部门共享相同的部门编号。`P12` 表示两个不同的部门要么名称不同，要么位置不同（或两者都不同）。

顺便问一下，你看出这两个谓词具有相同的形式了吗？你可以使用重写规则将谓词 `P11` 转换为全称量化形式，如下所示：

```
¬( ∃d1,d2∈T: d1≠d2 ∧ d1(deptno)=d2(deptno) )
⇔ ( ∀d1,d2∈T: ¬( d1≠d2 ∧ d1(deptno)=d2(deptno) ) )
⇔ ( ∀d1,d2∈T: ¬(d1≠d2) ∨ ¬(d1(deptno)=d2(deptno)) )
⇔ ( ∀d1,d2∈T: ¬(d1≠d2) ∨ d1(deptno)≠d2(deptno) )
⇔ ( ∀d1,d2∈T: d1≠d2 ⇒ d1(deptno)≠d2(deptno) )
```
这里我们使用了一个将存在量词转换为全称量词的重写规则、一个德·摩根定律，以及一个将析取转换为蕴含的重写规则。

谓词 `P11` 和 `P12` 都属于我们将在下一节讨论的数据完整性谓词的常见模式之一。

`清单 6-8`. 两个示例数据库谓词

```
P13(S) := { e(deptno)| e∈S(employee) } ⊆ { d(deptno)| d∈S(department) }
P14(S) := ( ∀d∈S(department):
  ( Σe2∈{ e1| e1∈S(employee) ∧ e1(deptno)=d(deptno) }: e2(sal) )
  ≤ d(salbudget) )
```
谓词 `P13` 表明，每位员工都受雇于部门表中已知的一个部门。

这种类型的数据库谓词（一个表的指定属性值集合是另一个表的指定属性值集合的子集）在数据库设计中很常见，将在下一节更详细地讨论。将数据库状态 `S1` 作为参数提供给谓词 `P13`，会得到以下命题：

`{10,12} ⊆ {10,11,12,13}`

这是一个**真**命题。左边的数字集合是右边数字集合的子集。我们再次说数据库状态 `S1` 满足（没有违反）数据库谓词 `P13`。一个更简洁的指定谓词 `P13` 的方式如下：`P13(S) := S(employee)⇓{deptno} ⊆ S(department)⇓{deptno}`

我们现在使用表投影。将状态 `S1` 作为参数的值，得到这个命题：

```
{ {(deptno;10)}, {(deptno;12)} } ⊆
{ {(deptno;10)}, {(deptno;11)}, {(deptno;12)}, {(deptno;13)} }
```
这同样是一个**真**命题。左边的两个函数的集合是右边四个函数集合的子集。

谓词 `P14` 表明，员工表中所有在特定部门工作的员工的工资总和不应超过部门表中找到的该部门的工资预算。数据库状态 `S1` 也满足 `P14`。

与元组谓词和表谓词类似，一些数据库谓词的情况被认为是退化情况。如果一个数据库谓词，比如 `P`，可以被重写为具有以下模式的谓词，我们就认为该数据库谓词是退化情况：`P(S) := Q(S(T))`

如果一个数据库谓词是上述形式，那么它只涉及一个表结构。

在右边的表达式中，`T` 代表数据库状态 `S` 中的一个表结构名。`S(T)` 则代表对应的表，`Q` 代表一个表谓词。只涉及一个表结构（在上面的例子中是 `T`）的数据库谓词被认为是退化的。这些退化情况实际上是在伪装一个表谓词。在下一章中，我们将要求这些情况中的 `Q` 被引入作为表谓词，而数据库谓词 `P` 则被丢弃。

不失一般性，我们再次将注意力限制在不能分解为多个合取项的数据库谓词上。如果你能将一个给定的数据库谓词 `P(S)` 重写为 `Pa(S) ∧ Pb(S)` 形式的表达式，那么你实际上是在处理两个数据库谓词。

##### 关于数据完整性谓词的一些说明

我们通过对前述三类数据完整性谓词的几点观察来结束本节的介绍。

正如你在各种示例中看到的，我们可以优雅地使用元组谓词、表谓词和数据库谓词来形式化定义现实世界的条件（业务规则），这些规则需要以某种方式反映在数据库设计的规范中。一个业务规则映射到的谓词类（元组、表或数据库）完全取决于给定数据库设计的数据库骨架。例如，谓词 `P8`（见 `清单 6-5`）是一个表谓词，因为正式指定“经理工资高于培训师，培训师工资高于文员”这一业务规则的所有信息都包含在员工表结构中。现在，如果出于某种原因引入了一个单独的表结构，比如 `trainer`，来保存所有培训师信息（使得 `employee` 仅保存经理和文员信息），那么这个现实世界的约束将映射到一个涉及 `employee` 和 `trainer` 两个表结构的数据库谓词。

你可能对我们本章介绍数据完整性谓词的方式有个疑问。在你看到的各种示例中，我们使用这些谓词来展示给定的元组、表或数据库状态是否满足它们。然而，在形式化指定数据库设计时，我们对单个数据库状态（或表或元组）并不感兴趣。我们感兴趣的是一个数据库状态空间：一个包含我们为数据库允许的所有值的集合。

我们打算对代表数据库变量值集的这种数据库状态空间进行数学建模（这将在第七章完成）。当我们这样做时，我们将以一种复杂的方式使用本章介绍的三类数据完整性谓词，使得它们代表在数据库状态空间级别起作用的数据完整性约束。
