# 第十章 数据操作（续）

除了数据检索，事务执行是数据库的第二个最重要的应用。通过执行事务，你可以维护数据库并确保其状态是（变化中的）现实世界的有效表示。在本节中，我们将演示如何形式化地指定事务。

#### 插入操作的形式化规范

让我们从一个简单的例子开始，使用第七章引入的数据库设计。看一下元组 `tcrs1`。它表示一个需要插入数据库的新课程：

```
tcrs1 := { (CODE;'AM4DP')
         , (DESC;'Applied Mathematics for Database Professionals')
         , (CAT;'DSG')
         , (DUR;5) }
```

为方便起见，我们在代码清单 10-1 中重复了特征定义 `chr_CRS`、元组宇宙 `tup_CRS` 和表宇宙 `tab_CRS` 的定义。所有关于 `CRS` 表结构的属性、元组和表约束都在此清单中。

**代码清单 10-1.** `chr_CRS`、`tup_CRS` 和 `tab_CRS` 的定义

```
chr_CRS :=
  { ( CODE; CRSCODE–TYP )
  , ( DESCR; varchar(40) )
  , ( CAT; { s | s∈varchar(3) ∧ s∈{'DSG','GEN','BLD'} } )
  , ( DUR; { n | n∈number(2,0) ∧ 1 ≤ n ≤ 15 } )
  }

tup_CRS :=
  { c | c∈Π(chr_CRS) ∧ c(CAT) = 'BLD' ⇒ c(DUR) ≤ 5 }

tab_CRS :=
  { C | C∈℘(tup_CRS) ∧ (∀c1,c2∈C: c1(CODE) = c2(CODE) ⇒ c1 = c2) }
```

`tcrs1` 并未违反 `CRS` 表结构的任何属性或元组约束；或者换句话说，`tcrs1` 是元组宇宙 `tup_CRS` 的一个元素。

以下是对将 `tcrs1` 插入 `CRS` 表结构这一事务进行形式化规范的首次尝试。我们可以将这个事务指定为一个函数，例如 `Tx1a`，定义在 `DB_UEX` 上。对于 `DB_UEX` 中的一个给定数据库状态，例如 `dbs`，函数 `Tx1a` 返回一个反映了将 `tcrs1` 插入 `CRS` 表结构后的数据库状态。

```
Tx1a(dbs) := { (EMP; dbs(EMP) )
             , (SREP; dbs(SREP))
             , (MEMP; dbs(MEMP))
             , (TERM; dbs(TERM))
             , (DEPT; dbs(DEPT))
             , (GRD; dbs(GRD) )
             , (CRS; dbs(CRS) ∪ {tcrs1} )
             , (OFFR; dbs(OFFR))
             , (REG; dbs(REG) )
             , (HIST; dbs(HIST)) }
```

如你所见，函数 `Tx1a` 生成了另一个数据库状态，该状态包含与 `dbs` 相同的十个表结构。这个结果数据库状态仅在 `CRS` 表结构上与 `dbs` 不同；元组 `tcrs1` 已被添加。对于所有其他表结构，结果数据库状态持有 `dbs` 所持有的相应表。

让我们将 `Tx1a` 应用于“空数据库状态”（即所有表结构都持有空表的数据库状态）。以下是这个空数据库状态，我们将它命名为 `db_empty`。我们在第七章的练习 12 中向你介绍了这个数据库状态。

```
db_empty := { (EMP; ∅)
            , (SREP; ∅)
            , (MEMP; ∅)
            , (TERM; ∅)
            , (DEPT; ∅)
            , (GRD; ∅)
            , (CRS; ∅)
            , (OFFR; ∅)
            , (REG; ∅)
            , (HIST; ∅) }
```

在第七章的练习 12 中，你已确定 `db_empty` 是 `DB_UEX` 的一个元素。因此，你可以将函数 `Tx1a` 应用于它。将函数 `Tx1a` 应用于 `db_empty`，记为 `Tx1a(db_empty)`，得到以下数据库状态：

```
Tx1a(db_empty) = { (EMP; ∅)
                 , (SREP; ∅)
                 , (MEMP; ∅)
                 , (TERM; ∅)
                 , (DEPT; ∅)
                 , (GRD; ∅)
                 , (CRS; {tcrs1} )
                 , (OFFR; ∅)
                 , (REG; ∅)
                 , (HIST; ∅) }
```

注意，将 `Tx1a` 应用于 `db_empty` 的结果是一个（再次）属于 `DB_UEX` 的数据库状态；结果数据库状态符合作为 `DB_UEX` 定义一部分的所有静态约束。然而，这通常并不成立。让我们看看将 `Tx1a` 应用于以下数据库状态 (`dbs1`) 的情况：

```
dbs1 :=
  { (EMP; ∅)
  , (SREP; ∅)
  , (MEMP; ∅)
  , (TERM; ∅)
  , (DEPT; ∅)
  , (GRD; ∅)
  , (CRS; { { (CODE;'AM4DP'), (DESC;'AM4DP workshop')
           , (CAT;'DSG'), (DUR;1) } } )
  , (OFFR; ∅)
  , (REG; ∅)
  , (HIST; ∅) }
```

#### 事务定义与约束讨论

##### 问题分析

以下数据表示存在约束违反：

```
{ { (CODE;'AM4DP'), (DESC;'AM4DP workshop'), (CAT;'DSG'), (DUR;1) }, { (CODE;'AM4DP'), (DESC;'Applied Mathematics for Database Professionals'), (CAT;'DSG'), (DUR;5) } }
```

此表明确违反了作为 `tab_CRS` 定义一部分指定的表约束，因此不是 CRS 表结构的可接受表。因此，在此情况下，`Tx1a(dbs1)` 不是 `DB_UEX` 的元素。显然，当前对此事务的定义是有缺陷的，因为我们不希望事务导致数据库最终处于违反任何数据完整性约束的状态。

##### 改进的事务定义

将 `tcrs1` 插入数据库的更合适的事务定义如下，命名为 `Tx1b`。请注意，在此定义中我们重用了函数 `Tx1a`，该函数在本节前面已指定。

`Tx1b(dbs) := dbs , if Tx1a(dbs) ∉ DB_UEX`
`:= Tx1a(dbs), otherwise`

事务 `Tx1b` 描述了一次插入 *尝试*。如果 `tcrs1` 的插入导致数据库状态违反任何数据完整性约束（即，结果数据库状态不是 `DB_UEX` 的元素），则该插入被拒绝（也称为 *回滚*），数据库状态保持不变。否则，`tcrs1` 的插入被允许（事务 *成功执行* 或 *提交*）。

在此特定示例中，给定数据库状态 `dbs`，仅当 `dbs` 中的 CRS 表不包含 `code` 属性值等于 `'AM4DP'` 的元组时，插入尝试才会成功执行。但是，如果该元组恰好是 `tcrs1`，则插入尝试成功，但在此情况下根本不会添加元组。通过集合并集将元组添加到已包含该元组的表中，不会向表中引入新元素（元组）。

##### 约束的考量

至此，我们仅关注属性、元组和表约束。

在检查 `Tx1a(dbs)` 是否是 `DB_UEX` 的元素时，您还应考虑涉及 CRS 表结构的数据库约束。

**■ 注意**
您可以通过在形式化数据库约束规范中简单扫描字符串 `'CRS'` 来确定在插入元组到 CRS 的事务中所涉及的数据库约束。

以下是涉及的数据库约束（使用清单 7-39 中引入的简称）：`PSSR6`、`PTIJ9`、`PTIJ10`、`PTIJ12`、`PTIJ14`、`PTIJ15` 和 `PODC3`。在此特定示例中，将新的 CRS 元组插入到有效的数据库状态（即 `DB_UEX` 中的数据库状态）永远不会违反这些数据库约束之一（您可能需要自行验证这一点）。

因此，在此示例的讨论中，我们无需考虑这些涉及的数据库约束。

**■ 注意**
`Tx1b` 的定义涵盖了所有静态约束：属性、元组、表和数据库。

##### 状态转换约束

最后——这应该不足为奇——*状态转换* 约束应包含在事务的形式化定义中。

看看元组 `toffr1`。它代表一个需要插入数据库的 `AM4DP` 课程的授课安排。

```
toffr1 := { (COURSE; 'AM4DP'), (STARTS; '01-FEB-2007'), (STATUS; 'CONF'), (MAXCAP; 20), (TRAINER; 1126), (LOC; 'UTRECHT') }
```

此元组符合为 `OFFR` 表结构指定的所有属性和元组约束（参见第 7 章中 `chr_OFFR` 和 `tup_OFFR` 的定义）。

假设您希望尝试将 `offr1` 元组插入到其开始状态当前符合 `DB_UEX` 定义中所有静态约束的数据库中。我们进一步假设此事务的开始数据库状态符合以下附加特征。括号内列出了导致我们列出该特征的涉及约束。

*   `CRS` 包含一个代表 `AM4DP` 课程的元组 (`PSSR6`)。
*   `OFFR` 中尚无任何授课安排；这是要插入的第一个 (`tab_OFFR` 约束 `PTIJ13`、`PTIJ14`、`PTIJ15`)。
