# 索引类型

假设你需要一种方法，能够搜索图书馆中所有书籍的目录。目录提供了书籍的层级视图。有章节概述书籍的主要部分，随后是子章节标题，提供章节内容的更多细节。这种关系模型与 XML 文档的设计相似；有节点以及它们之间的关系，共同定义了信息的结构。

正如在讨论卡片目录时提到的，逐一查看图书馆中的每一本书以找到 Jason Strate 撰写的书，这效率不高。如果为了查明任何书籍中的任何章节是否由 Ted Krueger 撰写，而逐一查看图书馆的所有书籍，效率就更低了。每本书可能都有不止一章，这意味着每本书都需要检查多个值，并且在检查之前无法确定需要查看多少章。

解决这个问题的一种方法是列出图书馆中的每一本书，并列出每本书的所有章节。每本书在列表中有一个或多个章节条目。这提供了与卡片目录相同的优点，但适用于一些非标准的信息。在数据库中，这正是*XML 索引*和为 JavaScript 对象表示法（JSON）建立索引所能实现的。

对于 XML 文档中的每个节点，都会在 XML 索引中创建一个条目。这些信息持久保存在内部表中，SQL Server 可以使用这些表来确定 XML 文档是否包含正在查询的数据。类似地，对于 JSON，要索引的值通过计算列上的索引具体化。

创建和维护 XML 索引可能相当昂贵。每次更新索引时，它都需要将 XML 文档的所有节点分解到 XML 索引中。XML 文档越大，这个过程的成本就越高。但是，如果 XML 列中的数据将经常被查询，那么创建和维护 XML 索引的成本可以通过在运行时无需分解所有 XML 文档而迅速抵消。

## 空间索引

每个图书馆都有地图。有些地图覆盖海洋；有些则覆盖大陆、国家、州或城市。图书馆里可以找到各种地图，每张地图可能对相同区域提供不同的视图和信息。所有这些地图都存在两个基本挑战。首先，你可能想知道哪些地图重叠或包含相同的信息。例如，你可能对所有包含明尼苏达州的地图感兴趣。第二个挑战是，当你想要找到图书馆中所有在特定地点编写或出版的书籍时。同样，在这种情况下，在明尼阿波利斯 25 英里范围内编写了多少本书？

这两者都带来了一个问题，因为传统上，数据库中的数据是相当一维的，这意味着数据代表离散的事实。在物理世界中，数据通常存在于不止一个维度。地图是二维的，建筑物和楼层平面图是三维的。为了解决这个问题，SQL Server 提供了创建*spatial indexes*的功能。

空间索引将提供的空间信息分解为数据的四级表示。这种表示允许 SQL Server 在记录中绘制空间信息（包括`geometry`和`geography`），以确定行在哪里重叠以及一个点与另一个点的接近程度。

空间索引存在一些限制。主要限制是空间索引必须创建在具有主键的表上。没有主键，空间索引的创建将不会成功。此外，在创建空间索引时，它们被限制使用并行处理；并且一次只能在一个表上构建一个空间索引，这影响了它们的创建速度。另外，空间索引不能在索引视图上使用。这些以及其他限制在第 5 章中有介绍。

与 XML 索引类似，空间索引有与其大小相关的前期成本和维护成本。好处是，当需要使用查询空间数据的特定方法来查询空间数据时，空间索引的价值可以很快体现出来。空间索引将在第 5 章中更深入地讨论。

## 哈希和范围索引

当书籍进入图书馆时，有时它们被归还的频率超过了被放回书架的速度。将书籍分类并放回原处需要时间。在这些时候，图书馆管理员通常会在那里跟踪归还的书籍。对于这些书籍，图书馆管理员通常能记住归还书籍队列中哪些书在哪里，并能在不使用卡片目录的情况下找到你想要的书。这本质上就是内存优化表使用哈希和范围索引所做的事情。唯一的区别是，借助哈希和范围索引，数百万行（或书籍）可以保存在内存中，而无需依赖基于磁盘的结构来支持它们。

哈希索引允许内存优化表提供表中数据的点查找。换句话说，索引（或图书馆管理员）每次需要时都能准确记住表中书籍的位置并为书建立索引。

或者，范围索引为内存优化表提供了高效识别项目范围的能力。例如，如果索引（或图书馆管理员）需要所有在上午 8 点到 12 点之间归还的书籍，索引将能够扫描行，而不是以逐行操作的方式访问行。

对于范围索引和哈希索引，有几点需要考虑。首先，哈希和范围索引只允许在内存优化表上使用。关于记住书籍位置的引用，正是哈希和范围索引与其他索引类型的不同之处。在磁盘和内存之间，聚集索引的结构相对不变。而对于哈希和范围索引，其结构是专门为快速内存访问而设计的，并仅利用磁盘来支持事务一致性以及在数据库上线时在内存中重建索引的能力。

## 全文搜索

最后一个要探讨的场景是在书中查找特定术语的想法。卡片目录在提供按作者、主题或标题查找书籍的信息方面做得很好。一本书的主题并不是你可能想用来搜索书籍的唯一关键词。许多书的后面都有关键词索引，帮助你在书中查找其他主题。当这本书完成时，将会有一个索引，其中会有`full-text search`这个条目，并附有指向本页以及本书中讨论此内容的其他页的引用。

试想一下，如果图书馆里的每本书都有一个关键词索引。此外，让我们把所有这些关键词放到它们自己的卡片目录中。有了这个卡片目录，你将能够找到图书馆中每一本提及`full-text search`的书籍，以及讨论此内容的每一页的引用。一般来说，这就是全文搜索实现所提供的功能，只是它几乎涵盖了书中的所有词语。
