# 1. 索引基础

本书的目标是帮助你通过使用索引来提升数据库的性能。为了达成这一目标，你必须首先理解什么是索引以及为什么需要它们。你需要理解聚集索引、列存储索引和堆表中数据存储方式的差异。你还将了解非聚集索引及其他索引类型的构建方式，以及索引如何与其他索引相互作用。本章将为你理解索引的逻辑设计奠定基础。

## 为何要构建索引？

任何企业拥有的最重要资产就是其数据。数据库的存在就是为了存储这些数据。提供数据的一个关键环节是高效地交付它。能够高效地访问数据，可以提升企业从数据中获得的价值。实现这一点的方法就是通过索引。

索引是在用户和数据之间提供高效访问路径的手段。通过提供这种访问路径，用户可以向数据库请求数据，而数据库将知道去哪里检索数据以及如何以最小的代价来完成。

为什么不直接将所有数据放在一张表中，并在需要时返回呢？为何要费心创建索引？在需要时返回数据实际上正是索引的目的所在；它们提供了以尽可能快的方式获取数据所必需的路径。如果没有索引来提供数据位置的地图，数据库系统就必须搜索所有可用数据才能确认所需的数据已被访问。在今天，太字节的数据已很常见，能够快速高效地获取所需数据至关重要。

为了说明这一点，让我们考虑一个常用来描述索引的类比——图书馆。当你去图书馆时，会看到一排排的书架。在这个图书馆里，一个反复执行的常见任务就是找书。大多数时候，你对需要的书有特定的要求，你有几种选择来找到那本书。

在图书馆中，书籍使用杜威十进分类法存储在书架上。该系统根据书籍的主题分配一个编号。一旦编号被分配，书籍就会按数字顺序存放在图书馆内。例如，科学类书籍的编号范围是 500-599。从那里，如果你想要一本数学书，你会查找分类为 510-519 的书籍。然后，要找一本几何书，你会查找编号为 516 的书。有了这个分类系统，查找任何主题的书籍都变得简单而高效。一旦你知道了要找书籍的编号，你可以直接去图书馆存放 516 号书籍的区域，而不是在图书馆里闲逛直到偶然发现几何类书籍。这正是索引的工作方式；它们提供了一种有序的信息存储方式，使用户能够轻松地找到数据。

然而，如果你想在图书馆中找到所有由杰森·斯特拉特编写的书，会发生什么？你可以做一个有根据的猜测，认为它们都被归类在数据库类下，但你必须确切知道这一点。唯一的办法就是走遍图书馆，检查每一个书架。图书馆对此有一个解决方案——卡片目录。

如今，大多数卡片目录都可以通过计算机终端获取，但在我的童年时代，它们由单独的卡片组成，按作者、标题、主题和类别排序。使用卡片目录，你可以找到任何一本书的杜威十进制编号。例如，按作者搜索，你可以找到杰森·斯特拉特编写的所有书籍。因此，你不必在书架间徘徊并检查每本书是否是我写的，而是可以直接去找图书馆里我写的特定书籍。本质上，这也是索引的工作方式。索引提供数据的位置，以便查询可以直接访问数据。

没有这些机制，在图书馆找书，或在数据库中找信息，都会变得困难。你不会直接获取信息，而是在图书馆的书架间穿梭，试图找到所需之物。在较小的图书馆，比如“免费小图书馆”，这不是什么大问题，因为书太少了。但随着图书馆变大并安置在一座建筑里，浏览所有书架就不再高效了。当需要进行研究和寻找书籍时，根本没有时间浏览所有东西。

希望这个类比让你理解了索引的目的和必要性。在接下来的章节中，我将更深入地剖析这个类比，并将其与 SQL Server 数据库中可用的不同索引选项配对说明。

## 主要索引类型

你可以用不同的方式对索引进行分类。然而，理解本节描述的四个主要类别至关重要：堆、聚集索引、列存储索引和非聚集索引。堆、聚集索引和聚集列存储索引直接影响底层表中数据的存储方式。非聚集索引独立于数据存储。理解索引的第一步是掌握这种分类方案。

### 堆表

正如图书馆类比中所提到的，在“免费小图书馆”中，可用的书籍经常变化；通常只有一两个短书架。在这种情况下，所有者不会花时间按照杜威十进系统整理书籍。相反，书籍在获得时就被放置在书架上。在这种情况下，图书馆中书籍的存储没有顺序。当 SQL Server 以类似方式在表中存储数据，即数据缺乏有序结构时，它被称为 *堆* 。

在堆中，添加到索引的第一行是表中的第一条记录，第二行是表中的第二条记录，第三行是表中的第三条记录，依此类推。数据中没有任何内容用于指定数据被添加的顺序。数据和记录存在于表中，没有任何特定的顺序。

当表首次创建时，初始的存储结构称为 *堆* 。这可能是最简单的存储结构。行按它们被添加的顺序插入到表中。表在堆结构下运行，直到在表上创建了聚集索引或聚集列存储索引，或者该表被创建为内存优化表（将在第 7 章讨论）。一个表可以是堆，仅当没有其他索引类型定义表上基础数据的存储方式。每个表只允许一个堆结构。

### 注意

大多数人并不认为堆是索引。这没关系。在本次讨论的语境中，我们将它们视为索引，因为它们有助于确定数据将位于何处以及查询将如何访问数据。



### 聚簇索引

在图书馆的类比中，你了解到杜威十进制系统定义了书籍在图书馆中如何排序和存储。无论书籍何时被添加到图书馆，在杜威十进制系统下，它都会根据其主题被分配一个编号，并放置在同主题书籍之间的书架上。书籍的位置由其主题决定，而非其添加时间。这种结构是图书馆内查找书籍最直接的方法。在数据库表的语境中，提供此功能的索引被称为`clustered index`。

使用`clustered index`时，会选择一个或多个列作为索引的`key columns`。`key columns`用于排序并决定数据在表中的存放位置。图书馆根据杜威十进制编号将书籍放置在书架上，而`clustered index`则基于索引`key columns`的逻辑顺序来决定表中记录的存储位置。

用作`clustered index`的`key columns`是基于访问表中记录的最常用方法来选择的。例如，在一个包含州和省的表中，查找记录最常见的方式可能是通过其缩写。在这种情况下，使用缩写作为聚簇键列将是最佳选择。对于大多数表，主键或业务键将用作`clustered index`的`key columns`。

与堆（heaps）一样，`clustered index`决定了数据在表中的存储位置。在`clustered index`中，键列之外的数据与键列存储在一起。这等同于`clustered index`决定了物理表本身，就像堆定义了表一样。因此，一个表不能有多个`clustered index`。

### 非聚簇索引

正如我的类比中所指出的，杜威十进制系统并不能涵盖人们可能需要查找书籍的所有方式。如果已知作者或书名但不知道主题，那么分类法就没什么价值。图书馆通过卡片目录解决了这个问题，它提供了一个地方，可以交叉引用书籍的分类号与作者姓名或书名。数据库也能够通过非聚簇索引解决这个问题。

在`nonclustered index`中，会根据其值来选择和排序列。这些列包含指向与之相关的数据所在的堆或聚簇索引位置的引用。这与图书馆中卡片目录的工作原理几乎完全相同。书籍（或表中的记录）的顺序不会改变，但会根据其他搜索条件创建一条指向数据的快捷路径。

`nonclustered index`没有与堆和聚簇索引相同的限制。一个表上可以有多个`nonclustered index`，实际上最多可以有 999 个。这允许为用户创建替代路径以获取所需数据，而无需遍历表中的所有记录。一个表可以有很多索引并不意味着它应该有，正如我将在本书后面讨论的那样。

#### 列存储索引

大型图书馆卡片目录的问题之一是，可能有几十张或几百张索引卡片匹配一本书的标题。每张这样的索引卡片都包含作者、主题、标题、国际标准书号（ISBN）、页数、出版日期以及杜威十进制编号等信息。在几乎所有情况下，这些额外的信息并不需要，但它可以帮助在必要时筛选索引卡片。

想象一下，如果不需要查看几十张或几百张索引卡片，而是只有几张卡片，上面只包含标题和杜威十进制编号，或者只包含主题和杜威十进制编号。基本上，不是将所有属性存储在一起，而是将它们分开存储，并包含一个标识符（如杜威十进制编号）以将它们重新链接起来。对于每个属性，以前需要查看几十张或几百张索引卡片，现在只剩下几张合并的索引卡片。这种类型的索引被称为`columnstore index`。

`columnstore index`在 SQL Server 2012 中是新增的，并在后续的 SQL Server 版本中得到了极大扩展。传统上，索引以基于行的组织存储，也称为`rowstore`。当请求单行或小范围行时，这种存储形式效率极高。当返回大范围行或所有行时，`rowstore`可能变得低效，特别是在存在聚合或只需要少数几列的情况下。`columnstore index`通过以列式组织存储数据，更利于返回大范围的行。

创建`columnstore index`时，会包含表中的所有列。这确保了所有列都能受益于列式组织的性能提升。在`columnstore index`中，不是将一条记录的所有列存储在一起，而是每个列单独存储，并与索引中的所有其他行放在一起。这种索引的好处是，只需读取查询所需的列和行。在数据仓库场景中，查询结果通常只需要索引中不到 15%的列。^(¹)

由于其结构，`columnstore index`对数据仓库具有重要价值。首先考虑，索引只访问执行查询所需的列。此外，压缩效果大大提高，因为单个列内的数据具有更高的相似性。结合这两方面，`columnstore index`提供了显著的性能改进。我将在后面的章节中更深入地讨论这些内容。

### 其他索引类型

除了刚刚讨论的索引类型外，还有许多其他索引类型可用。这些其他类型涵盖了不适合传统索引结构的专门搜索、数据和表类型。这些类型包括 XML、空间、哈希和范围以及全文搜索（FTS）索引，每种都有专门的章节来介绍其专门的索引结构。虽然这些不一定符合迄今为止概述的卡片目录场景，但在处理相关的数据和表类型时，它们是重要的选项。为了帮助说明，我将展示如何为图书馆添加一些新功能。后面的章节将进一步扩展这里介绍的信息。
