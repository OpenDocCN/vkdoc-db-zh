# 6. 递归子查询因子化

子查询因子化子句（有时也称为 `with` 子句或 CTE – 公用表表达式）在 Oracle 9.2 中引入。当时它不允许我们定义递归子查询，主要用于将逻辑分解为命名查询——将子查询提取出来，并在主查询中通过名称引用它们。CBO（基于成本的优化器）可以决定是物化提取出的子查询结果，还是将其作为内联视图插入。在前一种情况下，如果命名查询在主查询中被多次引用，可以提高性能；而在后一种情况下，可能会对性能产生负面影响，因为转换引擎不以与内联视图相同的方式处理命名查询。例如，在旧版本中，Oracle 可以合并内联视图，但不能合并具有完全相同文本的命名查询。

从 Oracle 11.2 开始，子查询因子化子句允许我们递归执行查询，如果子查询引用了其自身的名称，这在清单 6-1 中进行了示意性展示。

```
with rec as
(
anchor_query_text - 锚定成员
union all
recursive_query_text – 引用 rec 的递归成员
)
select *
from rec

```

执行时使用以下算法：

1.  运行锚定成员以获取基础结果集 `Set[0]`。
2.  使用前一次迭代的结果集 `Set[i-1]` 执行递归成员。
3.  重复步骤 2，直到返回空结果集。
4.  返回最终结果集，即 `Set[0] … Set[n]` 的 `union all`。

清单 6-2 中的查询遍历了上一章（清单 5-1）介绍的 `tree` 表中的树。你可能会注意到结果的顺序与 `connect by` 方法不同。

```
with rec(lvl, id, path) as
(
select 1 lvl, id, cast('->' || id as varchar2(4000))
from tree where id_parent in (1, 10)
union all
select r.lvl + 1, t.id, r.path || '->' || t.id
from tree t
join rec r on t.id_parent = r.id
)
select lvl,
rpad(' ', (lvl - 1) * 3, ' ') || id as id,
path
from rec;

2    13      ->11->13

```

要返回与 `connect by` 相同顺序的结果，你必须指定 `search depth first`——在这种情况下，节点的所有后代将在其层级上的其他节点之前返回——参见清单 6-3。默认排序是 `search breadth first`，因此先前层级的所有节点在当前层级的节点之前返回。正如将在“遍历层次结构”一节中展示的那样，顺序只影响结果的返回方式，而不影响遍历层次结构的方式。
