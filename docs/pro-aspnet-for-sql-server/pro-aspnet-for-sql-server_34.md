# 第 6 章

#### 缓存

也许加速应用程序的最佳方法是实施有针对性的缓存策略。当你对应用程序进行性能分析时，通常会发现从数据库提取数据的时间是页面请求周期中最显著的延迟。你需要从数据库提取的数据越少，应用程序的执行速度就越快。ASP.NET 内置的缓存机制为你提供了大量的缓存选项，它建立在 .NET 1.1 的坚实基础上，并在 .NET 2.0 中增加了更多功能。

在 ASP.NET 中，缓存由 `Cache` 对象处理，该对象是 `System.Web.Caching` 命名空间的一部分。尽管它位于 `System.Web` 程序集中，但它可以轻松地用于任何 .NET 应用程序，包括控制台和桌面应用程序。你可以像引用任何其他程序集一样引用它。而对于 ASP.NET 网站，`Cache` 对象可以在类库中使用，以创建可重用的数据访问层。

本章涵盖以下内容：

-   缓存的替代方案
-   缓存选项
-   缓存的问题
-   使缓存失效
-   性能策略

本章的重点主要放在 ASP.NET 上，这很大程度上是由于 HTTP 的无状态特性。已经创建了许多策略和技术来克服请求-响应周期的无状态性质，例如 Cookie 和会话，但事实仍然是网站以分离模式工作。相比之下，桌面应用程序可以加载数据并将其可靠地保存在内存中，并持续访问它。桌面应用程序也是单用户的，这允许应用程序为该单一用户占用所有资源。这些因素降低了桌面应用程序使用 Web 应用程序所采用的相同缓存技术的需求。

对于网站，你可能需要为多个会话提供服务，每个会话都保存用户的数据，每个会话都会占用内存空间。如果你为每个用户的会话加载占用 3 MB 空间的数据，并且有 1,000 个并发会话，那么你将已经占用 3 GB 的内存。默认情况下，会话将保持活动状态，直到在 20 分钟不活动后超时。虽然服务器拥有 4 GB 或更多内存并不少见，但占用那么多空间是不合理的。最好是在需要空间时（例如像网站上发生的那样，流量高峰期突然出现 5,000 个活动会话）让数据从内存中清除。

## 第 6 章 ■ 缓存

对于典型的访问，用户可能访问几个页面并在不到 10 分钟内离开。也许他们正在查看你网站上某个产品的价格或订单状态。他们希望快速获取信息然后离开。一旦他们离开，你希望有一种方法来清理他们使用的内存，以便其他用户可用。这正是缓存填补实际需求的地方。

还有一种情况是，特定数据集被许多用户同时使用。也许一个热门产品刚刚开始销售。每次对产品页面的访问都会触发一系列查询数据库以显示页面的操作。数据实际上并不经常变化，但在一秒钟内，你可能有 20 个用户访问该页面并运行相同的查询来加载页面。这会给数据库带来负载，导致越来越多的 I/O 延迟，因为数据库在短时间内执行大量工作。它还填满了数据库服务器和 Web 服务器之间的网络连接，如果网络没有足够的容量来处理带宽，这可能成为瓶颈。如果产品页面或查询被包装了缓存机制，即使它只使用五秒钟的超时，你也可以显著提高页面性能，并减少数据库服务器和网络的负载。这是缓存解决的另一个实际问题。
