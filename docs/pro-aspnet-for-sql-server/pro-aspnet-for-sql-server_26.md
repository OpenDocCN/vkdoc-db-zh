# 第 5 章 ■ SQL 提供程序

能够在 Web 窗体中使用您自己的控件来表示的提供程序。我发现，在构建模块化网站时，SQL 提供程序与控件的结合是一个强大的组合。

ASP.NET 附带的提供程序旨在解决经典 ASP 和 ASP.NET 1.1 中通过自定义代码反复执行的常见任务。此类常见任务包括管理用户和角色，以及创建控件以允许这些用户创建账户、登录和登出、更改和检索密码以及自定义用户账户。早在经典 ASP 时代，就可以使用表单身份验证和数据库来创建 ASP.NET 2.0 提供的所有控件，但您首先必须设计数据库架构，然后构建所有控件来处理用户账户所需的所有任务。由于缺乏通用的实现，开发人员不得不从头开始构建自己的版本，这既耗时，也无法提供与 ASP.NET 2.0 控件所提供的同样丰富、高度定制的功能。

从数据库到用户界面，有多种方法可以快速部署、自定义、混合搭配以及完全替换实现，同时仍能保持与系统内其他标准组件的兼容性。当您能够掌握这些组件可用的所有细微差别时，您将能够创建一个具备现代网站所有预期功能的网站，即使存在 ASP.NET 2.0 设计未曾预见的独特需求。它在创建时就考虑到了可扩展性。

在本章中，我将介绍设置三个最常用提供程序的基础知识，并向您展示它们的灵活性。然后，我将向您展示如何实现您自己的提供程序，包括一个从头开始构建、与 ASP.NET 2.0 提供程序和控件无缝集成的提供程序。

## SqlMembershipProvider

最常用的 SQL 提供程序无疑是 `SqlMembershipProvider`，它用于管理您网站的用户。在第一章中，您使用了 `aspnet_resql.exe` 实用工具将成员资格服务添加到数据库。借助这些表、视图和存储过程以及配置设置，您就能够使用各种利用 `Membership` API 来操作 SQL Server 中数据的控件。这里值得关注的控件是表 5-1 所示的 `Login` 控件。

**表 5-1.** `Login` 控件

| 控件 | 描述 |
| :--- | :--- |
| `Login` | 显示登录表单并处理身份验证 |
| `LoginView` | 使用两种模板之一（用于已认证用户或匿名用户）显示内容 |
| `LoginStatus` | 当用户为匿名时显示登录链接，当用户已认证时显示注销链接 |
| `LoginName` | 显示用户的登录名 |
| `PasswordRecovery` | 显示一个用于恢复丢失密码的表单 |
| `CreateUserWizard` | 显示一系列用于创建新用户账户的步骤 |
| `ChangePassword` | 显示一个允许用户更改其密码的表单 |

表 5-1 中的控件都通过 `Web.config` 文件中的设置以及控件本身的属性进行配置。默认值通常是合理的，但如果您的需求与默认值不同，更改其行为以满足您的需求并不困难。

`SqlMembershipProvider` 只是 Microsoft 随 ASP.NET 2.0 发布的两个提供程序之一，另一个是 Active Directory 实现。`SqlMembershipProvider` 符合基类 `MembershipProvider` 的必要接口，并准确执行与 SQL 实现等效的工作。正如名称所示，一个使用目录，另一个使用数据库来管理用户。这种通用接口是 ASP.NET 控件用来执行相同行为的依据，而不管底层实现如何。

### 接口还是抽象类？

提供程序实现所使用的基类实际上是抽象类，它们首先继承自 `ProviderBase` 类，该类添加了所有提供程序实现共有的基础方法和属性。由 `MembershipProvider` 定义且必须实现的方法和属性被标记为 `abstract`。提供程序之间的通用功能可以在抽象基类中实现，就像 `ProviderBase` 在 `Initialize` 方法中设置 `Name` 和 `Description` 属性一样。您使用此基类来定义必需的方法，同时也使开发人员更容易实现他自己的实现。

尽管 Active Directory 和 SQL 实现都处理了 `MembershipProvider` 基类的必需方法，但它们有几个与其自身实现相关的独特配置设置。例如，Active Directory 实现有一个名为 `attributeMapEmail` 的属性，SQL 实现不使用该属性。当您需要创建自定义实现，而其需求远超出这两个现有实现所预期的范围时，使用独特配置设置的能力将非常有用。

##### 使用 XML 实现

SQL 实现的另一种替代方案是 XML。您可以将 `DataSet` 序列化为磁盘上的 XML 文件，并在需要时加载它。当您创建自定义 SQL 提供程序时，必须创建表和存储过程，这比在内存中以编程方式构建 `DataSet` 并将其序列化输出到 XML 文件要复杂得多。通过这种方式，您可以构建自定义提供程序的快速原型，并且在您有控件通过提供程序接口使用它之后，您可以实现 SQL 实现以获得更好的可伸缩性。因为 XML 实现与提供程序实现兼容，即使在您拥有 SQL 实现之后，它仍然可以在开发期间使用。当您与数据库断开连接（无论是主动还是被动）时，XML 版本会非常方便。在 XML 和 SQL 提供程序之间切换只需更改配置中的 `defaultProvider` 属性。

> **注意：** Membership 提供程序的 XML 实现不会自动拥有与 SQL 实现相同的所有用户，但您可以通过创建一组满足您需求的示例用户，在开发期间在 XML 和 SQL 配置之间切换。在第 1 章中，清单 1-19 的示例展示了如何在应用程序启动时也自动创建用户和角色。

##### 设置数据库连接

将 `SqlMembershipProvider` 连接到数据库是通过 `connectionStringName` 配置属性完成的。请注意，这是一个名称，而不是连接字符串。在 .NET 1.1 中，通常将连接字符串作为 `appSettings` 中的另一个设置。在 .NET 2.0 配置文件中，`connectionStrings` 元素通过名称和 `connectionString` 属性表示数据库连接。这种正式的构造将连接字符串设置与应用程序的其他通用设置分离开来，使得可以通过编程方式访问这些值。在 `SqlMembershipProvider` 配置中，`connectionStringName` 指的是连接字符串的 `name` 属性，将提供程序实现绑定到该数据库。

您会发现，可以为数据库服务器中的不同数据库目录设置多个连接字符串，并将每个提供程序实现配置到不同的数据库。您可以选择将所有 ASP.NET 生成的资源隔离在与应用程序表分开的数据库中，以实现清晰的分离并便于维护。
