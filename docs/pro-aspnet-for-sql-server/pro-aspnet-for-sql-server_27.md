# 使用 SQL 提供程序

在开发包含这些提供程序配置的网站时，您可能会发现需要在工作站的本地数据库副本与办公室服务器上的远程数据库之间进行切换。您可以配置每一个数据库连接，并将提供程序配置所使用的名称切换为您想要激活的那个。我通常使用本地数据库副本进行工作，这样可以避免他人直接修改服务器上的数据库对我造成干扰。您可能会维护一个所有开发人员共用的过渡数据库，用于测试集成构建。当您在本地副本上完成一组功能后，可以将数据库更改部署到该过渡数据库，重新配置您的网站以连接过渡数据库，然后运行测试以验证您的更改是否与他人可能已做的更改正确集成，之后再将您的更改签入源代码控制。

##### 创建密码策略

为一个新网站配置 `SqlMembershipProvider` 时，您必须选择密码的处理方式。这是使用 `MembershipProvider` 的第一个障碍，因为默认的密码策略有些严格。它要求密码至少七个字符，并且至少包含一个非字母数字字符。像 `aspnet9` 这样的密码是不允许的，但 `aspnet9!` 则可以。在开发过程中，我经常创建新账户并登录，因此简化这个要求（至少在开发期间）是我的首要步骤之一。根据您的公共网站所需的安全级别，您可以放宽密码策略，不要求非字母数字字符，并可能将所需长度降至六个字符。或者，您可以配置一个正则表达式来验证新密码。表 5-2 展示了一些您可以使用的示例正则表达式。

**表 5-2.** 密码正则表达式

`^.{7,10}$`
密码可以包含任何字符，长度为七到十个字符。

`^[a-zA-Z]\w{7,14}$`
密码必须仅为字母，长度为七到十四个字符。

`^(?=.*\d).{7,12}$`
密码必须包含至少一个数字字符，长度为七到十二个字符。

`^(?=.*\d)(?=.*[a-z])(?=.*[A-Z]).{7,15}$`
密码必须至少包含一个大写字母、一个小写字母和一个数字。其长度必须为七到十五个字符。

当然，您可以使用更严格的正则表达式要求，但应将其与用户能接受的程度相平衡。如果它只是一个关于您梦幻橄榄球联盟的在线讨论论坛，您可以采用宽松的密码策略。但如果是一个通过 SSL 保护的、管理银行账户的网站，您应该倾向于使用最后一个示例。

**注意** 您可以在 `http://regexlib.com/` 上在线获取更多正则表达式示例。搜索 `password`，您将得到一个密码正则表达式列表。

## SqlRoleProvider

`SqlRoleProvider` 是 `RoleProvider` 的一个实现，它提供了一种对用户进行分组的方法。如果您正在使用 `SqlMembershipProvider`，则并非必须启用 `RoleProvider`。并且如果两者都启用，您甚至可以选择不同类型的提供程序。您可以将网站配置为使用 `Active Directory 成员资格提供程序` 和 `SqlRoleProvider`，它将通过提供程序接口按预期工作。

`SqlRoleProvider` 的功能相当简单，但对于大多数网站来说非常必要。用户与角色相关联，角色可以以各种方式用于授予或拒绝访问权限，以及定制网站的行为。

##### 按角色控制访问

在网站使用了 `SqlMembershipProvider` 和 `SqlRoleProvider` 之后，您可能希望保护它。要完全锁定网站，您可以拒绝所有匿名用户的访问。在 ASP.NET 中，您可以在 `Web.config` 文件中使用 `authorization` 元素来实现。该元素包含 `deny` 或 `allow` 元素，这些元素指定了您的身份验证要求，无论是基于角色、用户，还是已验证或匿名用户。清单 5-1 展示了如何锁定网站，使得只有经过身份验证的用户才能进入。

**清单 5-1.** 仅允许经过身份验证的用户

```xml
<authorization>
  <deny users="?"/>
</authorization>
```

问号代表匿名用户。默认的身份验证是允许所有用户，无论其登录状态如何，如清单 5-2 所示。

**清单 5-2.** 允许所有用户

```xml
<authorization>
  <allow users="*" />
</authorization>
```

您也可以组合使用这些指令，首先允许某些用户（可能通过角色），然后拒绝所有其他用户。您的访问控制规则还可以限制在特定位置。清单 5-3 展示了一个必须放在主 `system.web` 配置部分之外的配置，因为 `location` 块不能被 `system.web` 节包含。通常，您将在主 `system.web` 配置中设置全局访问设置，然后紧接着添加额外的、限制性更强的规则。

**清单 5-3.** 仅允许管理员用户

```xml
<location path="Admin">
  <system.web>
    <authorization>
      <allow roles="Admin"/>
      <deny users="*"/>
    </authorization>
  </system.web>
</location>
```

## 注意，并非所有内容都被保护！

当添加授权要求以防止匿名账户访问您的页面时，您可能并未保护所有内容。您实际上只保护了映射到 .NET 运行时运行的 ASP.NET 资源。扩展名为 `.aspx`、`.asmx` 和 `.ashx` 等文件被映射到 .NET 运行时，但图片和文本文件则不然。那些不由 .NET 运行时管理的文件也不受此授权机制的保护。通常，您的关键数据位于网页上，但如果您的 ASP.NET 应用程序生成包含关键数据的图片或其他文档到磁盘，请记住这一点。

然而，当您使用 Visual Studio 2005 时，您会发现所有资源都根据配置受到保护。这是因为开发 Web 服务器提供的所有内容都通过 .NET 运行时运行，它遵循 `Web.config` 中的设置。如果您确实希望保护您的文件，您可以选择将文件类型的扩展名专门映射到 .NET 运行时，或者为应用程序扩展部分中未专门映射的每个文件设置通配符应用程序映射。后一个选项可能会增加服务器的负载，因为 IIS 更适合提供静态图像文件，因此如果您希望保护生成的 Excel 文档，最好将 `.xls` 扩展名映射到 .NET 运行时。您可以复制 `.aspx` 扩展名的映射。它引用以 `aspnet_isapi.dll` 结尾的 .NET 2.0 安装文件夹。

授权机制有一个例外，那就是登录页面。必须允许此页面，以便表单身份验证可以授予用户访问权限。默认的身份验证模式是 `Windows`，但要使用 `SqlMembershipProvider` 和 `Login` 控件，您必须将其更改为 `Forms` 模式。您可以决定使用不同的页面，但 `Login.aspx` 是默认的。您还可以将超时时间从默认的 30 分钟更改为大约 2 周（20,160 分钟），并将其设置为使用可滑动过期，这样只要用户在两周内返回网站，她就保持登录状态。如清单 5-4 所示的这个配置，对于门户或购物网站非常有用。

**清单 5-4.** 登录页面配置

<authentication mode="Forms">

<forms loginUrl="Login.aspx"

name=".ASPXFORMSAUTH"

timeout="20160"

slidingExpiration="true"/>

</authentication>

用户成功登录账户后，会被分配一个令牌，该令牌以 Cookie 的形式保存在客户端。此 Cookie 即被称为表单身份验证令牌。在清单 5-4 中，Cookie 名称为 `.ASPXFORMSAUTH`（这是默认名称）。该令牌是客户端与后端会员系统之间的连接纽带。

##### 按角色控制行为

作为网站的管理员，你可能会将自己添加到一个名为 `Admin` 的角色中。在清单 5-5 中，可以看到如何使用 `Admin` 角色来限制只有属于 `Admin` 角色的用户才能访问管理区域。当你登录网站时，可以显示指向管理区域的链接，而普通用户则看不到该链接。你只需检查当前用户是否在 `Admin` 角色中，并据此设置超链接的 `Visible` 属性。

**清单 5-5.** 设置 AdminHyperLink 的可见性

```
protected void Page_Load(object sender, EventArgs e)
{
    AdminHyperLink.Visible = Roles.IsUserInRole(User.Identity.Name, "Admin");
}
```

你可以更进一步利用这个机制。例如，在一个电商网站中，当客户首次购买时，你可能会将其添加到 `Customer` 角色，这将开始在其账户页面上显示订单历史和货运追踪等内容。如果你的电商网站与为网站提供产品的多个供应商合作，你可以将他们添加到 `Vendor` 角色，并授权他们访问以更新库存数据和预计发货时间。

请记住，所有角色信息都与你的应用程序数据一起存储在数据库中。虽然最好使用 `RoleProvider` 来与此数据交互，但你也可以使用底层存储过程来与会员数据交互。对于电商网站，你可能希望每月运行一次报告，查找过去三个月内购买商品总额超过 1000 美元的所有用户。然后，你可以使用 `aspnet_UsersInRoles_AddUsersToRoles` 存储过程将他们添加到 `PreferredCustomer` 角色，使其有资格享受折扣和特别优惠。当订单进来时，你会记录订单的商品、数量和价格，但将处理工作留到当天晚些时候批量处理。那时，你可以验证用户是否在 `PreferredCustomer` 角色中，并使用 `aspnet_UsersInRole_IsUserInRole` 存储过程给予他折扣。回到网站上，该用户将开始在主页上看到其他人无法看到的特别优惠（希望能赢得用户的忠诚度），这一切都基于用户与特权角色的关联。

拥有这些可访问的存储过程的另一个优势是，许多平台都可以与 SQL Server 协同工作。你的公共网站可能运行的是 ASP.NET，但你的订单履行流程可能使用另一种无法直接使用 .NET 框架的语言或平台来实现。在关于面向服务架构（SOA）作为整合不同平台的手段的诸多讨论中，我认为数据库本身已经是一个极好的集成点。几年前我在构建一个网站时，团队中有几位开发人员使用另一个软件平台开发桌面应用程序，用于处理供应商申请和订单履行流程。我使用数据库作为在不同平台之间安全传递数据的方式。

由于存储过程对其他平台同样可用，我不必为让他们适应我正在构建的系统而做任何额外的工作。

## SqlProfileProvider

ASP.NET 配置文件提供程序向用户账户添加了额外的属性。我们在第 1 章中提到过这一点。`SqlProfileProvider` 的有趣之处在于，它不要求你必须同时使用 `MembershipProvider` 的实现。这似乎没什么意义，因为如果你可以自定义用户账户，却无法登录并恢复你的偏好设置。但这并不是 ASP.NET 身份验证的工作方式。会员提供程序实际上只是表单身份验证之上的一个层，而表单身份验证已经存在好几年了。事实上，你可以直接使用表单身份验证完全实现一个会员系统，并且它与 `SqlProfileProvider` 兼容。你甚至可以在 `Web.config` 文件中存储用户名和密码，如清单 5-6 所示。

**清单 5-6.** Web.config 中的用户名和密码

```
<authentication mode="Forms">
  <forms loginUrl="Login.aspx"
         name=".ASPXFORMSAUTH"
         slidingExpiration="true">
    <credentials passwordFormat="Clear">
      <user name="admin" password="easy12guess"/>
      <user name="user" password="abc123"/>
    </credentials>
  </forms>
</authentication>
```

这一切或许可以解释为什么会员、角色和配置文件接口似乎混杂在一起，有如此多的地方可以收集关于用户的必要信息。`MembershipUser` 对象使用自 .NET 1.0 以来就存在的 `User` 对象。但是，尽管你可以在不使用 `MembershipProvider` 的情况下使用 `SqlProfileProvider`，但几乎没有理由这样做。

在使用 `SqlProfileProvider` 后，你需要做一个选择。你可以选择允许任何访问你网站的用户匿名维护其配置文件，或者不允许。这是一个常见功能，但有一些值得考虑的问题。

**注意：** 请记住，已经配置了默认提供程序，你在配置网站时需要先清除它们。否则，你会遇到意外行为。虽然可以配置和使用多个提供程序实现，但这并不常见。在配置网站时，以 `clear` 指令开始你的提供程序配置应该是第一步。

## 为什么需要匿名配置文件？

匿名配置文件可以像已验证的配置文件一样，存储与配置文件相关的自定义设置。即使重启 Web 浏览器，这些设置也会跟随用户。也许你希望允许新用户在要求她创建账户之前就使用你的网站。电商网站上的一个常见功能是允许用户匿名将商品添加到她的购物篮中，在她选择购买这些商品后，在结账过程中为她创建一个账户，这样她以后就可以返回、登录并访问她的订单状态或使用之前的账户信息下新订单。

越来越多的网站还提供一些小功能来定制其内容以适应用户，但这些微小的偏好设置并不值得用户在另一个网站上再创建一个账户。一个例子是 CNN 网站上的字体大小调整按钮，它允许你增大或减小页面上的文本大小（见图 5-1）。当你在几天后返回时，CNN 网站会记住你的偏好。这个单一设置本可以单独保存为一个 Cookie，但当设置配置文件属性如此容易时，你就可以依赖匿名配置文件了。

您还可以利用匿名配置文件来跟踪用户行为，将其操作记录至数据库。每当用户返回网站时，系统将记录此次访问。当用户深入浏览某个栏目并查看具体文章时，您也可以记录该细节。后续可分析所有这些数据，从而掌握访问网站的用户画像。或许您之后会将用户细分为七个群体，将长期用户分配到其中一个群体，并开始主动调整网站的功能、内容和广告以适应这些群体。

##### 配置匿名配置文件

匿名配置文件默认未启用，因此您需要将该设置添加到配置中（参见清单 5-7）。

**清单 5-7.** 启用匿名配置文件

```
<anonymousIdentification
    enabled="true"
    cookieName=".ASPXANONYMOUS"
    cookieTimeout="144000"
    cookieSlidingExpiration="true"/>
```

此配置为匿名会话设置了 100 天的超时时间，并启用滑动过期。如果打算启用匿名配置文件，您需要权衡其利弊。虽然让成千上万的匿名配置文件用户在考虑是否购买商品时使用您的商务网站将商品添加到购物篮系统会带来好处，但让所有这些匿名配置文件占用数据库空间也会产生成本。

##### 管理匿名配置文件

如果将超时时间设置为四周并启用滑动过期窗口，您可以认为四周内没有任何活动的匿名账户已不活跃，因为用户将无法重新连接到该配置文件。识别出这些非活动配置文件后，您可以从系统中清除它们。如果某些配置文件未存储重要数据，甚至可以更早清除部分配置文件。您会发现搜索引擎和其他自动化程序生成了大量匿名会话。这类匿名配置文件将构成数据库中配置文件的很大一部分。

您可以编写查询来识别数据库中的配置文件，并编写脚本来清除这些配置文件，但幸运的是，`Membership API` 已预见到此需求，因此您无需完成所有这些工作。

清单 5-8 展示了 `Web Form` 上一个按钮的事件处理程序。它可以放置在网站管理部分某个页面的后台代码中。当您想要清除非活动配置文件时，可以使其执行以下代码行。

**清单 5-8.** 使用 `Button1` 删除非活动配置文件

```
protected void Button1_Click(object sender, EventArgs e)
{
    ProfileManager.DeleteInactiveProfiles(
        ProfileAuthenticationOption.Anonymous,
        DateTime.Now.AddMonths(-1));
}
```

此代码删除过去一个月内没有任何活动的所有匿名配置文件。`ProfileAuthenticationOption` 枚举还包括 `Authenticated` 或 `All` 值，以便您能够删除任何配置文件。这仍然是手动过程。最好安排脚本在每晚运行，以维持一个滚动窗口，持续删除所有非活动配置文件。当 `MembershipProvider` 以 `SQL Server` 为后端时，您可以利用 `SqlMembershipProvider` 调用的存储过程 `aspnet_Profile_DeleteInactiveProfiles`。您必须传入应用程序名称、配置文件选项以及配置文件处于活动状态以来的日期。配置文件选项 `0` 表示匿名，`1` 表示已认证，`2` 表示两种配置文件类型。清单 5-9 展示了删除非活动配置文件的代码。

**清单 5-9.** 使用 T-SQL 删除非活动配置文件

```
DECLARE @PurgeDate DATETIME
SET @PurgeDate = DATEADD(DAY, -60, GETDATE())
EXEC aspnet_Profile_DeleteInactiveProfiles 'appName', 0, @PurgeDate
```

将此脚本设置为每晚运行，应能保持您的数据库清除不必要的配置文件数据。脚本设置为回溯 60 天，但 14 天可能更合适。
