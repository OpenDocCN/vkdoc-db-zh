# 第九章 部署

提升应用程序性能的最大障碍是部署成本。这个过程伴随着一定程度的风险，因为它涉及更改生产环境，可能导致停机。因此，它需要大量的质量保证测试以及发布所需的资源，这些工作可能在非工作时间进行，以最小化对应用程序用户的影响。所有这些加起来构成了一个成本高昂的过程，这可能足以将性能改进推迟到几周或几个月后的某个发布窗口。采用正确的部署模型，风险和成本可以大大降低，从而使及时的更新能够被推送到生产环境。

本章涵盖以下内容：

*   使用`MSBuild`进行自动化
*   部署网站
*   部署数据库
*   自动化配置更改
*   自动化数据库更新
*   自定义配置节

部署应用程序发布版本有正确的方式和错误的方式。几年前，我曾与一位开发者合作，他会创建生产数据库的备份，下载它，在本地还原，然后对数据库进行更改。接着，他会创建一个新的备份，上传并将其还原到服务器上。这个过程明显存在一些问题，回想起来让我既好笑又心惊。我很庆幸自己从未不得不遵循同样的流程。如今，使用可用的工具，有好得多的方法来推送更新。

在接触`SQL Server`之前，许多开发者通过将数据库文件上传到网站可以直接访问的服务器，将`Microsoft Access`数据库连接到经典`ASP`应用程序。由于服务器上通常没有安装`Microsoft Access`，开发者会将应用程序下线，拉取数据库文件，更新它，然后将其放回原处并使应用程序重新上线。这个旧流程或许可以解释为什么前一个例子中的开发者对`SQL Server`数据库使用了非常相似的流程。他只是在做他一直做的事情，尽管有更好的方法。

## 使用 MSBuild 自动化

使用微小更改持续更新数据库可能是一项繁琐且重复的工作，但必须完成。幸运的是，这项工作可以以最小的努力实现自动化。自动化工具`MSBuild`是`.NET`平台的标准构建工具。它是一个脚本系统，使用`XML`来定义一系列包含一组任务的构建目标。目标通过依赖关系链接在一起。一旦您了解了基础知识，`Visual Studio`将为`MSBuild`脚本提供智能感知支持，帮助您入门。

`MSBuild`脚本几乎可以做任何事情。如果标准任务不能满足您的需求，您可以使用`Exec`任务调用命令行来运行其他工具，或者创建一个自定义任务。但您不希望过度使用`MSBuild`脚本。我在处理`MSBuild`脚本时遵循三个首要规则。

首先，我将脚本完成的工作保持在最低限度。自动化所有事情可能很诱人，但最终您将花费更多时间维护脚本，而不是在应用程序上完成工作，而应用程序应该是您的主要关注点。

其次，我只自动化那些我发现自己反复执行的耗时工作。

每天多次花费几分钟的任务是理想的自动化候选者。虽然向脚本添加功能可能需要一段时间，但最终将为每个发布版本节省数小时的工作。

最后，我将脚本与解决方案一起进行版本控制。随着构建过程随时间变化，您将更新构建脚本以适应这些更改。这些调整应该与应用程序的其余部分一起在源代码管理中保持最新。当您生成发布版本时，您将能够从源代码管理中拉取较旧的版本并进行构建，即使自该版本部署以来构建过程已经发生变化。通过在源代码控制系统中使用发布分支，您可以在继续在主分支上进行开发的同时，支持应用程序的生产版本。

### 发布分支

在源代码控制中管理项目的方法有很多。使用`Microsoft Team Foundation Server`和`Subversion`等现代源代码控制系统，您可以创建分支来隔离项目中的更改。

您可以为每个开发者、错误修复或您选择的任何其他原因创建分支。然而，大型团队最终采用的分支模型是*发布分支*。开发在主分支（主干）上进行，当准备发布时，会为该发布创建一个新分支，以便将其与持续开发隔离。然后应用程序从这个发布分支推送。之后，当部署的应用程序需要更新时，更改在分支上进行，并在必要时合并回主干。

`MSBuild`脚本是一个由三个主要元素组成的`XML`文档：`PropertyGroup`、`ItemGroup`和`Target`，它们都包含在根元素`Project`中。这是每个`MSBuild`脚本的基本结构。`PropertyGroup`元素定义了在脚本其余部分使用的属性。`ItemGroup`元素定义了属性的集合。`Target`元素包含多个任务，例如`MakeDir`和`Copy`。清单 9-1

展示了一个基本`MSBuild`脚本的框架。

**清单 9-1.** *基本 MSBuild 脚本框架*

```xml
<?xml version="1.0" encoding="utf-8"?>
<Project DefaultTargets="Build"
>
  <PropertyGroup>
    <Configuration Condition=" '$(Configuration)' == '' ">Debug</Configuration>
    <PackageName>MyProject</PackageName>
    <DatabaseDirectory>Database</DatabaseDirectory>
    <WebsiteBinDirectory>Website\bin</WebsiteBinDirectory>
  </PropertyGroup>
  <ItemGroup>
    <!-- 集合定义 -->
  </ItemGroup>
  <Target Name="Build">
    <!-- 构建任务 -->
  </Target>
</Project>
```

虽然`MSBuild`附带的标准任务涵盖了许多常见任务，但它们并未涵盖自动化构建和部署工作所需的所有内容。一个特别的任务是`Zip`任务，我经常使用它来创建文件集合，以便可以轻松地归档并在环境之间移动。幸运的是，有一个名为`MSBuild Community Tasks`的开源项目，其中包括`Zip`任务以及许多其他有用的任务（请参阅侧栏“MSBuild 社区任务”）。

### MSBuild 社区任务

`MSBuild`可以通过创建自定义任务来扩展。一个已成为自定义`MSBuild`任务聚集点的开源项目是`MSBuild Community Tasks`项目，它不断向已相当广泛的列表中添加更多自定义任务（参见 http://msbuildtasks.tigris.org/）。诸如`Zip`、`FxCop`和`Prompt`等任务是为此脚本环境添加关键功能的几个例子。
