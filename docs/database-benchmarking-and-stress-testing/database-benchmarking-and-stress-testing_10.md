# 5. 基准测试常见错误

## 工作负载捕获

有时我们很幸运，拥有相应的工具和能力来捕获实际工作负载，然后将其回放作为我们的数据库基准测试方法。之前提到的一些工具以及部分数据库供应商提供了这样的功能。此时，关键问题在于：选择哪个时间段来捕获工作负载才是合适的？再次强调，咨询了解应用程序的内部人员，远比询问外部顾问要好得多。你可能会发现需要捕获多种工作负载：一个典型的业务日、一个典型的业务夜，再加上一个或多个业务事件时段。第一个显而易见的问题是选择正确数量和时间点的工作负载时段进行捕获。但另一个常常被遗忘直到最后一刻的问题是，需要为捕获过程在生产服务器上产生的额外开销以及存储这些元数据所需的额外存储空间做好安排。有些生产服务器本身负载已经非常重，申请给系统增加 3-5%的负载都难以获得批准。关于工作负载捕获和回放，另一个需要提前准备的主要问题是，你必须制作生产数据库的完整副本才能运行捕获的工作负载。这意味着需要额外的磁盘空间来存放该副本，以及进行完整冷备份或离线备份所需的时间。因此，尽管工作负载捕获和回放看似是进行基准测试的最佳方式之一，但现实情况是，在许多情况下它并不可行。这还没考虑工作负载捕获和回放软件的额外成本，对于许多平台来说，这类软件相当昂贵。你可能非常想进行工作负载捕获和回放，但最终不得不退而求其次选择简单的基准测试。如果你的使命宣言及其所有问题的答案能够通过数据库基准测试而非工作负载捕获和回放来满足，那也无妨。

## 扭曲因素

在“准备”这个主题下，这一小节乍看可能有些奇怪。但在准备阶段，确实有许多因素需要提前考虑，它们可能对你的数据库基准测试结果产生重大的扭曲或干扰作用。如果你打算在共享存储（如 SAN 或 NAS）上进行测试，能否至少拥有自己的 LUN（逻辑单元号）？在共享的 SAN 或 NAS 上，能否避开诸如备份之类的常规维护任务时段？数据库服务器和基准测试机能否隔离在单独的网段上？有太多外围问题可能悄然出现并困扰你。让我举几个在基准测试前未考虑到、并彻底导致结果无效的例子。

有一次，在我们进行基准测试的一个周末，网络交换机进行了固件升级。我们对此并不知情，而且所有人都认为这不可能影响到我们的测试。大错特错！不知何故，`VNETs`和`Jumbo Frames`的设置在固件升级过程中被破坏了。我们花了两周时间徒劳无功，才查明并证实是某个外部因素扭曲了我们的结果。这本来不该发生，但它确实发生了。而且这类问题排查起来非常棘手。因此，当你一周的测试结果与另一周截然不同时，请务必寻找任何不常见的原因。因为一般来说，如果操作得当，数据库基准测试结果的变化幅度不应超过 5-10%。

另一次，在测试一个`Oracle 真正应用集群 (RAC)`配置时，我们某一周的结果又莫名其妙地出了问题。我们排查了将近四天才有人发现，我们的一块绑定网卡正在产生低层错误，而这些错误并未上升到操作系统日志中。而且因为是绑定网卡，它并未报错，只是运行速度大大降低，这给集群带来了许多问题，包括节点无故掉线。一度有位数据库管理员（DBA）想把这个问题正式记录为 Oracle 的一个缺陷，认为`RAC`无法可靠地扩展到 20 个节点。不要轻易将异常行为归咎于操作系统或数据库的缺陷。当然，这种情况可能发生。但那将是罕见的例外，而非普遍规律。人们很容易陷入这个思维陷阱。

最后一个例子又是另一个奇怪的硬件问题。是的，我知道我之前说过这类问题应该相当罕见，但如果你进行的基准测试足够多，你就会遇到更多罕见情况。我们有一个属于`RAID 5 LUN`的磁盘驱动器开始出现故障。磁盘故障还未达到危急程度，`RAID`仍在进行补偿。但在几乎每次 I/O 操作上增加的那些额外奇偶校验检查累积起来，足以扭曲测试结果。我们非常幸运，只花了大约四个小时就发现了这个问题。但当你处于一个基准测试项目的最后一周，得到暗示其他结果可能已被污染的数据时，这会让人极度不安。
