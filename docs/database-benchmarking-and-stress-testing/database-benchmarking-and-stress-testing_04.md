# 1. 基准测试基础

本书旨在向读者阐明与数据库`基准测试`、`压力测试`和`工作负载捕获/回放`相关的一切。此刻你可能觉得这些是同一个主题，或者仅仅认为它们之间的区别无关紧要。此外，你可能认为这些主题有些深奥，因此更适合学术界和理论家，但与你工作的现实世界并不真正相关。最后，你可能简单地认为进行此类测试活动所需的时间和精力是不值得的，或者至少很难证明其合理性。结果，许多人怀疑这些活动的真正价值。本书致力于改变这种错误的看法，或者至少让读者了解这些主题的真正价值。

本章提供了关于数据库`基准测试`、`压力测试`和`工作负载捕获/回放`的所有基本定义和我们的通用词汇。此外，它还解释了为什么、何时以及如何运用这些不同的技术是值得的。最后，也是最重要的，本章详细阐述了如何从这些不同的方法中做出最佳选择，因为我们通常都会发现，在生活中知道何时为特定工作使用特定工具是最有价值的。当我们看到涉及钉子的任务时，我们会想到锤子；当涉及螺丝时，我们会想到螺丝刀。一旦你对这些数据库技术也有了同样的、本能的“*直觉*”，你会发现每一种技术都是你工具带上不可或缺的工具。达到这种熟练程度需要一些时间，但非常值得。



## 什么是数据库压力测试？

这看似一个极其简单的问题，但其答案可能比你想象的更为重要。大多数人对被称为**心脏压力测试**的简单程序都有基本了解。这是指医生（通常是心脏病专家）将你连接到心脏监护仪上，让你在跑步机上快走或跑上一段时间。用外行的话来说，其目标是 `让你的心率上升` 并 `让你出汗`。医生通过将患者连接到心电图监护仪上，当然是为了获取更有价值的详细信息。虽然这样的测试通常无法识别具体的病症，但它对于检测值得关注的早期迹象具有很高的预测价值。简而言之，这种简单的医学测试非常宝贵，因此它如此频繁地被进行。

那么，你的数据库呢？它有多健康？虽然你的数据库在当今典型的、已知的工作负载下的性能可能不是问题，但如果你把数据库推到 `让它心率上升` 并 `让它出汗` 的地步，会发生什么？你可能会认为这种数据库压力测试属于简单而审慎的容量规划范畴。这将允许数据库管理员回答关键的管理问题，例如数据库应用程序的资源消耗何时会增长到需要订购更多计算资源的地步？然而，尽管这听起来很有用，但这通常并不是你被要求或认为明智进行任何数据库压力测试的原因（本章稍后会详述）。

那么，你究竟如何对数据库施加压力以 `让它心率上升` 并 `让它出汗`？与心脏病专家有明确界定且几乎普遍接受的既定最佳实践和程序不同，当所有数据库本质上都大不相同时，如何对数据库进行可靠且可重复的压力测试？对于许多数据库管理员来说，答案是他们在高峰使用时段监控数据库，并捕获数据库应用程序中那些看起来开销很大的、预期中显著的一部分语句。然后，这些语句被拼凑成一个单独的脚本或程序，数据库管理员可以执行它，以测量特定更改前后或一段时间内的结果，从而评估潜在的差异。

虽然这看起来相当合理，但这种简单的方法有几个缺点。首先，它具有很大的随机性。如果数据库管理员选择了错误的监控时间段，或者在监控期间遗漏了一些重要且开销大的数据库语句执行怎么办？其次，一旦被捕获，它就成为了数据库应用程序在被捕获那天状况的一个快照。如果应用程序的下一个版本在性质上发生重大变化怎么办？第三，不同的数据库管理员可能不会选择相同的监控时间段或设定哪些数据库语句执行是重要的。事实上，即使是同一个数据库管理员，如果重复进行第二次监控和捕获过程，也可能无法做到。因此，这种方法缺乏实际的可靠性和可预测性。它确实有一些有限的价值，但不能现实地作为先测量然后比较数据库性能的普遍接受的方法。需要更多（的手段）。

## 什么是数据库基准测试？

一般来说，数据库基准测试是一种定义明确、普遍接受、行业标准的数据库压力测试，具有清晰的性能度量指标，用于针对给定的数据库规模和并发用户工作负载测量和比较结果。此外，在相同参数下，这些压力测试原则上被认为是跨多次运行可靠的。这意味着，从统计学上讲，所归因的性能分数在不同测试运行之间不应有显著变化。而且，这种压力测试被认为在不同数据库版本、数据库供应商甚至显著不同的数据库部署场景（例如，在专用服务器 vs. 虚拟服务器、本地部署 vs. 云端等）中都是可行且可靠的。简而言之，数据库基准测试是数据库管理员在逻辑上等同于心脏病专家的跑步机心脏压力测试的东西。

那么，这些数据库基准测试从何而来？更重要的是，在众多来源中，哪些被认为是创造了真诚且持久的行业标准？最后，数据库管理员应将哪些来源以及由此产生的数据库基准测试添加到他的工具库中？这些都是重要的问题，需要简要回顾历史，以便为有意义的欣赏提供适当的背景和上下文。

随着 20 世纪 80 年代初基于结构化查询语言的关系数据库的出现，能够为新的数据库赋予某些性能指标变得重要得多。早期的网络和层次数据库通常性能很高，因为它们基于物理指针的基本性质。然而，基于科德 12 条规则的新型关系数据库则截然不同，它们依赖基于数据值的逻辑引用。这种方法使数据库程序员从专注于 `如何访问` 中解放出来，允许他们专注于 `要访问什么`。数据库系统现在负责存储数据的数据结构和解析查询的检索过程。这种从物理数据访问到逻辑数据访问的根本转变并非没有代价，因此产生了能够可靠地度量该成本的需求，从而产生了对数据库基准测试的新需求。

数据库基准测试的一个明显早期来源是学术界，因为关系数据库如此新颖，并且基于关系代数和关系演算的数学基础。其中一个早期例子是 1983 年^(¹)来自威斯康星大学计算机科学系的 `Wisconsin Benchmark`。其主要目标是提供一种测试关系数据库系统所有主要底层组件性能的方法。

然后在 1985 年，`Debit-Credit Benchmark`^(²)（后来被称为 TP1）首次亮相，由吉姆·格雷编写，学术界、数据库供应商甚至用户都做出了贡献。

随后在 1987 年，吉姆·格雷使用借记-贷记工作负载编写了 `Top Gun Benchmark`^(³)。

从 1985 年（在 *Datamation* 杂志发表一篇数据库基准测试文章之后）到 1988 年，一场数据库基准测试之战随之而来，硬件和软件供应商开始了一场争强斗胜的较量。格雷招募了其他人，并试图通过成立事务处理理事会（ [`www.tpc.org`](http://www.tpc.org) ）来为这种混乱带来秩序，其使命声明^(⁴)写道：

> TPC 是一个非营利性公司，旨在定义事务处理和数据库基准测试，并向行业传播客观、可验证的 TPC 性能数据。

数据库基准测试的另一个来源是大型硬件供应商（例如，吉姆·格雷在创作他的一些作品时就在 Tandem Computers 公司）。其中一些基准测试在数据库方面是相当中立的。一个很好的例子是，戴尔为 SQL Server、Oracle、MySQL 和 PostgreSQL 提供了非常流行的开源 `DVD Store Benchmark`。该基准测试的目的写道：



>*DVD Store Version 2 (DS2)* 是一个完整的**在线电子商务测试**应用程序，包含后端数据库组件、Web 应用层和驱动程序。设计数据库组件和中间层应用的目标在于充分利用许多高级数据库特性（事务、存储过程、触发器、参照完整性），同时保持数据库易于安装和理解。DS2 工作负载可用于测试数据库，也可作为任何目的的压力测试工具。

然而，虽然某些硬件供应商的数据库基准测试可以在不同数据库供应商之间进行一定程度的公平和可靠性的比较（如戴尔 DVD Store 基准测试），但那些侧重于特定组件（如 I/O 子系统）的其他硬件供应商数据库基准测试通常则不能。一个很好的例子是非常流行的 "`Silly Little Oracle Benchmark`" (SLOB)。该基准测试旨在突出基于闪盘系统的原始数据库写入性能。因此，它的设计更侧重于测试写入活动而非读取。本质上，它只发出两个命令：大规模更新和计数选择。因此，它实际上并不测试查询优化器或数据库引擎的任何其他高级方面——只测试它写入数据的速度有多快。这并不意味着它是一个糟糕的测试或需要避免使用它。但你需要了解它实际测试的是什么以及它真正报告的是什么。事实上，作者^(⁵) 在他的网页上明确指出：

>*`SLOB` 不是数据库基准测试。`SLOB` 是一个 Oracle I/O 工作负载生成工具包。我需要指出，由于习惯使然，许多 `SLOB` 用户将 `SLOB` 与基准测试和负载等术语互换使用。`SLOB` 旨在填补 `Orion` 和 `CALIBRATE_IO`（正如这里部分解释的，两者都不产生合法的数据库 I/O）与全功能事务基准测试（如 `Swingbench`）之间的空白。事务负载旨在测试数据库的事务处理能力。*

数据库基准测试的另一个来源是一些开源工作，特别是一些较大的开源数据库项目。例如，PostgreSQL 提供 "`pgbench`" 作为比较性能的标准工具。虽然它使运行测试变得相当容易，但它仅适用于 PostgreSQL，并且更重要的是，文档明确指出：

>*要进行基准测试，必须执行与生产负载和查询类型相似的测试。例如，`pgbench` 查询不执行连接、在文本字段上使用正则表达式的 where 子句，以及许多应用程序中会出现的其他查询类型。*

此外，MySQL 提供了 "`MySQL Benchmark Tool`"，它仅针对 MySQL 数据库运行三个数据库基准测试：
1.  `DBT2`
2.  `SysBench`
3.  `flexAsynch`

其中文档对这些数据库基准测试定义如下：

>*`DBT2` 是一个模拟拥有大量仓库的公司 OLTP 应用的开源基准测试。它包含处理新订单、订单录入、订单状态、支付和库存处理的事务。`FlexAsynch` 是专门开发用于测试 MySQL Cluster 可扩展性的基准测试。`Sysbench` 是一个流行的开源基准测试，用于测试开源 DBMS。*

到目前为止，数据库基准测试拥有一个漫长、多样且非常有趣的历史，这一点应该相当清楚了。然而，同样显而易见的是，只有 `TPC` 基准测试旨在客观地测试现实世界的数据库性能并提供公平的结果。我们将在第 2 章更全面地介绍所有相关的 `TPC` 数据库基准测试。

## 什么是工作负载捕获/回放？

对许多人来说，针对特定目的（如"`在线事务处理`" (`OLTP`) 或 "`数据仓库`" (`DW`)）的行业标准数据库基准测试这一概念已经足够。但对有些人来说，这些预先封装的"`合成`"测试还不够。他们希望根据其数据库应用程序的实际性质，获得更现实和有意义的东西。他们本质上渴望的是进行数据库压力测试的第一种方法（即监控和捕获命令），但希望是针对 100%的活动，而不仅仅是手动、有些随机的样本。他们想要全部。他们希望在一段定义的时间内获得其所有数据库活动的完整快照。此外，他们希望它以可按需回放的方式打包。简而言之，他们想要的就是所谓的**工作负载捕获和回放**。这基本上就是它的名字所表达的意思，对于许多数据库管理员来说，这是他们认为唯一有意义的基准测试。

很难反驳这种自然而合乎逻辑的观点。这完全符合常识。但正如谚语所说，"`通往地狱的道路由善意铺就`"，意思是说，良好的意愿在付诸行动时可能会产生意想不到的后果。为了使工作负载捕获/回放真正具有代表性，你必须使用大小完全相同且数据相同（或至少数据分布相同）的数据库，这样结果才准确。这意味着需要复制你整个 100TB 的生产数据库，保持数据原样进行测试，因此不能进行数据子集化或数据屏蔽。这转化为非常大的硬件和软件成本，此外，由于真实的生产数据现在在你的测试环境中完全可用，这构成了巨大的安全风险。你的管理层可能会对预算影响有点小题大做，但你的安全与合规人员会因为你这样做而让你丢掉工作。

工作负载捕获和回放的另一个主要问题是，并非每个数据库供应商都提供此功能，而对于那些提供的供应商来说，该功能可能成本高昂（Microsoft SQL Server 是一个显著的例外）。当然，数据库供应商的工作负载捕获/回放工具只适用于其自身的数据库。对于许多公司来说，这意味着需要采用多种工具，因为如今我们都部署不止一个数据库供应商的产品。也有一些第三方软件供应商，但数据库供应商通常不会共享其系统内部细节，因此有些人质疑在没有这种访问权限和洞察力的情况下，此类解决方案的有效性和效率如何。尽管如此，对于那些希望其数据库基准测试能反映其数据库应用程序实际性能的人来说，工作负载捕获/回放仍然是他们追求的圣杯。

## 为什么这些技术有用

如前所述，数据库基准测试有其真正的怀疑论者。然而，我们中没有人会像环保署 (`EPA`) 那样驾驶车辆来验证燃油经济性，但我们都会寻找购买任何新车时贴在车窗上的标签，当油价昂贵时，我们常常将那个数字视为重要信息。因此，我们清楚地看到了该基准测试的预测价值，并接受它所提供的洞察力。虽然我们不期望在自己的驾驶中达到相同的燃油经济性评级，但我们确实真诚地期望它在比较车辆时相对准确。作为人类，拥有这样一个对其可靠性有一定信念的比较是非常令人安心的，并使决策过程压力小一些。

在日常生活中，人们依赖的类似基准测试有几十种，然而数据库基准测试却常常受到极大的怀疑。所以，让我们来审视一些你可能需要回答的问题，对于这些问题，你可能需要基准测试可能提供的一些见解（当然，前提是选择了合适的基准测试）。关键在于，如果你在购买新车时会欣然依赖 `EPA` 的油耗基准测试，那么你一定相信某些基准测试的价值，那为什么对你的数据库不能同样如此呢？


## 这些技术何时具有价值

想象一下，作为数据库管理员，你的老板问你：“如果我们进行以下操作，预计会对生产数据库性能产生什么影响？”：

*   对操作系统和/或数据库应用主要版本升级？
*   从基于 RISC CPU 的专有 UNIX 系统迁移到基于 Intel CISC CPU 的 Linux 系统？
*   在戴尔、惠普和联想同等价位的服务器中选择？
*   从 Windows Server 迁移到 Linux，或反之亦然？
*   从物理服务器迁移到虚拟机监控程序？
*   从 VMware 等商业虚拟机监控程序迁移到 KVM 等开源方案？
*   按实例整合数据库？
*   从本地部署迁移到云端？
*   升级到下一个云定价层级？

这些只是其中少数几个问题。那么，你该如何回答这类问题呢？即使拥有超过 30 年的数据库经验，我也只能对其中少数几个问题提供合理的见解，而且可能无法结合当时所需的具体上下文，向管理层提供可靠的建议。因此，在这种情况下，经验丰富的数据库管理员通常会进行一些测试，通常被称为 `概念验证`（POC）项目。那时，你必须决定是进行简单的压力测试、数据库基准测试，还是工作负载捕获/重放。

一个常见的合理反驳是：“我们不经常做出此类决策，所以在遇到这类问题之前，这并不太重要或不值得担心。”我非常理解，数据库管理员总是被要求或被告知要用更少的资源做更多的事。他们根本没有足够的时间或精力去精通每一项技术。但过去十年数据库领域发生的变化表明，数据库的部署方式已经演变：先是虚拟化，现在正迁移到云端。因此，这类问题比过去发生的频率高得多。事实上，如今的数据库管理员或数据架构师很少不面对这类问题。

## 如何在这些选项中做选择

既然我们已经探讨了这些不同技术为何重要以及何时重要，剩下的问题就是：应该选择哪一种？就像生活中任何重大决定一样，一个合理而非轻率的回答是 `这要视情况而定。` 然而，我主张在许多（即使不是大多数）情况下，你应该给数据库基准测试一个实际的考虑。可以说，它比通过临时监控数据库来创建压力测试脚本、捕获你认为并希望是真正代表性的工作负载，要更快、更可靠。它也不需要使用可能涉及机密、甚至可能非法复制到测试环境中的真实数据。最后，它不会增加工作负载捕获/重放工具的额外成本，而且该工具也依赖于使用真实数据。

是的，你可能会争辩说数据库基准测试依赖于合成的工作负载。然而，存在许多数据库基准测试，可以近似模拟各种工作负载类型，包括在线事务处理（OLTP）、数据仓库（DW）查询、批处理系统，以及用户自定义的、加权组合的所有上述类型。因此，你确实应该能够快速、轻松地组合一系列基准测试，以有意义且可接受地近似你的系统。这将使你能够专注于快速、准确地比较 `同类` 场景。

对于资深的数据库管理员，你可能已经对你会做什么或者在提议的选项中有所偏好看法。我只是建议你以公正、开放的心态阅读本书，认识到数据库基准测试很可能会补充你现有的技能库。记住中国哲学家孔子的话：“开卷有益。”

对于更初级的数据库管理员或那些只是不习惯应对这些棘手的 `假设性` 数据库性能问题的人，你可能希望这本书能提供所有答案。然而，没有任何一本书能够拥有你对你的数据库及其数据的全部经验和洞察力。因此，我建议你阅读本书时，寻找你的目标与你所处的数据库上下文之间的最佳契合点。正如 `科学小子` 比尔·奈所说：“你遇到的每个人都知道一些你不知道的东西，因此你应该倾听他人并‘尊重他们的知识’。”
