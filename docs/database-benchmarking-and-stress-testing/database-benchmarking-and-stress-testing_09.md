# 4. 基准测试准备

此时，你可能认为自己已经完全准备好开始数据库基准测试工作了，凭借的仅是迄今为止提供的信息：数据库基础知识、各种行业标准数据库基准测试的概述，以及可用的 IO 子系统和数据库基准测试工具的阐明。因此，你们中的一些人可能觉得本章只是填充内容或花哨的装饰。但我要强调，本章实际上是整本书中最关键的一章，应该以最多的注意力来阅读。事实上，我将进一步说明，未能按本章所述进行准备，可以说是导致下一章出现失误和失败的最大因素。

当我年轻的时候，我曾是美国童子军的一员。虽然我可能没有成为鹰级童子军或获得很多徽章，但我确实学到了很多。那时深深烙印在我脑海里的是童子军的座右铭：时刻准备着！这两个简单的词蕴含着巨大的智慧。将这一理念延伸，本杰明·富兰克林说过：“不为成功做准备，就是在准备失败。”这些话在生活中的许多方面都千真万确，尤其是在数据库基准测试方面。想象一位美式橄榄球（不是足球）教练不准备比赛计划，当他输掉比赛时，你真的会感到惊讶吗？事实上，你可能会和其他球迷一起要求换掉教练，如果现任教练毫无计划或计划不佳就参加比赛。对教练有益的东西，对数据库基准测试专业人员同样有益。如果准备不足，那么或许就是换人的时候了。


## 基准测试准备

如同任何希望获得最大成功机会的商业项目一样，首先要清晰界定和阐明的，是任何数据库基准测试工作所要达成的**使命声明**。这一步骤常常被跳过或仅仅被边缘化。据我所知，没有哪家公司的使命声明仅仅是“赚钱”。同样，大多数有能力的橄榄球教练可能也不会说，比赛计划仅仅是为了赢而赢（尽管有一个值得注意、可以理解且完全被允许的例外——俄亥俄州立大学的教练，他们必须击败密歇根队，只因为对手是密歇根队）。

那么，在商业中，什么构成一个好的使命声明？关于这个问题，你会发现很多观点和说法。就我们的目的而言，最适用和最重要的四点包括：

*   能够建立预期
*   目标切实可行
*   具备审慎的适应性
*   简短但目的明确

让我们按照这四个标准来剖析下面这个使命声明示例：

> *验证将我们最大的数据仓库和决策支持数据库从本地服务器和存储区域网络（SAN）迁移到云端后，在合理甚至更优的总体成本下，性能表现至少一样好。*

哇——这看起来像一个值得尊敬的使命声明。事实上，它可能与你管理层已经在考虑或即将要求你处理的那些想法非常相似。实际上，就在过去的 12 到 18 个月里，我参与的所有数据库基准测试项目都或多或少遵循这个思路。这似乎是当下一个非常常见的主题。

在我们开始剖析这个使命声明之前，我们首先需要认同，它包含技术和非技术两方面，可能需要分工负责。与我共事的大多数数据库管理员实际上只负责技术层面——即，在什么样的云配置（规模）下，其性能表现达到或超过当前系统。这并不是说另一部分不重要，因为它涉及成本。但大多数 DBA 最适合的还是技术层面。因此，现在让我们只剖析这个使命声明，去掉与成本相关的那七个字。

预期似乎相当直接明了；云端的性能能否与当前系统一样好？但在“最大的数据仓库和决策支持数据库”这个短语中，隐含着几个重要问题。“大”究竟有多大？数据仓库（DW）和决策支持（DSS）工作负载是否主要是复杂查询，外加一些在非高峰时段的批量加载？终端用户的查询主要是来自预定义的报表，还是用户自定义的即席报表？我们仅仅是把当前的 Oracle 或 MS SQL Server 这样的关系型数据库迁移到云端，还是迁移至 Hadoop 或 Cassandra 这样的 NoSQL 数据库？如果存在 BI（商业智能）服务器，它是保留在本地，还是也迁移到云端？现在，基于所有这些问题，你可能会认为我要说这个使命声明示例在建立预期方面失败了。但如果它以一种能引发这些问题并获取有用答案的方式撰写，那么我会说，使命已经达成（至少对于第一个要点而言）。

接下来看目标。同样，使命声明示例中有一部分引发了许多问题：“迁移到云端”具体指哪里？“性能表现至少一样好”又是什么意思？我们是迁移到 Amazon AWS、Microsoft Azure、Google Cloud 还是 Oracle Cloud？我们是否知道允许使用何种类型和规模的虚拟机镜像？我们可以自由使用 NVMe 和 SSD，还是仅限于标准存储磁盘？我们打算如何测试性能？我们会复制数据库，在云端并行运行应用程序吗？或者我们会选择某个行业标准的数据库和/或应用程序基准测试并在云端运行？当前的服务级别协议（SLA）是什么？新的 SLA 必须满足哪些差异（如果有的话）？例如，大多数云供应商提供 99.9 或 99.99 或 99.999 的正常运行时间（例如，99.9999% 对比 99.99999%）。这种可靠性水平可能比当前的本地 SLA 要好，那么我们是否必须全面达到这个提升后的 SLA？所有这些问题实际上都归结为一个核心问题：我们如何判断通过还是失败？你可能已经注意到，这个使命声明再次提出了比答案更多的问题。这具有巨大价值。本章内容完全围绕准备展开。如果使命声明能在我们开始基准测试之前，就促使这些问题得到解答，那么我们将减少时间浪费，增加成功的机会。

这个使命声明的适应性又如何呢？获取迄今为止所有问题的答案，自然会引出关于预留空间以修订或调整方法的问题。例如，如果被告知必须使用 Amazon AWS，而最初的想法是使用`通用型`镜像类型，那么是否允许我们建议使用`内存优化型`或`存储优化型`，因为它们可能更符合数据库服务器的需求？在涉及时间估算时，适应性问题变得至关重要。如果被告知不行，镜像类型必须是通用型，那么你可以合理地预期时间表是相对固定的。另一方面，如果被告知从通用型开始探索最佳镜像类型，那么你的时间表就会延长——甚至可能延长两倍或三倍。作为适应性的一部分，还有其他隐藏的方面值得询问吗？例如，也许我们知道当前使用本地数据库的应用程序实际上并未利用任何`企业版`特性。那么，我们是否应该进行测试，以验证一个更低、更便宜的数据库版本是否足以满足需求，从而降低许可成本？所以，是的，还有更多需要答案的问题。但这正是“做好准备”的准确定义。换句话说，“*三思而后行*”，以免遭遇伊索寓言中山羊的类似命运：狐狸告诉它干旱将至，它跳进了井里，而狐狸的真实目的只是想找个垫脚石逃出井。将允许的适应性确定下来，对你大有裨益，因为它常常直接代表着范围蔓延。


## 项目使命陈述

最后一个要点，即使命陈述是否简短且目标明确，似乎是最容易理解的部分。什么是“大局”概念以及这样做的理由是什么？在这个例子中，这很明确：在不产生任何负面影响的情况下迁移到云端。然而，我曾参与过一个大型数据库基准测试项目，该公司想知道哪家硬件供应商能最好地运行他们首选的数据库。因此，实际目的其实是看未来 5-10 年内谁能获得约一万台服务器的订单。这个事实让我们肩负了更大的责任，以确保我们是在进行公平的比较，没有“调包计”之类的花招。这个真实的潜在动机显然改变了我们对该项目的看法和方法。虽然这项工作的成本可能只有 2 万美元，但它却是一个关系到任务成败、具有战略意义的问题，上升到了最高管理层。掌握了这一事实后，我们能够提供一个可以证明和论证的明确结论。所以，有时扮演牙医、深挖问题是值得的，因为有时那些看似简单无害的事情，只要稍微深究一下，就会发现其实是一个重大问题。

至此应该清楚了，使命陈述应该是你提出大量问题的脚本，目的是为了构建和验证一个检查清单，该清单将成为基准测试项目中资源分配、调度和预算的关键因素。这个检查清单也将确保最大的成功机会。然而，这个关键步骤几乎总是被跳过，导致交付成果缺失和时间超支。因此，我无法再强调这一步有多么关键。它可能是取得成功的唯一最大因素。

## 数据库准备

在与进行数据库基准测试项目的公司合作时，最令我震惊的领域之一往往是数据库准备的缺乏。事实上，这种情况发生的频率超乎我的想象。此外，它还能以多种方式表现出来。

在足够多的情况下（值得列出），我看到基准测试项目由业务分析师或应用项目负责人执行。我无法解释第一种情况，但它确实会发生。第二种情况更容易理解；应用程序是最终用户看到和感受到的东西，因此有时会要求应用程序团队测试将数据库应用程序从一种部署模型迁移到另一种的效果。无论哪种方式，最终效果都是要求非数据库管理员来完成一项工作，而其中一些最关键的要求恰恰属于不同资源的技能范畴。我也遇到过应用开发团队中有 DBA 的情况，但他们是一位应用 DBA，其工作是设计应用程序所需的表和索引。他们通常不是详细性能分析和优化技术方面的专家。无论由谁负责执行数据库基准测试项目，他们都应该包括一名在目标数据库和平台上拥有丰富技能的资深 DBA。我甚至更进一步地建议，未能包含一名胜任的 DBA 几乎普遍会保证项目失败。

另一个令人惊讶的怪事是，在我参与的数据库基准测试项目中，大约有 50%只是使用预先配置好的默认值安装数据库。在某些情况下，如前所述，团队可能缺乏具备部署技能的 DBA，但在其他情况下，这似乎仅仅是懒惰或对细节缺乏关注。并非所有数据库都能自动适应分配给服务器或虚拟机的 CPU 和/或 RAM（内存）的增加。同样，某些工作负载类型通常开箱就需要不同的配置。此外，基准测试的数据库规模或大小以及并发用户数也需要不同的设置。除非你只是进行简单的教育性或相对较小的基准测试工作，否则你需要正确配置数据库。关键点是，对于基准测试工作，你应该进行与生产系统完全相同的优化。

数据库准备中另一个经常被忽视的主要领域是存储分配。大多数数据库基准测试工具不理解你的存储逻辑单元或其 RAID 特性（例如条带宽度和深度）。大多数基准测试工具只是在用户运行创建和加载步骤的默认存储区域中创建所需的基准测试对象（表和索引）。其中一些工具确实提供了创建分区对象的选项，但同样，对你存储设置的“智能”考虑很少。我告诉人们的是，将任何数据库基准测试作为两个独立的主要步骤运行：创建/加载与执行。其思路是将第一步隔离运行，以便你可以修改或重新创建数据库对象，以最好地利用我们的存储布局。然后对那个布局合理的数据进行备份，以便对于同一执行的后续迭代，你只需进行恢复即可。我甚至见过一个足够智能的数据库基准测试工具，它提供了对此类方法的支持（Quest’s Benchmark Factory）。请相信我，当你只需要基准测试工具在第一次加载数据时，一个 3TB 的 TPC-H 会运行得更好、更快。我见过一个 3TB 的 TPC-H 初始加载花了三天时间，但恢复备份只用了三个小时。当你处于一个两到三周的日程安排中时，这一差异可能决定你工作的成败。

数据库准备中最后一个被忽视的领域是正确的统计信息和直方图收集，以及运行测试迭代比较报告前后的执行性能指标快照。这些工具都无法替代你的 DBA 的经验和见解。一些工具允许在前后运行脚本，这样你可以手动编码这些项并将其手动添加到项目中。很少有工具将其内置到它们的工作流或图形用户界面中。因此，你的 DBA 需要学习你的数据库基准测试工具，了解它能做什么、不能做什么，然后在适当的地方加入他们专业的操作。我曾见过忽视这个问题导致诊断或优化性能的能力很差。如果做得好，我见过它能带来高达 400%的差异。这非常值得关注。


## 基准测试时间规划

一个数据库基准测试项目应该花多长时间？这应该基于项目的目标声明以及它引发的数十个问题的答案（这些问题最好能得到解答）。然后，你应该制定一个项目计划，包含所有开始前必需的先决条件。然而，我参与的许多项目里，获批的计划却是我们只有两周时间来运行一个 `TPC-C` 或 `TPC-H` 测试，并显示出可接受的 `TPS`。有一个案例，他们要运行 `TPC-H`，但设备尚未部署，软件尚未安装，而且资源紧张，难以完成这项工作。然后他们告诉我，我们必须运行几次 3 TB 或更大规模的 `TPC-H` 测试。想象一下，当我说不如今天就报告说这次努力失败了，省时省力时，他们有多惊讶。当然，他们以为我在开玩笑，于是我们开始了项目。硬件和软件准备就绪花了四天。初始数据加载花了三天。所以到第二周的周一，我们只剩五天时间，却要从零开始。这并非个例；几乎每一个数据库基准测试项目似乎都被严重错误估算，以至于几乎扼杀了任何成功的合理机会。

在理想情况下，如果所有硬件和软件都已设置好，预先选定了合适的服务器或虚拟机规格，操作系统和数据库都经过优化，拥有用于执行和调整测试运行（以便进行迭代测试）的适当资源，并且目标明确，那么两周时间可能（我只说可能）是足够的。我告诉那些询问的人，初步预估应以三到四周为起点，此时大多数人会说他们没有那么多时间。当我质疑这个限制时，通常被告知我们只需开始，看看两周内能进展到哪里——然后就停止。我会问那些没有当场解雇我的人：他们是否愿意让一个外科医生开始手术，然后不管进行到哪一步，两小时后就直接停止？答案当然总是否定的，但与医生不同，这些专业人士大多不愿回去对管理层说“需要更多时间，否则就算了”。所以在这些情况下，我们尽力在两周内完成，但多半时候我们未能达成预期结果。

时间规划的另一个关键方面是：当分配的资源不正确或不充足时会发生什么？还记得我上面提到的那个 3 TB `TPC-H` 项目吗？嗯，为存储订购的硬件是一个直连存储柜，里面有 15 块机械磁盘。我解释说，`TPC-H` 基准测试的性质决定了其性能与机械磁盘数量直接相关，当时发布的所有同等规模的基准测试都是在不少于 100 块磁盘上完成的。当然，我们不可能再弄到 85 块磁盘，因为他们只负担得起增加一个存储柜，总共 30 块磁盘。这又花了一周时间才到位并准备好。所以，看到了吗，这些项目为什么几乎不可能在两周内完成？不要让自己在没有救生圈的情况下被扔进泳池的深水区。如果有人要求你在过于有限的时间内完成一个数据库基准测试项目，那就直接说“不”。

好吧——我知道向管理层说“不”几乎不可能，而且对职业生涯来说很鲁莽（尽管我已经做过很多次了，也许这就是我从未“发达”的原因）。但请努力争取更多时间，或降低期望值，或缩小范围。如果那个 3 TB `TPC-H` 项目愿意考虑将数据库规模缩减到 300 GB（这是该数据库基准测试的最低有意义规模），并添置第二个磁盘柜，我们本可以成功的。我们本可以在第一周搭建好并运行测试，使一切就绪；然后在第二周添加第二个存储柜并重新运行测试，以获得最佳最终数据。所以，要寻找创造性的解决方案。

## 基准测试规模设定

当人们问我规模设定的问题时，比如我的 `TPC-C` 数据库应该多大、应该运行多少并发用户？我总是感到惊讶。作为一个来帮忙的外人，我其实不了解他们现有的系统或最终用户池的规模。你作为内部人员，会比任何外部顾问都更有优势，但他们可以提供协助。所以，这个决策应该由你主导。

不久前，我帮助一家大型全国性保险公司进行他们的基准测试工作。他们正考虑从大型机转向客户端-服务器和关系型数据库。他们对于运行一个基于 `OLTP` 的基准测试作为衡量标准表示 `OK`。起初，他们只为基准测试运行 100 个并发用户会话。原因是当时使用的一个免费工具有 100 用户的限制。对于一家如此庞大的公司来说，这显然是不可接受的。管理层迅速购买了一个商业许可证，但数据库管理员（`DBA`）最初只选择运行 2000 个用户。当我询问（并随后真正追问）原因时，答案是运行基准测试工具在达到这个数量级前很容易，但要扩展到更高规模则需要更多的工作。于是我问：既然我知道这个系统必须处理你们整个全国代理网络的负载，那么你们目前看到的最大并发会话数是多少？答案是 15,000 到 20,000。于是，我们不得不购买额外的数据库基准测试工具许可证，以便将其部署为一个控制台，配以 10 个运行在其他服务器上的代理，每个代理运行 2000 个并发会话，总计 20,000 个并发会话。现在知道 `TPC-C` 规范规定 `每 10 个用户对应 1 个仓库`，我们知道规模因子必须设为 2000，这转化为 150 GB 的数据库规模。

在 `OLTP` 案例中，这一切听起来都很好，但如果你需要运行的是数据仓库（`TPC-H`）或决策支持（`TPC-DS`）类型的基准测试呢？首先也是最重要的，尽量提前弄清楚高级管理层到底想把多少数据保留在热的、可快速访问的存储上。我听过业务人员说我们要永久保留数据。但热数据通常真的只是一个可管理的子集。例如，也许这家零售杂货连锁店想对客户在一年内购买的商品进行购物篮或趋势分析，同时考虑天气条件、假日和体育赛事。以如今相当低廉的存储成本来看，答案可能高达五到七年的详细交易数据，这轻易就能达到数十甚至数百 TB。另外，你可能还需要保留聚合数据或摘要数据，以及索引和任何其他占用空间的结构。因此，为 3 到 30 TB 左右的这些需求进行基准测试可能是有意义的。你需要确定的另一个因素是，在这个环境中，高管们愿意为一个有意义的战略报告等待多久。记住，这些数据仓库和决策支持类型的查询往往需要处理数千万甚至数亿行数据才能形成答案。高级管理人员可能愿意等待四个小时来获得一个能直接并积极影响利润的结果。所以，你需要足够的数据来近似模拟这类环境。

你需要牢记的最重要的一点是：磁盘空间很便宜，因此企业希望数据集越来越大。将你的基准测试数据库规模定得太小对任何人都没有好处。只需记住，数据库基准测试工具在数据加载过程中非常缓慢，因此你会希望采用前面提到的拆分项目和备份方法以求快捷。
