# 3. 基准测试工具

在第 1 章中，我们区分了数据库基准测试、数据库压力测试以及工作负载捕获/重放。接着在第 2 章，我们涵盖了所有相关的数据库基准测试，包括当前和过去的（即已淘汰的）。这很像一个青少年学习驾驶汽车，你现在已经学习了交通规则并通过了笔试，获得了学习驾驶许可证。现在是时候运用这些知识，进行一些实际驾驶了（远在参加最终路考之前）。为此，这位既紧张又兴奋的青少年需要一辆车和一位坐在副驾驶的合法驾驶员（注意还有其他限制，例如不能载其他青少年，不能在夜间驾驶）。类似地，对于数据库基准测试，你需要一些工具，在获得足够经验之前，你需要在限制条件下进行一些测试驾驶，然后才能真刀真枪地干。因此，在第 3 章，我们将介绍一些可用的数据库基准测试工具以及如何最好地使用它们。此外，我们还将介绍一些你可能认为是基准测试工具，但更符合数据库压力测试或工作负载捕获/重放工具类别的其他工具。

汽车驾驶的例子也适用于描述各种数据库基准测试工具的实际使用复杂程度。有些车是自动挡，而另一些是手动挡。大多数人会同意自动挡更容易驾驶，但由一个经验丰富的司机驾驶手动挡应该能获得更好的油耗。同样，我们基本上都偏爱有动力转向和刹车的汽车，因为这些功能确实让驾驶变得轻松得多。当前的数据库基准测试工具也是如此；有些自动化程度高，因此易于使用，而另一些则更困难，但可能提供更大的灵活性。因此，需要根据要使用的工具类型进行适当的准备。此外，就像汽车一样，你应该能够使用任何一种类型到达你的目的地。当然，你的油耗和经验可能会有所不同。

最后，请注意，虽然本章旨在为你提供所有必备的基础知识以及一些有用的技巧和窍门，但合乎逻辑的是，要成为一名能够合理测试高级场景以获得可靠且**可重复**结果的专家，需要时间和毅力。请注意，`可重复`是最难也是最重要的目标。一辆车能跑一次 125 英里/小时，并不意味着它能轻易重复这一壮举。在此过程中有许多变量在起作用。结果必须与所有变量相关联，并且条件必须能在某个合理的接近范围内重复。否则，就只是一个非常幸运的`一次性`结果，几乎没有价值。

## 存储基准测试

在研究数据库基准测试工具之前，我们必须首先认识到那些即使非常流行但也不被归类为真正数据库基准测试工具的工具。有时，数据库管理员（通常与存储管理员合作）会专注于每秒的 IO 速率或 IOPS。他们对磁盘驱动器读写数据的速度比数据库引擎本身更感兴趣。因此，他们不关心实际的数据库事务，而是关心原始的 IO 速率。例如，仅通过了解发动机的 RPM（每分钟转速）你无法真正知道一辆车能跑多快。同样，仅通过了解磁盘的 IOPS，你也无法了解实际的数据库性能特征。是的，磁盘的 IOPS 越快，数据库可能也越快，但仅凭 IOPS 你确实只能**猜测**数据库性能。虽然这是一个影响因素，但它不是唯一的因素，甚至在很多情况下不是最重要的因素。就像汽车一样，我们需要知道除 RPM 之外的其他因素，例如变速箱齿轮比和轮胎尺寸。尽管如此，即使这些工具只能提供真实数据库性能的最基本概念，它们仍然非常受欢迎。

## 数据库模拟

数据库模拟的概念相当简单；数据库供应商知道其数据库引擎通常会产生什么样的 IO 模式，因此他们可以提供工具来模拟这些 IO 模式进行测试，而无需实际创建和基准测试数据库。虽然这类产品不是真正的基准测试工具，但对于一些想要在 IO 层面（通常是着眼于 IOPS）检查更低层性能的数据库管理员来说，它们仍然提供了有价值的见解。

### SQLIOSTRESS

如果你的数据库是 Microsoft SQL Server，那么一个已过时但仍被铭记的数据库模拟工具是 `SQLIOSTRESS` 实用程序。它已不再在 Microsoft 网站上提供，我也无法在任何免费软件网站上找到它。我在这里提及它只是为了完整性，因为它已被 `SQLIOSIM` 取代。



### SQLIOSIM

如果您的数据库是 Microsoft SQL Server，那么当前的数据库模拟工具是 SQLIOSIM 实用程序。您可以通过网络搜索获取 Microsoft 网站 URL 以下载当前版本。还需注意，SQLIOSIM 甚至不需要安装 SQL Server，因为它完全模拟了 SQL Server 会向 I/O 子系统请求的 I/O 命令。SQLIOSIM 是一个非常强大的工具，但其选项过多，无法在此详述。对于最简单的部署，只需将下载的文件解压缩到一个目录中。同时，也解压缩包含在该主 zip 文件中的配置文件。然后将默认配置文件 `sqliosim.default.cfg.ini` 复制为 `sqliosim.cfg.ini`。最后，编辑该复制的文件，在接近底部的位置取消注释两个 FILE 部分，以指定要模拟的设备上的 SQL Server 数据和日志文件。现在，您已准备好开始。您可以通过运行 `sqliosim.com` 在命令行中运行此实用程序，并通过运行 `sqliosim.exe` 在图形模式下运行，如图 3-5 所示。此实用程序运行需要相当长的时间（除非您减少配置文件中的 `TestCycleDuration` 值），日志文件包含有关结果的详细 HTML 报告。

### Orion

如果您的数据库是 Oracle，那么一个众所周知的数据库模拟工具是 ORION 实用程序（ORION 代表 ORacle IO Numbers）。它位于数据库和/或网格的 `$ORACLE_HOME/bin` 目录中。虽然它位于 Oracle 安装目录中，但它实际上并不需要 Oracle 数据库的存在。Orion 是一个非常强大且功能齐全的工具，因此建议在进行任何详细测试之前阅读 Orion 手册，但对于本书，我们可以看一个简单的例子。在我的 Windows 机器上，我的数据库数据文件位于 D: 盘上，这是一个三星 SATA-III SSD；因此命令将是 `$ORACLE_HOME\orion -run normal -testname d_drive`，其中测试名参数指定了带有 `.lun` 后缀的文本文件，该文件只是要测试的 LUN 或磁盘驱动器的列表。在 Linux 上，文件内容形式为 `/dev/sdb1`，而在 Windows 上则为 `\\.\D:`。图 3-6 显示了输出的示例。注意达到的两个关键值：514 MBS 和 10,472 IOPS。

### Calibrate IO

如果您的数据库是 Oracle，那么一个非常流行的数据库模拟工具是 `CALIBRATE_IO` 实用程序。根据 Oracle 数据库性能调整指南：“此过程向数据库文件发出 I/O 密集型只读工作负载（由 1 MB 的随机 I/O 组成），以确定存储子系统可以维持的最大 IOPS 和 MBPS。”调用此实用程序的代码非常简单，如下所示。

*调用 Oracle 的 Calibrate IO 实用程序*

```sql
SET SERVEROUTPUT ON
DECLARE
lat  INTEGER;
iops INTEGER;
mbps INTEGER;
BEGIN
-- DBMS_RESOURCE_MANAGER.CALIBRATE_IO (physical disks,
max latency, iops, mbps, lat);
DBMS_RESOURCE_MANAGER.CALIBRATE_IO (1, 10,
iops, mbps, lat);
DBMS_OUTPUT.PUT_LINE ('max_iops = ' || iops);
DBMS_OUTPUT.PUT_LINE ('latency  = ' || lat);
dbms_output.put_line('max_mbps = ' || mbps);
end;
/
```

在我的 Windows 机器上，我的数据库数据文件位于 D: 盘上，这是一个三星 SATA-III SSD；因此我调用 `CALIBRATE_IO` 实用程序时，第一个参数设置为一个磁盘，第二个参数设置为最大延迟 10 毫秒。显然，您的数据库服务器可能拥有更多的磁盘驱动器，并且随着 NVMe 闪存盘等新技术的出现，您的延迟也可能设置得更低。因此，您必须根据您的环境进行实验。我在笔记本电脑上运行了 `CALIBRATE_IO`，得到了如图 3-7 所示的结果。注意报告的数值：528 MBS 和 54,767 IOPS。

![../images/464117_1_En_3_Chapter/464117_1_En_3_Fig7_HTML.jpg](img/464117_1_En_3_Fig7_HTML.jpg)

图 3-7

运行 Oracle CALIBRATE_IO

现在，我不想阻止任何人使用 `CALIBRATE_IO`（如果它适合您的特定情况）。但您会发现许多作者和博主都会列出此实用程序方法的诸多缺点。以下是我从网上找到的关于此实用程序的一些转述示例（我完全同意）：

*   `CALIBRATE_IO` 也许更好地被描述为集成在数据库内部的 Oracle ORION 工具的简化版本
*   `CALIBRATE_IO` 的最大 IOPS、最大 MBPS 和观察到的延迟值的结果并不总是准确的
*   `CALIBRATE_IO` 的结果既不可靠也不可重复
*   `CALIBRATE_IO` 执行使用异步 I/O 调用缓冲在进程堆中的单块随机读取，这与绝大多数被称为 `db file sequential read` 的单块随机 I/O（在 SGA 中缓冲且是同步 I/O）毫无共同之处
*   `CALIBRATE_IO` 和 Orion 都不访问从存储中读取的数据，它们真正做的只是执行 I/O 并让缓冲区中的数据保持不变

尽管如此，您会发现许多 DBA 依赖并引用此实用程序，仿佛这些数字在某种程度上是福音。因此，即使这个工具易于使用且输出简单，也不要陷入过度依赖像 `CALIBRATE_IO` 这样单一、用途有限的工具的陷阱。请记住，简单往往不等同于好。

## 数据库基准测试



### HammerDB

#### 简介与资源

在众多适用于多种数据库平台的知名且广受欢迎的数据库基准测试工具中，HammerDB 当属其一。它的前身是 HammerORA，当时还只是一个 Oracle 专属工具。多年来我一直使用这款工具，并认为对于那些希望快速执行并获取行业标准数据库基准测试结果的人来说，尤其是在那些仅偶尔进行此类测试、并非数据库基准测试专家的人看来，它堪称最优秀的数据库基准测试工具之一。尽管 HammerDB 的图形用户界面（GUI）需要一点时间来适应，但其工作流程足够简单，大多数人无需阅读文档即可完成。

HammerDB 是一款开源的数据库基准测试工具，支持 Oracle、SQL Server、DB2、TimesTen、MySQL、MariaDB、PostgreSQL、Greenplum、Postgres Plus Advanced Server、Redis、Amazon Aurora 和 Redshift，以及 Hadoop 上的 Trafodion SQL。你可以在以下网址分别找到 HammerDB 的网页、SourceForge 项目以及 Git 源代码版本控制仓库。

*   `http://www.hammerdb.com`
*   `https://sourceforge.net/projects/hammerora`
*   `https://sourceforge.net/p/hammerora/code/ci/master/tree/`

#### 执行基准测试的流程

既然我们已经选择了数据库平台和基准测试，是时候深入了解使用 HammerDB 成功运行数据库基准测试的四个步骤了：

1.  构建数据库模式对象并加载数据。
2.  加载驱动脚本，其代码实现了所选的数据库基准测试。
3.  创建所有将运行所选数据库基准测试工作负载的并发虚拟用户会话。
4.  启动所有并发虚拟用户会话，开始运行所选数据库基准测试的工作负载。

## 步骤 #3：创建虚拟用户

我们现在继续展开 `虚拟用户` 树状视图节点，然后双击 `选项` 节点，这将启动虚拟用户选项，如图 3-19 所示。请注意，步骤 #3 也是一个更简短、更简单的步骤；因此，与第 #2 节类似，本节内容特意较为简略。

![../images/464117_1_En_3_Chapter/464117_1_En_3_Fig19_HTML.jpg](img/464117_1_En_3_Fig19_HTML.jpg)

图 3-19：HammerDB 虚拟用户选项

在这里，我们必须指定所需的并发虚拟用户数量、虚拟用户延迟或 `思考时间` 的长度、处理延迟或 `滞后时间` 的长度，以及每个虚拟用户重复整个工作负载的迭代次数。请注意，尽管图 3-18 中的这个屏幕看起来相当直观和简单，但您需要在此屏幕上花些时间，因为这些值可能是由规范定义和/或强制要求的，并且在此屏幕上可能不会默认设置得当。不注意这一点很可能导致意外且不可重复的结果。

例如，假设我们当时进行的是 `TPC-C` 基准测试，并且在数据库对象创建和数据加载阶段请求了，比如说，`10 个仓库`。`TPC-C` 规范明确定义了每个仓库允许的最大用户数量为 `10`。然而，这个屏幕会允许我们请求 `200 个并发虚拟用户`，并且进一步地，不会警告这超出了规范范围。这可能会导致不必要的数据库行争用，并且由于没有足够的、独立的行供用户处理而导致结果不正确。您甚至可能会看到数据库死锁错误。无论如何，结果很可能不可靠且不可重复。

这里还有一个重要的注意事项值得提及。虽然 HammerDB 提供了定义虚拟用户开始处理之间以及迭代之间延迟的功能，但其他工具可能没有。再次强调，这样的设置可能使两个不同的数据库基准测试工具看起来产生了不同的结果。但严酷的事实是，种瓜得瓜，种豆得豆。因此，请注意这类细节。

现在选择了虚拟用户选项后，是时候生成那些处于空闲状态但已准备就绪的虚拟用户会话进程了。为此，我们再次查看 `虚拟用户` 树状视图节点下方，但现在双击 `创建` 节点，这将生成或分叉那些空闲的虚拟用户会话，如图 3-20 所示。

![../images/464117_1_En_3_Chapter/464117_1_En_3_Fig20_HTML.jpg](img/464117_1_En_3_Fig20_HTML.jpg)

图 3-20：HammerDB 虚拟用户启动

图 3-20 中有几个值得注意的项目需要认识和理解。首先，生成的虚拟用户数量正好是请求数量，并且没有像数据库对象加载阶段那样的控制或监控附加项。其次，如果您请求了 `100 个虚拟用户`，那么请确保滚动浏览列表并验证您实际生成了请求数量的虚拟用户并已准备就绪。人们很容易忘记这一步，并且没有意识到结果不可重复的原因是虚拟用户数量由于某些不可预见的问题（如机器内存不足或数据库允许的并发连接数限制）而发生了变化。

### SLOB

Oracle DBA 最流行且广泛使用的工具之一是 **SLOB**，其全称是“小 Oracle 基准测试工具”（Silly Little Oracle Benchmark）。它是免费的，这进一步增加了其受欢迎程度。我称其为“高度使用”，是因为几乎每个 Oracle DBA 都知道并使用它。但这并不一定意味着他们理解了其真正的目的。引用该工具作者 Kevin Closson 的话：

> *SLOB 不是一个数据库基准测试工具。SLOB 是一个 Oracle I/O 工作负载生成工具包。我需要指出，由于习惯使然，许多 SLOB 用户会交替使用“基准测试”和“工作负载”等术语来指代 SLOB。SLOB 旨在填补 Orion 与 `CALIBRATE_IO` 以及全功能事务性基准测试之间的空白。*

因此，SLOB 实际上并不适合放在前面专门讨论存储基准测试的章节中，也不完全适合放在这里与数据库基准测试工具一起讨论。但正如我所说，大多数 Oracle DBA 都在使用它，甚至连该工具的作者也承认他们经常将其误解为数据库基准测试工具。所以，基于以上说明，我们将在本节末尾介绍它。

你可以在以下网址分别找到 SLOB 的网页和下载链接。

*   [`https://kevinclosson.net/slob/`](https://kevinclosson.net/slob/)
*   [`https://github.com/therealkevinc/SLOB_distribution`](https://github.com/therealkevinc/SLOB_distribution)

SLOB 是一系列 Linux shell 脚本和一些基于 C 语言的程序。要使用 SLOB，你需要了解两个脚本：`setup.sh` 和 `runit.sh`。还有一个名为 `slob.conf` 的文本文件，其中包含控制 SLOB 运行时行为的各种参数。你很可能需要根据你的环境编辑此文件。在我的案例中，我更改了 `SYSTEM` 密码并将快照机制改为 AWR。

第一个脚本 `setup.sh` 接受两个输入参数：用于创建其模式及其对象的表空间，以及要创建的模式数量。在我的案例中，我在笔记本电脑上运行一个 Linux 虚拟机，因此我希望最小化设置：使用现有的 `USERS` 表空间并仅加载一个模式。图 3-40 展示了使用这些值执行 `setup.sh` 的情况。

![../images/464117_1_En_3_Chapter/464117_1_En_3_Fig40_HTML.jpg](img/464117_1_En_3_Fig40_HTML.jpg)

图 3-40

SLOB `setup.sh` 执行
