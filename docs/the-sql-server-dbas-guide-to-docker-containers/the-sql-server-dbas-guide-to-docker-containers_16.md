# 7. 数据持久化

> *好记性不如烂笔头。*
>
> ——中国谚语

刚在我的书桌前安顿下来不久，我就向我的主日学老师要了一些纸和蜡笔。并不是因为我喜欢写字，而是作为一个生病的孩子，这几乎是为数不多我能做的事情之一。根据我的医生所说，我不能进行剧烈运动，不能做体育活动，甚至不能大笑（他们就是这么告诉我的）。所以，我只能读书、弹钢琴和写字——或者说涂鸦，因为我连自己写的字都认不出来。

我的主日学老师递给我一个 Magic Slate（魔法画板），据她说，这是一个可以反复使用的绘画工具。我四岁的小脑瓜试图理解它是什么以及它是如何工作的。而且，那时我只知道铅笔——有时是蜡笔——和纸。我被告知可以在上面画画，掀起那层薄薄的纸就能擦掉我的画，然后可以一遍又一遍地想画多少次就画多少次。它配有一支触控笔，但我的指甲就足够用来涂鸦了。于是，我开始涂鸦和画画。我在上面写下了我的名字。我甚至尽可能详细地试着画了一个孩子的脸。有趣的是，当我给我的“模特”看我这幅“艺术肖像”时，我们俩都笑了起来。

当他在魔法画板上掀起那层薄薄的纸时，我们的笑声戛然而止。我那幅花了好几个小时精心绘制的画，在不到我说出“本！”所需的时间里就消失了（如果你正在读这个，抱歉，本。总得有人当替罪羊，而我碰巧是讲这个故事的人）。我崩溃了。这怎么可能发生？我放声大哭，直到我的主日学老师走过来问我们发生了什么。当我试图解释我的画所表演的“消失术”时，她提醒我，魔法画板就是这么工作的。你在上面写写画画，然后掀起那层薄膜就能擦掉一切。它就是那样设计的。我花了足足半个小时才最终平静下来，重新要了纸和蜡笔。我回到了我知道有效的方法上，并发誓再也不碰那个魔法画板了。至少我当时是这么想的。

在你过于投入于将 SQL Server 数据库部署到容器上之前，我想确保你明确一点：Docker 容器最初被设计为**无状态**的。这意味着它们不具备数据持久性，当它们被移动到另一台服务器（例如你决定部署一个包含多个主机的 Docker Swarm 集群）或容器被销毁时，它们无法维持数据。对于 SQL Server 数据库专业人员来说，这是一个大忌。我们想要存储并持久化数据。我们想要保护它，备份它，并确保需要时它就在那里。这个 Docker 东西怎么可能适合 SQL Server 或任何类型的数据存储平台呢？

如果情况果真如此，我本可以跳过撰写本章。但我没有。本章涵盖了在容器中处理有状态应用程序（如关系数据库）的内容。我将向你介绍 Docker 卷，如何使用它们，以及如何使用卷来存储 SQL Server 数据库文件。一旦你完成配置，你就会看到 Docker 容器如何用于有状态应用，尤其是 SQL Server。

## 有状态应用 vs. 无状态应用

我在前一节提到，Docker 最初是被设计为无状态的。无状态应用不会保存在一个会话中产生的客户端数据供下一个会话使用。想象一下，每隔一周从你最喜欢的理发师那里理发。无状态意味着你每次去都要告诉理发师你想要什么样的发型。你的理发师不会记录你偏好的发型，也不会记住你在店里最喜欢的椅子，更不会给你他们招牌的洗发头皮按摩；你明白这个意思了。

相比之下，有状态应用则会保存每次会话中产生的客户端数据，并在客户端下次请求时使用这些数据。这就好比你的理发师，在你踏进店门的那一刻，就已经为你准备好了你最喜欢的椅子、将要用于招牌头皮按摩的洗发水，并且已经准备好为你做你偏好的发型。所有关于你和你的偏好的信息都存储并持久化在理发师的大脑中。然而，你可能会争辩说大脑是临时存储，因为如果你的理发师决定去度假，你就不得不告诉替代理发师你所有的偏好。更好的存储这些信息的方法是写在一张纸上供所有理发师查阅——或者，如果理发店足够高级，可以使用一个带有后端数据库的客户关系管理（CRM）系统。

我假装很了解发型和理发店，但我其实并不懂。我已经十年没去理发店了。请不要评判我。

## 容器是短暂且不可变的

我最喜欢的电影《黑客帝国 2：重装上阵》中的一个场景是尼奥与先知的对话。虽然他们讨论中传达的哲学思想更让我着迷，但他们对话中的这段特定问答引起了我的注意。对话是这样的：

*   *尼奥：那为什么一个程序会被删除呢？*
*   *先知：也许它崩溃了。也许一个更好的程序被创建出来替代它——这种事一直在发生。*

我不确定这段对话是否是容器这两个原则背后的灵感：**短暂**和**不可变**。但它确实是解释它们如何应用于容器的一个好方法。

短暂意味着容器的生命周期很短——是临时的。不可变意味着容器是不变的；它们在创建之后，状态和内容就无法被修改。就像先知向尼奥解释程序为什么会被删除一样，一个应用可能故障，也可能为了提供额外功能而被升级。先知说得对，这种事确实一直在发生。但我们过去处理服务器上运行的软件以应对这两种可能结果的方式，却与尼奥的做法大相径庭。当一个应用出现故障或需要升级时，我们不会删除它。我们会把它迁移到不同的服务器——或者不同的虚拟机（VM）上。我们维护、打补丁、更新、迁移等等。运行应用的服务器有更长的生命周期，并且会不断更新——但很少被删除。我们将它们带入软件的下一个版本，随之带来所有可能影响应用稳定性和性能的包袱和垃圾。难怪升级或迁移项目会耗费如此多的时间和资源。

另一方面，容器的生命周期非常短。想要在容器上运行应用的新版本？很简单。不要更新容器。只需删除它，然后部署一个包含新版本应用的新容器。听起来太简单了？你将不得不等到本章稍后部分才能看到实际操作。
