# 引言

那是 1999 年。我和我的前女友（现在的妻子）正在为一家即将开业的新网吧做最后的准备。我拆开我的全新 US Robotics Sportster 56K 拨号调制解调器，把它插进我的主机，然后等待测试网络连接。这是我期待已久的时刻。

那时，我对互联网还相对较新。大学期间接触互联网后，我便领略到了它的强大。我和同学们用它来通过电子邮件或简单的 FTP 互相发送物理实验报告。我跳过了图书馆，开始使用 Lycos 和 AltaVista 进行研究。我在 GeoCities 上创建了一个功能类似简历的简单网站。我知道我偶然发现了一座金矿，大学里（甚至我所在的城市）还没有人充分意识到这一点。但当我打开电脑连接到互联网时，所见的景象却与此截然不同。

雅虎邮箱里弹出了 Dialpad 的一则广告，内容是关于一项新服务：足不出户，免费拨打全球国际电话。起初我觉得这简直疯了。要知道，在那个年代，打国际电话，你得拨通一个连接到中继线或接线员的号码，由他们代你呼叫。而且，从菲律宾马尼拉打到美国旧金山的电话，平均每分钟要 5 美元。这个 Dialpad 的东西毫无道理可言。

但我也没什么可失去的。我把耳机连接到电脑上，在 Netscape 浏览器中启动了 Dialpad 小程序，拨打了我姑妈在旧金山的电话号码。几分钟后，我们就开始通话了——我能听到她的声音，她也能听到我的。我简直不敢相信。我以为的疯狂想法变成了现实。我迫不及待地想告诉父母。再也不用打昂贵的跨国电话了。但是……

有趣的是，一项始于四十多年前（发明于 1973 年，但直到 90 年代末才商业化）的技术创新，如互联网协议语音技术，竟创造了一个完整的产业。它创造了诸如 VoIP 工程师、网络工程师、开发利用 VoIP 应用程序的开发者等工作机会。它使得运行在 VoIP 之上的应用程序成为主流——你在 Instagram 上呼叫的朋友、在 Zoom 或 Skype 上主持的电话会议，或者你在 WhatsApp 上聊天的姑妈。

每一项解决重大业务痛点的技术创新，都经历了构想、被采纳、最终普及的相同过程。就 VoIP 而言，它解决了打国际电话昂贵且繁琐的问题。我们技术专业人士应该时刻关注下一个新事物，运用批判性思维思考它如何能创造下一个产业。我们需要学会观察风向，看清它的去向，并调整我们的风帆。否则，我们就会面临被淘汰、无法在职业道路上前进的风险。还记得大型机和 FORTRAN 吗？

但这并不意味着你就会被同行认可为科技界的预言家。事实上，恰恰相反。你会被嘲笑、讥讽、蔑视，甚至被视为疯子。当我告诉我妈我可以用 Dialpad 服务免费打美国电话时，她真的笑了。当我在 1996 年告诉同学们机器人和人工智能将成为主流时，我被嘲笑了。当我在 2003 年告诉我的同行们，他们应该期待在微软认证系统工程师认证中增加 Linux 考试选项时，他们说我疯了。史蒂夫·鲍尔默在 2001 年称 Linux 为“癌症”，这更是雪上加霜。

这就是为什么我描述了技术创新的生命周期。因为如果你知道它是如何运作的，别人怎么想、怎么说就无关紧要了。你完全确信它会按它应有的方式运作。回顾过去四十年的技术创新。虚拟化诞生于大型机时代，源于在大量用户中共享昂贵计算机资源的需求。如今，每个组织都利用虚拟化来运行应用程序。公共云的诞生，则源于将 IT 视为创收业务模式而非仅仅是一项业务成本的需求。如今，人们使用公共云提供商，而不是自建私有数据中心和购买硬件。

这把我们带到了 2017 年，也就是微软发布 SQL Server 2017 的那一年。这是 SQL Server 首次在 Linux 上运行。是的，那个史蒂夫·鲍尔默在 2001 年称为“癌症”的 Linux，根据 Gartner 的数据，现在正运行着世界上最流行的操作型数据库管理系统。就连微软 Azure 认证考试中也包含了 Linux 组件。微软加 Linux 不再是愚人节玩笑的话题了。这是真的。就像任何技术创新的生命周期一样，这也将成为主流。问题是，“当那一刻到来时，你准备好了吗？”



伴随着 SQL Server 在 Linux 上的发布，另一项早已兴起的技术创新也随之而来：容器技术。虽然容器最初是为 Linux 操作系统设计的，但它解决了一个每个 IT 组织都面临的独特业务问题：当软件从一个计算环境迁移到另一个计算环境时，能够可靠地运行。想象一下，升级或迁移从物理机到另一台物理机，或者到虚拟机，或者到公有云。甚至将代码从开发者的笔记本电脑移动到生产环境。我相信你肯定不止十几次听过你的开发者说“但它在我笔记本上是可以运行的”。容器技术就是来消除这些挑战的。

既然这两项技术都已可用，那么看到微软不仅支持在容器技术中运行的 Linux 版 SQL Server，还将其用于自身的内部开发和构建流程，也就不足为奇了。在容器上运行的 Linux 版 SQL Server 比我们过去二十五年来熟知的 Windows 版 SQL Server 变得更加普遍，这只是时间问题。

而这正是本书的目标。像你这样已经使用 SQL Server 一段时间的数据库管理员和开发人员，对关系数据库引擎已经略知一二。你已经知道 SQL Server 如何在数据页上存储数据，索引如何加速（或不加速）查询，以及它如何实现高可用性。我希望你带着这些相同的知识和经验，为迎接容器技术的下一波技术创新浪潮做好准备。想象你正在一段旅程中，这本书将是你的向导，每一章都为你铺就通往最终目的地的步骤。

## 本书结构

本书包含 11 个章节和一个附录，结构如下：

*   第 `1` 章向你介绍容器技术，简要回顾其起源历史，以及微软如何将其应用于 Windows Server 操作系统和 Linux 上的 SQL Server。
*   第 `2` 章将引导你如何将 Windows Server 操作系统配置为容器主机，让你能够利用现有的 Windows 知识。
*   第 `3` 章涵盖了一些关于 Linux 操作系统的基础知识，为你将其配置为容器主机做好准备。
*   第 `4` 章介绍 Docker 生态系统，以更好地理解容器技术的工作原理。
*   第 `5` 章向你介绍 Docker 镜像和容器，但重点在于如何将其与 SQL Server 配合使用。
*   第 `6` 章讲解容器的管理和运维，介绍操作容器时你需要执行的最常见任务。
*   第 `7` 章介绍在容器中持久化数据的概念，因为 SQL Server 的设计目的就是存储和持久化数据。
*   第 `8` 章涉及一些在 Linux 上使用 SQL Server 的内容，包括安装和管理体验，帮助你开始管理 Linux 上的 SQL Server，并最终构建自定义的 Linux 版 SQL Server 容器镜像。
*   第 `9` 章将引导你构建一个自定义的 Windows 版 SQL Server 容器镜像，可用于标准化部署。
*   第 `10` 章将引导你构建一个自定义的 Linux 版 SQL Server 容器镜像。
*   第 `11` 章介绍一些 Docker 网络知识，以便你更好地配置容器上的 SQL Server，使其能与其他应用程序交互。

我强烈建议你从`附录`开始。虽然现在在亚马逊 AWS、微软 Azure 或谷歌云平台等公有云基础设施上快速搭建一个实验环境非常容易，但我不希望你错过从头开始完整搭建一切的体验。听到资深 SQL Server DBA 没有安装 Windows Server 操作系统和配置 TCP/IP 的经验，我感到很惊讶。这本身没什么错。但很多时候，这恰恰决定了一个问题能否被轻松解决。

另外，由于公有云环境中的网络基础设施是以服务形式提供的，存在多个抽象层掩盖了真实情况。当你试图在一个安全的环境中测试新东西，结果却发现你浪费了那么多小时去解决一个你无法控制、或者只有提供信用卡信息才能解决的问题时，这可能会非常令人沮丧。你来这里是为了学习容器技术，而不是陷入财务困境。

那么，让我们开始吧，深入容器技术的世界。
