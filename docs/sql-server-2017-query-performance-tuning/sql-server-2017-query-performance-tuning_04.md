# SQL Server 性能杀手

现在，让我们来考虑可能降低 `SQL Server` 性能的主要问题领域。通过预先了解 `SQL Server` 中的主要性能杀手，你将能够把调优精力集中在可能的原因上。

一旦你优化了硬件、操作系统和 `SQL Server` 设置，`SQL Server` 中的主要性能杀手按粗略顺序（最严重的排在最前）如下：

- 索引不足或不准确
- 统计信息不准确
- 查询设计不当
- 生成的执行计划不佳
- 过度的阻塞和死锁
- 非基于集合的操作，通常是 `T-SQL` 游标
- 不适当的数据库设计
- 执行计划重新编译
- 查询频繁重新编译
- 游标使用不当
- 索引碎片过多

让我们快速浏览一下这些问题。

## 索引不足或不准确

索引不足通常是 `SQL Server` 中最大的性能杀手之一。同样糟糕，有时更糟的是索引错误。在查询缺乏适当索引的情况下，`SQL Server` 在执行查询时必须检索和处理更多的数据。这会对磁盘、内存和 `CPU` 造成巨大压力，显著增加查询执行时间。查询执行时间的增加进而可能导致 `SQL Server` 中过度的阻塞和死锁。你将在第 8–12 章学习如何确定索引策略和解决索引问题。

通常，索引被认为是数据库管理员 (`DBA`) 的责任。然而，`DBA` 无法主动定义如何使用索引，因为索引的使用是由开发人员编写的数据库查询和存储过程决定的。因此，定义索引必须是共同的责任，因为开发人员通常更了解要检索的数据，而 `DBA` 更了解索引的工作原理。不了解查询而创建的索引几乎毫无用处。

索引过多或错误的索引同样会造成诸多问题。大量索引会因为需要维护而减慢通过 `INSERT`、`UPDATE` 和 `DELETE` 进行的数据操作。性能下降会导致过度阻塞，进而再次引发死锁。不正确的索引不会被优化器使用，但仍然必须进行维护，在处理能力、磁盘存储和内存上付出代价。

## 注意

由于不了解查询而创建的索引几乎毫无用处，数据库开发人员需要至少像了解 `T-SQL` 一样理解索引。

## 统计信息不准确

`SQL Server` 高度依赖基于成本的优化，因此准确的数据分布统计信息对于有效使用索引极为重要。没有准确的统计信息，`SQL Server` 的查询优化器无法准确估计查询影响的行数。因为从表中检索的数据量对于决定如何优化查询执行至关重要，如果数据分布统计信息维护不准确，查询优化器的效果将大打折扣。统计信息可能因未更新而过时。你还会看到数据分布不均匀损害统计信息的问题。对于自增值（如 `date`）的列上的统计信息，随着新数据的添加，可能会过时。你将在第 13 章学习如何分析统计信息。



## 不当的查询设计

索引的有效性在很大程度上取决于你编写 SQL 查询的方式。从表中检索过多的行，或指定的过滤条件从表中返回了比实际需要更大的结果集，都可能导致索引失效。为了提高性能，你必须确保 SQL 查询的编写能够充分利用新的或现有的索引。未能编写出具有成本效益的 SQL 查询可能会阻止优化器选择合适的索引，从而增加查询执行时间和数据库阻塞。19 章详细介绍了如何编写有效的查询。

查询设计不仅涉及单个查询，还包括常用于实现数据库功能（例如队列读取器和写入器之间的队列管理）的一组查询。即使设计中使用的单个查询性能良好，数据库的整体性能也可能非常差。解决这类瓶颈需要对 `SQL Server` 的不同特性有广泛的理解，这些特性会影响数据库功能的性能。在本书中，你将看到如何使用 SQL 查询设计有效的数据库功能。

## 生成的执行计划不佳

让 `SQL Server` 能够建立高效的执行计划并重复使用该计划（而不是重新编译）的相同机制，在某些情况下可能对你不利。一个糟糕的执行计划可能是真正的性能杀手。当一种称为*参数探测*的过程出现问题时，经常会导致不准确和性能低下的计划。参数探测是一个过程，它源于查询优化器根据统计信息中的采样或特定值来确定最佳计划的机制。理解统计信息和参数如何结合以创建执行计划，以及你可以做些什么来控制它们，这一点很重要。统计信息在第 13 章中介绍，执行计划分析在第 15 章和第 16 章中介绍。第 17 章专门讨论不良的参数探测以及如何最好地处理它（结合第 11 章关于查询存储和计划强制的细节）。

## 过度的阻塞和死锁

因为 `SQL Server` 完全符合 `ACID` 规范，数据库引擎确保并发事务所做的修改彼此正确隔离。默认情况下，一个事务看到的数据要么是在另一个并发事务修改数据之前的状态，要么是在该事务完成之后的状态——它不会看到中间状态。

由于这种隔离性，当多个事务试图以不兼容的方式同时访问公共资源时，数据库中就会发生*阻塞*。两个进程无法同时更新同一数据。此外，由于 `SQL Server` 中的所有更新都建立在数据页（8KB 大小的行集）上，即使两个进程没有更新同一行，你也能看到阻塞发生。阻塞在确保正确的数据存储和检索方面是好事，但如果在错误的地方发生过多，就会拖慢速度。

与阻塞相关但实际上是另一个问题的是*死锁*，当两个资源试图升级或扩展锁定的资源并相互冲突时就会发生。查询引擎确定哪个进程回滚成本最低，并选择其作为*死锁牺牲品*。这需要重新提交数据库请求才能成功执行。死锁是一个根本性的性能问题，尽管许多人认为它是一个结构性问题。查询的执行时间会受到它所遇到的阻塞和死锁（如果有的话）数量的不利影响。

对于多用户数据库应用程序的可扩展性能，正确控制查询的隔离级别和事务范围以最小化阻塞和死锁至关重要；否则，查询的执行时间将显著增加，即使硬件资源可能利用率很低。我将在第 21 章和第 22 章深入讨论这个问题。

## 非基于集合的操作

`Transact-SQL` 是一种基于集合的语言，这意味着它操作的是数据集。这迫使你以列而不是行为单位进行思考。非基于集合的思维会导致过度使用游标和循环，而不是探索更高效的连接和子查询。`T-SQL` 语言提供了丰富的机制来操作数据集。为了发挥性能优势，你需要利用这些机制，而不是强行采用逐行处理的方法，那会扼杀性能。如何做到这一点的示例在本书中随处可见；另外，我在第 19 章讨论了 `T-SQL` 的最佳实践，在第 23 章讨论了游标。

## 不恰当的数据库设计

数据库应该充分规范化，以提高数据检索性能并减少阻塞。例如，如果你有一个规范化不足的数据库，将客户和订单信息放在同一个表中，那么客户信息将在该客户的所有订单行中重复。这种信息在每一行的重复会增加获取客户所有订单所需的页面读取次数。同时，处理客户订单的数据写入器将保留包含客户信息的所有行，从而可能阻塞所有其他试图访问客户资料的数据写入器/数据读取器。

数据库的过度规范化可能和规范化不足一样糟糕。过度规范化增加了检索数据所需的连接数量和复杂性。过度规范化的数据库包含大量列数很少的表。过度规范化并不是我经常遇到的问题，但当我遇到时，它对性能的影响非常严重。更常见的情况是处理结构规范化不足或规范化不当的问题。

查询中有太多连接也可能是因为数据库实体没有明确分区，或者查询是为了满足复杂的需求，而创建一个新的存储过程可能更好地满足这些需求。

数据库设计的另一个问题是实际实施 `主键`、`唯一约束` 和强制的 `外键`。这些机制不仅确保数据的一致性和准确性，而且查询优化器在决定如何解析特定查询时可以利用它们。然而，人们常常忽略创建主键或禁用他们的外键，无论是直接禁用还是通过使用 `WITH NOCHECK`。没有这些工具，优化器别无选择，只能创建次优的计划。

数据库设计是一个庞大的主题。我将在第 19 章和本书的其余部分提供一些要点。由于主题的规模，我无法以它所需的完整方式来处理它。但是，如果你想阅读一本侧重于介绍该主题的数据库设计书籍，我推荐 Louis Davidson 等人所著的《Pro SQL Server 2012 Relational Database Design and Implementation》(Apress, 2012)。


### 重新编译执行计划

为了以高效的方式执行查询，SQL Server 的查询优化器会消耗大量的 CPU 周期来创建一个具有成本效益的执行计划。好消息是，该计划会被缓存在内存中，因此一旦创建就可以重用。然而，如果生成的计划无法插入参数值，那么 SQL Server 每次在相同查询以不同值重新提交时都会创建一个新的执行计划。因此，为了获得更好的性能，以有助于 SQL Server 缓存和重用执行计划的形式提交 SQL 查询是极其重要的。我还将讨论诸如计划冻结、强制查询计划以及使用“即席工作负载优化”等主题。你将在第 16 章中详细了解如何提高执行计划的可重用性。

### 查询的频繁重新编译

确保可重用执行计划（独立于查询中使用的值）的标准方法之一是使用存储过程或参数化查询。使用存储过程来执行一组 SQL 查询允许 SQL Server 创建一个参数化的执行计划。

一个`parameterized execution plan`（参数化执行计划）独立于存储过程或参数化查询执行期间提供的参数值，因此具有很高的可重用性。查询的频繁重新编译会增加 CPU 压力和查询执行时间。我将在第 18 章中详细讨论存储过程和语句重新编译的各种原因和解决方案。

### 游标的不当使用

优先使用基于游标的（逐行）结果集——或者正如 Jeff Moden 精辟地称之为“痛苦逐行”（`Row By Agonizing Row (RBAR; pronounced "ree-bar")`）——而不是常规的基于集合的 SQL 查询，会给 SQL Server 增加大量开销。尽可能使用基于集合的查询，但如果被迫处理游标，请务必使用高效的游标类型，如`fast-forward only`（仅快速向前）。过度使用低效的游标会增加 SQL Server 资源的压力，从而减慢系统性能。如果你必须使用游标，我将在第 23 章讨论如何正确地使用它们。

### 过度的索引碎片

在分析数据检索操作时，通常可以假设数据是按索引指示的有序方式组织的。然而，如果包含数据的页面以无序方式碎片化，或者由于频繁的页面拆分而只包含少量数据，那么数据检索操作所需的读取操作次数将远高于原本所需的次数。由碎片引起的读取操作次数的增加会损害查询性能。在第 14 章中，你将学习如何分析和消除碎片。不过，值得一提的是，关于索引碎片有很多新的观点，认为它可能根本不是问题。你需要评估你的系统以检查这是否是一个问题。

## 总结

在这个介绍性的章节中，你了解到 SQL Server 性能调优是一个迭代过程，包括识别性能瓶颈、排查其原因、应用不同的解决方案、量化性能改进，然后重复这些步骤，直到达到所需的性能水平。为了协助这个过程，你应该创建一个系统基线，以便与你的修改进行比较。在整个性能调优过程中，你需要客观地看待你想要进行的调优量——你总是可以让一个查询运行得更快一点，但这种努力是否值得付出成本？最后，由于性能取决于用户活动和数据的模式，你必须定期重新评估数据库服务器的性能。

为了从 SQL Server 数据库系统中获得最佳性能，理解数据库应用程序对服务器造成的压力是极其重要的。在接下来的三章中，我将讨论如何分析这些压力，既在更高的系统层面，也在更低的 SQL Server 活动层面。然后我会展示如何将两者结合起来。

在本书的其余部分，你将深入探讨本章前面提到的最大的 SQL Server 性能杀手。你将了解这些单独的因素如果使用不当会如何影响性能，以及如何解决或避免这些陷阱。
