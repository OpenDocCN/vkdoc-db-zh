# 4. CPU 性能分析

本章通过讨论 CPU、网络和通用 SQL Server 指标，结束本书对系统的探索。CPU 是系统的工作引擎，维持一切运行。收集和交付数据、维护系统以及排序访问所需的所有不同计算都由 CPU 执行。CPU 遇到瓶颈可能是一个难以摆脱的过程。与内存（有时可以轻松安装更多）或磁盘（有时可以轻松添加更多或升级）不同，CPU 是你所运行系统的一个组成部分，通常只能通过购买更新的机器来升级。因此，你需要密切关注 CPU 使用率。网络很少成为 SQL Server 的主要瓶颈，当然在处理 Azure SQL 数据库时除外，但最好也留意一下。最后，有一些 SQL Server 内部进程你需要收集指标。本章涵盖以下主题：

*   如何收集处理器指标
*   通过 T-SQL 查询可用的附加指标
*   解决处理器瓶颈的方法

## 处理器瓶颈分析

SQL Server 大量使用任何可用的处理器资源。你更可能在 I/O 或内存上遇到瓶颈，但也可能遇到 CPU 问题。我们在此介绍的措施侧重于操作系统和 SQL Server。然而，在虚拟化环境中，我们关注的 CPU 度量指标不太可能反映现实情况。你需要处理你所使用的虚拟机管理程序或系统，以准确了解某些操作系统级别的 CPU 度量指标实际上是如何反映现实情况的。你可能正经历外部压力甚至外部限制，这些都不会在下面概述的计数器中显示出来。

你可以使用表 4-1 中的 `性能监视器` 计数器来分析处理器资源上的压力。

### % Processor Time（处理器时间百分比）

`% Processor Time` 不应持续过高（大于 80%）。任何持续超过 90% 的处理器时间，其效果实际上与 100% 相同。如果 `% Processor Time` 持续偏高，而磁盘和网络计数器值较低，你的首要任务必须是减轻处理器上的压力。请记住，这里的数字只是建议；人们可以基于合理的原因对这些数字持不同意见。将它们作为评估系统的起点，而不是具体的建议。

例如，如果 `% Processor Time` 为 85%，并且通过监控 I/O 计数器发现磁盘使用过多，那么很可能处理器时间的主要部分花费在了管理磁盘活动上。这将反映在处理器的 `% Privileged Time` 计数器中，如下一节所述。在这种情况下，优先优化磁盘瓶颈将是有利的。此外，请记住，磁盘瓶颈反过来可能是由于内存瓶颈造成的，正如本章前面所解释的。

你可以将处理器时间作为机器上所有处理器的总和进行跟踪，也可以单独跟踪特定处理器的使用率百分比。这允许你在 SQL Server 运行在四处理器机器的三个处理器上的情况下隔离数据收集。请记住，你可能会看到一个处理器达到最大负载，而另一个处理器负载很小。在这种情况下，平均值无法反映现实情况。将平均值仅作为一个指标，而将单个值视为系统上实际负载和处理的更准确度量。

在虚拟化环境中，CPU 可能是虚拟化的，因此你看到的情况并不准确。例如，在 `VMware` 系统中，如果安装了 `VMware Tools`，你将能够查看 `VM Processor` 计数器以查看主机的处理器使用率。使用这个度量指标，你可以判断你在操作系统中看到的 CPU 使用率是否反映在主机上，或者你是否只是用完了虚拟 CPU。另一方面，运行 `HyperV` 时，你需要查看 `\Hyper-V Hypervisor Logical Processor(_Total)\% Total Run Time` 以获取相同的度量指标。根据你使用的虚拟机管理程序，你还会遇到其他度量指标。


## % 特权时间

Windows 服务器上的处理有两种模式：`用户模式`和`特权模式`（或`内核模式`）。所有系统级活动，包括磁盘访问，都在特权模式下进行。如果你发现专用 SQL Server 系统上的 `% 特权时间` 达到 20% 到 25% 或更高，那么系统可能正在进行大量的外部处理。这可能是 I/O、加密服务等过滤器驱动程序、有缺陷的 I/O 组件，甚至是过时的驱动程序。专用 SQL Server 系统上的 `% 特权时间` 计数器最多应为 5% 到 10%，但请使用你的基线来确定在你的系统上什么行为看起来是正常的。

## 处理器队列长度

`处理器队列长度` 是处理器队列中的线程数。（即使在有多处理器的计算机上，也只有一个处理器队列。）与磁盘计数器不同，`处理器队列长度` 计数器不读取正在运行的线程。在 CPU 利用率较低的系统上，`处理器队列长度` 计数器通常为 0 或 1。

持续大于 2 的 `处理器队列长度` 计数器通常表示处理器拥塞。由于存在多处理器，你可能需要考虑处理处理器队列长度的调度器数量。超过调度器数量两倍（通常调度器数量与处理器数量 1:1）的处理器队列长度也可能表示处理器瓶颈。虽然高的 `% 处理器时间` 计数器表示处理器繁忙，但持续高的 `处理器队列长度` 计数器是更确定的指标。如果超过建议值，通常表示准备运行的线程数量多于当前处理器数量所能最优服务的数量。

## 上下文切换/秒

`上下文切换/秒` 计数器监视计算机上所有处理器从一个线程切换到另一个线程的综合速率。当运行线程自愿放弃处理器、被更高优先级的就绪线程抢占，或在用户模式和特权模式之间切换以使用执行体或子系统服务时，就会发生上下文切换。它是计算机上所有处理器中运行的所有线程的 `线程:上下文切换/秒` 之和，以切换次数衡量。

高数值在很大程度上取决于你的 CPU 速度，因此要随时间测量性能，并将此数字与你的基线进行比较，以了解何时可能出现偏差。

## 批处理请求/秒

`批处理请求/秒` 为你提供了一个很好的指标，可以了解系统上承载的负载量，这与处理器承载的负载有直接关联。由于你可能会在系统上看到许多低成本查询或少数高成本查询，因此你不能单独看这个数字，而必须参考本节定义的其他计数器；每秒 10,000 个请求将被视为繁忙系统。更大的值可能引起关注，但这完全取决于你的系统什么是正常的。了解哪个值在你自己的系统中具有意义的最佳方法是建立一个基线，然后从那里开始监控。请记住，这里的高数字不一定会引起关注。如果你的所有其他资源都在控制之中，并且你持续维持高批次请求数，这仅仅意味着你的服务器很繁忙。

## SQL 编译/秒

`SQL 编译/秒` 计数器在其聚合中同时显示了批编译和语句重编译。当服务器首次启动（或在故障转移或任何其他启动类型事件之后），这个数字可能非常高，但会随着时间的推移而稳定下来。稳定后，与基线度量相比显著或持续的编译峰值是引起关注的原因，并且肯定会在处理器中表现为问题，因为查询编译是一项昂贵的操作。如果你正在使用某种对象关系映射引擎，如 nHibernate 或 Entity Framework，高编译次数可能是正常的，尽管成本不低。第 14 章详细介绍了 SQL 编译。

## SQL 重编译/秒

`SQL 重编译/秒` 是批和语句重编译的度量。高数量的重编译可能导致处理器压力。因为语句重编译是此计数的一部分，所以它可能比 2005 年之前版本的 SQL Server 高得多。第 17 章详细介绍了查询重编译。

## 其他测量 CPU 性能的工具

你也可以使用 DMO 来捕获有关 CPU 的信息。这些 DMO 中的信息必须通过运行查询来捕获，然后作为你基线度量的一部分保存下来。

### Sys.dm_os_wait_stats

等待统计信息是了解系统上是否存在瓶颈的好方法。然而，你不能简单地说大于 *x* 就是一个坏数字。你需要随着时间的推移收集指标，以了解你的系统上什么是正常的。与该基准的偏差才是有趣的。针对此 DMO 的查询如果查找信号等待时间，则可以指示 CPU 瓶颈。

过去，`CXPACKET` 等待被认为对于测量 CPU 性能是浪费时间。然而，从 SQL Server 2016 SP2 和 SQL Server 2017 CU3 开始，等待统计信息出现了一个新的拆分。现在有一个 `CXPACKET` 等待（我们需要关注的）和一个 `CXCONSUMER` 等待（不重要的）。如果你运行的是 Azure SQL Database，你也会看到这种拆分。拆分的核心是为了涵盖并行性的消费者和生产者，这是某些系统上 CPU 瓶颈的一个主要原因。并行性的消费者，即接收数据的操作，通常具有可忽略但可测量的等待，即 `CXCONSUMER` 等待。现在，`CXPACKET` 测量的是生产者，即推送数据的操作符。这改变了情况。`CXPACKET` 现在是一个指示系统上影响你 CPU 的真实负载的等待。

### Sys.dm_os_workers 和 Sys.dm_os_schedulers

这些 DMO 显示 Windows 操作系统中的工作线程和调度器线程。定期对这些 DMO 运行查询将允许你获取处于可运行状态的进程数量。这是处理器负载的一个极好指标。

### 查询存储

`查询存储` 并非专门测量 CPU 的工具，但它捕获的信息确实包括相关查询的（聚合）CPU 使用率以及这些查询的等待统计信息，包括任何与 CPU 相关的统计信息。

### 在 Linux 中测量 CPU 行为

在 Linux 上运行时，你仍然可以使用 `sys.dm_os_wait_stats`。这将为你提供可以指示 CPU 负载的等待统计信息。否则，你需要进入 Linux 系统本身。查看 CPU 的通用推荐方法是使用 `top` 命令。该工具的输出记录在这里：`https://bit.ly/2KbZmuZ`。

## 处理器瓶颈解决方案

一些常见的处理器瓶颈解决方案如下：

*   优化应用程序工作负载
*   消除或减少过多的编译/重编译
*   使用更多或更快的处理器
*   不运行不必要的软件

让我们依次考虑每一个解决方案。



### 优化应用程序工作负载

要识别处理器密集型查询，请使用 `Extended Events` 会话（我将在下一章讨论）捕获所有 SQL 查询，然后在 `CPU` 列上对输出进行分组。另一种方法是利用 `Query Store`（在第 11 章 讨论）。您可以从 `sys.query_store_runtime_stats` 检索信息，以查看每个查询的多项聚合 `CPU` 指标。消耗 `CPU` 时间最多的查询对 `CPU` 压力的贡献最大。然后，您应该分析和优化这些查询，以减轻 `CPU` 的压力。通常，`CPU` 压力的原因并非查询中的大量计算，而是逻辑 I/O 中的争用。解决 I/O 问题通常也有助于解决 `CPU` 问题。您也可以直接查询 `sys.dm_exec_query_stats` 或 `sys.dm_exec_procedure_stats` 动态管理视图，以实时查看即时问题。最后，使用查询哈希和查询计划哈希，您可以识别和调整常见查询或常见执行计划（这将在第 14 章 详细讨论）。本书的大部分后续章节都与优化应用程序工作负载有关。

### 消除过多的编译/重新编译

一定数量的查询编译和重新编译是预期之中的，尤其是如前所述，在使用 ORM 工具时。只有当这些情况在持续期间内大量出现时，才表明存在问题。同样值得注意的是它们之间的比率。编译次数多而重新编译次数少，意味着系统内查询重用很少（查询重用在第 9 章 详细讨论）。大量的重新编译会导致处理器使用率高。解决重新编译的方法涵盖在第 17 章。

### 使用更多或更快的处理器

最简单的解决方案之一，也是您大多数时候会采用的方法，就是增加系统处理能力。然而，由于处理器升级涉及的成本较高，您应该首先尽可能优化 `CPU` 密集型操作。

可以通过提高单个处理器的性能或增加更多处理器来提升系统的处理能力。当 `% Processor Time` 计数器高而 `Processor Queue Length` 计数器低时，提高单个处理器的性能是合理的。如果 `% Processor Time` 计数器和 `Processor Queue Length` 计数器都高，则应考虑添加更多处理器。增加处理器数量允许系统同时执行更多请求。

### 不运行不必要的软件

公司政策通常要求在服务器上安装病毒检查软件。服务器上也可能运行其他产品。在可能的情况下，不应在与 `SQL Server` 相同的服务器上运行不必要的软件。与维护 `Windows Server` 或 `SQL Server` 无关的外部应用程序最好放在不同的机器上。

## 网络瓶颈分析

在 `SQL Server` OLTP 生产环境中，很少会发现性能问题是由网络问题引起的。在 OLTP 环境中遇到的大多数网络问题实际上是硬件或驱动程序限制，或者是交换机或路由器的问题。这些问题大多可以通过 `Network Monitor` 工具进行最佳诊断。不过，`Performance Monitor` 也提供了收集网络活动数据的对象，如表 4-2 所示。

表 4-2

用于分析网络压力的性能监视器计数器

| 对象(实例[,实例 N]) | 计数器 | 描述 | 值 |
| --- | --- | --- | --- |
| `Network Interface`(网卡) | `Bytes Total/sec` | 网卡上传输字节的速率 | 平均值 < 网卡容量的 50%，但需与基线比较 |
| `Network Segment` | `% Net Utilization` | 网段上使用的网络带宽百分比 | 平均值 < 网络带宽的 80%，但需与基线比较 |

### 字节总数/秒

您可以使用 `Bytes Total/sec` 计数器来确定网络接口卡 (NIC) 或网络适配器的性能。`Bytes Total/sec` 计数器应报告较高的值，以表示大量成功的传输。将此值与 `Network Interface\Current Bandwidth` 性能计数器报告的值进行比较，后者反映了每个适配器的带宽。

为了给流量高峰留出余地，通常平均不应超过容量的 50%。如果该数字接近连接容量，并且处理器和内存使用率适中，那么该连接很可能存在问题。

### % 网络利用率

`% Net Utilization` 计数器表示网段上正在使用的网络带宽百分比。此计数器的阈值取决于网络类型。例如，对于以太网，当 `SQL Server` 位于共享网络集线器上时，推荐的阈值是 30%。对于位于专用全双工网络上的 `SQL Server`，即使网络使用率接近 100% 也是可接受的，但将网络利用率保持在可接受的阈值以下是有利的，以便为负载高峰留出空间。

## 注意

您必须安装 `Network Monitor Driver` 才能使用 `Network Segment` 对象计数器收集性能数据。

在 `Windows Server 2012 R2` 中，您可以从网络适配器的本地连接属性中安装 `Network Monitor Driver`。`Network Monitor Driver` 在网络适配器的网络组件网络协议列表中可用。

您还可以查看 `sys.dm_os_wait_stats` 中与网络相关的等待统计信息。但是，经常出现的一个等待是 `ASYNC_NETWORK_IO`。虽然这可能表示与网络相关的等待，但它更常见地是反映了由于编程代码不佳（未能高效地使用结果集）而导致的等待。

## 网络瓶颈解决方案

一些常见的网络瓶颈解决方案如下：

*   优化应用程序工作负载
*   添加网络适配器
*   调节和避免中断

让我们更详细地考虑这些解决方案。

### 优化应用程序工作负载

要优化数据库应用程序和数据库服务器之间的网络流量，请在应用程序中进行以下设计更改：

*   不要发送长 SQL 字符串，而是为 SQL 查询创建一个存储过程。然后，您只需通过网络发送存储过程的名称及其参数。
*   将多个数据库请求分组到一个存储过程中。这样，对于存储过程中实现的一组 SQL 查询，只需通过网络进行一次数据库请求。在谈到 `Azure SQL Database` 时，这一点变得极其重要。
*   请求小型数据集。不要请求应用程序逻辑中未使用的表列。
*   将数据密集型业务逻辑移入数据库，作为存储过程或数据库触发器，以减少网络往返次数。
*   如果数据不经常更改，尝试在应用程序上缓存信息，而不是频繁调用数据库以获取与上次调用完全相同的信息。
*   最小化网络调用，例如返回未使用的多个结果集。一个常见问题是 `SQL Server` 返回的结果集中包含每个语句的行计数。您可以在查询顶部使用 `SET NOCOUNT ON` 来禁用此功能。

**表 4-6**

**用于分析执行计划可重用性的性能监视器计数器**

| Object(Instance[,InstanceN]) | Counter |
| --- | --- |
| SQLServer:SQL Statistics | SQL Re-Compilations/sec |

存储过程的重新编译会给处理器增加开销。您希望 `SQL Re-Compilations/sec` 计数器的值尽可能接近 0，但您可能永远看不到它。如果您持续看到偏离基线测量值或剧烈波动的数值，那么您应该使用 `Extended Events` 来进一步调查正在重新编译的存储过程。一旦识别出相关的存储过程，您应该尝试分析并解决重新编译的原因。在第 `17` 章中，您将学习如何分析和解决各种导致重新编译的原因。

## 用户连接数

多个只读 SQL Server 实例可以在负载均衡环境（SQL Server 分布在多台机器上）中协同工作，以支持大量的数据库请求。在这种情况下，最好监控 `User Connections` 计数器来评估用户连接在多个 SQL Server 实例上的分布情况。`User Connections` 的值可能随应用程序的正常行为而在整个范围内变化。这正是需要基线来确定预期行为的地方。您很快将看到如何建立这个基线。

## 批处理请求/秒

此计数器是 SQL Server 上负载的一个良好指标。根据系统资源利用率和 `Batch Requests/sec` 的水平，您可以估算 SQL Server 在不出现资源瓶颈的情况下可能能够支持的用户数量。此计数器在不同负载周期下的值，有助于您理解其与数据库连接数的关系。这也有助于您理解 SQL Server 与 `Web Request/sec`（即对于使用 Microsoft Internet Information Services (IIS) 和 Active Server Pages (ASP) 的 Web 应用程序而言的 `Active Server Pages.Requests/sec`）的关系。所有这些分析都有助于您更好地理解和预测用户负载变化时的系统行为。

该计数器的值在应用程序正常行为下可能跨越很宽的范围。一个正常的基线对于确定预期行为至关重要。

## 总结

在本章中，您学习了如何收集关于 CPU、网络和 SQL Server 整体的指标。所有这些信息都为您提供了在深入尝试调优查询之前理解系统状况的能力。请记住，CPU 会受到其他资源的影响，因为它必须管理这些资源，因此一些看起来像是 CPU 问题的情况，其实更适合解释为磁盘或内存问题。网络很少成为 SQL Server 的主要瓶颈。您有多种方法可以通过性能监视器计数器观察 SQL Server 内部行为，就像观察系统其他部分一样。关于各种系统指标的讨论到此结束。接下来，您将学习如何将所有信息整合在一起以创建一个基线。
