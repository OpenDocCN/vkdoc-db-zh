# 理解 SQLite 是什么

SQLite 处于公共领域，因此任何人都可以使用。更多详情和下载源代码的链接可在[sqlite.org](https://sqlite.org)找到。

你可以将源代码编译成一个库，然后在你的应用程序中使用。尽管你可能会看到将 SQLite 称为 DBMS 的说法，但从严格意义上讲，它只是一个库。包含该库的容器（一个类、一个框架或一个完整的 DBMS）提供了更广泛的 DBMS 功能。

### SQLite 是 DBMS 吗？

在某种程度上，这个问题只有在能以某种影响你设计或实现基于 SQLite 项目的方式回答时才显得重要。然而，如果你追溯关于 SQLite 的参考资料，你会看到它通常（正确地）被称为一个库或一个数据库引擎。

从一开始，其代码就被设计为紧凑和可复用的。

（这里提及它源自美国海军是有相关性的——当你在海上航行时，所有资源都是有限的，包括电力、空间和重量。此外，在现代舰船中，技术组件必须协同工作，即使它们来自不同的供应商。这就是 SQLite 的环境。）

无论你担心的是舰船上的有限资源，还是运行 Android 或 iOS 等操作系统移动设备上的有限资源，或是像可编程信标这样的小型移动设备上的有限资源……甚至超级计算机上的有限资源（“有限”总是相对的），SQLite 在许多情况下都是一个不错的选择。

以下部分重点介绍了 SQLite 实现这一使命的一些主要特性，这些特性在你将其用于你所选择的容器（DBMS、类、框架或基础库）时都很重要。当你为移动设备开展基于 SQLite 的项目时，这些特性对你很重要。

本节讨论的特性实际上都围绕着一个事实：SQLite 是一个被设计为供单一用户处理数据库功能的库。

（在你放弃之前，请继续阅读以了解 SQLite 如何在多用户和多进程环境中表现良好。）以下是需要考虑的特性：

*   SQLite 是为单一用户设计的
*   SQLite 是自包含的
*   SQLite 支持事务并符合 ACID 标准

> **注意** 你可以在[sqlite.org](https://sqlite.org)上找到关于这些主题的更多详细信息。

## SQLite 是为单一用户设计的

SQLite 与大多数 DBMS 的最大区别之一在于，SQLite 是为单一用户设计的。大多数 DBMS 管理多个用户，包括各种安全特性，允许或阻止对特定 SQL 命令和功能的访问。对许多人来说更重要的是，DBMS 管理资源争用，以便多个用户可以*显然*地同时使用相同的数据。

*显然*是这里的关键词，因为尽管每个用户通常认为他或她拥有对数据库的独占访问权，但实际上，在幕后，DBMS 管理着并发性，以便在某些情况下，当一个用户正在更新记录时，它会锁定该记录以防止第二个用户访问它。

### 单一用户不意味着单线程

SQLite 在自己的环境中管理并发性。这意味着它可能同时运行多个线程来执行其自身任务，但这些线程是在 SQLite 环境内部管理的。它们不代表单独的用户。

#### 在多用户环境中使用 SQLite

当 SQLite 是为单一用户设计时，你怎么能拥有多个用户呢？答案很简单：你自己管理多个用户。有多种方法可以做到这一点，但一般来说，你需要将多用户管理推送到嵌入 SQLite 的应用程序、类或语言中。应用程序通常具有相互通信的能力（受安全性和平台限制）。因此，尽管 SQLite 不会管理用户 A 和用户 B 试图同时修改相同*数据*的情况，但你的应用程序可以做到这一点。

典型的并发策略包括拥有一个管理并发性的主进程，或者拥有一种机制，使得多个独立进程无需主进程即可通信。你在诸如 Dropbox 之类的应用程序以及许多基于云的应用程序中可以找到许多独立进程的例子。（对于 Dropbox 和基于云的应用程序，可能有一个进程持续在云端的某处运行，并且可能在活动的客户端上运行单独的进程。控制权是驻留在中央进程中还是分布在客户端（和主机）中，取决于具体的实现。）

因此，SQLite 完全有能力在多用户世界中运行；它只需要在自身实现了多用户功能的应用程序或其他进程内运行即可。

### SQLite 是自包含的

SQLite 在两个方面是自包含的。

*   代码本身是自包含的 ANSI-C 代码。它对 C 库的使用极少。事实上，它只使用：
    *   `memset()`
    *   `memcpy()`
    *   `memcmp()`
    *   `strcmp()`
    *   `malloc()`
    *   `free()`
    *   `realloc()`
*   数据存储本身是自包含的、可移植的且与平台无关的。

#### 自包含代码

自包含代码意味着当你将 SQLite 库包含到你的项目中时（直接包含或通过语言、类或框架包含），你就拥有了所需的一切。你不需要包含额外的库。

你不必担心版本问题，并且一旦你拥有了一个编译好的 SQLite 库，你通常可以在不依赖于组件变更的情况下重用它。

（不过请记住，在大多数情况下，SQLite 是嵌入在语言、类、框架或 DBMS 中的，因此你需要时不时更新的是容器。）由于 SQLite 处于公共领域，你不必担心许可证或许可费用。（然而，你也可以按照[sqlite.org](https://sqlite.org)上的描述获得许可证。该选项是为一些需要向管理层证明他们确实有权使用 SQLite 的用户提供的。）

请记住，SQLite 通常包含在你正在使用的类、框架或语言中，因此这个特性（在很大程度上使得这种可包含性成为可能）对你可能是不可见的。

> **注意** 如果你访问[sqlite.org](https://sqlite.org)，你会发现 SQLite 源代码的各种下载选项。有选项可以下载分段的源代码，这些源代码在最终构建中会合并（*amalgamated*）。这样做主要是为了适应那些难以一次性处理完整构建的开发环境。

#### 自包含数据

SQLite 用于数据库的数据存储被设计为自包含且跨平台的。这意味着在大多数情况下，你可以将 SQLite 文件从一个环境传输到另一个环境而不会出现问题。（这个警告是必要的，因为你可能会在特定配置中遇到问题，但对于许多（如果不是大多数）SQLite 用户来说，验证数据库是否可以移动的第一步就是实际移动它——这通常是有效的。）在其最基本（和原始）的形式中，每个 SQLite 数据库都存储在一个单一文件中。

该文件在任何平台上都是可读和可写的（受文件系统环境约束）。例如，如果你将文件放在只读磁盘上，你无法用 SQLite 写入它。（这可能听起来很牵强，但这种事情确实会发生。）

由于数据库存储在单个文件中，用户可以看到它并在需要时移动它。
