# 第一章：快速掌握数据库与 SQLite

## 关系型数据库与 SQL 的救场

你可以编写代码来实现其中任何一条规则，但这仅仅是开始。

随着时间的推移，情况会变，用户会提建议，而掌权者（无论在你的生活中他或“他们”是谁）会决定更改关于就业、汽车、车牌号码适用条件或其他任何一时兴起想到的事情的规则。

那些只花了几行代码和几分钟写成的代码需要重写。而且，尽管对此没有明确的研究，但似乎这些变更总是在产品（或版本）发布前的最后一刻发生，并且必须立即完成。

你会用 Python 吗？懂 Scala 吗？

实现某个数据规则的那一两行代码，可能是由已经不在场的人写的。即使是你自己写的，时间可能已经过去，你也许记不清当时到底想实现什么（是的，给代码加注释是个好习惯，但生活总有意外...）。

那几行代码可能是模仿自朋友的某段代码，那段代码恰好（或几乎）能实现你想要的功能。维护即使是写得最好的代码，对从最初作者到更新者、再到管理员，并最终到提出“只是微调一下”的用户或经理来说，都是一个挑战。

如今，数据库在结构上已经高度标准化——或许比信息技术世界中几乎任何其他概念都要更标准化。关系型数据库的概念最早由 IBM 圣何塞研究实验室的 Edgar Codd 于 1970 年提出，他的提议已成为现代数据库实现的基础（参见题为《大型共享数据库的关系模型》的文章，发表于 `ACM` 通信，`13(6)`，`377–387`（1970 年 6 月））。

1974 年，Donald D. Chamberlin 和 Raymond F. Boyce（同样来自 `IBM`）为 `ACM`（计算机协会）`SIGFIDET` 数据描述、访问与控制研讨会撰写了一篇题为 “SEQUEL：一种结构化英语查询语言” 的论文。

关系模型（由 Codd 提出）与 `SQL` 一起，已成为现代数据库实现的基础。`C. J. Date` 等人的著作以及 Codd 关于关系模型的原始工作，至今仍是关系数据库理论的基石。

> **注意**：`SQL` 这个名字源自 `SeQUeL`，而后者后来被证明是霍克·西德利飞机公司的商标。如今，许多人认为 `SQL` 不是一个缩写或首字母缩略词，而本身就是名称。也有人认为 `SQL` 是结构化查询语言（Structured Query Language）的首字母缩略词。

`SQL` 的基本概念不多。你可以通过想象一个包含行和列的电子表格来形象化它们。（你也可以使用数学集合论和其他概念来理解它们，但对当今许多人来说，电子表格更容易思考。）

以下是基本的 `SQL` 概念：

*   **表**。表非常像一个简单的电子表格，有行和列。（它不像可能包含多个表格的复杂电子表格。在此讨论中，请考虑简单的情况——只是一个表中的行和列。）表有时被称为*关系*，但更常用*表*。
*   **列**。列代表一个单一的数据元素，如“地址”或“姓名”。在数据库世界中，列可以称为*字段*。在编程领域，列可以称为*属性*或*特性*。
*   **行**。行代表单个观测值或一组值，每列一个值。因此，这个假想的简单电子表格中的一行可能代表一个人的数据。在数据库世界中，行可以称为*记录*或*元组*。

要从关系数据库表中检索数据，你需要运行*查询*。查询是一组逻辑指令，以某种方式操作给定的表以检索你想要的数据。查询的结果是另一个表。结果表可能是空的（如果没有数据满足查询），可能是表中的部分或全部数据，在某些复杂的查询中，可能比基本表还大。查询的结果有时被称为*视图*或*结果集*。

这些概念和术语适用于当今的大多数数据库，无论它们是用什么语言实现的，或许更重要的是，无论它们是在什么数据库引擎或数据库管理系统（`DBMS`）中实现的。

此外，数据库有时作为框架和语言的一部分实现。因此，在 `PHP` 中，你可以从 `PHP 5.1` 开始使用 `PDO`（PHP 数据对象）来访问 `SQLite`。对于 `iOS`，你可以使用 `Core Data` 框架来处理 `SQLite` 和其他数据管理器。在这些以及其他情况下，其思路都是将尽可能多的东西抽象到框架或其他封装器中，这样切换数据库时就不需要大量重写代码。

`SQLite` 实现了大部分 `SQL` 标准。例外情况在此列出：[`sqlite.org/omitted.html`](https://sqlite.org/omitted.html)。

## 探究关系型表与查询内部

在本章及接下来的几章中，我们将使用一个简单的例子来学习如何创建 `SQL` 表和查询。表 `1-1` 展示了一个可以用来演示 `SQL`、查询和关系型表基本概念的简单表格。它显示了一些人的数据及其原籍国。列名（字段）是 `PK`、`Name` 和 `Origin`。`Name` 的值是 `Cecelia`、`Leif` 和 `Charlotte`；`Origin` 的值是 `Australia`、`Iceland` 和 `United States`。

`表 1-1. SimpleTable`

| 主键 | 姓名       | 原籍         |
|------|------------|--------------|
| 1    | Cecelia    | Australia    |
| 2    | Leif       | Iceland      |
| 3    | Charlotte  | United States|

> **注意**：在关系型数据库中，列有名称，但行没有名称。在表 `1-1` 所示的简单表格中，第一列（名为 `PK`）有三个值：`1`、`2` 和 `3`。这些值恰好对应于行号，但如果它们被命名为 `Chair`、`Tree` 和 `15`，这个例子同样有效。

你可以通过使用查询从表中检索数据。如前所述，查询的结果是一个表——可能是空的，可能是原始表的部分数据，在某些情况下，甚至可能比原始表的数据更多。本节展示了一些典型的基础查询。在后面的章节中，你将看到更多关于查询的内容以及如何构建它们的细节。现在，这只是一个关于查询能是什么以及能做什么的初步体验。

## 基础查询结构

最简单的形式下，查询由三部分组成。

> **注意**：本节中展示的代码是 `SQL` 代码。按照惯例，`SQL` 中的保留字用大写字母显示，尽管在大多数实现中，`SQL` 是不区分大小写的。

**`SQL` 动作：`SELECT`**

`SQL` 中有许多动词，但 `SELECT` 用于从表中选择数据。命令以这个词开头。

**`SQL` 要选择的数据：列名列表**

你可以通过列出你想要选择的列的名称来开始构建 `SELECT` 语句。因此，要从 `Name` 列检索数据，你会这样开始 `SELECT` 语句：

```sql
SELECT Name
```

要从多个列中选择数据，你可以指定一个逗号分隔的列表，如下所示：

```sql
SELECT Name, Origin
```

你可以用星号选择所有列，如下：

```sql
SELECT *
```

**`SQL` 数据源：表名**

你可以使用以 `FROM` 开头的子句来指定数据源。下面是一个例子：

```sql
SELECT Name FROM SimpleTable
```
