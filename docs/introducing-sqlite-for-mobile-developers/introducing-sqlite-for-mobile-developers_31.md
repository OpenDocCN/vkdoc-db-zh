# 第 8 章 ■ 在 iOS 和 OS X 中使用 SQLite 与 Core Data

尽管 SQLite 与 PHP、Android 和 iOS/OS X 紧密结合，但集成策略有所不同。在某种程度上，它们形成了一个逻辑上的递进关系。PHP 将 SQLite 作为语言的扩展来支持（在撰写本文时已不再是可选组件）。当你为 SQLite 编写 PHP 代码时，当今的最佳实践是使用 PHP 数据对象（`PDO`）类。

然而，你代码的其他部分很可能是传统的过程式代码：并没有假设你会全程编写面向对象的代码。

Android 在创建通用类（`android.database`）的理念上采取了稍有不同的方法。在 Android 的情况下，一个特定的子类数据库提供 SQLite 支持（`android.database.sqlite`），同时在`android.database`中提供通用的数据管理支持。

可以合理地认为，Android 中的 SQLite 实现比 PHP 更具面向对象性。（这是一个概括，并非旨在忽视或煽动争论。）尽管实现方式存在这些差异，但你在本书和其他地方看到的 SQLite 代码在 PHP 和 Android 中许多情况下都是可见的。

你将在 PHP 和 Android 应用的源代码中找到熟悉的 SQLite 命令——如`SELECT`、`DELETE`、`INSERT INTO`、`DELETE FROM`等命令。

在 iOS 和 OS X 的世界里，面向对象的方法走得更远。在正常情况下，作为开发人员，你甚至看不到 SQLite 代码。就像`PDO`和`android.database`一样，存在针对各种数据库定制的通用类，但 SQLite 代码被放置在这些通用类内部，很少对你可见。这意味着你不需要编写诸如`SELECT`、`DELETE`之类的语句或命令。相反，你编写的是与通用类及其为你的应用定制的子类交互的代码。

此外，与`PDO`和`android.database`中的实现相比，iOS 和 OS X 中的实现有一个非常关键的区别。这个区别与关系有关，它们在概念上与其他任何 SQL 环境中的关系完全相同，但实现方式却大不相同，你将在本章中看到。

最后，有一个关键点对 Core Data 至关重要，但不以同样的方式适用于其他环境。`PDO`和`android.database`封装特定的数据库并提供对它们的访问。Core Data 可以与多种数据库协作，但其数据库访问是通过 Core Data 本身进行的：底层数据库并不暴露。有些人认为 Core Data 本身就是一个数据库，但它不是。从技术上讲，它是一个对象图和持久化框架。它使用一个数据库（或者，在 SQLite 的情况下，一个数据库库），但它本身并不是一个数据库。它的职责是隐藏数据库，以便你可以使用 Core Data 基于其所使用的数据库中的数据管理的对象进行工作。


■ **注意** `Core Data` 能够与包括 `SQLite` 在内的多种数据库协同工作，但通过将这些数据库封装在其自身的框架类中，它为不同的数据库类型提供了一致性。因此，尽管在 `iOS` 和 `OS X` 上最常见 `Core Data` 用法中，底层的数据库是 `SQLite`，你并不能简单地将一个现有的 `SQLite` 数据库直接塞进 `Core Data`。如果你有一个现有的数据库（`SQLite` 或任何其他类型）并想将其迁移到 `Core Data`，最简单的方法是将其卸载，然后重新加载到你创建的 `Core Data` 模型中。在很多情况下，你可以使用实用工具来完成卸载；而重新加载可能需要一小段代码来与 `Core Data` 交互。在任何情况下，拥有卸载和重新加载任何类型数据库的能力都是一种最佳实践。这不仅在从一种数据库管理器迁移到另一种时很方便，对于调试、审计和其他需求也同样有用。

## Core Data 框架简介

与 `PDO` 和 `android.database` （非常）大致类似的通用代码就是 `Core Data` 框架。它由一系列类组成——本章将描述关键类以及用于数据模型的图形编辑器。

### Core Data 的起源

Core Data 是一个框架——一组协同工作、可以一起安装到你应用程序中的类。它与 `Cocoa`（用于 `OS X` 的用户界面 (`UI`) 框架）和 `Cocoa Touch`（用于 `iPad`、`iPod touch` 和 `iPhone` 的 `UI` 框架）配合使用。它也用于 `tvOS` 和 `watchOS`。它为标准数据库提供了一个完全面向对象的接口。它最初于 1994 年在 `Next` 公司编写为企业对象框架 (`EOF`)，并支持 `Oracle`、`Sybase`、`Informix` 和兼容 `ODBC` 的服务器以及 `OpenBase Lite`。如今，它在所有平台上都支持内置的 `SQLite` 库，在 `Mac` 上还支持 `XML`。

## 第 8 章 在 iOS 和 OS X 中结合 SQLite 使用 Core Data

Core Data 源自 `EOF`，但在许多方面有很大不同。仍然保留的主要相似之处之一是用于描述关系数据的图形用户界面。

如今，它作为 `Xcode` 的一部分，体现为 `数据模型检查器` 和 `Xcode Core Data 模型编辑器`。

*   两者都是完全面向对象的，尽管你可以编写代码来管理它们，但存储的对象及其属性在 `Xcode Core Data 模型编辑器` 中以图形方式表示。如果你确实编写代码，那是标准的 `Objective-C` 或 `Swift` 代码，而不是 `SQL` 或 `SQLite`。
*   除了存储的对象和属性本身之外，用于设计数据、其对象及其关系的图形用户界面直接源自 `EOF` 工具。你可以编写代码来管理数据及其对象，但该代码是 `Swift` 或 `Objective-C`，而不是 `SQL` 或 `SQLite`。

也许 `EOF` 和 `Core Data` 最重要的方面是非常相关的。

作为开发者，`Core Data` 有三个重要的方面。

*   用于构建和维护包括关系在内的数据模型的图形用户界面。这就是 `Xcode Core Data 模型编辑器`。它在 `Xcode` 内部运行。
*   使 `Core Data` 为你工作的运行时对象。它们统称为 `Core Data` 栈 (`Core Data stack`)。
*   你经常与 `Core Data` 一起使用的其他对象。

本章讨论第一个方面。第 9 章将讨论第二个和第三个方面。

## 使用 Core Data 模型编辑器

从一个 `Xcode` 项目开始，可以是基于内置的 `Xcode` 模板，也可以是基于你自己的项目。许多内置模板在设置项目时都有一个使用 `Core Data` 的选项。如果你选择了这个选项，你的项目中就已经有一个数据模型了。
