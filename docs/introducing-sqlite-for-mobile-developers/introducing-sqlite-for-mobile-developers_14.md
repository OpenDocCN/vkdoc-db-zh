# 快速掌握数据库与 SQLite

如果你问人们与 `数据库` 相关联的词汇，`庞大` 很可能是其中的回应之一。数据库处理海量数据，这一点众所周知。

有多大？这取决于你计算的是数据库中的条目数量，还是条目本身的规模。截至撰写本文时，美国国会图书馆在 858 英里长的书架上拥有 1.6 亿件藏品。其基于数据库的目录可在 [www.loc.gov/about/fascinating-facts/](http://www.loc.gov/about/fascinating-facts/) 在线查阅。对数据库感兴趣的人们都认同，美国国会图书馆的目录远非世界上最大的数据库。（很可能，最大的数据库并不为公众所见，因为它们包含机密的商业和政府信息。）

本章旨在介绍或回顾当今的数据库及其使用方式。如果你过去使用过数据库，那么变化已经很大；如果你过去用得不多，本章将为你提供一个快速概览。

## 超越 `庞大`

除了 `庞大`，你应该开始将数据库视为 `结构化` 和 `有组织` 的。

事实上，作为一名移动开发者，数据库的结构化和有组织性方面远比其规模重要。自 20 世纪 90 年代初网络早期以来，网络浏览器就一直使用数据库来存储和组织其数据。你可能会问，是什么数据？你认为浏览器是如何存储你访问过的网站的密码的？浏览器又是如何将网页存储在缓存中，以便在可能的情况下无需新的网络访问即可检索的？还有你对默认字体大小的偏好设置呢？或者你可以随时更改的文件下载文件夹？甚至在许多浏览器上，对某些用户或用户类型的限制？这些只是简单的例子，但其原理同样适用于当今最大（和最小）的数据库。

## 数据库是结构化和有组织的

所有这些例子都可以用数据库实现，因为所有这些条目都需要结构化和有组织。其中一些条目可能变成大量的数据（特别是如果你不选择偶尔清除旧信息的选项——这个选项本身也可以存储在偏好数据库中），但列出的一些偏好设置是非常少量的数据：即使下载文件夹的名称经过完全修改，128 个字符也足以存储这个数据元素。此外，大多数浏览器只存储你偏好的下载文件夹名称，而不存储下载文件夹的历史记录（上周偏好的文件夹、上个月使用的文件夹等）。

在思考数据库时，“庞大”通常相关，但很多时候它会误导人。最重要的是结构和组织。

而对于移动开发者来说，这两点至关重要。

正因为移动设备需要在资源受限（例如，有限的存储空间和电池电量）的环境中运行，所以对存储数据的结构化和组织就显得尤为重要。数据管理系统内置了各种优化措施，这些措施对大小数据库都很重要。例如，许多数据库可以存储字符或字符串数据。这类数据通常变化很大。即使数据库设计者指定一个文本字段可以包含 50、500 或 50,000 个字符，幕后通常会丢弃尾随空格。用户（甚至开发者）可能永远不会知道这一点，但这使得整个数据库的运行效率更高。（此类功能在幕后运行，在许多情况下，数据库设计者可以访问并启用或禁用它们。）

## 数据库是智能的

能够优化存储以处理文本字段中字符的存在或缺失，是一个强大的幕后工具，但数据库的智能是内建其中的。

在当今的大多数数据库引擎中，设计者可以指定字段的许多属性，例如以下内容：

*   **名称**。内部使用的名称通常与用户界面上显示的名称大不相同。
*   **类型**。你可以指定某个数据库字段必须是整数、字符串或数据库支持的任何其他类型。
*   **可选性**。有些字段是可选的，你可以在设置数据库时指定。（许多人拥有带牌照的汽车；许多其他人则没有车。）
*   **默认值**。可以设置数据库提供默认值——可以是简单值，也可以是根据数据库中其他数据计算得出的值。
*   **基数**。有时，数据库的某部分有多个值。（继续上面的例子，有些人没有车，有些人拥有一辆车，而另一些人则拥有几辆车。此外，有些人拥有一辆车和一辆自行车……你明白这个意思了。）
*   **值范围**。当数据库设计者指定数据库结构时，可以为每个字段设置值范围。驾照标识符若包含字母和数字以外的字符（取决于地区）可能不被允许。
*   **值关系**。在数据库设计中，你可以超越值范围。你可以创建属性和值的组合，以便例如，数据库可以强制执行一条规则：一名员工可以是已婚（与一人或无人），但如果员工已婚，其配偶的雇主不能与该员工的雇主相同。数据库中的这些限制反映了政策，但也可能存在例外（例如主管的批准），并且数据库可以被设计为支持这种情况。

这些只是数据库可以设置来强制执行逻辑规则的几种方式。
