`SQLite` 数据库存储在单个文件中的事实，导致了 `SQLite` 的一种常见用途：在用户看来只是一个简单文件的内部，实现结构和数据库功能。

## SQLite 支持事务并符合 ACID

`Transaction`（事务）在数据库领域有一个非常具体的含义，超越了其在英语中的通常含义。它指的是作为一个整体一起执行（通常是顺序执行，但不一定）的一系列步骤。这些步骤合在一起构成一个事务，重要的是整个事务要么成功，要么失败。如果任何步骤失败，整个事务就失败。事务失败的结果是，数据库被 `回滚` 到事务开始前的状态。

当一个数据库支持事务时，事务的任何部分失败都意味着整个事务失败，数据库被设置为事务开始前的状态。所有这些都发生在一个符合 `ACID` 的数据库中，例如 `SQLite`。

正如你在第 [1](http://dx.doi.org/10.1007/978-1-4842-1766-5_1) 章所见，使用数据库时，有很多代码你不需要编写。

#### ACID

`ACID` 是当今评判大多数数据库是否支持事务的标准。`ACID` 是一个首字母缩写词。

-   *原子性*。这是事务的"全有或全无"特性。它作为一个整体成功或失败，永远不可能部分成功。
-   *一致性*。在处理事务时，数据库将从一个有效状态开始，并以一个有效状态结束。换句话说，事务不会因为事务的结果而违反数据库规则（如属性的可选性），但在事务过程中可能会违反这些规则。
-   *隔离性*。这意味着事务可以顺序或并发地运行。它们不会相互干扰。（`隔离性` 要求数据库实现一种管理并发的方法，以便操作可以是顺序或并发的。）
-   *持久性*。这意味着，一旦完成，事务就持续存在。实际上，这通常意味着它被提交到磁盘。

在 `SQLite` 中（与许多其他 `SQL` 实现一样），事务由一系列 `SQL` 语句组成，并用以下语句括起来：

```
BEGIN TRANSACTION
```

和

```
END TRANSACTION
```

关于事务的更多内容见第 [4](http://dx.doi.org/10.1007/978-1-4842-1766-5_4) 章。你将了解如何定义事务以及如何指定失败事务回滚到的点。在该章中，你还会看到关于 `WAL` 如何实现原子性和持久性的更详细描述。

#### 移动设备上的并发和 ACID 事务

你可能认为，当你为手机等移动设备编写程序时，只有单一用户，因此关于事务和 `ACID` 合规性的讨论不适用。然而，并发问题在移动设备上或许比在其他情况下更值得关注。

为什么你必须考虑移动设备上的事务及其失败？原因在于你在移动设备上执行的许多最频繁的任务都涉及网络访问。正如任何曾在突然高速驶入隧道的火车上尝试打电话的人所知，网络可用性在移动设备上可能会突然消失。
