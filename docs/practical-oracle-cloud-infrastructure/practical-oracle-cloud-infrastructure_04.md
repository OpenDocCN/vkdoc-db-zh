# 2. 构建您的第一个云应用程序

本章展示了如何创建一个简单的云基础架构，并在负载均衡器后运行两个自定义 REST API 实例。本章中展示的 API 实现将作为一个示例。主要目标是指导您从头开始构建您的第一个基于云的解决方案。随后，在本书的课程中，您还将学习如何配置自动扩展的实例池、将应用程序部署到**Oracle Kubernetes Engine**的**Docker**容器中，以及在**Oracle Functions**上执行无服务器函数。然而，在此阶段，让我们专注于基础，使用标准的虚拟机。

有多种方式可以与**Oracle Cloud Infrastructure**的云管理平面进行通信。在本章中，我们使用`OCI 控制台`，这是一个用于管理`OCI 资源`的 Web 浏览器界面。此方法仅适用于希望演示特定功能或尝试对您而言是新功能的情况。在现实世界中，您要么使用动态的、过程化的命令行界面（`CLI`）来自动化整个供给过程，要么使用声明式的`Terraform`驱动的基础架构描述符来管理基础架构即代码。自动化是第三章的主要主题。此外，本书所有剩余章节中的绝大多数演练都展示了云自动化的实际应用。

## 实现 API 服务器与云基础设施设计

是时候仔细看看实现了。我选择使用 Flask 来实现 API 服务器，这是一个面向 Web 的 Python 微框架。我确信你已经听说过 Python 编程语言。它于 20 世纪 90 年代初推出，尤其是在过去十年中，其受欢迎程度迅速增长。Python 被认为是一种通用语言，广泛应用于 Web 开发、数据科学、机器学习、云生态系统，甚至物联网（IoT）等各个领域。它被归类为一种多范式、解释型的语言，具有动态类型和称为*垃圾回收*（`garbage collection`）的自动内存管理。支持多种编程范式意味着你可以选择自己喜欢的编码方式，无论是过程式、面向对象还是函数式。解释型语言不需要在执行前预编译成机器代码，而是直接在解释器上运行。这简化了交付流程，但会带来性能下降。动态类型和自动内存管理提高了编程效率，使语言更易于使用。另一方面，Flask 是一个 Web 开发微框架，主要侧重于处理 HTTP 请求并发送正确渲染的 HTTP 响应。它完全兼容 WSGI。WSGI 在 PEP3333 中定义，代表 Web 服务网关接口（Web Service Gateway Interface），这是 Python 生态系统的一个标准，指定了 Web 服务器和 Web 框架之间的接口。为方便起见，Flask 自带一个嵌入式 Web 服务器，非常适合开发目的。

清单 2-2 展示了一个 UUID 服务的实现。你不需要对这段代码做任何事。所有事情都将在云实例首次启动时自动发生。怎么做？继续阅读即可。

> **提示**
>
> 尽管你不需要对这段代码做任何事，但它仍可在本书的 Git 仓库中找到，路径为 `chapter02/uuid-service/app.py`。

```python
import flask
import uuid
import os
import socket
import logging
app = flask.Flask(__name__)
@app.route('/identifiers', methods=['GET'])
def identifiers():
    generator_name = os.getenv('UUID_GENERATOR_NAME', socket.gethostname())
    generator_uuid = uuid.uuid4()
    app.logger.info('Generator: [%s] UUID: [%s]', generator_name, generator_uuid)
    rsp = flask.jsonify(uuid=generator_uuid, generator=generator_name)
    rsp.status_code = 200
    rsp.headers['Content-Type'] = 'application/json'
    return rsp
@app.before_first_request
def setup_logging():
    logging.getLogger('werkzeug').disabled = True
    app.logger.removeHandler(flask.logging.default_handler)
    handler = logging.StreamHandler()
    handler.setFormatter(logging.Formatter('%(asctime)s - %(levelname)s - %(message)s'))
    app.logger.addHandler(handler)
    app.logger.setLevel(logging.INFO)
```
*清单 2-2 UUID 服务实现*

为了将重点放在本书的主题上，我将避免逐行解释实现细节。你应该理解的是，这个应用程序将所有通过 `/identifiers` URL 发送的 GET 请求路由到 `identifiers()` 处理函数。该函数通过组合一个新生成的 UUID 和 `generator` 的名称来准备响应。后者被设置为主机名，除非存在 `UUID_GENERATOR_NAME` 环境变量。响应以 JSON 格式发送，如下所示：

```json
{
  "generator": "uuid1",
  "uuid": "e22b5858-e70f-46ab-bd16-0da3d55aca73"
}
```

### API 服务器实现

#### Cloud Infrastructure

根据我们在上一节简要讨论的概念设计，我们知道将需要两个位于两个不同可用性域（availability domain）中的计算实例，它们位于一个高可用负载均衡器后面。负载均衡器将使用一个公共 IP，以便可以从互联网访问。我们将创建一些其他辅助基础设施资源，以确保云设计的完整性。

然而，事情要分轻重缓首。我们首先设计资源所在虚拟网络的布局。*虚拟云网络*（`VCN`）可以被视为一个虚拟的、私有的网络，它提供一个连续的 IPv4 地址块。建议选择通常被认为是私有的 IPv4 地址块。RFC1918 定义了这些地址。如果你想遵循此准则，请从以下地址范围中选择任何子集：

*   `192.168.0.0/16`
*   `172.16.0.0/12`
*   `10.0.0.0/8`

对于我们的云解决方案，我们将使用这个范围：`192.168.1.0/24`。

一个 VCN 被细分为一个或多个*子网*。属于同一个 VCN 的子网的 IP 地址范围必须在父 VCN 的 IP 地址范围内，并且不能与其他子网重叠。你在 Oracle Cloud Infrastructure 中运行的每个计算实例都将附加到特定的 VCN 子网。这是通过一个称为 `vNIC` 的虚拟网络接口卡完成的。

在这个阶段，我们对云网络布局有了相当清晰的构想。现在，让我们对计算实例做几个选择。

*`规格`* 是基于给定配置文件（`shape`）的计算实例所提供的硬件资源（CPU 核心、可用内存）的配置文件。Oracle Cloud Infrastructure 提供两个系列的计算实例配置。

*   **裸机主机**（`Bare-metal hosts`）：专用、单租户硬件，无管理程序（`hypervisor`）
*   **虚拟机**（`Virtual machines`）：通用目的、多租户硬件，有管理程序（`hypervisor`）


一些软件，如即时通讯平台、企业应用服务器或数据库引擎，据报道在专用的物理硬件上运行时更为稳定高效，无需涉及管理程序（hypervisor）虚拟化。你可以将它们托管在数量更少、性能更强的裸机主机上。而在所有其他通用用例中，你可能会使用更多虚拟机来利用水平扩展。

注意

裸机计算形态家族深深植根于 Oracle Cloud Infrastructure（OCI）的起源。OCI 实际上源自一个名为 Oracle Bare Metal Cloud Service、面向裸机的云服务而演变而来。

在此，我仅列举几个在撰写本文时提供的、随机选择的形态：

*   `VM.Standard2.1` 是最小的标准虚拟机形态，配备 1 个 `OCPU` 和 15GB 内存。它基于 X7 平台，这基本上意味着它使用的是 Intel Xeon Skylake 处理器。
*   `VM.Standard2.8` 是另一个标准虚拟机形态的例子，提供 8 个 `OCPU` 和 120GB 内存。
*   `BM.Standard2.52` 是一个标准的裸机形态，利用 52 个 `OCPU` 和 768GB 内存。
*   `BM.Standard.E2.64` 是一个使用 AMD EPYC 处理器的裸机形态，配备 64 个 `OCPU` 和 512GB 内存。
*   `BM.DenseIO2.52` 是一个专为大数据项目中存在的 I/O 密集型工作负载设计的裸机形态。它配备了本地附加的 SSD NVMe 驱动器，提供 51.2TB 的存储容量。
*   `BM.GPU3.8` 是一个旨在为机器学习流程和高性能计算（HPC）软件提供动力的裸机形态。除了 52 个 `OCPU` 和 768GB 内存外，其真正能力来自于八个 Nvidia Tesla V100 GPU。

在上面的列表中，我使用了一个新的缩写。一个 `Oracle 计算单元` (`OCPU`) 是一个逻辑术语，也是计算相关计费的最重要度量单位。你通常按 1 `OCPU`/小时付费。换句话说，一个运行 8 个 `OCPU` 的实例一小时所产生的计算相关成本，与一个仅运行 1 个 `OCPU` 的实例运行八小时的成本相同。

对于基于 Intel 的形态，单个 `OCPU` 对应于一个物理 Intel Xeon 核心。由于超线程技术，操作系统实际上为每个物理 CPU 核心看到两个虚拟核心。总结一下，一个基于 Intel 的 `OCPU` 意味着一个物理 CPU 核心和两个近乎并行执行线程的能力，有时也称为 vCPU。

你可以很容易地猜到，我们关于计算实例的第一个选择是为将托管 UUID 服务应用程序的节点选择一个形态。我们将使用可用的最小虚拟机：`VM.Standard2.1`。

一个 `镜像`（image）是一个模板，它指定了操作系统以及在实例启动期间必须预安装的任何附加软件。一个 `镜像` 用于初始化计算实例使用的新启动卷。启动卷与其计算实例保持关联，直到终止。启动卷可以看作是实例使用的主要存储，主要用于其操作系统和软件。

Oracle 提供的镜像会定期更新并获得新版本。然而，一旦你启动了一个实例，安排定期系统更新就成为你的责任。

在撰写本文时，Oracle Cloud Infrastructure 提供四种类型的基础操作系统镜像，包含以下操作系统：

*   Oracle Linux
*   CentOS
*   Ubuntu
*   Windows Server

在你的日常工作中，你最终可能会创建许多自定义镜像，这些镜像在所选基础镜像上添加了所需的软件、服务或计划脚本。也可以导入并非基于可用基础镜像构建的自定义镜像。在这种情况下，你将在虚拟机形态上使用准虚拟化（paravirtualization）或模拟（emulation）。然而，创建自定义镜像最简单的方法是依赖 Oracle 提供的基础镜像。

那么，再做一个决定如何？我们将使用最新的 Oracle Linux 基础镜像：`Oracle Linux 7.6`。你可能会在未来使用 `Oracle Linux 7` 的较新次版本。

#### 服务限额

你可能想知道，如果太多租户（云客户）在同一区域同时启动大量最强大的裸机形态会发生什么。即使在最大的数据中心中，这些机器的数量也肯定是有限的，不是吗？嗯，当然，确实是有限的。这正是服务限额（也称为 `配额`）这个巧妙想法的用武之地。

每个云账户都有一组服务限额，定义了允许同时运行多少何种类型的资源。如果你发现当前的限额无法满足你的需求，只需提交一个服务请求，就可以申请提高限额。这有助于 Oracle 规划和优化数据中心容量。简单而明智，不是吗？

提示

你可以在 OCI 控制台中随时验证你的使用情况与当前云账户的服务限额。只需转到 `菜单 ➤ 治理 ➤ 限额、配额和使用情况`。

在启动新的云项目之前，验证并酌情增加服务限额，应包含在你规划阶段的检查清单中。

### 安全列表

我们已经处理了虚拟网络、虚拟路由器和虚拟路由表。现在是时候添加一些安全措施了。每个 VCN 子网都必须至少有一个 `security list`（安全列表），它是一组安全规则的集合。你可以将这些规则视为虚拟防火墙的附加层，独立于你在每个单独计算实例上控制的操作系统防火墙。在请求到达计算实例的虚拟网络接口之前，会强制执行安全规则。

根据流量方向，有两种类型的安全规则：
*   `Ingress`（入站规则），验证进入 VCN 的入站流量。
*   `Egress`（出站规则），验证从 VCN 出去的出站流量。

VCN 附带一个默认的 `security list` 资源，其中包含一些基本规则，允许入站 SSH 和基本的 ICMP 流量以及所有出站流量。我们仍然需要添加一条规则，允许在 UUID 服务端口 (`5000`) 上的入站流量。以下是使用 OCI 控制台向现有安全列表添加新安全规则的方法：

1.  转到 Menu ➤ Networking ➤ Virtual Cloud Networks。
2.  确保选中 `Sandbox` 隔间。
3.  点击你的 VCN 名称。
4.  在 Resources 菜单中点击 `Security Lists`（安全列表）。
5.  点击默认安全列表的名称。
6.  在 Resources 菜单中选择 `Ingress Rules`（入站规则）。
7.  点击 `Add Ingress Rule`（添加入站规则）。
8.  提供所需的值：
    1.  在 Source CIDR 中填入 `0.0.0.0/0`。
    2.  在 Destination Port Range 中填入 `5000`。
9.  点击 `Add Ingress Rules`（添加入站规则）。

图 2-14 展示了我们正在添加的新入站安全规则。

![../images/478313_1_En_2_Chapter/478313_1_En_2_Fig14_HTML.jpg](img/478313_1_En_2_Fig14_HTML.jpg)
图 2-14

你可能想知道 Stateless 标签旁边的小复选框到底是什么意思。我们留空了这个复选框。结果，我们添加的新入站规则是一个 `stateful rule`（有状态规则）。每次通过这条有状态的入站规则接受端口 `5000` 上的入站流量（请求）时，相应的出站流量（回复）会自动放行，无需强制执行其他出站规则。如果你决定使用 `stateless rule`（无状态规则），你将还需要在相反方向创建另一条无状态规则来验证相关流量。

有状态规则听起来像是完美的解决方案。那么为什么我们仍然要使用无状态规则呢？嗯，有状态规则需要连接跟踪，这会消耗资源并引入一些延迟。此外，每个计算实例在单个时刻可跟踪的连接数是有限制的。这就是黄金法则所说：如果你能够将你的安全规则设计为无状态规则，那就这样做。

### 子网

一个 VCN 被划分为一个或多个 `subnets`（子网）。我们将计算实例，或者更准确地说，它们的 `vNICs`（虚拟网络接口），附加到子网。你可能还记得，在定义新的 VCN 时，我们没有对可用性域做任何选择。而子网是你可以在你选择的某一个可用性域中明确创建的。所有在特定子网中创建的计算实例最终都必须在创建该子网的可用性域中运行。或者，也可以创建跨越给定区域中所有可用性域的子网。这种子网让你能够以更灵活的方式启动实例，推迟为实例选择可用性域。你将在第 6 章中了解更多相关信息。目前，我们将坚持使用绑定到单个可用性域的子网。以下是使用 OCI 控制台创建新的 VCN 子网的方法：

1.  转到 Menu ➤ Networking ➤ Virtual Cloud Networks。
2.  确保选中 `Sandbox` 隔间。
3.  点击你的 VCN 名称。
4.  在 Resources 菜单中点击 `Subnets`（子网）。
5.  点击 `Create Subnet`（创建子网）。
6.  提供所需的值：
    1.  填入 `a-net` 作为显示名称。
    2.  选择 `Availability Domain-specific Subnet Type`（可用性域特定子网类型）。
    3.  在你的区域中选择第一个可用性域（后缀为 AD-1）。
    4.  填入 `192.168.1.0/28` 作为 CIDR 块。
    5.  确保选择了默认路由表。
    6.  选择 `Public Subnet`（公共子网）选项。
    7.  勾选 `Use DNS Hostnames in this subnet`（在此子网中使用 DNS 主机名）。
    8.  使用简短的单字符标签 `a` 作为子网的 DNS 标签。
    9.  确保选择了默认 DHCP Options 设置。
    10. 确保选择了默认安全列表。
7.  点击 `Create Subnet`（创建子网）。

有两种类型的子网：
*   `A public subnet`（公共子网）允许你为在此子网中配置的计算实例分配公共 IP 地址。你仍然可以在公共子网中创建没有公共 IP 地址的计算实例。

我们希望我们的 Oracle Linux 计算实例在机器启动并运行后，能够立即服务于 UUID 服务客户端。我们不希望执行任何手动安装或配置。我们将利用基础操作系统镜像中已预装在计算实例上的 `cloud-init` 包。`cloud-init` 的作用是在云实例首次启动时执行早期初始化。您的角色是在实例创建期间，以一种支持的格式上传一个 `用户数据` 文件。清单 2-3 展示了 `cloud-config` YAML 格式的 `cloud-init` 用户数据。您可以在 `git/chapter02/cloud-init` 目录中找到 `cloud-config` 文件。

```yaml
#cloud-config
yum_repos:
  ol7_developer_EPEL:
    name: Oracle Linux $releasever Developement Packages ($basearch)
    baseurl: http://yum.oracle.com/repo/OracleLinux/OL7/developer_EPEL/$basearch/
    enabled: true
    gpgcheck: true
    gpgkey: file:///etc/pki/rpm-gpg/RPM-GPG-KEY-oracle
package_upgrade: true
packages:
  - tree
  - wget
  - python36
  - python36-pip
write_files:
  - content: |
      [Unit]
      Description = Launching UUID Service API
      After = network.target
      [Service]
      Environment=FLASK_APP=/home/opc/uuidservice/app.py
      Environment=LC_ALL=en_US.utf8
      Environment=LANG=en_US.utf8
      ExecStart = /home/opc/uuidservice/venv/bin/flask run --host=0.0.0.0
      User = opc
      [Install]
      WantedBy = multi-user.target
    path: /etc/systemd/system/uuidservice.service
runcmd:
  - [ "mkdir", "-p", "/home/opc/uuidservice/venv" ]
  - [ "chown", "opc:opc", "/home/opc/uuidservice" ]
  - [ "python3", "-m", "venv", "/home/opc/uuidservice/venv" ]
  - [ "/home/opc/uuidservice/venv/bin/python3", "-m", "pip", "install", "--upgrade", "pip" ]
  - [ "/home/opc/uuidservice/venv/bin/python3", "-m", "pip", "install", "flask" ]
  - [ "wget", "-qO", "/home/opc/uuidservice/app.py", "https://raw.githubusercontent.com/mtjakobczyk/oci-book/master/chapter02/uuid-service/app.py " ]
  - [ "firewall-offline-cmd", "--add-port=5000/tcp" ]
  - [ "systemctl", "restart", "firewalld" ]
  - [ "ln", "-s", "/etc/systemd/system/uuidservice.service", "/etc/systemd/system/multi-user.target.wants/uuidservice.service" ]
  - [ "systemctl", "enable", "uuidservice.service" ]
  - [ "systemctl", "start", "uuidservice.service" ]
final_message: "UUID Service API node is running, after $UPTIME seconds"
```

清单 2-3： `vm.config.yaml`

稍后，您将把这个文件上传到 OCI 控制台实例创建向导。首先，让我们了解一下它的作用。如果您仔细查看清单 2-3，会发现它由六个命名节组成。

*   `yum_repos` 定义一个将要添加到 Yum 的新仓库。
*   `package_upgrade` 触发系统范围的包更新。
*   `packages` 列出了将在机器上安装的新包。
*   `write_files` 在给定路径使用提供的内容创建一个新文件。
*   `runcmd` 按提供的顺序执行命令。
*   `final_message` 在处理完成时打印一条消息。

但是，我们的云配置具体做了什么呢？首先，它将 Oracle Linux EPEL 仓库添加到 Yum。该仓库包含我们运行基于 Python 的 UUID 服务所需的 Python 3。`cloud-init` 将执行系统范围的包升级，并不仅安装 Python 3，还安装 `wget` 和 `tree` 实用程序。另一个有趣的部分是 `write_files` 节，在我们的案例中，它创建了一个新的 systemd 单元文件。文件内容的创建方式使得 UUID 服务能够作为 systemd 服务运行。通过这种方式，我们能够遵循一种在 Linux 中将应用程序作为操作系统服务运行的标准化方法。最后但同样重要的是，`runcmd` 节运行一系列命令。首先，为新的 Python 虚拟环境创建目录，然后使用 Python 3 的 `venv` 模块生成虚拟环境本身。接下来，在虚拟环境内部，我们升级 Python 包安装器 (`pip`) 并用它来安装 Flask。随后的步骤使用 `wget` 实用程序从本书相关的 GitHub 仓库下载 UUID 服务代码，并在操作系统防火墙中开放 5000 端口。最后三个命令有效地链接、启用并启动了 systemd 服务。这样，UUID 服务就作为一个操作系统服务运行，并在每次重启时自动启动。

您可能想知道为什么命令被指定为数组。`cloud-init` 有两种运行命令的方式。如果命令像这样包装在数组中，它将通过操作系统调用直接作为一个新进程执行：

```yaml
runcmd:
  - [ "mkdir", "-p", "/home/opc/uuidservice/venv" ]
```


如果简单地写成这样的字符串，命令会先被写入一个文件，然后传递给操作系统 shell 进行处理：

```
runcmd:
- mkdir -p /home/opc/uuidservice/venv
```

虽然从性能角度来看，前一种方式可能被认为稍快一些，但只有后一种方式允许你使用管道组合多个命令，像这样：

```
runcmd:
- 'echo $(date) | tee /home/opc/datemarker'
```

在同一个 `runcmd` 部分混合使用这两种风格是可以的。

### 横向扩展

*横向扩展*一个应用程序意味着添加更多通常是相同类型的实例。这是描述水平扩展的另一个术语。你想要扩展应用程序的原因有两个：

*   提高处理吞吐能力
*   增加容错能力

单个实例永远不足以提供可靠的云 API。然而，仍然可能出问题，摧毁你孤立的应用程序节点。如果你有多个执行相同职责的节点，并且分布在各个数据中心，那么即使单个数据中心发生故障，你也可能永远不会经历完全停机。此外，如果你有更多相同类型的实例，就可以并行处理更多的请求。事情就这么简单。

### 负载均衡器

负载均衡器为活动服务器或工作节点提供单一入口点和流量分发。我们将使用一个面向互联网的，即所谓的公共负载均衡器，它使用一个公共浮动 IP 来支持故障转移机制。这意味着您需要启动一对负载均衡器节点，每个节点位于不同的可用性域中。在任何给定时间，只有一个 LB 节点处于活动状态，并将流量转发到 UUID 服务 API 节点。第二个 LB 节点保持在备用模式。如果活动的 LB 节点发生故障，备用的 LB 节点将检测到此情况，并宣布自己为活动节点和当前 IP 地址持有者。

为避免不必要的成本或试用额度的消耗，无论它们多么低廉，让我们终止本章创建的资源。某些资源依赖于其他资源；因此，在删除时遵循所需的顺序很重要。在后续章节中这不会成为问题，因为在许多情况下，自动化将一次性为我们完成操作。然而，在本章中，我们在 `OCI Console` 中手动完成所有操作，因此必须遵循以下步骤：

1.  转到菜单 ➤ 网络 ➤ 负载均衡器。
2.  确保选择了 `Sandbox` compartment。
3.  点击三点菜单，点击“终止”，并等待资源被移除。
