# 5. 项目与按钮

本书迄今为止一直专注于应用程序如何向用户显示信息。在本章及下一章中，你将研究那些允许用户向应用程序发送回信息的组件。这些组件，即 `项目`（items）和 `按钮`（buttons），被称为 `窗体控件`（form controls），因为它们通常出现在网页表单中。
APEX 支持多种类型的项目，它们的外观和行为差异很大。然而，它们都可以被理解为几个基本概念的变体。本章的主要目的是探讨这些概念，并研究它们如何在不同的项目类型中得以体现。

## 面包屑区域

如果你想让一个页面显示面包屑对象，你必须为其创建一个区域。`Breadcrumb` 区域类型就是为此目的而设计的。虽然你可以将面包屑区域放置在页面上的任何位置，但它自然应该放在 `面包屑栏`（Breadcrumb Bar）位置，该位置位于导航栏正下方。通常，你需要遵循以下步骤向页面添加面包屑区域。

*   向页面添加一个新区域。
*   将区域的类型设置为 `Breadcrumb`。
*   将区域的源设置为 `Breadcrumb` 对象。
*   将该区域分配到 `面包屑栏`（Breadcrumb Bar）位置。
*   将区域模板设置为 `带属性的空白`（Blank with Attributes），这样它将在没有边框或标题的情况下显示。
*   设置区域模板选项，使面包屑显示时有足够边距。

## 全局页面

显示列表或面包屑对象的标准方式是将其放置在页面上的一个区域中。这种技术提供了相当大的灵活性——每个页面可以在不同位置、使用不同模板来显示列表或面包屑。然而，你通常不需要这么大的灵活性。假设你希望每个页面都以完全相同的方式显示列表或面包屑。实现这种统一性最简单的方法是使用 `全局页面`（global page）。
全局页面是一个特殊的页面，其内容会包含在应用程序的每一个其他页面中。如果你将图 4-30 中的面包屑区域或图 4-15 中的 `导航菜单`（Navigation Menu）区域放在全局页面中，这些区域将统一出现在应用程序的所有页面上。APEX 在应用程序创建时会创建全局页面；按照惯例，全局页面的页面编号为 0。
面包屑区域的一个优点是，APEX 只会在页面确实有面包屑时才在页面上渲染它。也就是说，如果你把面包屑区域放在全局页面中，它将只出现在那些实际拥有面包屑的页面上。

#### 小结

APEX 中主要的导航工具是链接列表，也称为 `列表`（list）。内置的 `桌面导航栏`（Desktop Navigation Bar）列表的条目显示在应用程序的导航栏中，而 `桌面导航菜单`（Desktop Navigation Menu）列表的条目则出现在应用程序的导航菜单中。列表管理器允许你编辑这些列表的内容，应用程序的 `用户界面`（User Interface）属性则允许你自定义它们的位置和样式。
你也可以使用列表管理器来创建自己的列表。你可以通过将列表放置在类型为 `List` 的区域中来在页面上显示它。每个列表区域都有一个 `模板`（Template）属性，让你可以自定义列表的显示方式。
列表可以是分层的，即一个列表条目可以是一个或多个子条目的父级。这种层次结构为列表条目提供了结构。这种结构对于导航菜单列表尤其有用，因为它允许菜单包含大量条目而不会让用户不知所措。
另一种导航工具是 `面包屑`（breadcrumb）。要为页面定义面包屑，你必须首先了解该页面如何适应所有应用程序页面的层次结构中。页面的面包屑是从该页面到根节点的层次结构中的路径。要在页面上显示面包屑，你必须将其放置在类型为 `Breadcrumb` 的区域中。通常将面包屑区域放在应用程序的全局页面（页面 0）中，以便它在每个页面的相同位置出现。

## 项目

一个 `项目`（item）是网页中保存值的元素。每个项目都有一个 `类型`（type），例如 `文本字段`（Text Field）、`复选框`（Checkbox）或 `选择列表`（Select List）。每种项目类型都有特定的方式来显示其值并让用户修改该值。
你在使用 APEX 的过程中已经遇到了几种项目类型。例如，图 5-1 显示了用于重命名 `DEPT` 表某一列的对象浏览器屏幕。此图显示了五个项目：两个只读项目（标记为 `Schema` 和 `Table`）、一个选择列表（`当前列名`（Current Column Name））、一个文本框（`新列名`（New Column Name））和一个复选框（`保留大小写`（Preserve Case））。通常，每次你在 APEX 屏幕中输入一个值，都是通过一个项目来完成的。

## 定位项

您的下一个任务是将其项定位在其区域内。APEX 以行的形式显示区域的项，位于区域的顶部或底部。区域的 `Item Display Position` 属性（在区域的 `Appearance` 部分中）可让您指定其项是出现在其内容上方还是下方。此属性对于 `Item Sampler` 页面无关紧要，因为其区域没有其他内容，但对于下一章的页面将变得有用。
项的 `Layout` 部分决定了其在区域内的位置。APEX 在区域内定位项的方式类似于在页面内定位区域的方式。项根据其 `Sequence` 属性的值从左到右放置到行中，每当其 `Start New Row` 属性被选中时，就移动到新行。区域的宽度被划分为 12 个网格点。每个项（包括其标签，如果显示在项的左侧）有一个起始网格点和一个以网格点为单位的宽度。

## Text-Based Items

基于文本的项以其输入框为特征，用户可以在其中键入值。APEX 提供了多种基于文本的项，其中五种显示在图 5-2 的 `Text-Based Items` 区域中。为了可读性，该区域在图 5-10 中重印。

### Text Fields

`Text Field` 是默认的项类型。它也是最简单的项类型，因为对输入框中可以输入的值没有限制。项的值就是你输入到输入框中的内容。

### Number Fields

项 `P8_EMPNO` 是一个数字字段；它与文本字段的不同之处在于其值只能是数字。数字字段项的 `Settings` 部分有几个特定于数字的属性。图 5-11 显示了 `P8_EMPNO` 的这些属性。`Minimum Value` 和 `Maximum Value` 属性允许你以明显的方式限制允许的项值。`Number Alignment` 属性允许你指定数字在输入框内的对齐方式。

图 5-11 与数字字段相关的属性

### Password Items

项 `P8_SALARY` 是一个密码字段；它与文本字段的不同之处在于其值不会显示在页面上。输入时，其字符会被圆圈替代，页面渲染时也以这种方式显示。

### Text Fields with Autocomplete

自动完成的文本字段是一个带有相关建议值列表的文本字段。当你在输入框中键入时，页面会显示一个匹配值的菜单。如果你点击一个显示的值，APEX 会自动将该值输入到输入框中。例如，项 `P8_JOB` 是一个自动完成的文本字段，其关联列表由五个当前的职位名称组成。当我在框中输入 `a` 时，APEX 显示了职位名称 `ANALYST`（因为它是唯一以 `a` 开头的职位）；然后我就可以点击该标题而无需进一步输入。

自动完成字段使用户可以轻松输入最常见的值，但不强制他们从这些值中选择。即使从列表中选择了一个建议值，用户也可以通过继续输入来自由更改该值。

你可以在项的 `List of Values` 部分指定自动完成值列表。图 5-12 显示了 `P8_JOB` 的这个部分。

## 颜色选择器

项目 `P8_COLOR` 的类型是 `Color Picker`（颜色选择器）。计算机上的颜色通常由其红、绿、蓝分量值表示，每个分量值的范围是 0 到 255。例如，在图 5-10 中，为 `P8_COLOR` 选择的值具有 (`50`, `53`, `235`) 的 (`red`, `green`, `blue`) 分量值。该颜色主要是蓝色，带有一点绿色和红色。

这些分量值可以紧凑地表示为一个十六进制数。因为一个两位的十六进制数可以表示 256 个不同的值，所以一个六位数可以表示所有可能的颜色。在图 5-10 中，该颜色由数字 `#3235EB` 表示。注意，`32` 是 50 的十六进制值，`35` 是 53 的十六进制值，`EB` 是 235 的十六进制值。

APEX 颜色选择器项允许您在已知颜色值时直接输入。或者，您可以点击滴管图标从颜色轮中选择所需的颜色，APEX 会在输入框中显示相应的数字值。颜色选择器下方的链接是 *快速选择*，稍后将进行讨论。

`Format Mask`（格式掩码）属性的工作方式与报表中相同。该属性的值是一个描述项值如何显示的表达式。在图 5-16 中，`Hire Date`（雇佣日期）项的格式掩码为 `Month DD, YYYY`。

默认情况下，最大长度为空，表示没有上限。一个好的设计规则是为所有基于文本的项指定一个最大长度。在第 13 章中，您将看到没有最大长度（或不切实际的大长度）的项如何可能导致严重的安全漏洞。

快速选择项非常实用，因为它们能让用户快速输入项目值。事实上，APEX 界面中的许多项目都包含快速选择功能。例如，图 5-16 中的 `CSS Classes` 和 `Format Mask` 属性，其文本框右侧就有一个上下文菜单，图 5-5 中的 `Type` 属性也是如此。点击上下文菜单会显示该项目的快速选择值列表。遗憾的是，尽管 APEX 界面提供了多种实现快速选择的方式（如上下文菜单或链接），但使用 APEX 开发的应用程序却仅限于一种方式（即显示在项目下方的链接）。
具有快速选择功能的项目在理念上类似于自动完成项。两者都允许用户通过从建议值中选择来消除（或减少）输入。不同之处在于，自动完成菜单会随着用户的输入进行过滤，而快速选择则是一次性全部呈现给用户。因此，一个项目可以拥有任意多个自动完成值，而 APEX 将一个项目的快速选择限制为十个。

## 基于列表的项目
与基于文本的项目不同，基于列表的项目没有输入框。相反，它向用户呈现一个可供选择的可能性列表。此外，用户选择的值不一定是分配给该项目的值。也就是说，该项目向用户展示一个显示值列表；当用户选择一个显示值时，一个对应的返回值会被分配给该项目。

例如，考虑图 5-2 中 `单值列表项` 的三个子区域，这些区域在图 5-18 中重新打印。每个区域专用于三种常见的基于列表的项目类型之一——单选按钮组、选择列表和弹出列表——并且包含三个该类型的项目。虽然从图中无法看出，但这九个项目都具有相同的可能返回值集合，即部门编号集合 {10, 20, 30, 40}。此外，图中显示的选择使得每个项目的值都相同，即 30。

![../images/335103_3_En_5_Chapter/335103_3_En_5_Fig18_HTML.jpg](img/335103_3_En_5_Fig18_HTML.jpg)
**图 5-18** 单值列表项

这九个项目具有相同的功能：用户选择一个部门，相应的部门编号就被分配给该项目。这些项目的区别仅在于其显示值是什么以及如何向用户呈现这些值。
首先考虑单选按钮组。在单选按钮组中，所有显示值都出现在页面上，用户通过点击来选择所需值。`单选按钮组` 区域中的每个单选按钮组都有不同的显示值：第一个组显示部门名称列表，第二个组显示部门编号列表，第三个组在括号中显示每个部门名称及其员工数量。
现在考虑选择列表。当用户点击选择列表时，会出现一个显示值菜单。第一个选择列表的显示值与第一个单选按钮组的显示值相同，另外两个选择列表也是如此。
最后，考虑弹出列表。当用户点击项目右侧的箭头时，会出现一个显示值菜单供用户选择。三个弹出列表的显示值与另外两个区域的相应显示值相同。
弹出列表与选择列表的不同之处在于它还有一个搜索框。用户可以在搜索框中输入所需值的前缀，然后点击放大镜；显示的值列表将被过滤，只包含具有该前缀的值。弹出列表使得从冗长的值列表中进行选择成为可能，而无需大量滚动。
单选按钮组项目的 `设置` 部分有一个 `列数` 属性。此属性指定单选按钮在页面上的排列方式。在图 5-18 中，你可以看到第一个单选按钮组有两列，第二个有四列，第三个有一列。
基于列表的项目最重要的问题是如何指定其显示值和返回值。你可以通过 SQL 查询动态指定这些值，或通过显式值静态指定。图 5-18 中的所有九个项目都是动态指定的，因此我将从这里开始讨论。

## 动态基于列表的项目
用于指定显示值和返回值之间对应关系的 SQL 查询应有两列：第一列给出显示值，第二列给出返回值。这些列的名称并不重要。

例如，考虑单选按钮组项目 `P8_RADIO_NAME`。它的查询应在第一列中包含部门名称，在第二列中包含部门编号。此外，项目 `P8_SELECT_NAME` 和 `P8_POPUP_NAME` 也将具有相同的显示值和返回值。因此，所有三个项目的值都可以通过以下查询指定：

```
select DName, DeptNo
from DEPT
order by DName
```

此查询在项目的 `值列表` 部分中指定。这与图 5-12 中指定自动完成文本项目的值列表的部分相同；唯一的区别是基于列表的项目需要一个两列查询。
现在你已经准备好配置项目 `P8_RADIO_NAME`、`P8_SELECT_NAME` 和 `P8_POPUP_NAME`。将它们的 `类型` 属性分别设置为 `单选按钮组`、`选择列表` 和 `弹出列表`，并将它们的 `值列表` 属性设置为前面的 SQL 查询。你还应该将 `P8_RADIO_NAME` 的 `列数` 属性设置为 `2`。



## 基于静态列表的项

现在考虑项 `P8_RADIO_NO`、`P8_SELECT_NO` 和 `P8_POPUP_NO` —— 也就是图 5-12 中每个区域的第二项。这些项具有相同的显示值和返回值。这里有趣的情况是显示值和返回值都是部门编号。如果你用直接的方式编写它们的 SQL 查询，那么两个输出列的名称将会相同，这在 SQL 中是非法的。为了使查询合法，你需要至少重命名其中一个列。查询应该类似于这样：

```sql
select DeptNo as DisplayVal, DeptNo as ReturnVal
from DEPT
order by DisplayVal
```

**注意**

列的名称可以是任何名称；SQL 只要求它们彼此不同。

你应该类似地配置项 `P8_RADIO_NO`、`P8_SELECT_NO` 和 `P8_POPUP_NO`，方式与前三个项相同。将它们的 `Type` 属性分别设置为 `Radio Group`、`Select List` 和 `Popup List`，并将它们的 `List of Values` 属性设置为前面的 SQL 查询。你还应该将 `P8_RADIO_NO` 的 `Number of Columns` 属性设置为 `4`。

项 `P8_RADIO_INFO`、`P8_SELECT_INFO` 和 `P8_POPUP_INFO` 的处理方式类似。它们的属性值与前几个项相同，只是查询不同。下面是该查询，它展示了如何创建复杂的显示值：

```sql
select d.DName || ' (' || count(e.EmpNo) || ' employees)' as Display,
d.DeptNo as ReturnVal
from DEPT d left join EMP e on d.DeptNo = e.DeptNo
group by d.DeptNo, d.DName
order by Display
```

查询中的左连接确保所有部门都被包含在内，即使是那些没有员工的部门。

## 多值列表项

当你使用查询来指定基于列表的项的值时，APEX 会在每次渲染页面时执行该查询。如果查询的输出永远不会改变，那么你可以使用静态表达式来避免数据库调用。

你指定静态表达式的方式与类型为 `Text Value with Autocomplete` 的项相同：首先，将 `List of Values` 部分中的 `Type` 属性设置为 `Static Values`；然后点击 `Static Values` 属性以调出“静态值”屏幕。接着，你可以为列表的每个成员输入显示值和返回值。图 5-19 展示了我如何为 `P8_RADIO_NAME`、`P8_SELECT_NAME` 和 `P8_POPUP_NAME` 填写值。

多值列表项的配置方式与单值项相同。特别是，它们使用 SQL 查询或静态表达式来指明其显示值和返回值。例如，该区域中四个项的 SQL 查询是：

```sql

from DEPT
order by DName
```

主要问题是如何为一个选择了多个显示值的项分配单个返回值。APEX 使用约定：多值项的值是一个字符串，由所选的返回值用冒号分隔组成。如果没有选择显示值，则该项的值为 `null`。例如，图 5-20 中的所有项的值都是 `40:30`。

让我们看看这四种项类型。第一个项 `P8_M_SELECT` 是一个选择列表；也就是说，与 `Single-Value List` 区域中的项类型相同。要指定该项为多值选择列表，你需要将其 `Settings` 部分中的 `Allow Multi Selection` 属性设置为 `Yes`。其 `Appearance` 部分中的 `Height` 属性的值决定了显示多少行菜单。所有其他属性的配置方式相同。

请注意，多值选择列表 `P8_M_SELECT` 的外观和行为与单值选择列表 `P8_SELECT_NAME` 不同。多值列表始终在页面上显示菜单，而不是按需显示。（并且如果列表中的项目多于页面上可以显示的数量，会出现滚动条，以便用户滚动浏览这些值。）用户可以通过按住 Control 或 Command 键（取决于操作系统）并点击来从多值列表中选择其他菜单项。

该区域中的第二个项 `P8_CHECKBOX` 是一个复选框组。复选框组的配置与单选按钮组完全相同。复选框组与单选按钮组的不同之处在于可以选择多个复选框，并且点击一个框会切换其值。

第三个项 `P8_SHUTTLE` 是一个穿梭列表，它与多值选择列表非常相似，配置方式也相同。它们之间的区别是，多值选择列表通过高亮显示来表示选定的值，而穿梭列表则有单独的菜单来显示已选定和未选定的值。

第四个项 `P8_LIST` 是一个列表管理器，它是自动完成文本框和弹出列表的交叉产物。与自动完成类似，用户可以在输入框中输入任何值，然后点击 `Add` 按钮将其添加到选定值列表中。或者，用户可以点击弹出箭头以调出一个值菜单进行选择，选中的值会被添加到输入框中。然而，重要的是要注意，虽然弹出列表显示的是显示值，但它将返回值放入输入框。因此，图 5-20 显示选择了部门编号 `30` 和 `40`，尽管弹出列表显示的是部门名称。

## 基于显示的项目

还有另外两种有用的项目类型未出现在`项目示例器`页面中：`仅显示`项目和`隐藏`项目。这些项目与你已见过的其他项目不同，因为用户无法为它们赋值；它们的值只能由 APEX 服务器分配。一个仅显示的项目有一个标签和一个文本值，类似于文本字段；不同之处在于该值不在输入框中且不可编辑。例如，再次参考图 5-1；标有`模式`和`表`的字段就是仅显示的。隐藏项目不可见，用户不知道它们的存在。它们通常用于保存操作的值，以便另一个操作可以使用它。你将在第 6 章中看到此技术的示例。

## 项目的初始值

当 APEX 渲染一个页面时，它会为页面上的每个项目分配一个初始值。如果项目是基于文本的，APEX 会将该值放入该项目的输入框中；如果项目是基于列表的，APEX 会使用该值来确定相应的初始显示值。APEX 通过两种方式之一确定此初始值：它可以计算项目的`源表达式`，或者可以使用先前存储在会话状态中的值。由于尚未介绍会话状态的概念，现在假设会话状态值始终为`null`。

你在属性编辑器的`源`部分指定项目的源表达式。图 5-24 显示了此部分的两个重要属性`类型`和`用法`，以及它们的默认值。

## 按钮

按钮在点击时执行一个`操作`。APEX 按钮有两种基本操作：它可以`重定向`或`提交`。`重定向`一词简单地意味着转到另一个网页。在这方面，按钮的作用类似于 HTML 链接。`提交`一词意味着让 Web 服务器处理当前页面。此处理可以涉及设置会话状态和在数据库上执行操作，以及移动到另一个页面。第 6 章到第 10 章将详细涵盖重定向和提交操作的用途。作为一个例子，让我们看看如何在`项目示例器`页面的`一些按钮`区域添加两个按钮：一个操作是`重定向到页面 8`的`取消`按钮和一个操作是`提交`的`提交`按钮。

## 总结：项目

项目是保存值的页面元素。有许多项目类型，它们在页面上显示值的方式以及帮助用户指定该值的方式上有所不同。最重要的是区分基于文本的项目和基于列表的项目。

基于文本的项目在其输入框中显示其值，用户可以通过在输入框中键入来指定值。基于文本的项目类型支持输入特殊值（如颜色和日期）、辅助数据输入（如自动完成）以及限制允许的值（如数字字段）。

基于列表的项目强制用户从给定的值列表中选择。该列表可以通过 SQL 查询动态定义，也可以通过显式值静态定义。在任何情况下，都存在显示值（呈现给用户的值）和返回值（项目实际持有的值）之间的区别。各种基于列表的项目类型主要区别在于它们如何在页面上呈现显示值。

项目的目的是允许用户向服务器发送一个值。第 6 章将研究这个问题，并提供展示项目在 Web 应用程序中典型用法的示例。

## 会话状态

用户与应用程序的交互称为 *会话*。一个会话可以包含多个页面请求。例如，在电子商务应用程序中，一个会话可能涉及浏览待售商品、将其中一些添加到购物车、输入付款和运输信息以及完成交易。

在会话期间，用户为页面项目赋值，可能涉及多个页面，并将这些值发送到服务器进行处理。给定会话的项目值构成其 *会话状态*。在本章中，您将研究应用程序可以修改和访问会话状态的各种方式，并了解一些利用此状态的常见页面设计技术。

## 会话 ID

在 Web 应用程序中构建会话功能是有问题的，因为浏览器请求是匿名的。服务器没有内置方法来知道两个浏览器请求是否属于同一个会话。APEX 通过为每个会话分配一个标识符来解决此问题。浏览器请求通过将会话 ID 添加到请求中来指示其会话。

回想一下，APEX URL 由字符串 `f?p=` 后跟冒号分隔的值字符串组成。前两个值是应用程序 ID 和页面 ID；第三个值是会话 ID。要查看实际操作中的会话 ID，请登录 APEX 并浏览其屏幕。查看与您访问的每个屏幕关联的 URL。您应该会发现每个 URL 都具有相同的会话 ID 值。现在从 APEX 注销。当您重新登录时，APEX 将为此次交互分配一个新的会话 ID。您访问的每个屏幕的 URL 现在都将具有这个新的会话 ID。

您也可以在您自己的应用程序中观察会话 ID。从 APEX 注销并运行您的 `Employee Demo` 应用程序。APEX 将分配一个会话 ID，该 ID 将出现在每个页面的 URL 中。您可以通过提交会话 ID 为 0 的请求来请求新的会话。（实际上，任何随机 ID 值都可以。）例如，我可以通过提交以下 URL 在新会话中查看我的应用程序的主页：

```
apex.oracle.com/pls/apex/f?p=91392:1:0
```

当页面加载时，URL 将包含一个新的会话 ID。



## 提交与重定向

假设你修改了网页上某个项目的值，可能是通过选择单选按钮或在文本字段中输入。你的浏览器知道如何直接显示这一更改，而无需请求服务器重新渲染页面。因此，服务器将对你对该值所做的更改一无所知，从而无法使用新值。

如果你希望应用程序知道页面上各项的值，你必须先将这些值上传到服务器。此操作称为 `提交`。服务器为你的应用程序的每个项目都持有一个变量。当你提交页面时，该页面上每个项目的值都会保存在对应的变量中。实际上，提交操作将会话状态与当前页面同步。

回顾一下，按钮的操作要么是 `提交`，要么是 `重定向`。这两种操作都会导致 APEX 服务器渲染一个目标页面并将其发送到浏览器。对于重定向，你在重定向操作中指定目标页面。对于提交，你通过单独的 `分支` 组件指定目标页面——如果未指定分支，则重新加载当前页面。分支在第 9 章中讨论，因此目前你可以假设提交操作总是重新加载当前页面。

提交和重定向的主要区别在于，提交会将页面的项目值上传到服务器，而重定向不会。如果用户更改了页面上某些项目的值，然后执行重定向，这些更改将会丢失——即使重定向到的是同一个页面。

为了看到这种差异的示例，考虑第 5 章中的 `项目采样器` 演示页面，如图 5-2 所示。回顾第 5 章，项目的初始值由其 `源` 属性决定。默认是使用会话状态值（如果它不为空）。这意味着如果你为项目分配一个非空值并提交，那么当页面重新加载时，该值将成为该项目的初始值。你可以通过更改页面上的一些值并点击 `取消` 按钮来验证这一事实。因为该按钮执行的是重定向，你的更改会丢失。现在再次修改这些值并点击 `提交` 按钮。项目值保持选择后的状态，因为提交操作将这些更改写入了会话状态，并在页面重新加载时检索了它们。如果你导航到不同的页面再返回，这些值仍然会存在。

回顾一下，当你在 APEX 中运行页面时，开发者工具栏会出现在屏幕底部。标记为 `会话` 的工具栏按钮显示当前的会话状态。点击它会打开一个窗口，显示所有项目及其当前的会话值。在调试页面时，这个窗口非常有用。在示例中，打开会话窗口并查看会话状态。然后更改一些项目并提交。再次打开会话窗口，验证项目值是否已更改。

重要的是要理解，指定项目的初始值会影响浏览器中的项目值，但绝不会改变其在会话状态中的值。你可以在 `项目采样器` 页面上进行一些实验来验证这一事实。转到属性编辑器，将项目 `P8_JOB` 的源设置为静态表达式 `CLERK`，并将其使用设置为 `始终，替换会话状态中的任何现有值`。运行页面，将项目的值更改为 `ANALYST`，然后点击 `提交` 按钮，检查每一步的会话状态。会话状态值在页面提交之前一直保持空白，在提交时变为 `ANALYST`。然而，当页面渲染时，浏览器会显示 `CLERK`。现在回到属性编辑器，将该项目的 `源使用` 属性更改为 `仅当会话状态中的当前值为空时` 并重复实验。（首先将项目值设为空白并点击 `提交` 按钮以清除会话状态。）你应该观察到会话状态值的行为与之前相同。然而，浏览器会显示 `CLERK`，直到你提交一个新值，之后它将显示该值。

请注意，这种修改会话状态的方式与提交操作完全不同，因为它不（且不能）使用浏览器中显示的值。也就是说，来自按钮的重定向可以将项目的会话状态值设置为常量或计算值，但无法将其设置为用户输入的浏览器值。

### 引用会话状态变量

既然你已了解如何设置会话状态变量的值，接下来需要知道如何让页面访问该值。主要有两种技术。如果你想在 SQL 代码中引用该变量，请在项目名称前加上冒号。这种引用称为 `绑定变量`。如果你想在 HTML 表达式中引用项目，请在项目名称前加上 `&` 符号，并在末尾加上句点。后一种引用称为 `替换字符串`，已在第 3 章介绍过。

例如，考虑 `Item Sampler` 页面上的项目 `P8_EMPNO`。要将其作为替换字符串引用，请写作 `&P8_EMPNO.`。要将其作为绑定变量引用，请写作 `:P8_EMPNO`。

在本书中，你已经看到了 SQL 查询的几种用途，例如作为报表或图表的源、项目的初始值，以及基于列表的项目的值定义。你也看到了 HTML 表达式的几种用途，例如页面或区域的标题，以及项目的标签。本章剩余部分将举例说明如何通过在这些 SQL 查询和 HTML 表达式中使用项目引用，来极大增强页面的功能。

在构建本章的任何页面之前，你可能需要转到 `桌面导航菜单` 的 `列表详情` 页面（如图 4-21 所示），并添加一个名为 `会话状态` 的根条目。然后，每次创建页面时，你都可以将其导航菜单条目的父项设置为 `会话状态`。

## 自定义报表

`自定义报表` 是其内容受一个或多个项目值影响的报表。用户在页面上为某些项目输入值并点击 `提交` 按钮。然后，页面将根据这些输入值显示一个报表。在本节中，你将构建三个自定义报表页面来阐述此技术。

### 按职位和部门筛选页面

第一个页面名为 `按职位和部门筛选`，是演示应用的第 9 页，见图 6-5。其思路是用户为 `职位` 和 `部门` 项目选择值，然后点击 `提交` 按钮。随后，报表将显示具有所选职位且属于所选部门的员工。例如，图 6-5 显示了研究部门中的办事员。

![../images/335103_3_En_6_Chapter/335103_3_En_6_Fig5_HTML.jpg](img/335103_3_En_6_Fig5_HTML.jpg)

图 6-5 按职位和部门筛选页面

此页面有两个区域：`选择值` 和 `匹配的员工`。`选择值` 区域是一个静态内容区域，包含两个选择列表 `P9_JOB` 和 `P9_DEPTNO`，它们的值都由 SQL 查询定义。`P9_JOB` 的值列表查询如下：

```sql
select distinct Job as DisplayVal, Job as ResultVal
from EMP
order by DisplayVal
```

`P9_DEPTNO` 的值列表查询如下：

```sql
select DName, DeptNo
from DEPT
order by DName
```

按钮的操作是 `提交`，其位置为 `编辑`。

`匹配的员工` 区域是一个报表区域。为了让报表显示具有所选职位和部门编号的 `EMP` 记录，其源查询需要引用两个项目的值。以下查询实现了这一点；请注意它如何使用绑定变量语法来引用这两个项目。

```sql
select EmpNo, EName, Job, Sal, DeptNo
from EMP
where Job = :P9_JOB  and  DeptNo = :P9_DEPTNO
order by EName
```

你可以通过允许用户为 `P9_JOB` 或 `P9_DEPTNO` 选择 `空` 值来增强此区域的功能。意图是，`P9_JOB`（或 `P9_DEPTNO`）的 `空` 值应表示查询不会使用该项目进行筛选。例如，如果两个项目都是 `空`，报表将显示所有员工。要实现此功能，请修改源查询以显式测试 `空` 项目值，如下所示：

```sql
select EmpNo, EName, Job, Sal, DeptNo
from EMP
where (:P9_JOB is null  or  Job = :P9_JOB)
and   (:P9_DEPTNO is null  or  DeptNo = :P9_DEPTNO)
order by EName
```

我想指出，此页面并非必须使用两个区域。事实上，很容易修改此页面以使用单个区域。首先，转到项目 `P9_JOB` 的 `布局` 部分，将其 `区域` 属性更改为 `匹配的员工`。然后对 `P9_DEPTNO` 和 `提交` 按钮执行相同操作。最后，转到 `匹配的员工` 的 `外观` 部分，将其 `项目显示位置` 属性设置为 `内容上方`，以便这两个项目显示在报表上方。（你也可以根据需要删除 `选择值` 区域。）现在运行该页面。你会发现其行为没有改变。也就是说，使用一个还是两个（甚至三个！）区域的决定纯粹基于美观。

```sql
select EmpNo, EName, Job, Sal, DeptNo
```

主从报表由两份报表组成。这两份报表是*链接*的，意味着明细报表的每一行在主报表中都有关联行。用户通过选择主报表中的一行来进行交互；明细报表则通过仅显示与所选主记录关联的明细行来响应。

在本节中，您将构建一个名为 `Employees by Department` 的主从页面，它是 `Employee Demo` 应用程序的第 12 页。其主报表包含每个部门的一行。单击某个部门行会导致明细报表显示该部门的员工列表。图 6-8 显示了单击会计部门链接后的页面。

思考如何实现此页面。主要问题在于，主报表需要让明细报表知道选择了哪一行。标准技术是使用一个隐藏项，其值就是所选行的键值。我们将此项命名为 `P12_DEPTNO`。主报表会将选中行的 `DeptNo` 值赋给 `P12_DEPTNO`。然后明细报表将使用此值来自行定制显示内容。

实现定制的明细报表很直接，其源查询如下：

```
select EmpNo, EName, Job, Sal
from EMP
where DeptNo = :P12_DEPTNO
order by EName
```

难点在于实现主报表。你需要回答三个问题：报表的源查询是什么？如何让它的第四列显示一个链接？以及如何将合适的值赋给 `P12_DEPTNO`？

主报表有四列。前三列是部门名称、位置和员工数；第四列包含指向明细表的链接。由于链接的目的是选择一个部门编号，因此该列的值为 `DeptNo` 是合理的。因此，你有以下源查询（这是一个右连接，以便主报表能显示没有员工的部门）：

```
select d.DName, d.Loc, count(e.EmpNo) as EmpCount, d.DeptNo
from EMP e right join DEPT d
on  e.DeptNo = d.DeptNo
group by d.DeptNo, d.DName, d.Loc
order by DName
```

第 3 章描述了如何为报表列指定链接。回顾一下，转到页面设计器并单击渲染树中的 `DEPTNO` 列。将列的类型设置为 `Link`。在属性编辑器中找到 `Link` 部分（参见第 3 章中的图 3-35），并将链接文本设置为 `Click for details`。单击 `Target` 框将进入 `Link Builder` 页面，你可以在其中将目标设置为第 12 页（当前页）。

等等。重定向到当前页面没有任何作用，那么链接的意义何在？它的真正目的是更改会话状态。回想一下，`Link Builder` 页面还有一个 `Set Items` 部分，你可以在其中将隐藏项 `P12_DEPTNO` 设置为所选的部门编号。回顾第 3 章，要引用当前行中列 `X` 的值，需写作 `#X#`。因此，你应该将 `P12_DEPTNO` 设置为 `#DEPTNO#`，如图 6-9 所示。

## 图表下钻

图表下钻与主从关系非常相似。图表是主项，报表是从项。实际上，其实现策略与主从报表的本质相同：

1.  创建一个隐藏项，此处名为 `P13_JOB`。
2.  创建由该隐藏项定制的明细报表。
3.  创建图表，并配置其在被点击时将所选职位分配给该隐藏项。

明细报表的源查询与 `按部门查看员工` 中的查询本质相同。唯一的区别是该查询执行了一个连接（join），以便报表能够显示部门名称。SQL 代码如下：

```sql
select e.EmpNo, e.EName, d.DName, e.Sal
from EMP e join DEPT d
on    e.DeptNo = d.DeptNo
where e.Job = :P13_JOB
order by EName
```

```sql
select Job, count(*) as EmpCount
from EMP
group by Job
order by EmpCount desc
```

## 级联列表

下一个任务是构建一个页面，使用户能够从选择列表中选择员工姓名，然后显示包含该员工信息的报表。该页面名为 `级联列表`，是 `员工演示` 应用的第 14 页，如图 6-16 所示。该页面有两个区域，每个区域都提供了一种解决方案。

![../images/335103_3_En_6_Chapter/335103_3_En_6_Fig16_HTML.jpg](img/335103_3_En_6_Fig16_HTML.jpg)

图 6-16：级联列表

### 顶部区域

顶部区域有一个名为 `P14_EMPNO` 的项。这是一个选择列表，显示所有员工的姓名，并将员工编号作为结果值。也就是说，它的值列表查询如下：

```
select EName, EmpNo
from EMP
order by EName
```

顶部区域没有按钮。相反，选择列表的操作是 `重定向并设置值`，这意味着从选择列表中选择一个员工姓名会导致该项的值在会话状态中被更改。

顶部区域的类型是 `经典报表`。它的源查询由 `P14_EMPNO` 的值定制，如下所示：

```
select e.EmpNo, d.DName, e.Job, e.Sal, d.Loc
from EMP e join DEPT d
on    e.DeptNo = d.DeptNo
where e.EmpNo = :P14_EMPNO
order by EmpNo
```

如果你尝试这个区域，你应该会发现，由于员工数量较少，浏览姓名列表很容易。然而，假设 `EMP` 表很大，有数千名员工可供选择。即使姓名已排序，滚动浏览选择列表也会很麻烦且耗时。解决此问题的常见方法是让用户通过一系列选择列表分步选择姓名。

### 底部区域

图 6-16 的底部区域使用两个选择列表来选择一名员工。第一个选择列表显示每个员工姓名的首字母，第二个选择列表显示姓名以该字母开头的员工姓名。这种技术称为 *级联列表*，其中第二个选择列表的显示值会根据为第一个列表选择的值而变化。这种技术可以扩展到多个选择列表——除第一个之外的每个选择列表的值都会根据其前面选择列表的值而变化。这种方法的优点是用户可以滚动浏览几个小的选择列表，而不是一个大列表。

底部区域中的两个选择列表名为 `P14_FIRST_LETTER` 和 `P14_EMPNO_CASCADE`。它们的定义如下。

选择列表 `P14_FIRST_LETTER` 不依赖于任何其他项，因此其查询很简单：

```
select FirstLetter as DisplayVal, FirstLetter as ResultVal
from  (select distinct substr(EName, 1, 1) as FirstLetter
from EMP)
order by DisplayVal
```

列表 `P14_EMPNO_CASCADE` 依赖于 `P14_FIRST_LETTER` 的值，其查询如下：

```
select EName, EmpNo
from EMP
where substr(EName, 1, 1) = :P14_FIRST_LETTER
order by EName
```

底部报表的源与顶部报表完全相同，只是 `P14_EMPNO` 被 `P14_EMPNO_CASCADE` 取代。如果你将这两个选择列表的操作都设置为 `重定向并设置值`，那么选择首字母会导致姓名列表更新，而选择姓名会导致报表更新。

## 条件渲染

在使用应用程序构建器的过程中，你遇到过无数次这样的情况：在属性编辑器中选择一个特定值会导致编辑器突然显示其他属性。图 6-15 所示的 `链接生成器` 页面就是一个例子——如果你选择目标类型 `URL`，那么 `设置项` 属性会突然消失，如果你将其值改回 `此应用程序中的页面`，那么那些项会重新出现。这是怎么回事？

虽然看起来这些项是当场创建和删除的，但事实并非如此；相反，这些项始终存在。当你在链接生成器中并选择值 `此应用程序中的页面` 时，APEX 会显示额外的项；当你选择值 `URL` 时，APEX 会隐藏它们。这种动态显示或隐藏项的能力称为 *条件渲染*。

可以为项、按钮和区域定义条件渲染。每个在属性编辑器中都有一个 `服务器端条件` 部分，让你可以从选择列表中选择一个条件类型。尽管有许多条件类型，但这里的重点是 `SQL 表达式` 类型。该类型的值是一个返回 `true`/`false` 的 SQL 表达式。表达式的值决定了项、按钮或区域是否被渲染。

#### 小结

APEX 服务器维护着每个用户会话的数据；这些数据被称为 *会话状态*。在本章中，你了解了会话状态如何被修改，并探讨了应用程序开发者利用它的几种方式。
会话状态为应用程序中的每个项都维护一个变量。这些变量的值会因为提交或重定向操作而被修改。
当页面被提交时，页面上所有项的值都将在会话状态中得到更新。当页面重定向到目标页面时，重定向操作可以为特定项赋值。特别是，当你配置按钮、报表链接或图表进行重定向时，属性编辑器将显示 `链接构建器` 向导，其中包含一个用于指定会话状态值的部分。
会话状态的一个常见用途是使用户能够自定义报表。其思路是用户指定一些值，这些值会影响报表显示哪些记录。你看到了用户指定这些值的几种方式示例：

*   用户可以直接将值输入到页面上的项中。
*   用户可以从主报表中选择一行。
*   用户可以通过图表向下钻取来选择一个值。

本章还探讨了会话状态的其他用途，例如 *级联列表*（其中为一个选择列表选择的值决定了另一个列表显示的值）和 *条件渲染*（其中一项的值决定了某个区域应该显示还是隐藏）。
会话状态是 APEX 中极其重要的一个方面。本书余下部分的大部分示例都将运用本章提到的技术。

## 7. 进程

正如你所见，仅使用项、报表和图表就可以编写功能强大的网页。但要实现更复杂的功能，你必须能够执行计算并与数据库进行更一般化的交互。*进程* 这个概念正是为了解决这两个问题。
本章将探讨一些需要使用进程的典型情况，并介绍 PL/SQL 的基础知识，以便你可以编写自己的代码来实现这些进程。本章还将探讨编写 PL/SQL 代码时可能遇到的陷阱，例如丢失更新的可能性。

## APEX 执行点与进程创建

图中显示了五个节点。前四个对应于提交操作期间可能发生的四种活动阶段；APEX 将这些阶段称为 **执行点**。这些阶段具有以下目的：

*   在 **提交后** 阶段，服务器计算额外的会话状态值，为主要处理活动做准备。这种计算偶尔有用，在第 10 章中有简要讨论。

*   在 **验证** 阶段，服务器运行验证检查，以确保提交请求是使用适当的数据调用的。如果检测到问题，服务器将中止进一步处理。验证是第 8 章的主题。

*   在 **处理** 阶段，服务器执行与提交请求相关联的进程。这些进程的定义和使用是本章及第 10 章的主题。

*   在 **处理后** 阶段，服务器使用分支来决定将哪个页面发送到浏览器。分支是第 9 章的主题。

图 7-1 中的第五个节点名为 `Ajax Callback`，它不是一个执行点；其目的是保存可以从 JavaScript 函数调用的代码。由于 APEX 中 JavaScript 的使用超出了本书的范围，我将不再进一步提及此节点。

### 使用 PL/SQL 访问数据库

PL/SQL 进程通常用于访问数据库。SQL 数据修改命令可以用作 PL/SQL 语句，这意味着修改数据库的进程可以仅由那一条语句组成。例如，以下 PL/SQL 块将部门 10 中每个人的工资增加 100。

```
begin
update EMP
set Sal = Sal + 100
where DeptNo = 10;
end;
```

### 用于计算值的 PL/SQL

本节探讨 PL/SQL 的编程语言方面。PL/SQL 的核心是一种传统的编程语言，具有变量、赋值语句、条件语句和循环。在这方面，它与 C、Java、Python 和类似语言没有太大区别。如果你有使用这些语言中的任何一种的经验，那么编写 PL/SQL 代码最困难的部分可能是找出合适的语法。

```
declare
v_amount int := :P18_SOLO_AMOUNT;
v_saleCommission number(7,2);
v_prevCommission number(7,2);
v_newCommission  number(7,2);
begin
-- 步骤 1：计算新佣金，前 100 美元按 15%计算，
-- 剩余销售价格按 10%计算。
if v_amount > 100 then
v_amount := v_amount - 100;
v_saleCommission := 15 + (v_amount * 0.1);
else
v_saleCommission := v_amount * 0.15;
end if;
-- 步骤 2：从 EMP 表中检索之前的佣金
-- 并计算新的佣金。
select Comm into v_prevCommission
from EMP
where EmpNo = :P18_SOLO_SALESMAN;
v_newCommission := v_prevCommission + v_saleCommission;
-- 步骤 3：更新员工记录。
update EMP
set Comm = v_newCommission
where EmpNo = :P18_SOLO_SALESMAN;
-- 步骤 4：计算各项的值。
:P18_SOLO_SALE_COMMISSION := v_saleCommission;
:P18_SOLO_PREV_COMMISSION := v_prevCommission;
:P18_SOLO_NEW_COMMISSION  := v_newCommission;
end;
```

### 局部变量
PL/SQL 块可以在 `begin` 关键字之前有一个 `变量声明部分`。声明部分以关键字 `declare` 开头，后跟一系列变量声明。清单 7-5 的代码声明了四个局部变量：`v_amount`、`v_saleCommission`、`v_prevCommission` 和 `v_newCommission`。每个声明的语法类似于 SQL 中声明表列的语法，可能的变量类型也相同。例如，后三个变量的类型为 `number(7,2)`，这与 `EMP` 表中 `Comm` 列的类型相同。
局部变量和项引用可以在 PL/SQL 块中互换使用。区别在于，局部变量按原样引用，而项引用需要前置冒号。例如，步骤 3 中的 SQL `update` 命令同时引用了局部变量 `v_newCommission` 和项 `P18_SOLO_SALESMAN`。
由于项是使用绑定变量语法引用的，因此很容易识别。但是，SQL 命令中的局部变量引用很难与列名或函数调用区分开来。为避免混淆，应始终为局部变量命名，以便人们知道它们是变量。这里采用的一个常见约定是，每个局部变量名以 `v_` 开头。

### 注释
与 SQL 一样，PL/SQL 用两个连续的减号表示注释。跟在两个减号之后的所有字符直到行尾都会被忽略。

### 赋值语句
PL/SQL 中的赋值语句使用 `:=` 作为赋值运算符。赋值也可以在 `declare` 部分中使用，以为局部变量分配初始值。

### 条件语句
PL/SQL 支持多种形式的条件语句；清单 7-5 步骤 1 中出现的形式适用于通用目的。请注意，关键字充当部分边界：条件表达式出现在 `if` 和 `then` 之间，当条件为 `true` 时要执行的语句出现在 `then` 和 `else` 之间，当条件为 `false` 时要执行的语句出现在 `else` 和 `end if` 之间。通常，如果在 `else` 和 `end if` 关键字之间没有语句，则可以省略 `else` 关键字。

### 处理多行 SQL 查询
现在让我们看看如何构建 `Shared Commission` 区域。其 `Enter Sale` 区域与 `Solo Commission` 区域基本相同。它的项名称是 `P18_SHARED_SALESMAN` 和 `P18_SHARED_AMOUNT`。
`Commission Info` 区域中的项被命名为 `P18_SHARED_SALE_COMMISSION`、`P18_SHARED_COMMISSION_EACH` 和 `P18_SHARED_NEW_COMMISSIONS`。
选择列表 `P18_SHARED_SALESMAN` 的属性值与 `P18_SOLO_SALESMAN` 相同；不同之处在于其 `Allow Multi Selection` 属性设置为 `Yes`，`Height` 值为 `4`。回顾第 5 章，此类项的值是一个字符串，其中包含以冒号分隔的返回值。例如，如果从多值选择列表 `P18_SHARED_SALESMAN` 中选择了 `ALLEN` 和 `TURNER`。由于他们对应的员工编号是 7499 和 7844，因此该项的值是字符串 `"7499:7844"`。
从该字符串中提取所需信息并不特别容易，因此需要一些巧妙的方法。你需要做两件事：确定所选员工的数量，并从 `EMP` 表中提取他们的记录。
请注意，所选员工的数量比字符串中的冒号数量多一。你可以通过使用 SQL `replace` 函数创建一个新字符串（其中冒号被替换为空字符串，从而有效地移除了冒号），然后比较两个字符串的长度，来确定冒号的数量。也就是说，如果 `P18_SHARED_SALESMAN` 包含该字符串，则员工数量由以下表达式给出：
```
1 + length(:P18_SHARED_SALESMAN) - length(replace(:P18_SHARED_SALESMAN, ':', ''))
```
要提取每个所选员工的记录，请使用 `instr` 函数，就像你为 `Filter by Possible Department` 页面所做的那样。以下查询将检索所有选定员工的 `EMP` 记录：
```
select *
from EMP
where instr( ':'|| :P18_SHARED_SALESMAN||':',  ':'||EmpNo||':' ) > 0
```
解决了这些问题后，你现在可以为该区域的 `Submit` 按钮编写处理过程。PL/SQL 代码如清单 7-6 所示。
```
declare
v_saleCommission number(7,2);
v_amount         int := :P18_SHARED_AMOUNT;
v_empCount       int;
v_commissionEach number(7,2);
v_empInfo        varchar2(25);
v_output         varchar2(250) := '';
begin
-- 步骤 1：计算新佣金，前 100 美元按 15%计算，
-- 剩余销售价格按 10%计算。
if v_amount > 100 then
v_amount := v_amount - 100;
v_saleCommission := 15 + (v_amount * 0.1);
else
v_saleCommission := v_amount * 0.15;
end if;
-- 步骤 2：确定共享佣金。
v_empcount := 1 + length(:P18_SHARED_SALESMAN) - length(replace(:P18_SHARED_SALESMAN, ':', '' ));
v_commissionEach := v_saleCommission / v_empcount;
-- 步骤 3：更新 EMP 表。
update EMP
set Comm = Comm + v_commissionEach
where instr( ':'||:P18_SHARED_SALESMAN||':',  ':'||EmpNo||':' ) > 0;
-- 步骤 4：使用循环收集员工信息。
for row in (
select EName, Comm
from EMP
where instr( ':'||:P18_EMPNO_SHARED||':',  ':'||EmpNo||':' ) > 0 )
loop
v_empInfo := row.EName || ': ' || row.Comm || chr(13);
v_output  := v_output || v_empInfo;
end loop;
-- 步骤 5：写入项值。
:P18_SHARED_SALE_COMMISSION  := v_saleCommission;
:P18_SHARED_COMMISSION_EACH  := v_commissionEach;
:P18_SHARED_NEW_COMMISSIONS  := v_output;
end;
```

## 并发数据库更新

### 丢失更新问题

步骤 1 计算销售佣金，与之前相同。步骤 2 计算每位员工的按比例佣金，即用佣金总额除以员工数量。步骤 3 更新所选员工的记录。步骤 5 将一个值写入`Commission`区域的每个项目。

最有趣的代码出现在步骤 4 中，您在此处理每个所选员工。问题在于，检索员工信息的 SQL 查询会返回多行，因此您不能像之前那样使用`into`子句。取而代之的技术是遍历查询中的记录，每次循环处理一行。

循环的基本结构如下：

```
for r in (  )
loop
-- 用于处理行 r 的语句
end loop;
```

变量`r`无需声明。每次循环时，它将保存查询的下一行。如果`C`是查询的一个输出列，那么表达式`r.C`将返回当前行的`C`值。例如，在清单 7-6 中，行变量名为`row`，查询的两个输出列是`EName`和`Comm`。因此，循环代码引用的是`row.EName`和`row.Comm`。

循环体处理每个所选员工。其任务是构建一个字符串，总结该员工的新佣金。代码通过将员工姓名与新的佣金值拼接来创建摘要字符串。表达式`chr(13)`表示一个换行符。

一个经常遇到的情况称为*丢失更新问题*。当两个用户在大约相同的时间更新同一条数据库记录，并且更新的时序导致数据库系统丢失了其中一个更新时，就会出现此问题。以下两种场景说明了该问题。

对于第一种场景，考虑图 7-5 的`员工数据录入`页面。假设两个用户使用`更新员工`区域来更新 Allen 的记录——用户 A 希望将工资改为`1800`，用户 B 希望将职位改为`ANALYST`。如果两个用户大约同时点击`GetInfo`按钮，那么他们都将从`EMP`表中检索到相同的记录，该记录显示 Allen 是一名工资为 1600 的销售员。当用户 A 更改工资并点击`Update`按钮时，该记录将被替换为一条记录，显示 Allen 是一名工资为 1800 的销售员。然而，用户 B 看不到更新后的工资。当用户 B 更改职位并点击`Update`按钮时，该记录将被替换为一条记录，显示 Allen 是一名分析师，工资为 1600。

无论谁先写入，此场景都会出现问题。如果用户 A 先修改，用户 B 将用其先前值覆盖新工资；如果用户 B 先修改，用户 A 将用其先前值覆盖新职位。无论发生什么，数据库系统都会丢失其中一个更新。

您可以自行验证此场景。在两个私密浏览器窗口中打开`员工数据录入`页面（使用私密窗口可防止 APEX 识别出两个窗口都属于您）。在每个窗口中选择同一个员工并点击`GetInfo`按钮。然后在每个窗口中进行一次更新并点击其`Update`按钮。现在查看报告中修改后的记录——您会发现您的第一次更新丢失了。

对于第二种场景，考虑图 7-7 的`销售佣金`页面。假设 Allen 独立销售了 100 美元和 200 美元，分别获得 15 美元和 25 美元的佣金。用户 A 和用户 B 各自将其中一笔销售输入页面的`单独佣金`区域，并同时点击其`提交`按钮。假设他们的 PL/SQL 进程执行几乎同时到达清单 7-5 的步骤 2，这意味着两个进程都会看到当前佣金为 300 美元。用户 A 的进程将在步骤 3 计算出 315 美元的新佣金，用户 B 的进程将计算出 325 美元的新佣金。后更新的进程将覆盖先更新进程的佣金。被覆盖的更新将会丢失。

您也可以自行验证此场景。这次，在您的两个私密浏览器窗口中打开`销售佣金`页面。在每个窗口中，选择同一个销售员，选择不同的销售额，然后点击`提交`按钮。很可能，两个更新都不会丢失。原因是，丢失更新场景只有在 APEX 服务器恰好在步骤 2 之后立即推迟执行第一个进程时才会发生，这种情况可能发生但概率较低。您可以通过在步骤 2 之后立即向进程中引入一条*休眠*语句来大幅提高概率，如下所示：

```
...
v_newCommission := v_prevCommission + v_saleCommission;
dbms_session.sleep(10);
-- 步骤 3：更新员工记录。
...
```

此语句使进程在开始步骤 3 之前休眠 10 秒。现在重复实验。如果您在 10 秒内点击两个`提交`按钮，则会出现问题场景，并且您提交的第一个佣金将会丢失。

在这两种场景中，丢失更新的发生完全取决于用户请求的时序。这种对时序的依赖意味着丢失更新往往在看似随机的时间点发生。例如，一个应用程序很可能运行数月（或数年）而没有发生丢失更新，然后却在短时间内连续发生几次。

丢失更新显然是非常糟糕的事情。丢失更新可能在发生很久后才被发现，到那时可能为时已晚。应用程序开发人员必须确保它们永远不会发生。其不可预测的性质使得通过测试难以发现问题。因此，任何更新数据库的 PL/SQL 进程都必须专门编写以避免丢失更新。

### 避免丢失更新

当两个并发进程读取相同数据然后对其进行冲突更新时，就会发生丢失更新。避免此问题的方法是强制冲突进程按顺序交互，以便第二个进程读取第一个进程写入的数据。

Oracle 数据库系统使用*写锁*来确保这种顺序性。您通过向 SQL 查询添加`for update`子句来指定写锁。当您的进程执行包含该子句的查询时，数据库系统会锁定该查询的输出记录，使其不能被任何其他进程读取或修改。然后，您的进程可以安全地修改这些记录而无需担心冲突。当您的进程完成时，数据库系统会释放锁。

例如，考虑前面的第二种场景，其中两个用户并发地更新`单独佣金`区域。清单 7-5 的步骤 2 包含有问题的 SQL 查询：

```
select Comm into v_prevCommission
from EMP
where EmpNo = :P18_EMPNO;
```

此查询的输出是指定`EMP`记录的`Comm`值，该值将在步骤 3 中稍后修改。为防止其他进程访问此记录，您可以通过将查询更改为以下形式来锁定它：

```
select Comm into v_prevCommission
from EMP
where EmpNo = :P18_EMPNO for update;
```


如果另一个进程试图访问此记录，而该记录已被锁定，数据库系统将延迟该进程，直到您的进程完成。因此，您进程的第 3 步可以修改该记录，确信不会覆盖另一个进程的修改。

转到`Employee Demo`应用程序的第 18 页，按此策略修改`Solo Commission`区域的进程。然后使用两个私密浏览器窗口验证其是否有效，如前一节所述。即使在进程的第 3 步之前添加`sleep`语句，页面也应永远不会丢失已更新的佣金。

### 并发更新场景

现在考虑第一个场景，即两个用户同时更新一个员工记录。在此场景中，更新记录的活动需要两个进程：`GetEmpInfo`进程从指定记录读取值，而`UpdateEmp`进程修改该记录。仅让`GetEmpInfo`锁定其记录是不够的，因为锁会在`GetEmpInfo`完成时释放（因此，在`UpdateEmp`开始之前）。

因此，这两个进程之间需要显式的协作。以下是一个好的策略：

`GetEmpInfo`进程读取所选记录的值，在页面上显示它们，并保存原始值以备将来参考。`UpdateEmp`进程从数据库重新读取值，并将其与保存的值进行比较。如果它们匹配，则进程可以执行更新。如果不匹配，则必定有另一个进程更新了该记录，这意味着`UpdateEmp`必须拒绝执行更新。

`Employee Demo`应用程序的第 19 页名为`Revised Employee Data Entry`，包含此策略的实现。它看起来与第 17 页相同，行为也相同。唯一的（目前）区别是其`GetEmpInfo`和`UpdateEmp`进程已修订，因此它们不会丢失更新。

### 使用哈希函数

尽管此修订代码有效，但该技术扩展性不佳。需要保存和比较所有原始值很繁琐，并且随着保存值数量的增加会变得更加繁琐。一种更简单的方法是将所有原始值组合成一个值；然后您可以保存并比较该单一值。首选技术是使用`hash function`，它将一组值转换为一个单一的固定长度值，称为`checksum`。哈希函数还会混淆其输入，这使其可用于编码用户密码——如第 13 章将看到的。

APEX 有一个内置的哈希函数，名为`apex_util.get_hash`。该函数的输入是一个包含您想要哈希的值的集合。您应该使用 APEX 内置类型`apex_t_varchar2`来保存此集合。作为一个说明性示例，考虑以下代码：

```
v_vals   apex_t_varchar2 := apex_t_varchar2('CLERK', 20);
v_result varchar2(1000)  := apex_util.get_hash(v_vals);
```

代码的第一行声明了一个类型为`apex_t_varchar2`的变量`v_vals`，并为其分配了由该类型的构造函数函数（也称为`apex_t_varchar2`）创建的集合。构造函数函数的参数是值`'CLERK'`和`20`。代码的第二行声明了一个类型为`varchar2(1000)`的变量`v_result`，并为其分配了`apex_util.get_hash`函数的结果。换句话说，变量`v_result`将包含值['CLERK', 20]的哈希值。


### 使用哈希函数增强数据完整性

您应该使用此哈希函数来修订 `GetEmpInfo` 进程的代码，以便将三个可更新的值编码为单个哈希值。此哈希值随后保存在隐藏项 `P19_HASH`（您需要创建此项）中。此代码出现在清单 7-9 中（粗体行是新增的）。

```sql
declare
  v_valuesToHash apex_t_varchar2;
begin
  -- 首先检索数据。
  select Job, Sal, DeptNo
  into :P19_UPDATE_JOB,
       :P19_UPDATE_SAL,
       :P19_UPDATE_DEPTNO
  from EMP
  where EmpNo = :P19_UPDATE_EMPNO;

  -- 然后保存这些值的哈希值。
  v_valuesToHash := apex_t_varchar2(:P19_UPDATE_JOB,
                                    :P19_UPDATE_SAL,
                                    :P19_UPDATE_DEPTNO);
  :P19_HASH := apex_util.get_hash(v_valuesToHash);
end;
```

`UpdateEmp` 进程的代码也应进行重新修订，使其重新从数据库读取可更新值，计算其哈希值，并将该哈希值与保存的哈希值进行比较。此代码出现在清单 7-10 中（粗体行是新增的）。

```sql
  v_valuesToHash apex_t_varchar2;

  into v_valuesToHash
  from EMP

  if :P19_HASH = apex_util.get_hash(v_valuesToHash)
  then

           Sal    = :P19_UPDATE_SAL,
           DeptNo = :P19_UPDATE_DEPTNO
    where  EmpNo = :P19_UPDATE_EMPNO;
  else
    -- 记录已更改，因此中止。
    raise_application_error(-20000,
                            '记录已过期。请重新获取。');
  end if;
end;
```

### 条件进程

## 单行视图页面

员工演示应用程序的第 20 页被称为 `Single Row View`，如图 7-17 所示。该页面包含两个区域，其工作方式类似于交互式报表的单行视图。`Employees` 区域是 `EMP` 记录的经典报表。用户可以通过点击其第一列中的链接来选择一行；APEX 会将该行的值放入 `Emp Info` 区域的项中。`Emp Info` 区域有三个按钮：底部的箭头按钮选择上一个或下一个记录，顶部的 `Report View` 按钮将控制权传回报表。

图 7-17 单行视图页面

尽管图 7-17 同时显示了 `Employees` 和 `Emp Info` 区域，但它们实际上永远不会以这种方式显示。`Employees` 区域仅在没有选定记录时显示，而 `Emp Info` 区域仅在有选定记录时显示。

当您尝试实现此页面时，会出现几个问题：`Emp Info` 区域中的项如何获取其值？页面如何知道是显示 `Employees` 区域还是 `Emp Info` 区域？箭头按钮是如何实现的？这些问题将在以下各节中讨论。

```sql

from EMP
order by EName
```

您应将 `EmpNo` 列的类型设置为 `Link`。要配置此链接，请单击该列的 `Target` 属性以调出 `Link Builder` 向导屏幕。图 7-18 显示了我填写后的屏幕。它指定当选择链接时，APEX 将把名为 `P20_EMPNO` 的项设置为选定的员工编号，并重定向到同一页面。为此，您当然需要将 `P20_EMPNO` 创建为隐藏项。

一旦构建了 `Emp Info` 区域，下一步就是编写一个进程来检索 `P20_EMPNO` 中指定的员工的值，并将它们放入 `Emp Info` 的项中。转到页面编辑器的 `Processing` 选项卡，创建一个名为 `FetchRow` 的进程，其 PL/SQL 代码如清单 7-15 所示。此代码类似于清单 7-4，即 `Employee Data Entry` 页面上 `GetEmpInfo` 按钮关联的代码。

```sql
begin
if :P20_EMPNO is not null then
select EName, Job, Sal, DeptNo
into :P20_ENAME, :P20_JOB, :P20_SAL, :P20_DEPTNO
from EMP
where EmpNo = :P20_EMPNO;
end if;
end;
```

清单 7-15 FetchRow 进程的 PL/SQL 代码

尽管此进程代码正确，但它的执行点错误。`Processing` 执行点意味着进程在发生提交操作时执行。但 `Employees` 区域不执行提交；而是调用链接，而链接执行重定向。您需要的是进程在重定向发生后执行——即在页面渲染期间。

在页面设计器中单击 `FetchRow` 的节点，并查看其 `Execution Options` 部分。`Point` 属性指定了进程的执行点。图 7-20 向您展示了可能的值。

![../images/335103_3_En_7_Chapter/335103_3_En_7_Fig20_HTML.jpg](img/335103_3_En_7_Fig20_HTML.jpg)

图 7-20 可能的进程执行点

如果一个进程的执行点是 `New Session`，它只会在会话开始时执行。从 `Before Header` 到 `After Footer` 的执行点表示页面渲染的六个阶段；分配到其中一个点的进程将在指定的渲染阶段执行。执行点 `After Submit` 和 `Processing` 是提交操作的两个阶段。如果您希望进程在验证阶段之前执行，则将其分配给 `After Submit` 点；如果您希望它在验证之后执行，则分配给 `Processing`。

尽管这些选项对于高级应用程序很有用，但许多执行点之间的差异很微妙，并且在大多数情况下无关紧要。就我们的目的而言，唯一有趣的区别在于在提交期间执行的进程和在渲染期间执行的进程之间。特别是，本书中的所有进程在页面渲染期间执行时都具有 `After Header` 执行点，而在提交期间执行时具有 `Processing` 执行点。

现在将注意力转回 `FetchRow` 进程，并将其执行点更改为 `After Header`。您应该观察到该进程的节点将从处理树中消失并重新出现在渲染树中，如图 7-21 所示。这是合理的，因为渲染树包含页面渲染期间发生的所有内容，其中包括进程。此图还显示 `After Header` 执行点是一个**预渲染**点——也就是说，它在区域渲染之前发生。这很重要，因为它确保 `FetchRow` 将在页面项被渲染之前为它们赋值。

![../images/335103_3_En_7_Chapter/335103_3_En_7_Fig21_HTML.jpg](img/335103_3_En_7_Fig21_HTML.jpg)

图 7-21 FetchRow 现在是一个渲染进程


`FetchRow` 进程会在页面每次渲染时被调用。但是，除非用户已选择了一行，否则让它去查询数据库是没有意义的——也就是说，它应该仅当 `P20_EMPNO` 非空时才执行其 SQL 语句。这正是清单 7-15 中 if 语句的目的。（一个同样好的等效解决方案是移除 if 语句，并更改进程的 `服务器端条件` 属性，使其仅在 `P20_EMPNO` 不为空时执行。）
与其将 `FetchRow` 创建为提交进程然后再更改其执行点，不如直接将其创建为渲染进程更为简便。你只需在渲染树中右键单击 `After Header` 节点，然后选择 `创建进程`。

### 在区域间切换
在你应用程序的这个阶段，你应该验证点击 `Employees` 报表上的链接是否能正确地将所选员工的值放入 `Emp Info` 的各项中。接下来的问题是要制定一个计划，让这些区域如何能“轮流”渲染自身。
关键思路是认识到 `P20_EMPNO` 的非 `空` 值表示已选择了一行。因此，要“取消选择”一行，自然地应该将 `空` 值放入 `P20_EMPNO`。这样，两个区域便可以利用该项的值来决定它们的条件渲染——当该项值为 `空` 时，`Employees` 区域应可见；当值不为 `空` 时，`Emp Info` 区域应可见。换句话说，`P20_EMPNO` 的值决定了页面是处于 *报表模式* 还是 *单行模式*。
这一思路也引出了实现 `ReportView` 按钮的自然方式。它应该简单地重定向到同一页面，并将 `P20_EMPNO` 设置为 `空`。进行这些更改，并再次测试你的页面。点击报表中的一行应显示该员工的 `Emp Info` 区域，而点击 `Report View` 按钮应再次显示报表。

### 查找上一条和下一条记录
最后一个问题是实现 `Previous` 和 `Next` 按钮，使得点击它们时，区域会显示报表中的上一条（或下一条）记录。本节考虑两种不同的实现此功能的方法。第一种方法更容易理解，但有一些局限性。第二种方法理解起来稍难一些，但优雅得多。

#### 方法一：分别为上一条和下一条使用不同进程
考虑 `Previous` 按钮，它需要选择上一条记录。我们暂时假设“上一条记录”意味着拥有次低员工编号的记录。清单 7-16 中的 PL/SQL 代码可以完成这项工作。

```sql
declare
  v_prevEmp number(4,0);
begin
  select max(EmpNo) into v_prevEmp
  from EMP
  where EmpNo < :P20_EMPNO;
  if v_prevEmp is not null then
    :P20_EMPNO := v_prevEmp;
  end if;
end;
```
*清单 7-16 方法一的“上一条”按钮简化代码*

if 语句对于处理尝试获取第一个员工的上一条记录的情况是必需的。在这种情况下，查询将返回一个 `空` 值。如果没有 if 语句，按钮会将该 `空` 值放入 `P20_EMPNO`，这会导致取消选择行并返回到报表模式。通过使用 if 语句，从第一行点击 `Previous` 按钮将不会有任何效果。

现在假设你想使用不同的列来确定上一条记录，例如 `EName`。前面的查询不容易扩展到这种情况。相反，你应该使用 SQL 分析函数 `lag`，它根据指定的排序顺序查找前一个值。清单 7-17 给出了相应的 PL/SQL 代码。

```sql
declare
  v_prevEmp number(4,0);
begin
  select PrevEmp into v_prevEmp
  from (select EmpNo, lag(EmpNo) over (order by EName) as PrevEmp
        from EMP)
  where EmpNo = :P20_EMPNO;
  if v_prevEmp is not null then
    :P20_EMPNO := v_prevEmp;
  end if;
end;
```
*清单 7-17 方法一的“上一条”按钮 PL/SQL 代码*

内部查询创建了一个包含两列员工编号的表，其中每个员工都与排序中紧接在其前面的员工配对。外部查询然后从该表中选择 `EmpNo` 值等于 `P20_EMPNO` 的行，并返回该记录的 `PrevEmp` 值。
除了 `lag` 函数，SQL 还有 `lead` 函数，其行为方式相同，只是它返回排序顺序中的下一个更高值。此函数可用于实现 `Next` 按钮，其 PL/SQL 代码与清单 7-17 类似（因此省略）。
因为 `Previous` 和 `Next` 按钮各自有自己的进程，你应该将它们的动作设置为 `提交`。你还应将每个进程配置为根据其按钮有条件地执行。虽然这不是此问题的最佳解决方案，但它运行良好，值得实现以查看其实际效果。

#### 方法二：统一的“上一条/下一条”进程
方法一的缺点是，当 `Previous` 和 `Next` 按钮到达第一条或最后一条记录时，它们没有任何可见效果；对用户来说，它们似乎停止工作了。另一个想法是在按钮不适用时将其隐藏。为此，页面需要“向前查看”。也就是说，除了跟踪当前员工编号外，页面还需要跟踪上一个和下一个值。这样，如果上一个值为 `空`，它就可以隐藏 `Previous` 按钮，对 `Next` 按钮同理。

要实现此方法，创建隐藏项 `P20_PREV` 和 `P20_NEXT` 来保存上一个和下一个员工编号。然后，每当选择一条记录时，一个进程可以“向前查看”并将适当的值赋给这些项。这个名为 `FindPreviousNext` 的进程如清单 7-18 所示。

```sql
begin
  if :P20_EMPNO is not null then
    select PrevEmp, NextEmp
      into :P20_PREV, :P20_NEXT
      from (select EmpNo,
                   lag(EmpNo)  over (order by EName) as PrevEmp,
                   lead(EmpNo) over (order by EName) as NextEmp
              from EMP)
     where EmpNo = :P20_EMPNO;
  end if;
end;
```
*清单 7-18 用于查找上一条和下一条值的 PL/SQL 代码*

此代码与清单 7-17 中的代码相似，区别在于它不特定于任何特定按钮。实际上，它必须在选择一行时执行，无论该选择是通过报表中的链接还是通过 `Previous` 或 `Next` 按钮之一进行的。换句话说，`FindPreviousNext` 进程应该在页面渲染期间执行，就像 `FetchRow` 进程一样。

因此，`Previous` 和 `Next` 按钮的配置可以简单得多。考虑 `Previous` 按钮（`Next` 按钮类似）。因为 `P20_PREV` 保存着上一条记录的 `EmpNo`，你可以将按钮配置为在被点击时重定向，将 `P20_PREV` 的值放入 `P20_EMPNO`。参见图 7-22。

![../images/335103_3_En_7_Chapter/335103_3_En_7_Fig22_HTML.jpg](img/335103_3_En_7_Fig22_HTML.jpg)
*图 7-22 配置“上一条”按钮的动作*

你也可以使用 `P20_PREV` 和 `P20_NEXT` 来自定义何时显示 `Previous` 和 `Next` 按钮。具体来说，当 `P20_PREV` 不为空时应显示 `Previous` 按钮，当 `P20_NEXT` 不为空时应显示 `Next` 按钮。这些条件可以使用条件渲染来表达。例如，图 7-23 显示了 `Previous` 按钮的 `服务器端条件` 部分。

### 方案总结

总结来说，方案 2 的实现“上一个/下一个”功能的方式如下：首先创建隐藏项 `P20_PREV` 和 `P20_NEXT`。然后创建名为 `FindPreviousNext` 的过程，它使用清单 7-18 的代码计算这两个项的值，并在页面呈现时执行。

最后，配置 `Previous` 和 `Next` 按钮，使其重定向到当前页面，并为 `P20_EMPNO` 赋值。`Previous` 按钮使用 `P20_PREV` 的值，而 `Next` 按钮使用 `P20_NEXT` 的值。每个按钮都使用条件渲染，仅当其关联的项值不为 `null` 时才显示。

结果是，整个页面由两个相对较小的过程组成：一个获取所选行的值，另一个计算上一条和下一条记录的雇员号。两个过程都在页面呈现时执行。用户可以通过点击报表链接或 `Previous`/`Next` 按钮来选择一行。无论哪种情况，页面都会确定所选记录的雇员号，将其保存到项 `P20_EMPNO` 中，并进行重定向，从而触发这两个过程的执行。整个过程无需任何提交操作。

最初，页面以报表模式显示 `Employees` 区域。从那里，用户可以点击 `Insert` 按钮进入插入模式，或者点击某个 `Edit` 链接进入更新模式。

在插入模式下，页面显示 `Insert Employee` 区域。该区域与 `Employee Data Entry` 页面中的相应区域基本相同。它的 `Create` 按钮创建一个具有指定值的新 `EMP` 记录，然后清除各项的值，为下一次插入做准备。`Cancel` 按钮返回到报表模式。

在更新模式下，页面显示 `Update/Delete Employee` 区域。该区域的各项最初填充了所选雇员的信息。用户可以更改区域中的项值，然后点击 `Update` 按钮来更新数据库。或者，点击 `Delete` 按钮会删除所选记录。删除记录后也会返回到报表模式，因为没有记录可再显示。点击 `Cancel` 按钮只是简单地返回到报表模式。

现在你了解了页面的工作原理，可以考虑如何实现每个区域了。以下小节将探讨这些问题。

### 实现报表区域

报表区域的实现与图 7-17 的 `Single Row View` 页面基本相同。一个区别是它还有一个名为 `P21_MODE` 的隐藏项，用于区分三种模式。该项的值将是 `Report`、`Insert` 和 `Update`（视情况而定），`null` 值表示报表模式。

`EmpNo` 列被实现为一个链接。该链接重定向到第 21 页，并在重定向时设置两个项的值：`P21_EMPNO` 设置为所选雇员号，`P21_MODE` 设置为 `Update`。`Insert` 按钮也会进行重定向，将 `P21_MODE` 设置为 `Insert`。该区域根据以下 SQL 表达式进行条件渲染：

```
:P21_MODE = ‘Report’ or :P21_MODE is null
```

### 实现插入区域

一条新的 `EMP` 记录有九个值，但用户只需指定其中五个。`Insert Employee` 区域为这五个值各设置了一个项。这些项被命名为 `P21_INSERT_ENAME`、`P21_INSERT_JOB` 等。该区域还有两个按钮，分别名为 `Create` 和 `CancelInsert`。区域根据以下 SQL 表达式进行条件渲染：

```
:P21_MODE = 'Insert'
```

`Create` 按钮执行提交操作。它有一个关联的过程，名为 `InsertEmp`，该过程执行两件事：执行一条 SQL 插入命令，并将其项设置为 `null`。它对 `HireDate`、`Comm` 和 `Office` 使用默认值。其代码见清单 7-19。

### 实现更新/删除区域

`Update/Delete Employee` 区域中的各项由一个名为 `FetchRowForUpdate` 的过程填充，该过程在页面呈现时执行，并基于 SQL 表达式进行条件判断。

```
:P21_MODE = 'Update'
```

该过程的代码见清单 7-20。该代码与清单 7-15 中的 `FetchRow` 过程基本相同，只是它还处理了丢失更新的问题。（此问题与 `Single Row View` 页面无关，因为该页面不执行任何更新。）它计算三个可修改项的哈希值，并将其保存在隐藏项 `P21_HASH` 中。

```
declare
v_valuesToHash apex_t_varchar2;
begin
if :P21_EMPNO is not null then
-- First, fetch the employee record and save its values.
select EName, Job, Sal, DeptNo
into :P21_UPDATE_ENAME,
:P21_UPDATE_JOB,
:P21_UPDATE_SAL,
:P21_UPDATE_DEPTNO
from EMP
where EmpNo = :P21_EMPNO;
-- Then save the hash of the updatable values.
v_valuesToHash := apex_t_varchar2(:P21_UPDATE_JOB,
:P21_UPDATE_SAL, :P21_UPDATE_DEPTNO);
:P21_HASH := apex_util.get_hash(v_valuesToHash);
end if;
end;
```

该区域的 `Delete` 按钮的 PL/SQL 代码名为 `DeleteEmp`。它执行一条 SQL `delete` 命令，类似于清单 7-1，然后返回到报表模式。其代码见清单 7-21。

## 总结

过程是一种页面组件，它使您能够访问数据库并执行计算。本章探讨了过程在应用程序中所扮演的角色。例如，执行提交操作的按钮通常有一个关联的过程，该过程将在按钮被点击时执行。过程也可以配置为在页面呈现时执行，并计算项的初始值。

本章完全集中于 PL/SQL 过程。要指定 PL/SQL 过程的行为，您需要编写 PL/SQL 代码。PL/SQL 将 SQL 命令直接集成到通用编程语言中，这使得编写数据库感知代码变得容易。第 10 章将探讨其他过程类型。

尽管 PL/SQL 过程可能非常有用，但它们需要谨慎编码。本章探讨了一个常见的陷阱，称为**丢失更新问题**。您学习了如何识别该问题以及如何编写必要的 PL/SQL 代码来避免它。
