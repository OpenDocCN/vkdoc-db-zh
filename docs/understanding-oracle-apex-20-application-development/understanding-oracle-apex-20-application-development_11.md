# 3. 区域

页面的内容被划分为称为*区域*的矩形区域。例如，考虑图 1-3 所示的 APEX SQL Workshop 主屏幕，它显示了七个区域。其中六个区域有标题：`Recently Created Tables`、`Recent SQL Commands`、`Recent SQL Scripts`、`About`、`Schema` 和 `Create Object`。第七个区域包含五个大按钮，没有标题。

每个区域都有一个*类型*，表示它可以包含何种内容。图 1-3 中的区域属于三种不同的区域类型：

*   `Recently Created Tables`、`Recent SQL Commands` 和 `Recent SQL Scripts` 区域包含报表，其类型为 `Classic Report`。
*   `Create Object` 区域包含一个链接列表，其类型为 `List`。
*   `About`、`Schema` 以及那个无标题区域的类型为 `Static Content`。

一个区域也可以包含*控件*——即项目和按钮。在图 1-3 中，`Schema` 区域包含一个选择列表项目和一个按钮，而无标题区域包含五个大按钮。由于 APEX 将区域的控件视为与其内容不同的部分，因此任何区域，无论类型如何，都可以包含控件。本章重点介绍区域的内容；控件将在第 5 章和第 6 章讨论。

APEX 支持多种不同的区域类型。本章涵盖四种基本类型：`Static Content`、`Classic Report`、`Chart` 和 `Interactive Report`。后续章节将介绍其他区域类型：第 4 章的 `Breadcrumb` 和 `List`，第 6 章的 `Region Display Selector`，第 10 章的 `Form`，以及第 11 章的 `Interactive Grid`。

## 布局部分

一个区域的 `Layout`（布局）部分包含指定该区域在其页面上位置的属性。这些属性如图 3-8 所示。
![../images/335103_3_En_3_Chapter/335103_3_En_3_Fig8_HTML.jpg](img/335103_3_En_3_Fig8_HTML.jpg)
图 3-8
区域的默认布局部分

`Parent Region`（父区域）和 `Position`（位置）属性提供了两种不同的方式来指定区域在页面上的位置。这两个属性中只能有一个有值。`Parent Region` 有值意味着该区域应位于一个指定的区域（称为其 `parent`，父区域）内。`Position` 有值意味着该区域应位于页面上的一个特定位置。默认情况下，新区域位于 `Content Body`（内容主体）位置，如图 3-8 所示。您可以通过从其下拉列表中选择来更改位置，如图 3-9 所示。
![../images/335103_3_En_3_Chapter/335103_3_En_3_Fig9_HTML.jpg](img/335103_3_En_3_Fig9_HTML.jpg)
图 3-9
可能的区域位置

这些位置按它们在页面上从上到下的出现顺序列出。大多数位置都有特定的用途。例如，`Content Body` 位置是通用区域的主要放置位置，`Page Navigation`（页面导航）位置用于页面左侧的导航菜单，而 `Breadcrumb Bar`（面包屑栏）用于面包屑区域。

注意：其中一些位置（即 `After Header`、`Body 1`、`Body 2`、`Body 3`、`Footer` 和 `Before Footer`）是 `legacy`（遗留）位置。也就是说，它们仅用于支持使用旧版 APEX 编写的应用程序。新应用程序不应使用它们。

可能的区域位置集合由页面的 `template`（模板）决定。创建页面时，会为其分配应用程序主题定义的默认模板。例如，`Universal` 主题的默认页面模板称为 `Standard`。

可以更改页面的模板。在渲染树中选择页面组件，然后查看 `Appearance`（外观）部分中的 `Page Template`（页面模板）属性。当前，模板设置为 `Theme Default`（主题默认）。您可以通过点击该属性的选择列表来更改模板。例如，将 `Region Practice`（区域练习）页面的模板更改为 `Right Side Column`（右侧栏），如图 3-10 所示。
![../images/335103_3_En_3_Chapter/335103_3_En_3_Fig10_HTML.jpg](img/335103_3_En_3_Fig10_HTML.jpg)
图 3-10
页面模板选项

现在选择一个区域（例如 `My Second Region`，我的第二个区域）并重新检查其 `Position` 值列表；您会发现除了图 3-9 中的值外，还有一个值 `Right Column`（右栏）。将 `My Second Region` 的位置设置为 `Right Column` 并重新运行页面。该区域现在应显示为右侧栏，如图 3-11 所示。（我为第二个区域的源添加了更多文本以使其更像侧边栏。）请注意，侧边栏区域在其左上角有一个箭头图标。点击该图标可在显示和隐藏侧边栏之间切换。
![../images/335103_3_En_3_Chapter/335103_3_En_3_Fig11_HTML.jpg](img/335103_3_En_3_Fig11_HTML.jpg)
图 3-11
向右侧栏添加区域

`Layout` 部分还有一个 `Sequence`（序列）属性，可以在图 3-8 中看到。如果两个区域位于页面的同一位置，它们的 `Sequence` 值决定了它们在该位置中的放置顺序。区域在创建时按顺序分配序列号，但您只需为 `Sequence` 分配不同的值即可更改该顺序。例如，`My First Region`（我的第一个区域）和 `My Third Region`（我的第三个区域）都位于 `Content Body` 位置，序列号分别为 `10` 和 `30`。如果您将 `My Third Region` 的序列号更改为 `5`，则渲染后的页面将类似于图 3-11，只不过 `My Third Region` 将出现在 `My First Region` 上方。

尽管尝试将区域放置在页面上的不同位置是很好的，但页面上的大多数区域通常位于同一位置（特别是 `Content Body`）。在这种情况下，`Layout` 部分的其余属性决定了区域在该位置内的相对位置。

一个位置中的区域被组织在网格中；每个区域被分配给该网格的一行和一列。APEX 根据区域的序列号来定位它们。第一个区域被放置在网格第一行的第一列。然后，每个后续区域按以下方式之一放置：

*   与前一个区域位于同一行和同一列（在这种情况下，它显示在前一个区域下方）
*   在当前行的下一列
*   在新行的第一列

`Layout` 部分的 `Start New Row`（开始新行）和 `New Column`（新列）属性允许您指定使用哪个选项。图 3-12 显示了一个被放置在同一行新列中的区域的这些属性。图 3-8 显示了一个区域开始新行的情况；请注意，在这种情况下，`New Column` 属性是不必要的，因此没有出现。
![../images/335103_3_En_3_Chapter/335103_3_En_3_Fig12_HTML.jpg](img/335103_3_En_3_Fig12_HTML.jpg)
图 3-12
指定行和列的区域定位

在将每个区域分配给一行和一列之后，APEX 会确定这些区域内位置中的大小和位置。为此，APEX 使用 `Column`（列）和 `Column Span`（列跨度）属性的值（如图 3-12 所示）来确定行中每个区域的偏移量和宽度。

`Column` 属性指定区域在其行内的偏移量。APEX 将一个位置的宽度划分为 12 个“网格点”；`Column` 的值是一个从 1 到 12 的数字，表示其中一个点。值为 `1` 表示区域从位置的极左侧开始；值为 `7` 表示区域从位置的中心开始。`Column Span` 属性指定列的宽度。其值也是一个从 1 到 12 的数字，表示以网格点为单位的宽度。

最常见的是 `Standard` 模板，它在顶部的边框中显示区域的标题。其他许多模板都有特殊用途，例如 `Alert`、`Inline Dialog`、`Login`、`Title Bar` 和 `Wizard Container`。

网格布局在设计时为你提供了渲染页面布局的视觉近似。当你更改区域的 `Position` 和 `Layout` 属性值时，网格布局也会相应改变。

如果在网格布局中点击区域的标题栏，该区域的属性将出现在属性编辑器中。你还可以将区域拖动到网格布局上的其他位置。拖动时，可用位置会以黄色高亮显示，这些区域内的放置点会由一个轮廓矩形指示。你可以将区域放置到空位置、另一个区域的 `Sub Regions` 部分（使该区域成为另一个区域的子区域），或者另一个区域的上方、下方、左侧或右侧（将该区域添加到该位置的网格中）。当你这样做时，你会看到区域的 `Layout` 属性相应地发生变化。

### 格式化文本与非格式化文本

静态内容区域的来源是要显示的文本。APEX 可以以两种方式解释此文本：一种是按书写原样显示的字符，另一种是需要格式化的 HTML 代码。指定此选项的属性称为 `输出为`，它出现在页面渲染树中区域 `属性` 节点的 `设置` 部分。图 3-19 展示了此部分以及 `输出为` 属性的两个可能值。

![../images/335103_3_En_3_Chapter/335103_3_En_3_Fig19_HTML.jpg](img/335103_3_En_3_Fig19_HTML.jpg)
图 3-19：静态内容区域的类型特定属性

`输出为` 的值为 `HTML` 指定该区域的源包含 HTML 代码。APEX 在渲染页面时会按原样将此代码发送到浏览器，以便浏览器可以相应地格式化源。

### 替换字符串

APEX 服务器有几个内置变量，其中包含有用的信息。以下是三个有用的变量：

*   变量 `APP_USER` 保存当前登录用户的用户名。如果应用程序不要求用户登录，则该变量的值为 `APEX_PUBLIC_USER`。
*   变量 `APP_ID` 保存当前应用程序的 ID。
*   变量 `APP_SESSION` 保存当前会话的 ID。

要从 HTML 文本中引用变量的值，您在变量名前添加字符 `&` 并在后附加句点字符 (`.`)。这样的表达式称为替换字符串。换句话说，名为 `X` 的变量的替换字符串写作 “`&X`.” 。当 APEX 服务器渲染页面时，它会用该变量的值文本替换该替换字符串。

例如，替换字符串 `&APP_USER.` 出现在 `演示应用` 区域的源中。当 APEX 服务器渲染页面时，它会用变量 `APP_USER` 的值（在本例中为 `SCIORE@BC.EDU`）文本替换该替换字符串。

```
代码清单 3-3：参考资料区域的源
```

HTML `<a>` 标签指定一个链接；其参数是目标位置的 URL。清单中的第一个链接是一个典型链接，它引用外部网站上的一个页面。第二个链接引用另一个 APEX 应用程序的一个页面——特别是 `asktom.oracle.com` 上 APEX 服务器中应用程序 100 的第 1 页。请注意该链接如何使用“f?p=”表示法，如第 2 章所述。

## 值列表

考虑格式化报表中的 `DEPTNO` 列。它显示来自 `DEPT` 表的值，尽管报表只提到了 `EMP` 表。这是怎么回事？答案是列中的值没有改变；它们仍然是部门编号。但是，你可以建立部门名称与编号之间的对应关系，并告诉 APEX 显示给定部门编号对应的名称。返回值（部门编号）与其显示值（部门名称）之间的这种对应关系称为 *值列表*。

```sql
select DName, DeptNo
from DEPT
```

这个表达式需要一些解释。首先，考虑一下您会为名为`FORD`的员工编写什么样的 HTML 表达式。要使名字变成大号粗体，可以将其放在`<b><big></big></b>`标签内。生成的表达式如下所示：

```
<b><big>FORD</big></b> the ANALYST
```

这个表达式几乎就是`HTML 表达式`值应有的样子——唯一的区别是员工的姓名和职位需要根据报表的每一行而变化。APEX 通过以下约定使其成为可能：对于名为`X`的列，术语`#X#`表示当前行中`X`的值。因此，您应该将上述表达式中的`FORD`替换为`#ENAME#`，将`ANALYST`替换为`#JOB#`，从而得到图 3-33 中的表达式：

```
<b><big>#ENAME#</big></b> the #JOB#
```

请注意，表达式`#JOB#`是有意义的，即使`JOB`列在报表中不可见。实际上，在报表中设置不可见列的常见原因之一，就是为了能在报表的其他地方使用它们的值。

### 链接列

图 3-24 中的`HIREDATE`值被格式化为链接。具体来说，点击一个雇佣日期（例如`Dec 03, 1981`）会将浏览器重定向到 URL [`http://www.infoplease.com/year/1981.html`](http://www.infoplease.com/year/1981.html)，该页面提供了当年的时事信息。

实现此功能的一种方法是用适当的 HTML 表达式包装雇佣日期。例如，1981 年 12 月 3 日的链接将是：

```
<a href="http://www.infoplease.com/year/1981.html" target="_blank">#HIREDATE#</a>
```

请注意，`<a>`标签的第二个参数是`target="_blank"``。这个参数导致目标页面在新的浏览器标签页中打开，这在本例中似乎是合理的。这个 HTML 表达式的问题在于，无论雇佣日期是什么，它总是重定向到 1981 年的页面。我们需要一种方法从雇佣日期中提取雇佣年份，不幸的是，这在 HTML 中无法实现。

然而，在 SQL 查询中提取年份是可行的。思路是让 SQL 源查询计算一个新列，其值为雇佣年份。以下查询可以完成这项工作：

```sql
select e.*, extract(year from e.HireDate) as HireYear
from EMP e
```

使用这个新的源查询的一个结果是，`HIREYEAR`现在将成为报表的一列。由于您不希望它在报表中显示，您应该将其类型设置为`隐藏列`。

将`格式化报表`区域的源修改为此查询后，您可以通过在其`HTML 表达式`属性中使用`#HIREYEAR#`来格式化`HIREDATE`列，如下所示：

```
<a href="http://www.infoplease.com/year/#HIREYEAR#.html" target="_blank">#HIREDATE#</a>
```

尽管此技术有效，但将列格式化为链接的更好方法是将其`类型`属性设置为`链接`。这样做会使属性编辑器显示一个`链接`部分。此部分包含`目标`属性，其初始值为`未定义链接目标`。点击该值会打开一个模态的`链接生成器`页面，在其中您可以指定目标是 APEX 应用程序的页面还是任意 URL。在本例中，选择`URL`并在出现的文本框中输入目标 URL，如图 3-34 所示。然后单击确定按钮返回属性编辑器。

## 条件格式化

回想一下，图 3-24 将超过 2500 的薪资显示为红色。使用`HTML 表达式`属性将`SAL`值格式化为特定颜色是相当容易的。例如，以下表达式使所有薪资值变为红色：

```
<font color="red">#SAL#</font>
```

但是，如何编写一个表达式使得只有部分值变为红色呢？因为 HTML 本身无法执行计算，您必须再次依赖 SQL 来完成。思路是让 SQL 源查询计算另一个新列，名为`SalColor`，其值为薪资值的颜色。以下是所需的查询（新代码以粗体显示）：

```sql
select e.*, extract(year from e.HireDate) as HireYear,
case when e.Sal>2500 then 'red' else 'black' end as SalColor
from EMP e
```

将`格式化报表`区域的源更改为使用此查询后，您可以通过将以下代码分配给`SAL`列的`HTML 表达式`属性来格式化该列：

```
<font color="#SALCOLOR#">#SAL#</font>
```

您可以对`OFFSITE`列应用相同的技术。在格式化报表中，`Offsite`值如果其值为**Y**则显示绿色对勾图片，否则显示红色对勾图片。假设这两张图片已保存为 APEX 应用程序文件`checkgreen.jpg`和`checkred.jpg`。（您可以从 Apress 网站下载这些文件。）那么，值为**Y**时应格式化为以下 HTML 表达式：

```
<img src="#IMAGE_PREFIX#checkgreen.jpg">
```

值为**N**时应格式化为：

```
<img src="#IMAGE_PREFIX#checkred.jpg">
```

区分这两个文件名的方法是扩展 SQL 源查询，使其计算一个新列，名为`OffsiteImage`，其值为`'checkgreen.jpg'`或`'checkred.jpg'`。SQL 查询现在如下所示（新代码以粗体显示）：

```sql
select e.*, extract(year from e.HireDate) as HireYear,
case when e.Sal>2500 then 'red' else 'black' end as SalColor,
case when e.Offsite = 'Y' then 'checkgreen.jpg'
else 'checkred.jpg' end as OffsiteImage
from EMP e
```

将报表的源更改为该查询后，您可以通过将以下代码分配给`OFFSITE`列的`HTML 表达式`属性来格式化该列：

```
<img src="#IMAGE_PREFIX##OFFSITEIMAGE#">
```

最后，您应该将`SALCOLOR`和`OFFSITEIMAGE`列的类型设置为`隐藏列`。

## 使用源查询进行格式化

尽管前面的章节重点介绍了如何使用 APEX 格式化源查询的行，但重要的是要意识到，通过使用更精确的源查询，你也可以获得许多相同的功能。以下是`格式化报表`演示区域的一些示例。

与其隐藏一列，不如直接在查询中不包含它可能更合适。特别是，`EmpNo`列在格式化报表中没有作用，最好完全排除在其源查询之外。另一方面，`Job`、`SalColor`、`OffsiteImage`和`HireYear`列是格式化其他列所必需的，因此它们必须出现在源查询中，并且必须是`隐藏列`类型。

与其在属性编辑器中更改列的标题，不如在源查询中更改列的名称。一个例子是`EName`列，它被重命名为`Name`。替代方法是在源查询中使用`AS`关键字，如下所示：

```
select EName as Name, ...
from EMP
```

如果你不希望用户能够以交互方式更改记录的排序顺序，在源查询中指定排序顺序比单独设置每个列的可排序性属性更容易。

与其使用值列表来格式化`DeptNo`列，你可以修改源查询，使其成为`EMP`和`DEPT`表的连接，如下所示：

```
select e.EName as Name, ..., d.DName, ...
from EMP e join DEPT d
on e.DeptNo = d.DeptNo
```

在这种情况下，很难说这两种方法哪个更好。值列表方法需要 APEX 为报表的每一行搜索值列表，这可能比直接进行连接的开销更大。另一方面，值列表方法更简单，更直接地传达了设计者的意图。

经验法则是，源查询应指定你想要显示的*内容*，而列属性应指定你想要显示的*方式*。在某些情况下（例如隐藏列），这种区分是明确的。在其他情况下（例如使用值列表），则不那么明确。

### 配置单个系列

`Identification`（标识）部分允许您为系列选择一个名称。由于此系列将保存每个职位的最低工资，我将其命名为 `MinSal`。（您还需要创建另一个系列来保存最高工资，但现在还不需要。）

## 图表属性配置

最重要的属性是 `Show`，它控制图例在页面上的可见性。通常其他属性使用默认值就足够了，但你应该尝试各个选项来看看效果。特别值得注意的是 `Hide and Show Behavior`。如果其值为 `None`，则点击图例没有任何作用。否则，在图例上点击某个系列名称会切换该系列在图表中的可见性，并相应地重新计算图表。这种行为对于饼图尤其有趣，因为图例标识的是扇区而非系列。作为一个实验，请启用 `Salary Info` 图表的图例，并将 `Hide and Show Behavior` 的值设置为 `Rescale`。然后运行页面，点击一些图例值，观察图表如何自我调整。

另一种图表定制方式是直接在图表中显示数据值。例如，在图[3-37]中，最低和最高薪资显示在其条形图的上方。在图[3-38]中，员工数量和标签名称显示在每个扇区的外部。相关属性位于系列属性的 `Label` 部分。图[3-49]展示了我是如何为饼图填写这些属性的。

## 4. 导航

Web 应用程序的用户应能轻松地在其页面间导航。您在使用 APEX 时已经遇到过三种导航功能：链接、面包屑和导航菜单。本章将详细研究这些功能，并向您展示在应用程序中使用它们的不同方式。

### 列表

在通常的表述中，`列表`就是事物的集合。然而，在 APEX 中，列表是链接的集合。列表是一个有用的导航工具，原因显而易见：单击链接会带您进入新页面。

以 APEX 列表的一个好例子来说，再次考虑 SQL Workshop 主屏幕，如图 1-3 所示。它的`创建对象`区域是一个列表，可快速访问各种对象创建向导页面。该列表为每种 SQL 对象类型设有一个条目，其链接将带您进入该对象的创建向导。

另一个例子是`员工演示`应用程序的主页，如图 3-17 所示。它的`参考`区域是一个包含链接的列表，这些链接将带您访问选定的网页。

也许列表最显著的例子是导航菜单。它的每个条目都是一个链接，可带您进入应用程序的指定页面。

APEX `列表管理器`让您可以查看、编辑和创建列表。要访问列表管理器，请从应用程序的主屏幕开始，单击页面顶部的大`共享组件`按钮。搜索`导航`部分并单击`列表`。您将被带到列表管理器的主屏幕，如图 4-1 所示。

将链接存储在列表中，并在列表区域中显示，是一项重要的技术；它远比手动使用 HTML 格式化链接更优越，就像您在主页的 `引用` 区域所做的那样。其中一个原因是，列表区域的模板负责格式化列表条目；您只需选择所需的模板。换言之，列表区域将内容（源列表）与格式（列表模板）分离。编写 HTML 代码来实现列表不仅繁琐，而且会将内容与格式混在一起。任何更改条目格式的决定都需要您重写 HTML 代码。

### 层级列表

`员工演示` 应用程序当前的 `桌面导航菜单` 列表中有六个条目。由于条目数量较少，简单地将它们线性地显示在页面侧面（或顶部）是合理的。然而，随着条目数量的增加，一个长长的线性列表会变得难以管理。常见的解决方案是分层显示条目。

APEX 导航栏（如图 1-3 所示）就是一个很好的例子。它的第一个选项卡（包含徽标）对应于 APEX 主屏幕；其他选项卡对应于 APEX 站点的通用类别。每个类别选项卡都有一个主目标页面，作为该类别的主屏幕。点击类别选项卡的主要部分会带您前往该页面。每个类别选项卡还有一个下拉箭头；点击它会显示该类别的次要选项卡菜单。这样就形成了一个功能齐全且界面整洁的效果，次要选项卡在需要时才会显示。

在层级列表中，每个列表条目都可以有一个 `父级` 条目。没有父级的条目定义了一个层级结构，被称为该层级的 `根`。一个层级可以有多个级别，这意味着一个条目的父级也可能有其父级。例如，图 4-20 展示了 APEX 导航栏的 `应用构建器` 条目。它的子级之一 `工作区实用工具` 也有自己的子级。

![../images/335103_3_En_4_Chapter/335103_3_En_4_Fig20_HTML.png](img/335103_3_En_4_Fig20_HTML.png)
*图 4-20：APEX 导航栏中的应用构建器菜单*

让我们看看如何类似地重构 `员工演示` 应用程序。它的 `桌面导航菜单` 列表将包含主页以及两个层级结构：

*   一个 `区域` 层级，包含页面 `区域练习`、`经典报表`、`图表` 和 `交互式报表`
*   一个 `列表` 层级，包含页面 `列表练习`

每个层级对应本书的一个章节——`区域` 层级包含在第 3 章开发的页面，`列表` 层级包含在第 4 章开发的页面。随着本书的进展，我将遵循此约定，为每章开发的页面创建一个新的层级结构。

但在开始创建层级之前，您需要考虑如何配置每个根条目。显然，其 `父列表条目` 属性的值应为 `null`，表示它是一个根。但是，应该为其 `目标` 属性赋予什么值呢？有三种合理的策略：您可以创建一个专用的“主屏幕”作为目标；也可以将目标设置为其某个子级的目标；或者可以将目标设置为 `null`。APEX 采用了第一种策略——它为应用构建器、SQL 工作坊等分别设置了主屏幕。采用第二种策略的效果是使所选目标成为该部分的默认页面。而采用第三种策略则无法选择父级，从而迫使用户选择一个子级。本书将遵循以下约定：拥有单个子级的根条目将把该子级作为其目标，而拥有多个子级的根条目将采用空目标。

要创建此层级列表，请先转到 `桌面导航菜单` 的 `列表详细信息` 屏幕，该屏幕如图 4-2 所示。创建两个新的列表条目作为两个层级的根，并根据前述约定设置它们的目标页面：`区域` 条目的目标应为空，`列表` 条目的目标应为 `列表练习` 页面。

现在需要为五个子条目设置父级。直接的方法是转到每个条目的 `列表条目` 屏幕（如图 4-3 所示）并修改其 `父列表条目` 属性。不过，`列表详细信息` 屏幕有一个向导可以加速此过程。点击 `列表详细信息` 屏幕右侧标题为 `重新分配此列表中的列表条目` 的链接。该链接会带您到一个 `重新分配列表条目` 屏幕，从那里您可以快速指定给定父条目的子级。无论您采用哪种方式，桌面导航菜单的最终条目应如图 4-21 所示。

## 导航

标记为 `Parent Navigation Menu Entry` 的项会显示导航菜单列表中的每一个条目。（你需要滚动列表才能看到全部。）其默认值为 `-No parent selected-`，这将使新的菜单条目成为根条目。如果你希望新条目成为另一个条目的子项，就必须指定一个父项。理想情况下，你应该能够直接从列表中选择你想要的条目。然而，有些条目是灰显且不可选择的。具体来说，可选的条目是那些已经是父项的条目，或者是“潜在父项”（即没有目标的条目）。大多数情况下，你想要的父项是可选的。但如果你希望新条目是某个灰显条目的子项，你必须通过列表管理器为其分配父项。

### 巨型菜单

分层导航菜单通过两种方式改善用户体验。首先，它们对菜单条目进行结构化，使用户更容易理解有哪些可用操作。其次，它们减少了菜单所需的屏幕空间占用。

一个较新的设计思路是保持分层结构，但一次性显示所有菜单条目，这被称为 *巨型菜单*。例如，图 4-24 展示了带有巨型菜单的员工演示应用程序的外观。点击标志左侧的图标可以切换菜单的可见性。每个菜单条目都是指向其目标页面的链接。

### 页面层级

一个大型 Web 应用程序包含的页面太多，无法全部放入导航菜单。因此，导航菜单包含应用程序的 *着陆页*——那些标志着最突出任务开始的页面。

通常，应用程序中的所有页面集合也具有层级结构。这种页面层级的显式呈现被称为 *站点地图*。例如，考虑一个 APEX 站点中的页面。它们在层级中的根是 APEX 主页。根的子项对应于应用程序的各种类别，如应用构建器、SQL 工作室和团队开发工具。每个类别都有一个“主页”，它领导着包含其类别页面的子树。

图 4-26 列出了对应于 APEX 应用构建器的部分页面层级。由于空间限制，该图省略了许多页面。

![../images/335103_3_En_4_Chapter/335103_3_En_4_Fig26_HTML.jpg](img/335103_3_En_4_Fig26_HTML.jpg)

图 4-26 应用构建器的页面层级

### 面包屑导航

你可以通过给出页面层级中从该页面到主页路径上的页面序列，来描述应用程序中任何页面的位置。例如，`Application Attributes` 页面的位置是

```
App Builder > Workspace Utilities > Cross Application Reports > Application Attributes
```

此路径被称为 *面包屑导航*。并非所有 APEX 屏幕都显示其面包屑导航，但如果它们显示，通常会显示在导航栏下方。请注意，APEX 面包屑导航中的第一个条目显示为带圈的向上箭头，这表示层级的根页面。

作为练习，导航到 APEX 站点的其他页面并注意它们的面包屑导航。事实上，看看你是否能找到图 4-26 中列出的所有页面。面包屑导航是可点击的。点击其路径的任何部分都会将你带到相应的页面。

导航菜单和面包屑导航是互补的导航工具。导航菜单对所有页面都是一样的，其条目涵盖了应用程序的全部功能。通过查看菜单，用户可以确定任何所需操作的适当起始页面。另一方面，页面的面包屑导航特定于该页面，并提供有关该页面在页面层级中位置的上下文信息。例如，假设你正在查看应用构建器的 `Application Attributes` 页面。它的面包屑导航告诉你该页面是跨应用程序报告之一。该信息可能会促使你点击面包屑导航来探索其他跨应用程序报告。

在 APEX 中，整个页面层级被称为 *面包屑对象*。面包屑对象本质上是一个分层列表——它为层级中的每个页面包含一个条目，每个条目（根除外）都有一个父项。

APEX 会自动为每个应用程序创建一个面包屑对象（名为 `Breadcrumb`）。APEX *面包屑管理器* 允许你编辑此对象并创建其他面包屑对象。要进入面包屑管理器，请从应用程序的主页开始，选择页面顶部的 `Shared Components` 按钮。搜索 `Navigation` 部分，点击 `Breadcrumbs`，然后点击 `Breadcrumb` 图标。你将被带到该面包屑对象的面包屑管理器。图 4-27 显示了我的 `Employee Demo` 应用程序的面包屑管理器。

![../images/335103_3_En_4_Chapter/335103_3_En_4_Fig27_HTML.jpg](img/335103_3_En_4_Fig27_HTML.jpg)

图 4-27 面包屑管理器

该图显示只有一个面包屑条目，对应于主页。此条目是在我创建应用程序时自动创建的。请注意，即使我从主页删除了其面包屑区域，该条目仍然存在。

当我创建其他页面时，我告诉向导不要为新页面创建面包屑条目。相反，我现在可以使用面包屑管理器来创建面包屑条目。点击 `Create Breadcrumb Entry` 按钮。图 4-28 显示了生成的屏幕。

![../images/335103_3_En_4_Chapter/335103_3_En_4_Fig28_HTML.jpg](img/335103_3_En_4_Fig28_HTML.jpg)

图 4-28 面包屑条目屏幕

此屏幕允许你指定页面及其父项，以及有关面包屑如何在页面上显示的信息。具体而言：

*   `Page` 属性（在 `Breadcrumb` 部分中）指定页面。
*   `Parent Entry` 属性指定父项。
*   `Short Name` 属性指定页面在面包屑上的显示方式。选择页面的名称通常是合理的。
*   `Page` 属性（在 `Target` 部分中）指定用户点击该面包屑条目时发生的情况。它通常与前面的 `Page` 属性值相同。

让我们为第 7 页，即 `List Practice` 页面创建一个面包屑条目，它将是主页的子项。在 `Breadcrumb Page` 和 `Target Page` 属性中输入 7，在 `Parent Entry` 属性中输入 `1`，在 `Short Name` 属性中输入 `List Practice`。
