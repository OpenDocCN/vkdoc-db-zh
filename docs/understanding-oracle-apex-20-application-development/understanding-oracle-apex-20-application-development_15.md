# 10. 表单

第 7 章中的许多进程都涉及某些基本任务，例如根据键检索单条记录的值、查找给定记录的前后记录、更新表中的记录以及修改会话状态。

APEX 有一个区域类型，称为`Form`，它使得无需任何 PL/SQL 编程即可执行这些任务。表单区域包含两个通用进程——一个在渲染期间执行，一个在处理期间执行——您可以配置它们以满足您的许多需求。

本章探讨表单区域的使用、其进程以及 APEX 内置的一些相关进程。它还评估了一些用于创建基于表单的页面的 APEX 向导。了解如何以及何时使用 APEX 表单区域和向导，可以免除您为更常见的页面开发任务编写 PL/SQL 代码的需要。



## 表单区域

表单区域与报表区域类似，两者都有一个源查询。它们的区别在于输出记录的显示方式。报表以表格形式一次显示多条输出记录，而表单一次只显示一条记录，每个值位于一个项中。报表是只读的，但表单可以修改数据库。

回顾第 7 章，`单行视图`页面的`员工信息`区域（参考图 7-17）被用作表单，`单行更新`页面的`插入员工`和`更新/删除员工`区域（参考图 7-24）也是如此。这些区域当时的类型是`静态内容`，并使用 PL/SQL 代码来实现其功能。本节将介绍`表单`区域类型，并展示如何无需任何 PL/SQL 编码即可实现相同的功能。

### 更新记录

`更新员工`区域的工作原理如下：用户从选择列表中选取一名员工。该员工的`EMP`记录将被提取，其值用于初始化该区域的各项。随后，用户可以根据需要修改项值。点击`更新`按钮将使用这些值修改该记录。点击`删除`按钮将删除该记录。

要创建此区域，首先创建一个新的空白页面，命名为`表单更新`，页面编号为 30。然后创建一个类型为`表单`的新区域，并将其命名为`更新员工`。将数据源设置为 SQL 查询：

```sql
select EmpNo, Job, Sal, DeptNo
from EMP
```

查看生成的渲染树。APEX 已为查询中的每个字段创建了一个项，以及一个初始化过程。这与你在`表单视图`页面中遇到的情况相同。趁还未忘记，请转到`P30_EMPNO`的`数据源`属性（如图 10-4 所示），并开启其`主键`属性。

现在，你需要自定义表单以匹配图 10-9。具体而言，将`P30_EMPNO`的模板更改为`可选`，将其他三个项的模板更改为`可选-上方`。将`P30_EMPNO`、`P30_JOB`和`P30_DEPTNO`的类型更改为`选择列表`。`P30_EMPNO`的值列表查询应为：

```sql
select EName, EmpNo
from EMP
```

`P30_JOB`的查询应为：

```sql
select distinct Job as DisplayVal, Job as ResultVal
from EMP
```

而`P30_DEPTNO`的查询应为：

```sql
select DName, DeptNo
from DEPT
```

你还需要创建两个按钮，分别命名为`删除`和`更新`。它们的位置是`编辑`，操作是`提交`。此时，你的表单应能成功检索所选员工的信息。你剩下的任务是让按钮执行预期的数据库修改操作。这个结果出乎意料地简单。

表单区域修改数据库的能力默认是禁用的。要启用它，请转到表单区域的渲染树，点击其`属性`节点，并查看`编辑`部分中的`启用`属性。默认情况下，此属性是关闭的。当你开启它时，将显示其他属性，如图 10-10 所示。这些复选框允许你指定该区域应允许哪些编辑操作。`更新员工`区域执行更新和删除操作，因此你应确保只勾选了`更新行`和`删除行`。

图 10-10 表单区域的编辑属性

## 清除会话状态

花些时间运行你的 `Form Update` 页面。选择一条记录，修改其值，然后点击 `Update` 按钮。一切应该正常工作。但是，你将发现 `Delete` 按钮存在一个问题。当你点击它时，APEX 会显示一条错误信息。问题在于删除操作后，项 `P30_EMPNO` 仍然包含被删除的员工编号。因此，在页面渲染期间，初始化过程将尝试从该不存在的记录中获取值，从而产生错误。解决方案是在删除操作发生后将 `P30_EMPNO` 设置为 null。事实上，这正是你在 Listing 7-21 所示的 `DeleteEmp` 过程的 PL/SQL 代码中所做的。

但是如何让第 30 页上的 `Delete` 按钮设置一个项的值呢？由于你无法触及内置的 ARP 进程，你唯一的选择是创建另一个进程，该进程将在删除操作发生后执行。我们称这个进程为 `ClearEmpNo`。

实现 `ClearEmpNo` 的一种方法是使用以下代码作为 PL/SQL 进程：

```
begin
:P30_EMPNO := null;
end;
```

我们这些熟悉 PL/SQL 编码的人意识到这是一个直接、简单的解决方案。但 APEX 为那些不熟悉 PL/SQL 的人提供了另一种解决方案，即 `Clear Session State` 进程类型。以下是使用方法。

将你的进程 `ClearEmpNo` 的类型更改为 `Clear Session State`。图 10-14 显示了其初始属性。`Settings` 部分的 `Type` 属性指定要清除哪些项。默认情况下，其值为 `Clear all items on the Current Page`。该属性的其他值允许你清除单个项、指定页面上的所有项或所有页面上的所有项。

![../images/335103_3_En_10_Chapter/335103_3_En_10_Fig14_HTML.jpg](img/335103_3_En_10_Fig14_HTML.jpg)

图 10-14：清除会话状态进程的属性

虽然默认值完全没问题，但将类型更改为 `Clear Items` 也是合理的。如果你这样做，那么你将能够为 `Item(s)` 属性输入明确的项名称，如图 10-15 所示。

![../images/335103_3_En_10_Chapter/335103_3_En_10_Fig15_HTML.jpg](img/335103_3_En_10_Fig15_HTML.jpg)

图 10-15：配置进程以清除 P30_EMPNO

无论你如何创建 `ClearEmpNo` 进程，你必须执行以下两个步骤以确保它仅在删除操作后执行：

*   你必须将其 `When Button Pressed` 属性的值设置为 `Delete`。
*   你必须将其序列值（如图 10-14 所示）设置为大于 ARP 进程的值。

## 插入记录

图 10-9 中 `Form Update` 页面的第二个区域称为 `Insert Employees`。其工作方式如下：用户将值输入到该区域的五个项中，然后点击 `Insert` 按钮。APEX 随后会将一条记录插入到 EMP 表中，对于未提及的字段使用默认值。

一个 ARP 进程只能插入出现在页面项中的值。因此，重要的是 `Insert Employees` 区域要为 `EMP` 的每个字段都有一个项。换句话说，表单的数据源应该是查询 `select * from EMP`。然后你可以隐藏那些将接收默认值的项。

当你创建具有前述数据源的表单区域时，APEX 会立即通知你一个问题。问题是其生成的几个项名称重复。生成的项的名称完全基于其对应字段的名称，因此出现在页面上多个表单中的字段名将会有多个同名的项。

幸运的是，解决方案很简单；只需重命名有问题的项。图 10-16 显示了我的策略，即为 `Insert Employees` 区域中的所有项（无论是否重复）添加“`INSERT`”。

*   创建一个名为 `Insert` 的按钮。将其 `Action` 属性配置为 `Submit Page`，并将其 `Database Action` 属性配置为 `SQL INSERT action`。
*   转到名为 `Process form Insert Employees` 的 ARP 进程的属性编辑器，并将其 `When Button Pressed` 属性的值设为 `Insert`。
*   根据你的喜好设置项的格式和布局。

如果你尝试使用新的区域，你将发现另一个问题，即表单期望你始终为 `P30_EMPNO` 和 `P30_INSERT_EMPNO` 提供值。当 `Update Employees` 是唯一区域时，这个要求是合适的，因为没有员工编号，`Delete` 和 `Update` 操作没有意义。但当按下 `Insert` 按钮时，`P30_EMPNO` 可以（也应该）是 `null`。同样，当按下 `Delete` 和 `Update` 按钮时，`P30_INSERT_EMPNO` 应该为 null。因此，你需要禁用这两个项在 `Validation` 部分的 `Value Required` 属性。

## 计算

图 10-9 所示的 `Insert Employees` 区域仅显示 `EName`、`Job`、`Salary` 和 `DeptNo` 列的项；`HireDate`、`Comm` 和 `Offsite` 的项是隐藏的。其想法是这些隐藏项将包含默认值——特别是，`P30_INSERT_HIREDATE` 将包含当前日期，`P30_INSERT_COMM` 将为 `0`，`P30_INSERT_OFFSITE` 将为 `'N'`。ARP 进程在执行插入操作时将使用这些项的值。

那么，问题是如何将默认值放入隐藏项中。答案是使用 APEX *计算*。计算是一种页面组件，它为某个项赋值。其思路是为每个需要默认值的项创建一个计算，并安排这些计算在 ARP 进程执行之前执行。这样，计算出的值将成为会话状态的一部分，ARP 进程在创建新记录时将使用它们。

## 自定义报表和表单

尽管`Report with Form`向导能生成功能完整的页面，但这些页面是原始且未加修饰的。在实际应用中，你不太可能直接使用它们。相反，你应该将其视为构建更广阔、可定制化应用的基础。这些定制化主要分为三类：外观定制、功能定制和组织结构定制。

外观定制可改善应用的外观和布局。例如，你可以使用第 3 章的技术来改进报表及其列的格式，以及使用第 5 章的技术来优化按钮和项的外观与布局。你还可以更改默认的区域标题、按钮文本和项标签。

另一个外观问题涉及表单项的可见性。回想图 10-25，`Report with Form`向导要求你指定应在表单页面中显示的项。问题在于，表单页面同时用于插入和更新操作，而这两种操作有不同的需求。例如，插入操作需要为所有列设置项，但你可能希望隐藏那些接收默认值的列的项。此外，更新操作应仅显示可更新列的项。因此，为每个项提供一个`Server-side Condition`表达式，指定该项何时可见，将是一个好主意。

或者，你可以选择禁用项而不是隐藏它。每个项都有一个`Read Only`属性部分，你可以像`Server-side Condition`部分那样指定一个条件。`Server-side Condition`会隐藏项，而`Read Only`则使其不可修改。因此，例如，在更新模式下，显示整个记录但将不可修改的项（如`EName`）格式化为只读，可能是一个好主意。

功能定制为向导生成的页面添加功能。例如，你可以向表单页面添加`Previous`和`Next`按钮。你也可以修改插入和更新操作，使其在更改数据库后保留在表单页面，如图 7-24 的`Single Row Update`页面所示。

与组织结构相关的定制涉及添加或删除页面和区域，以及在它们之间移动现有的页面组件。例如，可以将报表和表单区域移动到同一个页面。或者，你可以将表单区域拆分为两个区域——一个用于插入，一个用于更新/删除——如`Single Row Update`页面所示。你甚至可以有三个表单区域，如图 7-5 的`Employee Data Entry`页面所示。

### 表单页面

`Report with Form`向导创建的两个页面连接非常松散。它们唯一的交互是当点击报表页面上的链接或其`Create`按钮时，这两者都会设置表单页面上`P32_EMPNO`项的值。因此，表单页面可以与其他类型的页面结合使用，唯一的前提是其他页面能向表单页面提供员工编号。

#### 实现树页面

尽管 APEX 有一个页面向导来创建树，但自己创建页面更具指导性（也更简单！）。因此，将第 33 页创建为一个名为`Management Hierarchy Tree`的空白页面，并创建一个类型为`Tree`的区域。其数据源应为`EMP`表。

配置树的属性位于该区域的`Settings`部分，你可以从渲染树中的`Attributes`节点进入。图 10-36 显示了图 10-34 中树的`Settings`属性。

![../images/335103_3_En_10_Chapter/335103_3_En_10_Fig36_HTML.jpg](img/335103_3_En_10_Fig36_HTML.jpg)

图 10-36
树区域设置

名为`Hierarchy`的属性应设置为`Computed with SQL`，这指示 APEX 根据以下设置计算层次结构：

*   `Node Label Column`是其值作为树节点标签的字段。
*   `Node Id Column`是主键字段。
*   `Parent Key Column`是定义层次结构的字段。记录的此字段值指定其父记录的主键。
*   `Start Tree With`指定层次结构的根。设置`Value is NULL`断言层次结构的根是父键列值为`null`的记录。

此时运行你的页面，并探索树。如果一个节点有子节点，其左侧将有一个箭头。点击这些箭头将展开或收缩其子节点列表。

最后的配置步骤是指定当用户点击树中的节点时发生什么。你需要的属性是`Link`，它位于`Settings`部分更靠下的位置。点击其框会弹出`Link Builder`向导。图 10-37 显示了我如何配置该链接。它指定点击节点将重定向到第 34 页，并将`P34_EMPNO`的值设置为该节点的`EMPNO`值。请注意，这里使用了替换字符串语法来引用节点字段的值。

向导生成的表单页面应该能与树页面无缝协作，无需你进行任何额外的配置——当点击树节点时，树会重定向到表单；当点击按钮时，表单会重定向到树。图 10-35 中我的表单页面，仅仅是更改了向导生成的项的类型和布局后的结果。

我想强调一点，由 `Form` 向导生成的表单页面与由 `Report with Form` 向导生成的表单页面完全相同——这两个向导之间的唯一区别是，`Report with Form` 还会生成一个包含 `创建` 按钮和报表的页面。此外，由 `Form` 向导生成的页面与一个带有 `表单` 区域的空白页面加上几个按钮和属性设置也完全相同。图 10-23 中的其他基于表单的向导与 `Report with Form` 类似。了解这一点可以帮助你决定，对于任何给定的任务，是使用其中一个向导还是自己编写页面。

页面 40 和 41 是表单页面，与您通过调用 `Create Form` 向导获得的结果相同。因此，主要问题是如何实现报表页面。

这三个报表区域，也许令人惊讶，是 `Classic Report` 区域。它们外观不同是因为使用了不同的区域模板。主报表模板是 `Media List`。此模板显示两个值，分别称为 `List_Title` 和 `List_Text`。如果您希望报表行成为链接，则源查询还必须有一个 Link 字段，其值是要重定向到的 URL。向导生成的源查询很复杂，但其本质如下：

```sql
select DeptNo, DName as List_Title, Loc as List_Text,
apex_page.get_url(p_items  => 'P39_DEPTNO',
p_values => DEPTNO) as Link
from DEPT
```

函数 `apex_page.get_url` 返回一个指向 APEX 页面的 URL。默认情况下，URL 会重定向到当前页面。参数 `p_items` 和 `p_values` 指定了应如何更改会话状态。
图 10-42 中标题为 `Dept` 的区域使用了模板 `Value Attribute Pairs – Column`。此模板将每个值显示在其自己的行上，并在前面加上列名。在图中，该区域的源有一行两列，因此其报告显示两行。
标题为 `Emp` 的区域使用了 `Standard` 模板，因此其报告以标准方式显示。
除了外观不同，`Dept` 和 `Emp` 报表以通常的方式与其表单页面交互，即通过让它们的按钮和链接重定向到相应的表单页面。
底线是，即使是像 `Master-Detail` 这样复杂的向导，也只是创建了一个包含三个报表区域的页面，其中两个区域链接到表单页面。如果您愿意，可以自定义这些区域以满足您的需求。

#### 小结

本章的第一部分探讨了 APEX 表单区域。一个表单区域包含两个进程——一个初始化进程和一个 ARP 进程——它们允许您的页面访问数据库，而无需编写 PL/SQL 代码。APEX 还为一些常见任务提供了内置进程，这些进程与两个表单进程配合使用。几个例子展示了这些进程相对于相应 PL/SQL 进程的优点和局限性。
本章的第二部分探讨了一些创建表单的 APEX 页面创建向导。这些向导各不相同，从创建旨在与其他页面结合使用的单页面的 `Form` 向导，到将 `Form` 页面与简单报表页面配对的 `Report with Form` 向导，再到将两个 `Form` 页面与一个包含三个报表区域的页面匹配的复杂 `Master-Detail` 向导。在每种情况下，重点都是揭开这些向导的神秘面纱，使您能够自信地创建所需的页面，无论是否使用它们。

## 11. 交互式网格

报表是只读的。如果您正在查看报表并想要更改其内容，则必须使用表单。例如，考虑图 10-27 中的 `Report with Form 1` 页面。假设用户想要修改员工报表中的一行。会发生以下步骤：
*   用户单击所需行的链接，这会重定向到 `Report with Form 2` 页面（图 10-29），并将选定的员工号放入该页面的一个项中。
*   然后，表单的初始化进程从数据库中检索该员工的信息，并将其存储在相应的项中。
*   然后用户修改这些项的内容，并单击 `Apply Changes` 按钮执行更新。

这些步骤的奇特之处在于，第二步没有利用员工报表。相反，初始化进程重新从数据库中检索所需的员工记录，即使这些值就在报表中。
*交互式网格* 的概念通过允许用户直接更新报表值来解决这种情况。交互式网格本质上是一个“可更新的报表”，在一个区域中结合了报表和表单的功能。本章探讨了构建和自定义交互式网格所需的技术。

### 确保一致性
第 8 章讨论了页面如何保持数据一致性的方法。一种方法是使用验证来检查不合适的项值；另一种方法是限制可输入的值。以下小节展示了如何将这些解决方案应用于交互式网格。

#### 网格验证
交互式网格中的验证是一个必须对网格的每一行都为真的断言。您定义交互式网格验证的方式与定义基于项的验证相同；唯一的区别是它将应用于网格中的每一行，而不仅仅是一行。

例如，考虑如何创建一个验证，以确保员工的经理要么是总裁，要么是同一部门的员工。要创建此验证，请转到页面编辑器的`处理`选项卡，右键单击`验证`节点，然后选择`创建验证`。图 11-14 显示了您应该指定的相关属性值：

*   将验证名称设置为`同一部门经理`。
*   将可编辑区域设置为`员工`。
*   将验证类型设置为`返回的行`，查询为

```
select * from EMP
where EmpNo = :MGR
and (DeptNo = :DEPTNO or Job = 'PRESIDENT')
```

*   给出适当的错误消息。

![../images/335103_3_En_11_Chapter/335103_3_En_11_Fig14_HTML.jpg](img/335103_3_En_11_Fig14_HTML.jpg)

图 11-14
网格验证的属性

这里的`可编辑区域`属性至关重要。通过为此属性提供值，您指定了这是一个交互式网格验证，而不是基于项的验证。因此，当发生提交操作时（例如通过单击`保存`按钮），APEX 将对网格的每个已更改行应用该验证。
给定一个员工行，如果经理是总裁或与该员工在同一部门，验证查询会返回该员工的经理。换句话说，如果存在输出记录，则该行有一个合适的经理，从而满足`返回的行`验证类型。请注意，该查询使用绑定变量语法引用修改行的值；也就是说，它将修改行的单元格视为项。

#### 限制交互式网格输入
使用交互式网格进行更新的一个问题是，您可能希望某个单元格对于某些操作是可更新的，而对于其他操作则不可更新。例如，考虑`员工`网格中的`员工姓名`列。假设用户应该能够在插入时输入员工姓名，但随后不应能够编辑它。如何强制执行此限制？

想法是利用该列的`只读`属性。特别是，您需要指定该列的单元格对于除更新行之外的所有行都是只读的。要表达此规范，您需要知道每行的更新状态。

APEX 有一个名为`APEX$ROW_STATUS`的内置变量用于此目的。它对网格的每一行都有一个值，该值可以是四个值之一：

*   如果该行已被编辑，则状态为‘`U`’。
*   如果该行是插入行，则状态为‘`C`’（表示“已创建”）。
*   如果该行被标记为删除，则状态为‘`D`’。
*   如果该行未被更改，则状态为`null`。

图 11-15 展示了如何使用`APEX$ROW_STATUS`来指定`员工姓名`列的`只读` SQL 表达式。另请注意，`执行`属性的值为`每一行`，这告诉 APEX 将此表达式应用于网格中的每一行。

### 网格进程

每个网格区域都有自己的 ARP 进程，该进程为该区域实现数据库修改操作。如果你希望网格具备额外功能，则需要为其创建额外的进程。

第 7 章和第 10 章讨论的进程在每次提交请求时执行一次。然而，交互式网格的进程需要针对网格中每一行被修改的行执行一次。举个例子，考虑代码清单 7-13 中的`LogChanges`进程。该进程用于`修订后的员工数据录入`页面——每当用户单击`Delete`（删除）、`Insert`（插入）或`Update`（更新）按钮时，该进程就会向`EMPLOG`表中插入一条记录，标明请求、用户和日期。

### 修订 LogChanges 进程

你可以轻松地为`交互式网格`页面重新创建此进程。图 11-19 显示了相关的进程属性。除了一个属性外，这些属性值与第 7 章的进程完全相同——`可编辑区域`属性的值现在是`Employees`（员工）。这个值告诉 APEX，当进程执行时，其代码应应用于`Employees`网格中每个被更改的行。

![../images/335103_3_En_11_Chapter/335103_3_En_11_Fig19_HTML.jpg](img/335103_3_En_11_Fig19_HTML.jpg)

**图 11-19** LogChanges 进程

通过修改`Employees`网格的一些数据，然后查看 SQL Workshop 中的`EMPLOG`表来测试你的进程。该表应该为你所做的每次修改都有一条记录。不幸的是，所有记录看起来都一样——`Request`（请求）值要么是`APEX_AJAX_DISPATCH`（如果你使用了工具栏上的内置`SAVE`按钮），要么是`SAVE`（如果你使用了为图 11-13 编写的`SAVE`按钮）。这还不够好；`Request`字段的值应该更具信息性。

因此，让我们修改进程代码，使得`Request`字段的值包含行状态和被修改行的`EmpNo`值，例如"`update of 8830`"（8830 的更新）或"`insertion of 8200.`"（8200 的插入）。代码清单 11-1 包含了修订后的代码。注意，它使用`APEX$ROW_STATUS`来确定发生了哪种操作。

```sql
declare
  v_request varchar(20);
begin
  case :APEX$ROW_STATUS
    when 'U' then v_request := 'Update ';
    when 'C' then v_request := 'Insert ';
    when 'D' then v_request := 'Delete ';
  end case;
  v_request := v_request || :EMPNO;
  insert into EMPLOG (Request, UserName, RequestDate)
  values (v_request, :APP_USER, sysdate);
end;
```

### 网格的自定义插入

你可能遇到的另一种情况是，网格内置的数据库修改代码不能满足你的需求。在这种情况下，APEX 允许你编写自己的 PL/SQL 代码来替换它。

举个简单的例子，考虑向一个不包含表所有字段的网格中插入新行的问题。内置的插入进程会给缺失的字段赋空值。但如果这还不够好呢？如果新记录需要非空的默认值怎么办？

```sql
select EmpNo, EName, Job, Sal, DeptNo
from EMP
```

别忘了打开`EMPNO`列的`主键`属性，并关闭其`需要验证`属性。然后转到`属性`节点的`编辑`部分并启用编辑。图 11-20 显示了生成区域的顶部。

手头的任务是修改该区域的插入代码，以便将默认值放入缺失的字段中。操作方法如下。

转到页面编辑器的`处理`选项卡，选择名为`Custom Emps-Save Interactive Grid Data`的进程节点。这是`Custom Emps`网格区域的内置 ARP 进程。图 11-21 显示了其`标识`和`设置`属性。这里的关键属性是`设置`部分中的`目标类型`。默认值`区域源`告诉 APEX 使用内置的修改进程。要将其替换为你的代码，请将该属性值更改为`PL/SQL 代码`。属性编辑器将显示一个文本框，你可以在其中输入代码。

你的自定义 PL/SQL 代码必须处理所有三种修改操作，即使你只更改了插入操作。代码清单 11-2 展示了我的代码。它使用 case 语句来确定需要执行的操作。删除和更新的代码是直接的 SQL 命令，与 ARP 进程的操作没有不同。插入的代码为四个缺失的列确定默认值：新员工的经理是总裁，雇佣日期是今天，初始佣金为 0，且员工不在外地工作。`insert`语句末尾的`returning`子句导致新生成的员工编号被保存在网格的会话状态中。

请注意，你的进程只需要处理单行。APEX 将遍历网格，为每个更改的行调用你的代码，并处理丢失的更新检测。

```sql
begin
  case :APEX$ROW_STATUS
    when 'C' then
      insert into EMP(EName, Job, Sal, DeptNo,
                      Mgr, HireDate, Comm, Offsite)
      values (:ENAME, :JOB, :SAL, :DEPTNO,
              7839, sysdate, 0, 'N')
      returning EmpNo into :EMPNO;
    when 'U' then
      update emp
      set EName  = :ENAME,
          Job    = :JOB,
          Sal    = :SAL,
          DeptNo = :DEPTNO
      where EmpNo  = :EMPNO;
    when 'D' then
      delete emp
      where EmpNo = :EMPNO;
  end case;
end;
```

## 总结

本章重点介绍了交互式网格——它们是什么、能做什么以及如何构建。首先探讨了其类报表的特性。在这方面，它们的外观和行为与交互式报表非常相似。交互式网格还具有一些额外的功能，最重要的是能够将两个网格区域链接成主从关系。

随后，本章讨论了如何使用交互式网格来更新数据库的问题。交互式网格拥有一个复杂且易于使用的界面，你可以在其中临时就地编辑值、添加行并初始化其值、删除现有行，然后根据需要撤销或保存这些更改。

交互式网格具有验证功能，以确保用户输入的适当性。你指定这些验证的方式与第 8 章相同。区别在于，APEX 会将验证代码应用于网格的每一行已修改行。你还可以使用网格的 `只读` 属性来限制用户更改指定行或列的值。

每个交互式网格都有一个内置进程来处理数据库的插入、删除和更新操作。如果你需要额外的功能，可以编写自己的进程。与验证类似，进程的代码仅引用网格中的一行；APEX 会将该进程应用于网格的每一行已修改行。实际上，你甚至可以替换掉默认的数据库处理代码，用自己的代码来代替。
