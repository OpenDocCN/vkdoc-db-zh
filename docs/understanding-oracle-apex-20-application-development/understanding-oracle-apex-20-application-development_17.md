# 13. 安全
当你创建 `Employee Demo` 应用程序时，你选择将访问权限限制为那些在你工作区拥有账户的人。此限制是 APEX 用于限制访问的几种功能之一。这些功能可分为 *认证* 和 *授权*：认证用于识别合法用户，授权则指定每个合法用户被允许查看哪些信息。本章将探讨这些功能，以及如何保护数据免受恶意用户侵害的相关问题。

## 认证
认证是识别应用程序当前用户的能力。APEX 用户通过提供用户名和密码来表明身份。APEX 支持几种可能的 *认证方案*，它们以不同的方式并在不同的组织级别维护用户/密码列表。

以下是四种常见的认证方案，从最通用到最不通用排序：

*   *LDAP 目录方案*：用户/密码列表在组织的 LDAP 目录中维护。此方案允许组织为每个人分配一个账户，用于该组织维护的所有系统。由此带来的管理便利性通常使 LDAP 成为许多组织的首选认证方案。
*   *数据库账户方案*：用户/密码列表在 Oracle 数据库系统中维护。此方案允许用户为多个基于 Oracle 的系统（包括 APEX 应用程序）使用一个账户。此方案的缺点是用户也将拥有直接访问 Oracle 数据库的权限，这可能不是期望的。
*   *Application Express 账户方案*：这是当前 `Employee Demo` 应用程序的授权方案。用户/密码列表在应用程序的工作区内维护。工作区的 APEX 管理员管理该列表。如果一个人需要访问不同工作区的应用程序，则需要为每个工作区拥有一个账户。
*   *自定义方案*：应用程序所有者维护用户/密码列表。每个人需要为每个此类应用程序拥有一个独立的账户。自定义方案常用于允许人们创建和管理自己账户的应用程序。

此外，还有一种方案是应用程序不执行任何认证。APEX 将其视为 *无认证* 认证方案。

### 编写认证函数

既然您了解了自定义认证的工作原理，是时候实现一个更有用的认证函数了。以下部分讨论了三个重要问题：如何存储用户名和密码信息、如何管理用户账户以及如何对用户进行认证。

#### 存储用户名/密码信息

存储用户名/密码信息的标准方法是创建一个表；我们称这个表为 `USERS`。`USERS` 表将有三列：`UserName`、`Password` 和 `IsAdministrator`。这第三列指示用户是否具有管理权限。此外，您应该创建两个“内置”用户：一个用户 `ADMIN`，他是密码为 `1234` 的管理员；以及一个用户 `GUEST`，他不是管理员且密码为 `null`。这些任务都可以在 SQL 命令工具中执行。代码清单 13-1 给出了相应的 SQL 语句，应单独执行。

```
create table USERS (UserName varchar2(12), Password varchar2(1000),
IsAdministrator char(1));
insert into USERS (UserName, Password, IsAdministrator)
values ('ADMIN',
apex_util.get_hash(apex_t_varchar2('ADMIN','1234'), null),
'Y');
insert into USERS (UserName, Password, IsAdministrator)
values ('GUEST',
apex_util.get_hash(apex_t_varchar2('GUEST',null), null),
'N');
```

## 密码处理

密码的处理方式需要一些说明。出于安全考虑，密码应始终以编码形式存储，而非明文。清单 13-1 使用了函数 `apex_util.get_hash` 来执行编码。该函数在第 7 章中介绍，并曾用于丢失更新检测。回想一下，该函数接受一个值集合作为输入，并生成一个字符串作为输出。构造函数 `apex_t_varchar2` 从其参数创建值集合。例如，`ADMIN` 的编码密码是调用哈希函数时以集合 `['ADMIN','1234']` 作为输入的输出结果。

使用哈希函数进行密码编码的原因在于它能混淆输入值——仅凭函数的输出，他人实际上无法确定其输入。因此，您可以放心，即使数据库系统遭到入侵（例如，硬盘被盗），入侵者也无法轻易利用已编码的密码信息。

清单 13-1 中的哈希密码将用户名作为哈希值的一部分。这种技术通过挫败以下攻击，在一定程度上提高了安全性：入侵者创建一个账户，然后为其创建一系列密码，并记下每个密码的哈希值。然后，入侵者可以将这些哈希密码与从 `USERS` 表中窃取的密码进行比较。假设检测到匹配项。如果用户名不是哈希值的一部分，那么入侵者很可能发现了某个用户的密码。但是，如果用户名是哈希值的一部分，那么检测到匹配项对入侵者来说毫无意义。

`apex_util.get_hash` 函数有两个参数：第一个是要哈希的值集合，第二个参数是一个布尔值。清单 13-1 中的第二个参数是 `null`，这被视为 `false`。在第 7 章中，该函数仅使用一个参数调用，这意味着第二个参数值为 `true`。第二个参数指示函数是否应将会话 ID 添加到输入值中。如果哈希值包含会话 ID，安全性会更高，但这样它就只能在该会话的范围内使用。因为丢失更新检测发生在一个会话内，所以使用 `true` 作为第二个参数是合理的，这正是第 7 章中的情况。然而，密码编码是跨会话的，因此必须使用 `false`（或 `null`）作为第二个参数。

*图 13-5 演示应用程序的管理用户页面*

`所有用户` 区域是一个经典报表，其数据源是 SQL 查询：

```sql
select UserName, IsAdministrator, Password
from USERS
```

我将编码后的密码放在报表中仅用于演示目的。在真实的应用程序中，不应显示编码后的密码——这些值没有有用的管理用途，显示它们会增加密码被破解的可能性。

`创建用户` 区域包含四个项目和一个提交按钮。这些项目定义如下：

*   `P46_NAME` 的类型为 `纯文本`。
*   `P46_ADMIN` 的类型为 `开关`。
*   `P46_PASSWORD` 和 `P46_CONFIRM` 的类型为 `密码`。

图 13-5 显示了在添加名为 `JOE` 的非管理员用户（密码为“`joe`”）之前此页面的内容。虽然此用户名是以小写输入的，但为了确保用户名不区分大小写，它将在表中以大写保存。当然，密码将使用哈希函数保存为编码字符串。

当您点击提交按钮创建新用户账户时，将执行两项验证和一个处理过程。一项验证确保所选用户名尚未出现在 `USERS` 表中。此验证的类型为 `未返回行`，并由以下查询定义：

```sql
select * from USERS
where UserName = upper(:P46_NAME)
```

另一项验证确保两个密码相同。此验证的类型为 `SQL 表达式`，并由以下代码定义：

```sql
(:P46_PASSWORD = :P46_CONFIRM) or
(:P46_PASSWORD is null and :P46_CONFIRM is null)
```

验证的第二行考虑了密码可能为 `null` 的情况。

该处理过程将用户名转换为大写，对密码进行哈希处理，然后向 `USERS` 表插入一条记录。其代码见清单 13-2。

```sql
declare
    v_username     varchar2(20) := upper(:P46_NAME);
    v_valuesToHash apex_t_varchar2 :=
        apex_t_varchar2(v_username, :P46_PASSWORD);
begin
    insert into USERS (UserName, Password, IsAdministrator)
    values (v_username,
            apex_util.get_hash(v_valuesToHash, null),
            :P46_ADMIN);
end;
```

## 用户身份验证

既然您有了创建用户的方法，就需要创建一个方案来对他们进行身份验证。返回到应用程序的 `身份验证方案` 屏幕，创建一个新的 `自定义` 方案，其身份验证函数如清单 13-3 所示。

```sql
function custom_authentication (p_username in varchar2,
                                p_password in varchar2) return boolean
is
    v_userName     varchar2(20) := upper(p_username);
    v_valuesToHash apex_t_varchar2 :=
        apex_t_varchar2(v_username, p_password);
    v_userCount    int;
begin
    select count(*) into v_userCount
    from USERS
    where UserName = v_userName
    and   Password = apex_util.get_hash(v_valuesToHash, null);
    return v_userCount = 1;
end;
```
*清单 13-3 自定义身份验证函数*

此身份验证函数将提供的用户名转换为大写，以确保用户名不区分大小写。然后它执行一个查询，检查 `USERS` 表中是否存在具有该用户名和编码密码的记录。如果存在，则返回 `true`。创建此身份验证方案后，请将其设为当前方案。然后运行您的应用程序。练习以 `ADMIN`、`GUEST` 和 `JOE` 的身份登录。转到 `管理用户` 页面创建另一个用户，然后注销并以该用户身份重新登录。

## 授权

认证方案规定了哪些用户被允许访问应用程序的非公开页面。这是一个良好的开端，但往往还不够。一个 Web 应用程序可以有多种类型的用户，而某些页面（或页面的部分内容）只适合特定类型的用户。因此，你需要一种方法来指定哪些用户可以访问什么信息。安全的这一方面被称为`授权`。

APEX 通过允许每个页面或页面组件关联一个`授权方案`来实现授权，该方案指定了一组允许的用户。如果一个页面（或组件）有关联的授权方案，那么它将只对方案指定的用户可见。

## 创建授权方案

回想一下，内置变量`APP_USER`保存着当前用户的用户名。一个 APEX 授权方案由一个布尔表达式定义，通常涉及`APP_USER`的值；当表达式返回`true`时，用户即被授权。

授权表达式与验证表达式类似，它们的类型也相似。例如，`Exists SQL Query`类型在查询输出至少包含一条记录时返回`true`。`PL/SQL function returning Boolean`类型是明确返回`true`或`false`的 PL/SQL 代码。

举个例子，让我们为`Employee Demo`应用程序创建三个授权方案：

*   `Administrators`，如果当前用户拥有管理权限则返回`true`。
*   `Guests`，如果当前用户名为`GUEST`则返回`true`。
*   `Built-in Users`，如果当前用户名为`GUEST`或`ADMIN`则返回`true`。

`Administrators`方案可以通过以下`Exists SQL`查询定义：

```sql
select *
from USERS
where UserName = :APP_USER and IsAdministrator = 'Y'
```

`Guests`方案可以通过以下 PL/SQL 函数定义：

```sql
return :APP_USER = 'GUEST';
```

`Built-in Users`方案可以通过以下 PL/SQL 函数定义：

```sql
return :APP_USER = 'GUEST' or :APP_USER = 'ADMIN';
```

要创建一个授权方案，请转到应用程序的主页，点击`Shared Components`按钮，然后从`Security`部分选择`Authorization Schemes`。你将进入一个列出了为你的应用程序创建的授权方案的屏幕；我的初始屏幕如图 13-7 所示。我的 APEX 服务器会自动为我创建的每个应用程序创建`Administration Rights`方案。我的`Employee Demo`应用程序不需要它，所以我选择删除它。如果你的应用程序也有一个，请随意删除你的。

例如，考虑由以下 PL/SQL 函数定义的授权方案：

```sql
return :APP_USER = 'ADMIN'
and extract(hour from current_timestamp) >= 9
and extract(hour from current_timestamp) < 17;
```

此方案仅对`ADMIN`用户返回`true`，且仅在上午 9 点到下午 5 点之间。如果授权方案的目的是将访问限制在这些时间，其评估点需要设置为`Once per page view`。

## 授权方案选项与应用

第一个选项是`–Select–`，表示不需要授权。接下来的三个选项是我创建的三种授权方案。最后三个是它们的否定形式，由 APEX 自动生成。`Must Not Be Public User`方案是 APEX 内置的，将在后面讨论。

例如，考虑图 7-5 中的`Employee Data Entry`页面。回想一下，这个页面有四个区域：一个显示`EMP`表报告的区域，以及用于删除、插入和更新表行的区域。假设任何人都可以查看员工报告，但不允许访客修改。在这种情况下，你会将授权方案`Not Guests`分配给删除、插入和更新区域，并将`–Select–`分配给报告区域。结果是，以`GUEST`身份验证的用户将只看到报告区域；其他三个区域对他们将是隐藏的。

再举一个例子，再次考虑图 13-5 中的`Manage Users`页面。如果你希望页面上的两个区域只能由管理员访问，请为每个区域分配授权方案`Administrators`。结果是，管理员将看到如图所示的页面；非管理员将看到一个只有导航栏和菜单但没有内容的页面。

因为向用户显示一个基本空白的页面是尴尬的，所以页面应该向非管理员显示一些内容。一个解决方案是创建一个显示消息“此页面仅供管理员使用”的区域，并为其分配授权方案`Not Administrators`。另一个解决方案是将授权方案分配给整个页面，稍后讨论。

将授权方案分配给组件会导致该组件根据当前用户显示或隐藏。这种行为与你在第 6 章研究的条件渲染非常相似。所以问题是：为什么要使用组件授权——为什么不使用条件渲染来完成所有操作？

答案是，是的，可以使用条件渲染来执行授权。如果你创建一个与授权方案定义相同的条件，该区域的行为将完全相同。然而，使用显式授权有三个更好的理由：

*   一个授权方案可以用于多个组件，确保一致性。
*   每个授权方案都有一个名称，这使应用程序开发人员更容易理解授权条件的效果。
*   授权和条件渲染在应用程序中有着完全不同的目的，将它们分开会更清晰。

你可以将授权方案分配给页面本身，而不是分配给页面的组件。每个页面在其`Security`部分都有属性`Authorization Scheme`，其选项与图 13-11 中的相同。

### 非公共用户方案

授权方案`Not Public User`是 APEX 内置的。它对任何已登录用户返回`true`，如果用户未登录则返回`false`。它的主要用途是有选择地隐藏公共页面上的组件。

例如，假设你应用程序的主页是一个公共页面。然而，并非页面的所有部分都应该是公共的；特别是，`Quick Link`区域应该只对登录用户可见。这种情况可以通过将该区域的授权设置为`Not Public User`来处理。


### 避免恶意使用

一个 Web 应用的认证和授权方案详细规定了谁被允许访问哪些页面的哪些组件。每个经过认证的用户都对数据拥有一个明确定义的、有限的接口——当然，前提是用户以预期的方式访问页面。

问题在于，用户有各种方式来破坏 Web 应用的预期功能。本节重点介绍三种此类技术：SQL 注入、跨站脚本攻击和 URL 篡改。在**SQL 注入**中，用户提交一个伪装成输入数据的 SQL 代码片段，并更改正在处理这个伪造“数据”的查询。在**跨站脚本攻击**中，用户将恶意的 JavaScript 代码片段保存为表数据值的一部分；当另一个用户随后显示该值时，JavaScript 代码也会执行。在**URL 篡改**中，用户以服务器期望的形式向服务器发送一个 URL；然而，该 URL 请求的操作是用户本不应执行的。

APEX 具有若干特性，可帮助你防范恶意使用。事实上，利用这些特性构建一个防篡改的应用程序相对简单。但要利用它们，你需要了解威胁是什么以及 APEX 如何防范它们。

## SQL 注入

回顾图 12-1 中的`Report Builder`演示页面。此页面的意图是根据输入到`Enter Condition`文本区域的条件，显示`EMP`表或`DEPT`表的子集。然而，恶意用户可能输入一个条件，导致页面以相当意外的方式显示其他表中的数据。例如，图 13-14 显示了一个显示`USERS`表内容的条件。

![../images/335103_3_En_13_Chapter/335103_3_En_13_Fig14_HTML.jpg](img/335103_3_En_13_Fig14_HTML.jpg)

图 13-14 Report Builder 页面的意外使用

请注意，输入到`Enter Condition`项中的字符串不是一个合法的 SQL 条件。相反，它由一个合法条件`EmpNo<0`加上一些额外的 SQL 代码组成。要理解输入的目的，你必须了解清单 12-1 中的 PL/SQL 函数如何构建报告的源查询。回想一下，该代码生成了一个针对`EMP`的查询开头，然后将指定的条件附加到其末尾。因此，为图 13-14 生成的源查询变成如下形式：

```sql
select EName, Job, Offsite
from EMP where EmpNo < 0
union
select UserName, substr(Password,1,9), IsAdministrator
from USERS
```

请注意，用户利用了`union`关键字来扩大查询的范围。原本针对`EMP`表的预期查询现在只是整个查询的一个子查询。此外，条件`EmpNo<0`导致该子查询不返回任何记录，这意味着整个查询的输出完全来自第二个子查询。

第二个子查询可以是任意的，但受限于它必须与第一个子查询具有相同的列数，并且每个输出值必须匹配第一个子查询对应列的类型。这一限制解释了为什么第二个子查询只提取密码的前九个字符——`Job`列被定义为`varchar2(9)`，因此尝试选择更多字符会产生 SQL 错误。

恶意用户可能利用此漏洞获取用户的编码密码，以便随后破解它们并获得对应用的未授权访问。当然，仅拥有编码密码的前九个字符是不够的。然而，恶意用户可以多次运行此漏洞，每次获取密码的接下来九个字符，从而发现剩余字符。

这种技术被称为**SQL 注入**。当恶意用户在基于文本的项中输入 SQL 以改变查询的预期目的时，就发生了 SQL 注入。前面的例子表明 SQL 注入可能产生严重后果。因此，应用程序开发者必须意识到 SQL 注入的可能性，并不惜一切代价避免它。

从某种意义上说，`Report Builder`页面是在自找麻烦，因为`Enter Condition`项期望 SQL 代码。SQL 注入的真正危害在于，即使一个项期望一个值，它也可能成功。例如，让我们构建一个名为`SQL Injection`的页面，它将是`Employee Demo`的第 48 页。图 13-15 显示了此页面的预期用途。

图 13-16 中`P48_COLUMN1`的值如下：

```
Password from USERS where UserName = 'ADMIN' --
```

当 APEX 使用此字符串构造报表源时，它会得到以下 SQL 查询：

```
select distinct Password from USERS where UserName = 'ADMIN' -- from EMP
```

回想一下，字符“`--`”在 PL/SQL 中表示注释。因此，SQL 代码中的`from EMP`部分被完全忽略。换句话说，SQL 注入将针对`EMP`表的查询转变成了针对`USERS`表的查询。

`获取员工信息`区域使用了本质上相同的注入技巧。图 13-16 中`P48_COLUMN2`的值如下：

```
Password from USERS where UserName = 'ADMIN' or :1 < 0 --
```

当 APEX 使用此字符串构造报表源时，它会得到以下 SQL 查询：

```
select Password from USERS
    where UserName = 'ADMIN' or :1 < 0 --from EMP where EmpNo = :1
```

同样，注释字符之后的所有内容都被忽略，因此注入最终创建了一个对`USERS`表而非`EMP`表的查询。请注意，恶意用户必须仔细构造注入代码，以使生成的 SQL 字符串合法。考虑`获取员工信息`示例。`execute immediate`语句将员工号传递给 SQL 查询；虽然恶意查询不会使用此值，但它仍然需要一个形式参数来接收它。表达式`:1<0`实现了这个目标。指定的员工号被赋值给参数`:1`。因为员工号总是正数，这个表达式总是求值为`false`，所以它不会影响查询的输出。

一般来说，只要查询字符串是从原始的、未检查的用户输入构造的，就可能发生 SQL 注入。在这种情况下，恶意用户可能能够输入一个代码片段作为输入，从而改变构造查询的目的。因此，应用程序开发人员应始终检查用户输入，以确保其符合预期的格式。这里有三种策略：

*   避免基于文本的输入。请注意，如果用户被迫从列表中选择一个列，图 13-16 的 SQL 注入技术将不起作用（尽管 URL 修改技术可以规避此策略，稍后会看到）。
*   为输入编写一个 APEX 验证，以检测可能的代码片段。例如，在图 13-16 中，你可以拒绝接受包含空格或注释字符的列名。
*   将输入转换为“净化”形式。也就是说，编写一个函数将用户值转换为可接受的值。在图 13-16 中，该函数可能会删除输入中的所有空格，或者将每个空格转换为转义的 HTML 字符`&nbsp;`。

让我们看看如何使用这些策略来改进图 13-14 所示的`报表生成器`页面。问题当然是，基于文本的项`输入条件`要求用户输入 SQL 代码。第一种策略建议你尝试一种非基于文本的方式来指定条件。`计数和删除`演示页面的`版本 2`区域（如图 12-4 所示）展示了如何做到这一点，至少对于简单条件是这样。APEX *query builder*提供了一个更复杂的解决方案。

## 跨站脚本

在*跨站脚本*技术中，恶意用户将 JavaScript 代码作为数据库值的一部分保存。这段代码就像一个陷阱：当另一个用户显示包含该值的页面时，用户的浏览器会在用户不知情的情况下执行 JavaScript 代码。JavaScript 代码可能会做诸如将用户的会话信息通过电子邮件发送给攻击者（从而让攻击者冒充用户）或让 APEX 服务器执行指定函数等事情。

跨站脚本之所以如此邪恶，是因为 JavaScript 代码是*以受害者的授权级别*执行的。因此，如果管理员成为跨站脚本攻击的受害者，攻击者将获得数据库的管理员级访问权限。

成功的跨站脚本攻击有三个要求：首先，某个表必须有一个类型为`varchar2(N)`的列，其中`N`足够大以容纳 JavaScript 代码；其次，应用程序的某个页面必须有一个用于向该字段输入数据的基于文本的项；第三，应用程序的某个页面必须配置为以“原始”未转义形式显示被污染的值。

就目前情况而言，`员工演示`应用程序不易受到跨站脚本攻击，因为`EMP`和`DEPT`都没有足够大的列。因此，为了本示例的目的，你应该通过将`DEPT`表的`Loc`列扩展为`varchar2(80)`来修改它。所需的 SQL 如下：

```
alter table DEPT
modify Loc varchar2(80)
```

如果你仔细想想，这个结果是有一定道理的——毕竟，报表只是显示查询的输出。然而，实际情况比这更复杂。考虑 APEX 如何渲染报表：它生成一个 HTML 表格，并将输出值放入其中。因此，例如，显示报表第一行的 HTML 代码可能如下所示：

```html

  <td>ACCOUNTING</td>
  <td><b>NEW YORK</b></td>
</tr>
```

尽管 `<b>` 和 `</b>` 标签是作为数据输入到表格中的，但浏览器最终会将它们视为 HTML。为了使 APEX 能在报表中显示这些标签，它必须做额外的工作。具体来说，它会将 `<` 和 `>` 字符转换为恰好显示相同的其他字符。这种转换称为 *转义特殊字符*。具体来说，字符 `<` 被替换为 `&lt;`，字符 `>` 被替换为 `&gt;`。每个报表列在其 `Security` 部分都有一个 `Escape special characters` 属性。默认情况下它是开启的，这会导致 APEX 转义特殊字符，并如 13-18 图所示显示 HTML 标签。现在转到 `Loc` 列并关闭此属性。`Loc` 值中的标签将不会被转义，因此将被作为 HTML 处理。报表现在将看起来像图 13-17。

这种在 SQL 源查询中格式化数据的能力非常强大，可以让你完成通过 `HTML Expression` 属性无法实现的事情。例如，你可以选择将一列中的某些值格式化为文本，而将其他值格式化为链接或图像。然而，这种强大功能是有代价的。一旦你关闭了一列的 `Escape special characters` 属性，你就使自己面临跨站脚本攻击的风险。该属性的帮助屏幕对此点特别强调；参见图 13-19。

前面的 JavaScript 代码并无恶意。`alert` 函数只是在用户浏览器中显示一个警告窗口。恶意的 JavaScript 字符串几乎肯定不会弹出警报，而是会做一些更复杂和恶意的事情。很可能，当你发现问题时，为时已晚。因此，你需要确保这种情况不会发生。

消除跨站脚本攻击的唯一方法是确保页面不显示未转义的数据。有三种方法可行，我们将依次考虑：

-   拒绝不适当的数据
-   在将数据保存到数据库之前对其进行转义
-   在显示数据之前对其进行转义

### 拒绝不适当的数据

在任何数据录入页面中，防止不适当的数据被输入数据库是良好的实践。这就是数据库约束和 APEX 验证的用途。检查输入数据是否包含意外 HTML 的行为遵循相同的原理。你可以采取两种方法。

第一种方法是使用项目 `Security` 部分中的 `Restricted Characters` 属性。此属性允许你指定输入中允许哪些字符。图 13-21 显示了可能的选项。你可以要求输入只能是字母数字，或者可以限制使用某些特殊字符。

![../images/335103_3_En_13_Chapter/335103_3_En_13_Fig21_HTML.jpg](img/335103_3_En_13_Fig21_HTML.jpg)

在当前示例中，假设部门位置是字母数字；在这种情况下，将 `P49_LOC` 的 `Restricted Characters` 属性设置为 `Whitelist for a–Z, 0–9, and space` 是合理的。如果不是，那么你至少应该将其设置为 `Blacklist HTML command characters`。

```plsql

  v_LocEscaped varchar2(100);
begin
  v_LocEscaped := apex_escape.html(:P49_LOC);
  if :P49_LOC = v_LocEscaped then
    return true;
  else
    return false;
  end if;
end;
```

## URL 修改
第三种恶意攻击利用了网页的访问方式。在一个典型的非恶意场景中，用户将应用程序主页的 URL 输入浏览器。从此之后，URL 在很大程度上就不重要了；用户通过点击标签、链接和按钮导航到其他页面。
这个场景对应于应用程序的预期用途。页面上的导航链接旨在帮助用户决定接下来要查看哪些页面。事实上，衡量应用程序质量的一个标准是导航在多大程度上有助于形成连贯的用户体验。
当然，用户不一定遵循这个场景。了解应用程序 URL 结构的用户可以通过简单地构造一个有效的 URL 来访问任意页面。这种可能性给 APEX 开发人员带来了几个困难。以下小节将探讨这些困难及其解决方案。

### APEX URL 结构
APEX 应用程序支持两种不同的 URL 格式：一种是旧版语法，另一种是更新的“友好”语法。本书中使用的语法一直是旧版语法，其中应用程序和页面通过其编号来指定。例如，我的 `Employee Demo` 应用程序的编号是 91392，主页的编号是 1。因此，我的主页的旧版 URL 是

```
apex.oracle.com/pls/apex/f?p=91392:1
```

友好 URL 语法则通过名称来指定工作区、应用程序和页面，而不是编号。例如，我的主页的友好 URL 是

```
apex.oracle.com/pls/apex/understanding_apex/r/employee-demo/home
```

尽管在本章中我将继续使用旧版 URL，但大多数概念也适用于友好 URL。（即使您偏好友好 URL，您仍然需要了解恶意用户如何滥用旧版 URL。）
一个旧版 URL 支持九个冒号分隔的参数；如果某个参数的值缺失，则使用默认值。您已经知道了前三个参数：应用程序 ID、页面 ID 和会话 ID。另外两个相关的参数是第七个和第八个，它们用于设置会话状态。

关于这些参数的示例，请考虑第 6 章中的 `Employees by Department` 页面，该页面再次出现在图 13-22 中。

### GET 与 POST
浏览器向 Web 服务器发出的请求可以是不同类型。最常见的两种类型是 GET 和 POST，它们在向服务器传递信息的方式上有所不同。POST 请求将信息放置在请求数据包内的一个特定区域。而 GET 请求则通过其 URL 将信息传递给服务器。
在 APEX 中，POST 请求由提交操作发出。为了处理一个提交，浏览器提取页面项的值，将它们打包到 POST 请求中，然后将其发送到服务器。服务器随后使用这些值来更改会话状态。如果您想验证这一点，请转到任何带有提交按钮的页面（例如 `Filter by Job and Department`）。点击提交按钮，并观察 URL 没有改变。这证明了计算的输入不是所提交 URL 的一部分，这也是您无法将提交操作的结果加入书签的原因。
另一方面，APEX 中的重定向操作发出的是 GET 请求。重定向由多种 APEX 组件执行，例如按钮、基于列表的项、列链接和分支。作为重定向的一部分，APEX 允许您将会话状态中的指定项设置为特定值。例如，在图 13-22 的 `Employees by Department` 页面中，`Click for Details` 列链接将 `P12_DEPTNO` 和 `P12_DNAME` 分别设置为当前记录的部门编号和名称。浏览器通过将此信息编码在 URL 内将其发送到服务器，如代码清单 13-11 所示。
由于 APEX URL 的结构是众所周知的，一个经过身份验证的用户有可能构造一个 URL，以应用程序开发人员未曾意图的方式访问应用程序。接下来的章节将考虑两种技术。

### 页面扫描


## 页面扫描与项目修改安全

由于没有好的方法来消除页面扫描，开发者的唯一对策是使用授权来限制可访问性。换句话说，页面扫描的可能性意味着为每个页面定义授权方案至关重要，无论其在导航菜单中的可见性如何。

### 项目修改

第二种技术，*项目修改*，可用于攻击那些通过 URL 设置项目值的页面。用户获取该页面的现有合法 URL，修改其赋值，并提交修改后的 URL。

例如，假设公司有第五个部门，其部门编号为 50，名称为 `SECRET`。假设你不希望该部门的成员公开，因此你将图 13-22 中 `Departments` 报告的数据源设置为排除该部门。不幸的是，很容易使用 URL 修改来获取此隐藏信息。用户只需修改清单 13-11 中的 URL，使其请求第 50 号部门的内容，如下所示：

```
f?p=91392:12:98765::::P12_DEPTNO,P12_DNAME:50,SECRET
```

再举一个例子，考虑图 10-9 的 `Form Update` 页面。图 13-23 显示了我在 `Choose Employee` 项目中选择 BLAKE 后，其 `Update Employees` 区域的情况。这个名为 `P30_EMPNO` 的项目执行了一个 `Redirect and Set Value` 操作。重定向操作生成的 URL 是：

APEX 执行请求的过程如下。它首先通过将项目值从浏览器复制到会话状态来执行提交；然后，它将 URL 中指定的值复制到两个指定的项目中，并呈现页面。因此，页面上选择的 `P9_JOB` 和 `P9_DEPTNO` 的值将被 URL 中的值覆盖。也就是说，生成的报告将列出会计部门的经理，而不管选择列表中显示的值是什么！

项目修改的可能性意味着你不能依赖基于列表的项目来对输入数据施加任何类型的限制。这对 SQL 注入和跨站脚本攻击有影响。回想一下，对这些技术的讨论指出，攻击者需要足够长的基于文本的项目进行数据输入，因此建议使用基于列表的项目。然而，攻击者可以使用项目修改直接将攻击代码输入到 URL 中。因此，除非有办法禁止项目修改，否则使用基于列表的项目无法提供安全性（并且实际上会给人一种虚假的安全感）。

### 防范项目修改

APEX 有一项名为 *页面访问保护* 的功能，其唯一目的是防止项目修改攻击。有多个保护级别。本节将仅讨论最通用的一种：`Arguments Must Have Checksum`。它的工作原理如下。

假设一个页面在此级别受到保护。如果该页面的 URL 分配了项目值，则该 URL 还必须包含这些值的校验和。当 APEX 服务器收到请求时，它会提取参数值，计算它们的校验和，并将其与 URL 中的校验和进行比较；如果不同，则拒绝该请求。

其理念是校验和函数只有 APEX 知道。当 APEX 为重定向操作生成 URL 时，它会计算校验和并将其添加到 URL 中。如果用户以任何方式修改了项目值，现有的校验和将不再正确；此外，用户将无法知道如何计算正确的值。同样，用户无法从零开始创建保存项目的 URL，因为无法确定其校验和。因此，项目修改是不可能的。

为页面启用页面访问保护有两个步骤。第一步是启用应用程序的 `Session State Protection` 属性。从应用程序的主屏幕开始，单击 `Edit Application Properties` 按钮，然后单击 `Security` 选项卡。该属性位于 `Session State Protection` 部分。很可能，该属性已经启用；如果未启用，请选择 `Enabled`。第二步是转到该页面的页面属性。`Page Access Protection` 属性位于 `Security` 部分；选择选项 `Arguments Must Have Checksum`。

作为测试，为 `Employees by Department` 页面设置页面访问保护。然后转到该页面并单击会计部门的链接。你应该看到一个类似以下的 URL：

```
f?p=91392:12:98765::::P12_DEPTNO,P12_DNAME:10,ACCOUNTING&cs=10FzVgvoxRFQBVRnEe9Ww2QSzoVM
```

请注意，此 URL 与清单 13-11 中的 URL 相同，只是末尾多了校验和信息。现在尝试对此 URL 执行项目修改（无论是否更改校验和值）。APEX 会检测到修改，并返回图 13-25 中显示的错误消息。

## 安全性

### 页面访问保护的复杂性

还有一个额外的复杂情况：如果你想启用页面访问保护，就需要为应用程序的每一个页面都启用它，因为一个页面的 URL 可以设置另一个页面的项。

例如，假设`按部门查看员工`页面（演示应用的第 12 页，图 13-22）已受到保护。现在考虑一下应用程序的首页，即第 1 页。由于它没有任何项，你可能会认为没有必要要求校验和。但是，一个聪明的用户可以发出如下 URL 请求：

```
f?p=91392:1:98765::::P12_DEPTNO,P12_DNAME:10,ACCOUNTING
```

这个请求在加载首页之前，就设置了`P12_DEPTNO`和`P12_DNAME`的会话状态值。然后用户可以从首页正常导航到`按部门查看员工`页面。如果这些项的值来源于当前的会话状态值（通常情况如此），页面将使用用户指定的值进行渲染。换句话说，尽管你本意良好，用户还是成功修改了项。

## 总结

本章重点讨论了如何确保应用程序的安全性。它讨论了如何配置应用程序，使得每个用户只能执行他们应该执行的操作——不多也不少。安全性问题涉及三个方面：认证、授权和建立防护措施。

认证确保应用程序知道其用户的身份。用户提供用户名和密码来表明自己的身份；应用程序的认证方案负责将用户名/密码与主列表进行匹配。APEX 支持多种认证方案，它们根据管理用户/密码列表的组织部分不同而有所区别。你学习了如何构建一个自定义认证方案，其中应用程序本身负责维护该列表。

授权指定了每个已认证的用户可以做什么。基本的授权机制是授权方案。每个授权方案指定了一组用户。通过将授权方案与页面（或页面组件）关联，你可以将该页面（或组件）的访问权限限制在方案指定的用户。

防护措施对于防止用户执行他们未被授权的操作是必要的。本章涵盖了三种众所周知的技术：SQL 注入、跨站脚本和 URL 修改。在 SQL 注入中，用户将 SQL 代码片段作为“数据”提交，导致底层过程执行错误的查询。在跨站脚本攻击中，用户将恶意的 JavaScript 代码片段作为“数据”保存；当另一个用户试图显示该数据时，恶意代码就会执行。在 URL 修改中，用户向 APEX 服务器发送一个服务器期望的 URL 形式；然而，该 URL 请求的操作却是用户未被授权请求的。你研究了防范这些技术的方法。关键在于，如果你意识到这些风险，确保应用程序的整体安全性就相对直接了。
