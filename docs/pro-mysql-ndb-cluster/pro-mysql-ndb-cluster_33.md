提示

检查核心转储文件时，您经常会看到 `<optimized out>` 字符串。这表明所讨论的变量仅存储在寄存器中，而未存储在内存中。在这种情况下，您无法直接从核心转储文件中检索变量的值。


##### OOM Killer

若发现`mysqld`进程消失且错误日志中无任何输出，则该进程很可能被其他程序终止。在 Linux 系统上，导致`mysqld`被终止的最常见原因是 OOM killer（内存不足终止程序）。

由于 Linux 内核允许内存超量分配，可能会在进程获得分配器返回的堆内存指针后出现内存短缺。此时，内核可能仅分配了虚拟内存空间，而到物理内存的映射可能尚未完成。物理内存是在进程实际访问该内存区域时才进行分配的。这种行为被称为**延迟页面分配**或**乐观内存分配**。在某些情况下，该机制可以节省内存，因为并非所有分配的内存都必然被使用——进程最终可能只访问了已分配内存的一部分。这同时也节省了 CPU 时间，因为物理内存分配是一项繁重的操作。该机制还提高了进程的可用性，因为即使分配的内存大小超过实际可用内存，在进程访问已分配的内存区域之前，都不会检测到内存不足错误。另一方面，如果内核预期内存将会短缺，它会返回一个内存不足错误。因此，延迟页面分配在大多数情况下都能正常工作。

延迟页面分配的主要缺点是，即使内核对内存分配返回了成功，内存短缺问题也可能在事后才显现出来，因为它是乐观的。在这种情况下，系统可能由于内存短缺而无法继续执行。OOM killer 程序通过终止一个正在运行的进程来解决这种问题。`mysqld`进程以及`ndbmtd`和`ndbd`很可能是 OOM killer 的目标，因为这些进程会消耗大量内存。
