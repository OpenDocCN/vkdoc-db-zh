# 4. 配置

在系统上安装 MySQL NDB Cluster 软件包之前，本章将讨论其配置。你可以按喜好先执行配置或先安装软件包。但是，配置必须在首次启动集群之前完成。虽然大多数选项可以通过滚动重启来更改，但初始配置必须至少足够实用。由于滚动重启是一项耗时的操作，因此尽可能仔细地审查选项以减少配置更改的必要性是个好主意。

### 配置概述

在 MySQL NDB Cluster 上，有两种类型的配置：

*   `config.ini`：主要用于定义安装布局和资源分配。此文件由管理节点读取并分发给其他类型的节点。此配置文件定义了集群的拓扑结构。
*   命令行选项和 `my.cnf`：主要用于定义每个进程的启动行为并微调 SQL 节点。所有类型的进程都有自己的命令行选项，并从 `my.cnf` 读取选项。SQL 节点有多种命令行选项用于定制 SQL 的各个方面。

当使用 MySQL Cluster Manager (MCM) 时，配置集群的方式不同，并且这些文件不会被直接编辑。有关 MCM 的更多信息，请参见第 13 章。

定义集群安装的整体蓝图是一个重要过程，因为之后不容易更改。微调 SQL 节点和启动选项不如 `config.ini` 重要，因为对 `my.cnf` 的更改可以通过重启 SQL 节点来应用，因此它们可以随意更改。

以下各节详细说明了主要的配置选项。由于本章内容较长，你可以现在跳过细节，等到需要一个关于主要选项的一站式参考时再返回本章。

#### `config.ini` 的格式

在讨论每个配置选项的细节之前，我们先介绍 `config.ini` 的内容：它的组织方式以及如何正确编写选项。`config.ini` 由以方括号括起的节名开头的节组成。有两种类型的节：

*   用于单个节点或连接的配置。通常，一个节对应一个节点。节名与节点类型相同，例如 `NDBD`。表 4-1 显示了可用的节名。

    表 4-1.

    config.ini 中的节名

    | 节名 | 描述 |
    | --- | --- |
    | `MGM` 或 `NDB_MGMD` | 管理节点的配置。 |
    | `NDBD` | 数据节点的配置。 |
    | `API` 或 `MYSQLD` | SQL 节点或 NDB API 客户端应用程序的配置。 |
    | `TCP` | 节点互连的配置。 |
    | `SCI` | SCI 连接的配置。 |
    | `COMPUTER` | 为每个节点命名，不同于其主机名或 IP 地址。 |
    | `SHM` | 共享内存连接的配置。 |

*   一种节点类型的默认配置。此类型的节必须位于所有单个节点节之前。节名是节点类型加上 `DEFAULT` 字符串，例如 `NDBD DEFAULT`。

在本书中，我们不讨论以下节。

*   `SCI`：对 SCI 的支持已过时。此外，使用 SuperSockets 时不需要此节。
*   `COMPUTER`：不常用。
*   `SHM`：共享内存连接目前是实验性的。

通常，`config.ini` 以几个默认节开始，后跟各个节点节。每个节由选项名和值组成。与 `my.cnf` 相比，`config.ini` 不能包含重复的选项。每个选项应只指定一次。

列表 4-1 展示了一个小型系统的 `config.ini` 示例。

```
[MGM]
NodeId = 255
HostName = mgmhost
DataDir = /var/lib/mysql-cluster
[NDBD DEFAULT]
NoOfReplicas = 2
DataDir = /var/lib/mysql-cluster
DataMemory = 4G
IndexMemory = 512M
[NDBD]
NodeId = 1
HostName = host1
[NDBD]
NodeId = 2
HostName = host2
[MYSQLD]
NodeId = 49
HostName = host1
[MYSQLD]
NodeId = 50
HostName = host2
清单 4-1.
一个小系统的 config.ini 示例
```

在此示例中，只配置了五个节点——一个管理节点、两个数据节点和两个 SQL 节点。数据节点和 SQL 节点的主机名重叠，因此它们被放置在相同的主机（`host1`，`host2`）上。管理节点放置在单独的主机（`mgmnost`）上，因此总共涉及三台主机。

当每个数据节点和每个 SQL 节点连接到管理服务器时，配置会分发给它们。当一个节点连接到管理节点时，管理节点通过比较连接节点的主机名和每个节下的 `HostName` 选项来确定哪个槽位与该连接节点匹配。例如，如果 `ndbmtd` 从 `host2` 连接到管理节点，则其节点 ID 被识别为 2，因为有一个 `NDBD` 节的配置为 `NodeId = 2` 和 `HostName = host2`。如果有另一个 `ndbmtd` 试图从 `host2` 连接到管理节点，连接将失败，因为没有可用的空闲数据节点槽位给从 `host2` 连接的节点。但是，即使数据节点的槽位被占用，`mysqld` 仍然可以从 `host2` 连接到管理节点，因为 `host2` 上有一个可用的 SQL 节点槽位。



### 重启类型

在集群启动后应用配置时，需要重启节点。在以下章节中，每个选项都配有重启类型的说明，这指的是目标节点的重启方式。在 MySQL NDB Cluster 中，定义了四种重启类型：

*   **节点重启 (`N`)**: 一次重启一个节点，无需指定额外指令。若要将配置更改应用到整个集群，应依次重启每个节点。此操作也称为滚动重启。在滚动重启期间，集群保持运行，应用程序可以访问数据。当 `NoOfReplicas = 1` 时，无法执行节点重启，因为同一节点组内没有存活的数据节点。
*   **系统重启 (`S`)**: 所有节点停止（集群完全关闭），然后集群再次启动。在系统重启期间，应用程序无法访问数据，因为集群已完全停止。
*   **初始节点重启 (`IN`)**: 此重启类型与节点重启类似，但每个数据节点在重启时使用 `--initial` 选项。此选项指示目标数据节点清除其数据，然后从同一节点组内的另一个节点复制全部数据。例如，当更改了与文件相关的选项时，就需要这种重启类型。
*   **初始系统重启 (`IS`)**: 此重启类型与系统重启类似，但所有数据节点在重启时使用 `--initial` 选项。这会导致所有数据被清除并丢失。如果您在重启后需要保留现有数据，请在执行系统初始重启之前进行完整备份。

重启类型和操作的详细信息在第 10 章中讨论。

### 管理节点选项

本节涵盖管理节点选项。管理节点的选项并不多。在本节中，仅列出主要选项。

#### 管理节点的主要选项

##### NodeId

| 默认值 | 自动 |
| :--- | :--- |
| 范围 | 1 – 255 |
| 所属节 | `[MGM]` |
| 重启类型 | 节点重启 |

每个节点的标识符。在之前的版本中，`Id` 选项与 `NodeId` 含义相同，但在 MySQL NDB Cluster 7.5 中已被移除。即使不指定 `NodeId` 也可以配置每个节，因为如果未显式指定，其值会自动分配。然而，省略此参数并不是一个好的做法，因为它会导致配置的歧义。

##### HostName

| 默认值 | 无（接受来自任何主机的连接） |
| :--- | :--- |
| 范围 | 任意主机名或 IP 地址 |
| 所属节 | `[MGM]` |
| 重启类型 | 系统重启 |

节点所在的主机名或 IP 地址。您可能会疑惑“为什么管理节点需要知道它自己的主机名？”。当安装了多个管理节点时，需要此主机名。

##### DataDir

| 默认值 | 进程启动时的当前目录 |
| :--- | :--- |
| 范围 | 任意目录路径名 |
| 所属节 | `[MGM]` 或 `[MGM DEFAULT]` |
| 重启类型 | 节点重启 |

管理节点生成的数据（如日志文件和 PID 文件）存储在此目录中。

##### PortNumber

| 默认值 | 1186 |
| :--- | :--- |
| 范围 | 任意端口号 |
| 所属节 | `[MGM]` 或 `[MGM DEFAULT]` |
| 重启类型 | 节点重启 |

这是管理节点监听的端口号。在大多数情况下，无需更改此值。当默认端口号 (1186) 被另一个程序占用时，您应将其更改为不同的值以避免端口冲突。例如，如果您在本地网络中安装了多个集群，您可能希望将管理节点整合到单个主机上以节省主机数量。这种配置是可行的，因为管理节点需要的计算机资源非常少。请注意不要弄错每个节点连接字符串中的端口号。如果指定了错误的连接字符串，节点将连接到不同的管理节点。

##### ArbitrationRank

| 默认值 | 1 |
| :--- | :--- |
| 范围 | 0 – 2 |
| 所属节 | `[MGM]` 或 `[MGM DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定成为仲裁器的可能性。此参数的范围是 0、1 和 2。每个值的含义如下：

*   `0`: 该节点永远不会被用作仲裁器。
*   `1`: 该节点具有高优先级；即，与低优先级节点相比，它将优先被选为仲裁器。
*   `2`: 表示低优先级节点，仅当所有更高优先级的节点都不可用时，它才会被用作仲裁器。

如果您有多个管理节点，并且对哪个节点应成为仲裁器有偏好，请将不太理想的管理节点的此选项设置为 2。请注意，不仅管理节点，SQL 节点也可以成为仲裁器。即使您只有一个可用的管理节点，通过使用 SQL 节点作为额外的仲裁器，仲裁器也可以实现容错。注意不要使用与数据节点位于同一主机上的 SQL 节点，因为仲裁器应位于与任何数据节点分开的主机上。


##### LogDestination

| 默认值 | `FILE:filename=ndb_node_id_cluster.log,maxsize=1000000,maxfiles=6` |
| 范围 | `CONSOLE&#124;SYSLOG&#124;FILE` 及其他附加参数 |
| 所属章节 | `[MGM]` 或 `[MGM DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定集群日志的输出目标。集群日志是 MySQL NDB Cluster 的集中式事件日志文件，所有事件报告都聚合为一个连续的日志。目标有以下三种类型：

*   `CONSOLE`：日志条目输出到 `ndb_mgmd` 进程的标准输出。
*   `SYSLOG`：日志条目发送到 `syslog` 工具。
*   `FILE`：日志条目写入日志文件。日志文件以循环方式复用。

`SYSLOG` 和 `FILE` 需要附加参数来指定日志规范的详细信息。

对于 `SYSLOG` 输出，应指定工具。工具指明了生成日志消息的组件。工具不是任意字符串，而是在 RFC 3164 的 syslog 协议中定义的。已定义的工具列于表 4-2。如清单 4-2 所示，使用关键字指定工具。

表 4-2. Syslog 工具

| 数值代码 | 关键字 | 工具 |
| --- | --- | --- |
| 0 | kern | 内核消息 |
| 1 | user | 用户级消息 |
| 2 | mail | 邮件系统 |
| 3 | daemon | 系统守护进程 |
| 4 | auth | 安全/授权消息 |
| 5 | syslog | 由 syslogd 内部生成的消息 |
| 6 | lpr | 行式打印机子系统 |
| 7 | news | 网络新闻子系统 |
| 8 | uucp | UUCP 子系统 |
| 9 |   | 时钟守护进程 |
| 10 | authpriv | 安全/授权消息 |
| 11 | ftp | FTP 守护进程 |
| 12 |   | NTP 子系统 |
| 13 |   | 日志审计 |
| 14 |   | 日志告警 |
| 15 | cron | 时钟守护进程（调度守护进程） |
| 16 ∼ 23 | local0 ∼ local7 | 本地使用 0 ∼ 7 |

清单 4-2. 使用 SYSLOG 的 LogDestination
```
LogDestination = SYSLOG:facility=local0
```

工具通常用于日志过滤。良好的做法是指定一个不与其他应用程序、系统程序或内核重叠的工具。`local0` ∼ `local7` 适用于此目的。它们是所谓的自定义日志消息，操作系统不使用。默认工具是 `user`。

可以将具有特定工具的日志消息存储到单独的日志文件中。在这种情况下，您可以根据需要自定义日志文件名、日志轮换等。

甚至可以使用 syslog 功能将集群日志定向到远程主机（日志服务器）。当您有多个集群安装时，这种配置很有用。您可以使用单个日志实体监控所有集群。更多信息请参考 syslog 手册。有几种 syslog 实现。具体配置取决于实现方式。

对于 `FILE` 输出，存在以下三个参数：

*   `filename`：日志文件名
*   `maxsize`：每个日志文件的最大大小
*   `maxfiles`：保留的日志文件的最大数量

`FILE` 输出的默认值如清单 4-3 所示。

清单 4-3. 使用 FILE 的 LogDestination
```
LogDestination = FILE:filename=ndb_{node_id}_cluster.log,maxsize=1024000,maxfiles=6
```

清单 4-3 中的 `{node_id}` 表示管理服务器的节点 ID。您不一定需要指定所有参数，但可以指定参数的任意组合。例如，您可以只指定 `maxsize`，而保持 `filename` 和 `maxfiles` 不变。

我建议您将集群日志的大小增加约 10 倍。默认大小太小，日志轮换太快。

#### 管理节点配置的要点

管理节点没有很多选项。绝大多数选项都可以在 `config.ini` 文件中省略。但是，为了稳健的设置，我建议至少设置以下三个选项：

*   `NodeId`：节点 ID 默认是自动分配的；然而，这可能导致后续分配不同的标识符。
*   `HostName`：如果未显式指定主机名，管理节点可以从任何主机连接。
*   `DataDir`：默认数据目录是启动管理节点时所在的当前目录。当从 shell 启动管理节点进程时，该目录可能会改变。

显式设置这些选项将消除管理节点配置的模糊性。

### 数据节点选项

我们开始讨论数据节点选项。如第 2 章所述，数据节点是 MySQL NDB Cluster 的核心。配置数据节点是设置 MySQL NDB Cluster 时最困难的阶段之一，因为存在大量选项。然而，如果换个角度看，数据节点的配置是 MySQL NDB Cluster 生命周期中最具挑战性和最令人兴奋的任务之一。

由于数据节点选项开发非常活跃，它们经常发生变化或新增。如果选项在 7.2 或更新版本中发生变化或引入，章节中会标明有效的版本号。

#### 基本选项

以下选项在数据节点配置中是强制性的。为了稳健的安装，请显式设置这些选项。

##### NodeId

| 默认值 | 自动分配 |
| 范围 | 1 – 48 |
| 所属章节 | `[NDBD]` |
| 重启类型 | 节点重启 |

这是节点的标识符，与其他节点类型的 `NodeId` 含义相同。但是，可能的取值范围是 1 – 48，因为数据节点的最大数量应小于或等于 48。因此，良好的做法是让其他节点类型的 `NodeId` 大于 48。

##### HostName

| 默认值 | 无（接受来自任何主机的连接） |
| 范围 | 任意主机名或 IP 地址 |
| 所属章节 | `[NDBD]` |
| 重启类型 | 系统重启 |

节点所在的主机名或 IP 地址。您可以不设置此选项，但我不建议您这样做。为了使数据节点配置稳健，您需要设置此选项，并使主机名对该插槽是唯一的。

`HostName` 和 `NodeId` 用于在节点连接时标识插槽。因此，这些选项对于每个插槽是唯一的，不能写在 `[NDBD DEFAULT]` 部分，因为它们不能在多个插槽之间共享。

##### ServerPort

| 默认值 | 无（自动分配） |
| 范围 | 1 – 64K |
| 所属章节 | `[NDBD]` 或 `[NDBD DEFAULT]` |
| 重启类型 | 系统重启 |

此选项通过端口号指定，用于接受来自其他节点的连接。默认情况下，数据节点请求操作系统将端口号分配为临时端口号。例如，在 POSIX 系统上，`bind(2)` 系统调用会执行此操作。

##### DataDir

| 默认值 | 启动进程时所在的当前目录 |
| 范围 | 任意目录路径名 |
| 所属章节 | `[NDBD]` 或 `[NDBD DEFAULT]` |
| 重启类型 | 初始节点重启 |

此选项指定数据所在的目录。

由于数据节点处理各种类型的数据，因此可以根据数据类型指定不同的目录。`DataDir` 被视为特定目录路径选项（如 `FileSystemPath`）的默认值。

日志文件（不是事务日志文件，而是 `ndbd` 或 `ndbmtd` 进程的日志文件）也存储在 `DataDir` 下，无法单独指定。

并非所有数据节点的 `DataDir` 都必须相同。

##### `NoOfReplicas`

| 默认值 | 2 |
| --- | --- |
| 范围 | 1 – 4（实际上，2 是唯一的选择） |
| 所属节 | `[NDBD DEFAULT]` |
| 重启类型 | 初始系统重启 |

此选项指定数据的副本数量。理论上，副本数量越多，集群就越健壮。然而，三副本或四副本尚未经过官方测试和支持。一个副本意味着没有冗余；单节点故障会立即导致系统故障。目前你应选择两个副本。未来的版本可能会支持三或四副本，但目前还不确定。

有关副本的更多信息，请参阅第 2 章和第 3 章。

#### 内存数据存储选项

以下选项与内存容量规划相关。内存容量规划的策略并不难；除非系统发生内存交换，否则应分配系统拥有的尽可能多的内存。请注意，用于模式和事务处理的对象也会消耗一定量的内存。重要的是，不要在此节中为缓冲区分配过多的内存。要为它们留出余地。

##### `DataMemory`

| 默认值 | 80M |
| --- | --- |
| 范围 | 1M – 1024M (字节) |
| 所属节 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定存储行数据和有序索引数据的内存区域的大小。此选项决定了数据大小的上限。人们常误以为有序索引存储在 `IndexMemory` 中，但实际上它们存储在 `DataMemory` 中。从技术上讲，可以在 `[NDBD]` 节下写入此选项，但这没有意义。数据会在所有数据节点之间平均分配，每个数据节点上的数据量大致相同。对于本章中以下选项，列表中只写了 `[NDB DEFAULT]` 节，但适用类似的推理。

整个集群的总可用内存大小按以下公式计算：

```
DataMemory × number_of_node_group
```

节点组的数量是数据节点数除以 `NoOfReplicas`。因此，整个集群在没有单点故障的情况下，理论上的最大可用数据内存是 24TB（48 个数据节点，每个数据节点 1TB，2 个副本）。`DataMemory` 的数量也对 LCP、GCP 和备份的大小有影响。内存大小对存储需求也有很大影响。

##### `IndexMemory`

| 默认值 | 18M |
| --- | --- |
| 范围 | 1M – 1024M (字节) |
| 所属节 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定用于哈希索引的内存大小，哈希索引用于主键和唯一哈希索引。键数据和额外开销（包括指向行数据的指针）存储在 `IndexMemory` 中。所需的 `IndexMemory` 大小大致取决于行的数量。还存在一些其他开销，除非你有大量小表，否则它们可以忽略不计。每行的数据大小可使用以下公式估算：

```
Number_of_Rows * 18 * NoOfReplicas
```

计算出的值表示整个集群所需的总内存量。因此，每个节点所需的内存量通过将其除以数据节点数来计算。例如，如果表有 1,000,000,000 行，集群有 10 个数据节点，且 `NoOfReplicas` = 2，则该表每个数据节点所需的 `IndexMemory` 大小如下：

```
1000000000 * 18 * 2 / 10 = 3.6GB
```

此公式仅是近似值，实际分配的内存量很可能小于计算值。在 MySQL NDB Cluster 7.5 系列或更高版本上，你可以使用 `ndbinfo.memory_per_fragment` 表查询确切的已分配内存大小。有关 `ndbinfo` 模式的更多信息，请参阅第 16 章。

##### `StringMemory`

| 默认值 | 25 |
| --- | --- |
| 范围 | 1 – 100 (百分比) 或 101 – 4294967039 (字节) |
| 所属节 | `[NDBD DEFAULT]` |
| 重启类型 | 系统重启 |

此选项指定用于对象名称的内存。当值小于 100 时，表示相对于根据需要名称的对象数量（如 `MaxNoOfTables`、`MaxNoOfAtrributes`、`MaxNoOfOrderedIndexes`、`MaxNoOfUniqueHashIndexes` 和 `MaxNoOfTriggers`）计算出的理论最大所需内存大小的百分比。这些选项将在本章后面描述。在大多数情况下，默认值是足够的。

当 `StringMemory` 不足时，你在创建新表或添加列时会看到以下消息：

```
Got error 773 'Out of string memory, please modify StringMemory config parameter' from NDBCLUSTER
```

在这种情况下，你需要增加 `StringMemory`。

##### `SharedGlobalMemory`

| 默认值 | 128M |
| --- | --- |
| 范围 | 0 – 64T |
| 所属节 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定共享全局内存的大小，该内存用于多种用途：

*   磁盘数据表的撤销日志缓冲区
*   日志文件组和表空间的元数据
*   下推连接
*   模式事务

当你使用磁盘数据表和/或并行执行大量下推连接时，请增加此选项。`SharedGlobalmemory` 的实际所需大小通常比预期的要大。

##### `LockPagesInMainMemory`

| 默认值 | 0 |
| --- | --- |
| 范围 | 0 – 2 |
| 所属节 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项控制是否使用 `mlock(2)` 锁定内存。当内存被锁定时，在系统内存不足时，数据节点进程不太可能被换出。例如，当复制大量文件并消耗大量文件系统缓存时，可能会发生内存短缺。

此选项接受以下值之一：

*   0：不锁定内存。
*   1：在分配所需内存区域后锁定内存。
*   2：在分配内存之前设置锁定模式。

1 和 2 之间的区别微不足道。无论如何都需要足够的系统内存。

##### `Numa`

| 默认值 | 1 (True) |
| --- | --- |
| 范围 | 0, 1 (布尔值) |
| 所属节 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项在 Linux 系统上导致内存分配交错。

如第 3 章所述，近期的 x86_64 CPU 内部集成了内存控制器，每个内存控制器拥有自己的内存。如果服务器有多个 CPU 芯片，每个 CPU 对目标内存的访问速度取决于内存位于哪个 CPU 上，这种访问速度是非均匀的。这种架构被称为非统一内存访问（NUMA）。默认情况下，Linux 会尝试在运行线程的同一个 CPU 上分配内存。这种策略对于小程序效果很好，因为本地内存访问比远程内存访问更快，单线程运行的程序所需内存较小。然而，对于需要多线程和大量内存的服务器程序来说，这会导致问题，因为内存在 NUMA 节点之间的分配是非均匀的。

此选项导致内存分配交错，这意味着内存从所有 NUMA 节点均匀分配。默认值启用 NUMA 交错，你通常不需要更改它。一个不寻常的情况是，如果你在单个 NUMA 机器上运行多个数据节点，并将每个数据节点绑定到单独的 NUMA 节点。

#### 模式对象选项

在 MySQL NDB 集群上，模式对象的元数据存储在数据节点启动时分配的固定大小数组中。各种对象的最大允许数量由以下选项配置。重要的是要事先为每个模式对象分配所需的大小。模式对象设计将在第 18 章中介绍。

##### MaxNoOfTables

| 默认值 | 128 |
| 范围 | 8 – 20320 |
| 小节 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定集群中允许的最大表数。请注意，它包括系统表，例如 `ndb_schema`、`ndb_apply_status` 等。唯一哈希索引会创建一个内部支持表，因此它消耗一个表对象，并根据应用需求确定最佳表数。

##### MaxNoOfAttributes

| 默认值 | 1000 |
| 范围 | 32 – 4294967039 |
| 小节 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定属性的数量。属性是属于表的元素，例如列或索引。每个对象消耗的属性数量因对象类型而异，具体描述如下：

*   非 BLOB 列：1
*   BLOB (TEXT) 列：5
*   主键中的哈希索引：0
*   唯一哈希索引：1 加上索引中的列数
*   有序索引：1 加上索引中的列数

您可能想知道为什么主键中的哈希索引不消耗属性。它不消耗 `MaxNoOfAttributes`；相反，它消耗 `MaxNoOfTables`。

请注意，除非在 DDL 语句中指定 `USING HASH` 关键字，否则有序索引会随主键和唯一哈希索引一起创建。

由于属性数据需要在所有数据节点之间共享，因此它不像行数据那样按节点组划分。`MaxNoOfAttributes` 的值应根据模式对象的总量来决定。

每个表实际消耗的属性数量可以使用 `ndb_desc` 命令显示。但是，默认情况下，`ndb_desc` 命令的输出中不包括关于 BLOB 列和唯一哈希索引的信息。您需要指定 `--blob-info` 选项才能看到关于 `BLOB` 列的信息。`BLOB` 列的行存储在一个单独的支持表中，该表消耗四个属性。唯一哈希索引也存储在单独的支持表中，但无法查看分配给唯一哈希索引的属性数量。

##### MaxNoOfOrderedIndexes

| 默认值 | 128 |
| 范围 | 0 - 4294967039 |
| 小节 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定有序索引的数量，这些索引用于二级索引、主键和唯一键。顾名思义，有序索引用于有序索引扫描。实际上，除了全表扫描和主键或唯一键查找之外的所有操作都作为有序索引扫描处理。有序索引是处理各种类型查询的关键对象。

所需的有序索引数量取决于模式设计。您可以使用以下规则确定有序索引的数量：

*   非唯一二级索引：1
*   不带 `USING HASH` 子句的唯一二级索引：1
*   不带 `USING HASH` 子句的主键：1

每个有序索引也消耗一个 `MaxNoOfAttributes`。

##### MaxNoOfUniqueHashIndexes

| 默认值 | 64 |
| 范围 | 0 - 4294967039 |
| 小节 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定唯一哈希索引的数量。顾名思义，每个唯一索引消耗此对象的数量为 1。请根据您的应用需求增加此选项的值。

每个唯一哈希索引还消耗一个 `MaxNoOfTables` 和若干个 `MaxNoOfAttributes`，具体数量由构成索引的列数决定。

##### MaxNoOfTriggers

| 默认值 | 768 |
| 范围 | 0 - 4294967039 |
| 小节 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定数据节点内部使用的内部触发器的数量。请注意，此处的触发器与 SQL 中的触发器不同。在数据节点中，内部触发器用于各种进程：

*   更新有序索引条目
*   更新唯一哈希索引条目
*   更新 BLOB 列
*   外键检查以及父表和子表的更新
*   备份
*   复制（生成二进制日志）
*   表重组

在正常操作期间，无需修改此值，因为如果它太小，系统会自动调整。所需触发器的数量使用以下选项在内部计算：`MaxNoOfTables`、`MaxNoOfOrderedIndexes` 和 `MaxNoOfUniqueHashIndexes`。

#### 事务选项

由于 MySQL NDB Cluster 是实时数据库系统，它不会动态分配内存。相反，它在启动时分配内存。这包括事务和数据操作使用的各种缓冲区。

##### MaxNoOfConcurrentTransactions

| 默认值 | 4096 |
| 范围 | 32 - 4294967039 |
| 小节 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定在每个数据节点中分配的事务记录数。事务记录是事务协调器（TC）使用的内存缓冲区。如第 2 章所述，某个数据节点上的一个 TC 负责处理每个事务的整个生命周期。每个数据节点上都有 TC。因此，整个集群总共有 `(MaxNoOfConcurrentTransactions * 数据节点数量)` 条事务记录。

尽管其名称如此，`MaxNoOfConcurrentTransactions` 并不表示事务的数量。相反，它表示事务记录的数量。它们在逻辑上是不同的。事务记录是 TC 中用于处理事务而消耗的对象。TC 根据以下规则消耗事务记录：

*   每个活动事务一条
*   事务访问的每个表一条

因此，整个集群所需的事务记录数量可以使用以下公式计算：

```
max_connections *
(平均每个事务访问的表数 + 1) *
sql 节点数量
```

每个事务平均访问的表数因应用程序中使用的查询而异。因此，如果不检查应用程序，则无法估算。

`MaxNoOfConcurrentTransactions` 指定每个数据节点的事务记录数。它的值应是由公式计算出的值除以数据节点数。但是，在发生节点故障时，同一节点组中的幸存节点必须处理故障节点的事务。良好的做法是将该值除以节点组的数量，而不是数据节点的数量：

```
MaxNoOfTransactions =
max_connections *
(平均每个事务访问的表数 + 1) *
sql 节点数量 / 节点组数量
```

##### MaxNoOfConcurrentOperations

| 默认值 | 32K |
| 范围 | 32 - 4294967039 |
| 小节 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定操作记录的数量。在一个事务中，每更新、删除、插入或锁定一行都需要一个操作记录。此选项对每个数据节点操作的行数设定了上限。因此，在整个集群上可以操作 `(MaxNoOfConcurrentOperations * 数据节点数量)` 行。与 `MaxNoOfTransactions` 一样，为估算此选项的值，考虑节点故障，请使用节点组的数量而不是数据节点的数量。

请注意，由唯一哈希索引和 BLOB 列创建的支持表中的行也会消耗操作记录。例如，当更新具有唯一哈希索引的主表中某行的非键值时，则会共享锁定其支持表中的一行。


#### MaxNoOfConcurrentScans（最大并发扫描数）

| 默认值 | 256 |
| 范围 | 2 – 500 |
| 部分 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定每个数据节点的扫描记录数。每次扫描操作都需要一个扫描记录。正如在 `MaxNoOfOrderedIndexes` 部分所讨论的，除了主键或唯一键查找外的所有操作都是扫描操作。此外，全表扫描也需要一个扫描记录。对于整个集群，它能处理的扫描数量由以下公式计算：

```
MaxNoOfConcurrentScans * number_of_data_nodes
```

当然，对此选项的估算应考虑到节点故障。

在大多数情况下，所需的扫描记录数量会比预期的多。因此，我建议你从一开始就将其增加到最大值。即使你将其增加到 500 并频繁遇到 `“活动扫描过多”` 错误，也请考虑添加更多数据节点。

#### MaxNoOfLocalScans（最大本地扫描数）

| 默认值 | 自动 |
| 范围 | 32 - 4294967039 |
| 部分 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定本地扫描记录的数量。与负责处理访问整个集群的扫描的扫描记录不同，本地扫描记录负责访问其自身数据节点中的数据。

默认情况下，此选项使用以下公式自动计算：

```
MaxNoOfConcurrentScans * number_of_data_nodes * 4 + 2
```

在大多数情况下，默认的自动计算值是足够的，你无需更改它。然而，在旧版本（< 7.2）中，公式不同，计算出的值远小于当前版本。如果在旧版本中观察到 `“活动扫描过多”` 错误，请增加此选项。

#### MaxParallelScansPerFragment（每个分片的最大并行扫描数）

| 默认值 | 256 |
| 范围 | 1 - 4294967039 |
| 部分 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定每个分片的最大扫描数。如果应用程序的扫描访问集中在某些表上，请考虑增加此值以获得更高效的 CPU 利用率。

#### TransactionDeadlockDetectionTimeout（事务死锁检测超时）

| 默认值 | 1200 |
| 范围 | 50 – 4294967039 (毫秒) |
| 部分 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定从某个事务被阻塞到被确定为超时所经过的毫秒数。尽管它的名称如此，但它并不表示死锁超时，而是锁等待超时。因此，此选项等同于 `InnoDB` 存储引擎的 `innodb_lock_wait_timeout` 选项。

然而，MySQL NDB 集群没有死锁检测功能。因此，死锁会作为锁等待超时被检测到，因为处于死锁中的事务无论如何都无法继续进行。

通过增加此选项，事务因锁等待超时而中止的可能性会降低。但是，不建议设置过大的值，因为处于死锁中的事务也会持续很长时间。处于死锁中的事务可能会阻塞其他无辜的事务，这可能导致系统吞吐量严重下降。在非常繁忙的系统上，请考虑缩短超时时间，以避免因死锁导致的性能下降。

#### RedoBuffer（重做日志缓冲区）

| 默认值 | 32M |
| 范围 | 1M - 4294967039 (字节) |
| 部分 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定重做日志缓冲区的大小，该缓冲区用于在提交事务之前临时存储重做日志内容。所有更改在写入重做日志之前都应保存在重做日志缓冲区中。因此，重做日志缓冲区的大小应根据并发事务的数据更改大小来计算。如果你增加了 `MaxNoOfConcurrentOperations`，那么你也需要增加重做日志缓冲区的大小。

请注意，每个 LDM 线程都会分配一个大小为 `RedoBuffer` 的重做日志缓冲区。因此，重做日志缓冲区的总大小为 `RedoBuffer * number_of_ldm_threads`。

#### RedoOverCommitLimit（重做日志提交覆盖限制）

| 默认值 | 20 |
| 范围 | 0 – 4294967039 (秒) |
| 部分 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

MySQL NDB 集群有一种机制，允许针对重做日志的写入和同步操作有一定的延迟。它可以在磁盘 I/O 延迟的情况下防止性能下降。

当然，重做日志记录对于 MySQL NDB 集群不丢失已提交事务至关重要，因此不应被延迟。如果重做日志记录比事务所做的更新慢，未刷新的重做日志数据量将持续增加。事务无法比重做日志记录的速度更快地进行。然而，磁盘 I/O 速度限制了重做日志记录的速度。因此，在负载突然激增的情况下，磁盘 I/O 可能成为瓶颈。

为了解决这个问题，MySQL NDB 集群实现了对重做日志的提交覆盖。它允许未完成的磁盘刷新操作继续进行。由于 MySQL NDB 集群对节点故障具有容错能力，未刷新的重做日志中包含的数据在节点故障时不会丢失，因为同一节点组中的其他节点拥有相同的数据。

尽管重做日志记录具有容错性，但允许刷新操作无限期延迟并不是一个好主意，因为较大的值会增加在另一个数据节点不可用时丢失数据的机会。`RedoOverCommitLimit` 定义了针对重做日志磁盘刷新操作的超时（以秒为单位）。此选项与 `RedoOverCommitCounter` 结合使用。如果重做日志刷新操作耗时超过 `RedoOverCommitLimit` 秒并发生了 `RedoOverCommitCounter` 次，则挂起的事务将被中止。发生重做日志记录超时时，API 节点将按照本章后面描述的 `DefaultOperationRedoProblemAction` 中定义的方式采取操作。

#### RedoOverCommitCounter（重做日志提交覆盖计数器）

| 默认值 | 3 |
| 范围 | 0 – 4294967039 |
| 部分 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定重做日志刷新操作的超时次数。更多信息请参阅 `RedoOverCommitLimit` 的解释。

#### 总内存消耗估算

数据库管理员能够估算分配给数据库服务器的总内存大小非常重要。在 MySQL NDB 集群数据节点进程上，会分配以下类型的内存：

*   程序映像内存
*   全局缓冲区
*   模式对象和事务对象数组
*   各种缓冲区

接下来的章节将详细解释每种内存区域。

##### 程序映像内存

通常，`ndbd` 和 `ndbmtd` 会为程序可执行文件、共享库和栈内存分配内存。程序映像的总内存大小通常小于 20MB，可以忽略不计。你可以在 Linux 上使用 `/proc` 文件系统确认用于此目的的内存大小。参见 `VmExe`、`VmLib` 和 `VmStk`。


##### 全局缓冲区

全局缓冲区是为特定目的分配的大块内存。全局内存缓冲区的类型和分配的内存大小列于表 4-3。

**表 4-3. 全局内存缓冲区类型与内存大小**

| 缓冲区类型 | 内存大小 | 是否超额分配 |
| --- | --- | --- |
| 数据缓冲区 | `DataMemory + IndexMemory` | 否 |
| 作业缓冲区 | `(number_of_threads) ^ 2 + 1MB` | 否 |
| 传输缓冲区 | `SendBufferMemory * (number_of_nodes - 1)` 或 `TotalSendBufferMemory + ExtraSendBufferMemory` | 是 |
| 文件缓冲区 | `RedoBuffer` + 每个日志部分 1MB | 否 |
| 磁盘页缓冲区 | `DiskPageBufferMemory` | 否 |
| Undo 缓冲区 | 在 `CREATE LOGFILE GROUP` 语句中定义 | 是 |
| 模式事务内存 | 2MB | 是 |
| 共享全局内存 | `SharedGlobalMemory` | 否 |

当需要更多内存时，某些全局缓冲区内存可能会被超额分配。由于分配是从共享全局内存中完成的，因此在正常操作期间，数据节点进程的总内存大小不会增加。

请注意，当配置了大量线程时，作业缓冲区是一个很大的内存区域。当配置了最大线程数（72）时，它大约消耗 5GB 内存。

##### 模式和事务对象数组

MySQL NDB Cluster 有各种用途的固定大小内存区域。这些缓冲区的大小并非微不足道，因此您应该考虑这些内存大小，以避免内存超额分配。表 4-4 显示了固定大小数组的列表以及每个单位的内存大小（每个选项每增加一个单位所带来的内存大小增量）。

**表 4-4. 按选项划分的内存大小增量**

| 选项 | 每单位内存大小 |

##### 各种缓冲区

还存在更多缓冲区，例如 `LongMessageBuffer`、`UndoDataBuffer` 等。将这些缓冲区汇总用于估算是很混乱的。最好是从运行的进程确定实际的内存大小作为起点。您可以根据要修改的选项来估算内存大小。

#### 检查点选项

检查点是 MySQL NDB Cluster 中一个非常重要的过程，因为它使数据在无论计划与否的整个系统关闭情况下都保持持久性。从性能角度来看也很重要，因为它限制了写入事务的吞吐量。客户端无法提交比检查点更快的事务。因此，对于写入密集型集群，检查点选项非常重要。

正如第 2 章所讨论的，有两种类型的检查点：

*   LCP（本地检查点）：特定时间点的整个集群数据。
*   GCP（全局检查点）：所有更改的历史记录。

有关 MySQL Cluster 中检查点的更多信息，请参见第 2 章。本章稍后将介绍如何调整检查点的选项。

##### FileSystemPath

| 默认值 | `DataDir` |
| 范围 | 任意目录路径名 |
| 部分 | `[NDBD]` 或 `[NDBD DEFAULT]` |
| 重启类型 | 初始节点重启 |

此选项指定存储备份和磁盘数据表文件的目录。除了检查点，还有其他选项可用于微调目录布局。

##### NoOfFragmentLogFiles

| 默认值 | 16 |
| 范围 | 3 – 4294967039 |
| 部分 | `[NDBD DEFAULT]` |
| 重启类型 | 初始节点重启 |

此选项指定重做日志文件的数量。重做日志以循环方式写入。有关重做日志的更多信息，请参见第 2 章。

重做日志的总文件大小使用以下公式计算：

```
NoOfFragmentLogFiles * NoOfFragmentLogParts * FragmentLogFileSize
```

这些选项的默认值分别为 16、4 和 16M。`16 * 4 * 16M = 1G` 是重做日志总大小的默认值。

##### FragmentLogFileSize

| 默认值 | 16M |
| 范围 | 4M – 1G（字节） |
| 部分 | `[NDBD DEFAULT]` |
| 重启类型 | 初始节点重启 |

此选项指定每个重做日志文件的大小。更多信息请参见 `NoOfFragmentLogFiles`。如果您需要更多重做日志空间，请考虑首先增加此选项，因为每个日志文件都需要一个内存缓冲区。

##### NoOfFragmentLogParts

| 默认值 | 4 |
| 范围 | 4 – 32（仅允许 4 的倍数） |
| 部分 | `[NDBD DEFAULT]` |
| 重启类型 | 初始节点重启 |

此选项指定组内重做日志文件的数量。更多信息请参见 `NoOfFragmentLogFiles`。

##### InitFragmentLogFiles

| 默认值 | `SPARSE` |
| 范围 | `SPARSE`, `FULL` |
| 部分 | `[NDBD]` 或 `[NDBD DEFAULT]` |
| 重启类型 | 初始节点重启 |

此选项指定重做日志文件的初始化方式。默认值为 `SPARSE`，这意味着重做日志文件被创建为稀疏文件。稀疏文件是一种节省不必要的 I/O 操作和文件空间的机制。如果重做日志被创建为稀疏文件，每个文件的大小看起来与 `FragmentLogFileSize` 相同，但数据块并未实际分配。如果程序从稀疏区域读取，则会读取填充了零的数据，直到文件的长度。当数据写入稀疏区域时，才会分配数据块。

稀疏文件的主要缺点是，只有在数据实际写入时，文件系统的空闲空间大小才会减少。因此，您可能会在正常操作期间而非初始化时注意到文件系统空间不足。请确保您的文件系统有足够的空闲空间。

`FULL` 表示重做日志文件被真正初始化，数据块被实际分配。后续进行重做日志记录时，您将不会看到“文件系统已满”的错误。

此选项对数据节点的首次启动没有影响。在重启时更改此选项的值没有意义。

##### CompressedLCP

| 默认值 | 0 |
| 范围 | 0, 1（布尔值） |
| 部分 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

当此选项为 true 时，会导致 LCP 以压缩格式存储。这节省了一定的磁盘空间，但在 LCP 和重启时会消耗更多的 CPU 时间。在繁忙的系统上最好不要压缩 LCP。CPU 资源应保留给事务处理。

不建议为不同的数据节点设置不同的此选项值。所有数据节点上的可用资源应相同，以避免瓶颈。

##### ODirect

| 默认值 | 0 |
| 范围 | 0, 1（布尔值） |
| 部分 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

当此选项为 true 时，会导致检查点的写入操作以 `O_DIRECT` 模式进行，即直接 I/O。顾名思义，直接 I/O 是直接进行的 I/O 操作，不经过文件系统缓存。它可以节省一定的 CPU 资源。最好在使用 2.6 或更高版本内核的 Linux 系统上将此选项设置为 true。

##### DiskCheckpointSpeed

| 默认值 | 10M |
| 范围 | 1M – 4294967039（字节） |
| 部分 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |
| 生效版本 | 7.2.0 – 7.4.0 |

此选项决定检查点写入操作的速度，即每秒写入的数据量。此选项在 7.4.1 中已弃用，并在 7.5 系列中移除。在 7.4.1 或更新的系列上，请改用 `MinDiskWriteSpeed` 和 `MaxDiskWriteSpeed`。在 7.4 系列（版本大于或等于 7.4.1）上，可以设置此选项，但它没有效果。

##### DiskCheckpointSpeedInRestart

| 默认值 | 100M |
| 范围 | 1M – 4294967039 (字节) |
| 章节 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |
| 生效版本 | 7.2.0 – 7.4.0 |

此选项决定了在重启操作期间作为一部分的本地检查点（LCP）每秒写入的数据量，即写操作的速度。此选项在 7.4.1 版本中已弃用，并在 7.5 系列中被移除。在 7.4.1 或更新系列上，请改用 `MaxDiskWriteSpeedOtherNodeRestart` 和 `MaxDiskWriteSpeedOwnRestart`。在 7.4 系列（版本大于或等于 7.4.1）上，此选项仍可设置，但不再起作用。

##### MinDiskWriteSpeed

| 默认值 | 10M |
| 范围 | 1M – 4294967039 (字节) |
| 章节 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |
| 生效版本 | 7.4.1 和 7.5.0 或更新版本 |

此选项在 MySQL NDB Cluster 7.4.1 中添加。此选项决定了本地检查点（LCP）和备份 I/O 速度的下限。重做日志记录的 I/O 速度不受此选项影响。

数据节点在以下条件下调整 LCP 和备份的 I/O 速度：
*   如果重做日志记录延迟（> 2 秒），则减慢速度
*   如果 CPU 使用率低于 90%，则加快速度
*   如果 CPU 使用率高于 95%，则减慢速度
*   如果 CPU 使用率高于 97% 或 99%，则进一步减慢速度

速度的调整在每个周期（大约 1 秒）内逐步完成。LCP 的速度不会低于 `MinDiskWriteSpeed`，也不会高于 `MaxDiskWriteSpeed`。在此范围内，备份可能占用 I/O 速度的 `BackupDiskWriteSpeedPct` 百分比。

##### MaxDiskWriteSpeed

| 默认值 | 20M |
| 范围 | 1M – 4294967039 (字节) |
| 章节 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |
| 生效版本 | 7.4.1 和 7.5.0 或更新版本 |

此选项决定了本地检查点（LCP）和备份 I/O 速度的上限。更多信息请参阅 `MinDiskWriteSpeed`。

##### MaxDiskWriteSpeedOtherNodeRestart

| 默认值 | 50M |
| 范围 | 1M – 4294967039 (字节) |
| 章节 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |
| 生效版本 | 7.4.1 和 7.5.0 或更新版本 |

此选项决定了当另一个数据节点正在重启时的 I/O 速度。当其他数据节点重启时，LCP 的 I/O 必须具有更高的优先级，因为数据节点在重启过程中会执行 LCP。正在重启的数据节点上的 LCP 要等到进行中的 LCP 完成后才会启动。因此，必须加快幸存节点上正在进行的 LCP 速度，使其比通常情况更快。

##### MaxDiskWriteSpeedOwnRestart

| 默认值 | 200M |
| 范围 | 1M – 4294967039 (字节) |
| 章节 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |
| 生效版本 | 7.4.1 和 7.5.0 或更新版本 |

此选项决定了作为重启过程一部分执行的 LCP 的 I/O 速度。由于数据节点在重启期间不需要执行全局检查点（GCP），所有 I/O 带宽都可以分配给 LCP。因此，此选项的值会高于其他 `MaxDiskWriteSpeed` 选项。

##### TimeBetweenLocalCheckpoints

| 默认值 | 20 |
| 范围 | 0 – 31（以 2 为底的对数表示的写入数据字数） |
| 章节 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项决定了本地检查点（LCP）之间写入数据的最小大小。尽管名称如此，此选项并不指定 LCP 之间的时间间隔。

此选项的数据大小以字数的 2 为底的对数来指定。每个字的大小为四字节。因此，默认值 20 表示 `4 * 2²⁰` 字节 = 4MB。这意味着如果在上一次 LCP 之后，集群上进行的更新非常少且写入数据量小于 4MB，则不会启动新的 LCP。

为此选项设置较大的值可以防止在集群不繁忙时进行不必要的 LCP。特别是在使用 SSD 作为数据磁盘时，这是更可取的，因为与 HDD 相比，SSD 的写入次数限制较小。如果将此选项设置得非常大，则必须确保有足够的重做日志空间。在繁忙的系统上，无论如何新的 LCP 都会在上一个 LCP 结束后立即开始。

##### TimeBetweenGlobalCheckpoints

| 默认值 | 2000 |
| 范围 | 20 – 32000（毫秒） |
| 章节 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定了一个全局检查点（GCP）与另一个 GCP 之间的时间间隔。与 `TimeBetweenLocalCheckpoints` 不同，此选项以毫秒为单位指定时间。在 GCP 期间，重做日志会被写入并同步到磁盘，从而使其持久化。

##### TimeBetweenEpochs

| 默认值 | 100 |
| 范围 | 0 – 32000（毫秒） |
| 章节 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定了微全局检查点（Micro-GCP）之间的时间间隔。微 GCP 是指每次写入文件的重做日志单元，它在特定周期内同时提交。包含在一个微 GCP 中的一组事务被称为一个纪元（epoch）。换句话说，纪元每隔 `TimeBetweenEpochs` 毫秒生成一次。确保每个纪元的内容在所有数据节点之间同步。因此，他们别称微 GCP 为数据同步过程。由于纪元在所有数据节点之间同步，因此写入重做日志是安全的，并且可用于崩溃恢复。

顾名思义，微 GCP 的执行频率高于 GCP。微 GCP 用于在 SQL 节点上生成二进制日志。这使得数据节点能够比使用 GCP 更频繁、更快速地向 SQL 节点发送数据以生成二进制日志。这减少了复制延迟。

微 GCP 保留在重做缓冲区中，直到 GCP 完成，因此在节点故障时它不具有持久性。未被 GCP 写入磁盘的重做缓冲区内容将在崩溃后丢失。尽管微 GCP 不具有持久性，但如果 `NoOfReplicas` 为 `2` 或更大，它是容错的。即使一个数据节点崩溃，并且崩溃节点所属节点组中的任何节点仍然存活，任何已提交的事务都不会丢失。但是，如果微 GCP 没有通过 GCP 刷新到磁盘，在整个集群故障期间，它将真正丢失。

##### TimeBetweenEpochsTimeout

| 默认值 | 0 |
| 范围 | 0 – 256000（毫秒） |
| 章节 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定了一个微 GCP 与另一个微 GCP 之间的超时时间（以毫秒为单位）。默认值 0 表示超时被禁用。一旦微 GCP 开始，在进行中的微 GCP 完成之前，无法提交新的事务。因此，卡住的微 GCP 会严重拖慢系统的写入吞吐量。

由于微 GCP 可能因各种原因（如故障的网卡）而卡住，MySQL NDB Cluster 具有通过切断无法按时完成微 GCP 的数据节点来防止微 GCP 滞后导致系统变慢的功能。错过此超时的数据节点将被强制关闭。

如果您的应用程序需要良好的响应时间，请考虑将此选项设置为非零值。但是，不要将此选项设置得太小。如果太小，会导致因偶然的缓慢而发生不必要的节点关闭。对于在虚拟机上运行的集群，不要将此选项设置为非零值，因为虚拟机由于 CPU 调度器的限制，响应经常滞后。

在以前的版本（7.1 系列或更早）中，默认值是 4000。



#### 估算重做日志大小

选择一个最优的重做日志文件大小对于 MySQL NDB 集群来说是一项重要任务，因为当重做日志空间不足时，新事务将无法提交。重做日志越大，空间耗尽的可能性就越小。然而，设置一个不必要的过大重做日志并非良策，因为应避免浪费宝贵的文件系统空间。因此，您需要确定一个最优的重做日志大小。在确定最优大小时，请考虑以下因素。

*   数据大小
*   I/O 速度

崩溃恢复需要重做日志，它是针对局部检查点应用的。LCP 展示了某个时间点的完整数据快照，而重做日志则显示了按时间序列排列的增量差异。因此，在最新 LCP 之后生成的所有重做日志条目都是完美必需的。这意味着现有的重做日志条目在下一次 LCP 完成之前都无法被释放。这表明所需的重做日志文件最大尺寸等于两次 LCP 之间生成的重做日志条目总大小。要计算它，需要以下指令。

##### 计算两次 LCP 之间的理论最长时间

两次 LCP 之间的时间大约等于完成一次 LCP 所需的时间。可以使用以下公式计算：

```
data_size / average_disk_write_speed_for_lcp
```

除非磁盘速度不足，平均磁盘 I/O 速度将是 `MinDiskWriteSpeed` 和 `MaxDiskWriteSpeed` 之间的中间值。由于您需要为最坏情况做准备，计算时应采用 `MinDiskWriteSpeed`。因为 I/O 带宽在 LCP 和备份之间共享，LCP 最多可占用 `BackupDiskWriteSpeedPct` 百分比的带宽。数据大小受限于 `DataMemory`。因此，完成 LCP 所需的时间为：

```
DataMemory / (MinDiskWriteSpeed * BackupDiskWriteSpeedPct / 100)
```

##### 计算重做日志条目的理论最大大小

写入重做日志时，I/O 速度没有上限。全局检查点写入的数据量取决于写请求的量。您可以通过 `ndbinfo` 模式下的 `disk_write_speed_aggregate` 表来确定 GCP 写入的数据量，该表在 7.4.1 版本中添加。最好运行一些写入基准测试并监控此 `ndbinfo` 表，以查看重做日志记录的 I/O 速度。

一旦确定了重做日志记录的速度，就可以使用以下公式计算重做日志条目的理论最大大小：

```
time_taken_to_complete_lcp * io_speed_of_redo_logging
```

或者，可以假设每秒写入的数据量作为最坏情况下的物理磁盘 I/O 速度。然而，物理磁盘速度通常过快且不切实际。毕竟，从实际系统获取的真实值更为可取。

##### 计算重做日志大小选项

如本章前面所述，重做日志的总大小使用以下公式计算。如何确定这些选项的值呢？

```
NoOfFragmentLogFiles * NoOfFragmentLogParts * FragmentLogFileSize
```

上一节计算的重做日志条目的理论最大大小必须与此重做日志的总大小相匹配。最好留有一些安全余地。如果满足此条件，您可以为这些选项选择任意值。

##### 计算示例

例如，假设 `DataMemory` 为 18GB，`MinDiskWriteSpeed` 为 32M。在最坏情况下，完成一次 LCP 所需的时间为 10 分钟。如果重做日志记录速度为 80M/秒，则在一个 LCP 期间将生成 48GB 的重做日志条目。加上 33% 的余量，所需的重做日志大小约为 64GB。假设 `NoOfFragmentLogParts` 设置为 32，这是运行尽可能多的 LDM 线程所必需的。那么，以下选项值的组合满足要求。

```
FragmentLogFileSize = 500M
NoOfFragmentLogFiles = 4
NoOfFragmentLogParts = 32
```

#### 多线程选项

根据摩尔定律，单个 CPU 芯片内的晶体管数量仍在增加。然而，最近每个 CPU 核心的性能并没有提高，因为：

*   时钟速度没有提升
*   单靠增加晶体管数量并不能提升单核性能

因此，近年来的高端 CPU 芯片每颗都有许多核心。为了实现高性能，软件必须利用多个 CPU 核心。所以，程序必须并行运行。从并行编程的角度来看，有两种选择——多线程或多进程。MySQL NDB 集群采用了前者，即多线程。为了尽可能实现高性能，优化多线程选项是关键。

##### MaxNoOfExecutionThreads

| 默认值 | 2 |
| 范围 | 2 – 8 (7.2.0 – 7.2.4) 2 – 36 (7.2.5 – 7.3.2) 2 – 72 (7.3.3 – 最新版) |
| 配置段 | `[NDBD DEFAULT]` |
| 重启类型 | 初始系统重启 |

此选项指定了多线程版本数据节点 `ndbmtd` 用于数据处理的线程数。多线程数据节点具有以下类型的线程。

*   LDM 线程：用于实际数据访问（如键查找和扫描）的线程。
*   TC 线程：用于处理事务的线程。
*   发送线程：向其他数据节点发送信号。
*   接收线程：从其他数据节点接收信号。
*   主线程：此线程处理各种任务，如检查点、仲裁、故障切换等。
*   Rep 线程：此线程将来自微 GCP 的变更发送到特定的 SQL 节点以生成二进制日志。
*   I/O 线程：此线程处理文件 I/O 请求。
*   看门狗线程：此线程确定每个线程是否运行良好。

此选项指定了这些类型线程的总数，每种线程的数量会自动调整。当然，对于数据库管理员来说，自动值虽然方便，但可能并非最优。如果您想达到最高性能，请改用 `ThreadConfig`。

请注意，如果为此选项设置了较大的值，则必须相应增加 `NoOfFragmentLogParts`。LDM 线程的数量不得大于 `NoOfFragmentLogParts`。表 4-5 显示了针对 `MaxNoOfExecutionThreads` 的可接受的 `NoOfFragmentLogParts` 值。请注意，表 4-5 针对 `MaxNoOfExecutionThreads` 值有几种变体，因为自 7.4.2 起线程分配发生了变化。

表 4-5.
`MaxNoOfExecutionThreads` 所需的 `NoOfFragmentLogParts`

| `MaxNoOfExecutionThreads` | `NoOfFragmentLogParts` |

##### LockExecuteThreadToCPU

| 默认值 | 无 |
| 范围 | 以逗号分隔的 CPU 列表。每个 CPU 有一个从 0 到 64K 的标识符。 |
| 配置段 | `[NDBD]` 或 `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项通过一个以逗号分隔的 CPU 列表，指定要绑定执行线程的 CPU。目标线程类型是除 I/O 线程和看门狗线程之外的所有线程。

当目标 CPU 的序列标识符连续无间隔时，您可以使用短横线指定一个范围内的多个 CPU 进行绑定。例如，`2-5` 等同于 `2,3,4,5`。无法使用此选项进行微调。例如，您无法指定哪个线程绑定到哪个 CPU。如果您想指定此类配置，请改用 `ThreadConfig`。

在虚拟化环境中将线程绑定到 CPU 并非良策，因为从虚拟机看到的 CPU 是虚拟核心，并且无论如何都可能发生变化。



##### LockMaintThreadsToCPU

| 默认值 | 无 |
| 范围 | 逗号分隔的 CPU 列表。每个 CPU 的标识符范围为 0 – 64K |
| 所属段 | `[NDBD]` 或 `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定将 I/O 线程和看门狗线程绑定到哪些 CPU，采用逗号分隔列表格式。使用此选项无法进行微调。若要指定哪个线程绑定到哪个 CPU，请改用 `ThreadConfig`。

##### ThreadConfig

| 默认值 | 无 |
| 范围 | 见下文 |
| 所属段 | `[NDBD]` 或 `[NDBD DEFAULT]` |
| 重启类型 | 初始系统重启 |
| 影响版本 | 7.3.0 – 最新 |

此选项允许对线程配置进行微调。您可以为每种线程类型同时指定线程数量及其 CPU 亲和性。此选项的值由逗号分隔的属性列表组成，格式为 `thread_type={property=val[,property=val...]}`。已定义的线程类型有 `ldm`、`tc`、`recv`、`send`、`io`、`watchdog`、`main` 和 `rep`。这些线程类型与 `MaxNoOfExecutionThreads` 中描述的相同。表 4-6 中的属性可以为每种线程类型设置。

**表 4-6. ThreadConfig 中的线程属性**

| 属性名称 | 范围 | 描述 |
| --- | --- | --- |
| `count` | 数字 | 每种类型线程的数量。更多信息见表 4-7。 |
| `cpubind, cpuset, cpubind_exclusive, cpuset_exclusive` | CPU 列表 | 逗号分隔的 CPU 列表，与 `LockExecuteThreadToCPU` 和 `LockMaintThreadsToCPU` 相同。`cpubind` 使每个线程绑定到一个 CPU。`cpuset` 使每个线程绑定到一组 CPU。排他性变体（仅在 Solaris 上可用）不允许其他线程或进程绑定到相同的 CPU。 |
| `spintime` | 微秒时间 | 当没有信号需要处理时，线程进入睡眠前的自旋时间。一旦线程进入睡眠，它将无法快速响应。自旋可以提高响应速度，但会消耗 CPU 时间。 |
| `realtime` | 0 或 1 | 设置为 1 时，将使用实时调度器。这可能会提高性能。 |
| `thread_prio` | 0 – 10 | 10 为最高优先级。当多个线程共享相同的 CPU 时，此属性才有意义。 |

默认情况下，线程数量为最小值，没有线程绑定到任何 CPU，没有线程进行自旋，并且所有线程具有相同的优先级。`realtime` 属性的默认值是设置给 `RealTimeScheduler` 选项的值。

```

**清单 4-4. 32 核系统上的 `ThreadConfig` 配置示例**

在清单 4-4 的示例中，CPU 0 没有绑定到任何线程。这是为了给操作系统和其他系统进程执行的任务留出空间。请注意，清单 2-2 中的 `ThreadConfig` 看起来是多行的，但实际上写在一行中，不包含换行符。这样非常长的行无法在书籍的单页中呈现。

确定最优设置非常困难。您可能需要反复进行基准测试，并通过试错法来确定最佳设置。根据经验法则，以下配置是一个良好的起点：

1.  将一半可用的 CPU 核心分配给 `ldm` 线程。
2.  将 `ldm` 线程数量的四分之一分配给 `tc` 线程。
3.  将 `ldm` 线程数量的四分之一分配给 `recv` 和 `send` 线程。如果可用线程数为奇数，则分配更多给 `recv` 线程。
4.  将其他 CPU 核心分别分配给其他类型的线程。
5.  至少留出一个 CPU 核心供系统使用。
6.  将每个线程绑定到特定的 CPU，这可以通过确定 CPU 使用率来监控每个线程的繁忙程度。

##### CPU 属性考虑因素

由于 CPU 必须响应各种请求，因此需要进行一些调整才能发挥其潜在性能。对于 MySQL NDB 集群，性能是最重要的需求。

###### 节能

最好关闭节能功能，例如 Linux 系统上的 `cpuspeed`。启用节能功能后，操作系统会根据工作负载降低 CPU 频率。对于具有后台工作负载的系统（例如台式机）来说，这是合理的。然而，这对于繁忙的服务器机器有负面影响。当 CPU 频率较低时，无法快速处理突发工作负载，因为 CPU 频率恢复到原始状态存在延迟。这会损害数据访问的响应能力。

在 Windows 上，请从电源选项设置中选择“高性能”电源计划。

###### 中断

来自设备的中断在大多数情况下不可忽略。例如，当数据流量很大时，网卡会产生大量中断。默认情况下，在大多数操作系统上，中断由特定的 CPU 处理。这使得某些 CPU 比其他 CPU 更繁忙。从节能角度来看，这是一个不错的选择。但是，它可能会导致性能瓶颈。在最坏的情况下，单个 CPU 核心会因中断而达到 100% 的繁忙状态。如果系统有许多设备，那么该 CPU 核心将无法执行任何其他任务。

防止此问题的一个选项是将中断引起的工作负载均匀地分散到所有 CPU。将中断重新分配给其他 CPU 的能力称为中断亲和性。要实现这一点，Linux 系统的选择是使用 `irqbalance`。如果中断仅集中在特定的 CPU 上，它会将中断请求重新分配给其他 CPU。这可以防止特定的 CPU 成为瓶颈。

另一个选项是将中断请求重新分配给特定但多个 CPU。当使用 `ThreadConfig`、`LockExecuteThreadToCPU` 或 `LockMaintThreadsToCPU` 选项将线程绑定到特定的 CPU 时，此策略尤其有效。将中断分配给数据节点中未被线程绑定的 CPU，这样中断就不会消耗数据节点线程的 CPU 资源。在这种情况下，Linux 的选择是 `irqbalance`。它可以在特定的 CPU 内重新平衡中断工作负载。

在 Windows 上，需要编辑注册表来配置中断亲和性。在 Solaris 上，`pcitool` 是用于配置的工具。更多信息请参阅给定操作系统的手册。

##### 超线程

总体而言，超线程技术通过牺牲响应能力来提升整体吞吐量。当启用超线程时，多个逻辑核心会共享同一个物理核心。如果超线程不可用，在以下示例场景中可能会浪费部分 CPU 周期：

1.  一个线程发出加载（从内存读取数据）指令。
2.  数据是后续指令执行所必需的。
3.  在内存响应并完成加载指令之前，CPU 无法执行更多指令。

超线程技术旨在改善此类场景下的性能。借助超线程，CPU 可能能够执行运行在另一个逻辑核心上的另一个线程的指令，从而充分利用 CPU 核心。

超线程技术存在几个主要缺点：

*   单线程性能可能下降：随着更多逻辑核心共享同一个物理核心，当一个线程正在执行指令时，另一个线程无法执行新指令。
*   CPU 缓存内存的效率可能降低：物理缓存内存的一个实例必须存储来自多个线程的数据。这与使用上下文切换在单个 CPU 核心上运行多个线程时的问题相同。然而，数据节点不依赖于上下文切换。理想情况下，数据节点上的每个线程应独占一个绑定的 CPU 核心。
*   CPU 使用率指标会产生误导：虽然操作系统看不到物理核心的使用情况，但 CPU 资源的容量很可能低于预期。

那么，你到底应该使用超线程吗？答案是“视情况而定”。这取决于各种因素，例如应用程序负载类型和硬件规格。因此，最好在更改超线程设置的同时运行基准测试。你可以从 BIOS 设置中开启或关闭它。

如果你觉得关闭它很麻烦，那就不要为每个物理核心分配超过一个逻辑核心。在这种情况下，可以配置线程分配，使某些类型的线程使用超线程，而其他线程则不使用。例如，`ldm` 线程在不使用超线程时往往表现更好，但 `recv` 和 `send` 线程在使用超线程时性能更佳。

#### 备份选项

毋庸置疑，备份是数据库管理系统最重要的任务之一。任何存储介质都可能发生故障。即使没有硬件故障，操作失误或应用程序中的错误也可能导致重要数据丢失。为了保护重要数据，必须频繁进行备份。MySQL NDB Cluster 具有原生备份功能，这是进行在线备份的唯一方式。然而，由于需要处理大量数据，在线备份是资源消耗型过程。默认设置通常没问题，但有时为了获得更好的性能，你需要进行微调。有关备份的更多信息，请参阅第 8 章。

##### BackupDataDir

| 默认值 | `FileSystemPath` |
| 取值范围 | 任意目录路径名 |
| 配置段 | `[NDBD]` 或 `[NDBD DEFAULT]` |
| 重启类型 | 初始节点重启 |

此选项指定存储备份数据的目录。备份文件通常很大，因此常与主数据分开存储在非昂贵磁盘上。从 I/O 带宽角度考虑，使用独立磁盘也是更优选择。

##### BackupDiskWriteSpeedPct

| 默认值 | 50 |
| 取值范围 | 0 – 90 （百分比） |
| 配置段 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |
| 生效版本 | 7.4.8 – 最新 |

此选项指定 LCP（本地检查点）和备份的 I/O 带宽速率。它们共享由 `MinDiskWriteSpeed` 和 `MaxDiskWriteSpeed` 指定的相同 I/O 带宽。这意味着在备份期间，LCP 的速度将受到限制。在繁忙的系统上，应为此选项指定较小的值。

有关这些选项的更多信息，请参阅本章前面描述相关选项的部分。

##### CompressedBackup

| 默认值 | 0 |
| 取值范围 | 0, 1 （布尔值） |

##### BackupWriteSize

| 默认值 | 256K |

##### BackupMaxWriteSize

| 默认值 | 1M |
| 取值范围 | 2K – 4294967039 (字节) (< 7.4.8) ; 256K – 4294967039 (字节) (>= 7.4.8) |
| 配置段 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定备份数据和日志的写入单元的最大尺寸。

##### BackupDataBufferSize

#### 传输器选项

基本上，传输器的属性是针对每个单独的传输器配置的。但是，有少数选项是按数据节点设置的。请注意，数据节点通常有多个传输器。

##### TotalSendBufferMemory

| 默认值 | 0 |
| 取值范围 | 0, 256K – 4294967039 (字节) |
| 配置段 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定所有传输器的发送缓冲区总内存大小。无论节点类型如何，每个节点都需要一个传输器。默认情况下，发送缓冲区是为每个传输器分配的。因此，发送缓冲区所需的总内存大小与 `(节点数量 – 1)` 成正比增加。当配置了大量节点时，这会导致内存消耗巨大。

此选项可以限制内存消耗，并使发送缓冲区内存按需分配。当连接了大量非繁忙的 API 节点或 SQL 节点时，请考虑设置此选项。


##### `ExtraSendBufferMemory`

| 默认值 | 0 |
| 范围 | 0, 32G (字节) |
| 段落 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定了当传输器发送缓冲区即将耗尽时额外分配的内存大小。拥有额外的发送缓冲区内存可以提高数据节点的稳定性。目前，`TotalSendBufferMemory`和`ExtraSendBufferMemory`在内存使用方式上没有区别。这些选项的总和将被分配给一个缓冲区。

##### `InitialLogFileGroup`

| 默认值 | 无 |

InitialLogFileGroup = name=LG1; undo_buffer_size=1G; undo1.log:5G; undo2.log:5G
```

`name`和`undo_log_buffer`属性是可选的。它们的默认值分别是`DEFAULT-LG`和`64M`。如果配置了多个撤销日志文件，可以使用分号列出它们。只要分配了足够的容量，撤销日志文件的数量并不重要。因此，您无需列出小文件。

此选项是为磁盘数据表设置所需日志文件组的一种便捷方法。此选项仅在集群首次启动时生效。您可以使用诸如`CREATE LOGFILE GROUP`之类的 DDL 语句来设置日志文件组。更多信息请参见第 18 章。

##### `InitialTableSpace`

| 默认值 | 无 |
| 范围 | 指定日志文件组的字符串 |
| 段落 | `[NDBD DEFAULT]` |
| 重启类型 | 初始系统重启 |

此选项指定在系统启动时自动创建的磁盘数据表表空间的属性。该选项由两个或多个由分号分隔的部分组成。在以下示例中，名为`TS1`的表空间配置了四个 16GB 的数据文件。这些数据文件在`FileSystemPathDataFiles`下创建。

```
InitialTableSpace = name=TS1; data1.dat:16G; data2.dat:16G; data3.dat:16G; data4.dat:16G
```

虽然表空间必须与日志文件组关联，但通过此选项创建的表空间需要`InitialLogFileGroup`选项创建的日志文件组。

此选项是为磁盘数据表设置所需表空间的一种便捷方法。此选项必须与`InitialLogFileGroup`一起使用。此选项仅在集群首次启动时生效。您可以使用诸如`CREATE TABLESPACE`之类的 DDL 语句来设置日志文件组。更多信息请参见第 18 章。

##### `DiskPageBufferMemory`

| 默认值 | 64M |
| 范围 | 4M – 1T |
| 段落 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定为磁盘页缓冲区分配的内存大小。此缓冲区用于缓存磁盘数据表上的行数据。如果您大量使用磁盘数据表，则需要为此缓冲区分配大量内存。另一方面，如果您不使用磁盘数据表，则可以将此选项减小到最小值以节省内存。

##### `DiskIOThreadPool`

| 默认值 | 2 |
| 范围 | 0 – 4294967039 |
| 段落 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定了用于针对表空间和撤销日志文件进行磁盘 I/O 的线程数量。您应该使用多个线程的主要原因是磁盘 I/O 是非常耗时的任务，文件操作系统调用不会立即响应。当一个线程等待系统调用的响应时，它无法处理其他任务。因此，当一个线程正在处理其他任务时，它无法发出文件操作系统调用。这会浪费 CPU 时间。

使用适当数量的线程访问文件可以缓解这个问题。如果用户进程中发出多个 I/O 请求，请求会在内核和/或磁盘控制器中排队。I/O 请求按照内核和/或磁盘控制器中的 I/O 调度程序决定的顺序进行处理。这可以最大化磁盘 I/O 性能。

在以下场景中可能需要增加此选项：
*   将 I/O 负载分散到独立的磁盘上：表空间和撤销日志文件放置在多个磁盘上。在这种情况下，可以利用更多线程来提高 I/O 并行性。
*   使用高性能磁盘：使用高性能磁盘作为底层存储。需要更高的 I/O 并行性来填满 I/O 带宽。

您可能知道，这类问题可以通过异步 I/O 解决，例如 Linux 上的 epoll。但是没有与异步 I/O 相关的选项，因为 MySQL NDB Cluster 尚未实现它。它使用大量线程进行磁盘 I/O。

#### 心跳和看门狗选项

如第 1 章所述，MySQL NDB Cluster 采用故障早期策略，该策略会强制停止无响应的数据节点，以避免整个集群变慢。为了检测无响应的数据节点，MySQL NDB Cluster 利用心跳和看门狗。

##### `HeartbeatIntervalDbDb`

| 默认值 | 5000 |
| 范围 | 10 – 4294967039 (毫秒) |
| 段落 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定了心跳之间的时间间隔。数据节点向另一个数据节点发送心跳信号（`CM_HEARTBEAT`），并每`HeartbeatIntevalDbDb`毫秒检查一次该心跳信号是否已到达。数据节点有两个独立的计时器分别用于心跳信号的发送和检查，但它们的间隔是相同的。如果一个数据节点检测到来自另一个数据节点的心跳信号连续四次缺失，则另一个数据节点将被标记为死亡，这将导致故障转移或系统关闭，具体取决于是否有足够的幸存节点继续集群操作。

如果集群运行在不稳定的网络上，请稍微增加此选项。如果您的应用程序需要最佳响应时间并且使用专用高速网络，请考虑将此选项减少到 1500，这是旧版本（< 7.2.0）的默认值。



#### HeartbeatIntervalDbApi

| 默认值 | 1500 |
| 范围 | 100 – 4294967039（毫秒） |
| 章节 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定从 API（SQL）节点发送的心跳检查之间的时间间隔。所有 API 节点每 100 毫秒发送一次心跳（`API_REGREQ`）信号。数据节点每隔 `HeartbeatIntervalDbApi` 毫秒检查一次是否已收到心跳信号。如果一个 API 节点连续错过四次心跳信号，该 API 节点将被标记为死亡。随后，该 API 节点将被断开连接，所有进行中的事务都将被中止。

当您的集群运行在不稳定的网络上时，请略微增大此选项的值。

#### HeartbeatOrder

| 默认值 | 0 |
| 范围 | 100 – 4294967039 |
| 章节 | `[NDBD]` |
| 重启类型 | 系统重启 |

此选项定义了心跳信号的发送顺序。心跳信号以循环方式发送。这意味着每个数据节点仅从一个节点接收心跳信号，同时，一个数据节点也仅向一个节点发送心跳信号。当数据节点数量超过两个时，心跳信号的源节点（左节点）和目标节点（右节点）是不同的。否则，它们是同一个节点。

心跳信号的顺序在每个 `[NDBD]` 部分（而不仅仅是 `[NDBD DEFAULT]` 部分）中被指定为一个整数值。心跳信号从一个节点发送到下一个具有该选项更大数值的数据节点。具有最高 `HeartbeatOrder` 值的数据节点向具有最低值的数据节点发送心跳信号。

在大多数情况下，您无需更改此选项。当网络拓扑结构不对称且每个传输器的稳定性存在差异时，不稳定的网络更有可能丢失心跳。在这种情况下，请配置心跳顺序，确保每个节点组内至少有一个数据节点拥有稳定的心跳源，以避免因节点组故障导致整个系统关闭。

#### TimeBetweenWatchDogCheck

| 默认值 | 6000 |
| 范围 | 70 – 4294967039（毫秒） |
| 章节 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定每次看门狗检查之间的时间间隔。看门狗是一种检查数据节点是否冻结的机制。如果看门狗线程观察到一个作业线程在连续三个检查周期内保持同一状态，它将立即关闭该节点。

#### ArbitrationTimeout

| 默认值 | 7500 |
| 范围 | 10 – 4294967039（毫秒） |
| 章节 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定仲裁的超时时间（毫秒）。如第 1 章和第 3 章所述，当集群发生网络分区时会执行仲裁。如果数据节点在超过 `ArbitrationTimeout` 毫秒的时间内无法从仲裁器获得回复，它将立即终止。

#### StartPartialTimeout

| 默认值 | 30000 |
| 范围 | 0 – 4294967039（毫秒） |
| 章节 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定等待所有其他节点准备就绪的秒数。0 表示无限期等待。一旦 `StartPartialTimeout` 毫秒过去，且仍有一个或多个数据节点缺失，集群将强制继续启动过程，而不会等待缺失的节点。

#### StartPartitionedTimeout

| 默认值 | 60000 |
| 范围 | 0 – 4294967039（毫秒） |
| 章节 | `[NDBD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定等待网络分区解决的秒数。0 表示无限期等待。一旦 `StartPartitionedTimeout` 毫秒过去，且仍有任何数据节点缺失，集群进入网络分区状态，即使潜在的网络分区问题尚未解决，集群也将强制继续启动过程。

我建议将此选项设置为 0 或一个非常大的值。以潜在的网络分区状态启动是非常危险的。这很可能导致网络分区，因为在此超时发生时不会执行仲裁。因此，当超时发生时，两个独立的集群可能同时启动。必须避免这种情况，以保护数据免遭损坏。

#### 日志记录选项

管理节点记录集群日志，但其内容（例如其事件）的来源是在数据节点上生成的。有几个与数据节点上事件报告相关的选项。有关日志级别的更多信息，请参阅第 16 章。

#### MemReportFrequency

| 默认值 | 0 |
| 范围 | 0 – 4294967039（秒） |
| 章节 | `[NDBD DEFAULT]` |
| 重启类型 | 系统重启 |

当此选项设置为非零值时，集群日志会定期记录内存使用情况。如果未激活定期使用情况报告，集群日志将在内存使用率达到 80%、90% 和 100% 时进行报告。

#### StartupStatusReportFrequency

| 默认值 | 0 |
| 范围 | 0 – 4294967039（秒） |
| 章节 | `[NDBD DEFAULT]` |
| 重启类型 | 系统重启 |

当此选项设置为非零值时，集群日志会记录初始启动阶段 4 的进度。如果重做日志的大小很大，此阶段可能需要很长时间，因为必须在此阶段初始化重做日志。

#### 推荐的配置策略

配置集群时最重要的事情是，您应该根据需求选择合适的服务器机器。不要根据服务器机器的规格来确定您的需求。应用需求大致可分为以下几类——容量、性能、响应速度和可用性。

例如，在容量方面，不可能将 `DataMemory` 设置得大于服务器机器的物理内存大小。您不应在确定所需内存量之前就选择服务器机器。

数据节点的配置应遵循以下步骤：

1.  估算所需的计算机资源，例如内存大小、重做日志的磁盘空间、I/O 速度、网络速度和 CPU 速度。
2.  在一定的余量范围内选择合适的计算机硬件。
3.  微调数据节点配置，使数据节点能够充分利用计算机资源。
4.  为将来容量扩展时添加数据节点做好准备。

在估算内存大小时，线程数量非常重要，因为它是增加所需内存大小的最重要的间接因素。多线程是现代计算机硬件的必备条件。精心定义的 `ThreadConfig` 与随手设置的 `MaxNoOfExecutionThreads` 相比，可能会使性能翻倍。

仔细估算 LCP 速度和重做日志文件大小。它们是定义事务总体写入速度上限的主要因素。

配置和系统规划并非孤立的过程。我建议通读第 3 章和第 4 章。

### SQL 节点选项

本节讨论 SQL 节点和 API 节点的主要选项。章节名称可以是 `MYSQLD` 或 `API`。无论指定哪个章节名称，这些章节都被标识为相同的节点类型。在以下描述中，为方便起见，使用 `MYSQLD`。

#### SQL 节点的主要选项

##### NodeId

| 默认值 | 自动 |
| 范围 | 1 – 255 |
| 章节 | `[MYSQLD]` |
| 重启类型 | 节点重启 |

节点的标识符，其含义与其他节点类型的 `NodeId` 相同。

##### HostName

| 默认值 | 无（接受来自任何主机的连接） |
| 范围 | 任意主机名或 IP 地址 |
| 章节 | `[MYSQLD]` |
| 重启类型 | 系统重启 |

节点所在的主机名或 IP 地址。您可以不设置此选项，这意味着该插槽允许从任何主机连接。如果您不希望无特权节点连接，请显式设置此选项。



##### ArbitrationRank

| 默认值 | 0 |
| 范围 | 0 – 2 |
| 配置节 | `[MYSQLD]` 或 `[MYSQLD DEFAULT]` |
| 重启类型 | 节点重启 |

`ArbitrationRank` 选项指定了节点成为仲裁节点的可能性。该参数的可选值为 0、1 和 2。SQL 节点的默认值为 0，这意味着默认情况下 SQL 节点不会成为仲裁节点。要使 SQL 节点成为额外的仲裁节点，请设置一个非零值。1 的优先级高于 2。请注意，仲裁节点不应与数据节点位于同一台主机上。更多信息请参考本章前面对 `ArbitrationRank` 的解释，以及第 1 章和第 3 章的内容。

##### DefaultOperationRedoProblemAction

| 默认值 | `QUEUE` |
| 范围 | `ABORT` 或 `QUEUE` |
| 配置节 | `[MYSQLD]` 或 `[MYSQLD DEFAULT]` |
| 重启类型 | 系统重启 |
| 生效版本 | 7.2.10 或更新 |

此选项定义了由于超出重做操作超限阈值而中止的事务如何处理。每个值的含义如下：

*   `ABORT`：来自已中止事务的任何挂起操作也将被中止。这适用于你希望在应用中自行处理重试的情况。
*   `QUEUE`：来自已中止事务的挂起操作将被排队以进行重试。

请参阅本章前面对 `RedoOvercommitLimit` 和 `RedoOvercommitCounter` 的解释。

##### BatchSize

| 默认值 | 64 (7.2.0) 256 (>= 7.2.1) |
| 范围 | 1 – 992 (记录数) |
| 配置节 | `[MYSQLD]` 或 `[MYSQLD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定了扫描操作的批处理大小（以记录数为单位）。由于扫描操作会返回许多行，为了性能考虑，将记录打包成适当大小的批次非常重要。批处理大小受此选项和下面描述的 `BatchByteSize` 选项的限制。

##### BatchByteSize

| 默认值 | 32K (7.2.0) 16K (>= 7.2.1) |
| 范围 | 1K – 1M (字节) |
| 配置节 | `[MYSQLD]` 或 `[MYSQLD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定了扫描操作的批处理大小（以字节为单位）。

##### MaxScanBatchSize

| 默认值 | 256K |
| 范围 | 32K – 16M (字节) |
| 配置节 | `[MYSQLD]` 或 `[MYSQLD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定了一个 SQL 节点一次可以从数据节点接收的数据总量。SQL 节点有可能一次性从数据节点接收过多记录，从而消耗过多内存。此选项旨在保护 SQL 节点免受此类过度内存消耗的影响。

##### TotalSendBufferMemory

| 默认值 | 0 |
| 范围 | 0, 256K – 4294967039 (字节) |
| 配置节 | `[MYSQLD]` 或 `[MYSQLD DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定了所有传输器的发送缓冲区总内存大小。无论节点类型如何，每个节点都需要一个传输器。默认情况下，每个传输器都会分配一个发送缓冲区。因此，所需的发送缓冲区内存总大小与 `(节点数量 – 1)` 成正比。请注意，SQL 节点之间不相互连接。但是，即使 SQL 节点只连接到数据节点和管理节点，当配置了大量数据节点时，也会导致较大的内存消耗。请参阅本章前面对 `TotalSendBufferMemory` 的解释。

SQL 节点也定义了 `ExtraSendBufferMemory` 选项。这些选项配置的内存被分配给同一个缓冲区，且缓冲区使用情况不会按选项区分。

#### SQL 节点配置要点

在大多数情况下，SQL 节点选项的默认值就足够了。通常不需要经常更改它们。如果 SQL 节点的服务器机器拥有高速 CPU 和大内存，可以考虑增加批处理大小以最大化性能。虽然 `config.ini` 中可供 SQL 节点配置的选项不多，但 `mysqld` 上存在大量选项。

请注意，当 `--ndb-cluster-connection-pool` 选项设置为大于 1 时，SQL 节点将拥有多个 NDB API 连接。在这种情况下，缓冲区总大小也会与连接数量成比例增加。

### TCP 传输器选项

在 MySQL NDB 集群中，可以对单个传输器（两个节点之间的网络路径）进行微调。可以配置以下模式的传输器：

*   两个数据节点之间
*   数据节点和 SQL（API）节点之间

请注意，SQL（API）节点之间不相互连接。管理节点的传输器无法配置。

在大多数情况下，当传输器使用专用网络路径（例如使用交叉以太网电缆的直接连接）连接时，才会对单个传输器进行配置。

随着节点数量的增加，传输器总数将急剧增加。每个任意两个数据节点的组合，或每个数据节点与 SQL 节点的组合，都存在一个传输器。两个数据节点之间的传输器数量可以用组合数 [`n`] `C` [`2`] 来计算，其中 `n` 代表数据节点的数量。数据节点的最大数量是 48，因此两个数据节点之间的最大传输器数量是 [`48`] `C` [`2`] `= 1128`。数据节点与 SQL 节点之间的最大传输器数量是 `48 * (255 – 48 – 1) = 9888`。（在此情况下，有 48 个数据节点，1 个管理节点和 206 个 SQL 节点。）因此，最大传输器总数为 `1128 + 9888 = 11016`，数量庞大。由此可见，在大型集群环境中定义单个传输器是一项艰巨的任务。

#### 传输器主要选项

##### NodeId1, NodeId2

| 默认值 | 无 |
| 范围 | 1 – 255 |
| 配置节 | `[TCP]` |
| 重启类型 | 节点重启 |

这两个选项使用节点 ID 来标识传输器。任意两个节点的组合恰好对应一个传输器。换句话说，你需要为每两个节点的组合列出 `[TCP]` 配置节。

##### HostName1, HostName2

| 默认值 | 无 |
| 范围 | `NDBD`、`API` 或 `MYSQLD` 配置节中出现的主机名或 IP 地址 |
| 配置节 | `[TCP]` |
| 重启类型 | 节点重启 |

这些选项指定了要使用的主机名或 IP 地址。如果一台服务器机器有多个网络接口，那么它们应该对应多个 IP 地址和主机名。因此，例如，可以按节点类型分离网络路径。这种配置将在本节后面讨论。

##### SendBufferMemory

| 默认值 | 2M |
| 范围 | 256K – 4294967039 (字节) |
| 配置节 | `[TCP]` 或 `[TCP DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定了每个传输器用于缓冲待发送数据的内存大小。信号数据在发送到其他节点之前存储在此缓冲区中，因此该缓冲区称为发送缓冲区。当发送缓冲区变满时，数据节点无法将信号写入缓冲区，导致信号丢失。这种情况对数据节点而言是严重的，节点将会关闭。

在大多数情况下，默认值是足够的。

##### ReceiveBufferMemory

| 默认值 | 2M |
| 范围 | 16K – 4294967039 (字节) |
| 配置节 | `[TCP]` 或 `[TCP DEFAULT]` |
| 重启类型 | 节点重启 |

此选项指定了每个传输器用于接收数据的内存大小。从其他数据节点接收到的信号数据在调度到 NDB 内核块之前存储在此缓冲区中。此缓冲区称为接收缓冲区。

在大多数情况下，默认值是足够的。



#### 通过独立网络路径的传输通道

###### 过载限制

| 默认值 | 0 |
| --- | --- |
| 范围 | 0 – 4294967039 (字节) |
| 所属章节 | `[TCP]` 或 `[TCP DEFAULT]` |
| 重启类型 | 节点重启 |

当发送缓冲区中未发送的数据超过此选项设定的值时，`DBLQH` 内核块将中止更多请求并返回错误 `1218`。在发送缓冲区几乎满时拒绝更多请求是一种合理的实现。将此选项设置为发送缓冲区大小的大约 80% 将降低因发送缓冲区不足而导致关闭的概率。默认值为 `0`，这意味着 `DBLQH` 不检查过载状态。建议在 `[TCP DEFAULT]` 部分适当设置此选项。

#### 通过独立网络路径的传输通道

在没有显式配置时，节点使用在 `HostName` 选项上指定的网络接口相互通信。但是，可以在 `[TCP]` 部分指定与 `HostName` 不同的其他网络接口。然后，您可以实现以下场景：

*   独立的网络允许数据节点在网络故障时访问仲裁器。这可以避免整个系统关闭。
*   在低成本交换机之间分配工作负载，以获得更好的吞吐量和更低的成本。
*   将属于同一节点组的数据节点之间的通信重定向到物理服务器的直接连接，以获得更快的响应。
*   将所有数据节点之间的通信重定向到专用的快速网络交换机。

尽管定义大量的 `[TCP]` 部分是一项繁琐的任务，但这是值得的，因为网络很可能是故障和性能瓶颈的来源。

##### 连接字符串

除了像 `ndb_print_file` 这样的离线工具外，所有 NDB 程序在启动时都会连接到管理服务器。因此，连接字符串是一个关键且常见的选项。

当节点尝试连接时，管理服务器会将客户端的主机名与每个插槽上的 `HostName` 选项进行比较。如果有匹配的插槽可用，则接受连接。要连接到集群，您需要指定一个与至少一个可用插槽匹配的连接字符串。不需要身份验证或其他任何东西。

> 注意：没有 `HostName` 选项的插槽将接受来自任何主机的连接。

连接字符串是表示管理节点位置的字符串。指定连接字符串的选项是 `--ndb-connectstring`（或 `--connect-string`）。虽然可以使用 `--ndb-mgmd-host` 指定管理服务器的主机名，但我不建议您使用它。因为它有限制，例如不能指定端口号。我推荐使用 `--ndb-connectstring`。连接字符串的格式如下：

```
[nodeid=#,]{hostname_or_ipaddr[:port][hostname_or_ipaddr[:port], ...]}
```

请注意，方括号中的参数是可选的，可以省略。例如，如果管理服务器运行在名为 `mgm1` 的主机上，使用默认端口且不需要特定的节点 ID，则连接字符串为：

```
mgm1
```

如果管理服务器运行在名为 `mgm1` 和 `mgm2` 的主机上，使用网络端口 `1188`，并且需要的节点 ID 是 `20`，则连接字符串为：

```
nodeid=20,mgm1:1188,mgm2:1188
```

##### 配置缓存

管理节点拥有一种机制，可以将配置数据以二进制格式缓存，而非读取并解析 `config.ini` 文件。此功能称为配置缓存。它能增强集群配置在变更时的稳定性。这是一个便捷的功能，但需稍加留意。

配置缓存默认启用。在启动时，`ndb_mgmd` 会读取最新的配置缓存条目，而无需读取 `config.ini`，即使 `config.ini` 已被更新。要使 `config.ini` 被读取，必须指定 `--reload` 选项。此选项会使 `ndb_mgmd` 读取 `config.ini` 文件，并将其内容与最新的配置缓存条目进行比较。若有任何变更，则会从 `config.ini` 的内容创建一个新的缓存条目。

配置缓存条目具有以下文件名格式：
```
ndb_node-id_config.bin.seq-number
```
例如，节点 ID 为 255 的管理节点上的第三个配置缓存条目是 `ndb_255_config.bin.3`。由于文件带有序列号，很容易识别最新的缓存条目。如果您在更改配置后遇到问题，可以通过删除最新的缓存条目文件并重启管理节点来恢复到之前的配置。

`--initial` 选项将删除所有缓存条目。当您将集群安装迁移到另一个环境，并希望从头刷新配置时，此选项非常方便。

请注意，即使指定了 `--skip-configuration-cache`，如果存在配置缓存条目，它仍然会被读取。要完全禁用配置缓存，您必须删除所有缓存条目并同时指定 `--skip-configuration-cache` 选项。

##### 在多个管理节点下读取新配置

当存在多个管理节点时，在配置变更后重启需要遵循特殊流程，因为如果存在一个正在运行的管理节点，管理服务器会从该节点读取配置。简单的滚动重启（逐个重启管理节点）会失败。`ndb_mgmd` 的重启必须使用以下步骤完成：

1.  修改 `config.ini`。
2.  停止所有管理节点，一次一个。
3.  使用 `--initial` 或 `--reload` 选项启动第一个 `ndb_mgmd`。
4.  如果您使用 `--initial` 选项启动了第一个 `ndb_mgmd`，则使用 `--initial` 选项启动剩余的 `ndb_mgmd` 进程。
5.  如果您使用 `--reload` 选项启动了第一个 `ndb_mgmd`，则在不指定 `--initial` 或 `--reload` 选项的情况下启动剩余的 `ndb_mgmd` 进程。它们将从第一个 `ndb_mgmd` 读取配置。

#### 初始启动

如第 2 章所述，数据节点有一种称为初始启动的启动类型。在初始启动时，数据节点会在启动时擦除所有数据。初始启动分为以下类型：初始节点重启和初始系统重启。这两种初始重启类型的区别在于，前者是逐个重启数据节点，后者是在整个系统关闭后重启集群。请参阅本章前面关于重启类型的解释。

##### 部分启动

有时，由于硬件故障等原因，需要在缺少部分数据节点的情况下启动集群。即使缺少一些数据节点，只要每个节点组至少有一个可用的数据节点，集群仍然是可以启动的。这种启动称为部分启动。

在执行部分启动时，必须指定 `--nowait-nodes` 选项。其参数是一个以逗号分隔的缺失节点的节点 ID 列表。例如，如果节点 ID 为 3、5 和 7 的数据节点缺失，并且集群可以使用剩余的数据节点运行，则使用 `--nowait-nodes=3,5,7` 选项启动所有剩余的数据节点。如果是初始系统重启，则还需要 `--initial-start` 选项。

##### 非自动启动

个人建议使用此功能。当指定了 `--nostart` 选项时，数据节点在连接到管理节点后会暂停其执行。要恢复启动过程，必须从 `ndb_mgm` 客户端发出 `START` 命令。

默认情况下，数据节点在连接到管理节点后会自动启动。在系统启动时，所有数据节点必须一次性启动。如果缺少某些数据节点且尚未准备好启动，所有正在启动的数据节点都会等待缺失的节点。当所有数据节点（由 `--nowait-nodes` 选项指定的数据节点除外）都准备就绪，或者 `StartPartialTimeout` 毫秒数已过时，正在启动的数据节点将继续启动过程。如果缺失的数据节点能够组成一个可运行的集群，换句话说，如果每个节点组至少缺失一个数据节点，则集群可能会陷入网络分区。在这种情况下，正在启动的数据节点将等待 `StartPartitionedTimeout` 毫秒数。

在自动启动后照顾所有已启动的数据节点是费时费力的。我建议使用 `--nostart` 选项禁用自动启动，然后检查所有数据节点是否已连接，并使用以下命令从 `ndb_mgm` 客户端逐个启动它们。
```
ndb_mgm> ALL START
```

#### 选项文件的位置

按照惯例，MySQL 程序如果存在多个选项文件，可能会读取它们。可以在不同的选项文件中多次指定同一个选项。如果同一个选项被指定多次，则接受最后读取的值。因此，程序读取选项文件的顺序很重要。

MySQL 程序读取选项文件的位置和顺序取决于编译时的配置。无论程序是客户端还是服务器，您都可以使用系统的程序二进制文件和 `--help` 选项来确认选项文件的位置和顺序。以下命令输出是 UNIX 类系统上 `mysql` CLI 的示例。

```
shell$ mysql --help | grep /my.cnf
/etc/my.cnf /etc/mysql/my.cnf /usr/local/mysql/etc/my.cnf ~/.my.cnf
```

在 Windows PowerShell 上，您可以做同样的事情。

```
PS C:\Program Files\MySQL\MySQL Cluster 7.5\bin> ./mysql --help –verbose | Select-String "my.ini"
C:\WINDOWS\my.ini C:\WINDOWS\my.cnf C:\my.ini C:\my.cnf C:\Program Files\MySQL\MySQL Cluster 7.5\my.ini C:\Program Files\MySQL\MySQL Cluster 7.5\my.cnf
```

MySQL 程序在 Windows 上接受两种类型的配置文件名——`my.cnf` 和 `my.ini`。您可以随意选择文件名。

除了配置文件之外，MySQL 程序在读取所有选项文件后，还会从命令行读取选项。因此，在命令行上指定为参数的命令行选项具有最高的优先级。

此外，有几个选项会影响如何读取选项文件，描述如下：

*   `--no-defaults`：不读取任何选项文件，只接受命令行选项。
*   `--defaults-file`：只读取作为此选项参数给出的选项文件。任何其他选项文件将被忽略。
*   `--defaults-extra-file`：除了选项文件外，还读取作为参数给出的选项文件。

这些选项是互斥的，因此您一次只能指定其中一个。这些选项必须指定为第一个命令行选项。任何其他选项都不能放在这些选项之前。

在 UNIX 类系统上，MySQL Server 还会读取由环境变量 `$MYSQL_HOME` 指定目录下的选项文件。如果未设置该变量，`mysqld --help --verbose` 将不会显示它。读取选项文件的规则相当复杂。表 4-12 显示了选项文件位置的列表。选项文件按照从上到下的顺序读取。我特意从表 4-12 中省略了 Windows 的 `.cnf` 文件，因为它们是冗余的，并且具有相似的优先级（`.cnf` 文件在对应的 `.ini` 文件之后立即读取）。例外是 `.mylogin.cnf`，因为它没有 `.ini` 变体。

### 本章总结

本章详细讨论了管理节点、数据节点和 SQL 节点的主要配置选项。在您检查集群配置时，本章可作为参考。

正如您在本章中所见，MySQL NDB Cluster 有许多选项。配置如此众多的选项是一项艰巨的任务。数据节点尤其拥有众多配置选项，且其配置至关重要。在生产系统中，并非所有数据节点选项都必须进行配置。如本章所述，数据节点配置中有几个至关重要的类别：
*   **内存消耗**：明智地利用服务器机器上可用的全部内存至关重要。内存既用于存储，也在 MySQL NDB Cluster 操作期间用于缓冲处理中的数据。内存大小配置是 MySQL NDB Cluster 最关键的配置项。
*   **重做日志大小**：在繁忙的系统中，MySQL NDB Cluster 需要相对较大的重做日志，因为它必须容纳两次连续本地检查点之间所做的所有修改。如果重做日志空间耗尽，在正在进行的本地检查点完成之前，客户端无法提交新事务。
*   **检查点速度**：检查点速度对于在合理时间内完成本地检查点非常重要。加快检查点速度将节省重做日志所需的空间。
*   **多线程**：若未在数据节点上正确配置多线程，无论服务器机器有多少 CPU 核心，都不可能获得良好性能。

通常，配置必须以自上而下的方式进行。微调可以稍后进行，但最初必须选择正确的方向。
