## 基础知识

## 1. 架构与核心概念

MySQL NDB Cluster 是一个采用无共享架构的分布式实时数据库管理系统。它擅长处理大量小型事务的工作负载，旨在通过两阶段提交和同步复制来提供高可用性，确保没有单点故障并实现数据复制。高可用性实现的一部分还包含对实时性的承诺；也就是说，MySQL NDB Cluster 力求提供一致的响应时间。为避免一个节点上的问题导致集群其余部分出现延迟，MySQL NDB Cluster 采用了“尽早失败”策略，这主要影响数据节点，是用户应当牢记的一点。

集群内部的通信使用 NDB API（一种 NoSQL API）进行，该 API 最初是与 NDB Cluster 的其他部分在 1990 年代一同开发的。然而，自 2003 年 NDB Cluster 成为 MySQL 的一部分以来，已开发出一种使用可插拔存储引擎 API 接入 MySQL Server 的存储引擎。因此，如今与 NDB Cluster 交互最常用的方法是通过类似于 MySQL 中其他存储引擎（如 `InnoDB`）的 SQL 语句。

> 注意
>
> 对于 MySQL NDB Cluster，术语 NoSQL API 指的是一种允许绕过 SQL 层与后端存储引擎通信的应用程序编程接口。无论使用何种 API，底层存储都是相同的，并且相同的数据可以使用不同的 API（包括 SQL API）并发访问。章节 18 和 19 会更详细地讨论这些 API。

随着持续的发展，如今的 MySQL NDB Cluster 是一个通用存储引擎，支持广泛的功能，包括 MySQL 中其他任何存储引擎都不支持的特性。

本章通过审视 MySQL NDB Cluster 的一般特性和功能、局限性、各种节点类型以及内置的高可用性，为读者提供了其全貌。章节 2 将通过关注后端存储——数据节点，继续展开讨论。但首先，让我们快速了解一下 MySQL NDB Cluster 中的术语。

### 特性与特点

MySQL NDB 集群是一项将传统独立式 MySQL 服务器进行扩展，以提供高可用性存储引擎的技术。存储引擎的特性和功能集使其区别于其他引擎。本节将介绍一些高层次的特性、功能、限制以及适用场景。

#### 架构

MySQL NDB 集群的一些重要架构特性包括：

*   主要基于内存的数据库：内存中的数据提供快速且一致的访问。磁盘数据也支持非索引列。
*   磁盘数据：可以使用磁盘表空间来容纳海量数据。
*   分布式系统：集群通过在一个或多个主机上部署节点来创建。这包括将数据分散到多个进程中。
*   无共享架构：正如“内置高可用性”一节所讨论的，MySQL NDB 集群采用无共享架构，以确保没有单点故障。
*   商品化硬件：MySQL NDB 集群可运行于商品化硬件之上，使其硬件成本相对低廉。
*   数据复制：数据可以被复制，以确保单个节点故障不会中断数据访问。数据复制通过同步复制和两阶段提交来完成。

#### 功能

MySQL NDB 集群继承了许多来自 MySQL 服务器的功能，但也有几项功能是 MySQL NDB 集群特有的。本小节概述了最重要的功能。本书的后续章节将对这些功能进行更详细的介绍。

##### 符合 ACID 的事务

**ACID** 代表（原子性、一致性、隔离性、持久性），是事务型数据库的一个重要特性。原子性意味着一个变更要么被完全应用，要么完全不应用，事务内的所有变更将被视为一个整体变更。一致性意味着从数据库角度看，内容始终是正确的（例如，满足约束条件）。隔离性确保两个并发事务只能做出彼此隔离的变更（即，结果与它们按顺序执行相同）。持久性是指已提交的事务在随后发生崩溃时不会丢失的属性。在将数据存储在内存中的同时如何确保持久性，将在第 2 章中讨论。

##### 外键

`NDBCluster`存储引擎支持外键，包括对分区表的支持。这使得`NDBCluster`成为 MySQL 中唯一支持在分区表上使用外键的存储引擎。外键涉及的两张表都必须是`NDBCluster`表，并且在父表和子表的被引用列上都必须存在索引。

##### 内置高可用性

“高可用性”一词有多种含义。在 MySQL NDB 集群中，它意味着集群能够执行查询，并且响应时间是可预测的。MySQL NDB 集群从设计之初就是为了提供高可用性，而不是在现有产品之上附加高可用性功能。

响应查询的能力源于无共享架构以及对数据副本和每种节点类型多副本的支持。这使得即使某个节点（或在某些情况下是多个节点）不可用，集群也能继续运行。一致的响应时间则通过多种手段确保，例如将数据存储在内存中、不允许数据节点内部作业处理阻塞超过 10 毫秒，以及关闭速度过慢的节点。

总体而言，这意味着使用 MySQL NDB 集群的高可用性功能不需要特殊软件。本章后续将更详细地讨论高可用性。

##### 自动故障切换

当一个节点关闭时，剩余节点会自动处理与该节点故障相关的任何任务。网络分区问题通过仲裁解决，必要时会采用“直接杀死对方节点”的策略。故障切换完成后，剩余节点可以继续运行。

##### SQL 与 NoSQL 访问

可以选择多种 API 来访问集群中的数据：SQL 节点、`NDB API`、`ClusterJ`、`Node.js`和`NDB-memcached`。（`NDB-memcached`在此上下文中较为特殊，因为它需要在客户端/应用程序和数据节点之间运行一个守护进程。）应用程序可以混合使用多种 API，例如通过 SQL 节点执行复杂查询，并使用`ClusterJ`进行较简单的查询。这为开发人员提供了灵活性，可以为每项任务使用功能最合适的 API。

##### 自动分区与自动分片

数据的分布基于集群配置自动完成。数据节点越多，为数据节点配置的并行度（线程数）越高，表的分区数就越多。分片操作在数据节点之间进行，因此每个数据节点只存储一部分数据。自动化意味着应用程序无需关心分区和分片的具体实现方式。

##### 水平可扩展性

水平扩展系统是通过向系统添加更多节点来实现的。（另一方面，垂直扩展则通过提升每个节点的硬件性能来实现。）可以在集群在线时添加数据节点、管理节点和 API 节点。添加数据节点尤其能提高集群的存储容量，但对于某些工作负载也可能提升性能（参见第 3 章和第 20 章）。随着需求变化，可以添加更多 API/SQL 节点，以帮助在节点间分配负载，从而提高吞吐量。

##### 在线操作

广泛支持在线执行操作，这也是高可用性策略的一部分。如前所述，所有三种节点类型（数据节点、管理节点和 API 节点）都可以在集群在线时添加。还支持在线进行模式变更——事实上，MySQL NDB 集群是 MySQL 中第一个支持在线模式变更的存储引擎。最后，升级和降级都可以在集群在线时完成。这包括更改补丁发布版本（例如，从 7.5.4 到 7.5.5）以及跨主要版本迁移（例如，从 7.4.14 到 7.5.5）。

##### 地理复制

地理复制使得可以设置另一个集群或 MySQL 服务器实例来存放数据副本。这基于 MySQL 服务器的标准（异步）复制。当在复制源服务器上进行变更时，会通过`二进制日志`发送到复制副本服务器。复制副本可以位于同一数据中心，也可以远在另一个大洲。

复制副本有多种用途。它们允许应用程序扩展读取操作，可用于提高系统的整体吞吐量。另一个用途是拥有一个备用集群，当复制源服务器关闭时可以启用——这提高了可用性。最后，某些工作负载并非最适合 MySQL NDB 集群，因此复制副本可以使用`InnoDB`存储引擎来处理，例如，联机分析处理工作负载。

复制是一个重要主题，并且为了支持 MySQL NDB 集群的分布式特性，复制功能有若干变更。第 6 章专门讨论复制的工作原理。




#### 主主复制中的冲突解决

在跨地域复制场景中，可以允许多个集群同时进行写入操作。由于复制是异步的，这会导致一个问题：写入不同集群的数据可能会产生冲突。从事务提交到数据变更完成复制之间存在延迟，因此这种冲突直到两个事务都提交后才会被检测到。传统上，解决这类冲突需要人工干预——通常需要重建复制设置中的某一端。

`MySQL NDB Cluster` 支持自动处理冲突，可以通过判定哪些更新胜出，和/或将冲突更新的详细信息记录下来，以便数据库管理员更轻松地决定后续处理方式。

### 限制

与所有其他数据库实现一样，`MySQL NDB Cluster` 也存在限制。了解其优势和限制非常重要，这样才能确保应用程序设计能够充分利用其优势，同时尽量减少限制带来的影响。存在限制的原因有多种。一个常见的原因是需要在两个特性之间做出权衡，例如性能与功能。这也源于当年的设计决策，有些决策甚至是在二十年前做出的。随着新功能的开发以及软硬件的进步，这些限制会随时间而改变。`NDBCluster` 存储引擎的限制可分为两大类：模式限制和其他限制。

注意：`MySQL NDB Cluster` 的限制在参考手册中有详细记载，地址为 `https://dev.mysql.com/doc/refman/5.7/en/mysql-cluster-limitations.html`。

##### 模式限制

模式限制涉及表、索引以及由数据库用户创建的其他对象。主要的模式限制包括：

*   所有表必须具有主键。如果未提供，`MySQL NDB Cluster` 将添加一个隐藏的自增列作为主键。但是，这个隐藏的主键会给表带来一些限制：隐藏主键不能用于过滤条件，包括在复制从库上执行复制语句时；具有隐藏主键的表不支持在线模式变更；如果表包含一个或多个 `BLOB`、`TEXT` 或 `JSON` 列，该表将不会被包含在二进制日志中。
*   任何索引的宽度不能超过 3072 字节。
*   支持的索引类型仅有：用于有序索引的 `T-Tree` 索引（类似于 `B-Tree` 索引，但为内存使用进行了优化）和用于唯一索引的哈希索引。索引类型也在第 18 章有更详细的讨论。
*   不能在 `BLOB`、`TEXT` 和 `BIT` 列上创建索引。`BLOB` 和 `TEXT` 数据使用辅助表存储（详情将在下一章介绍），多个 `BIT` 列会被组合存储在一个存储位置。这种设计阻止了在这些数据类型的列上创建索引。
*   不支持前缀索引（即仅索引列的前 N 个字符/字节）。
*   索引列不能存储在磁盘上；它们总是存储在内存中以实现最佳性能。有序索引的 `T-Tree` 索引实现假设索引存储在内存中，哈希索引则部分存储在专用的内存段（索引内存）中。
*   如果外键引用的是父表的主键，则不能设置为 `ON DELETE CASCADE`。
*   最大行大小为 14000 字节。行大小的计算方式为：每个列的宽度之和，其中 `BLOB`、`TEXT` 和 `JSON` 数据类型各贡献 264 字节。每个列的贡献按 4 字节对齐；例如，一个使用 `UTF8` 的 `VARCHAR(12)` 列，内容需要 `12 * 3` 字节，存储长度需要 1 字节，共 37 字节。但是，由于 4 字节对齐，该列对行大小的贡献为 40 字节。
*   `MySQL NDB Cluster` 中数据库、表和索引的总数限制为 20320。
*   `NDBCluster` 不能用于临时表（`CREATE TEMPORARY TABLE …` 语句）。

##### 其他限制

还有其他一些限制分布在不同的类别中，例如事务支持和二进制日志记录。最重要的限制包括：

*   仅支持 `READ COMMITTED`（读已提交）事务隔离级别。这与 Oracle DB 类似，但与 `InnoDB` 存储引擎不同。
*   不支持保存点。
*   对 `NDBCluster` 中数据变更的二进制日志事件始终使用基于行（ROW）的格式记录。
*   不支持全局事务标识符（`GTID`）。
*   在数据节点重启期间，不能进行任何模式变更（`DDL`）操作。
