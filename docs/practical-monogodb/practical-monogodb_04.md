# 3. MongoDB 简介

> “MongoDB 是领先的 NoSQL 文档存储数据库之一。它使组织能够处理大数据并从中获得有价值的洞见。”

一些领先的企业和消费级 IT 公司已在他们的产品和解决方案中利用了 MongoDB 的能力。MongoDB 3.0 版本引入了可插拔的存储引擎和 Ops Manager，这扩展了最适合 MongoDB 的应用程序范围。

MongoDB 的名字源于单词“humungous”（巨大的）。与其他 NoSQL 数据库一样，MongoDB 也不遵循 RDBMS 原则。它没有表、行和列的概念。同时，它也不提供 ACID 合规性、JOIN、外键等功能。

MongoDB 以二进制 JSON 文档（也称为 BSON）的形式存储数据。文档可以具有不同的模式，这意味着模式可以随着应用程序的发展而改变。MongoDB 专为可扩展性、高性能和高可用性而构建。

在本章中，我们将简要介绍 MongoDB 的创建和设计决策。我们将在后续章节中探讨 MongoDB 的关键特性、组件和架构。

## 3.1 历史

2007 年下半年，Dwight Merriman、Eliot Horowitz 和他们的团队决定开发一项在线服务。该服务的意图是提供一个开发、托管和自动扩展 Web 应用程序的平台，与 Google App Engine 或 Microsoft Azure 等产品非常相似。很快他们意识到，没有开源数据库平台能满足该服务的要求。

“我们感觉很多现有数据库并不真正具备你希望它们拥有的‘云计算’原则：弹性、可扩展性……以及易于管理，同时对开发人员和操作人员也易于使用，”Merriman 说。“[MySQL] 并不具备所有这些特性。”¹ 因此，他们决定构建一个不遵循 RDBMS 模型的数据库。

一年后，用于该服务的数据库已可使用。该服务本身从未发布，但团队在 2009 年决定将该数据库作为 MongoDB 开源。2010 年 3 月，MongoDB 1.4.0 版本的发布被认为已具备生产可用性。最新的生产版本是 3.0，于 2015 年 3 月发布。MongoDB 是在总部位于纽约的初创公司 10gen 的赞助下构建的。

## 3.2 MongoDB 设计哲学

Eliot Horowitz 在一次演讲中提到，MongoDB 并非在实验室中设计，而是从构建大规模、高可用和健壮系统的经验中诞生的。在本节中，我们将简要了解一些导致 MongoDB 如今形态的设计决策。

### 3.2.1 速度、可扩展性和敏捷性

设计团队在设计 MongoDB 时的目标是创建一个快速、高度可扩展且易于使用的数据库。为了在分区数据库中实现速度和水平可扩展性，正如 CAP 定理所解释的那样，必须在一致性和事务支持上做出妥协。因此，根据该定理，MongoDB 以牺牲一致性和事务支持为代价，提供了高可用性、可扩展性和分区能力。在实际应用中，这意味着 MongoDB 使用文档而非表和行，以使其具备灵活性、可扩展性和快速性。
