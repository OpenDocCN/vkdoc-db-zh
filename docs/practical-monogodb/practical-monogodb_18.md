# 10. MongoDB 用例

> “MongoDB：它对我有用吗？”

在本章中，我们将提供 MongoDB 的特性与其适合解决的业务问题之间急需的连接。我们将使用两个用例来分享用于解决此类问题的技术和模式。

## 10.1 用例 1 - 性能监控

在本节中，您将探索如何使用 MongoDB 存储和检索性能数据。您将重点关注用于存储数据的数据模型。检索将仅包括从相应集合中读取。您还将了解如何应用分片和复制以获得更好的性能和数据安全性。

我们假设有一个监控工具正在以 CSV 格式收集服务器定义的参数数据。通常，监控工具要么将数据作为文本文件存储在服务器上的指定文件夹中，要么将其输出重定向到任何报告数据库服务器。在此用例中，有一个调度程序将读取此共享文件夹路径并将数据导入 MongoDB 数据库。

### 10.1.1 模式设计

设计解决方案的第一步是决定模式。模式取决于监控工具捕获的数据格式。

日志文件中的一行可能类似于表 10-1。

表 10-1. `日志文件`

| 节点 UUID | IP 地址 | 节点名称 | MIB | 时间戳（毫秒） | 指标值 |
| --- | --- | --- | --- | --- | --- |
| 3beb1a8b-040d-4b46-932a-2d31bd353186 | 10.161.1.73 | corp_xyz_sardar | IFU | 1369221223384 | 0.2 |

以下是将每一行存储为文本的最简单方式：

```json
{
  "_id": ObjectId(...),
  "line": "10.161.1.73 - corp_xyz_sardar [15/July/2015:13:55:36 -0700] \"Interface Util\" ..."
}
```

虽然这捕获了数据，但对用户来说毫无意义，因此如果您想找出特定服务器的事件，则需要使用正则表达式，这将导致对集合进行全扫描，效率非常低。

相反，您可以从日志文件中提取数据，并将其作为有意义的字段存储在 MongoDB 文档中。

请注意，在设计结构时，使用正确的数据类型非常重要。这不仅节省空间，而且对性能有显著影响。

例如，如果您将日志的日期和时间字段存储为字符串，它不仅会占用更多字节，而且执行日期范围查询也会很困难。如果不使用字符串，而是将日期存储为 UTC 时间戳，它将占用 8 个字节，而字符串需要 28 个字节，因此执行日期范围查询会更容易。如您所见，为数据使用正确的类型增加了查询的灵活性。

您将使用以下文档来存储您的监控数据：

```json
{
  "_id": ObjectID(...),
  "Host": "",
  "Time": ISODate(""),
  "ParameterName": "aa",
  "Value": 10.23
}
```

注意

实际的日志数据可能有额外的字段；如果您捕获所有字段，它将导致文档过大，这是对存储和内存的低效使用。在设计模式时，您应该省略不需要的详细信息。识别出为满足您的要求必须捕获哪些字段非常重要。

在您的场景中，满足报告应用程序要求的最重要信息是以下内容：

`主机` `时间戳` `参数` `值`

### 10.1.2 操作

设计好文档结构后，接下来您将查看需要在系统上执行的各种操作。



#### 10.1.2.1 插入数据

用于插入数据的方法取决于您的应用程序的写入关注级别。

如果您追求快速的插入速度，并且可以在数据安全性上做出妥协，则可以使用以下命令：

```javascript
db.perfpoc.insert({Host:"Host1", GeneratedOn: new ISODate("2015-07-15T12:02Z"),ParameterName:"CPU",Value:13.13},w=0)
```

尽管此命令是可用的最快选项，但由于它不会等待操作是否成功的任何确认，您可能会面临丢失数据的风险。

如果您只想要一个至少数据正在被保存的确认，您可以执行以下命令：

```javascript
db.perfpoc.insert({Host:"Host1", GeneratedOn: new ISODate("2015-07-15T12:07Z"),ParameterName:"CPU",Value:13.23},w=1)
```

尽管此命令确认了数据已保存，但它不会提供防止任何数据丢失的保护，因为它没有启用日志记录。

如果您的主要关注点是在增加插入速度和数据安全保证之间进行权衡，您可以执行以下命令：

```javascript
db.perfpoc.insert({Host:"Host1", GeneratedOn: new ISODate("2015-07-15T12:09Z"),ParameterName:"CPU",Value:30.01},j=true,w=2)
```

在此代码中，您不仅确保数据正在被复制，还启用了日志记录。除了复制确认外，它还会等待成功的日志提交。

**注意**
虽然这是最安全的选项，但它对插入性能有严重影响，因此是最慢的操作。

#### 10.1.2.2 批量插入

在使用严格的写入关注级别时，批量插入事件总是有益的，正如您的情况，因为这使 MongoDB 能够将产生的性能损失分摊到一批插入中。

如果可能，应使用批量插入来插入监控数据，因为数据量将会很大，并且在几秒钟内生成。将它们分组在一起作为一组进行插入会有更好的效果，因为在相同的等待时间内，多个事件将被保存。因此，对于此用例，您将使用批量插入将多个事件分组。

#### 10.1.2.3 查询性能数据

您已经看到了如何插入事件数据。维护数据的价值在于，您能够通过查询数据来响应特定的查询。

例如，您可能希望查看与特定字段（例如 `Host`）关联的所有性能数据。

您将查看一些用于获取数据的查询模式，然后了解如何优化这些操作。

#### 查询 1：获取特定主机的性能数据

```javascript
db.perfpoc.find({Host:"Host1"})
```
```
{ "_id" : ObjectId("553dc64009cb76075f6711f3"), "Host" : "Host1", "GeneratedOn": ISODate("2015-07-18T12:02:00Z"), "ParameterName" : "CPU", "Value" : 13.13 }
{ "_id" : ObjectId("553dc6cb4fd5989a8aa91b2d"), "Host" : "Host1", "GeneratedOn": ISODate("2015-07-18T12:07:00Z"), "ParameterName" : "CPU", "Value" : 13.23 }
{ "_id" : ObjectId("553dc7504fd5989a8aa91b2e"), "Host" : "Host1", "GeneratedOn": ISODate("2015-07-18T12:09:00Z"), "ParameterName" : "CPU", "Value" : 30.01 }
```
如果需求是分析主机的性能，则可以使用此查询。

在 `Host` 上创建索引将优化上述查询的性能：

```javascript
db.perfpoc.ensureIndex({Host:1})
```

#### 查询 2：获取 2015 年 7 月 10 日至 2015 年 7 月 20 日日期范围内的数据

```javascript
db.perfpoc.find({GeneratedOn:{"$gte": ISODate("2015-07-10"), "$lte": ISODate("2015-07-20")}})
```
```
{ "_id" : ObjectId("5302fec509904770f56bd7ab"), "Host" : "Host1", "GeneratedOn"
...............
```
如果您想要考虑和分析特定日期范围内收集的数据，这一点很重要。在这种情况下，"时间"上的索引将对性能产生积极影响。

```javascript
db.perfpoc.ensureIndex({GeneratedOn:1})
```

#### 查询 3：获取特定主机在 2015 年 7 月 10 日至 2015 年 7 月 20 日日期范围内的数据

```javascript
db.perfpoc.find({GeneratedOn:{"$gte": ISODate("2015-07-10"), "$lte": ISODate("2015-07-20")},Host: "Host1"})
```
```
{ "_id" : ObjectId("5302fec509904770f56bd7ab"), "Host" : "Host1", "GeneratedOn"
.................
```
如果您想要查看特定时间段内主机的性能数据，此查询非常有用。

在此类涉及多个字段的查询中，所使用的索引对性能有显著影响。例如，对于上述查询，创建复合索引将是有益的。

还需注意复合索引中字段的顺序有影响。让我们通过一个示例来理解差异。创建如下复合索引：

```javascript

```

#### 查询 4：按主机和日期获取性能数据的数量

列出数据很好，但通常对性能数据执行查询是为了在分析时找出计数、平均值、总和或其他聚合操作。在这里，您将看到如何使用 `aggregate` 命令来选择、处理和聚合结果，以满足强大的临时查询需求。

为了进一步解释这一点，让我们编写一个按月计数数据的查询：

```javascript
db.perfpoc.aggregate(
... [
... {$project: {month_joined: {$month: "$GeneratedOn"}}},
... {$group: {_id: {month_joined: "$month_joined"}, number: {$sum:1}}},
... {$sort: {"_id.month_joined":1}}
... ]
... )
```
```
{ "_id" : { "month_joined" : 7 }, "number" : 4 }
```
为了优化性能，您需要确保筛选字段有索引。您已经创建了一个覆盖相同内容的索引，因此在此场景中您无需创建任何额外的索引。



### 10.1.3 分片

性能监控数据集非常庞大，因此迟早会超过单台服务器的容量。因此，您应考虑使用分片集群。

在本节中，您将了解哪种分片键适合您的性能数据使用场景，以便将负载分散到集群中，避免单台服务器过载。

分片键控制着数据的分布方式，并直接影响系统处理查询和写入的能力。理想的分片键应具备以下两个特点：

*   插入操作在分片集群中保持平衡。
*   大多数查询可以被定向到部分分片上得到满足。

让我们看看哪些字段可以用作分片键。

**时间字段：** 选择此选项，虽然数据会在分片间均匀分布，但无论是插入还是读取都不会平衡。就像性能数据的情况一样，时间字段是单调递增的，因此所有插入最终都会落到单个分片上，写入吞吐量将与独立实例相同。假设您经常需要查看最新数据，那么大多数读取操作最终也会落在同一个分片上。

**哈希值：** 您也可以考虑使用随机值来应对上述情况；可以考虑将 `_id` 字段的哈希值作为分片键。虽然这能解决上述的写入问题（即写入会被分布开），但会影响查询。在这种情况下，查询必须广播到所有分片，无法被定向到特定分片。

**使用均匀分布的字段，例如主机名。** 这具有以下优点：如果查询选择了 `host` 字段，读取将是选择性的，并且仅限于单个分片，写入也是平衡的。然而，最大的潜在缺点是，由于所有文档都具有相同的分片键，为单个主机收集的所有数据必须进入同一个块。如果数据是跨所有主机收集的，这不会成为问题；但如果监控为某个主机收集了不成比例的大量数据，您最终可能会得到一个非常大的块，这个块可能完全无法再分裂，导致负载不平衡地集中在一个分片上。

**结合方案二和三的优点，** 您可以使用一个复合分片键，例如 `{host:1, ssk: 1}`，其中 `host` 是文档的主机字段，`ssk` 是 `_id` 字段的哈希值。在这种情况下，数据主要根据 `host` 字段进行分布，使得访问 `host` 字段的查询被局部化到单个分片或一组分片上。同时，使用 `ssk` 确保了数据在集群中均匀分布。在大多数情况下，这种键不仅能确保写入在集群中理想地分布，还能确保查询仅访问与其相关的特定数量的分片。

尽管如此，最好的方法是分析应用程序实际的查询和插入模式，然后选择一个合适的分片键。

### 10.1.4 管理数据

由于性能数据量巨大且持续增长，您可以定义一个数据保留策略，规定您将维护数据一段指定的时期（例如 6 个月）。

那么如何删除旧数据呢？您可以使用以下模式：

**使用多个集合存储数据：** 第三种模式是创建按天分的集合，其中包含存储当天性能数据的文档。这样，您最终会在一个数据库中有多个集合。虽然这会使查询复杂化（为了获取两天的数据，您可能需要从两个集合中读取），但删除集合的操作很快，并且空间可以被有效重用，不会产生数据碎片。在您的用例中，您正在使用此模式来管理数据。

**使用固定集合：** 虽然固定集合可用于存储性能数据，但无法对固定集合进行分片。

**使用 TTL 集合：** 此模式创建一个类似于固定集合的集合，但它可以被分片。在这种情况下，会在集合上定义一个生存时间索引，使 MongoDB 能够定期从集合中 `remove()` 旧的文档。然而，这不具备固定集合的性能优势；此外，`remove()` 操作可能导致数据碎片。

## 10.2 用例二 – 社交网络

在本节中，您将探讨如何使用 MongoDB 来存储和检索社交网站的数据。

此用例基本是一个友好的社交网站，允许用户分享他们的状态和照片。为此用例提供的解决方案基于以下假设：

*   用户可以选择是否关注其他用户。
*   用户可以决定与哪些圈子的用户分享更新。圈子选项包括：朋友、朋友的朋友、公开。
*   允许的更新包括状态更新和照片。
*   用户资料显示兴趣、性别、年龄和感情状况。



### 10.2.1 模式设计

你所提供的解决方案旨在最小化展示任意给定页面所需加载的文档数量。所讨论的应用程序有两个主要页面：第一个页面展示用户的个人墙（旨在展示由特定用户创建或指向该用户的帖子），以及一个社交新闻页面，该页面展示所有关注该用户的人或该用户所关注的人的所有通知和活动。

除了这两个页面，还有一个用户个人资料页面，用于展示用户与个人资料相关的详细信息，并包含其好友圈（关注他的人或他关注的人）的信息。为了满足此需求，此用例的模式包含以下集合。

第一个集合是 `user.profile`，它存储用户的个人资料相关数据：

```json
{
    _id: "用户定义的唯一标识符",
    UserName: "用户名",
    ProfilDetaile: {Age:.., Place:.., Interests: ...etc},
    FollowerDetails: {
        "User_ID":{name:..., circles: [circle1, circle2]}, ....
    },
    CirclesUserList: {
        "Circle1": {"User_Id":{name: "username"}, ......},
        .......
    },
    ListBlockedUserIDs: ["user1",...]
}
```

*   在这种情况下，你是手动指定 `_id` 字段。
*   `Follower` 列出了关注此用户的用户。
*   `CirclesUserList` 包含此用户正在关注的好友圈。
*   `Blocked` 包含用户已阻止查看其更新的用户。

第二个集合是 `user.posts` 集合，其模式如下：

```json
{
    _id: ObjectId(...),
    by: {id: "用户 id", name: "用户名"},
    VisibleTocircles: [],
    PostType: "帖子类型",
    ts: ISODate(),
    Postdetail: {
        text: "",
        Comments_Doc: [
            {Commentedby: {id: "user_id", name: "用户名"}, ts: ISODate(), Commenttext: "评论内容"}, .....
        ]
    }
}
```

*   此集合基本上用于展示用户的所有活动。`by` 提供发布帖子的用户信息。`Circles` 控制帖子对其他用户的可见性。`Type` 用于标识帖子内容。`ts` 是帖子创建的日期时间。`detail` 包含帖子文本，其中内嵌了评论。
*   一个 `comment` 文档包含以下详情：`by` 提供评论用户的用户 id 和姓名详情，`ts` 是评论时间，`text` 是用户发布的实际评论。

第三个集合是 `user.wall`，它用于在瞬间渲染用户的个人墙页面。此集合从第二个集合获取数据并以汇总格式存储，以便快速渲染墙页面。该集合格式如下：

```json
{
    _id: ObjectId(...),
    User_id: "用户 id",
    PostMonth: "201507",
    PostDetails: [
        {
            _id: ObjectId(..), ts: ISODate(), by: {_id: .., Name:.. }, circles: [..], type: ....,
            detail: {text: "..."}, comments_shown: 3,
            comments: [
                {by: {_id:., Name:....}, ts: ISODate(), text:""}, ......
            ]
        },....
    ]
}
```

*   如你所见，你是按用户、按月维护此文档。首次显示的评论数量是有限的（此例中为 3 条）；如果需要为该特定帖子显示更多评论，则需要查询第二个集合。
*   换句话说，它是一种用于快速加载用户个人墙页面的汇总视图。

第四个集合是 `social.posts`，它用于快速渲染社交新闻屏幕。这是显示所有帖子的屏幕。与第三个集合类似，第四个集合也是一个依赖集合。它包含了许多与 `user.wall` 信息相同的内容，因此为了清晰起见，此文档已被缩写：

```json
{
    _id: ObjectId(...),
    user_id: "用户 id",
    postmonth: '2015_07',
    postlists: [ ... ]
}
```
