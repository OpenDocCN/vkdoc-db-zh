# 数据库镜像：自动故障转移与客户端连接性

在进行辅助服务器的滚动升级时，需要执行故障转移，然后升级之前的主服务器。请务必记住，故障转移后不可能回滚到旧版本的 SQL Server。同时，请记得在升级后使用 `sp_updatestats` 存储过程更新数据库中的所有统计信息。

同步数据库镜像提供两种不同的模式：*高保护* 和 *高可用性*。这两种模式的唯一区别在于对自动故障转移的支持。SQL Server 在高可用性模式下支持自动故障转移；但是，它需要您拥有第三个 SQL Server 实例，即 *见证服务器*，它有助于在配置中建立仲裁。

> **注意**
>
> 仲裁是指参与数据库镜像会话的服务器就它们的角色达成一致；即，哪台服务器充当主体服务器，哪台服务器充当镜像服务器。实际上，只要至少两台服务器（来自主体、镜像和见证服务器）能够相互通信，就可以建立仲裁。我们将在本章后面讨论当一台或多台服务器不可用时镜像会发生什么情况。

您可以使用任何版本的 SQL Server（包括 Express 版）作为见证服务器。然而，至关重要的是，见证实例必须安装在另一台物理服务器上，以避免单台物理服务器的硬件故障导致多个 SQL Server 实例离线并阻止建立仲裁的情况。

表 32-1 显示了不同数据库镜像模式之间的异同。

***表 32-1.** 数据库镜像模式*

| | **高性能** | **高保护** | **高可用性** |
| :--- | :--- | :--- | :--- |
| **提交** | 异步 | 同步 | 同步 |
| **SQL Server 版本** | 仅企业版 | 企业版和标准版 | 企业版和标准版 |
| **数据丢失** | 可能 | 当数据库处于 `SYNCHRONIZED` 状态时不可能 | 当数据库处于 `SYNCHRONIZED` 状态时不可能 |
| **自动故障转移** | 不支持 | 不支持 | 在有见证服务器的情况下支持 |
| **性能影响** | 无 | 网络和镜像 I/O 子系统延迟 | 网络和镜像 I/O 子系统延迟 |

让我们看看高可用性模式下几个可能的故障转移场景。这里的关键点是，在任何时间点，服务器都必须拥有仲裁，因此至少两台服务器必须能够相互连接。

首先，假设主体服务器和见证服务器失去了与镜像服务器的连接。主体和见证服务器仍然拥有仲裁，因此主体服务器继续在 `SUSPENDED` 状态下进行镜像工作。如果在此阶段主体服务器失去了与见证服务器的连接，因此没有仲裁，主体服务器将关闭。

> **提示**
>
> 考虑将见证实例放置在靠近主体服务器的位置，以避免它们之间的连接问题以及不必要的故障转移和关闭。当服务器重新联机时，将故障转移回靠近见证实例的前主体服务器也是有益的。

现在，假设主体服务器脱机。在这种情况下，镜像和见证服务器可以相互看见，因此拥有仲裁，于是发生自动故障转移，镜像服务器成为新的主体服务器。如果旧的主体服务器重新上线并看到这两台服务器，它将成为镜像服务器并与新的主体服务器同步自身。否则，它将自行关闭以避免 *脑裂* 情况，即两台不同的服务器允许客户端连接到同一数据库的不同副本，并同时更改数据。

如果见证服务器脱机，镜像将继续工作，但无法执行自动故障转移。这类似于高保护模式，不同之处在于，如果主体服务器失去与镜像服务器的连接，而见证服务器又不可用，主体服务器将关闭以避免脑裂情况。
