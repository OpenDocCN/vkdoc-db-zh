# 第 28 章：系统故障排除

## 分析内存授予问题

如果发生查询需要如此多内存授予的情况，需分析其原因。

最后，`DBCC MEMORYSTATUS` 命令提供了有关 SQL Server 内存使用情况的信息，以及内存节点和内存 clerk 的统计信息。尽管这些信息非常详细，但在许多情况下，使用 `sys.dm_os_memory_clerks` 和 `sys.dm_os_memory_nodes` 视图在查询中执行筛选、分组和聚合操作会更为简便。

## 当服务器无响应时该怎么办

SQL Server 停止响应或不接受用户请求的情况并不常见。然而，它们确实有时会发生，而首要且最重要的规则是不要惊慌。SQL Server 始终将数据一致性视为最高优先级，数据发生问题的可能性极低。

作为第一步，你应该确认问题并非基础设施相关。你应该检查服务器和网络是否正常运行，并且问题并非局限于特定的客户端工作站或网络子集。问题很可能与 SQL Server 完全无关。例如，防火墙配置的更改或网络交换机故障可能会阻断 SQL Server 与客户端应用程序之间的通信。

接下来，你应该检查 SQL Server 错误日志。某些情况，如长时间的工作者线程短缺，会在日志中留下错误消息，通知系统管理员有关问题。此外，此类情况可能引发未处理的内部异常和小型转储文件。不幸的是，无法保证 SQL Server 在此类异常后会恢复，在某些情况下你需要重启它。

然而，重启的关键点在于对问题进行根本原因分析。你需要分析错误日志和默认跟踪，进行研究，并在某些情况下向 Microsoft 提交支持案例，以确保问题被发现并得到解决。

> **注意：** 未处理的异常通常是由于 SQL Server 中的错误引起的，这些错误可能已在最新的服务包和累积更新中得到修复。考虑应用这些更新，如果问题仍未解决，再向 Microsoft CSS 提交支持案例。

你可能需要连接到 SQL Server 以进行进一步的故障排除。幸运的是，SQL Server 2005 引入了一种特殊的连接，称为 `专用管理员连接 (DAC)`，可用于此目的。SQL Server 为 DAC 保留了一个私有调度器和少量内存，即使 SQL Server 不接受常规连接，这也能让你连接进去。

默认情况下，DAC 仅在本地可用。在某些情况下，当服务器完全过载时，操作系统没有足够的资源来处理用户会话，这会阻止你在本地模式下使用 DAC。你可以使用清单 28-16 所示的代码更改配置设置以允许远程 DAC 连接。显然，最好在初始服务器配置期间启用此设置，而不是等到问题发生。

### 清单 28-16：启用远程管理员连接

```sql
exec sp_configure 'remote admin connections', 1
go
reconfigure
go
```

你可以通过 Management Studio 连接框中使用 `ADMIN:` 服务器名称前缀，或在 `sqlcmd` 中使用 `-A` 选项来通过 DAC 连接到 SQL Server。只有 `sysadmin` 服务器角色的成员才允许连接，并且任何时候只能有一个会话使用 DAC 连接。

> **重要提示：** 当你在 Management Studio 中使用 DAC 时，应使用从查询窗口启动的连接对话框。对象浏览器设计上使用多个数据库连接，因此它无法使用 DAC。在尝试此连接之前，请确保已禁用 `Intellisense` 和其他 Management Studio 插件。
