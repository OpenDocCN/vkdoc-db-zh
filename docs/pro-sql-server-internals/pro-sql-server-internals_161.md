`点对点复制`（如`图 32-10`所示）允许你构建一个分布式且可扩展的解决方案，其中包含驻留在不同服务器（称为`节点`）上的多个可更新数据库。它是企业版的一项功能，基于事务复制，因此在节点之间分发更改的延迟非常低。它适用的一个场景是具有多个数据中心的系统。你可以在每个数据中心托管单独的 SQL Server 实例，并将客户端重定向到最近的实例。点对点复制在所有节点之间同步数据，并处理某个节点暂时失去与其他节点连接的情况。

`图 32-10. 点对点复制`

复制最大的缺点是其复杂性。设置和监控复杂的复制拓扑结构，迄今为止比其他高可用性解决方案要复杂得多。此外，它通常需要实现复杂的冲突解决机制，并且可能需要更改应用程序逻辑和数据库模式以最小化冲突。

我建议避免将复制用于高可用性目的，除非你需要保护数据库中非常小的一部分数据，或者有其他除了高可用性之外也能从复制中受益的用例。

#### 设计高可用性策略

## 第 32 章 ■ 高可用性技术

设计高可用性策略的过程融合了艺术、科学与政治。这是一个收集并经常调整需求、设定合理预期、并构建符合预算的解决方案的迭代过程。

*需求收集*是该过程的第一阶段。如同备份策略一样，你需要处理 `RPO` 和 `RTO` 指标。通常，你可以从服务等级协议（`SLA`）中获得它们。或者，如果这些指标未出现在 `SLA` 中，你应该与系统的利益相关者共同定义它们。

> **注意**
> 系统可用性需求通常以"九的个数"来衡量。例如，`五个九` ，即 99.999%的可用性，意味着系统应在 99.999%的时间内可用，相当于每年 5.26 分钟的停机时间。`四个九` ，即 99.99%的可用性，相当于每年 52.56 分钟的停机时间。`三个九` ，即 99.9%的可用性，允许每年 8.76 小时的停机时间。

与利益相关者协作是一个微妙的过程。虽然利益相关者通常希望零停机和零数据丢失，但这既在技术上不可行，在财务上也不可行。例如，现有的高可用性技术都无法提供零停机。在故障转移过程中，总有一段时间系统是无法访问的。

另一方面，零数据丢失是可以实现的，但这需要付出代价。数据库镜像或 `AlwaysOn 可用性组` 中的同步提交会给修改数据的事务增加开销和额外延迟。在某些情况下，对于高端 `OLTP` 系统，这种开销是不可接受的。

无论哪种情况，预算都是另一个需要考虑的关键因素。实施高可用性策略总会导致额外支出。在大多数情况下，你需要购买新的服务器、软件许可证以及网络和存储设备。这些采购反过来又需要额外的机架空间，并为新硬件及其空调消耗更多 `AC` 电力。此外，你还需要有可用的人力来实施和维护该解决方案。

预算限制了你能实现的目标。如果预算不允许你购买所需的硬件和软件许可证，那么实现 99.999%甚至 99.99%的可用性是不可能的。你应该与系统的利益相关者合作，根据需要调整需求和预期，或争取额外的预算。

另一个重要的行动是定义高可用性解决方案的范围。例如，明确所需的可用性水平是需要全天候达到，还是仅在营业时间达到，这一点非常重要。另一个需要解决的重要问题是解决方案是否应具备地理冗余。该要求可能极大地增加解决方案的复杂性和成本。

在收集并分析所有需求（包括预算限制）之前，切勿开始实施，这一点至关重要。综合起来，这些需求将决定你能够在实施中使用哪种或哪些技术。
