## 与内存相关的等待类型

此外，任何 OLTP 系统都有一些合理的复杂查询能够从并行处理中受益。将`Cost Threshold for Parallelism`（并行度的成本阈值）配置选项调高，比将`MAXDOP`设置为 1 来禁用并行处理要好。这样做可以在处理复杂且开销大的查询时利用并行度，同时让低成本的 OLTP 查询以串行方式运行。

对于如何设置`Cost Threshold for Parallelism`的值，没有通用的建议。默认情况下，该值被设为 5，这在当今是非常低的。你应该分析系统中查询的活动和成本，以找到此设置的最佳值。检查你希望串行和并行运行的查询的成本，并相应地调整阈值。你可以在执行计划中根（顶部）操作符的属性里看到该成本。

说到`MAXDOP`设置，通常它不应超过每个硬件 NUMA 节点的逻辑 CPU 数量。然而，在一些数据仓库系统中，你可以考虑使用超过此数量的`MAXDOP`设置。同样，你应该分析和测试你的工作负载，以找到此设置的最佳值。

SQL Server 从缓冲池中一个称为`workspace memory`（工作区内存）的特殊部分分配查询内存授予。`workspace memory`的最大大小限制为缓冲池大小的 75%。默认情况下，最大查询内存授予大小不能超过`workspace memory`的 25%；不过，你可以通过资源调控器工作负载组中的`REQUEST_MAX_MEMORY_GRANT_PERCENT`设置来控制它。

*Required memory*（必需内存）是存储操作所需的内部数据结构所必需的。没有此内存可用，查询将无法运行。

*Additional memory*（附加内存）用于在操作期间将数据行存储在内存中。附加内存量基于行大小和基数估计。如果附加内存量不足，查询仍可运行，但会在需要时将数据溢出到 tempdb。

必需内存量也受并行度影响。每个工作线程都需要为`sort`或`hash`操作符创建自己的一套内部数据结构。此外，交换操作符将需要一些内存来缓冲行。

在计算完内存授予的大小后，SQL Server 会检查它是否超过最大大小限制，并在需要时减少它。之后，它会向`MEMORYCLERK_SQLQERESERVATIONS`内存 clerks 请求内存，该 clerk 使用一个称为`Resource Semaphore`（资源信号量）的线程同步对象来分配内存。我们将在本章后面讨论内存 clerks 和 SQL Server 内存分配。

当无法分配内存时，`Resource Semaphore` 会将查询放入等待队列，这会导致 `RESOURCE_SEMAPHORE` 等待。在内部，`Resource Semaphore` 使用两个等待队列，根据内存授予大小和查询成本对查询进行排序。一个队列称为 `Small-query Resource Semaphore`（小查询资源信号量），存储需要少于 5 MB 且成本低于 3 个成本单位的查询。第二个队列存储所有其他查询。

当 `Resource Semaphore` 收到新请求时，它首先检查是否有任何查询在等待，然后按照先到先得的原则处理请求。它偏向于小查询队列而非常规队列，这减少了对不需要大量内存的小查询的等待时间。
