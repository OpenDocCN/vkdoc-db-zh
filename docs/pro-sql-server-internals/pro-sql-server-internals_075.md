# 第 25 章 ■ 查询优化与执行

执行计划由物理操作符组成，除了`Parallelism`（并行操作符），它们都使用基于轮询的、逐行的模型。每个父操作符逐行地向其子操作符请求数据。从`SQL Server` 2012 开始，还有另一种批处理模式执行模型可用，该模型与`列存储索引`和一些数据仓库查询一起使用。

操作符有两种类型：阻塞和非阻塞。非阻塞操作符在获得行后立即将其返回给父操作符。阻塞操作符在将行返回给父操作符之前，先从子操作符获取并缓存所有行。

阻塞操作符需要内存来存储数据。在内存估计不正确的情况下，数据会溢出到`tempdb`。这种溢出会降低查询性能，并且可以通过`SQL Trace`和`Extended Events`中的`排序和哈希警告`进行监控。

内存授予大小不正确的两种最常见情况是基数估计和行大小估计不正确。你可以通过保持统计信息最新，并将可变长度数据列的大小定义为实际存储数据的大约两倍来改进这些估计。你还应该避免在连接条件中使用`SARGable 谓词`，尤其是在查询连接大量表时。
