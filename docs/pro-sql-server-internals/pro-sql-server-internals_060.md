# 第 24 章 设计事务策略

#### 注意事项与代码模式

在同时运行 DDL 和 DML 操作的系统中。

SQL Server 在拥有 16 个或更多逻辑处理器的系统上采用锁分区机制。通过锁分区，SQL Server 在每个处理器的基础上维护独立的锁结构。意向锁和架构稳定性锁保存在单个锁分区中，而其他类型的锁则在所有分区中获取并持有。这增加了存储锁信息所需的内存，并且在同时运行 DDL 和 DML 语句时可能增加发生死锁的概率。

SQL Server 2014 和 2016 支持低优先级锁，可用于在在线索引重建和分区切换操作期间减少阻塞。当这些低优先级锁正在等待获取时，它们不会阻塞其他请求不兼容锁类型的会话。

这章内容相当简短，提供了一套通用指南，说明如何设计事务策略并提高系统中的并发性。

当多个会话竞争相同资源集时会发生阻塞。会话尝试在这些资源上获取不兼容的锁，从而导致锁冲突和阻塞。

如你所知，SQL Server 在`处理`数据时获取锁。需要修改或返回给客户端的行数并不重要。重要的是 SQL Server 在执行语句期间访问了多少行。很可能一个只选择或更新单行的查询，由于执行计划中的`索引扫描`操作符，获取了数千甚至数百万个锁。

适当的查询优化和索引调整可减少 SQL Server 在查询执行期间需要访问的行数。这进而减少了要获取的锁的数量以及发生锁冲突的机会。

> **提示** 优化查询。这将有助于提高系统中的并发性、性能和用户体验。

另一种减少锁冲突机会的方法是缩短锁的持有时间。排他（`X`）锁总是持有到事务结束。在`可重复读`和`可序列化`隔离级别下，共享（`S`）锁也是如此。锁持有时间越长，发生锁冲突和阻塞的可能性就越大。

你需要使事务尽可能简短，并避免在事务活动期间通过 UI 进行任何长时间操作或与用户交互。在处理使用`CLR`或链接服务器的外部资源时也需要小心。例如，当链接服务器宕机时，可能需要很长时间才会发生连接超时，而你希望避免锁在此期间一直保持的情况。

> **提示** 使事务尽可能简短。

在尽可能接近事务结束时更新数据。这减少了排他（`X`）锁的持有时间。在某些情况下，使用临时表作为暂存区，将数据插入其中，然后在事务的最末尾更新实际表，可能是有意义的。

这种技术特别有用的一种情况是`UPDATE`语句无法或不切实际进行优化。考虑这样一种情况：该语句扫描大量行但只更新其中少数行。你可以更改代码，将需要更新行的聚集索引键值存储在临时表中，然后基于这些收集的键值运行`UPDATE`语句。

#### 选择事务隔离级别

***清单 24-1.*** 使用临时表减少阻塞：原始语句

```sql
update dbo.Orders
set Cancelled = 1
where
    (PendingCancellation = 1) or
    (Paid = 0 and OrderDate < @MinUnpaidDate) or
    (Status = 'BackOrdered' and EstimatedStockDate > @StockDate)
```

你可以将代码修改为类似于清单 24-2 中所示的形式。`SELECT`语句要么获取共享（`S`）锁，要么根本不获取行级锁，这取决于隔离级别。`UPDATE`语句经过优化，只获取少量的更新（`U`）和排他（`X`）锁。

***清单 24-2.*** 使用临时表减少阻塞：使用临时表暂存待更新操作的关键值

```sql
create table #OrdersToBeCancelled
    ( OrderId int not null primary key );

insert into #OrdersToBeCancelled(OrderId)
select OrderId
from dbo.Orders
where
    (PendingCancellation = 1) or
    (Paid = 0 and OrderDate < @MinUnpaidDate) or
    (Status = 'BackOrdered' and EstimatedStockDate > @StockDate);

update dbo.Orders
set Cancelled = 1
where OrderId in (select OrderId from #OrdersToBeCancelled);
```

你需要记住，虽然这种方法有助于减少阻塞，但创建和填充临时表会引入显著的 I/O 开销，尤其是在涉及大量数据时。此方法应作为最后的手段；在大多数情况下，创建正确的索引是更好的选择。

**提示：** 尽可能在事务接近结束时修改数据。

应避免在同一事务中多次更新同一行，特别是当`UPDATE`语句修改不同非聚集索引中的数据时。请记住，当更新索引行时，SQL Server 会按索引基础获取锁。多次更新会增加当其他会话访问被更新行时发生死锁的可能性。

**提示：** 不要在单个事务中多次更新数据行。

你需要了解锁升级是否影响你的系统，特别是在 OLTP 工作负载的情况下。可以监控对象级阻塞条件和锁等待情况，并将其与*锁升级*扩展事件和跟踪事件关联起来。请记住，锁升级有助于减少内存消耗并提高系统性能。在做出任何决定之前，应分析锁升级发生的原因及其对系统的影响。在许多情况下，修改代码和工作流程比禁用它更好。

**提示：** 监控系统中的锁升级。

一般来说，应避免在同一事务中混合可能导致行级锁和对象级锁的语句，特别是混合 DML 和 DDL 语句。这种模式可能导致死锁，以及意向锁与完整对象级锁之间的阻塞。当服务器具有 16 个或更多逻辑 CPU 时，这一点尤为重要，因为这会启用锁分区。

**提示：** 不要在单个事务中混合 DDL 和 DML 语句。

如果系统中存在死锁，你需要分析其根本原因。在大多数情况下，查询优化和代码重构将有助于解决它们。然而，在某些情况下，特别是涉及锁分区时，可以考虑在系统的关键用例周围实现重试逻辑。

**提示：** 找出死锁的根本原因。如果查询优化和代码重构无法解决它们，则实现重试逻辑。

选择正确的事务隔离级别并非易事。你需要在阻塞和 tempdb 开销之间，以及所需的数据一致性和系统隔离级别之间找到适当的平衡。系统必须为客户提供可靠的数据，你不应该为了减少阻塞而选择无法保证这一点的隔离级别。

应选择提供所需数据一致性要求的*最低要求*隔离级别。



在许多情况下，默认的 `READ COMMITTED`（读已提交）隔离级别已经 **足够好**，特别是在查询经过优化且不执行不必要的扫描时。在 OLTP（在线事务处理）系统中，应避免使用 `REPEATABLE READ`（可重复读）或 `SERIALIZABLE`（可序列化）隔离级别，除非你有正当理由这样做。这些隔离级别会持有共享（S）锁直到事务结束，这可能导致在处理易变数据时出现严重的阻塞问题。它们还会在扫描过程中触发共享（S）锁升级。

作为一般规则，最好避免使用 `READ UNCOMMITTED`（未提交读）隔离级别。尽管许多数据库专业人员为了减少阻塞而转向此隔离级别，无论是显式使用还是使用 `NOLOCK` 提示，但这很少是正确的选择。首先，`READ UNCOMMITTED` 并不能解决由写入者引入的阻塞问题。它们在扫描期间仍然会获取更新（U）锁。然而，最重要的是，通过使用 `READ UNCOMMITTED`，你是在声明根本不需要数据一致性，而这不仅仅是关于读取未提交数据的问题。SQL Server 可能选择在大型表上使用`分配映射扫描`的执行计划，这可能导致在繁忙且数据易变的系统中出现行丢失和因页面拆分导致的重复读取。

在大多数情况下，乐观隔离级别，特别是 `READ COMMITTED SNAPSHOT`（读已提交快照），是比 `READ UNCOMMITTED`、`REPEATABLE READ` 或 `SERIALIZABLE` 更好的选择，即使是在 OLTP 系统中。它在不涉及读者/写入者阻塞的情况下，提供了语句级别的数据一致性。历史上，由于其 `tempdb` 开销，我在 OLTP 系统中建议使用 `RCSI` 时一直非常谨慎；然而，随着现代硬件和基于固态硬盘的磁盘阵列的出现，如今这已经是一个较小的问题了。不过，在你的分析中，仍应考虑额外的索引碎片和 `tempdb` 开销。同样值得注意的是，`READ COMMITTED SNAPSHOT` 在 Microsoft Azure SQL 数据库中是默认启用的。

作为一般规则，我建议你不要在 OLTP 系统中使用 `SNAPSHOT`（快照）隔离级别，因为它会过度使用 `tempdb`，除非绝对需要事务级别的一致性。对于数据仓库和报表系统（数据在大部分时间是静态的），它可能是一个不错的选择。

如果你在数据库中启用了 `SNAPSHOT` 隔离级别，那么在事务管理方面需要非常小心。错误和未提交的事务可能会阻止 `tempdb` 版本存储的清理，并导致 `tempdb` 数据文件的过度增长。即使系统中没有使用 `SNAPSHOT` 事务，只要数据库设置 `ALLOW_SNAPSHOT_ISOLATION` 被启用，这种情况也可能发生。

然而，乐观隔离级别常常会`掩盖`系统中优化不佳的查询。尽管这些查询导致了糟糕的系统性能，但它们不涉及阻塞情况，因此常常被忽视。人们通过启用 `READ COMMITTED SNAPSHOT` 来`解决`读者/写入者阻塞，而后却不再处理阻塞的根本原因，这种情况并不少见。你应该记住这一点，并且无论系统中是否存在阻塞，都应进行查询优化。

对于数据仓库系统，事务策略在很大程度上取决于数据如何更新。对于静态的只读数据，任何隔离级别都适用，因为读者不会阻塞其他读者。你甚至可以将数据库切换到只读模式以减少锁开销。否则，乐观隔离级别是一个不错的选择。它们为报表查询提供事务级别或语句级别的一致性，并消除了与涉及的写入者可能发生的阻塞。

最后但同样重要的是，在一个系统中甚至在同一个事务中使用不同的隔离级别是完全正常的。你需要分析用例，并根据具体情况选择正确的事务策略。
