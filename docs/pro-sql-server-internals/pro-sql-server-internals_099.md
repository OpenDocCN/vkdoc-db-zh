# 第 28 章 ■ 系统故障排除

查询通常会处理大量数据，这导致了繁重的 I/O 活动，并刷新了缓冲池的内容。总的来说，这对其他系统的性能产生了负面影响。

`清单 28-1`展示了如何获取基于每个数据库的缓冲池使用情况信息。类似地，你可以使用`sys.dm_io_virtual_file_stats`函数获取每个数据库文件的 I/O 活动信息。我们将在本章后面更详细地讨论这个函数。

***清单 28-1.*** 基于每个数据库的缓冲池使用情况

```sql
select database_id as [DB ID], db_name(database_id) as [DB Name]
,convert(decimal(11,3),count(*) * 8 / 1024.0) as [Buffer Pool Size (MB)]
from sys.dm_os_buffer_descriptors with (nolock)
group by database_id
order by [Buffer Pool Size (MB)] desc;
```

在整合数据库时，你也应该分析安全需求。一些安全功能，例如`审计`，在服务器作用域工作，并为服务器上的所有数据库增加性能开销。`透明数据加密（TDE）`是另一个例子。尽管它是一个数据库级别的功能，但当任何一个数据库启用了`TDE`时，SQL Server 就会加密`tempdb`，这也会为其他系统引入性能开销。

作为一般规则，你应该避免将具有不同安全需求的数据库整合到同一个 SQL Server 实例上。使用多个 SQL Server 实例（可能对它们进行虚拟化）是更好的选择，即使这些实例或虚拟机运行在同一台服务器/主机上。

#### 数据库选项

每个`生产`数据库都应该禁用`自动收缩`选项。正如我们已经讨论过的，`自动收缩`会周期性地触发数据库收缩过程，这会带来不必要的 I/O 负载和严重的索引碎片。此外，这个操作实际上没什么用，因为进一步的数据修改和索引维护会使数据库文件再次增长。

`自动关闭`选项强制 SQL Server 在数据库没有任何连接用户时，从内存中移除所有与数据库相关的对象。正如你可以猜到的，这会导致额外的物理 I/O 和当用户重新连接到数据库时的查询编译。除了非常罕见的、访问频率极低的数据库例外，`自动关闭`设置应该被禁用。

在文件组中为易失性数据设置多个数据文件是更好的做法。这有助于避免分配映射争用，类似于`tempdb`中发生的情况。我们将在本章后面讨论这种争用的症状。

## 我的五分钟配置检查清单

在初始系统分析阶段，我通常会检查几个项目。这些项目使我能够快速定位一些 SQL Server 和数据库的配置问题。

*   **操作系统版本和版本。** 重大的危险信号是使用旧版（Windows Server 2003/2008）和/或 32 位版本的操作系统。
*   **SQL Server 版本和版本。** 与操作系统一样，我会检查系统是否使用 64 位 SQL Server。我还会验证服务包和`累积更新`级别。我不主张立即升级到最新的`累积更新`；但是，重要的是查明系统是否运行在受支持的产品/服务包版本上，以及该版本的已知问题是什么。
*   `即时文件初始化`是否启用？在大多数系统中都需要启用它。
*   使用了哪些跟踪标志？在 SQL Server 2016 之前的大多数系统中，我会启用`T1118`（禁用混合区分配）和`T2371`（使统计信息更新阈值动态化）。我还建议你启用`T4199`（启用查询优化器修补程序），尽管这可能需要回归测试和系统监控。正如你将记得的，在 SQL Server 2016 中，对于兼容级别为`130`的数据库，不需要`T2371`和`T4199`。另一个有用的跟踪标志是`T3226`，它可以防止 SQL Server 在日志中存储成功的备份信息，从而避免其大小激增。
