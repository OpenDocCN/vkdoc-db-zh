## 引言

四年前，当我刚开始着手编写《Pro SQL Server Internals》第一版时，许多人问我：“你为什么决定再写一本关于这个主题的书？已经有很多关于内部原理的书出版了。”这是一个——而且事实上现在仍然是——非常合理的问题，我觉得有义务回答。

当我开始着手这个系列时，我为自己设定了两个目标。首先，我希望以最实用的方式解释 SQL Server 的工作原理，展示 SQL Server 内部原理的特定方面与你的系统行为之间的依赖关系。这或许需要一些解释。

SQL Server 社区里有个笑话：“你如何区分初级和高级数据库专业人员？只需问他们任何关于 SQL Server 的问题。初级人员会给你一个直接的答案。而高级人员总是回答：‘这要看情况。’”

尽管听起来很奇怪，但这是正确的。SQL Server 是一个非常复杂的产品，拥有大量相互依赖的组件。对于任何问题，你很少能给出直接的*是*或*否*的答案。每个决定都有其自身的优缺点，并会导致影响系统其他部分的结果。

《Pro SQL Server Internals》涵盖了“这要看情况”中的具体情况。我想为你提供足够多关于 SQL Server 如何工作的信息，并通过各种示例展示特定的数据库设计和代码模式如何影响 SQL Server 的行为。我尽量避免基于最佳实践的泛泛建议。尽管这些建议很好，并且在许多情况下都有效，但总有例外情况。我希望在读完这个系列后，你将能够识别这些例外，并为你的特定系统做出有利的决定。

我的第二个目标基于一个坚定的信念：数据库管理和开发之间的界限非常模糊。不了解 SQL Server 内部原理，就不可能成为一名成功的数据库开发人员。同样，不具备开发能力，也不可能成为一名成功的数据库管理员。


## 引言

设计高效的数据库模式并编写良好的`T-SQL`代码。这些知识有助于开发人员和管理员更好地理解彼此并进行协作，这在当今敏捷开发和多 TB 级数据库时代尤为重要。

这一信念源于我的个人经验。我从应用开发人员开始了我的 IT 职业生涯，多年来逐渐转向后端和数据库开发。在某个时候，我发现除非我理解`SQL Server`如何执行代码，否则不可能写出好的`T-SQL`代码。这一发现迫使我学习`SQL Server`内部原理，从而开启了一段新的生活：我设计、开发和调优各种数据库解决方案。我不再编写客户端应用程序；然而，我完全理解应用开发人员在处理`SQL Server`时面临的挑战。我“亲身经历过这一切”。

在转向*内部原理*世界的旅程中，我最大的挑战是寻找好的学习资料。有大量优秀的书籍；然而，它们的内容都有明确的分离。它们期望读者要么是开发人员，要么是数据库管理员——而不是两者兼备。我在本书中试图避免这种分离。显然，有些章节更面向 DBA，而另一些则更倾向于开发人员。尽管如此，我希望任何使用`SQL Server`的人都能发现这些内容很有用。

然而，你不应将`Pro SQL Server Internals`视为`SQL Server`教程。它也不是一本入门级书籍。我期望你具备使用关系型数据库的经验，最好是`SQL Server`。如果你想从本系列中获得最大收益，你需要了解 RDBMS 概念，熟悉不同类型的数据库对象，并且能够理解 SQL 代码。

正如你可能已经注意到的，本书涵盖了多个`SQL Server`版本，从`SQL Server 2005`一直到最近发布的`SQL Server 2016`。除了少数例外，我没有特别涵盖 Microsoft Azure `SQL Databases`；然而，它们基于最新的`SQL Server`代码库，并且本书的大部分内容都适用于它们。

我还需要提到，我在`SQL Server 2016` RTM 发布后不久就完成了手稿。最近的开发流程变化使微软显著提高了敏捷性，我们应该预期增强功能和改进将通过服务包甚至累积更新发布。其中一些甚至会出现在产品的先前版本中，正如我们已经看到的`SQL Server 2012 SP3`和`SQL Server 2014 SP2`。

考虑到开发的敏捷性质以及微软采用的*云优先*模式，我预计`SQL Server 2016`新功能在 RTM 版本中存在的一些限制将在未来得到解决。请查阅最新的文档，不要仅仅依赖本书作为你的信息来源。虽然与一个不断演进的产品打交道并为其撰写内容具有挑战性，但这确实是一个值得拥有的好挑战。

两年前半，当`Pro SQL Server Internals`第一版即将出版时，我极其紧张。我不知道自己是否能达成目标。我很高兴发现你们中的许多人喜欢这本书并认为它有用。我希望你们会喜欢第二版，我*主观地*认为它甚至比第一版更好。

最后，我想再次感谢你们所有的反馈、鼓励和支持——最重要的是，感谢你们对我的信任。没有你们的帮助，我根本无法完成这本书！

### 本书结构

本书在逻辑上分为八个不同的部分。尽管所有这些部分都相对独立，但我仍然鼓励你从`第一部分“表与索引”`开始阅读。这部分解释了`SQL Server`如何存储和处理数据，这是理解`SQL Server`内部原理的关键。本书的其他部分都依赖于这一理解。



本书包含以下几个部分：

## 第一部分：表与索引
涵盖 SQL Server 如何处理数据。它解释了数据库表的内部结构；讨论了 SQL Server 使用索引的方式与时机；并为你提供了关于如何设计和维护索引的基本指南。本书第二版新增了一章关于 SQL Server 2016 新特性的内容，并在其他章节中补充了一些与 SQL Server 2016 相关的额外变更。

## 第二部分：其他重要事项
概述了不同的 T-SQL 对象并概述了它们的优缺点；同时提供了用例，说明何时应该或不应该使用这些对象。它还包括一个关于数据分区的、专注于架构的长篇讨论。第二版增加了关于 `JSON` 支持和空间类型增强的内容，并对其他领域进行了几项次要改进。

## 第三部分：锁定、阻塞与并发
讨论 SQL Server 的并发模型。它解释了 SQL Server 中各种阻塞问题的根本原因，并向你展示了如何在系统中排查和解决它们。最后，本部分为你提供了一套指南，指导如何以提高系统并发性的方式来设计事务策略。这一领域在 SQL Server 2016 中没有变化；不过，我重写了其中几章以使其更完善。

## 第四部分：查询生命周期
讨论 SQL Server 中查询的优化和执行。此外，它解释了 SQL Server 如何缓存执行计划，并演示了系统中常遇到的与计划缓存相关的若干问题。与 SQL Server 并发模型类似，SQL Server 2016 在这方面变化不大；然而，我尝试在各处改进了内容。

## 第五部分：实用故障排除
概述了 SQL Server 的执行模型，并解释了你如何能够快速诊断系统并精准定位问题的根本原因。第二版新增了一章，介绍 SQL Server 2016 中一项新颖且令人兴奋的功能，称为 `查询存储`。此外，“系统故障排除”一章也已扩展并改进。

## 第六部分：事务日志内部机制
解释 SQL Server 如何处理事务日志，并为你提供了一套指南，指导如何在系统中设计备份和高可用性策略。第二版增加了关于 SQL Server 2016 和 Microsoft Azure 在这些领域改进的内容。

## 第七部分：列存储索引
概述了 `列存储索引`，它可以显著提高数据仓库解决方案的性能。SQL Server 2016 在该领域增加了很多改进，包括在操作型分析场景中使用 `列存储索引`，这些内容现已被第二版涵盖。

## 第八部分：内存 OLTP 引擎
讨论了 `内存 OLTP` 在 SQL Server 2014 和 2016 中的实现。本书描述了 SQL Server 2016 中的许多技术改进。

同样值得注意的是，本书中的大多数图表和示例都是在 SQL Server 2012-2016 的企业版中创建的，并在服务器级别禁用了并行处理，以简化最终的执行计划。在某些情况下，当你在使用不同版本 SQL Server 的环境中运行脚本时，可能会得到略有不同的结果。
