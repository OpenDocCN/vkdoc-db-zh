# 第二十八章：系统故障排除

在试用/试点阶段，应在轻负载下评估系统功能。数据库可能在试验期间被部署在次级硬件或资源不足的虚拟机上，即使在全面部署后仍保持这种状态。

`SQL Server` 是一个 I/O 密集型应用，缓慢或配置不当的 I/O 子系统常常成为性能瓶颈。一个常被忽视的重要设置是分区对齐。旧版 `Windows` 在磁盘上紧接 63 个隐藏扇区后创建分区，这导致磁盘分配单元被分散在 `RAID` 阵列的多个条带单元上。在这种配置下，向磁盘控制器发出的单个 I/O 请求可能导致多次 I/O 操作，以便从不同的 `RAID` 条带中访问数据。

幸运的是，在 `Windows Server 2008` 及更高版本中创建的分区默认已对齐。然而，当您升级操作系统或将磁盘连接到服务器时，`Windows` 不会重新对齐在旧版 `Windows` 中创建的现有分区。通过修正不正确的分区对齐，无需对系统进行任何其他更改，即可实现 20% 到 40% 的 I/O 性能提升。

`Windows` 分配单元大小也起着重要作用。大多数 `SQL Server` 实例将受益于 `64 KB` 的单元；但是，您应该考虑 `RAID` 条带大小。使用制造商推荐的 `RAID` 条带大小；但要确保 `Windows` 分配单元位于单个 `RAID` 条带上。例如，`1 MB` 的 `RAID` 条带大小与 `64 KB` 的 `Windows` 分配单元配合良好，当磁盘分区对齐时，每个条带可容纳 16 个分配单元。

`提示` 您可以在 `http://technet.microsoft.com/en-us/library/dd758814.aspx` 阅读更多关于分区对齐的信息。

最后，您需要分析网络吞吐量。网络性能取决于拓扑中最慢的链路。例如，如果 `SQL Server` 和 `SAN` 之间的路径上的某个网络交换机只有两吉比特的上行链路，那么即使拓扑中所有其他网络组件速度更快，网络吞吐量也将被限制在两吉比特。这在基于网络的存储情况下尤其重要，因为每次物理 I/O 操作都会利用网络，并且通常您希望网络吞吐量快于磁盘性能。此外，请始终牢记考虑信息在网络中传输的距离。访问远程数据会增加额外的延迟并减慢通信速度。

#### 操作系统配置

接下来，您应该查看操作系统配置。这在 32 位 `OS` 情况下尤为重要，因为进程可用的用户内存量有限。至关重要的是，您需要检查 32 位版本的 `SQL Server` 是否能够使用扩展内存，是否启用了“`使用 AWE 内存`”设置，以及 `SQL Server` 启动账户是否具有 `锁定内存页` 权限。尽管如此，32 位版本的 `SQL Server` 只能为缓冲池使用扩展内存。这限制了其他组件（如计划缓存和锁管理器）可以使用的内存量。升级到 64 位版本的 `SQL Server` 总是有益的，特别是因为从 `SQL Server 2016` 开始，`Microsoft` 已停止支持 32 位 `SQL Server` 版本。

您应该检查服务器上安装了哪些软件以及正在运行哪些进程。非必要进程会使用内存并增加服务器 `CPU` 负载。以杀毒软件为例。通过限制用户访问和撤销管理员权限来保护服务器免受病毒侵害，比在服务器上持续运行杀毒软件更好。如果公司政策要求您必须启用并运行杀毒软件，请确保系统数据库和用户数据库被排除在

```
if (condVar > someVal) {console.log("xxx")}
```


扫描范围。如果您在系统中使用了 `FILESTREAM` 和 `FILETABLES` 技术，还应该将包含这些数据的文件夹排除在扫描之外。

在服务器上本地使用开发和故障排查工具是另一个常见的错误。开发人员和数据库管理员在部署和故障排查期间，经常在服务器上运行 Management Studio、SQL Profiler 和其他工具。这些工具会减少可用于 SQL Server 的内存容量，并增加不必要的负载。只要可能，始终应远程访问 SQL Server。

此外，请检查 SQL Server 是否已虚拟化。虚拟化有助于降低 IT 成本，提高系统的可用性，并简化管理。然而，虚拟化在进行性能故障排查时增加了另一层复杂性。应与系统管理员协作或使用第三方工具，确保主机即使在来宾虚拟机中性能指标显示正常时也不会过载。

另一个与虚拟化相关的常见问题是资源过度分配。例如，可以这样配置一台主机：为所有来宾虚拟机分配的内存总量超过了主机上安装的物理内存量。这种配置会导致人为的内存压力，并给虚拟化的 SQL Server 带来性能问题。同样，您需要与系统管理员合作来处理这种情况。

## SQL Server 配置

在 SQL Server 实例上托管多个数据库是很典型的情况。数据库整合通过减少需要许可和维护的服务器数量，有助于降低 IT 成本。然而，所有这些数据库都使用相同的 SQL Server 资源池，增加了其负载，并相互影响。来自一个系统的繁重 SQL Server 工作负载可能会对其他系统的性能产生负面影响。

您可以通过在服务器范围内检查资源密集型且频繁执行的查询来分析这种情况。如果您检测到大量此类查询来自不同的数据库，您可以考虑优化所有这些查询，或者将数据库分离到不同的服务器上。我们将在本章后面讨论如何检测此类查询。

您还应该检查同一服务器上是否运行着多个 SQL Server 实例，以及它们如何相互影响。这种情况检测起来有点棘手，需要您分析来自多个实例的各种性能计数器和 DMO。在这种情况下，最常见的问题之一是多个 SQL Server 实例竞争内存，从而相互造成内存压力。根据需求和负载，为每个实例设置并微调最小和最大内存设置可能是有益的。

同样值得注意的是，各种 Microsoft 和第三方产品通常会在您不知情的情况下安装独立的 SQL Server 实例。在非专用服务器上，请务必检查是否存在这种情况。
