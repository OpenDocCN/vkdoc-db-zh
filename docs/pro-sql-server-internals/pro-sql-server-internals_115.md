# 第 28 章 系统故障排除

可以执行某些操作。例如，`DAC` 不支持并行查询执行或备份/恢复功能。它是为故障排除而设计的，你应该仅为此目的使用 `DAC`。

我们已经讨论过工作线程饥饿是 `SQL Server` 可能变得无响应的一个原因。另一种可能性是*失控查询*，它们消耗了服务器上的大部分资源。你可以通过 `sys.dm_exec_requests` 视图检测此类查询，如代码清单 28-17 所示。

***代码清单 28-17.*** 检测失控查询

```sql
select top 10
er.session_id, er.start_time, er.cpu_time, er.status, er.command, er.blocking_session_id
,er.wait_time, er.wait_type, er.last_wait_type, er.logical_reads
,substring(qt.text, (er.statement_start_offset/2)+1,
(( case er.statement_end_offset
when -1 then datalength(qt.text)
else er.statement_end_offset
end - er.statement_start_offset)/2)+1) as SQL
from
sys.dm_exec_requests er with (nolock)
cross apply sys.dm_exec_sql_text(er.sql_handle) qt
order by cpu_time desc
```

你可以使用 `KILL` 命令终止一个包含失控查询的会话。然而，你应该谨慎行事，并分析该会话正在做什么。当你终止会话时，`SQL Server` 会回滚活动会话事务，在大量数据修改的情况下，这可能会非常耗时且消耗资源。完全有可能，允许会话完成任务是更快、更好的选择。

你还可以考虑使用 `资源调控器` 来防止任务消耗所有 `SQL Server` 资源。如果一台服务器托管属于多个系统的多个数据库，这可能特别有用。你可以将不同系统的连接分离到不同的资源池中，配置方式要确保为每个系统都留出一些可用资源。

## 使用基准线工作

正如你已经注意到的，我在本章中经常提到*基准线*。创建基准线是任何数据库和 IT 专业人员的必备任务。它使你能够积极主动，在问题变得明显并影响系统健康与性能之前的早期阶段就发现它们。

许多性能计数器和指标本身用途非常有限。其中一些具有指示问题的阈值或*不良*值；然而，*良好*值并不总是能保证系统是健康的。观察动态变化和趋势，监控值的变化总是有益的。

考虑*页面预期寿命*计数器。对于拥有 64 GB 内存的服务器来说，10,000 的值是非常健康的。然而，如果上周它是 50,000，那就表明有情况发生了变化。也许上次部署删除了一些索引或引入了未优化的查询，从而引发了沉重的 I/O 负载。随时间监控页面预期寿命值使你能够积极主动，并在问题开始影响系统的其他部分之前启动调查和处理。

另一个很好的例子是 I/O 子系统性能。每个 I/O 子系统都有某个临界点，超过该点后性能会随着负载的增加而呈指数级下降。在初始部署之前确定极限，并监控 I/O 负载随时间的变化，确保仍有增长空间，这总是有益的。基准线将有助于你进行监控和分析。

■ **提示** 你可以在初始部署之前使用 `DiskSpd` 实用程序对 I/O 子系统进行压力测试。你可以从 [`gallery.technet.microsoft.com/DiskSpd-a-robust-storage-6cd2f223`](https://gallery.technet.microsoft.com/DiskSpd-a-robust-storage-6cd2f223) 下载它。

市场上有很多工具可以帮助你自动化基准线的创建和监控。然而，你也可以轻松地手动实现，通过使用各种数据管理对象和通过 `sys.dm_os_`


`performance_counters`（性能计数器）视图。我们已经讨论了其中相当多的计数器，显然，你可以根据需要扩展其他信息。

捕获系统工作负载的信息至关重要，这包括连接数、批处理数、每秒事务数、数据库大小以及其他类似指标。这将帮助你分析趋势，将工作负载与系统负载关联起来，并在需要时进行容量分析。

捕获系统关键代码部分性能信息也非常有益。应用程序开发者可以收集并持久化最关键查询和/或存储过程的响应时间，这将使你能够监控趋势，确保关键代码始终表现令人满意。

## 第 28 章 ■ 系统故障排除

最后，创建基线是系统故障排除中非常有用的第一步。它帮助你评估是否达到了预期的结果，然后你可以向管理层或客户进行展示。

#### 总结

数据库并非在真空环境中运行。它们是一个大型生态系统的一部分，该系统包含各种硬件和软件组件。客户端应用程序的响应迟缓和无法响应，不一定与数据库或`SQL Server`相关。问题的根本原因可能存在于系统中的任何地方，从硬件配置错误到不正确的应用程序代码。

在故障排除过程中，检查整个系统基础设施作为初始步骤非常重要。这包括硬件的性能特征、网络拓扑和吞吐量、操作系统和`SQL Server`配置、进程以及服务器上运行的数据库。

`SQL Server`由几个主要组件组成，包括协议层、查询处理器、存储引擎、实用工具和`SQL Server 操作系统 (SQLOS)`。`SQLOS`是 Windows 与所有其他`SQL Server`组件之间的层，负责调度、资源管理以及其他一些低级任务。

`SQLOS`创建的调度器数量等于系统中的逻辑处理器数量。每个调度器负责管理一组执行工作的工作者。在执行期间，每个任务被分配给一个或多个工作者。

任务在执行期间主要保持三种状态之一：`running`（正在调度器上执行）、`runnable`（等待调度器执行）和`suspended`（等待资源）。`SQL Server`跟踪不同类型等待的累计等待时间，并将此信息提供给用户。等待统计信息分析是一种常见的性能故障排除技术，它分析主要的系统等待类型并消除等待的根本原因。

通过收集和监控系统中的各种性能和负载指标来创建基线至关重要。基线帮助你在问题开始影响用户之前，主动在早期阶段检测和解决问题。它显示了系统行为和负载如何随时间变化，这有助于容量分析并防止系统规模超出硬件能力的情况。

## 第 29 章 查询存储

`查询存储`是`SQL Server 2016`的一个新组件，用于收集系统中查询的执行计划和运行时统计信息。它帮助你检测次优查询，展示执行计划如何随时间演变，并允许你强制使用特定计划以解决参数嗅探相关问题。

本章概述了`查询存储`，解释了它如何集成到查询处理流水线中，并演示了如何在系统监控和性能故障排除期间使用它。
