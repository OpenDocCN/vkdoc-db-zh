这种比较是必须的，因为计算哈希值的算法并不能保证各个键的哈希值是唯一的，这会导致多个键生成相同哈希值的情况，即*哈希冲突*。尽管由于哈希冲突可能带来额外比较操作的开销，但这种情况相对较少。

清单 25-9 展示了内连接哈希连接的算法。

***清单 25-9.*** 内连接哈希连接算法
```
/* 构建阶段 */
for each row R1 in input I1
begin
    calculate hash value on R1 join key
    insert hash value to appropriate bucket in hash table
end

/* 探测阶段 */
for each row R2 in input I2
begin
    calculate hash value on R2 join key
    for each row R1 in hash table bucket
        if R1 joins with R2
            return join (R1, R2)
end
```

如你所料，哈希连接需要内存来存储哈希表。哈希连接的性能在很大程度上取决于内存授予估算是否正确。当内存估算不正确时，哈希连接会将一些哈希表桶存储在 `tempdb` 中，这会大大降低操作符的性能。当这种情况发生时，SQL Server 会跟踪桶的位置：是在内存中还是在磁盘上。对于来自第二个输入的每一行，它都会检查哈希桶的位置。如果在内存中，SQL Server 会立即处理该行。否则，它会将该行存储在 `tempdb` 的另一个内部临时表中。

第一次处理完成后，SQL Server 会丢弃内存中的桶，用磁盘上的桶替换它们，并对存储在内部临时表中来自第二个输入的所有剩余行重复探测阶段。如果仍然没有足够的内存来容纳所有的哈希桶，其中一些桶会再次溢出到磁盘上。

这种情况发生的次数称为*递归级别*。SQL Server 会跟踪这个级别，并最终切换到一个特殊的*备用*算法，该算法效率较低，但保证在某个时刻能够完成。

■ **提示** 你可以使用 SQL 跟踪和扩展事件中的 `Hash Warnings` 来监控哈希表溢出到 `tempdb` 的情况。

与合并连接类似，哈希连接要求连接条件中至少有一个相等谓词。

## 连接类型比较

通常，连接操作符的选择属于“视情况而定”的范畴。每种连接类型都有其自身的优缺点，这使得它适用于某些场景，而不适用于其他场景。

表 25-2 比较了不同场景下的各种连接类型。

***表 25-2.*** 连接类型比较

| 属性 | 嵌套循环连接 | 合并连接 | 哈希连接 |
|-----------|------------------|------------|-----------|
| 最佳用例 | 小型输入。内层表在连接键上有索引时更优。 | 中大型输入，在索引键上已排序。 | 中大型输入。 |
| 要求输入已排序 | 否 | 是 | 否 |
| 要求相等谓词 | 否 | 是 | 是 |
| 阻塞操作符 | 否 | 否 | 是（仅构建阶段） |
| 使用内存 | 否 | 否 | 是 |
| 使用 tempdb | 否 | 否（多对多连接除外） | 是，在溢出情况下 |
| 保留顺序 | 是（外部输入） | 是 | 否 |

人们在进行性能调优时常犯的一个错误是严格依赖查询产生的逻辑读数量。尽管这个数字是一个很好的性能指标，但在涉及连接的情况下，它可能会产生误导。例如，完全有可能哈希连接比嵌套循环连接产生的读取次数更少。然而，它并未考虑 CPU 使用率和内存开销，或者 `tempdb` 溢出和备用算法带来的性能影响。
