# 第 28 章 ■ 系统故障排除

与其他类型的闩锁不兼容，除了保留（`KP`）闩锁。

`DT` – 销毁闩锁是销毁数据结构所必需的。例如，当 `lazy writer` 进程从缓冲池中移除一个数据页时，会获取一个销毁闩锁。这些闩锁与任何其他类型的闩锁都不兼容。

当线程无法在数据结构上获得闩锁时，它会被放入 FIFO 队列，在那里保持挂起状态，生成一种与闩锁相关的等待类型，直到可以获得闩锁。我们将在稍后讨论这些类型。

在具有 32 个或更多逻辑处理器的系统中，SQL Server 可以基于每个 CPU 对某些闩锁进行分区。这些分区的闩锁称为 `超级闩锁`，有时也称为 `子闩锁`。在这种情况下，每个逻辑 CPU 维护其自己的闩锁对象状态和等待者列表，这提高了在引用结构上获取共享（`SH`）闩锁的性能。另一方面，获取排他（`EX`）闩锁需要跨所有超级闩锁分区进行同步，因此，与常规闩锁相比开销更大。

SQL Server 根据活动情况动态地将闩锁提升为超级闩锁或从超级闩锁降级。对频繁读取的数据结构（例如，索引的根页）上的闩锁会快速提升为超级闩锁。繁重的修改操作（如页面拆分）可能会将这些超级闩锁降级回常规闩锁。

闩锁产生的等待类型可以属于三个不同的类别。此外，在每个类别中，SQL Server 会根据闩锁类型使用不同的等待类型。你可以通过等待类型名称的后缀来确定闩锁类型。例如，`PAGELATCH_EX` 等待类型表示数据页结构上的排他（`EX`）闩锁，而 `PAGELATCH_SH` 表示共享（`SH`）闩锁。

这三种等待类型类别如下：

`PAGEIOLATCH` – 表示与 I/O 相关的闩锁。SQL Server 在等待数据页从磁盘读入缓冲池时使用这些闩锁/等待类型。此类等待类型的高比例可能表明存在大量未优化的查询和/或次优的磁盘系统。我们已经在本章中介绍了如何对这些情况进行故障排除。

`PAGELATCH` – 表示与缓冲池相关的闩锁，当线程需要访问或修改缓冲池中的数据和分配映射页时发生。

`LATCH` – 所有与缓冲池无关的其他闩锁。

导致 `PAGELATCH` 等待的两种主要情况。第一种是分配映射争用，最常发生在 `tempdb` 中，或者有时发生在具有高度易变数据的用户表中。正如我们已经在本书中讨论过的，你可以通过增加 `tempdb` 和/或受影响文件组中的数据文件数量来解决此问题，并且在 SQL Server 2016 之前的版本中，可以通过启用跟踪标志 `T1118` 来防止混合区分配。

第二种情况涉及数据上的始终递增或始终递减的索引，且并发插入活动非常高。考虑这样一种情况：你有一个表，该表在标识列上有一个索引，并且每秒接受数百或数千次插入。虽然这种设计大大减少了索引碎片，但所有会话都将数据插入到相同的数据页中，获取排他的 `PAGELATCH_EX` 闩锁并相互阻塞。这种情况称为 `热区`，解决它的唯一方法是更改数据库架构并移除始终递增/始终递减的索引。

当你看到 `PAGELATCH` 等待占比较高时，你应该定位发生争用的资源。你可以监控 `sys.dm_exec_requests` 视图中的 `wait_resource` 列或 `sys.dm_os_waiting_tasks` 视图中的 `resource_description` 列，以获取相应的等待类型。这些列中的信息包括数据库 ID、文件 ID 和页号，这将使你能够识别问题的根本原因。例如，`tempdb` 中的分配映射争用通常发生在 `PFS (2:1:1)` 和 `SGAM (2:1:3)` 页上。
