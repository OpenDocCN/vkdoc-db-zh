# 事务日志内部机制

事务日志保证了数据库中数据的事务一致性。它由数据修改操作生成的*日志记录*流组成。每条日志记录都有一个唯一的、自增的*日志序列号 (LSN)*，并且它也描述了数据变更。日志记录包含有关操作和受影响行的信息；数据的旧版本和新版本；执行修改的事务等等。此外，一些内部操作，例如`CHECKPOINT`，会生成它们自己的日志记录。

每个数据页都保留了最后修改它的日志记录的 LSN。在恢复阶段，SQL Server 可以比较日志和数据页中的日志记录的 LSN，从而找出最近的更改是否已保存到数据文件中。日志记录中存储了足够的信息，以便在需要时撤销或重做操作。

SQL Server 使用*预写式日志*，这保证了在脏数据页保存到数据库之前，日志记录已写入日志文件。在第 1 章中，我提到日志记录与数据修改同步保存，而数据页在`CHECKPOINT`过程中异步保存。然而，这并不完全准确。SQL Server 将日志记录缓存在一个名为*日志缓冲区*的小型内存缓存中，一次保存多条日志记录。这有助于减少所需的物理 I/O 操作次数。

在内部，日志缓冲区由 128 个 60KB 的结构组成，称为*日志块*。SQL Server 在一次 I/O 操作中将日志块写入事务日志文件。但这并不意味着 SQL Server 会等到整个日志块都满了才写入；写入请求中的数据大小可以从 512 字节到 60 KB 不等。此外，SQL Server 在队列中可以有多个未完成的日志写入请求。允许的最大请求数取决于 SQL Server 版本。

不幸的是，SQL Server 文档有些混乱，经常将日志块称为日志缓冲区，指出每个数据库有很多个。最终，重要的是每个数据库在将日志记录刷新到磁盘（批次大小最多 60 KB）之前，都会在内存中缓存它们。
