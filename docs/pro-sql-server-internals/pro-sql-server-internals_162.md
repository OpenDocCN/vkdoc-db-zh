# 高可用性技术

| 特性 | 同步 | 故障转移时间 | 性能开销 |
| :--- | :--- | :--- | :--- |
| 丢失 | 无 | 最佳情况：分钟级（实例中所有数据库的崩溃恢复） | 无开销 |
| 延迟 | 无 | 不适用 | 无开销 |
| 存储 | 无 | 秒级（单个数据库的崩溃恢复） | 同步提交的开销 |
| 日志备份 | 无 | 秒级（AG 中所有数据库的崩溃恢复） | 同步提交的开销 |
| 镜像 | 是 | 不适用 | 对事务日志的额外负载 |
| 单点故障 | 无 | 自动 | 手动 |
| 故障转移 | 自动 | 手动 | 自动（需要见证） |

显然，您并不局限于使用单一的高可用技术。结合使用多种技术通常是有益的，将它们一起使用以防范不同类型的故障。

## 技术组合

例如，如果由于 SQL Server 版本或版本不兼容而无法使用 `AlwaysOn Availability Group`，您可以将故障转移集群与数据库镜像或日志传送结合使用。故障转移集群可以保护您免受服务器故障的影响，而第二种技术可以保护您免受存储系统故障的影响。

## 数据零丢失选项

在数据丢失绝对不允许的情况下，选择仅限于数据库镜像或采用同步提交的 `AlwaysOn Availability Groups`。尽管故障转移集群使用数据库的单个副本，因此您不会因为复制（或同步）延迟而丢失数据，但它并不能防止存储故障。不幸的是，在某些极端情况下，同步提交可能会引入不可接受的延迟。

这是一个需要与利益相关者合作并达成妥协的情况示例。例如，在某些情况下，拥有一个故障转移集群，数据存储在高度冗余的磁盘阵列上，并通过异步提交到另一台服务器，可能就 *足够好了*。

## 保护单元

保护单元是另一个需要考虑的非常重要的因素。如果 `AlwaysOn Availability Groups` 不可用，同步数据库镜像可能是一个很好的选择，它能保证零数据丢失且没有单点故障。然而，它在单个数据库范围内工作，这对于必须驻留在同一服务器上的多个数据库组成的系统来说可能存在问题。故障转移集群是除了 `AlwaysOn Availability Group` 之外，唯一能保证多个数据库始终一起故障转移的选项。

## 镜像的变通方案

您仍然可以在此类场景中使用数据库镜像，通过实现一个例程来监控主服务器数据库位置，并在需要时故障转移数据库。一种可能的实现是创建一个 `SQL Agent` 作业，该作业每分钟运行一次，并查询 `sys.databases` 视图中组内某个数据库的 `State` 或 `State_Desc` 列。当该作业检测到数据库处于 `RESTORING` 状态（这意味着它已被故障转移到不同的服务器）时，它可以故障转移组中的其他数据库。

## 测试与验证

在生产环境实施后，测试您的高可用性策略并执行故障转移是极其重要的。一切第一次就完美运行的情况极为罕见。您可能会遇到安全问题、应用程序连接字符串中的错误设置、服务器上缺少对象，以及许多其他阻止系统在故障转移后按预期工作的问题。尽管测试故障转移过程可能导致系统停机，但最好是在所有人员待命的情况下进行有计划的停机，而不是在发生未计划的灾难后系统无法工作的情况。

最后，您应该定期重新评估和测试您的高可用性与灾难恢复策略。数据库规模和活动量的增长可能会使您的 `HA` 实现失效，从而无法满足 `RPO` 和 `RTO` 要求。当备用（辅助）服务器的性能低于主服务器时，这一点尤为重要。在这种情况下，系统在故障转移后很可能无法跟上负载。
