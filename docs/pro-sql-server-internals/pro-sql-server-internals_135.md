# 第 30 章 ■ 事务日志内部原理

## 概述

SQL Server 使用事务日志来存储对数据库所做所有数据修改的信息。它使 SQL Server 能够保持数据库的事务一致性，即使在发生意外关闭或崩溃的情况下也是如此。

SQL Server 采用**预写日志记录**机制，这保证了日志记录总是先被保存到日志文件中，然后才会将更新后的数据页保存到数据文件。SQL Server 使用一个小型缓冲区在内存中缓存日志记录，并在需要时一次性保存所有记录。

事务日志是一个**循环文件**，其内部由多个**虚拟日志文件**组成。每个虚拟日志文件可以是**活动**或**非活动**状态。事务日志截断会将一些 VLF 标记为非活动状态，使其可以被重用。在 `SIMPLE` 恢复模式下，SQL Server 在 `CHECKPOINT` 时截断事务日志。在 `FULL` 和 `BULK LOGGED` 恢复模式下，SQL Server 在日志备份期间截断事务日志。

有多种问题可能阻止事务日志截断。最常见的原因是在 `FULL` 和 `BULK LOGGED` 恢复模式下缺少日志备份，或是长时间运行的未提交事务。你可以通过分析 `sys.databases` 视图中的 `log_reuse_wait_desc` 列来检查是什么阻止了日志截断。

你应当避免事务日志拥有过多或过少的 VLF。这两种情况都会对系统性能产生负面影响。对于需要大型事务日志文件的数据库，你可以按 4,000 MB 的块来预分配事务日志，这样会产生大约 16 个每个约 250 MB 的 VLF。

建议你手动管理事务日志大小，以避免日志自动增长。但是，你仍应保持自动增长功能启用，以避免 `"9002: 事务日志已满"` 错误。自动增长大小应以 MB 为单位指定，而不是百分比。你需要根据系统的 I/O 性能来微调大小。

当将多个活动数据库存储在单个磁盘阵列上时，日志 I/O 访问会变为随机而非顺序。在测试你的 I/O 子系统时，你应考虑到这种行为，并选择能代表你预期生产环境负载的测试场景。

最重要的是，你应当将事务日志存储在**高度冗余**的磁盘阵列上。如果事务日志损坏，则不可能将数据库恢复到事务一致的状态。

快速的事务日志吞吐量对于良好性能至关重要，尤其是在 OLTP 系统中。你必须将事务日志存储在快速的磁盘阵列上，以最小化写入延迟。最重要的是，该阵列必须是冗余的。如果事务日志损坏，则不可能将数据库恢复到事务一致的状态。
