# 第 26 章：计划缓存

#### 计划缓存概述

*查询优化*是一个资源密集型过程，可能会在繁忙的服务器上引入显著的 CPU 负载。`SQL Server` 试图通过在进程内存的一个特殊部分（称为 `plan cache`）中缓存计划来最小化这种负载。

本章将详细讨论计划缓存，并由两部分组成。第一部分提供了计划缓存的高层次概述，并讨论了与之相关的几个问题。第二部分深入探讨了计划缓存的内部结构，并讨论了可用于计划缓存监控的各种数据管理视图（DMV）。

`SQL Server` 通过在一个称为 `plan cache` 的特殊内存区域中缓存计划，来防止查询不必要的重编译。除了预编译的参数化查询和即席查询与批处理外，它还缓存各种对象的计划，例如存储过程、触发器、用户定义函数以及其他一些对象。

`SQL Server` 缓存的不是实际的执行计划，而是一组其他与计划相关的实体，主要是 `compiled plans`（编译计划）。每次需要执行查询时，`SQL Server` 都会从编译计划生成一个实际的执行计划，与创建编译计划相比，这是一个开销较低的操作。

执行计划是运行时结构，并且对于每次查询执行都是唯一的；也就是说，如果多个会话需要执行相同的编译计划，将会生成多个执行计划，每个会话一个。

■ **注意** `SQL Server` 文档和其他资源常常忽略编译计划和执行计划之间的区别。它们经常将计划缓存称为缓存执行计划的内存区域。这完全正常，您不应对此描述感到困惑。

编译计划是为整个批处理生成的，并包含该批处理中各个语句的计划。在本章中，我通常指的是查询级别或语句级别的计划；但是，多语句批处理的计划行为方式相同。

除了编译计划外，`SQL Server` 还缓存其他结构，例如 `compiled plan stubs`（编译计划存根）、`shell queries`（外壳查询）以及另外几种。我们将在本章后面详细讨论所有这些。

缓存计划的数量并不直接影响 `SQL Server` 的性能。然而，计划缓存会使用内存，因此可能会减小 `buffer pool`（缓冲池）的大小，这反过来又会增加物理读取次数并降低系统性能。

`SQL Server` 使用不同的算法来确定在内存压力下应从缓存中移除哪些计划。对于即席查询，此选择严格基于计划被重用的频率。对于其他类型的计划，计划生成的成本也会被纳入决策考量。我们将在本章后面讨论计划缓存的内存管理。

当 `SQL Server` 怀疑当前缓存的计划不再有效时，它会重新编译查询。一种情况是计划引用的对象的架构发生更改。这可能包括在表中创建或删除列、索引、约束、触发器和统计信息。

另一种情况与过时的统计信息有关。当 `SQL Server` 从缓存中查找计划时，它会检查统计信息是否过时，如果过时，则重新编译查询。该重编译反过来会触发统计信息更新。

### 统计信息更新阈值

当表的行数少于 500 行时，SQL Server 在统计列发生 500 次更改后使统计信息过时。

当表的行数达到或超过 500 行时，如果数据库兼容级别低于 130（SQL Server 2016），SQL Server 在统计列发生 500 次以上更改（或表中总行数的 20%）后使统计信息过时。对于兼容级别为 130 或启用了跟踪标志 `T2371` 的数据库，该阈值是动态的，基于表中的总行数。

然而，对于临时表，存在另一个值为 6 次更改的阈值，这在某些情况下可能导致不必要的重编译。`KEEP PLAN` 查询提示消除了该阈值，并使得临时表的行为与常规表相同。

另一个查询提示 `KEEPFIXED PLAN` 可防止在统计信息过时时触发查询重编译。查询仅在底层表的架构发生更改或强制重编译时才会被重编译；例如，当使用 `WITH RECOMPILE` 子句调用存储过程时。

### 计划缓存与重用

计划缓存可以为相同的查询、批处理或 T-SQL 对象存储多个执行计划。一些 `SET` 选项，例如 `ANSI_NULL_DLFT_OFF`、`ANSI_NULL_DLFT_ON`、`ANSI_NULL`、`ANSI_PADDING`、`ANSI_WARNING`、`ARITHABORT`、`CONCAT_NULL_YELDS_NULL`、`DATEFIRST`、`DATEFORMAT`、`FORCEPLAN`、`DATEFORMAT`、`LANGUAGE`、`NO_BROWSETABLE`、`NUMERIC_ROUNDABORT` 和 `QUOTED_IDENTIFIER` 会影响计划重用。使用一组 `SET` 选项生成的计划不能被使用不同 `SET` 选项集的会话重用。

不幸的是，不同的客户端库和开发环境默认使用不同的 `SET` 选项。例如，默认情况下，`ARITHABORT` 在 ADO.Net 中为 OFF，而在 Management Studio 中为 ON。当排查客户端应用程序提交的低效查询时请记住这一点。你在 Management Studio 中运行这些查询时可能会得到不同的执行计划。当你的数据库与使用不同语言开发的多个客户端应用程序协同工作时，你应该考虑在建立与 SQL Server 的连接后，在会话级别以相同方式指定 `SET` 选项。

> **提示**：你可以通过 Management Studio 中“工具”菜单的“选项”菜单项，更改 Management Studio 中运行的查询的默认 `SET` 选项以匹配客户端应用程序。

导致缓存中出现重复计划的另一个常见原因是使用了未限定的对象名称（未指定对象的架构）。在这种情况下，SQL Server 根据数据库用户的默认架构来解析对象，因此，像 `SELECT * FROM Orders` 这样的语句对于不同的用户可能引用完全不同的表，从而阻止了计划重用。相比之下，`SELECT * FROM Sales.Orders` 无论用户的默认数据库架构如何，始终引用同一个表。

> **重要提示**：引用表和存储过程时务必指定架构。这可以减小计划缓存的大小并加快编译过程。

### 其他缓存限制

最后，如果批处理或对象包含大于 8 KB 的字符串字面量，SQL Server 将不会缓存其计划。例如，当 `WHERE` 子句中使用的常量超过 8,192 个字符时，以下查询的计划将不会被缓存：
```
SELECT * FROM Table1 WHERE Col='<insert more than 8,192 characters here>'
```
