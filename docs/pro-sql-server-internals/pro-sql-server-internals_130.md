事务日志截断并不会减小磁盘上日志文件的大小。截断意味着事务日志的部分（一个或多个 VLF）被标记为不活动并准备重用。它清理了日志的内部空间，同时保持日志文件大小不变。

事务日志是一个可循环写入的文件。当逻辑日志文件的末尾到达物理文件的末尾时，日志会循环回开头，如图 30-9 所示。

**图 30-9.** 事务日志是一个可循环写入的文件

每次日志增长时，SQL Server 都会创建新的虚拟日志文件。VLF 的数量取决于新分配的空间大小和 SQL Server 版本。SQL Server 2014 之前的分配算法如表 30-1 所示。

**表 30-1.** 分配大小与创建的 VLF 数量（SQL Server 2014 之前）

| 分配大小 | 创建的 VLF 数量 |
| :--- | :--- |
| < 64 MB | 4 个 VLF |
| 64 MB – 1 GB | 8 个 VLF |
| > 1 GB | 16 个 VLF |

在 SQL Server 2014 和 2016 中，算法略有变化。它会分析日志文件当前的大小，如果增长量小于当前日志大小的 1/8，则只生成一个 VLF 文件。否则，它会使用旧的算法。

你可以使用 `DBCC LOGINFO` 命令来检查虚拟日志文件。图 30-10 展示了在某个 SQL Server 实例上对 master 数据库运行此命令的输出。它显示该数据库有一个物理日志文件（`FileId = 2`）和三个虚拟日志文件。其他列指示以下信息：

- `Status` 是 VLF 的状态。值 `0` 和 `2` 分别表示不活动和活动的 VLF。
- `FileSize` 是 VLF 的大小（以字节为单位）。
- `StartOffset` 是 VLF 在文件中的起始偏移量。
- `CreateLSN` 是创建 VLF 时的 LSN（日志序列号）。`零` 表示该 VLF 是在数据库创建时生成的。
- `FSeqNo` 是 VLF 的使用顺序。具有最高 `FSeqNo` 的 VLF 是当前日志记录写入的文件。
- `Parity` 可以是两个可能值之一：`64` 和 `128`。每次重用 VLF 时，SQL Server 会切换奇偶校验值。SQL Server 使用奇偶校验值在崩溃恢复期间检测处理日志记录的停止位置。

## 数据库恢复模型

有三种影响事务日志管理和截断行为的数据库恢复模型：`SIMPLE`（简单）、`FULL`（完整）和 `BULK_LOGGED`（大容量日志）。尽管无论恢复模型如何，SQL Server 都会记录足够的信息以回滚事务和/或执行崩溃恢复，但这些模型控制着日志何时被截断以及 VLF 何时变为不活动状态。你无法访问并重做来自日志不活动部分的任何操作，因此如果数据文件不可用，截断将影响潜在的数据丢失量。

这里需要再次提及，事务日志截断并不会减小日志文件的大小，而是将 VLF 标记为不活动并准备重用。

在 `SIMPLE` 恢复模型中，SQL Server 在 `CHECKPOINT`（检查点）时截断事务日志。假设你有一个系统，其中有三个活动的 VLF，如图 30-11 所示。最旧的活动 LSN 位于 `VLF4` 中。因此，如果发生事务回滚，SQL Server 可能需要访问来自 `VLF4` 和 `VLF5` 的日志记录，这要求 SQL Server 保持 `VLF4` 和 `VLF5` 处于活动状态。

`VLF3` 中虽然没有来自活动事务的日志记录，但缓冲池中的一些脏数据页可能对应存储在那里的日志记录。在崩溃恢复的情况下，SQL Server 需要访问这些记录以重做更改；因此，`VLF3` 也应保持活动状态。
