# 第 25 章 ■ 查询优化与执行

## 图 25-6. 基于行的模型：获取输出中的第一行

让我们看看 SQL Server 如何基于逐个运算符来执行这样的查询。作为执行计划中的父运算符，`SELECT` 运算符调用 `Top` 运算符的 `GetRow()` 方法。

`Top` 运算符则调用 `nested loop join`（嵌套循环联接）的 `GetRow()` 方法。

如您所知，一个联接需要从两个不同的来源获取数据才能产生输出。作为第一步，它调用 `Sort`（排序）运算符的 `GetRow()` 方法。为了进行排序，SQL Server 首先需要读取所有行；因此，`Sort` 运算符多次调用 `Clustered Index Scan`（聚集索引扫描）运算符的 `GetRow()` 方法，以累积结果。`Scan`（扫描）运算符是执行计划树中最低层的运算符，每次调用从 `dbo.Customers` 表返回一行。为简洁起见，图 25-6 仅显示了两次 `GetRow()` 调用。

当 `dbo.Customers` 表的所有数据都被读取后，`Sort` 运算符执行排序操作，并将第一行返回给 `Join`（联接）运算符。随后，`Join` 运算符调用 `dbo.Addresses` 表上 `Clustered Index Seek`（聚集索引查找）运算符的 `GetRow()` 方法。如果存在匹配项，`Join` 运算符将来自两个输入的数据拼接起来，并将结果行传递给 `Top` 运算符，后者继而将其传递给 `SELECT`。

`SELECT` 运算符将一行返回给客户端，并再次调用 `Top` 运算符的 `GetRow()` 方法来请求下一行。此过程会重复，直到选中了前十行。值得一提的是，各运算符保持了它们的状态，`Sort` 运算符保留了已排序的数据，可以返回所有后续行，而无需再次访问 `Clustered Index Scan` 运算符，如图 25-7 所示。
