系统中有两个与事务日志相关的等待类型需要监控。`WRITELOG`等待发生在 SQL Server 将日志块刷新到磁盘以持久化时。过多的此类等待表明日志驱动器没有足够的吞吐量来处理日志生成速率。

`LOGBUFFER`等待发生在 SQL Server 等待日志缓冲区中的空间来保存记录时。尽管在数据库启动后 SQL Server 构建日志缓冲区时可能短暂发生此类等待，但通常这些等待表明日志缓冲区已满，SQL Server 正在等待记录刷新到磁盘，这并行地生成了`WRITELOG`等待。

当系统中出现这些等待时，您应该分析事务日志驱动器的性能和吞吐量。在分析期间，您也可以查看来自`sys.dm_io_virtual_file_stats`和磁盘相关性能计数器的数据。

您可以通过减少生成的日志记录和写操作来提高事务日志吞吐量。在许多情况下，这可以通过改进客户端应用程序中的事务管理来实现。在隐式单个事务中进行大量数据修改会产生大量日志记录，并迫使 SQL Server 在每次提交时刷新日志缓冲区。将代码修改为在单个显式事务中执行数据修改可以解决此问题。

或者，如果您使用的是 SQL Server 2014-2016 并且可以容忍少量数据丢失，您可以切换到事务的延迟持久性，我们接下来将讨论这一点。

## 延迟持久性

`延迟持久性`，也称为`lazy commit`，在 SQL Server 2014 中引入。如前所述，默认情况下，提交操作是同步的。SQL Server 在提交时刷新日志缓冲区的内容，将日志记录持久化到日志文件中，并且只有在提交记录写入磁盘后才会向客户端发送确认。延迟持久性通过使提交操作异步来改变此行为。客户端会立即收到事务已提交的确认，而无需等待提交记录持久化到磁盘。提交记录保留在日志缓冲区中，直到其内容被刷新，这发生在以下情况之一：

- 日志块已满。
- 同一数据库中的一个完全持久性事务被提交。该事务的提交记录会将日志块的内容刷新到磁盘。
- 发生`CHECKPOINT`操作。
- `sp_flush_log`存储过程成功完成。

如果 SQL Server 在提交记录持久化之前崩溃，则在恢复时，来自该事务的数据修改将被回滚，就好像该事务从未提交过一样。然而，其他事务能够在该事务提交后到 SQL Server 崩溃之间的时间段内看到该事务所做的数据修改。

在常规 SQL Server 关闭时也可能发生数据丢失。尽管 SQL Server 会尝试在关闭时刷新日志缓冲区，但无法保证此操作会成功。
