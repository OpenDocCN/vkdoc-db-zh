# 第六部分

## 事务日志内部机制

### 第 30 章

#### 事务日志内部原理

如你所知，SQL Server 中的每个数据库除了数据文件外，还有一个或多个事务日志文件。事务日志存储了数据库中所有更改的信息，并允许 SQL Server 在发生意外关闭或崩溃时将数据库恢复到事务一致的状态。

在本章中，我们将探讨事务日志的内部结构，讨论 SQL Server 如何记录数据修改，并回顾其如何执行数据库崩溃恢复。我们还将介绍如何诊断事务日志的过度增长，并讨论一些与日志管理和 I/O 文件放置相关的最佳实践。

#### 数据修改、日志记录与恢复

SQL Server 始终将数据库保持在事务一致的状态。事务中进行的数据修改必须全部提交或全部回滚。SQL Server 绝不允许仅应用未提交事务的部分更改，从而导致数据在事务上不一致。

即使在 SQL Server 意外关闭时也是如此。每次 SQL Server 重新启动时，它都会对实例中的每个数据库运行恢复过程。SQL Server 会回滚（`undo`）所有未提交事务所做的更改，并重新应用（`redo`）所有已提交事务所做的更改（如果在关闭或崩溃时尚未保存到数据文件中）。

从备份还原数据库时也会发生相同的过程。无法保证在运行备份时所有事务都已完成。因此，SQL Server 需要在还原过程的最后一步恢复数据库。

#### 总结

查询存储是 SQL Server 2016 的“飞行数据记录器”，它捕获系统中查询的执行计划和统计信息。它完全集成到查询处理管道中，不依赖于计划缓存。

在内部，查询存储由两个存储组成：计划存储（包含有关执行计划的信息）和运行时统计信息存储（收集按特定时间间隔聚合的运行时执行统计信息）。这两个存储都包含内存缓存和磁盘表。新收集的数据存储在内存中，并根据计划刷新到磁盘。查询存储的磁盘表存储在主文件组中。

当你需要解决与参数嗅探相关的性能问题时，查询存储非常有用。它向你展示计划如何随时间演变，并允许你强制特定的执行计划应用于查询。你可以通过 Management Studio 中提供的一组交互式报告或通过 T-SQL 中的一组数据库视图来使用查询存储。

查询存储不应给系统带来明显的性能开销。你可以通过一组性能计数器和扩展事件来监控其影响。你还应防止查询存储达到最大大小并变为只读，如果自动清理任务被禁用，这种情况可能发生。

`SQL Server:Query Store` 对象中的性能计数器允许你跟踪查询存储的 CPU 使用情况和磁盘活动。

查询存储公开了大量扩展事件。其中之一 `query_store_plan_forcing_failed`，在无法应用强制计划的情况下触发。一个可能发生的情况是更改数据库名称时。SQL Server 使用三部分对象引用来保留执行计划，重命名数据库将使这些计划失效。

最后，SQL Server 2016 RTM 公开了 19 个与查询存储相关的等待类型，你可以通过名称中的 `QDS` 前缀来识别。除了 `QDS_PERSIST_TASK_MAIN_LOOP_SLEEP` 和 `QDS_ASYNC_QUEUE` 等待外，系统中不应出现大量这些等待。这些等待是正常的，在等待统计分析期间，你应将它们作为非必要等待过滤掉。
