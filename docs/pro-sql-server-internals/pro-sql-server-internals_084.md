# 第 26 章 ■ 计划缓存

对于参数化和即席查询，`object_id` 是查询或批处理的内部哈希值。完全有可能一个存储桶存储了同一对象或查询的多个计划。正如我们已经讨论过的，不同的 `SET` 选项、数据库用户以及许多其他因素都会阻止计划重用。`SQL Server` 在缓存中查找正确计划时会比较多个计划属性。我们将在本章后面讨论如何分析计划属性。

为多语句批处理缓存的编译计划基本上是各个语句级计划的数组。当批处理中的某个语句需要重新编译时，`SQL Server` 会重新编译该单个语句，而不是整个批处理。

`SQL Server` 将缓存的批处理计划视为一个单元。整个批处理必须与产生缓存计划的原始批处理逐字符匹配，该计划才能被重用。`SQL Server` 为整个批处理从编译计划生成执行计划。

计划缓存可使用的内存量取决于所使用的 `SQL Server` 版本（参见 表 26-1）。

*可见内存* 在 `SQL Server` 的 32 位和 64 位实例中是不同的。`SQL Server` 的 32 位实例最多有 2 GB 或 3 GB 的可见内存，具体取决于 `boot.ini` 文件中是否存在 `/3GB` 开关。即使正在使用 `AWE`（地址窗口扩展）内存，也只能将 4 GB 以上的内存用于缓冲池。在 `SQL Server` 的 64 位实例上不存在此类限制。

在内存压力下，`SQL Server` 开始从缓存中移除计划。有两种内存压力：`local`（本地）和 `global`（全局）。当某个缓存存储增长得过大并开始使用过多的 `SQL Server` 进程内存时，会发生本地内存压力。当 `Windows` 强制 `SQL Server` 减少其物理内存使用量，或者所有缓存存储的组合大小达到计划缓存压力限制的 80% 时，会发生全局内存压力。

当某个缓存存储开始使用过多内存时，会触发本地内存压力。在 `SQL Server 2005-2008R2` 中，单页和多页分配被分开处理，当缓存存储在单页分配中达到计划缓存压力限制的 75% 或在多页分配中达到 50% 时，会发生内存压力。在 `SQL Server 2012` 及更高版本中，只有一个内存分配器，称为 `any-size page allocator`（任意大小页面分配器），当缓存存储增长到计划缓存压力限制的 62.5% 时，会触发内存压力。

本地内存压力也可能基于 `SQL` 和 `Object Plan` 缓存存储中的计划数量来触发。该数字大约是哈希表大小的四倍，在 32 位和 64 位实例上分别为 40,000 或 160,000 个计划。

本地和全局内存压力都使用称为 `eviction policy`（驱逐策略）的算法从缓存中移除计划，该算法基于计划成本。对于即席计划，成本从零开始，并在每次计划重用时增加一。其他类型的计划衡量生成它们所需资源的成本。它基于 `I/O`、内存和上下文切换，单位称为 `ticks`（滴答），如下所示：

`I/O`：每次 `I/O` 操作成本为 1 tick，最大为 19。

`Memory`：每 16 页内存成本为 1 tick，最大为 4。

`Context Switches`：每次切换成本为 1 tick，最大为 8。
