# 第 28 章

## 系统故障排除

事情并不总是按预期发展。随着数据量和负载的增加，系统性能可能会随时间推移而下降，或者有时服务器可能变得无响应，完全停止接受任何连接。无论哪种情况，你都需要在压力和紧张的工作状态下，快速发现并修复这些问题。

在本章中，我们将讨论 SQL Server 的执行模型，并基于**等待统计分析**来探讨系统故障排除。我将向你展示如何检测系统中经常遇到的常见问题。

### 从全局视角审视

尽管本章重点讨论与数据库相关问题的故障排除，但你需要记住，数据库和 SQL Server 永远不会孤立存在。总有客户在使用客户端应用程序。这些应用程序与来自一个或多个 SQL Server 实例的单个或多个数据库进行交互。而 SQL Server 则运行在物理或虚拟硬件上，其数据存储在通常与其他客户和数据库系统共享的磁盘上。最后，所有系统组件都使用网络进行通信和基于网络的存储访问。

从客户的角度来看，大多数问题表现为一般的性能问题。客户端应用程序感觉缓慢且无响应，查询超时，在某些情况下，应用程序甚至无法连接到数据库。然而，问题的根本原因可能在任何地方。硬件可能故障或配置不当；数据库可能有低效的模式、索引或代码；SQL Server 可能过载；或者客户端应用程序可能存在错误或设计问题。

■ **重要提示** 在故障排除期间，你应该始终查看系统的所有组件，以确定问题的根本原因。

系统的性能取决于其最慢的组件。例如，如果 SQL Server 使用 SAN 存储，你应该同时查看存储子系统和网络的性能。如果网络吞吐量不足以传输数据，那么提高 SAN 性能帮助不大。通过优化网络吞吐量或通过额外的索引或数据库模式更改来减少网络流量，你可以取得更好的效果。

另一个例子是客户端数据处理，当有大量数据需要传输到客户端应用程序时。虽然通过升级网络可以提高应用程序性能，但通过将数据处理转移到 SQL 和/或应用程序服务器，从而减少通过网络传输的数据量，你可以获得更好的效果。

#### 硬件与网络
