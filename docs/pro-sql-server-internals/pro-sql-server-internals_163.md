# 第 32 章 ■ 高可用性技术

尽管高可用性与灾难恢复策略相互关联，但它们并非同一概念。

高可用性策略通过向用户透明地处理硬件或软件故障，来提升系统的可用性。灾难恢复策略则用于处理高可用性策略未能应对的情况，以及在灾难发生后需要恢复系统时。

SQL Server 故障转移群集通过使用 SQL Server 实例作为共享资源来实现群集模型，从而保护你免受服务器故障的影响。在任何给定时间，只有一个服务器/节点可以处理用户请求；然而，一个 Windows Server 故障转移群集可以托管多个 SQL Server 群集。尽管运行多个 SQL Server 故障转移群集实例是降低解决方案成本的常见做法，但你应避免出现群集没有备用被动节点，以及故障转移后多个 SQL Server 实例在同一节点上运行且性能不可接受的情况。

SQL Server 故障转移群集使用共享存储，这成为单点故障。你应将故障转移群集与其他将数据存储在不同存储设备上的高可用性技术结合使用，以最大程度地减少因存储故障导致数据丢失的可能性。

数据库镜像和 AlwaysOn 可用性组允许你通过持续通过网络发送事务日志记录，在另一台服务器上维护数据库的字节级副本。使用同步提交时，SQL Server 会等到日志记录在辅助服务器上固化后，才在主服务器上提交事务。这种方法保证了已提交事务不会丢失数据，尽管它增加了事务的额外延迟。使用异步提交时，日志记录是异步发送的，可能存在数据丢失。即使使用同步提交，如果辅助服务器离线或数据未完全同步，数据丢失也是可能的。

AlwaysOn 可用性组允许创建一个具有一台处理读写活动的主服务器和多台允许对数据库进行只读访问的辅助服务器的基础设施。AlwaysOn 可用性组应安装在 Windows Server 故障转移群集之下，尽管每个节点使用独立的存储来存放数据库。

日志传送允许通过应用连续的日志备份流，在多个辅助服务器上维护数据库的副本。它不能防止自上次备份应用以来的时段内发生数据丢失。

复制允许你跨多个数据库复制数据库中的数据子集，允许在每个位置进行读写访问。事务复制在将更改分发到订阅者方面延迟较低。然而，设置和监控复杂的复制拓扑是一项极具挑战性的任务。

设计高可用性策略是一个迭代和互动的过程，需要你与技术团队的其他成员以及利益相关者合作。你必须确保恢复时间目标和恢复点目标要求是现实的，并在项目分配的预算内可以实现。

高可用性技术的选择取决于需求和预算，以及安装的 SQL Server 版本和版本。你不局限于单一技术——结合几种技术通常是有益的。

你应考虑使用同步提交的技术的性能影响，特别是当系统有性能服务等级协议规定某些联机事务处理事务的延迟时。

在生产环境实施后，测试你的高可用性技术并执行故障转移是极其重要的。在受控环境中发现并修复任何问题，比在灾难发生后与之抗争要好得多。



你应该定期根据数据库规模和活动增长情况，重新评估已实施的高可用性解决方案，尤其是当你的备用服务器性能低于主服务器时。
