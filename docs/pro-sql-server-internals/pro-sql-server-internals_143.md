# 第 31 章 ■ 备份与恢复

#### 设计备份策略

假设你误删了希望恢复的表。这种情况下，你可以结合使用 `STOPBEFOREMARK` 和 `STANDBY` 选项执行多次恢复，并查询数据库，直到找到正确的恢复点。

或者，你可以将 `STANDBY` 选项与 `STOPAT` 结合使用，以分析特定时间点的数据库状态。

每个生产系统都有两个影响和塑造备份策略实施的要求。

第一个是 `恢复点目标`，它规定了在灾难情况下可接受的数据丢失量。第二个要求是 `恢复时间目标`，它定义了恢复过程可接受的停机时间。

`RPO` 和 `RTO` 指标通常包含为系统定义的服务级别协议中。当 `RPO` 和 `RTO` 未正式记录时，你可以通过与利益相关者面谈并收集他们期望的信息来确定它们。

非技术利益相关者在定义 `RPO` 和 `RTO` 要求时，常常抱有不切实际的期望。他们经常要求零数据丢失和零系统停机。在现实生活中，保证或实现这样的目标是不可能的。此外，非常小的 `RPO`/`RTO` 会给服务器增加额外负载，并且通常不切实际且实施成本高昂。你的职责是教育利益相关者，并与他们合作，根据业务需求定义现实的 `RPO` 和 `RTO`。

`RPO` 决定了数据库应使用的恢复模式。表 31-1 显示了在假设备份文件可用且备份链完整的前提下，不同数据库恢复模式下可能的数据丢失和恢复点。显然，如果数据和日志文件都损坏，无论使用哪种恢复模式，恢复最后一次备份都是唯一的选择。

### 表 31-1. 基于数据库恢复模式的数据丢失情况

| 恢复模式 | 描述 | 数据文件损坏 | 日志损坏 |
| :--- | :--- | :--- | :--- |
| `SIMPLE` | 不支持日志备份。自上次完整或差异备份以来的更改必须重做。 | 数据库可以恢复到上次完整或差异备份的时间点。 | |
| `FULL` | 所有操作均完整记录在事务日志中。 | 无数据丢失 | 自上次 `LOG` 备份以来的更改必须重做。 |
| `BULK LOGGED` | 大容量复制操作被最小化记录。所有其他操作均完整记录。 | 如果自上次日志备份以来未发生大容量复制操作，则无数据丢失。否则，自上次 `LOG` 备份以来的更改必须重做。 | |

在 `SIMPLE` 恢复模式下，自上次完整或差异备份以来的所有更改都必须重做。因此，对于需要保护的易失性数据，此模型并非最佳选择。

然而，当数据是静态的时，`SIMPLE` 恢复模型是完全可以接受的；例如，在数据仓库和/或报表系统中，这些系统的数据会根据某个计划刷新。你可以在每次数据刷新后执行完整数据库备份，从而使用 `SIMPLE` 恢复模型。

`SIMPLE` 恢复模型的另一个可能用例是，数据库中的数据可以轻松、快速地从其他来源重建。在这些情况下，你可以考虑使用此模型以避免事务日志维护。同样值得注意的是，处于 `SIMPLE` 恢复模型的数据库不支持依赖于事务日志扫描的功能，例如数据库镜像、`AlwaysOn 可用性组`、日志传送等。

`FULL` 和 `BULK LOGGED` 恢复模型对常规（非大容量复制）操作的日志记录方式彼此相同，并且具有相同的事务日志维护要求。尽管 `BULK LOGGED` 恢复模型通过最小化日志记录提高了大容量复制操作的性能，但在数据文件损坏的情况下，它面临数据丢失的风险。因此，你应该避免使用 `BULK LOGGED` 恢复模型。尽管如此，你仍可考虑将数据库从 `FULL` 恢复模型切换到 `BULK LOGGED` 恢复模型
