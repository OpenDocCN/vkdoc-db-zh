# 第 31 章 ■ 备份与恢复

## 数据库恢复策略

```sql
restore database RecoveryDemo with recovery;
```

在恢复数据库时，可以采取多种恢复路径。除了刚才展示的方法，你还可以使用差异备份 `D1` ，并应用日志备份 `L3` – `L7` 以及尾日志备份。作为另一种选项，你可以在还原完整备份后，仅使用日志备份，完全不使用任何差异备份。然而，恢复过程所需的时间很大程度上取决于需要重放的事务日志记录量。差异备份有助于减少所涉及的时间量，并加快恢复过程。

你应该设计一个备份策略，并找到完整的、差异的和日志备份的正确组合，以便能在 `RTO` 需求定义的时间内恢复数据库。这里的关键点是定义完整备份和差异备份的时间安排，因为日志备份的频率取决于 `RPO` 和可能的数据丢失。

■ **提示** 记得启用 `即时文件初始化` ，这可以防止在恢复的数据库创建阶段将数据文件置零。

你应该足够频繁地创建差异备份，以最小化在需要恢复时需要还原的日志备份数量和需要重放的日志记录量。然而，差异备份是累积的，你应该避免它们存储自上次完整备份以来被修改的大量数据的情况。在这种情况下，更频繁地执行完整备份会更好。

## 示例场景

举例来说，考虑一个数据库，它从外部源收集一些数据，保留最近一周的数据，并使用滑动窗口模式实现每日清除。在这种模式下，每天有七分之一的数据在变化。

假设每周进行一次完整备份，每天进行一次差异备份。如果完整备份的大小是 `1 TB`，那么差异备份将以每天 `140–150 GB` 的速度增长。在这种情况下，如果在上次完整备份后的第七天发生灾难，你需要先还原 `1 TB` 的完整备份和大约 `850 GB` 的差异备份，然后才能应用日志备份，这非常耗时且冗余。在这种情况下，每天执行一次完整备份会高效得多。

## 备份与恢复的关键考量

### 备份文件位置

备份文件的位置是影响恢复时间的另一个重要因素。通过网络复制大量数据可能非常耗时。在合适的情况下，请考虑保留备份文件的多个副本——异地、本地，甚至可能就在服务器本地。

确保磁盘上有足够的可用空间来存储备份文件，并在备份失败时实现系统告警。请记住，失败的日志备份将阻止事务日志的截断，并会迫使事务日志增长。

当快速的系统恢复至关重要时，你可以考虑将备份条带化分布在多个本地 `DAS` 驱动器上，之后将备份文件复制到其他服务器和异地位置。这将保护你免受各种类型的故障影响，并提供最佳的备份和恢复过程性能。

### 性能与压缩

I/O 子系统和网络性能通常是备份和恢复期间最大的瓶颈。备份压缩有助于减少需要通过网络传输或从磁盘读取的数据量。如果数据库未加密，并且服务器能够处理压缩带来的额外 CPU 负载，请始终使用备份压缩。

你应该记住，备份压缩会影响备份和恢复操作的持续时间。SQL Server 会花费额外的时间来压缩和解压缩数据；然而，这可以通过更小的备份文件以及因此更少的通过网络传输和/或从磁盘读取的数据量来弥补。你需要在系统中启用备份压缩后，验证是否仍能实现 `RTO`。
