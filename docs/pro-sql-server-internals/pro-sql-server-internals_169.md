如你所见，列存储索引显著降低了系统中的 I/O 负载以及查询的 CPU 时间和已用时间。这种差异在`批处理模式执行`的情况下尤为明显，与`行模式聚集索引扫描`相比，查询运行速度快了几个数量级。

`SQL Server`的每个新版本都增加了支持`批处理模式执行`的运算符和用例数量。例如，`SQL Server 2012`中的`批处理模式`支持极其有限。它不支持除内部哈希联接以外的任何联接类型；不支持标量聚合，也不支持 `union all` 运算符，还有其他相当多的限制。

在 `SQL Server 2014` 中，`批处理模式执行`支持所有联接类型和外联接，在查询优化阶段探索不同的联接顺序，并支持标量聚合和 `union all` 运算符。此外，各种运算符的执行算法也得到了改进。例如，`SQL Server 2014`中的哈希联接运算符现在可以溢出到`tempdb`而无需切换到`行模式执行`，这在`SQL Server 2012`中是不可能的。

`SQL Server 2016` 进一步改善了这种情况。它支持带有`排序`运算符的`批处理模式执行`，在某些情况下允许将字符串谓词推送到`扫描`运算符，并包含许多其他增强功能，包括能够在串行执行计划中使用`批处理模式执行`。

所有这些改进使得在数据仓库环境中升级到最新的`SQL Server`版本是值得的，特别是在`SQL Server 2012`的情况下，其`批处理模式执行`支持非常有限，并且需要进行复杂的查询重构。

#### 基于列的存储

不同版本的`SQL Server`提供多种类型的列存储索引。尽管它们的需求和行为各异，但底层都采用基于列的存储格式。我们将在后续章节中讨论不同类型的列存储索引。

需要重点提及的是，无论列存储索引的类型如何，`SQL Server`都不允许为一个表定义多个列存储索引。

## 存储格式

基于列的存储中，每个数据列都单独存储在一组称为`行组`的结构中。每个`行组`最多存储大约一百万行，准确地说，是 `2²⁰=1,048,576` 行。`SQL Server`在创建索引时会尝试完全填充`行组`，只留下最后一个`行组`部分填充。例如，如果一个表有五百万行，`SQL Server`会创建四个包含 `1,048,576` 行的`行组`和一个包含 `805,696` 行的`行组`。
