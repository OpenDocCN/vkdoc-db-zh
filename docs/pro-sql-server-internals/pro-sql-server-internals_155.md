# 第 32 章 ■ 高可用性技术

## 图 32-6. 异步提交

使用异步提交时，网络延迟不会影响主服务器的性能，尽管在主服务器崩溃时，基于发送队列中的日志记录存在数据丢失的可能性。

尽管主服务器和辅助服务器数据库是彼此逐字节的副本，但该过程在将日志记录硬化到事务日志时并不会更新数据文件。SQL Server 通过异步重放日志记录将更改应用到辅助服务器上的数据文件，无论提交模式如何。`同步提交仅保证日志记录被同步硬化到事务日志中。它不保证也不提供数据文件的同步更改。`

在辅助服务器上，SQL Server 使用称为 `重做线程` 的线程集来重放日志记录并将更改应用到数据文件。活动重做线程的数量取决于技术、SQL Server 版本、系统中的工作线程数量，以及最重要的，同步数据库的数量及其工作负载。大量的镜像或同步数据库可能会耗尽工作线程池并影响系统性能。

尚未重放的事务日志部分称为 `重做队列`。您应该同时监控主服务器上的发送队列和辅助服务器上的重做队列的大小。发送队列的大小表明在主服务器发生故障时可能的数据丢失量。重做队列的大小表明需要重放多少日志记录，从而表明在故障转移后使镜像数据库重新联机可能需要多长时间。

`SQLServer:Database Mirroring` 性能计数器提供有关数据库镜像性能以及发送和重做队列统计的信息。`SQL Server:Availability Replica` 和 `SQL Server:Database Replica` 计数器提供与 AlwaysOn 可用性组相关的信息。

您需要测试数据库维护如何影响重做队列的大小。一些操作，例如索引重建或数据库收缩，可能会生成大量的日志记录，这反过来会使重做队列变得非常大。这可能导致在发生故障转移时崩溃恢复过程过长，从而可能使您无法满足 SLA 中定义的可用性要求。

**提示** 有关数据库镜像监控的更多详情，请参见 `https://msdn.microsoft.com/en-us/library/ms190030.aspx`。有关 AlwaysOn 可用性组监控的更多信息，请访问 `https://msdn.microsoft.com/en-us/library/ff877954.aspx`。
