# 20. Oracle 架构

在本章中，我们将讨论 Oracle 数据库架构。我们将探讨构成 Oracle 数据库的内存结构、进程和文件。了解 Oracle 如何工作非常重要，否则它只是一个放在架子上的黑匣子。

考虑一辆汽车及其发动机。对许多人来说，汽车是一个黑匣子。他们启动它，然后驾驶。直到有一天，发动机无法启动。如果你完全不懂如何修理汽车引擎，你会怎么做？你会把它交给这个领域的专家——汽车修理工。他们知道如何修复这个每天载你上下班的黑匣子。本书的某些读者可能对汽车发动机了解甚多，无论什么部件损坏，他们都能轻松修复。另一些读者则介于两者之间，可能对内燃机略知一二，足以解决小问题，但不足以处理大故障。

作为一名 Oracle 数据库管理员，您的组织依赖您来修复这个他们用来存储宝贵数据的黑匣子。就像机械师需要深入了解发动机部件如何协同工作以使汽车行驶一样，DBA 也需要理解 Oracle 的各个组件如何相互配合工作。您的组织雇佣您，就是让您成为数据库的机械师。

如果您正在阅读本书，那么您可能不属于那种完全了解 Oracle 工作原理的专家，但您也不是将 Oracle 视为黑匣子的人。您很可能处于中间水平。如果是这样，那么本章就是为您准备的。毕竟，您很可能阅读本书是因为您希望从初级 DBA 或更高级别起步，一直成长为该领域的专家。

在第 13 章中，我建议您阅读 `Oracle 数据库概念指南`。Oracle 文档中的这本书提供了关于 Oracle 数据库引擎工作原理的丰富信息。请务必阅读该指南。在本章中，我将从我的视角讨论相同的话题。我还将涉及我认为对理解 Oracle 架构至关重要的领域。本章无法替代阅读 `数据库概念指南`。当您查阅 Oracle 文档时，也请一并阅读 `数据库管理员指南`，正如我在第 13 章中所推荐的那样。

随着您在职业生涯中不断使用 Oracle 数据库，您将对这一架构有越来越深入的了解。当我刚开始成为一名 Oracle DBA 时，试图理解 Oracle 如何工作是需要解决的最大难题，也是最大的乐趣所在。尽管我使用 Oracle 已经超过 20 年，但我仍在学习有关 Oracle 架构的新细节。我不仅越来越深入 Oracle 内部，而且 Oracle 在每个新版本中都会对架构做些许更改。

## 数据库 vs. 实例

在 Oracle 架构中，Oracle 数据库和 Oracle 实例之间有很大区别。它们不是一回事。作为数据库管理员，您需要理解这两者、它们的含义以及它们如何交互。

您会听到人们说类似“我无法连接到数据库”这样的话。从技术上讲，您永远无法连接到一个 Oracle 数据库。软件应用程序通常运行在另一台机器上，甚至可能是一台 Web 服务器，通过浏览器访问。应用程序无法直接访问这些文件，因此它无法连接到数据库。这就是 DBA 需要了解其受众的地方。对于最终用户来说，Oracle 只是一个黑匣子，他们会使用“数据库”这个术语。当 DBA 与最终用户交谈时，即使 DBA 知道这个术语在事实上并不准确，也应该使用相同的术语来促进交流。

用户和应用程序连接到的是 Oracle `实例`，而不是数据库。实例是启动 Oracle 软件后产生的一组进程和分配的内存。用户会话在数据库服务器上生成进程，这些进程与分配的内存以及数据库文件进行交互，正如我们将在本章中看到的那样。

当 DBA 启动 Oracle 时，他们是在启动实例。当 DBA 关闭实例时，数据库仍然存在于磁盘上。`实例` 是进程和内存的集合。`数据库` 是磁盘上的文件。当与其他 Oracle 专业人士交谈时，您需要使用正确的术语。在与其他人交谈时，可以随意使用“数据库”这个术语。



## 文件类型

对于 Oracle 数据库，共有八种不同的文件类型，每种类型详述如下：

### 数据文件
每个表空间由磁盘上的一个或多个数据文件组成。一个数据文件只能属于一个表空间，但一个表空间可以拥有多个数据文件。

### 临时文件
任何临时表空间都由一个或多个称为临时文件的特殊文件组成。Oracle 会区分用于在实例关闭后保留数据的数据文件，以及在实例启动时被清空的临时文件。了解这两种文件类型的区别后，Oracle RMAN 可以对它们进行不同的备份。RMAN 无需浪费时间备份临时文件的内容，因为当实例再次启动时，这些数据将不复存在。当 SQL 语句执行的排序或连接操作过大，超出其内存空间时，就会使用临时文件。这些工作会被拆分成多个片段，并临时存储在 TEMP 表空间中。

### 联机重做日志
当发生事务时，Oracle 需要能够重放或重做该事务，以辅助任何恢复工作。重做信息被写入联机重做日志（ORLs）。ORL 以组为单位创建。通常应至少有三个，且一般不超过五个 ORL 组。当写入重做信息时，所有信息都会写入一个 ORL 组。当该组写满后，Oracle 将执行一次 `日志切换`，将重做信息写入下一个 ORL 组。一旦最后一个 ORL 组写满，下一次日志切换将回到第一个 ORL 组，覆盖其原有内容。理想情况下，ORL 组应足够大，以便在实例事务活动高峰时段，每小时只发生三到四次日志切换。ORLs 对数据库操作至关重要。因此，ORL 组应被 `多重化`，这意味着 Oracle 会维护该文件的镜像副本。两个副本不应位于同一磁盘上。

### 归档重做日志
如上所述，ORL 组最终将被覆盖。如果数据库管理员需要将重做记录保存更长时间，则必须将数据库配置为归档日志模式。当发生日志切换时，Oracle 会将之前的 ORL 复制到一个归档日志目标位置。在第 7 章中，我们为测试数据库启用了归档日志模式。

### 撤销表空间文件
撤销表空间拥有自己的数据文件。从技术上讲，Oracle 认为撤销表空间的文件与常规数据文件无异。然而，撤销表空间中的数据在实例关闭后不会保留，这使得撤销表空间的数据文件更类似于临时文件。我之所以将这些文件与常规文件分开分类，至少是为了知道它们与常规数据文件不同。

### 控制文件
控制文件是数据库的主指针。当 Oracle 实例启动时，实例并不知道数据库文件的位置。但实例确实从必需的 `CONTROL_FILES` 初始化参数中知道控制文件的位置。实例从控制文件中读取数据文件的位置。控制文件还包含实例启动前 Oracle 可能需要的任何其他信息。控制文件还包含有关 RMAN 备份、数据库名称和唯一标识符的信息。创建生产数据库时，应创建三个控制文件，并全部放置在不同的磁盘单元上。

### 参数文件
当 Oracle 实例启动时，它首先需要找到控制文件。实例还需要知道要分配多少内存，以及受各种影响 Oracle 的参数控制的若干其他设置。这些参数存储在参数文件中。从技术上讲，参数文件不属于“数据库”，但它是一个需要了解的重要文件。

### 口令文件
每当有人尝试从不同于数据库服务器的机器连接到实例，并且他们尝试进行 `SYSDBA` 或 `SYSOPER` 连接时，就会使用这个可选文件。同样，此文件不是数据库的一部分，但你需要理解其作用。

## 内存结构

当 Oracle 实例启动时，它会在数据库服务器上分配内存。Oracle 使用的内存分为两个不同的类别：程序全局区（`PGA`）和系统全局区（`SGA`）。两者的大小调整方式不同。`PGA` 的使用是动态的，响应用户需求的变化。`SGA` 的大小通常是固定的，但如果数据库配置正确，数据库管理员可以动态更改 `SGA` 的大小。



### PGA

程序全局区是最容易理解的。用户的会话连接到 Oracle 实例。该会话需要内存来存储其会话数据和控制信息。当用户执行需要临时排序空间的操作时，PGA 中占用空间最多。任何时候会话发出包含 `ORDER BY`、`GROUP BY` 的查询，甚至连接两个或多个表时，Oracle 都需要空间来存储中间结果。Oracle 使用会话的 PGA。

程序 `全局区` 这个名称有点误导性。每个会话都会分配自己的 PGA。会话无法查看另一个会话的 PGA 内容。PGA 中的内存不共享，因此术语“全局”可能并不恰当。随着每个会话分配 PGA 内存，实例的总 PGA 会增长。当会话断开与实例的连接时，实例 PGA 总量会下降。清单 20-2 中的示例展示了如何计算 Oracle 实例的总分配 PGA。我重复了几次查询以显示 PGA 总分配量的升降情况。

### SGA

系统全局区 (`SGA`) 是在 Oracle 实例中所有会话之间全局共享的内存。这个内存结构被划分为许多不同的组件，每个组件都有不同的工作职责。在本节中，我们将讨论 `SGA` 的各个组件。


### 缓冲区缓存

缓冲区缓存将数据文件中的块副本存储在内存中，以便快速高效地检索。如果数据库引擎每次需要数据时都必须从磁盘读取，性能将非常低下。从内存中读取数据则快得多。实例启动时，缓冲区缓存初始为空。当向数据库发出 SQL 语句时，这些语句将尝试读取或修改数据库中的数据。在数据被修改或返回给用户之前，数据文件的块会被读入缓冲区缓存。其理念是，既然你的会话需要访问这些数据，那么其他会话在不久的将来可能也需要类似的访问。如果我们将数据缓存在内存中，就可以省去为后续任何请求从磁盘读取数据的昂贵操作。

在与非 Oracle 专业人士交谈时，我简单地将此结构称为“数据库的缓存”。许多人都熟悉缓存的概念，因此使用简单的术语有助于我们更有效地沟通。我也会将 SGA（系统全局区）中的其他缓存结构归入这一术语，我们很快会讨论到。在与 Oracle 数据库管理员交谈时，我会非常具体地说明我指的是 SGA 中的哪个缓存组件。

除非数据库相对较小，否则服务器可能缺乏足够的内存来缓存整个数据库的内容。缓冲区缓存需要管理其内容，以便最常请求的数据保留在内存中。Oracle 使用一种改进的“最近最少使用”算法来确定当需要为另一个请求腾出空间时，应从缓存中清除哪些数据。

在某些情况下，特别是当 Oracle 引擎认为数据只会被请求一次时，可能会跳过缓冲区缓存。数据直接从磁盘读取并传递到用户的会话，Oracle 称之为 `直接路径读取`。直接路径读取可以避免在从磁盘读取数据之前查找并可能清理出部分缓冲区缓存所需的时间。但是，如果数据被再次访问，则必须再次从磁盘读取。

该结构之所以被称为缓冲区缓存，是因为缓存中的每个空间块都被称为 `缓冲区`。缓冲区的大小等于数据库的 `DB_BLOCK_SIZE` 初始化参数。如果你的数据库块大小为 8K，那么缓存中的每个缓冲区就是内存中连续的 8K 块。缓冲区缓存就是这些块或缓冲区的集合。

如前所述，实例启动时，缓冲区缓存为空。缓存中的每个缓冲区状态都是 `未使用`。然后，一个会话发出 `SELECT` 语句从表中读取数据。块从磁盘读取并放入缓存中的缓冲区。此时，数据尚未被修改，因此缓冲区状态为 `干净`。如果会话更改了缓冲区中的数据，状态则变为 `脏`。当事务提交更改后，缓冲区仍保留在缓存中，并标记为脏。

需要注意的是，提交事务的操作并不会导致 Oracle 将脏缓冲区的内容写入磁盘。脏缓冲区仅在 Oracle 执行检查点时才会被写入数据文件。以下事件将触发检查点：

*   重做日志切换到下一组。
*   达到 `LOG_CHECKPOINT_INTERVAL` 指定的间隔。
*   超过 `LOG_CHECKPOINT_TIMEOUT` 指定的值。
*   有人执行了 `ALTER SYSTEM CHECKPOINT` 命令。

另一个需要注意的重要事项是，即使修改缓冲区的事务尚未提交，脏缓冲区也可能被写入磁盘。当缓冲区缓存存在空间压力时，就可能发生这种情况。

### 共享池

另一个重要的 SGA 组件是共享池。共享池是内存结构的集合，全部归入此类。我们可以分别考虑其子组件，但出于某种原因，Oracle 决定将它们分组到共享池中。这些子组件，也称为子池，定义如下。

### 库缓存

库缓存是 Oracle 存储所有已处理 SQL 语句的地方。当用户向实例发出 SQL 语句时，在 Oracle 开始返回数据之前会进行大量工作。必须解析该语句以确保其语法正确。接下来，Oracle 优化器必须确定高效的执行计划，以尽快返回数据。如果另一个用户发出相同的 SQL 语句，能够受益于之前的工作将非常有益。Oracle 将所有这些信息存储在库缓存中。当用户提交 SQL 语句时，Oracle 首先检查库缓存以查看该语句是否已存在。如果它在缓存中，则可以跳过所有预处理步骤。

### 数据字典缓存

缓冲区缓存存储来自应用程序表的数据。请记住，数据字典也存储在表中，由 `SYS` 用户拥有。数据字典表对于数据库操作至关重要，因此它们拥有自己的缓存，即共享池的一个子组件。数据字典缓存不保存块或缓冲区，而是保存数据行。因此，你在一些较早的 Oracle 文献中有时会看到它被称为 `行缓存`。

### 结果缓存

如果 SQL 语句返回一致的结果，DBA 可以利用结果缓存来存储该结果，而不必在每次执行 SQL 语句时重新计算结果。下次用户发出相同的 SQL 语句时，Oracle 会查看库缓存，并发现执行它的最佳方式是直接从结果缓存获取结果。结果缓存可以极大地提高这些 SQL 语句的性能。此功能最适用于基础数据不经常更改且结果集不非常大的 SQL 语句。只有极少数 SQL 语句会受益于结果缓存，但对于那些适用的语句，Oracle 需要内存，而结果缓存就是存放它们的地方。

### 重做日志缓冲区

重做日志缓冲区，也称为 `日志缓冲区`，是 SGA 的另一个组件。当事务对数据进行更改时，Oracle 必须存储信息以便能够重放或重做该事务。重做记录将被写入联机重做日志，但直接将重做记录写入磁盘将花费太多时间。事务将重做信息写入内存，即重做日志缓冲区。

重做日志缓冲区是一个循环内存结构。如果有足够多的数据写入缓冲区，最终较旧的内容将被覆盖。Oracle 需要在缓冲区填满之前将重做记录写入联机重做日志。

### 大型池

大型池是为大于其他共享池需求的内存需求而预留的空间。通常，大型池用作并行 SQL 语句和 RMAN 从属进程的缓冲区空间。

### Java 池

Java 池是 Oracle 存储在数据库 Java 虚拟机（JVM）内执行的 Java 代码的地方。

### 固定 SGA

SGA 需要一定空间用于内部管理活动。此空间大小固定且无法更改，因此得名 `固定 SGA`。但是，新的 Oracle 版本可能需要更大的固定 SGA。
