# 4. 磁盘布局

在首次安装 Oracle 数据库软件之前，我们需要规划磁盘布局策略。本章将首先讨论 Oracle 的**最佳灵活架构**（OFA），然后给出一些关于在磁盘上何处创建数据库的一般性指导原则。

正如我之前在本书所述，在学习材料的过程中，我们会看到一些最佳实践。本章充满了最佳实践。如果你正在生产环境中设置 Oracle，你会希望尽可能遵循这些最佳实践。对于简单的试验台，可能更倾向于将所有文件放在一个大磁盘驱动器上，而不必过多考虑本章内容。很多时候，尽管我们知道试验台并未达到生产就绪状态，但我们仍希望尽可能快地启动它。另外，如果你使用的是预构建的 VirtualBox 镜像，甚至是 Docker 或其他容器上的 Oracle，磁盘布局已经为你决定好了。

正如我们将在本章中讨论的，OFA 是一套关于在磁盘上放置 Oracle 文件的指导方针，总有需要偏离标准实践的时候。我去过许多站点，有的严格遵守 OFA，也有的站点根本没考虑过 OFA，更不用说在数据库服务器上实施了。无论你选择哪条路径，都要确保在整个企业内保持一致。数据库管理员在一个数据库服务器上工作，在一个位置寻找文件，然后转移到另一个数据库服务器工作时，却不得不因为文件位于完全不同的位置而到处搜寻，这是非常耗时的。将所有数据库服务器的文件保持在相同的位置，将节省时间并减少人为错误，防止数据库管理员意外删除他们认为不属于 Oracle 环境的文件或目录。在设置我的 Oracle 数据库时，我尽可能遵循 OFA 指导原则，即使是开发和测试环境也不例外。对于只有我一个用户的试验台，我通常只是随意地将文件放在磁盘上。

### 提示

保持整个企业内文件位置的一致性。

虽然 OFA 指导原则提供了磁盘上的具体位置，但你可能会想使用其他位置。对于文件位置，我建议对于个人试验台之外的任何用途，尽可能遵循 OFA 指导原则。将来可能会有某个时候，你的组织聘请 Oracle 顾问来协助你的工作。顾问很可能知道 OFA 指导原则，并期望在特定位置找到文件。如果你使用了不同的位置，顾问将浪费宝贵的时间来了解你企业的配置。由于顾问是按小时收费的，这种浪费的时间将给组织带来额外的成本。同样，未来也可能有某个时候，你的组织会雇佣另一位 DBA 加入你的团队。如果这位 DBA 有先前经验，他们很可能会熟悉 OFA 指导原则。坚持 OFA 指导原则将有助于顾问或新 DBA 在你的环境中更快上手。例如，我曾被叫去帮助解决一个 Oracle 系统的数据库问题，这个系统我以前从未见过。当时负责该系统的数据库管理员恰好不在，而且他没有遵循 OFA 指导原则。因此，我花了更长时间才找到解决该问题所需的 alert 日志和其他跟踪文件，这反过来导致解决问题的时间延长了。

## 最佳灵活架构

**最佳灵活架构**（OFA）是 Oracle 公司提供的一套指导原则，帮助我们定义在磁盘上放置 Oracle 文件的位置。这些文件包括数据库文件、Oracle 软件文件和任何跟踪文件，所有这些都将在本节中讨论。

请记住，OFA 提供的是指导原则，而非规则。你不必遵循它们。对于快速、简单地设置 Oracle（通常是个人试验台），你可以忽略这些指导原则。如果你计划将你的 Oracle 安装用于开发、测试或生产目的，那么请遵循 OFA 指导原则作为最佳实践。

### ORACLE_BASE

在我们安装任何 Oracle 软件之前，需要有一个基础位置。这个位置称为 Oracle Base，在许多 Unix 和 Linux 环境中，`$ORACLE_BASE` 环境变量被明确定义为此位置。OFA 指导原则建议 `ORACLE_BASE` 在 Unix 和 Linux 机器上位于 `/u01/app/oracle`。在 Windows 上，`ORACLE_BASE` 通常位于 `C:\oracle` 目录。如前所述，这只是一个指导原则，你可以将 `ORACLE_BASE` 放在任何其他位置，但除非另有说明，本书将假设遵循了这些指导原则。

更准确地说，OFA 指导原则建议 `ORACLE_BASE` 位于 `/*磁盘单元*/app/*所有者*`。磁盘单元通常为 `/u01`，但你也可能看到 `/u02`、`/u03` 等等。在过去，存储管理员挂载不同编号的磁盘单元是很常见的。但在当今的存储区域网络和网络文件系统环境下，`/u01` 很常见，其他则较为少见。你的存储管理员可能会使用不同的磁盘单元命名约定，如 `/disk1` 和 `/disk2`。只需要求你的存储管理员保持一致即可。在 Windows 机器上，磁盘被挂载为驱动器盘符，因此你可能会看到 `ORACLE_BASE` 在 `D:\app\oracle` 或 `E:\app\oracle`，但许多站点将其简化为 `D:\oracle` 或 `E:\oracle`。无论你使用哪种约定，都要保持一致。

Oracle Base 是 Oracle 软件及其日志文件的根目录。Oracle 数据库软件将安装在该基础目录的子目录中。Oracle 在 `$ORACLE_BASE/oraInventory` 中维护一个关于 Oracle 软件及其补丁程序的内部清单。Oracle 数据库文件也可以位于该基础目录的子目录中。

记住，在第 3 章中，我们使用 Oracle 的 VirtualBox 设置了一台 Linux 服务器。因此，本书大部分内容将讨论 Linux 环境。如果你将在 Windows 服务器上使用 Oracle，其翻译非常相似。在 Windows 机器上，环境变量名为 `%ORACLE_BASE%`，通常指向 `C:\oracle` 或 `D:\oracle`。


### ORACLE_HOME

你可以在单个服务器上运行多个 Oracle 数据库，每个数据库使用不同的 Oracle 版本。如今，OFA 最大的（虽然不是唯一的）好处是便于在同一服务器上部署不同的 Oracle 版本。借助 OFA，你可以轻松安装 Oracle 11.2.0.4、12.1.0.2 和 12.2.0.1 版本，而不会发生冲突。

Oracle 通过将每个版本独立安装在其自己的主目录（称为 `$ORACLE_HOME`，在 Windows 上为 `%ORACLE_HOME%`）中来处理它们。OFA 指南将 `$ORACLE_HOME` 放置在命名方案类似于 `$ORACLE_BASE/product/*version*/*type*` 的文件夹中，其中 `*version*` 表示已安装的 Oracle 版本，`*type*` 通常对于数据库软件是 `*dbhome_X*`，对于 Oracle 客户端软件是 `*client_X*`，对于 Oracle Grid Infrastructure 是 `*grid_X*`。以下是一些 `$ORACLE_HOME` 目录的示例：

*   `/u01/app/oracle/product/11.2.0/dbhome_1`
*   `/u01/app/oracle/product/11.2.0/dbhome_2`
*   `/u01/app/oracle/product/12.1.0/dbhome_1`
*   `/u01/app/oracle/product/12.1.0/dbhome_2`

这些示例说明了我一直以来在 OFA 指南方面遇到的一个问题，也是我稍有偏离 OFA 的地方之一。看看前两个例子。两者都显示版本 11.2.0，一个是类型 `dbhome_1`，另一个是类型 `dbhome_2`，因此我们确实可以知道两个目录都包含 11.2.0 版本的 Oracle 数据库软件。我的问题是，Oracle 数据库版本号用三个小数点表示，但 OFA 指南只使用了两个。如果我告诉你其中一个路径用于 Oracle 11.2.0.3，另一个用于 11.2.0.4，你仅通过查看目录结构无法分辨哪个是哪个。最后两个例子情况类似。哪个用于 Oracle 12.1.0.1，哪个用于 12.1.0.2？你无法分辨。

这就是我总是略微偏离指南的地方。我再添加一个小数点，并在路径中明确说明完整的 Oracle 版本。因此，我的 Oracle 主目录看起来像这样：

*   `/u01/app/oracle/product/11.2.0.3/dbhome_1`
*   `/u01/app/oracle/product/11.2.0.4/dbhome_1`
*   `/u01/app/oracle/product/12.1.0.2/dbhome_1`
*   `/u01/app/oracle/product/12.1.0.1/dbhome_1`

在目录路径中包含完整版本后，只要相同版本仅安装一次，类型始终是 `dbhome_1`。我经常省略类型，得到更简洁的路径名：

*   `/u01/app/oracle/product/11.2.0.3`
*   `/u01/app/oracle/product/11.2.0.4`
*   `/u01/app/oracle/product/12.1.0.1`
*   `/u01/app/oracle/product/12.1.0.2`

省略类型是可选的。我省略类型只是为了使 Oracle 主目录路径更短一些。你一天中会多次输入这个路径。无论你怎么做，只要在整个企业中保持一致即可。如果你在一个较大的数据库团队中工作，每个人都需要知道并理解你们组织使用的约定。

有了不同的 Oracle 主目录，我们可以轻松地为未来的升级做准备。例如，假设我想将现有数据库升级到 12.2.0.1 版本。我在 `/u01/app/oracle/product/12.2.0.1` 安装新版本，Oracle 软件就在其自己的主目录中，没有与其他版本的冲突。当我将数据库升级到新版本时，旧版本可能会变得不再使用，我可以将它们删除。

在某些情况下，组织可能有要求，某些 Oracle 软件需要由不同的用户拥有。这称为“职责分离”。在 Linux 机器上，Grid Infrastructure 由 “grid” 用户拥有，而 Oracle 数据库软件由 “oracle” 用户拥有，这种情况非常常见。对于负责集群软件（Grid Infrastructure）的团队和负责数据库软件的团队来说，凭证分离非常重要。通过让两个软件组件拥有不同的所有者，我们可以确保每个团队只能管理他们自己的组件，而无法访问另一个组件。OFA 对我们有帮助，因为每个 `$ORACLE_HOME` 目录对于该软件类型和该版本都是自包含的。

### 数据库管理文件

数据库管理员需要许多位于软件安装目录之外的文件。许多管理文件位于 `$ORACLE_BASE/admin` 中。表 4-1 显示了此位置的子目录。

**表 4-1 管理文件目录**

| 目录 | 内容 |
| --- | --- |
| `/u01/app/oracle/admin/audit` | 审计跟踪记录 |
| `/u01/app/oracle/admin/create` | 创建数据库的脚本 |
| `/u01/app/oracle/admin/dpdump` | Data Pump 文件位置 |
| `/u01/app/oracle/admin/scripts` | DBA 脚本 |

### 诊断文件

我们将在第 12 章更详细地探讨诊断文件，但现在，了解文件的位置很重要。诊断文件位于 `$ORACLE_BASE/diag` 目录中。下一级子目录是诊断文件的类型。对于我们的案例，我们关注的是数据库的诊断文件，因此我们将查看 `$ORACLE_BASE/diag/rdbms`。还存在用于 Oracle Listener、客户端软件等的诊断文件的子目录。在 `rdbms` 子目录中，我们的文件将在 `$ORACLE_BASE/diag/rdbms/*db_name*/*instance_name*` 中找到。如果你没有运行 Oracle Real Application Clusters (RAC)，实例名将与数据库名相同。位置示例可能是：

*   `$ORACLE_BASE/diag/rdbms/orcl/orcl`

在上面的示例中，我们的数据库名称是 `orcl`。
