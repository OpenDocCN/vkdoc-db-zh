# 2. 作为数据库管理员工作

在上一章，我介绍了成为数据库管理员的职业路径，以及 DBA 的一般职责和责任，包括他们作为**数据守护者**的角色。在本章中，我们将花时间讨论如何作为数据库管理员进行工作。本章的信息本质上并非技术性的，也适用于非 Oracle DBA。本章将重点介绍所有优秀数据库管理员都应具备的软技能。本章中的许多信息可以在 DBA 角色之外使用，并应用于组织中的其他职位。

## 粗鲁的数据库管理员

取决于你询问的对象，数据库管理员在 IT 世界中通常是一个备受尊敬的职业。正如我们在上一章讨论的，DBA 通常来自应用程序开发或系统管理背景。一个人大学一毕业就直接成为数据库管理员的情况比较少见。通常，DBA 是逐步晋升到这个职位，并展现了高超的技能才得以成功。

当你作为一名 DBA 工作时，请试着记住，你不仅代表你自己和你的组织，你也代表着这个职业。我一直认为数据库管理员是 IT 职业阶梯上的顶级职位之一。因此，你将与组织内部的许多人互动。如果你遵循后续章节的建议，你还将与全球 Oracle 社区的许多人互动。以积极的方式展现自己，其对你整个职业生涯的影响，将远远大于你表现出不良行为所带来的影响。遗憾的是，我遇到过太多这样的 DBA，他们总是脾气暴躁，对他人持阻挠态度——这些特质并非 DBA 独有。例如，一位 DBA 无论业务需求如何，也不给出解释，就拒绝授予特权或创建新数据库。我听过太多关于 DBA 是个混蛋、难以共事的故事。一些 DBA 持有这种态度的原因可能有很多，但请不要成为这类 DBA。如果你收到一个特权或创建新数据库的请求，并且你认为这个请求不应该被满足，与其用某种粗鲁的方式拒绝请求，更好的方法是弄清楚请求者真正需要什么，以及你如何能帮助他们完成工作。不要回应“你的请求被拒绝”，而是说“抱歉，但按照你概述的方式满足该请求会导致其他问题。我们能否换种方式，既能让你完成工作，又能解决我的顾虑？”

### 提示

不要做那个粗鲁的 DBA。要做乐于助人的那一个。

我一直有一种未经证实的观点：粗鲁的 DBA 是当今非常流行的无模式数据库兴起的次要原因之一。应用程序开发者声称，像 MongoDB 这样的数据库很棒，因为他们可以自己安装、设置并开始编码，而无需 DBA 的任何协助。我的反驳观点是，通过避开 DBA，开发者会错过数据库管理员通常会考虑的许多其他方面。其中一些方面涉及数据库安全以及经过验证的备份与恢复实施。应用程序开发者不习惯成为数据守护者，这是我们在上一章讨论过的概念。在第七章（7）中，我们将讨论 DBA 应该在备份之前就考虑恢复，这一观点是大多数应用程序开发者甚至不会考虑的。2017 年，超过 25,000 个未受保护的 MongoDB 数据库遭到了勒索软件^(¹)的攻击。这并非因为 MongoDB 是一个不安全的数据库平台。相反，这些部署使用了弱密码和默认用户名。DBA 被绕过了，安全从未被考虑，公司为此付出了代价。如果 DBA 参与了这些项目，数据库很可能会得到保护，因为一个好的 DBA 会考虑这些问题。我很好奇，那 25,000 多个 MongoDB 实例所属的公司中，现在有多少家希望当初是由他们的 DBA 负责管理其数据库平台。

这个例子并非意在指责 MongoDB。它是一个很棒的产品，我目前也管理着几个 MongoDB 数据库。开箱即用，MongoDB、Oracle 和其他数据库平台都不安全。无论部署哪种数据库平台，锁定安全设置都是数据库管理员的职责。这个例子的重点是，人们试图避开粗鲁的 DBA，并会寻找替代方案，而这些方案往往缺乏对良好数据治理至关重要的安全性和其他控制措施。

### 提示

粗鲁的 DBA 是负债。乐于助人的 DBA 是资产。

乐于助人并不意味着 DBA 需要满足同事的任何请求。有很多时候我会拒绝请求。每当我拒绝一个请求时，我都会保持礼貌，并提供后续指导，让请求者能够继续工作。请求者需要完成他们的工作。他们知道数据库中的某些东西目前阻碍了他们完成任务。根据我的经验，大多数请求者在如何正确绕过数据库障碍方面的知识不如数据库管理员丰富。这正是 DBA 必须帮助将请求塑造成能同时满足双方需求的方案的时刻。

## 跨平台工作

如今的数据库管理员（DBA）通常需要与多种数据库平台打交道。只专注于一个数据库平台的管理员已经很少见了。在我 DBA 职业生涯早期，我受雇于一家只使用 Oracle 数据库系统的公司。我们的 DBA 小团队日复一日地与 Oracle 系统打交道。有一天，我们的经理告诉我们，SQL Server 即将引入公司，我们中需要有一个人来快速掌握这个产品。我立刻抓住了这个机会。我有些惊讶，当时团队里并没有为谁来管理这个新产品而争抢。虽然我的主要专长是 Oracle，但从那天起，我之后的每一份工作都需要同时管理 Oracle 和 SQL Server。就在那一天，我知道能够管理多个平台对我的未来职业前景是有好处的。我发现，能够成功地与多个数据库平台合作，对我的职业成长更为有益。在学会了 SQL Server 之后，我并没有止步于此。我继续学习更多关于其他数据库产品的知识，尤其是在社区中推出新产品时。

多年来，我使用过许多不同的数据库引擎：Oracle、SQL Server、MongoDB、MySQL、PostgreSQL 和 DB2 等。如今的数据库管理员也被要求这样做。很多时候，由于第三方软件的部署，你会被要求支持不同的数据库平台。例如，一个产品在 Oracle 上运行，而另一个产品只支持 SQL Server。随着 NoSQL 数据库的兴起，你可能会发现你信赖的关系型数据库并不是满足不断增长的数据量的合适解决方案。因此，你的职业生涯很可能会让你在多个数据库平台上工作。那么你该如何做呢？你如何与这些彼此差异巨大的产品一起工作？

我在第一次参加 SQL Server 管理课程时，就学到了该如何应对这个新系统。在开始 DBA 职业生涯之前，我选择上大学并获得了计算机科学学位，专注于数据库系统。在大学里，我学习了数据库系统如何工作的理论。我学习了范式、事务的属性、关系数据库理论等等。当我学习成为一名 Oracle DBA 时，我并没有过多思考这些理论。当我坐在第一次 SQL Server 课堂上，讲师正在讲述如何在 SQL Server 数据库中创建表时，我的思绪开始集中：在 Oracle 中，我们在表空间中创建表，一个数据库由多个表空间组成，但 SQL Server 有些不同。然后我想起大学里的一位教授曾谈到，数据库引擎负有将存储与数据库设计抽象开来的责任。那一刻我意识到，我应该依靠大学里学到的关系数据库理论，而我需要做的就是弄清楚每个数据库平台是如何实现这个理论的。第二天 SQL Server 管理课，我们将讨论备份和恢复。我知道这在 Oracle 中是如何工作的。我记得大学时学过，数据库系统拥有事务日志，而一份备份加上日志可以让我将数据恢复到某个时间点。从 Oracle 的角度来看，那个事务日志就是在线和归档的重做日志。我在课堂上了解到，SQL Server 为每个数据库保留一个事务日志。我大学课程中学到的关于数据库系统如何运作的基础知识，让我在进入职场后，能够轻松地看懂 Oracle 是如何实现细节的。面对一个新平台，我立刻明白我只需要弄清楚该系统是如何实现事务日志的，之后我就可以顺利上手了。

### 小贴士

如果你通晓通用的数据库理论，那么遇到不同的平台就意味着你只需要了解该平台是如何实现这个理论的。

对我而言，大学时光已是久远的记忆。从那时起，数据库系统已经发生了巨大的变化。数据库厂商不断实现新功能，比如物化视图，这些我在大学里从未学过。Oracle 引入了物化视图，我曾利用它们在某些情况下成功地提升了应用性能。有一天，我需要在 SQL Server 上做点什么，我觉得物化视图会是个不错的选择——如果我能弄清楚 SQL Server 是如何实现这个功能的话。在这种情况下，当我知道在 Oracle 中要利用哪个功能，并想在另一个数据库引擎中实现它时，我会进行一次简单的网络搜索，输入 Oracle 功能名称和另一个平台的名称。快速搜索“SQL Server 物化视图”让我迅速找到了一个名为索引视图的 Microsoft 功能——名称不同，但功能非常接近。然后我搜索“SQL Server 索引视图”，很快就在另一个平台上顺利上手了。这方法反过来也适用。如果你来自 SQL Server 背景，需要使用 Oracle，搜索“Oracle 索引视图”会很快让你找到物化视图这个功能名称。

当我确定了新数据库引擎中哪个功能有望提供我所需要的功能后，我就会转向该产品的官方文档以了解更多。

## 技巧

如果你了解某个平台上的某个功能，并想在另一个平台上找到类似功能，你可以搜索“其他平台名称 功能名称”，这样你就能快速找出另一个平台上的对应功能。

只要你知道在一个平台上用什么功能来完成工作，通常花不了多长时间就能找出另一个数据库平台上类似功能的名称。这让你能够快速地在可能不熟悉的平台上执行任务。

另一个技巧是搜索“为新平台管理员准备的老平台指南”。这会提供一些指南，帮助你将已知平台的知识迁移到你刚开始使用的另一个平台。例如，你可以搜索“Oracle for the SQL Server DBA”或反之亦然。当我需要快速熟悉一个新的数据库平台时，我经常进行类似的搜索。通常花不了太长时间就能找到一份（或几份）“速查表”。速查表能让我快速了解新数据库引擎所需的术语。但是，你应该只把这些结果当作指南。获取最佳信息来源，应使用产品的官方文档。我经常发现，有些速查表是由对一种数据库引擎非常了解、而对另一种产品只有粗浅了解的人编写的，他们常常遗漏一些非常重要的信息。例如，在对比 Oracle 的“数据库”和 SQL Server 的“数据库”时，速查表通常无法清晰地解释“数据库”这个术语在这两个不同平台上的含义是不同的。一份速查表通常只用一句话来强调差异，而要更准确地解释这些差异，可能需要一整段话。因此，你应该参考官方文档来了解更多。

最后，扎实的数据库理论基础并非必需，但它确实有帮助。Oracle 是一种关系数据库。因此，Oracle 数据库管理员应该学习 Edgar F. Codd 博士的关系理论以及 Christopher J. Date 的工作。Codd 和 Date 是我们今天使用的关系数据库的先驱。正如我之前提到的，如果你了解引擎**应该**如何工作的理论，那么你就能轻松地专注于引擎如何实现该理论。

让我用一个汽车的例子来说明这一点。你会把你的丰田车只交给一个只知道如何修理雪佛兰 5.7 升发动机的修理工吗？我更愿意把车交给一个了解内燃机**工作原理**的修理工。汽油与氧气混合后进入气缸。火花点燃油气混合物，推动活塞，从而使发动机运转。一辆丰田轿车可能使用的是燃油喷射，而不是老式的化油器。那个雪佛兰 5.7 升发动机修理工对燃油喷射可能一无所知。但理解内燃机工作基础的修理工知道，燃油喷射是将燃料和氧气输送到气缸的另一种方法。

回到关系数据库引擎，制造商们努力遵循许多 Codd 和 Date 的理论基础。正如我之前所说，如果你了解其背后**应该**如何工作的理论，那么你只需要弄清楚特定供应商是如何实现该细节的。这使得使用不同的数据库平台变得容易得多。即使你只使用一种数据库平台，扎实的数据库理论基础也会对你的职业生涯有所帮助。遗憾的是，如今太多大学专注于向学生传授“可就业技能”，这意味着他们教授行业最急需的具体知识，而忽略了理论基础。别误会，可就业技能固然很好。但我见过太多大学只关注 Oracle，而完全跳过了 Codd 和 Date 的工作。学生毕业了，凭借在大学学到的知识，很容易就找到了一份 Oracle 数据库管理员的工作。大学和学生皆大欢喜。然后学生在职业生涯中晋升，被要求使用 MySQL。对这位学生来说糟糕的是，他们只是一个“雪佛兰 5.7 升发动机修理工”。他们从未接受过适当的数据库理论基础教育，很可能很难适应不同的数据库平台。

回顾一下你自己的教育背景。如果你没有接受过良好的数据库理论教育，花些时间投资自己，学习更多关于它**应该**如何工作的理论。然后将该理论应用到你管理的数据库中，看看它是**实际**上如何工作的。

## 数据库管理员的典型回答

我观察到，当有人向数据库管理员提问时，给出的回答似乎经常属于以下三类：

*   是的，我可以做这个。
*   不，我不能做这个。
*   这要看情况。

第一个回答是每个人都爱听的。他们向你提出了要求，而你将满足他们的请求。

中间的回答是每个人都讨厌听到的。在本章前面，我描述了粗鲁的数据库管理员。通常，正是这个回答开始定义了那位难以共事的粗鲁管理员。如前所述，如果你需要向某人提供这个回答，请找一种更委婉的方式表达，并且务必努力为他们提供一些帮助，以便他们能继续完成自己的工作。在那些你需要说“不”的场合，务必提供一个充分的理由说明为什么请求无法满足。如果你能解释拒绝的原因，你就不会被认为是那个粗鲁的管理员。要准备好在日后为你的拒绝辩护。请求者不可避免地会对别人说，因为你的拒绝妨碍了他们的工作。管理层可能会来找你，问你为什么拒绝该请求。你需要有一个充分、可靠、基于事实的拒绝理由。没有人喜欢听到“不”，但有时它不得不说。

这就引出了最后一个回答，这也是每位数据库管理员最喜欢的回应：“这要看情况。”通常的情况是，某人带着一个请求来找数据库管理员，但却遗漏了管理员需要知道的关键细节。管理员需要进行一些调查工作以了解更多信息。无论出于何种原因，数据库系统中的决策往往是一种平衡行为。在天平的一端增加某些东西，另一端就会受到影响，反之亦然。例如，一位应用程序开发人员问：“在这个表上添加索引会提升我的性能吗？”这个回答，可想而知，是“这要看情况。”如果`SELECT`语句使用了该索引，它很可能会更快完成。但天平的另一端是`INSERT`性能，如果在表上创建了过多的索引，性能可能会受到负面影响。即使是回答的前半部分也取决于尚未明确的细节。索引不一定能提升`SELECT`性能。表中的行数以及需要返回的行数是问题中缺失的关键细节。有时，表面上看起来简单的请求，需要进行更多分析才能给出恰当的回答。

每当给出“这要看情况”的回答时，通常最好提出一些引导性的问题，以帮助给出最佳可能的答案。请求者通常不知道他们需要提供哪些细节才能得到正确的答案。如果请求者知道得足够多，能提供这些细节，他们自己就能回答这个问题了。他们是来向数据库管理员求助的，很多时候，管理员需要手把手地引导他们。如果数据库管理员仍然需要给出“这要看情况”的回答，管理员应该解释其中的权衡取舍，以便请求者能掌握尽可能充分的信息。


## 了解你的受众

作为数据库管理员（DBA），你是组织内处理数据库相关事务的最佳资源。DBA 需要能够与公司许多不同领域的人沟通。成功的 DBA 是能够根据听众调整其措辞的人。你与首席执行官（CEO）和首席信息官（CIO）的谈话方式会不同于与其他 IT 人员的谈话方式，而这又会不同于与其他数据库管理员的谈话方式。例如，如果你与 IT 管理层交谈，并开始抛出像 `buffer cache` 或 `shared pool` 这样的术语，你将是在他们理解范围之上说话，他们将无法跟上你。相反，使用术语 `cache`。大多数 IT 管理层理解什么是缓存。当与可能不知道缓存含义的 CEO 讨论相同主题时，使用术语 `database memory`，他们可能会有更好的理解。然而，如果你在与另一位 Oracle DBA 交谈时使用术语 `cache`，他们会问“哪一个？” DBA 精通 `shared pool` 和 `buffer cache` 的不同功能，因此你可以更具体。了解你的听众以及如何快速提供他们能够理解的术语将使你成为更有效的沟通者。表 `2-1` 说明了你如何根据交谈对象改变你的术语。

表 2-1

听众数据库术语对照

| 与 DBA 交谈 | 与 IT 人员交谈 | 与非 IT 人员交谈 |
| --- | --- | --- |
| `Buffer cache` | `Cache` | `Database memory` |
| `View` | `View` | `Stored query` |
| `Instance` | `Database` | `Database` |
| `Redo logs` | `Transaction logs` | `Transactions journal` |
| `Table` | `Table` | `Table` |
| `Index` | `Index` | `Index` |
| `Tablespace` | `Data files` | `Database` |

表 `2-1` 绝对不是一个完整的列表，但它确实让你了解了如何根据听众选择词汇。大多数与数据库交互的人都理解 `table` 或 `index` 的概念，因此无论听众是谁，DBA 通常都可以使用这些词。然而，作为 DBA，在与非 IT 人员交谈时，你真的需要关注他们的肢体语言。有很多非 IT 人员使用数据库并理解一些基本概念，如表、索引甚至事务。当你与他们交谈时，观察他们对你术语的反应。你可能已经简化了你的用词，但仍然超出了他们的理解范围。如果他们理解 `table` 这个术语有困难，可以说一些类似这样的话：“`table` 就像 Excel 电子表格中数据按列和行排列的方式。”大多数人都使用过 Excel 或类似软件，因此就能理解这个概念。如果他们理解事务的概念有困难，给他们举一个从银行账户取款或转账的例子。尽最大努力保持冷静。如果与不了解数据库技术的人交谈时需要额外时间，请不要感到沮丧。分析你是否准确地传达了你的观点，并准备好提供更多的背景信息。准备好成为他们的导师，如果你愿意的话。

对于像 `shared pool` 这样的 Oracle 内存结构，我们在与其他 Oracle DBA 交谈时会用其名称指代该结构，但对 IT 人员会改为 `cache`，对非 IT 人员则改为 `database memory`，正如我们已经讨论过的那样。Oracle DBA 在彼此交谈时提到连接到 `instance`，但对其他所有人来说，它就是 `database`。

如果听众的技能水平未知，最好从假设他们属于非 IT 类别开始说话。尽可能简化与数据库相关的术语。在你的讨论中，倾听他们说的话。很多时候，他们使用的术语可以提示你需要如何调整你的词汇以适应他们的技能水平并更有效地沟通。在处理数据库服务请求时，DBA 应做的第一件事是倾听请求的细节。记住，并非每个人都使用与 DBA 相同的术语。不要立即拒绝他们的请求。他们可能不知道如何正确地提出他们的需求，因为他们不懂如何说你的语言。这是粗鲁的 DBA 需要真正重点关注并加以改进的一个方面。
