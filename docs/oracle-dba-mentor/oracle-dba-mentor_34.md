# 19. 容量规划

在过去的两章中，我们通过应用补丁和执行升级使我们的 Oracle 数据库保持前进，这两项对于数据库管理员来说都是至关重要的任务。有时，应用补丁是修复问题的被动任务。大多数时候，应用补丁是一项主动的努力。升级数据库是保持最新、维持支持并享受新功能的方式，这些新功能使数据库能够持续发展以支持基础设施。

在本章中，我们将讨论 Oracle 数据库的容量规划。就其本身而言，容量规划不会影响你当前的环境。相反，容量规划是你今天执行的一项活动，以帮助你为未来做出更好的决策。



## 复杂性

如果你与容量规划领域的专家交流，他们会非常乐意让你了解他们的工作有多么复杂和重要，以确保当今和未来拥有成功的运营环境。虽然我并非意在贬低或轻视他们的努力，但我对容量规划采取了一种更为简化的方法，正如我将在本章概述的那样。在我的 DBA 职业生涯中，这一方法一直对我颇有裨益。阅读本章后，你可能会发现你需求更多，并希望在容量规划工作中更加细致。如果本章内容对于你未来进行的容量规划来说有所不足，那么请务必自行深入学习该主题，在读完本章后，达到比我通常做法更深入的程度。对你们中的许多人来说，本章的信息将提供你所需的大部分知识，你很可能凭借这些知识就能维持你的数据库管理员职业生涯。

当我刚进入这个行业时，计算机系统的资源非常有限。软件开发人员会精打细算每一个字节的内存。将代码重构为执行相同任务但更高效、更精简的代码体，以节省内存，曾是一项常见任务。那时，磁盘空间极其有限，DBA 只存储他们需要的信息，不多存一分。如果你要求更多内存或更多磁盘空间，你必须明确说出需要增加多少。组织不会购买超出必要的资源。

我记得当我刚开始担任 Oracle DBA 时，我发现了一个 Excel 电子表格。只需输入几个数值，这个表格就能准确算出一个索引会有多大。你输入索引列的数据类型和精度，加上预期的行数，这个表格就会结合所有 Oracle 索引所需的开销进行计算，然后其中一个单元格会神奇地给出一个以 KB 为单位的数字。老实说，对于那个年代的 Oracle 版本，这个电子表格实际上非常好用。我记得用它来估算索引大小，然后在实际中创建索引，结果证明，一次又一次，估算都非常准确。

在一个项目中反复使用这个电子表格后，我意识到这种方法存在许多问题。首先，这种方法非常耗时。DBA 必须在电子表格中填入数值，得到估算结果，然后为每一个索引重复这一过程。我发现更简单的方法是：用样本数据填充一张表，创建一个索引，然后直接询问 Oracle 需要多少空间。如果样本数据量大约是预期数据量的 10%，那么我只需将索引大小乘以 10，就能得到一个很好的估计值。这样我不仅得到了索引大小的估计值，也得到了表大小的估计值。这意味着我也可以抛弃那个帮我估算表大小的电子表格了。

电子表格方法的另一个问题是，虽然它看起来非常精确并给出了一个确切的数字，但其输入值本身有时就是估算值。如果输入值不精确，那么无论工具多好，输出值也不会精确。所以，让我们把所有详细的分析留给更好的方法吧。相反，让我们尽可能轻松地得到一个答案。毕竟，这只是一个估算。

最后，尽管你在初步估算中可能做了些精细的工作，但根据我的经验，估算值在确定后不久就会变得不准确。例如，我曾参与一个系统项目，该系统将接收来自一颗卫星的数据点。我的任务是创建和设计一个能够处理来自这个绕地球运行物体的数据的数据库。卫星的制造者精确知道每天每分钟捕获的数据量。我对预期数据量的估算完全准确。我加载了一些样本数据，并能够精确计算出每月需要多少磁盘空间。当我们在生产环境中开始向数据库填充数据时，我生成的估算值与我们的实际系统情况相符。然后，需求变了。由于系统运行得非常好，于是有人提议将另一个系统的数据也加入到这个数据库中。好吧，这应该很简单。只需将预期增长乘以 2 就行了。然而，这个新系统会有原卫星所缺乏的额外数据。就这样，我那精确的估算瞬间泡汤了。关键在于，无论你的初始估算多么准确，需求和数据使用模式都可能发生变化。我们中很少有人维护的数据库系统，其数据流入是稳定且规律可循的，能够可靠地据此长期准确推断数据总量。

在我的职业生涯中，我学到的一个道理是，有时在对系统预期使用情况进行估算时，不必过于追求精确。与其陷入细枝末节的纠缠，我常常采取一种完全相反的方法，那就是从更高的层面来看待系统使用情况。


## 容量规划

在我们开始之前，我们需要明确 `容量规划` 的定义。这个术语的名称可能显而易见，但我发现不同的人对它可能有不同的理解。在本章深入学习之前，这个术语需要被明确定义，以确保我们都在同一页面上，双关语无意冒犯。

所有计算机系统都有一套执行工作的资源。在大多数情况下，我主要关注的资源只有 CPU 使用率、内存使用率和磁盘使用率。后者可以进一步细分为已使用的磁盘量，并由此推论出空闲磁盘空间量，以及处理请求所花费的时间。计算机系统中当然还有其他资源，但 CPU、内存和磁盘是最重要的三大资源。

当系统收到请求时，资源就会被使用。一旦请求完成，资源就可用于处理下一个请求。不幸的是，系统上的资源是有限的。对任何一个资源的需求过多，性能就会受到影响。如果我的请求占用了所有 CPU 资源，你的请求就不得不等待，而你完全有理由抱怨系统运行缓慢。资源不仅有限，而且需要同时被多个用户和进程共享。你作为数据库管理员的职责，就是确保最终用户有足够的资源，使他们不会察觉到任何性能问题。换句话说，你需要确保有足够的 `容量` 来承载工作负载。

数据库管理员的部分职责是确定系统初始上线时需要多少资源。换一种说法，DBA 需要 `规划` 资源利用率。在初始上线之后，DBA 需要预测未来的资源利用率。资源需求随时间增长是很常见的情况。DBA 需要为未来的增长 `做计划`，以便在需求产生之前就能提供相应的资源。

`容量规划` 是一种活动，旨在确保当前和未来有足够的资源来满足预期的需求。这是一种积极主动的做法。如果你发现自己是在被动模式下进行容量规划，那可能为时已晚，并且可能会失去客户。

对我而言，容量规划最困难的部分不是预测未来，而是为一个未知的产品设置系统并进行初始上线。预测未来是查看当前状况，乘以一个最佳估算系数，然后看图表上的趋势线指向何处。这是一项简单的任务。如果产品是已知的，那么你可以查看该产品在以往部署中的资源需求。如果产品来自第三方，供应商通常会提供资源建议以确保其产品正常运行。

为一个未知的系统进行首次设置更为困难，因为我们对它的资源利用率知之甚少。管理层的某人可能会说这样的话：“我们预计第一个月有 10 万用户，六个月内达到 100 万用户。”这些数字是有一些参考价值，但它们并没有告诉我们用户会消耗多少 CPU 或内存。简而言之，这是一个未知的系统，其资源需求也是未知的。对于未知系统，DBA 的最佳行动方案是：首先学习所有已定义的已知需求。然后，尝试参考他们过去参与过的、可能具有类似细节的其他系统，并以其作为指导。如果没有任何类似的系统存在，DBA 只能做出一个初步的猜测。无论哪种情况，DBA 都需要让团队中的其他人知道，初步的估计缺乏足够的背景信息来保证其高度可信。

多年来，我见过人们提出类似这样的问题：“支持一百万用户需要多少内存？”我很抱歉要让他们失望了，但并没有一个神奇的公式或表格能回答这个问题。我的系统中的一百万应用用户，可能与另一个系统中的一百万用户有着完全不同的资源需求。

由于这是一个未知的系统，在投入生产环境之前，它需要被开发和测试。在开发期间，DBA 应该关注资源利用率，以确定初步的猜测是否仍然有效，还是需要重新定义。一旦系统进入测试阶段，DBA 应该观察系统在负载下的表现。自动化测试工具很有帮助，它们可以模拟大量并发用户而无需真实人员的参与。你可以使用 `Swingbench` 或 `HammerDB` 来模拟测试负载，但请记住，这些模拟可能并不能代表你的应用程序。我过去曾使用 `JMeter` 来测试一个基于 Web 的应用程序，模拟多个用户。

如果可能的话，让一些真实用户来操作应用程序。我曾经时不时地寻求整个 IT 部门的帮助，让他们同时操作应用程序，即使只有 10 或 15 分钟。这样做能让我在真实世界接触系统之前，更好地了解资源在实际环境中的利用情况。
