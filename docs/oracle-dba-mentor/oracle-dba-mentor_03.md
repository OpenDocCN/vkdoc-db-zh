# 第一部分：初始设置

## 1. 引言

如果您已经拿起这本书，那么您很可能已经获得（或希望获得）一份 Oracle 数据库管理员（DBA）的工作，并且正在寻求职业生涯下一步的指导。市场上有许多书籍会详细介绍 Oracle 数据库工作原理的技术信息。我的书架上就有很多这类书籍。它们提供了大量有用的信息，是极好的资源。

正如本书标题所示，它旨在成为您 DBA 职业生涯中的一位导师。导师通常会与学员携手同行，指导他们的职业发展。我无法亲自在您身边，因此我认为次好的选择是写下我多年来传授给他人的建议和见解。我希望这本书比一般的计算机书籍更具个人色彩，更像任何其他导师与学员之间的关系，所以我经常使用“我”和“我们”这样的词。本书很可能是您职业生涯中众多导师中的一员。许多初级数据库管理员会与一位资深 DBA 一起工作，后者是他们的主要导师。本书应作为您主要导师指导的补充。您还会从遇到的其他 Oracle 专业人士那里获得建议，他们虽非正式导师，却会对您的职业成长产生影响。

您正在阅读的这本书之所以与众不同，是因为它的目标是教您如何找到其他同类书籍中没有的答案。如果您需要更多信息该怎么办？该去哪里寻找？最常见的答案是查阅 Oracle 文档，这是个很好的应对方法。但很多时候，人们因为 Oracle 文档庞大复杂而不敢使用。在本书的第 13 章，我将教您如何有效利用文档。可惜的是，Oracle 文档并不总是完备的，那么接下来该去哪里？当信息不易获得时，本书将为您指明下一步的方向。

有时，寻找答案的最佳资源就在您自己身上。我从许多初级 Oracle DBA 那里最常听到的一类问题大致是：“我能在 Oracle 中做*X*吗？”如果您问了这个问题，那么答案显然是“不能！”，因为这表明您甚至没有尝试过。在本书的学习过程中，我们将构建一个系统，让我们能够为自己创造许多问题的答案。在构建这个系统的过程中，我们将涵盖许多最佳实践。如果您遵循本书的建议，您不仅将拥有一个可以使用的测试环境，还会充满信心，确信您正在创建和管理的 Oracle 系统能够应对系统所需的多种不同需求。

本书并非高度技术性的书籍。它旨在供您在数据库管理职业生涯初期使用。有些人可能称之为*初级 DBA*级别。中级 DBA 也可能从中发现一些有用的信息。如果您是资深 DBA，您可能已经了解本书中呈现的大部分信息。然而，我发现有太多资深 DBA 不得不以艰难的方式学习这些知识，并希望有人能在他们职业生涯早期就向他们传授本书中的知识（这也是我写作本书的部分动机）。这本书仅仅是对您职业生涯漫长旅程的一个开始。没有人能一夜之间成为伟大的 DBA。您在职业生涯中可能已经遇到过一些杰出的 DBA。他们是如何达到如此崇高地位的？通过努力工作和持续学习。我担任 Oracle DBA 已超过 20 年，至今仍在不断学习新东西。对于许多我遇到的顶级数据库管理员来说，解决复杂数据库问题的挑战以及学习如何利用数据库引擎的特性，是这份职业最令人欣慰的方面。

本书的大部分内容源于多年来指导我的众多 DBA 们传授给我的东西，但遗憾的是，在我早期缺乏一位真正手把手的导师。在我该领域的第一份工作中，我是唯一的 DBA，所以没有人指导我。我的大多数“导师”甚至从未见过我，可能也不知道我的名字，或者他们曾帮助过我的职业生涯。但在我的职业生涯中，我从许多人那里汲取了点点滴滴，并用这些构建了您将在这里阅读的内容基础。

现在，这是我给您的第一条建议，以帮助您发展职业生涯：付出多少，就收获多少。听起来很简单，对吧？然而，付出需要时间、精力和自律，因此人们通常无法充分分享他们本可以分享的知识。我多年来学到的一个秘诀是，与他人分享知识的过程，教会我的和我教给他们的一样多。当您必须整理思路，尤其是以书面形式表达时，大脑中会发生一些变化，让您比没有写下任何东西时更好地掌握这些材料。Oracle 社区中的大部分知识共享都是通过书籍、博客文章、白皮书和论坛等书面形式完成的。

虽然本书专注于 Oracle，但您可以将这些相同的原则和技术用于处理其他数据库系统。例如，在第 13 章，我们将花时间学习 Oracle 文档。如果您是 SQL Server DBA，您只需将这些知识转换到微软的在线手册（Books Online）中。虽然本书涵盖了与 Oracle 相关的论坛，但也有许多针对 SQL Server 的论坛。建议是相同的，您只需要将其转换为适用于其他数据库引擎即可。

在您的职业生涯中，极有可能被要求使用多种数据库平台。在我当前的工作中，我主要使用 Oracle，但也使用 SQL Server 和 MongoDB。多年来我还使用过 PostgreSQL、MySQL 和 DB2。每当我接触一个新的数据库系统时，我都会尝试弄清楚如何在 Oracle 中完成某项任务，然后看看其他数据库引擎是如何做同样的事情的。例如，所有数据库引擎都有某种形式的事务日志。在 Oracle 中，它是联机重做日志。在 SQL Server 中，它被称为事务日志。两种引擎具有相同的功能。区别在于它们实现该功能的方式。



### 成为一名 DBA

通常来说，有两条职业路径可以成为数据库管理员。一般来说，初级的数据库管理员在从事 `DBA` 工作之前，要么是应用程序开发人员，要么是系统管理员 (`SysAdmin`)。成为 `DBA` 的应用程序开发人员通常有开发以数据库为中心的应用程序的经验，并且已经接触过表、查询和其他数据库活动。前系统管理员了解操作系统、磁盘子系统和基本的网络技能，这些都是数据库管理员需要理解的关键知识。无论 `DBA` 在开启数据库职业生涯之前走过这两条路径中的哪一条，他们在晋升填补公司初级 `DBA` 职位之前，很可能已经展现出高水平的技能。有时，`DBA` 会在其组织内参与与数据库相关的项目，即使还不是正式的数据库管理员。他们可以凭借这份工作经历，在另一家公司谋得一个入门级的 `DBA` 职位。虽然也可能，但一个人在没有过去担任过开发人员或 `SysAdmin` 的情况下获得 `DBA` 职位的情况更为罕见。那些直接从 `DBA` 起步、而非来自上述两条路径的人，通常在大学里专注于数据库系统的学习。

我常对人说，数据库引擎位于应用程序和服务器之间。因此，从一端或另一端招聘初级 `DBA` 是合理的。理想情况下，候选人应同时具备服务器和应用程序开发的经验，但这通常更难找到。

获得第一份数据库管理员工作可能很困难。首先，这份工作在技术上很复杂，但这也是它令人兴奋的原因之一。正如前面讨论过的，除了理解数据库引擎如何工作外，`DBA` 还需要对系统管理和应用程序开发有良好的了解。`DBA` 还需要理解网络和存储，并对 IT 安全有透彻的掌握。今天的 `DBA` 应具备云技术方面的经验。数据库管理员面临的挑战之一是胜任这份工作所需知识的广度。但这也对许多 `DBA` 非常有吸引力——这份工作永远不会枯燥，因为有太多东西要学，有太多不同的领域可以探索。多样性、复杂性和探索新功能的机会，是这份工作对许多人来说具有吸引力的特质。

获得第一份 `DBA` 工作的另一个绊脚石是公司内部数据的重要性。数据是一种组织资产。它需要被保护，同时又要保持可用以支持业务。许多公司不愿意将管理、维护和保护其 IT 基础设施中最重要组成部分之一的任务托付给经验不足的个人。我们将在本章后面的“数据守护者”一节进一步讨论这一点。

那么，你如何获得第一份 `DBA` 工作呢？对大多数人来说，如前面提到的，你会从系统管理或应用程序开发开始你的职业生涯。如果你从 `SysAdmin` 起步，那么尽可能多地学习关于网络、存储和 IT 安全的知识。如果你从应用程序开发人员起步，那么花时间尽可能多地了解如何创建良好的数据库应用程序。学习表设计的细节、编写 `SQL` 语句，以及如何通过利用索引、物化视图等适当的数据库特性来获得最佳的应用程序性能。

接下来，尝试将你的职业生涯重心向数据库方向倾斜。与你的经理谈谈你希望更多地了解数据库管理的意愿。主动承担小型的数据库相关项目，即使这些项目超出了你传统职责的范围。正如你将在本书后面学到的，你可以下载 `Oracle` 数据库，安装在测试系统上，并开始试用这个产品。你现在就可以开始构建你的 `Oracle` 数据库知识和经验，没有任何东西能阻止你！在你职业生涯的某个阶段，你很可能还会培养与 `DBA` 角色相关的其他技能，包括软技能和技术技能。

一旦你有机会在一些 `DBA` 任务上小试牛刀，就开始留意初级 `DBA` 的职位空缺。很多时候，如果你的经理知道你的兴趣，他们可能愿意给你一个机会，作为一个好员工，在你期望的方向上发展你的职业生涯。一个好的经理明白让好员工保持满意的价值，即使员工在公司内部转岗时需要更新他们的技能组合。

并非所有经理都愿意帮助员工发展他们的职业道路。另一个障碍可能仅仅是当前公司内部缺乏机会。在这种情况下，经理们不能仅仅因为一个员工想换岗位就凭空创造一个职位空缺。如果这两种情况中任何一种适用于你，那么也许是时候在当前公司之外寻找工作机会了。归根结底，推动你的职业生涯前进取决于你自己。你可能希望在当前公司继续前进，或者，如果你的公司无法帮助你实现职业抱负，你可能需要决定做出更大的改变。

如果你正在其他公司面试一个新职位，你不可避免地会被问到类似 `你认为五年后自己会在哪里？` 这样的问题。这是你向潜在的新雇主表明你渴望朝着 `DBA` 职位努力的机会。抓住时机！明确表达，让你的意图为人所知。有些公司可能会根据这个答案认定你不适合他们，但很多其他公司会愿意，因为他们理解投入且热情的员工的价值。根据我自己的经验，我宁愿为一家有兴趣帮助我朝期望方向推进职业生涯的公司工作，也不愿得到一份看不到未来职业发展的工作。有太多人被困在他们讨厌的工作中。我宁愿拥有一份能让我满足的职业，做有挑战性和回报的工作。最终，你必须审视自己的内心，找出什么能激励和让你兴奋。对我来说，那就是成为一名 `DBA`。我无法想象做其他工作。既然你在读这本书，你可能也有同感。无论你选择哪条道路去获得第一份 `DBA` 工作，祝你好运！


### 典型的 DBA 一日

本节的标题有些用词不当，因为对于 DBA 来说，并不存在所谓“典型”的一天。当然，与我交谈过的每一位 DBA 都是以喝杯咖啡或其他晨间饮料、然后检查电子邮件中是否有需要立即处理的事务来开始一天的，但在此之后，这一天很可能就与“典型”二字毫不沾边了。这种多样性对许多 DBA 来说很有吸引力，因为他们不会日复一日地 `困在` 同样的事务中。

大多数数据库管理员确实有一套相似的职责组合，例如：

*   检查备份作业状态
*   确保数据库安全
*   创建新数据库
*   创建新数据库对象
*   确保数据持续可用
*   调优数据库性能
*   设计数据库
*   规划数据库容量
*   与其他 IT 员工和应用程序开发人员紧密协作
*   参与值班轮换
*   安装软件
*   执行 `提取、转换和加载 (ETL)` 活动
*   故障排除

这绝非一份详尽无遗的清单，DBA 很可能还负有其他职责。很多时候，你并不会在一段长时间内持续、积极地处理某项分配的任务。例如，我每年只进行几次容量规划活动，但检查备份作业状态是每日的例行工作。事务有起有落，存在 `季节性` 变化，这些都使得工作保持趣味性。

DBA 的职责也会因公司而异。不同的组织根据自身需求和目标对该职位进行了定制。例如，我曾在一家只使用第三方软件应用程序的公司工作。在这种情况下，DBA 负责与应用程序供应商合作，为软件提供数据库服务。在另一家公司，我需要与应用程序开发人员紧密合作，协助他们编写 `SQL` 语句。这两份工作都给了我 DBA 的头衔，但工作的性质却大相径庭。

## 数据守护者

我在上一节中已有所提及，但掌握这个主题对于你作为数据库管理员取得成功至关重要。DBA 是公司的 `数据守护者`。然而，当我问同行 DBA 们他们认为自己最重要的职责是什么时，典型的回答是“备份”。对于 DBA 最重要职责是什么这个问题，我的回答是“保护数据”。备份只是整个数据保护使命中的一个组成部分。

数据是一项宝贵的资产。像 `Google` 和 `Facebook` 这样的公司免费向你提供他们的产品。你有没有想过为什么？简单的答案是，他们从你为他们生成的数据中获利。你搜索什么、关注谁，这些行为都会生成数据点，从而构建出你——这个用户的画像。这些数据点随后会被以多种方式使用。例如，`Google` 和 `Facebook` 可以利用这些画像为其他公司生成定向广告。他们也可以将数据出售给第三方。

即使你的公司不从事买卖数据的业务，其数据库中的数据仍然是一项宝贵的企业资产。一家销售产品的公司可能希望更多地了解其客户。我在大学时了解到一个著名案例，一家杂货店挖掘了客户购买模式的数据，发现在周末，啤酒和尿布经常被一起购买。杂货店推测，是父亲被派去商店为宝宝买尿布，同时也买了一些啤酒带回家。这家连锁杂货店开始将啤酒和尿布并排摆放在货架末端，从而提高了两者的销量。了解客户是企业利用其数据资产的一种方式。

即使你认为自己公司的数据没有价值，黑客们肯定不这么认为。黑客会尝试各种方法来获取访问你公司数据的权限。公司投入了大量资金用于硬件、许可证和工资，以存储数据并使其可供业务使用。然而，作为数据守护者，我们不能允许对这些数据进行无限制的访问。这样做可能会危及业务运营。我们决不能让任何人随意更改数据。例如，一家销售体育用品的公司拥有其销售产品的库存数据。如果公司有十根棒球棒库存，那么数据库就应该反映这一事实。如果有人恶意或意外地将数据更改为库存中有零根球棒，那么客户将无法订购该产品，公司就会失去潜在的销售额。相反，如果有人将数据更改为库存中有 100 根棒球棒，那么销售系统可能会尝试销售根本不存在的 90 根球棒，导致这些订单无法履行，从而引起客户不满。客户不满会导致未来的销售损失，这是必须避免的。这家体育用品公司需要在其数据库中准确反映其产品的库存。

当今许多数据库系统还存储着因其敏感性质而自身就具有价值的信息，例如 `个人身份信息 (PII)`。`PII` 是可用于识别特定个人的数据，例如姓名和出生日期、税号、信用卡号等。数据库中的任何 `PII` 都需要被加密，这样即使黑客获得了数据库访问权限，也无法从这些数据中获取任何价值。

未能保护数据对业务在多个层面上都是有害的，正如我们从近期众多关于又一数据泄露事件的头条新闻中所知。例如，2013 年，`Target Corporation` 遭受了一次众所周知的数据泄露，据报道，这次事件使该公司损失了超过 3 亿美元！`Yahoo` 在 2013-2014 年遭受了超过 30 亿账户的数据泄露。当 `Yahoo` 被 `Verizon` 收购时，据估计，收购价比如果没有发生此次泄露事件要低 3.5 亿美元。据估计，遭受数据泄露的小公司中有 60% 会倒闭。

我一直秉持的观点是，作为一名 DBA，我就是我公司的数据守护者。DBA 需要成为那个时刻思考着保护数据的人。这意味着要确保只有那些具有合法业务需求的人才能访问数据，并且只有那些具有适当修改权限的人才能修改数据。我们尽可能限制对数据的访问。DBA 不断被要求为业务中的新途径提供数据访问权限。随着公司学会将数据作为一项资产加以利用，他们正在探索利用这些数据驱动收入的新方法。数据不仅会以新颖而令人兴奋的方式被使用，DBA 可能还需要只为业务部门提供他们所需的数据，不多不少，以保护数据。DBA 可能还需要在另一个业务部门获得访问权限之前对数据进行清洗。

数据保护远不止备份。DBA 需要确保数据库引擎应用了最新的安全补丁。DBA 在数据库中授予和撤销权限。DBA 需要熟悉 `SQL` 注入和其他数据库黑客技术。DBA 需要学习如何加密数据库中的数据，以及如何保护加密密钥。希望本节能让你相信，DBA 最重要的工作不是备份。相反，DBA 必须精通数据保护的各个方面。DBA 需要成为数据守护者。

## 继续前进

正如之前所述，本书与其他书籍不同，因为它不仅专注于成为一名成功的 Oracle 数据库管理员所必需的技术信息，还为你提供如何学习那些在任何书籍中都找不到的知识的建议。本书可以作为你职业发展道路上众多导师之一，助你推进数据库管理员的生涯。我设计本书的目的，不仅是帮助你学习更多关于管理 Oracle 数据库的知识，更是帮助你掌握如何学习那些在其他书籍、甚至 Oracle 官方文档中也无法找到的内容。

下一章可能并不包含你所认为的技术信息。相反，它将重点介绍作为一名数据库管理员执行工作所必需的**软技能**。请不要跳过这一章，因为它仍然包含非常重要的信息。
