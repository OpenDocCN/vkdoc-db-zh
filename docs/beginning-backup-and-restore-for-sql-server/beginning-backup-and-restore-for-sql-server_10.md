# 5. 完整还原

按照前面章节的格式，我们将通过深入探讨本书的第二部分，继续备份与还原之旅。第一部分，包含第 1 至 4 章，涵盖了备份部分；这第二部分，包含第 5 至 8 章，将涵盖还原部分。

至此，我们认识到数据库备份是整个数据恢复过程的基础。没有备份，就不可能有还原。因此，确保所执行的备份是正确的至关重要，这一点我们之前已经讨论过。现在，基于这些基础知识，我们将进一步探讨执行完整数据库备份还原的各种方法。

需要重点指出的是，数据库还原主要在 `SQL Server Management Studio` 中完成。我想在这里强调“主要”这个术语，因为它并非还原数据的唯一方式。例如，如果您的 `master` 数据库已损坏，则必须在 SQL Server 正确启动之前从完整备份中还原它。这是因为 `master` 存储了 SQL Server 的所有配置信息。`master` 数据库损坏带来的一个主要问题，除了明显的 SQL Server 无法启动之外，还在于所有其他系统数据库（`msdb` 和 `model`）都依赖于 `master` 才能正常运行。如果 `master` 损坏严重到一定程度，那么将需要单独重建系统表，这是一件非常麻烦的事情。



#### 什么是完全恢复？

简而言之，完全恢复正如其名，但又并非完全如此；它恢复的是先前执行的完整数据库备份中包含的所有数据，但并不恢复整个数据库。回想一下，完整数据库备份只备份数据，而不备份系统设置。数据库备份中只包含指定数据库的表数据。

注意

如果你需要备份与 SQL Server 安装相关的系统设置，你必须拥有 `master`、`msdb` 和 `model` 数据库的当前备份。这些是系统数据库，对于 SQL Server 的平稳运行极其重要。

我刚开始接触 SQL Server 时，正值 SQL Server 7 和 SQL Server 2000 之间。我多少已经习惯了 MySQL 在界面和可用性方面的工作方式，所以一开始切换过来有点困难，但我逐渐掌握了窍门。MySQL 有一点我非常喜欢，那就是它们的“备份”就是巨大的 `.sql` 文件，里面包含大量的 `INSERT` 命令。没错，整个数据库的内容都被按顺序导出；先是针对某张表的 `DROP` 和 `CREATE` 命令，然后是该表中所有数据的单独 `INSERT` 语句。在有些人看来，这似乎是浪费空间，但要我说，精妙之处就在于细节。这样，时间充裕的数据库管理员就可以仔细检查那个备份，并在 SQL Server 中无法实现的细粒度级别上，实际掌控数据的插入。我的意思是，SQL Server 不会以这种格式创建备份；SQL Server 中的备份是专门为 SQL Server 引擎执行而创建的，不是为了方便人类阅读。是的，在 SQL Server 备份中，数据当然在那里，但无法通过在记事本中打开等方式读取。

完全恢复的另一个方面是，它们可用于创建现有数据库的完美副本。在开发环境中能够拥有这种功能对我来说绝对非常宝贵，因为我可以轻松地创建一个基于生产数据库的完整开发数据库，主要用于测试。然后，我可以创建另一个数据库，它是开发数据库的完全复制品，但将其命名为不同的名称。之后，如果我需要恢复数据，我可以轻松地在第一个开发数据库中 `TRUNCATE` 任何表，并使用 `INSERT INTO` 语句快速替换数据，其中第二个开发数据库作为源，第一个开发数据库作为目标。我将在本章后面详细展示如何操作。

首先，让我们暂时将注意力转向系统数据库。有三个主要的系统数据库，名为 `master`、`msdb` 和 `model`。这三个数据库承载着你的 SQL Server 安装的配置信息，因此它们对于整个安装的健康状况至关重要。从技术上讲，还有第四个名为 `tempdb` 的系统数据库，但此数据库在每次为特定 SQL Server 实例创建该实例时都会被销毁和创建，因此无需担心备份它。

#### 恢复 Master 数据库

如果你不得不恢复 `master` 数据库或任何系统数据库，我真诚地希望这不是因为严重的损坏问题。我可以亲口告诉你，这是一项重大的任务，必须极其谨慎和细致地完成。本节将展示从完整数据库备份恢复 `master` 数据库的各个步骤和说明。这些步骤可应用于恢复任何系统数据库，但现在，我将只关注 `master` 数据库。

本节依赖于存在一个当前的 `master` 数据库备份。通常，最好在每次更改表的结构或数据库的基础架构时都对 `master` 数据库进行一次备份。备份 `master` 数据库并确保在紧急情况下可用，将缓解未来的许多问题，我建议将这些系统数据库的备份作为你正常备份维护计划的一部分。我在前面讨论备份的章节中没有涉及这一点，因为本书的内容是基于单个数据库备份，而不是更高级别的数据库管理，在后者中我们也会处理系统数据库的备份。继续之前，请快速运行一次 `master` 数据库的备份，或使用你已准备好的备份。

要恢复 `master` 数据库，我们必须做的第一件事是在单用户模式下启动 SQL Server。对于不熟悉此操作的用户来说，单用户模式以有限的连接数（一个）启动 SQL Server，以便可以在不给服务器增加额外负担的情况下进行紧急工作。


##### 以单用户模式启动 SQL Server

以下是将 SQL Server 置于单用户模式的逐步说明：

*   打开 **SQL Server 配置管理器**，或者如果你使用的是 Windows 10，则打开开始菜单并输入 `sqlservermanager13.msc`，按 Enter 键，然后在右窗格中找到 `SQL Server (MSSQLSERVER)` 项。
*   右键单击并选择 **属性**，然后选择 **启动参数** 选项卡。
*   在 **指定启动参数** 字段中输入 `-m`。
*   单击 **添加** 按钮。
*   单击 **应用**，你会注意到出现一条消息，告诉我们更改在重启服务后才会生效。我们将立即重启服务。在此消息上单击 **确定**，然后在 `SQL Server (MSSQLSERVER) 属性` 窗口上单击 **确定** 将其关闭。
*   单击 **SQL Server Agent** 项，然后在工具栏中单击 **停止服务** 按钮。
*   重启 `SQL Server (MSSQLSERVER)` 服务。

这些步骤将我们的 SQL Server 实例设置为单用户模式。不过，让我们更深入地了解这些步骤的细节，以便真正理解我们的操作目标。

##### 将 SQL Server 恢复为多用户模式

在我看来，恢复 SQL Server 的访问权限比限制访问更容易。将 SQL Server 恢复到其原始状态（即多用户模式）的分步说明如下：

*   再次打开 `sqlservermanager13.msc`，并在右窗格中找到 `SQL Server (MSSQLSERVER)` 项。
*   右键单击并选择 `属性`，然后选择 `启动参数` 选项卡。
*   突出显示 `-m` 条目，然后单击 `删除` 按钮。
*   单击 `应用`，请注意会出现一条消息，提示我们重新启动 SQL Server 服务。单击此消息上的 `确定`，然后在 `SQL Server (MSSQLSERVER) 属性` 窗口上单击 `确定` 以关闭它。
*   单击 `SQL Server Agent` 项，然后按工具栏中的 `启动服务` 按钮。
*   再次单击 `SQL Server (MSSQLSERVER)` 项，然后按工具栏中的 `重新启动服务` 按钮。

这些步骤将我们的 SQL Server 实例设置回多用户模式。恭喜！您刚刚已将 SQL Server 设置为单用户模式、还原了 master 数据库，并将 SQL Server 恢复为多用户模式。

#### 从完整备份还原

到本章此处为止，您可能已经对如何从完整备份还原数据库有了很好的了解。我认识的每一个 DBA 都曾不得不在某个时候运行还原操作，因此这应该不是什么新鲜事。万一这是新内容，本节不仅将介绍如何使用 SSMS 和 T-SQL 从完整备份还原，还将介绍一些还原数据的要点，例如还原到日志备份中的特定点（称为时间点还原）。

##### 使用 SSMS 执行完整还原

还原数据库最常用的方法是使用与 SQL Server 最常关联的图形界面。SQL Server Management Studio 提供了大量可用于操作和管数据库的工具，因此它几乎是每位 SQL Server 管理员的首选。请注意，我这里说的是“几乎”；因为任何规则都有明显的例外，这绝不是侮辱或嘲笑。例如，我曾见过 SQL Server 管理员使用旧版 SSDT，因为他们获得了与数据库的（大部分）相同交互，并且额外获得了可用于编写 DTSX 包或任何他们需要增强界面来完成的任务的 Visual Studio 界面。

我们首先要做的就是找到最新的备份集。如果您像我一样设置了目录，那么它们位于 `E:\SQL Server\Backup`。

接下来，启动 SSMS 并展开 `数据库`。在 `数据库` 菜单内，您将找到您的用户数据库，以及 `ReportServer` 和 `ReportServerTempDB`（如果您安装了 SQL Server Reporting Services）。找到您要处理的数据库（本例中为 `backrecTestDB`），右键单击它，将鼠标悬停在 `任务` 上，然后将鼠标悬停在 `还原` 上，并查看我们可用的选项。图 5-12 显示了此时的界面。

首先，为什么事务日志选项是禁用的？它只能在数据库是使用完整备份还原的，并且在还原时启用了 `RESTORE WITH STANDBY` 的情况下启用。然后数据库将处于待机状态（在 `RecoveryPending` 模式下），等待事务日志尾部的还原。如果您使用 `RESTORE WITH STANDBY` 恢复了一个数据库，然后想要还原事务日志，那么事务日志选项将是唯一可用的选项，其他三个选项将被禁用。

##### 紧急全量恢复示例

最近，我有一个具体的实例，因为一个脚本中的错误，我不得不恢复生产数据。让我们假设你有一个包含数百条记录的表，每条记录都有一个唯一标识符。这些记录中的每一条都涉及一组特定的数据，这些数据对于特定项目通过工作流的正常运行至关重要。让我们进一步假设，在生产实例上运行的更新脚本没有定义 `WHERE` 子句。我相信你能看出问题出在哪里……我将表中所有行更新为了一个特定值，而不是仅仅更新我需要更新的那一行。现在，在这个例子中，有很多种方法可以恢复数据，但我选择了最快，而不一定是最干净的方法。该方法包含以下步骤：

*   定位最新的备份集（完整备份、差异备份和事务日志备份）
*   将最新的完整备份还原到一个名为 `fixYourMistake` 的新数据库
*   将最新的差异备份还原到 `fixYourMistake` 数据库
*   将自差异备份以来的所有事务日志备份还原到 `fixYourMistake` 数据库
*   在生产数据库中创建一个名为 `(tablename)_archive` 的备用表，其中 `(tablename)` 是我的表名
*   编写一条 `INSERT INTO…SELECT` 语句，将原始表 `(tablename)` 的内容写入归档表
*   编写一条 `TRUNCATE` 语句来清空原始表
*   编写一条 `INSERT INTO…SELECT` 语句，将 `fixYourMistake` 数据库中已恢复的数据插入到生产数据库

你可能会问自己，为什么我没有直接将最新的事务日志还原到生产数据库以更快地完成操作。我之所以没有这样做，是因为我不确定数据库是否还会有其他残留的损害，所以我想要有一个可靠的备份以便恢复，以防万一。我也不想还原整个数据库，因为在此期间还有关键操作正在录入。在我还原了有问题的表并验证数据恢复正确且没有其他损坏后，我在数据库服务器上还有一点清理工作要做。为了在这次恢复后进行清理，我只需要 `DROP` 掉 `fixYourMistake` 数据库和之前创建的归档表。之后，数据库就愉快地运行起来了，信不信由你，除了我自己，甚至没有人意识到数据库曾出过问题。

我想指出，如果没有一个可靠的备份和恢复结构，这是不可能做到的。如果没有先前的备份来恢复数据，我就无法还原到上一个事务日志，这意味着我将丢失自上次完整备份以来——在一个生产环境中的数据。这是非常非常糟糕的消息，我认为不言而喻，自那以后，在生产环境上执行的脚本都受到了更为严格的审查。所以，我想你可以说，当这一切发生时，我既是自己最大的敌人，同时也是自己最好的朋友。在未来，我想少一些“敌人”会是件好事。

## 6. 差异备份恢复

回到第 2 章，我们介绍了差异备份的基础知识，以及差异基准对整个恢复策略的重要性。这些同样的原则也适用于差异恢复，只不过我们现在可以说是在处理交易的另一端。恢复模式的选择是决定可用恢复方法有效性的最重要方面。回想一下，我们的测试数据库使用了完整恢复模型；这意味着我们可以使用数据库的全套备份和恢复功能。如果我们选择了大容量日志恢复模型，我们将获得大部分相同的功能，但如果事务日志中有大容量日志事务，我们将牺牲像第 5 章中那样进行时间点恢复的能力。大容量日志恢复模型将只允许我们恢复到备份的末尾；换句话说，每个备份都必须完整恢复，否则根本不会进行任何数据恢复。

在 SQL Server 中从差异备份恢复数据与从完整备份恢复数据几乎相同，但有一些明显的区别。首先，最明显的一点是，差异备份只能还原到对应的完整备份。这是因为差异备份是从上一次完整备份（在第 2 章中称为“基准”）以来，从已备份的事务日志和更改数据中获取的最后一组数据。因此，如果你尝试按错误顺序还原差异备份，它将失败。数据的序列将无法对齐，SQL Server 将停止恢复操作。其次，数据库必须处于 `NORECOVERY` 或 `STANDBY` 模式才能还原差异备份。这样做的原因是数据库只能将差异备份还原到完整备份；如果不先还原完整备份，就无法单独还原差异备份。一旦完整备份被还原，数据库就可以被置于 `NORECOVERY` 或 `STANDBY` 模式。

让我们简要回顾一下在恢复数据库时可用的恢复状态选项。这些选项控制恢复完成后数据库的状态，并且可以随着每次恢复类型而变化。

*   `RESTORE WITH RECOVERY`：这是默认选项，意味着数据库在恢复过程完成后立即可供使用。这意味着整个数据库恢复操作已完成，意味着无法再恢复额外的事务日志。
*   `RESTORE WITH NORECOVERY`：此选项不会使数据库处于就绪状态，这意味着在恢复操作完成后数据库不可操作。使用此选项，可以继续正常恢复事务日志。在对数据库执行使用 `WITH RECOVERY` 选项的最终命令之前，数据库无法进入就绪状态。
*   `RESTORE WITH STANDBY`：最后，此选项使数据库处于待机模式，本质上是只读的。任何尚未提交到事务日志的事务都将被撤销，并且撤销操作保存在一个待机文件中。

还记得第 5 章中我们如何勾选了 `将源数据库保持为还原状态 (WITH NORECOVERY)` 选项吗？这使我们仍然可以选择恢复更多备份，特别是差异备份和任何后续的事务日志备份，我们将在第 7 章中介绍这些内容。



#### 什么是差异还原？

简而言之，差异还原就是从差异备份中恢复数据。我们知道，差异备份仅包含自上次完整备份以来发生变化的数据，因此它的文件比完整备份要小。

那么，这如何融入我们的备份策略呢？我们的完整备份在每天凌晨 12:00 运行，差异备份每六小时运行一次，事务日志备份则每小时运行一次。换句话说，我个人认为这是一个相当稳健的备份计划。使用完整或简单恢复模型，以及实施差异备份和还原，有以下几个非常显著的意义：

*   我们可以拥有更少的事务日志，从而在备份计划中总体文件更少，因为我们可以删除最后一次完整备份之前的所有事务日志，前提是这些事务日志中的数据已经包含在差异备份中。
*   我们拥有更多的恢复点，或者说，我们可以还原到的位置更多了。
*   备份可以在白天进行，而无需每次都运行完整备份。这将减少服务器 CPU 时间并节省空间。

我们可以看到，在我们的常规数据库恢复计划中使用差异备份和还原，对于我们之前确定的理想恢复场景来说是最佳选择。

#### 使用 SSMS 进行还原

我希望到目前为止，我们都对使用 SQL Server Management Studio 进行数据库还原有了一点经验。如果你读过我的第一本书，我严重依赖维护计划的创建来处理日常数据库维护的全部工作。这包括一种最终可用于数据恢复的备份方法，包含完整备份、差异备份和事务日志备份。我们现在还不想走得那么远，因为这是我们将在后续章节中整合所有内容时才会涉及的，但我们确实想看看如何在 SQL Server Management Studio 中处理差异还原。

如果还没有启动 `SSMS`，请现在开始。你应该会看到左侧有对象资源管理器的界面，而主区域是空的。我通常会在主区域保持一个“新建查询”窗口打开，但这不是必须的。一旦你打开了 `SSMS` 并准备好继续，请展开 `数据库`，右键单击你的测试数据库名称（我的是 `backrecTestDB`），将鼠标悬停在 `任务` 上，然后选择 `还原`，最后选择 `数据库…` 以继续。图 6-1 显示了该菜单项的位置。

在你的界面中，验证列出的三个备份（或你拥有并想要还原的任意数量的备份）旁边的所有复选框是否都已选中。由于我们要还原差异备份，我们必须先还原完整备份，并且为了能够继续还原差异备份和事务日志，我们必须将数据库保持在 `正在还原` 状态。如果你取消选中所有恢复选项，你将看到可以在此数据库内完成的可用还原组合。这些组合如下：

*   完整备份
*   完整备份 + 差异备份
*   完整备份 + 事务日志
*   完整备份 + 差异备份 + 事务日志

换句话说，当我们使用 `SSMS` 时，每次想要执行还原操作都必须从完整备份开始。可以这样想：如果下午 3:05 你的数据库出现错误，在我的当前备份计划下，你需要还原当天早上的最后一个完整备份，然后是中午 12:00 的差异备份，最后是每小时的事务日志，直到恢复到你需要的数据位置。

请注意，我们位于 `还原数据库` 屏幕的 `常规` 选项卡上。我们已经浏览过这些选项卡中的各种选项，所以不需要再重复一遍。请注意，我们还将备份事务日志的尾部，这是一个默认选项，以便运行还原过程的事务保存在新的事务日志中，而之前的事务则保存在新备份的事务日志尾部。

由于我们要还原差异备份，我们只需确保在 `常规` 选项卡上选中了完整备份和差异备份选项，然后按 `OK`。然后我们会等待几秒钟，或者可能更长时间，然后看到如图 6-3 所示的内容。

#### 使用 T-SQL 进行还原

使用 `T-SQL` 还原差异备份的过程出奇地简单。到目前为止，我并没有包含大量的 `SQL` 代码，因为我希望这些书籍更侧重于数据库管理的可视化方面，但我认为这里值得花一点时间。

使用 `T-SQL` 的真正优势，除了自动化目的之外，在于如果你无法访问 `SQL` 界面但仍然可以访问数据库服务器，你可以使用 `sqlcmd` 来执行 `SQL` 语句。

使用逻辑备份设备且没有其他文件需要还原时，进行差异还原的基本语法如清单 6-1 所示。

```sql
RESTORE DATABASE backrecTestDB
FROM backrecTestDB
WITH NORECOVERY
清单 6-1：RESTORE DATABASE SQL 脚本
```

没有定义备份设备，或者当你需要从文件系统而非逻辑备份设备还原，并且没有其他文件需要还原时，进行差异还原的基本语法如清单 6-2 所示。

```sql
RESTORE DATABASE backrecTestDB
FROM DISK = 'E:\SQL Server\Backup\backrecTestDB\backrecTestDB_DIFF.bak'
WITH NORECOVERY
清单 6-2：RESTORE DATABASE SQL 脚本
```

请记住，在还原差异备份时，当我们指定 `NORECOVERY`，我们表达的意思是我们希望先还原最后一个完整备份，然后再还原差异备份。这样做的原因在于，正如我们之前讨论的，所有还原都必须有一个逻辑起点，而唯一能做到这一点的地方就是最后一个完整备份，它是该时间段内的最高级别还原。从这一点出发，我们就可以筛选差异备份，最后进入事务日志，以到达数据还原的确切位置。如果我们还有额外的事务日志需要还原，我们也希望指定 `NORECOVERY`；否则，我们将指定 `RECOVERY`。

如果你执行清单 6-1 中的代码，考虑到你已经定义了一个逻辑备份设备，并且没有其他文件需要还原（即额外的事务日志），你将得到与我们之前执行差异还原相同的结果。清单 6-2 也会产生相同的结果，如果你需要从文件系统而非逻辑备份设备还原。


#### 摘要

本章是本书中最简短的一章，原因很简单：差异恢复的概念是建立在先前完整备份的基础之上的。换言之，只要你拥有完整备份，你就能希望开始恢复所需的数据，无论该数据是位于差异备份还是事务日志中。恢复差异备份可以被视为恢复完整备份的逻辑延伸，因为差异恢复的存在以完整备份已先被恢复为前提；但正如我们之前提到的，如果你需要的是事务日志中的数据，你随时可以改为从事务日志恢复。

我们介绍了如何通过 `SSMS` 和直接使用 `SQL` 语句来恢复差异备份，并简要概述了与此类恢复相关的不同选项。请确保在你所需编写的恢复脚本中，包含我们在清单 6-1 和清单 6-2 中介绍的语法。

下一章，我们将探讨如何恢复事务日志，以及在此领域可以启用或禁用的不同选项。
