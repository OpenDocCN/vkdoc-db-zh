# 8. 还原方案示例

本章中，我们将把第 5 章到第 7 章学到的内容整合成一个连贯的结构，用于创建一个典型的还原场景。请注意，在之前的章节中我们并未涉及数据还原的所有概念；这将在本章中弥补，因为我将更侧重于介绍 SSMS 中各种屏幕上显示以及通过使用 `sqlcmd` 可用的各种条件和选项。因此，本章将是对 SQL Server 中数据还原的各个角度的一次全面审视，从基本的完整还原到事务日志还原的精确时间点。

建立还原场景时，要牢记的最重要的事情之一是，在将其投入生产环境之前，你需要**练习实施**。你应该有关于如何执行还原的分步说明，并且这些说明应足够简单，以便非 DBA 人员也能坐在服务器控制台前成功执行还原场景。我知道这看起来像是一项艰巨的任务，但本章末尾我将帮助你编写此流程。

执行还原场景必须做的第一件事是定义还原标准。我们需要列出一系列需要还原的情况，然后为每种情况定义最适合的还原类型。直接说还原到事务日志的最新点很容易，因为对于我们 DBA 来说，大约 99% 的情况可能都是如此。但有些情况下这并不是最佳方法，而这正是我们需要为之准备的。

让我们定义几种情况，然后是每种情况所需的适当还原类型。在这些场景中，我们希望：

-   将生产数据库转移到开发环境进行测试
-   从前一天晚上的未知时间点发生的灾难性错误中恢复
-   恢复 15 分钟前被覆盖的数据
-   在常规备份计划之外进行一次备份，作为“已知良好的起点”

正如你所见，对于至少一半的情况，我们只需从事务日志的最后一次备份进行还原即可完成。这就是数据还原的优势（除了实际恢复数据之外）；在大多数情况下，数据还原确实不需要太多预先规划。你知道数据存在于备份中，所以还原备份就行了。非常直接，从那里开始，我们可以根据需要保持简单或使其变得复杂。

在开始构建完整的还原场景之前，让我们先回顾一些在前几章中不太适合讨论、但在讨论数据还原时绝对值得一席之地的主题。并非所有的还原都是通过对象资源管理器中的数据库选项完成的（即右键单击数据库名称，悬停在“任务”上，然后选择“还原”，再选择“数据库”选项）。此菜单中还有两个我们尚未讨论的选项，即“文件和文件组”以及“页面”。这意味着我们不仅可以从数据库还原，还可以还原单个文件和文件组，或者细化到页面级别。

#### 页面还原

对于页面还原，要记住的重要一点是，`msdb` 数据库中的 `suspect_pages` 表是可能的损坏信息所在的位置。该表的结构如表 8-1 所示。

表 8-1 `msdb..suspect_pages` 结构

| 列名 | 描述 |
| --- | --- |
| `database_id` | 发生损坏的数据库的 ID。 |
| `file_id` | 损坏数据库中文件的 ID。 |
| `page_id` | 疑似损坏的页面的 ID。 |
| `event_type` | 表示错误类型的数字。可能的值如下：这是一个错误值的“全能”类型。这意味着发生了 823 错误（由于硬件故障）或 824 错误（其他数据库损坏）。校验和不正确。页面已损坏（torn page）。已还原的页面（被标记为可疑之后）。已由 DBCC 修复（被标记为可疑之后）。已由 DBCC 释放（被标记为可疑之后）。 |
| `error_count` | 错误发生的次数。 |
| `last_update_date` | 条目的日期和时间。 |

在想要执行页面还原时，有几件事需要牢记。例如，你只能还原数据库页面。这可能看起来很明显，但其含义是你不能还原事务日志或任何“特殊”页面：特别是分配页、全局分配映射页、共享全局分配映射页和页面可用空间页。微软在 [`https://docs.microsoft.com/en-us/sql/relational-databases/backup-restore/restore-pages-sql-server`](https://docs.microsoft.com/en-us/sql/relational-databases/backup-restore/restore-pages-sql-server) 上说“仅限数据库页面”时使用了非常精确的语言。

##### 查询 msdb..suspect_pages

`msdb..suspect_pages` 表是告诉你哪些页面被怀疑损坏的关键，希望这能引导你制定一个可靠的计划来还原那些损坏的页面。换句话说，可能还没有什么真正具有破坏性的事情发生，但未来可能会有。很多时候，可以通过常规的数据库恢复操作来修复数据库，这些操作默认会覆盖页面。在数据库被还原的情况下，重要的是要认识到，如果损坏的页面已经被备份到正在还原的备份集中，那么损坏的数据将一直存在，直到被修复。

打开一个新的查询窗口，输入 `SELECT * FROM msdb..suspect_pages;` 并按键盘上的 F5。当我运行此查询时，返回了零个结果，这是件好事。如果我得到了任何结果，我会立即采取表 8-2 中所示的行动。

表 8-2 页面还原操作

| 步骤 | 描述 |
| --- | --- |
| 1 | 从 `msdb..suspect_pages` 查询输出中获取损坏页面的 `page_id`。 |
| 2 | 使用包含相应损坏页面的完整备份、文件备份或文件组备份开始页面还原。如果从 T-SQL 而非 SSMS 运行，请使用以下语法：`RESTORE DATABASE <database_id> PAGE = '<file_id: page_id>' FROM <backup device or file> WITH NORECOVERY` 请注意，`database_id`、`file_id` 和 `page_id` 值均来自先前运行的 `msdb..suspect_pages` 查询。 |
| 3 | 还原最新的差异备份。 |
| 4 | 还原相关的事务日志备份。 |
| 5 | 备份事务日志的尾部。 |
| 6 | 还原步骤 5 中的事务日志。 |

按照这些步骤操作，你就可以在 T-SQL 中安全地还原页面了。


#### 使用 T-SQL 完成数据库完整恢复

回想一下，由于我们是在完整恢复模型下工作，因此拥有更多用于备份和恢复数据的功能可供使用。特别是，我们能根据当前情况，对需要恢复的数据量进行精细控制。完整恢复模型允许我们在需要时恢复整个数据库，更重要的是，正如我们在第 5 章所见，它可以恢复到特定时间点。

在完整恢复模型下恢复数据库时，需要遵循一组基本操作步骤。通用步骤如清单 8-1 所示。

##### 清单 8-1. 数据库恢复操作步骤

*   切换到 master 数据库。
*   使用 `BACKUP LOG backrecTestDB TO DISK = 'E:\SQL Server\Logs\backrecTestDB_TLOG_TAIL.bak' WITH NORECOVERY;` 创建尾日志备份。
    *   SSMS 响应：`为数据库 'backrecTestDB' 的文件 'backrecTestDB_log' 处理了 18 页。BACKUP LOG 成功处理了 18 页，耗时 0.069 秒（1.981 MB/秒）。`
*   使用 `RESTORE DATABASE backrecTestDB FROM DISK = 'E:\SQL Server\Backup\backrecTestDB_FULL.BAK' WITH NORECOVERY;` 恢复最新的完整备份。
    *   SSMS 响应：`为数据库 'backrecTestDB' 的文件 'backrecTestDB' 处理了 107968 页。为数据库 'backrecTestDB' 的文件 'backrecTestDB_log' 处理了 2 页。RESTORE DATABASE 成功处理了 107970 页，耗时 21.669 秒（38.927 MB/秒）。`
*   使用 `RESTORE DATABASE backrecTestDB FROM DISK = 'E:\SQL Server\Backup\backrecTestDB_DIFF.BAK' WITH NORECOVERY;` 恢复最新的差异备份。
    *   SSMS 响应：`为数据库 'backrecTestDB' 的文件 'backrecTestDB' 处理了 136 页。为数据库 'backrecTestDB' 的文件 'backrecTestDB_log' 处理了 2 页。RESTORE DATABASE 成功处理了 138 页，耗时 0.283 秒（3.795 MB/秒）。`
*   使用 `RESTORE LOG backrecTestDB FROM DISK = 'E:\SQL Server\Logs\backrecTestDB_backup_2017_10_16_190004_6842741.trn' WITH NORECOVERY;` 恢复自最新差异备份以来最早的事务日志备份。
    *   SSMS 响应：`为数据库 'backrecTestDB' 的文件 'backrecTestDB' 处理了 0 页。为数据库 'backrecTestDB' 的文件 'backrecTestDB_log' 处理了 16 页。RESTORE LOG 成功处理了 16 页，耗时 0.059 秒（2.118 MB/秒）。`
*   继续恢复事务日志备份，直至恢复到故障发生点。
    *   SSMS 响应：`为数据库 'backrecTestDB' 的文件 'backrecTestDB' 处理了 0 页。为数据库 'backrecTestDB' 的文件 'backrecTestDB_log' 处理了 8 页。RESTORE LOG 成功处理了 8 页，耗时 0.047 秒（1.329 MB/秒）。`
*   使用 `RESTORE DATABASE backrecTestDB WITH RECOVERY;` 恢复数据库。
    *   SSMS 响应：`RESTORE DATABASE 成功处理了 0 页，耗时 0.873 秒（0.000 MB/秒）。`

请注意，我在这些命令的 `DISK` 属性中使用了虚构的名称。这样做是为了让您更容易理解我在一组备份中要查找的是哪个备份。

如果您注意清单 8-1 中的 SSMS 响应要点，在事务日志恢复部分，我在恢复差异备份之后恢复了第一个日志，然后恢复了之后的下一个日志，此时我收到了处理了 `0` 页的消息，这意味着我已经到达了适用事务日志的末尾。在此日志恢复后，可能还有更多需要恢复的日志，但在本例中，这就是需要恢复的数据的终点。在实际场景中，您可能需要在一天中的特定时间恢复数据，因此请记住，您可能需要恢复比本示例所示更多的日志。只需为您需要的事务日志恢复重写脚本即可。

考虑到这些操作步骤，可以组合出一个良好的恢复脚本，如清单 8-2 所示。

##### 清单 8-2. 恢复脚本

```sql
-- 切换到 master 数据库
USE master
GO
-- 备份日志尾部
BACKUP LOG backrecTestDB
TO DISK = 'E:\SQL Server\Logs\backrecTestDB_TLOG_TAIL.bak'
WITH NORECOVERY;
GO
-- 恢复完整备份
RESTORE DATABASE backrecTestDB FROM DISK = 'E:\SQL Server\Backup\backrecTestDB_backup_2017_10_16_000002_2492163.bak'
WITH NORECOVERY;
GO
-- 恢复差异备份
RESTORE DATABASE backrecTestDB FROM DISK = 'E:\SQL Server\Backup\backrecTestDB_backup_2017_10_16_180001_7394135.bak'
WITH NORECOVERY;
GO
-- 恢复事务日志备份
RESTORE LOG backrecTestDB FROM DISK = 'E:\SQL Server\Logs\backrecTestDB_backup_2017_10_16_190004_6842741.trn'
WITH NORECOVERY;
GO
RESTORE LOG backrecTestDB FROM DISK = 'E:\SQL Server\Logs\backrecTestDB_backup_2017_10_16_200001_9609626.trn'
WITH NORECOVERY;
GO
RESTORE DATABASE backrecTestDB WITH RECOVERY;
GO
```

运行清单 8-2 所示的脚本后，您始终可以运行清单 8-3 所示的脚本来验证您的数据库是否已重新联机。

##### 清单 8-3. 数据库状态验证

```sql
SELECT databasepropertyex ('backrecTestDB', 'Status');
```

这应该在 SSMS 的结果窗格中仅返回值 `ONLINE`。一旦看到这个，您就可以放心，所有内容都已成功恢复，并且您的数据库正在平稳运行。

#### 数据库快照

快速还原整个数据库的一个好方法是使用数据库快照。这是数据库在特定时间点的真实快照，并且与该时间点的实际数据库一致。这意味着快照充当了一种复制的数据库，因为当源数据库更新时，快照也会更新。从拍摄快照的那一刻起，每个后续事务都会在快照和源数据库中捕获。你还可以为同一数据库创建多个快照，这是一个特别好的做法。如果管理得当，这可以成为管理数据恢复的好方法。例如，如果快照存在很长时间且未被删除，它将继续增长，直到消耗掉它所在的磁盘空间。这显然不理想，因此必须维护快照。

让我在这里停一下，提请注意：数据库快照绝不能用于替代备份和还原场景。如果我们不应该将它们用于备份或恢复，那么它们的目的是什么？使用数据库快照的主要目的是为了让 `DBA` 能够快速拍摄其数据库的快照，对数据库执行可能搞砸的操作，然后测试以确保数据库更改实际有效。如果无效，那么 `DBA` 只需还原快照，他们就回到了几分钟前的状态。执行此操作的时间通常比计划还原操作、执行所有还原步骤以及确定要恢复哪些事务日志要少得多。只需还原快照，工作就完成了。

数据库快照的另一个好用途是用于报告。使用我们的命名约定，我们可以创建一个基于一天中不同时间拍摄的快照数据的标准报告。这可以用于诸如跟踪销售量或监控数据库中随时间记录的各种美元交易等目的。报告服务将获得一个滚动的 24 小时数据窗口，并将此数据量化为管理层易于阅读的仪表盘是数据库快照的一个绝佳用途。`SQL Server` 中的报告主题略微超出了本书的范围，因此我建议阅读 Kathi Kellenberger 的 *Beginning SQL Server Reporting Services*，可在 [`www.apress.com/us/book/9781484219898`](http://www.apress.com/us/book/9781484219898) 获取。对于那些正涉足 `SQL Server` 报告服务世界但不太知道从何开始的 `DBA` 来说，这是一本绝对的必备书。

##### 数据库快照如何工作？

数据库快照的概念对不熟悉的用户来说可能有些令人生畏。我听过最好的解释是，它们在理论上最接近静态完整数据库备份。创建时存在的整个数据库作为只读副本驻留在此快照中，并且该快照保持为源数据库的只读副本，直到快照被删除为止。

现在我们看到有两个不同的数据库，包含两组不完全相同的数据，我们可以开始将快照恢复到源数据库之上的过程，基本上撤销自拍摄快照以来所做的任何操作。

```sql
-- RESTORE DATABASE 语法
```

*   只要没有到数据库实例的开放连接，代码就会运行。如果有开放连接，请运行以下代码关闭连接，然后再运行清单 8-8 中的代码：

*   当您验证数据正确后，您必须删除源数据库的任何其他快照。除非只有一个快照，否则恢复操作将无法继续。
*   我们无法使用快照进行时间点恢复。您得到的就是数据库中的内容，仅此而已。因此，您必须确保要恢复的是正确的数据版本。

## 9. 完整备份与恢复解决方案

在“现实世界”中，业务上绝对有需要能够随时提供可靠备份。数据损坏的可能性几乎是必然存在的，因此经验丰富的数据库管理员必须确保其数据库能够快速、正确地恢复。备份的可用性和可靠性至关重要，无论备份是存储在远程位置，还是存储在数据库服务器本地。在我们的备份和恢复实践中，我们经常需要处理服务级别协议，或称为 SLA。这类文档涉及广泛的细节，例如备份运行的频率、数据恢复的条件以及如何恢复数据。您可能在一个不涉及这些内容的地方工作，这也没问题。例如，如果您是作为网络托管套餐的一部分提供数据库存储即服务，那么您很可能会受到一份 SLA 的约束，该 SLA 描述了该系统的可用性细节。无论您作为数据库管理员的日常职责是否受到 SLA 的约束，有一件事是不变的，那就是您对数据库及其所承载数据的健康所负有的责任。在我迄今为止所写的每一本书中，我都试图传达一个信息，本书也不例外：作为数据库管理员，我们最大的责任莫过于保护我们的数据。

根据我的个人经验，我能想到的完全使用 T-SQL 编写完整备份场景的数据库管理员，总共只有一个。此人的理由是，SQL 可以从 SSMS 外部运行，因此在 SSMS 之外可用是理想的。虽然我部分同意这一点，但另一方面我认为，我们应该利用我们掌握的工具，包括（尤其是）那些可以用于自动化任务（如备份）的工具。对我来说，让 SSMS 完成大部分工作是有意义的，但仍然准备好 T-SQL 代码，以备不时之需，比如当我们需要备份或恢复数据库而 SSMS 因故不可用时，像我那位 DBA 朋友一样的代码爱好者可以使用。因此，本章我们将重点介绍如何在 SQL Server Management Studio 中创建一个用于完整备份的维护计划，然后编写相应的 T-SQL，以备需要时使用。需要理解的是，这并不是说 T-SQL 作为经验丰富的或初级的数据库管理员的主要脚本工具没有其用武之地；相反的说法是没有意义的，因为 T-SQL 是数据库管理员与数据库通信的基础语言。通过简单的谷歌搜索可以找到许多 T-SQL 解决方案，其中特别值得一提的是 Ola Hallengren 的备份脚本解决方案，位于 [`https://ola.hallengren.com/`](https://ola.hallengren.com/)。

到目前为止，我们一直专注于备份与恢复之间的区别，并通过不同的章节在我们的脑海中将它们区分开来。现在我们需要开始将这些不同的概念连接成一个连贯的整体，而我能想到的最好方法是创建两个独立的维护计划，一个用于备份，一个用于恢复，来管理计划的不同部分。拥有独立维护计划的明显含义是，每个备份部分将有不同的时间表。

###### 理解 SQL Server 备份中的块大小与最大传输大小

块大小和最大传输大小这两个选项听起来非常相似，不是吗？它们的区别在于，`块大小` 选项允许你告诉 SQL Server：“我知道你通常取一定数量的字节，但我希望你取不同数量的字节并将它们放入备份位置。”而 `最大传输大小` 选项则允许你告诉 SQL Server：“当你将字节移动到备份位置时，我希望那个数据块是一个特定的大小。你可以在该传输大小中打包任意多的块，但一旦达到我设定的限制，你就需要再跑一趟。” 换句话说，可以将 `块大小` 想象成单个火车车厢，而 `最大传输大小` 则是整列火车。因此，`最大传输大小` 选项总是大于 `块大小` 选项。

例如，勾选 `块大小` 和 `最大传输大小` 复选框。然后在 `块大小` 下拉菜单中选择 `4096`。接下来，在 `最大传输大小` 区域输入一个数值；我选择了 `512`。最后，按 `Tab` 键移出 `最大传输大小` 区域。你看到数值变回 `65536` 了吗？接着，点击 `最大传输大小` 区域的向上箭头，它会变为 `131072`。再点击向下箭头两次，它将默认变回 `65536`，但不会变得更低。

在我们继续之前，请确保你取消勾选了 `块大小` 和 `最大传输大小` 复选框。保持它们被勾选可能不会造成太大麻烦，但我不想引入任何可能导致错误的东西。我在这里选择的选项是 `验证备份完整性` 和 `执行校验和`。图 9-10 显示了更新后的 `选项` 选项卡界面。

#### SSMS 中的完整恢复计划

本节将详细介绍如何从可用的备份文件中制定一个可靠的恢复计划。本节的好处在于，它完全依赖于存在可恢复的备份数据集；如果没有可恢复的备份，我们就无法恢复任何数据。

在 SSMS 中，没有我们可以运行的“还原数据库”任务。相反，我们需要使用用于还原数据库的 T-SQL 命令，并将该命令包装在我们能够按需执行的 SQL Server 代理作业中。我们计划实现此目标的基本结构很简单，如清单 9-13 所示。

清单 9-13. 数据恢复结构

*   使用第 5 章中已知可用的 `RESTORE DATABASE` 命令作为 T-SQL 语法的基础
*   创建一个 T-SQL 类型的 SQL Server 代理作业
*   创建当前完整备份的副本并将其重命名为 `backrecTestDB.bak`
*   正常恢复数据

我想在这里暂停一下，并快速说明一点：我们复制和重命名备份文件的原因是因为我不想破坏维护计划已建立的恢复链。我想使用相同的数据，但脚本必须在恢复脚本中引用一个静态的文件名，这就是我选择复制和重命名文件的原因。我确定处理这一方面有不同的方法，如果你有不同的处理方法，那非常好！

#### T-SQL 恢复命令

参考第 5 章，我们用来恢复数据的命令是

```
RESTORE DATABASE backrecTestDB
FROM DISK = 'E:\SQL Server\Backup\backrecTestDB\backrecTestDB.bak'
WITH REPLACE
GO
```

我们将在恢复作业中使用相同的命令，因此请记下 `FROM DISK` 的位置，因为我们在本节重命名文件时需要使用该位置。
