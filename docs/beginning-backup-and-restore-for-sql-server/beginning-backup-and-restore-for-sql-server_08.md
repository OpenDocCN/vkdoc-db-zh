# 3. 事务日志备份

在本章中，我们将重点讨论备份事务日志的不同技术。这是一个完整备份解决方案（由完整备份、差异备份和事务日志备份组成）的最后一块拼图。这一部分很重要，因为如果没有事务日志备份，就无法进行时间点恢复。事实上，除了上次差异备份中的内容之外，无法进行任何恢复，而自上次差异备份以来提交到数据库的任何数据都将永久丢失，因为这些数据仅存在于事务日志中。因此，事务日志通常被视为灾难恢复中最重要的部分，因为它们提供了自上次差异备份以来缺失的数据片段。

如果没有事务日志备份，我们将无法执行除完整或差异备份之外的任何备份（不考虑本书未讨论的备份类型，包括文件和文件组备份、镜像备份、部分备份和仅复制备份）。虽然这看起来可能不太糟糕，但重要的是要记住，必须定期备份事务日志才能保持数据库以最佳状态运行。

#### 什么是事务日志？

事务日志是一个文件，包含所有数据库事务中所有已更改或更新数据的日志。每当由于数据修改而导致数据库状态改变时，与数据库的交互都会被记录在日志中。日志中的事务按接收顺序写入，因此所有事务都是顺序的。考虑到这一点，可以根据特定的时间范围“倒回”数据库事务的状态，直到达到所需状态。这被称为时间点恢复。

注意：时间点恢复可能非常耗时，但优点是事务日志的粒度使得可以针对单个事务进行操作以实现所需结果。

事务日志还具有在相对较短的时间内变得非常大的独特特性。例如，一旦运行了差异备份，该备份就不会继续增长。类似地，事务日志将一直增长，直到它被备份。然后，这些事务日志备份将在指定的备份间隔内持续进行，只要有磁盘空间或直到它们被关闭。每个事务日志都能够恢复到最后一个差异备份，而不能恢复到任何其他备份。因此，为了确保数据损失最小化，所有备份——完整备份、差异备份和事务日志备份——必须 100% 正常运行。如果不验证我们的事务日志备份是否正常运行，随着时间的推移，事务日志最终将占据整个磁盘。显然，让事务日志占据整个存储卷不是任何人所认为的理想解决方案，因此，我们需要确保备份平稳且正确地运行。为了帮助实现这一目标，下一节将向我们展示如何正确查看 SQL Server 中驻留的事务日志状态。


#### 查看事务日志状态

有几种不同的方法可以查看系统中每个数据库的事务日志的状态或大小。最常见的查看状态的方法是运行一个简单的查询。在 `SSMS` 中打开一个新查询窗口并输入以下查询：

```
DBCC SQLPERF(LOGSPACE);
```

回顾一下，无论是完整备份、差异备份还是事务日志备份，备份的目的都是能够将数据恢复到先前的状态。最基本地说，一个适当的备份策略通过以下步骤完成：

*   规划备份策略
*   实施备份策略
*   验证备份策略

换句话说，首先你必须通过确定所需的数据可用性级别来规划备份策略。然后，你必须在 `SQL Server` 中实施该策略。最后，你必须验证备份策略是否正常运行，以及在紧急情况下备份是否可以按预期恢复。

对于事务日志备份，验证的原则比完整或差异备份更为突出，因为事务日志是灾难恢复的关键。因此，我个人倾向于稍微更多地关注事务日志，而不是完整或差异备份。这并非贬低完整或差异备份的重要性，但事务日志作为数据库中事务的直接记录，在我看来似乎重要得多。

##### 备份事务日志

正如预期的那样，我们可以通过几种方式来运行事务日志备份：编写自定义 `SQL` 脚本，或使用熟悉的 `SSMS` 界面。在本节中，我将详细介绍这两种场景的不同部分，并将其留作练习，让读者自行决定哪种方法适合他们的特定应用。需要记住的重要一点是，这些技术既可以作为维护计划的一部分使用，也可以作为提供更高数据安全级别的定制方法。不过，我始终建议使用 `SQL Server Agent` 来管理你的备份，通过一个经过适当规划和管理的维护计划来实现。

##### 通过脚本进行事务日志备份

如果我们没有在维护计划中安排备份，或者即使安排了但我们只想再创建一个事务日志备份，我们可以通过使用以下脚本手动运行事务日志备份：

```
BACKUP LOG backrecTestDB
TO DISK = 'E:\SQL Server\Logs\backrecTestDB.trn'
```

只需在脚本的首尾部分填入你的数据库名称，你就可以手动发出你自己的备份命令。如果你遇到事务日志在自动备份计划运行前不知何故已填满的问题，此脚本特别有用。

对于此脚本，可以在 `TO DISK` 之后添加属性。它们是：

*   `WITH PASSWORD = 'password'`
    *   将 `password` 替换为你选择的密码。恢复此备份时必须输入此密码。
*   `WITH STATS = X`
    *   将 `X` 替换为你选择的整数。这是已完成进度的百分比间隔。例如，`WITH STATS = 5` 将在每完成 `5%` 后显示进度。
*   `WITH DESCRIPTION = 'text'`
    *   将 `text` 替换为你希望的描述文本。

让我们稍作停顿，回顾一下刚刚介绍的两个事务日志选项之间的区别。在引用事务日志备份时，有两点很重要：事务日志可以被截断，或者可以备份事务日志的尾部。那么区别是什么？

第一个选项，`截断事务日志`，专门处理截断日志，仅此而已。截断事务日志并不会减小物理事务日志文件的大小。这个概念起初对我来说似乎违反直觉，但后来我意识到，要减小日志文件的大小，您必须收缩日志文件。截断只是删除日志文件中未使用的部分；收缩日志文件才是实际使日志变小的操作。另外需要记住的重要一点是，截断会在检查点之后（在简单恢复模型中）以及日志备份之后（在完整或大容量日志恢复模型中）自动发生。这意味着，如果已经正确建立了备份过程，截断将作为常规备份过程的一部分自动发生。如果没有正确建立备份过程，那么您的事务日志将永远不会被截断（除非手动操作），并且最终会增长到存储介质的整个大小。

第二个选项，`备份日志尾部，并使数据库处于还原状态`，是两个选项中更强大、更有用的一个。此选项允许您备份尚未备份的最新事务，然后使数据库准备就绪，以便在必要时立即还原。这被称为尾日志备份。

出于本示例的目的，请确保选择第一个选项`截断事务日志`以及`完成后验证备份`。

### 第四章 备份方案示例

本章将作为第一章（关于完整备份）、第二章（关于差异备份）和第三章（关于事务日志备份）的总结。我们将首先深入探讨如何创建一个可持续的维护计划，使用一组预设的时间表在一个主要区域集中管理我们的备份，而不是手动运行备份，或者让完整、差异或事务日志备份在不同时间运行，然后我们将运行该计划并进行审查。

#### 设置维护计划

为了自动化备份，我们需要让 `SQL Server Agent` 知道需要运行的例程以及何时运行。这是通过维护计划实现的。自动化备份计划的每个方面都可以在维护计划中进行配置，所以让我们逐步了解一个可靠备份计划的内部运作机制。

#### 总结

在本章中，我们探讨了设置完整备份、差异备份和事务日志备份计划的最终步骤。这些备份计划完全取决于相关数据库的恢复模式，并且必须设置为完整或大容量日志模式，因为简单模式不允许进行事务日志备份。

我们了解了运行经过完整性验证的备份的重要性，以及每次创建备份时未验证备份完整性可能带来的风险。

我们根据特定环境的需求，清晰且正确地定义了备份计划，并将这些需求转化为可量化的时间表，从而按预期创建备份。

最后，我们学习了如何使最终的备份维护计划更易于人类阅读，同时保持任务的复杂性。

我们将继续进入第 5 章，探讨与我们在第 1 章创建的备份场景相对应的还原场景。这是备份场景的逻辑延伸，因为如果无法还原备份，那么拥有备份就没有意义。
