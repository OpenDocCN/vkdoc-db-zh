# 1. `Apache Cassandra` 简介

`Apache Cassandra` 与典型的**关系型数据库管理系统**（`RDBMS`），如 `MySQL` 或 `Oracle` 数据库，有相当大的不同。本章将阐述为何总体上需要 `NoSQL` 数据库，以及为何需要 `Cassandra`。

此外，本章解释了 `Cassandra` 数据库的独特特性，并介绍了几个 `Cassandra` 特有的关键概念和术语。

你将学习使 `Cassandra` 变得特别的关键概念。本章还介绍了 `Cassandra` 的数据建模，并解释了关键的 `Cassandra` 术语。

本书是关于如何管理 `Cassandra` 的，因此我也解释了 `Cassandra` 管理的显著特性、用于管理 `Cassandra` 集群的工具，以及 `Cassandra` 数据库管理的基础知识。

## 什么是 `Cassandra`？

`Apache Cassandra` 是一个分布式的 `NoSQL` 数据库系统，基于 `Amazon` 的 `Dynamo` 和 `Google` 的 `Bigtable`。`Cassandra` 是一个快速、分布式的数据库，具有高度的容错性和可扩展性。它提供高可用性和线性可扩展性，这是传统关系型数据库在处理非常庞大的数据集时无法同时满足的两个目标。

云应用需要高度可扩展的后端数据库，能够跨服务器集群处理分布式的、大规模的工作负载。这些应用需要非常快的数据访问速度，以满足各种应用程序对数据存储的交互式使用以及临时查询。`Cassandra` 正是为大容量、低延迟的云应用而设计的。

`Facebook` 开发了 `Cassandra` 用于搜索收件箱，该项目于 2008 年 7 月开源。2009 年 3 月，该项目被接受加入 `Apache 孵化器`，并于 2010 年 2 月成为 `Apache` 顶级项目。`Cassandra` 这个名字源自古希腊预言家 `Cassandra`。

在以下各节中，我将简要描述 `Cassandra` 的主要特性：

*   开源特性
*   容错性
*   高性能
*   分布式数据库架构
*   高可扩展性
*   弹性特性
*   处理多种类型数据的能力

### 开源数据库

`Cassandra` 并非商业产品。它是一个纯粹的开源产品，这意味着你不仅无需付费，还可以根据自己的需求对其进行定制。然而，如果你正在使用生产环境的 `Cassandra` 集群，了解 `DataStax`——这家“`Cassandra` 公司”很重要。`DataStax` 是一家由 `Cassandra` 数据库主要创建者运营的公司。`DataStax` 提供诸如 `Cassandra DevCenter` 和 `Cassandra Enterprise` 之类的商业产品，以便你更轻松地设置和运行 `Cassandra` 集群。`DataStax` 还提供多门关于 `Cassandra` 数据建模、开发和运维的免费课程，并通过其文档、课程和会议，成为 `Cassandra` 的主要倡导者。

### 容错性

`Cassandra` 会自动将数据复制到多个节点。这与传统关系型数据库形成鲜明对比，后者需要你设置复杂的复制架构才能将数据传播到多个节点以实现容错。对于每次更新，你都可以选择同步或异步复制。

由于数据库运行在一组相似的节点上，因此 `Cassandra` 集群中不存在**单点故障**。正如你将在本书后面学到的，基于 `Cassandra` 的应用程序即使在整个 `Cassandra` 数据中心（定义与传统的组织数据中心不同）丢失的情况下也能继续运行。

### 高性能

`Cassandra` 是一个高性能的数据库，基准测试表明其性能超越了其他 `NoSQL` 数据库。在 2012 年进行的一项主要的综合性 `NoSQL` 基准测试中，`Cassandra` 在多个实验中针对最大节点数量实现了最高吞吐量。

`Cassandra` 提供非常高的吞吐量，尤其是在写入时。可预测的高性能意味着你可以满足要求低延迟的严格业务 `SLA`（服务水平协议）。

尽管工作负载增加，或向集群中添加大量节点，`Cassandra` 仍能持续提供高性能，这得益于数据库的线性可扩展性。

### 分布式配置与高韧性

`Cassandra` 安装使用多个节点，且不存在单点故障。节点之间不存在主从关系；`Cassandra` 集群中的所有节点都是相同的。

`Cassandra` 会自动将数据复制到多个节点。你可以将节点放置在多个机架中，或者，如果你使用的是像 `Amazon Web Services` 这样的云设置，则可以放置在不同的可用区中。将数据分布在多个地理区域将确保数据始终可用。

`Cassandra` 使用“数据中心”的概念，它是一组节点。数据中心可以是逻辑的或物理的，因此你可以拥有遍布全球的数据中心。这一特性，加上数据复制，提供了非常高的可用性，因为 `Cassandra` 不仅能承受一个或多个节点的丢失，还能承受整个数据中心的丢失。

### 灵活性

`Cassandra` 使用极其灵活的数据模型。一个表中的不同行可以拥有不同数量的列。用户还可以在运行时更改模式。`Cassandra` 的**可调一致性**特性允许你在读或写操作级别调整一致性。你可以根据你的主要目标是可靠性还是速度来调整复制和读/写一致性保证。

### 使用廉价服务器

与关系型数据库不同（大多数关系型数据库需要高度稳健且昂贵的硬件），`Cassandra` 可以在商用硬件上运行。商用硬件比托管 `Oracle` 等数据库的高端服务器便宜得多。虽然更容易发生故障，但商用服务器工作得非常好，因为 `Cassandra` 的架构不涉及单点故障。

### 易于管理

由于其简单的架构和缺乏过多复杂的组件，`Cassandra` 易于管理。`Oracle` 和其他关系型数据库具有许多功能，这会导致数据库臃肿，进而可能导致数据库不稳定。

与管理像 `Oracle` 数据库这样的复杂数据库相比，管理 `Cassandra` 数据库，尤其是通过像 `DataStax` 这样的框架，要简单得多。在 `Oracle` 数据库中，备份和恢复数据库、优化 `SQL` 查询以及管理复制都是复杂的任务。`Cassandra` 通过保持简单性，使你能够快速上手。

当你通过添加更多节点来扩展 `Cassandra` 集群时，需要记住的一个关键点是：你不需要为管理 `Cassandra` 的团队增加更多成员。管理 20 个节点集群的团队同样可以处理 1000 个节点的集群。因此，随着 `Cassandra` 数据库的扩展，管理成本会下降。



### 高度可扩展

Cassandra 具有高度可扩展性。要分析更多数据，您只需向部署中添加更多节点即可。Cassandra 提供**线性可扩展性**，这意味着，如果需要处理三倍增长的写入量，您只需将节点数量增加三倍。

您可以轻松扩展 Cassandra，无需任何停机或中断正在进行的工作。

大型 Cassandra 生产部署包括 Netflix（拥有超过 2,500 个节点和 420TB 数据）和 Apple（拥有超过 75,000 个节点和超过 100PB 数据）。

Cassandra 高性能的关键之一在于其**水平可扩展**性。与依赖于提升服务器处理、存储和内存能力的垂直扩展策略不同，水平扩展是向集群中添加更多节点。

随着您向 Cassandra 集群中添加新机器，读取和写入吞吐量都会线性增长；也就是说，例如，通过将集群中的机器数量翻倍，您可以使吞吐量翻倍。此外，您无需付出服务中断的代价，因为在集群规模扩展期间，数据库保持可用，您的应用程序继续无障碍运行。

### 这还是熟悉的 SQL！

最后，Cassandra 查询语言（`CQL`）的语法结构与我们大多数人已经熟悉的 SQL 语言相似，这使得开发人员和管理员可以相当容易地掌握 `CQL`。

与 Cassandra 数据库交互最简单的方式是通过 `CQL` shell，通过 `cqlsh` 工具调用。`CQL` 使用与常规 SQL 相似的语法。您可以通过 `cqlsh` 执行数据操作（DML）和数据定义语言（DDL）任务。

### 关系型数据库带来的问题

当您尝试扩展到非常大的数据量时，会遇到关系型数据库的各种问题。关系型数据库所提供的高可用性概念有时也值得商榷。

当您使用 Linux 实用程序如 `sed` 和 `awk` 或简单的 Python 脚本来分析数据时，大多数时候您甚至不需要数据库。更常见的情况是，您会处理中等规模的数据，这时可以使用诸如 `MySQL`、`Oracle`、`PostgreSQL` 或 `SQL Server` 之类的数据库。这些数据库提供众所周知的 `ACID` 保证。

随着数据量的增长，您可以通过添加更强大的 CPU 和更多 RAM 来垂直扩展这些关系型数据库，但这些关系型数据库的扩展性存在一个有限的上限，而且由于数据库不是分布式的，费用会变得失控。

### 关系数据库管理系统与大数据

关系型数据库通常采用**主从格式的复制**来解决读取密集型工作负载的问题。当您为关系型数据库（如 `MySQL`）添加复制时，`ACID` 保证就变得难以实现。例如，当您在 `MYSQL` 中复制数据时，主节点和从节点之间通常存在复制延迟。如果在从节点复制到数据之前客户端从从节点读取，则一致性保证（`ACID` 中的 C）就丧失了。

##### 由第三范式引起的性能问题

在关系型数据库中，通常将数据置于众所周知的**第三范式**中。第三范式会对查询性能产生若干影响，使您难以扩展。查询变得越来越复杂，涉及大量的连接（join）以及资源消耗大的分组操作，如 `ORDER BY` 和 `UNION`。

查询是临时的（adhoc），用户希望立即得到结果。如果您的数据集大于分配给数据库的内存，由于大量的磁盘寻道，结果将会很慢。

摆脱这种性能困境的一种方法是，在读取时创建多个反范式化（denormalized）的表，这样读取操作就不会涉及太多昂贵的连接。然而，这意味着您最终将拥有重复的数据集，这违反了第三范式。

##### 由分片（Sharding）引起的问题

分片是指，您不再拥有一个数据库，而是将数据分散到多个数据库中。

分片意味着您将所有东西都反范式化，并且不能再进行任何聚合或连接。您的数据分布在众多机器上，而聚合操作需要您访问每一个分片。添加分片需要您移动数据。由于分片会抑制性能，您被迫进行反范式化，这意味着您需要保留经常查询的关键数据的多个副本，例如一个副本基于 `Customer Id`，另一个基于 `State`。

当您有多个分片时，模式（Schema）更改是困难的，因为您必须将更改传播到所有分片。此外，如果您想将数据库的规模扩大一倍或两倍，您需要重新分片所有数据。

##### 高可用性的神话

高可用性是使用复制数据库的用户经常追求的主要好处。然而，高可用性并不简单。高可用性涉及众多活动部件，并带来巨大的开销。尽管成本高昂，高可用性仍然很难实现，因为主从环境确实会因配置更改、驱动器和电源故障等原因而频繁停机。

除了计划外停机，您还应该处理由于操作系统补丁和升级导致的计划内停机。故障转移过程，无论是自动还是手动的，都不是无误的，并且通常涉及启动从节点或备用数据库的延迟，从而导致停机。

管理跨多个数据中心运行的 `MySQL` 等关系型数据库并非易事；它需要一个由高技能数据库专业人员和 Linux 工程师组成的团队持续关注。

Cassandra 通过提供以下功能简化了您的数据库架构：

*   **对等（Peer-to-Peer）架构**：Cassandra 使用对等架构，没有主实例和从实例的概念。摒弃主/从概念意味着活动部件更少。这种策略导致了一个简单的架构，其中所有节点都是平等的。
*   **无故障转移**：对等架构意味着不存在**故障转移**的概念。数据库不必处理复杂的状态维护操作，例如领导者选举。
*   **可靠性**：数据库完全预期到故障的发生，架构中单个组件的故障永远不会成为阻碍。您的用户完全不会察觉到有一个或多个节点、甚至整个数据中心崩溃或因其他原因不可用。
*   **数据局部性**：Cassandra 旨在实现**数据局部性**，而不是收集分散在大量节点上的数据。
*   **更低成本**：Cassandra 降低了扩展的成本（使用传统关系型数据库扩展可能很昂贵）。由于 Cassandra 运行在商用服务器上，您的成本远低于使用强大服务器扩展关系型数据库的成本。您无需购买几台价值数十万美元的服务器，而是购买数百台每台仅需几千块的服务器。

Cassandra 是 NoSQL 数据库阵营中的领先成员，因此，回顾一下对 NoSQL 数据库的需求、各种类型的 NoSQL 数据库以及它们所解决的问题类型，是一个好主意。



### NoSQL 数据库的兴起

正如关系型数据库的出现是为了解决数据异常和同一数据库难以适用于新应用的问题一样，NoSQL 数据库的兴起也有其成为大数据领域重要参与者的原因。关系模型在扩展性上的失败，使其无法满足日益增长的海量读写操作需求，从而导致了 NoSQL 数据库的出现。

NoSQL 数据库可以运行在单台服务器或多台服务器上；当它们运行在多台服务器上时，被称为分布式数据库。大多数 NoSQL 数据库处理大型数据集，因此通常是分布式数据库。

与关系型数据库设计不同，在关系型数据库中，模型是核心，实体的结构和它们之间的关系驱动着数据库的设计；而在 NoSQL 数据库中，性能才是关键。你仍然会有实体和关系，但维护关系并非神圣不可侵犯；目标是为了性能。

有几种类型的 NoSQL 数据库，每种都旨在解决不同类型的问题。以下各节简要描述了 NoSQL 数据库的主要类型。

### 键值数据库

键值数据库采用基于键集合和值集合的简单数据模型。键是用于查找数据的标识符，值是与键相关联的数据。

键值数据库采用的简单模型意味着没有表以及相关的实体，如列和约束。这些数据库不支持使用 SQL 查询数据库。

亚马逊的`DynamoDB`、`Riak`、`Oracle BerkeleyDB`，以及缓存数据库如`Redis`和`MemcacheD`都是键值数据库。

### 文档数据库

文档数据库存储文档，即你以灵活结构存储的数据项集合。文档只是一个数据结构，数据库将其存储为字符串或其二进制表示。这里的文档是半结构化实体，采用标准格式，如`JavaScript Object Notation (JSON)`或`Extensible Markup Language (XML)`。

文档数据库可以在单个文档中存储多个属性，而不是像键值数据库那样为每个实体属性使用单独的键。

与键值数据库类似，文档数据库使用标识符，但其值（数据库存储为文档）通常比键值数据库中的简单键更复杂。

你不需要在向文档数据库添加数据之前定义模式。添加文档将创建支持该文档所需的必要数据结构。由于没有固定模式，与关系型数据库相比，开发者在使用这些数据库时拥有更大的灵活性。

流行的文档数据库包括`MongoDB`、`CouchDB`和`CouchBase`。

### 列族数据库

列族数据库借鉴了关系型数据库的一些特性，将数据组织成列的集合。然而，为了提升性能，它们牺牲了关系型数据库的一些基本功能，例如连接表的能力。

列族数据库中的行可以有不同的列。当你处理大量列时，可以将列分组为相关列的集合。这些相关列的集合称为列族。一个列族数据库包含数百万列并不罕见。

与文档数据库一样，列族数据库不需要你预先定义固定的模式。

通常，列族数据库是反范式化的，你可以在一个宽行中找到关于一个对象的所有信息。`Cassandra`中的一行可以拥有多达二十亿个列。用于处理列族数据库的查询语言类似于 SQL，并使用类似 SQL 的语句，如`SELECT`、`UPDATE`和`INSERT`。

`Apache Cassandra`和`Apache Hbase`（通常与`Hadoop`发行版捆绑）是两个著名的列族数据库。

### 图数据库

图数据库是一种专门的 NoSQL 数据库，它使用称为节点和关系的结构（正式术语中称为顶点和边），而不是用列和行来建模数据。图数据库专为对对象及其间关系进行建模而设计。

图数据库与你通常称为图表的可视化工具无关。它们的名字来源于图论，这是数学的一个分支，通过将对象表示为顶点并将它们之间的关系表示为边来研究对象。

图数据库对对象间的邻接关系进行建模。图数据库中的一个节点包含指向与它相邻的对象的指针。这些指针使得需要遍历图中路径的操作能够快速执行。

图数据库中的节点是一个具有标识符和一组属性的对象。两个节点之间的关系是一个包含该关系属性的链接。

在涉及节点间多条路径的模型中，图数据库使你能够高效地查询数据。使用数据的表格表示来查询这类数据要困难得多，因为那些格式需要递归 SQL 结构，例如`Oracle`的`CONNECT BY`子句来查找路径。

图数据库可以表示现实世界中的广泛实体。例如，一个城市可以是一个节点，关于城市间距离和旅行时间的信息可以存储在城市之间的关系中。

`Neo4j`是一个流行的图数据库。

### Cassandra 的特殊能力

在接下来的章节中，我将重点介绍`Cassandra`的一些特殊能力，这些能力使其有别于其他 NoSQL 数据库。当然，这些能力并非`Cassandra`独有，因为一些替代方案也提供了其中某些功能，但没有其他数据库能在提供所有这些功能方面与之相提并论。

### 分布式与去中心化数据库

`Cassandra`是完全分布式的。`Cassandra`集群中的每台机器在数据处理方面承担相等的负载。在诸如`Hadoop`的大数据框架中，情况则大不相同。有一个名为`NameNode`的主节点，负责管理所有数据的读写。同样，`MongoDB`有它的`mongos`，与常规节点相比需要特殊处理，因为它们可能成为瓶颈。

与其他集群相比，在`Cassandra`集群中进行安装和故障排除要容易得多，因为所有节点在功能上都是相同的。`Cassandra`的对等策略意味着增加容量是一项简单的操作。

你可以通过将一些节点设置为主节点，其他节点设置为从节点来扩展`MySQL`等数据库。`Cassandra`被设计为一个去中心化数据库，这意味着所有节点都是相同的；没有主/从节点的概念。这也意味着没有单点故障，因为没有特殊的主机，无论节点是否发生故障，集群都能继续运行。

### 日志结构存储引擎

`Cassandra`使用一个高效的日志结构引擎，将更新操作转化为顺序 I/O。

`Cassandra`的存储引擎永远不需要或读取现有数据。它只追加更新后的数据，从而使更新操作非常快速。



### 本地管理存储

Cassandra 在本地管理所有存储。初看之下这似乎微不足道，但与 `Hbase` 等依赖底层存储系统（如 `HDFS` Hadoop 分布式文件系统）进行复制的 NoSQL 数据库相比，Cassandra 提供了更多功能。

你可以使用本地管理存储策略，将 Cassandra 的 `commitlog` 放置在独立的磁盘上。你还可以在同一个集群中混合使用 `SSD`（固态硬盘）和 `HDD`（机械硬盘）。

你可以根据访问模式减少 I/O，以降低使用极快存储的成本。DataStax 提供了 `DSE` 分层存储，允许你将减少的 I/O 转化为存储端的成本节约。

`DSE` 可以透明地将老化数据从快速的 `SSD` 层移动到较慢的存储层。时序数据和社交互动数据都是可以从 `DSE` 分层存储中受益的数据类型的良好示例。

### 处理多种类型数据的能力

Cassandra 能处理各种类型的数据，例如结构化、半结构化和非结构化数据。

### 面向行

尽管 Cassandra 属于列式数据库类型（其数据存储按列组织），但它实际上是一个分区数据存储，"分区"指的是数据库对每一行使用唯一键，将行分布到多个节点上。它以稀疏哈希表的形式存储数据，"稀疏"暗指并非所有行都拥有相同的列。

Cassandra 最有趣和最显著的特性之一是 `tunable consistency`（可调一致性），我将专门用一整节来讨论数据库一致性，因为透彻理解这个主题至关重要。

### 一致性、ACID 要求、CAP 定理与 BASE

一致性是数据库的一种属性，确保读取操作总是检索到最新的值。所有拥有相同数据的节点，在写入后应处于一致的状态。

与提供严格一致性以确保事务总是正确的关系型数据库不同，NoSQL 数据库提供了一种称为 `eventual consistency`（最终一致性）的机制。

在 `NOSQL` 数据库中，一致性和可用性之间存在权衡。Cassandra 牺牲了一些一致性以提供高可用性。更准确地说，应该说 Cassandra 提供了 `tunable consistency`（可调一致性），这意味着你可以选择最适合你的一致性级别，并在可用性方面做出相应妥协。

要理解 Cassandra 的这一关键特性 `tunable consistency`，深入探究传统关系型数据库的 `ACID` 要求基础、Brewer 的 `CAP` 定理以及 `BASE` 特性是很有帮助的。

### ACID 要求

与其他文档数据库以及广泛归属于 NoSQL 数据库范畴的其他数据库一样，Cassandra 不支持关系型数据库所提供的著名 `ACID` 特性。`ACID` 特性是所有现代事务型数据库（如 `MySQL` 和 `Oracle`）的标志，指以下一组原则：

*   `Atomicity`（原子性）：这是"全有或全无"的原则，要求如果事务中哪怕有一个元素失败，整个事务也将失败。只有当数据库成功执行所有任务时，事务才算成功。
*   `Consistency`（一致性）：此属性确保数据库完全完成所有事务，要求数据库在事务开始和结束时都必须处于一致的状态。
*   `Isolation`（隔离性）：每个事务都应独立于其他事务。任何事务都不应访问其他处于未完成状态的事务。
*   `Durability`（持久性）：一旦事务完成，它就是"永久的"。也就是说，数据库会将事务记录在持久化存储中，确保事务在系统故障（如断电或磁盘故障）后仍然存在。

虽然 `ACID` 要求在许多年里很好地服务了传统关系型数据库，但诸如非结构化数据、非关系型数据等非关系型数据的流行，以及分布式计算系统的激增，引发了对数据库必须满足的事务特性的新看法。

### CAP 定理

一致性、可用性和分区容错性（`CAP`）定理旨在完善你在现代分布式计算系统中实施应用程序所需满足的要求。`CAP` 定理代表以下原则：

*   `Consistency`（一致性）：与 `ACID` 的一致性属性相同。满足此要求意味着数据存储的所有客户端都获得"合理"的响应。如果客户端 A 向节点 X 写入 1 然后写入 2，另一个客户端 B 不能先读取到 2 然后读取到 1。
*   `Availability`（可用性）：系统在被请求时必须可用。对数据存储的所有操作最终必须成功返回。
*   `Partition Tolerance`（分区容错性）：如果网络无法在两组节点之间传输消息，系统必须继续正确工作。分布式系统中单个节点的故障不得导致整个系统故障。即使存在部分数据丢失或局部系统故障，系统也必须可用。

问题是，在任何给定时间，分布式系统通常只能支持这里列出的三个要求中的两个。这意味着在使用 NoSQL 数据库等分布式数据存储时，权衡通常是不可避免的。这就是为什么我们有那句名言："一致性、可用性、分区容错性，三者选其二。"

大多数现代分布式数据存储，如 Cassandra，提供 `availability`（可用性）和 `partition tolerance`（分区容错性），而牺牲了 `consistency`（一致性）。这里的理念是，应用程序短时间的异常行为比数据库短时间不可用更可取。

如果两个节点无法通信，而你要求严格一致性，系统将表现为宕机。像 `Cassandra` 这样的数据库选择在网络分区（当节点间网络丢弃消息时）期间保持高可用性，而不是满足一致性原则。

严格一致性，特别是当你的数据中心分布在全球各地时，由于数据中心之间不可避免的延迟，你可能永远无法满足。可用性是分布式数据库可以尝试满足的一个更实际的原则。

当然，出于可靠性考虑，满足 `availability`（可用性）和 `partition tolerance`（分区容错性）要求至关重要。这意味着 `consistency`（一致性）要求常常面临风险。然而，像 Cassandra 和 Amazon 的 `DynamoDB` 这样的领先 NoSQL 数据库能很好地处理一致性损失。如何做到的呢？这是由于这些数据库采用了称为 `BASE` 的系统，这是一套修改后的 `ACID` 要求，以适应现代 NoSQL 及相关的非关系型数据库。`BASE` 代表如下：

*   `Basically Available`（基本可用）：系统保证数据的可用性，即它会响应任何请求。然而，响应可能是一个"失败"（未能获取请求的数据集），或者返回的数据集可能处于不一致或变化的状态。
*   `Soft state`（软状态）：系统的状态始终是"软"的，意味着"最终一致性"（最后一个要求）可能在任何给定时间导致系统状态发生变化。
*   `Eventually Consistent`（最终一致性）：一旦系统停止接收新的数据输入，它最终会变得一致。只要系统在接收输入，它在转向下一个事务之前不会检查每个事务的一致性。

支撑 Amazon 购物车技术的 `DynamoDB` 强调高可用性，意味着它可以在一致性方面放宽要求。这类数据库摒弃了传统意义上支持一致性所需的复杂查询，转而追求最终一致性的目标。
