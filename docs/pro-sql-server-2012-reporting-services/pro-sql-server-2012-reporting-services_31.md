# 第 10 章

## 管理报表

在 SSRS 之前的许多报告解决方案中，报表管理通常仅限于通过文件共享或嵌入到第三方应用程序中将完整的报表文件交付给最终用户。SSRS 是一个完整的报告环境，具有诸如计划报表执行、报表订阅服务、快照、内容缓存和按需 Web 访问等功能。

这些额外的好处也带来了管理责任的额外层面。根据组织的规模，一些管理任务可以委派给其他用户，例如部门经理（他们可能会维护其部门的报表文件夹），以及系统管理员和 DBA。幸运的是，SSRS 提供了多种在各个级别管理报表服务器的方法。在本章中，你将继续使用内置工具，如报表管理器和命令行实用程序，以及利用 SSRS 编程模型的自定义 .NET 管理工具来管理 SSRS 部署。我们还将演示 SSRS 的“Reporting Services 配置管理器”。当连接到 SSRS 实例时，它提供了报表管理器“站点设置”部分和命令行实用程序中可用的大部分（如果不是全部）功能。使用 SSRS 2000 和 2005 时，某些管理任务（如配置报表订阅）可以通过 SSMS 完成。然而，自 2008 版本以来，此功能已被移除，现在只能使用报表管理器。因此，我们选择在本章的大多数示例中使用报表管理器。报表管理器可以通过 Web 浏览器访问，不需要在本地安装 SQL 客户端工具。



## 探索 SSRS 部署中的管理角色

您可以将 SSRS 部署的管理角色细分为三个基本类别：

*   内容管理
*   性能
*   报表执行

在将任何应用程序或服务投入生产环境之前，始终进行测试部署是最佳实践。由于我们公司通过互联网为大量用户提供服务，一些特殊的管理考虑——例如如何为不同公司订阅同一份报表——变得至关重要，我们需要测试各种场景以确保在每种情况下功能都正常。让我们首先看看这三个管理类别，以及我们如何实施和测试它们，以确保在部署到生产环境时，能尽量避免（或至少减少）意外情况。我们将展示如何使用 `SSRS` 和 `SQL Server 2012` 的内置管理工具来执行所有这些任务。本章稍后部分，我们将展示如何用 `.NET` 构建一个管理应用程序界面，以提供安排报表计划的能力。

### 内容管理

有效管理 `SSRS` 报表服务器上的内容，要求您熟悉可用的管理任务。报表管理的几个方面只有在报表部署后才可用。在我们展示如何为选定用户部署医疗保健报表时，我们将详细涵盖每个方面：

*   共享计划
*   报表参数和数据源
*   报表快照、历史记录和缓存
*   订阅

对于每个内容管理任务，我们将提供具体的现实场景，继续以医疗保健机构为例。至此，您已经部署了多个报表、数据源以及其他报表项，例如在本书中开发的图形图像和代码。现在，是时候戴上管理员的帽子，充分利用所有使 `SSRS` 成为一个独特而强大的报表传递系统以及丰富报表创作环境的功能了。

## 设置共享计划

一般来说，共享计划类似于共享数据源，因为它对有权访问它的用户在整个系统范围内可用。您可以专门为某个作业类型创建共享计划。可以将重复的共享计划配置为按小时、天、月、周执行，或仅运行一次。在此示例中，财务报表将在每月末执行。为这些报表维护历史记录非常重要，这样您就可以在任何时间点冻结值，或用于审计功能，例如了解哪些用户查看了报表以及他们在报表中看到了什么。

您将创建一个共享计划，用于在每月的最后一天运行财务报表，以提供以下好处：

*   您可以将报表安排在预定时间执行。
*   您可以存储每份报表的快照，以维护数据的历史视角。

`注意` 报表快照是在特定时间执行的报表，无论是由用户启动还是作为计划的一部分执行，它们共同构成了报表历史记录。我们稍后将更详细地介绍快照。

客户可能在月末运行的一份财务报表是 `AR 对账` 报表。此报表列出了当前会计期间（例如 2009 年 10 月 的 10-2009）发生的财务交易。此报表可能是需要按相同计划执行的几份报表之一。其他财务报表可能包括 `账龄试算平衡表` 报表和 `AR 账龄分析` 报表。我们将在下一节中使用 `AR 对账` 报表来展示与设置共享计划和创建报表快照相关的管理任务。我们将 `AR 对账` 报表包含在 `Pro_SSRS` 报表项目下载中，该下载可在 Apress 网站的 Source Code/Download 部分获取 ([`www.apress.com`](http://www.apress.com))。报表定义文件是 `AR Reconciliation.rdl`。在本节中，我们将展示如何使用已部署的 `AR 对账` 报表版本来配置其参数值，以使其与每月计划执行时间保持一致。

要部署 `AR 对账` 报表，请在 `BIDS` 中打开 `Pro_SSRS` 项目，然后打开 `解决方案资源管理器`。但是，在部署此报表之前，您需要更改控制报表将部署到报表服务器上哪个文件夹的项目属性。在 `AR 对账` 示例中，您将把报表部署到名为 `月末财务` 的文件夹中。在 `解决方案资源管理器` 中，右键单击 `Pro_SSRS` 报表项目，然后选择 `属性`。将 `TargetReportFolder` 值从 `Pro_SSRS` 更改为 `月末财务`，然后单击 `确定`。现在，当您在 `解决方案资源管理器` 中右键单击 `AR 对账` 报表并选择 `部署` 时，将在您的报表服务器上创建包含 `AR 对账` 报表的新文件夹。

## 创建共享计划

要在 `报表管理器` 中创建共享计划，请单击 `站点设置`，然后在页面左侧导航选项卡的底部选择 `计划`。单击 `新建计划` 按钮链接，并将计划命名为 `月末财务`。在 `计划详细信息` 部分，您将看到标准的计划选项：`小时`、`天`、`周`、`月` 或 `一次`。对于此报表，请选择 `月`。

配置共享计划在每月最后一天运行时的第一个挑战是克服 `计划` 表单上的内置数据验证。虽然可以告诉 `SSRS` 在每个月的最后一个星期日执行报表，但无法选择每个月的最后一天，因为最后一天是可变的（即，根据月份不同，可能是 28、29、30 或 31 日）。嗯，创建一个涵盖所有四个日期的单一计划应该是可行的，对吧？并不完全正确！如果您尝试如 图 10-1 所示保存 `共享计划`，选择 31 日并在选中所有月份时会导致错误，因为并非所有月份都有 31 天。

`注意` 您必须以属于 `SSRS 2012 系统管理员` 角色成员的用户身份登录才能添加新计划。如果您尚未被添加为站点的 `系统管理员`，但您是本地 `管理员` 组的成员，则可能需要使用提升的权限运行浏览器。如果使用 `IE`，您可以右键单击 `IE` 图标并选择 `以管理员身份运行`。

本例中的解决方案——因为我们知道在每月最后一天午夜 12:00 之后不会有任何活动发生——是将计划设置为在每月第一天的凌晨 12:01 运行。这实质上相当于选择了每个月的最后一天，并且您知道每月的第一天总是 1 日。

![Image](img/9781430238102_Fig10-01.jpg)

*图 10-1. 选择每月最后一天*

接下来，为计划选择合适的开始日期和结束日期（本例中，不要指定结束日期），然后单击 `确定` 按钮。现在您可以继续前进，准备报表本身以使用新计划。请注意，`SSRS` 中的计划依赖于 `SQL Server Agent` 服务，因此，在您能够成功创建计划之前，该服务必须正在运行。

`注意` 请注意，当在 `SSRS` 中计划运行作业（例如订阅或执行快照）时，会使用 `SQL Server Agent` 创建一个 `SQL Server` 作业。可以通过 `SSMS` 或 `报表管理器` 监视作业。



## 配置报告以使用共享计划

AR 对账报告包含可用于限制报告显示数据的报告参数。其中两个参数 `AcctPeriodYear` 和 `AcctPeriodMonth` 在驱动存储过程中用于将财务数据限制在特定的会计期间。当从共享计划执行 AR 对账报告时，确保所有报告参数都具有默认值非常重要。首先，AR 对账报告中的所有八个参数都被设置为允许 `NULL` 值，如 图 10-2 所示，这导致执行时参数的默认值被设为 `NULL`。这是可以接受的，因为使用这些参数的主存储过程逻辑知道在传递 `NULL` 值时返回所有数据。

![Image](img/9781430238102_Fig10-02.jpg)

*图 10-2. AR 对账报告参数*

然而，对于参数 `AcctPeriodYear` 和 `AcctPeriodMonth`，你需要添加默认的非 `NULL` 参数值，以便返回的数据集仅包含当前会计期间的记录。为此，你使用两个函数：

> *   `DATEPART`: 返回一个表示日期组成部分（如年、季度、月或日）的整数
> *   `TODAY`: 返回当前日期

在表达式中一起使用这些函数，可以为这两个参数设置所需的默认值。对于 `AcctPeriodYear`，只需将默认值设置为当前年份，如下所示：

`=DATEPART("yyyy", TODAY())`

对于 `AcctPeriodMonth`，本质上做同样的事情，但需要从表达式值中减去 1 以返回正确月份的数据。例如，表达式 `=DATEPART("m", TODAY())`，如果在 11 月 1 日凌晨 12:01 运行，将返回值 11，而当前的会计期间是 10 月。为了考虑一月份的情况，你添加一个 `IIF` 函数来评估 `DATEPART`，这样当返回 1（表示一月）时，函数返回 12，让报告知道处理 12 月的会计期间。因此，正确的表达式如下：

`=IIF(DATEPART("m", TODAY()) = 1, 12, DATEPART("m", TODAY()) -1)`

![Image](img/square.jpg) **注意** 我们的示例数据库仅包含来自过去某个时间点的数据快照。因此，如果你正在按照示例操作，则需要输入示例数据库中包含的年份和月份。大多数记录介于 2009 年 9 月至 2009 年 12 月之间。然而，在需要月末频率的报告的真实场景中，请使用上述表达式。

## 为报告历史创建快照

AR 对账报告的目标是允许其在每月的指定时间（出于性能考虑，选择非高峰时段）执行，并保存每月处理的历史记录。为此，我们在本章前面创建了“月末财务报告”共享计划。现在是时候利用 SSRS 的一个有益功能了，即能够将报告作为快照处理。

快照是报告的静态“时间点”副本。SSRS 中使用两种类型的快照：作为执行快照生成的（从预先执行的时间点渲染单个报告），以及为存储在报告历史中而生成的（可以包含在给定时间点的多个报告副本）。在本节中，我们将重点展示如何配置 AR 对账报告以生成报告历史的快照，从而可以生成一系列历史财务报告。

你将使用`报表管理器`来配置 AR 对账报告的历史属性，以便每次报告按照“月末财务报告”计划处理时，都会生成一个报告的快照。

让我们从查看存储报告历史的可用设置开始。如图 10-11 所示，几个设置不仅影响快照的创建，还影响快照在报告历史中的存储方式。在`报表管理器`中，你可以通过将光标悬停在报告上、点击向下箭头并选择`管理`（如前面导航到属性时所述）来导航到“快照选项”屏幕。进入报告属性屏幕后，点击左侧的`快照选项`选项卡。

![Image](img/9781430238102_Fig10-11.jpg)

`图 10-11.` 报告快照选项

对于 AR 对账报告，设置快照选项如下：

*   用户将无法为报告历史创建快照。将此项设置为仅由计划运行的报告才会出现在报告历史中。

## 处理报告与执行缓存

`AR Reconciliation`报告现已设置为按计划交付、从快照渲染并保存在报告历史记录中。员工也会按需运行此报告，这意味着用户可以查看包含最新数据的报告。由于这可能是一份资源密集型的财务报告，您应确保在多个用户同时渲染报告时，性能不会受到影响。您将使用`报告管理器`来配置控制报告处理方式的设置。

第一步是导航到`AR Reconciliation`报告的`属性`选项卡，然后在左侧窗格中选择`处理选项`。图 10-14 显示了可用于报告处理的设置。

![Image](img/9781430238102_Fig10-14.jpg)

*图 10-14. 处理选项属性*

第一个处理选项“始终从最新数据运行此报告”具有多个控制报告缓存的设置。

缓存是`SSRS`的一项功能，允许存储报告的临时副本并将其呈现给用户。创建报告缓存副本的主要优势是性能提升。访问设置为缓存的报告的用户会生成第一个缓存实例。如果满足某些条件，每个后续用户都会收到该缓存副本。用户收到缓存副本的条件如下：

> *   后续用户必须在缓存报告设置为过期的时间间隔内访问报告。当缓存报告过期后，下一个用户必须创建一个新副本。
> *   如果缓存报告具有在后续执行期间更改的参数，并且每个用户基于该参数接收新报告，那么生成的每个报告都会成为特定于该参数值的缓存副本。
> *   报告的数据源未设置为`Windows 身份验证`或未设置为提示用户登录凭据。

对于我们的报告，指示`SSRS`缓存报告的临时副本，并在十分钟后使缓存副本过期。十分钟通常是保留缓存报告的一个良好时长，尽管这确实取决于您数据的时间敏感性。缓存报告中的数据当然会反映报告被渲染的时间，而不是当前时间，而且除非在报告上打印执行时间，否则用户无法知道他们正在查看的是缓存报告还是实时报告。

第二个选项“从报告快照渲染此报告”不应与创建报告历史记录的快照混淆。执行快照与历史记录快照不同，它可以从`报告管理器`中的`报告`文件夹查看，就像查看按需报告一样。另一方面，历史记录快照是从报告的`报告历史记录`选项卡查看的，并且可以积累许多副本。

快照不会像缓存报告那样过期；相反，它们会在指定的时间间隔刷新。如果您选择为报告生成执行快照，则该报告无法被缓存。在此示例中，请不要选择此选项。

最后一个报告执行选项设置报告的超时间隔，可以是默认设置（通常为 1,800 秒，即 30 分钟），也可以是指定值。这是一个重要的设置，因为长时间运行的报告会使用宝贵的系统资源。对于`AR Reconciliation`报告，您将使用默认值。

我们提到了在使用具有时间敏感性数据的报告缓存时涉及的权衡。选择使用快照或缓存报告时的另一个重要考虑因素是磁盘空间。随着时间的推移，设置为较长过期时间的历史记录和缓存报告可能会积累起来。然而，与随之而来的性能提升和生产力提高相比，磁盘存储的成本微不足道，不应阻止任何人利用`SSRS`的这些有益功能。

## 管理数据驱动订阅

标准订阅满足了许多公司为其员工和客户设置自定义订阅的需求。然而，您可以使用另一种更灵活的方法来传递报告：数据驱动订阅。数据驱动订阅允许管理员或内容管理器查询数据源（例如 `SQL Server` 表），以返回符合特定条件的订阅者列表。这是向广泛的订阅者列表传递报告的理想方式。此外，您只需为所有订阅者管理一个订阅，并且这些订阅者可能具有用于生成个性化报告的不同参数。

我们知道我们希望让客户及其员工能够利用数据驱动订阅，幸运的是，我们很久以前就构建了我们的应用程序数据库，以包含正好用于此目的的员工信息。通过存储员工的电子邮件地址以及其他数据（如地理位置和认证），我们拥有了通过电子邮件向出差员工提供灵活传递系统所需的一切。我们最初的目标员工是临床医生，他们有每日要查看的患者日程安排。大多数临床医生在日常工作中使用笔记本电脑或 `PDA`。


## 设计订阅查询

第一步是重新设计一个`SSRS`报表，使其能够为临床员工提供每日日程安排，并对其进行参数化处理，以便每次执行时都能根据特定员工进行定制。作为数据驱动订阅的一部分，该报表将被处理并以可嵌入、可打印的格式（如`PDF`或`WORD`）发送给员工，如果他们在线，也会提供一个连接到`SSRS`报表服务器的链接。图 10-21 展示了我们创建的报表——“每日活动”。

![Image](img/9781430238102_Fig10-21.jpg)

**图 10-21.** 每日活动报表

由于数据驱动订阅正是基于数据，因此一个驱动订阅的查询至关重要。设计查询时包含选择性条件非常重要，因为`SSRS`会为数据源返回的每条记录交付一份报表副本。

对于我们的收件人列表，我们使用清单 10-2 中的查询。本质上，该查询返回所有拥有电子邮件地址且在报表执行日期的次日有已安排活动的员工。该报表在下班后进行处理和交付。在我们的环境中，员工的日程在晚上 9 点之后变更并不常见，因此我们将报表设置为在该时间执行。

**清单 10-2.** 用于返回订阅者列表的 T-SQL 查询

```sql
SELECT DISTINCT
        EmployeeTblid, Email, HWUserLogin, ActivityDate
FROM
        Employee E (NOLOCK)
        JOIN Activity A (NOLOCK) ON E.EmployeeTblid = A.ProviderID
WHERE
        E.Email IS NOT NULL
        AND A.ActivityDate BETWEEN GETDATE() AND GETDATE() + 1
```

当这个查询执行时，其输出会产生六行数据，如表 10-1 所示，这表明有六位临床医生在次日有活动。格式化和比较日期时间值的方法有很多。然而，在本例中，使用 `GETDATE` 函数将当前日期与 `ActivityDate` 字段值进行比较是最佳选择。有必要使用 `BETWEEN` 和 `GETDATE`，因为 `ActivityDate` 值的时间部分默认为 `00:00:00`，而 `GETDATE` 返回的是当前时间。这两种值在一对一比较时无法匹配。

> **注意** `Activity` 表中的数据是静态的，因此在使用 `GETDATE()` 函数时可能无法按预期工作，因为在您运行查询时可能没有行可以返回。您可以轻松地更新活动日期以反映您当前的日期时间，或者修改查询本身，使其回退 (n) 天，其中 n 代表要回退的天数。

## 设计日志报告

我们知道需要另一份包含所有执行日志信息且便于管理员分析的报告。因此，我们利用单次查询的数据，创建了一份名为**报告执行日志**的矩阵式报告。

要根据执行日志中的信息衡量性能，你需要掌握几项统计数据：

> *   **检索数据总时长**：数据检索花了多长时间？
> *   **处理总时长**：报告处理花了多长时间？
> *   **渲染总时长**：报告渲染花了多长时间？
> *   **字节数**：报告包含多少字节？
> *   **行数**：报告包含多少行数据？

此外，了解报告的执行时间也很有用。对于矩阵中的行分组，我们希望看到执行了哪份报告、由谁运行报告，以及报告来自哪台客户端机器。对于列分组，我们希望提供两种选择：渲染格式（例如 HTML 4.0、Word 或 PDF）或源类型（即 SSRS 如何生成报告，如实时、缓存或快照）。`Source_Type` 是一个需要监控的重要字段，因为 SSRS 生成报告的方式直接影响性能。从缓存副本或快照（两者都是报告的预处理副本）生成报告，对性能是有益的。如果 SSRS 总是为用户生成实时或按需报告，性能可能会受到影响。

为了实现矩阵中的动态列分组，你将使用一个名为 `Column_Group` 的参数，该参数接受查询中字段名的值：`Format` 或 `Source_Type`。你将使用 `Format` 作为默认值，以便在预览时自动渲染报告。列分组和标题值都将使用以下表达式，根据参数使列动态变化：

```
=Fields(Parameter!Column_Group.Value).Value
```

当报告被渲染时，如图 10-27 所示，它将默认使用 `Format` 字段，但你可以通过将参数下拉选项更改为 `Source_Type` 来动态改变它。

### 监控性能

当然，没有人愿意经历这样的挫折：在开发环境中构建了一个稳固的报告解决方案，却在部署给广大用户时发现它无法承受压力。通常，最佳实践是对服务器施加模拟负载，以便更好地了解系统的运行情况。同样，当你推出一个完整解决方案时，常见的做法是分批次、分阶段地向有限数量的用户推出功能。这正是我们在在线模型中所采用的方法。

推出策略还应包括一份计划，明确哪些报告将提供按需访问，哪些将通过报告快照或订阅提供，正如你在本章前面所做。结合高峰时段和非高峰时段的报告处理策略，将极大地提升性能。SSRS 性能的另一个考量点在于分离 SSRS Web 服务和数据库服务的负载。也就是说，如果整个 SSRS 安装都位于同一系统上，可能会对性能产生负面影响。

在本节中，我们将展示我们进行压力测试的结果，该测试访问了两个独立服务器 `RS05` 和 `HWC04` 上的两个报告服务器实例。许多工具（如 SSRS）可用于对 Web 应用程序进行压力测试；幸运的是，Visual Studio 2010 的 Ultimate 版内置了 Web 压力测试工具，我们使用它对两台服务器施加了模拟负载，最多可模拟 250 个虚拟用户。你可以在 [`www.microsoft.com/visualstudio`](http://www.microsoft.com/visualstudio) 了解更多关于该 Web 测试工具的信息。

我们还将向你展示如何使用 `rsconfig` 将 SSRS 服务器加入 Web 场，以了解将资源卸载到另一系统将如何提升性能。Reporting Services 配置管理器也可用于此目的。

我们首先对 `RS05` 运行了 15 个用户的模拟负载，所有用户都执行单个报告。我们通过任务管理器监控服务器，快速评估了性能影响。幸运的是，当对 Reporting Services 服务器运行模拟负载时，并没有造成太大的性能负担。然而，在某些情况下，你可能会遇到 CPU 达到极限的情况。在这些情况下，你很可能会发现占用处理器的单个进程是 SQL Server 和 `ReportingServicesService.exe`。图 10-28 显示了测试运行期间，CPU 使用率峰值在 25% 到 85% 之间跳跃。

我们知道我们的测试用 SSRS 服务器是一个四处理器系统，拥有超过 3GB 的内存。在你的 CPU 使用率达到极限的情况下，这告诉我们瓶颈在于 CPU。由于我们 SSRS 的生产部署不会镜像 `RS05` 的设置（换句话说，生产服务器将是至少拥有 32GB 内存的高端系统），我们可以将这一点考虑在内。

然而，还有一个因素会对生产和测试环境之间的性能差异产生重大影响。在测试环境中，SSRS 服务和 SQL Server 位于同一系统 `RS05` 上。如果我们将 SSRS 服务配置为使用远程 SQL Server 实例作为其数据库会怎样？虽然通过网络访问 `ReportServer` 数据库而不是本地数据库可能会导致任何性能下降，但如果 CPU 利用率百分比下降到更易于管理的水平，这种影响将可以忽略不计。

![Image](img/9781430238102_Fig10-28.jpg)

**图 10-28.** 任务管理器

如果你有两台 SSRS 服务器，那么将 SSRS 服务器从一个实例移动到另一个实例是很简单的。我们在测试环境中有两台 SSRS 服务器，`RS05` 和 `HWC04`，因此移动相当容易。为了指示 `RS05` 上的 SSRS 服务使用 `HWC04` 上的 SSRS 数据库，我们使用了命令行实用程序 `rsconfig`。当首次将一个 SSRS Web 服务加入使用相同 `ReportServer` 数据库的 Web 场时，需要使用 `rsconfig` 命令或 Reporting Services 配置管理器。`rsconfig` 命令的语法如下：


## 安排报表

现在您有了可用调度选项的列表，您需要添加一些代码来处理用户选择根据共享计划之一或根据快照创建来计划报表交付的情况。为此，您将使用报表服务 Web 服务的另一个方法，`CreateSubscription`。API 的 `CreateSubscription` 方法接受六个参数：

> *   `Report`：要为其创建订阅的报表的完整路径名。
> *   `ExtensionSettings`：一个传递扩展，包含特定于该扩展的设置列表。SSRS 2012 附带两个内置扩展：电子邮件传递扩展和文件共享传递扩展。
> *   `Description`：显示给用户的有意义的描述。
> *   `EventType`：触发订阅的事件类型。有效值为 `TimedSubscription` 和 `SnapshotUpdated`。
> *   `MatchData`：与指定的 `EventType` 参数关联的数据。此参数由事件用于将订阅与已触发的事件进行匹配。
> *   `Parameters`：一个 `ParameterValue[]` 对象数组，包含报表的参数列表。

在您的报表调度器中，您将创建一个新方法 `ScheduleReport`，每当用户选择安排报表时都会调用此方法。此方法将这些参数设置为适当的值，然后调用 SSRS 2012 报表服务器 Web 服务的 `CreateSubscription` 方法。大多数值只是字符串，设置起来很简单。

检查用户是否选择了订阅，如果是，检查它是基于共享计划还是快照。您将据此设置 `EventType`。如果用户选择了共享计划，则将变量 `matchData` 设置为 `ScheduledlD`。如果不是，则将该变量设置为 `null` 以告诉 SSRS 2012 基于快照触发它。

要设置订阅，您必须向 SSRS 2012 提供有关如何传递订阅的一些信息。为此，通过 `ExtensionSettings` 对象设置传递扩展，该对象本身包含 `ParameterValue` 对象。`ParameterValue` 对象本质上是名称-值对，使得 `ExtensionSettings` 对象本质上是一个名称-值对数组。

## 实现调度报告功能

以下代码展示了 `ScheduleReport` 方法的实现，该方法负责创建报告订阅。

```csharp
private void ScheduleReport()
{
     // See whether the user wants to schedule this versus run it now
     if (sharedSchedules.SelectedValue.ToString() != "NS")
     {
          string desc = "Send report via email";
          string eventType = String.Empty;
          string matchData = String.Empty;

          // If the user selected SnapShot, then
          // set up the parameters for a snapshot
          if (sharedSchedules.SelectedValue.ToString() == "SS")
          {
               eventType = "SnapshotUpdated";
               matchData = null;
          }
          // otherwise the user is using a subscription
          else
          {
               eventType = "TimedSubscription";
               matchData = sharedSchedules.SelectedValue.ToString();
          }

          ParameterValue[] extensionParams = new ParameterValue[8];
          extensionParams[0] = new ParameterValue();
          extensionParams[0].Name = "TO";
          extensionParams[0].Value = "someone@company.com";
          extensionParams[1] = new ParameterValue();
          extensionParams[1].Name = "ReplyTo";
          extensionParams[1].Value = "reporting@company.com";
          extensionParams[2] = new ParameterValue();
          extensionParams[2].Name = "IncludeReport";
          extensionParams[2].Value = "True";
          extensionParams[3] = new ParameterValue();
          extensionParams[3].Name = "RenderFormat";
          extensionParams[3].Value = "PDF";
          extensionParams[4] = new ParameterValue();
          extensionParams[4].Name = "Subject";
          extensionParams[4].Value = "@ReportName was executed at @ExecutionTime";
          extensionParams[5] = new ParameterValue();
          extensionParams[5].Name = "Comment";
          extensionParams[5].Value = "Here is your @ReportName report.";
          extensionParams[6] = new ParameterValue();
          extensionParams[6].Name = "IncludeLink";
          extensionParams[6].Value = "True";
          extensionParams[7] = new ParameterValue();
          extensionParams[7].Name = "Priority";
          extensionParams[7].Value = "NORMAL";

          // Configure the extension settings required
          // for the CreateSubscription method
          ExtensionSettings extSettings = new ExtensionSettings();
          extSettings.ParameterValues = extensionParams;
          extSettings.Extension = "Report Server Email";

          // Get the report parameters using the GetParameters form
          GetParameters reportParameters = new GetParameters(url);
          reportParameters.ShowDialog();
          Microsoft.Reporting.WinForms.ReportParameter[] rps = reportParameters.Parameters;

          // Convert the Winforms.ReportParameter returned
          // from the GetParameters to ParameterValues required for
          // the CreateSubscription method
          int i = 0;
          foreach (Microsoft.Reporting.WinForms.ReportParameter rp in rps)
          {
               if (rp.Values.Count != 0) i++;
          }

          ParameterValue[] pvs = new ParameterValue[i];
          int j = 0;
          foreach (Microsoft.Reporting.WinForms.ReportParameter rp in rps)
          {
               if (rp.Values.Count != 0)
               {
                    pvs[j] = new ParameterValue();
                    pvs[j].Name = rp.Name;
                    pvs[j].Value = rp.Values[0]; j++;
               }
          }

          // Now set up the subscription
          try
          {
               rs.CreateSubscription(report, extSettings, desc, eventType, matchData, pvs);
          }
          catch (SoapException ex)
          {
               MessageBox.Show(ex.Detail.InnerXml.ToString());
          }
     }
}
```
