## 使用 User!UserID 过滤报告内容

对于每日计划报告，假设你决定让用户只能查看自己的日程安排。SSRS 允许你通过创建一个报告过滤器来实现这一点，该过滤器使用执行报告的用户的登录账户值。登录名值是从 SSRS 中的一个全局集合返回的。你一直在使用全局集合——例如，当你使用表达式`=Fields!FieldName.Value`时，你实际上是从 `Fields` 全局集合中返回一个值。你将用于报告过滤器的全局集合是 `User`，而你感兴趣的值是 `UserID`。因此，表达式将是`=User!UserID`。

要在过滤器中使用 `User!UserID`，你需要在数据集中有一个等于 `UserID` 值的字段。在每日计划报告的数据集中，你可能还记得有一个名为 `HWUserLogin` 的字段，你可以用于此目的。当过滤器进行比较时，这两个值将是相同的——一个值随数据集提供，另一个在报告执行时提供。应用过滤器后，报告将仅显示那些执行报告的员工的用户名与数据集每条记录返回的 `HWUserLogin` 字段值相匹配的记录。

与参数不同，过滤器不能通过报告管理器设置。要设置过滤器，你需要修改报告本身，可以直接在 RDL 文件中，或通过报表生成器或 Visual Studio 进行，如 图 11-20 所示。请注意，你可以使用 `RTRIM` 函数来去除尾随空格；否则，比较可能会失败。确保员工登录名与你创建的用户匹配，这样比较才能匹配。在本书示例中，这是 `PROSSRS\Jyoungblood`，但你的示例可能不同。将过滤器添加到组属性中，以过滤掉除查看用户之外的任何其他用户。

**图 11-20.** 每日计划报告的 `UserID` 过滤器

## 设置数据源安全性

一旦你将数据源部署到报告服务器，就可以指定其连接属性。这是一个重要的步骤，因为属性设置决定了用户和 SSRS 如何连接到数据源。在执行无人参与的报告时——例如，用于用户订阅——SSRS 将控制向数据源传递身份验证凭据，并且必须有权访问有效的身份验证凭据。

在报告管理器中，数据源有四种连接选项可用：

> *   *由运行报告的用户提供的凭据：* 使用此选项时，用户在执行报告时始终会被提示登录数据源。
> *   *安全存储在报告服务器中的凭据：* SSRS 使用存储在 `ReportServer` 数据库中的身份验证凭据。敏感的登录信息是加密的。
> *   *Windows 集成安全性：* 此选项将当前用户的登录信息传递给数据源。如果数据源将用于无人参与的安装，或者 Windows 域未配置 `Kerberos`，则不要选择此选项。在从 SQL Server 进行多次跳转时使用此选项也会出现问题。除非 `Kerberos` 在整个域中正确安装和配置，否则这可能无法在所有情况下正常工作。
> *   *不需要凭据：* 这是最不安全的选项，在数据源不需要身份验证时使用。

## 审核报告

能够了解报告执行的详细信息，特别是对于不需要的访问，是安全难题的重要组成部分。在本章中，我们将使用内置的报告执行日志和一个报告管理表来构建一个报告，该报告将显示谁运行了什么报告以及更多信息。拥有报告执行的审计跟踪对于深入了解用户行为和可能的安全漏洞至关重要。

## SSRS 审计

在本节中，我们将展示如何使用 `ReportServer` 数据库中的内置视图来审计 SSRS 中的以下活动：

> *   *报告执行活动：* 哪位用户在何时执行了哪份报告，以及报告以何种状态完成（以发现可能的问题）
> *   *报告的参数输入：* 用户输入了哪些参数

后者很重要，因为尽管 SSRS 对某些类型的参数（如 `Integer` 和 `DateTime`）包含验证功能，但当报告参数使用字符串时，它就容易受到 SQL 注入攻击。当网页或服务从用户或程序接收的输入中可能包含可在 SQL Server 机器上执行的注入代码时，SQL 注入攻击就成为可能。这些攻击本质上可能是恶意的，也可能导致报告或页面传递超出预期的数据。

我们简单地将报告从矩阵改为了表格，这更符合用于性能分析的数据透视表。在这个名为 `报告执行审计` 的修改版报告中（可在 `Pro_SSRS` 第 11 章 文件中找到），您还将包含用户已选择的参数。请上传此报告以及一个指向您的 `ReportServer` 数据库的新数据源，将其放在 SSRS 安装根目录下一个名为 `Management` 的新文件夹中。您还需要将报告指向您正在创建的这个新数据源。您可以在代码清单 11-1 中查看我们用于提取此信息的查询。

**代码清单 11-1.** 执行日志查询

```sql
SELECT
          Catalog.name
        , ExecutionLog.UserName
        , ExecutionLog.Parameters
        , ExecutionLog.TimeStart
        , ExecutionLog.TimeEnd
        , ExecutionLog.Status
 FROM ExecutionLog
INNER JOIN Catalog ON ExecutionLog.ReportID = Catalog.ItemID
```

在图 11-23 中，您可以看到所选报告的执行时间以及输入的参数名称和值。除了通过 `参数` 字段显示用户如何与报告交互外，`报告执行审计` 报告还显示了报告的开始和结束时间。拥有这种级别的审计是维护机密信息安全斗争中的宝贵盟友。对于 HIPAA，还需要维护用户数据访问的审计跟踪。如果您怀疑有人正在访问其无权查看的信息，此报告可以与其他常规审计程序（如 Windows 事件日志）一起作为审计跟踪。

## 日志文件审计简介

除了报告审计外，SSRS 还支持另外两种类型的日志记录：标准的 Windows 事件日志记录和跟踪文件日志记录。您可以使用这两种资源来搜索错误和警告，以及其他重要信息（如安全信息）。

所有跟踪文件都存储在默认安装位置 `驱动器:\Program Files\ Microsoft SQL Server\MSRS11.MSSQLSERVER\Reporting Services\LogFiles`。存在几种类型的日志文件：`ReportServerService` 和 `ReportServer`。每个文件都以时间戳命名，例如 `ReportServerService_05_31_2012_17_00_30.log` 和 `ReportServerService_HTTP_05_31_2012_17_00_30.log`，并包含特定于其各自服务的信息。您可以收集五个级别的跟踪日志信息——0 到 4 级——这些级别在 `Reporting Services\ReportServer\Bin` 文件夹中的 `ReportingServicesService.exe.config` 文件中控制。选择 `0` 将禁用跟踪，选择 `4` 将启用详细模式。每当 SSRS 出现问题时，管理员通常可以通过查看事件日志或某个跟踪文件来隔离问题。

### 小结

在本章中，您研究了确保 SSRS 部署安全所必需的三项安全任务：加密、身份验证和审计。在内部，只有需要访问机密数据的员工才应该拥有该权限，您可以通过为报告部署应用特殊过滤器来确保这一点。作为一家处理大量机密患者信息的公司，我们必须遵守 HIPAA 规定的法规。然而，在其他行业，也存在许多类似的法规。对于任何公司来说，制定安全政策和程序都是良好的实践，即使不是在严格的监管下工作。幸运的是，SSRS 设计用于通过 Windows 身份验证利用您组织中已存在的核心级安全机制，并且在需要时可以扩展以支持其他自定义身份验证方式。

## 使用 SSRS 交付商业智能

大多数公司都会积累业务数据，如果分析得当，这些数据可以为公司指明实现最终成功的方向提供洞察。商业智能（BI）的主要目的是以可立即用于决策的方式提供数据。微软的 BI 平台包含许多协同工作的服务和应用程序，以促进关键业务数据的分析与交付。`SQL Server`是 BI 模型的核心，提供数据存储、数据转换、通知、调度、分析和报表服务。

拥有正确的数据只是构建有效 BI 系统挑战的第一部分。第二部分是确保这些数据以有效且易于访问的方式交付给所有需要查看它们以做出正确决策的人。这正是`SSRS`发挥作用的地方。在我们的组织中，我们发现通过将`SSRS`与 BI 平台的许多其他组件集成，我们能够通过在员工需要时、无论他们身在何处都能提供必要信息，从而显著改善整体业务战略。

在本章中，您将了解我们为我们的软件开发公司扩展并纳入`SSRS`报表的以下应用程序：

*分析服务*：能够对数据进行多维度“切片”常常能产生意想不到且有意义的结果。当`OLAP 服务`随`SQL Server 7.0`引入时，我们被要求构建一个数据仓库，将我们的`OLTP`数据转换为`OLAP`多维数据集。我们使用随`SQL Server 2000`一起发行的`分析服务`版本来维护该项目，并增加了一些新功能，例如数据挖掘模型。自那时起，微软一直在`OLAP`技术上大力投入。其中一个增强功能是`VertiPaq`引擎，它与`SharePoint 2010`、`PowerPivot`和`Excel 服务`一起使用。使用`PowerPivot`模型并将其部署到已配置好的`SharePoint 2010`服务器上，可以在内存中快速生成一个`OLAP`多维数据集，以实现对数亿条记录的快速切片和切块。

*SharePoint 服务*：作为 Microsoft Office 的一部分，`SharePoint 服务`为我们公司提供了一个按部门划分的内网门户。任何与单个部门或整个公司相关的信息都会被索引并可搜索。将`SSRS`报表与`SharePoint`集成，使我们的员工能够轻松找到他们完成工作所需的数据。我将向您展示如何将`SSRS`报表添加到`SharePoint`。由于`SSRS`构建在`Windows SharePoint 服务`之上，您在本章中完成的工作也将适用于`Windows Server 2003`附带的`Windows SharePoint 服务`。对于`SharePoint 2010`和`Reporting Services 2012`的安装，您将需要`Windows Server 2008 SP2`的 64 位版本。

大多数公司都有与我们描述的应用程序类似的应用程序。我们提供这些示例是为了让您了解，只需付出少量努力，`SSRS`就能轻松增强这类业务应用程序。我们的目的不一定是提供逐步指南，而是展示您的公司可能如何使用`SSRS`。如果您使用了本章提到的任何技术，您可以轻松地将本章的想法整合到您自己的环境中。

### 为 SQL 分析服务构建 SSRS 报表

当我们开始使用`SQL Server 2000 分析服务`为我们的医疗保健应用程序构建`OLAP`解决方案时，我们渴望在第一天早上就直接开始分析数据。我们很快成为了开发`SQL`查询来查询我们`OLTP`数据库的常驻专家。由于这是我们将要构建数据仓库的源数据库，我们的团队认为这不过是添加几个查询并处理多维数据集这么简单。但事实证明并非如此简单。然而，回想起来，从一个已知数据源创建`OLAP`多维数据集的过程是值得的，因为我们能够将学到的技能应用到许多其他项目中。随着`分析服务`的每个新版本发布，都交付了越来越多的功能。起初，这可能让人应接不暇，仅仅因为增强功能的数量庞大，以及不仅需要熟悉技术，还需要掌握它所需的时间。然而，在添加新功能的同时，微软也添加了许多简化耗时任务的新工具。以`Analysis Services 2005`为例，这些新工具包括一个图形化的`MDX`查询生成器和一个多维数据集向导，它自动化了许多创建`分析服务`解决方案复杂部分的步骤。`SQL Server Analysis Services 2008`和`2008 R2`更进一步，增加了节省时间的功能、增强的数据分析以及与`SharePoint`集成的能力。`SQL Server 2012 分析服务`带来了新的增补功能，比如前面提到的`VertiPaq`引擎和`表格模型`，后者基于新的`商业智能语义模型`（`BISM`）。

在本节中，我们将展示如何使用一个简单的`分析服务`多维数据集作为数据源来构建和部署`SSRS` BI 报表。该多维数据集基于一个`SQL Server`数据库，该数据库作为贯穿本书的医疗保健应用程序的数据仓库。多维数据集填充了与医疗机构患者入院相关的数据。虽然我们设计该报表是为了分析患者入院历史的许多方面，例如多次重复入院的患者、变化的诊断和患者转诊来源，但我们将展示如何创建一个专门提供关于患者被转诊到机构与其实际入院之间时间间隔的分析信息的报表。随着时间的推移，收集到的数据可以帮助决策者定位问题区域，并改进可能导致患者转诊时间效率低下的流程。首先，让我们看一下您将创建的报表所基于的数据库和多维数据集。

您将用作`分析服务`项目源的数据仓库数据库名为`HW_Analysis`。这是一个简单的数据库，仅包含八个表，并使用`SQL Server 集成服务`（`SSIS`）填充数据。使用`SSIS`准备数据仓库数据库的典型过程是：从源`OLTP`数据库导出数据，转换数据使其更易于`SSAS`分析，最后将这些转换后的数据加载到数据仓库或数据集市中。

我们已经构建了`HW_Analysis`数据库和名为`Patient Referral`的`SSAS`多维数据集，并将部署这两个关键组件所需的文件以及详细的安装说明包含在本书代码下载附带的`ReadMe.txt`文件中。源代码和相关文件可在 Apress 网站（[`http://www.apress.com`](http://www.apress.com)）的本书`Source Code/Download`区域获取。一旦您按照`ReadMe.txt`文件中的说明还原了`HW_Analysis`数据仓库数据库，就可以打开包含`Patient Referral`多维数据集的`Pro_SSRS`项目。图 12-1 显示了`BIDS`中简单的`Patient Referral`多维数据集结构。该多维数据集有六个维度和两个度量值。

在这六个维度中，有多个您应该感到熟悉，因为您已在本书其他章节中使用过它们。维度包含描述事实数据的文本属性。您可以将多维数据集中的维度视为预聚合的数据组或类别，这些数据可与一个或多个度量值相关联。度量值，例如 `入院时间`（`Time to Admission`）和 `转诊事实计数`（`Fact Referral Count`），是在维度各层级计算出的值。存储在多维数据集中的多维数据由维度和度量值组成，当使用 `多维表达式`（`MDX`）进行查询时，不仅可以像 `T-SQL` 查询的行列那样进行二维分析，还能通过下钻和切片在多个维度上进行分析。

想象一下 1974 年发明的魔方，我们很多人小时候都玩过。它有三层高、三层宽、三层深。基于这个画面，假设魔方的绿色面代表 `入院日期`（`Date of Admission`）维度，红色面代表 `分支机构`（`Branch`）维度，黄色面代表 `入院时间`（`Time to Admission`）度量值。比解开魔方容易得多，您可以创建一个 `MDX` 查询来返回这三个点的交集，如 图 12-4 所示。

将在 Patient Referral 多维数据集中使用的维度如下：

> *   `县`（`County`）
> *   `转诊来源`（`Referral Source`）
> *   `诊断`（`Diagnosis`）
> *   `支付方`（`Payer`）
> *   `分支机构`（`Branch`）
> *   `入院日期`（`Date of Admission`）

## 使用图形化 MDX 查询生成器

在 `SQL Server 2005` 的 `SSRS` 中引入并在 `2008` 版中进一步增强的一个重要新特性，就是图形化的 `MDX` 查询生成器。`MDX` 是一门相当庞大且复杂的语言，容易出现语法错误。您必须对其有内在的理解，才能真正从 SSAS 多维数据集中提供精确的数据。拥有并使用图形化查询工具来构建基础 `MDX` 查询（其方式与开发人员使用图形化查询设计器构建 `T-SQL` 查询非常相似），可以减少常见的语法错误并加快查询开发速度。

打开“平均转诊至入院时间”报表的报表数据查询窗口，注意到列出了几个维度和度量值元素：`BranchName`、`Year`、`MonthName`、`Time to Admission` 和 `Fact Referral Count`。默认情况下，报表将在图形设计模式下打开，如 图 12-7 所示，而不是通用查询设计器。设计模式通过工具栏右侧的“设计模式”按钮进行切换。在图形设计模式中，维度元素可以直接从列出这些元素的“元数据”窗格拖放到窗口中。如果选择了“自动执行”按钮（默认情况下已选择），则每当拖放元素时，查询就会执行，并显示数据结果。

![Image](img/9781430238102_Fig12-07.jpg)

图 12-7. 图形化查询生成器的输出

如同您在“患者转诊”多维数据集浏览器中查询同一多维数据集时所看到的数据多维视图一样，`SSRS` 以二维方式（行和列）看待数据。`SSRS` 报表可以从多维数据集访问数据；然而，有必要以二维方式排列数据，因为报表在求和及总计度量值（本例中为 `Time to Admission` 和 `Fact Referral Count`）时将以此方式聚合数据。我们还对查询应用了筛选器，以便仅选择 2010 年至 2012 年的数据。

正如您可能在 图 12-7 的左下角所注意到的，您可以使用工具栏上的几个按钮来修改查询的属性，例如显示空单元格或添加计算成员。计算成员是 `MDX` 查询的一部分，通过将一个或多个元素组合成一个可作为新元素独立使用的值而创建。例如，您可以创建计算成员，不仅显示每个维度元素（如 `BranchName` 和 `Year`）的平均入院时间，还显示使用 `MIN` 函数计算的最短入院时间，或使用 `MAX` 函数计算的最长入院时间。这些计算成员将成为整个查询的一部分，可作为新的度量值使用，并在每个维度级别返回数据值。换句话说，如果您需要在报表层创建一个在多维数据集本身中并未创建为计算的计算，您可以在报表的数据集层创建一个“计算成员”。对于那些不擅长编写 `MDX` 计算的人来说，这是一种创建报表级计算的简便方法。
