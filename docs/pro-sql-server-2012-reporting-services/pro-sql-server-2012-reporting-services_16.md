# 安全部署 SSRS

安全是当今企业的首要任务之一。为客户提供安全可靠的计算环境不仅是良好实践，而且在许多情况下是强制性要求，受到严格的联邦法规约束。就我们的情况而言，这意味着必须遵守《健康保险流通与责任法案》（HIPAA）。这要求制定相应的政策和程序，以确保机密的患者信息不仅能够安全传输，而且只能由有权查看的人员访问。为此，我们必须确保通过网络连接（尤其是互联网）传输的数据在源头即被加密。

SSRS 是一个基于角色的应用程序，它通过定义的角色（例如只能查看报表和报表数据的“内容浏览者”）来访问其存储的对象。这些角色与基于 Windows 的登录账户相关联，因此 SSRS 主要依赖 Windows 作为其身份验证源。可以扩展 SSRS 的安全模型以支持其他身份验证方法，例如基于表单的身份验证，用户可以使用 Windows 外部维护的账户登录以访问报表服务器。由于 SSRS 具有多个身份验证点——即在报表服务器级别通过 `http.sys`，以及数据访问级别（SQL 或 Windows 身份验证）——因此当修改默认的基于 Windows 角色的安全模型时，会存在特定的安全风险。其一，需要设置 `http.sys` 以允许显式访问和对用户凭据进行自定义验证。另一个风险是，SSRS 一次只能支持一个安全扩展。换句话说，单个 SSRS 报表服务器可以扩展为支持非默认的身份验证模型，或保留为默认的 Windows 身份验证，但不能同时利用两种模型。根据您对自定义安全的需求程度——例如，您需要在面向互联网的服务器上部署 SSRS，或者您的应用程序已经支持表单身份验证，而要适应 Windows 身份验证的限制过于困难——那么您可能需要考虑自定义安全扩展。而我们的需求使得我们可以轻松地将 SSRS 纳入现有的 Windows 身份验证模型。

另一种处理安全的方法是通过具有自身身份验证层的基于 Windows 或 Web 的应用程序。在 Visual Studio 的 Windows 窗体和 Web 窗体设计器中，使用 `ReportViewer` 控件，您可以将应用程序作为进入报表服务器的门户。只要应用程序处理了安全，您就可以通过使用类似 `DOMAIN\ServerName$` 的 Active Directory 计算机账户，授予应用程序服务器访问 Reporting Services 内所需对象的权限。

在本书中，我们将介绍两种部署场景：

*   使用虚拟专用网 (VPN) 和防火墙技术进行内网部署，以允许访问 SSRS 报表服务器
*   使用终端服务安全连接到 SSRS 报表服务器的 Internet 托管应用程序

在第 11 章中，我们将引导您使用提供所需加密级别和用户身份验证的技术来保护 SSRS 部署模型。除了我们介绍的两种模型外，我们还将简要讨论集成基于表单的身份验证方法的方式，允许客户端通过 Internet 直接连接到 SSRS。

## 总结

通过为 SQL Server 2005、2008 和 2008 R2 创建和部署了众多 SSRS 项目，我们与其他 SQL Server 社区成员一样，热切期待着 SQL Server 2012 的发布。在您阅读本书的过程中，我们将指出 SSRS 2008 R2 和 SSRS 2012 中发布的增强功能。然而，与本书的其他版本一样，我们的主要目标是向您展示如何利用高级功能，提供有用的示例，使您能够在真实环境中运用 SSRS，而您所部署的报表和应用程序的用户将对解决方案的成功拥有最终决定权。
