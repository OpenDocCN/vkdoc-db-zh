# 使用报表生成器 1.0、2.0 和 3.0 创建报表

在涉及到用户自行开发的报表时，我们常常需要处理这样一个问题：如何在以受控且安全的方式提供底层数据的同时，又给予足够的自由度，使这些数据能为商业决策提供有效帮助。在 `SQL Server 2005` 中，`Reporting Services` 包含了第一个版本的 `报表生成器`。`报表生成器` 是一种应用程序，它将部分繁琐的报表设计任务转移给了报表使用者自己。

在 `报表生成器 1.0` 中，拥有适当权限的用户可以创建和发布他们自己的报表，但有一个前提：数据必须通过预先设计好的报表模型来提供。这个报表模型创建了一个抽象层——如果你愿意这样理解的话——它允许用户将提供的属性用作报表元素。就像软件开发中的任何新进展一样，我们常常会发现某些功能已被弃用。在这方面，`SQL Server 2012` 也不例外。我们不能再使用 2010 版的 `BIDS`、`SSDT` 或 `Visual Studio` 创建报表模型，但我们仍然可以在报表中将它们用作数据源。

随着 `SQL Server 2008` 的发布，`报表生成器 2.0` 应用程序也应运而生。凭借其类似于其他 `Microsoft Office` 产品的 `功能区` 技术，`报表生成器 2.0` 提供了熟悉的界面，并且超越了 `报表生成器 1.0`，它允许用户编写查询并使用存储过程直接从数据库访问数据，而无需报表模型这个额外的抽象层。这个新界面直观易用，其功能远超 `报表生成器 1.0`。

2010 年 4 月，`SQL Server 2008 R2` 发布，随之而来的是另一个版本的 `报表生成器`，它提供了数据集缓存功能，并与增强的可视化报表控件（如 `地图`、`迷你图` 以及 `指标`）相集成。既然你现在已经熟悉了使用 `BIDS` 制作报表，那么在本章中，我们将把注意力转向使用 `报表生成器 1.0`、`2.0` 和 `3.0` 应用程序来创建实用的现实世界报表。

在我们开始之前需要注意的一点是，在本书撰写时，绝大多数组织仍在使用先前版本的 `SQL Server`。因此，我们决定在本书此版本中保留报表模型和 `报表生成器 1.0`。如果 `SQL Server 2012` 是你组织拥有的唯一版本，或者如果你的公司正计划将所有东西迁移到 `SQL Server 2012`，你可以跳过关于报表模型和 `报表生成器 1.0` 的章节。如果你的公司属于那些正迁移到 `2012` 的公司之一，请注意，目前没有将报表模型迁移到 `SQL Server 2012` 的途径。在澄清了这一点之后，让我们开始吧。

首先，我们将探索 `报表生成器 1.0` 及其组件。正如你将看到的，尽管编译报表的用户需要做一些设计工作，但管理员仍然需要设计向 `报表生成器 1.0` 客户端应用程序提供数据的报表模型。我们将从两个角度引导你完成使用 `报表生成器 1.0` 的过程——首先基于用户输入开发和发布报表模型，然后基于该模型创建你自己的即席报表。我们将涵盖的三个步骤是：收集用户反馈、创建并发布报表模型，最后通过使用 `报表生成器 1.0` 创建报表来测试报表模型：

> *获取反馈*：由于我们与开发人员合作多年，我们知道交付新应用程序的最大问题之一是，有时用户拿到应用程序后期望它做更多的事情，或者期望它以不同于交付的方式来做某事。从用户的角度来看，期望是系统应该精确地做用户需要的事情。但对于开发人员来说，如果版本 1 没有做到，就必须在版本 2 中实现。你可以通过实施标准的设计实践来避免这种冲突，这些实践总是预先考虑用户反馈。当同一个应用程序将被交付给拥有不同背景用户的的不同公司，而每个用户对流程应如何运作都有自己的理解时，这个过程尤其重要。在过去的几十年里，系统分析师会从用户群体那里获取信息，并将其转化为开发人员可以慢慢理解的技术设计。如今，大多数系统分析工作由开发人员直接完成，他们在项目的设计阶段与用户互动。在我们介绍如何构建供报表设计器使用的报表模型之前，我们将提供一些我们收到的关于用户希望看到何种报表的反馈样本。
>
> *构建报表模型*：报表模型是 `报表生成器 1.0` 的核心。报表模型充当底层数据源的语义描述。因为报表模型可以通过选择单个表、字段和值来简化可能复杂的整体数据库架构，所以它可以被比作一个 `SQL Server` 视图。确实，正如你将在本章中看到的，报表模型的一个关键组成部分是 `数据源视图`。一旦 `DBA` 或开发人员创建了一个包含满足用户需求数据的报表模型，他们就可以将其发布到报表服务器，供 `报表生成器 1.0`（即客户端报表设计器）使用。提供给 `报表生成器 1.0` 的所有数据都来自已发布的报表模型。我们将在本章中展示如何基于用户反馈创建和发布模型。
>
> *使用报表生成器 1.0*：`报表生成器 1.0` 是几种设计应用程序之一，可用于创建和发布 `SSRS` 报表，而无需像 `Visual Studio` 或 `BIDS` 那样完整的开发环境。`报表生成器 1.0` 依赖预先发布的报表模型作为其数据源。正如你将看到的，`报表生成器 1.0` 的界面非常直观；事实上，对于需要基于预定义报表模板快速创建报表的用户来说，它是理想的选择。`报表生成器 1.0` 不仅允许用户设计报表，还充当将完成的报表直接部署到报表服务器的发布工具。

当我们开始讨论 `报表生成器 2.0` 时，用户对其访问的数据有更多控制权，你会发现你在 `报表生成器 1.0` 中学到的许多相同概念仍然适用。开发人员可能会发现自己仍然需要根据最终设计报表的用户的反馈来创建视图和存储过程。同样，许多开发人员可能会认为 `报表生成器 2.0` 已具备他们所需的所有功能，并会选择它作为其首选的开发环境。


## 获取用户反馈

在本书的这一阶段，你已经构建了一个名为 `Employee Service Cost` 的报表，该报表在多个层级上分解了医疗就诊的成本。你使用 `BIDS` 设计了此报表，它包含多个分组级别、交互式排序和向下钻取功能。该报表的数据集是一个存储过程，它从 `Pro_SSRS` 数据库的多个表中提供所需字段。就像你将使用 `Report Builder 1.0` 构建的报表一样，这个存储过程和报表是基于需要它的用户的反馈创建的。创建存储过程的一个目标是，不仅要使其满足一个报表的需求，还要使其具有足够的通用性，以便能够重用于其他类似报表。在展示如何构建报表模型以满足用户以及将使用 `Report Builder 1.0` 创建和发布报表的报表设计者的需求时，我们将讨论这个问题。

在医疗保健行业（即，诸如家庭保健和临终关怀等急性期后护理领域），你必须遵循许多针对州和联邦机构的报告要求。从成本报告到患者入院和出院信息，不同部门因不同原因需要不同的报表。每个部门都使用的一组报表是患者普查报表。该报表显示了当前入住医疗保健机构并正在接受服务的所有患者。该报表可以简单到按字母顺序排列的患者姓名列表，也可以详细到每位患者的所有人口统计信息，例如他们的居住地以及紧急情况下的联系人。

由于患者普查报表是最常被请求定制的报表，我们将在本章中将其用作报表模型的示例，确保我们提供足够的数据以满足当前和未来的需求。

对于患者普查报表示例，假设用户要求查看每日入住该机构的患者总数。这类患者都有一些在查看普查数据时很重要的共有数据关联。这些信息回答了这类问题：

*   患者的主要诊断是什么？
*   患者当前入住的是哪个分院？
*   患者在机构内的住院时长是多少？
*   目前每个分院有多少活跃患者？
*   患者的地址、电话号码和其他关键人口统计信息是什么？（这种类型的普查报表，通常被称为“脸谱表”，会在紧凑的位置显示大量关于患者的信息。）
*   如果患者已出院，何时发生的以及原因是什么？是因为他们康复了，还是转院了？

![Image](img/square.jpg) `注意` 我们在本章讨论的信息类型被认为是受保护的健康信息（PHI），根据 `HIPAA`，这些信息需要严格保护，并且只能由获准使用的人员访问。第 11 章 涵盖了如何保护通过 `SSRS` 交付的数据的访问安全。

你可以通过单个查询来回答所有这些问题甚至更多，这可以构成报表模型的基础。因此，在 `Pro_SSRS_2008R2` 数据库中，你将把几个表连接成一个数据源视图，以提供交付患者普查所需的信息。在开发其他报表时，你已经使用了这些表中的大部分，但在展示如何构建报表模型之前，我们将先回顾一下相关表的数据库模式。这些是你在本章示例中将使用的表：

*   `Admissions`
*   `Branch`
*   `Diag`
*   `Discipline`
*   `Documentlmage`
*   `PatEMRDoc`
*   `Employee`
*   `PatDiag`
*   `Dischg`
*   `Patient`

每个表都包含与个体患者入院历史相关的具体信息，它们共同提供了足够的数据，可以为医疗机构内的许多部门创建多种普查报表。你知道需要回答的问题类型，而这些表的设计就是为了基于用户的直接反馈和请求来提供这些信息。现在，让我们继续设计查询并构建报表模型。

## 报表模型简介

仅仅因为你不懂得如何将汽车引擎的所有部件组装在一起的复杂性，并不意味着你不能开车，对吧？对于将使用即席式 `Report Builder 1.0` 来设计和发布报表的报表设计者来说，情况也是如此。在这种情况下，为了构建报表，你几乎不需要了解数据库结构的各个部分。不过，*某个人*必须理解数据库模式是不可避免的，就像机械师必须理解刹车片和风扇皮带一样。构建一个通用的报表模型（报表设计者将使用它来创建报表）很可能属于 `DBA`、数据分析师或 `SQL` 开发人员的职责范围，他们对数据源有深入的了解。在本节中，我们将展示如何利用从报表使用者那里收集的反馈，为患者普查报表开发一个报表模型，该模型旨在通过逻辑且直观的字段名称使数据易于访问。

你将执行以下步骤来为患者普查报表模型创建报表模型：

1.  使用 `BIDS/SSDT` 创建报表模型项目。
2.  创建数据源。
3.  创建数据源视图。
4.  基于数据源视图创建患者普查报表模型。
5.  将患者普查报表模型发布到 `SSRS` 服务器。

构建报表模型的过程包含许多单独的步骤，其中许多步骤只需通过向导逐步操作即可完全控制。报表模型可以是复杂的，也可以是简单的，这取决于你的个人需求。为了简洁起见，我们将尽可能保持示例模型的简单性；你将能够回答患者普查的反馈问题，并且你将能够毫无问题地创建和发布你自己的报表模型。
