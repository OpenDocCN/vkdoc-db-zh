# 后台进程

正如我们之前讨论的，数据库后台进程会根据优先级自动分配到内置（内部）IORM 消费者组。下表显示了这些特殊的 IORM 消费者组及其用途描述。

表 7-7. 资源管理器内部消费者组

| 消费者组 | 描述 |
| --- | --- |
| `_ASM_` | 与 Oracle ASM 卷管理相关的 I/O |
| `_ORACLE_BG_CATEGORY_` | 数据库后台进程发出的高优先级 I/O 请求 |
| `_ORACLE_MEDPRIBG_CATEGORY_` | 数据库后台进程发出的中等优先级 I/O 请求 |
| `_ORACLE_LOWPRIBG_CATEGORY_` | 数据库后台进程发出的低优先级 I/O 请求 |

重要的是要认识到，高敏感度 I/O 请求（如日志写入器写入）的优先级是通过（内部）消费者组强制执行的。但这在事件链中可能为时已晚。回顾 IORM 的评估步骤：
类别/配置文件 -> 跨数据库计划 -> 可插拔数据库 -> 数据库内计划

您可以看到，尽管该组被命名为高优先级 I/O 请求组，但 IORM 的评估对于该组排在第四位，这意味着如果资源已经在较早的步骤（如跨数据库计划）中被分配出去，数据库可能只剩下很少的优先级。这意味着即使被命名为高优先级 I/O 组，在 I/O 压力出现时的整体情况下，它仍然可能拥有非常低的优先级。当然，这确实意味着，相对于同一数据库中其他消费者组执行的 I/O，该组为其 I/O 获得了最高优先级。

当启用类别计划时，情况会有所改变。启用类别计划后，会创建内部类别计划，如上述内部消费者组，后台和 ASM 请求将被直接确定优先级。在内部计划中，优先级划分基于跨数据库计划的优先级。
