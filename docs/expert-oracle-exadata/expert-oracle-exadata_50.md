# 第 11 章 Exadata 性能指标

Oracle Exadata 相较于传统的数据库服务器架构是一次巨大的飞跃；然而，尽管拥有诸多创新特性，它运行的仍然是标准的 Oracle 数据库软件。大多数经验丰富的数据库性能规则仍然适用，同时增加了一些能够体现 Exadata 功能优势的规则，例如智能扫描、存储节点连接过滤和闪存缓存。在本章中，您将了解到更多关于 Exadata 特有的及相关的性能主题、指标以及一些相关的内部原理。

幸运的是，Oracle 无论在数据库层还是存储节点层都提供了大量的性能指标供我们使用。然而，在考察任何指标时，您都应该清楚为什么要监控它，以及要寻找什么样的数值。换句话说，如何知道一切正常无需采取行动，又如何知道情况糟糕需要采取行动？为了避免浪费精力解决错误的问题，衡量真正重要的事情至关重要。对于数据库性能而言，没有什么比响应时间更重要，即最终用户（或连接的系统）等待响应所花费的实际时间。因此，如果您想在数据库中让某些操作变得更快，就应该专注于衡量并优化响应时间。所有其他指标和指示器，如 I/O 请求数或缓存命中率，都是次要的。就像等待报表生成的最终用户，他们只关心需要等待的时间，而不是像 CPU 利用率或 I/O 速率这类次要指标。尽管如此，这些次要指标通常对于理解和解释性能问题非常有用。

分解数据库响应时间的关键指标是 Oracle 等待接口的事件。一些关键的等待事件在 第 10 章 中讨论，而利用这些事件的性能监控工具将在第 12 章 中介绍。然而，Exadata 还提供了其他有用的指标，例如智能扫描返回的数据字节数、以及由于存储索引而实际避免的 I/O 量等等。这些指标提供了关于 SQL 执行期间数据库和存储节点内部发生情况的非常重要的附加信息。在本章中，您将读到更多关于这些指标的内容，并学习如何访问和显示它们，最重要的是，理解它们的含义。即使您对每个指标的含义不感兴趣，我们仍然建议您阅读本章，因为它解释了数据库代码和存储服务器中 Exadata 特有部分的一些重要内部原理和设计决策。

本章的信息将帮助您很好地理解 Exadata 数据库和存储服务器的一些关键内部工作机制，并为您学习第 12 章 打下基础，在那一章中，这些知识将在监控和排除 Exadata 性能故障时派上用场。

## 衡量 Exadata 的性能指标

在深入研究 Exadata 特有的性能指标之前，让我们先审视一些内部原理，并回顾 Exadata 特有功能和指标的一些重要元素。在尝试使用这些信息进行监控或优化之前，必须理解这些性能数字实际代表什么。

至此，您已经知道 Exadata 数据库节点本身并不执行物理磁盘 I/O，而是通过 ASM 层请求存储节点代为执行这些请求。在智能扫描的情况下，存储节点还会处理读取的数据块：提取其内容、过滤行等。因此，从数据库的角度来看，存储节点就像一个黑盒。数据库层请求数据块或数据块范围，而存储节点对数据库层透明地执行物理 I/O 工作，并返回匹配行的所需列。值得庆幸的是，Oracle Exadata 的架构师、工程师和开发人员在 `cellsrv` 及相关组件中加入了数量惊人的仪表化功能。存储节点会跟踪它们完成了多少工作，并且还可以将指标与结果一起发送回数据库层。这使我们——DBA、开发人员和故障排查人员——能够全面了解在服务用户请求或运行查询时，数据库和存储节点中发生的情况。作为 DBA，如果您尝试从现有的存储基础设施中获取如此详细的性能信息，您会很快更加欣赏 Exadata。

例如，当通过智能扫描执行查询时，如果您在数据库层查询 `V$SQLSTATS`，您仍然会看到熟悉的统计信息，如 `physical_reads`，尽管数据库层本身并没有直接执行任何物理读取。另一个例子是“存储索引节省的物理 I/O 字节数”统计信息，它是在存储节点层而非数据库层计数的。这些数字之所以能在数据库层可见，得益于存储节点在返回查询数据之外，还发送了有用的性能指标。有些指标目前同时在存储节点和数据库级别报告数据，您可能会看到一些重复计数的情况。

除了标准的 Oracle 性能工具外，本章还将介绍两个在本章和第 12 章 中讨论的定制工具。它们更适合灵活和高级的性能分析，允许您超越标准的等待事件和 SQL 语句级别的统计信息。第一个工具是 Oracle Session Snapper，它是一个仅包含匿名 PL/SQL 块的脚本，可以从 `V$SESSION`、`V$SESSION_EVENT`、`V$SESS_TIME_MODEL`、`V$SESSTAT` 等视图中测量详细的性能指标。列表中的最后一个性能视图 `V$SESSTAT` 对于高级性能分析尤为重要——它包含了实例中每个会话的数百个动态性能计数器（在 11.2.0.2 版本中有超过 600 个，在 Oracle 12.1.0.2 中有 1178 个）。除了使用等待事件进行常规监控外，深入分析 `V$SESSTAT` 可以让我们更好地了解 Oracle 会话正在进行何种工作，例如它们每秒执行多少 I/O 请求、每秒进行多少次全段扫描、在扫描过程中必须获取多少已迁移/链接的行等等。

### 会话计数器的简要历史

Oracle 性能计数器作为数据库引擎的一部分已经存在很长时间了。Oracle 为每个会话维护的计数器数量允许对正在发生的事情进行非常细粒度的分析。如果您计算起始快照和结束快照之间的差值，通常可以很好地概览数据库活动。

## Oracle 会话性能计数器的演变

Oracle 持续地通过每个补丁为引擎增加仪表化功能，这非常出色。你可以通过查询 `v$statname` 来观察不断增加的数量。这个视图包含一个名为 `class` 的列，该列在 Oracle 参考文档中有记载。

为了展示数据库仪表化不断演进的例子，我们使用以下查询检查了一些主流版本：

```sql
with stats as (
select name, decode(class,
1, 'USER',
2, 'REDO',
4, 'ENQUEUE',
8, 'CACHE',
16, 'OS',
32, 'RAC',
64, 'SQL',
128, 'DEBUG',
'MULTI-CATEGORY'
) as decoded_class from v$statname
)
select count(decoded_class), decoded_class
from stats
group by rollup(decoded_class)
order by 1
```

结果很有启发性。在 11.2.0.3 版本中，发现了如下分布：

```
COUNT(DECODED_CLASS) DECODED
-------------------- -------
9 ENQUEUE
16 OS
25 RAC
32 REDO
47 MULTI-CATEGORY
93 SQL
107 USER
121 CACHE
188 DEBUG
638
```

Oracle 11.2.0.4（在 12.1.0.1 之后发布）显示如下：

```
COUNT(DECODED_CLASS) DECODED
-------------------- -------
9 ENQUEUE
16 OS
25 RAC
34 REDO
48 MULTI-CATEGORY
96 SQL
117 USER
127 CACHE
207 DEBUG
679
```

如你所见，增加了相当多的计数器，但与 12.1.0.2 相比就不算什么了：

```
COUNT(DECODED_CLASS) DECODED
-------------------- -------
9 ENQUEUE
16 OS
35 RAC
68 REDO
74 MULTI-CATEGORY
130 SQL
130 USER
151 CACHE
565 DEBUG
1178
```

几乎是 11.2.0.3 的两倍！公平地说，其中大量计数器与内存中选项相关，但尽管如此，仪表化能力的增强仍然是一个受欢迎的补充。

## 重新审视 Exadata 智能扫描的前提条件

在本节中，您将被温和地介绍 Oracle 数据库内核仪表化向我们提供的更详细的指标。重点是 Exadata 相关的性能统计。Exadata 等待事件的细节在此不涉及，因为这些已在第 10 章中解释过。在适用的情况下，会补充回顾如何使用等待事件来更好地理解数据库性能，您可能会发现一些已在第 2 章中介绍过的卸载和智能扫描材料。这是一个为了帮助您理解本章内容而无需翻回很多页的复习。在理解本章以及监控和排除 Exadata 性能问题的背景下，理解细节很重要。

Exadata 平台上数据仓库和报表工作负载的主要性能提升之一就是 Exadata 智能扫描。OLTP 工作负载则极大地受益于 Exadata 智能闪存缓存的使用。本节的第一部分详细说明了如何衡量您的工作负载是否从这两项功能中受益。这些知识构成了后续章节中数据库和查询性能监控的基础。在第二部分，您可以阅读更多关于一些 Exadata 内部性能计数器的内容。

### Exadata 智能扫描性能

由于智能扫描是一项如此令人兴奋的功能，我们首先介绍其指标。在讨论任何指标之前，让我们回顾一下 Oracle 中决定执行智能扫描是如何做出的。请注意，智能扫描可用于常规表段，也可用于物化视图段——它们在物理上与常规表没有区别。智能扫描也可用于全扫描 B*树索引段（索引快速全扫描）以及位图索引段（位图索引快速全扫描）。使用“强力”的多块读取方式扫描索引段，而不是采用通常的单块 I/O 方法，这与对表执行全表扫描非常相似。主要区别在于，索引段有一个称为索引分支块的附加结构，在扫描时需要跳过并忽略，此外还有 ASSM 位图块，在表和索引扫描中也都需要跳过。对表和索引的分区及子分区进行的智能扫描，在内部与对非分区对象的扫描没有区别。一个分区段实际上只是较小段的逻辑分组。请记住，智能扫描可以扫描各种类型的段。

无论段类型如何，智能扫描始终要求 SQL 执行引擎在执行时选择直接路径读取，这不是优化器的决定。因此，在排除为何未使用智能扫描的故障时，您必须首先检查是否使用了直接路径读取。您应该先检查执行计划，看看那里是否报告了全扫描。这里您可以看到在不同段类型上发生全段扫描的简单示例，从全表扫描开始：

```sql
SELECT AVG(line) FROM t WHERE owner LIKE 'S%'
-----------------------------------------------------------------------------------
| Id  | Operation                  | Name | Rows  | Bytes | Cost (%CPU)| Time     |
-----------------------------------------------------------------------------------
|   0 | SELECT STATEMENT           |      |       |       |   295K(100)|          |
|   1 |  SORT AGGREGATE            |      |     1 |    11 |            |          |
|*  2 |   TABLE ACCESS STORAGE FULL| T    |  5743K|    60M|   295K  (1)| 00:59:01 |
-----------------------------------------------------------------------------------
Predicate Information (identified by operation id):
---------------------------------------------------
2 - storage("OWNER" LIKE 'S%')
    filter("OWNER" LIKE 'S%')
```

请注意，这里的 `TABLE ACCESS STORAGE FULL` 行中的 `STORAGE` 并不意味着 Oracle 试图进行智能扫描。这个关键字仅仅表明 Oracle 知道这个段驻留在存储单元上，而不是诸如 NFS 挂载或 iSCSI 设备之类的地方，并且它正在使用能够执行智能扫描的表扫描代码路径。是否实际使用了智能扫描取决于其他多个因素，这些因素在第 2 章中讨论过，并且这里也将简要演示。执行计划中是否显示 `storage` 关键字也取决于初始化参数 `cell_offload_plan_display`。此外，智能扫描是一个广义的术语，涵盖了过滤（可能利用存储索引）、列投影、HCC 压缩 CU 的解压，以及利用布隆过滤器进行哈希连接提前消除等操作——所有这些都在存储单元中完成。即使您没有利用将过滤谓词卸载到存储（过滤）的功能，存储单元也可能能够减少返回到数据库的数据量。如果使用了智能扫描，列投影在存储单元中完成，它们只返回所需的列，而不是包含整行长数据的整个数据块。
