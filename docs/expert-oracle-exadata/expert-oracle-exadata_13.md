如你所见，压缩处理减慢了足够多的速度，以至于抵消了需要读取的数据块减少所带来的收益。这是由于所执行工作的 CPU 密集型性质所致。

## 接下来是另一个使用 I/O 非常密集型查询的示例。

此测试使用了一个没有`where`子句的查询，该查询大部分时间用于通过智能扫描从存储层检索数据。首先是针对基线表的查询——该表大小为 80 GB。为了抵消闪存缓存中数据自动缓存的影响，此功能已在所有表上禁用。此外，还禁用了存储索引，以确保各次执行的结果具有可比性。

```sql
SQL> select  /*+ parallel(16) monitor gather_plan_statistics */
  2  /* hcctest_io_001 */
  3  sum(id) from t3;

SUM(ID)
----------
1.0240E+15

Elapsed: 00:00:07.33
```

在表 3-5 中，我们将此时间与其他所有表进行比较。

### 表 3-5. 压缩对全表扫描的影响

| 表名 | 压缩类型 | 压缩比 | 运行时间 | 运行时间比 |
| --- | --- | --- | --- | --- |
| T3 | 无 | 1.0 | 00:00:23.54 | 1.0 |
| T3_BASIC | 基础 | 1.1 | 00:00:19.27 | 0.82 |
| T3_OLTP | OLTP | 1.0 | 00:00:19.59 | 0.83 |
| T3_QL | 查询低 | 11.6 | 00:00:02.64 | 0.11 |
| T3_QH | 查询高 | 25.5 | 00:00:01.86 | 0.08 |
| T3_AL | 归档低 | 25.5 | 00:00:01.86 | 0.08 |
| T3_AH | 归档高 | 35.8 | 00:00:01.90 | 0.08 |

单元闪存缓存上的数据自动缓存可以显著提高性能。由于`T3`被用作创建所有其他表的基础，因此其大部分数据都存在于闪存缓存中。最初的执行时间是 00:00:07.33。执行记录在表 3-5 中的运行时间的查询时，禁用了单元闪存缓存和存储索引。但由于数据自动缓存对整体查询性能非常有益，因此你不应该为表禁用单元闪存缓存。对于单元版本 11.2.3.3.0 及更高版本，通常保持其默认设置就足够了。

执行这些查询不需要计算节点上大量的 CPU 使用率。由于这些操作被下推执行，只有存储单元是繁忙的。执行时间的大幅减少主要归因于需要扫描的数据量变小了。虽然智能扫描是在存储单元上执行的，但由于存储索引被禁用，它们无法跳过读取数据。系统必须扫描基线中大约 80GB 的数据，而对于查询高，只需扫描大约 3GB 的数据。以下是某次查询期间的 CPU 使用情况示例：
