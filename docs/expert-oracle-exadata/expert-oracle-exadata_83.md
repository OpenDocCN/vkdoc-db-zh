哎呀——最后一个例子清楚地显示了严重的性能下降。然而，在这一阶段必须重点指出，这是最极端的例子，因为每一行都发生了链接。事实上，许多记录跨越了三个块。在这种极端情况下，避免下推确实显著提高了性能。当然，在可能的情况下消除链接行将是最佳解决方案。

## 非常繁忙的单元格

当存储单元因 CPU 操作变得非常繁忙时，存储软件可能会开始拒绝为部分请求执行下推活动。也就是说，如果某个特定单元格变得极其繁忙，并且收到来自数据库服务器（该服务器并不特别繁忙）的可下推请求，它可能会决定将部分未处理或仅部分处理的数据发送回数据库服务器。在某些情况下，可能完成了列投影但没有完成过滤；在其他情况下，`cellsrv`可能会退回到将整个块传输回数据库层。虽然这个问题不会导致下推完全停止，但它会减少在存储层完成的工作量。

这是一个极其复杂的话题，并且很难直接观察到该行为。该特性的目标是利用可用的 CPU 资源，无论它们在数据库层还是存储层。此行为是在`cellsrv`版本 11.2.2.3.1 中引入的，适用于运行 11.2.0.2 及以上版本并安装了 Bundle Patch 6 的数据库。在 11.2.0.2 及更高版本的 BP 6 中，有一个名为`cell physical IO bytes pushed back due to excessive CPU on cell`的统计信息，它显示了这种情况正在发生。请注意，该统计信息名称后来更改为`cell physical IO bytes sent directly to DB node to balance CPU`。该特性旨在提高非常繁忙系统的吞吐量，但也可能导致某些语句的性能出现一定程度的不稳定。如果你的`cellsrv`错误地决定它太忙而无法承担额外工作，可以禁用此特性；然而，一般来说，如果你观察到这种行为，你可能已经接近系统的极限了。如果无法减少创建的负载，在存储层增加资源（更多的存储单元）可能是一个可行的选择。


#### 提示代码

提示（Hint）对于引导优化器按预期工作非常有用。遗憾的是，提示的文档记录并不完善，理解程度就更低了。许多情况下，提示被用来解决因数据库某些配置错误而引发的问题（然后在升级时被遗忘，可能会对新主版本发布造成问题）。其设计本意是让人类在优化器自身无法有效（或稳定）完成任务的情况下，帮助其做出正确选择。这种情况发生在优化器（目前）不够智能，无法在特定情境下得出最佳执行计划时。然而，即使提示使用恰当且生成了预期行为，它们也可能妨碍 Exadata 利用其部分内置能力。迁移到 Exadata 时，最佳方法是预留充足的测试时间。如果您的应用程序使用了提示，测试计划中的一个重要步骤应是测试其在**没有**提示时的行为。这可以通过在升级/回归测试的会话中，将隐藏参数 `_optimizer_ignore_hints` 设置为“true”来轻松实现。只有在**不必要的**提示被清除之后，系统才应该在 Exadata 上投入生产。最终，在迁移测试期间移除提示比系统上线后要容易得多。毕竟，测试阶段您拥有更多的资源和时间！

#### 索引

这看起来可能是个奇怪的主题，但索引也可能对智能扫描（Smart Scans）产生不利影响。如果有索引可用，优化器会尝试使用它们。在纯粹的数据仓库环境中，分析性索引可能完全不需要。接下来在关于混合工作负载的章节中，您会读到更多关于索引的内容，但重要的是要理解：使用索引通常意味着**下推卸载**（offloading）发生的可能性降低。

#### 逐行处理

我们经常在非常大的数据集中看到逐行处理（row-at-a-time processing），这令人惊讶。这种类型的编码在非 Exadata 平台上很少是个好主意，在 Exadata 上则**绝对不是**个好主意。实际上，由于平台内置的优化，性能差异在 Exadata 上甚至可能更加显著。

### 其他注意事项

在使用数据仓库（DW）系统时，还有几点需要牢记：Exadata 智能闪存缓存、压缩和分区的使用。

#### Exadata 智能闪存缓存：保留还是不保留

Exadata 智能闪存缓存（`ESFC`）主要被认为是为延迟敏感的 SQL 语句提供好处，通过更快的闪存设备满足单块读取需求。对单块 I/O 有益的事情，对多块 I/O 也不会有害。在此背景下，重要的是要记住，从闪存进行扫描也能显著提高扫描性能。在 Exadata 版本 `11.2.3.3.0` 之前，智能扫描默认忽略 `ESFC`，仅使用硬盘进行扫描。在 `11.2.3.3.0` 及更高版本中，智能扫描也能够并将会透明地将数据缓存在闪存缓存中。这个特定的 Exadata 功能对平台上扫描的执行方式产生了根本性影响。从该版本开始，扫描性能提升了数个数量级。透明地将整个段或其部分内容存储在 `ESFC` 中，将使智能扫描以及非下推卸载的单块和多块读取都受益。这项新功能很大程度上消除了通过使用段的存储子句对关键处理对象进行闪存缓存使用的微调。

如果您想证明 `ESFC` 在当前 Exadata 版本中对表扫描的有用性，您需要扭转 `11.2.3.3.0` 之前的情况：不是将段固定（pinning）到闪存缓存以使智能扫描能从 `ESFC` 中受益，而是必须**明确禁止**某些段使用它。以下两个表是 100%相同的，唯一的区别是表 `T1_NOCOMPRESS_NOESFC` 被禁止从 `ESFC` 中受益。
