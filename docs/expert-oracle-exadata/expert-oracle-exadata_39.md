# 9. Exadata 的恢复

你可能听过“磁盘会转动，也会死亡”这种说法。虽然我们不愿去想，但从你启动新系统的那一刻起，你的磁盘就开始老化了。磁盘在过去 30 年里取得了长足的进步，其预期寿命也显著提高了。但归根结底，问题是磁盘“何时”会发生故障，而不是“是否”会发生。而且我们都知道，许多磁盘的故障远早于其应有寿命。通常，诊断磁盘故障以及故障发生时如何应对，是系统管理员或存储管理员的职责。对许多数据库管理员（DBA）而言，Exadata 将改变这一现状。现有的许多 Exadata 系统完全由 DBA 团队管理。无论你们数据中心的情况是否如此，在 Exadata 上从磁盘故障中恢复的流程，将与你以往习惯的方式有所不同。

传统的 Oracle 数据库服务器需要两种备份：操作系统备份和数据库备份。Exadata 计算节点依赖于行业标准的硬件 RAID 和 Linux 逻辑卷，以确保其能抵御硬件故障并易于管理。Exadata 在此基础上增加了存储单元，随之而来的是一个全新的、需要保护且偶尔需要恢复的子系统。存储单元是相当坚固的硬件设备，它采用 Linux 软件 RAID 来保护操作系统文件系统。因此，单个磁盘故障很可能不需要进行操作系统恢复。更可能的原因包括人为错误、补丁安装失败或软件缺陷。请记住，这些物理磁盘设备也包含了网格磁盘（数据库卷），因此丢失其中一个磁盘很可能也意味着数据库存储的丢失。Oracle 在 Exadata 中设计了多种功能来保护你的数据并减少此类故障的影响。本章将讨论一些更常见的存储故障场景、如何诊断它们，以及如何以最小的停机时间进行恢复。

**注意**

撰写本章最具挑战性的一个方面是，我们讨论的命令和脚本变化迅速。在许多情况下，恢复任务会让你非常贴近 Exadata 的硬件层进行操作。因此，在阅读本章时请牢记，随着 Exadata 硬件和软件的每个新版本发布，本章讨论的命令和脚本可能会发生变化。请务必查阅 Oracle 文档，获取此处讨论的命令和脚本的最新更新。

## Exadata 诊断工具

Exadata 是一个高度复杂的硬件和软件组合体，它们协同工作，形成一个极其强健的交付平台。这个平台的复杂性起初可能有点令人望而生畏。要有效地维护这个平台，需要理解的组成部分实在太多了。Oracle 提供了丰富的诊断工具，可用于验证、分析和报告有关系统配置和健康状况的重要信息。在本节中，我们将讨论其中一些工具及其使用方法。

### Sun 诊断工具：sundiag.sh

每台 Exadata 数据库服务器和存储单元上都安装了 `sundiag.sh` 脚本，位于 `/opt/oracle.SupportTools` 目录中。在较新版本的 Exadata 存储服务器软件上，该脚本通过 `exadata-sun-computenode` 或 `exadata-sun-cellnode` RPM 包安装。如果出于某种原因在你的系统上找不到它，可以从 My Oracle Support 下载。请参考 MOS 文档 ID 761868.1。该脚本以 root 账户运行，收集用于排查硬件故障所需的诊断信息。它收集的文件以熟悉的 `tar` 格式打包，然后使用 bzip2 压缩。

#### sundiag.sh 的输出

`sundiag.sh` 脚本会创建一个包含主机名、序列号和运行时间戳的存档文件。例如，在我们的实验室系统上运行该脚本会产生一个如下命名的输出文件：

`/tmp/sundiag_enkcel05_XXXXXXXX_2014_11_17_12_23.tar.bz2`

现在，让我们看看 `sundiag.sh` 收集的诊断文件。文件压缩在一个存档中，包含以下组件的文件夹：

*   `asr`：此目录包含与自动服务请求配置相关的文件。
*   `cell`：此目录包含各种 `CellCLI` 命令的输出以及与 Exadata 存储服务器软件堆栈相关的日志文件。输出包括单元磁盘、网格磁盘和闪存缓存的配置信息，以及 cellsrv 和管理服务进程的日志文件。当 `sundiag.sh` 在 Exadata 计算节点上运行时，此目录不会出现。
*   `disk`：此目录包含与硬盘相关的二进制文件，由 LSI 磁盘控制器生成。
*   `ilom`：如果使用了 ilom 或 snapshot 选项，此目录将包含 ILOM 数据收集输出。
*   `messages`：此目录包含来自 syslog 工具的 dmesg 和 messages 系统日志的副本。
*   `net`：此目录包含与节点上各种网络相关的诊断信息。文件包括 InfiniBand 诊断、任何防火墙规则列表以及网络设备配置文件。此外，还包含 `ethtool` 命令的输出。
*   `raid`：`raid` 目录包含来自 `parted`、`fdisk` 和 `mdstat` 命令的磁盘控制器配置信息，以及来自 `MegaCli64` 命令的 RAID 控制器输出。
*   `sysconfig`：不属于其他类别的文件存放在这里。文件以其生成命令命名。示例包括 `df-hl.out`、`lspci-vvv.out` 和 `CheckHWnFWProfile.log`。

`sundiag.sh` 脚本创建的一些更重要的文件描述如下。这些文件在计算节点和存储服务器上都可以找到。

*   `messages`：这是系统 `/var/log/messages` 文件的副本。`messages` 文件会由操作系统自动轮转和清除。如果你的系统已经运行了一段时间，你将看到多个按升序排列的文件，从当前（`messages`）到最旧（`messages.4`）。此文件由 `syslog` 守护进程维护，包含有关操作系统健康状况和操作的重要信息。
*   `dmesg`：此文件由 `dmesg` 命令创建，包含来自内核环缓冲区的诊断级内核信息。内核环缓冲区包含发送到或接收自连接到系统的外部设备（如磁盘驱动器、键盘、视频等）的消息。
*   `lspci`：此文件包含系统中所有 PCI 设备的列表。
*   `lsscsi`：`lsscsi` 文件包含系统中所有 SCSI 设备的列表。
*   `fdisk-l` 和 `parted`：`fdisk-l` 和 `parted` 文件包含系统中所有磁盘设备分区的列表。
*   `megacli64`：`sundiag.sh` 脚本使用各种选项运行 `MegaCli64` 命令，该命令会查询 MegaRAID 控制器以获取有关磁盘控制器和连接磁盘驱动器的配置和状态信息。MegaRAID 控制器收集的丰富信息可以很容易地通过 `MegaCli64` 命令获取。例如，`megacli64-PdList_short.out` 文件显示了计算节点上磁盘驱动器的 RAID 配置摘要：

```
Slot 00 Device 11 (HITACHI H106030SDSUN300GA3D01247NLV9ZD  ) status is: Online,
Slot 01 Device 10 (HITACHI H106030SDSUN300GA3D01247NGXRZF  ) status is: Online,
Slot 02 Device 09 (HITACHI H106030SDSUN300GA3D01246NLV1JD  ) status is: Online,
Slot 03 Device 08 (HITACHI H106030SDSUN300GA3D01247NH06DD  ) status is: Online,
```
