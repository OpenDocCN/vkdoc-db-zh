# 13. 迁移到 Exadata

重大时刻终于来临。您的 Exadata 数据库一体机已完成安装、配置、优化和调试，随时可以投入使用。至此，您可能已经投入了大量时间学习 Exadata 的知识、向公司论证其价值，并规划如何充分利用这一强大的数据库平台。毫无疑问，这是一段漫长的旅程，但您尚未完全抵达终点。现在，真正的工作才开始——迁移。毕竟，预算负责人需要看到，所有投入的资金都得到了良好的利用并产生了回报！

这一章比预期的要难写得多，因为几乎无法统计 Enkitec 多年来完成的所有迁移项目。但考虑到 Oracle 的各种版本、可用的迁移工具以及这些工具从一个版本到另一个版本所经历的变化，显然需要缩小范围。因此，为了使内容更有趣并节省篇幅，本章大部分内容将聚焦于 11.2 和 12.1 版本（企业版）。在此过程中，您可以了解更多关于如何充分利用 Oracle 数据库先前版本中可用功能的知识。“Oracle 的先前版本”总是相对于当前受支持的版本而言。在撰写本文时，11.2.0.3+ 和 12.1.0.1+ 处于错误修正支持阶段，不过您应查阅 Doc ID 742060.1 以获取更准确的概览。您也可以查看 My Oracle Support Note 888828.1 了解 Oracle 各个发布版本的支持状态，以及 My Oracle Support 上的认证矩阵。

有多种方法、工具和技术可用于将数据库从传统硬件迁移到 Exadata，但一般来说，它们可以分为两大类：物理迁移和逻辑迁移，并且这两者可以结合使用。虽然决定最佳方法的因素有几个，但决策过程通常由一个主要因素主导：可用于完成迁移的停机时间。次要因素是您团队的技能组合。本章涵盖的某些内容超出了数据库管理员日常业务的预期范围。可能需要花些时间提升团队成员的技能，使他们能够自信地使用其中一些工具。

好消息是，有多种策略可以帮助您迁移到 Exadata。每种方法都有其自身的优缺点。在本章中，您将详细了解每一种方法。您将了解到选择某种方法而非其他方法的原因、相对优缺点，以及需要注意的常见陷阱。

注意

将您的应用程序从非 Oracle 平台迁移到 Oracle Exadata 不在本章讨论范围之内。

### 迁移策略

一旦您很好地理解了 Exadata 是什么以及它如何工作，就可以开始思考如何迁移数据库了。让我们回到两个大类——逻辑迁移和物理迁移。逻辑迁移涉及从一个数据库中提取数据并加载到另一个数据库中。物理迁移则是指将数据库逐块从一台数据库服务器提取并移动到另一台。在决定最佳迁移方法时，数据库的数据访问特性是一个关键考虑因素，这主要是由于在 Exadata 上访问数据的方式。OLTP 数据库倾向于使用单块读取并在所有表中更新数据，而数据仓库（DW）数据库通常针对全表扫描进行优化，并且只更新当前数据（如果有的话）。

Exadata 在存储节点上使用智能闪存缓存来优化单块读取，并提升 OLTP 数据库的整体性能。对于 DW 数据库，Exadata 使用智能扫描技术来优化全表扫描。从 Exadata 版本 11.2.3.3.x 开始，智能扫描会同时扫描闪存和磁盘存储以实现更高的吞吐量。智能扫描和闪存缓存优化方法的细节在第 2 章和第 5 章以及第 16 章中有介绍。逻辑迁移让您有机会对数据库进行更改，并针对 Exadata 平台进行优化。这些更改可能包括调整区大小、实施或重新设计当前的分区方案，以及使用混合列压缩压缩表。对于大表，这些都是非常重要的存储考量，对于 DW 数据库尤其如此。

由于 OLTP 应用程序倾向于在整个数据库中更新数据，根据您的分区方案实现方式，HCC 压缩可能不太适用。在最坏的情况下，HCC 甚至可能降低性能。虽然大区对 DW 数据库有益，但对于主要使用基于索引的访问和“随机”单块读取的 OLTP 数据库来说，益处较少。从 11.2.0.2 开始，Oracle 将分区创建为 8MB 的区，以实现更好的全段扫描性能（记录在 My Oracle Support Doc ID 1295484.1 中）。

物理迁移本质上不允许更改数据库中表和索引的存储参数，而逻辑迁移在重新定义存储、压缩、分区等方面提供了更大的灵活性。

将数据库迁移到 Exadata 最令人印象深刻的方式是结合逻辑和物理迁移，正如 Oracle 网站上的各种白皮书所述。其中一个例子涉及使用 Golden Gate 在大型字节序平台和 Exadata 之间进行近零停机时间迁移，描述了如何利用一个中间数据库作为暂存区。如果您将中间数据库前滚到一个已知的 SCN，您就可以在源端从同一个 SCN 开始提取事务。当 Golden Gate 提取进程正在捕获源端的更改时，您使用可传输表空间技术（在本章后面描述）将中间数据库转换为 Exadata 格式。字节序转换完成并且表空间插入到 Exadata 上一个原本为空的数据库后，您就开始传输并应用 Golden Gate 在源端挖掘到的已捕获事务。Golden Gate 应用进程将使用迁移期间源端发生的所有更改来更新 Exadata 数据库。在深入探讨组合逻辑和物理迁移的技术细节之前，让我们先从基础开始。



### 逻辑迁移

无论使用何种技术，逻辑迁移都包括从源数据库提取对象并将其重新加载到目标数据库。尽管逻辑迁移策略往往比物理策略更复杂，但由于以下优点，它们通常是更可取的选择：

*   分阶段迁移：不再接收更新的表和分区可以移出迁移窗口，从而减少最终切换期间需要移动的数据量。
*   选择性迁移：源数据库中通常包含过时的用户账户和不再需要的数据库对象。使用逻辑方法时，这些对象可以从迁移中简单地省略。旧数据库可以保留一段时间，以防之后需要某些未迁移的内容。
*   平台差异：数据会自动转换为目标数据库的块大小。自动处理从大端序到小端序的转换。
*   Exadata 混合列压缩（HCC）可以在数据移动前进行配置。也就是说，可以在 Exadata 数据库中定义表时使用 HCC，这样数据在加载到新数据库时就会被压缩。
*   区大小设定：可以在数据移动前，以最优的区大小预先创建目标表、分区和索引（例如，自 11.2.0.2 起，Oracle 为分区默认使用 8MB）。有关区大小对智能扫描相关性的讨论，请参阅第 16 章。
*   允许合并数据库：当 Exadata 用作整合平台时，此特性尤为重要。如果您的 Exadata 是 V2 或 X2-2 型号，数据库服务器上的内存可能是一个限制因素。V2 数据库服务器每台配置 72GB RAM，而 X2-2 每台服务器配备 96GB RAM（可升级）。在处理 10 个或更少的中大型数据库时，这可能是足够的内存，但在 Oracle 11.2 中，在一台服务器上托管 15 个或更多数据库已变得相当普遍。例如，Enitec 曾参与一个项目，使用 Exadata 来托管 PeopleSoft HR 和财务数据库。实施者为此请求了 15 个数据库。再加上他们计划用于其他应用程序的 10 个数据库，SGA 内存就成了一个真正的问题。解决方案当然是将这些其他独立数据库合并在一起，使它们能更高效地共享内存。数据库合并的问题在 Oracle 12c 及其可插拔数据库中得到了解决。这些可以是合并和整合数据库的完美工具。对于 X3 及以后的计算节点，内存短缺不再是大问题。它们默认配备 256GB RAM，可升级到每节点 768GB。X3-8 和 X4-8 每台服务器拥有 2TB DRAM。每个 X4-8 计算节点最多可配备 6TB DRAM，每个集群总内存可达 12TB。
*   预排序：如果使用“创建表为选择”方法或“插入为选择”方法（`CTAS` 或 `IAS`）通过数据库链接进行，在数据加载到目标数据库时也可以对数据进行排序，以提高索引效率、优化 Exadata 存储索引并实现更好的压缩率。

逻辑迁移基本上有两种方法。一种涉及从源数据库提取数据并加载到目标数据库。这通常被称为“提取与加载方法”。此方法常用的工具是 `Data Pump`、`Export/Import` 以及通过数据库链接的 `Create Table as Select`（`CTAS`）（或 `Insert ... as Select` - `IAS`）。另一种方法是在正常业务操作期间复制源数据库。当需要切换到新数据库时，取消复制并将客户端应用程序重定向到新数据库。这种技术通常被称为“基于复制的方法”。基于复制方法常用的工具是 `Oracle Streams`、`Oracle Data Guard`（逻辑备用）和 `Oracle Golden Gate`。也可以结合使用物理和逻辑迁移，例如在最终切换前很久就将（主要是）只读表空间复制过去，并通过某种复制方法（如 `Streams`）应用对它们的更改。请注意，`Oracle Streams` 在 12c 中已弃用，且未收到任何增强。

### 提取与加载

一般来说，在所有迁移策略中，提取与加载方法需要的停机时间最多，因为一旦开始提取并在整个迁移期间，所有 DML 活动都必须停止。数据仓库环境是例外，因为数据通常以“移入/移出”的方式组织。由于数据通常按日期范围分区，静态数据与仍在变化的数据是分开的。这些“只读”数据可以提前迁移，在最终迁移窗口之外，甚至可能在业务时间内进行。提取与加载策略的最大优势是其简单性。大多数 DBA 出于某种原因都使用过 `Data Pump` 或 `CTAS`，因此工具集是熟悉的。另一个巨大优势是它赋予您的控制权。`Exadata` 带来的一个重要新特性是混合列压缩（HCC）。由于您可以完全控制数据如何加载到目标数据库，因此在加载时使用 HCC 压缩表是一项相对简单的任务。提取与加载还允许您实施分区或更改分区策略。使用 `CTAS` 加载数据允许您在加载时对数据进行排序，这提高了 `Exadata` 存储索引的效率。有人可能会说所有这些都可以在迁移后完成，这没错，但如果可以在迁移过程中就完成，为什么还要移动两次数据呢？在某些情况下，如果不应用压缩，甚至可能无法将数据装入平台。在接下来的几节中，您可以阅读几种执行提取与加载迁移的方法。
