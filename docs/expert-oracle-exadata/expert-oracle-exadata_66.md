# 数据泵

数据泵是在数据库间迁移大量数据的优秀工具。数据泵由两个程序组成：`expdp` 和 `impdp`。两者均在《实用程序指南》中有文档说明。`expdp` 命令用于从源数据库提取数据库对象。它可以用于转储整个数据库的内容，或者更具选择性地按模式或表进行转储。与其前身导出工具 (`exp`) 类似，数据泵提取数据并将其保存到一个可移植的数据文件中。然后，该文件可以被复制到 Exadata，并使用 `impdp` 命令加载到目标数据库中。数据泵首次亮相于 Oracle 10g，因此如果你的数据库版本是 9i 或更早，你将需要使用旧的导出/导入 (`exp`/`imp`) 工具。

## 导出与导入

导出和导入自 Oracle 7 以来就已存在，尽管它们已经有些过时，但它们仍然是将数据和对象从一个数据库迁移到另一个数据库的非常有效的工具。尽管 Oracle 多年来一直在谈论弃用 `exp` 和 `imp`，但它们仍然是基础 12c 安装的一部分。

首先，你可以阅读关于数据泵的内容，了解如何使用它迁移到 Exadata。在介绍完这套新工具之后，你可以看看使用导出和导入迁移旧数据库的方法。请记住，数据泵的每个主要版本都会添加新特性和参数。请查阅 Oracle 文档，了解特定于你数据库版本的功能和特性。

## 数据泵参数

在继续之前，让我们先了解一下一些最相关的数据泵参数。以下是一些对数据库迁移有用的关键参数。

*   `FULL`, `SCHEMAS`, `TABLES`：这些选项是互斥的，用于指定导出是针对整个数据库、选定的模式，还是选定的单个表。请注意，在执行完整数据库导出时，某些模式（如 `SYS`、`MDSYS`、`CTXSYS` 和 `DBSNMP`）永远不会被导出。`FULL` 关键字在 12c 中得到了增强，在适用的情况下允许你结合使用可传输表空间技术。
*   `PARALLEL`：`PARALLEL` 参数指示数据泵将工作拆分成多个部分并并发运行。`PARALLEL` 可以极大地提升导出过程的性能。如果你计划使用 `PARALLEL` 关键字，请不要忘记在转储文件名中使用 `%U` 标识符来指定多个输出文件。
*   `NETWORK_LINK`：此参数指定目标数据库中要用于导出的数据库链接。它允许你从远程服务器导出数据库，通过数据库链接（在目标数据库中定义）直接通过网络拉取数据。根据《实用程序指南》，当与 `impdp` 一起使用此参数时，不涉及转储文件。网络链接被 Grid Control 用于通过“从数据库导入”过程自动化迁移过程，也用于跨平台备份的可传输表空间以及新的 12c 完全可传输导入。
*   如果你选择使用 `FLASHBACK_SCN`，你可以通过运行以下查询获取当前数据库的 SCN：
    ```
    SQL> select current_scn from v$database;
    ```
*   `COMPRESSION`：数据泵压缩是 11g 的新特性。在 10g 中，你只能压缩元数据，但在 11g 中，此功能扩展到了表数据。有效选项包括 `ALL`、`DATA_ONLY`、`METADATA_ONLY` 和 `NONE`。使用 `COMPRESSION=ALL` 选项，数据泵将一次测试导出的大小从 13.4G 减少到 2.5G，压缩率超过五倍。这是对存储空间非常显著的节省。在启用压缩进行测试时，我们完全预期它会减慢导出速度，但相反，它实际上将我们的导出时间从 39 分钟减少到了 9 分钟多一点。当然，不能期望总是如此。压缩效果高度依赖于实际数据。在我们的测试系统上，导出明显受 I/O 限制。但这确实表明，压缩可以显著减少导出数据库的存储需求，而不一定会减慢过程。遗憾的是，实时压缩表数据的功能直到 11gR1 版本才被引入。如果你的数据库是 10g，并且需要在将转储文件传输到 Exadata 之前进行压缩，你将需要使用外部工具如 `gzip`、`zip` 或 `compress` 来完成。请注意，在数据泵中使用数据 `COMPRESSION` 选项需要 Oracle Advanced Compression 许可。
*   `COMPRESSION_ALGORITHM`：Oracle 12c 让你在压缩转储文件时对所使用的算法有更细粒度的控制。你可以选择 `basic`、`low`、`medium` 和 `high` 压缩级别。与任何压缩一样，你是在用文件大小换取（CPU）时间。你应该测试是否值得更积极地压缩转储文件。使用此参数需要 Advanced Compression 选项。
*   `FLASHBACK_TIME`, `FLASHBACK_SCN`：数据泵默认不保证导出的读一致性，这可能会让你感到惊讶。要导出数据库的读一致性映像，你必须使用 `FLASHBACK_SCN` 或 `FLASHBACK_TIME` 参数。如果你使用 `FLASHBACK_TIME`，数据泵会查找与你指定时间相对应的最接近的系统更改号 (SCN)，并导出截至该 SCN 的所有数据。`FLASHBACK_TIME` 可以按如下方式传递给数据泵：
    ```
    FLASHBACK_TIME="to_timestamp('05-JUN-2014 21:00:00','DD-MON-YYYY HH24:MI:SS')"
    ```

## 导出策略考虑

在导出时，创建多个导出转储文件可能是个好主意。第一个可以是整个数据库的仅元数据导出转储。这样，你可以创建数据库或其子集的有效 DDL 文件 (`SQLFILE`=...)。例如，巧妙地组合使用 `SQLFILE` 和 `INCLUDE`/`EXCLUDE` 参数，可以让你创建所有用户及其权限的 DDL，而无需使用 `DBMS_METADATA` 提取这些信息。这对于从视图中移除提示也很有用：`DBA_VIEWS` 中 `TEXT` 列使用的 `LONG` 数据类型在 11.2 中需要一些巧妙的编码才能读取完整文本（如果视图的代码适合 4000 个字符，在 12.1 中可以从 `DBA_VIEWS.TEXT_VC` 检索）。将某个模式的所有视图的 DDL 转储到单个文件中，可以让你使用 `perl`、`awk` 或 `sed` 进行全局搜索和替换文本。另一次导出可以是你要导入的模式的小样本。`SAMPLE` 关键字可以在导出期间用于限制转储文件中的数据量。你可以在开发系统上使用这样的文件来完善流程。最终的导出文件将包含你迁移 Exadata 系统所需的一切。

**注意**
在 Exadata 上使用转储文件时，无论是在导出还是导入期间，你都应考虑使用数据库文件系统 (`DBFS`) 来存储文件。`DBFS_DG` 中可用的磁盘数量大大超过了计算节点上的内部磁盘数量。从 Grid Infrastructure 12.1.0.2 开始，你还可以选择使用 ASM 集群文件系统 (`ACFS`)。
