# Oracle 数据迁移：Streams 与 Golden Gate 对比

需要注意的是，`append_values`提示仅适用于 Oracle 11gR2 及以上版本。与本书第一版所述不同，Golden Gate 现在支持读取压缩表。要支持 Basic、Advanced (OLTP) 和 EHCC 压缩，唯一的方法是配置新的集成捕获机制。与挖掘在线重做日志的经典捕获不同，集成捕获直接与日志挖掘服务器交互，以逻辑更改记录 (LCR) 的形式接收数据更改。

> **注意**
>
> 您不能在 Oracle 12c 多租户数据库中使用经典捕获。

对于 Streams，文档指出其在 10g 数据库中不支持直接路径插入。11g 通过直接路径 API 提供了有限的支持。再次强调，请务必查阅 Oracle 文档以了解您的数据库版本支持哪些功能。如有疑问，请运行一些测试，查看是否获得了预期的压缩级别。然而，一个很大的改进是，Oracle 11.2 允许 Streams 捕获使用基本表压缩和 OLTP 压缩的表的数据。从 Oracle 11.2.0.2 补丁集开始，Streams/XStream 也可以捕获对 HCC 压缩表所做的更改。Oracle Streams 概念与管理指南在附录 B 中对此有更详细的解释。

Streams 支持 Oracle 的大多数数据类型，并且列表随每个版本而增加。以下是 Oracle Streams 和 Golden Gate 共有的优势摘要：

*   支持故障转移 (Oracle RAC 环境)
*   支持多租户数据库
*   支持表分区 (目标表)
*   支持 HCC 压缩，尽管与 CTAS 或 IAS 相比，您可能无法获得最佳的加载压缩率
*   支持表和索引的存储更改 (目标表)
*   能够更改索引策略 (目标表)
*   迁移期间对目标表的只读访问
*   支持源和目标之间的字节序差异
*   支持源和目标之间的（某些）字符集差异
*   支持源和目标之间不同的数据库块大小
*   支持在目标服务器上进行变更捕获，减少对源系统的性能影响
*   设置不复杂
*   与数据库版本无关
*   切换停机时间非常短
*   Oracle 企业管理器中存在支持。

Streams 和 Golden Gate 共有的缺点包括：

*   复制的表必须具有主键或唯一键。如果源表上没有主键，则必须提供一组能保证唯一性的列。如果不存在这样的组合，则无法复制该表。
*   某些数据类型不支持复制。
*   源系统上的日志挖掘会影响性能。考虑将变更捕获移到目标系统。
*   逻辑复制通常比其他迁移策略更复杂。

Streams 相对于 Golden Gate 的缺点：

*   Streams 在 Oracle 12c 中已弃用。它仍然可用（不像变更数据捕获 — CDC），但如 Streams 概念与管理指南附录 B 所述，它不支持多租户数据库。Golden Gate 通常被视为 Streams 的替代品。
*   数据挖掘可能严重影响源数据库的性能。
*   设置和维护更复杂。
*   支持 DDL 但存在问题。
*   如果复制中断，修复过程可能很麻烦。通常从头重建目标更容易。
*   SQL Apply 比 Golden Gate 更容易落后于源。
*   Streams 往往比 Golden Gate 需要更多的 CPU 资源。

Streams 相对于 Golden Gate 的优势：

*   Streams 包含在您的数据库许可中，而 Golden Gate 需要单独购买。
*   所有进程信息都存储在数据库中，提供了方便的远程访问。

那么，您应该使用 Streams 还是 Golden Gate？一方面，是技术层面。我们曾与使用过这两种产品的客户和同事交流。在每种情况下，他们都将该产品用于类似数据库迁移的简单复制到远程站点。我们得到的反馈压倒性地倾向于 Golden Gate。这些用户引用了 Golden Gate 易用性、稳定性和性能作为最重要的原因。除此之外，Golden Gate 支持非 Oracle 数据库源。Streams 运行在 Oracle 数据库内部，并且没有被移植到任何其他数据库平台。如前所述，Streams 包含在您的 Oracle 数据库许可中。Golden Gate 则不是，其基于源和目标服务器总核心数的成本不容小觑。如果您的数据复制需求超出了 Exadata 迁移，这项投资可能是非常值得的。如果您面临一次性迁移，考虑到成本（忽略团队需要熟悉该工具的事实），Streams 可能更合适。但另一方面，Streams 已被 Golden Gate 取代。许多读者可能仍在 11g Release 2 上，因此 Streams 被弃用的事实可能不会立即影响您。然而，当 Streams 对某些更有趣的 12c 特性缺乏支持成为限制因素时，那一天终将到来。

##### 何时使用 Streams 或 Golden Gate

Streams 和 Golden Gate 都提供了极大的灵活性和接近零的停机时间。它们非常适合需要更改目标表的迁移。这包括调整表和索引的区大小、实现分区策略和压缩表。它们还允许您更改模式、表空间和磁盘组等。

##### 使用 Streams 和 Golden Gate 的注意事项

由于 Streams 和 Golden Gate 通过重做日志传播数据库更改，您必须防止源数据库上的 `NOLOGGING` 操作。这可以在表空间级别使用 `alter tablespace force logging` 或在数据库级别使用 `alter database force logging` 来完成。以下是您必须防止的一些 `NOLOGGING` 操作示例：

*   `CREATE TABLE newtab NOLOGGING AS select * from ...`
*   `insert /*+ append */` ... (针对处于 nologging 模式的表) 或
*   `insert /*+ append nologging */` (专门覆盖默认的日志记录操作)
*   直接路径 SQL*Loader (针对处于 nologging 模式的表)

请注意，一些大型数据仓库数据库设计为使用 `NOLOGGING` 进行大规模数据加载，以节省 CPU 和磁盘空间（生成的归档日志更少）。这些数据库依赖于增量备份加上数据重新加载作为恢复策略。在此类数据库中，强制所有操作进行重做日志记录可能会导致不可接受的性能下降和归档日志生成。在这些情况下，配置 ETL 引擎在过渡期间对新旧 DW 系统同时执行加载是有意义的。尽管从 nologging 更改为 logging 加载操作看起来很简单，但在孤立的组织中，这可能被证明是无法改变的。相反，如果您使用依赖于挖掘重做日志的复制方法，则可能必须使用 `alter database force logging`。

#### 逻辑备库

另一种可能对数据库迁移有用的复制类型是逻辑备库数据库。逻辑备库是 Oracle 数据库的一个内置功能。它在 9i release 2 版本中作为 Data Guard（前身为备用数据库）的一个扩展或附加功能被引入。表面上，此策略听起来与 Streams 和 Golden Gate 非常相似，因为它基本上执行相同的任务。源数据库的更改从重做日志（或归档重做日志）中挖掘出来，转换为 SQL 语句，并在目标数据库上执行。与其他逻辑复制方法一样，数据库在复制运行时处于打开且可用状态。

逻辑备库比 Streams 或 Golden Gate 限制要多得多。因为逻辑备库实际上是从源数据库的物理、逐块副本实例化而来的，所以有几个规则开始起作用。源数据库和目标数据库的数据库版本必须相同，甚至补丁级别也要一致。

Oracle 支持在同一 Data Guard 配置中为 primary 和逻辑备库数据库进行异构配置。主说明可以在 My Oracle Support 上找到：Data Guard Support for Heterogeneous Primary and Logical Standbys in Same Data Guard Configuration (Doc ID 1085687.1)。阅读此说明会揭示 Oracle Data Guard 支持的操作系统组合矩阵。然而，在你抱有过高期望之前，对于“不能在不同平台甚至不同字节序之间使用 Data Guard”这一规则，只有极少数例外。

你可能会惊讶地发现，虽然索引和物化视图可以在目标数据库中轻松创建，但实施分区策略则更为困难。你必须暂停复制，使用分区重新创建表，然后恢复复制。如果你的复制表与主库不同步，那你就麻烦了。通常用于使一切恢复正常的 `DBMS_LOGSTDBY.INSTANTIATE_TABLE` 过程会移除备用数据库上的现有表，并将其替换为主库上存在的表。

表压缩也不是 100% 支持的。请记住，HCC 压缩仅适用于直接路径加载，而逻辑备库使用常规插入应用更改。更糟的是，批量更新是通过逐行更新来复制的。实施 EHCC 通常是 Exadata 一个非常重要的性能策略。缺乏对此功能的支持是逻辑备库和一次性迁移方法的一个显著缺点。当然，在切换完成后，你可以执行一系列 `alter table move` 命令来达到正确的压缩率。

不支持的数据类型列表可以在 Data Guard 概念与管理指南的 附录 C 中找到。

创建逻辑备库比创建其近亲物理备库更为复杂。配置和实例化逻辑备库有很多步骤。列出所有步骤会进一步扩展本章内容。高级步骤已足够：

```
1.  创建一个物理备库数据库（请注意，这将在后面的“物理备库”部分稍后介绍）。
2.  停止物理备库数据库上的重做应用。
3.  在主数据库中配置 Log Miner。
4.  将物理备库数据库转换为逻辑备库。
5.  打开逻辑备库数据库并重启重做应用。
6.  验证逻辑备库数据库正在应用源数据库的更改。
```

逻辑备库使用 SQL 应用（SQL Apply）而非重做应用（Redo Apply）技术将更改从源数据库复制到目标。如前所述，源数据库中的更改被转换为 SQL 语句并传送到目标端执行。虽然这听起来相当直接，但由于其实现方式，仍有一些有趣的含义需要考虑。对于 DML 语句，有几件事需要记住：

*   批量更新一次执行一行。因此，如果你用一条命令更新 100,000 条记录，你将在备用数据库上应用 100,000 条单独的命令。
*   直接路径插入以常规方式执行。
*   并行 DML 不会并行执行。

DDL 语句的复制没有任何特殊的配置要求。然而，对于 DDL 语句有一些注意事项：

*   并行 DDL 不会并行执行。
*   CTAS（Create Table As Select）执行为 `create table`、`insert`、`insert`、`insert`。

你应该密切关注备用数据库上的应用过程，以确保批量操作不会过多干扰性能。当在逻辑备库数据库上复制对表的大规模批量插入时，你会看到相当多的开销，并且由于撤销应用和块清理，逻辑备库上查询的性能会受到影响。

##### 何时使用逻辑备库

逻辑备库有一份非常窄的、使其区别于其他更可能的迁移方法的益处清单。它提供完整的数据库复制，并允许在复制期间对备用数据库进行读取访问。因此，在准备好最终切换之前，备用数据库可用于卸载生产数据库上资源密集型的报告操作。可以在备用数据库上创建表和物化视图，并且可以创建或删除索引以提高性能。

##### 逻辑备库需要注意什么

`NOLOGGING` 操作不会产生足够的重做日志，这意味着它们不会被复制到备用数据库。与任何 Data Guard 设置一样，请务必使用 SQL*Plus 在主库和逻辑备库数据库上执行 `alter database force logging` 命令来防止 `NOLOGGING` 操作。考虑到它缺乏对修改目标表存储、压缩和分区的支持，你可能会选择使用物理备库方法。如果你不需要完整的数据库复制，Streams 和 Golden Gate 是很好的替代方案，它们提供了许多逻辑备库数据库所缺乏的功能支持。

#### 并行加载

这有点像“自己动手”的迁移方法。在此策略中，数据更改通过内部开发的应用程序馈送给数据库。因此，配置应用程序以与当前生产数据库并行更新 Exadata 表是一项小任务。然后，在正常业务时间内将历史数据迁移到 Exadata。根据数据量，这可能需要几天到几周的时间。一旦所有表都迁移到 Exadata，对旧生产系统的馈送就被切断，Exadata 成为新的生产数据库。

#### 逻辑迁移总结

在大多数情况下，逻辑迁移为重新配置区大小、实施或修改分区策略以及使用 HCC 压缩表提供了最大的灵活性。这种灵活性是以复杂性和限制为代价的，应该与本章介绍的一些物理迁移策略的简单性进行权衡和考量。归根结底，你需要为大型表正确调整区大小，实施良好的分区策略，并使用 HCC 压缩只读表。在你切换到 Exadata 之前，使用本文讨论的大多数逻辑迁移策略就可以完成这些工作。


### 物理迁移

物理数据库迁移，顾名思义，就是创建源数据库（或数据库部分）的逐块副本并将其移动到`Exadata`的过程。物理迁移比本章前面讨论的一些逻辑迁移策略要简单得多。正如您可能预料的那样，除了选择不迁移某些不必要的表空间外，它不允许对目标数据库进行任何更改。这意味着您将无法修改表和索引的区大小、更改索引策略、实现分区或应用 HCC 表压缩。此外，如果您的数据库诞生于`7.x`或`8i`时代，由于当时文件系统的限制，您可能拥有数百甚至数千个数据文件。您还记得`2GB`的文件大小限制吗？

所有旨在简化或"Exadata 化"您的数据库的任务都必须在迁移后完成。然而，毫无疑问，物理迁移是将数据库迁移到`Exadata`的最快方式。对于所有物理迁移策略，除了可传输表空间（`TTS`）之外，新的`Exadata`数据库开始时都是单实例数据库。迁移后需要执行步骤，将数据库及其所有实例注册到集群就绪服务（网格基础架构）。因为物理迁移创建的是源数据库的精确副本，所以限制列表非常短：

*   源数据库版本必须是`11.2`或更高。
*   一种选择是在复制到`Exadata`之前将生产数据库升级到`11.2`或更高版本。升级通常是一个快速过程，不依赖于数据库表的大小，而是依赖于数据库中的对象数量和配置的选项。（然而，升级后`PL/SQL`包和视图的重新编译可能需要一些时间。）
*   源平台必须获得运行`Oracle 11.2`或更高版本的认证。您可以查看`My Oracle Support 认证矩阵`，了解哪些 Linux 发行版分别获得了`Oracle 11.2`和`12.1`的认证。
*   源平台必须是小端格式。当使用在`Exadata`上创建新数据库并使用跨平台可传输表空间的混合解决方案时，源数据库可以位于具有不同字节序的不同平台上（支持的平台列在`v$transportable_platform`视图中）。

执行物理迁移有三种策略：备份与恢复、物理备机以及带增量备份的跨平台可传输表空间。在本节中，我们将讨论这些策略、它们的工作原理以及它们最适用的场景。

## 备份与恢复

备份与恢复策略使用`Oracle`的恢复管理器（`RMAN`）创建源数据库的完整备份，然后将其恢复到`Exadata`平台。除非您计划在此过程中关闭数据库，否则源数据库必须在`归档日志`模式下运行。备份和恢复过程可以通过一次完整备份完成。这是将较小数据库迁移到`Exadata`的最佳方式，因为较小的数据库所需时间要少得多。较大的数据库可能需要数小时进行备份和恢复。对于这些数据库，可以通过先进行完整备份，然后进行增量备份，分两次完成该过程。

完整备份和恢复是众所周知的过程，此处不详细列出。`RMAN`备份和恢复/还原的一条规则是不能跨平台使用。虽然在`MOS 文档 ID 1079563.1`" `RMAN DUPLICATE/RESTORE/RECOVER 混合平台支持`"中有记录的少数例外情况，但它们几乎不值一提，因为实在太少了。更有趣的是一个新的`12c`特性，您可以使用备份集创建跨平台备份，并将转换卸载到目标数据库执行。当前的实现不支持移动到不同的字节序，这使得该特性对于迁移到`Exadata`的吸引力降低，因此，出于同样的原因，此处不予讨论。

请注意，除非您的源数据库是`12c`数据库，否则您将无法立即使用新的多租户数据库架构。`11.2`数据库必须先迁移到`12c`，然后才能将其转换为可插拔数据库（`PDB`）。如果您乐意继续将数据库用作非`CDB`，则无需采取进一步措施。但是，如果您正在将多个数据库整合为一个作为`PDB`的容器数据库，则需要进一步的步骤。


## 全备份与恢复

使用 RMAN 创建要迁移的数据库的全备份。备份文件暂存在文件系统中，或以其他方式使其对 Exadata 可访问。然后，将备份恢复到 Exadata 平台。请注意，本节介绍的是全恢复，它无法克服数据库字节序的差异。但是，只要字节序保持一致，您可以执行跨平台转换。这涉及本章未涵盖的额外步骤。换句话说，您不能将数据库从大端转换为小端，但可以将 Windows 转换为 Linux。在 Exadata 上恢复备份的高级步骤如下：

执行迁移前任务：
   为 Exadata 上的数据库在 `tnsnames.ora` 文件中创建一个条目。
   将密码文件和参数文件复制到 Exadata。
   限制用户和应用程序对数据库的访问。
   执行数据库全备份。
   将文件复制到 Exadata。
   在 Exadata 上，使用调整后的服务器参数文件将数据库启动到 `nomount` 模式。
   恢复控制文件。
   挂载数据库。
   恢复数据库。
   恢复数据库。
执行迁移后任务：
   将数据库转换为 RAC 并创建服务名。
   重新配置客户端的 `tnsnames.ora` 文件、用于连接到 Exadata 数据库的配置文件。
   使数据库可供用户和应用程序使用。

如您所见，这是一个相当直接的过程。遵循这些步骤后，您就得到了一个在 Exadata 上运行的、与生产数据库完全一致的副本。不过，有几件事需要注意。如果您的源数据库使用文件系统进行数据库存储，或者您使用了 ASM 但 Exadata 上的磁盘组名称不同，您可以在开始恢复过程之前，通过更改 `init.ora` 文件中的 `db_create_file_dest` 和 `db_recovery_file_dest` 参数，将恢复的文件重定向到新的磁盘组名称。表 13-4 展示了如何使用这些参数将磁盘组名称从源数据库重映射到 Exadata。

表 13-4.
磁盘组重映射

| Init.ora 参数             | 源数据库            | Exadata 数据库 |
| :------------------------ | :------------------ | :------------- |
| `db_create_file_dest`     | `’+DATA_FILES’`     | `’+DATA’`      |
| `db_recovery_file_dest`   | `’+RECOVERY_FILES’` | `’+RECO’`      |

如果您的数据库使用多个 ASM 磁盘组来存储数据库文件，请使用 RMAN 的 `db_file_name_convert` 子句将文件名重映射到 ASM 磁盘组。例如：
```
db_file_name_convert= \
  (’+DATA_FILES/exdb/datafile/system.737.729723699’,’+DATA’)
db_file_name_convert= \
  (’+DATA_FILES1/exdb/datafile/sysaux.742.729723701’,’+DATA’)
```

细心的读者现在无疑会指出，使用 RMAN 的 `duplicate` 命令可以实现完全相同的事情，这当然是正确的。然而，根据我们的经验，管理层对生产环境的访问权限通常极为谨慎，即使对于复制过程也是如此。上面解释的额外步骤通常是值得的——备份可能已经存在，只需要使用归档日志进行前滚。而且，还有一个额外的好处，甚至不需要“接触”生产系统，这个操作通常需要精心设计的变更控制流程。

#### 增量备份

如果您需要迁移的数据库很大，且迁移时间有限，您可以考虑使用增量备份和恢复策略，以减少最终切换到 Exadata 的停机时间。遗憾的是，增量方法也不支持字节序格式转换。基本步骤如下：

执行迁移前任务：
   为 Exadata 上的数据库在 `tnsnames.ora` 文件中创建一个条目。
   将密码文件和参数文件复制到 Exadata。
   创建数据库的 0 级备份。
   将备份文件复制到 Exadata。
   在 Exadata 上，将数据库启动到 `nomount` 模式。
   恢复控制文件。
   挂载数据库。
   恢复数据库。
   限制用户和应用程序对数据库的访问。
   创建源数据库的增量 1 级备份。
   将增量备份文件复制到 Exadata。
   恢复数据库。
执行迁移后任务：
   将数据库转换为 RAC 并创建服务名。
   重新配置客户端的 `tnsnames.ora` 文件。
   使数据库可供用户和应用程序使用。

您肯定已经注意到，增量备份方法的步骤与全备份方法几乎相同。重要的区别在于所需的停机时间。迁移数据库的大部分时间花在全备份本身上。这可能需要数小时才能完成，而在全备份方法中，这全部都是数据库停机时间。增量备份方法使用 0 级备份代替全备份。实际上，0 级备份与全备份相同，只是它具有特殊属性，可以作为应用增量备份的基线。因此，使用增量方法，数据库在迁移的最长部分期间可以保持在线供用户访问。

在拍摄增量 0 级备份之前，确实应该在源数据库上激活块更改跟踪。启用块更改跟踪后，Oracle 会通过翻转一个小位图文件中的位来跟踪自上次 0 级备份以来所有已更改的块。这意味着当您执行增量 1 级备份时，Oracle 不必扫描数据库中的每个块来查看哪些块已更改。它只需在块更改跟踪文件中查找它们即可。使用块更改跟踪文件几乎可以保证减少增量备份所需的时间。我们见过这样的案例：一个 13TB 的数据仓库使用 BCT 文件进行增量备份只需 15 分钟，而之前需要 9 小时。要查看块更改跟踪是否处于活动状态，请在 SQL*Plus 中执行以下查询：
```
SQL> SELECT status FROM v$block_change_tracking;
STATUS
----------
DISABLED
```
您不必关闭数据库来激活块更改跟踪。它可以随时完成。要启用块更改跟踪，请执行以下命令：
```
SQL> ALTER DATABASE ENABLE BLOCK CHANGE TRACKING;
Database altered.
```
默认情况下，块更改跟踪文件创建在 `db_create_file_dest` 位置。如果未设置此参数，您需要设置它或为块更改跟踪文件指定文件名：
```
ALTER DATABASE ENABLE BLOCK CHANGE TRACKING
USING FILE ’/u01/app/oracle/admin/<ORACLE_SID>/bct/bct.dat’;
```
迁移后任务对于将单实例 Exadata 数据库转换为多实例 RAC 数据库是必需的。为此，您需要为实例到在线重做日志线程添加映射，并添加额外的撤销表空间。将新的 RAC 数据库注册到 Clusterware 中也是非常稳妥的做法。一旦数据库在 Exadata 上恢复并运行，您应该执行一次数据库全备份。假设数据库运行在归档日志模式下，您可以在数据库重新上线并服务于最终用户和应用程序之后进行此操作。

> 注意
> 增量备份的一个轻微变体是使用 Exadata 数据库作为 Data Guard 物理备库来自动执行前滚副本的过程。本章稍后将介绍此内容。


#### 何时使用备份与恢复

使用 RMAN 执行简单的数据库备份与恢复，是所有 DBA 应当能够驾轻就熟的操作。这使得备份与恢复成为一种极具吸引力的策略。它最适用于那些暂时不需要分区和 HCC 压缩的 OLTP 数据库。它同样适用于已经运行在小端序、64 位平台上的数据仓库数据库，尽管迁移后可能还需要执行一些后续步骤。

#### 考虑备份与恢复策略时的注意事项

增量备份与恢复不支持字节序转换。如果这是必需的要求，您可能更适合使用可传输表空间迁移策略，或者重新考虑本章前面介绍过的一些逻辑迁移策略。迁移到 Exadata 后，对象需要重建才能利用其存储特性，例如 HCC。数据文件也是如此。如果您的数据库包含大量版本历史，那么您可能拥有许多小的数据文件。在 Solaris 2.5 和早期 AIX 版本的时代，文件系统限制规定您不能创建大于 2GB 的数据文件。在 Exadata 时代，重新组织这些分散的数据文件及其上存储的段是明智之举。

与此相关的问题还包括物化视图、微分区表、块碎片等，可以在新的 Exadata 平台上重新审视这些问题，以求简化并优化数据库。如果一个数据库拥有大量历史数据且从未进行过物理重组，那么行很可能跨越多个块形成行链。

#### 可传输表空间

可传输表空间 (TTS) 可用于将源数据库的子集迁移到 Exadata。为此，您需要在 Exadata 上运行一个数据库来承载这些数据库子集。我们经常将 TTS 描述为一种“修剪与嫁接”过程。此方法允许将一组表空间从源数据库复制并安装到一个活动的 target 数据库中。与前面描述的完整备份与恢复过程不同，您可以将单个表空间插入到 12c 可插拔数据库中。这样，您就省去了将非 CDB 转换为 PDB 的部分工作。如果源数据库中已实施了模式级整合，此任务将进一步简化。假设您的模式都各自良好地包含在自己的（大文件）表空间中。将它们迁移到 Exadata 是一种非常优雅的迁移方式，不是吗？

标准的 TTS 方法基于数据文件的映像副本。该过程相当简单，尤其是在没有 Data Guard 物理备用数据库的情况下，但在开始之前您需要注意一些重要事项。该过程的重要限制如下所述，如第 15 章中提到：

*   TTS 支持不透明类型，如 `RAW` 和 `BFILE`，但这些类型不进行跨平台转换。这些类型的结构仅对应用程序可知，字节序格式的任何差异都必须由应用程序处理。无论对不透明类型的使用是直接的还是间接的，类型和对象都受此限制。
*   只要目标数据库的版本与源数据库相同或更高，就支持数据库版本差异。
*   待迁移的表空间在迁移过程中必须置于只读模式。
*   理想情况下，源数据库和目标数据库的字符集应相同。此规则也有例外。如果您计划在迁移过程中更改字符集，请参阅 Oracle 文档。
*   具有底层依赖关系（如物化视图和表分区）的对象不可传输，除非它们包含在同一组中。表空间组是在一个操作中一起移动一组相关表空间的方式。
*   在传输加密表空间之前，必须先将 Oracle wallet 复制到目标系统并为目标数据库启用。一个数据库只能有一个用于 TDE 的 Oracle wallet。如果您要迁移到的数据库已有 wallet，则需要使用 Data Pump 来导出和导入表数据。
*   未使用块级加密但包含使用列加密的表的表空间，无法使用 TTS 传输。在这种情况下，Data Pump 可能是最佳替代方案。
*   自 10g Release 2 起，TTS 支持包含 XML 数据类型的表空间。在该版本中，您需要使用原始的 `exp`/`imp` 实用程序来提取元数据。从 11.1 开始，您必须使用 `expdp`/`impdp`。传输 XML 数据还有其他限制和注意事项。完整列表请参阅 Oracle XML DB 开发人员指南。要列出包含 XML 数据类型的表空间，请运行以下查询：
    ```sql
    SQL> select distinct p.tablespace_name
           from dba_tablespaces p,
                dba_xml_tables  x,
                dba_users       u,
                all_all_tables  t
          where t.table_name = x.table_name
            and t.tablespace_name = p.tablespace_name
            and x.owner = u.username;
    ```

跨平台可传输表空间 (XTTS) 支持大多数但并非所有平台之间的转换。要确定您的平台是否受支持，请运行以下查询：

```sql
SQL> SELECT * FROM V$TRANSPORTABLE_PLATFORM ORDER BY PLATFORM_NAME;
```

```
PLATFORM_ID PLATFORM_NAME                               ENDIAN_FORMAT
----------- ----------------------------------- --------------
          6 AIX-Based Systems (64-bit)                  Big
```
