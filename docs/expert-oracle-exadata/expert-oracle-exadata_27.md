# 6. Exadata 并行操作

Exadata 在执行并行操作方面并没有什么特殊方法，这是运行 `11gR2` 和 `12c` 的其他平台也具备的。然而，并行处理是 Exadata 的一个关键组件，因为高效处理数据仓库工作负载是 Exadata 的主要设计目标之一。此外，因为卸载/`智能扫描` 依赖于直接路径读取，而并行查询从进程可以使用直接路径读取，所以并行操作具有了全新的重要性。传统上，高效使用并行查询需要仔细控制并发性，以便在不使系统过载的情况下最大限度地利用可用资源。Oracle 先前为在多用户环境中使用而尝试对并行操作进行节流的做法并不完全成功。`11gR2` 版本提供了一些控制并行操作的新功能。特别是引入了一种排队机制，允许更有效地管理并发并行进程的数量。与以往的尝试相比，这种方法似乎更适合在不使可用资源过载的情况下实现高度并行化。`11gR2` 还引入了 Oracle 自动根据每条语句计算并行度的能力。`12c` 版本引入了 `并行语句排队` 增强功能和进程内存使用控制。`12c` 版本还引入了一个 `数据库资源管理器 (DBRM)` 指令，允许关键查询绕过并行语句队列，以及一个新的数据库参数来限制总的 `程序全局区 (PGA)` 大小，其目的是避免因过度使用物理内存而导致高交换率。

在本章中，我们将重点讨论并行查询。这里将不涵盖数据库提供的其他形式的并行操作，例如恢复并行性。我们将简要介绍并行查询机制，并具体演示排队机制和自动并行度如何在 Exadata 上工作。我们还将简要介绍 `11gR2 内存中并行执行` 功能，并讨论 Exadata 存储如何并行化 `I/O` 操作。我们不会涵盖并行查询或并行 `DML` 的所有细节，并假设读者熟悉基本的 Oracle 并行概念。另请注意，讨论和示例将参考 Oracle 数据库 `11g 第 2 版 (11.2.0.4)` 和 `12c 第 1 版 (12.1.0.2.0)` 中的行为。在大多数情况下，这些说明也同等适用于 `11.2.0.1` 版本。例外情况会被明确指出。
