# 在 Oracle 执行计划中识别 Smart Scan 迹象

在执行计划中找到`STORAGE`关键字固然重要，但检查执行计划下方的“谓词信息”部分是否显示了`storage()`谓词同样关键。这是判断执行计划是否有能力进行智能扫描的谓词下推（智能过滤）的一个良好指标。遗憾的是，这并不意味着谓词下推一定已被尝试。在某些情况下，即使存在`storage()`谓词，也不能保证谓词下推一定会发生。这时，Oracle 性能指标就会很有帮助，我们稍后会详细讨论。

总之，执行计划中必须同时包含`STORAGE`关键字和`storage()`谓词，一个全表扫描访问路径才有能力进行智能扫描的谓词下推。如果你在执行计划中看到了`STORAGE`行，但其下没有`storage()`谓词，那么谓词下推甚至不会被尝试，不过你仍然可能受益于列投影下推，即只有所需的列由存储单元返回。在真实的查询计划中，你很可能会看到多个`storage()`谓词（以及多个全表扫描操作），因为你将进行多表连接。

## `STORAGE`关键字与`storage()`谓词的含义

如果你在执行计划中看到了`STORAGE`选项和`storage()`谓词，那么很有可能会尝试进行智能扫描。谓词下推在原则上是可能的，但不能保证每次运行查询时都会发生智能扫描。另一方面，如果在执行计划中没有看到`STORAGE`关键字，那么在该执行计划步骤中，相应的段上就不可能发生智能扫描。当执行计划行中没有`STORAGE`关键字时，意味着此行源操作符无法使用存储单元的智能特性，因此也无法将任何`storage()`谓词下推到存储单元中。

## 在物化视图和索引段上的应用

你之前可能了解到，智能扫描也可以在物化视图表段上进行。这是因为物化视图表段在物理上与常规表完全相同。以下是对应的执行计划摘录：

```sql
select count(*) from mv1 where owner like 'S%'
--------------------------------------------------------------------
| Id  | Operation                     | Name | E-Rows | Cost (%CPU)|
--------------------------------------------------------------------
|   0 | SELECT STATEMENT              |      |        |   139K(100)|
|   1 |  SORT AGGREGATE               |      |      1 |            |
|*  2 |   MAT_VIEW ACCESS STORAGE FULL| MV1  |   2089K|   139K  (1)|
--------------------------------------------------------------------
Predicate Information (identified by operation id):
---------------------------------------------------
2 - storage("OWNER" LIKE 'S%')
    filter("OWNER" LIKE 'S%')
```

以下是可以尝试智能扫描的另外两个例子。第一个是当扫描常规 B*树索引段时，请求了索引快速全扫描：

```sql
SELECT /*+ INDEX_FFS(t2) */ AVG(LENGTH(owner)) FROM t2 WHERE owner LIKE'S%'
---------------------------------------------------------------------
| Id  | Operation                     | Name  | E-Rows | Cost (%CPU)|
---------------------------------------------------------------------
|   0 | SELECT STATEMENT              |       |        |  5165 (100)|
|   1 |  SORT AGGREGATE               |       |      1 |            |
|*  2 |   INDEX STORAGE FAST FULL SCAN| T2_I1 |    597K|  5165   (2)|
---------------------------------------------------------------------
Predicate Information (identified by operation id):
---------------------------------------------------
2 - storage("OWNER" LIKE 'S%')
    filter("OWNER" LIKE 'S%')
```

在第二个例子中，在位图索引段上请求了索引快速全扫描：

```sql
SELECT /*+ INDEX_FFS(t1) */ AVG(LENGTH(owner)) FROM t1 WHERE owner LIKE'S%'
Plan hash value: 3170056527
-----------------------------------------------------------------------------------
| Id  | Operation                             | Name        | E-Rows | Cost (%CPU)|
-----------------------------------------------------------------------------------
|   0 | SELECT STATEMENT                      |             |        |   505 (100)|
|   1 |  SORT AGGREGATE                       |             |      1 |            |
|   2 |   BITMAP CONVERSION TO ROWIDS         |             |    597K|   505   (0)|
|*  3 |    BITMAP INDEX STORAGE FAST FULL SCAN| BI_T1_OWNER |        |            |
-----------------------------------------------------------------------------------
Predicate Information (identified by operation id):
---------------------------------------------------
3 - storage("OWNER" LIKE 'S%')
    filter("OWNER" LIKE 'S%')
```

在这两种情况下，你看到段都是使用快速全扫描进行扫描的，这就像在索引段上进行全表扫描一样。并且，全扫描操作上存在`STORAGE`选项和`storage()`谓词，这表明可以尝试进行智能扫描谓词下推。

## 谓词下推的尝试与实际执行

你可能会奇怪为什么反复读到“可以尝试智能扫描谓词下推”。措辞如此谨慎的原因很简单：在某些情况下，智能扫描要么未被采用，要么在运行时启动但未完成。是的，执行计划的结构可能会让你相信某个查询将使用智能扫描，但智能扫描是否实际执行，首先取决于是否选择了直接路径读取来扫描该段。此外，即使选择了直接路径读取并执行了智能扫描，在智能扫描执行过程中的某个时刻（或开始时），也可能做出不同的决定。这取决于多种因素，我们将在后面讨论其中一些。这时，会话统计数据就真正发挥作用了，它能帮助性能分析师确定所观察到行为的原因。

这个听起来相当抽象的概念最好用一个例子来解释。你可以先尝试在索引组织表段上进行智能扫描，以了解额外的 Oracle 指标有多大价值。请注意，在撰写本文时当前的 Oracle 版本中，Oracle 尚未在索引组织表段上实现智能扫描功能。因此，这是练习使用智能扫描相关指标的一个好例子。

请查看此处显示的执行计划；它来自一个在索引组织表的索引段上使用索引快速全扫描的查询：

```sql
select avg(length(state)) from t1_iot where date_created >
to_date('07.11.2013','dd.mm.yyyy')
Plan hash value: 379056979
-------------------------------------------------------------------------------------------
| Id  | Operation             | Name              | Rows  | Bytes | Cost (%CPU)| Time     |
-------------------------------------------------------------------------------------------
|   0 | SELECT STATEMENT      |                   |       |       |   402 (100)|          |
|   1 |  SORT AGGREGATE       |                   |     1 |    17 |            |          |
|*  2 |   INDEX FAST FULL SCAN| SYS_IOT_TOP_19445 |   740K|    12M|   402   (6)| 00:00:01 |
-------------------------------------------------------------------------------------------
Predicate Information (identified by operation id):
---------------------------------------------------
2 - filter("DATE_CREATED">TO_DATE(' 2013-11-07 00:00:00', 'syyyy-mm-dd
              hh24:mi:ss'))
```


与之前列出的执行计划不同，在该执行计划行源（第 2 行）中未列出 `STORAGE` 选项，也没有 `storage()` 谓词，这表明存在将筛选谓词下推至存储单元的机会。该计划无法使用任何 Smart Scan 功能；它将进行传统的块 I/O。可以立即跳过查看额外的 Exadata 和单元指标，因为执行计划本身表明它无法使用兼容 Smart Scan 的行源代码路径。
