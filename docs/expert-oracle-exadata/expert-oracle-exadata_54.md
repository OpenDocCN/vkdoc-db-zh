# 行迁移与链接：对性能的影响及处理策略

行迁移是另一种情况，需要处理因更新操作导致行数据从当前位置移出并进入另一个数据块的情形。在这种情况下，只有行头片段保留在原位置，并包含一个指向行其余部分的前向指针。通常，更新操作是导致行迁移的元凶。在执行更新前，整行数据位于同一个数据块中，如下所示：

```
Start dump data blocks tsn: 5 file#:5 minblk 131 maxblk 131

block_row_dump:

tab 0, row 0, @0x1f7a

tl: 6 fb: --H-FL-- lb: 0x0  cc: 1

col  0: [ 2]  c1 02

tab 0, row 1, @0x1f71
```

而在更新之后，同一个数据块看起来则大不相同：

```
Start dump data blocks tsn: 5 file#:5 minblk 131 maxblk 131

block_row_dump:

tab 0, row 0, @0x1f77

tl: 9 fb: --H----- lb: 0x2  cc: 0

nrid:  0x01603167.0

tab 0, row 1, @0x1f6e
```

迁移后的行在原始位置甚至不包含任何列数据（`cc: 0`），仅保留了一个`NRID`（下一个行片段地址）。

链接的行对智能扫描构成一个问题。链接行的“下一个”行片段可能位于段内的任何位置。由于`ASM`条带化机制，无法保证链接行的下一个行片段与行头片段位于同一个存储单元中。因此，一个链接行在物理上可能被分散到多个不同的存储单元。考虑到存储单元在智能扫描期间从不相互通信，那么当需要时，如何能够构建出完整的行呢？目前`cellsrv`（单元服务）的解决方式是：每当智能扫描遇到一个链接行（并意识到需要获取其下一个行片段）时，如果无法在当前正在扫描的数据块中定位到`NRID`，`cellsrv`就会回退到对该行执行常规的数据块 I/O，并将整个数据块送回数据库层进行常规处理。在此情况下——即`NRID`已找到时——统计项“`chained rows processed by cell`”（存储单元处理的链接行）会递增。

如果`NRID`不在智能扫描当前处理的数据块中，那么该数据块必须被发送到`RDBMS`层进行处理。一旦收到，数据库层可以从行头片段中提取下一个行片段的数据块地址，并向`ASM`条带化物理存放该数据块的相应存储单元发出数据块读取 I/O。此优化背后的原因和基本问题，与为何一致性读回滚必须在数据库层而不是在存储单元中完成类似——此操作所需的一些数据块可能正好位于另一个存储单元中，而存储单元之间从不通信。`RDBMS`层随后必须对`NRID`指示的数据块发出单数据块读取，并进一步处理它。

这种行为意味着，如果你的智能扫描遇到大量链接行并且必须获取它们的下一个行片段，其性能可能会显著下降。如果你比较幸运，只需要访问存在于行头片段中的列，那么对于这些数据块就不必回退到数据库数据块 I/O 模式，你的智能扫描将会保持高速。

> 注意
> 下次为表添加列时请记住这一点！从智能扫描的角度来看，执行表重组并将访问最频繁的列放在前面可能更高效。或者，如果你的数据访问热度不高，可以考虑使用`HCC`（混合列压缩），因为其中的这些限制会被放宽。

如果你必须持续获取下一个行片段，那么你的智能扫描将不断被中断，回退到数据块 I/O，数据库层开始进行逻辑读取（并且如果这些数据块没有缓存在缓冲区缓存中，可能还会进行单数据块物理读取）。这意味着你的查询最终大部分时间都在等待随机的单数据块读取，而不是高性能的智能扫描。以下是一个会话在五秒钟内的快照（一个虚构且不切实际的最坏情况），该会话不得不与大量链接行作斗争：

此链接行性能问题仅适用于常规数据块以及使用常规块级压缩（`BASIC`或`ADVANCED`）压缩的数据块。幸运的是，对于`HCC`压缩的表来说，这完全不是问题，因为在`EHCC`（扩展混合列压缩）中，行和列在物理上是以不同方式组织的。此外，当全表扫描通过一个段时，此问题也不适用于迁移的行。全扫描/智能扫描会直接忽略迁移行的行头片段，因为整行数据在物理上位于其他地方。请注意，更新可能对`HCC`压缩表产生副作用，如第 3 章所述，尽管链接行不是问题。

另一个有趣的链接行特性情况是当一个表有超过 255 列时。即使总行大小足够小，可以放入单个数据块中，但超过 255 列，Oracle 仍会进行块内链接，其中行被链接，但所有行片段在物理上都位于同一个数据块内。这是必要的，因为 Oracle 在 Oracle 8.0 中将列限制从 255 增加到 1000 时，希望保持向后兼容性。行片段中的“列计数”字节只有一个字节，允许每个行片段最多 255 列，但通过链接，你可以在下一个行片段中拥有更多列。合著者`Tanel Poder`在他的博客`blog.tanelpoder.com`上有一个与此情况相关的有趣案例研究。

#### `chained rows rejected by cell`（存储单元拒绝的链接行）
此统计信息显示有多少链接行未在存储单元中处理。此统计信息应该很少递增。在智能扫描处理中，存储单元会将处理完的数据块结果放在一个输出缓冲区中，然后再将结果发送到查询会话的`PGA`。每当在存储单元上处理链接行的结果无法放入此输出缓冲区时，此统计信息就会递增 1。


#### 单元跳过的链式行

这是处理 Exadata 上链式行时最重要的统计数据，与“表提取继续行”统计项一同作用。一个简单的测试可以更好地演示这一点。以下是对一个在每个数据块中都存在行链接的表执行的 SQL 查询：

```sql
SQL> select id, count(e) from chaines_big where a like 'a%' and id < 11 group by id;
```

在另一个会话中，执行了一个专门只包含本节要涵盖的两个统计信息的五秒快照：

```sql
SQL> @snapper4 all,gather=s,sinclude=fetch|chain|consistent|cell 5 1 297
```

从上面的输出可以看出，Oracle 在单元中跳过了大约 28,000 个链式行，并将它们发送到 RDBMS 进行常规处理。另一个问题指标是 `table fetch continued row` 的值相对较高。

另请注意，`consistent gets` 的值（计算了数据库和单元通过智能扫描完成的所有一致性读取）略多于 45,000。同时，子计数器 `consistent gets from cache`（由数据库层从缓冲区缓存执行的一致性读取）和 `consistent gets direct`（绕过缓冲区缓存的一致性读取）显示了在单元级别完成了一致性读取工作量，以及在数据库级别额外完成了多少。

关于链式行的结论是，下次当你看到你的智能扫描正在等待大量的 `cell single block physical reads` 并且在数据库层内（`consistent gets from cache`）执行逻辑 I/O 时，需要检查的事情之一就是上述统计信息，以查看是否遇到了链式行并必须在数据库中处理它们。当然，别忘了智能扫描只卸载大型段扫描工作负载，但如果你的查询计划包含其他行源、索引范围扫描或小型缓存表扫描，看到逻辑 I/O 和单块物理读是预期的。你可以根据你的许可，使用 ASH、`V$SESSION` 或 SQL 跟踪中的等待接口数据来查看这些单块读访问了哪些对象。ASH 中的 `current_obj#` 列和原始 SQL 跟踪文件中的 `obj#` 字段指的是会话正在读取的表（或索引或分区）的 `object_id`。
