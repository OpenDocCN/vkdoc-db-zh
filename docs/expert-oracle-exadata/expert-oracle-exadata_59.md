# Exadata IORM 页面可用于简单地按数据库监控时间序列工作负载分布，或配置 IO 资源管理设置。该页面布局清晰，能让您了解各数据库的平均节流时间、利用率、小 IO 延迟和磁盘 IO 目标等关键信息，这些信息是制定通用资源管理和 IORM 计划时所必需的。结合前两个页面（存储服务器网格主页和性能视图），可以更容易地验证性能下降的原因是由于资源争用、容量问题，还是数据库内部的问题。I/O 资源管理器将在第 7 章中更详细地介绍。

## 应使用哪些单元格指标？

使用哪些单元格指标是一个重要问题。Oracle 提供了种类繁多的各种指标——来自数据库级别、主机服务器级别和 Exadata 单元格级别。不仅在 12.1.2.1 版本中指标数量已经非常庞大，而且还在不断增长。因此，很容易在指标中迷失方向，或者更糟的是，被误导并排查一个错误或不存在的问题。在性能故障排查中，规则是从关键因素——用户响应时间——开始，然后从那里深入分析，正如本章前面章节所解释的那样。但是，通用效率监控、利用率概览和容量规划有时是检查单元格级别聚合指标的充分理由。

如果您期望更多，这可能会令人惊讶，但实际上并没有一长串需要您在单元格中监控的秘密且特殊的 Exadata 指标。单元格只是一个服务器，带有 RAM、CPU 和磁盘，通过网络链路发送数据。您希望在单元格级别监控的基本指标与在任何服务器上使用的指标相同：CPU 利用率、系统负载、内存利用率和磁盘利用率，以及 I/O 吞吐量和延迟。最重要的指标可在 `OEM Exadata Storage Server Plug-in` 中获取，其余的可以从 `ExaWatcher` 日志文件中提取。高级指标仅在处理错误或一些罕见的、非常具体的性能问题时才会发挥作用，因此除非出现问题，否则不应该主动监控它们。

有一个主题值得详细说明——CPU 使用率监控，包括在数据库和单元格中。数据仓库和报告系统与 OLTP 数据库不同，通常不需要用户查询有非常快的响应速度。报告系统对带宽敏感，而不是对延迟敏感。当然，查询完成得越快越好，但在 DWH 环境中，如果查询运行了 33 秒而不是 30 秒，人们通常不会注意到。相反，典型的 OLTP 用户如果偶尔遇到他们的一秒钟查询花了六秒钟，则肯定会注意到。这就是为什么在 OLTP 服务器上，您不希望 CPU 利用率持续达到 100% 的原因之一。您无法做到这一点，同时还期望保持稳定、可靠的性能。在 OLTP 系统中，您必须留出一些余量。然而，在 DW 服务器上，性能的小幅波动不会被注意到，为了充分利用投资，您可以负担得起让 CPU 利用率达到 100%。

然而，Exadata 使事情变得复杂。除了拥有多个数据库节点之外，您还有另一整层服务器：单元格。当使用高度并行的 `Smart Scans` 访问 `EHCC` 表时，情况变得有趣起来。请回顾第 3 章和第 11 章，卸载解压缩需要在单元格中占用大量的 CPU 周期。因此，对于某些工作负载，您的单元格 CPU 可能会 100% 繁忙，无法足够快地向数据库层回送数据。数据库层的 CPU 可能一半处于空闲状态，而单元格可能确实需要一些额外的 CPU 容量。为了改善这种情况，Oracle 计算节点和单元格服务器会执行握手协议。

由于单元格利用率达到 100% 的风险，Oracle 使 `cellsrv` 能够跳过某些数据块的卸载处理，并将这些块直接传回数据库（从 `cellsrv` 11.2.2.3.0 开始）。单元格会检查其 CPU 利用率是否超过 90%，以及数据库 CPU 利用率（根据数据库的资源管理器统计信息发送）是否低于此值。如果是，则某些块不会在单元格中处理，而是直接传递到数据库。然后数据库将解密（如果需要，此处理模型与 `HCC` 没有严格关系）和解压缩块，并在数据库层执行投影和过滤。这使您能够充分利用 Exadata 集群两层中的所有 CPU 容量。然而，这自动意味着，如果某些块突然在数据库层处理（而不是卸载到单元格），当单元格过于繁忙时，您可能会看到数据库层出现意外的 CPU 利用率峰值。在大多数 DW 系统上，尤其是在正确配置资源管理器设置的情况下，这应该不是问题，但在 Exadata 上运行 OLTP 或其他低延迟系统时，您需要留意这一点。

像往常一样，Oracle 提供了关于此直通功能的良好指标。每当在 `Smart Scan` 期间跳过某些块的卸载处理，并且这些块被送回数据库层进行处理时，统计信息“cell physical IO bytes pushed back due to excessive CPU on cell”或较新的“cell physical IO bytes sent directly to DB node to balance CPU”会在 `V$SESSTAT`/`V$SYSSTAT` 和 `AWR` 报告中递增。有关此功能和统计信息的更多信息，请参阅第 11 章。

## 监控 Exadata 存储单元格操作系统级别指标

Oracle 数据库和 `cellsrv` 软件在收集性能指标方面做得很好，但在某些情况下，您仍可能希望使用操作系统工具。原因之一是，通常 Oracle 中的 `V$` 视图告诉您 Oracle 认为它正在做什么。然而，如果您遇到错误或 Oracle 内置工具的某些其他限制，这可能并非真实情况。其中一个限制是低级 I/O 测量。



#### 使用 `iostat` 监控存储单元服务器的 I/O 指标

Oracle 数据库和存储单元都会测量 I/O 完成时间；换句话说，就是 I/O 的响应时间。对于同步 I/O 操作，Oracle 的 I/O 等待时间仅仅是系统调用（如 `pread`）的完成时间。对于异步 I/O，响应时间的测量则更为复杂，因为异步 I/O 提交系统调用不会阻塞等待；它会在微秒内立即返回，稍后执行某个 I/O 回收系统调用来标记该 I/O 操作已完成。Exadata 存储单元会跟踪每个异步 I/O 请求（例如，何时提交的），一旦 I/O 被成功回收，它们会将回收的时间戳与该 I/O 的提交时间戳进行比较，从而得知其持续时间。

除了这种额外的 `cellsrv` 级别的 I/O 监控外，存储单元仍然没有将 I/O 响应时间分解为两个重要组成部分：该 I/O 请求在发送到 SCSI 设备之前，在操作系统 I/O 队列中无谓等待了多长时间；以及一旦 I/O 请求发送到硬件后，实际的硬件服务时间有多长。我们在 Exadata 12.1.2.1.0 中注意到许多新的单元指标，其名称中包含“等待时间”和“服务时间”，但这些都与 IO 资源管理器相关，而非单个网格磁盘。因此，比较 I/O 服务时间与等待时间仍然很有用。这种比较提供了一个重要线索，可以判断存储硬件本身是否响应缓慢——如果是这种情况，服务时间会高于你对现代磁盘驱动器的预期。

**注意**
那么，磁盘的正常 I/O 服务时间应该是多少呢？这取决于你使用的磁盘类型、寻道延迟、磁头从上一个位置需要寻道多远、磁盘的旋转延迟、转速（RPM）以及通过线缆（或光纤）传输刚读取的位的速度有多快。计算磁盘的良好服务时间并没有什么魔法；这完全基于磁盘的物理能力，而这些能力在供应商的磁盘规格文档中都有说明。你可以在维基百科上阅读有关各种延迟的更多信息：[`http://en.wikipedia.org/wiki/Hard_disk_drive#Access_time`](http://en.wikipedia.org/wiki/Hard_disk_drive#Access_time)。

当然，可能还存在其他开销，比如 SAN 存储网络的往返时间，以及存储控制器、交换机等引起的额外延迟。但请记住，Exadata 不连接 SAN 存储；磁盘是直接连接到存储单元上的。除了 X5-2 高性能服务器外，每个存储单元内部都有一个小型 LSI MegaRaid 控制器，磁盘连接到该控制器上。这是因为在该服务器类型中也没有硬盘，因此关于硬盘机械特性的讨论不适用。

此外，在磁盘和存储（单元）之间没有复杂的网络组件（例如中间结构交换机），这些组件可能会增加延迟或发生故障。更重要的是，没有“外部”数据库或应用程序连接到存储，这些连接你无法控制，但它们可能会突然开始使磁盘饱和而不发出警告。在这种情况下，你唯一能看到的现象就是当存储阵列过载时，磁盘 I/O 服务时间突然上升。

总之，对现代 15000RPM 磁盘驱动器进行小型 I/O 请求（最大 256KB）时，理想情况下每次 I/O 请求的平均服务时间应为 5–7 毫秒。这源于 3.5 毫秒的平均寻道延迟、2 毫秒的平均旋转延迟，加上 0.4 毫秒用于通过 SAS 通道传输 256KB 数据的时间。理论上，这个平均寻道延迟是一个悲观的数字，它假设整个磁盘已满，磁盘读/写头必须在磁盘柱面的两端轨道之间进行随机寻道。但你的配置可能将磁盘划分为较热和较冷的区域，这通常被称为“短行程”。Exadata 默认安装就会创建这样的设置，其中 `DATA` 磁盘组位于硬盘磁盘片的外部较快区域；`RECO` 磁盘组位于磁盘片的中部区域，访问频率要低得多；而 `DBFS_DG` 位于磁盘片的内部区域，除了前两个磁盘外，操作系统分区就放在这两个磁盘上，并通过软件 RAID 进行镜像。实际上，这种数据放置方式减少了实际的平均寻道时间。因此，当你的磁头不必寻道太远时，你可能会得到 2–3 毫秒这样低的服务时间；但在实际运行真实生产工作负载时，10 毫秒的平均 I/O 服务时间是完全可以接受的。

**注意**
关于为获得最佳性能而使用短行程磁盘的讨论不适用于全闪存的 X5-2 高性能单元，因为这些服务器没有任何机械磁盘。

到目前为止，整个解释的重点其实是：对于 Exadata 单元的磁盘驱动器，了解理想的磁盘服务时间应该是可能的。如果服务时间持续远高于此值，则存储硬件存在问题，可能是控制器问题、磁盘故障或由于数据碎片导致的性能下降等多种因素的组合。对于 SAN 存储，操作系统级别的高服务时间也可能意味着 SAN 网络或存储阵列内部存在队列（例如，如果上千台其他服务器正在用 I/O 猛烈冲击同一个存储阵列）。但是，如上所述，Exadata 存储单元内部拥有专用的存储；只有当前的单元操作系统可以访问此存储，并且所有 IO 在 `iostat` 中都是可见的。从版本 11.2.3.2 开始，Exadata 单元会尝试检测性能不佳的磁盘，并在可能的情况下将其从磁盘组中移除。所有这些操作都会在单元警报日志文件中被跟踪，如果单元配置得当，还会发送警报。在大多数情况下，工作负载的激增会导致磁盘被暂时离线并进行检查，进入受限状态。如果彻底检查表明磁盘性能可预见地低下，则可以将其剔除。否则，它会被重新添加到配置中，就像什么也没发生过一样。关于此操作的详细说明可以在 MOS 注释 1509105.1 中找到。
