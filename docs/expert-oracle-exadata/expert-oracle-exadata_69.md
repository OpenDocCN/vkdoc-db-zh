`Exadata` 是小端格式、64 位的 `Linux`（或 `Solaris x86-64`），因此如果源数据库也是小端格式，表空间可以像平台相同一样进行传输。如果源平台是大端格式，则需要一个额外的步骤。

要将表空间从一个平台转换到另一个平台，请使用 `RMAN` 的 `CONVERT TABLESPACE` 或 `CONVERT DATAFILE` 命令。你可以在备份期间使用以下命令转换文件的字节序格式：

```
RMAN> CONVERT TABLESPACE payroll_data,payroll_mviews
TO PLATFORM 'Linux x86 64-bit'
FORMAT '/u01/shared_files/%U';
```

在这个例子中，`RMAN` 将数据文件转换为与 `Exadata` 兼容的字节序格式。转换后的数据文件会被自动唯一命名（`%U`）并保存在 `/u01/shared_files` 目录中。这个转换操作可以选择在源系统（如上所示）或目标系统（`Exadata`）上进行。以下命令在 `Exadata` 的恢复操作期间转换字节序格式：

```
RMAN> CONVERT DATAFILE payroll_data.dbf, payroll_mviews.dbf
FROM PLATFORM 'Solaris[tm] OE (64-bit)'
DB_FILE_NAME_CONVERT
'/u01/shared_files/payroll_data.dbf','+DATA',
'/u01/shared_files/payroll_mviews.dbf','+DATA';
```

这为你提供了同时将文件移动到 `ASM` 的额外优势。一个表空间可以单独传输，也可以作为传输集的一部分进行传输。传输集更常见，因为表空间之间通常存在对象依赖关系。例如，一个表空间可能包含表，而另一个表空间可能包含依赖的物化视图或索引。

为了传输表空间，你必须首先将其置于只读模式。然后，使用 `Data Pump` 导出表空间元数据，并指定 `transportable_tablespaces` 参数。你还应该指定 `TRANSPORT_FULL_CHECK` 参数以确保被传输表空间的严格包含性，这能确保没有依赖对象（如索引）存在于传输集之外。

然后，使用 `RMAN` 对传输集中的表空间进行备份。字节序格式的转换可以在 `RMAN` 备份期间进行，也可以在目标系统上恢复时进行。

以下是将表空间传输到 `Exadata` 的步骤：

1.  识别表空间对象依赖关系。
2.  将表空间设置为只读模式。
3.  使用 `Data Pump` 导出传输集的元数据。
4.  对传输集中的表空间进行 `RMAN` 备份。
5.  将导出文件与数据文件备份一起复制到 `Exadata`。
6.  从 `RMAN` 备份中将数据文件恢复到你的 `Exadata` 数据库。如果需要字节序转换，请使用 `CONVERT DATAFILE` 命令同时恢复和转换数据文件。
7.  将表空间重新设置为读写模式。
8.  使用 `Data Pump`，通过 `transport_datafiles` 参数将表空间元数据导入到 `Exadata` 数据库。你还可以选择使用 `remap_schema` 参数重映射表空间内容的模式。

#### 何时使用可传输表空间

`TTS` 和 `XTTS` 对于利用 `RMAN` 的速度迁移源数据库的部分内容非常有用。如果你的数据库中部分数据已准备好迁移到 `Exadata`，而其他部分尚未准备好，`TTS` 可能是一个不错的选择。但要小心，数据字典中的某些对象（如权限、特权、`PL/SQL` 代码）不会被迁移，因此必须单独导出。此外，在将表空间插入数据库之前，你还需要预先创建账户。还记得仅包含元数据的 `Data Pump` 导出吗？这可能是创建 `SQLFILE` 并执行必要 `DDL` 命令的一个好策略。

#### 可传输表空间策略的注意事项

请查阅你的 `Oracle` 文档，了解可能适用于你数据库的具体限制或注意事项。注意目标（可插拔）数据库上可能存在表空间和模式名称冲突。表空间必须置于只读模式才能迁移，这将对需要读写访问这些表空间的应用程序造成停机时间。这个停机窗口不容低估！表空间必须在将副本迁移到 `Exadata` 所需的时间内保持只读。对于一个 `TB` 级的数据库，这个持续时间可能非常显著，即使不是数周（取决于网络吞吐量或其他将数据传输到 `Exadata` 系统的方式），也可能需要数天。

你可以在表空间处于读写模式时运行 `RMAN` 备份，以查看此操作需要多长时间，然后再决定 `TTS` 是否是你迁移的合适方法。一个无需借助逻辑复制工具即可缩短停机窗口的选项是使用**具有增量备份的跨平台可传输表空间**。这不会完全消除停机窗口，但有潜力显著缩短它。

如果你确实需要零停机迁移，你需要使用逻辑迁移工具，在字节序转换到 `Exadata` 的过程中捕获系统上的任何更改。只有在可传输表空间集完全导入到 `Exadata` 之后，才能打开闸门，开始在 `Exadata` 上进行这些更改的复制。



### 跨平台可传输表空间与增量备份

另一个物理迁移数据库子集的选择是使用一个鲜为人知的 My Oracle Support (MOS) 文档及其记载的步骤。该流程本身并无全新内容，因为它建立在数据库已有功能之上——例如，你刚刚读过的关于 XTTS 的介绍。该 MOS 文档标题为“11G - Reduce Transportable Tablespace Downtime using Cross Platform Incremental Backup”，其文档 ID 为 1389592.1。其中描述的流程适用于 Oracle 11.2。该 MOS 文档最近已更新至 12c 版本，指向另一个文档 ID：2005729.1。正如刚刚提到的，12c 的流程要求源数据库和目标数据库的初始化参数 `compatible` 设置为 12.1.0 或更高。鉴于你很可能是在旧硬件上进行迁移，假设你在迁移到 Exadata 之前先将源系统原地升级到 12c 是不现实的。因此，本节将重点介绍 11.2 的流程。一旦数据库安全迁移到 Exadata 系统上，就可以采取进一步步骤将其升级到更高版本。

顾名思义，整个想法基于跨平台可传输表空间。XTTS 并非新技术——那么为何本节要讨论增量备份？虽然 XTTS 确实广为人知，但新颖之处在于你使用增量备份来向前滚动数据文件的映像副本。这本身也不新鲜，因为这正是数据库恢复时发生的事情。真正新颖的是，你在源系统上进行增量备份，然后将其应用到目标 Exadata 数据库上。考虑到你的源系统很可能是 Big Endian 架构，例如基于 Power、SPARC 或安腾的系统，你会开始体会到所提出解决方案的优雅之处。

总而言之，跨平台可传输表空间 (x-platform TTS) 加上增量备份，使你能够将 Big Endian 数据库从旧硬件迁移到 Exadata，且停机成本可能非常低（指停机时间短）。你将产生的停机时间预计大致与你作为可传输表空间迁移的一部分需要导入的元数据量成正比。你正在阅读的流程专注于这些步骤，其中大部分通过一个名为 `xttdriver.pl` 的 perl 脚本启动：

*   从设置阶段开始，在此阶段设置所有需要的结构、数据库和脚本。
*   在准备阶段，创建要迁移的数据文件的初始映像副本，并将它们传输到 Exadata 系统。
*   在滚动更新阶段，对源系统进行增量备份，并将它们应用到目标系统的映像副本上。
*   当切换周末最终到来时，将要迁移的表空间置于只读模式，创建最后一个增量备份，应用它，然后插入表空间。
*   你可以在切换并将其置为读写模式之前，在 Exadata 数据库中验证数据文件。

如果你仔细阅读了上述要点，你会注意到源系统在切换日期之前保持在线和可用。与“传统”TTS 方法相比，这是一个巨大的优势，在传统方法中，你需要将表空间置于只读模式，直到其所有数据文件转换完成。

在 2013 年圣诞节之前（以及 Oracle 11.2.0.4 之前），你现在读到的流程要稍微复杂一些。除了源数据库和目标数据库之外，你还需要一个转换实例。如果你的目标数据库是 11.2.0.4 或更高版本，则不再需要该实例。如果你想迁移到 11.2.0.3（我们希望你不必这么做），则需要一个额外的“转换”实例。此外，该流程过去仅当你迁移到 Exadata 时才受支持。这一限制现已解除，你也可以使用它迁移到 64 位 Oracle Linux。

**警告**

跨平台可传输表空间 + 增量备份流程无法施展任何魔法，也无法绕过 TTS 的限制。请务必阅读本章前一节以及 Oracle 文档集，以了解 TTS 的限制。

#### 设置与配置

对于本节，请假设源数据库位于 SPARC 架构的 Solaris 上（意味着 Big Endian），并将被迁移到 Exadata（意味着 Little Endian）。源数据库的 `ORACLE_SID` 为 “solaris”，使用 Oracle 11.2.0.3，而目标数据库 “EXA” 是 11.2.0.4。因此，不需要转换实例。在 Data Guard 环境中要小心——新的数据文件不会自动复制到备用数据库。除了主数据库之外，你有责任确保这些文件物理存在于所有备用数据库上。

通过查阅文档 ID 1389592.1 并下载其附件 zip 文件来开始准备。在撰写本文时，文件名为 `rman_xttconvert_2.0.zip`。将 zip 文件暂存在源主机上。在以下示例中，我们将在 oracle 用户的主目录中使用一个名为 `xtt` 的新目录来存放脚本和配置文件。

使用版本 2 的脚本时，你可以选择使用 `DBMS_FILE_TRANSFER` 或 `RMAN` 来传输文件。前者的优点是你无需担心如何将数据文件导入 Exadata。另一方面，它对最大文件大小有限制，不能超过 2 TB。该限制使得 `DBMS_FILE_TRANSFER` 流程存在问题，尤其是在使用大文件表空间时。因此，本示例选择了 `RMAN` 方法。

使用 `RMAN` 方法时，你必须考虑如何在源系统和目标系统之间物理传输文件。对于位置较近的系统，NFS 可能是一个选择。但防火墙和到各种 VLAN 设置的路由问题常常阻碍 NFS 的使用。备选方案——特别是对于长距离传输——是 SCP。根据你的网络带宽，你可以并行启动多个 SCP 会话，甚至可以在数据通过管道传输时进行压缩。使用 RMAN 方法的缺点也不应隐瞒：你需要在源端和目标端都有一个暂存区来保存映像副本以及任何增量备份。值得庆幸的是，对于那些一直在为他们的 `DBFS_DG` 寻找用途的 Exadata 用户，这里有一个用途：使用数据库文件系统选项，将 `DBFS_DG` 磁盘组中的一个数据库作为暂存区！而且 Oracle 12.1.0.2 Grid Infrastructure 允许你使用 ACFS 作为 DBFS 的替代方案。

作为准备工作的一部分，你需要编辑 `xtt.properties` 文件。它包含了关于源和目标数据库、数据文件备份位置等所有信息，这些都是下一步所需的。以下是一个剥离了所有注释的配置文件示例：

```
oracle@solaris:∼/xtt$ grep -i ’^[a-z]’ xtt.properties
tablespaces=DATA_TBS
platformid=2
dfcopydir=/u01/stage/src
backupformat=/u01/stage/src
stageondest=/u01/stage/dest
storageondest=+DATA/EXA/datafile/
backupondest=+RECO
asm_home=/u01/app/11.2.0.4/grid
asm_sid=+ASM1
parallel=8
rollparallel=4
```



此处，`tablespaces` 列出了要迁移的表空间。可以列出多个表空间，但需要用逗号分隔。平台 ID 是源数据库的 `v$database.platform_id`。`dfcopydir` 所指示的目录是准备阶段（稍后详述）期间表空间映像副本将存放的位置。同样地，`backupformat` 需要指定增量备份创建所在的目录。在 Exadata 上，`stageondest` 是该过程期望存放初始映像副本的位置。转换后的数据文件随后将被存储在 `storageondest` 中。`backupondest` 所指示的位置指示驱动脚本在向前滚阶段写入增量备份。ASM 参数是自解释的。你还可以利用某些步骤的并行执行：`Parallel` 是数据文件转换期间的并行度，而 `roll parallel` 定义了使用来自源的增量备份向前滚动映像副本的并行级别。根据你的需求调整这些参数。如果你决定使用 `DBMS_FILE_TRANSFER` 方法，你将需要设置一些其他参数。为保持简单，此处未显示这些参数。

注意

是的，没错——你可以在源（大端）平台上进行增量备份，并将其应用到 Exadata 上的映像副本。这就是我们如此喜欢这个流程的原因。

在源系统上完成配置后，将 `xtt` 目录复制到驱动 Exadata 计算节点的 Oracle 用户主目录中。确保所有目录都存在。当然，你也可以利用 NFS——在这种情况下，你由 `dfcopydir`、`backupformat`、`stageondest` 指示的目录将是相同的。

#### 实现

步骤完成后，将映像副本传输到 Exadata 系统的 `stageondest` 目录。如果你使用 NFS，此步骤是可选的。你还需要将新创建的元数据文件和脚本复制到 Exadata 上的 `∼/xtt` 目录。数据文件传输完成后，将其转换为 64 位 Linux x86-64 格式。请注意，此过程耗时多久并不重要。表空间在源数据库上始终处于在线读写状态，因此对源的应用可用性完全没有影响。

下一阶段称为向前滚。你对要迁移的表空间进行增量备份，目的是将之前拍摄的映像副本向前滚动。再次说明，这是此技术的最大卖点：你在大端系统上进行备份，以将已转换为小端格式的数据文件向前滚动。备份文件及其元信息和脚本需要传输到 Exadata 并在那里应用。这个过程可以重复多次，直到下一阶段开始：传输阶段。

传输阶段与传统的跨平台 TTS（可传输表空间）技术并无太大不同：你在源上将要迁移的表空间更改为只读模式，进行最后一次增量备份，将其传输并应用到 Exadata。一旦你将源表空间设置为只读模式，就会在源上造成停机。与常规跨平台 TTS 的不同之处在于，你不需要将数据文件传输到 Exadata 并进行转换，这可能大大节省时间。最后一次增量备份确保你 Exadata 数据库中的映像副本是 100% 最新的。如果你想导入的表空间的数据所有者尚不存在，现在就创建它们，包括授权、角色和权限。一旦最终的向前滚命令完成，你就可以将表空间元数据导入到 Exadata 数据库。

导入完成后——其完成时间与你导入的表空间相关的字典元数据量成正比——你可以验证表空间的完整性，并在目标端将其设置为读写。

#### 物理备用数据库

在物理备用策略中，目标数据库是从源数据库的完整备份中实例化而来的。该备份以与备份和恢复策略相同的方式还原到 Exadata。数据库还原到 Exadata 后，它将以挂载模式启动并保持持续恢复状态。随着源数据库中发生变化，会产生重做信息并传输到备用数据库，在那里更改首先被写入备用重做日志，然后才会被应用（重做应用）。与备份和恢复策略不同，此数据库会保持恢复模式一段时间。由于重做信息不断应用到备用数据库，因此不支持从大端到小端格式的转换。备用数据库自 Oracle 版本 7 以来就已存在。该功能内置于数据库架构中。当然，那时你必须编写脚本来监控源系统上的归档日志目标，然后将它们（通常通过 FTP）复制到备用系统，在那里另一个脚本负责将它们应用到备用数据库。在版本 9i 中，Oracle 引入了一个名为 Data Guard 的新产品，用于管理和自动化那些管理监控备用环境的繁琐任务。如今，Data Guard 提供以下服务：

*   重做传输服务
    *   处理重做信息到目标系统的传输
    *   解决因网络故障导致的归档重做日志间隔
    *   通过重新传输替换日志来检测和解决缺失或损坏的归档重做日志

*   应用服务
    *   尽可能实时地自动将重做信息应用到备用数据库
    *   可以在重做应用期间允许对备用数据库进行只读访问，这需要 Active Data Guard 选项的许可
    *   提供角色过渡管理：源数据库和备用数据库的角色可以临时或永久切换
    *   切换（Switchover）：平滑地切换源数据库和备用数据库的角色。不会对 Data Guard 配置中的数据库产生任何影响。如果在不涉及版本更改的情况下，可用于撤销到旧硬件的迁移。
    *   故障转移（Failover）：角色的更改，通常是站点故障的结果。需要对前主数据库重新实例化，可以通过可用的闪回数据库功能，或者完全重建。通常不用于迁移。

*   Data Guard 代理（Data Guard Broker）：
    *   简化备用数据库的配置和管理
    *   Oracle 企业管理器提供了一个图形用户界面，可以帮助创建 Data Guard 配置
    *   用于监控和管理备用数据库环境的集中控制台
    *   简化备用数据库的切换/故障转移
    *   同样也存在一个非常通用的命令行界面。

Data Guard 提供了三种备用数据库保护模式：


### Oracle Data Guard 保护模式与迁移注意事项

*   **最大可用性**：在“最大性能”模式采用的放任方法与严格的“最大保护”设置之间，这是一个很好的折中方案。为了让事务在主数据库提交，至少一个已同步的备用数据库必须已在其某个备用重做日志中接收到相关的重做信息。如果无法实现，主数据库仍将继续运行。从 12c 开始，可以在`FASTSYNC`模式下配置最大可用性。在此模式下，备用数据库向主数据库确认已接收到重做信息，而无需等待 I/O 完成（将信息写入磁盘）。此模式提供的稍高性能可能会使您面临更高的数据丢失风险。
*   **最大性能**：Data Guard 中的默认保护模式。它完全不影响主数据库的可用性，主数据库和备用数据库彼此独立运行。通过实时重做应用，备用数据库仍能与主数据库完全同步。
*   **最大保护**：唯一能保证零数据丢失的 Data Guard 配置。这听起来是一个非常理想的功能，但它是以可能损失服务为代价的。如果主数据库无法将重做信息写入备用数据库，它将会关闭。很少有客户实施此保护模式。在此保护模式下运行可能会导致主数据库出现性能损失。

请记住，虽然 Data Guard 是数据库迁移的优秀工具，但这并非其唯一目的。Data Guard 的强大之处在于保护数据库免受介质损坏、灾难性介质故障和站点故障的影响。它是 Oracle 最大可用性架构中不可或缺的组成部分。因此，上述某些复制模式在数据库迁移的上下文中没有意义，也就不足为奇了。**最大性能**是最适合数据库迁移的复制模式，因为它对源数据库的性能没有影响。在最大性能模式下，源数据库将继续按其原有方式运行。事务不会因网络问题或下游复制问题而延迟。同样值得一提的是，Data Guard 完全支持单实例备用数据库或 Oracle RAC 备用数据库。并且从 11gR1 开始，如果您购买了 Active Data Guard 选项，可以在备用数据库以只读模式打开时应用重做。它与 Enterprise Manager Grid Control 也有很强的集成。

Data Guard 12c 中一些有趣的新特性包括 Far-Sync 备用配置，它允许数据库更容易地将重做发送到真正遥远的位置。对级联备用数据库的支持也得到了极大增强，但这些对于普通的数据库迁移没有增加价值，因此这里不作介绍。

### 使用物理备用数据库

物理备用数据库不允许对数据库进行任何更改（数据文件名更改除外）。因此，它最适用于不需要对目标数据库进行更改的数据库迁移。在这方面，它与本章前面介绍的完整备份和恢复非常相似。如果需要进行更改，则必须在迁移后完成。一般来说，这对于 OLTP 数据库来说问题较小，因为诸如迁移到大型表分区大小、实施分区和实施 HCC 等更改，不像对大型 DW 数据库那样有益。如果您的目标是快速安全地迁移到 Exadata，物理备用可能是一个合适的选择。凭借 Exadata 的性能和可扩展性，迁移后任务的实施时间可能比您预期的要短得多。

### 物理备用策略的注意事项

物理备用策略没有太多复杂之处。您应该关注网络稳定性和性能。虽然可以使用 Oracle 的跨平台物理备用功能在某些平台之间进行低停机时间的迁移，但无法在具有不同字节顺序（ endian ）的平台之间进行此类迁移。此外，还存在一些 Oracle 版本特定的限制。（更多内容请参阅 My Oracle Support 说明 413484.1 “同一 Data Guard 配置中异构主数据库和物理备用数据库的 Data Guard 支持”。）如果需要在不兼容的平台之间进行低停机时间迁移，您应考虑逻辑备用、Streams 或 Golden Gate 策略。

#### 物理迁移部分总结

如果您的应用程序模式足够复杂，以至于您不想采用任何逻辑迁移路径，那么物理迁移可能被证明是一种简单的迁移选项。它也非常适合将大量数据库迁移到 Exadata。如果您需要快速见效，“物理迁移”（即“直接迁移”）是可行之道，但不要忘记在有机会时进行简化和优化。

此外，通过将生产副本作为物理备用数据库恢复到 Exadata 数据库，并应用生产归档日志直到切换并使备用数据库成为新的生产数据库，物理迁移有可能以极低的停机时间完成。但是，此方法不能用于字节顺序不同的平台之间，并且还有一些 Oracle 版本特定的限制。

#### 处理旧的初始化参数

从较旧版本的 Oracle 迁移时，您可能会倾向于保留所有旧的（未记录的）`init.ora`参数以用于“调优”或“稳定性”。事实是，自 10g 以来，Oracle 为其参数提供了非常好的默认值，特别是在 11.2 及更高版本中，而您的 Exadata 集群很可能运行在此版本或更高版本上。多年前通过设置这些未记录参数所解决的任何问题，很可能已经在数据库代码中得到了修复。此外，迁移到 Exadata 带来的变化远大于任何参数调整所能带来的影响，因此稳定性这一点也站不住脚。因此，建议不要从旧数据库迁移任何未记录的参数，除非您的应用程序（如 Oracle Siebel、SAP）文档明确要求您迁移到的新 Oracle 版本需要这些参数。

#### 规划超大型数据库迁移的停机时间

在估算停机时间或数据迁移时间时，您应仅依赖于在您的环境中、使用您的数据、您的网络连接、数据库设置和压缩选项进行的实际测量。虽然根据 Oracle 的说法，可以以 5TB/小时（Exadata V2）甚至 21.5 TB/小时（Exadata X5-2）的速度将原始数据加载到满配置的 Rack，但当目标表使用`ARCHIVE`或`QUERY HIGH`压缩时，您可能无法获得这样的加载速度。无论您从本书或官方规格中找到什么数字，您都必须自己进行端到端的测试。有一些临时的解决方法可以提高加载速率。例如，如果有足够的磁盘空间，您可以在停机时间窗口内先加载到未压缩或`EHCC QUERY LOW`压缩的表中，一旦投入生产，再用更高的压缩率重新压缩各个分区。

## 摘要

在本长篇章节中，你了解了大量可用于将数据迁移到 Exadata 的可用工具。你应该选择尽可能最简单的方法，因为这可以降低风险，也能节省你自己的时间。你可能也会考虑评估一下 `Data Pump`；它常被忽视，但它可以传输整个数据库或仅一个模式。在 `12c` 中，`Full Transportable Export/Import` 在底层利用了 `Transportable Tablespaces`。它可以快速完成工作，甚至在你想在此过程中调整对象 DDL 元数据时也很灵活。然而，这种方法通常不够快（网络带宽！）或需要过多的临时磁盘空间。这完全取决于你需要传输多少数据以及允许多少停机时间。当移动大小为数十甚至数百 TB 的 VLDB（超大型数据库）时，你可能需要发挥创意，使用不那么直接的方法，如数据库链接、提前复制只读数据，或者使用某种复制方法进行完全增量迁移。跨平台可传输表空间加上增量备份的方法非常有用，如果元数据导出不消耗太多时间，它能帮助你实现低停机时间窗口。

每个企业的数据库环境和业务需求都不同，但你可以将本章解释的方法作为构建模块。如果要迁移的数据库非常大且允许的停机时间很短，你可能需要组合使用多种技术。无论你使用哪种技术，最重要的事情是记住要测试一切——最好测试多次——以解决过程中出现的任何问题。得益于系统化测试，你很可能会提前发现并修复许多问题。祝你好运！
