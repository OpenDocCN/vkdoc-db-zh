# 捕获工作负载

作为诊断数据收集步骤的一部分，您必须定义一个`扩展事件`会话来捕获数据库服务器上的工作负载。您可以使用第 6 章推荐的工具和方法来完成此操作。表 27-1 列出了可用于测量查询资源使用情况的具体事件。

**表 27-1**

捕获关于高成本查询信息的事件

| 类别 | 事件 |
| --- | --- |
| 执行 | `rpc_completed`，`sql_batch_completed` |

如第 6 章所述，对于生产数据库，建议将`扩展事件`会话的输出捕获到文件中。将输出捕获到文件有几个显著优势：

*   由于您打算在工作负载捕获后分析`SQL`查询，因此在捕获时无需显示这些`SQL`查询。

*   通过`SSMS`运行会话无法提供对跟踪过程的灵活时间控制。

让我们更仔细地看看时间控制。假设您想在晚上 11 点开始捕获事件，并记录`SQL`工作负载 24 小时。您可以使用`GUI`或`T-SQL`定义一个`扩展事件`会话。但是，您不必在准备好之前启动该进程。这意味着您可以在`SQL 代理`或使用其他调度工具中创建命令，通过`ALTER EVENT SESSION`命令来启动和停止该进程。

```sql
ALTER EVENT SESSION 
ON SERVER
STATE = ;
```

在此示例中，我在会话上设置了一个筛选器，仅捕获来自`AdventureWorks2017`数据库的事件。该文件将仅捕获针对该数据库的查询，从而减少了我需要处理的信息量。这对于您的系统可能也是一个不错的选择。虽然`扩展事件`会话的成本可以非常低（特别是与较旧的跟踪事件相比），但它们并非免费。应始终应用良好的筛选以确保最小影响。
