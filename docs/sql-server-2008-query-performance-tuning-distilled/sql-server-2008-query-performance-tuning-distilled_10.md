# 4. CPU 性能分析

本章通过讨论 `CPU`、网络和一般 `SQL Server` 指标，结束本书对系统的探索。`CPU` 是系统的工作引擎，保持一切运行。收集和传递数据、维护系统以及排序访问所需的所有不同计算都由 `CPU` 执行。在 `CPU` 上遇到瓶颈可能是一个难以摆脱的过程。不像内存，你有时可以轻松安装更多，或者磁盘，你有时可以轻松添加更多或升级，`CPU` 是你所运行系统的一个组成部分，并且通常只能通过购买新机器来升级。因此，你需要密切关注 `CPU` 使用情况。网络很少是 `SQL Server` 的主要瓶颈，当然在处理 `Azure SQL Database` 时除外，但密切关注它们也是好的。最后，还有一些 `SQL Server` 内部进程，你需要收集相关指标。本章涵盖以下主题：

*   如何收集有关处理器的指标
*   通过 `T-SQL` 查询可用的其他指标
*   解决处理器瓶颈的方法

## 处理器瓶颈分析

SQL Server 大量使用任何可用的处理器资源。你更可能在 `I/O` 或内存上遇到瓶颈，但你也可能会遇到 `CPU` 问题。我们在此介绍的措施侧重于操作系统和 SQL Server。然而，在虚拟化环境中，我们正在查看的 `CPU` 措施不太可能反映现实。你需要处理你用于虚拟化的任何管理程序或系统，以确切了解某些操作系统级别的 `CPU` 措施实际上如何反映现实。你可能正在经历外部压力甚至外部限制，这些都不会在以下概述的计数器中可见。

### % Processor Time

`% Processor Time` 不应持续偏高（大于 80%）。任何持续超过 90% 的处理器时间的效果实际上等同于 100%。如果 `% Processor Time` 持续偏高，而磁盘和网络计数器值较低，你的首要任务必须是减少处理器上的压力。请记住，这里的数字只是建议；人们可能出于有效原因不同意这些数字。将它们作为评估系统的起点，而不是具体的建议。

例如，如果 `% Processor Time` 是 85%，并且通过监控 `I/O` 计数器发现磁盘使用过多，那么很可能很大一部分处理器时间用于管理磁盘活动。这将在处理器的 `% Privileged Time` 计数器中反映出来，如下一节所述。在这种情况下，首先优化磁盘瓶颈将是有利的。此外，请记住，磁盘瓶颈反过来可能是由于内存瓶颈造成的，正如本章前面所解释的。

你可以跟踪机器上所有处理器的聚合处理器时间，也可以单独跟踪特定处理器的使用百分比。这允许你在 `SQL Server` 运行在四处理器机器中的三个处理器上的情况下隔离数据收集。记住，你可能会看到一个处理器满负荷运行，而另一个处理器负载很小。在这种情况下，平均值不会反映现实。使用平均值仅作为一个指标，而使用单个值作为系统上实际负载和处理的更准确度量。

在虚拟化环境中，`CPU` 可能是虚拟化的，因此你所看到的并不准确。所以，例如，在 `VMware` 系统中，如果你安装了 `VMware Tools`，你将能够查看 `VM Processor` 计数器以查看主机机器的处理器使用情况。使用这个度量，你可以判断你在操作系统中看到的 `CPU` 用法是否反映在主机机器上，或者你是否实际上只是用完了你的虚拟 `CPU`。另一方面，运行 `Hyper-V` 时，你需要查看 `\Hyper-V Hypervisor Logical Processor(_Total)\% Total Run Time` 以获得相同的度量。根据你使用的管理程序，你还会看到其他度量。



### % 特权时间

在 Windows 服务器上，处理分为两种模式：*用户模式* 和 *特权*（或*内核*）模式。所有系统级活动，包括磁盘访问，都在特权模式下进行。如果你在专用 SQL Server 系统上发现 `% 特权时间` 达到 20% 到 25% 或更高，那么该系统可能正在进行大量外部处理。这可能是 I/O、过滤驱动程序（如加密服务）、有缺陷的 I/O 组件，甚至是过时的驱动程序造成的。在专用 SQL Server 系统上，`% 特权时间` 计数器应最多为 5% 到 10%，但请使用你的基线来确定你的系统上什么表现看起来是正常的。

### 处理器队列长度

`处理器队列长度` 是处理器队列中的线程数量。（即使在具有多个处理器的计算机上，也只有一个处理器队列。）与磁盘计数器不同，`处理器队列长度` 计数器不读取已经在运行的线程。在 CPU 利用率较低的系统上，`处理器队列长度` 计数器通常为 0 或 1。

持续大于 2 的 `处理器队列长度` 计数器通常表示处理器拥堵。由于存在多个处理器，你可能需要考虑处理处理器队列长度的调度器数量。超过调度器数量（通常与处理器数量 1:1 对应）两倍的处理器队列长度也可能表明存在处理器瓶颈。虽然高的 `% 处理器时间` 计数器表示处理器繁忙，但持续高的 `处理器队列长度` 计数器是一个更确定的指标。如果超过建议值，这通常表示准备运行的线程数量超出了当前处理器数量所能最优处理的范围。

### 上下文切换/秒

`上下文切换/秒` 计数器监视计算机上所有处理器从一个线程切换到另一个线程的组合速率。当正在运行的线程自愿放弃处理器、被更高优先级的就绪线程抢占，或在用户模式和特权模式之间切换以使用执行体或子系统服务时，就会发生上下文切换。它是计算机中所有处理器上所有运行线程的 `线程:上下文切换/秒` 的总和，以切换次数衡量。

较高的数值主要由你的 CPU 速度决定，因此请随时间测量性能，并将此数字与你的基线进行比较，以了解你何时可能偏离正常状态。

### 批处理请求/秒

`批处理请求/秒` 很好地指示了系统上承载的负载量，这与处理器上的负载量有直接关联。由于你可能在系统上看到大量低成本查询或少数高成本查询，因此你不能单独看这个数字，而必须参考本节定义的其他计数器；每秒 10,000 个请求将被视为繁忙系统。更高的数值可能值得关注，这完全取决于你的系统什么是正常的。了解哪个值在你自己的系统中具有意义的最佳方法是建立一个基线，然后据此进行监控。请记住，这里的高数字不一定值得关注。如果你的所有其他资源都在掌控之中，并且你维持着较高的 `批处理请求/秒` 数字，那只是意味着你的服务器很忙。

### SQL 编译/秒

`SQL 编译/秒` 计数器在其聚合中显示了批编译和语句重编译。当服务器首次启动（或在故障转移或任何其他启动类型事件之后），这个数字可能非常高，但它会随着时间的推移而稳定下来。一旦稳定，与基线测量相比出现显著或持续的编译峰值是值得关注的原因，并且肯定会在处理器中表现为问题，因为查询编译是一项昂贵的操作。如果你正在使用某种对象关系映射引擎，例如 nHibernate 或 Entity Framework，高编译次数可能是正常的，尽管成本同样高昂。第 14 章详细介绍了 SQL 编译。

### SQL 重编译/秒

`SQL 重编译/秒` 是批处理和语句重编译的度量。大量的重编译可能导致处理器压力。由于语句重编译是此计数的一部分，因此它可能比 2005 年之前的 SQL Server 版本高得多。第 17 章详细介绍了查询重编译。

## 用于测量 CPU 性能的其他工具

你也可以使用 DMO 来捕获有关 CPU 的信息。这些 DMO 中的信息需要通过运行查询来捕获，然后将其作为基线测量的一部分保存。

### Sys.dm_os_wait_stats

等待统计是了解系统是否存在瓶颈的好方法。不过，你不能简单地说某个大于 *x* 的数字就是坏的。你需要随时间收集指标，以了解什么在你的系统上代表正常。偏离正常值的部分是有趣的。针对此 DMO 的查询，如果查找信号等待时间，可以指示 CPU 瓶颈。

过去，`CXPACKET` 等待被认为是在测量 CPU 性能时浪费时间。然而，随着 SQL Server 2016 SP2 和 SQL Server 2017 CU3 的发布，等待统计出现了一个新的拆分。现在有一个 `CXPACKET` 等待（我们需要关注的），和一个 `CXCONSUMER` 等待（不重要的）。如果你运行的是 Azure SQL Database，你也会看到这种相同的拆分。这种拆分的核心是为了涵盖并行性的消费者和生产者，这是某些系统上 CPU 瓶颈的主要原因。并行性的消费者，即接收数据的操作，通常具有可忽略但可测量的等待，换句话说，就是 `CXCONSUMER` 等待。现在，`CXPACKET` 衡量的是生产者，或推送数据的操作符。这改变了情况。`CXPACKET` 现在是一个指示系统上影响你的 CPU 的实际负载的等待。

### Sys.dm_os_workers 和 Sys.dm_os_schedulers

这些 DMO 显示 Windows 操作系统中的工作线程和调度器线程。定期运行针对这些 DMO 的查询将使你能够获取处于可运行状态的进程数量。这是处理器负载的一个极好指示。

### 查询存储

查询存储并不是专门测量 CPU 的工具，但它捕获的信息确实包括相关查询的聚合 CPU 使用率，以及这些查询的等待统计，包括任何与 CPU 相关的统计。

### 在 Linux 中测量 CPU 行为

在 Linux 上运行时，你仍然可以使用 `sys.dm_os_wait_stats`。这将为你提供可以指示 CPU 负载的等待统计。否则，你需要进入 Linux 系统本身。查看 CPU 的通常推荐方法是使用 `top` 命令。该工具的输出记录在这里：[`https://bit.ly/2KbZmuZ`](https://bit.ly/2KbZmuZ)。

## 处理器瓶颈解决方案

以下是几个常见的处理器瓶颈解决方案：

*   优化应用程序工作负载
*   消除或减少过多的编译/重编译
*   使用更多或更快的处理器
*   不运行不必要的软件

让我们依次考虑这些解决方案。



### 优化应用程序工作负载

要识别处理器密集型查询，请使用扩展事件会话（将在下一章讨论）捕获所有 SQL 查询，然后按 CPU 列对输出进行分组。另一种方法是利用查询存储（在第 11 章讨论）。您可以从 `sys.query_store_runtime_stats` 检索信息，以查看每个查询的多个聚合 CPU 指标。CPU 时间最高的查询对 CPU 压力的贡献最大。您应随后分析并优化这些查询，以减轻 CPU 的压力。通常，CPU 压力的原因并非查询内的大量计算，而是逻辑 I/O 内的争用。解决 I/O 问题通常也有助于解决 CPU 问题。您还可以直接查询 `sys.dm_exec_query_stats` 或 `sys.dm_exec_procedure_stats` 动态管理视图，以实时查看即时问题。最后，使用 `query hash` 和 `query plan hash`，您可以识别并调优常见查询或常见执行计划（这将在第 14 章详细讨论）。本书的大部分章节都涉及优化应用程序工作负载。

### 消除过多的编译/重新编译

一定数量的查询编译和重新编译是预料之中的事，特别是如前所述，当使用 ORM 工具时。但当这些操作在持续期间内大量出现时，就存在问题。同样值得注意的是它们之间的比例。编译数量高而重新编译数量低，意味着系统内查询重用较少（查询重用在第 9 章详细讨论）。大量的重新编译将导致处理器使用率高。解决重新编译的方法在第 17 章介绍。

### 使用更多或更快的处理器

最简单的解决方案之一，也是您大多数时候会采用的方案，就是增加系统处理能力。然而，由于处理器升级涉及高昂成本，您应首先尽可能优化 CPU 密集型操作。

可以通过增加单个处理器的性能或添加更多处理器来提高系统的处理能力。当您的 `% Processor Time` 计数器较高而 `Processor Queue Length` 计数器较低时，增加单个处理器的性能是合理的。如果 `% Processor Time` 计数器和 `Processor Queue Length` 计数器都高，则应考虑添加更多处理器。增加处理器数量允许系统同时执行更多请求。

### 不运行不必要的软件

公司政策通常要求在服务器上安装病毒检查软件。服务器上也可能运行着其他产品。在可能的情况下，不应在 SQL Server 所在的同一服务器上运行不必要的软件。与维护 Windows Server 或 SQL Server 无关的外部应用程序最好放在不同的机器上。

## 网络瓶颈分析

在 SQL Server OLTP 生产环境中，您很少会遇到因网络问题导致的性能问题。在 OLTP 环境中遇到的大多数网络问题实际上是由硬件或驱动程序限制、或交换机或路由器问题引起的。大多数此类问题最好使用网络监视器工具进行诊断。但是，性能监视器也提供了收集网络活动数据的对象，如表 4-2 所示。

表 4-2

用于分析网络压力的性能监视器计数器

| 对象（实例[,实例 N]） | 计数器 | 描述 | 值 |
| --- | --- | --- | --- |
| 网络接口（网卡） | `字节总数/秒` | 网络接口卡（NIC）上传输字节的速率 | 平均值 < NIC 容量的 50%，但需与基线比较 |
| 网络段 | `网络利用率百分比` | 网络段上正在使用的网络带宽百分比 | 平均值 < 网络带宽的 80%，但需与基线比较 |

### 字节总数/秒

您可以使用 `字节总数/秒` 计数器来确定网络接口卡（NIC）或网络适配器的性能。`字节总数/秒` 计数器应报告较高的值，表示大量成功的传输。将此值与 `网络接口\当前带宽` 性能计数器报告的值进行比较，后者反映了每个适配器的带宽。

为了为流量高峰留出空间，通常平均不应超过容量的 50%。如果此数字接近连接容量，并且处理器和内存使用率适中，则该连接很可能存在问题。

### 网络利用率百分比

`网络利用率百分比` 计数器代表网络段上正在使用的网络带宽百分比。此计数器的阈值取决于网络类型。例如，对于以太网，当 SQL Server 位于共享网络集线器上时，建议阈值为 30%。对于位于专用全双工网络上的 SQL Server，即使网络利用率接近 100% 也是可以接受的，但将网络利用率保持在可接受的阈值以下是有利的，以便为负载高峰留出空间。

### 注意

必须安装网络监视器驱动程序才能使用 `网络段` 对象计数器收集性能数据。

在 Windows Server 2012 R2 中，您可以从网络适配器的本地连接属性安装网络监视器驱动程序。网络监视器驱动程序在网络适配器的网络组件的网络协议列表中可用。

您还可以查看 `sys.dm_os_wait_stats` 中与网络相关的等待统计信息。但是，经常出现的一个等待类型是 `ASYNC_NETWORK_IO`。虽然这可能表明存在与网络相关的等待，但它更常见地反映了因编程代码不佳（未高效地使用结果集）而导致的等待。

## 网络瓶颈解决方案

一些常见的网络瓶颈解决方案如下：

*   优化应用程序工作负载
*   添加网络适配器
*   调节和避免中断

让我们更详细地考虑这些解决方案。

### 优化应用程序工作负载

要优化数据库应用程序和数据库服务器之间的网络流量，请在应用程序中进行以下设计更改：

*   不要发送长的 SQL 字符串，而是为 SQL 查询创建一个存储过程。然后，您只需通过网络发送存储过程的名称及其参数。
*   将多个数据库请求分组到一个存储过程中。然后，对于存储过程中实现的一组 SQL 查询，只需通过网络发出一次数据库请求。当谈论 Azure SQL 数据库时，这一点变得极其重要。
*   请求小型数据集。不要请求应用程序逻辑中未使用的表列。
*   将数据密集型业务逻辑移到数据库中作为存储过程或数据库触发器，以减少网络往返次数。
*   如果数据不经常更改，尝试将信息缓存在应用程序上，而不是频繁调用数据库获取与上次调用完全相同的信息。
*   最小化网络调用，例如返回未使用的多个结果集。一个常见的问题是由 SQL Server 返回的结果集包含每个语句的行计数引起的。您可以在查询顶部使用 `SET NOCOUNT ON` 来禁用此功能。

### 用户连接

多个只读的 SQL Server 实例可以在负载均衡环境（其中 SQL Server 分布在多台机器上）中协同工作，以支持大量的数据库请求。在这种情况下，最好监控 `User Connections` 计数器，以评估用户连接在多个 SQL Server 实例间的分布情况。`User Connections` 在正常的 application 行为下可能覆盖很宽的范围。这正是基线对于确定预期行为至关重要的地方。您很快将看到如何建立此基线。

### 批请求/秒

此计数器是 SQL Server 上负载的一个良好指标。根据系统资源利用率和 `Batch Requests/sec` 的水平，您可以估计 SQL Server 在不出现资源瓶颈的情况下可能支持的用户数量。此计数器在不同负载周期下的值，有助于您理解其与数据库连接数量的关系。这也有助于您理解 SQL Server 与 `Web Request/sec` 的关系，即对于使用 Microsoft Internet Information Services (IIS) 和 Active Server Pages (ASP) 的 Web 应用程序来说，是 Active Server Pages.Requests/sec。所有这些分析有助于您更好地理解和预测随着用户负载变化而产生的系统行为。

在正常的 application 行为下，此计数器的值可能覆盖很宽的范围。一个正常的基线对于确定预期行为至关重要。

## 总结

在本章中，您学习了如何收集关于 CPU、网络和 SQL Server 本身的一般指标。所有这些信息都有助于您在深入尝试调整查询之前，理解系统上正在发生的事情。请记住，CPU 会受到其他资源的影响，因为它需要管理这些资源，因此一些看似 CPU 问题的情况，可能更适合解释为磁盘或内存问题。网络很少成为 SQL Server 的主要瓶颈。您可以通过性能监视器计数器，像观察系统其他部分一样，使用多种方法来观察 SQL Server 内部行为。关于各种系统指标的讨论到此结束。接下来，您将学习如何将这些内容整合起来以创建基线。
