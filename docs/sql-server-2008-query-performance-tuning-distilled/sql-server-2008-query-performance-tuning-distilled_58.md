# 25. Azure SQL Database 和 SQL Server 中的自动调优

尽管许多查询性能调优涉及对系统、查询、统计信息、索引以及本书中提出的所有其他信息的详细了解，但查询调优的某些方面本质上是相当机械化的。发现缺失索引建议，然后测试该索引对查询是有帮助还是损害，以及是否会损害其他查询，这个过程可以实现自动化。对于某些类型的错误参数嗅探（其中一种计划明显优于另一种）同样如此。微软已开始着手自动化查询调优的这些方面。此外，它还在 Azure SQL Database 和 SQL Server 中引入了其他形式的自动化机制，这些机制将通过即时调优查询的某些方面来帮助你。别担心，本书的其余部分仍将非常有用，因为这些方法只能解决相当有限范围的问题。你仍然会有大量具有挑战性的工作要做。

在本章中，我将涵盖以下内容：

*   自动计划更正
*   Azure SQL Database 自动索引管理
*   自适应查询处理

## 自动计划更正

SQL Server 2017 和 Azure SQL Database 能够自动更正执行计划的机制，最好这样总结：微软利用了现在可获取的数据（这要归功于 `Query Store`；有关 `Query Store` 的更多详情，请参见 `第 11 章`），并将这些数据加以利用，实现了一些惊人的功能。随着数据变化，你的统计信息可能会改变。你可能有一个编写良好的查询和适当的索引，但随着时间的推移，随着统计信息的变动，你可能会看到引入一个新的执行计划，从而损害性能，这基本上就是 `第 17 章` 概述的错误参数嗅探问题。其他因素也可能导致良好的查询性能变差，例如 `累积更新` 改变了引擎行为。无论原因如何，你的代码和结构仍然支持良好的性能，只要你能采用正确的计划。显然，你可以自己使用 `Query Store` 提供的工具来识别性能下降的查询，以及哪些计划提供了更好的性能，然后强制使用那些计划。然而，请注意上述整个过程是非常直接的。

1.  监控查询性能，并注意一个过去未曾改变的查询突然发生性能变化。
2.  确定该查询的执行计划是否已更改。
3.  如果计划已更改且性能下降，则强制使用之前的计划。
4.  衡量性能，看其是下降、改善还是保持不变。
5.  如果性能下降，则撤销该更改。

简而言之，这就是在 SQL Server 内部发生的事情。SQL Server 中的一个进程通过 `Query Store` 观察查询性能。如果它看到查询本身保持不变，但执行计划改变时性能下降，它会将此记录为建议的计划回归。如果你启用了查询自动调优功能，当识别出回归时，它将自动强制使用上一个良好的计划。然后，自动进程会观察行为，以判断强制使用计划是否是一个错误的选择。如果是，它会纠正问题并记录该事实供你以后查看。简而言之，微软通过利用 `Query Store` 中可用的数据，借助自动计划强制功能，自动化了修复诸如错误参数嗅探之类的问题。
