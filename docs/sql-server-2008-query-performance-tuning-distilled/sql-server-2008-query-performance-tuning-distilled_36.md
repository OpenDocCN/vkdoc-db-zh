# SQL 查询优化最佳实践

## 显式参数化查询的可变部分

查询通常会运行多次，每次运行的唯一区别在于可变部分的取值不同。如果能够将查询的静态部分与可变部分分离开来，它们的执行计划便可以被重用。尽管 SQL Server 提供了简单参数化和强制参数化功能，但它们有严重的局限性。因此，应始终使用标准的预执行工作负载技术来显式地执行参数化。

## 创建存储过程以实现业务功能

如果你已经显式地参数化了查询，那么将其放入存储过程将带来最佳的可重用性。由于只需要将参数随存储过程名称一起发送，因此网络流量得以减少。同时，因为存储过程从缓存中被重用，它们可以比即席查询运行得更快。

如同其他事情一样，好事也可能过头。有些业务过程理应属于数据库，但也有些业务过程绝不应放在数据库内。例如，在存储过程中格式化数据通常由应用程序完成更好。基本上，你的数据库及其周围的查询应专注于直接的数据检索和数据存储。任何其他处理都应在别处进行。

## 使用 `sp_executesql` 编码以避免存储过程部署

如果存储过程所需的对象部署成为一个考虑因素，或者你正在使用客户端生成的查询，那么可以使用 `sp_executesql` 将查询作为预执行工作负载提交。与存储过程模型不同，`sp_executesql` 不会在数据库中创建任何持久对象。它适用于执行单次查询或小型批处理查询。

在存储过程中实现的完整业务逻辑也可以作为一个大的查询字符串通过 `sp_executesql` 提交。然而，随着业务逻辑复杂性的增加，为完整逻辑创建和维护一个查询字符串会变得困难。

此外，使用带有适当参数的 `sp_executesql` 和存储过程可以防止服务器上的 SQL 注入攻击。

尽管如此，我仍然强烈建议在可能的情况下在数据库中使用存储过程。

## 实施准备/执行模型以避免重新发送查询字符串

`sp_executesql` 每次重新执行查询时都需要通过网络发送查询字符串。它还需要在服务器上进行查询字符串匹配的开销，以在计划缓存中识别相应的执行计划。在 ODBC 或 OLEDB（或 OLEDB .NET）应用程序的情况下，你可以使用准备/执行模型来避免在多次执行期间重新发送查询字符串，因为只需要提交计划句柄和参数。在准备/执行模型中，由于计划句柄会返回给应用程序，计划可以被其他用户连接重用；它不局限于创建该计划的用户。

## 避免即席查询

不要设计使用即席查询的新应用程序！为即席查询创建的执行计划在查询以可变部分的不同值重新提交时无法被重用。尽管 SQL Server 具有简单参数化和强制参数化功能来隔离查询的可变部分，但由于 SQL Server 在参数化方面的严格保守性，该功能仅限于简单查询。为了获得更好的计划可重用性，请将查询作为预执行工作负载提交。

有些系统完全建立在即席查询的概念之上。这在功能上可行，并且可以在 SQL Server 中运行，但正如你所见，它带来了大量的额外开销，你需要为此进行规划。此外，即席查询通常是 SQL 注入被引入系统的方式。

## 对于动态查询，优先使用 `sp_executesql` 而非 `EXECUTE`

在存储过程或数据库应用程序中动态生成的 SQL 查询字符串应使用 `sp_executesql` 执行，而不是使用 `EXECUTE` 命令。`EXECUTE` 命令不允许显式地参数化查询的可变部分。

为了理解前面 `sp_executesql` 和 `EXECUTE` 之间的比较，请考虑用于在 `adhocsproc` 中执行 `SELECT` 语句的动态 SQL 查询字符串。

```
DECLARE @n VARCHAR(3) = '776',
@sql VARCHAR(MAX);
SET @sql
= 'SELECT * FROM Sales.SalesOrderDetail sod  ' + 'JOIN Sales.SalesOrderHeader soh  '
+ 'ON sod.SalesOrderID=soh.SalesOrderID ' + 'WHERE    sod.ProductID="' + @n + "";
-- 使用 EXECUTE 语句执行动态查询
EXECUTE (@sql);
```

`EXECUTE` 语句将查询与 `d.ProductID` 的值一起作为即席查询提交，因此可能不会导致简单参数化。你可以通过查看缓存自行检查输出。

```
SELECT deqs.execution_count,
deqs.query_hash,
deqs.query_plan_hash,
dest.text,
deqp.query_plan
FROM sys.dm_exec_query_stats AS deqs
CROSS APPLY sys.dm_exec_sql_text(deqs.plan_handle) AS dest
CROSS APPLY sys.dm_exec_query_plan(deqs.plan_handle) AS deqp
WHERE dest.text LIKE 'SELECT * FROM Sales.SalesOrderDetail sod%';
```

为了提高计划缓存的可重用性，请使用 `sp_executesql` 将动态 SQL 字符串作为参数化查询执行。

```
DECLARE @n NVARCHAR(3) = '776',
@sql NVARCHAR(MAX),
@paramdef NVARCHAR(6);
SET @sql
= 'SELECT * FROM Sales.SalesOrderDetail sod  ' + 'JOIN Sales.SalesOrderHeader soh  '
+ 'ON sod.SalesOrderID=soh.SalesOrderID ' + 'WHERE    sod.ProductID=@1';
SET @paramdef = N'@1 INT';
-- 使用 sp_executesql 系统存储过程执行动态查询
EXECUTE sp_executesql @sql, @paramdef, @1 = @n;
```

使用 `sp_executesql` 将查询作为显式参数化查询执行，会为该查询生成一个参数化计划，从而增加执行计划的可重用性。

## 谨慎地参数化查询的可变部分

将查询的可变部分转换为参数时要小心。某些变量的值范围可能变化非常剧烈，以至于某个值范围的执行计划可能不适用于其他值。这可能导致糟糕的参数嗅探（在第 17 章中介绍）。

## 不允许在查询中隐式解析对象

SQL Server 允许在不同的架构下创建具有相同名称的多个数据库对象。例如，表 `t1` 可以使用两个不同的架构（`u1` 和 `u2`）在它们各自的所有权下创建。大多数系统中的默认所有者是 `dbo`（数据库所有者）。如果用户 `u1` 执行以下查询，那么 SQL Server 首先尝试查找表 `t1` 是否存在于用户 `u1` 的默认架构中。

```
SELECT *
FROM tl
WHERE cl = 1;
```

如果不存在，则尝试查找表 `t1` 是否存在于 `dbo` 用户下。这种隐式解析允许用户 `u1` 在不同的架构下创建另一个表 `t1` 的实例并临时访问它（使用相同的应用程序代码），而不影响其他用户。

在生产数据库上，我建议使用架构所有者并避免隐式解析。否则，使用隐式解析会在生产服务器上增加以下开销：

*   需要更多时间来识别对象。
*   会降低计划缓存可重用性的有效性。


## 概述

SQL Server 基于成本的查询优化器不仅根据查询的确切语法，还基于使用不同处理策略执行查询的成本来决定有效的执行计划。对不同处理策略的成本评估在多个优化阶段中进行，以避免在优化查询上花费过多时间。然后，执行计划会被缓存，以便在重新执行相同查询时节省生成执行计划的成本。为了提高缓存计划的可重用性，SQL Server 支持在变量部分取不同值重新运行查询时，使用不同的技术来重用执行计划。

使用存储过程通常是提高执行计划可重用性的最佳技术。SQL Server 为存储过程生成参数化的执行计划，以便在使用相同或不同参数值重新运行存储过程时可以重用现有计划。但是，如果存储过程的现有执行计划失效，则无法在不重新编译的情况下重用该计划，从而降低了计划缓存的可重用性。
