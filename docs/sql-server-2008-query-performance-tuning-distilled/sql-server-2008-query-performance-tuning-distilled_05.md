# 内存瓶颈分析

## 识别瓶颈

资源瓶颈之间通常存在关联关系。例如，处理器瓶颈可能是由糟糕的执行计划导致的**过度分页**（内存瓶颈）或**磁盘速度慢**（磁盘瓶颈）的症状。如果一个系统内存不足，导致过度分页，同时磁盘速度又慢，那么最终结果之一就是处理器利用率过高，因为处理器需要花费大量的 CPU 周期来在内存中换入换出页面，并管理因此产生的大量 I/O 请求。用更快的处理器替换现有处理器可能略有帮助，但这并非最佳的全局解决方案。在这种情况下，增加内存是更合适的解决方案，因为它将减轻磁盘和处理器的压力。实际上，升级磁盘可能比升级处理器是更好的解决方案。如果可能，减少工作负载也会有帮助；当然，调整查询以确保最高效率也是一种选择。

定位瓶颈的最佳方法之一，是识别那些**等待其他资源完成其操作**的资源。你可以使用性能监视器计数器或 DMV（动态管理视图），例如 `sys.dm_os_wait_stats` 来收集此类信息。一个由资源处理的请求的响应时间，包括请求在资源队列中等待的时间以及执行请求所花费的时间，因此最终用户的响应时间与系统中的排队量成正比。

另一种识别瓶颈的方法是参考系统的响应时间和容量。例如，对磁盘的吞吐量通常应该接近供应商所建议的磁盘处理能力。因此，测量诸如“磁盘秒/传输”这样的信息，将能指示出磁盘是否因负载过高而变慢。

并非所有资源都有专门显示排队级别的计数器，但大多数资源都有一些代表该资源**过载**的计数器。例如，内存没有这样的计数器，但大量的**硬页错误**代表了物理内存的过载（硬页错误将在本章后面的“页/秒和页错误/秒”一节中解释）。其他资源，如处理器和磁盘，有特定的计数器来指示排队级别。例如，计数器 `Page Life Expectancy` 表示一个页面在缓冲池中未被引用的情况下能存活多久。这指示了 SQL Server 管理其内存的能力，因为生存期越长，意味着缓冲池中的数据会一直可用，等待下一次引用。然而，生存期越短，则意味着 SQL Server 正在快速地将页面移入移出缓冲池，这可能表明存在内存瓶颈。

稍后你将看到在分析每种类型的瓶颈时应使用哪些计数器。

## 瓶颈解决

一旦识别出瓶颈，你可以通过两种方式来解决它们。

*   **增加资源容量。**
*   **减少到达该资源的请求速率。**

增加容量通常需要额外的资源，例如内存、磁盘、处理器或网络适配器。你可以通过更严格地筛选到达资源的请求来减少其到达速率。例如，当遇到磁盘子系统瓶颈时，你可以选择增加磁盘子系统的容量，或者减少 I/O 请求数量。

增加容量意味着添加更多磁盘或升级到更快的磁盘。减少到达速率则意味着识别导致磁盘子系统高 I/O 请求的原因，并采取措施来减少其数量。例如，你可能能够通过在表上添加适当的索引来限制访问的数据量，或者通过编写 T-SQL 语句在 WHERE 子句中包含更多或更好的过滤条件，来减少 I/O 请求。
