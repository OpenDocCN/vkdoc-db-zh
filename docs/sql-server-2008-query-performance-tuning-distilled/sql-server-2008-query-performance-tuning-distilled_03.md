# 性能与成本

之前提到过一点，为了获得越来越小的性能提升，您需要投入越来越多的时间和金钱。因此，为确保获得最佳投资回报率，在优化性能时应保持客观。始终考虑以下两个方面：

*   您的应用程序可接受的性能是什么？
*   投资是否值得其带来的性能提升？

## 性能目标

为实现最高效率，您必须切合实际地估算您的性能需求。您可以遵循许多最佳实践来提高性能。例如，您可以将数据库文件放在最高性能的磁盘子系统上。然而，在应用最佳实践之前，您应考虑可能从中获得多少收益，以及该收益是否值得投资。这些性能需求通常由他人设定，要么是应用程序开发人员，要么是数据的业务消费者。查询调优的一个基本部分将涉及与这些相关方沟通，以确定一套足够好且切合实际的需求。

有时，在没有实际进行改进的情况下，确实很难估算性能提升。这使得正确识别性能瓶颈的来源变得更加重要。您的瓶颈是`CPU`、`内存`还是磁盘？是代码、数据结构或索引的问题，还是您已达到硬件的极限？您是否有一个糟糕的路由器、配置不当的`I/O`路径，或是应用不当的补丁导致网络运行缓慢？您平台上的服务层是否设置为适当的级别？请确保您能够从已知点出发做出这些可能成本高昂的决定，而不是靠猜测。一种实际的方法是逐步增加资源，并分析应用程序在增加资源后的可扩展性。如果该资源确实是可扩展性的瓶颈，那么可扩展的应用程序将从资源的逐步增加中按比例受益。如果结果令人满意，那么您可以承诺进行全面的改进。经验在这里也起着重要作用。

然而，有时您正承受着性能方面的痛苦，需要尽一切努力来缓解这种痛苦。进行全面的根本原因分析可能并不总是可行。尽管它仍然是保护生产系统的首选途径，但人们也承认，您并不总是能够做到这一点。



### “足够好”调优

与其将系统调优到理论上的最大性能，目标应该是调到系统性能“足够好”为止。这是一种普遍采用的性能调优方法。超过此点后的成本投入，相对于性能提升而言，通常是呈指数级增长的。80/20 法则非常适用：投入你 20%的资源，你可能获得 80%的潜在性能提升；但为了剩下 20%的潜在性能提升，你可能需要额外投入 80%的资源。因此，在设定性能目标时保持现实很重要。请记住，“足够好”是由你、你的客户以及与你合作的业务人员定义的，并没有一个所有人都遵循的标准标准。

业务获益并非通过考虑纯粹的性能，而是通过考虑性能的价格。然而，如果目标是为了找出应用程序的可扩展性限制（出于各种原因，包括针对竞争对手的营销），那么投入尽可能多的资源可能是值得的。即使在这样的情况下，使用第三方压力测试实验室也可能是更好的投资决策。

虽然在某些情况下需要深挖每一个可能的微秒级性能提升，但对我们大多数人来说，在大多数情况下，这完全没有必要。相反，专注于确保我们恰当地执行标准的最佳实践，就能达到我们需要的目标。你可能会遇到例外情况，但一般来说，这种情况不会发生。请首先专注于正确的标准。

## 性能基准

性能分析的主要目标之一是了解不同硬件和软件子系统底层的系统使用水平或压力。这些知识在以下方面对你有所帮助：

*   允许你分析资源瓶颈。
*   使你能够通过比较系统利用率模式与预先建立的基准来进行故障排除。
*   协助你在容量规划和安排硬件升级时做出准确的估算。
*   帮助你识别出数据库管理活动最适合执行的低利用率时期。
*   帮助你估算可能的硬件缩减或服务器整合的性质。公司为何会缩减规模？嗯，公司可能租用了一个非常高端的系统，期望强劲的增长，但由于增长不佳，他们现在希望缩减系统规模。那么整合呢？公司有时会购买过多的服务器，或者意识到维护和许可成本太高。这将使得使用更少的服务器变得非常有吸引力。
*   有些指标只有在与先前记录的数值进行比较时才有意义。没有之前的测量值，你将无法理解这些信息。

因此，为了更好地理解应用程序的资源需求，你应该为应用程序的硬件和软件使用情况创建一个基准。一个`baseline`作为你系统当前使用模式的统计数据，也是与未来统计数据进行比较的参考。基准分析帮助你理解应用程序在稳定期间的行为、在此期间硬件资源的使用方式以及软件的特性。有了基准，你可以执行以下操作：

*   衡量当前性能并阐述应用程序的性能目标。
*   比较其他硬件和软件组合，或者将平台服务层级与基准进行比较。
*   衡量工作负载和/或数据如何随时间变化。这包括了解业务周期，如年度续费或销售活动。
*   确保你了解服务器上的“正常”状态是什么，这样就不会将任意数字误解为问题。
*   评估应用程序的高峰和非高峰使用模式。此信息可用于在非高峰时段有效分配数据库管理活动，例如完整数据库备份和数据库碎片整理。

你可以使用内置于`Windows`的`Performance Monitor`来为`SQL Server`的硬件和软件资源利用创建基准。你也可以通过使用动态管理视图和动态管理函数来获取此信息的快照。类似地，你可以使用`Extended Events`对`SQL Server`查询工作负载进行基准测试，这可以帮助你在条件稳定时理解`SQL`查询的平均资源利用率和执行时间。你将在第 2 章到第 5 章详细学习如何使用这些工具和查询。一个平台系统可能有不同的度量标准，例如`Azure SQL Database`的数据库事务单元（`DTU`）。

另一个选择是利用众多工具之一在给定的服务器或数据库上生成人工负载。有许多第三方工具可用。微软提供了`Distributed Replay`，这将在第 25 章中详细介绍。


## 应将精力集中于何处

当你对特定系统进行调优时，请特别关注数据访问层（即由你的代码执行或通过你的对象关系映射引擎执行的、用于访问数据库的数据库查询和存储过程）。你通常会发现，在数据访问层上，你能对性能产生的积极影响，远大于你花费同等时间去琢磨如何调优硬件、操作系统或 `SQL Server` 配置所产生的影响。尽管硬件、操作系统和 `SQL Server` 实例的合理配置对于数据库应用程序获得最佳性能至关重要，但这些专业领域已经高度标准化，因此通常只需花费有限的时间即可为性能正确配置系统。然而，诸如查询设计和索引策略之类的应用程序设计问题是你的代码和数据集所独有的。因此，通常在数据访问层中有比硬件、操作系统、`SQL Server` 配置或平台更多需要优化的地方。图 1-3 展示了一项对 346 名数据专业人员进行的调查结果（已获得 Paul Randal 的许可：[`http://bit.ly/1gRANRy`](http://bit.ly/1gRANRy) ）。

如你所见，两个最常见的问题是 `T-SQL` 代码和糟糕的索引。六个最常见的问题中，有四个都与 `T-SQL`、索引、代码和数据结构直接相关。我的经验与其他受访者的相符。通过首先审视数据访问领域，包括逻辑/物理数据库设计、查询设计和索引设计，你可以获得数据库应用程序性能的最大提升。

当然，如果你专注于硬件配置和升级，可能会获得令人满意的性能提升。然而，无论你拥有多少硬件资源，应用程序发出的一个糟糕的 `SQL` 查询都可能耗尽所有可用的硬件资源。因此，糟糕的应用程序设计可能使硬件升级的需求变得非常高，甚至超出你的成本限制。在存在繁重 `SQL` 工作负载的情况下，专注于硬件配置和升级通常投资回报率很低。

你应该在两个层面上分析应用程序对 `SQL Server` 数据库造成的压力。

*   `高层`：分析数据库应用程序对各个硬件资源以及 `SQL Server` 安装的整体行为造成了多少压力。衡量此问题的最佳指标是各种等待状态以及像 `Azure` 这样的平台的 `DTU`。这些信息可以从两方面帮助你。首先，它帮助你确定在 `SQL Server` 应用程序性能不佳时应重点关注哪个领域。其次，它帮助你识别在更高层次上是否存在配置不当的问题。然后，你可以决定可能需要升级哪种硬件资源。

*   `底层`：识别应用程序中确切的元凶——换句话说，就是那些造成你在整体高层面上可见的大部分压力的 `SQL` 查询。这可以通过使用 `Extended Events` 工具和各种动态管理视图来完成，如第 6 章所述。

## SQL Server 性能杀手

现在，让我们考虑可能降低 `SQL Server` 性能的主要问题领域。通过预先了解 `SQL Server` 中主要的性能杀手，你将能够把调优工作集中在可能的原因上。

一旦你优化了硬件、操作系统和 `SQL Server` 设置，`SQL Server` 中的主要性能杀手如下（按大致顺序排列，最严重的排在前面）：

*   索引不足或不准确
*   统计信息不准确
*   查询设计不当
*   生成的执行计划不佳
*   过度的阻塞和死锁
*   非基于集合的操作，通常是 `T-SQL` 游标
*   数据库设计不当
*   重新编译执行计划
*   查询频繁重新编译
*   游标使用不当
*   索引碎片过多

让我们快速看一下这些问题。

### 索引不足或不准确

索引不足通常是 `SQL Server` 中最大的性能杀手之一。同样糟糕，有时更糟的是，存在错误的索引。如果查询没有适当的索引，`SQL Server` 在执行查询时必须检索和处理更多的数据。这会导致磁盘、内存和 `CPU` 承受巨大压力，显著增加查询执行时间。增加的查询执行时间随后可能导致 `SQL Server` 中出现过度的阻塞和死锁。你将在第 8 章至第 12 章中学习如何确定索引策略和解决索引问题。

通常，索引被认为是数据库管理员 (`DBA`) 的职责。然而，`DBA` 无法主动定义如何使用索引，因为索引的使用是由开发人员编写的数据库查询和存储过程决定的。因此，定义索引必须是共同的责任，因为开发人员通常对要检索的数据了解更多，而 `DBA` 则更了解索引的工作原理。不了解查询而创建的索引几乎毫无用处。

索引过多或仅仅是错误的索引会造成同样多的问题。大量的索引会通过 `INSERT`、`UPDATE` 和 `DELETE` 操作减慢数据操作速度，因为必须维护这些索引。性能下降会导致过度的阻塞，再次引发死锁。错误的索引优化器根本不会使用，但仍然需要维护，从而在处理能力、磁盘存储和内存方面付出代价。

### 注意
因为不了解查询而创建的索引几乎毫无用处，所以数据库开发人员至少需要像了解 `T-SQL` 那样了解索引。

### 统计信息不准确

`SQL Server` 严重依赖基于成本的优化，因此准确的数据分布统计信息对于有效使用索引极为重要。没有准确的统计信息，`SQL Server` 的查询优化器无法准确估计查询影响的行数。因为从表中检索的数据量对于决定如何优化查询执行至关重要，如果数据分布统计信息没有准确维护，查询优化器的效果就会大打折扣。统计信息可能因未更新而过时。你也可能看到数据以倾斜方式分布的问题损害了统计信息。关于自动递增值（例如日期）的列的统计信息可能会随着新数据的添加而过时。你将在第 13 章学习如何分析统计信息。

## 性能考量

### 不恰当的查询设计

索引的有效性很大程度上取决于你编写 SQL 查询的方式。从表中检索过多的行，或者指定的过滤条件返回了比实际需求更大的结果集，都会导致索引失效。为了提高性能，你必须确保 SQL 查询的编写能充分利用新的或现有的索引。编写低效的 SQL 查询可能会阻止优化器选择合适的索引，从而增加查询执行时间和数据库阻塞。第 19 章详细介绍了如何编写高效的查询。

查询设计不仅涉及单个查询，还包括通常用于实现数据库功能（例如在队列读者和写者之间进行队列管理）的一组查询。即使设计中使用的单个查询性能良好，数据库的整体性能也可能非常差。解决此类瓶颈需要对 SQL Server 不同的特性有广泛的理解，这些特性会影响数据库功能的性能。本书通篇将介绍如何使用 SQL 查询来设计有效的数据库功能。

### 生成不佳的执行计划

让 SQL Server 能够建立高效的执行计划并重复使用该计划（而不是重新编译）的相同机制，在某些情况下可能对你不利。糟糕的执行计划可能是一个真正的**性能杀手**。不准确和性能低下的计划通常是由于一个名为 `参数嗅探` 的过程出现问题时导致的。`参数嗅探` 是查询优化器基于统计信息中的抽样值或特定值来确定最佳计划的机制所产生的一个过程。理解统计信息和参数如何结合以创建执行计划，以及你能做些什么来控制它们，这一点很重要。统计信息在第 13 章介绍，执行计划分析在第 15 和 16 章介绍。第 17 章专门讨论糟糕的 `参数嗅探` 以及如何最好地处理它（同时结合了来自第 11 章关于查询存储和计划强制的一些细节）。

### 过多的阻塞和死锁

因为 SQL Server 完全支持原子性、一致性、隔离性和持久性（ACID），所以数据库引擎确保并发事务所做的修改彼此之间得到适当的隔离。默认情况下，一个事务看到的数据要么是另一个并发事务修改之前的状态，要么是该事务完成之后的状态——它看不到中间状态。

由于这种隔离性，当多个事务试图以不兼容的方式并发访问一个公共资源时，数据库中就会发生 `阻塞`。两个进程不能在同一时间更新同一份数据。此外，由于 SQL Server 中的所有更新都建立在数据页（8KB 大小的行组）之上，你会看到即使两个进程没有更新同一行，也可能发生 `阻塞`。`阻塞` 在确保正确的数据存储和检索方面是一件好事，但在错误的地方发生过多的 `阻塞` 会拖慢速度。

与 `阻塞` 相关但实际上是另一个问题的是 `死锁`，它发生在两个资源试图升级或扩展锁定的资源并相互冲突时。查询引擎确定回滚代价最小的进程，并选择它作为 `死锁牺牲品`。这需要重新提交数据库请求才能成功执行。`死锁` 是一个根本性的性能问题，尽管许多人认为它是一个结构性问题。查询的执行时间会受到其所面临的 `阻塞` 和 `死锁`（如果有的话）数量的不利影响。

为了实现多用户数据库应用程序的可扩展性能，正确控制查询的隔离级别和事务范围以最小化 `阻塞` 和 `死锁` 至关重要；否则，即使硬件资源可能未被充分利用，查询的执行时间也会显著增加。我将在第 21 和 22 章深入探讨这个问题。

### 非基于集合的操作

Transact-SQL 是一种基于集合的语言，这意味着它对数据集进行操作。这迫使你以列而非行的思维来思考问题。非基于集合的思维会导致过度使用游标和循环，而不是探索更高效的连接和子查询。T-SQL 语言提供了丰富的机制来操作数据集合。为了发挥良好的性能，你需要利用这些机制，而不是强制在代码中采用逐行处理的方式，那会扼杀性能。如何做到这一点的例子遍布全书；另外，我将在第 19 章讨论 T-SQL 最佳实践，并在第 23 章讨论游标。

### 不合适的数据库设计

数据库应进行充分的规范化，以提高数据检索性能并减少 `阻塞`。例如，如果你有一个规范不足的数据库，将客户和订单信息放在同一张表中，那么客户信息将在该客户的所有订单行中重复出现。这种每行信息重复会增加获取客户所有订单所需的页面读取次数。同时，处理客户订单的数据写入器将锁定包含客户信息的所有行，从而可能阻塞所有其他试图访问客户资料的数据写入器/数据读取器。

数据库的过度规范化可能和规范化不足一样糟糕。过度规范化增加了检索数据所需的连接的数量和复杂性。过度规范化的数据库包含大量只有少数列的表。过度规范化并不是我经常遇到的问题，但当我看到它时，它确实会严重影响性能。更常见的是处理结构规范化不足或规范化不当的问题。

查询中包含过多连接也可能是因为数据库实体没有明确分区，或者查询服务于复杂的需求集，或许可以通过创建新的存储过程更好地满足这些需求。

数据库设计的另一个问题是实际实现 `主键`、`唯一约束` 和强制执行的 `外键`。这些机制不仅确保数据的一致性和准确性，而且查询优化器在决定如何解析特定查询时可以利用它们。但人们常常忽略创建 `主键` 或禁用其 `外键`，无论是直接禁用还是使用 `WITH NO_CHECK` 禁用。没有这些工具，优化器别无选择，只能创建次优的计划。

数据库设计是一个庞大的主题。我将在第 19 章以及本书的其余部分提供一些要点。由于该主题的篇幅，我无法以其所需的方式完整地论述它。但是，如果你想阅读一本侧重于介绍该主题的数据库设计书籍，我推荐 Louis Davidson 等人所著的 *Pro SQL Server 2012 关系数据库设计与实现*（Apress，2012）。

### 重新编译执行计划

为了以高效的方式执行查询，SQL Server 的查询优化器会花费相当多的 CPU 周期来创建一个经济高效的执行计划。好消息是，该计划会被缓存于内存中，因此一旦创建，你就可以重用它。但是，如果设计的计划不允许你插入参数值，那么 SQL Server 每次使用不同的值重新提交相同的查询时，都会创建一个新的执行计划。因此，为了获得更好的性能，以有助于 SQL Server 缓存和重用执行计划的形式提交 SQL 查询至关重要。我还将讨论诸如计划冻结、强制查询计划以及使用"即席工作负载优化"等主题。你将在第 16 章中详细看到如何提高执行计划的可重用性。

### 查询的频繁重新编译

确保执行计划可重用（与查询中使用的值无关）的标准方法之一是使用存储过程或参数化查询。使用存储过程来执行一组 SQL 查询，允许 SQL Server 创建参数化的执行计划。

一个*参数化执行计划*独立于在执行存储过程或参数化查询时提供的参数值，因此具有高度的可重用性。查询的频繁重新编译会增加 CPU 的压力并延长查询执行时间。我将在第 18 章中详细讨论存储过程和语句重新编译的各种原因和解决方案。

### 不当使用游标

相比于常规的基于集合的 SQL 查询，偏好基于游标（一次一行）的结果集——或者正如 Jeff Moden 精辟地称之为"Row By Agonizing Row (RBAR; 发音为 'ree-bar')"——会给 SQL Server 增加大量开销。尽可能使用基于集合的查询，但如果被迫处理游标，请务必使用高效的游标类型，如仅快速只进游标。过度使用低效的游标会增加 SQL Server 资源的压力，从而拖慢系统性能。如果你必须使用游标，我将在第 23 章讨论如何正确地使用它们。

### 过度索引碎片

在分析数据检索操作时，通常可以假设数据是按有序方式组织的，正如数据检索操作所使用的索引所指示的那样。但是，如果包含数据的页面以非有序方式碎片化，或者由于频繁的页面拆分而只包含少量数据，那么数据检索操作所需的读取操作次数将远高于原本可能需要的次数。由碎片引起的读取操作次数增加会损害查询性能。在第 14 章中，你将学习如何分析和消除碎片。不过，值得一提的是，关于索引碎片有很多新的观点，认为它可能根本不是问题。你需要评估你的系统以检查这是否是一个问题。

## 总结

在这个介绍性章节中，你了解到 SQL Server 性能调优是一个迭代过程，包括识别性能瓶颈、排查其原因、应用不同的解决方案、量化性能改进，然后重复这些步骤，直到达到所需的性能水平。为了协助这个过程，你应该创建一个系统基线，以便与你的修改进行比较。在整个性能调优过程中，你需要客观地看待你想要进行的调优量——你总是可以让查询运行得再快一点，但付出的努力是否值得其成本？最后，由于性能取决于用户活动和数据的模式，你必须定期重新评估数据库服务器的性能。

为了从 SQL Server 数据库系统中获得最佳性能，理解数据库应用程序对服务器造成的压力至关重要。在接下来的三章中，我将讨论如何分析这些压力，既从较高的系统层面，也从较低的 SQL Server 活动层面。然后，我将展示如何将两者结合起来。

在本书的其余部分，你将深入研究本章前面提到的最大的 SQL Server 性能杀手。你将学习如果使用不当，这些单独因素如何影响性能，以及如何解决或避免这些陷阱。
