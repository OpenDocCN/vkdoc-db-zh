# 3. 磁盘性能分析

#### 将内存表移回标准存储

SQL Server 2014 引入了一种名为 `内存` 表的新表类型。这将表的存储从磁盘转移到内存，从而显著提升了性能。但是，并非所有表或所有工作负载都能从这一新功能中受益。你需要密切关注 `内存` 表的常规查询性能指标，并利用特定的动态管理视图来监控 `内存` 表。我将在第 24 章详细讨论这些内容。如果你的工作负载与 `内存` 表配合不佳，或者系统内存不足，你可能需要将这些 `内存` 表移回磁盘存储。

### 增加系统内存

SQL Server 的内存需求取决于 SQL 活动处理的数据总量。它与数据库的大小或传入的 SQL 查询数量没有直接关联。例如，如果一个内存密集型查询在两个小表之间执行交叉连接，且没有任何过滤条件来缩小结果集，就可能给系统内存带来巨大压力。

最简单、最快捷的解决方法之一就是通过购买并安装更多内存来直接增加系统内存。然而，找出消耗物理内存的原因仍然很重要，因为如果应用程序工作负载极其耗费内存，你可能很快就会受限于系统可访问的最大内存量。要确定哪些查询使用了更多内存，可以查询 `sys.dm_exec_query_memory_grants` 动态管理视图，并收集查询及其 I/O 使用情况的指标。但在使用 `JOIN` 或 `ORDER BY` 语句对此动态管理视图运行查询时需要小心；如果你的系统已经处于内存压力之下，这些操作可能导致你的查询本身需要分配内存。

#### 从 32 位处理器更改为 64 位处理器

将物理服务器从 32 位处理器切换到 64 位处理器（以及随之进行的 Windows Server 软件升级）会彻底改变 SQL Server 的内存管理能力。SQL Server 的内存限制从 3GB 提升到取决于操作系统版本和特定处理器类型的高达 24TB 的限制。

在 SQL Server 2012 之前，可以通过使用地址窗口扩展插件向 SQL Server 实例添加最多 64GB 的数据缓存。这些功能在 SQL Server 2012 中被移除，因此 32 位的 SQL Server 实例被限制为只能访问 3GB 内存。由于这一限制，只有小型系统才应该运行 2017 年之前的 32 位版本 SQL Server。

SQL Server 2017 不支持 `x86` 芯片组。你必须使用 64 位处理器才能运行 2017 版本。

#### 压缩数据

数据压缩在存储和检索信息方面有许多显著的优点。它还有一个许多人没有意识到的好处：当压缩信息存储在内存中时，它会保持压缩状态。这意味着可以在使用更少系统内存的情况下传输更多信息，从而提高整体内存吞吐量。所有这些确实需要付出一定的 CPU 代价，因此你需要密切关注 CPU 使用情况，以确保你不仅仅是转移了压力。有时，由于数据的性质，你可能看不到太大的压缩效果。

#### 启用 3GB 进程地址空间

标准的 32 位地址最多可以映射 4GB 内存。因此，32 位 Windows 操作系统进程的标准地址空间被限制为 4GB。在这 4GB 的进程空间中，默认情况下，上部的 2GB 预留给操作系统，下部的 2GB 提供给应用程序使用。如果你在 32 位操作系统的 `boot.ini` 文件中指定 `/3GB` 开关，则操作系统仅保留 1GB 的地址空间，应用程序可以访问最多 3GB。这也被称为 *4GB 调整*（4GT）。此目的不需要新的 API。

因此，在一台拥有 4GB 物理内存且采用默认 Windows 配置的机器上，你会发现可用内存约为 2GB 或更多。为了让 SQL Server 使用最多 3GB 的可用内存，你可以在 `boot.ini` 文件中添加 `/3GB` 开关，如下所示：

```
[boot loader]
timeout=30
default=multi(o)disk(o)rdisk(o)partition(l)\WINNT
[operating systems]
multi(o)disk(o)rdisk(o)partition(l)\WINNT=
"Microsoft Windows Server 2016 Advanced Server"
/fastdetect /3GB
```

对于物理内存超过 16GB 的系统（如下一节所述），或需要更多内核内存的系统，不应使用 `/3GB` 开关。

运行在 64 位系统上的 SQL Server 2017 在 `x64` 平台上最多可支持 24TB 内存。将生产系统，特别是企业级生产系统，放在 32 位架构上已经没有太大意义了，而且使用 SQL Server 2017 也无法做到。

#### 处理碎片化

虽然存储碎片化听起来可能不像一个性能问题，因为 SQL Server 是从磁盘将信息检索到内存中，但访问的是信息页。如果你有高水平的碎片化，它将直接转化为你的内存管理问题，因为你必须原样存储从磁盘检索到内存中的页，包括空白空间。因此，碎片化不仅会影响存储，也会影响内存。我将在第 17 章讨论碎片化问题。
