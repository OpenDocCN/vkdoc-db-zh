# 21. 阻塞与被阻塞进程

理想情况下，您希望数据库应用程序能够随着数据库用户数量和数据量的增加而线性扩展。然而，常见的情况是，随着用户数量的增加和数据量的增长，性能会下降。导致性能下降的一个原因，特别是与规模不断扩大相关的，就是阻塞。事实上，数据库阻塞通常是数据库应用程序可扩展性的最大敌人之一。

本章将涵盖以下主题：

*   SQL Server 中阻塞的基础知识
*   事务性数据库的 ACID 属性
*   数据库锁的粒度、升级、模式和兼容性
*   ANSI 隔离级别

*   分析阻塞所需的信息

*   解决阻塞问题和建议
*   自动化阻塞检测和信息收集过程的技术

## 阻塞基础知识

在理想世界中，每个 SQL 查询都能够并发执行，而不会被其他查询阻塞。然而，在现实世界中，查询*确实*会相互阻塞，类似于汽车在交叉路口绿灯时通行，会阻挡其他等待通过路口的汽车。在 SQL Server 中，这种交通管理以 `锁管理器` 的形式实现，它控制对数据库资源的并发访问以维护数据一致性。对数据库资源的并发访问是跨多个数据库连接进行控制的。

在继续之前，我想先明确一点。数据库中使用了三个听起来相同、相互关联但含义不同的术语。这些术语经常被混淆，人们经常错误地、互换地使用它们。这些术语是 `锁定`、`阻塞` 和 `死锁`。锁定是 SQL Server 管理多个会话过程中不可或缺的一部分。当会话需要访问一段数据时，会对该数据放置某种类型的锁。这与阻塞不同，阻塞是指一个会话或线程需要访问一段数据，并且必须等待另一个会话的锁被释放。最后，死锁是指两个会话或线程形成了有时被称为 `致命拥抱` 的状态。它们相互等待对方释放锁。死锁也可以被称为永久性阻塞情况，但它是即使等待任何时间段也不会自行解决的阻塞。死锁将在第 22 章中更详细地介绍。因此，锁可能导致阻塞，而锁和阻塞都在死锁中起作用，但这三个是截然不同的概念。请理解这些术语之间的区别并正确使用它们。这将有助于您理解系统、排查故障以及与其他数据库管理员和开发人员进行交流。

在 SQL Server 中，数据库连接由一个会话 ID 标识。连接可能来自一个或多个应用程序，以及这些应用程序上的一个或多个用户；就 SQL Server 而言，每个连接都被视为一个独立的会话。两个会话同时访问同一数据时发生的阻塞是 SQL Server 中的自然现象。每当两个会话试图以冲突的方式访问同一数据库资源时，锁管理器会确保第二个会话等待，直到第一个会话结合系统内事务的管理完成其工作。例如，一个会话可能正在修改表记录，而另一个会话试图删除该记录。由于这两个数据访问请求不兼容，第二个会话将被阻塞，直到第一个会话完成其任务。

另一方面，如果两个会话试图同时读取一个表，则两个请求都被允许执行而不会发生阻塞，因为这些数据访问请求彼此兼容。

通常，阻塞对会话的影响很小，并且不会显著影响其性能。然而，有时由于查询和/或事务设计不佳（或者可能是运气不佳），阻塞会显著影响查询性能。在数据库应用程序中，应尽一切努力最小化阻塞，从而增加可以使用数据库的并发用户数量。

随着 SQL Server 2014 引入内存中表，锁定（至少对于这些表而言）呈现出全新的维度。我将在第 24 章中单独介绍它们的行为。

## 理解阻塞

在 SQL Server 中，数据库查询可以作为独立的工作逻辑单元执行，也可以参与更大的工作逻辑单元。更大的工作逻辑单元可以使用 `BEGIN TRANSACTION` 语句以及 `COMMIT` 和/或 `ROLLBACK` 语句来定义。每个工作逻辑单元必须符合一组称为 `ACID` 属性的四个属性：

*   原子性
*   一致性
*   隔离性
*   持久性

我将在接下来的章节中介绍这些属性，因为理解事务的工作原理是理解阻塞的基础。


### 原子性

一个逻辑工作单元必须是原子的。也就是说，逻辑工作单元中的所有操作要么全部完成，要么不保留任何效果。要理解逻辑工作单元的原子性，请考虑以下示例：

```sql
USE AdventureWorks2017;
GO
DROP TABLE IF EXISTS dbo.ProductTest;
GO
CREATE TABLE dbo.ProductTest (ProductID INT
CONSTRAINT ValueEqualsOne CHECK (ProductID = 1));
GO
--All ProductIDs are added into ProductTest as a logical unit of work
INSERT INTO dbo.ProductTest
SELECT p.ProductID
FROM Production.Product AS p;
GO
SELECT pt.ProductID
FROM dbo.ProductTest AS pt; --Returns 0 rows
```

SQL Server 将前面的 `INSERT` 语句视为一个逻辑工作单元。`dbo.ProductTest` 表中 `ProductID` 列上的 `CHECK` 约束只允许值为 `1`。虽然 `Production.Product` 表中的 `ProductID` 列以值 `1` 开头，但它也包含其他值。因此，`INSERT` 语句根本不会向 `dbo.ProductTest` 表添加任何记录，并且由于 `CHECK` 约束而引发错误。因此，SQL Server 会自动确保原子性。

到目前为止，一切都很好。但在更大的逻辑工作单元情况下，你应该注意 SQL Server 的一个有趣行为。想象一下，前面的插入任务由多个 `INSERT` 语句组成。这些语句可以组合起来形成一个更大的逻辑工作单元，如下所示：

```sql
BEGIN TRAN
--Start:  Logical unit of work
--First:
INSERT  INTO dbo.ProductTest
SELECT  p.ProductID
FROM    Production.Product AS p;
--Second:
INSERT  INTO dbo.ProductTest
VALUES  (1);
COMMIT --End:   Logical unit of work
GO
```

在前面脚本中已创建 `dbo.ProductTest` 表的情况下，`BEGIN TRAN` 和 `COMMIT` 语句对定义了一个逻辑工作单元，表明事务内的所有语句本质上都应是原子的。然而，SQL Server 的默认行为并不确保用户定义事务范围内的某个语句失败时会撤销先前语句的效果。在前面的事务中，第一个 `INSERT` 语句会如前所述失败，而第二个 `INSERT` 语句则完全正常。即使第一个 `INSERT` 语句失败，SQL Server 的默认行为也允许第二个 `INSERT` 语句执行。如下面的代码所示，一条 `SELECT` 语句将返回由第二个 `INSERT` 语句插入的行：

```sql
SELECT  *
FROM    dbo.ProductTest; --Returns a row with t1.c1 = 1
```

用户定义事务的原子性可以通过以下两种方式确保：

*   `SET XACT_ABORT ON`
*   显式回滚

让我们简要地看看这些方法。

#### SET XACT_ABORT ON

你可以使用 `SET XACT_ABORT ON` 语句来修改前面一节中 `INSERT` 任务的原子性。

```sql
SET XACT_ABORT ON;
GO
BEGIN TRAN
--Start:  Logical unit of work
--First:
INSERT  INTO dbo.ProductTest
SELECT  p.ProductID
FROM    Production.Product AS p;
--Second:
INSERT  INTO dbo.ProductTest
VALUES  (1);
COMMIT
--End:   Logical unit of work GO
SET XACT_ABORT OFF;
GO
```

`SET XACT_ABORT` 语句指定当事务中的某个语句失败时，SQL Server 是否应自动回滚并中止整个事务。第一个 `INSERT` 语句的失败将自动暂停整个事务，因此第二个 `INSERT` 语句将不会执行。`SET XACT_ABORT` 的作用范围是连接级别，在重新配置或关闭连接之前一直有效。默认情况下，`SET XACT_ABORT` 为 `OFF`。

#### 显式回滚

你也可以通过使用 SQL Server 中的 `TRY/CATCH` 错误捕获机制来管理用户定义事务的原子性。如果 `TRY` 代码块中的某个语句产生错误，则 `CATCH` 代码块将处理该错误。如果发生错误并激活了 `CATCH` 块，则可以回滚用户定义事务的全部工作，并阻止进一步执行语句，如下所示：

```sql
BEGIN TRY
BEGIN TRAN
--Start: Logical unit of work
--First:
INSERT INTO dbo.ProductTest
SELECT p.ProductID
FROM Production.Product AS p
Second:
INSERT INTO dbo.ProductTest (ProductID)
VALUES (1)
COMMIT --End: Logical unit of work
END TRY
BEGIN CATCH
ROLLBACK
PRINT 'An error occurred'
RETURN
END CATCH
```

`ROLLBACK` 语句回滚事务中到该点为止执行的所有操作。有关如何在基于 SQL Server 的应用程序中实现错误处理的详细描述，请参阅 MSDN 库文章 "Using TRY...CATCH in Transact SQL" ( [`http://bit.ly/PNlAHF`](http://bit.ly/PNlAHF) )。

由于原子性属性要求逻辑工作单元的所有操作要么全部完成，要么不保留任何效果，SQL Server 通过授予事务对受影响资源的独占访问权，将其工作与其他事务*隔离*。这意味着事务可以在需要时安全地回滚其所有操作的效果。授予事务对受影响资源的独占访问权会在此期间阻止所有其他试图访问这些资源的事务（或数据库请求）。因此，虽然原子性是维护数据完整性所必需的，但它会引入不希望的副作用：阻塞。

### 一致性

一个工作单元应使数据库的状态从一个*一致*状态转移到另一个一致状态。在事务结束时，数据库的状态应完全一致。SQL Server 始终通过自动应用受影响数据库资源的所有约束（作为事务的一部分）来确保数据库的内部状态正确且有效。SQL Server 确保事务之后内部结构（如数据和索引布局）的状态是正确的。例如，当修改表的数据时，SQL Server 会自动识别表上的所有索引、约束和其他依赖对象，并作为事务的一部分对所有依赖的数据库对象应用必要的修改。这意味着 SQL Server 将维护数据和对象的物理一致性。

数据的逻辑一致性由业务规则定义，并应由数据库开发人员实施。业务规则可能要求对多个表进行更改、限制特定类型的数据或任何其他数量的要求。数据库开发人员应相应地定义一个逻辑工作单元，以确保考虑到业务规则的所有标准。此外，开发人员将确保已设置适当的结构来支持已定义的业务规则。SQL Server 提供了不同的事务管理功能，数据库开发人员可以使用它们来确保数据的逻辑一致性。

因此，SQL Server 与逻辑的、业务定义的、确保面向业务的数据一致性的约束协同工作，以在底层结构上创建物理一致性。逻辑工作单元的一致性特性会在此期间阻止所有其他试图访问受影响对象的事务（或数据库请求）。因此，即使一致性是维护数据库有效逻辑和物理状态所必需的，它也会引入相同的副作用：阻塞。

### 隔离性

在多用户环境中，可能会有多个事务同时执行。这些并发事务应该彼此隔离，这样一个事务所做的中间更改不会影响其他事务的数据一致性。一个事务所需的`隔离性`程度可能各不相同。SQL Server 提供了不同的事务隔离级别特性，来实现事务所需的隔离程度。

> 事务隔离级别将在本章后面的“隔离级别”一节中解释。

一个数据库资源上操作的事务的隔离要求，可能会阻塞其他试图访问该资源的事务。在多用户数据库环境中，通常会有多个事务同时执行。至关重要的是，一个正在进行的事务所做的数据修改必须免受其他事务修改的影响。例如，假设一个事务正在修改表中的若干行。在此期间，为了维护数据库一致性，你必须确保其他事务不会修改或删除相同的行。SQL Server 通过适当地阻塞其他事务，在逻辑上将一个事务的活动与其他事务的活动隔离开来，这允许多个事务同时执行，而不会破坏彼此的工作。

由隔离引起的过度阻塞会对数据库应用程序的可扩展性产生不利影响。一个事务可能会无意中长时间阻塞其他事务，从而损害数据库并发性。由于 SQL Server 使用锁来管理隔离，因此理解 SQL Server 的锁架构非常重要。这有助于你分析阻塞场景并实施解决方案。

> 数据库锁的基础知识将在本章后面的“捕获阻塞信息”一节中解释。

### 持久性

一旦事务完成，该事务所做的更改应该是`持久的`。即使在事务完成后立即切断机器的电源，事务中所有操作的效果也应该被保留。SQL Server 通过在事务日志中跟踪被修改数据的所有修改前和修改后映像来确保持久性。事务完成后，SQL Server 立即确保该事务所做的所有更改都被保留——即使 SQL Server、操作系统或硬件发生故障（日志磁盘除外）。在重新启动时，SQL Server 运行其数据库恢复功能，该功能从事务日志中识别已完成事务的未决更改，并将其应用到数据库资源。此数据库功能称为`前滚`。

恢复间隔时间取决于在重新启动时需要应用于数据库资源的未决更改数量。为了缩短恢复间隔时间，SQL Server 会按照恢复间隔选项的配置，间歇性地应用正在运行的事务所做的中间更改。恢复间隔选项可以使用 `sp_configure` 语句进行配置。间歇性应用中间更改的过程被称为`检查点`过程。在重新启动时，恢复过程会识别所有未提交的更改，并利用事务日志中数据的修改前映像将它们从数据库资源中移除。

从 SQL Server 2016 开始，`TARGET_RECOVERY_TIME` 的默认值已从 0（这意味着数据库将进行所有自动检查点）更改为一分钟。自动检查点的默认间隔也是一分钟，但现在默认情况下通过 `TARGET_RECOVERY_TIME` 值进行控制。如果需要更改检查点操作的频率，请使用 `sp_configure` 更改恢复间隔值。设置此值意味着数据库正在使用间接检查点。你可以使用间接检查点，而不是依赖自动检查点。这是一种基本上让检查点持续发生以满足恢复间隔的方法。对于数据修改量极高的系统，你可能会因为间接检查点而看到高 I/O。从 SQL Server 2016 开始，所有新创建的数据库都自动使用间接检查点，因为已设置了 `TARGET_RECOVERY_TIME`。从早期版本迁移的任何数据库将使用其在早期版本中使用的检查点方法。你可能也希望更改它们的行为。对于大多数系统来说，使用间接检查点可以带来更一致的检查点行为和更快的恢复。

持久性特性并不是大多数阻塞的直接原因，因为它不要求一个事务的操作与其他事务的操作隔离。但间接地，它会增加阻塞的持续时间。由于持久性特性要求将被修改数据的修改前和修改后映像保存到磁盘上的事务日志，这增加了事务的持续时间，从而增加了阻塞的可能性。

SQL Server 2014 引入了一项功能，可以通过修改给定数据库的持久性行为来减少延迟（等待查询提交并写入日志的时间）。你现在可以使用延迟持久性。这意味着当一个事务完成时，它会立即向应用程序报告为成功事务，从而减少了延迟。但此时日志的写入尚未发生。这也可能允许在等待系统将所有输出写入事务日志的同时，完成更多事务。虽然这可能会提高系统内部的表观速度，并可能减少事务日志 I/O 上的争用，但这本质上是一个危险的选择。这是一个难以给出的建议。微软提出了三种可能使其具有吸引力的情况。

*   你不在意可能丢失部分数据：由于你可能会遇到需要从日志备份恢复到某个时间点的情况，选择将数据库置于延迟持久性模式后，在必须进行恢复操作时可能会丢失一些数据。
*   你在日志写入期间存在高度争用：如果你在事务写入日志时看到大量等待，延迟持久性可能是一个可行的解决方案。但是，如前所述，你也需要能够容忍数据丢失。
*   你正经历整体资源高度争用：服务器上的大量资源争用归根结底是由于锁被持有时间过长。如果你看到大量争用，并且看到日志写入时间长或也在日志上遇到争用，同时你对数据丢失有很高的容忍度，那么这可能是一种有助于减少系统争用的可行方法。

换句话说，我建议仅当你符合所有这些标准时才使用延迟持久性，其中第一条最为重要。另外，不要忘记前面提到的对检查点行为的更改。如果你处于一个高流量系统，有大量数据更改，你可能还需要调整恢复间隔以辅助系统行为。

> 在 ACID 四大特性中，`隔离性`特性也是确保原子性和一致性的主要手段，它是 SQL Server 数据库中阻塞的主要原因。在 SQL Server 中，隔离性是通过锁实现的，这将在下一节中解释。

## 锁

当一个会话执行查询时，SQL Server 会确定需要访问的数据库资源；并且，如果需要，锁管理器会授予该会话不同类型的锁。如果另一个会话已经获得了锁，该查询就会被阻塞；然而，为了同时提供事务隔离和并发性，SQL Server 使用了不同的锁粒度，如下文各节所述。

### 锁粒度

SQL Server 数据库在物理磁盘上以文件的形式维护。对于传统的非数据库文件（例如桌面上的 Excel 文件），一次只能由一个用户写入。任何其他用户尝试写入该文件都会失败。然而，与非数据库文件有限的并发性不同，SQL Server 允许多个用户同时修改（或访问）内容，只要他们不影响彼此的数据一致性即可。这减少了阻塞并提高了事务间的并发性。

为了提高并发性，SQL Server 在以下资源级别按此顺序实现锁粒度：

*   行 (`RID`)
*   键 (`KEY`)
*   页 (`PAG`)
*   区 (`EXT`)
*   堆或 B 树 (`HoBT`)
*   表 (`TAB`)
*   文件 (`FIL`)
*   应用程序 (`APP`)
*   元数据 (`MDT`)
*   分配单元 (`AU`)
*   数据库 (`DB`)

让我们更详细地了解这些锁级别。

### 行级锁

此锁维护在表中的单行上，是数据库表上的最低级别锁。当查询修改表中的一行时，会授予该查询一个针对该行的 `RID` 锁。例如，考虑在以下测试表上的事务：

```sql
DROP TABLE IF EXISTS dbo.Test1;
CREATE TABLE dbo.Test1 (C1 INT);
INSERT INTO dbo.Test1
VALUES (1);
GO
BEGIN TRAN
DELETE dbo.Test1
WHERE C1 = 1;
SELECT dtl.request_session_id,
dtl.resource_database_id,
dtl.resource_associated_entity_id,
dtl.resource_type,
dtl.resource_description,
dtl.request_mode,
dtl.request_status
FROM sys.dm_tran_locks AS dtl
WHERE dtl.request_session_id = @@SPID;
ROLLBACK
```

```text
FileID:PageID:Slot(row)
```

在图 21-1 中针对 `sys.dm_tran_locks` 的查询输出中，`DatabaseID` 单独显示在 `resource_database_id` 列下。对于 `RID` 类型，`resource_description` 列值将 `RID` 资源的剩余部分表示为 `1:121321:0`。在此情况下，`FileID` 为 `1` 表示主数据文件，`PageID` 为 `121321` 表示属于 `dbo.Test1` 表的一页（通过 `C1` 列标识），而 `Slot (row)` 为 `0` 表示该行在页面内的位置。您可以通过执行以下 SQL 语句获取表名和数据库名：

```sql
SELECT OBJECT_NAME(1940201962),
DB_NAME(6);
```

行级锁提供了非常高的并发性，因为阻塞仅限于受影响的行。

### 键级锁

这是索引内的行锁，被标识为 `KEY` 锁。如您所知，对于具有聚集索引的表，表的数据页和聚集索引的叶页是相同的。由于对于具有聚集索引的表，两者的行是相同的，因此在访问表（或聚集索引）中的行时，只会在聚集索引行或有限的行范围内获取一个 `KEY` 锁。例如，考虑在 `Test1` 表上有一个聚集索引。

```sql
CREATE CLUSTERED INDEX TestIndex ON dbo.Test1(C1);
```

接下来，重新运行以下代码：

```sql
BEGIN TRAN
DELETE  dbo.Test1
WHERE   C1 = 1 ;
SELECT  dtl.request_session_id,
dtl.resource_database_id,
dtl.resource_associated_entity_id,
dtl.resource_type,
dtl.resource_description,
dtl.request_mode,
dtl.request_status
FROM    sys.dm_tran_locks AS dtl
WHERE   dtl.request_session_id = @@SPID ;
ROLLBACK
```

`sys.dm_tran_locks` 的相应输出显示的是 `KEY` 锁而非 `RID` 锁，如图 21-2 所示。

![../images/323849_5_En_21_Chapter/323849_5_En_21_Fig2_HTML.jpg](img/323849_5_En_21_Fig2_HTML.jpg)
图 21-2：`sys.dm_tran_locks` 的输出，显示授予 `DELETE` 语句的键级锁

当您查询 `sys.dm_tran_locks` 时，您将能够检索数据库标识符 `resource_database_id`。您也可以从 `resource_associated_entity_id` 获取有关被锁定对象的信息；但是，要获取特定资源（本例中是键所在的页），您必须查看 `resource_description` 列的值，即 `1:34064`。在此情况下，`Index ID` 为 `1` 表示 `dbo.Test1` 表上的聚集索引。您还会看到发出的请求类型：S、IX、X 等。我将在即将到来的“锁模式”部分更详细地介绍这些。

> 注意：您将在本章的“索引对锁的影响”部分了解 `IndId` 列的不同值以及如何确定对应的索引名称。

与行级锁类似，键级锁也提供非常高的并发性。

### 页级锁

页级锁维护在表或索引中的单个页上，被标识为 `PAG` 锁。当查询请求一个页内的多行时，可以通过在各个行上获取 `RID/KEY` 锁，或者在整个页上获取一个 `PAG` 锁，来维护所有请求行的一致性。锁管理器会从查询计划中确定获取多个 `RID/KEY` 锁的资源压力，如果发现压力很高，锁管理器会改为请求一个 `PAG` 锁。

`PAG` 锁锁定的资源在 `sys.dm_tran_locks` 的 `resource_description` 列中可能表示为以下格式：

```text
FileID:PageID
```

页级锁可以通过减少其锁定开销来提高单个查询的性能，但它会阻塞对页面中所有行的访问，从而损害数据库的并发性。

### 区级锁

区级锁维护在一个区（八个连续数据或索引页的组）上，被标识为 `EXT` 锁。例如，当对表执行 `ALTER INDEX REBUILD` 命令，并且表的页可能从现有区移动到新区时，就会使用此锁。在此期间，使用 `EXT` 锁来保护区的完整性。

### 堆或 B 树锁

堆或 B 树锁用于描述何时可能对这两种对象类型之一加锁。目标对象可以是无序堆表（即没有聚集索引的表），也可以是 B 树对象（通常指分区）。`ALTER TABLE` 中的一个设置允许您控制锁升级（在“锁升级”部分介绍）对分区的影响方式。由于分区可以跨多个文件组，每个分区必须有自己的数据分配定义。这就是 `HoBT` 锁发挥作用的地方。它就像表级锁，但作用于分区本身，而不是整个表。


#### 表级锁

这是表上的最高级别锁，标识为`TAB`锁。表级锁保留对整个表及其所有索引的访问权限。

当执行查询时，锁管理器会自动确定在较低级别获取多个锁的锁定开销。如果确定在行级或页级获取锁的资源压力很高，则锁管理器会直接为查询获取表级锁。

`TAB`锁锁定的资源将在`resource_description`中以以下格式表示：

```
ObjectID
```

与其他锁相比，表级锁所需的开销最小，因此可以提高单个查询的性能。另一方面，由于表级锁会阻塞整个表（包括索引）上的所有写请求，因此可能会显著损害数据库并发性。

有时，应用程序特性可能受益于对查询中引用的表使用特定的锁定级别。例如，如果在非高峰时段执行管理查询，则表级锁可能不会对系统用户造成太大影响；但是，它可以减少查询的锁定开销，从而提高其性能。在这种情况下，查询开发人员可以使用锁定提示覆盖锁管理器对查询中引用表的锁定级别选择。

```
SELECT * FROM <table_name> WITH(TABLOCK)
```

但是，在像这样从 SQL Server 拿走控制权时要小心。在实施之前要进行彻底的测试。

### 数据库级锁

数据库级锁在数据库上维护，并标识为`DB`锁。当应用程序建立数据库连接时，锁管理器会为相应的`session_id`分配一个数据库级共享锁。这可以防止用户在其他用户连接到数据库时意外删除或恢复数据库。

SQL Server 确保在一个级别请求的锁尊重在其他级别授予的锁。例如，一旦用户在表行上获得了行级锁，另一个用户就无法获取可能影响该行完整性的任何其他级别的锁。第二个用户可以在其他行上获取行级锁，或在其他页上获取页级锁，但不会向其他用户授予包含该行的、不兼容的页级或表级锁。

用户或数据库管理员无需指定应应用锁的级别；锁管理器会自动确定。当访问少量行以辅助并发性时，它通常倾向于行级锁和键级锁。但是，如果多个低级别锁的锁定开销非常高，锁管理器会自动选择适当的高级别锁。

## 锁操作和模式

由于 SQL Server 需要执行的操作多种多样，因此维护了同样庞大且复杂的锁定机制集。除了不同类型的锁之外，还有一条升级路径用于从一种类型的锁更改为另一种类型的锁。以下部分描述了这些模式和过程，以及它们的用途。

### 锁升级

当执行查询时，SQL Server 确定查询所引用的数据库对象所需的锁定级别，并在获取所需锁后开始执行查询。在查询执行期间，锁管理器跟踪查询请求的锁的数量，以确定是否需要将锁定级别从当前级别升级到更高级别。

锁升级阈值由 SQL Server 在事务过程中确定。当事务超过其阈值时，行锁和页锁会自动升级为表锁。将锁定级别升级到表级锁后，表上所有较低级别的锁都会自动释放。锁管理器的这种动态锁升级功能优化了查询的锁定开销。

可以对给定表上的锁定机制建立一定程度的控制。例如，你可以控制是否发生锁升级。以下是进行该更改的 T-SQL 语法：

```sql
ALTER TABLE schema.table
SET (LOCK_ESCALATION = DISABLE);
```

此语法将完全禁用表上的锁升级（除了少数特殊情况）。你也可以将其设置为`TABLE`，这将导致每次升级都转到表锁。你还可以将表上的锁升级设置为`AUTO`，这将允许 SQL Server 确定锁定模式和任何必要的升级。如果该表已分区，你可能会看到升级更改为分区级别。同样，在修改标准 SQL Server 行为时要谨慎使用这些类型的修改。

你还可以通过使用跟踪标志 1224 在更广泛的范围内禁用锁升级。这会基于锁的数量禁用锁升级，但保留基于内存压力的锁升级。你也可以通过使用跟踪标志 1211 来同时禁用基于内存压力和基于锁数量的锁升级，但这是一个危险的选择，可能导致系统错误。我强烈建议在使用这两种选项之前进行彻底测试。

### 锁模式

不同事务所需的隔离程度可能有所不同。例如，如果两个事务同时读取数据，数据的一致性不会受到影响；但是，如果允许两个事务同时修改数据，则会影响一致性。根据请求的访问类型，SQL Server 在锁定资源时使用不同的锁模式。

*   共享 (S)

*   更新 (U)

*   排他 (X)

*   意向锁
    *   意向共享 (IS)

    *   意向排他 (IX)

    *   共享意向排他 (SIX)

*   架构锁
    *   架构修改 (Sch-M)

    *   架构稳定性 (Sch-S)

*   批量更新 (BU)

*   键范围

#### 共享 (S) 模式

共享模式用于只读查询，例如`SELECT`语句。它不会阻止其他只读查询同时访问数据，因为数据的完整性不会因并发读取而受损。但是，为了维护数据完整性，会阻止对数据的并发数据修改查询。`(S)`锁在数据被读取之前一直保持。默认情况下，`SELECT`语句获取的`(S)`锁在数据读取后立即释放。例如，考虑以下事务：

```sql
BEGIN TRAN
SELECT  *
FROM    Production.Product AS p
WHERE   p.ProductID = 1;
--其他查询
COMMIT
```

在默认隔离级别`read_committed`下，`SELECT`语句获取的`(S)`锁不会一直保持到事务结束；而是在`SELECT`语句读取数据后立即释放。`(S)`锁的这种行为可以通过使用更高的隔离级别或锁定提示来改变。
