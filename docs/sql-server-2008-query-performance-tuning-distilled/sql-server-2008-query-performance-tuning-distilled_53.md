# 滚动锁并发

具有滚动锁并发的游标在底层行上持有 (U) 锁，直到获取了另一个游标行或游标关闭。这可以防止在游标获取该行时其他用户修改它。滚动锁并发模型使游标可更新。

以下 T-SQL 语句创建了一个采用滚动锁并发模型的 T-SQL 游标：
```sql
DECLARE MyCursor CURSOR SCROLL_LOCKS FOR
SELECT adt.Name
FROM Person.AddressType AS adt
WHERE adt.AddressTypeID = 1;
```

由于在引用行期间（直到获取另一个游标行或关闭游标）一直持有锁，因此会阻塞所有在此期间尝试修改该行的其他用户。这损害了数据库并发性，但确保了如果通过游标修改数据则不会出错。

## 游标类型

游标可以分为以下四种类型：
*   只进游标
*   静态游标
*   键集驱动游标
*   动态游标

我们将在接下来的章节中更详细地探讨这四种类型。

### 只进游标

只进游标具有以下特征：
*   它们直接在基表上操作。
*   通常直到使用游标 `FETCH` 操作获取游标行时，才会从底层表检索行。但是，具有以下附加特性的数据库 API 只进游标类型会首先从底层表检索所有行：
    *   客户端游标位置
    *   服务器端游标位置且只读游标并发
*   它们仅支持通过游标向前滚动（`FETCH NEXT`）。
*   它们允许通过游标进行所有更改（`INSERT`、`UPDATE` 和 `DELETE`）。此外，这些游标会反映对底层表所做的所有更改。

数据库 API 游标和 T-SQL 游标实现只进特性的方式不同。数据访问层将只进游标特性实现为前面列出的四种游标类型之一。但 T-SQL 游标并未将只进游标特性实现为一种游标类型；而是将其实现为定义游标可滚动行为的一个属性。因此，对于 T-SQL 游标，只进特性可用于定义其余三种游标类型之一的可滚动行为。

T-SQL 语法提供了一个特定的游标类型选项 `FAST_FORWARD` 来创建快速只进游标。`FAST_FORWARD` 游标的昵称是 `fire hose`，因为它是通过游标移动数据的最快方式，并且所有信息单向流动。但是，当“消防水管”仍然不如传统的基于集合的操作快时，请不要惊讶。以下 T-SQL 语句创建了一个快速只进 T-SQL 游标：
```sql
DECLARE MyCursor CURSOR FAST_FORWARD FOR
SELECT adt.Name
FROM Person.AddressType AS adt
WHERE adt.AddressTypeID = 1;
```

`FAST_FORWARD` 属性指定了一个只进、只读的游标，并启用了性能优化。

#### 静态游标

静态游标具有以下特征：
*   它们在打开游标时在 `tempdb` 数据库中创建游标结果的快照。此后，静态游标在 `tempdb` 数据库中的快照上操作。
*   数据在打开游标时从底层表检索。
*   静态游标支持所有滚动选项：`FETCH FIRST`、`FETCH NEXT`、`FETCH PRIOR`、`FETCH LAST`、`FETCH ABSOLUTE n` 和 `FETCH RELATIVE n`。
*   静态游标始终是只读的；不允许通过静态游标进行数据修改。此外，对底层表所做的更改（`INSERT`、`UPDATE` 和 `DELETE`）不会反映在游标中。

以下 T-SQL 语句创建了一个静态 T-SQL 游标：
```sql
DECLARE MyCursor CURSOR STATIC FOR
SELECT adt.Name
FROM Person.AddressType AS adt
WHERE adt.AddressTypeID = 1;
```

一些测试表明，静态游标可以表现得与只进游标一样好，有时甚至更快。在必须使用游标的情况下，请务必在您自己的系统上测试此行为。

#### 键集驱动游标

键集驱动游标具有以下特征：
*   键集游标由一组称为 `keyset` 的唯一标识符（或键）控制。键集由一组唯一标识结果集中行的列构建而成。
*   这些游标在打开游标时在 `tempdb` 数据库中创建行的键集。
*   游标中的行成员资格仅限于打开游标时在 `tempdb` 数据库中创建的行键集。
*   在获取游标行时，数据库引擎首先查看 `tempdb` 中的行键集，然后导航到相应底层表中的数据行以检索剩余的列。
*   它们支持所有滚动选项。
*   键集游标允许通过游标进行所有更改。在游标外部执行的 `INSERT` 不会反映在游标中，因为游标的行成员资格仅限于打开游标时在 `tempdb` 数据库中创建的行键集。通过游标的 `INSERT` 会出现在游标末尾。在底层表上执行 `DELETE` 时，当游标导航到已删除行会引发错误。对底层表的非键集列执行的 `UPDATE` 会反映在游标中。对键集列的 `UPDATE` 会被视为对旧键值的 `DELETE` 和对新键值的 `INSERT`。如果更改使某行不符合成员资格或影响行的顺序，则该行不会消失或移动，除非游标关闭并重新打开。

以下 T-SQL 语句创建了一个键集驱动的 T-SQL 游标：
```sql
DECLARE MyCursor CURSOR KEYSET FOR
SELECT adt.Name
FROM Person.AddressType AS adt
WHERE adt.AddressTypeID = 1;
```

#### 动态游标

以下是动态游标的特性：

*   动态游标直接在基表上操作。
*   由于直接在基表上操作，游标中的行成员关系不是固定的。
*   与仅向前游标类似，基础表中的行在通过游标 `FETCH` 操作获取游标行之前不会被检索。
*   动态游标支持除 `FETCH ABSOLUTE n` 以外的所有滚动选项，因为游标中的行成员关系不固定。
*   这些游标允许通过游标进行所有更改。此外，对基础表所做的所有更改都会反映在游标中。
*   动态游标不支持数据库 API 游标实现的所有属性和方法。诸如 `AbsolutePosition`、`Bookmark` 和 `RecordCount` 等属性，以及 `clone` 和 `Resync` 等方法，不受动态游标支持。相反，它们由键集驱动游标支持。

以下 T-SQL 语句创建了一个动态 T-SQL 游标：

```sql
DECLARE MyCursor CURSOR DYNAMIC FOR
SELECT adt.Name
FROM Person.AddressType AS adt
WHERE adt.AddressTypeID = 1;
```

动态游标在所有情况下绝对是最慢的游标类型。它需要获取更多的锁并持有更长时间，这极大地增加了其糟糕的性能。在设计系统时请考虑这一点。

### 游标成本比较

既然你已经看到了不同类型的游标，现在让我们来看看它们的成本。如果你必须使用游标，你应该总是使用能满足应用程序要求的最轻量级的游标。接下来将详细说明不同游标特性之间的成本比较。

#### 游标位置的成本比较

客户端和服务器端游标各自具有其成本效益和开销，如下节所述。

##### 客户端游标

与服务器端游标相比，客户端游标具有以下成本效益：

*   *更高的可伸缩性*：由于游标元数据维护在连接到服务器的各个客户端机器上，维护游标元数据的开销由客户端机器承担。因此，服务更多用户的能力不受服务器资源的限制。
*   *更少的网络往返*：由于 `SELECT` 语句返回的结果集被传递到维护游标的客户端，因此在从游标检索行时不需要额外的网络往返服务器。
*   *更快的滚动*：由于游标在客户端本地维护，遍历游标行可能更快。
*   *高度可移植*：由于游标是使用数据访问层实现的，它适用于多种数据库：SQL Server、Oracle、Sybase 等。

客户端游标具有以下成本开销或缺点：

*   *对客户端资源压力更大*：由于游标在客户端管理，它增加了客户端资源的压力。但这可能并不那么糟糕，因为大多数时候客户端应用程序是 Web 应用程序，并且使用标准的负载均衡解决方案可以轻松地横向扩展 Web 应用程序（或 Web 服务器）。另一方面，横向扩展事务性 SQL Server 数据库仍然是一门艺术！
*   *仅支持有限的游标类型*：不支持动态和键集驱动游标。
*   *一个连接上只能有一个活动的基于游标的语句*：结果集的尽可能多的行以网络包的形式安排并发送到客户端应用程序，直到客户端网络缓冲区满为止。因此，在应用程序获取所有游标行之前，数据库连接保持忙状态，将行推送到客户端。在此期间，其他基于游标的语句不能使用该连接。通过利用多个活动结果集（MARS）可以抵消这一点，它允许一个连接拥有第二个活动游标。

#### 服务器端游标

服务器端游标具有以下成本效益：

*   *一个连接上可以有多个活动的基于游标的语句*：使用服务器端游标时，游标操作之间连接上没有未完成的结果。这释放了连接，允许在一个连接上同时使用多个基于游标的语句。在客户端游标的情况下，如前所述，连接在应用程序获取所有游标行之前保持忙状态。这意味着它们不能被多个基于游标的语句同时使用。
*   *靠近数据的行处理*：如果行处理涉及与其他表的连接以及相当多的集合操作，那么使用服务器端游标靠近数据执行行处理是有利的。
*   *对客户端资源压力更小*：它减少了对客户端资源的压力。但这可能不那么可取，因为如果服务器资源耗尽（而不是客户端资源），那么将需要横向扩展数据库，这是一个困难的命题。
*   *支持所有游标类型*：客户端游标在支持的游标类型上有限制。服务器端游标没有限制。

服务器端游标具有以下成本开销或缺点：

*   *更低的可伸缩性*：它们消耗服务器资源来管理游标，从而降低了服务器的可伸缩性。
*   *更多的网络往返*：如果在客户端应用程序中进行游标行处理，它们会增加网络往返。可以通过在存储过程中处理游标行或使用数据访问层的缓存大小特性来优化网络往返次数。
*   *可移植性较差*：使用 T-SQL 游标实现的服务器端游标不容易移植到其他数据库，因为管理游标的数据库代码语法在不同数据库之间是不同的。

#### 游标并发的成本比较

正如预期的那样，并发模型更高的游标在数据库中创建的阻塞最少，并支持更高的可伸缩性，如下节所述。

##### 只读

只读并发模型提供以下成本效益：

*   *最低的锁开销*：只读并发模型对数据库引入的锁和同步开销最少。由于在获取游标行后不会在底层行上持有（S）锁，因此其他用户不会被阻止访问该行。此外，通过在游标的 `SELECT` 语句中使用 `NO_LOCK` 锁定提示，可以避免在获取游标行时获取底层行的（S）锁，但这仅在你不关心由于脏读而返回的数据类型时才适用。
*   *最高的并发性*：由于不持有对底层行的额外锁，只读游标不会阻止其他用户访问底层表。共享锁仍然会被获取。

只读游标的主要缺点如下：

*   *不可更新*：不能通过游标修改底层表的内容。
