# SQL Server 跟踪与分布式重放工具

## 背景介绍

我将这个模板连接到测试服务器而不是生产机器，因为一旦设置妥当，您必须点击 **运行** 来启动跟踪。我不会在生产系统上那样做。然而，在测试系统上，您可以观察屏幕以确保获取您认为应该获得的事件。它会显示事件，同时将它们捕获到文件中。当您确信其正确后，可以暂停跟踪。接下来，单击 **文件** 菜单，然后选择 **导出** ➤ **脚本化跟踪定义**。最后，选择 **For SQL Server 2005 – 2014**（参见图 26-3）。

![../images/323849_5_En_26_Chapter/323849_5_En_26_Fig3_HTML.jpg](img/323849_5_En_26_Fig3_HTML.jpg)

**图 26-3** 输出跟踪定义的菜单选择

此模板允许您将刚刚创建的跟踪保存为 T-SQL 文件。一旦拥有了 T-SQL，您就可以将其配置为在任何您喜欢的服务器上运行。文件路径必须替换，并且您可以通过脚本内的参数重置停止时间。以下脚本展示了用于设置服务器端跟踪事件的 T-SQL 过程的开头部分：

```
/****************************************************/
/* Created by: SQL Server 2017 Profiler          */
/* Date: 05/08/2018  08:27:40 PM         */
/****************************************************/
-- Create a Queue
declare @rc int
declare @TraceID int
declare @maxfilesize bigint
set @maxfilesize = 5
-- Please replace the text InsertFileNameHere, with an appropriate
-- filename prefixed by a path, e.g., c:\MyFolder\MyTrace. The .trc extension
-- will be appended to the filename automatically. If you are writing from
-- remote server to local drive, please use UNC path and make sure server has
-- write access to your network share
exec @rc = sp_trace_create @TraceID output, 0, N'InsertFileNameHere', @maxfilesize, NULL
if (@rc != 0) goto error
```

您可以编辑其中写着 `InsertFileNameHere` 的路径，并为 `@DateTime` 提供不同的值。此时，您的脚本可以在任何 SQL Server 2017 服务器上运行。您很可能可以将相同的脚本一直运行到 SQL Server 2008R2；自那时起，跟踪事件的变化如此之少，以至于它现在已成为固定的标准。然而，为了安全起见，请务必进行测试。

您收集的信息量实际上取决于您想要运行哪种测试。对于标准的性能测试，收集至少一个小时的信息可能是个好主意；然而，在大多数情况下，您不希望捕获超过两到三小时的数据。另外，必须强调的是，跟踪事件不如扩展事件轻量级，因此捕获数据的时间越长，对生产服务器的负面影响就越大。捕获更多数据将意味着管理大量数据，这也意味着您计划运行测试很长时间。这完全取决于您打算在测试中处理的业务和应用程序行为。

在捕获数据之前，您确实需要考虑将在哪里运行测试。假设您不担心磁盘空间，并且不需要保护法律上需要审计的数据（如果您有这些问题，您需要单独解决它们）。如果您的数据库不是 **完整恢复** 模式，那么您无法使用日志备份将其恢复到某个时间点。如果是这种情况，我强烈建议将数据库备份作为启动跟踪数据收集的一部分。原因在于，您需要数据库在开始记录事务时处于相同的条件。如果不是，您可能会遇到大量错误，这可能会严重改变您的性能测试运行方式。例如，尝试选择或修改不存在的数据会影响 I/O 和 CPU 在测试中的测量结果。如果您的数据库保持在您开始跟踪时或接近开始时的状态，那么您应该很少（如果有的话）遇到错误。

准备好数据库副本和一组跟踪数据后，您就可以运行分布式重放工具了。

## 用于数据库测试的分布式重放

假设您使用了分布式重放模板来捕获跟踪信息，您应该准备好开始处理文件了。如前所述，第一步是将跟踪文件转换为不同的格式，一种可以分发到多台客户端机器上进行重放的格式。但这不仅仅是针对您的文件运行可执行文件那么简单。您还需要就您希望分布式重放如何运行做出一些决定；这些决定是在预处理跟踪文件时做出的。

这些决定相当直接。首先，您需要决定是否要重放系统进程以及用户进程。除非您正在处理特定的系统问题，否则我建议将此值设置为 `否`。这也是默认值。其次，您需要决定如何处理空闲时间。您可以使用调用数据库频率的实际值；或者，您可以输入一个以秒为单位的值，将等待时间限制为不超过该值。这实际上取决于您要运行的重放类型。假设您使用 **同步** 模式重放（最适合进行直接性能测量的模式），通过将值设置为较低的值（例如三到五秒）来消除空闲时间是个好主意。

如果您选择使用默认值，则不需要修改配置文件。但如果您选择包含系统调用或更改空闲时间，则需要更改配置文件 `DReplay.Exe.Preprocess.config`。这是一个简单的 XML 配置文件。我使用的文件如下所示：

```
<MaxdIdleTime>5</MaxdIdleTime>
```

我只做了一处更改，调整了 `MaxdIdleTime` 以限制重放期间的任何停顿时间。

在运行预处理之前，请确保已安装 `DRController` 并且 `DReplay` 服务正在您的系统上运行。如果满足条件，您只需要调用 `DReplay.exe` 来执行预处理。

```
dreplay preprocess –i c:\perfdata\dr.trc –d c:\DRProcess
```

在上面的代码中，您可以看到 `DReplay` 运行了预处理事件。输入文件由 `–i` 参数提供，用于存放输出的文件夹则通过 `–d` 参数提供。跟踪文件将被处理，输出将转到指定的文件夹。输出内容将类似于图 26-4。

预处理完成后，您就可以继续运行分布式重放过程了。但在这样做之前，您需要确保一台或多台客户端系统已准备就绪。



### 配置客户端

客户端机器需要配置为与分布式重放控制器协同工作。首先，将客户端安装到不同的机器上。仅用于演示目的，我在单台机器上运行所有组件；但是，即使使用多台机器，设置过程也并无不同。你需要配置客户端以与某个控制器协同工作，且一个客户端一次只能与一个控制器协同工作。你的系统还需要为两项内容预留空间。首先，你需要一个位置来存放每次重放时被覆盖的工作文件。其次，如果你想从该客户端收集执行数据，你还需要为客户端输出的跟踪文件预留空间。你还需要决定客户端进程的日志记录级别。所有这些设置都在另一个 XML 配置文件 `DReplayClient.config` 中完成。以下是我的配置：

```
PerfTune
C:\DRClientWork\
C:\DRClientOutput\
CRITICAL
```

目录和日志记录级别一目了然。我还必须将客户端指向运行分布式重放服务的服务器。对于多个客户端协同工作，不需要其他设置；你只需要确保它们指向了正确的控制器系统。

### 运行分布式测试

至此，你已经完成了所有配置并捕获了数据。接下来，你需要回到命令行，通过 `Dreplay.exe` 可执行文件来运行。大部分控制是通过配置文件完成的，因此可执行文件本身几乎不需要输入。你使用以下命令调用测试：

```
Dreplay replay –d c:\data –w DOJO
```

你需要输入预处理输出的位置，这意味着你需要以逗号分隔的列表形式列出参与测试的客户端机器。执行的输出类似于图 26-5。

运行 `DReplay.exe` 的输出

如图所示，捕获了 62 个事件，并且所有 62 个事件都成功重放。另一方面，如果出现错误或事件失败，你可能需要查明有关某些事件失败原因的信息。这些信息可在日志中找到。然后，只需重新配置测试并再次运行即可。拥有可重复测试过程的核心理念在于你可以反复运行它。前面的例子代表了针对我的本地 AdventureWorks2017 副本执行的一次轻负载运行，捕获过程大约持续了五分钟。但是，我配置了空闲时间的限制，因此重放仅用了 26 秒就完成了。

从这里开始，你可以根据需要重新配置测试，重置数据库，并反复运行测试。请注意，更改配置文件将需要重启相关服务，以确保这些更改在下一组测试中生效。处理此类测试的最佳方法之一是启用查询存储。你可以捕获一组结果，重置测试，对系统进行任何你计划做的更改，然后从第二次测试中捕获另一组结果。接着，你可以轻松地查看关于性能回退的查询、消耗资源最多的查询等的报告。

## 结论

通过包含分布式重放工具，SQL Server 现在使你能够对数据库执行负载和功能测试。你通过服务器端跟踪以简单的方式捕获代码来完成此操作。但是，如果你打算利用此功能，请务必验证你根据本书提出的原则对查询所做的更改是否确实有效，并且是否有助于提高系统的性能。你还应确保重置数据库以尽可能避免错误。
