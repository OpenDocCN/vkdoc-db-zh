# 数据库索引碎片整理技术

你会注意到，大多数行组仍然有些碎片化。其中两个行组（29,30）的状态尚未从`COMPRESSED`更改为`TOMBSTONE`。这意味着这些行组将在稍后的某个时间在后台被移除。简而言之，只有少数行组被合并，几乎未处理任何已删除的行。这是因为`REORGANIZE`命令仅当行组中超过 10%的数据被删除时，才会清理已删除的数据。让我们从表中删除更多数据，以使一些行组的删除比例超过 10%。

```
DELETE dbo.bigTransactionHistory
WHERE Quantity BETWEEN 8
AND     17;
```

现在，当我们查看碎片化结果时，会看到更多的活动，如图 14-21 所示。

![../images/323849_5_En_14_Chapter/323849_5_En_14_Fig21_HTML.jpg](img/323849_5_En_14_Fig21_HTML.jpg)

图 14-21 针对更碎片化的索引执行不带压缩的`REORGANIZE`

你可以看到，更多的行组已被标记为`TOMBSTONE`，并且所有新页面的已删除行数为零。你会注意到，对于已压缩的行组，生成了新的`row_group_id`值。旧的`row_group_id`不会被重用。如果你在清理完成后查看表，将只看到`COMPRESSED`行组，由于数据被移除，总数降至 30，但你会看到空隙。

如果我们重新运行`REORGANIZE`命令但包含行组，命令将如下所示：

```
ALTER INDEX cci_bigTransactionHistory
ON dbo.bigTransactionHistory
REORGANIZE
WITH (COMPRESS_ALL_ROW_GROUPS = ON);
```

命令`COMPRESS_ALL_ROW_GROUPS`将确保增量存储中任何`OPEN`或`CLOSED`的行组都会被移入列存储索引，经历压缩以及列存储索引相关的所有其他过程。

不过，在运行此命令之前，让我们再删除一些数据，以将碎片化程度达到 10%的行组推过阈值。

```
DELETE dbo.bigTransactionHistory
WHERE Quantity BETWEEN 6
AND     8;
```

如图 14-22 所示的结果包括移除`TOMBSTONE`，以及完全重组索引。

![../images/323849_5_En_14_Chapter/323849_5_En_14_Fig22_HTML.jpg](img/323849_5_En_14_Fig22_HTML.jpg)

图 14-22 列存储索引的压缩和碎片整理

实际上，行组的顺序并不重要。当它们被碎片整理时，会从索引中的原始位置移动到之后的新位置。

如果你不想处理 10%的限制，可以使用列存储索引上的`REBUILD`选项，但你将不得不面对在此过程中索引将脱机的事实。

表 14-1 总结了行存储索引上这四种碎片整理技术的特性。

表 14-1 行存储碎片整理技术的特性

| 特性/问题 | 删除并创建索引 | 使用`DROP_EXISTING`创建索引 | `ALTER INDEX REBUILD` | `ALTER INDEX REORGANIZE` |
| --- | --- | --- | --- | --- |
| 重建因聚集索引碎片化而失效的非聚集索引 | 两次 | 否 | 否 | 否 |
| 缺失索引 | 是 | 否 | 否 | 否 |
| 带约束的索引碎片整理 | 非常复杂 | 中等复杂 | 简单 | 简单 |
| 一起碎片整理多个索引 | 否 | 否 | 是 | 是 |
| 与其他进程的并发性 | 低 | 低 | 中等，取决于并发用户活动 | 高 |
| 中途取消 | 危险（无事务保护） | 进度丢失 | 进度丢失 | 进度保留 |
| 碎片整理程度 | 高 | 高 | 高 | 中等至低 |
| 应用新的填充因子 | 是 | 是 | 是 | 否 |
| 更新统计信息 | 是 | 是 | 是 | 否 |

你还可以通过压缩页面内的更多行来减少内部碎片，从而减少页面内的空闲空间。索引叶级页面内可进行的最大压缩量由填充因子控制，如下一节所述。

在处理大型数据库及相关索引时，可能需要通过分区将表和索引分散到磁盘上。分区上的索引也可能随着分区内数据的变化而变得碎片化。处理分区索引时，你需要确定是希望作为`ALTER INDEX`命令的一部分，重新组织或重建（分别使用`REORGANIZE`或`REBUILD`）一个、部分还是所有分区。分区索引不能在线重建。请记住，执行影响所有分区的操作可能代价高昂。

如果在索引（即使是分区索引）上指定了压缩，则必须确保在执行`ALTER INDEX`操作时将压缩设置为其之前的值；否则，压缩设置将会丢失，你将不得不重建索引。这对于非聚集索引尤其重要，因为它们不会从表继承压缩设置。

### 碎片整理与分区

如果你拥有大型数据库，有效管理数据的一个标准机制是将其分解为分区。虽然在极少数情况下，分区可能有助于提高性能，但它们首先是用于管理数据。但是，索引和分区的一个问题是，如果重建索引，在重建期间它将不可用。这意味着对于大型索引上的分区，在重建期间，你可能预期数据的主要部分将离线。SQL Server 2012 引入了在线重建的能力。如果你有一个分区索引，它将如下所示：

```
ALTER INDEX i1 ON dbo.Test1
REBUILD PARTITION = ALL
WITH (ONLINE = ON);
```

这可以重建整个分区，并将其作为在线操作进行，这意味着它在重建期间使索引基本保持可用。但是，对于某些分区来说，这是一个巨大的任务，可能会导致服务器负载过高，并需要更多的 tempdb 存储。SQL Server 2014 引入了新功能，允许你指定单个分区。

```
ALTER INDEX i1 ON dbo.Test1
REBUILD PARTITION = 1
WITH (ONLINE = ON);
```

这降低了重建操作的开销，同时在重建期间仍保持索引基本可用。我强调是“基本”在线，因为在重建期间仍会发生一定程度的锁定和争用。这不是一个完全免费的操作。它只是比替代方案有了极大的改进。

谈到分区重建操作涉及的锁定，SQL Server 2014 还引入了另一项新功能。你可以通过调整`REBUILD`命令来修改重建操作期间使用的锁优先级。

```
ALTER INDEX i1
ON dbo.Test1
REBUILD PARTITION = 1
WITH (ONLINE = ON (WAIT_AT_LOW_PRIORITY (MAX_DURATION = 20,
ABORT_AFTER_WAIT = SELF)));
```

这样做是设置重建操作愿意等待的持续时间（以分钟为单位）。然后，它允许你确定中止哪些进程以为索引重建清除系统。你可以让它停止自己或阻塞进程。最有趣的是，等待进程被设置为低优先级，因此它不会使用大量系统资源，并且任何传入的事务都不会被此进程阻塞。



## `填充因子`的重要性

在行存储索引上，通过让索引的每个叶级页面容纳更多行来减少索引的内部碎片。在一个叶级页面中获得更多的行，减少了索引所需的总页数，进而降低了检索一系列索引行所需的磁盘 I/O 和逻辑读取。另一方面，如果索引键值具有高度的事务性，那么完全填满的索引页会导致页面拆分。因此，对于一个事务性表，需要在最大化页面中的行数和避免页面拆分之间取得良好的平衡。

SQL Server 允许你通过使用 `填充因子` 来控制索引叶级页面内的可用空间量。如果你知道表上会有大量的 `INSERT` 查询或对索引键列的 `UPDATE` 查询，那么你可以使用填充因子预先在索引叶级页面中添加可用空间，以最小化页面拆分。如果表是只读的，你可以创建具有高填充因子的索引以减少索引页的数量。不过，在处理针对 `IDENTITY` 列（或任何包含有序数据、可能形成热点页面的索引键）的插入时，保留一些可用空间是个好主意。

默认的填充因子是 0，这意味着叶级页面被填充到 100%，尽管在 B 树结构的分支节点中会保留一些可用空间。索引的填充因子仅在索引创建时应用。随着键的插入和更新，索引中行的密度最终会稳定在一个狭窄的范围内。正如你在前一章关于 `UPDATE` 和 `INSERT` 导致的页面拆分的章节中所看到的，当发生页面拆分时，通常会将原始页面的一半移动到一个新页面，这与索引创建时使用的填充因子无关。

为了理解填充因子的重要性，让我们使用一个包含 24 行的小型测试表。

```sql
DROP TABLE IF EXISTS dbo.Test1;
GO
CREATE TABLE dbo.Test1 (C1 INT,
C2 CHAR(999));
WITH Nums
AS (SELECT 1 AS n
UNION ALL
SELECT n + 1
FROM Nums
WHERE n < 24)
INSERT INTO dbo.Test1 (C1,
C2)
SELECT n * 100,
'a'
FROM Nums;
```

通过使用默认填充因子创建聚集索引来增加叶级（或数据）页面中的最大行数。

```sql
CREATE CLUSTERED INDEX FillIndex ON Test1(C1);
```

由于平均行大小为 1,010 字节，一个聚集索引叶级页面（或表数据页）最多可包含八行。因此，24 行至少需要三个叶级页面。你可以在图 14-23 所示的 `sys.dm_db_index_physical_stats` 输出中确认这一点。

![../images/323849_5_En_14_Chapter/323849_5_En_14_Fig23_HTML.jpg](img/323849_5_En_14_Fig23_HTML.jpg)
*图 14-23：填充因子设置为默认值 0*

请注意，`avg_page_space_used_in_percent` 为 100%，因为默认填充因子允许页面中压入最大行数。由于一个页面无法包含部分行来完全填满页面，即使使用默认填充因子，`avg_page_space_used_in_percent` 也通常会略低于 100%。

为了减少由 `INSERT` 和 `UPDATE` 操作引起的初始页面拆分频率，通过使用如下填充因子重新创建聚集索引，在叶级（或数据）页面内创建一些可用空间：

```sql
ALTER INDEX FillIndex ON dbo.Test1 REBUILD
WITH  (FILLFACTOR= 75);
```

因为每个页面总共有八行的空间，75% 的填充因子将允许每页六行。因此，对于 24 行，叶级页面的数量应增加到四个，如图 14-24 所示的 `sys.dm_db_index_physical_stats` 输出。

![../images/323849_5_En_14_Chapter/323849_5_En_14_Fig24_HTML.jpg](img/323849_5_En_14_Fig24_HTML.jpg)
*图 14-24：填充因子设置为 75*

请注意，`avg_page_space_used_in_percent` 约为 75%，正如填充因子所设定的。这允许在每个页面中再插入两行而不会引起页面拆分。你可以通过向第一组六行（`C1` = 100 – 600，包含在第一页中）添加两行来确认这一点。

```sql
INSERT  INTO dbo.Test1
VALUES  (110, 'a'),  --第 25 行
(120, 'a') ;  --第 26 行
```

图 14-25 显示了当前的碎片情况。

![../images/323849_5_En_14_Chapter/323849_5_En_14_Fig25_HTML.jpg](img/323849_5_En_14_Fig25_HTML.jpg)
*图 14-25：新记录后的碎片情况*

从输出中可以看到，添加这两行并没有为索引增加任何页面。因此，`avg_page_space_used_in_percent` 从 74.99% 增加到 81.25%。随着向第一组六行添加两行，第一页应该完全填满（八行）。在前八行的范围内进一步添加任何行都将导致页面拆分，从而使索引页数量增加到五。

```sql
INSERT  INTO dbo.Test1
VALUES  (130, 'a') ;  --第 27 行
```

现在，`sys.dm_db_index_physical_stats` 在图 14-26 中显示了差异。

![../images/323849_5_En_14_Chapter/323849_5_En_14_Fig26_HTML.jpg](img/323849_5_En_14_Fig26_HTML.jpg)
*图 14-26：页数增加*

请注意，即使索引的填充因子是 75%，`平均页面密度（满）` 也下降到了 67.49%，这可以计算如下：

```
平均页面密度（满）
= 平均每页行数 / 每页最大行数
= (27 / 5) / 8
= 67.5%
```

从前面的例子中，你可以看到填充因子在索引创建时应用。但后来，随着数据的修改，它就不再具有意义了。无论填充因子如何，每当发生页面拆分时，原始页面的行都会分布在两个页面之间，`avg_page_space_used_in_percent` 也会相应稳定下来。因此，如果你使用非默认的填充因子，你应该确保定期重新应用填充因子以保持其效果。

你可以通过重新创建索引，或使用 `ALTER INDEX REORGANIZE` 或 `ALTER INDEX REBUILD` 来重新应用填充因子，如前所示。`ALTER INDEX REORGANIZE` 会考虑索引创建时指定的填充因子。`ALTER INDEX REBUILD` 也会考虑原始填充因子，但如果需要，它允许指定一个新的填充因子。

如果不进行填充因子的定期维护，无论是对于默认还是非默认的填充因子设置，索引（或表）的 `avg_page_space_used_in_percent` 最终都会稳定在一个狭窄的范围内。

有一种观点认为，与其一遍又一遍地尝试碎片整理索引（并带来所有相关的开销），你可能更倾向于确定一个填充因子，使得索引中页面的数据分布相当标准。有些人确实使用这种方法，牺牲一些读取性能和磁盘空间来避免页面拆分及其导致的相关问题。你需要在自己的系统上进行测试，以找到合适的填充因子，并确定该方法是否有效。



## 自动维护

在一个拥有大量事务的数据库中，表和索引会随着时间推移而碎片化（假设你没有使用前面提到的填充因子方法）。因此，为了提升性能，你应该定期检查表和索引的碎片化程度，并对碎片化程度较高的对象进行碎片整理。你可能还需要考虑工作负载，并根据负载以及索引的碎片化水平来决定何时整理索引碎片。你可以通过以下步骤对一个数据库进行此分析：

1.  识别当前数据库中所有需要分析碎片化的用户表。
2.  确定每个用户表和索引的碎片化程度。
3.  根据以下注意事项，确定需要进行碎片整理的用户表和索引：
    *   碎片化程度较高，即 `avg_fragmentation_in_percent` 大于 20%
    *   表/索引并非非常小——即 `pagecount` 大于 8
4.  对碎片化程度高的表和索引进行碎片整理。

对于一个功能全面的脚本，我强烈建议使用位于 [`http://bit.ly/2EGsmYU`](http://bit.ly/2EGsmYU) 的 Minion Reindex 应用程序，或者 Ola Hollengren 的脚本 [`http://bit.ly/JijaNI`](http://bit.ly/JijaNI)。

除了这些脚本，你也可以使用 SQL Server 内置的维护计划。但是，我不推荐它们，因为为了那一点点易用性，你放弃了太多控制权。使用前面推荐的一套脚本，你会对获得的结果满意得多。

## 总结

正如你在本章所学，在一个高度事务性的数据库中，由 `INSERT` 和 `UPDATE` 语句引起的页面拆分可能会使表和索引碎片化，从而增加数据检索的成本。你可以通过使用填充因子在页面内预留空闲空间来避免这些页面拆分。由于填充因子仅在索引创建时应用，因此你应该定期重新应用它以保持其有效性。对列存储索引的数据操作也会导致碎片化和性能下降。你可以使用 `sys.dm_db_index_physical_stats`（针对行存储索引）或 `sys.column_store_row_groups`（针对列存储索引）来确定索引（或表）的碎片化程度。在确定碎片化程度较高后，你可以根据所需的碎片整理量、数据库并发性以及你处理的是行存储还是列存储索引，选择使用 `ALTER INDEX REBUILD` 或 `ALTER INDEX REORGANIZE`。

碎片整理会重新排列数据，使其在磁盘上的物理顺序与表/索引中的逻辑顺序一致，从而提高查询性能。然而，除非优化器为查询决定一个有效的执行计划，否则即使在碎片整理后，查询性能也可能仍然不佳。因此，让优化器使用高效的技术来生成高性价比的执行计划非常重要。
