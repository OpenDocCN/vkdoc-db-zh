# T-SQL 代码格式化与最佳实践

#### 在复杂操作中使用显式事务

多年来，我发现将复杂的 `DELETE` 语句包装在显式事务中是最佳做法。默认情况下，在 SQL Server 上执行 T-SQL 代码时，我们使用的是隐式事务。这意味着 SQL Server 会在我们执行完事务后自动提交它。我们也可以选择指定一个显式事务。在这种情况下，直到我们向 SQL Server 发送 `COMMIT` 命令，SQL Server 才会完成 T-SQL 代码的执行。尤其是对于删除操作，但通常对于任何复杂的代码，我发现始终保持谨慎是最佳策略。我知道我可能要么运行 `BEGIN TRAN` 和不带 `COMMIT` 的查询，要么在首次执行类似清单 [3-12] 中的查询时以 `ROLLBACK` 结束整个语句。这让我能够核实受影响的记录数量。这是我确认查询是否按预期工作的最后机会。如果返回的记录数量与我预期的不同，我可以发出 `ROLLBACK`，SQL Server 将撤销我试图运行的代码。这只在使用显式事务时才可能实现。同样值得提及的是，如果你使用多级事务（称为嵌套事务），`ROLLBACK` 的功能可能与预期不同。

#### 格式化的考量因素

这引出了设计 T-SQL 格式时的另一个考量。你需要了解你的 T-SQL 代码将如何编写和存储。如果所有 T-SQL 代码都将手动编写，并且无法由第三方工具格式化，那么你可能需要保持编码标准非常简单，并限制对特殊情况的判定标准。然而，如果你有可用的第三方工具，你可以创建一种该工具能够处理的、尽可能复杂的格式。

#### 一致性至关重要

构成一名优秀开发者或工程师的因素之一是纪律性。编写 T-SQL 时也是如此。在某种程度上，你使用哪种特定的格式风格并不重要。重要的是**坚持使用那种格式**。理想情况下，你应该尝试让整个团队就一种标准的 T-SQL 格式化方法达成一致。

当整个团队都以相同的方式编写 T-SQL 代码时，查看他人的代码会容易得多。你不再需要同时解析格式和编码风格，这可以带来更快的分析和更快的问题解决。格式化 T-SQL 代码以使其更具可读性，完全关乎代码的呈现方式。

你应该致力于让公司编写的所有 T-SQL 代码具有一致的格式。由于无论谁编写代码，格式都将相同，这也意味着每个阅读代码的人都会越来越熟悉并快速理解他们的代码。正如你在清单 [3-13] 中看到的，我正在创建一个用户定义的表类型。

#### 大小写规则

关于大小写，至少需要考虑两件事：与关键字相关的大小写，以及所有其他术语的大小写。我倾向于将所有保留字大写。T-SQL 保留字列表可能相当长。我选择将任何不属于数据库模式组成部分的词都视为保留字。我按照数据库对象和列创建时的大小写形式进行大写。当我创建表别名时，我使用小写。你可以在清单 [3-16] 中看到保留字被大写，而数据库对象名称每个单词的首字母被大写。

```sql
CREATE TRIGGER dbo.LogIngredientCostHistory
ON dbo.IngredientCost
AFTER INSERT, UPDATE
AS
IF (ROWCOUNT_BIG() = 0)
RETURN;
INSERT INTO dbo.IngredientCostHistory (IngredientCostID, Cost, DateCost, DateCreated)
SELECT inserted.IngredientCostID, inserted.Cost, GETDATE()
FROM inserted;
```

#### 别名的使用

确定 T-SQL 格式化标准时需要考虑的另一个因素是是否以及如何使用别名。别名是一种允许你创建一个简短名称来引用表的方法。在编写查询时，也可以在 `SELECT` 语句中为列名创建别名。如何使用别名取决于被别名化的是表还是列。然而，整体概念是相同的。如果为表名创建了别名，则必须在该查询的整个生命周期中使用该别名代替表名。对于列，别名通常用于重命名列或使输出对用户更友好。通常列别名不会被引用。

在编写查询时，我唯一会引用列别名的情况是，如果我必须按一个已被别名化的列进行排序，特别是当该列包含的逻辑不仅仅是重命名原始列名时。可以在 `ORDER BY` 子句中提供列的数字顺序表示，而不是提供列名。虽然这是一种快速排序数据的方法，但不建议将其用作永久应用程序代码的一部分，因为 `SELECT` 语句中列的顺序可能会更改，而 `ORDER BY` 子句却没有更新。这将导致返回的数据可能以非预期的顺序排列。

#### 逗号的格式

关于 T-SQL 代码格式化的另一个有争议的话题是如何在编写代码时格式化逗号。有些人喜欢将逗号放在每行的开头。这可以提高可读性并帮助其他人快速识别这是多行中的一行。将逗号添加到每行的开头确实简化了调试，因为很容易注释掉以逗号开头的单行，而查询的其余部分仍能正确解析。我更喜欢将我的逗号放在末尾，这样我可以忽略逗号，专注于查询中返回的列。

#### WHERE 子句的格式

当涉及到 `WHERE` 子句中的多个条件时，我确实更喜欢与那些偏好逗号在开头的人相同类型的方法。我希望 `WHERE` 子句具有与 `SELECT` 语句相同的风格。然而，这并不总是意味着每行的逻辑与逗号的逻辑相同。`WHERE` 子句中的某些行可能包含 `AND` 或 `OR`，而不仅仅是单一的比较。这种额外的逻辑将要求你根据自己的阅读舒适度做出一些决定。我确实希望那些行能突出显示，以便我意识到 `WHERE` 子句中每一行使用的逻辑。
