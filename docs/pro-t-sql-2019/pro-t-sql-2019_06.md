# 第二部分 构建高性能 T-SQL

## 5. 基于集合的设计

了解如何设计 T-SQL 是编写专业代码的基础之一。一旦你能够编写易于理解的 T-SQL 代码，就可以开始专注于提升 T-SQL 代码的性能。如果你是在工作中非正式地学习的 T-SQL，或者你的主要职能是为软件应用程序编写代码，那么在编写 T-SQL 代码时，你可能不会刻意考虑基于集合的设计。你或许是一位经验丰富的数据库开发人员，并希望了解更多关于基于集合设计的知识。

在本章中，我将讨论如何处理数据。第一步是熟悉与数据交互的各种方式。一旦你理解了与数据交互的不同方式，就要开始思考如何为查询准备最佳格式的数据。然后，你应该能够以利用 SQL Server 天然优势的方式来编写这些查询。

### 基于集合的设计简介

### 过程式代码与集合式设计

过程式代码不仅描述了要告诉系统做什么，还描述了怎么做。在许多公司中，软件开发人员或软件工程师负责编写 `T-SQL` 代码。我发现的一件事是，在编写应用程序代码时，这种代码被设计为最适合过程式处理。大部分代码在迭代处理数据时表现良好。在过程式应用程序代码和数据库 `T-SQL` 代码之间切换上下文的成本可能很高。总体而言，编写基于集合的代码所引发的思维过程与迭代处理代码非常不同。

为什么我们甚至有兴趣使用基于集合的设计？为什么它值得这么麻烦？这与 `SQL Server` 引擎的工作方式以及如何检索查询数据有关。当发出查询时，`SQL Server` 需要检索数据记录。`SQL Server` 会前往缓冲区缓存查看这些数据是否可用。图 5-1 显示了如果数据在缓冲池中缓存，`SQL Server` 作为查询请求的一部分检索数据所经历的过程。

**图 5-1** 从单个查询中检索数据

如果查询是按过程式编写的，即每次检索新行时都执行，那么 `SQL Server` 将需要为每条记录都去访问缓冲区缓存。一次检索单行的过程会增加 `SQL Server` 的工作负载。在图 5-2 中，您可以看到一次检索一行所需的额外工作负载。

![../images/480547_1_En_5_Chapter/480547_1_En_5_Fig2_HTML.jpg](img/480547_1_En_5_Fig2_HTML.jpg)

**图 5-2** 当查询一次检索一行时的数据检索

图 5-2 显示了在数据存储在缓冲池中的情况下，一次检索单个记录的查询过程。但是，如果查询是为基于集合的设计编写的，那么将只有一次对缓冲池的调用。在小型数据集上，这可能不会产生很大的影响。当处理数千或数百万条记录的数据集时，执行时间的差异会非常显著。这里还涉及其他因素，这与数据如何存储在缓冲区缓存中有关。当 `SQL Server` 执行查询并取回数据时，它只取回存在于缓冲区缓存中的数据。从 `SQL Server 2019` 开始，有一个称为混合缓冲池的功能。混合缓冲池允许将存储在 `RAM` 和持久内存（`PMEM`）中的数据都视为缓冲区缓存的一部分。否则，如果数据在 `RAM` 或 `PMEM` 中不可用，那么 `SQL Server` 将需要访问磁盘来检索数据。检索到的数据随后将被存储在内存中。图 5-3 显示了这些数据如何从磁盘检索并存储到缓冲区缓存中。

![../images/480547_1_En_5_Chapter/480547_1_En_5_Fig3_HTML.jpg](img/480547_1_En_5_Fig3_HTML.jpg)

**图 5-3** 从磁盘到缓冲区缓存的数据检索

当查询执行时，该数据在那个时间点可能甚至不在缓冲区缓存中。`SQL Server` 必须访问磁盘 `I/O` 子系统来查找该数据。一旦找到该数据，该数据就会被存储在缓冲区缓存中。

如果每一行都被单独拉取，那么对磁盘 `I/O` 的调用就必须为每条记录单独发生。从缓冲区缓存拉取数据并非瞬时完成。从磁盘 `I/O` 获取数据并将其放入缓冲区缓存以供查询检索，则更加耗费资源。最终，这创造了一种场景，您希望专注于基于集合的设计，以获得执行查询时尽可能最佳的性能。

使用数据集最大的挑战之一与插入数据记录有关。大多数情况下，当你插入一条数据记录时，每次只插入单条记录。这可能导致你养成只处理一次插入单个记录的习惯。当你设计新表或移动数据时，可能会遇到可以从数据集角度考虑插入的情况。这通常发生在你创建查询将数据插入另一个表时。

你可以使用食谱中的成分来决定如何更新食谱上的餐食类型。思考你想要使用的逻辑会很有帮助。如果你想根据每个食谱的主要成分将餐食类型更新为一个通用类别，你首先需要确定每个食谱的 `主要成分` 是什么。你还需要为你的 `餐食类型` 确定通用类别。在前面的场景中，我可能会有 `肉类`、`蛋类`、`水果`、`蔬菜`、`谷物` 和 `糖类`。一旦我掌握了这两部分信息，我可能会创建一个临时表来将成分映射到特定类别，除非我已经设计了 `成分表` 为每种成分记录该特定类别。

## 6. 硬件使用

在考虑如何设计查询时，我已经介绍了基于集合设计的要素。在使用 SQL Server 时，还有其他需要考虑的事项。你所面对的是一个超越其各组成部分的系统。SQL Server 涉及诸多方面：存储的数据、访问数据的方式，以及如何访问数据的处理过程。SQL Server 不仅是软件，还涉及该软件如何与你的计算机或服务器的硬件进行交互。

当谈到硬件使用时，SQL Server 会利用几个不同的部分。其中一些可能看起来很直接，而另一些起初可能会令人惊讶。SQL Server 被称为关系数据库管理系统（`RDBMS`）。作为关系数据库管理系统的一部分，SQL Server 的目标是实现高效的数据存储和检索。这意味着，运行 SQL Server 的系统上，最重要的硬件之一是内存。当谈到数据的长期存储时，硬盘驱动器变得最为关键。在本章中，我们将这些称为“存储”，因为存储不仅限于硬盘。然而，还有一些不太明显的要素，比如`CPU`。

### T-SQL 设计中的内存考量

当你刚开始在 SQL Server 中处理数据时，内存是最重要的因素之一可能会让你感到惊讶。这很大程度上与内存在 SQL Server 中的用途有关。原因之一是内存的速度和多功能性。另一个因素与内存成本与存储成本的对比有关。这种成本差异增加了有效使用内存的重要性。内存中存储着许多不同类型的事物。SQL Server 用于访问满足查询条件所需数据的过程，会以“执行计划”的形式保存在内存中。为查询结果检索到的数据也存储在内存中。你还可以让数据库对象存在于内存中，例如表变量或临时表。

首先，让我解释一下为什么在处理 SQL Server 时内存如此重要。你经常听到`OLTP`（联机事务处理）这个术语。它的真正含义是你希望系统能够执行高容量的写入和更新活动。这些正是应用程序中频繁发生的活动类型。在提到写入和更新活动时，以食谱数据库为例，每次访问食谱、对食谱进行的每一项更改，或向系统添加的每一种新食材，都属于此类活动。

虽然系统中存储着大量的数据，但通常人们只想处理其中非常小的一部分子集。技巧在于试图确定哪些数据应该保持随时可用。这将提高添加新条目、更新现有条目或访问已有条目的速度。这正是对硬件有更深入了解能帮助你的地方。

我们最常想到的内存是服务器或机器上可用的`RAM`。然而，还有其他类型的内存存在。甚至比`RAM`更接近`CPU`且速度更快的是`L2 缓存`和`L3 缓存`。`L2`和`L3 缓存`存在于处理器上，其延迟比`RAM`更低。这种延迟可能比`RAM`低十倍。然而，SQL Server 可用的`L2 缓存`或`L3 缓存`容量，通常远小于你的应用程序有效运行所需的数据量。`L2 缓存`的延迟可能只有`L3 缓存`的一半，但速度的优势是以缓存大小为代价的。`L2 缓存`通常以数百 KB 为单位衡量，而`L3 缓存`通常以数十 MB 为单位。我们如何才能尽可能快速且经济高效地访问数据？这正是正确使用内存的卓越之处。

SQL Server 的设计旨在利用内存。有一个称为“缓存”的过程将信息保留在内存中。在 SQL Server 中，有两种类型的缓存非常重要。第一种与缓存“执行计划”有关。这是 SQL Server 保存如何在未来再次执行某个存储过程或即席查询的指令的地方。其目标是利用 SQL Server 在计算执行特定查询的最佳方式时已经获得的知识。第二种与查询设计相关的缓存形式可能更为重要。它与信息如何存储在内存中有关。这被称为“缓冲池”。



### SQL Server 中的内存管理：计划缓存与缓冲池

虽然执行计划和数据都可以保存在缓存中，但 SQL Server 会执行特定计算来决定为计划缓存或缓冲池分配多少内存。此外，还有复杂的计算用于确定在计划缓存或缓冲池中保留某项内容多长时间。其真正目标是确保不常访问的存储过程或查询从计划缓存中删除，从而不占用不必要的空间。同样地，对于缓冲池，目标是仅保留应用程序正在积极使用的数据，并清除应用程序不再需要的数据。在这两种情况下，项目只有在缓存已满且需要为新的执行计划或数据腾出空间时才会被清除。SQL Server 2014 引入了缓冲池扩展功能，提供了另一种解决方案。缓冲池扩展允许您将缓冲池扩展到使用 RAM 和固态硬盘（SSD）。考虑到 SSD 比购买额外的 RAM 更具成本效益，这有助于解决 I/O 瓶颈问题。

#### 计划缓存的特定考量

谈到计划缓存，需要记住一些特定的考量。默认情况下，SQL Server 针对存储过程和预处理语句进行了优化。这种对存储过程的优化也是使用计划缓存时的一个因素。在理想情况下，您的计划缓存中应有大量存储过程，而即席查询很少。然而，这并非对每个业务都适用。您的公司可能使用 ORM 或其他技术（如 LINQ）在应用程序代码中编写即席查询。虽然这对开发人员来说很方便，但它们给 SQL Server 带来了额外成本。如果这些查询没有转换为预处理语句，情况可能尤其如此。

可能发生的情况是，计划缓存会被即席查询填满。这可能导致应用程序更急需的存储过程从计划缓存中被删除。如果有存储过程因应用程序而从计划缓存中被删除，那么当下次调用该存储过程时，SQL Server 将需要重新计算其执行计划。这将给系统带来额外成本。有各种方法可以处理计划缓存中存在大量即席查询的情况。然而，这超出了本书的范围。

#### 缓冲池与数据保留

我在与 SQL Server 交互时遇到的另一个问题与缓冲池以及其中保存的数据有关。我不止一次听说，在处理非常大的表时，您将总是陷入必须处理访问大量数据的困境。如果表设计、索引、查询或上述任何组合未得到优化，这种情况确实可能发生。然而，有方法可以防止它。

### 编写高效查询

使用内存和 SQL Server 时需要考虑的主要方面是，当您编写查询时，这些查询将返回正确的结果并在应用程序可接受的时间内完成。然而，根据整体表设计或查询的编写方式，这些查询访问的数据量可能远远超出必要。乍看之下，这可能不是一个重大问题。当数据表是新的且没有填满数据时，通常不会出现问题。清单 6-1 中的查询用于返回应用程序中的一些数据。

在这些情况下，应用程序经常访问的数据可能正从缓冲池中被清除。当有存储过程运行且必须筛选大量数据时，通过查看与查询关联的逻辑读取数通常可以看出这一点。您可能正在处理一个只返回五个结果的查询；然而，它必须经过 200 万次逻辑读取才能返回结果。这是一个信号，表明您可能正在使用比预期更多的缓冲池。

这种行为可能导致缓冲池中的大量数据被清除。SQL Server 中有一个称为**页面寿命预期**的值，它显示数据页在缓冲池中停留的平均时间。我已开始使用页面寿命预期作为指标，表明有些查询可能在 SQL Server 中读取的数据量远多于结果集中返回的数据量。这可能导致这些查询使用比查询所需更多的内存。通常，当我处理与应用程序相关的查询时，它们通常一次不涉及超过十条记录。如果查询用于某种形式的报告，通常一次也不超过几千条记录。即使我得到的结果是查看 20 万条记录，我仍然不希望为了获得这些结果而处理数千万次逻辑读取。

真正的关键是尝试限制 SQL Server 必须处理的数据量。在编写查询时，这可以通过连接逻辑或 `WHERE` 子句条件来处理。您要确保访问尽可能少的数据。访问所需最少的数据量可以让您在使用 T-SQL 时最小化内存使用。前面在清单 6-1 中引用的查询用于返回一些数据。然而，此查询中包含了一些不需要的额外列和连接。我可以更新查询以移除这些额外的列，如清单 6-2 所示。

### 结合 T-SQL 设计考虑存储

我已经介绍了如何与计划频繁访问的数据进行交互；您还需要考虑大部分数据的存储方式。当涉及到处理存储时，考虑哪些活动导致您必须访问磁盘上的数据至关重要。这些正是我们希望找出如何最小化或改进的活动类型。在某些情况下，当我看到大量的磁盘 I/O 时，这可能表明 T-SQL 设计的某些其他部分需要改进。这可能只是症状而非根源。

在上一节中，我谈到了数据存储在 `缓冲池` 中。如果我们需要的数据不在 `缓冲池` 中，会发生什么？这可能就是 `SQL Server` 内部情况变得有点复杂的地方。`SQL Server` 只与 `缓冲池` 内的数据交互。因此，如果需要的数据不在 `缓冲池` 中，第一步将是确定哪些数据使用频率较低。然后将这些数据从 `缓冲池` 中移除。这些初始步骤已经让我们对内存使用情况有所了解。如果有某些东西不必要地填满了 `缓冲池`，我们现在就知道关键数据可能会被清除出 `缓冲池`，以便为这些新数据腾出空间。

一旦数据被判定为非优先，`SQL Server` 就可以转到磁盘并获取所需的数据。这些数据随后将被存储在 `缓冲池` 中，直到未来某个时间点被判定不再需要。从硬件速度的角度来看，这个过程本身已经引入了延迟。如果数据不存在于 `缓冲池` 中，还需要时间来确定哪些记录需要从 `缓冲池` 中丢弃。将这些记录从 `缓冲池` 中丢弃并转到磁盘，会在从磁盘检索数据所需的时间方面产生额外成本。

如果这些数据仅用于每 4 到 6 小时运行一次的作业，那么这些数据中的大部分对于其他所有进程都可能被视为无关紧要。这导致了从 `缓冲池` 中移除的数据高度更替。新的数据可以从磁盘获取并保存到 `缓冲池`。然后，这些新数据可能被移除出 `缓冲池`，并被从磁盘检索到的更新的数据所取代。

除了处理大量数据相关的这些开销之外，还有其他与磁盘或存储交互的活动。当新数据被写入系统时，该数据将被存储在 `事务日志` 中。数据会不断添加到 `事务日志` 中，直到 `事务日志` 被清空。当事务提交并且信息被写入磁盘时，可以发生日志刷新。在系统中存在大量数据添加、更新或删除的情况下，这可能导致存储的大量使用。

了解这些数据如何存储，以及 `SQL Server` 如何通过 `索引` 和存储在磁盘上的其他数据页轻松找到这些数据，这一点很重要。当信息被添加到 `索引` 或 `索引` 需要更新时，这些信息也会产生存储成本。这不仅涉及存储数据所需的时间，还涉及访问磁盘进行这些添加或更改所涉及的时间。

在使用 T-SQL 时，您需要考虑您的查询如何影响数据存储。正如我们在前一节中看到的，这可能意味着多件事情。您需要确保在给定时间内，您没有访问、插入、更新或删除超出您所需的数据。然而，编写 T-SQL 时，这并非唯一的考虑因素。您可以采取一些措施来监控与查询相关的存储使用情况。其中大多数涉及查看 `物理读取`、`预读` 或查询相关的 `I/O 成本`。



如果你已确认正在处理的是你希望交互的最小数据量，那么可能无法再通过任何操作来降低与你的 `T-SQL` 查询相关的存储成本。在某些情况下，就像内存问题一样，发现大量的存储使用量可能指向其他潜在问题。这些问题可能包括数据在磁盘上的保存方式，也可能包括索引的设计方式。

在某些情况下，你可能会发现频繁的索引更新导致了显著的存储成本。这可能是由于索引根据其设计方式需要不断重组造成的。也存在涉及页拆分（page splits）的情况，你可能会看到与此活动相关的显著存储成本。你的表的排序方式或索引的设计方式可能导致大量的页拆分。你也可能会看到与此活动相关的存储成本增加。

关于存储成本如何运作的更多信息，了解 `SQL Server` 存储引擎的运作机制可能会有所帮助。在处理 `T-SQL` 查询时，有一个方面更频繁地依赖于存储引擎。这与访问当前不在缓冲池（buffer pool）中的数据有关。

存储引擎有几种不同的功能。这些功能涉及访问不在缓冲池中的数据。存储引擎的另一个功能与需要锁时如何处理数据有关。存储引擎的第三个功能涉及与应用程序使用的、非运行 `T-SQL` 相关的命令。

存储引擎在访问数据时可能会遇到性能瓶颈。这些瓶颈反过来又会在访问存储的数据时导致性能问题。一个例子是当查询访问数据表中的行或与索引交互时。在其他时候，存储引擎可能需要处理与为缓冲池检索数据相关的页分配。如果你的公司决定实施行版本控制（row versioning），这也有可能导致存储相关的性能问题。虽然这超出了本书的范围，但如果你正在使用行版本控制并且遇到与存储性能相关的问题，你可能需要做一些额外的研究来确认行版本控制不是原因。

当访问数据行或索引时，`SQL Server` 将利用存储引擎。被访问的数据可能存储在缓冲池中。由存储引擎负责跟踪这些行在 `B` 树（B-tree）中的位置。对于在 `SQL Server` 中管理、添加、更新或删除索引条目，也会发生相同的过程。这在存储引擎中统称为行和索引操作维护。

`SQL Server` 使用索引来组织数据，并允许轻松快速地检索数据。对于大于几个数据页的表，`SQL Server` 会尝试使用索引来帮助遍历各个数据页并快速找到所需的数据。然而，可能存在必要索引不存在的情况。这导致 `SQL Server` 必须读取更多的数据页，有时甚至是表中的所有数据，以确定需要哪些行来满足查询要求。

在编写查询以访问数据时，`SQL Server` 将需要访问的数据称为读取（reads）来跟踪。如果数据在内存中可用，访问该数据的行为称为逻辑读取（logical read）。有时，访问数据可能需要 `SQL Server` 转到磁盘或存储来访问数据页。如果从磁盘访问数据，则该数据称为物理读取（physical read）。如果页被移到缓冲池，则该操作称为预读（read-ahead read）。在任何一种情况下，`SQL Server` 都需要转到磁盘来访问请求的记录。当你编写一个查询并且它需要检索不存在于缓冲池中的记录时，`SQL Server` 将需要访问磁盘上的那些记录。对于 `SQL Server` 来说，这可能是一个非常正常的过程。然而，在查询需要读取的记录多于查询结果所需记录的情况下，这会导致磁盘上发生比所需更多的活动。一种监控从磁盘检索数据所花费时间的方法是使用闩锁等待（latch waits）。这就是 `SQL Server` 访问不存在于缓冲池中的数据所花费的时间量。

除了 `SQL Server` 在存储引擎内管理数据访问外，存储引擎还负责管理与锁定数据相关的活动。这类活动可能会受到服务器、应用程序或事务上配置的并发方法的影响。在许多情况下，并发级别配置通常由与负责编写 `T-SQL` 代码的人员不同的团队管理。存储引擎还负责处理 `SQL Server` 中锁的处理方式。

可能导致与存储相关的性能问题的其他因素与查询设计中可能与硬盘交互的其他领域有关，例如，需要将大量活动记录到事务中的数据库操作。这可能包括插入、更新或删除大量数据。`SQL Server` 被设计为能够及时处理将这些更改记录到事务日志中。然而，这些修改也需要被记录到磁盘上。根据碎片或其他使用硬盘的进程情况，这可能会导致额外的性能问题。

`Tempdb` 是 `SQL Server` 的另一个部分，它可能产生与存储相关的显著开销。可能存在一些查询在 `tempdb` 内引发了大量的活动。在许多情况下，通过将 `tempdb` 放在单独的驱动器上，可以减轻对系统其余部分的影响。在这些情况下，`tempdb` 上的任何存储开销应该只影响其他也依赖 `tempdb` 的查询。

编写 `T-SQL` 查询从根本上说与 `SQL Server` 如何与已存储的数据交互相关联。这种交互的某些方面是可以控制的，例如确保你访问尽可能少的数据。还有其他因素，例如 `SQL Server` 如何访问数据或管理锁定，可能超出你的控制范围。无论哪种情况，重要的是要意识到所有这些部分是如何组合在一起的，并且可能影响你的查询性能。



### 考虑 CPU 与 T-SQL 设计

SQL Server 不仅与内存和存储交互，还与服务器 CPU 进行交互。这是设计查询时需要考虑的最后环节之一。在为 SQL Server 考虑许可授权时，CPU 可能是一项重大成本。部分原因可能是对 SQL Server 所需的最佳 CPU 缺乏了解。这可能导致采购的服务器 CPU 看似比其他 CPU 更有优势，但实际上，CPU 核心数更多并不意味着性能更好。对于 SQL Server，核心数更多的 CPU 可能运行速度反而更慢。因此，市场上的一些 CPU 可能比其他 CPU 更昂贵，但当它们被 SQL Server 使用时，其效果可能还不如其他 CPU。CPU 在 SQL Server 中有多种不同的用途。其中之一是作为 T-SQL 代码的一部分执行计算。

正如本章前面所见，内存的大量使用有时会导致 I/O 问题。同样，当 I/O 活动量非常大时，也可能导致 CPU 问题。在讨论整体 T-SQL 设计中的内存考虑时，我提到了两种类型的缓存。在那一部分，我主要关注了缓冲区缓存或缓冲池。在本节中，我将简要谈谈执行计划的缓存如何可能导致 CPU 性能问题。

#### 执行计划缓存

当 SQL Server 确定如何执行查询时，它会创建一个执行计划。为了让 SQL Server 知道如何执行查询，它必须利用 CPU。这被视为一种 CPU 成本。预期是这类成本不应频繁导致性能问题，因为经常执行的预处理语句和存储过程它们的计划已经保存在缓存中，不需要以任何显著的频率重新创建这些执行计划。

但这引出了什么会导致 SQL Server 判定需要一个新的执行计划。这与理解 SQL Server 在创建执行计划时所做的事情直接相关。创建执行计划的一部分是创建一个允许 SQL Server 快速找到先前创建的执行计划的值。在计划缓存中，这个值被称为`计划哈希`。虽然您可能有多个查询看起来相似或返回相同的结果，但 SQL Server 可能不会给它们相同的`计划哈希`。只有看起来完全相同的查询才具有相同的`计划哈希`。

这种对`计划哈希`的依赖有助于确定是否需要创建新的执行计划。这导致我们依赖于使用参数。这些参数可以存在于存储过程或预处理语句中。如果正在执行的查询具有与计划缓存中相同的`计划哈希`，SQL Server 将不会重新创建新的执行计划。创建执行计划的这个过程称为编译。这可能是您的查询使用的 CPU 资源超出预期的众多原因之一。

SQL Server 也可能判定需要为一个在执行缓存中已有相关`计划哈希`的查询创建新的执行计划。这种情况通常发生在 SQL Server 识别到查询涉及的数据发生了足够显著的变化，需要一个新的执行计划时。这可能是频繁数据更改的结果，也可能是参数嗅探的结果；无论哪种情况，都会导致 SQL Server 识别为一次`重新编译`。就像查询编译有相关成本一样，`重新编译`也会带来 CPU 上的成本。

#### 统计信息与索引碎片

如果查询处理的 I/O 量非常大，这可能会蔓延并导致 CPU 性能问题。虽然查询可能访问了比必要更多的数据，但这可能并非由于整体的 T-SQL 查询设计，而可能是由于统计信息过时。尽管管理统计信息的概念超出了本书的范围，但您需要意识到统计信息对执行计划的影响。统计信息能让 SQL Server 快速了解特定数据表或索引内的数据整体分布情况。其理念是，正确的统计信息有助于 SQL Server 对如何创建执行计划做出最佳猜测。如果这些值过时或陈旧，那么 SQL Server 可能会决定一个不如拥有最新统计信息时高效的执行计划。由于统计信息过时，您可能会看到估计读取次数与实际读取次数之间存在显著差异。这可能对 CPU 造成性能影响。您现在可以看到，统计信息已经更新，估计返回的行数与实际返回的行数保持相对一致。

另一种可能性是，SQL Server 不得不遍历比其最初预期更多的硬盘或存储空间。这可能是由于涉及碎片的情况。在 CPU 活动异常高的情况下，这可能是由于索引碎片造成的。这意味着 SQL Server 可能仅根据索引未被碎片化的假设，估计了检索数据所需的一定工作量。然而，由于索引存在碎片，SQL Server 为访问所请求的数据可能不得不做比预期更多的工作。

#### 数据类型与隐式转换

T-SQL 查询设计中另一个常见的问题涉及数据类型。在 SQL Server 中有很多不同的地方需要定义数据类型。这些定义可以出现在表、临时表、参数、存储过程、参数，甚至通过应用程序代码指定的数据类型中。正如第 1 章前面所讨论的，SQL Server 可以在后台将一种数据类型转换为另一种；这个过程称为隐式转换。虽然 SQL Server 可以处理隐式转换并且有一个结构化的过程来执行它，但强制 SQL Server 执行隐式转换也可能带来相关的 CPU 成本。清单 6-3 中的代码是一个隐式转换的例子。

## 7. SQL Server 性能与执行计划

一些公司可能也在使用数据压缩。数据压缩的好处之一是能够在内存中保存更多数据，因为压缩数据所占用的空间比未压缩数据更小。在 SQL Server 中使用数据压缩也有一些缺点。其中一个缺点与整个数据压缩过程中产生的 CPU 成本有关。这并不意味着你不应该使用数据压缩。我提到这一点是为了让你意识到可能出现的任何潜在问题。

这就引出了并行的概念。最容易观察到 CPU 使用率升高的常见方式之一，就是当你的查询使用了并行处理时。并行是一种概念，指的不是在一个单一线程上执行查询，而是可以在多个线程上执行查询。在这种情况下，假设你的 CPU 有四个核心。每个核心可能有两个线程。如果一个查询只在一个线程上运行，它将只使用你 CPU 四分之一个核心的资源。然而，如果查询以并行方式运行，它可能能够在多个线程上运行。例如，如果它在四个线程上运行，那么这个查询就可以占满你 CPU 的所有核心。这会导致整体的 CPU 处理量增加。

SQL Server 在决定是否在执行查询时使用并行处理时，会考虑许多因素。这很大程度上与 SQL Server 的配置方式有关。虽然配置 SQL Server 超出了本书的范围，但我希望你能了解当 SQL Server 创建执行计划时它是如何工作的。SQL Server 会生成执行计划并计算其总成本。SQL Server 还有一个配置值，用于指示在 SQL Server 决定实施并行处理之前，执行计划应达到的最小成本。这个成本被称为**并行度的成本阈值**。如果执行计划的成本大于并行度的成本阈值，那么 SQL Server 可以判断是否存在一个更好的、以并行方式运行的执行计划。

如果 SQL Server 决定创建一个使用并行的执行计划，它还需要确定给定执行计划允许使用的最大线程数。还有一个配置值用于指示可用于并行处理的最大线程数。这个值被称为**最大并行度**。`MAXDOP` 是最大并行度的缩写。这并不意味着 SQL Server 必须使用与最大并行度中指定的相同数量的线程。它只表示这是执行计划以并行方式运行所允许的最大线程数。

SQL Server 将在执行时确定将在并行执行的执行计划使用的线程数。这将基于查询执行开始时未使用的线程数量来决定。在涉及并行时，可能影响 CPU 性能的一个因素发生在查询并行运行期间使用了更多线程的情况下。并行的基本思想是，虽然总体上会使用更多的 CPU 处理能力，但持续时间会短得多。如果有更多查询进入队列并且需要运行，需要更多线程处于活动状态，这可能会导致 CPU 在较长时间内保持高于正常水平的运行状态。

确保你的 T-SQL 不会对 CPU 使用率产生负面影响的主要原因之一，是与 CPU 相关的货币成本。自 SQL Server 2012 以来，SQL Server 的许可一直与运行 SQL Server 的核心数量相关。在更高版本的 SQL Server 中，这被修改为与服务器可用的核心数量相关，即使这些核心没有明确分配给 SQL Server。当公司开始遇到性能问题时，常见的解决方案之一是增加更多硬件。虽然这可能在短期内解决性能问题，但根据需要添加的硬件情况，这也可能是一个非常昂贵的解决方案。由于每个核心的许可费用可能超过 1000 美元，因此最好尝试通过减少与 CPU 使用率相关的性能瓶颈来最小化对额外 CPU 的需求。

在设计 T-SQL 查询时，有许多方面需要考虑。其中一些因素包括意识到 T-SQL 可能对运行 SQL Server 的物理硬件产生的影响。在编写 T-SQL 查询时，你需要注意访问的数据量，以便最好地利用可用的内存。最小化数据访问的另一个额外好处是减少了存储资源的消耗。确保你的统计信息是最新的并且有适当的索引，将有利于 CPU 使用率。了解你的查询如何影响硬件，不仅可以减少对额外硬件的需求，还可以让你当前的硬件更好地支持你的应用程序。

### 7.1 执行计划简介

当我刚开始编写 T-SQL 时，我专注于使用正确的语法。我努力尝试记住像 `GROUP BY` 子句是否在 `ORDER BY` 子句之前这样的事情。随着我对编写 T-SQL 越来越熟悉，我面临了新的挑战，比如确保我的结果是准确的。随着时间的推移，我发展到了不仅编写具有复杂逻辑的查询，还必须编写几个更简单的查询来验证我的结果。我开始调试其他数据库开发人员编写的 T-SQL 代码。我也负责研究各种应用程序问题。

在许多情况下，这种研究的结果归结为一些在应用程序崩溃或硬件使用率飙升的同时运行的 T-SQL 代码。在那个时候，压力很大，需要尽快找出导致性能问题的原因。最初，我学会了依赖自己对业务需求和数据库设计的理解。虽然这可能非常有帮助，但它确实需要大量的领域知识，这可能需要数年时间才能获得。有更好的、更快的、更简单的方法来找到查询中的痛点。

### 阅读执行计划

当我编写 T-SQL 时，我的最终目标是确保获得准确的结果。在某些情况下，我编写的查询简单而直接。作为一名数据工程师，我发现自己经常需要对与报表相关的查询进行性能调优，或者为编写新的报表查询。由于这些查询是为了提供事务数据的报告，我通常必须比编写应用程序查询时加入更多的连接（join）和更多的逻辑。虽然这并不理想，但对于许多公司来说，这是一个非常常见的情况。

许多公司已经开始意识到其系统中数据的重要性，或者它们希望增加监控以确保应用程序按预期工作。无论哪种情况，你都很有可能在环境中遇到一个对你的应用程序产生负面影响的查询。那时，你会想知道可以采取什么措施来解决这个问题。在本节中，你将了解为什么执行计划很重要，以及何时需要使用它们。你将学习如何阅读执行计划，包括常用的符号。你还将了解执行计划中的属性和警告。

我的第一位导师曾经告诉我，要编写好的 T-SQL，我需要理解 SQL Server 的工作原理。这是我得到过的最明智的建议之一，并且我至今仍在努力更好地理解 SQL Server 内部机制。考虑到这个建议，我们能更好地理解 SQL Server 的一种方式，就是理解 SQL Server 将如何执行我们编写的任何查询。虽然我不会涵盖与执行查询相关的所有内部机制，但我会向你展示 SQL Server 确定用于执行你的 T-SQL 代码的指令。只要计划仍在计划缓存中，这些指令就可以在执行计划中找到。

#### 访问权限

一旦你决定要开始研究某段 T-SQL 代码的执行计划，你应该确认你在 SQL Server 中拥有查看执行计划的访问权限。你有几个选择。如果你打算在 T-SQL 代码执行时获取**实际执行计划**，你需要确保你有执行该代码的权限，并且对于 T-SQL 代码中涉及的数据库，你拥有 `SHOWPLAN` 权限。你也可以从计划缓存中获取执行计划，但你需要能够查询动态管理视图 `sys.dm_exec_cached_plans`。既然你知道了如何验证访问权限，我想接下来介绍一下可用的不同类型的执行计划。

#### 类型与检索方法

有多种方法可以检索执行计划，当时的具体需求将有助于确定使用哪种模型最合适。当你试图排查性能问题时，最佳选择是从 `sys.dm_exec_cached_plans` 检索执行计划。如果你想了解之前使用的执行计划的最佳情况，这是你想使用的方法。如果执行计划已从计划缓存中清除，你可能无法在此处找到该查询或存储过程。此外，根据性能问题发生的时间，缓存中的计划可能已被清除。这可能意味着你能找到一个执行计划，但它可能不是之前使用的同一个计划。

如果你发现执行计划不在计划缓存中，或者自性能问题发生后执行计划已被更改，你可以尝试查找当前的执行计划。使用此方法有两种选择。一种选择是获取 SQL Server 将用作执行计划的假设性方案。这可以在服务器上不实际运行 T-SQL 代码的情况下完成。这被称为**估计执行计划**。使用此方法的最大挑战是，仅仅因为 SQL Server 估计了一个执行计划，并不能保证这就是查询运行时 SQL Server 将使用的实际执行计划。我在日常操作中通常不使用估计执行计划。我只用过几次估计执行计划，那是在我因为担心导致生产环境（Production）中断而犹豫是否要运行 T-SQL 代码的时候。

除了查看计划缓存中的执行计划之外，下一个最佳选择是查看如果现在运行查询，执行计划会是什么样子。在执行 T-SQL 代码时请求**实际执行计划**，将使该执行计划在查询执行完成后可用。在 SQL Server 2014 及更高版本中还有一个更新的选项，即能够在查询执行时查看数据流经执行计划的流程。你不是查看实际执行计划，而是要求 SQL Server 向你显示**实时查询统计信息**。虽然此选项对于查询较慢且存在一个特定关键瓶颈的情况可能有帮助，但我经常发现许多查询完成得太快，无法使用此方法有效诊断我试图解决的许多问题。

#### 总结

通过阅读执行计划，您可以快速识别查询中存在性能问题的地方，以及可以采取哪些步骤来解决这些问题。这包括知道如何访问执行计划，并了解估算执行计划与实际执行计划之间的区别。执行计划还具有多项功能，可帮助您快速识别可能的性能问题。通过查看箭头的大小，您可以判断执行计划中每个步骤通过的数据的相对数量。排序操作符可能为排序数据提供见解，而不仅仅是为了返回有序的结果集。您还可以查看估算和返回的实际行数，以帮助确定 SQL Server 是否有足够的信息来为您提供良好的执行计划。一旦您熟悉了执行计划的这些方面，您可能想研究 SQL Server 是如何使用您的索引的。



### 执行计划中的索引使用

在管理 `T-SQL` 方面，我观察到各公司间最大的差异之一就是索引的管理方式。某些情况下，开发团队负责编写所有包含索引创建的 `T-SQL` 代码。也有一些公司由数据库团队处理所有的索引创建和维护工作。由于数据库管理员通常对开发的访问权限少于开发人员，我认为让两个团队协同工作可能最有帮助。数据库管理员可以发现性能不佳的索引，并且可能在为多个 `T-SQL` 查询设计索引方面更有经验。另一方面，开发人员可能更了解表的设计方式以及 `T-SQL` 代码如何使用这些表。

#### 识别索引问题

存在许多查询、健康检查和监控工具可以帮助你识别索引问题。你可以通过查询来找出哪些存储过程使用了相同的数据库对象，或者你可能心里已经有了一个特定的查询来改进性能。一旦你从缓存中获取执行计划或生成当前的实际执行计划，你就可以开始调查潜在的性能问题。如前一节所述，你知道可以检查每个步骤的相对数据记录量，查看各种运算符和警告，并验证估计的和实际返回的行数。你还可以查看索引在执行计划中是如何被使用的。

#### 执行计划中的逻辑连接类型

除了阅读执行计划和理解索引，你的 T-SQL 代码还可以用来确定执行计划将如何生成。在某些情况下，表如何连接在一起与数据如何存储在表中同样重要。有时，连接列的方式也可能影响你的执行计划。`WHERE`子句中的某些 T-SQL 代码会影响查询执行中使用的逻辑连接类型。如果多个查询被组合，也可能改变查询的执行方式。最终，T-SQL 代码的编写方式与 SQL Server 决定执行查询的方式之间存在关系。

SQL Server 中有几种不同的逻辑连接。其中一些逻辑连接在 T-SQL 中很容易看到。这包括`INNER JOIN`、`LEFT OUTER JOIN`、`RIGHT OUTER JOIN`和`FULL OUTER JOIN`。这些并不是 SQL Server 中存在的唯一逻辑连接。使用某些 T-SQL 命令时，SQL Server 会将一个表与另一个表进行比较，寻找匹配或不匹配的记录。在这些情况下，SQL Server 不执行完整的连接，而是执行`SEMI JOIN`。这种连接类型未被指定为 T-SQL 命令，但一些 T-SQL 命令如`EXISTS`或`NOT EXISTS`会指示一个`SEMI JOIN`。可用的半连接包括`LEFT SEMI JOIN`、`LEFT ANTI-SEMI JOIN`、`RIGHT SEMI JOIN`和`RIGHT ANTI-SEMI JOIN`。最后，还有一些逻辑连接与在同一事务中组合两个或多个查询结果有关。这可以包括通常与`UNION ALL`相关的`CONCATENATION`，或者在 T-SQL 中使用`UNION`时可能发生的`UNION`逻辑连接。根据逻辑连接，SQL Server 将决定可以使用哪些物理连接。通过这种方式，逻辑连接可用于影响查询性能。

有一些物理连接操作符可以与逻辑连接相关联。这四个物理连接操作符分别称为合并连接、哈希连接、嵌套循环和自适应连接。使用`INNER JOIN`时，这四种物理连接操作符中的任何一种都可以作为执行计划的一部分使用。根据数据在表中的存储方式以及表的相对大小，可能会影响使用哪个物理操作符。如果被比较的两个表中的数据已排序，则可以对来自两个表的记录进行并排比较。这将使 SQL Server 能够快速找到匹配或不匹配的记录，具体取决于查询要求。当这种情况发生时，它被称为合并连接。一个出现在执行计划中的合并连接物理操作符如图 7-22 所示。

![../images/480547_1_En_7_Chapter/480547_1_En_7_Fig22_HTML.jpg](img/480547_1_En_7_Fig22_HTML.jpg)

**图 7-22**

嵌套循环

从 SQL Server 2017 开始，有一种新的物理连接操作符可以在 SQL Server 确定查询的执行计划时使用。这个新的物理连接操作符旨在帮助解决给定表中存储的数据可能根据`WHERE`子句中的标准而有显著差异的情况。这种物理连接允许 SQL Server 根据为查询选择的数据来决定是使用哈希匹配还是嵌套循环。这被称为自适应连接。在 SQL Server 2017 中，自适应连接仅在列存储索引上可用。这对于 SQL Server 2019 已经发生了变化。

了解了可用的逻辑连接后，接下来是`LEFT OUTER JOIN`和`RIGHT OUTER JOIN`。在某种程度上，这两种逻辑连接类型是相同但也不同的。例如，嵌套循环只能使用`LEFT OUTER JOIN`。如果查询是使用`RIGHT OUTER JOIN`编写的，SQL Server 将产生额外的开销来将`RIGHT OUTER JOIN`转换为`LEFT OUTER JOIN`。除了这个特定场景，所有物理连接类型都支持使用`LEFT OUTER JOIN`。在 SQL Server 的早期版本中，根据查询中使用的是`LEFT OUTER JOIN`还是`RIGHT OUTER JOIN`，SQL Server 可能生成两个不同的执行计划。这是由于在连接重新排序方面的限制造成的。

下一种逻辑连接是`FULL OUTER JOIN`。在所有连接类型中，这种类型对于何时可能使用一种物理连接类型而非另一种有最多的特定条件。在第一个例子中，我将按标识符对表进行排序，如清单 7-5 所示。

```
ALTER TABLE dbo.RecipeIngredient
ADD CONSTRAINT PK_RecipeIngredient_RecipeIngredientID
PRIMARY KEY CLUSTERED (RecipeIngredientID)
Listing 7-5
Primary Key to Sort Data by RecipeIngredientID
```

```

Outer Join Between Recipe and RecipeIngredient

AND recing.RecipeID = 2
)
```
