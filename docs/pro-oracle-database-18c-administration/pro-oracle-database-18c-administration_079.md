# 安排任务自动运行

要安排一个任务，你必须在你的 `cron` 表中添加一行，指定你希望任务执行的时间。有两种方法可以在你的 `cron` 表中添加条目：

- 直接编辑 `cron` 表文件
- 从文件加载 `cron` 表

这两种技术将在以下部分中描述。

## 直接编辑 Cron 表

你可以使用 `crontab` 命令的 `-e`（编辑器）选项直接编辑你的 `cron` 表：

```
$ crontab -e
```

发出上述命令后，你将看到一个要编辑的文件。这个文件被称为你的 `cron` 表（`crontab`）。要安排一个名为 `backup.bsh` 的脚本在每天晚上 11:05 运行，请将以下行输入到你的 `cron` 表中：

```
5 23 * * * /home/oracle/bin/backup.bsh
```

这里，5 指定任务将在整点过 5 分时运行。23 是军用时间，指定任务应该在 2300 时段运行（在此示例中，是整点过 5 分）。三个星号 (`* * *`) 表示任务应该在每月的每一天、每年的每一个月和每周的每一天运行。

退出 `cron` 表文件。如果你的默认编辑器是 `vi`，则输入 `wq` 退出。当你退出 `crontab` 时，你的 `cron` 表会为你保存。要查看你的 `cron` 条目，请使用 `crontab` 命令的 `-l`（列表）选项：

```
$ crontab -l
```

要完全移除你的 `cron` 表，请使用 `r` 选项：

```
$ crontab -r
```

在运行上述命令之前，你应该将你的 `cron` 表保存到一个文本文件中，如下所示：

```
$ crontab -l > saved.cron
```

这样，如果你并非有意删除 `cron` 表，可以参考已保存的文件。

> **提示**
> “我曾经和一位认为 `crontab -r` 意思是‘读取 cron 表’的 DBA 共事过。千万别犯那个错误。总是将你的 crontab 备份到文本文件是个好习惯，这样可以通过 `crontab -l > crontab.backup` 来确保计划表在这些 `crontab -r` 引起的问题中是安全的。这个备份文件也可以用于加载到其他服务器上。”

## 设置默认编辑器

用于修改 `cron` 表的默认编辑器取决于你的 `VISUAL` 操作系统变量的值。在我当前的环境中，`VISUAL` 变量被设置为 `vi`：

```
$ echo $VISUAL
vi
```


## 从文件加载 Cron 表

修改 `cron` 表的另一种方法是通过文件名直接加载。这样你可以保存 `cron` 作业，并通过直接加载文件来确保每台服务器上的作业保持一致，语法如下：

```bash
$ crontab
```

在此，`crontab` 实用程序会将指定文件的内容加载到你的 `cron` 表中。使用此方法修改 `cron` 表的建议步骤如下：

1.  修改 `cron` 表前，先将当前 `cron` 表内容填充到一个文件中；例如，
2.  接下来，复制先前创建的文件（`mycron.txt`）。这样，如果你引入了错误且无法轻易找出问题所在，可以回退到原始文件。同时，这也为你提供了 `cron` 表变更的审计跟踪：

    ```bash
    $ crontab -l > mycron.txt
    ```

3.  现在，用你喜欢的文本编辑器编辑 `mycron.txt` 文件：

    ```bash
    $ cp mycron.txt mycron.jul29.txt
    ```

    ```bash
    $ vi mycron.txt
    ```

    例如，要将一个名为 `backup.bsh` 的脚本安排在每天晚上 11:05 运行，请添加以下几行：

    ```bash
    #--------------------------------------------------------------------------
    # 文件备份，dk：2014 年 1 月 20 日，插入。
    5 23 * * * /home/oracle/bin/backup.bsh
    #--------------------------------------------------------------------------
    ```

4.  当你编辑完成后，将 `mycron.txt` 的内容加载到 `cron` 表中，如下所示：

    ```bash
    $ crontab mycron.txt
    ```

如果你的文件不符合 `cron` 语法，你将收到类似以下的错误：

```bash
"mycron.txt":6: bad day-of-week
errors in crontab file, cannot install.
```

在这种情况下，要么更正语法错误，要么重新加载原始的 `cron` 表副本。

## 典型的 Cron 表示例

这里列出了数据库服务器上 `cron` 表的一个示例条目：

```bash
#----------------------------------------------------------------------------
# 分(0-59) 时(0-23) 日(1-31) 月(1-12) 周(0/7-6) 命令或脚本
#----------------------------------------------------------------------------
# RMAN 备份，dk：2012 年 5 月 1 日，更新。
1 16 * * * /u01/oracle/bin/rmanback.bsh INV >/u01/oracle/bin/log/bck.log 2>&1
#-----------------------------------------------------------------
# 表空间检查，sp：2012 年 12 月 17 日，创建。
5 * * * * /u01/oracle/bin/tbsp_chk.bsh INV 10 1>/u01/oracle/bin/log/tbsp.log 2>&1
#-----------------------------------------------------------------
```

请注意此条目中的几个方面。在每个 `cron` 表（以及每个数据库服务器）的顶部放置一行，简要解释日期调度特性的含义：

```bash
# 分(0-59) 时(0-23) 日(1-31) 月(1-12) 周(0/7-6) 命令或脚本
```

用注释行分隔每个条目。这使条目更易读：

```bash
#-----------------------------------------------------------------
```

此外，包含一个简短的注释（附上作者或修改者的姓名首字母），描述 `cron` 作业以及编辑的时间：

```bash
# RMAN 备份，dk：2018 年 1 月 1 日，更新。
```

如果你管理着数十台数据库服务器（每台都有自己的 `cron` 表），并且有多位数据库管理员，你将需要某种机制（不一定是复杂的）来跟踪谁在何时进行了更改。

## 重定向 cron 输出

每当你运行一个 Linux shell 命令时，默认情况下，（该命令的）标准输出将显示在你的屏幕上。同样，如果产生任何错误消息，它们默认也会显示在你的屏幕上。你可以使用 `>` 或 `1>`（它们是同义词）将任何标准输出重定向到操作系统文件。此外，你可以使用 `2>` 将任何错误消息重定向到文件。符号 `2>&1` 指示 shell 将任何错误消息发送到与标准输出相同的位置。即使在发送脚本执行的输出时，也要确保脚本具有适当的日志记录和报告功能，无论是记录到日志文件还是数据库表中，以反映成功或失败。

当你创建 `cron` 作业时，你可以使用这些重定向功能将 shell 脚本的输出发送到日志文件。例如，在下面的 `cron` 表条目中，`backup.bsh` shell 脚本生成的任何标准输出和错误消息都被捕获到一个名为 `bck.log` 的文件中：

```bash
11 12 * * * /home/oracle/bin/backup.bsh 1>/home/oracle/bin/log/bck.log 2>&1
```

如果你不重定向 `cron` 作业的输出，那么任何输出都会通过电子邮件发送给拥有该 `cron` 表的用户。你可以通过直接在 `cron` 表中指定 `MAILTO` 变量来覆盖此行为。通过以下几行代码，`cron` 的输出将发送给 `root` 用户：

```bash
MAILTO=oraclebkup
11 12 * * * /home/oracle/bin/backup.bsh
```

如果你不希望输出发送到任何地方，那么将其重定向到众所周知的“位桶”。以下条目将标准输出和标准错误发送到 `/dev/null` 设备：

```bash
11 12 * * * /home/oracle/bin/backup.bsh 1>/dev/null 2>&1
```

## 排查 cron 故障

如果你有一个运行不正常的 `cron` 作业，请按照以下步骤排查问题：

1.  复制你的 `cron` 条目，将其粘贴到操作系统命令行中，然后手动运行该命令。通常，目录或文件名中一个轻微的拼写错误可能就是问题的根源。手动运行命令会突出显示此类错误。

2.  如果脚本运行的是 Oracle 实用程序，请确保你在脚本内为所需的 OS 变量进行了 source（设置）操作，例如 `ORACLE_HOME` 和 `ORACLE_SID`。通常，这些变量在你登录时由启动脚本（例如 `HOME/.bashrc`）设置。因为 `cron` 不会运行用户的启动脚本，所以任何所需的变量都必须在脚本中显式设置。

3.  确保任何从 `cron` 调用的 shell 脚本的第一行指定了用于解释脚本中命令的程序名称。例如，`#!/bin/bash` 应该是 Bash shell 脚本的第一条条目。因为 `cron` 不运行用户的启动脚本（如 `HOME/.bashrc`），你不能假设操作系统的默认 shell 将被用来运行从 `cron` 调用的命令或脚本。

4.  确保 `cron` 后台进程正在运行。从操作系统发出以下命令进行验证：

    ```bash
    $ ps -ef | grep cron
    ```

    如果 `cron` 守护进程（后台进程）正在运行，你应该看到类似以下的内容：

    ```bash
    root      2969     1  0 Mar23 ?        00:00:00 crond
    ```

5.  检查你在服务器上的电子邮件。当有 `cron` 作业运行不正常时，`cron` 实用程序通常会向操作系统账户发送一封电子邮件。

6.  检查 `/var/log/cron` 文件的内容是否有错误。有时，此文件包含有关未能运行的 `cron` 作业的相关信息。

7.  验证作业设置的运行时间，并确保所有列都与日期、时间等对齐。
