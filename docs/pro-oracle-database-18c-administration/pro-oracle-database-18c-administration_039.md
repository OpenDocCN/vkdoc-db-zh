# Oracle RMAN 备份配置与实践指南

## 1. 本地与远程运行 RMAN
当在本地运行并且数据库环境变量配置完成后，可以像这样运行 RMAN：

```
$ rman target /
```

如果您远程运行 RMAN，需要确保远程的 `rman` 可执行文件与目标数据库兼容。例如，您可能确认正在运行的远程 `rman` 可执行文件是 Oracle Database 12c 版本的 RMAN 客户端，并且它与几个早期版本的 Oracle 数据库兼容。如果您在目标服务器上本地运行 `rman` 客户端，则永远不会出现兼容性问题，因为 `rman` 客户端总是与目标数据库版本相同。

## 2. 指定备份用户
如前所述，RMAN 要求您使用具有 `SYSDBA` 权限的数据库用户。无论我是从命令行运行 RMAN 还是在脚本中调用 RMAN，在大多数场景下，使用一个备份用户是合适的。例如，以下是如何从命令行连接到 RMAN：

```
$ rman target BACKUPUSER/$password
```

一些 DBA 不使用这种方法；他们选择设置一个独立于 `SYS` 的用户，并以安全考虑作为理由。密码变量可以从一个安全的加密文件中读取密码，或者使用其他方法来获取密码。本章的其他示例可能会展示 `SYS` 用户的用法：

```
$ rman target
```

## 3. 使用联机或脱机备份
大多数生产数据库都有 24-7 的可用性要求。因此，您唯一的选择是联机 RMAN 备份。您的数据库必须处于归档日志模式才能进行联机备份。您需要仔细考虑如何放置归档日志、如何格式化它们、备份频率以及删除前保留多长时间。这些主题将在后续章节中讨论。

> **注意**
> 如果您进行脱机备份，您必须使用 `IMMEDIATE`、`NORMAL` 或 `TRANSACTIONAL` 选项关闭数据库，然后将其置于加载模式。RMAN 需要数据库处于加载模式才能读写控制文件。

## 4. 设置归档日志目标位置与文件格式
启用归档重做日志模式是进行联机备份的先决条件（有关归档日志目标位置和格式的架构决策以及如何启用/禁用归档日志模式的完整讨论，请参见第 5 章）。

当启用归档日志模式时，Oracle 会将归档日志写入以下一个或多个位置（您可以配置归档日志写入 FRA 以及通过初始化参数手动设置的其他几个位置）：

*   默认位置
*   FRA（快速恢复区）
*   通过 `LOG_ARCHIVE_DEST_N` 初始化参数指定的位置

如果您不使用 FRA，并且也没有通过 `LOG_ARCHIVE_DEST_N` 初始化参数显式设置归档日志目标位置，那么默认情况下，归档日志将被写入一个依赖于操作系统的默认位置。在许多 Linux/Unix 系统上，默认位置是 `ORACLE_HOME/dbs` 目录。归档日志的默认文件名格式是 `%t_%s_%r.dbf`。这个格式是引用日期、备份集编号等的简写。其中 `%t` 是时间戳，`%s` 是日志序列号，`%r` 是重置日志 ID。

如果您启用了 FRA（并且没有设置 `LOG_ARCHIVE_DEST_N`），那么默认情况下，归档日志会被写入 FRA 中的一个目录。在 FRA 中创建的归档日志文件的默认文件名格式是 Oracle 托管文件格式。这些文件存储在一个与数据库唯一名同名的子目录中；例如：

```
///archivelog//o1_mf_1_1078_68dx5dyj_.arc
```

推荐使用 FRA。要使用 `LOG_ARCHIVE_DEST_N` 参数设置归档日志文件的位置，请遵循以下示例：

```
log_archive_dest_1='LOCATION=/oraarch1/CHNPRD'
```

这里还有一个推荐的默认归档日志文件名格式：

```
log_archive_format='%t_%s_%r.arc'
```

有时，DBA 对数据文件和归档日志文件都使用 `.dbf` 作为扩展名。我更喜欢对归档日志文件使用 `.arc`。`.arc` 扩展名避免了将文件识别为归档日志文件还是实时数据库数据文件的潜在混淆任务。

## 5. 配置 RMAN 备份位置与文件格式
当您为基于磁盘的备份运行 `BACKUP` 命令时，RMAN 会在以下位置之一创建备份片：

*   默认位置
*   FRA
*   通过 `BACKUP...FORMAT` 命令指定的位置
*   通过 `CONFIGURE CHANNEL...FORMAT` 命令指定的位置

### 5.1 默认位置
如果您没有配置任何 RMAN 变量，也没有设置 FRA，默认情况下 RMAN 会分配一个基于磁盘的通道，并将备份文件写入默认位置。例如，您可以不配置任何 RMAN 参数直接运行以下命令：

```
RMAN> backup database;
```

默认位置因操作系统而异。在许多 Linux/Unix 环境中，默认位置是 `ORACLE_HOME/dbs`。所创建备份文件的默认名称格式是 OMF 格式；例如：

```
/dbs/01ln9g7e_1_1
```

> **提示**
> 默认位置对于小型开发数据库来说是可以的。然而，对于大多数其他环境（尤其是生产环境），您需要提前规划备份所需的磁盘空间，并通过其他方法之一（例如实施 FRA 或 `CONFIGURE CHANNEL`）显式设置备份位置。

### 5.2 FRA
当备份到磁盘时，如果您没有明确指示 RMAN 将备份写入特定位置（通过 `FORMAT` 或 `CONFIGURE` 命令），并且您正在使用 FRA，RMAN 会自动将备份文件写入 FRA 中的目录。当您使用 FRA 时，RMAN 在给定日期首次备份数据库时会自动创建单独的目录。这些文件存储在一个与数据库唯一名同名的子目录中。此外，在 FRA 中创建的备份文件的默认名称格式是 OMF 格式；例如：

```
///backupset//o1_mf_nnndf_TAG20100907T025402_68czfbdf_.bkp
```

可以使用 `SYSTEM` 动态参数设置 FRA 的大小或位置。这在备份集仍在被清除或归档日志填满空间时非常有用；与其等待备份或文件移动完成，不如调整 FRA 的大小，然后再将其缩小回原始大小。

### 5.3 BACKUP...FORMAT
如果您已配置了 FRA，但不希望 RMAN 备份文件自动放入 FRA，您可以在发出 `BACKUP` 命令时直接指定希望备份放置的位置；例如：

```
RMAN> backup database format '/u01/O18C/rman/rman_%U.bkp';
```

以下是 RMAN 生成的一个对应文件：

```
/u01/O18C/rman/rman_0jnv0557_1_1.bkp
```

其中的 `%U` 指示 RMAN 为备份文件名动态构造一个唯一字符串。在大多数情况下需要唯一的名称，因为 RMAN 不会覆盖已存在的文件。这一点很重要，因为如果您指示 RMAN 并行写入，它需要为每个通道创建唯一的文件名；例如：

```
RMAN> configure device type disk parallelism 2;
```

现在，当您运行 `BACKUP` 命令时，会看到这条消息：

```
RMAN> backup database format '/u01/O18C/rman/rman_%U.bkp';
```

RMAN 分配多个通道并并行写入两个不同的备份文件。格式字符串中的 `%U` 保证创建的文件名是唯一的。

## 6. 设置控制文件的自动备份

你应始终配置 RMAN，使其在以下任一操作后自动备份控制文件：运行任何 RMAN `BACKUP` 或 `COPY` 命令后，或在你对数据库进行物理更改（例如添加/删除数据文件）并导致控制文件更新后。使用 `SHOW` 命令可以显示控制文件自动备份的当前设置：

```
RMAN> show controlfile autobackup;
```

以下是一些示例输出：

```
RMAN configuration parameters for database with db_unique_name O18C are:
CONFIGURE CONTROLFILE AUTOBACKUP ON;
```

下面这行代码展示了如何启用控制文件的自动备份功能：

```
RMAN> configure controlfile autobackup on;
```

控制文件的自动备份总是存储在它自己的备份集中。当启用了控制文件的自动备份时，如果你正在使用 `spfile`，它将随控制文件一起被自动备份。

如果出于任何原因，你希望禁用控制文件的自动备份，可以按如下方式操作：

```
RMAN> configure controlfile autobackup off;
```

注意
如果控制文件的自动备份处于关闭状态，那么每当你备份数据文件 1 (`SYSTEM` 表空间数据文件) 时，RMAN 会自动备份控制文件。

## 7. 指定控制文件自动备份的位置

当你启用控制文件的自动备份时，RMAN 会在以下位置之一创建控制文件的备份：

*   默认位置
*   FRA
*   通过 `CONFIGURE CONTROLFILE AUTOBACKUP FORMAT` 命令指定的位置

如果你未使用 FRA，或者未指定控制文件自动备份的位置，则控制文件自动备份会被写入一个依赖于操作系统的默认位置。在 Linux/Unix 环境中，默认位置是 `ORACLE_HOME/dbs`；例如，

```
/u01/app/oracle/product/18.1.0.1/db_1/dbs/c-3423216220-20130109-01
```

如果你已启用 FRA，那么 RMAN 会使用 OMF 格式作为文件名，自动将控制文件自动备份文件写入 FRA 中的目录；例如，

```
///autobackup//o1_mf_s_729103049_68fho9z2_.bkp
```

控制文件的备份可以放置在数据库备份所在的同一目录中。下面是一个示例：

```
RMAN> configure controlfile autobackup format for device type disk to '/u01/O18C/rman/rman_ctl_%F.bk';
```



如果要将自动备份格式设置回默认值，请按如下方式操作：
```sql
RMAN> configure controlfile autobackup format for device type disk clear ;
```

## 8. 备份归档日志

您应该定期备份归档日志。在至少备份一次之前，不应从磁盘中删除归档日志文件。我通常喜欢保留在上次成功的`RMAN`备份之后生成的所有归档日志。

通常，我会指示`RMAN`在备份数据文件的同时备份归档日志。这在大多数情况下是一个足够的策略。以下是与数据文件一起备份归档日志的命令：
```sql
RMAN> backup database plus archivelog;
```

有时，如果您的数据库生成大量重做日志，您可能需要以与数据文件不同的频率备份归档日志。DBA 可能会每天备份归档日志两到三次；在日志备份后，DBA 会删除它们，以便为更新的归档日志文件腾出空间。

在大多数情况下，您不需要上次成功备份之前生成的任何归档日志。例如，如果某个数据文件经历了介质故障，您需要从备份中恢复该数据文件，然后应用在该数据文件备份期间及之后生成的任何归档日志。

在某些情况下，您可能需要上次备份之前生成的归档日志。例如，您可能遇到介质故障，尝试从上次成功的备份中恢复数据库，但发现该备份已损坏，因此需要从更早的备份中恢复。此时，您需要自该旧备份制作以来生成的所有归档日志的副本。

## 9. 确定快照控制文件的位置

`RMAN`需要控制文件的读一致性视图来完成以下任务：
*   与恢复目录同步
*   备份当前控制文件

`RMAN`会创建当前控制文件的快照副本，在执行这些任务时将其用作读一致性副本。这确保了`RMAN`使用的是一个未被修改的控制文件副本。

快照控制文件的默认位置是操作系统特定的。在 Linux 平台上，默认位置/格式是`ORACLE_HOME/dbs/snapcf_@.f`。请注意，默认位置不在`FRA`中。

您可以使用`SHOW`命令显示当前快照控制文件的详细信息：
```sql
RMAN> show snapshot controlfile name;
```

以下是一些示例输出：
```
CONFIGURE SNAPSHOT CONTROLFILE NAME TO
'/ora01/app/oracle/product/18.1.0.1/db_1/dbs/snapcf_o18c.f'; # default
```

在大多数情况下，快照控制文件的默认位置和格式是足够的。此文件占用空间不大，也没有密集的`I/O`要求。我建议您使用默认设置。

如果您有充分的理由需要将快照控制文件配置到非默认位置，可以按如下方式操作：
```sql
RMAN> configure snapshot controlfile name to '/u01/O18C/rman/snapcf.ctl';
```

如果您不小心将快照控制文件位置配置到了一个不存在的目录，那么在运行`BACKUP`或`COPY`命令时，控制文件的自动备份将失败，并出现此错误：
```
ORA-01580: error creating control backup file ...
```

您可以将快照控制文件设置回默认值，如下所示：
```sql
RMAN> configure snapshot controlfile name clear;
```

## 10. 使用恢复目录

`RMAN`总是将其最新的备份操作存储在目标数据库控制文件中。您可以设置一个可选的恢复目录来存储有关`RMAN`备份的元数据。恢复目录是一个独立的模式（通常位于与目标数据库不同的数据库中），它包含存储`RMAN`备份信息的数据库对象（表、索引等）。恢复目录不存储`RMAN`备份片——只存储备份元数据。

使用恢复目录的主要优点如下：
*   为`RMAN`元数据提供了一个辅助存储库。如果您丢失了所有控制文件及其备份，仍然可以从恢复目录中检索`RMAN`元数据。
*   存储`RMAN`元数据的时间比仅使用控制文件作为存储库可能的时间要长得多。
*   提供对所有`RMAN`功能的访问。使用恢复目录时，某些恢复功能会更简单。

使用恢复目录的缺点是，这是另一个需要您设置、维护和备份的数据库。此外，当您开始备份并尝试连接到恢复目录时，如果恢复目录因任何原因（服务器宕机、网络问题等）不可用，您仍然可以继续进行备份，而不使用恢复目录。

使用恢复目录时，您还必须注意版本问题。您需要确保用于存储恢复目录的数据库版本与目标数据库的版本兼容。升级目标数据库时，请确保（如果需要）恢复目录也已升级。

> **注意**
> 有关如何实现恢复目录的详细信息，请参见第 18 章。

## 11. 使用介质管理器

`RMAN`要直接备份到磁带需要介质管理器。多家供应商提供此功能（需付费）。介质管理器用于大型数据库环境，例如数据仓库，在这些环境中可能没有足够的空间将数据库备份到磁盘。您也可能有直接备份到磁带的灾难恢复要求。

如果您有此类要求，则应购买并实施介质管理软件包。如果不需要直接备份到磁带，则无需实现介质管理器。`RMAN`直接备份到磁盘工作良好。备份文件可以复制到磁带以进行异地存储，并且根据保留策略进行管理。如果复制到磁带，您可以安排一个作业运行`RMAN`备份，然后启动复制到磁带的操作。备份文件需要是完整的，而不是处于备份过程中；否则，存在仅将部分文件复制到磁带的风险。

> **提示**
> 有关如何实现 Oracle Secure Backup 作为介质管理层的详细信息，请参见第 20 章。

## 12. 设置 CONTROL_FILE_RECORD_KEEP_TIME 初始化参数

`CONTROL_FILE_RECORD_KEEP_TIME`初始化参数指定了控制文件中可重用记录在可被覆盖之前保留的最短天数。`RMAN`元数据存储在控制文件的可重用部分中，因此最终会被覆盖。

如果您正在使用恢复目录，那么您不需要担心此参数，因为`RMAN`元数据在恢复目录中是无限期存储的。因此，当您使用恢复目录时，您可以访问任何历史的`RMAN`元数据。

如果您仅使用控制文件作为`RMAN`元数据存储库，那么存储在那里的信息最终将被覆盖。`CONTROL_FILE_RECORD_KEEP_TIME`的默认值为 7 天：
```sql
SQL> show parameter control_file_record_keep_time
NAME                                 TYPE        VALUE
------------------------------------ ----------- --------------------------
control_file_record_keep_time        integer     7
```
