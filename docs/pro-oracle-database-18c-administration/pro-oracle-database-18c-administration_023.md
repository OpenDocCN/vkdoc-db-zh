# 摘要

索引是独立于表的关键对象；它们能大幅提升数据库应用程序的性能。你的索引架构应经过精心规划、实施和维护。要审慎选择为哪些表和哪些列创建索引。尽管索引能极大提升查询速度，但它们也可能拖慢 DML 语句的执行，因为随着表数据的变化，索引也必须进行维护。索引还会消耗磁盘空间。因此，仅应在需要时创建索引。

Oracle 新版本的推出和更多资源的投入，改善了索引或表扫描的性能表现。随着数据库系统的升级和变化，应重新审视旧索引是否仍然需要。SQL 调优中验证索引的步骤至关重要，而使用诸如将索引设为`invisible`之类的工具，有助于在删除索引前验证其是否必要。

Oracle 的 B 树索引是默认索引类型，对大多数应用来说已经足够。然而，你应当了解其他索引类型及其用途。特定功能，如位图索引和基于函数的索引，应在适用场景中加以应用。本章讨论了实施和维护索引的各个方面。表 8-3 总结了本章涵盖的指导原则与技术。

## 表 8-3：创建索引的指导原则总结

| 指导原则 | 原理说明 |
| :--- | :--- |
| 根据需要创建尽可能多的索引，但尽量保持数量最少。明智地添加索引。首先测试以确定可量化的性能提升。 | 索引能提升性能，但也消耗磁盘空间以及 DML 语句的处理资源和时间。不要不必要地添加索引。 |
| 针对表执行的查询所需性能应构成你索引策略的基础。 | 为 SQL 查询中使用的列创建索引最有助于提升性能。 |
| 考虑使用 SQL 调优顾问或 SQL 访问顾问获取索引建议。 | 这些工具能提供建议，并为你的索引决策提供第二双“眼睛”。 |
| 为所有表创建主键约束。 | 这将自动创建一个 B 树索引（如果主键中的列尚未被索引）。 |
| 在适当情况下创建唯一键约束。 | 这将自动创建一个 B 树索引（如果唯一键中的列尚未被索引）。 |
| 为外键列创建索引。 | 在连接表时，外键列通常包含在`WHERE`子句中，从而提升 SQL `SELECT`语句的性能。在外键列上创建 B 树索引也能减少更新和插入子表时的锁问题。 |
| 仔细选择和测试小表（小表指少于几千行）上的索引。 | 即使是小表，索引有时也能比全表扫描表现更好。 |
| 使用正确的索引类型。 | 正确使用索引能最大化性能。详情参见表 8-1。 |
| 如果使用其他索引类型无法带来可验证的性能提升，请使用基本的 B 树索引类型。 | B 树索引适用于大多数具有高基数列值的应用程序。 |
| 考虑在数据仓库环境中使用位图索引。 | 这些索引是低基数列的理想选择，前提是这些列的值不常更新。在星型模式事实表的外键列上，位图索引效果很好，尤其是当经常运行使用`AND`和`OR`连接条件的查询时。 |
| 考虑为索引使用单独的表空间（即与表分开）。 | 表和索引数据可能有不同的存储或备份恢复要求，或两者兼有。使用单独的表空间可以让你独立于表来管理索引。 |
| 让索引从其所在表空间继承存储属性。 | 这使得索引存储更易于管理和维护。 |
| 使用一致的命名标准。 | 这使维护和故障排除更容易。 |
| 除非有充分的理由，否则不要重建索引。 | 重建索引通常是不必要的，除非索引已损坏或不可用，或者你想将索引移动到不同的表空间。 |
| 监视你的索引，并删除那些未使用的索引。 | 这样做可以释放物理空间并提升 DML 语句的性能。 |
| 在删除索引前，考虑将其标记为`invisible`。 | 这使你能够在删除索引前更好地确定是否存在任何性能问题。这些选项让你可以在不需要 DDL 创建语句的情况下重建或重新启用索引。 |

在数据库中创建和管理索引时，请参考这些指导原则。这些建议旨在帮助你正确使用索引技术。

利用自动调优流程和来自 Oracle 数据库的统计信息，将提供关于索引需求和使用情况的信息。结合对索引使用策略的理解，这些工具有助于管理对象并利用索引来辅助查询性能。
