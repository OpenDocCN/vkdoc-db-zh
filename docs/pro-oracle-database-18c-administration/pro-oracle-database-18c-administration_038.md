# RMAN 架构组件与备份类型

## 关键术语

*   **备份集 (Backup set)**：当你运行 RMAN 的 `BACKUP` 命令时，默认会创建一个或多个备份集。备份集是 RMAN 的一种逻辑结构，用于分组备份片文件。你可以将备份集与备份片的关系，类比为表空间与数据文件的关系：一个是逻辑结构，另一个是物理文件。

*   **备份片文件 (Backup piece file)**：RMAN 的二进制备份文件。每个逻辑备份集由一个或多个备份片文件组成。这些是 RMAN 在磁盘或磁带上创建的物理文件。它们是专有的二进制格式文件，只有 RMAN 能够读写。一个备份片可以包含来自许多不同数据文件的块。备份片文件通常比数据文件小，因为备份片只包含数据文件中已使用的块。

*   **映像副本 (Image copy)**：通过 `BACKUP AS COPY` 命令发起。这是一种备份类型，其中 RMAN 创建数据文件、归档日志文件或控制文件的完全相同的副本。映像副本可以被操作系统实用程序（如 Linux 的 `cp` 和 `mv` 命令）操作。映像副本是增量更新映像备份的一部分。有时，如果你需要快速恢复，使用映像副本比备份集更可取。

*   **恢复目录 (Recovery catalog)**：一个可选的数据库模式，包含用于存储 RMAN 备份操作相关元数据信息的表。Oracle 强烈建议使用恢复目录，因为它提供了更多的备份和恢复选项。该目录通常是远程的，不必存在于每个数据库中。

*   **介质管理器 (Media manager)**：允许 RMAN 直接备份文件到磁带的第三方软件。当磁盘空间不足无法直接备份，或者灾难恢复要求必须备份到易于运离现场的存储时，备份到磁带是理想的选择。

*   **快速恢复区 (Fast Recovery Area, FRA)**：RMAN 可用于备份的磁盘区域。你还可以使用 FRA 来多路复用控制文件和在线重做日志。你通过数据库初始化参数 `DB_RECOVERY_FILE_DEST_SIZE` 和 `DB_RECOVERY_FILE_DEST` 来实例化一个快速恢复区。

*   **快照控制文件 (Snapshot control file)**：当备份控制文件或与恢复目录（如果使用）同步时，RMAN 需要控制文件的一致性视图。在这种情况下，RMAN 首先创建控制文件的临时副本（快照）。这允许 RMAN 在备份控制文件或与恢复目录同步期间，使用一个保证不会更改的控制文件版本。

你可以使用 RMAN 制作几种类型的备份：

*   **完全备份 (Full backup)**：备份与数据文件相关的所有已修改块。完全备份并非整个数据库的备份。例如，你可以对一个数据文件进行完全备份。

*   **增量级别 0 备份 (Incremental level 0 backup)**：备份的块与完全备份相同。级别 0 备份与完全备份的唯一区别在于，级别 0 备份可以与其他增量备份一起使用，而完全备份不行。

*   **增量级别 1 备份 (Incremental level 1 backup)**：仅备份自上次备份以来修改过的块。级别 1 增量备份可以是差异备份或累积备份。差异级别 1 备份是默认选项，备份自上次级别 0 或级别 1 备份以来所有修改过的块。累积级别 1 备份则备份自上次级别 0 备份以来所有更改过的块。

*   **增量更新备份 (Incrementally updated backup)**：首先创建数据文件的映像副本，随后的备份是增量备份，并与该映像副本合并。这是使用映像副本进行备份的一种高效方式。使用增量更新备份进行介质恢复速度很快，因为恢复时使用的是数据文件的映像副本。

*   **块更改跟踪 (Block change tracking)**：跟踪数据库中已更改块的数据库功能。已更改块的记录保存在一个二进制文件中。RMAN 可以利用该二进制文件的内容来提高增量备份的性能：无需扫描数据文件中的所有已修改块，RMAN 即可通过二进制块更改跟踪确定哪些块已更改。

*   **归档日志备份 (Archivelog backups)**：此操作备份归档日志并允许释放归档日志目录中的空间。归档日志备份通常作为数据文件备份的一部分，但也可以单独运行以管理归档日志的磁盘空间。

## 启动 RMAN

既然你已经了解了 RMAN 架构组件和可制作的备份类型，现在可以启动 RMAN 并根据你的环境进行配置。

要连接到 RMAN，你需要建立：

*   操作系统环境变量
*   访问有特权的操作系统账户或具有 `SYSBACKUP` 权限的数据库用户

连接到 RMAN 最简单的方法是，以 Oracle 软件所有者（在 Linux/Unix 系统上通常名为 `oracle`）的身份登录到目标数据库所在的服务器。当你以 `oracle` 登录时，在使用 `rman` 和 `sqlplus` 等实用程序之前，需要建立几个操作系统变量。设置这些必需的操作系统变量在第 2 章中有详细说明。RMAN 也可以从安装了 Oracle 软件的另一台服务器运行。连接到目标数据库执行备份需要服务名或 SID 名。

至少需要设置 `ORACLE_HOME` 和 `ORACLE_SID`。此外，如果 `PATH` 变量包含 `ORACLE_HOME/bin` 目录会很方便。该目录包含 Oracle 实用程序。

建立操作系统变量后，可以从操作系统调用 RMAN，如下所示：

```
$ rman target /
或者
$ rman target backupuser@ora18c
```

连接到 RMAN 时，无需指定 `AS SYSDBA` 子句（就像在 SQL*Plus 中以特权用户连接数据库时那样）。这是因为 RMAN 始终要求你以具有 `SYSDBA` 权限的数据库用户身份连接。任何用户都需要被授予 `SYSBACKUP` 角色才能执行备份操作。

**提示**：Oracle Database 12c 新增的 `SYSBACKUP` 权限允许你仅向用户分配执行备份和恢复操作所需的权限。`SYSBACKUP` 权限包含执行此类操作所需的 `SYSDBA` 权限子集。



之前的 RMAN 登录示例使用了操作系统（OS）认证。这类认证意味着，如果您能登录到一个授权的 OS 账户（例如 Oracle 软件的所有者，通常是`oracle`），那么您无需提供用户名和密码即可连接到数据库。您通过为 OS 账户分配特殊组来管理 OS 认证。在 Linux/Unix 环境中安装 Oracle 二进制文件时，需要在安装时指定被授予`SYSDBA`、`SYSOPER`、`SYSBACKUP`数据库特权的 OS 组名称——通常分别是`dba`、`oper`和`backupdba`组（详见章节 1）。作为企业备份解决方案的一部分，建议创建一个单独的备份用户，以便执行到磁盘或磁带的备份并安排自动运行。

## （不）从 SQL*PLUS 调用 RMAN
在 SQL*PLUS 会话中意外尝试 RMAN 命令，或者试图从 SQL*PLUS 中运行 RMAN，这种情况非常典型。然而，这行不通：
```sql
SQL> rman
SP2-0042: unknown command "rman" - rest of line ignored.
```
答案很简单：`rman`客户端是一个 OS 实用程序，而非 SQL*Plus 函数。您必须从 OS 命令行提示符调用`rman`客户端。

## RMAN 架构决策

如果您的数据库启用了归档（关于归档的详细信息，请参见章节 5），您可以开箱即用地使用 RMAN 运行如下命令来备份整个目标数据库：

```
$ rman target /
RMAN> backup database;
```

如果遇到介质故障，您可以按如下方式还原所有数据文件：

```
RMAN> shutdown immediate;
RMAN> startup mount;
RMAN> restore database;
```

数据库还原后，您可以进行完全恢复：

```
RMAN> recover database;
RMAN> alter database open;
```

这样就万事大吉了，对吗？不，不完全如此。对于简单的备份需求，RMAN 的默认属性设置是相当合理的。RMAN 的开箱即用设置可能适用于小型开发或测试数据库。但是，对于任何类型的业务关键型数据库，您需要仔细考虑备份存储的位置、在磁盘或磁带上存储备份的时长、哪些 RMAN 功能适用于该数据库等等。本章接下来的几节将引导您了解在生产环境中实施 RMAN 所需的许多备份和恢复架构决策。RMAN 拥有丰富而强大的选项，可用于自定义备份、管理备份文件以及执行恢复；通常，您并不需要实施 RMAN 的许多功能。然而，每次您实施 RMAN 来备份生产数据库时，都应该仔细考虑每个决策点，并决定是否需要某个属性。

表 17-1 总结了后续章节中的 RMAN 实施内容。许多 DBA 对其中的一些建议会有不同意见；这没关系。关键在于您需要考虑每个架构方面，并确定什么最适合您的业务需求。

### 表 17-1 架构决策与建议概览

| 决策点 | 建议 |
| :--- | :--- |
| 1. 远程还是本地运行 RMAN 客户端 | 在目标数据库服务器上本地运行客户端。 |
| 2. 指定备份用户 | 创建一个用于执行备份的用户。 |
| 3. 使用联机还是脱机备份 | 取决于您的业务需求。大多数生产数据库需要联机备份，这意味着您必须启用归档。 |
| 4. 设置归档日志目标位置和文件格式 | 如果您使用了 FRA（快速恢复区），归档日志将以默认格式写入其中。若要使用`LOG_ARCHIVE_DEST_N`初始化参数明确地将位置设置在 FRA 之外。 |
| 5. 配置 RMAN 备份位置和文件格式 | 取决于您的业务需求。有些环境要求磁带备份。如果使用磁盘，请将备份放在 FRA 中，或通过通道设置指定位置。使用 FRA 可以动态增加存储大小。在创建备份集后可以运行到磁带的备份，这通常效率更高。 |
| 6. 设置控制文件的自动备份 | 始终启用控制文件的自动备份。 |
| 7. 指定控制文件自动备份的位置 | 将其放在 FRA 中，或配置一个位置。将控制文件的自动备份写入与数据库备份相同的位置是合理的。 |
| 8. 备份归档日志 | 取决于您的业务需求。对于许多环境，归档日志的备份可以每天进行一次，使用与备份数据库相同的命令，或者在空间需要时触发。 |
| 9. 确定快照控制文件的位置 | 使用默认位置。 |
| 10. 使用恢复目录 | 取决于您的业务需求。Oracle 建议您使用恢复目录。如果 RMAN 的保留策略时间大于`CONTROL_FILE_RECORD_KEEP_TIME`，那么我建议您使用恢复目录。 |
| 11. 使用介质管理器 | 这是直接备份到磁带所必需的。 |
| 12. 设置`CONTROL_FILE_RECORD_KEEP_TIME`初始化参数 | 通常，默认的 7 天是足够的。 |
| 13. 配置 RMAN 的备份保留策略 | 取决于您的数据库和业务需求。对于许多环境，我使用 1 或 2 的备份保留冗余度。 |
| 14. 配置归档日志的删除策略 | 取决于您的数据库和业务需求。在许多场景中，将备份保留策略应用于归档日志就足够了（这是默认行为）。 |
| 15. 设置并行度 | 取决于可用的硬件资源和业务需求。对于大多数具有多个 CPU 的生产服务器，基于可用的 CPU 数量，配置 2 或更高的并行度。 |
| 16. 使用备份集还是镜像副本 | 备份集通常比镜像副本更小，且更易于管理。 |
| 17. 使用增量备份 | 当大型数据库在两次备份之间只有很小比例的数据发生变化，并且您希望节省磁盘空间时，请使用增量备份。在大型数据库和数据仓库类数据库中使用增量备份。 |
| 18. 使用增量更新备份 | 如果您需要数据文件的镜像副本，请使用此方法。 |
| 19. 使用块更改跟踪 | 使用此功能来提高增量备份的性能。对于大型数据仓库类数据库，块更改跟踪可以显著节省备份时间，因为它跟踪发生更改的数据块，因此备份不必读取块头来确定块是否已更改。 |
| 20. 配置二进制压缩 | 取决于您的业务需求。压缩备份占用空间更少，但在备份和恢复操作中需要更多的 CPU 资源（和时间）。 |
| 21. 配置加密 | 取决于您的业务需求。 |
| 22. 配置其他设置 | 您可以设置许多与通道相关的属性，例如备份集大小和备份片大小。根据需要配置。 |
| 23. 配置信息输出 | 配置 OS 变量`NLS_DATE_FORMAT`以显示日期和时间。使用`SET ECHO ON`和`SHOW ALL`来显示 RMAN 命令和设置。 |

## 1. 远程还是本地运行 RMAN 客户端

可以从远程服务器运行`rman`实用程序，并通过 Oracle Net 连接到目标数据库：

```
$ rman target sys/foo@remote_db
```

这允许您从一个中心位置在不同的远程服务器上运行 RMAN 备份。当您远程运行 RMAN 时，备份文件总是在目标数据库服务器上创建。
