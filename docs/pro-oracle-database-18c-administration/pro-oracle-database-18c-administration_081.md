# 摘要

自动化例行数据库作业是成功 DBA 的关键策略。自动化作业确保任务可重复、可验证，并在出现问题时能快速通知您。例如，作为 DBA，您的工作在很大程度上依赖于成功执行备份和确保数据库的高可用性。本章包含多个脚本和示例，详细说明了如何以定义的频率运行例行作业。

Oracle 提供了 Oracle 调度程序工具（通过`DBMS_SCHEDULER` PL/SQL 包实现）来调度作业。此工具可用于自动化任何类型的数据库任务。您也可以根据系统事件或其他计划作业的成功/失败来启动作业。

如果您是在 Linux/Unix 环境下工作的 DBA，您应该熟悉`cron`工具。这个调度程序简单易用，并且几乎普遍可用。即使您当前的任务不使用`cron`，在将来的工作环境中您肯定会遇到它的使用。

这些脚本大多只是检查和告警，下一步是如果发现问题需要处理或修复时，实际自动化一个操作。纠正操作可能是解锁账户或为归档日志空间向 FRA 添加空间，以便立即处理问题，并留出时间审查和验证该操作。

至此，在本书中，您已经学习了如何实现和执行 DBA 所需的许多任务。凭借这些知识和理解，数据库变得更加熟悉，可能现在就在列出需要自动化的任务。这些知识将延续到云迁移和云端数据库管理中。即使您只管理一个数据库，毫无疑问，您也陷入了大量的故障排除活动中。下一章重点介绍诊断和解决 DBA 遇到的许多问题。

## 21. 数据库故障排除

`数据库故障排除`是一个模糊且通用的术语，适用于广泛的主题。它可以指从调查数据库连接问题到详细的性能调优等任何事情。在本章中，将涵盖以下故障排除活动：

*   快速评估数据库可用性问题
*   使用操作系统工具识别系统性能问题
*   查询数据字典视图以显示资源密集型 SQL 语句
*   使用 Oracle 性能工具识别消耗资源的 SQL 语句
*   识别和解决锁问题
*   排除打开游标问题的故障
*   调查撤销和临时表空间的问题

前面的列表并未涵盖您将遇到的所有类型的数据库故障排除和性能问题。相反，它是一个您可能遇到的数据库问题的抽样，并展示了解决问题的有用技术。

### 快速分类处理

当接到数据库出现问题的电话时，能够提问并知道该问什么问题是至关重要的。理解问题是第一步，并且必须快速完成，以便进行其他故障排除步骤。DBA 将被要求排查数据库和非数据库问题、服务器、连接和网络问题，因为这些都是数据库系统的一部分，可能只看到的是数据返回不够快或根本没有返回。

以下是一些有用的问题：

*   这是应用程序的问题还是直接连接到数据库的问题？
*   这是一个新的进程、查询、代码段吗？
*   这种情况变慢多久了？以前发生过吗，还是第一次？
*   有返回任何数据吗？
*   您收到任何错误消息吗？

当答案陆续传来时，您可以检查告警日志、例行作业的脚本输出、ping 数据库和数据库服务器，以及看看您是否能够登录。这应该能为您开始故障排除问题提供一个良好的起点。

**提示**
请记住，您应该自动化执行诸如验证数据库可用性等任务的作业（有关自动化 DBA 任务的示例，请参见第 19 章）。自动化作业帮助您主动处理问题，以免它们演变成数据库停机。

### 检查数据库可用性

最初的几次检查不需要登录到数据库服务器。相反，它们可以通过 SQL*Plus 和操作系统命令远程执行。通过网络远程执行初始检查可以确定所有系统组件是否正常工作。

一个快速检查远程服务器是否可用、数据库是否启动、网络是否正常以及监听器是否接受传入连接的方法是，通过 SQL*Plus 客户端通过网络连接到远程数据库。初始测试可以使用非 DBA 账户。以下是一个通过网络连接到远程数据库的示例，使用用户`michelle`和密码`ora123`；网络连接信息直接嵌入到连接字符串中（其中`dwdb1`是服务器，1521 是端口，`dwrep1`是数据库服务名）：

```bash
$ sqlplus michelle/ora123@'dwdb1:1521/dwrep1'
```

如果可以建立连接，那么远程服务器是可用的，数据库和监听器已启动并正在工作。此时，结合之前问的问题，我们可以验证连接问题是否与应用程序或数据库以外的其他因素有关。

如果前面的`sqlplus`命令不起作用，请尝试确定远程服务器是否可用。在此示例中，向名为`dwdb1`的远程服务器发出`ping`命令：

```bash
$ ping dwdb1
```

如果`ping`成功，您应该看到类似这样的输出：

```bash
64 bytes from dwdb1 (192.168.254.215): icmp_seq=1 ttl=64 time=0.044 ms
```

如果`ping`不起作用，可能是网络或远程服务器有问题。如果远程服务器不可用，是时候联系系统管理员或网络管理员了。

如果`ping`确实有效，那么请检查是否可以通过监听器监听的端口访问远程服务器。使用`telnet`命令来完成此操作：

```bash
$ telnet IP
```

在此示例中，尝试连接到服务器 IP 地址的 1521 端口：

```bash
$ telnet 192.168.254.215 1521
```

如果可以在指定端口访问 IP 地址，您应该在输出中看到“Connected to ...”，如下所示：

```bash
Trying 192.168.254.216...
Connected to ora04.
Escape character is '^]'.
```

如果`telnet`命令不起作用，请联系系统管理员或网络管理员。

如果`telnet`命令有效，并且通过指定端口到服务器存在网络连接，那么使用`tnsping`命令测试使用 Oracle Net 到远程服务器和数据库的网络连接。此示例尝试连接到`DWREP1`远程服务：

```bash
$ tnsping DWREP1
```

如果成功，输出应包含`OK`字符串，如下所示：

```bash
Attempting to contact (DESCRIPTION = (ADDRESS = (PROTOCOL = TCP)(HOST = DWDB1)(PORT = 1521)) (CONNECT_DATA = (SERVICE_NAME = DWREP1)))
OK (20 msec)
```

如果`tnsping`有效，意味着远程监听器已启动并正在工作。这并不一定意味着数据库已启动，因此您可能需要登录到数据库服务器进行进一步调查。如果`tnsping`不起作用，则监听器或数据库已关闭或挂起。

要进一步调查问题，请直接登录到服务器执行其他检查，例如挂载点已满。希望如果有其他问题，其中一个日志或监控脚本已经报告了；但是，此时可能需要在服务器上进行检查。
