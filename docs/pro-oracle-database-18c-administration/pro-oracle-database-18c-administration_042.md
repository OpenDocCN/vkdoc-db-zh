# 15. 设置并行度

如果你的数据库服务器配备了支持多通道的硬件，你可以显著提高 RMAN 备份和恢复操作的性能。如果你的服务器有多个 CPU 和多个存储设备（磁盘或磁带设备），那么可以通过启用多个备份通道来提高性能。
如果你需要从备份和恢复操作中获得更好的性能，并且拥有支持并行操作的硬件，那么你应该启用并行度并进行测试以确定最佳程度。如果你的硬件可以利用并行 RMAN 通道，启用并行度几乎没有缺点。

如果你有多个 CPU，但只有一个存储设备位置，你可以启用多个通道向一个位置写入和读取。例如，如果你备份到 FRA，仍然可以通过启用并行度来利用多个通道。假设你的服务器上有四个 CPU，并希望启用相应的并行度：

```
RMAN> configure device type disk parallelism 4;
```

你还可以通过配置与不同挂载点关联的多个通道来并行写入单独的位置；例如，

```
RMAN> configure device type disk parallelism 4;
RMAN> configure channel 1 device type disk format '/u01/O18C/rman/rman1_%U.bk';
RMAN> configure channel 2 device type disk format '/u02/O18C/rman/rman2_%U.bk';
RMAN> configure channel 3 device type disk format '/u03/O18C/rman/rman3_%U.bk';
RMAN> configure channel 4 device type disk format '/u04/O18C/rman/rman4_%U.bk';
```


此代码配置了四个通道，分别写入磁盘上的不同位置。当为不同位置配置单独通道时，请确保启用并行度以匹配已配置的设备通道数量。如果分配的通道数超过指定的并行度，RMAN 将仅向并行度指定数量的通道写入数据，并忽略其他通道。

如果需要清除并行度，可以按如下方式操作：
```
RMAN> configure device type disk clear;
```

类似地，要清除通道设备类型，请使用 `CLEAR` 命令。此示例清除通道 4：
```
RMAN> configure channel 4 device type disk clear;
```

## 16. 使用备份集或映像副本

当你发出 RMAN `BACKUP` 命令时，可以指定备份为以下类型之一：
*   备份集
*   映像副本

备份集是默认的 RMAN 备份类型。一个备份集包含备份片段，这些是只有 RMAN 才能写入或读取的二进制文件。备份集通常是可取的，因为它们通常比正在备份的数据文件要小。RMAN 会自动尝试创建带有未使用块压缩的备份片段。在此模式下，RMAN 读取位图以确定哪些块已分配，并且只从数据文件中读取这些块。此功能仅适用于基于磁盘的备份集和 Oracle Secure Backup 磁带备份。

注意：RMAN 也可以使用真正的二进制压缩来创建备份集。这是你从操作系统压缩实用程序（如 `zip`）获得的那种压缩。Oracle 支持多个级别的二进制压缩。`BASIC` 压缩算法无需额外许可即可使用。Oracle 通过 Oracle Advanced Compression 选项提供了进一步的压缩功能（有关如何启用二进制压缩的详细信息，请参阅本章后面的“配置二进制压缩”部分）。

当你创建备份集形式的备份时，二进制备份片段文件只能由 RMAN 进程操作。一些数据库管理员认为这是一个缺点，因为他们必须使用 RMAN 来备份和恢复这些文件（你无法直接访问或控制备份片段）。但这些看法是没有根据的。除非你遇到一个罕见的错误，否则 RMAN 是可靠的，并且在所有备份和恢复情况下都能可靠地工作。

将备份集与映像副本进行对比。映像副本为每个数据文件创建逐字节相同的副本。创建映像副本的优点是（如有必要）你可以不使用 RMAN 而操作映像副本（如使用操作系统复制实用程序）。此外，发生介质故障时，映像副本是恢复数据文件的快速方法，因为 RMAN 只需将文件从备份位置复制回来（无需重构数据文件，因为它是一个精确副本）。

备份到磁盘的大小几乎总是需要关注的问题。备份集在磁盘空间消耗方面更高效。因为备份集可以利用 RMAN 压缩，与映像副本相比，涉及的 I/O 也更少。在许多环境中，减少 I/O 以免影响其他应用程序是一个需要考虑的问题。但是，如果你觉得需要直接控制 RMAN 创建的备份文件，或者你所处的环境中恢复过程的速度至关重要，请考虑使用映像副本。

## 17. 使用增量备份

增量备份策略适用于大型数据库，其中只有一小部分数据库块在从一次备份到下一次备份之间发生变化。如果你处于数据仓库环境中，可能需要考虑增量备份策略，因为它可以大大减小备份的大小。例如，你可能希望运行每周一次的 0 级备份，然后运行每日一次的 1 级增量备份。

术语 `RMAN level 0 incremental backup` 本身描述得也不是很确切。0 级增量备份备份的块与完全备份相同。换句话说，以下两个命令备份数据库中的相同块：
```
RMAN> backup as backupset full database;
RMAN> backup as backupset incremental level=0 database;
```

之前两个命令之间的唯一区别是，增量 0 级备份可以与其他增量备份结合使用，而完全备份不能参与增量备份策略。因此，我几乎总是更喜欢使用 `INCREMENTAL LEVEL=0` 语法（而不是完全备份）；它让我可以灵活地将 0 级增量备份与后续的 1 级增量备份一起使用。

## 18. 使用增量更新备份

增量更新备份是实现映像副本备份策略的一种高效方法。此技术指示 RMAN 首先创建数据文件的映像副本；然后，下次备份运行时，RMAN 不是创建一组新的映像副本，而是进行一次增量备份（自映像副本创建以来块的更改），并将该增量备份应用到映像副本上。

如果你有足够的磁盘空间存放数据库的完整映像副本，并且希望在发生介质故障时能够直接使用映像副本的灵活性，请考虑此备份策略。

这种方法的一个潜在缺点是，如果要求你恢复并恢复到过去的某个时间点，你只能恢复并恢复到映像副本上次使用增量备份更新的时间点。

## 19. 使用块更改跟踪

此功能跟踪数据库块何时发生更改。其思想是，如果你正在使用增量备份策略，则可以提升性能，因为通过实施此功能，RMAN 无需扫描数据文件中的每个块（在高水位线下）以确定是否需要备份。相反，RMAN 只需访问块更改跟踪文件，即可找出自上次备份以来哪些块已更改，并直接访问这些块。如果你在大型数据仓库环境中工作并且正在使用增量备份策略，请考虑启用块更改跟踪以提升性能。

注意：有关实施增量和块更改备份及恢复的更多详细信息，请参见第 18 章。

## 20. 配置二进制压缩

你可以配置 RMAN 在生成备份集时使用真正的二进制压缩。你可以通过以下两种方式之一启用压缩：
*   在 `BACKUP` 命令中指定 `AS COMPRESSED BACKUPSET`。
*   使用一次性的 `CONFIGURE` 命令。

以下是在发出 `BACKUP` 命令时使用压缩进行备份的示例：
```
RMAN> backup as compressed backupset database;
```

此示例为磁盘设备配置了压缩：
```
RMAN> configure device type disk backup type to compressed backupset;
```

如果需要清除设备类型压缩，请发出此命令：
```
RMAN> configure device type disk clear;
```

事实证明，默认的压缩算法非常高效。对于典型的数据库，备份通常比常规备份小大约四到五倍。当然，你的压缩结果可能会有所不同，具体取决于你的数据。

为什么不压缩所有备份？压缩备份消耗更多 CPU 资源，并且创建和从中恢复所需时间更长，但它们导致的 I/O 更少，并且分布在更长的时间段内。如果你有多个 CPU，并且制作备份的速度不是问题，那么你应该考虑压缩你的备份。

你可以使用 `SHOW` 命令查看启用的压缩类型：
```
RMAN> show compression algorithm;
```

以下是一些示例输出：
```
CONFIGURE COMPRESSION ALGORITHM 'BASIC' AS OF RELEASE 'DEFAULT'
OPTIMIZE FOR LOAD TRUE ; # default
```
