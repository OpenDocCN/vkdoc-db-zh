# 从决策到行动

既然您已经很好地理解了在实施 RMAN 之前应该做出哪些类型的决策，那么查看一个实现了其中一些组件的脚本是很有指导意义的。脚本可用于自动化 RMAN 备份。这些 shell 脚本通过诸如 `cron` 之类的调度工具实现自动化。RMAN 也可以从 Oracle 企业管理器运行。此外，如果有用于将文件复制到磁带或其他调度程序的工具，RMAN 脚本可以包含在这些作业中；但是，如果 RMAN 脚本没有保留在数据库端，灵活性就会降低。

本节包含一个典型的用于 RMAN 备份的 shell 脚本。该脚本在输出中带有行号，以便在讨论编写脚本时所做的架构决策时参考。（如果您复制该脚本，请在运行前删除行号。）


以下是脚本内容。表 17-2 详细列出了本章涵盖的每个 RMAN 架构决策点、它在 shell 脚本中的实现（或未实现）方式，以及在 shell 脚本中对应的行号。该脚本并未涵盖 RMAN 使用的方方面面。如果您要使用此脚本，请务必根据您自身环境的需求和 RMAN 标准对其进行修改：

## 表 17-2 架构决策的实现

```bash
1 #!/bin/bash
2 HOLDSID=${1}  # SID 名称
3 PRG=`basename $0`
4 USAGE="用法: ${PRG}  "
5 if [ -z "${HOLDSID}" ]; then
6    echo "${USAGE}"
7    exit 1
8 fi
9 #----------------------------------------------
10 # 源环境变量（关于 oraset 的详细信息请参见第 2 章）
11 . /etc/oraset $HOLDSID
12 BOX=`uname -a | awk '{print$2}'`
13 MAILX='/bin/mailx'
14 MAIL_LIST='dkuhn@gmail.com'
15 export NLS_DATE_FORMAT='dd-mon-yyyy hh24:mi:ss'
16 date
17 #----------------------------------------------
18 LOCKFILE=/tmp/$PRG.lock
19 if [ -f $LOCKFILE ]; then
20   echo "锁文件已存在，退出..."
21   exit 1
22 else
23   echo "请勿删除，$LOCKFILE" > $LOCKFILE
24 fi
25 #----------------------------------------------
26 rman nocatalog <<EOF
27 connect target /
28 set echo on;
29 show all;
30 crosscheck backup;
31 crosscheck copy;
32 configure controlfile autobackup on;
33 configure controlfile autobackup format for device type disk to
'/u01/O18C/rman/o18c_ctl_%F.bk';
34 configure retention policy to redundancy 1;
35 configure           device type disk parallelism 2;
36 configure channel 1 device type disk format '/u01/O18C/rman/o18c_%U.bk';
37 configure channel 2 device type disk format '/u02/O18C/rman/o18c_%U.bk';
38 backup as compressed backupset incremental level=0 database plus archivelog;
39 delete noprompt obsolete;
40 EOF
41 #----------------------------------------------
42 if [ $? -ne 0 ]; then
43   echo "RMAN 出现问题..."
44   echo "请检查 RMAN 备份" | $MAILX -s "RMAN 问题: $ORACLE_SID 于 $BOX" $MAIL_LIST
45 else
46   echo "RMAN 运行正常..."
47 fi
48 #----------------------------------------------
49 if [ -f $LOCKFILE ]; then
50   rm $LOCKFILE
51 fi
52 #----------------------------------------------
53 date
54 exit 0
```



关于此脚本，有几个方面需要进一步讨论。

## 设置操作系统变量

第 11 行通过运行名为`oraset`的脚本来设置所需的 OS 变量（有关运行`oraset`和获取操作系统变量的详细信息，请参见第 2 章）。许多 DBA 选择将 OS 变量（如`ORACLE_HOME`和`ORACLE_SID`）硬编码到脚本中。但是，您应该避免硬编码变量，而应使用脚本来获取所需的变量。运行脚本要灵活得多，尤其是在一台服务器上安装了多个不同版本 Oracle 的数据库时。

## 设置 `NLS_DATE_FORMAT`

第 15 行设置了 `NLS_DATE_FORMAT` OS 变量。这在调试和诊断问题时特别有用。默认情况下，RMAN 只显示日期部分。仅知道命令运行的日期通常不足以确定命令实际执行的时间点。至少，您需要看到小时和分钟（连同日期）。

## 使用锁文件

第 18 至 24 行检查锁文件是否存在。如果脚本已经在运行，您不希望再次执行它。脚本会检查锁文件，如果存在，则脚本退出。备份完成后，锁文件会被删除（第 49 至 51 行）。

## 配置 RMAN 以进行调试

第 28 行将 `ECHO` 参数设置为 `on`。这指示 RMAN 在运行命令前在输出中显示该命令。这对于调试问题可能非常宝贵。第 29 行显示所有可配置变量。这对于故障排除也很有用，因为您可以在执行任何命令之前查看 RMAN 变量被设置为什么值。

## `CONFIGURE` 命令的作用

第 32 至 37 行使用了 `CONFIGURE` 命令。这些命令在每次脚本执行时都会运行。为什么要这样做？`CONFIGURE` 命令只需要运行一次，它就会存储在控制文件中——您不必再次运行它，对吧？没错。然而，我曾偶尔因此吃亏：当一个有不良习惯的 DBA 为数据库配置了某个设置却没有告诉任何人，而我直到尝试进行另一次备份时才发现这个错误配置。我强烈倾向于将 `CONFIGURE` 命令放在脚本中，这样无论其他 DBA 在脚本之外做了什么，行为都能保持一致。脚本中的 `CONFIGURE` 设置也充当了一种文档形式：我可以轻松查看脚本并确定设置是如何配置的。

## 运行 `CROSSCHECK` 命令

第 30 和 31 行运行了 `CROSSCHECK` 命令。为什么要这样做？有时，文件会丢失，或者某个“不走寻常路”的 DBA 可能通过 RMAN 之外的操作系统命令从磁盘上删除了归档日志文件。当 RMAN 运行时，如果它找不到它认为应该存在的文件，就会抛出错误并停止备份。我更倾向于运行 `CROSSCHECK` 命令，让 RMAN 调和它认为应该在磁盘上的文件与实际在磁盘上的文件。这能保持 RMAN 顺畅运行。

## 删除过时的备份

您在第 39 行运行 `DELETE NOPROMPT OBSOLETE`。这将删除所有被 RMAN 根据保留策略标记为 `OBSOLETE` 的备份文件和归档日志文件。这使得 RMAN 可以管理哪些文件应该保留在磁盘上。我倾向于在备份完成后运行 `DELETE` 命令（而不是在备份之前运行）。保留策略定义为 1，因此如果您在备份后运行 `DELETE`，RMAN 会在磁盘上保留一份备份副本。如果您在备份前运行 `DELETE`，RMAN 会在磁盘上保留一份备份副本。备份运行后，磁盘上会有两份备份副本，而我的服务器上没有足够空间存放。

## 使用 `cron` 进行调度

您可以从 Linux/Unix 的调度工具 `cron` 执行此 shell 脚本，如下所示：

```
0 16 * * * $HOME/bin/rmanback.bsh INVPRD >$HOME/bin/log/INVPRDRMAN.log 2>&1
```

该脚本在数据库服务器上每天 16:00（24 小时制）运行。会创建一个日志文件 (`INVPRDRMAN.log`) 来捕获与 RMAN 作业相关的所有输出和错误。有关通过 `cron` 自动化作业的详细信息，请参见第 21 章。

再次说明，本节中的脚本是基础版；您无疑会希望根据您的需求对其进行增强和修改。此脚本为您提供了一个起点，包含了具体的 RMAN 建议及其实现方法。

## 小结

RMAN 不仅是 Oracle 数据库的备份工具，还是恢复管理器。如果您仍在使用较旧的、用户管理的备份技术，那么我强烈建议您切换到 RMAN。RMAN 包含一组强大的功能，是其他任何可用的备份工具都无法比拟的。RMAN 易于使用和配置。在实施一个坚如磐石的恢复和备份策略时，它将为您节省时间和精力，并让您高枕无忧。

如果您是 RMAN 的新手，可能并不清楚哪些功能应该始终启用和实施，同样，哪些方面您很少需要。本章包含一个清单，引导您完成每个架构决策点。许多策略需要进行讨论，以确保满足业务需求。关键在于，您应该仔细考虑每个组件以及如何实现有意义的各项功能。
