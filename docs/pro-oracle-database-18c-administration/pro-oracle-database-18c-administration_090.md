# 摘要

一名高级数据库管理员（DBA）必须擅长高效地确定数据库不可用和性能问题的根源。识别并解决问题的能力定义了专业级别的 DBA。任何人都可以谷歌一个主题（最糟糕的事情莫过于与一位正在谷歌并推荐随机解决方案的经理一起参加故障排除电话会议）。除了确认正在谷歌的是正确的 Oracle 版本外，找到合适的解决方案并自信地将其应用到生产数据库环境中，才是你创造巨大价值的方式。

诊断问题有时需要一些系统和网络管理员的技能。此外，一名高效的 DBA 必须知道如何利用 Oracle 数据字典来识别问题。作为策略的一部分，你还应该主动监控导致数据库不可用的常见原因。理想情况下，你应该在其他人发现问题之前就察觉到，并主动解决问题。

没有一本书能涵盖所有的故障排除活动。本章包含了一些最常用的识别和处理问题的技巧。通常，基本的操作系统实用程序将帮助你确定挂起数据库的根源。在几乎所有情况下，都应该检查 `alert.log` 和相应的跟踪文件。找到问题的根本原因往往是最困难的任务。采用一致且有条理的方法，你将在诊断和解决问题方面成功得多。

本章旨在传授一些技巧和方法，帮助你在最混乱的数据库环境中也能生存下来。总结这些想法，这可以说是一份 DBA 的宣言：

*   通过脚本和调度程序实现自动化和监控。成为第一个知道系统出问题的人。
*   在流程和脚本中力求可重复性和高效。保持一致性。
*   保持简单。如果一个模块超过一页长，那就太长了。不要编写其他 DBA 无法理解或维护的脚本或功能。有时，简单的解决方案就是正确的解决方案。
*   无论灾难如何，保持冷静。保持尊重。
*   不要害怕寻求或采纳建议。欢迎反馈和批评。倾听他人。接受自己可能是错的这一想法。
*   利用图形化工具，但始终要懂得如何手动实现功能。
*   预期故障，预测故障，为故障做准备。你不知道什么会出问题，但你知道一定会出问题。为你已为故障做好准备而感到高兴吧。最好的教训往往是痛苦的。
*   测试并记录你的操作流程、备份和恢复步骤。这将帮助你在压力巨大的数据库宕机情况下保持冷静（相对而言）并专注于问题。
*   不要为实现数据库供应商已提供解决方案（如复制、灾难恢复、备份与恢复等）的功能而编写代码。
*   熟练掌握 SQL、过程化 SQL 和操作系统命令。这些技能区分了弱者与强者。最优秀的 DBA 兼具系统管理员（SA）和开发者的专长。
*   不断研究新功能和新技术。学习是一个永无止境的过程。质疑一切，重新评估，并寻找更好的方法。通过可重复的、同行评审的测试来验证你的解决方案。记录并自由分享你的知识。
*   知道该问什么问题，以便更好地理解问题或情况。
*   寻求协助数据库迁移上云、迁移到合适的数据库以及利用自治数据库流程。
*   全力以赴完成任务。你现在是在与全世界竞争。更努力、更聪明地工作。

DBA 的工作可能非常有回报，但也可能非常痛苦和充满压力。希望本书记录的技巧能帮你从压力重重的状态转变为偶尔快乐的状态。
