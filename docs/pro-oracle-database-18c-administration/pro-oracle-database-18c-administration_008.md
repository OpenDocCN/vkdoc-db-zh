# 数据库 vs. 实例

虽然数据库管理员（DBA）经常将 `database` 和 `instance` 这两个术语互换使用，但它们指的是架构中截然不同的组成部分。在 Oracle 中，`database` 一词指的是构成数据库的物理文件：数据文件、在线重做日志文件和控制文件。而 `instance` 一词则指代后台进程和内存结构。

例如，你可以先创建一个实例而无需存在数据库。在物理创建数据库之前，你必须使用 `STARTUP NOMOUNT` 语句启动实例。在此状态下，你拥有后台进程和内存结构，但没有任何关联的数据文件、在线重做日志或控制文件。数据库文件直到你发出 `CREATE DATABASE` 语句后才会创建。

另一个需要记住的重点是，一个实例只能关联一个数据库，而一个数据库可以关联许多不同的实例（如 Oracle Real Application Clusters [RAC]）。实例只能挂载并打开一个数据库一次。每次你停止并启动一个数据库时，都会有一个新的实例与之关联。先前创建的后台进程和内存结构永远不会与一个数据库关联。

为了演示这个概念，使用 `ALTER DATABASE CLOSE` 语句关闭一个数据库：

```
SQL> alter database close;
```

如果你尝试重新打开数据库，会收到错误：

```
SQL> alter database open;
ERROR at line 1:
ORA-16196: database has been previously opened and closed
```

这是因为一个实例只能挂载并打开一个数据库。你必须停止并启动一个新实例，然后才能挂载并打开该数据库。

## 使用 dbca 创建数据库

你也可以使用 `dbca` 实用程序来创建数据库。该实用程序有两种模式：图形模式和静默模式。要使用图形模式的 `dbca`，请确保已安装合适的 X 软件，然后发出 `xhost +` 命令，并确认你的 `DISPLAY` 变量已设置；例如，

```
$ xhost +
$ echo $DISPLAY
:0.0
```

要以图形模式运行 `dbca`，请在操作系统命令行中输入 `dbca`：

```
$ dbca
```

图形模式非常直观，它会引导你完成创建数据库的所有方面。如果你是 Oracle 新手并希望有明确的选项提示，可能会更喜欢使用此模式。

你也可以使用响应文件以静默模式运行 `dbca`。在某些情况下，使用图形模式的 `dbca` 并不可行。这可能是由于网络缓慢或缺少 X 软件造成的。要使用静默模式的 `dbca` 创建数据库，请执行以下步骤：

1.  找到 `dbca.rsp` 文件。
2.  复制一份 `dbca.rsp` 文件。
3.  修改这份 `dbca.rsp` 文件副本以适应你的环境。
4.  以静默模式运行 `dbca` 实用程序。

首先，导航到你复制 Oracle 数据库安装软件的位置，并使用 `find` 命令定位 `dbca.rsp`：

```
$ find . -name dbca.rsp
./18.0.0.0/database/response/dbca.rsp
```

复制文件，这样你就不会修改原始文件（这样，你始终会有一个完好的原始文件）：

```
$ cp dbca.rsp mydb.rsp
```

现在，编辑 `mydb.rsp` 文件。至少，你需要修改以下参数：`GDBNAME`、`SID`、`SYSPASSWORD`、`SYSTEMPASSWORD`、`SYSMANPASSWORD`、`DBSNMPPASSWORD`、`DATAFILEDESTINATION`、`STORAGETYPE`、`CHARACTERSET` 和 `NATIONALCHARACTERSET`。以下是 `mydb.rsp` 文件中修改后的值示例：

```
[CREATEDATABASE]
GDBNAME = "O18DEV"
SID = "O18DEV"
TEMPLATENAME = "General_Purpose.dbc"
SYSPASSWORD = "foo"
SYSTEMPASSWORD = "foo"
SYSMANPASSWORD = "foo"
DBSNMPPASSWORD = "foo"
DATAFILEDESTINATION ="/u01/dbfile"
STORAGETYPE="FS"
CHARACTERSET = "AL32UTF8"
NATIONALCHARACTERSET= "UTF8"
```

接下来，使用响应文件以静默模式运行 `dbca` 实用程序：

```
$ dbca -silent -responseFile /home/oracle/orainst/mydb.rsp
```

你应该会看到类似以下的输出：

```
Copying database files
1% complete
...
Creating and starting Oracle instance
...
62% complete
Completing Database Creation
...
100% complete
Look at the log file ... for further details.
```

如果你查看日志文件，请注意 `dbca` 实用程序使用 `rman` 实用程序来恢复数据库所需的数据文件。然后，它会创建实例并执行安装后步骤。在 Linux 服务器上，你还应该在 `/etc/oratab` 文件中为你的新数据库有一个条目。

许多 DBA 会启动 `dbca` 并在图形模式下配置数据库，但只有少数人利用响应文件提供的选项。通过有效利用响应文件，你可以持续自动化数据库创建过程。你可以修改响应文件以在 ASM 上构建数据库，甚至创建 RAC 数据库。此外，你几乎可以控制响应文件的每个方面，类似于在图形模式下启动 `dbca`。

**提示**
你可以通过 help 参数查看 `dbca` 的所有选项：`dbca -help`

## 使用 dbca 生成创建数据库语句

你可以使用 `dbca` 实用程序来生成 `CREATE DATABASE` 语句。你可以通过图形界面交互式地执行此操作，也可以通过静默模式执行。关键在于选择“自定义数据库模板”并同时指定“生成数据库创建脚本”选项。此示例使用静默模式生成一个包含 `CREATE DATABASE` 语句的脚本：

```
$ dbca -silent -generateScripts -customCreate -templateName New_Database.dbt \
-gdbName DKDEV
```

上面的代码指示 `dbca` 创建一个名为 `CreateDB.sql` 的脚本，并将其放置在 `ORACLE_BASE/admin/DKDEV/scripts` 目录中。`CreateDB.sql` 文件中包含一个 `CREATE DATABASE` 语句。同时还会创建一个用于初始化实例的 `init.ora` 文件。

在此示例中，创建数据库所需的脚本已为你生成。直到你手动运行这些脚本，才会实际创建数据库。

此技术为你提供了一种生成 `CREATE DATABASE` 语句的自动化方法。如果你是 Oracle 新手，不确定如何构造 `CREATE DATABASE` 语句，或者你使用的是数据库的新版本并希望由 Oracle 实用程序生成一个有效的 `CREATE DATABASE` 语句，这将特别有用。

## 删除数据库

如果你有一个不再使用的数据库需要删除，可以使用 `DROP DATABASE` 语句来完成此操作。这样做会移除与该数据库关联的所有数据文件、控制文件和在线重做日志。

不用说，在删除数据库时要极其谨慎。在删除数据库之前，请确保你在正确的服务器上，并且连接到正确的数据库。在 Linux/Unix 系统上，在操作系统提示符下发出以下 OS 命令：

```
$ uname -a
```

接下来，连接到 SQL*Plus，并确保你连接到要删除的数据库：

```
SQL> select name from v$database;
```

验证你处于正确的数据库环境后，从具有 `SYSDBA` 权限的账户发出以下 SQL 命令：

```
SQ> shutdown immediate;
SQL> startup mount exclusive restrict;
SQL> drop database;
```

**注意**
显然，在删除数据库时要非常小心。删除数据库时不会出现提示，并且在撰写本文时，没有 `UNDROP ACCIDENTALLY DROPPED DATABASE`（意外删除数据库后撤消删除）命令。删除数据库时要极其谨慎，因为此操作会移除数据文件、控制文件和在线重做日志文件。

`DROP DATABASE` 命令在你需要移除一个数据库时非常有用。它可能是一个测试数据库或一个不再使用的旧数据库。`DROP DATABASE` 命令不会移除旧的归档重做日志文件。你必须使用操作系统命令（例如 Linux/Unix 中的 `rm`，或 Windows 命令提示符下的 `del`）手动删除这些文件。你也可以指示 RMAN 删除归档重做日志文件。

## 一个服务器上有多少个数据库？



有时，在创建新数据库时，会遇到这样一个问题：应该在一台服务器上放置多少个数据库？一种极端情况是每个数据库服务器只运行一个数据库。图 2-3 展示了这种架构，其中显示了两台不同的数据库服务器，每台服务器都有自己的`Oracle`二进制文件安装。这种设置对硬件供应商有利，但在许多环境中并非资源的经济利用方式。

**图 2-3** 一个服务器对应一个数据库的架构

如果您有足够的内存、中央处理器（CPU）和磁盘资源，那么应考虑在一台服务器上创建多个数据库。您可以为每个数据库创建一个新的`Oracle`二进制文件安装，也可以让多个数据库共享一套`Oracle`二进制文件。图 2-4 展示了一台服务器上多个数据库共享一套`Oracle`二进制文件的配置。当然，如果您有使用不同版本`Oracle`二进制文件的需求，您就必须拥有多个`Oracle 主目录`来容纳这些安装。

![../images/214899_3_En_2_Chapter/214899_3_En_2_Fig4_HTML.jpg](img/214899_3_En_2_Fig4_HTML.jpg)

**图 2-4** 一台服务器上的多个数据库共享一套`Oracle`二进制文件

如果您没有足够的 CPU、内存或磁盘资源来在一台服务器上创建多个数据库，可以考虑使用一个数据库来托管多个应用程序和用户，如图 2-5 所示。在此类环境中，请注意不要使用公共同义词，因为应用程序之间可能会发生冲突。在这种环境中，通常会为不同的应用程序创建不同的方案和表空间。

![../images/214899_3_En_2_Chapter/214899_3_En_2_Fig5_HTML.jpg](img/214899_3_En_2_Fig5_HTML.jpg)

**图 2-5** 一个数据库被多个应用程序和用户使用

使用`Oracle 数据库 18c`，您可以选择使用`可插拔数据库`功能。该技术允许您在一个`容器数据库`中容纳多个`可插拔数据库`。这些`可插拔数据库`共享实例、后台进程、`撤销`和`Oracle`二进制文件，但作为完全独立的数据库运行。每个`可插拔数据库`都有自己的一组表空间（包括`SYSTEM`表空间），这些表空间对于同一`容器数据库`内的任何其他`可插拔数据库`都不可见。这使您能够安全地实现一个与其他数据库共享资源的隔离数据库。图 2-6 描绘了这种架构（有关如何实现`可插拔数据库`的详细信息，请参见第 23 章）。

![../images/214899_3_En_2_Chapter/214899_3_En_2_Fig6_HTML.jpg](img/214899_3_En_2_Fig6_HTML.jpg)

**图 2-6** 一个包含多个`可插拔数据库`的`容器数据库`

在决定是否使用一个数据库托管多个应用程序和用户时，必须考虑几个架构方面：

*   应用程序生成的重做量是否差异巨大，以至于可能需要不同大小的在线重做日志？
*   应用程序使用的查询是否差异足够大，需要不同数量的撤销空间、排序空间和内存？
*   应用程序类型是否需要不同的数据库块大小，例如联机事务处理（OLTP）数据库使用 8KB，而数据仓库使用 32KB？
*   是否有任何安全性、可用性、复制或性能要求需要将某个应用程序隔离？
*   应用程序是否需要`Oracle`企业版才有的功能？
*   应用程序是否需要使用任何特殊的`Oracle`特性，例如`Data Guard`、分区、`Streams`或`RAC`？
*   每个应用程序的备份和恢复要求是什么？一个应用程序需要联机备份而另一个不需要吗？一个应用程序需要磁带备份吗？
*   任何应用程序是否依赖于特定的`Oracle`数据库版本？是否会有不同的数据库升级计划和要求？

表 2-5 描述了关于如何使用`Oracle`数据库和应用程序的这些架构考虑因素的优缺点。这仅是查看数据库实例，而未使用包含`容器`和`可插拔数据库`的多租户技术。当我们利用`容器`时（这允许您整合到更少的服务器上），我们将重新审视这些缺点。这将在第 22 章讨论。

**表 2-5** `Oracle`数据库配置的优缺点

| 配置 | 优点 | 缺点 |
| :--- | :--- | :--- |
| 每台服务器一个数据库 | 为使用数据库的应用程序提供专用资源；将应用程序彼此完全隔离； | 最昂贵；需要更多硬件 |
| 每台服务器多个数据库和多个`Oracle 主目录` | 需要更少的服务器 | 多个数据库竞争磁盘、内存和 CPU 资源 |
| 一台服务器上多个数据库和一套`Oracle`二进制文件安装 | 需要更少的服务器；不需要多个`Oracle`二进制文件安装 | 多个数据库竞争磁盘、内存和 CPU 资源 |
| 一个数据库和一个`Oracle 主目录`服务于多个应用程序 | 只需要一台服务器和一个数据库；成本低 | 多个数据库竞争磁盘、内存和 CPU 资源；多个应用程序依赖于一个数据库；存在单点故障 |
| 包含多个`可插拔数据库`的`容器数据库` | 成本最低；允许多个`可插拔数据库`安全地共享一个父`容器数据库`的基础设施 | 多个数据库竞争磁盘、内存和 CPU 资源；多个应用程序依赖于一个数据库；存在单点故障 |
