# 13. 配置 RMAN 的备份保留策略

你可以将该值设置为 0 到 365 天之间的任何值。将值设置为 0 意味着 RMAN 元数据信息可以在任何时候被覆盖。
`CONTROL_FILE_RECORD_KEEP_TIME`参数在旧版 Oracle 中更为关键，在那些版本中，如果元数据被覆盖，用 RMAN 信息重新填充控制文件并非易事。`CROSSCHECK`命令可用于快速让控制文件感知到 RMAN 备份文件。
如果你进行每日备份，那么我建议你将此参数保留为 7 天。但是，如果你每月只备份一次数据库，或者由于某种原因，你的保留策略大于 7 天，并且你没有使用恢复目录，那么你可能需要考虑增加此值。增加此参数的缺点是，如果你有大量 RMAN 备份活动，这可能会增加控制文件的大小。

RMAN 保留策略允许你指定希望保留备份多长时间。RMAN 有两种相互排斥的方法来指定保留策略：

*   恢复窗口
*   备份数量（冗余度）

## 恢复窗口

通过恢复窗口，你可以指定一个过去的天数，以便能够恢复到该窗口内的任何时间点。例如，如果你指定一个 5 天的保留策略窗口，那么 RMAN 不会将那些能够恢复到该 5 天窗口内任何时间点所需的数据文件和归档日志备份标记为过时：

```
RMAN> configure retention policy to recovery window of 5 days;
```

对于指定的恢复窗口，RMAN 可能需要比 5 天窗口更早的备份，因为它可能需要一个更早的备份作为起点，才能恢复到指定的恢复点。例如，假设你上一次良好的备份是 6 天前进行的，而现在你想恢复到 4 天前。对于这个恢复窗口，RMAN 需要 6 天前的备份来还原并恢复到指定的点。

## 冗余度

你还可以指定 RMAN 保留的最小备份数量。例如，如果将冗余度设置为 2，那么 RMAN 不会将数据文件和归档日志文件的最新两个备份标记为过时：

```
RMAN> configure retention policy to redundancy 2;
```

我发现，基于冗余度的保留策略更容易操作，并且在备份保留时间方面更具可预测性。如果我设置冗余度为 2，我知道 RMAN 不会将最新的两个备份标记为过时。相比之下，恢复窗口保留策略依赖于备份频率和窗口长度来确定备份是否过时。

## 基于保留策略删除备份

你可以报告 RMAN 根据保留策略已确定为过时的备份，如下所示：

```
RMAN> report obsolete;
```

要删除过时的备份，请运行 `DELETE OBSOLETE` 命令：

```
RMAN> delete obsolete;
```

系统会提示你：

```
Do you really want to delete the above objects (enter YES or NO)?
```

如果你在编写脚本执行该过程，可以指定删除操作不提示输入：

```
RMAN> delete noprompt obsolete;
```

我通常将 `DELETE NOPROMPT OBSOLETE` 命令编码到备份数据库的 Shell 脚本中。这指示 RMAN 根据保留策略删除任何过时的备份和过时的归档日志（有关如何使用 Shell 脚本自动删除过时备份的示例，请参阅本章后面的“从决策到行动的过渡”部分）。

## 清除保留策略

默认的保留策略是冗余度为 1。你可以通过 `TO NONE` 命令完全禁用 RMAN 保留策略。

```
RMAN> configure retention policy to none;
```

当策略设置为 `NONE` 时，没有任何备份会被视为过时，因此无法通过 `DELETE OBSOLETE` 命令移除。这通常不是你想要的行为。你希望让 RMAN 根据窗口或备份数量的保留策略删除备份。

要将保留策略设置回默认值，请使用 `CLEAR` 命令：

```
RMAN> configure retention policy clear;
```
