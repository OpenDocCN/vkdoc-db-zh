# CBO 的计划选择质量

让我们来看看 CBO 成本估算过程的准确性，以及为什么它有时重要，有时又无关紧要。

## CBO 成本估算的准确性

你可能会惊讶地发现，CBO 的成本计算经常是极不准确的！幸运的是，这些错误往往被证明是无关紧要的。假设 CBO 正在为一个查询在两个可能的执行计划之间做选择。它认为计划 A 需要 5 秒运行，计划 B 需要 10 秒，因此选择了计划 A。现在，当使用计划 A 运行查询时，实际上可能需要 50 秒，CBO 的估算误差达到了 10 倍。嗯，如果对计划 B 的估算也是一个很大的低估，那么这个误差就不会产生任何后果。只要使用计划 B 运行查询时，速度没有明显快于使用计划 A，那么就没有造成损害：尽管做出了极不准确的成本估算，CBO 仍然做出了可接受的计划选择。

在许多情况下，特别是对于交互式联机事务处理 (OLTP) 风格的查询，计划的选择是显而易见的，即使对象统计信息非常不准确或不存在，CBO 也不可能出错。例如，如果你使用所有主键列上的相等谓词从一个巨大的表中读取一行，很难想象 CBO 会不使用主键索引。全表扫描的替代方法可能需要多花数百甚至数千倍的时间来运行，因此，即使 CBO 的成本估算偏差很大，仍然会选择正确的计划。

## 次优执行计划

尽管 CBO 通常会选择可接受的执行计划，但它也经常会严重地选择次优计划，即使统计信息收集正确且是最新的。换句话说，CBO 选择的计划运行时间比另一个替代计划要长得多。我将在本书后面解释为什么这种情况经常发生。

这些糟糕的计划选择常常不被注意的主要原因有两个：

*   第一个原因是许多 SQL 语句的性能无关紧要。假设你有一个在夜间运行的五小时批处理作业，执行了数千个 SQL 语句。如果一个 SQL 语句需要运行 30 秒而不是 1 秒，那通常无关紧要，特别是如果批处理中的其他语句在并发运行，而这个语句不在关键路径上。
*   第二个原因是无法轻易知道一个 SQL 语句应该运行多长时间；人们的期望通常很低。当然，例外是当一个计划变得更糟时；你会知道新计划很差，因为你体验过之前的好计划！我冒昧地保守猜测，大多数未经专业调优的夜间批处理作业的运行时间，如果有足够的分析时间，不添加任何代码（除了添加优化器提示），至少可以减少一半。在现实生活中，调优工作通常涉及物理数据库更改，例如添加索引和重写一些代码。在这些情况下，批处理时间减少 10 倍或更多并不少见！

关于 CBO 经常出错的断言可能会令人惊讶。但请记住，CBO 只是对什么是正确的计划做出猜测。它无法确定。常识告诉我们，当你一次又一次地猜测时，有时会猜错，而在某些猜错的情况下，你会错得非常、非常离谱。
