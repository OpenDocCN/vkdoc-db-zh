## 为什么事情会出错？

正如我在第 1 章开头所提到的，SQL 是一种声明式编程语言。理论上，你只需说明你想要完成什么，然后某个“东西”会设法找出如何去做。对于 Oracle 数据库而言，这个“东西”就是 CBO，它应该为你设计出合适的执行计划。理论上，你不必担心得到错误的结果，也不必担心性能。既然你正在阅读这本书，你几乎肯定已经意识到生活并非那么简单，事实上，你确实需要相当多的担心。除了 bug 之外，正确编写的 SQL 语句的结果将是正确的，但最优性能，甚至是可接受的性能，远非有保证，即使是来自一个没有 bug 的数据库也是如此。

那么为什么性能常常如此糟糕呢？我们在本书中已经触及了许多问题。没有正确的扩展统计信息集合，就会发生基数错误，而基数错误会完全迷惑 CBO。但迷惑 CBO 的不仅仅是基数错误。我们在第 4 章讨论了标量子查询缓存。CBO 无法评估标量子查询缓存的收益，也无法知道查询运行时哪些数据块会在缓冲区缓存中。

缓存只是信息缺失的一个例子。有时，只有在确定被索引的列不为空时才能使用索引。CBO 可能知道某列不为空，因为该列上有 `NOT NULL` 约束，或者 `WHERE` 子句中出现了 `IS NOT NULL` 谓词。在缺乏此类信息的情况下，CBO 将不得不假设该列实际上可能为 `NULL`，从而避免使用索引。

那么争用呢？你是否认为，当另一个会话正在更新你正在查询的索引时，你的性能可能会受到影响？我可以告诉你，它可能会。我见过真实案例，我不得不使用一个对 CBO 来说似乎是较差选择的索引，但查询性能良好，因为在我查询时，我选择的索引没有被其他进程更新。另一方面，CBO 选择的索引正被其他进程更新，导致了性能不佳。

声明式编程语言的概念意味着 CBO 应该是全知的，因此我们可以轻易地将糟糕执行计划的大部分责任归咎于 CBO 的不足。然而，设计你的物理数据库不在 CBO 的职责范围内；如果缺少索引，CBO 无法为你创建它。

你可以看到，本章涉及的许多概念在本书前面已经讨论过，但我想把它们汇集在一处并展开说明。让我们首先重新审视 CBO 面临的最大问题：基数错误。

### 基数错误

术语 *基数错误* 指的是在候选执行计划中，CBO 对行源操作返回行数的估计值与实际选择并运行该执行计划时将返回的实际行数之间存在显著差异。基数估计是 CBO 在执行最终状态优化时生成的最重要的信息。基数错误有几个原因。让我从最大的基数错误来源开始：相关列。
