# 第 18 章 结束与第 19 章 开始

即使最后一个例子中的所有转换都是基于成本的，CBO 可能仍然无法正确处理。一方面，CBO 完全专注于减少耗时，而不关心临时表空间的使用情况；CBO 假设临时表空间的空间永远不会耗尽。另一方面，CBO 目前无法衡量标量子查询缓存的好处，因此看不出保留第一次排序的意义。

当然，这个例子在几个方面存在严重缺陷。例如，即使我们想要分步执行聚合，我们也可能会想办法一次聚合多个国家的数据。这里的挑战，正如本书中经常出现的那样，是找到简单的例子来生成用于复杂情况的技术。在这个特定的例子中，它可能有点过于简化了。然而，这个例子所演示的原则是完全有效的：有时你需要以某种方式构建 SQL，以强制使用特定的查询评估方式，从而完全放弃在第 1 章开头引入的声明式编程语言的概念。在这种情况下，你需要应用提示来确保你精心构建的代码不会被 CBO 解构。

## `v$database_block_corruption` 视图

正如你从第 9 章中了解到的（如果你之前不知道的话），要求 CBO 在对象统计信息过时或其他情况下不合适时提出一个合理的执行计划，就像是要求它双手被绑在背后战斗一样。然而，几年前我不幸遇到了一个情况，CBO *总是* 双手被绑在背后战斗！

如果你从数据字典视图 `V$DATABASE_BLOCK_CORRUPTION` 中选择行，我真诚地希望，为你好，你得不到任何返回行！在一个健康的数据库中，这个视图不返回任何行，但如果 RMAN 在执行备份时检测到损坏的块，它会用有关损坏块的信息填充底层数据结构，你可以通过查看这个视图看到。

当你发现数据库中有损坏的块时，首先闪过你脑海的问题之一是：*哪些段是损坏的*？Oracle 支持网站上有一篇著名文章，472231.1，其中包含一个查询，该查询连接 `V$DATABASE_BLOCK_CORRUPTION`、`DBA_SEGMENTS` 和 `DBA_EXTENTS` 来识别损坏的块。

当你运行该说明中提供的查询时，CBO 会查看数据库对象 `X$KCCBLKCOR`（`V$DATABASE_BLOCK_CORRUPTION` 引用的对象）的统计信息，并发现它显然没有行，因为上次收集数据字典统计信息时 `X$KCCBLKCOR` 中没有行。当然，除非确实有损坏的块，否则运行说明 472231.1 中的查询是没有意义的，因此你总是需要给这个查询加提示才能让它快速运行！

顺便说一下，说明 472231.1 中的查询有第二个弱点，即它报告了段高水位标记以上的损坏，这不应该引起关注。我确实有一个修改后的脚本（当时提交给了 Oracle），它消除了误报并包含提示以提供合理的执行计划。这个脚本太长，无法在此处打印，但可以在可下载材料中作为 `corruption.sql` 找到。然而，请记住，我还没有机会在真实的损坏数据库上验证 `corruption.sql`，因此我不能保证其正确性或适用性。

## 总结

有一些提示是基于版本的重定义使用的，还有一些主要针对运行时引擎，例如 `APPEND` 提示。然而，绝大多数提示是优化器提示，它们分为两类。

一方面，一些优化器提示确实只是提示，在调查执行计划性能不佳的原因以及在 SQL 调整练习中识别可能的替代方案时非常有用。然而，由于其不可预测的行为，在生产系统上使用非规定性提示通常是不可取的。

另一方面，规定性提示，即真正作为指令的提示，可以合法地用于生产系统以解决某些类别的性能问题。在生产系统上使用提示应该只在经过深思熟虑后才进行，不是因为提示可能在未来的版本中停止工作，也不是因为 Oracle 的支持水平会受到负面影响，而是因为支持文档通常非常差或根本不存在。这种文档的缺失往往会导致误解和误用。

本章和前三章涵盖了在优化 SQL 性能过程中可供你使用的大部分技术。然而，还有一两种更先进的技术可能在特殊情况下有用。让我们继续到第 19 章。
