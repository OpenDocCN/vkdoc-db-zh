## 优化器转换

我们在本书中已经看到，有许多方法可以将一条 SQL 查询（或 DML 子查询）转换为另一条，而完全不改变其含义。对于除最简单的查询之外的所有查询，CBO 都会考虑多种此类转换，有时会估算成本，以确定用于执行我们在第 12 章中刚刚介绍的最终状态优化的最佳 SQL 语句。本章致力于讨论优化器转换的主题；您将会看到，这类转换的数量令人眼花缭乱。虽然并非所有的优化器转换都包含在本章中，但您在日常工作中遇到的绝大多数转换，以及一些您很少遇到的转换，都已涵盖。

您可能想知道为什么要在 CBO 转换上投入如此多的精力。毕竟，您可能会觉得以高效的方式编写 SQL 是程序员的责任，并且您的这种感觉是正确的。然而，随着本章的展开，您将看到，有些重写 SQL 语句的方式是普通程序员不太可能考虑到的，而这些深奥的重写有时会产生非常高效的执行计划。

然而，一些 CBO 转换，比如连接消除（我将在本章后面介绍），似乎只适用于编写得非常糟糕的代码。这类转换之所以重要，有以下几个原因：

*   查询经常引用为多种目的而设计的数据字典视图。一旦这些视图被展开到语句中，结果往往明显低效。
*   有时 SQL 代码是由图形工具或其他语言处理器生成的。自动生成的代码常常显得极其简单。
*   嗯，实际上，世界上有很多糟糕的 SQL 程序员！

有一类特殊的转换，我称之为“无需思考的”转换，我将很快讨论它们。除了“无需思考的”转换之外，所有优化器转换都可以通过 `NO_QUERY_TRANSFORMATION` 提示一键禁用。这个提示有时对分析和调试很有用，但我从未在生产 SQL 语句中使用过它。然而，在某些情况下，您需要在特定上下文中强制或禁用特定的转换。除了“无需思考的”转换之外，所有优化器转换都可以在合法的情况下通过特定于该转换的提示来强制执行，并且可以通过另一个提示来禁用，该提示通常（但不总是）通过在提示开头加上字符 `NO_` 来生成。例如，我们已经在第 8 章讨论了视图合并：`MERGE` 提示强制视图合并，而 `NO_MERGE` 提示则禁用它。此规则有两个例外：

*   或展开转换通过 `USE_CONCAT` 提示强制，通过 `NO_EXPAND` 提示禁用。
*   提取子查询物化通过 `MATERIALIZE` 提示强制，通过 `INLINE` 提示禁用。

本章解释的大多数查询转换都可以用合法的 SQL 来表达；换句话说，您可能一开始就自己编写了转换后的 SQL。然而，我们已经在第 11 章中看到，子查询去关联可以生成半连接和反连接，这些连接没有等价的合法 SQL 语法，而这里我们将讨论分组下推，这是另一个无法用合法 SQL 表达的转换示例。在阅读本章中旨在演示各个转换的清单时，请记住，转换后的查询文本可能不是精确的表示。这些清单显示了原始查询以及（在可能的情况下）转换后的查询，同时附带相关的执行计划以及强制和禁用相关转换的提示。
