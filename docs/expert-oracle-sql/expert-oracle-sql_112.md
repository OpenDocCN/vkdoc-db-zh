# 第十四章 剩余问题

我是那种喜欢在事情出错时责怪他人或其他事物的人之一。我妻子会告诉你，在我心里，从来不是我的错。这正是我喜欢 **CBO**（基于成本的优化器）的一点。当性能不佳时，它是一个很好的替罪羊！当我的 SQL 执行得很差时（是的，我有时会写出性能不佳的 SQL），我通常可以声称 CBO 本应发现我的错误并修复它。然而，**CBO** 不能为缺失的索引背锅！我得找别人来怪罪这个！

当然，物理数据库设计远不止是确定要创建哪些索引。其中最重要的考虑之一是确定要删除哪些索引。下一章，第十五章，将更详细地探讨物理数据库设计的主题，所以我现在就停笔。我只希望你将糟糕的物理设计加入那份不断增长的“出错原因”清单中。

## 争用

当 **CBO** 考虑为一条语句使用何种执行计划时，它不会假设同时可能有其他数据库活动在进行。对并发活动的考虑可能会影响最优执行计划。

考虑以下场景。我们连续运行十个作业。第一个作业向表`T1`中插入数据，其`JOB_ID=1`。第二个作业插入`JOB_ID=2`的数据，依此类推。假设在执行最后一个作业时，我们决定查询`JOB_ID=1`的部分数据。我们的查询指定了`WHERE JOB_ID=1 AND PRODUCT_CODE=27`。**CBO** 对任何争用问题浑然不觉，它选择使用`PRODUCT_CODE`上的索引，因为该索引更具选择性且聚簇因子较低。但第十个作业正在大量插入`PRODUCT_CODE=27`的行。这意味着我们的查询可能需要做大量工作来回滚这些插入，并将数据块重构成查询开始时的样子。

现在假设我们提示查询使用`JOB_ID`上的索引。使用选择性较低的`JOB_ID`索引可能更高效，因为查询运行时，`JOB_ID=1`的那部分索引不会被更新。此外，被访问的表数据块早已填满且不会被更新。通过`JOB_ID`索引访问的索引和表数据块在我们的查询过程中均未被更新这一事实，可能限制甚至消除回滚活动。我曾经处理过一个运行数小时的查询，通过更改查询使用一个表面上看起来很差的索引，执行时间降到了几分钟！

## 总结

本章可能让你感到沮丧。所有这些关于似乎永无止境的、阻止 **CBO** 制定出最优执行计划的原因的负面谈论，可能令人望而生畏。而我的清单只是触及了表面。还有许多导致出错的原因我甚至没有在此涵盖。一条 SQL 语句越复杂，特别是其谓词越复杂，**CBO** 猜对的可能性就越小。别太沮丧。本章的主要目的是摘掉你可能一直戴着的玫瑰色眼镜。一旦你理解了事情确实会出错以及它们出错的主要原因，你就处于更有利的位置去采取行动！让我们现在就开始着手解决吧。进入第十五章，看看良好的物理数据库设计如何帮助提升 SQL 性能。
