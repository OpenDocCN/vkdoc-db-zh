## 物理数据库设计

为了让运行时引擎以足够的性能执行 SQL 语句，你必须为它提供一台配置合理的机器，具有足够的 **SGA**（系统全局区）和 **PGA**（程序全局区）。然而，物理数据库设计也至关重要：即使机器配置合理、SQL 语句编写良好、**CBO** 完美地做出了所有基数估计，除非你关注物理数据库设计，否则也不太可能获得良好的性能。

### 关于 EXADATA 的一点说明

有人认为 **Exadata** 的一个好处是，你无需担心物理数据库设计：全表扫描成本极低，硬件资源又如此充足，谁会在乎呢？

我觉得这有点夸张。是的，花数月时间设计物理模型可能不划算，但人们总会希望有效利用硬件。考虑物理数据库设计几乎总是一个好主意。

本章回顾了物理数据库设计中更为关键的方面，这是我们无法归咎于 **CBO** 的少数 SQL 性能问题之一。

人们常常将“改进物理数据库设计”等同于“添加索引”。这与事实相去甚远，我们将在本章开头比较索引的积极和消极方面。一旦你断定需要一个或多个索引，你仍面临许多决策：索引中要包含哪些列，是否使用位图索引等等。我们将探讨其中的一些问题。

分区选项是提高性能的一个流行且有效的工具，我将介绍我认为最重要的三个分区性能优势：对表的部分进行全表扫描、使用分区连接的能力以及并行化的更多可能性。

在某些圈子里，“反规范化”这个词被视为邪恶，就像“提示”一样。而就像提示一样，反规范化在 SQL 性能调优中有一席之地。我将在本章后面介绍物化视图和其他形式的反规范化。我将用一些关于压缩和 **LOB**（大型对象）的评论来结束本章。

然而，在我们开始之前，我想再次强调，物理数据库设计几乎从来不是针对单个 SQL 语句进行调优。你通常是寻找优化整个应用程序的方法。在我们讨论索引时，这个想法应该牢记在你脑海中。

## 添加与删除索引

想象一下，你正在查看一个查询的执行计划。你看到一个索引被用于访问表，但该索引的选择性并不高：经常访问了一个表数据块，然后因为未索引列上的谓词，索引中指定的行被丢弃。“啊哈，”你说，“让我创建一个新索引，包含我的查询中所有谓词引用的列，这样一切就都会好了。”你创建了新索引（或向现有索引添加了列）……瞧！你的查询运行得很快，你拍拍自己的背。这种行为太常见了，它反映了一种不恰当的思考索引和物理数据库设计的方式——你不应该零敲碎打地添加索引来优化单个查询。你需要思考的是，为支持**所有**查询且不过度负担 **DML**（数据操作语言）操作，特定表所需的最优索引集合。让我们首先从表的另一面看索引策略：优化 DML。
