# Oracle SQL 提示（Hints）的使用与考量

*提示（Hints）是在 Oracle7 中引入的，当时如果优化器生成了次优的执行计划，用户几乎束手无策。现在，Oracle 提供了多种工具，包括 SQL 调优建议器、SQL 计划管理器和 SQL 性能分析器，来帮助你解决那些优化器未能解决的性能问题。Oracle**强烈建议你使用这些工具，而不是提示**。这些工具**远优于提示**，因为如果持续使用，它们能在你的数据和数据库**环境发生变化时提供***新的解决方案***。**

**提示应谨慎使用**，并且仅在对相关表收集了统计信息，并使用`EXPLAIN PLAN`语句评估了不使用提示时的优化器计划之后。数据库条件的改变以及后续版本中查询性能的增强，都可能显著影响你代码中提示对性能的效果。*

*——摘自 Oracle 12c SQL 语言参考手册*

一位知名的 SQL 专家承认提示的使用是现实的一部分，但他将提示分类为“好、坏和丑”，这意味着，根据提示的不同，在生产代码中使用提示可能会让你面临相当大的风险。

这类观点中所表达的情绪不应被轻易否定，但有时会被误解。事实是，在某些情况下，在 SQL 语句中使用一个或多个提示提供了最佳、有时甚至是唯一的方式来确保性能足够且稳定。而这类情况的数量，远比你在阅读所有这些可怕的警告后想象的要多。

本书的许多章节都包含了提示如何被合法使用的示例，最值得注意的是第 14 章，它解释了优化器为何可能无法自行得出最优执行计划的诸多原因。在本章中，我想通过几个案例研究，展示一些将提示作为一种优化技术的合法示例，这些示例在本书其他章节中并未出现。但在那之前，让我退一步，试图对那些“末日预言者”的观点提供一些平衡，以帮助你形成自己关于如何在生产代码中处理提示的看法。这都归结为一个问题：**提示是可支持的吗？**

## 提示是可支持的吗？

我知道有些人持有这样一种假设：如果他们遇到一个包含提示的 SQL 语句的问题，Oracle 在移除那些提示之前不会为他们提供任何支持。这种推理暗示提示本质上是 SQL 语言中一个不可支持的特性。让我通过几个自己的真实经历，来试着判断这种观点的合理性。让我们从重新审视我在第 13 章中介绍过的连接子查询下推优化器转换开始。

### `PUSH_SUBQ`的故事

如果你正在从事从 9i 到 10g 的数据库升级工作，并且你的代码包含了`PUSH_SUBQ`或`NO_PUSH_SUBQ`提示来控制子查询下推，你可能会大吃一惊。在 9i 中，提示被放在*外部*查询块中，并应用于被包围查询块引用的*所有*子查询。在 10g 中，提示的行为得到了增强，允许对每个子查询进行不同的处理；现在你需要将提示放在*每个单独的子查询中*，*而不是*放在外部查询块中。

这个 10g 中针对在 9i SQL 参考手册中有文档记载的提示所做的更改，并未向后兼容现有的 9i 代码，因此所有包含`PUSH_SUBQ`提示的语句在升级时都需要编辑。正是这样的经历让一些人得出结论，认为 Oracle 对待优化器提示的方式不同于对待 SQL 语言的其他部分，因此，使用优化器提示是一种有风险的事情。但在你决定是否同意这些结论之前，思考一下我的下一个故事。

### DML 错误日志记录的故事

PL/SQL `FORACLE`语句中的一个合法子句是`SAVE EXCEPTIONS`，这在 Oracle 12c PL/SQL 语言参考手册的清单 12-13 中有所演示。`SAVE EXCEPTIONS`子句实现了所谓的*批量错误模式*。在 PL/SQL 中，批量错误模式会导致填充集合`SQL%BULK_EXCEPTIONS`，它标识出那些因为完整性约束违规等原因而无法插入或更新的行。解决同一问题的另一种方法是使用 SQL 语言参考手册中作为`INSERT`、`UPDATE`和`MERGE`语句一部分描述的*DML 错误日志记录*。在同一条语句中同时使用批量错误模式和 DML 错误日志记录是不合逻辑的，但它在 10g 中是有效的。许多客户发现，当他们从 10g 升级到 11g 时，突然收到了 ORA-38909 错误信息，告知他们之前可以工作的代码现在非法了。只要搜索一下这个错误代码，就能看到它引起的麻烦。

大多数遇到 ORA-38909 错误代码的客户使用的是 PL/SQL，他们可以修改代码以移除冗余的`SAVE EXCEPTIONS`子句。然而，我的客户使用的是 ODP.NET，它*无条件地*使用批量错误模式。在 ODP.NET 中使用批量错误模式意味着无法调用包含错误日志记录子句的 DML。

我代表客户提交了服务请求，大约三个月后，Oracle 承认这确实是一个 bug。然后 Oracle 询问这个 bug 的影响有多大，以便他们可以优先安排修复。由于我的客户在中间的三个月里已经重新编写了他们的应用程序代码，并且没有其他客户报告此问题，因此不再有业务需求去修复它。在 12.1.0.1 版本中，仍然无法从 ODP.NET 调用带错误日志记录子句的 DML！

但是，这个故事与优化器提示有什么关系呢？关键在于，有时 Oracle 对其产品已记录特性的更改并非向后兼容。如果这些更改影响了大量的客户，那么 Oracle 将不得不提供修复或变通方案。变通方案可能涉及客户更改他们的代码。对于有文档记载的优化器提示或 Oracle 数据库产品的任何其他有文档记载的特性，情况并无不同：99.9%的情况下升级是向后兼容的，但偶尔会发生一些不可预见的事情。

但本书讨论了众多*未记录的*提示。我们对这些提示的态度是否应该与对有记录提示的态度不同呢？现在让我们来讨论这一点。

## 有记录提示与未记录提示

SQL 语言参考手册中有一章题为“Oracle SQL 的基本要素”，其中包含对注释、特别是提示的描述。有几十个提示是有文档记载的，但一个完整（或近乎完整）的提示列表可以在视图`V$SQL_HINT`中找到，它有 314 行！

你可能会假设 Oracle 对有记录提示的使用做出了一定的隐式保证，但对于未记录的提示，Oracle 可以在任何时候移除或更改其行为而不受道义上的追究。事实上，甚至要弄清楚什么构成一个有记录的提示也并非易事。例如，《数据仓库指南》提到了我们在第 13 章中介绍过的`EXPAND_GSET_TO_UNION`提示，以及其他提示，如`NO_MULTIMV_REWRITE`和`NO_BASETABLE_MULTIMV_REWRITE`。这些提示都没有出现在当前的 SQL 语言参考手册中，尽管`EXPAND_GSET_TO_UNION`出现在该手册的旧版本中。Oracle 是否支持使用这些提示呢？


或许讨论最多的未公开提示就是`CARDINALITY`提示了。在撰写本书时，Tom Kyte 是`CBO`（基于成本的优化器）的产品经理。尽管 Tom 担任此职位时间不长，但他对 Oracle 数据库产品的了解却广为人知。他已出版多本关于该产品的优秀书籍，并维护着著名的*Ask Tom*网站，时间之久超出了大多数人的记忆。2010 年，在 Ask Tom 网站上回答问题编号为 2233040800346569775 时，Tom 表示：

…正确的[`cardinality`提示]并未公开。然而，在我看来，它是少数几个可以“安全”使用的未公开功能之一，因为它的使用不会导致数据损坏、错误答案或不可预测的结果。如果它起作用——它会影响查询计划；如果不起作用——则不会。仅此而已——在这方面它相当“安全”。

嗯，事实是绝大多数提示都是优化器提示，关于`CARDINALITY`提示安全性的说法当然可以扩展到每一个优化器提示，不是吗？优化器提示要么会改变执行计划……要么不会！事实上，我个人认为`CARDINALITY`提示是在生产系统上使用的最*危险*的优化器提示之一，因为它的效果实际上是不可预测的。我稍后会介绍`CARDINALITY`提示的语义。

实际上，很多提示未公开，仅仅是因为 Oracle 还没抽出时间来记录它们。我在第 4 章中讨论了`GATHER_PLAN_STATISTICS`提示，但这个提示直到 12c 才出现在 Oracle 官方文档中！再举一个例子，很明显 Oracle 希望我们能够使用提示来控制连接顺序和连接方法，因此文档记录了`LEADING`、`USE_NL`、`USE_MERGE`和`USE_HASH`提示。但任何时候，如果我们同时使用`LEADING`和`USE_HASH`提示，我们绝对*必须*同时使用`SWAP_JOIN_INPUTS`或`NO_SWAP_JOIN_INPUTS`提示，以使我们的意图明确无误。`SWAP_JOIN_INPUTS`和`NO_SWAP_JOIN_INPUTS`未被记录的唯一原因，仅仅是 Oracle 还没抽出时间来记录它们。这完全不涉及可支持性问题。

讨论到这个阶段，你可能会认为我的观点是，你应该毫无顾忌地使用任何提示，无论是否公开，而且应用程序代码中漂浮着多少此类提示，都不会影响应用程序的可支持性。这就有些极端了。事实上，使用提示*确实*存在可支持性方面的影响，但它们并非我们迄今为止讨论的那些。让我用一个类比来说明我的观点。

`MODEL`子句的推论

如果你觉得需要在 SQL 语句中使用`HAVING`子句，尽管去用。如果你想使用分析函数，也不必有所顾虑。应用程序中 1%还是 90%的 SQL 语句包含`HAVING`子句，并不影响应用程序的可支持性，尽管如果 90%的语句都有`HAVING`子句，那会是个奇怪的应用！如果有人阅读你的代码却不理解 SQL 语言的基本特性，那是他们的问题。他们应该回家去读手册。

但如果你突然想使用`MODEL`子句怎么办？你应该像使用分析函数那样，几乎不假思索地开始使用它吗？绝对不行。`MODEL`子句在极少数应用程序中使用，大多数程序员对它一无所知。你的同事不可能回家前读几页关于`MODEL`子句的资料，睡一觉醒来就成了专家。这是个复杂的野兽！

所以，如果你想使用`MODEL`子句，先深呼吸。你能用传统的 SQL 语法以足够的性能获得你需要的结果吗？如果可以，并且`MODEL`子句在你的应用程序中尚未使用，那么你或许应该避免成为第一个引入它的人。另一方面，如果你有明确的需求（例如我在第 7 章中讨论的移动中值示例），那么你不必仅仅因为它不熟悉而害怕使用`MODEL`子句。只需确保你给代码加上注释，并向同事解释你在做什么以及为什么。

当你有冲动想在注释开头加上那些加号时，也应该经历类似的思考过程。你应该问自己这类问题：

*   我知道为什么需要给这段代码加提示吗？为什么`CBO`选择了一个与我认为的最佳方案不同的计划？
*   我知道给代码加提示是否真的会产生积极影响，还是我在做一个可能不正确的假设？
*   有替代方法吗？这些替代方法比使用提示更好还是更差？
*   我理解我打算使用的提示的作用吗？
*   这些提示是否合适且完整？
*   我的同事能理解我所做的和为什么这么做吗？

如果你发现对以上所有问题都有满意的答案，那就去做。如果没有，那就尝试*获得*满意的答案，或者为你的问题找到替代解决方案。

那么，所有的类比和轶事把我们引向何方？以下是我个人关于使用提示的一些思考总结。

可支持性结论

如果你的 Oracle 数据库产品存在功能或性能问题，你从 Oracle 获得的回应将取决于你的业务影响程度以及受类似问题影响的客户数量。就其性质而言，单个 SQL 语句的性能问题几乎总是远不如数据库无法启动之类的数据损坏问题严重。因此，无论问题是由于优化器提示还是数据库产品的其他特性引起，也无论提示是否公开，Oracle 对 SQL 性能问题提供的支持程度是有限的。

这并非对 Oracle 的指责。商业运作就是如此。事实上，你会发现，像`ORDERED`和`AND_EQUAL`这样的已弃用提示，尽管多年前已从 SQL 语言参考手册中移除，但在许多客户多年前编写的代码中仍然广泛使用，并且在 12c 中依然运行良好。

与提示可支持性相关的真正问题，更多地聚焦于我们自身，涉及你和你的同事。可悲的事实是，即使是已公开的提示通常也记录得不够好，并且存在各种隐藏的限制，这意味着我们都在某种程度上处于摸索状态。

在 2013 年的 Oracle Open World 大会上，Jonathan Lewis 在他的一次演讲中说了“*we don't know how to hint*”（我们不知道如何使用提示）这句话。这句话至今仍在我的脑海中回响，因为它*并非*虚假的谦逊！没有人，我是说没有人，了解每一个优化器提示的所有来龙去脉。这意味着，除了最简单的 SQL 语句外，没有人能绝对肯定地说，他们拥有一套正确的提示，能够永远提供最优和稳定的性能。

然而，撰写本书让我意识到，编写优秀的 SQL 非常困难，提出一个好的物理数据库设计也是如此。所以，我们不仅不知道如何使用提示，而且*也不知道如何编写 SQL*，*也不知道如何设计我们的数据库*！我们只能根据具体情况，基于我们的知识，为业务问题提供我们能找到的最简单、最可支持的解决方案。
