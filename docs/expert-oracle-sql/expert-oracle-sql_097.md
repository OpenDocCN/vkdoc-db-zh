到目前为止，我展示的所有涉及区块范围粒度的示例都指定了未分区的表。为了表明分区表虽然非必需，但也并非被禁止，清单 11-21 将 `T_PART2`（一个分区表）与 `T1` 进行了连接。从 `PSTART` 和 `PSTOP` 列可以看出，对分区列 `T_PART2.C2` 的过滤谓词允许进行分区消除，因此并非扫描表中的所有块。但 `T_PART2` 的剩余块在 `PQSS2` 的成员之间进行分发时，并未考虑它们来自哪个分区。

因为连接条件使用了 `T_PART2.C4`，而 `T_PART2` 是使用 `T_PART2.C2` 进行分区的，所以部分分区级连接不可用。作为替代，我们使用了哈希分发。

我想明确区分哈希*分发*和哈希*连接*，因为清单 11-21 中同时使用了这两种技术。哈希分发通过 `PQ_DISTRIBUTE (T_PART2 HASH HASH)` 指示强制指定，并在清单 11-21 的第 5 行和第 9 行通过 `PX SEND HASH` 操作实现。在清单 11-21 中，对列 `T1.C1` 使用了分发哈希函数，以确保 `T1` 中的每一行都由 `PQSS1` 准确地发送到 `PQSS2` 的一个成员。当 `PQSS1` 扫描 `T_PART2` 时，对 `T_PART2.C4` 使用了相同的分发哈希函数，以便将来自 `T_PART2` 的行发送到持有任何来自 `T1` 的匹配行的 `PQSS2` 的那个成员。

![image](img/sq.jpg) **注意** 尽管哈希分发避免了因分区大小变化引起的负载不均衡，但负载不均衡仍可能因哈希函数而产生。例如，如果 `T1.C1` 只有一个值，那么来自 `T1` 的所有行都将被发送到 `PQSS2` 的同一个成员。

连接哈希函数通过 `USE_HASH (T_PART2)` 指示强制指定，并在第 3 行通过 `HASH JOIN BUFFERED` 操作实现。与串行哈希连接或其他分发机制的并行连接一样，`PQSS2` 的成员使用连接哈希函数在 `T1.C1` 和 `T_PART2.C4` 上对收到的行进行匹配。哈希分发可以与归并连接一起使用，但它不支持嵌套循环连接。

你会注意到哈希分发涉及三个 DFO：一个 DFO 用于获取驱动行源的数据，第二个 DFO 用于获取探测行源的数据，第三个用于连接本身。在清单 11-21 中，`PQSS1` 先读取 `T1`，然后转而读取 `T_PART2`。连接操作由 `PQSS2` 执行。

这看起来很棒，那么有什么不喜欢的呢？嗯，我精心设计了一个非常顺利的场景，一切都没有出错。但是存在一个性能威胁——一个很大的威胁。线索就在第 3 行连接操作的名称中。哈希连接总是涉及一个工作区，但第 3 行的 `HASH JOIN BUFFERED` 操作涉及两个工作区！我稍后会解释为什么，但首先我想向你展示 12cR1 版本中对哈希分发进行的一项不错的改进。

执行计划中的更改在清单 11-22 中以粗体显示。这里发生的情况是，`STATISTICS COLLECTOR` 从 `T1` 缓冲一定数量的行（当前 DOP 的两倍），如果未达到该数量，则将原始的哈希分发机制更改为广播。

你可能会认为，一旦来自 `T1` 的行被广播，执行连接的同一个 DFO 将访问 `T_PART2`，并且 `:TQ10001` 将不会被使用。这当然是我最初的假设。不幸的是，在 12cR1 中，来自 `T_PART2` 的行仍然通过一个 TQ 发送！据我所知，广播来自 `T1` 的行的唯一优点是使用了轮询分发机制而不是哈希分发，因此可以避免由哈希函数带来的负载不均衡。希望这个冗余的 DFO 在未来某个时候能被移除。

考虑到我在第 6 章中关于自适应行为的评论，你可能会惊讶地发现我其实挺喜欢这个功能。这个功能与迄今为止所有其他自适应功能的最大区别在于，决定使用哈希分发还是广播分发是在查询每次运行时动态做出的。因此，你可以在一次执行时使用哈希分发，下一次执行时广播，第三次执行时又回到哈希分发。如果适应性决策在首次做出后不是固定不变的，我是否会更热衷于自适应连接方法这类功能呢？我当然会。

## 验证自适应连接方法

如果你查看 *VLDB 和分区指南* 中关于*使用并行执行*的章节，你会发现一些可用于研究并行执行行为和性能的视图信息。其中最重要的是 `V$PQ_TQSTAT`，可下载的脚本在测试用例中使用了这个视图，以证明广播与哈希分发的选择是在游标每次执行时动态决定的。

现在是时候解释这个 `HASH JOIN BUFFERED` 操作是什么，以及为什么它是对性能的巨大威胁了。

### 数据缓冲

当 `T1` 是 3GB 而 `T_PART2` 也是 3GB 时，串行执行的哈希连接必然溢出到磁盘，但使用哈希分发并行实现的哈希连接可能不会。另一方面，假设 `T1` 是 500MB 而 `T_PART2` 是 30GB——串行哈希连接可能不会溢出到磁盘，但假设同样有 15GB 的 PGA 可用，这个并行执行计划将会溢出到磁盘！



### 布隆过滤

花朵会绽放，有时你需要修剪它们。但如果你认为布隆过滤与分区修剪有关，那可就错了！事实上，根据维基百科关于布隆过滤的文章，这项技术是由伯顿·霍华德·布隆在 1970 年发明的。该技术并非 Oracle 数据库或一般数据库所特有。但 Oracle 在优化并行连接时对其进行了巧妙应用。

我们试图避免的场景如下：我们将一行数据从探头行源通过 TQ 发送到消耗端 DFO，后者将其与驱动行源执行连接。不幸的是，消耗端 DFO 发现它接收到的行与驱动行源中的任何行都不匹配，于是丢弃了这行来自探头行源的数据。多么浪费啊！

通过某种方式，可以由生成端 DFO 在通过 TQ 发送大多数不匹配的探头行源数据之前就将其丢弃！就我而言，这简直是纯粹的魔法，但如果你真想了解伯顿的发明是如何应用于 Oracle 数据库的并行连接的，只需谷歌搜索“Christian Antognini Bloom Filtering”，你首先找到的文章将解释一切。

布隆过滤涉及从驱动行源生成少量数据，并将其反向通过 TQ 发送，以便用于过滤行。这种过滤有时不会显示在操作表或过滤谓词中，但有时会。代码清单 11-23 回到了代码清单 11-18 的部分分区智能连接，以查看当我们对探头行源而非驱动行源进行分区时会发生什么。

## 总结

尽管在查询或子查询中连接行源只有四种基本方法，但在执行计划中可能出现的操作实际上要多得多。这些变体是由于哈希连接输入交换以及需要支持不同类型连接而产生的：内连接、外连接、全外连接、分区外连接以及多种类型的半连接和反连接都需要独立的连接操作。

当查询或子查询中的多个行源被连接时，连接树会作为 CBO 选择连接顺序的结果而创建，但存在许多限制约束了连接顺序的选择。特别地，CBO 从不选择丛状连接，并且外连接中的保留行源必须先于可选行源。这些限制无法被覆盖，即使使用提示也不行，尽管在某些情况下哈希连接输入交换是一种替代方案。
