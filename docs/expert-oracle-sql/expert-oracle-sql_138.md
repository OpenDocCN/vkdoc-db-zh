# 基于时间的列统计信息复杂性

## 偏斜的列值与范围谓词

清单 9-8 展示了如何为表设置直方图，以处理在相等谓词中为不同值需要不同执行计划的偏斜数据。我在第 9 章中也提到，当涉及范围谓词时，你可能需要为高低端点指定现实的值（而不是清单 9-8 中的那些离谱值）。但是，如果你遇到基于时间的偏斜数据和/或基于时间的范围谓词，该怎么办呢？

考虑一个保存了 12 个月工资支付记录的工资系统。绝大多数交易发生在每月 20 号（除非 20 号是周末），因为那是公司的发薪日。然而，有极少数交易发生在其他日期，用于处理高管薪酬、错误、调整或其他异常事件。

在这个工资应用中，热门的值会随时间变化：每个月会添加一个新的标准发薪日，同时移除一个旧的。就我个人而言，我还没有处理过带有此类基于时间的直方图的应用程序，但弄清楚如何应对这样的需求似乎是一项有益的练习。我建议在添加新的支付日期和移除旧日期时，手动更新直方图。

请注意，用新值手动更新直方图与收集统计信息有很大不同；前者变更受到严格控制，不存在执行计划发生不良变化的风险，但在后者情况下，我们又回到了玩俄罗斯轮盘赌的状态。我们假想的工资系统的维护过程需要每月至少调用一次 `TSTATS` 包中的自定义过程，以保持直方图的时效性。

这种针对特定应用程序的 `TSTATS` 定制是典型的需要你考虑的情况，这也说明了为什么 `TSTATS` 通常不是即插即用的解决方案。现在让我们来看基于时间的列统计信息的最后一个复杂性：相关性。

## 相关列与表达式

清单 9-11 和 9-12 展示了如何设置扩展统计信息来处理谓词中的相关列和表达式。值得注意的是，你也可以为扩展统计信息创建直方图，而且扩展统计信息也可能是基于时间的。在列组统计信息上手动设置直方图远非易事。理论上，应该能够使用 `SYS_OP_COMBINED_HASH` 函数来设置直方图的值，但我一直未能成功实现。此外，当直方图中缺少值时，CBO 根本不会使用扩展统计信息。

很容易想象更复杂的情况，那时扩展统计信息甚至会更不实用。我最近参与的一个应用程序有如此多的相关列，以至于可能需要数百个扩展统计信息。然而，通过对性能不佳的查询进行仔细分析，我找到了另一种解决方案。该方案是伪造某些列的统计信息。

为了说明这一点，你可以看到清单 20-2 创建的 `PAYMENTS` 表包含一个名为 `JOB_DESCRIPTION` 的列，并且每个职位描述都与 `PAYGRADE` 高度相关。我们可能知道，应用程序中所有包含 `JOB_DESCRIPTION` 谓词的查询也都包含 `PAYGRADE` 谓词。既然如此，伪造 `JOB_DESCRIPTION` 的 `NUM_DISTINCT` 值可能是可行的。通过将 `NUM_DISTINCT` 设置为一个非常低的值，我们可以最小化涉及 `JOB_DESCRIPTION` 的谓词导致 CBO 错误地降低基数估计的程度。

但是，让我们喘口气。与其使用大量代码来操作列统计信息以解决复杂问题，不如精心构建样本数据。让我演示一下。

## 利用样本数据解决复杂统计问题

现实数据的问题在于其不可预测性。某个月你的工资异常比例可能是 2%，下个月可能是 1%，再下个月可能是 3%。但是，如果你手动创建样本数据并在该样本数据上收集统计信息，就可以消除这种变化。在某些情况下，在伪造的数据上收集统计信息比直接伪造统计信息更容易。一旦收集完成，来自样本数据的列统计信息就可以复制到现实表中。

无论我们是伪造统计信息还是伪造样本数据，都必须从分析需求开始。因此，让我基于对使用清单 20-2 创建的 `PAYMENTS` 表的 SQL 语句的分析，提出以下结果。

### PAYGRADE 列分析

我们发现，处理 `PAYGRADE` 为 1 的两名员工的工资支付的代码，与处理 `PAYGRADE` 为 2 到 `PAYGRADE` 为 10 的 9998 名员工的工资支付的代码不同。我们需要一个直方图，显示支付给 `PAYGRADE` 1 员工的少量支付记录，但我们不希望直方图反映剩余工资等级之间的真实差异，因为除了 1 以外的所有工资等级都使用相同的 SQL 进行处理。我们决定，我们的直方图应显示 `PAYGRADE` 1 的行数比其他工资等级少 100 倍。这意味着：

*   `PAYGRADE` 1 的选择性为 1/901
*   工资等级 2 到 10 各自的选择性为 100/901

记住，我们想要一个能产生有效且稳定执行计划的直方图。直方图没有准确反映 `PAYMENTS` 表中的数据分布并不重要。

### PAYMENT_DATE 列分析

```sql
DBMS_STATS.set_column_stats (ownname         => r.owner
                                     ,tabname         => r.table_name
                                     ,colname         => r.column_name
                                     ,distcnt         => v_num_distinct
                                     ,density         => 1 / v_num_distinct
                                     ,nullcnt         => r.num_nulls
                                     ,srec            => NULL -- No HIGH_VALUE/LOW_VALUE
                                     ,avgclen         => r.avg_col_len
                                     ,no_invalidate   => FALSE
                                     ,force           => TRUE);
      END LOOP;
   END adjust_column_stats_v2;
-- 其他过程未显示
END tstats;
/

EXEC tstats.adjust_column_stats_v2(p_table_name=>'PAYMENTS');
SELECT *
  FROM payments
 WHERE special_flag = 'Y';
```

| Id  | 操作              | 名称     | 行数   |
| --- | ----------------- | -------- | ------ |
|   0 | SELECT STATEMENT  |          |   1200 |
|   1 |  TABLE ACCESS FULL| PAYMENTS |   1200 |

你可以看到，`TSTATS.ADJUST_COLUMN_STATS_V2` 现在将 `SPECIAL_FLAG` 的 `NUM_DISTINCT` 设置为一个略大于 1 的非整数值，并将 `DENSITY` 设置为 `1/NUM_DISTINCT`。现在，一个列的 distinct 值数量为非整数在逻辑上是不合理的，但 CBO 似乎对此并不在意，并将这个非整数值代入其计算中。正如我们通过查询的基数估计所看到的，我们避免了 CBO 代码中不期望的特殊代码路径，并且我们的基数估计值恢复到了一个合理的（在这种情况下是绝对正确的）值。
