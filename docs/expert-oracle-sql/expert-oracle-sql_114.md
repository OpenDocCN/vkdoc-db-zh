# 索引维护成本与识别

如果你在一个表上有十几个索引，那么每次向该表添加一行时，都需要添加十几个索引条目。再加上对表块本身的更改，总共会产生 13 个块更改。由于这些更改是针对不同的数据库对象进行的，因此这 13 个块更改保证会涉及不同的块。不要低估这种开销。索引占用`SGA`中的空间，增加了所需的`redo`量，并经常产生大量`I/O`。你是否见过`INSERT`语句持续等待“`db file sequential read`”事件？你可能假设这些等待事件与`INSERT`语句的查询部分有关。检查这一点最简单的方法是查看`V$ACTIVE_SESSION_HISTORY`或`DBA_HIST_ACTIVE_SESS_HISTORY`中的`CURRENT_OBJ#`列。这个数字代表与被访问对象相关的`DBA_OBJECTS`中的`OBJECT_ID`。你很可能会发现，大部分或全部这些等待事件都与你正在插入行的表的索引块有关；你需要在添加索引条目之前，将索引块读入`SGA`。如果你能减少一个表上的索引数量，你就可以显著提高`DML`的性能。以下是《性能调优指南》中的一段话：

> 使用这个简单的估算指南来理解索引维护的成本：`INSERT`、`DELETE`或`UPDATE`索引键所维护的每个索引，其资源消耗大约是实际在表上执行的`DML`操作的三倍。因此，如果你向一个有三个索引的表执行`INSERT`，那么该插入操作大约比向一个没有索引的表执行`INSERT`慢 10 倍。对于`DML`，尤其是`INSERT`密集型的应用程序，索引设计应被认真审视，这可能需要在查询性能和`INSERT`性能之间做出妥协。

根据我见过的许多拥有数十个索引的表来看，许多开发员并没有意识到上述问题。要么如此，要么在解决查询性能问题的紧要关头忘记了这些信息。
