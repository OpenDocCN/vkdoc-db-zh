# 基本执行计划概念

既然您了解了 CBO 是什么以及它的作用，您需要开始学习如何解读它的输出：执行计划。您经常会听到执行计划被称为 `explain plan`，因为 SQL 的 `explain plan` 语句是查看执行计划最重要方法之一的关键组成部分。本章将详细探讨 `explain plan` 语句以及其他查看执行计划的方法。

但是，如果无法理解显示的内容，那么展示执行计划就没什么意义。我将在本章中解释解读执行计划的过程。不幸的是，有些执行计划极其复杂，因此在介绍更多基础知识后，我将在第 8 章中探讨更高级的执行计划解读主题。

回到本章，让我们从查看保存执行计划信息的各种视图开始。

## 显示执行计划

执行计划可以在 Oracle 数据库的多个位置找到，但就本书目的而言，以下三个位置最为重要。

*   `SYS.PLAN_TABLE$`：这个系统拥有的全局临时表通常通过公共同义词 `PLAN_TABLE` 引用，它包含 `EXPLAIN PLAN` 语句的结果。
*   `V$SQL_PLAN`：此视图显示当前正在运行或最近完成运行的语句的执行计划。该视图显示的数据保存在共享池的*游标缓存*中。
*   `DBA_HIST_SQL_PLAN`：此视图显示过去执行过的执行计划的一个有用子集。该视图显示的数据保存在 AWR 中，由 MMON 后台进程从共享池中收集。

虽然可以直接查看该表和两个视图，但这样做并不方便。对于 `OTHER_XML` 列尤其如此，正如您可能猜到的，它包含未格式化的 XML。使用 `DBMS_XPLAN` 包中的函数要容易得多，这些函数以更易于阅读的格式呈现执行计划。在接下来的章节中，我将讨论如何解读该包中三个函数的默认输出；我将在第 4 章和第 8 章中探讨非默认输出格式。

## 显示 EXPLAIN PLAN 的结果

清单 2-2 创建了四个表，然后查询它们。该查询被显式编写为建议先连接 `T1` 和 `T2`；接下来连接 `T3` 和 `T4`；然后连接两个中间结果。然而，CBO 应用了 `简单视图合并`，并用粗体突出显示的四个表的简单连接替换了原始查询。我将在第 3 章详细解释这一点以及如何生成和分析执行计划，但现在您只需要关注整个查询的估计成本，该成本在计划的第 0 行报告为 293。清单 2-3 添加了一些提示。

清单 2-3。抑制了简单视图合并

```sql
WITH t1t2
     AS (SELECT /*+ NO_MERGE */
               t1.c1, t2.c2
           FROM t1, t2
          WHERE t1.c1 = t2.c2)
    ,t3t4
     AS (SELECT /*+ NO_MERGE */
               t3.c3, t4.c4
           FROM t3, t4
          WHERE t3.c3 = t4.c4)
SELECT COUNT (*)
  FROM t1t2 j1, t3t4 j2
 WHERE j1.c1 + j1.c2 = j2.c3 + j2.c4;
```

两个 `NO_MERGE` 提示抑制了简单视图合并转换，现在所选执行计划的成本为 11！这很有趣，不是吗？大多数人认为应用优化器提示的唯一原因是让 CBO 选择一个它认为不是最优但用户知道是最优的计划。然而，在这种情况下，CBO 认为带提示的计划比它自己选择的计划更好。它自己没有选择带提示的计划，因为简单视图合并转换的无条件性质意味着最优计划甚至没有被考虑！查询在实际运行中表现如何？所有这些关于 CBO 成本计算的讨论都非常有趣，但归根结底，您最感兴趣的是实际的耗时，而不是成本估计。事实上，在这次的成本算法并没有太大问题。如果您尝试运行这些语句，很可能会发现带提示的查询运行速度比不带提示的快得多，但可能不像成本估计所暗示的那样快 26 倍。
