# 使用 `V$OBJECT_USAGE` 识别未使用的索引

作为解决高开销`DML`的短期修复方法，你可以寻找未被使用的索引。这可以通过在表的所有索引或模式（schema）的所有索引上启用索引监控来实现，使用`ALTER INDEX ....MONITORING USAGE`命令。该命令会在视图`V$OBJECT_USAGE`中创建一行。`V$OBJECT_USAGE`视图包含一个`USED`列，该列由`ALTER INDEX`命令初始化为`NO`值，当索引在执行计划中被使用时，它会被更新为`YES`。其思路是，如果`USED`列的值永远保持为`NO`，那么该索引就没有被使用，因此可以安全地删除。当然，“永远”是很长的时间，但你的系统在没有使用该索引的情况下运行得越久，你对该索引永远不会被当前版本的应用程序使用的置信度就越高。

关于`V$OBJECT_USAGE`，你应该记住以下几点：

*   你只能在`V$OBJECT_USAGE`中查看你拥有的索引。换句话说，你必须作为模式所有者登录才能使用`V$OBJECT_USAGE`视图。
*   对被监控索引进行统计信息收集不会导致`V$OBJECT_USAGE`中`USED`列的值发生变化。
*   与动态性能视图不同寻常的是，重启数据库不会导致`V$OBJECT_USAGE`中任何索引的`USED`列值发生变化。
*   通过`DML`添加或删除索引条目不会导致`V$OBJECT_USAGE`中`USED`列的值发生变化。
*   重建或合并索引不会导致`V$OBJECT_USAGE`中`USED`列的值发生变化。
*   当索引用于强制执行主键、唯一键或外键约束时，`V$OBJECT_USAGE`中`USED`列的值不会改变。

最后一点值得强调：一个索引在执行计划中未被使用，并不意味着它没有履行其职责，因此在审查你考虑删除的索引时要格外小心。

## 移除外键约束列上的索引

在 20 世纪 90 年代，一个关于所谓*最佳实践*的神话完全失控了。这个神话就是，*总是*需要在外键约束的引用列上创建索引。这个神话影响如此之大，以至于许多数据库设计工具会自动生成`DDL`来在这些列上创建索引。

以`OE`示例模式中`ORDER_ITEMS`表的`ORDER_ID`列为例——该列被`ORDER_ITEMS_ORDER_ID_FK`外键约束使用，该约束引用了`ORDERS`表中的`ORDER_ID`列。现在，确实有必要在`ORDERS`表的`ORDER_ID`列上创建索引，因为`ORDER_ID`是`ORDERS`表的主键，但**没有必要**在`ORDER_ITEMS`表的`ORDER_ID`列上创建索引。

那么，*总是*应该在像`ORDER_ITEMS.ORDER_ID`这样的列上创建索引的神话是如何盛行的呢？嗯，如果你从`ORDERS`表中`DELETE`行（或者你用`UPDATE`或`MERGE`语句更改主键），那么除非`ORDER_ITEMS.ORDER_ID`列上有索引，否则将需要一个表锁。但是，如果你不执行这样的`DELETE`语句，或者`DELETE`语句的性能并不关键，那么你就不需要这个索引。

别误会我的意思——尽管`ORDER_ITEMS`中的`ORDER_ID`列被用于外键约束，无论如何索引该列可能仍然是个好主意。如果你使用两个表上的`ORDER_ID`列将`ORDERS`与`ORDER_ITEMS`连接起来，并且你希望从`ORDERS`表驱动一个嵌套循环进入`ORDER_ITEMS`，那么在`ORDER_ITEMS.ORDER_ID`上创建索引可能是个好主意。事实上，`OE.ORDER_ITEMS`表确实有一个名为`ITEM_ORDER_IX`的索引位于`ORDER_ID`上。

但是，所有这些删除未使用索引的操作都只是一个快速修复。最好的做法是透彻地理解你的索引需求，并只保留那些必要的索引。
