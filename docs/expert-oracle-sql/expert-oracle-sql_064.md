## 对象统计信息

尽管 CBO 使用来自各种来源的数据来决定如何优化一条语句，但 CBO 所需的大部分重要信息来自对象统计信息。本章大部分内容致力于解释对象统计信息，不过我们将在本章结尾简要谈谈 CBO 使用的其他信息来源。

原则上，并且在很大程度上在实践中，将对象统计信息用作优化的基础是理想的。例如，我们需要某种方法来区分一个只有十行的小型配置表和一个拥有 TB 级数据的事务表——对象统计信息可以为我们做到这一点。请注意，如果 CBO 被设计为从表的实际大小获取输入，而不是使用单独的统计信息，我们将无法控制对 CBO 输入数据的更改；每次我们向表添加一行时，语句的执行计划都可能改变。

我们可以用两种方式对对象统计信息进行分类：按对象类型和按级别。我们有 `表统计信息`、`索引统计信息` 和 `列统计信息`，并且对于每个表、列或索引，我们可能在 `全局`、`分区` 或 `子分区` 级别保存数据。

![image](img/sq.jpg) **注意** 如果你使用 `域索引`，你可以创建自己的对象统计信息类型。详情请参阅《Data Cartridge Developer’s Guide》。还有各种与用户定义对象类型和对象表相关的统计信息。相关信息请参阅《Object-Relational Developer’s Guide》。本书将不涵盖域索引或对象关系概念。

本章将解释每个统计信息是什么以及它是如何使用的，并解释为什么我们有三种不同级别的统计信息。但我想首先谈谈对象统计信息应用的一些一般性要点，并解释它们最初是如何获得的。

### 对象统计信息的目的

让我们假设 CBO 正在考虑使用全表扫描（`FTS`）访问一个表。CBO 需要估算三个关键指标：

*   **成本。** 换句话说，CBO 需要知道 `FTS` 可能需要多长时间。如果很快，那么 `FTS` 可能是个好主意；如果耗时很长，那么可能就不是。
*   **基数。** 在决定是执行 `FTS` 还是使用索引访问表时，返回的行数并不直接相关；无论我们如何访问表，返回的行数都是相同的。另一方面，输入到*父*操作的行数可能至关重要。例如，假设 `FTS` 的结果随后被用作与另一个行源进行连接的驱动行源。如果 `FTS` 只返回一两行，那么嵌套循环可能是个好主意。如果 `FTS` 返回 10,000 行，那么可能哈希连接会更好。`FTS` 的基数可以分两步确定。首先，我们需要知道表中的**行数**，其次，我们需要知道谓词的**选择率**。因此，例如，如果一个表有 1,000 行，我们有 10% 的选择率（也可以表示为 0.1 的选择率），那么全表扫描返回的行数将是 1,000 的 10%，即 100。
*   **字节数。** 与基数类似，无论使用 `FTS` 还是索引，表访问返回的字节数都是相同的。然而，如果 CBO 想要对两个各有 10,000 行的表执行哈希连接，那么将根据两个表中较小的那个（以字节衡量）来选择驱动行源。

尽管我用 `FTS` 作为例子，但同样的三个指标对所有行源操作都至关重要，而对象统计信息的主要目的就是帮助 CBO 尽可能准确地估算它们。
