# 有几个初始化参数你可以随意更改。这些参数是：

*   `DB_FILE_MULTIBLOCK_READ_COUNT`
*   `PARALLEL_DEGREE_POLICY`
*   `OPTIMIZER_FEATURES_ENABLE`
*   `OPTIMIZER_ADAPTIVE_FEATURES`
*   `OPTIMIZER_ADAPTIVE_REPORTING_ONLY`

`DB_FILE_MULTIBLOCK_READ_COUNT` 的默认行为，说得委婉些，有点令人困惑。你应该将此参数显式设置为你的平台和块大小所允许的最大 I/O 大小。对于大多数平台，最大 I/O 大小是 1MB，这意味着在块大小为 8K 的情况下，该参数的最佳值是 128。通过显式设置此参数，你可以让 CBO 认为多块读是高效的——而事实也确实如此。

在 第 2 章 中，我解释了如何通过将 `PARALLEL_DEGREE_POLICY` 设置为 `AUTO` 来设置自动并行度。我对这一功能的体验极其正面，如果将来某个时候 `AUTO` 成为 `PARALLEL_DEGREE_POLICY` 的默认设置，我一点也不会感到惊讶。

如果你将数据库升级到新版本，并且没有机会进行彻底测试（或者你做了测试但遇到了问题），你可以将 `OPTIMIZER_FEATURES_ENABLE` 设置为非默认值，以禁用新的 CBO 功能，并将这些新 CBO 功能干扰执行计划的风险降至最低。

最后，我只想提一下随 12cR1 提供的两个新的初始化参数。将 `OPTIMIZER_ADAPTIVE_FEATURES` 设置为 `FALSE`，可以关闭 12cR1 引入的所有新自适应功能，这些功能我在 第 6 章 讨论过。作为更改 `OPTIMIZER_ADAPTIVE_FEATURES` 的替代方案，你可以将 `OPTIMIZER_ADAPTIVE_REPORTING_ONLY` 设置为 `TRUE`。采用此设置后，会收集自适应优化所需的信息，但不会采取任何行动来更改计划。换句话说，自适应计划将始终选择默认（由优化器选择的）计划，但会收集信息，以了解在非报告模式下应适应哪个计划。我必须承认，我目前的经验尚不足以对如何设置这些参数给出明确建议，但我的初步感觉是，至少生产系统通常应运行在 `OPTIMIZER_ADAPTIVE_FEATURES` 设置为 `FALSE` 的状态下，直到自适应功能更加成熟。

以防你好奇，我并不建议将 `所有` 初始化参数都保留为默认值！不，你的数据库不一定非要命名为 `orcl`，而且你可能需要更改 `SGA_TARGET` 或 `MEMORY_TARGET` 的值！我在本节中的评论纯粹是关于刻意尝试影响 CBO 的行为。

## 系统统计信息

哪个耗时更长：一次 1MB 的多块读还是 40 次顺序发出的单块读？理论上，该问题的答案取决于你的硬件。实际上，考虑到企业存储上可能存在的缓存，你的答案也可能取决于工作负载。理论上，CBO 可能需要知道这个问题的答案。但 CBO 不仅需要在多块 I/O 和单块 I/O 之间做出选择；它偶尔还需要在例如内存中的 CPU 密集型聚合和读取物化视图的磁盘访问之间做出选择。CBO 再次希望了解 CPU 和磁盘子系统的相对能力。

于是就有了系统统计信息。系统统计信息在 10gR1 中引入，并受到了适度的欢迎。然而，我对专家进行的相当不科学的民意调查表明，专家意见的钟摆已经非常坚定地摆向了反对使用系统统计信息的一方。首先，执行计划因系统统计信息的存在而改变的可能性极小，即使真的改变，你也不太可能看到因此带来的性能大幅改善。

那些劝阻人们不要通过调整初始化参数来影响 CBO 的论点，同样可以用来劝阻人们不要收集或设置系统统计信息：你遇到某种错误或其他差异的风险，超过了使用系统统计信息可能带来的任何好处。我的建议是调用 `DBMS_STATS.DELETE_SYSTEM_STATS`。这个过程调用将恢复所谓的 `默认空载统计信息`，地球上大多数 Oracle 数据库都在使用这些统计信息。

如果你确实选择使用非默认的系统统计信息，至关重要的是使用 `DBMS_STATS.EXPORT_SYSTEM_STATS` 和 `DBMS_STATS.IMPORT_SYSTEM_STATS` 将这些非默认的系统统计信息从你的性能测试系统复制到生产系统。否则，你的生产系统可能会选择与你精心测试过的计划不同的执行计划。

## 其他数据字典信息

显然，CBO 需要知道存在哪些表和索引，以及这些表和索引中有哪些列。这类模式信息、与之相关的对象统计信息，以及任何可能已收集或设置的系统统计信息，都保存在数据字典中。然而，CBO 可能从数据字典中提取的信息至少还有另外两部分。

我在 第 2 章 提到过，当 `PARALLEL_DEGREE_POLICY` 处于默认值 `MANUAL` 时，你可以通过指定查询要使用的并行度来修饰表或索引。此信息保存在数据字典中，不被视为对象统计信息。CBO 也可能对表中的分区数量感兴趣。我们将在 第 20 章 回到最后这一点。我在此提及只是为了完整性。

## 小结

对象统计信息是一个复杂的主题，这冗长的一章仍然只提供了高层次的概述。重要的是要认识到，我们需要的统计信息是能让 CBO 做出合理且可供我们测试的行为的统计信息，而不一定是反映现实的对象统计信息，因此有时需要对统计信息进行操作。另一方面，如果你手动更改了统计信息，你应该非常小心，确保统计信息是 `自洽的`。例如，你不应该将索引的 `NUM_ROWS` 统计信息设置为大于表的 `NUM_ROWS` 的值。这种行为很可能会引导 CBO 走向无人测试过的代码路径。

本章结束了本书涉及高级概念的部分，现在是时候进入 第 3 部分，更详细地探讨 CBO 的工作了。第 10 章 通过介绍 CBO 必须选择的访问方法来开启这一部分。
