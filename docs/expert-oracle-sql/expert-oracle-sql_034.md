## 基于成本的优化器

第 1 章介绍了`CBO`。`CBO`的工作是确定如何着手运行你的 SQL 语句。执行 SQL 语句的方法被称为*执行计划*，对于非常简单的语句，可能只有一种方式来完成任务。因此，可能只有一种执行计划。然而，对于绝大多数 SQL 语句，存在不同的可能执行计划，`CBO`必须从中进行选择。在英语中，*优化*意味着从多个选项中挑选最佳方案，而在这里，`CBO`正试图从它识别出的所有可能计划中挑选最优的执行计划。

## 基于规则的优化器

当数据库已挂载但未打开时，查询不能使用`CBO`，而使用基于规则的优化器（`RBO`）。在其他情况下也可以使用`RBO`，但自 Oracle 数据库 10gR1 版本发布以来，`RBO`已不再支持通用用途。因此，在本书中我将不再进一步讨论`RBO`。

`CBO`的绝大部分工作围绕优化查询或嵌入在`DML`或数据定义语言（`DDL`）语句中的子查询展开。对于`DML`和`DDL`语句，还有一两件其他事情需要考虑，例如是执行直接路径插入还是常规插入，但这些通常由相对固定的规则决定。为简单起见，我将忽略这些，并继续假设`CBO`的存在纯粹是为了优化查询。

本章首先讨论`CBO`用来确定最优执行计划的标准。之后，我将介绍成本的概念，这是`CBO`量化一个计划相对于另一个计划的收益的方式。最后，我将简要概述`CBO`用来识别和评估其必须从中选择的一组计划的过程。我将在本书后面重新讨论本章介绍的主题，并进行更详细的阐述。

## 最优执行计划

那么，如果优化是关于挑选最优执行计划，理解什么是最优执行计划就很重要。幸运或者说不幸的是，`CBO`对什么构成最优有一个非常简单的看法：最优执行计划是运行最快的那一个。就是这样。嗯，*几乎*就是这样。如果优化器使用`FIRST_ROWS`优化器模式之一运行，那么最优计划是最快返回前几行的计划，而不是最快返回所有行的计划。尽管如此，关键在于`CBO`不会让自己背负对资源消耗等问题的担忧。因此，例如，如果一个串行执行计划估计需要 4.98 秒完成，而一个涉及 128 个并行查询从属进程的计划估计需要 4.97 秒完成，那么`CBO`将选择后者！也就是说，如果你允许它这样做的话。我将在本章稍后讨论优化过程，作为该讨论的一部分，我将回到并行操作的主题。

## 成本的定义

因此，如果最优执行计划是最早完成的那个，那么`CBO`需要估计每个计划运行所需的时间，然后选择预测执行时间最短的那个。这个估计的执行时间被称为计划的*成本*。现在你可能会假设执行时间的单位可能是纳秒、毫秒、秒或类似的标准度量。不幸的是，由于历史原因，成本的单位被定义为单个块读取完成所需的时间长度！所以，如果`CBO`认为一个查询需要 5 秒运行，并且单个块读取需要 10 毫秒，那么分配的成本将是 500，因为这是 5,000 毫秒除以 10 毫秒。

### **CBO 成本与实际成本**

本书中使用的术语*成本*专指`CBO`对 SQL 语句执行时间的估计。当然，这个估计可能不准确。重要的是要认识到，在调优（即试图超越`CBO`的人工优化过程）时，目标是减少语句运行的实际耗时。只要语句运行速度快，`CBO`的成本估计有多高并不重要；同样，如果语句运行时间很长，即使成本估计很低，也不会令人满意。

为了得出执行计划的成本，`CBO`不仅仅是估计单块读取的数量。它还试图计算计划中任何多块读取操作需要多长时间，以及中央处理器（`CPU`）相关操作（如内存排序）需要多长时间，并将这些时间纳入总成本。然而，奇怪的是，它将所有这些时间都转换为等效的单块读取。这里有一个例子。假设一个特定计划估计涉及：

*   400 次单块读取
*   300 次多块读取
*   5 秒的`CPU`处理

让我们进一步假设`CBO`估计：

*   一次单块读取需要 10 毫秒
*   一次多块读取需要 30 毫秒

该计划的成本计算如下：

```
((400 x 10) + (300 x 30) + 5,000)/10 = 1800
```

如今，执行计划通常打印时会同时显示成本和以小时、分钟、秒为单位的估计耗时，所以你不必像以前那样做那么多的数学运算了！
