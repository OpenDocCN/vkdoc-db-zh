# 提示

*   `FS` 是因子化子查询的名称，允许主查询像引用表一样引用它。
*   `QB1` 是查询块的名称。这是由 `QB_NAME` 提示分配的，并且仅与全局提示相关。
*   `MYALIAS` 是在主查询的 `FROM` 子句中分配的对象别名。其主要目的是限定主查询块中的列名，但它在 `NO_MERGE` 提示中也有用途。对象别名的存在使得进一步引用标识符 `FS` 变得非法。

考虑到这一点，让我们依次看看这四个提示。

*   在第一个查询中，`NO_MERGE` 提示没有参数。它是一个局部提示，指示 CBO 不要合并出现此提示的块。与任何局部提示一样，它必须放置在它所适用的块中，本例中是子查询。
*   第二个查询也使用了局部 `NO_MERGE` 提示，但这次提供了一个参数。该参数指定了不允许合并的行源的对象别名。因此，正确的参数值是 `MYALIAS`。与任何局部提示一样，它必须放置在它所适用的块中，本例中是主查询，因为行源出现在主查询的 `FROM` 子句中。
*   第三个变体使用全局提示语法。提供的查询块名称 `QB1` 表明该提示应像在 `QB1` 中的局部提示一样操作。由于除了查询块名称外没有其他参数，该提示的操作与第一个变体完全相同，只是提示可以出现在语句中的任何位置。
*   第四个变体也是一个全局提示。提供的查询块名称 `QB2` 表明该提示应像放置在 `QB2` 中的局部提示一样操作。第四个变体中的第二个参数 `MYALIAS` 被视为第二个变体中的第一个参数。与任何全局提示一样，它在语句中的位置无关紧要——该提示始终作用于主查询块 `QB2`。

现在你是否感到困惑了？别担心。当然，你非常明智，不会使用第四个提示变体，但请保留本页内容，以便当你遇到其他人难以理解的提示时参考！

## 总结

本章涵盖了三个重要的相关主题。我讲解了通过使用非默认格式化参数调用 `DBMS_XPLAN` 函数而获得的执行计划中大部分先前未解释的部分。我还解释了如何解读并行运行的语句的执行计划，并阐述了我对全局提示的看法。

全局提示让我们得以一窥出现在 `DBMS_XPLAN` 函数输出中、并构成 Oracle 计划稳定性功能基础的概要（outline）是如何运作的。全局提示还显著提高了我们在不使业务逻辑难以阅读或不影响数据字典视图其他用户的情况下控制 CBO 行为的能力。

虽然 SQL 编程新手可能从不使用提示，普通 SQL 编程员也只是偶尔使用局部提示，但通常在所有人都放弃时，才会请来专家的调优专家。对于这个人来说，掌握并行执行计划和全局提示技巧，对于理解 CBO 的意图以及让它做正确的事情至关重要。

但为什么需要所有这些复杂的操作呢？在第 9 章中，我们将探讨对象统计信息，它们是 CBO 决策过程的主要输入。理解什么是对象统计信息以及 CBO 如何使用它们，将使我们走上理解 CBO 为何经常能做对、而有时又让我们失望的道路。
