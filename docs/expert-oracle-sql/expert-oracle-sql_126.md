# 排序更少的列

既然我们已经确保只在需要时才排序，就可以开始专注于优化剩余的排序。我们可以采取的一个步骤是尽量减少排序的列数。排序的列越少，我们的排序就越有可能适应工作区。让我们从对未建立索引的列排序开始，看看我们能做什么。

## 利用 ROWID

想象你在汽车展厅工作，想按价格顺序物理排列五十辆汽车。你会开车移动它们来试图排好顺序吗？可能不会。你可能会把汽车的名称和价格写在纸片上，然后对这些纸片排序。一旦你把纸片排序好，就可以按正确的顺序一辆一辆地挑选实际的汽车。

有时，你可以在 Oracle 数据库中使用类似的方法对行进行排序。如果行非常宽而排序键相对较窄，最好将排序不需要的列替换为一个 `ROWID`。`ROWID` 排序后，你可以使用这些 `ROWID` 来检索表中的其余列。

是时候看一个例子了。代码清单 17-11 展示了一个代价相当高的排序，它是由一个编写得相当自然的查询产生的。

代码清单 17-11. 按未建立索引的列排序

```sql
 SELECT /*+ cardinality(s 1e9) */
         *
    FROM sh.sales s
         JOIN sh.customers c USING (cust_id)
         JOIN sh.products p USING (prod_id)
ORDER BY cust_id;

| 标识  | 操作              | 名称      | 行数  | 字节数 |临时空间| 成本 |
|-----|------------------------|-----------|-------|-------|-------|------|
|   0 | SELECT STATEMENT       |           |  1000M|   364G|       |    59|
|   1 |  MERGE JOIN            |           |  1000M|   364G|       |    59|
|   2 |   SORT JOIN            |           |  1000M|   188G|   412G|    59|
|*  3 |    HASH JOIN           |           |  1000M|   188G|       | 21907|
|   4 |     TABLE ACCESS FULL  | PRODUCTS  |    72 | 12456 |       |     3|
|   5 |     PARTITION RANGE ALL|           |  1000M|    27G|       | 12955|
|   6 |      TABLE ACCESS FULL | SALES     |  1000M|    27G|       | 12955|
|*  7 |   SORT JOIN            |           | 55500 |    10M|    25M|  2709|
|   8 |    TABLE ACCESS FULL   | CUSTOMERS | 55500 |    10M|       |   415|
```

代码清单 17-11 在 `SH.SALES` 事实表和两个维度表之间进行了自然连接。生成的宽行随后被排序。我再次使用了基数提示来尝试模拟一个更大的事实表。如你所见，CBO 估计我们需要 412.025GB 的临时表空间来执行两次排序。CBO 使用了 `MERGE JOIN`，因为这种连接的结果会自动排序，并且我们的大连接不需要来自 `SH.CUSTOMERS` 表的列。聪明，是的，但重大胜利？不是。
