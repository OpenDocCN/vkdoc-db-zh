# 服务消费者与可预测的性能

IT 服务的消费者与其他任何服务的消费者并无不同。他们想要**保证**。这些保证体现在服务水平协议（SLA）中，这些协议可能是正式文件，也可能是非正式的理解。这些 SLA 不会写上类似“批处理任务必须在 90%的情况下在一小时内完成”甚至“批处理任务在一个月内的平均完成时间必须少于一小时”这样的话。这类性能指标对任何服务消费者都毫无用处。不，SLA 总是表述为体现这种情绪：批处理任务“绝对、必须在一小时内完成”。

在绝大多数情况下，消费者并不关心一小时的批处理任务是在 15 分钟还是 59 分钟内完成，就像你不会关心你的信件是在凌晨 4:30 送达一样。IT 服务的商业消费者极少会打电话给在家的员工说：“嘿，隔夜批处理提前完成了，为什么不赶早班车提前开始工作呢？”服务消费者根据保证的性能水平制定计划，通常无法利用更高水平的性能（如果它们偶然出现）。

当然，一小时批处理任务的**服务提供者**对 15 分钟和 59 分钟运行时间的差异非常关注。前者有充足的应急时间来处理意外事件，而后者则完全没有。尽管如此，消费者的需求是第一位的，只有在建立了可预测的性能概况后，才能努力减少批处理的耗时。

## 非数据库部署策略

鉴于服务消费者的要求，IT 行业已经建立了一些技术，用以确保 IT 服务的功能和性能水平是可预测的。目前存在多种软件开发方法，但它们都有几个共同的属性：

*   在交付给一般用途消费之前，新功能会以某种方式进行测试，以提供信心，证明软件功能正确且性能水平令人满意。
*   涉及生产服务的软件或关键配置数据的变更，会以某种方式进行控制以最小化风险。通常，这些控制涉及测试、影响分析、授权和审计的某种组合。

所有这些流程和程序都是为了做一件基本的事情；它们是为了让消费者绝对、肯定地获得他们所承诺的服务。当然，我们并不总是成功，但这至少是我们的目标。

## CBO 的战略方向

到目前为止，我一直在谈论向消费者交付服务。我详细地谈了这一点，并刻意避免提及数据库性能。正如你可能已经猜到的，所有这些铺垫是因为，在我看来，Oracle，特别是 CBO 开发团队所遵循的战略，并未反映出可预测服务交付的基本要求。

### 战略特性的历史

CBO 在 Oracle 7 中作为一个可选特性被引入。你不必使用它，我们大多数人也确实没有使用。其采用缓慢有两个原因。一些客户担心，像任何重大的新特性一样，CBO 起初可能工作得不好。他们说得有道理。采用缓慢的第二个原因是，CBO 在设计上就是不可预测的。你可能今天得到一个执行计划，然后在收集了对象统计信息后，你可能会得到一个完全不同的执行计划，这个计划可能工作得一样好，也可能不好。

旧的基于规则的优化器（RBO）的一个关键点是，相同的语句每次都会产生相同的执行计划。而 CBO 在设计上是**可适应的**。CBO 每次都试图获得最优的执行计划，如果对象统计信息表明数据已经改变，那么 CBO 会很乐意考虑一个新的执行计划。

尽管客户从一开始并持续至今都在提出对可预测性的担忧，但 CBO 的战略方向仍然基于适应性。动态采样和绑定变量窥探等特性开始出现。当 11g 到来时，很明显 CBO 团队正朝着更加适应和更不可预测的行为迈进。这些信号以 11gR1 中的**自适应游标共享**和 11gR2 中的**基数反馈**的形式出现。自适应游标共享和基数反馈都可能导致为语句第一次执行选择的执行计划在第二次及后续执行时被“升级”，即使对象统计信息没有改变！通常，这些改变是为了变得更好，但并没有**保证**。

好像这还不够糟糕，12cR1 引入了**自适应执行计划**。这个特性可能导致一个语句的执行计划在中途改变！我们不想要这些特性！

*   在测试系统上，我们不想要这些特性，因为它们可能掩盖会影响生产服务的问题。
*   在生产系统上，我们不想要这些特性，因为无论 CBO 怎么想，我们已经测试过的执行计划是令人满意的；我们不想冒险使用未经测试的、可能导致性能下降的计划。

### CBO 战略的影响

既然我们知道了 CBO 开发团队的战略是什么，现在让我们看看它对我们意味着什么。我将通过两位著名 Oracle 专家：Connor McDonald 和 Dave Ensor 的评论来探讨这一点。

**Connor McDonald 的观点**

如果你问谁是 Oracle 数据库技术方面知识最渊博的专家，你可能会很快发现自己陷入一场争论。有时甚至是相当激烈的争论！然而，世界上最具娱乐性的 Oracle 演讲者肯定是 Connor McDonald。如果你提出另一个名字，你可能没听过 Connor 的演讲！哦，以防你好奇，Connor 对 Oracle 数据库技术也略知一二！

我有幸只听过 Connor 演讲两次，在第一次演讲中，他讨论了一个话题，正好是我们本章讨论问题的核心。Connor 以对软件程序的一个非常简单的修改建议开始了他的演示。一个下拉列表有两个条目：“MALE”和“Female”。“MALE”全是大写字母，但“Female”只有首字母大写。一位开发者希望通过将“MALE”改为“Male”来纠正这一点。需要做什么？嗯，Connor 列举了一长串虚构开发者可能需要完成的任务：编写商业案例、获得商业案例批准、生成测试计划、进行单元测试、子系统测试和系统测试、记录测试结果、通过质量保证审查……这个列表没完没了。在此期间，Connor 的笔记本电脑不断弹出会议、文档和各种其他事物的图片，为了不破坏你观看他演讲的乐趣，我就不一一提及了！当 Connor 讲完他的列表时，我和大多数听众一样，笑得相当大声。毕竟，我们大多数人都曾在某个时刻经历过类似的事情！



Connor 随后的演讲则采取了不同的思路。他提出了一个虚构的服务影响事件及其后续的复盘分析。该事件由一个变更的执行计划引起，而执行计划的变更又源于对象统计信息的变化。什么！一个导致事件的变化！这个变更经过授权了吗？有变更记录吗？该追究谁的责任？

当然，这个虚构的执行计划变更是由一个显式运行的、用于收集统计信息的作业引起的。没有授权，没有影响分析，没有测试，什么都没有。Connor 提出了一个非常合理的问题：为何 IT 组织总是会贸然引发未经测试的执行计划变更来影响他们的数据库系统，与此同时，却将大多数其他类型的变更置于最严格的审查之下。在 Connor 看来，这个世界简直疯了。我深表同意！

## **戴夫·恩索尔悖论**

如果说**康纳·麦克唐纳**是世界上最有趣的 Oracle 演讲者，那么我会把“最令人难忘的精辟言论”奖颁给**戴夫·恩索尔**！我不知道戴夫最初是在何时何地说出“唯一安全收集统计信息的时候就是收集它也不会改变什么的时候”这句话的。无论他是在何时何地说出这句如今闻名于世（或者说声名狼藉）的话，它现在已被全球业界称为“戴夫·恩索尔悖论”。

我必须立刻指出，戴夫的断言完全是错误的，我相信戴夫本人也会第一个承认这一点。然而，他的“悖论”中有两个引人深思的方面值得讨论。第一个方面是他原始断言的依据。

当你收集统计信息时，你的 SQL 语句的执行计划可能会发生两种情况。第一种情况是**什么都没发生**；基于新统计信息的执行计划与上次语句运行时使用的执行计划完全一致。既然没有任何变化，你显然没有造成任何损害。第二种情况是**执行计划发生了变化**。我们无法确定新计划在实际中会如何表现，因此当结果是后者时，收集统计信息就不安全了。这是一个有力的论点。

戴夫·恩索尔悖论的第二个有趣之处在于其论证中存在的致命缺陷。我们很快就会谈到这一点。

这个实验得出了一个惊人的结论。接下来的这句话是本书中最重要的一句。

我们在生产系统上收集统计信息的原因是为了**防止**执行计划发生变化，而不是为了确保执行计划**发生**变化。

让我们暂停一下，思考一下这句话的巨大含义。现在可以看出 Dave Ensor 悖论中的缺陷了。有时，收集统计信息所带来的“差异”是为了阻止一个原本会发生的执行计划变更！因此，Dave Ensor 悖论的正确版本应该是：

> 收集统计信息唯一安全的时刻，是当这样做不会导致执行计划与上次 SQL 语句运行时相比产生任何变化。

听起来就不那么顺耳了，对吧？我们现在的处境似乎很绝望。一方面，如果我们在生产系统上不收集统计信息，那么迟早——就像鸡蛋注定会孵出小鸡一样——执行计划会发生变化，而且几乎肯定会变得更糟。另一方面，如果我们真的收集统计信息，那就像在玩俄罗斯轮盘赌；迟早某个表、列或索引统计信息的某种变化会导致一个不期望的执行计划变更，从而引发影响服务的事故。我们该怎么办？在回答这个问题之前，我们还需要理清一两件事。

## 我们需要多久改变一次执行计划？

Connor McDonald 和 Dave Ensor 都隐含地假设：可预测的性能来自于不变的执行计划。这真的正确吗？如果我们的数据库在增长，表的大小发生变化，难道我们不需要在某个时间点改变我们语句的执行计划吗？Wolfgang Breitling 可以帮助我们回答这个问题。

## Wolfgang Breitling 的基于基数反馈的调优

2006 年，Wolfgang Breitling 发表了一篇题为“基于基数反馈的调优 (Tuning by Cardinality Feedback, TCF)”的论文。你可以在这里找到他的论文：`http://www.centrexcc.com/Tuning%20by%20Cardinality%20Feedback.pdf`。这篇论文可能是迄今为止对 Oracle SQL 调优科学的最大单一贡献。

![image](img/sq.jpg) **注意** 我需要指出，我之前提到的 11gR2 基数反馈功能与 Wolfgang 的论文绝对没有任何关系。不过，我想 Wolfgang 对 Oracle 似乎用他的发明来命名一个功能感到暗自高兴！

简而言之，Wolfgang 的论点如下：

*   如果 CBO 能够正确确定执行计划中操作的基数，那么 CBO 将得出最优的执行计划。
*   当 CBO 产生的执行计划比最佳可能计划运行得慢很多时，是因为一个或多个基数估计值偏离了*好几个数量级*；偏离两倍或三倍的基数估计可能导致错过最优执行计划，但所选计划的性能几乎与最优计划相当。
*   要确定一条语句的最优执行计划，我们只需为 CBO 提供正确的基数估计，剩下的 CBO 会处理好。

虽然这些说法并不总是成立，但在大多数情况下是正确的！现在，Wolfgang 的论文是关于 SQL 调优的，所以我在专门讨论对象统计信息和部署的章节中提到它可能显得有些奇怪。我之所以这样做，是因为 TCF 推论。

## TCF 推论

让我们假设执行计划中的一个操作实际返回 10,000 行。TCF 说的是，如果 CBO 估计值是 5,000 或 20,000，那么无需担心；CBO 选择的计划会是合理的。只有当 CBO 估计值低于 100 或高于 1,000,000 时，CBO 才可能得出一个性能非常差的计划。

这个论点的推论是，如果一个表的大小翻倍或翻三倍，则不需要改变执行计划。假设我们为一个有 10,000 行的表收集了统计信息，CBO 根据准确的基数估计 10,000 得出了一个计划。现在假设表增长到 20,000 行。TCF 说，当 CBO 使用基于（现在不准确的）基数估计 10,000 的计划时，它仍然可以正常工作！这是理论。理论在实践中并不总是有效，但在这个情况下，我很高兴地报告它确实有效！

让我澄清一点。假设我们有一条 SQL 语句连接了 `T1` 和 `T2`，然后对结果进行聚合。一方面，如果 `T1` 和 `T2` 的大小都翻倍了，那么这条 SQL 语句几乎肯定会运行得更慢。另一方面，通过改变执行计划来避免这种耗时增加是不太可能的；无论最初的最优执行计划是什么，在表大小增加后它很可能仍然是最优的，或接近最优。



在**绝大多数**现实案例中，数据库表通常不会频繁地增长（或缩减）100 倍或 1000 倍；我们可能多年都使用同一套执行计划，而不会出现任何与执行计划相关的性能问题！此原则的主要例外是将空表部署到生产环境后其数据快速增长的情况，例如我在第 5 章中提到的`AUDIT_TRAIL`表。本章稍后，在理清主要原则后，我们将处理这个例外。

## 并发执行计划

我们现在已确定，如果一个执行计划某天运行良好，那么第二天它很可能也会运行良好。但是，对于每条 SQL 语句，我们是否总能始终只依赖一个执行计划？让我们看几个似乎表明我们无法这样做的案例。

## 偏斜数据与直方图

许多数据库表包含偏斜数据。我指的是，某一列中有限数量的数值出现频率非常高，而其他数值的出现频率则低得多。假设贵公司 60%的业务是与一家名为 AJAX 的公司进行的，其余 40%的业务是与其他 100 家客户（包括 ACME）进行的。从表中选择 AJAX 所有交易的查询很可能最适合采用全表扫描。另一方面，选择 ACME 所有交易的查询可能更适合使用索引，因为只有大约 0.4%的交易会被选中。为了使 CBO 能够区分 AJAX 客户与其他所有客户，可以使用`直方图`。

这里需要认识到的主要点是，绑定变量和直方图应被视为**互斥的**。如果你使用绑定变量的一个值（例如我的示例中的 AJAX）来解析语句，那么当使用该绑定变量的其他值（例如 ACME）时，可能会应用该执行计划。为了防止这种情况，你应该确保为你需要的每个执行计划使用不同的`SQL_ID`。清单 6-6 展示了如何实现这一点：

### 清单 6-6. 使用多个 SQL_ID 以支持多个执行计划

```
CREATE OR REPLACE PROCEDURE listing_6_6 (p_cust_last_name VARCHAR2)
IS
   cnt   NUMBER;
BEGIN
   IF p_cust_last_name ='AJAX'
   THEN
      SELECT COUNT (*)
        INTO cnt
        FROM sh.sales s JOIN sh.customers c USING (cust_id)
       WHERE c.cust_last_name = 'AJAX'; -- 为 AJAX 使用一个执行计划
   ELSE
      SELECT COUNT (*)
        INTO cnt
        FROM sh.sales s JOIN sh.customers c USING (cust_id)
       WHERE c.cust_last_name = p_cust_last_name; -- 为其他所有人使用另一个执行计划
   END IF;
END listing_6_6;
/
```

上面的过程不执行任何实际操作，但展示了使用直方图时正确的编码方式。PL/SQL 过程中的两条 SQL 语句将具有不同的`SQL_ID`，因此，两个不同的执行计划可以同时保存在游标缓存中。

在 11gR1 中，Oracle 引入了自适应游标共享，它允许具有相同`SQL_ID`的两个或多个执行计划保存在游标缓存中。这可能会让你认为清单 6-6 中展示的复杂性不再需要。自适应游标共享的问题在于，一条 SQL 语句必须先至少**执行**得不好一次，才会考虑备选执行计划。如果你希望**每次**都能获得高性能的 SQL，那么你仍然需要使用具有独立`SQL_ID`的独立 SQL 语句。

## 工作负载变化

我还遇到过另一种情况，显然需要为同一条 SQL 语句使用多个执行计划。让我解释一下这个**极其罕见**的场景。

我当时在处理一个 ETL 流程，该流程将数据从远程数据库中的一张表`R1`加载到一张全局临时表`TEMP1`中。然后，使用一条`INSERT`语句将`TEMP1`与永久表`T1`连接，再将连接后的数据加载到`T2`中。该 ETL 流程有两种操作模式，有趣之处正在于此。

ETL 流程的正常操作模式称为`增量`。只有`R1`中更改的数据被加载到`TEMP1`中。`增量`加载每 15 分钟运行一次，加载到`TEMP1`的行数从零到几十行不等。第二种操作模式是`完全`。`完全`加载仅在特殊条件下执行，涉及将`全部`行从`R1`加载到`TEMP1`中。一次`完全`加载会向`TEMP1`添加数百万行，但无论是`增量`还是`完全`加载，都使用相同的`INSERT`语句将数据插入`T2`。

根据为`TEMP1`设置的统计信息或动态采样的结果，CBO 可能会选择两种可能的执行计划。第一种计划涉及由`TEMP1`驱动的`嵌套循环`连接，这对`增量`加载运行良好。不幸的是，当`完全`加载使用`嵌套循环`计划时，ETL 过程需要大约两天时间！备选执行计划是使用`哈希连接`。使用此计划进行`完全`加载时，它能在几小时内完成，考虑到其不常使用，这已经非常令人满意。不幸的是，当`哈希连接`用于`增量`加载时，该过程需要大约一小时，这对于设计为每 15 分钟运行一次的过程来说非常不合适！

现在，正如你可以想象的那样，`完全`加载和`增量`加载**从**不会同时运行，但它们偶尔会在几分钟内相继运行，因此我们偶尔会观察到相同的执行计划被用于不同的模式。我们本可以结合使用共享池刷新、动态采样和统计信息操作来解决这个问题，但依赖操作人员遵循文档是一个应尽可能避免的风险。清单 6-7 向你展示了我们在实践中所做的事情。

### 清单 6-7. 使用提示创建多个 SQL_ID

```
CREATE OR REPLACE PROCEDURE listing_6_7 (load_mode VARCHAR2)
IS
BEGIN
   IF load_mode = 'INCREMENTAL'
   THEN
      INSERT INTO t3 (c1, c2)
         SELECT /*+ leading(temp1) use_nl(t3) */
               temp1.c3, t3.c2
           FROM temp1, t3
          WHERE temp1.c1 = t3.c1;
   ELSE -- FULL
      INSERT INTO t3 (c1, c2)
         SELECT /*+ leading(t3) use_hash(temp1) no_swap_join_inputs(temp1) */
               temp1.c3, t3.c2
           FROM temp1, t3
          WHERE temp1.c1 = t3.c1;
   END IF;
END listing_6_7;
/
```

从代码的高亮部分可以看到，我们使用了优化器提示。现在不必担心这些提示的具体含义。此阶段的关键点是，这些提示起到了三个作用：

*   由于两条语句的`SQL_ID`现在不同了，来自`完全`加载的游标缓存中的执行计划不会被用于`增量`加载，反之亦然。
*   提示指示 CBO 使用哪个执行计划。在我们理清问题的其余细节时，我们知道每种情况需要什么计划，因此无需冒险让 CBO 出错。
*   这些提示提供了有用的文档说明，让后来者能够理解为何使用如此晦涩的逻辑。当然，代码中也添加了几段解释性注释。



在过去的几年里，我接触过大量基于 Oracle 技术的应用程序，其中每一个都可能包含数万条 SQL 语句。在那段时间里，我只遇到过一条确实需要这种特殊处理的 SQL 语句！不过，它确实是一个非常好的教学案例。

## 并发执行计划总结

似乎确实存在一些特殊情况，对于同一条 SQL 语句，需要不同的执行计划来应对所访问数据的变化。无论幸运与否，我们确保在不同场景下使用不同执行计划的唯一方法，就是重新编写我们的应用程序，以便在每种场景下使用具有不同 `SQL_ID` 的不同 SQL 语句。好消息是，一旦接受了这个现实，我们就有了一个可以自我实现的预言，并可以据此制定策略：每条 SQL 语句我们只需要一个执行计划！

## Oracle 的计划稳定性特性

尽管 CBO（基于成本的优化器）策略的核心是适应性，但 Oracle 和其他任何公司一样，必须采取一些措施来满足其最大且最直言不讳的客户的需求。以下是 Oracle 为应对客户对可预测性的关切而采取的战术性计划稳定性措施。

### 存储概要

对客户关于稳定执行计划需求的第一个回应是在 8i 版本中引入了存储概要。粗略地说，存储概要涉及一个执行计划的存储库，每个 SQL 语句对应一个计划。该存储库保存在 `OUTLN` 方案的表中。存储概要有一些缺点，但两个关键缺点是：

*   无法确定哪些存储概要不再需要，因为它们引用的 SQL 语句已被更改或删除。
*   更致命的是，无法确定应用程序中哪些语句缺少存储概要。

### SQL 配置文件

SQL 配置文件出现在 10g 版本中，与 `SQL 调优顾问` 紧密相关。尽管我在 Listing 5-7 中给出的例子涉及创建新索引，但大多数 `SQL 调优顾问` 的建议都与基数错误有关，通常只涉及采用不同的执行计划。为了启用新的执行计划，你需要“接受” `SQL 调优顾问` 为你创建的 SQL 配置文件。由于 SQL 配置文件会覆盖 CBO 原本会使用的执行计划，我们大多数人最初认为 SQL 配置文件是某种计划稳定性的方案。但它们并不是！即使你接受了 SQL 配置文件，下次收集统计信息时，那个原始的、性能不佳的计划可能又会出现！你甚至可能得到一个性能未知的第三种计划。

SQL 配置文件实际上是为了改进 CBO 的猜测，而不是为了消除猜测。SQL 配置文件不是 CBO 战略方向的一部分，也不提供计划稳定性。SQL 配置文件处于两难境地，CBO 团队已向我们保证，未来不会对 SQL 配置文件进行大量投入。

### SQL 计划基线

SQL 计划基线在 11gR1 版本中伴随着一片混乱出现。简单来说，SQL 计划基线是存储概要的直接替代品。一个明显的改进，至少从安全角度来看，是执行计划的存储库现在保存在数据字典中，而不是保存在一个可能泄露密码的方案里。

SQL 计划基线相当易于使用，许多客户正在成功地使用它们。然而，它们存在与存储概要相同的主要缺点，即你无法将执行计划存储库与应用程序中的语句进行核对。在其他大多数重要方面，SQL 计划基线具有与存储概要相同的优点和缺点。

混淆出现在许可证方面。直到发布前夕，SQL 计划基线还计划成为一个收费功能。事实上，包括我在内的许多参加过 Oracle 大学 11g 新特性课程的学生都被错误地告知，SQL 计划基线需要 `调优包`。实际上，SQL 计划基线功能中唯一需要许可证的部分叫做 `计划演进`，这是一个允许执行计划更改的功能！即便如此，许可证也仅当 `计划演进` 由 `SQL 调优顾问` 执行时才需要。因此，我们免费得到了我们想要的东西（稳定性部分），而必须为不想要的东西（适应性部分）付费。难怪 Oracle 员工对此都感到困惑！

## 介绍 TSTATS

TSTATS 是一种用于管理性能的部署方法，它使我们能够充分利用 CBO 最好的方面，同时防止它随意更改执行计划。不涉及执行计划的存储库，因此没有核对问题，也不涉及使用隐藏初始化参数或其他不支持的功能。

## 致谢

TSTATS 这个名字代表“托尼的统计信息”，是我的客户发明的名字，用来指代我当时成功说服他们采纳的方法。虽然这个名字满足了我的虚荣心，但如果我说 TSTATS 完全是我的发明，那就错了。在前言中，我详细介绍了 TSTATS 的历史，并对 Jonathan Lewis 和 Wolfgang Breitling 表示了感谢。然而，最大的贡献来自 Adrian Billington，他提出的从对象统计信息中删除基于时间数据的想法是 TSTATS 的核心。现在让我们来探讨这一点。

## 调整列统计信息

如果 CBO 缺乏可预测性是基于时间敏感的数据，换句话说，是仅在有限时间段内有效的统计信息，那么我们为什么不直接删除那些统计信息呢？事实证明，尽管有几个原因表明从对象统计信息中删除时间敏感数据可能不是个好主意（我们将在 第 20 章 中讨论这些），但在大多数情况下，我们可以放心地这样做。你无法通过更新列统计信息来移除 `LOW_VALUE` 和 `HIGH_VALUE` 统计信息，并且出于我无法理解的原因，这多年来一直是我思维上的一个障碍。这个问题的解决方案（事后看来显而易见）是删除列统计信息，然后重新插入它们，并去掉时间敏感的部分！Listing 6-8 展示了如何做到这一点。

Listing 6-8. 使用 `DBMS_STATS` 更新列统计信息

```sql
DECLARE
   distcnt   NUMBER;
   density   NUMBER;
   nullcnt   NUMBER;
   srec      DBMS_STATS.statrec;
   avgclen   NUMBER;
BEGIN
   DBMS_STATS.get_column_stats (
      ownname   => SYS_CONTEXT ('USERENV', 'CURRENT_SCHEMA')
     ,tabname   => 'T3'
     ,colname   => 'C2'
     ,distcnt   => distcnt
     ,density   => density
     ,nullcnt   => nullcnt
     ,srec      => srec
     ,avgclen   => avgclen);

   DBMS_STATS.delete_column_stats (
      ownname   => SYS_CONTEXT ('USERENV', 'CURRENT_SCHEMA')
     ,tabname   => 'T3'
     ,colname   => 'C2');

   DBMS_STATS.set_column_stats (
      ownname   => SYS_CONTEXT ('USERENV', 'CURRENT_SCHEMA')
     ,tabname   => 'T3'
     ,colname   => 'C2'
     ,distcnt   => distcnt
     ,density   => density
     ,nullcnt   => nullcnt
     ,srec      => NULL
     ,avgclen   => avgclen);
END;
/
```

Listing 6-8 操作了 `T3.C2`，关键点是加亮的那一行。`SREC` 结构保存着本应存入 `LOW_VALUE` 和 `HIGH_VALUE` 统计信息的数据，在重新插入统计信息时省略了此结构。



一旦完成这些更改并刷新共享池，无论你提供什么日期，`清单 6-1` 至 `6-5` 中使用的示例语句都将使用全表扫描！此外，在删除了时间敏感的统计数据后，成本和基数的计算结果将与删除前针对当代日期的计算结果相同。

## TSTATS 简述

`TSTATS` 的核心思想是从对象统计信息中移除时间敏感数据。基于此理念，可以应用以下流程要点：

1.  在选定的“主”测试系统上收集统计信息。
2.  为所有全局临时表伪造统计信息。
3.  从对象统计信息中移除时间敏感数据。
4.  在主测试系统上进行初始性能测试，并根据需要进行调整。
5.  将统计信息复制到所有其他测试系统，并最终复制到生产环境。
6.  在所有系统上锁定应用程序模式的对象统计信息。
7.  在所有系统上删除所有针对应用程序模式的统计信息收集作业。
8.  `TSTATS` 仅适用于应用程序模式，因此任何收集字典统计信息的作业均不受影响。

遵循这些步骤并采取一些其他预防措施，你可以确保所有测试系统上的执行计划与生产系统相匹配，并且相同的 SQL 语句将永远具有相同的执行计划，或者至少在你决定改变之前是如此！

这个方法听起来熟悉吗？它应该熟悉；这是部署软件、DML 和配置更改的标准方法。你为软件和 DML 建立的任何变更控制程序都应应用于对象统计信息。通常，你会将软件、DML 和对象统计信息作为发布的一部分同时交付。

## TSTATS 的替代方案

使用 `TSTATS` 方法来稳定应用程序性能并非易事，需要投入一定的时间和精力。正如你将在 `第 20 章` 看到的，可能存在简化 `TSTATS` 以解决特定问题的方法，让你能够快速轻松地解决关键稳定性问题。即便如此，如果你要管理数百个数据库的庞大体系，那么将 `TSTATS` 的任何变体作为标准采用可能都不切实际。只有在 `CBO` 团队做出某些使得类似 `TSTATS` 的方法变得更直接的改变（尽管这种可能性不大）时，这种情况才会改变。

2012 年，Oracle 的真实世界性能团队进行了一次环球巡讲，试图破除一些关于 Oracle 性能的误解，并教育人们如何进行性能管理。我参加了爱丁堡的研讨会，发现它非常发人深省。其中最有趣的建议之一来自 Graham Wood。Graham 观察到，许多 Oracle 客户收集统计信息的频率过高。我想明确说明，Graham 并没有说收集统计信息只是为了防止执行计划改变，他当然也没有将收集统计信息比作俄罗斯轮盘赌！然而，Graham 的建议确实（非常正确地）指出，收集统计信息存在一些风险，而减少其频率可以显著降低这种风险。

## 你需要多久收集一次统计信息？

如果你的表中的数据日期范围越广，`CBO` 降低其基数估计的速度就越慢，并且只有当基数估计降低到非常小的数字时，执行计划才可能改变。因此，如果你保存一个月的数据，每周收集一次统计信息，应该没问题。如果你保存一年的数据，那么你不需要超过每三个月收集一次统计信息！

## 优化后 SQL 的部署选项

在 `第 5 章` 中，我描述了一个系统化的 SQL 语句调优过程，涉及运行一系列实验。我明确指出，实验性更改和生产环境部署更改可能不同，我现在想谈的就是确定后者的过程。无论你是使用传统的对象统计信息管理技术还是 `TSTATS` 的某种变体，所涉及的考虑因素都是相同的；主要考虑因素是你是否希望生产环境的更改也影响其他语句。

### 当仅需更改一个 SQL 语句时

你可能希望生产环境的更改仅限于一个特定 SQL 语句的原因有几个。可能你的性能问题涉及仅在一个地方出现的构造，并且你知道没有其他 SQL 语句需要修复。你甚至可能怀疑其他 SQL 语句可能也存在同样的问题，但这些语句并不关键，测试更改影响所需的时间和精力无法被证明是合理的。最后，你可能发现了一个影响多个关键 SQL 语句的问题，但你现在需要将修复仅部署到一个 SQL 语句，而其他语句的修复则推迟到下一个版本。

无论出于何种原因决定将更改限制在一个 SQL 语句，这个决定都是确定部署策略的关键。

### 转换 SQL 代码

这个很简单。如果成功的优化实验涉及以某种方式重写 SQL 语句，那么大多数情况下，在生产环境中进行相同的代码更改只会影响力一个 SQL 语句。由于这正是我们期望的效果，你可以直接将实验性的代码更改部署到生产环境。

为了限制更改范围，你应该尽量避免更改视图定义。更好的做法是创建一个新视图或使用带因子的子查询。

### 向 SQL 代码添加提示

如果你的成功优化实验涉及向 SQL 语句添加提示，并且你想要一个将更改限制在该 SQL 语句的部署选项，那么你应该毫不犹豫地将提示部署到生产环境。这个建议可能会让你感到惊讶，因为在书籍、会议论文和博客中有大量材料建议你应该非常不愿意在生产代码中部署提示。我强烈不同意这种观点，我将在 `第 18 章` 回到这个话题进行更详细的讨论。

仅仅因为你使用一组提示获得了期望的执行计划，并不意味着你应该为生产使用相同的提示集。例如，你可能认真采纳了 Wolfgang Breitling 的“基于基数反馈的调优”理念，并向代码中应用了一个或多个 `CARDINALITY` 提示来纠正一些 `CBO` 基数错误。如果你不使用 `TSTATS`，那么当下次收集统计信息时，这些提示将无法可靠地防止执行计划改变。最好识别出 `CARDINALITY` 提示带来的具体变化，然后直接使用其他提示来指定访问方法、连接顺序等。在 `第 8 章`，我将向你展示如何确定这些提示是什么，并且我们将在 `第 3 部分` 讨论其中大多数提示的含义。

## 物理数据库更改

你应该非常谨慎地仅为加速一个 SQL 语句而进行物理数据库更改。我已经在上一章解释过，添加索引会给所有 `DML` 操作带来额外开销。此外，当你添加索引或进行其他物理数据库更改时，你无法确定对其他 SQL 语句执行计划的影响。你应该仅在最后考虑为优化一个 SQL 语句而进行物理数据库更改。

### 当需要更改多个 SQL 语句时



当你处于重大发布的早期阶段，并且前方仍有大量测试周期时，你可能希望进行一些会影响多个 SQL 语句的更改。这些更改甚至可能带来整个应用程序的全面性能提升。

我最近接到一项任务，需要将一个耗时六小时的夜间批处理作业的运行时间缩短至两小时。该批处理涉及数百甚至数千条 SQL 语句，逐一优化排名前 10%的语句都是一项无望完成的任务。即便如此，我仍然会首先查看运行时间最长的 SQL 语句。但一旦理解了该语句的问题所在，我就会尝试寻找修复方法，并希望该方法也能使其他语句受益。当性能问题与某种常见结构相关时，这通常是可行的。

请记住，如果你做了一项更改，它提升了 100 条语句的性能，却导致另外两条语句的性能下降，你总可以对这两条性能下降的语句使用**提示（hint）**！

## 转换 SQL 代码与添加提示

我已经说过，当你试图限制变更范围时，修改视图定义是个坏主意。由此可推论，当需要全面修改时，修改视图定义就是个好主意。

除了这个例外，更改单条 SQL 语句（包括添加提示）不太可能对整个应用程序产生巨大影响，除非该语句运行得非常频繁或耗时特别长。

## 物理数据库更改

物理数据库更改对提升应用程序整体性能具有巨大潜力，并且在进行此类更改时，需要考虑的是应用程序的整体性能。应用程序性能不佳，往往是因为多年来为解决个别查询的问题而添加了大量索引，这种情况并不少见。也许提升你应用程序性能的方法是删除其中许多索引！添加索引并不是解决长时间运行的全表扫描的唯一方法，我们将在第 15 章中讨论多种其他方法。

## 更改统计信息

我曾大力强调这样一个事实：当对象统计信息发生变化时，执行计划可能会改变。在生产系统的上下文中，执行计划的改变是坏事；但在测试系统上调优整个应用程序的上下文中，此类改变可能是潜在的大好事。

如果你的成功优化实验涉及向 SQL 语句添加提示，并且你怀疑许多其他 SQL 语句会受到类似影响，你应该尝试寻找一个不同的、能在全局解决问题的方案。更改统计信息可以提供帮助。

## 扩展统计信息

正如 Wolfgang Breitling 的 TCF 论文所解释的，CBO（基于成本的优化器）得出糟糕执行计划的最常见原因之一是它做出了错误的**基数估算**。而 CBO 做出错误基数估算的一个主要原因，是它没有意识到两列之间的相关性。因此，如果你为 CBO 提供相关性信息，那么你可能有助于提升多条 SQL 语句的性能。

想象一下，你有一个包含 40,000 行的表`T1`。`T1`有两个列`C1`和`C2`。`C1`有 20 个不同的值，每个值对应 2,000 行。`C2`也有 20 个不同的值，每个值对应 2,000 行。CBO 可能需要知道有多少行会同时匹配`C1`的一个特定值和`C2`的一个特定值。CBO 知道`C1`和`C2`有 400 种可能的组合，并通常假设这 400 种组合出现的可能性均等。因此，CBO 会将 40,000 除以 400，得出基数估算为 100。

现在假设`T1`的真实名称是`ORDERS`，而`C1`和`C2`的真实名称分别是`ORDER_DATE`和`DELIVERY_DATE`。`ORDER_DATE`的值从 3 月 1 日到 3 月 20 日不等，而`DELIVERY_DATE`的值从 3 月 2 日到 3 月 21 日不等。你管理严格，每一笔订单的`DELIVERY_DATE`都是`ORDER_DATE`的次日！这意味着实际上表中只有 20 种`ORDER_DATE`和`DELIVERY_DATE`的组合——每种组合对应 2,000 行。`ORDER_DATE`和`DELIVERY_DATE`是相关的。

此类问题可以通过使用`扩展统计信息`来解决，我将在第 9 章中详细讲解扩展统计信息。通过创建扩展统计信息而不是添加提示，你可能会提升使用相同列组合的其他 SQL 语句的性能。

## 手工创建的直方图

CBO 得出错误基数估算的另一个原因是`数据倾斜`。我在本章前面已经介绍了数据倾斜的概念：它指的是少数列值出现频率远高于其他值的情况。虽然常规的 TSTATS（统计信息收集工具）对列统计信息的处理方式是移除关于特定值的所有信息，但对于需要直方图的列来说，这并非正确方法。只要应用程序没有为存在数据倾斜的列的值使用绑定变量，那么部署直方图是完全合理的事情。

根据我的经验，在一个完全规范化的模式中，需要直方图的列数量非常少。当然，在实际生活中，一定程度的反规范化是常见的，需要直方图的特定列会被复制到多个表中。在这些情况下，被复制的列通常具有相同的列名，易于识别，并且需要相同的直方图。无论你是否使用 TSTATS，我的建议始终是分析你的数据并手动设置此类直方图。我们将在第 20 章中详细讨论这一点。然而，此时的关键信息一如既往：统计信息的更改可以用于同时提升多条 SQL 语句的性能。

## 生成虚假的对象统计信息

扩展统计信息和直方图的使用是 Oracle 员工经常讨论并鼓励你使用的功能。然而，关于显式设置或用特定值覆盖其他类型统计信息的讨论则少得多。我推测缺乏讨论的原因是，将统计信息设置为特定值所需的知识和专业水平，比设置扩展统计信息或直方图要高得多。

此时我想强调的是，将统计信息设置为特定值是完全支持的，并且相关技术在`DBMS_STATS`包的文档中有记载。有一些特殊情况需要将对象统计信息设置为不反映现实的值，我们将在第 20 章中讨论这些情况。

![image](img/sq.jpg) **注意** 对象统计信息存在的目的是帮助 CBO 做出正确的决策。如果准确的统计信息导致了糟糕的 CBO 决策，而不准确的统计信息反而带来了良好的决策，那么你希望部署的是不准确的统计信息。

与任何统计信息操作一样，你应该考虑你的更改对其他 SQL 语句的影响。通常，了解你的应用程序中盛行的查询类型以及极少（如果有的话）出现的查询类型，对于判断伪造对象统计信息是否有益至关重要。
