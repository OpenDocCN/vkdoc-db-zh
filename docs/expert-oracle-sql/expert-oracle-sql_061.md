# 数据库并行查询机制与优化

## 数据加载机制

清单 8-24 中使用的数据加载机制在数据主要目标为一到两个分区时最为有效，因为每个服务器可以将数据加载到所有分区中。由于直接路径写入的工作方式，当行数据均匀分布在各分区之间时，这种机制会是低效的。

## 相关子查询分发机制

在 12cR1 版本之前，对无法解嵌套的相关子查询的评估要么以串行方式完成，要么由一个完全独立的 DFO 树完成。12cR1 版本引入了多种可以并行评估相关子查询的方法。它还引入了 `PQ_FILTER` 提示，让你能对该机制进行一定控制。清单 8-25 展示了四种选项中的两种。

清单 8-25. 相关子查询评估的并行执行

```
SELECT /*+                                    SELECT /*+
parallel(t1) full(t1)                         parallel(t1) full(t1)
pq_filter(NONE)                               pq_filter(HASH)
*/                                            */
       *                                              *
  FROM t1                                        FROM t1
 WHERE EXISTS                                   WHERE EXISTS
          (SELECT null                                   (SELECT null
             FROM t2                                         FROM t2
            WHERE t1.c1 != t2.c2);                          WHERE t1.c1 != t2.c2);
------------------------------------------  ---------------------------------------------
| Id  | Operation             | Name     |  | Id  | Operation                | Name     |
------------------------------------------  ---------------------------------------------
|   0 | SELECT STATEMENT      |          |  |   0 | SELECT STATEMENT         |          |
|   1 |  PX COORDINATOR       |          |  |   1 |  PX COORDINATOR          |          |
|   2 |   PX SEND QC (RANDOM) | :TQ10000 |  |   2 |   PX SEND QC (RANDOM)    | :TQ10001 |
|   3 |    FILTER             |          |  |   3 |    BUFFER SORT           |          |
|   4 |     PX BLOCK ITERATOR |          |  |   4 |     FILTER               |          |
|   5 |      TABLE ACCESS FULL| T1       |  |   5 |      PX RECEIVE          |          |
|   6 |     TABLE ACCESS FULL | T2       |  |   6 |       PX SEND HASH       | :TQ10000 |
------------------------------------------  |   7 |        PX BLOCK ITERATOR |          |
                                            |   8 |         TABLE ACCESS FULL| T1       |
                                            |   9 |      TABLE ACCESS FULL   | T2       |
                                            ---------------------------------------------
```

清单 8-25 中的查询并行读取 `T1`，并使用一个无法解嵌套的相关子查询对结果进行过滤。

假设 `T1` 非常大，相关子查询的评估成本很低，且该过滤器能过滤掉 `T1` 中的大多数行。在这种情况下，左侧使用 `PQ_FILTER (NONE)` 提示会是一个好主意，因为生成的执行计划使用读取 `T1` 的同一个并行查询服务器来评估子查询，并且只有一个 DFO；我们不需要通过 TQ 发送所有未过滤的数据。这在大多数情况下可能是最明智的方法。

另一方面，假设 `T1` 并没有那么大，但相关子查询执行的计算非常昂贵。在这种情况下，你可以安排由一个独立的 DFO 与 `T1` 的并行全表扫描并发地完成过滤。这是通过在清单 8-25 的右侧使用 `PQ_FILTER (HASH)` 提示来实现的。

清单 8-25 右侧的执行计划显示，当 PQSS1 从 `T1` 读取行时，它会基于 `T1.C1` 的哈希值将这些行发送给 PQSS2 的一个成员。请注意，第二个 DFO 的存在意味着数据在过滤后需要缓冲，然后再发送给 QC。并且，冒着听起来重复的风险，我想再次提醒你，操作 3 尽管表面上看涉及排序，但实际上并不包含排序。

`PQ_FILTER` 提示参数还有两个其他合法值。`PQ_FILTER (SERIAL)` 会导致所有相关子查询评估都由 QC 执行，就像 11gR2 及更早版本那样。如果关联列 `T1.C1` 的不同值数量很少，让 QC 执行过滤可能是个好主意，因为标量子查询缓存可能会防止子查询被重复评估。

如果你指定了 `PQ_FILTER (RANDOM)` 提示，行会以轮询方式从 `PQSS1` 发送到 `PQSS2`，而不是像 `PQ_FILTER (HASH)` 分发机制那样基于 `TQ1.C1` 的哈希函数。根据文档，当数据存在偏斜时应使用这种分发机制，但这对我来说没有意义。假设 `T1` 中所有行的 `T1.C1` 值完全相同。清单 8-25 右侧执行计划第 6 行显示的 `PX SEND HASH` 操作会将所有行发送到 PQSS2 的同一个成员，但标量子查询缓存会阻止使用 `PQ_FILTER (RANDOM)` 时会出现的子查询多次评估。

如今，无法解嵌套的子查询已经很少见，而需要并行评估它们的场合则更为罕见。我必须承认，我对这个 12cR1 特性没有实际经验；以上分析纯属理论，可能存在缺陷。至少现在你和我一样了解它了！

## 其他并行分发机制

`PQ_DISTRIBUTE` 提示不仅用于控制不同的数据加载方法。事实上，迄今为止，这个提示最常见的用途是控制并行连接表的不同方法。在介绍完串行连接表的不同方法后，我将在第 11 章中介绍这些分发机制。

生产者 PQSS 通过 TQ 分发数据还有另外一两种方式，但这些方式适用于特定场景，并且没有分发机制可供选择；`PQ_DISTRIBUTE` 提示既不合法也不需要。

例如，你将在清单 8-22 的第 7 行看到 `PX SEND RANGE` 操作。这是因为 TQ 的使用者并行执行排序。第 17 章专门讨论排序；我们将在那里研究排序的并行化。同一份清单在第 3 行显示了一个 `PX SEND QC (RANDOM)` 操作，在第 16 行显示了一个 `PX SEND QC (ORDER)` 操作。这两种分发机制都涉及将数据发送给 QC，但在前一种情况下，PQSS 成员向 QC 发送数据的顺序无关紧要——PQSS 成员只需通知 QC 它们已完成数据加载，而 QC 只想知道所有操作何时完成。在后一种情况下，PQSS 成员向 QC 发送行的顺序很重要，因为最终结果集需要按 `CNT_C2` 排序。

## 为何强制并行查询可能不生效

`PARALLEL` 和 `PARALLEL_INDEX` 提示用于*授权* CBO 使用并行执行来并行执行操作，并可选择指定要使用的 DOP（并行度）。但是，如果 CBO 有权选择一个无法并行执行的操作，例如索引范围扫描，那么它可能会选择该访问方法并串行运行它，而不是使用另一个更昂贵的并行操作。
