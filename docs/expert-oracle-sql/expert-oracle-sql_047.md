# 调优 SQL 语句

本章将介绍本书两大主题中的第一个：调优 SQL 语句。本章描述的系统化过程旨在确定一种执行 SQL 语句的方法，确保其在合理的时间内完成，且不会过度消耗资源。在此阶段，我将仅对该过程进行概述。更多细节将在本书的第 4 部分中深入探讨。

需要明确的是，用于确定使 SQL 语句快速运行的方法的技术，可能与在生产系统中实施或强制执行该方法的技术不同。我们将在下一章开始探讨部署相关的内容。

## 理解问题

在生活中的许多方面，准确理解你试图解决的问题是找到解决方案所需的大部分工作。调优 SQL 语句就是一个典型的例子。你对调优问题的理解越深入，就越能自信地做出调优决策。

### 理解业务问题

这是一本技术书籍，并非管理技巧的培训手册，但厘清业务问题至关重要，因此我不得不花少量篇幅加以讨论。当你首次被要求检查一个或多个 SQL 语句的性能问题时，很容易立即着手深入排查。但你应该先**冷静一下**。

假设你半夜被叫醒，因为一个夜间批次处理中的查询运行超时。你需要得到的第一个答案是：“那又怎样？”你可能想把问题表达得更委婉一些，但这确实是你需要弄清楚的实际问题。查询延迟运行业务影响是什么？是否有需要满足的服务等级协议（`SLA`），如果违反协议会怎样？`SLA` 何时会被违反，或者是否已被违反？

理解业务背景对你的整个调优工作方法至关重要，原因有几。其一，你需要在完成查询的需求与为根本原因分析（`RCA`）收集诊断信息的需求之间取得平衡，而 `RCA` 对于防止此问题再次发生本身也很重要。

理解业务问题之所以重要的第二个原因，是让你能够优先分配资源。你可能会观察到系统正忙于运行其他进程。这些其他进程有多重要？能否终止它们？问题越严重，业务部门就越能接受采取严厉措施，反之亦然。一方面，有时可能涉及数十亿美元，甚至有时人命关天。在这种危急情况下，你可能需要请一两个人协助处理。另一方面，我也曾半夜被叫醒，结果发现查询运行很久实际上没有任何后果；我只是被告知“以防你想知道”。最好在开始前就了解清楚这类情况。

对生产查询进行补救性性能调优并非唯一需要理解业务问题的情形。如果开发人员向你咨询一个运行 10 分钟的查询，你也应从礼貌地询问“那又怎样”问题的不同变体开始。该查询是用于交互式使用还是批处理的一部分？业务部门的期望是什么？如果该查询作为年终夜间批处理的一部分每年运行一次，那么也许它并不重要——当然，除非去年的年终批处理是一场灾难！

开发场景引出了理解业务问题的第三个重要原因：那就是知道何时停止！假设你有一个 75 分钟的 `SLA`，并且你已经将查询调优到能稳定在 15 分钟内完成。你可以看到，再稍加努力，也许能将运行时间缩短到 12 分钟。这值得吗？几乎可以肯定，你的时间最好花在别处，因为你的服务使用者从这项额外工作中获益甚微。

完整且权威地理解业务背景可能相当困难，在许多情况下甚至极不现实。你不能花费 30 分钟去弄清楚一个可能只需 10 分钟就能修复的问题；你的“专业精神”本身可能就会导致服务等级协议（`SLA`）被违反！然而，问题看起来越紧急，恐慌情绪似乎越弥漫，就越需要弄清楚真正的问题是什么，以便正确地集中资源。

这里需要相当多的判断力，我们都会随着经验积累而做得更好。关于状态报告，我将以一个小建议结束本话题：你可能想写一封简短的电子邮件，说明你被要求做什么以及为什么。将此类邮件的副本发送给各个关键人员，是让这些关键人员了解你被要求做出的假设的好方法，并提供了有用的审计跟踪。

### 理解技术问题

技术问题似乎显而易见：让 SQL 语句运行得快。对吗？再一次，你需要**冷静一下**。

我曾花费两天时间调优一个从未完成的查询，结果发现查询本身是错误的！开发人员刚接触她团队编码标准中规定的 `ANSI` 语法，每次本意是写 `RIGHT JOIN` 时都写成了 `LEFT JOIN`，反之亦然！我本应为自己没有检查而懊恼不已，我很想说我再也没有犯过这种错误，但我不能！就在不久前，一位开发人员在将 SQL 语句交给我调优之前，特意修改了它。修改是出于好意。提供给我处理的测试系统上的数据不具有生产环境的代表性，而实际的生产查询在那个测试系统上运行良好。查询被善意修改以适应测试数据，但这一改变完全改变了调优的可选方案。

确保你被要求查看（或你自己编写）的语句是正确的，并不是你需要弄清楚的唯一技术事项。你还需要了解 SQL 语句是静态的还是动态的。换句话说，你需要知道该语句每次运行时是否都会变化。你需要了解这是唯一需要调优的语句，还是存在数百个类似的语句。你需要了解同时还会运行什么其他任务，以便避免资源争用并适当分配资源。可能还有其他会困扰你的事情，所以不要将此视为详尽清单。

## 理解 SQL 语句


在第 1 章中，我介绍了**因式分解子查询**，因为我感觉能够阅读 SQL 语句并理解其功能对于调优至关重要。你可能需要完全重写一个查询以获得完全相同的结果。还记得从代码清单 1-1 到代码清单 1-2 的改动吗？如果你对 SQL 语句试图做什么毫无头绪，那么这类转换就非常难以看出来。也许查询中有一个`ORDER BY`子句。你知道它为什么在那里吗？它满足了真正的需求，还是仅仅因为开发者更容易验证输出正确而被放进去的？无处不在的`SELECT COUNT (*) FROM`查询通常仅用于确定匹配行的存在，而确切的计数无关紧要。如果查询是你自己写的，那么希望你知道这些问题的答案，但如果不是，花些时间尝试理解查询的功能和原因通常会加速整个调优过程。

## 理解数据

在测试系统上让你的查询运行得很快固然很好，但测试数据是否代表了你将在生产系统上找到的数据？你生产系统上的数据量是否可能增长？如果会，增长速度如何？我承认，关于数据的具体问题可能在调优练习的最初阶段难以提出。除非你对性能问题是什么以及可能的执行计划是什么有了一定了解，否则可能很难知道查询的哪些特定方面对数据量或数据分布敏感。然而，一旦你发现查询的某些方面对数据量或数据分布特别敏感，最好暂停并提一些问题。

让我给你一个现实生活中的例子，说明当不检查关于数据的假设时会发生什么问题。一项监管要求导致了一个我称之为`AUDIT_TRAIL`的表被添加到数据库中。`AUDIT_TRAIL`表中的行填充了诸如`USER_ID`、`PRODUCT_ID`等列。向该表添加行的软件经过了测试，大约有半打行出现在`AUDIT_TRAIL`表中。然后编写了一份报告。它从`AUDIT_TRAIL`表中取出数据，并与`USERS`和`PRODUCTS`表进行连接，以便用更具可读性的`USER_NAME`和`PRODUCT_NAME`列替换无意义的`USER_ID`和`PRODUCT_ID`列。报告运行良好！

应用程序部署到生产环境的第一天，报告就引起了一些担忧，因为它似乎运行得比预期的要长一些。第二天，担忧的声音更强烈了，因为报告现在花费的时间长得无法接受。报告花费如此长时间的原因是，每天大约有 100,000 行被添加到`AUDIT_TRAIL`表中！谁知道呢？显然报告的测试者不知道，他在报告了大约六行后就签字确认了测试！

## 理解问题总结

关于调优问题各个方面的讨论，不应被视为强制性的检查清单或详尽无遗的问题列表。我花这么长时间讨论这个问题是为了强调一个事实：没有任何 SQL 调优任务像最初看起来那么简单。如果你过早开始进行技术分析，你很可能会做出错误的假设，并提供一个因其他原因而不适用或不合适的解决方案。如果你最终确实提供了一个用户满意的解决方案，如果你先问对了问题，你会更快速、更高效地完成。最后，我想强调的是，一旦你开始分析，就应该始终警惕基于未经证实的假设从观察中得出结论。不要害怕暂停并核实你的事实。

## 分析

在透彻理解问题陈述之后，调优过程的下一阶段是分析 SQL 语句的行为。这是一个迭代过程的一部分；你进行分析，做出一些更改，然后重新分析。

## 让语句运行完成

尽管在 SQL 语句运行时，你可以通过使用 SQL 性能监视器对其工作方式有所了解，但直到 SQL 语句执行完毕，你才能真正声称掌握了全貌。而且，如果你想尝试几件事，你需要让语句在最多 5 到 10 分钟内完成。否则，你要么会空等太久，要么最终不得不同时处理其他事情，并最终忘记自己进行到哪里了。

假设一个查询已经运行了两分钟，你查看 SQL 性能监视器报告，发现表`T1`正在进行全表扫描，并且只完成了 1%。天哪。你没想到全表扫描会花这么长时间。按照这个速度，全表扫描需要 200 分钟才能完成，而整个语句完成还需要更长时间，谁知道呢。作为临时措施，你需要减少从表中选择的数据量。代码清单 5-1 展示了可用于减少你所操作数据集的众多方法之一。

代码清单 5-1. 使用 `SAMPLE` 关键字减小数据集大小

```
CREATE TABLE t1
AS
       SELECT 1 c1
         FROM DUAL
   CONNECT BY LEVEL <= 100;

CREATE TABLE t2
AS
       SELECT 1 c2
         FROM DUAL
   CONNECT BY LEVEL <= 100;

SELECT *
  FROM t1, t2
 WHERE t1.c1 = t2.c2;

WITH q1
     AS (SELECT *
           FROM t1 SAMPLE (5) SEED(0))
SELECT *
  FROM q1 t1, t2
 WHERE t1.c1 = t2.c2;

DROP TABLE t1;
DROP TABLE t2;
```

代码清单 5-1 中的第一个查询展示了一个连接两个表的典型查询。第二个查询使用`SAMPLE`关键字，在本例中选择`T1`中 5%的数据，试图在合理的时间内获得某种结果。`SEED`关键字用于生成可复现的结果。当然，在某个时刻你将必须移除这个`SAMPLE`关键字，但只有在你弄清楚查询为什么花费这么长时间之后。

## 分析耗时

语句执行完毕后，你需要获取该耗时的明细分解。在第 4 章中，我解释了如何确保视图`V$SQL_PLAN_STATISTICS_ALL`从运行时引擎填充了有关 SQL 语句中各个操作的信息。这是分析的主要信息来源。

### 当耗时总和不匹配时

有时，查看`V$SQL_PLAN_STATISTICS_ALL`的数据时，你会发现总耗时与你等待查询完成的时间长度相去甚远。可能的原因有以下几种：



## 影响 SQL 执行时间的其他因素

通常，大多数 SQL 语句的执行时间都花在了 CPU 处理、磁盘 I/O 和 SQL*Net 消息传递上。但有时会出现这样的情况：尽管统计数据显示 CPU、I/O 和 SQL*Net 的总时间与语句的总执行时间相符，但你仍然觉得某些操作花费了比预期更长的时间。这可能是因为时间被“隐藏”在了以下几个地方：

*   **递归 SQL。** 如果被测试的 SQL 语句包含了对某些函数的调用，而这些函数又包含了对其他 SQL 语句的调用，那么执行这些递归 SQL 语句所花费的时间**不会**包含在所报告语句的执行时间内。如果一个 DML 语句最终触发了一个或多个触发器，那么执行这些触发器所花费的时间同样会被排除在外。
*   **向客户端进程发送和接收数据。** 一旦查询返回了行数据，就需要将它们发送到客户端进程。如果客户端进程跟不上，就会阻塞服务器的处理。慢速网络或缓慢的客户端会在`V$ACTIVE_SESSION_HISTORY`中体现为大量的“SQL*Net message to client”等待事件。
*   **解析。** 对于运行时间长达数秒的 SQL 语句来说，这很少是个问题，但我曾遇到过一个应用，其中 CBO 花了超过 4 秒来制定一个查询计划！这个查询计划本身很好，运行时间不到 1 秒，但这于事无补！

大多数情况下，时间丢失的原因很快就会变得清晰。只有在极少数你看不到时间丢失在何处的情况下，我在第 4 章中提到的 10046 SQL 跟踪才变得至关重要。

## 当时间确实累积起来时

一旦`V$SQL_PLAN_STATISTICS_ALL`中的数据讲得通了，你就需要首先聚焦于最昂贵的操作。看看清单 5-2 中的测试脚本做了什么。

**清单 5-2. 分析一个简单的两表连接的性能**
```
CREATE TABLE t1 (c1 PRIMARY KEY)
ORGANIZATION INDEX
AS
   SELECT ROWNUM c1 FROM DUAL;

CREATE TABLE t2
PCTFREE 99
PCTUSED 1
AS
   WITH q1
        AS (    SELECT ROWNUM c2
                  FROM DUAL
            CONNECT BY LEVEL <= 100)
   SELECT a.c2, RPAD ('X', 2000) vpad
     FROM q1 a, q1;

BEGIN
   DBMS_STATS.gather_table_stats (SYS_CONTEXT ('USERENV', 'CURRENT_SCHEMA')
                                 ,'T1'
                                 ,no_invalidate   => FALSE);
   DBMS_STATS.gather_table_stats (SYS_CONTEXT ('USERENV', 'CURRENT_SCHEMA')
                                 ,'T2'
                                 ,no_invalidate   => FALSE);
END;
/

INSERT INTO t2 (c2, vpad)
   WITH q1
        AS (    SELECT ROWNUM c2
                  FROM DUAL
            CONNECT BY LEVEL <= 300)
   SELECT a.c2, RPAD ('X', 2000) vpad
     FROM q1 a, q1;

SET LINES 200 PAGES 900 TIMING ON

ALTER SESSION SET statistics_level=all;

BEGIN
   FOR r IN (SELECT *
               FROM t1, t2
              WHERE t1.c1 = t2.c2)
   LOOP
      NULL;
   END LOOP;
END;
/

COLUMN id FORMAT 99
COLUMN operation FORMAT a18
COLUMN options FORMAT a11
COLUMN actual_time FORMAT 99.999 HEADING "Actual|Time"
COLUMN object_name FORMAT a17 HEADING "Object|Name"
COLUMN last_starts FORMAT 9999999 HEADING "Last|Starts"
COLUMN actual_rows FORMAT 9999999 HEADING "Actual|Rows"

SELECT id
        ,LPAD (' ', DEPTH) || operation operation
        ,options
        ,last_elapsed_time / 1000000 actual_time
        ,object_name
        ,last_starts
        ,last_output_rows actual_rows
    FROM v$sql_plan_statistics_all
   WHERE sql_id = 'cwktrs03rd8c7'
ORDER BY id;
```

清单 5-2 包含一些 SQL*Plus 特有的语法，它首先创建了两个表。`T1`是一个索引组织表，`T2`是一个普通的堆表。在收集了两个表的统计信息之后（在 12cR1 及更高版本中这是多余的，我将在本书后面解释），更多的行被添加到表`T2`中。`T2`的统计信息现在稍微过时了。这也许有点不寻常；你不能每次向表中插入数据都重新收集统计信息，但通常也不会在重新收集统计信息之前让行数增加好几倍！然后，这两个表在一个 PL/SQL 块中被连接，并且出于测试目的，输出被丢弃。在我的笔记本电脑上，这个查询大约花了 25 秒。清单 5-2 中的最后一条语句使用我通过第 1 章描述的技术单独确定的 SQL_ID，检查了`V$SQL_PLAN_STATISTICS_ALL`中的一些关键列。

清单 5-2 中最后一个查询的输出显示在清单 5-3 中。

**清单 5-3. 来自两表连接的运行时统计信息**
```
 Actual Object                Last   Actual
 ID OPERATION          OPTIONS        Time Name                Starts     Rows
--- ------------------ ----------- ------- ----------------- -------- --------
  0 SELECT STATEMENT                25.157                          1      400
  1  NESTED LOOPS                   25.157                          1      400
  2   TABLE ACCESS     FULL         24.806 T2                       1   100000
  3   INDEX            UNIQUE SCAN    .258 SYS_IOT_TOP_97615   100000      400
```

我们可以看到，操作 1 表面上花费了大约 25.16 秒，但你还记得我说过分配给子操作的时间包含在父操作的时间中吗？操作 2 和 3 总共花了大约 24.81 秒，所以`NESTED LOOPS`本身只负责 0.35 秒。这是可以预料的，因为`NESTED LOOPS`连接只是从驱动行源获取行，然后调用探测行源进行匹配。或许更令人惊讶的是，尽管我们可以看到操作 3 被调用了 100,000 次（我们从`LAST_STARTS`列知道），但总耗时仍然只有 0.26 秒！

**提示** 除了我将在第 11 章中介绍的半连接和反连接之外，`NESTED LOOPS`连接的探测行源（具有较高 ID 的子操作）的执行次数（`LAST_STARTS`）总是与驱动行源（具有较低 ID 的子操作的`LAST_OUTPUT_ROWS`列）返回的行数完全相同。这是因为对于第一个子操作返回的每一行，第二个子操作都会被探测一次。在这个案例中，`T2`的全表扫描返回了 100,000 行，因此`T1`的索引被探测了 100,000 次。

显然，在那 100,000 次探测的底层发生了大量复杂的优化，这可能会在某天变得有趣，但目前主要的结论是你不需要担心操作 3。不，主要的关注点是操作 2。尽管它只执行了一次，但那次全表扫描占用了我们语句总耗时 24.16 秒中的 24.81 秒。我们需要将注意力集中在这个全表扫描上。

## 解决问题

一旦你确定了 SQL 语句中消耗时间最多的具体操作，接下来要做的就是想办法节省部分时间。请始终记住，在此阶段，我们只是试图找出需要修复什么，而不是如何在生产环境中部署修复方案。

### 检查统计信息
