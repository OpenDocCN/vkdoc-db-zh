## 对象统计信息与部署

本章介绍了本书两个重要主题中的第二个：如何在生产环境中管理 SQL 语句的性能。这无疑是本书中最具争议性的一章，因为一些专家可能不同意我在此表达的观点，特别是当这些观点与 Oracle 声明的策略背道而驰时。尽管如此，我绝非唯一得出这些结论的人，我所描述的技术已被许多人用于稳定许多 Oracle 数据库的性能。让我们从关于性能管理的一些通用观察开始。

### 性能管理的原则

我相信有一个首要原则适用于所有类型服务的性能。它不一定必须是与数据库相关的服务，甚至不一定在 IT 领域。让我举几个非 IT 相关的例子来把这一点说清楚。

#### 英国皇家邮政的例子

大约 20 年前，我听了一位英国皇家邮政员工的餐后演讲。英国皇家邮政是英国的邮政服务，演讲者开始讨论他的雇主对其企业客户进行的一项调查。英国皇家邮政想知道他们的客户是否愿意接受下午 2:00 的送达时间，或者他们是否愿意为上午 9:00 的送达支付额外费用。客户群体的回应是，英国皇家邮政问错了问题！

当时，英国皇家邮政的可靠性不是特别好。我对所谓的“一类邮件”的个人经验是，信件大约有 90%的概率会在第二天到达。有 10%的情况邮件至少两天内无法送达。这种服务对业务使用者来说根本没用。假设一家律师事务所将法律文件寄给其在伦敦办公室的员工 John。John 想与同样在伦敦的律师事务所客户 Jim 讨论这些文件。你能想象 John 可以打电话给 Jim 说，“嘿，Jim，过来聊聊这些法律文件怎么样？我有 90%的几率到那时已经收到它们了！”这不是业务运作的方式。John 需要一个送达时间的*保证*。如果那个保证的时间是下午 2:00，那么 John 也许能把与 Jim 的会议安排在下午 3:00。如果 Jim 只有上午有空，那可能需要另一家承运商，但无论 John 使用哪家服务提供商，该服务提供商都必须（套用美国一家包裹服务公司的著名广告语）确保邮件**绝对、肯定、必须**在承诺的任何时间送达。

当然，在现实生活中，没有*绝对*的保证。总有一些我们无法控制的因素会引入小程度的风险。但如果消费者要容忍这些例外情况，它们就必须非常少。

#### 机场的例子

我开车送我成年的儿子 Thomas 去机场，并告诉他我试图解释的性能管理原则。我问他是否能想到第二个大多数人都能理解的例子。我们吃饭时，他有点努力地想找出一个合适的例子。我们吃完饭后，Thomas 通过了机场安检，我开车回家。大约一小时后，我接到了 Thomas 的电话。他有一个好消息和一个坏消息。坏消息是他错过了他的航班，但好消息是我的书有了第二个例子！

我曾经是个经常飞行的人，我绝对讨厌在机场浪费时间。我在 Thomas 的航班预定起飞前大约 30 分钟让他去过安检。这应该给他足够的时间，而且他也不会在候机厅困太久。我以为我是个好的“服务提供商”。不幸的是，Thomas 智能手机上的电子登机牌图像出了问题，他不得不去打印一份纸质副本。等 Thomas 回到安检处时，他的航班已经停止登机了。他向我解释说，优化机场体验以最小化等待时间固然值得赞赏，但下次我必须确保他**绝对、肯定**不能错过航班！
