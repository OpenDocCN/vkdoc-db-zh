# 《EXPLAIN PLAN` 与 `DBMS_XPLAN.DISPLAY`

`EXPLAIN PLAN` SQL 语句存在的理由，是在不实际执行语句的情况下为其生成执行计划。`EXPLAIN PLAN` 语句仅将执行计划的表示形式写入一个表，它本身不显示任何内容。为此，必须使用 `DBMS_XPLAN.DISPLAY` 函数。此函数接受多个参数，所有参数都是可选的；目前我将使用默认值。目的是将该函数的结果传递给 `TABLE` 操作符。清单 3-1 展示了一个示例，该示例对一个使用 `SCOTT` 示例模式中表的查询执行 `EXPLAIN PLAN`，然后使用 `DBMS_XPLAN.DISPLAY` 显示结果。

清单 3-1. EMP 与 DEPT 表的连接

```
SET LINES 200 PAGESIZE 0 FEEDBACK OFF

EXPLAIN PLAN
   FOR
      SELECT e.*
            ,d.dname
            ,d.loc
            ,(SELECT COUNT (*)
                 FROM scott.emp i
                WHERE i.deptno = e.deptno)
                dept_count
        FROM scott.emp e, scott.dept d
       WHERE e.deptno = d.deptno;

SELECT * FROM TABLE (DBMS_XPLAN.display);

Plan hash value: 4153889731

| Id  | Operation          | Name | Rows  | Bytes | Cost (%CPU)| Time     |

|   0 | SELECT STATEMENT   |      |    14 |  1638 |     7  (15)| 00:00:01 |
|   1 |  SORT AGGREGATE    |      |     1 |    13 |            |          |
|*  2 |   TABLE ACCESS FULL| EMP  |     1 |    13 |     3   (0)| 00:00:01 |
|*  3 |  HASH JOIN         |      |    14 |  1638 |     7  (15)| 00:00:01 |
|   4 |   TABLE ACCESS FULL| DEPT |     4 |   120 |     3   (0)| 00:00:01 |

```

被解释的语句显示了每位员工的详细信息，以及部门的详细信息，包括通过选择列表中的关联子查询获得的部门员工估计数量。`DBMS_XPLAN.DISPLAY` 的输出以计划哈希值开始。如果两个计划具有相同的哈希值，它们将是同一个计划。事实上，只要操作、对象名称和谓词都相同，相同的执行计划在不同的数据库上将具有相同的哈希值。

在计划哈希值之后，您可以看到操作表，该表提供了运行时引擎预期如何执行 SQL 语句的详细信息。

*   第一列是 ID。这只是一个数字，用于唯一标识语句中的操作，以供计划的其他部分使用。您会注意到操作 2 和 3 的 ID 前面有一个星号。这是因为在后面的输出中引用了它们。为简洁起见，我将从现在开始说“操作 2”而不是“ID 为 2 的操作”。
*   第二列是操作。大约有 200 个操作码，其中很少有官方文档记录，并且每个版本列表都在变长。幸运的是，在绝大多数执行计划中的绝大多数操作都来自一小部分几十个操作，随着您阅读本书，这些操作将会变得熟悉，如果它们还不熟悉的话。
*   操作表中的第三列称为 `name`。这个相当不起眼的列标题指的是被操作的数据库对象的名称，可以这么说。此列并非适用于所有操作，因此经常为空白，但在 `TABLE ACCESS FULL` 操作的情况下，它是被扫描表的名称。
*   下一列是 `rows`。此列提供操作最终将产生的估计行数，该数字通常被称为*基数估计*。例如，您可以看到操作 2 在 `EMP` 上使用 `TABLE ACCESS FULL` 操作，并且将读取与在同一对象上执行相同操作的 ID 5 相同数量的行。然而，ID 5 返回表中的所有 14 行，而操作 2 预计返回一行。您稍后将看到这种差异的原因。有时，在 SQL 语句的单次执行过程中，单个操作会被执行多次。重要的是要理解，估计的行数通常适用于该操作的单次调用，而不是所有调用返回的行总数。在这种情况下，操作 1 和 2 对应于选择列表中的子查询，预计每次调用时平均返回约一行。
*   `bytes` 列显示操作的一次调用返回的估计字节数。换句话说，它是平均行数乘以行的平均大小。
*   `cost` 列和 `time` 列都显示语句的估计耗时。`cost` 列以单块读取单位表示此耗时，而 `time` 列以更熟悉的术语表示。当显示较大的值时，可以通过将时间除以成本来计算出 CBO 认为单块读取需要多长时间。这里的数字较小且进行了四舍五入，使得这很困难。您还可以看到 `cost` 列显示了操作在 CPU 上花费的时间百分比估计值（与从磁盘读取数据相反）。

## 四舍五入问题

此表中显示的数字始终是整数或秒，但事实上 CBO 保留的是分数估计值。`rows` 列向上取整，并且几乎总是至少为 1，即使 CBO 认为操作返回行的可能性微乎其微。`cost` 列向上取整到最接近的整数，`time` 列向上取整到最接近的秒。因此，根据经验，总是将基数为 1 的估计理解为“0 或 1”。

操作表中可能还有其他列。这些列包括与临时空间利用、分区消除、并行查询操作等相关的列。然而，这里没有显示它们，因为它们与此查询无关。
