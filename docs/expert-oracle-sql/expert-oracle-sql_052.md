## 查询块与子查询

对于 Oracle 性能专家而言，查询块的概念非常重要，因为它是 **CBO** 和运行时引擎的基本处理单元。然而，在讨论查询块如何工作之前，我们需要澄清一些术语。

### 术语

*SQL 语言参考手册* 中对查询块的正式定义相当复杂，但非正式地，查询块可以很直接地解释为“跟随关键字 `SELECT` 的内容”。该手册定义了子查询，它有四种可能的构成方式：

*   一个查询块
*   一个查询块后跟一个 `ORDER BY` 子句
*   多个通过集合运算符连接的查询块
*   多个通过集合运算符连接的查询块，后跟一个 `ORDER BY` 子句

集合运算符包括 `UNION`、`UNION ALL`、`INTERSECT` 和 `MINUS`。

**注意** 我对子查询概念的解释与 *SQL 语言参考手册* 略有不同，是为了强调：集合运算符的操作数中包含 `ORDER BY` 子句是不合法的。

事实证明，**CBO** 似乎将所有子查询都视为查询块。它处理方式如下：

*   涉及集合运算符的子查询被视为一种特殊类型的查询块，我将其称为集合查询块。
*   `ORDER BY` 子句被视为其所属查询块的一部分。

从现在起，在本书中使用术语“查询块”时，应理解为 **CBO** 的解释；当我使用术语“子查询”时，我指的是其通常的非正式含义，即 `SELECT` 语句中除主查询块之外的查询块。

### 查询块的处理方式

在 `SELECT` 关键字之后，查询块按以下顺序编写：

*   可选的 `DISTINCT` 或 `UNIQUE` 操作
*   选择列表
*   除 `ORDER BY` 子句外的其他所有内容
*   可选的 `ORDER BY` 子句

然而，查询块的逻辑执行顺序却不同：

*   其他所有内容
*   选择列表
*   可选的 `DISTINCT` 或 `UNIQUE` 操作
*   可选的 `ORDER BY` 子句

好消息是，除了极少数例外，我笼统称为“其他所有内容”的各个部分，其逻辑处理顺序与它们出现的顺序一致。特别是，以下子句（其中一些是可选的）的编写顺序与其逻辑处理顺序相同：

*   `FROM` 子句
*   `WHERE` 子句
*   `GROUP BY` 子句
*   `HAVING` 子句
*   `MODEL` 子句

当然，只要最终生成的行集与逻辑顺序的结果相同，**CBO** 可以自由更改其要求运行时引擎执行操作的顺序。

## 函数

如果你的应用程序构建时，大部分业务逻辑都在数据库之外实现，那么你在 SQL 中可能很少使用函数。然而，大多数商业和科学领域的 SQL 语句都充斥着函数调用。这些函数调用的使用和滥用常常导致大量的处理工作，尤其是排序和聚合；这种开销超过传统的访问和连接表的数据库开销的情况并不少见。因此，在本节中，我想要确保我们都清楚理解主要函数类别的工作方式。

*SQL 语言参考手册* 定义了以下函数类别：

*   单行函数
*   聚合函数
*   分析函数
*   对象引用函数
*   模型函数
*   OLAP 函数
*   数据 cartridge 函数

本节重点介绍单行函数、聚合函数和分析函数。让我们从聚合函数开始。

**注意** 在本节中，我将使用内置函数作为示例，但重要的是要理解，你可以编写自己的标量、聚合或分析函数，并且我将要讨论的同样原则也适用。

## 总结

本章介绍了一种称为 **TSTATS** 的部署方法论，该方法论旨在满足生产环境中对执行计划稳定性的需求，而无需维护一个执行计划仓库。该方法的设置较为复杂，将在本书的第 20 章中详细讨论，但存在替代方法。你可能会使用简化版的 TSTATS，但对于绝大多数行为良好的应用程序，你可能应该尽量减少对象统计信息收集的次数。

与物理数据库设计一起，对象统计信息是部署优化 SQL 时的一个重要考虑因素。在发布的早期阶段，调整统计信息是同时改善应用程序多个部分性能的好方法。然而，在测试生命周期的后期阶段，当目标是限制变更的影响范围时，应尽量减少对对象统计信息的更改，一个合理的替代方案是向你的 SQL 语句中添加提示。
