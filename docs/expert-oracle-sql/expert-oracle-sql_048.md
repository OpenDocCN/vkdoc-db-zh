# 对象统计信息与查询优化

在继续之前，你需要了解你的对象统计信息是否合适。对象统计信息是 CBO 的关键输入，我将在第 6 章解释为什么“合适”不一定意味着“最新”。关键是确保你不会在查询语句性能不佳纯粹是由于过时或不合适的统计信息时，却花费大量精力去调整查询。

如果你正在一个具有生产类数据的测试系统上工作，并且该测试系统上的对象统计信息已过时，那么简单的方法可能就是重新收集它们。但是，如果你的生产系统上有一些扩展统计信息或手工创建的直方图，最好确保测试系统上也存在相同的扩展统计信息和直方图。也许最好的做法是将统计信息从生产系统复制到测试系统。

我将在下一章更详细地讨论对象统计信息，现在不想在这里深入太多。我这里想强调的主要观点是，在调优过程的早期就需要考虑对象统计信息的状态。

## 修改代码

以功能等效的方式重写查询可能有助于 CBO，这有几个原因。我现在提几个。我将在第 16 章更深入地讨论这个话题。

### 添加信息

有很多方法可以向 CBO 提供信息以帮助它。这里只是一个例子。

除非一个列被保证总是`NOT NULL`，否则你不能为它添加`NOT NULL`约束。然而，在你选择的特定时间，你选择的特定行可能被保证是`NOT NULL`的，你可以添加一个谓词，如“`WHERE c1 IS NOT NULL`”。这样，CBO 就多了一点可能有用的信息，可以将其纳入执行计划的准备中。在 Listing 5-2 这个简单查询的例子中，我们无法添加任何有用的信息。

### 转换查询

正如我在 Listings 1-1 和 1-2 中所展示的，有时你可以以完全不同的方式重写查询，使 CBO 产生完全不同的执行计划。似乎没有适用于 Listing 5-2 中查询的有用转换，所以我们继续。

### 添加提示

虽然我认为提示是在生产系统上使 SQL 语句快速运行的一种完全可以接受的方式，但现在不是就这个话题进行辩论的时候。在这个阶段，我们只是试图通过一系列实验来识别一个性能良好的执行计划。你会发现所有知名的 Oracle 专家都会同意，*仅用于实验目的*，你在寻找最有效的执行计划时，不应以任何方式抑制自己使用提示，无论是文档化的还是未文档化的。

和你们大多数人一样，我经常问自己这样的问题：“为什么 CBO 使用了全表扫描，而不是使用我专门为这类查询创建的索引？”好吧，在花几天时间进行理论推导之前，我通常会在查询中添加一个`INDEX`提示，并在可能的情况下重新运行查询。有几种可能的结果：
- 添加`INDEX`提示后，CBO 仍然使用了全表扫描。这表明索引的使用在某种程度上是非法的。我现在就得到了问题的一半答案。另一半是：“为什么索引的使用是非法的？”
- 索引被使用了，但查询实际上并没有变快，甚至可能更慢。我现在有了一个新问题：“我做了什么错误的假设，让我相信索引会更快？”
- 索引被使用了，查询的耗时减少了。这个实验结果引出了一个不同的新问题：“CBO 做了什么错误的假设，让它认为索引不会更快？”如果有数百个类似的 SQL 语句，那么这个新问题就极其重要。不过，在这一点上，这些担忧并不太让我担心；我可以继续调整查询的其他部分，同时保留这个提示。

在 Listing 5-2 的查询案例中，表`T2`上还没有任何索引。唯一可能有帮助的提示`PARALLEL`在我的笔记本电脑上不起作用，因为单个本地硬盘一次只能做一件事。我们继续吧。

## 对数据库进行物理更改

人们为改善查询性能而进行的最常见物理数据库更改是添加一个或多个索引。然而，还有许多其他物理设计更改可能有助于改善查询性能，我将在第 15 章介绍这些。这里只举几个例子。

如果你拥有必要的许可证，你可能会实施分区或压缩。你可能会创建物化视图或执行其他形式的反规范化。当然，在 Listing 5-2 显示的特定情况下，我们可以简单地移除我特意添加以减慢全表扫描的`PCTFREE 99`！这在现实生活中可能不常是一个选项，但你可能会偶尔发现一个表的块中有大量空闲空间，通过移动表，你可以显著减小其大小并改善访问时间。

即使你决定添加索引将有助于查询性能，仍然有几个问题需要问。像任何物理设计更改一样，你脑海中浮现的第一个想法应该是评估添加索引对 DML 语句的影响。每次向表中插入一行、每次从表中删除一行、每次更新表中的索引列时，你都需要调整表的索引。索引越多，DML 所需的时间就越长。不仅如此，索引还会占用缓冲区缓存和磁盘上的空间，这些空间可能更适合用于其他目的。



我在此自相矛盾了一下。毕竟，你现在仍处于实验阶段，因此在这个阶段无需把所有事情都想周全。但某些物理设计变更需要大量工作，并且会影响到其他测试人员，因此，在你投入大量精力去实现一个在实际中可能相当不切实际的想法之前，值得考虑一下你的提议可能带来的更广泛影响。

我们假设你仍然热衷于添加索引的想法。然后，你需要问自己：要对哪些列建立索引？顺序如何？也许在索引中额外添加一列会对其他查询有帮助。另一件需要考虑的事情是使用一个或多个位图索引的可能性，如果数据是批量加载且之后很少（如果有的话）更新，这并不是一个坏选择。对于窄表，改为使用`索引组织表`也可能是一种选择。

添加索引时最后要考虑的是要压缩多少列（如果有的话）。以清单 5-2 中的表`T2`为例，我们知道被索引的列会重复很多次，因此压缩它似乎是合理的。清单 5-5 在`T2.C2`上添加了一个索引，并显示了来自`V$SQL_PLAN_STATISTICS_ALL`的修正后数据。

清单 5-5. 添加压缩索引

```
CREATE INDEX t2_i1
   ON t2 (c2)
   COMPRESS 1;
```

--- ------------------ -------------- ------- --------------- -------- --------
  0 SELECT STATEMENT                     .004                        1      400
  1  HASH JOIN                           .004                        1      400

哇！我们的查询现在只需几分之一秒就能运行！我们看到，添加索引后，执行计划几乎变得面目全非。关键点在于，我们的全表扫描已被操作 7 所取代，该操作仅对`T2`进行了 400 次单块访问。

### 修改环境

如果你的查询似乎已经调优得非常完美，但性能仍然无法接受，那该怎么办？当然，升级硬件总是一个选项，但通常还有远不那么激烈的措施可用。你可以尝试增大缓冲区缓存的大小以减少磁盘读取次数。你可以增大 PGA 的大小或将`WORAKAREA_SIZE_POLICY`设置为`MANUAL`，以便排序能在内存中完成。我们将在第 17 章详细讨论排序。你可能还想并行运行查询，我们将在第 8 章讨论并行执行计划。

### 运行 SQL 调优顾问

一个不幸的事实是，DBA 通常负责大量的应用程序和数据库，但往往在最后一刻、常常是在非工作时段被叫去处理他们几乎没有背景知识的性能问题。在这些情况下，很难像我在此所倡导的那样，为性能问题构建详细的业务和技术背景。

幸运的是，帮助可能近在眼前。`SQL 调优顾问`会对它在数据库中能看到的内容执行自动化检查。它检查统计信息是否过时或具有误导性。它查找列中隐藏的相关性，并进行一些其他基本检查。我相信它的工作方式是运行部分 SQL 语句，以查看 CBO 的理论想法在实践中效果如何。

关于`SQL 调优顾问`，有好消息也有坏消息。坏消息是，`SQL 调优顾问`无法解决除最基本调优问题之外的所有问题：
*   它并没有比 CBO 拥有更多的 SQL 转换方法，并且执行与 CBO 相同类型的成本计算，因此它经常得出与 CBO 相同的结论。
*   它可以推荐一个新索引，但其建议应持保留态度，因为 CBO 实际上并不创建索引，因此对诸如*聚簇因子*之类的索引统计信息一无所知。

![image](img/sq.jpg) `提示` 当查询涉及临时表时，切勿运行`SQL 调优顾问`。在实际应用中，应用程序会在调用查询前填充临时表，但`SQL 调优顾问`不会。因此，生成的建议通常完全错误。

好消息是，虽然`SQL 调优顾问`只能帮助解决最基本的调优问题，但许多调优问题实际上相当基础！正因如此，即使在处理我非常熟悉的应用程序时，我有时也会使用`SQL 调优顾问`。当然，现在我通常能立即看到简单调优问题的解决方案，因此我使用`SQL 调优顾问`现在主要限于午餐时间！如果我在准备出去吃饭时遇到一个调优问题，在出发前启动`SQL 调优顾问`是相当直接的。有时，等我吃完午餐回来，答案已经在等着我了！

你可以从*企业管理器* (`EM`) 调用`SQL 调优顾问`，但当你登录到`EM`时，你本可以从`SQL*Plus`启动任务然后去吃午餐了。清单 5-6 显示了一个你可以使用的通用 SQL 脚本。

清单 5-6. 从 SQL*Plus 创建 SQL 调优顾问任务

```
VARIABLE t VARCHAR2(20)

BEGIN
   :t := DBMS_SQLTUNE.create_tuning_task (sql_id => '&SQL_ID');
   DBMS_SQLTUNE.execute_tuning_task (:t);
END;
/

SET LINES 32767 PAGES 0 TRIMSPOOL ON VERIFY OFF LONG 1000000 LONGC 1000000

SELECT DBMS_SQLTUNE.report_tuning_task (task_name => :t, TYPE => 'TEXT')
  FROM DUAL;

EXEC dbms_sqltune.drop_tuning_task(:t);
SET LINES 200
```

思路与我之前展示的 SQL 性能监视器脚本相同。你使用`SQL*Plus`的`DEFINE`命令来指定`SQL_ID`，然后调用你保存在某个方便位置的脚本。你看到突出显示的部分`TYPE => 'TEXT'`了吗？在这种情况下，唯一支持的类型是`'TEXT'`，因此该脚本不是生成供浏览器查看的`HTML`报告，而是将报告发送到终端。

`DBMS_SQLTUNE.CREATE_TUNING_TASK`过程有几个你可能想查看的选项。例如，分配给调优任务的默认时间是 30 分钟，这常常是不够的。当`SQL 调优顾问`达到其时间限制时，所有发生的事情就是它生成一个简短的报告，说明已超时。你可能想增加超时时间，特别是如果你计划吃一顿长午餐的话！

我们要看看`SQL 调优顾问`对清单 5-2 中的查询有什么建议吗？请看清单 5-7。

清单 5-7. 删除清单 5-5 中的索引并运行 SQL 调优顾问

### 重新思考需求

有时，解决方案的成本和复杂性似乎与所要解决的问题完全不成比例。这真的值得投入所有的时间和精力吗？让 200 个并行查询从进程全速运行半小时，是否会对系统其他部分产生太大影响？确实有时需要提出这类问题。或许你被要求调优的那个每日报告，可以在系统空闲的周末只运行一次。也许查询可以简化，同时仍能为业务使用者提供完全可用的数据。

尽管在时机合适时不应害怕提出这类问题，但你需要确保在提问前掌握事实。你不希望你的老板让你的同事“再看一下”，然后他或她找到了一个快速的解决方案。那会很尴尬！如果你有任何疑问，何不先与同事一起复核一下你的发现？

## 总结

本章概述了系统化调优 SQL 语句的方法。首先需要尽可能透彻地理解手头的问题。下一步是分解时间消耗在何处。最后，使用迭代的更改和重新测试过程，直到性能达到可接受的水平。

在现阶段，我还没有给你太多关于查询转换的示例，没有关于如何正确使用提示的信息，并且只是非常简要地讨论了物理数据库转换。这些主题都将在本书后面更详细地讨论。

请始终记住，调优是一个实验过程，旨在确定哪里出了问题以及为什么出问题。这与在生产系统中的部署关系不大。现在让我们进入第 6 章的这个主题。
