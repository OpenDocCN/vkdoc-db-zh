本章阐述了一种可用于在生产系统上管理对象统计信息的激进方法。我的一位客户将这种方法命名为 TSTATS，意为“Tony 的统计信息”，我必须说我很喜欢这个名字。它满足了我的虚荣心！然而，正如我已经说过的，我并非唯一考虑本章所解决问题的人，其他人也独立开发了类似的解决方案，例如 Adrian Billington。

我在第 6 章解释了 TSTATS 要解决的问题：在生产系统上收集统计信息是一项有风险的活动，收集到的统计信息可能导致执行计划性能不佳。但是，如果按照 Oracle 推荐的传统方法来管理对象统计信息，这种风险是不可避免的。TSTATS 允许你消除在生产系统上收集统计信息这一有风险的活动，让你能够高枕无忧，知道在你休息时执行计划不会变得更糟。

关于 TSTATS，有好消息也有坏消息。坏消息是 TSTATS 不是一个快速的“即插即用”解决方案。对于一个复杂的应用程序，你可能需要花费数周甚至数月的时间来准备部署 TSTATS。你需要一位 Oracle 性能专家的服务来启动 TSTATS。请注意，我说的是*专家*，而不是*有经验的*。如果这样的专家不能全职在岗，你需要安排他来审查执行计划并确保其正常运行。此外，你很可能会遇到同事和管理层的反对；有些人会（相当错误地）声称 TSTATS 会使你的数据库无法获得 Oracle 支持。

### TSTATS 与 Oracle 支持

`DBMS_STATS` 包中用于手动创建或更改统计信息的过程是完全文档化并受支持的。使用这些提供的过程与手动更新数据字典中的表有很大不同：使用提供的过程是受支持的，但直接更新数据字典中的行则不受支持。

确实，如果你对对象统计信息的更改导致执行计划比未更改前更差，Oracle 支持不太可能会特别同情。但是，如果你遇到任何其他类型的问题（例如执行期间出现 ORA-0600 错误），无论你是否使用 TSTATS，你都应该拥有向 Oracle 寻求支持的同等权利。

关于 TSTATS 的好消息是，当它被正确开发时，它是有效的！在初始设置之后，你可能永远不再需要在任何开发、测试或生产系统上收集统计信息了，当然，除非是为新添加到应用程序中的表或索引生成统计信息。

![image](img/sq.jpg) **注意** 我在 TSTATS 方面犯的最大错误是在从 10g 升级到 11g 时重新收集了统计信息。这完全没有必要，只是制造了许多不必要的工作。在成功部署 TSTATS 后，对表重新收集统计信息要非常谨慎。

成功的 TSTATS 部署的两个关键是管理层的支持和正确完成这项工作的承诺——一个考虑不周的实施只会让反对者欢呼“我早就告诉过你”，而你将永远不会再有第二次机会。

考虑到这些，在继续之前，你需要思考以下问题的答案：

*   你的性能稳定性问题是否严重到值得投入 TSTATS？
*   是否有更简单的解决方案，例如减少收集统计信息的频率？比如作为季度发布的一部分来收集？
*   是否有办法*安全地*简化 TSTATS 方法，或者你真的需要本章描述的所有花哨功能？
*   你的问题是整个应用程序范围的，还是可以在某种程度上限制 TSTATS 的范围？
*   本章是否解释了解决你问题所需的所有内容，或者你的应用程序是否有需要额外考虑的特殊情况？例如，在特定情况下，数据量是否会有多个数量级的变化？

当你回答这些问题时，请记住没有绝对的对错答案。例如，如果你决定将 TSTATS 的范围缩小到几个关键表，决定不考虑直方图，并对所有使用范围谓词的查询使用提示，你可能低估了需要做的事情。就像本书及生活中其他地方提出的所有问题一样，决策最好在掌握尽可能多的信息后做出，因此请与你的同事交流并慢慢来。

### 管理列统计信息

当你停止在生产系统上收集统计信息时，朝着稳定执行计划迈出的最重要一步是，按照清单 6-8 所示的方式消除列统计信息的高值和低值。然而，还有几个其他问题需要考虑，例如：

*   只有一个值的列
*   具有倾斜数据值的列
*   相关的列和表达式
*   用于范围谓词的列

重要的是要认识到，大多数表中的大多数列不需要任何特殊考虑：与未在 `WHERE` 或 `HAVING` 子句谓词中使用的列相关的统计信息不会被 CBO 使用，即使一个列用于谓词中，它也可能不是基于时间的，可能没有任何倾斜数据，可能不会在谓词中与任何相关列一起使用，也可能不用于范围谓词。重要的是不要低估手头的任务，但同样重要的是不要高估它。
