# 引言

### 本书内容
我们中有多少人曾在某个不合时宜的时段被叫醒，只因一条原本运行良好的 SQL 语句因执行计划变更而突然表现糟糕？在大多数情况下，Oracle 可能会建议遭受此类问题反复困扰的客户研究使用 SQL 计划基线，这是 Oracle 用于稳定生产环境执行计划的战略特性。本书则介绍 `TSTATS`，这是我的一位客户给一项颇具争议的技术起的名字，该技术完全绕过了在生产系统上收集对象统计信息的需要，可以被视为 SQL 计划基线的一种替代方案。

尽管第 6 章和第 20 章专门讨论在生产环境中部署和管理统计信息的问题，但本书的主题是针对 Oracle 数据库的 SQL 调优。市面上还有其他优秀的书籍涵盖 Oracle 数据库的整体性能，但本书专门聚焦于 SQL 调优。

在我看来，找到 SQL 调优问题的最佳解决方案，关键在于完全理解你正在处理的问题以及你掌握的技术。本书的大部分内容致力于研究基于成本的优化器（`CBO`）和运行时引擎如何工作，以及如何获取和解读诊断数据，例如由 `DBMS_XPLAN` 包中的函数显示的执行计划。

有些读者可能会感到惊讶的是，我很少提及著名的 `10046` 和 `10053` 跟踪，它们构成了许多 SQL 性能书籍的基础。在实践中，我大约每年使用一次 `10046` 跟踪，大约每三年使用一次 `10053` 跟踪。在我看来，对于绝大多数 SQL 调优问题，有更容易的诊断方法，我将详细解释我所使用的技术。

你会注意到我在最近几段中使用了两次“在我看来”，在本书中我还会多次使用这个词。害怕疏远读者使许多作者不敢表达有争议的观点，特别是当这些观点与 Oracle 推荐的最佳实践不完全一致时。但看待一个主题的方式往往不止一种，我希望本书能为你提供一些新的思考素材。

但别太担心本书全是哲学和夸夸其谈。本书中有大量你在其他书籍或博客中找不到的技术内容，以及丰富的示例来帮助你学习。

### 我为何写这本书？



促使我写作本书的过程始于 2010 年英国 Oracle 用户组（UKOUG）全国大会上凯尔·海利（`http://www.oraclerealworld.com/`）的一次演讲。凯尔演讲的具体主题并不重要，但他顺带提及了一本名为《SQL Tuning》的书，该书由丹·托于 2003 年撰写，奥莱利出版社出版。当时我坐在听众席的乔纳森·刘易斯（`http://jonathanlewis.wordpress.com/`）旁边，他也同意凯尔的看法，认为这是一本极好的书，是他博客上仅推荐的少数几本书之一。我感到有义务购买这本书，并可以确认它是一部杰出的出版物。

我在 2014 年阅读丹的这本书时遇到的一个小困难是，它几乎完全专注于一种科学且万无一失的方法，来确定一组表的正确连接顺序。尽管连接顺序仍然是一个性能关注点，但由于以下几个原因，它在 2014 年的重要性已不如 2003 年：

*   自从 CBO（基于成本的优化器）在 Oracle 数据库第 7 版中引入以来，它识别正确连接顺序的能力已变得越来越强。
*   新的选项，例如右深层连接树和星型转换（我将分别在第 11 章和第 13 章中介绍），意味着 CBO 选择灾难性连接顺序的可能性进一步降低。
*   我们现在可以使用沃尔夫冈·布赖特林的《基于基数反馈的调优》（`http://www.centrexcc.com/Tuning%20by%20Cardinality%20Feedback.pdf`）调优技术（我将在第 6 章简要讨论），它为解决像连接顺序这样的简单问题提供了一种更简单的方法。沃尔夫冈的方法远不如丹的方法正式和万无一失，但它在 90%的情况下都有效，并且更容易掌握。

尽管在 2014 年，连接顺序已不像 2003 年那样是个大问题，但新的挑战出现了。基础数据库产品（无需额外许可的部分）现在包含了分析函数、并行处理和`MODEL`子句，所有这些都为数据库层，特别是 SQL 语句中的业务逻辑，开启了更复杂的选项。获得许可的选项，例如分区，也有助于解决一些原本可能非常繁琐（至少）的问题。所有这些优秀的新特性都带来了复杂性，伴随着这种复杂性，就需要理解 SQL 语句行为方式的更多方面。例如，第 17 章专门讨论了排序这一主题。

于是，本书的构思便诞生了。2011 年 12 月，在下一届 UKOUG 大会上，我仍在反复思考写作本书的想法，并向我认识的其他作者寻求一些鼓励。我得到了好坏参半的回应。是的，一本关于 SQL 调优的新书会很好。但考虑到所涉及的工作量，如果我承接这项工作，那一定是疯了。

但我当时已在情感上做出了承诺，不久之后，我请《Troubleshooting Oracle Performance》的作者克里斯蒂安·安托尼尼将我介绍给他在 Apress 出版社的编辑。
