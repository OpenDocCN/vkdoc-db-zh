# 第 4 章

## 错误与异常

作为软件开发人员，我们常常幻想一个没有缺陷软件的乌托邦世界——在遥远岛屿的棕榈树荫下，啜饮着长长的水果鸡尾酒，轻松完成开发。但是，唉，回到现实世界，大批开发人员坐在格子间里，喝着苦涩的咖啡，与那些并不总是他们的错或完全在他们控制范围内的 Bug 作斗争。

即使在经过最严格测试的软件中，异常也可能发生，仅仅是因为不可能预先检查每一个错误条件。例如，你知道如果一名清洁工在清洁数据中心地板时，不小心将一些拖把水溅入数据库服务器的风扇外壳会发生什么吗？它可能会崩溃，也可能不会；它可能只是导致应用程序深处某个组件发生故障，并发出一条奇怪的错误消息。

尽管大多数异常不会如此远离可测试的范围，但理解当它们发生时如何处理它们肯定是重要的。对于 `SQL Server` 开发人员来说，理解如何处理错误——无论是服务器本身抛出的错误，还是专门为应用程序运行时发生的问题构建的自定义错误——也是至关重要的。

### 异常与错误

术语 `exception`（异常）和 `error`（错误）虽然经常被开发人员互换使用，但实际上指的是略有不同的情况：

`error` 可能在程序运行过程中出现问题时发生，即使它本质上可以是纯信息性的。例如，程序显示消息告知用户问号是文件名的无效字符，这被视为错误消息。然而，这可能意味着也可能不意味着程序本身处于无效状态。

另一方面，`exception` 是由 `异常情况` 导致的错误。例如，如果正在使用一个网络库来发送数据包，而网络连接因有人拔掉网线而突然断开，该库可能会抛出一个异常。异常告知调用代码某处出错了，例程意外中止。如果调用者不 `handle`（处理）该异常（即捕获它），其执行也将中止。这个过程将不断重复，直到异常被处理，或者它到达调用栈的最高层，此时整个程序将失败。

另一种思考异常和错误的方式，是将错误视为程序 `预期` 会发生的情况。当文件名包含无效字符时显示的错误消息本质上是信息性的，因为程序的开发者预测到会发生此类事件，并专门创建了一条代码路径来处理它。另一方面，网络连接中断可能由无数种情况引起，因此更难以具体处理。相反，解决方案是引发一个异常并失败。然后，该异常可以由调用栈中更高级别的例程处理，该例程可以决定采取何种行动方案来解决问题。
