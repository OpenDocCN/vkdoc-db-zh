# 数据库世界的软件开发方法论

## 重大的平衡之举

归根结底，软件开发的真正目标是生产出客户愿意使用的可工作软件，同时，该软件应能随着时间推移和需求变化而易于修复或扩展。然而，在开发软件时，存在限制实际可实现成果的硬性约束。没有任何项目拥有无限的时间或金钱，因此通常需要在一个领域做出牺牲，以便优先满足另一个领域更高的要求。

在大多数情况下，数据库是其驱动的应用程序的核心。数据在很大程度上控制着应用程序，没有数据，应用程序的价值将大打折扣。

同样，数据库也常常是应用程序在性能、可维护性和其他关键成功因素方面面临真正挑战的地方。应用程序开发人员通常倾向于将这些问题尽可能推到数据层去解决。在没有数据架构师的情况下，数据库开发人员就成了负责平衡整个应用程序需求的人。

要试图取得恰当的平衡，通常需要在以下几个领域进行权衡取舍：

-   性能
-   可测试性
-   可维护性
-   安全性
-   为未来需求预留空间

平衡这些相互竞争方面的要求并非易事。接下来，我将对这些问题提出一些初步思考；本书其余部分的示例将更详细地阐释这些内容。

### 性能

我们生活在一个越来越缺乏耐心的社会。客户和管理层的要求必须*立刻*（或者有时是昨天）得到满足。我们想要快餐、快车和快捷的服务，并一直在寻求各种即时满足。这种对速度的需求无疑适用于数据库开发领域。用户似乎持续不断地感觉应用程序的性能未达到应有的水平，即使这些应用程序正在执行大量的工作。有时感觉用户似乎更愿意以尽可能快的速度获取*任何*数据，而不是在需要稍等片刻的情况下获取*正确的*数据。

当然，问题在于性能并非易事，并且可能打破整个平衡。

构建一个真正高性能的应用程序通常需要牺牲。功能可能需要削减（应用程序需要做的工作越少，速度就会越快），安全性可能需要降低（更少的授权循环意味着更少的工作），或者低效的代码可能需要用晦涩、难以维护的方式重写，以便从服务器中榨取每一个 CPU 周期。

那么，我们如何调和这种对极致性能的需求——许多人似乎只关心这个而忽略其他一切——与对开发最佳实践的需求呢？不幸的是，答案是有时我们只能尽力而为。大多数时候，如果我们发现自己处于用户抱怨性能的境地，并且如果不解决就会损失金钱或工作，用户不想听到为什么修复性能问题会增加耦合度和降低可维护性。用户只想要软件运行得快——而我们别无选择，只能交付。

坚持最佳实践的一个幸运事实是，出于多种原因，它们通常被认为是处理事情的最佳方式。在开发周期中密切关注耦合、内聚和正确封装等问题，不仅可以减少性能问题的发生，还会使修复大多数问题变得容易得多。而在那些你需要打破一些“完美”代码以使其达到必要速度的少数情况下，请知道这不是你的错——是社会把你置于这种境地！

### 可测试性

不经过彻底测试就发布任何产品，至少可以说不是明智之举。然而，开发人员利用反模式导致难以或无法进行适当测试的情况很常见。许多这类问题源于试图创建“灵活”模块或接口的尝试——与其正确划分功能并密切关注内聚性，有时人们会忍不住创建“包罗万象”、单体的例程，试图包办一切。

开发这类例程会产生永远无法完全测试的软件。单个例程可能用例的组合爆炸可能非常巨大——尽管在大多数情况下，应用程序用户实际使用的组合要有限得多。

在仅仅为了灵活性而实施灵活的解决方案之前，请仔细考虑。它真的需要那么灵活吗？这些功能是会立即被完全利用，还是可以以后根据需要慢慢扩展？

### 可维护性

在应用程序的整个生命周期中，各种模块和例程都需要以增强和错误修复的形式进行维护和修订。影响例程可维护性高低的问题与影响可测试性的问题相似，但也有一些不同之处。

在确定给定例程的可测试性时，我们通常只关心接口是否足够稳定以允许编写测试用例。而在确定可维护性水平时，我们也关心暴露的接口，但原因略有不同。从可维护性的角度来看，最重要的接口问题是耦合。紧密耦合的例程往往具有更高的维护成本，因为任何更改都必须传播到多个例程，而不是在一个地方进行更改。

可维护性的问题也超出了接口，延伸到实际的实现。一个例程可能拥有稳定、简单的接口，但其实现却可能复杂、缺乏文档，难以处理。一般来说，例程中的代码行数越多，维护就越困难；但由于大型例程也可能预示着内聚性问题，如果开发人员在设计过程中有所留意，这类问题应该能在早期被发现。

值得注意的是，我所提出的“数据库即 API”思维方式要求使用存储过程作为数据的接口，但并不涉及你使用何种协议来访问这些存储过程的细节。许多软件公司发现，Web 服务是提供标准、跨平台接口层的好方法，例如使用 `ADO.NET data services` 基于 `entity data model` 生成 `RESTful web service`。使用 Web 服务是否优于其他协议，必须视具体情况而定；像任何其他技术一样，它们当然也可能被错误地使用或在错误的场景中使用。请记住，Web 服务需要更多的网络带宽，并且遵循与 `SQL Server` 支持的其他协议不同的身份验证规则——它们的使用最终可能带来的问题比解决的更多。

通过使用具有正确定义接口和完全信息封装的存储过程，应用程序与数据库之间的耦合将大大降低，从而形成一个更易于随时间维护和演进的数据库系统。

仅用几段话很难强调存储过程在一个设计良好的 `SQL Server` 数据库系统中扮演的重要性。为了强化数据库必须被视为 API 而非仅仅是持久层的观点，本书将在后续章节中结合处理外部系统接口的示例，重新审视这个主题。



正如可测试性一样，可维护性在某种程度上也受到创建“灵活”接口尝试的影响。一方面，接口的灵活性可能通过要求调用者过多了解参数组合、可重写选项等，而增加例程之间的耦合度。
另一方面，具有灵活接口的例程有时可能更容易维护——至少在项目初期是如此。在某些情况下，使例程尽可能通用化，可能导致系统所需的总例程数量减少，从而需要维护的代码量也随之减少。然而，随着功能的增加，这些通用例程易于修改的特性往往会因每个新选项或参数带来的复杂度增加而减弱。因此，在项目早期旨在实现一定灵活性，然后在可维护性开始下降时再进行重构，通常是有利的。
