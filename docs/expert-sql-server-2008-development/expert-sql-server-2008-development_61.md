# 第五章 „ 权限与授权

## 概述

SQL Server 安全性是一个广泛的主题领域，有足够的潜在探索途径，以至于有整本的书籍专门讨论这个话题。本章的目标不是涵盖创建端到端安全产品所必需的全部安全知识，而是聚焦于在软件设计和开发过程中最重要的那些领域。

广义上讲，数据安全可以分为两个领域：
- **身份验证**：验证系统用户身份的行为。
- **授权**：授予用户访问系统控制资源的行为。

这两个领域在许多情况下可以分别委托；只要身份验证部分工作正常，用户就可以在会话的剩余部分移交给授权机制处理。

SQL Server 身份验证本身就是一个大课题，包含网络安全性、操作系统安全性以及所谓的服务器“表面区域控制”等多样化子主题。虽然生产环境的 DBA 应该非常关注这类问题，但身份验证是一个开发者在很大程度上可以忽略的领域。开发者需要更关注的是身份验证*之后*发生的事情：即用户如何被授权访问数据，以及数据如何免受未授权用户的侵害。

本章从开发者的视角，介绍了 SQL Server 中数据权限和授权的一些关键问题。包括对权限的初步讨论，以及使用 SQL Server 权限保护数据的通用指南和实践。一个相关的安全主题是数据加密，这将在下一章详细讨论。

请注意，尽管本文中通常忽略身份验证问题，但在日常开发工作中，你不应完全无视它们。开发环境为了简化设置，往往安全性非常宽松，但一个扎实的开发流程应包括一个测试阶段，在此阶段应用完整的身份验证限制。这有助于确保部署到生产系统时，不会以用户甚至无法登录的惨烈失败告终！

## 用户登录与应用程序登录

本章涵盖的主题涉及在 SQL Server 内部处理的各种权限和授权场景。然而，在许多数据库应用程序设计中，授权是在应用层而非数据库层处理的。在此类应用程序中，用户通常使用其个人凭证连接并登录应用程序，但应用程序随后使用一个共享的应用程序登录名连接到数据库。该登录名被授予执行数据库中与该应用程序相关的所有存储过程的权限，而确定任何特定用户可以执行哪些操作则由应用程序本身的授权例程决定。

使用这种方法有一些好处，例如能够利用不同会话之间的连接池。然而，这也意味着 SQL Server 提供的处理每用户安全性的任何功能都不适用。如果应用程序中存在错误，或者与应用程序登录名关联的凭证被泄露，用户就有可能执行应用程序有权执行的任何数据库查询。
