## 性能监控工具

使用单元测试、功能测试和回归测试进行验证，对于彻底测试应用程序行为是否正确极为重要，但真正吸引大多数开发人员注意力的还是性能测试。性能测试对于确保积极的用户体验至关重要。用户不希望等待超过绝对必要的时间来获取数据。

性能测试依赖于收集、审查和分析系统不同方面的性能数据。因此，在深入探讨如何分析系统性能之前，有必要先了解一些可用于捕获此类数据的工具。`SQL Server 2008` 提供了许多内置工具，允许 `DBA` 和开发人员存储或查看关于服务器上活动的实时信息，包括以下工具：

*   `SQL Server Profiler`
*   服务器端跟踪
*   系统监视器控制台
*   动态管理视图 (DMVs)
*   扩展事件
*   数据收集器

此外，还有许多第三方监控工具可用，它们可以以不同的方式测量、聚合和呈现性能数据。在本节中，我将讨论一些不同的性能监控方法，以及它们可以适用的情况。

### 实时客户端监控

随 `SQL Server 2008` 提供的 `Profiler` 工具非常有用且易于使用。只需加载 `Profiler` 应用程序并将其指向你想要监控的 `SQL Server` 实例，它就会基于大约 200 种不同事件报告实时信息。然而，对于大多数性能监控工作，你只需要关注少数几个关键事件。

在最初为应用程序建立基线时，我通常从只查看 `SQL:BatchCompleted` 和 `RPC:Completed` 事件开始。这两个事件都在查询完成时触发；它们之间的唯一区别是，`RPC:Completed` 在远程过程调用 (`RPC`) 完成时触发，而 `SQL:BatchCompleted` 在 SQL 批处理完成时触发——不同的访问方法，相同的结果。

对于这两个事件，最有价值的列是 `CPU`、`Reads`、`Writes` 和 `Duration`：

*   `CPU` 列报告用于解析、编译和执行查询的 CPU 时间（以毫秒为单位）。由于该列包含



进入`compilation time`阶段，通常可以观察到报告的时间在连续查询中呈下降趋势，这得益于计划缓存。

- `Reads`列报告查询执行的`logical`读取次数。`logical` I/O 发生在 SQL Server 的查询引擎请求数据时，无论是从物理磁盘还是从缓冲区缓存获取。如果在此列中看到高数值，不一定表示性能问题，因为大多数数据可能从缓存中读取。然而，即使从内存中读取数据也会消耗服务器的 CPU 时间，因此最好尽量减少任何类型的读取。
- `Writes`列报告查询执行的`physical`写入次数。这意味着只有在查询过程中实际持久化到磁盘的写入操作才会被报告。
- `Duration`列报告调用总耗时，以毫秒为单位。查询的持续时间直接反映了用户体验，因此通常从此入手分析。如果应用程序运行缓慢，您可以使用此列找到性能最差的查询。

通过查看这些列中包含的高层级信息，您可以识别出值得进一步调查的潜在候选对象。首先，考虑系统中任何给定查询需要设定的限制。查询允许运行的最长时间是多少？平均运行时间应该是多少？通过对`Duration`列进行聚合分析，您可以确定这些时间限制是否已被超出。

一旦您隔离了可能的问题区域（请参阅本章后面的“Granular Analysis”部分），您可以使用更详细的事件集进行深入探究。例如，`Scan:Started`事件可用于识别可能未有效使用索引并因此可能导致 I/O 问题的查询。另一方面，`SP:Recompile`事件表示查询被查询优化器重新编译，因此可能消耗了比必要更多的 CPU 时间。

## 第三章：测试数据库例程

### 服务器端跟踪

虽然 SQL Server Profiler 是一个方便且有用的工具，但它有一些限制。主要问题是，为了便于实时数据收集和显示，SQL Server 需要持续将数据流式传输回该工具——这样做会产生开销。因此，尝试监控性能的行为可能对被测量系统的性能产生负面影响，导致结果出现偏差（即“观察者效应”）。

在高事务吞吐量的性能测试中，您应该通过使用`server-side traces`而不是 Profiler 工具来最小化监控对测试结果的影响。`server-side trace`在 SQL Server 后台运行，将其结果保存到服务器上的本地文件，而不是流式传输到客户端。

可以手动在 T-SQL 中定义`server-side trace`的参数，但这样做是一个繁琐且不必要的过程。更好的方法是基于从 SQL Server Profiler 工具导出的跟踪定义来创建`server-side trace`，如以下步骤所述：

1.  首先，使用 SQL Server Profiler 定义跟踪所需的事件、列和筛选器。
2.  选择 File > Export > Script Trace Definition > For SQL Server 2005 – 2008，并选择文件名以保存脚本。
3.  保存脚本后，在 SQL Server Management Studio 中打开它。此文件包含基于提供的参数启动跟踪所需的 T-SQL 代码。
4.  编辑脚本的以下行，通过指定跟踪结果的有效输出路径和文件名：
   ```
   exec @rc = sp_trace_create @TraceID output, 0, N'InsertFileNameHere', @maxfilesize, NULL
   ```

**注意** 指定的文件名不应包含任何类型的扩展名。跟踪将自动添加一个。



## 3.1 操作追踪脚本

5.  你可能还需要修改`@maxfilesize`参数的值，该参数默认设置为 5MB。增加最大文件大小有助于最小化在追踪期间创建的滚动文件数量。我通常将其设置为 200MB 作为起点。
6.  完成编辑后，执行脚本。追踪将在后台开始收集数据，生成的脚本将返回一个追踪标识符`TraceID`，你应该记下它，因为稍后控制追踪时将需要它。

按照这些步骤操作后，追踪将会运行，所有指定事件的输出都将被写入日志文件，所以现在是执行你的测试套件的时候了。当你完成追踪后，必须使用`sp_trace_setstatus`存储过程停止并关闭追踪，该过程需要提供启动追踪时返回的`TraceID`追踪标识符。以下代码清单演示了这一点（此例中追踪标识符列为 99）：

```sql
EXEC sp_trace_setstatus @traceid=99, @status=0;
EXEC sp_trace_setstatus @traceid=99, @status=2;
GO
```

一旦追踪被停止并关闭，就可以使用`sys.fn_trace_gettable`函数从追踪文件中读取数据。此函数接受两个参数：追踪文件名的完整路径（包括 SQL Server 自动添加的`.trc`扩展名），以及要读取的最大滚动文件数。以下 T-SQL 将用于从路径`C:\Traces\myTrace.trc`读取追踪文件。滚动文件的数量设置得足够高，以便即使追踪多次滚动到新文件，也能读回所有数据：

```sql
SELECT *
FROM sys.fn_trace_gettable('C:\Traces\myTrace.trc', 999);
GO
```

以这种方式选中后，追踪数据可以像 SQL Server 中的任何其他数据一样使用。它可以被插入到表中、查询或以多种方式进行聚合，以评估哪些查询可能正在引发问题。

## 3.2 系统监控

要从更宏观的角度了解服务器的整体性能，系统性能计数器是一种宝贵的资源。可以使用“系统监视器”控制台（也称为“性能监视器”或`perfmon.exe`，具体取决于你的 SQL Server 实例运行的操作系统）来读取这些计数器，尽管许多负载测试工具集成了系统计数器收集和报告机制。与 SQL Server 追踪事件类似，有数百个计数器可供选择——但在对 SQL Server 安装进行初步性能评估时，通常只需要监控其中少数几个。

以下计数器是确定应关注何种性能问题的良好起点：

*   `Processor:% Processor Time`：报告相对于服务器可用容量的总处理器时间。如果此计数器在高峰负载期间高于 70%，可能值得开始调查哪些例程大量使用了 CPU 时间。
*   `PhysicalDisk:Avg. Disk Queue Length`：指示进程是否必须等待使用磁盘资源。当磁盘正在处理请求（即读写数据）时，无法立即满足的请求会被排队。过多的并发请求会导致等待时间，这可能意味着查询性能问题。尽可能确保队列长度保持在 1 以下（这实际上意味着没有队列）是个好主意。
