# 第四章：错误与异常

### 记录用户抛出的异常

一旦定义了异常文本后，使用 `sp_addmessage` 也很容易进行修改。操作时，可传入可选参数 `@Replace`，并将其值设为 `'Replace'`，如下列 T-SQL 所示：

```sql
EXEC sp_addmessage
    @msgnum = 50005,
    @severity = 16,
    @msgtext = 'Problem with ProductId numbers %i, %i, %i',
    @Replace = 'Replace';
GO
```

> **注意：** 除了能够添加消息和设置严重级别外，`sp_addmessage` 还支持为不同语言本地化消息。此处的示例未显示本地化；相反，消息将为用户的默认语言创建。有关本地化消息的详细信息，请参阅 SQL Server 2008 联机丛书。

`RAISERROR` 另一个有用的功能是能够将消息记录到 SQL Server 的错误日志中。这在处理自动化代码时尤其方便，例如通过 SQL Server 代理作业运行的 T-SQL。

为了记录任何异常，需要使用 `RAISERROR` 函数的 `WITH LOG` 选项，如下列 T-SQL 所示：

```sql
RAISERROR('This will be logged.', 16, 1) WITH LOG;
```

请注意，记录错误需要特定的访问权限。执行 `RAISERROR` 函数的用户必须是 `sysadmin` 固定服务器角色的成员，或者具有 `ALTER TRACE` 权限。

### 使用跟踪监控异常事件

一些应用程序开发人员在处理异常时做得太过头，最终创建的应用程序通过捕获发生的每个异常并且不报告来隐藏问题。在这种情况下，如果不了解是否抛出了异常，调试问题可能极其困难。如果您发现自己处于这种情况，可以使用 Profiler 跟踪来监控 SQL Server 中发生的异常。

为了监控异常，需要启动一个跟踪并选择 `Exception` 和 `User Error Message` 事件。对于大多数严重级别大于 10 的异常，这两个事件都会触发。`Exception` 事件将包含与该异常关联的所有数据，但实际的消息除外。这包括错误号、严重级别、状态和行号。`User Error Message` 事件将包含发送给客户端的格式化错误消息。

对于警告（严重级别小于 11 的消息），只会触发 `User Error Message` 事件。

您可能还会注意到错误 208 异常（“找不到对象”）而没有相应的错误消息事件。这些异常是 SQL Server 查询优化器在编译的作用域解析阶段内部使用的，可以安全地忽略。

## 异常处理

理解 SQL Server 何时、为何以及如何抛出异常固然很好，但真正的目标是在异常发生时实际*做一些事情*。**异常处理**指的是在异常发生时能够**捕获**它，而不是简单地让它冒泡到下一级作用域的能力。

### 为什么要在 T-SQL 中处理异常？

T-SQL 中的异常处理应被视为与其他任何语言中的异常处理没有不同。一个普遍接受的编程实践是在*尽可能低的范围*处理异常，以防止它们与更高层的应用程序交互。如果异常可以在较低层次被捕获并在那里处理，较高层的模块将不需要特殊代码来处理该异常，因此可以专注于它们自身的功能。这意味着应用程序中的每个例程都变得更简单、更易于维护，因而很可能也更健壮。

换句话说，异常应该尽可能地*封装*起来——了解其他模块的内部异常是另一种形式的耦合，与本书第一章讨论的某些类型没有太大不同。

请记住，异常的封装确实需要根据具体情况处理。但基本规则是，如果您能够以某种方式“修复”异常而不让调用者知道它曾经发生过，那可能就是一个适合封装的地方。

### 使用 @@ERROR 进行异常“处理”

SQL Server 2005 之前的版本没有真正的异常处理能力。发生的任何异常都会被传递回调用者，无论存储过程或查询的代码采取了何种操作（即使它抛出了异常）。尽管 SQL Server 2008 现在大多提供了更好的替代方案，但在早期版本的 SQL Server 中用于“处理”错误的一般方法在某些情况下仍然有用——而且大量遗留代码还将存在相当长一段时间——因此快速回顾一下绝对有必要。

> **注意：** 如果您正在按顺序尝试本章中的示例，请确保在尝试以下示例之前已关闭 `XACT_ABORT` 设置。

`@@ERROR` 函数非常简单：如果批处理中的最后一条语句没有抛出严重级别为 11 或更高的错误，则返回 `0`。如果最后一条语句确实抛出了错误，则返回错误号。例如，考虑以下 T-SQL：

```sql
SELECT 1/0 AS DivideByZero;
SELECT @@ERROR AS ErrorNumber;
GO
```

第一条语句返回以下消息：

```
Msg 8134, Level 16, State 1, Line 1
Divide by zero error encountered.
```

第二条语句返回一个包含单个值的结果集，该值是与上一个错误相关的错误号：

```
ErrorNumber
-----------
8134
```

通过检查 `@@ERROR` 的值是否非零，可以执行一些非常初级的错误处理。不幸的是，由于 `@@ERROR` 的特性以及它仅作用于批处理中*最后一条*执行的语句，这种方法也相当容易出错。许多刚接触 T-SQL 的开发人员对以下批处理的输出感到非常惊讶：

```sql
SELECT 1/0 AS DivideByZero;
IF @@ERROR <> 0
    SELECT @@ERROR AS ErrorNumber;
GO
```

此代码的第一行产生了与之前相同的错误消息，但在此时，`SELECT @@ERROR` 的结果是：

```
ErrorNumber
-----------
0
```

原因是紧接在 `@@ERROR` 之前执行的语句不是除零错误，而是 `IF @@ERROR <> 0` 这一行，它没有产生错误。解决此问题的方法是在批处理中每个需要错误处理的语句之后，将 `@@ERROR` 的值赋给一个变量，然后检查该变量而不是 `@@ERROR` 本身的值。当然，即使只遗漏了一条语句，策略中也可能留下漏洞，一些错误可能会逃脱注意。

即使存在这些问题，`@@ERROR` 在 SQL Server 2008 中可以说仍然占有一席之地。它是最近添加到 T-SQL 语言中的完整异常处理能力的一个简单、轻量级的替代方案，并且它还有一个额外的好处，即*不*捕获异常。在某些情况下，完全封装并非最佳选择，使用 `@@ERROR` 将允许开发人员采取一些操作（例如，记录异常），同时仍然将其传递回调用者。

### SQL Server 的 TRY/CATCH 语法

包括 T-SQL 在内的许多编程语言中，标准的错误处理结构被称为 *try/catch*。这种结构背后的思想是设置两个代码部分（也称为**块**）。第一部分，即 **try 块**，包含要“尝试”的可能引发异常的代码。第二部分包含在 try 块中的代码失败并发生异常时应执行的代码。这被称为 **catch 块**。一旦 try 块内发生任何异常，代码执行立即跳转到 catch 块。这也被称为**捕获**异常。

在 T-SQL 中，`try/catch` 使用以下基本形式实现：

```sql
BEGIN TRY
    -- 此处放置要尝试的代码
END TRY
BEGIN CATCH
    -- 如果抛出异常，则在此处执行代码
END CATCH
GO
```
