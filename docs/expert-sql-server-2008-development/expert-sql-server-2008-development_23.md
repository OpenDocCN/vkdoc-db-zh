# 第 1 章：面向数据库世界的软件开发方法论

鉴于对数据和数据库的这种依赖性，专注于该领域的开发者别无选择，只能成为既胜任软件开发又精通数据访问与管理的绝对专家。数据是核心的控制性因素，决定着任何应用能为用户带来的价值。没有数据，应用便无存在必要。

本书的主要目的是鼓励微软 SQL Server 开发者更深入地融入主流软件开发。本书强调严谨的测试、周密的架构设计，以及对相互依赖关系的仔细考量。对这些领域的恰当重视是专业软件开发者的标志——而数据库专业人员作为任何软件开发团队的核心成员，绝对不能缺乏这种专长。

在本章中，我将概述适用于数据库应用领域的软件开发与架构相关事项。其中涵盖的一些主题在开发社区中备受争议，我会尝试兼顾双方观点，即便是在阐述我认为最具说服力的论点时也是如此。尽管如此，我仍鼓励你们仔细思考这些问题，而非将我的——或任何其他人的——言论奉为绝对真理。软件架构是一个不断变化的领域。只有通过针对具体情况进行仔细的反思，你才有望为任何特定情境识别并理解“最佳”可能的解决方案。

## 架构再探

软件架构是一个庞大而复杂的主题，部分原因是软件架构师往往喜欢把事情弄得尽可能复杂。事实是，编写一流软件所需的复杂度远没有许多架构师会让你相信的那么多。仅仅通过理解并应用一些基本原则，就能实现极高质量的设计。每个软件开发者为了取得成功必须掌握的三个最重要概念是 `耦合性`、`内聚性` 和 `封装性`：

- `耦合性` 指的是系统中一个模块对同一系统内另一个模块的依赖程度。它也可以指不同系统之间存在的依赖程度。当模块或系统之间的相互依赖程度达到其中一个的改变必然导致另一个也随之改变时，它们就被称为 `紧密耦合`。这显然是不希望出现的，因为它可能在系统的不同模块间形成一个复杂（有时甚至是隐晦）的依赖网络，导致一个模块中看似简单的更改可能需要识别并对整个应用程序中众多不同的模块进行关联更改。软件开发者应努力实现相反的目标：创建 `松散耦合` 的模块和系统，它们可以轻松地被隔离和修改，而不影响系统的其余部分。

- `内聚性` 指的是特定模块或组件为整个应用程序提供单一、定义明确的功能方面的程度。`高内聚` 的模块只具有单一功能，通常被认为比执行多种操作、因而可能可维护性和可重用性较低的 `低内聚` 模块更可取。

- `封装性` 指的是模块的底层实现对系统其余部分隐藏的程度。正如你将看到的，这个概念本质上是松散耦合和高内聚的结合。如果一个模块的方法或属性没有暴露其内部行为的设计决策，则称该逻辑被 `封装` 在该模块内。

不幸的是，这些定性的定义有些难以应用，在实际系统中，判断一个给定模块是否与另一个模块紧密耦合、一个例程是否内聚、或者逻辑是否被恰当封装，涉及大量的主观性。没有客观的方法来衡量应用程序中的这些概念。

通常，开发者会使用比较性术语来讨论这些概念——例如，一个模块在它的接口被 `重构` 之后，可能被认为与另一个模块的耦合性 *较之前更低*。但是，如果没有某种方法来比较其耦合的性质，可能很难绝对地说一个给定模块 *是否* 与另一个模块紧密耦合。让我们看几个例子来澄清一下。

### 什么是重构？

重构是在不添加任何新特性或改变功能的前提下，审查和修订现有代码的实践——本质上，是清理现有代码使其运行得更好。这是管理团队往往比较厌恶的领域之一，因为从销售的角度来看，它没有为应用程序增加有形的价值，并且需要重新审视那些之前被认为是“已完成”的代码部分。

### 耦合性

首先，我们看一个说明基本耦合性的例子。下面这个类可能被定义为模拟一家汽车经销商的库存（为使例子简单，本节中的代码清单将使用一种简化且精简的类似 C# 的语法）：

```
class Dealership
{
    // 经销商名称
    string Name;
    // 经销商地址
    string Address;
    // 经销商拥有的汽车
    Car[] Cars;

    // 定义 Car 子类
    class Car
    {
        // 汽车品牌
        string Make;
        // 汽车型号
        string Model;
    }
}
```

这个类有三个字段：经销商的名称和地址都是字符串，但经销商汽车的集合是基于一个子类 `Car` 来类型的。在一个没有人的世界里，这个类工作得很好——但不幸的是，它的建模方式迫使我们将任何拥有汽车实例的类与经销商紧密耦合。以车主为例：

```
class CarOwner
{
    // 车主姓名
    string name;
    // 车主的汽车
    Dealership.Car[] Cars
}
```

注意，`CarOwner` 的汽车实际上是 `Dealership.Car` 的实例；为了拥有一辆车，似乎预设了必须有一个经销商参与其中。这没有给车主直接出售的汽车——或者就此而言，被盗的汽车——留下任何空间！有多种方法可以修复这种耦合，最简单的是不将 `Car` 定义为子类，而是将其作为一个独立的类。这样做意味着 `CarOwner` 将与 `Car` 耦合，`Dealership` 也会与 `Car` 耦合——但 `CarOwner` 和 `Dealership` 之间将完全没有耦合。这是有道理的，并且更准确地模拟了现实世界。

### 内聚性

为了演示内聚性原则，考虑一个可能在银行应用中定义的以下方法：

```
bool TransferFunds(
    Account AccountFrom,
    Account AccountTo,
    decimal Amount)
{
    if (AccountFrom.Balance >= Amount)
        AccountFrom.Balance -= Amount;
    else
        return(false);

    AccountTo.Balance += Amount;
    return(true);
}
```

请记住，这段代码高度简化，缺乏真实银行应用中必需的基本错误处理和其他特性，请思考这个方法的基本功能是从 `AccountFrom` 账户提取资金并将其存入 `AccountTo` 账户。
