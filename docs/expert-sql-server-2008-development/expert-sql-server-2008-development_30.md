# 面向数据库世界的软件开发方法论

### ORM：一个制造诸多问题的解决方案

数据库中建模继承的一种极其常见的设计是：创建一个单一表，其中包含超类型的所有列，再加上每个子类型的所有列（后者可为空）。这种设计问题重重，应当避免。根本问题在于，构成子类型的属性会混杂在一起，从而导致混淆。例如，仅查看表无法分辨哪些属性属于书籍而非 DVD。唯一的确定方法是查阅文档（如果存在的话）或评估代码。此外，数据完整性几乎丧失殆尽。难以强制规定某些子类型的特定属性必须非空，而在应该为空的属性实际非空时，要弄清楚如何处理则更加困难——例如，NTSC 制式对一本书意味着什么？这是由于代码缺陷导致的错误填充，还是这本书真的拥有播放制式？在一个建模得当的系统中，这个问题本不应存在。

克服关系型系统与面向对象系统之间问题的一个解决方案是转向称为`对象关系映射器（ORMs）`的工具，它们试图自动将对象映射到数据库。

此类工具众多，包括开源的`nHibernate`项目和`微软`自家的`Entity Framework`。这些工具各有其特性和功能，但在大多数情况下基本思路相同：开发者将`ORM`工具"插入"现有的面向对象系统，并告知工具数据库中的哪些列映射到每个类的每个字段。`ORM`工具会审视对象系统以及数据库，以弄清楚如何编写`SQL`将数据以对象形式检索出来，并在数据变更时将其持久化回数据库。这一切都是自动完成的，并且在一定程度上是无缝的。

有些工具更进一步，如果数据库尚不存在，则会为已有对象创建一个数据库。这些工具基于这样的假设运作：在大多数情况下，类与表可以一一对应映射，但正如前文所述，这通常*并非*事实。因此，这些工具最终往往会产生极其糟糕的数据库设计。

我曾合作过的一家公司，在其电子商务应用中使用了一款流行的基于`Java`的`ORM`工具。该工具从对象中心而非表中心的角度来映射"拥有"关系，结果导致数据库中`Products`表包含一个指向`Orders`表的外键。为该公司工作的`Java`开发者被迫在系统中插入虚假订单，以允许公司销售新产品。

尽管`ORM`确实有一些优点，并且脱离任何特定数据库的抽象有助于创建可移植代码，但我认为当前可用的工具集尚不够成熟，不足以使其成为企业软件开发的可行选择。除了那些根据类创建数据库表的工具存在的问题之外，我最关心的两个主要问题都与性能有关：首先，`ORM`工具倾向于从对象而非相关数据的集合（即表）角度思考。每个类都有由`ORM`工具生成的自己的数据访问方法，每次需要数据时，这些方法都会在细粒度级别上查询仅必要的行。这意味着（取决于连接池的处理方式）大量数据库连接会定期打开和关闭，而检索数据的整体接口相当"零碎化"。`SQL` `DBMS`在批量返回数据方面往往比逐行返回高效得多；通常，一次性查询一个产品及其所有相关数据，比先请求产品，再通过单独查询请求相关数据要好得多。

其次，如果过度依赖`ORM`工具，查询调优可能会很困难。在`SQL`数据库中，编写任何给定的查询通常有多种逻辑上等效的方式，每种方式可能具有截然不同的性能特征。当前这代`ORM`工具并未智能地监控并自动修复由编写不当的查询可能引起的问题，使用这些工具的开发者常常因为系统因编写不当的查询而无法扩展而措手不及。

`ORM`工具在过去几年中已经有了显著改进，并且随着时间的推移无疑将继续如此。然而，即使在最新的`微软` `Entity Framework`（`.NET 4.0 Beta 1`）版本中，其生成的`SQL`代码也存在大量缺陷，导致数据库查询充其量是糟糕的，并且常常不是最优的。我认为，任何此类自动生成的`ORM`代码在性能上永远无法与手动精心编写的查询相媲美，通过手工仔细设计对象-数据库接口可以获得更好的投资回报。

## 引入数据库即 API 的思维模式

在编写对象系统与数据库系统之间的数据交换接口时，迄今为止最需要警惕的问题是耦合。对象系统及其用作后端的数据库应仔细分区，以确保在大多数情况下，对一层的更改不会强制要求对另一层进行更改。这在两个领域都很重要；如果数据库的更改需要应用程序更改，重新编译和重新部署应用程序的成本通常很高。

同样，如果应用程序逻辑的更改需要数据库更改，那么更改数据结构或约束可能对其他可能需要相同数据的应用程序产生何种影响，可能很难预知。

为了应对这些问题，数据库开发者必须决心严格遵守一套在数据库系统与对象之间封装的稳固接口。我称之为`数据库即 API`的思维模式。

`应用程序编程接口（API）`是一组允许一个系统与另一个系统交互的接口。`API`旨在成为其所暴露系统的完整访问方法。

在数据库术语中，这意味着`API`将公开用于从数据库检索数据、向数据库插入数据以及在数据库中更新数据的公共接口。

一套数据库接口应遵循与其他接口相同的基本设计规则：已知的、标准化的输入集，产生已知的、标准化的输出集。这套接口应完全封装所有实现细节，包括表和列名、键、索引和查询。使用数据库数据的应用程序不应需要了解内部信息——应用程序只需知道可以使用某些方法检索和持久化数据。

为了定义这样的接口，第一步是为所有外部数据库访问定义存储过程。直接访问表的数据显然违反了正确的封装和接口设计原则，而视图可能够用也可能不够用。在`SQL Server`中，存储过程是唯一能够提供全面数据`API`所需类型接口的构造。
