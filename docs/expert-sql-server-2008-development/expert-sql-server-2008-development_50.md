# 性能测试：从何处开始

一旦设置好了性能计数器和跟踪，你就准备好开始实际的性能测试了。但这引出了一个问题，“从哪里开始？”尤其是在一个大型遗留应用程序中，运行端到端的性能测试可能是一项艰巨的任务。

第一步是确定存在哪些类型的单元测试和功能测试，并评估它们是否可以作为性能测试的起点。一些负载测试工具，例如 Microsoft 的 `Visual Studio Team System 2008`，能够直接对预构建的单元测试进行负载测试。然而，大多数商业负载工具是为直接操作应用程序或 Web 代码而设计的。尝试收集尽可能多的测试，以覆盖应用程序最常用的部分。绝对覆盖固然好，但在许多情况下是不现实的。

下一步是使用预构建的单元测试和功能测试来实现一个负载测试套件。这可能会花费一些时间，具体取决于你使用的负载工具。关键是要确保负载工具向任何返回数据库的函数传递随机或半随机的输入，以模拟真实流量，并确保缓存不会对数字产生过大影响而扭曲结果。非随机输入可能会掩盖由缓冲区缓存回收引起的磁盘 I/O 问题。

## 设置目标

为测试设定目标，以确定你需要测试的负载水平。如果你在一个镜像应用程序生产环境的系统上进行测试，尝试在等于应用程序在高峰时段遇到的负载水平下进行测试。如果测试服务器的性能不如生产系统，则应相应地降低负载水平。请注意，在服务器规模与生产系统不同的情况下进行测试可能很困难。例如，如果生产数据库系统有八个处理器并连接到专用存储区域网络（`SAN`），而测试数据库系统有四个处理器和内部驱动器，则可能会出现 I/O 不匹配。在这种情况下，建议修改测试系统上 `SQL Server` 的处理器亲和性，使可用的处理器功率减少，从而使可用处理器与可用磁盘 I/O 的比率与代码需要运行的实际环境保持一致。

除了确保测试和生产系统规模相似外，还要确保两个系统中的 `SQL Server` 配置相似。例如，确保 `maximum degree of parallelism` 设置相似，以便在测试和生产系统上查询对处理器的使用方式相同。同样，你应该监控 `RAM` 选项，以确保它们在两个系统上配置为等效的百分比——因此，如果你的生产系统有 `16GB` 的 `RAM`，但 `SQL Server` 的 `maximum server memory` 设置是 `12GB`，你也会希望将测试系统设置为使用 `75%` 的可用 `RAM`。

## 执行测试

一旦用户目标设定，负载测试通常应配置为缓慢增加负载，而不是立即用峰值用户数冲击服务器。在测试期间逐渐增加负载可以更准确地模拟典型的生产环境，在这种环境中，服务器可能在维护期间重启，然后在遇到更活跃时段的更多请求之前，缓慢接受用户请求并预热其缓存。请注意，如果你正在测试诸如集群故障转移之类的情况，逐步测试可能不是一个准确的描绘，在那种情况下，服务器可能在启动后立即承受满负荷。

整体测试的目标是查看系统整体扩展情况。尝试查看性能计数器的总体趋势，以确定系统是否能处理负载峰值，或者通常能承受长期持续的负载（再次强调，如果可能，应根据实际的应用程序使用模式）。在此期间捕获的数据可用于以后对特定组件进行更细致的分析。

## 细致分析

如果整体测试的结果显示某些领域需要改进，通常需要对特定例程进行更细致的调查。使用从全系统测试收集的聚合跟踪数据，评估单个查询和作为一组的长时间运行或资源密集型查询是很重要的。

虽然通常很容易只看最糟糕的查询——例如，那些持续时间最长的查询——但这可能无法说明全部情况。例如，你可能会注意到在高峰负载期间某些存储过程比其他过程运行时间更长。这些过程可能导致更长的用户界面等待时间，但持续时间最长的单个存储过程不一定表示单个最长的用户界面等待时间。这是因为许多应用程序在界面需要更新附加数据时会调用多个存储过程。在这些情况下，将同时调用的过程分组并聚合它们的总体资源利用率非常重要。

在聚合一组查询的综合性能影响时，考虑这些过程的调用方式很重要：

*   如果存储过程是按顺序调用的，则应将其各自的持续时间相加，以确定该组的总用户等待时间，并记录组中任何过程的最大资源利用率。
*   另一方面，如果所有过程都是同时并行调用的（例如，在不同的连接上），则应总计资源利用率，以确定该组对系统的总体影响，并记录运行时间最长的单个查询的持续时间。

例如，假设在一个给定的系统中，每当用户登录时，都会调用三个不同的存储过程来获取第一个屏幕的数据。表 3-1 显示了为这些存储过程收集的平均数据。

*表 3-1. 登录后调用的存储过程及其平均数据*

| 存储过程 | 持续时间 (ms) | CPU | 读取 | 写入 |
| :--- | :--- | :--- | :--- | :--- |
| LogSessionStart | 422 | 10 | ... | ... |
| GetSessionData | 224 | 210 | ... | ... |
| GetUserInfo | 305 | 166 | ... | ... |

如果系统按顺序调用这些存储过程，则应为该组记录的总持续时间为 `951` ms (`422 + 224 + 305`)。由于每个都是单独调用的，因此在任何给定时间，对系统的总影响将仅达到给定列中的最大值。因此，我们记录 `CPU` 为 `210`，`Reads` 为 `6408`，`Writes` 为 `1`。

另一方面，如果这些存储过程是同时调用的，影响将会大不相同。总持续时间将仅为三个过程中运行时间最长的那个——`422`ms（当然，假设系统有足够的资源来同时处理所有三个请求）。然而，运行期间的 `CPU` 时间应记录为 `386`，`Reads` 为 `9932`，`Writes` 为 `1`。

通过对存储过程进行分组，可以评估特定功能的总体影响。可能的情况是，单个长时间运行的存储过程并非主要的性能罪魁祸首，而实际上被一组看似资源密集度较低的存储过程所掩盖。对于执行大量非常小的 `fetch` 操作的游标来说，这也可能是一个问题。每个单独的 `fetch` 可能难以察觉，但作为一个整体来看，可能会清楚地看到该游标正在使用大量资源。

这种分组的另一个好处是可以进行进一步的聚合。例如，给定这些数据，可以确定一定数量的用户同时登录对系统的影响有多大。在尝试达到特定的可扩展性目标时，这些信息可能很有用。

## 解决问题：关注显而易见的部分就足够了吗？


当评估系统性能并试图确定从何处着手解决问题时，人们往往倾向于先关注那些最明显的“罪魁祸首”。然而，为了有效利用时间，应保持一定的审慎；在许多情况下，那些看似显而易见的问题，实际上是其他更微妙问题的副作用。

在分析性能问题时，仅看 `Duration` 通常是最容易犯的错误。

`Duration` 讲述了故事的主要部分，但它不一定表明该 `stored procedure`（存储过程）本身存在性能问题。它可能意味着该查询不得不等待其他一个或多个查询完成，或者该查询正在与其他查询竞争资源。进行性能调优时，最好对那些运行时间长但报告的资源利用率非常低的查询保持怀疑。这些往往根本不是真正的元凶。

通过使用细粒度分析技术和聚合方法，通常更容易找到真正的罪魁祸首。例如，在一个交易量相当高的系统中，有一个 `procedure`（过程）会不时地被调用，向磁盘写入 `10MB` 数据。该 `procedure` 报告了较高的 `Duration`，这被解读为可能的性能问题。不幸的是，该 `stored procedure` 本身并没有太多可调优之处，但进一步的聚合分析揭示了系统中另一个 `stored procedure`，它每分钟被调用超过 `1,000` 次，每次调用向磁盘写入多达 `50KB` 的数据。对第二个 `stored procedure` 的每次调用所报告的 `Duration` 都很小，似乎并未引起性能问题，但事实证明它在其他区域引发了问题。通过对其进行调优并减少每次调用写入的数据量，第一个 `stored procedure` 的平均 `Duration` 显著降低了。

## 总结

软件测试是一个复杂的领域，但开发人员有必要充分了解它，以使开发过程更有效。通过在开发过程中实施测试流程，可以用更低的成本交付更健壮的软件。

与应用程序开发人员一样，数据库开发人员必须学会利用单元测试来提高软件项目的成功率。数据库例程无疑将从单元测试中受益，尽管性能测试也极其重要——而且更受管理团队的青睐。

在性能测试期间，务必仔细分析数据。通过基于应用程序使用模式以各种方式重新组合数字，通常可以发现原始数据中并不明显的性能问题。`SQL Server 2008` 提供了多种收集和呈现性能计数器的方法，我将在本书后面的章节中使用这些方法来分析不同方法所采取的相对性能。

测试是区分优秀开发者和真正卓越开发者的重要因素之一。如果你尚未在开发过程中测试你的软件，那么本文介绍的方法论可以帮助你实施一套流程，让你更接近下一个层次。
