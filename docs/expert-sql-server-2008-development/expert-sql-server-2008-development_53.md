# 第四章：错误与异常

这种类型的异常行为也会在`作用域解析`过程中发生。本质上，SQL Server 分两个阶段处理查询。第一阶段解析并验证查询，确保 T-SQL 语法正确。第二阶段是编译阶段，在此阶段构建执行计划并解析查询中引用的对象。如果查询是通过应用程序的即席 SQL 或存储过程内的动态 SQL 提交给 SQL Server 的，这两个阶段会同时发生。

然而，在存储过程的上下文中，SQL Server 采用了`后期绑定`技术。这意味着解析阶段在存储过程创建时发生，而编译阶段（因此也包括作用域解析）仅在存储过程执行时发生。

为了理解其含义，请创建以下存储过程（假设当前数据库中不存在名为 `SomeTable` 的表）：

```sql
CREATE PROCEDURE NonExistentTable
AS
BEGIN
    SELECT xyz
    FROM SomeTable;
END;
GO
```

尽管 `SomeTable` 不存在，该存储过程仍被成功创建——T-SQL 代码解析时没有报错。但是，当执行该存储过程时，会抛出一个异常：

```sql
EXEC NonExistentTable;
GO
```

这将导致：

> 消息 208，级别 16，状态 1，过程 NonExistentTable，第 4 行
> 无效的对象名 'SomeTable'。

与解析异常类似，作用域解析异常在同一作用域内表现类似于批处理级异常，而在外部作用域则类似于语句级异常。由于存储过程创建了一个新的作用域，在该过程内遇到此类异常会中止过程的剩余部分，但在调用批处理中，执行存储过程之后遇到的任何 T-SQL 语句仍会继续运行。

例如：

```sql
EXEC NonExistentTable;
PRINT '这将会打印！';
GO
```

会导致以下结果：

> 消息 208，级别 16，状态 1，过程 NonExistentTable，第 4 行
> 无效的对象名 'SomeTable'。
> 这将会打印！

### 连接和服务器级异常

SQL Server 抛出的某些异常可能非常严重，以至于会终止整个连接，或导致服务器本身崩溃。这些类型的连接级和服务器级异常通常由 SQL Server 内部错误引起，所幸相当罕见。在撰写本文时，我无法提供此类异常的示例，因为我不了解 SQL Server 2008 中存在任何可重现导致此类问题的条件。

## XACT_ABORT 设置

尽管用户对 SQL Server 抛出异常的行为控制有限，但有一个设置可以按连接进行修改。启用 `XACT_ABORT` 设置会使所有语句级异常表现得像批处理级异常。这意味着在查询执行期间，只要 SQL Server 抛出异常（且该异常未被处理），控制权总是会立即返回到客户端。

要为连接启用 `XACT_ABORT`，使用以下 T-SQL：

```sql
SET XACT_ABORT ON;
```

此设置将保持启用状态，直到使用以下 T-SQL 将其禁用：`SET XACT_ABORT OFF;`。即使它是在较低级别的作用域（例如在存储过程或动态 SQL 中）设置的，也会在整个连接期间保持有效。

为了说明此设置对异常行为的影响，让我们回顾一下已经介绍过的几个异常。回想一下，以下整数溢出异常是在语句级运行的：

```sql
SELECT POWER(2, 32);
PRINT '这将会打印！';
GO
```

在运行此 T-SQL 之前启用 `XACT_ABORT` 设置会改变输出结果，导致 `PRINT` 语句不会被执行：

```sql
SET XACT_ABORT ON;
SELECT POWER(2, 32);
PRINT '这将不会打印！';
GO
```

运行此批处理的输出如下：

> 消息 232，级别 16，状态 3，第 2 行
> 针对数据类型 "int" 的算术溢出错误，值 = 4294967296.000000。
