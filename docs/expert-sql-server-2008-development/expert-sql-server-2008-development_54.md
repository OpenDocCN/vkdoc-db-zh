# 第四章：错误与异常

请注意，`XACT_ABORT` 仅影响运行时错误的行为，对编译期间生成的错误无效。回顾之前的示例，该示例演示了使用 `EXEC` 函数在较低作用域中引发的解析异常：

```
EXEC('SELECTxzy FROM SomeTable');
PRINT 'This will print!';
GO
```

无论 `XACT_ABORT` 设置如何，此代码清单的结果都将保持不变，导致即使在异常发生后，`PRINT` 语句仍会被执行。

除了控制异常行为，`XACT_ABORT` 还会修改异常发生时事务的行为方式。有关更多信息，请参阅本章后面的“事务与异常”部分。

## 剖析错误消息

SQL Server 异常有几个不同的组成部分，每个部分都在错误消息文本中有所体现。每个异常都有一个关联的错误号、错误级别和状态。错误消息还可能包含其他诊断信息，包括行号和发生异常的过程名称。

### 错误号

异常的错误号列在错误文本中的 `Msg` 之后。例如，以下异常的错误号为 156：

```
Msg 156, Level 15, State 1, Line 1
Incorrect syntax near the keyword 'FROM'.
```

SQL Server 通常随异常一起返回错误消息，因此从解决问题的角度来看，拥有错误号通常没有额外帮助。然而，在某些情况下，知道错误号会很有用。例如，在使用 `@@ERROR` 函数时，或在使用 `TRY/CATCH` 语法进行专门的错误处理时（有关这些主题的详细信息，请参阅本章后面的“使用 `@@ERROR` 进行异常‘处理’”和“SQL Server 的 `TRY/CATCH` 语法”部分）。

错误号也可用于从 `sys.messages` 目录视图中查找错误消息的本地化翻译。`message_id` 列包含错误号，`language_id` 列可用于获取正确语言的消息。例如，以下 T-SQL 返回错误 208 的英文文本：

```
SELECT text
FROM sys.messages
WHERE
message_id = 208
AND language_id = 1033;
GO
```

此查询的输出是一个错误消息模板，如下所示：

```
Invalid object name '%.*ls'.
```

有关错误消息模板的更多信息，请参阅“SQL Server 的 `RAISERROR` 函数”部分。

### 错误级别

错误消息中的 `Level` 标签表示一个介于 1 到 25 之间的数字。这个数字有时可用于对异常进行分类或确定其严重性。不幸的是，关键词是“有时”：SQL Server 分配的错误级别高度不一致，通常不应将其用于决定如何处理异常。

根据其错误消息，以下异常的错误级别为 15：

```
Msg 156, Level 15, State 1, Line 1
Incorrect syntax near the keyword 'FROM'.
```

每个异常的错误级别可以从 `sys.messages` 视图中查询，使用 `severity` 列。严重性小于 11 表示消息是**警告**。如果严重性大于或等于 11，则该消息被视为**错误**，并可分为以下有文档记录的类别：

*   错误级别 11 到 16 被文档记录为“用户可纠正的错误”。SQL Server 抛出的大多数异常都在此范围内，包括约束违规、解析和编译错误以及大多数其他运行时异常。
*   错误级别 17 到 19 是更严重的异常。这些包括内存不足异常、磁盘空间异常、内部 SQL Server 错误及其他类似违规。其中许多在抛出时会自动记录到 SQL Server 错误日志中。您可以通过检查 `sys.messages` 表中的 `is_event_logged` 列来识别那些被记录的异常。
*   错误级别 20 到 25 是致命的连接和服务器级别异常。



## 第四章：错误与异常

包含各种类型的数据损坏、网络、日志记录及其他关键错误。此层级的几乎所有异常都会被自动记录。

虽然构成每个范围的错误级别在联机丛书（http://msdn2.microsoft.com/en-us/library/ms164086.aspx）中有单独说明，但这些信息在许多情况下不一致或错误。例如，根据文档，严重级别 11 表示“给定对象或实体不存在”的错误。然而，错误 208“对象名无效”是一个级别 16 的异常。许多其他错误也具有同样不可预测的级别，建议不要在客户端软件编程中依赖错误级别来处理逻辑。

除了不同错误的相对严重性存在不一致外，大多数情况下，错误的严重级别与该错误是在语句级还是批处理级行为之间，也没有可辨别的模式。例如，错误 245（“转换失败”）和错误 515（“无法将值 NULL 插入...列不允许 NULL 值”）都是级别 16 的异常。然而，245 是一个批处理级异常，而 515 则在语句级行为。

### 错误状态

每个异常都有一个 `State` 标签，其中包含有关该异常的信息，供 SQL Server 内部使用。SQL Server 为此标签使用的值未被记录，因此该标签通常没有帮助。以下异常的状态为 1：

```sql
Msg 156, Level 15, State 1, Line 1
关键字 'FROM' 附近的语法不正确。
```

### 附加信息

除了错误号、级别和状态外，许多错误还带有附加信息，包括异常发生的行号以及（如果相关）发生异常的过程。以下错误消息表明在过程 `NonExistentTable` 的第 4 行引用了一个无效的对象名：

```sql
Msg 208, Level 16, State 1, Procedure NonExistentTable, Line 4
对象名 'SomeTable' 无效。
```

如果异常不是发生在过程中，行号指的是导致异常的语句所在的批处理中的行号。

注意不要将用 `GO` 分隔的批处理与单个批处理混淆。考虑以下 TSQL：

```sql
SELECT 1;
GO
SELECT 2;
GO
SELECT 1/0;
GO
```

在此情况下，尽管除零异常发生在代码清单本身的第 5 行，但异常消息将报告异常发生在第 1 行：

```sql
(1 行受影响)
(1 行受影响)
Msg 8134, Level 16, State 1, Line 1
遇到除以零的错误。
```

行号重置的原因在于，`GO` 实际上并非 T-SQL 命令。`GO` 是 SQL Server 客户端工具（例如，SQL Server Management Studio 和 `SQLCMD`）识别的标识符，用于告诉客户端将查询拆分为批处理，并依次将每个批处理发送给 SQL Server。前面示例中报告的看似错误的行号，是因为每个批处理被单独发送到查询引擎。SQL Server 不知道在客户端（例如，在 SQL Server Management Studio 中）这些批处理都在屏幕上一起显示。就 SQL Server 而言，这是三个完全独立的 T-SQL 单元，只是碰巧通过同一连接发送。

## SQL Server 的 `RAISERROR` 函数

除了 SQL Server 自身抛出的异常外，用户还可以在 T-SQL 中使用名为 `RAISERROR` 的函数来引发异常。该函数的通用形式如下：

```sql
RAISERROR ( { msg_id | msg_str | @local_variable }
    { ,severity ,state }
    [ ,argument [ ,...n ] ] )
    [ WITH option [ ,...n ] ]
```

第一个参数可以是字符串或变量形式的临时消息，也可以是来自 `sys.messages` 表中 `message_id` 列的有效错误号。如果指定了字符串，它可以包含格式说明符，然后可以使用函数调用末尾指定的可选参数来填充这些说明符。
