# 关于灵活性与接口的核心要义

我希望在此传达的核心信息是，在许多情况下，极致的灵活性与稳固、可维护的接口可能无法并行不悖。如果你的目标是开发真正健壮的软件，你往往会发现必须削减灵活性。但请记住，在大多数情况下，都存在完全合理的变通方案，而无需牺牲原始接口所预期的任何真实灵活性。例如，在这个例子中，可以通过多种方式重写接口以保留所有可能的功能。其中一个版本如下所示：

```sql
CREATE PROCEDURE GetAllEmployeeData
@OrderByName int = 0,
@OrderByNameASC bit = 1,
@OrderBySalary int = 0,
@OrderBySalaryASC bit = 1,
-- 其他列 ...
```

在这个修改后的接口版本中，使用者可以选择用于排序的每个列都有两个相关联的参数：一个参数指定列的排序顺序，第二个参数指定是升序还是降序排序。因此，如果使用者为 `@OrderByName` 参数传递值 2，并为 `@OrderBySalary` 参数传递值 1，结果将首先按工资排序，然后按姓名排序。使用者可以通过操作 `@OrderByNameASC` 和 `@OrderBySalaryASC` 参数来进一步修改排序，为每个列指定排序方向。

这个版本的接口没有暴露关于存储过程内部实现的任何信息。开发者可以自由使用他或她选择的任何技术，以最有效的方式返回正确的结果。此外，使用者无需了解 `Employees` 表的实际列名。包含员工姓名的列可能叫 `Name`，也可能叫 `EmpName`。或者，可能有两列，一列包含名字，一列包含姓氏。由于使用者不需要知道这些名称，它们可以根据数据变化进行必要的修改，并且由于使用者并不耦合于基于例程的列名知识，因此无需对使用者进行任何更改。请注意，同样的推理也可以用来建议，最终用户和应用程序应该只访问作为视图公开的数据，而不是直接访问数据库中的基表。视图可以提供一个抽象层，使得可以对底层表进行更改，同时保持视图的属性不变。

请注意，此示例仅讨论了接口的输入。请记住，输出（例如，结果集）同样重要，这些也应该在契约中记录。我建议始终根据需要使用 `AS` 关键字创建列别名，以便即使底层表发生变化，接口也能继续返回相同的输出。如前所述，我还建议开发者避免根据输入参数返回额外数据，例如额外的列或结果集。这样做可能会产生难以测试和维护的存储过程。
