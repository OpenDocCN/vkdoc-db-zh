# 第三章：测试数据库例程

开发团队应努力建立一套测试套件，至少在每次应用程序发布时运行一次，如果可能则运行更频繁。这些测试应自动化且易于执行。理想情况下，测试套件应是模块化的，这样当开发人员负责应用程序的某一部分时，可以轻松运行仅适用于该部分的测试子集，以验证任何更改。

### 持续测试

一旦建立了一套自动化测试，距离全自动测试环境就仅一步之遥了。这样的环境应能从源代码控制仓库中获取最新代码，运行相应的构建脚本来编译可工作的应用程序版本，并执行整个测试套件。许多软件开发团队使用这种技术每天多次运行测试，如果提交了有问题的代码，几乎能立即发出警报。这种严格的自动化测试被称为**持续集成**，它是将部分测试负担从开发人员手中转移出去、同时确保所有测试按需（甚至更频繁地）运行的绝佳方式。一个帮助在 .NET 环境中建立持续集成的优秀免费工具是 `CruiseControl.NET`，可在 http://sourceforge.net/projects/ccnet 获取。

测试人员还必须特别注意用于执行测试的任何数据。生成包含应用程序可能遇到的所有可能情况的测试数据集通常是有益的。这样的数据集可以保证测试运行之间的一致性，因为它可以恢复到其原始状态。它还可以保证测试到那些原本可能不会遇到的罕见边界情况。

还建议在任何给定测试周期接近尾声时，使用实际生产数据的副本（如果可用）进行测试，而不是依赖人工生成的测试数据。通常，生成的数据集可能缺乏真实性，无法揭示只有真实用户才能从应用程序中引出的隐蔽问题。

#### 为什么测试很重要？

可以说，软件的唯一目的是被最终用户使用，因此测试的唯一目的是确保这些最终用户不会遇到问题。因此，测试希望实现两个重要目标：

*   测试发现需要修复的问题。
*   测试确保没有需要修复的问题。

最终，所有软件都必须经过测试。如果开发人员或质量保证团队没有进行充分测试，应用程序将被试图使用该软件的最终用户测试。不幸的是，这是失去可信度的好方法；用户通常对存在缺陷的软件感到不满。

开发和质量保证团队进行的测试验证了软件。执行的每一种测试都验证了难题的特定部分，如果使用了完整的测试套件（并且测试通过），团队可以相当确定软件具有最少的错误、性能缺陷和其他问题。由于数据库在大多数应用程序中是日益重要的组件，测试数据库是有意义的；如果数据库有问题，它们会传播到应用程序的其他部分。

## 哪种测试重要？

从数据库开发人员的角度来看，在大多数情况下，只有少数几种测试是真正必要的。数据库应针对以下问题进行测试：

*   应验证 `接口一致性`，以保证应用程序具有稳定的数据访问结构。
*   `数据可用性` 和 `授权测试` 类似于接口一致性测试，但更侧重于*谁*可以从数据库获取数据，而不是*如何*检索数据。
*   `身份验证测试` 验证有效用户是否可以登录，以及无效用户是否被拒绝访问。只有当数据库用于验证用户身份时，这类测试才重要。
*   `性能测试` 对于验证用户体验是否积极以及用户是否不必为数据等待超过必要时间至关重要。性能测试可能涉及 `负载测试`（监控数据库在给定负载下的性能）、`饱和测试`（通过不断增加负载和/或移除资源直到系统崩溃来试图使系统不堪重负）以及 `耐力测试`（在持续时间内对数据库施加持续需求）。
*   `回归测试` 涵盖了所有其他类型的测试，但通常侧重于发现先前已修复的问题。回归测试是验证修复是否仍然有效的测试。

#### 需要多少测试？

尽管大多数开发团队缺乏足够数量的测试来彻底测试应用程序，但在某些情况下，情况恰恰相反。测试过多可能与测试不足一样是个问题；编写测试可能很耗时，并且每当功能发生变化时，测试必须与软件的其余部分一起维护。在彻底测试的需求与时间和金钱限制的现实之间取得平衡非常重要。

数据库测试的一个良好起点是为每个接口参数“类”或输入组创建一个单元测试。例如，考虑以下存储过程接口：

```sql
CREATE PROCEDURE SearchProducts
    SearchText varchar(100) = NULL,
    PriceLessThan decimal = NULL,
    ProductCategory int = NULL
```

此存储过程根据三个参数返回有关产品的数据，每个参数都是可选的，基于以下（已记录的）规则：

*   用户可以搜索产品描述中的文本。
*   用户可以搜索价格低于给定输入价格的产品。
*   用户可以将文本搜索或价格搜索与特定产品类别的额外过滤器结合使用，以便仅返回该类别的结果。
*   用户不能同时搜索文本和价格。此条件应返回错误。
*   任何其他输入组合都应导致错误。

为了验证存储过程的接口，需要为上述每个条件编写一个单元测试。传递有效输入参数的单元测试应验证存储过程根据其隐式契约返回有效的输出结果集。针对无效参数组合的单元测试应验证在使用这些组合时会发生错误。已知错误是接口隐式契约的一部分（有关此主题的更多信息，请参见第 4 章）。

除了这些单元测试之外，还应为存储过程内已修复的每个已知问题生成一个额外的回归测试，以确保该过程的功能不会随时间推移而退化。

尽管这看起来像是大量的测试，但请记住，这些测试可以——并且应该——共享相同的基础代码。各个测试只需将正确的参数传递给参数化的基础测试即可。

## 管理层会支持吗？

一个不幸的事实是，许多管理团队认为测试要么是不必要的时间浪费，要么根本不应该是软件开发过程中一个整合良好的部分。许多软件公司，尤其是规模较小的公司，没有专门的质量保证人员，而且开发进度如此紧张，以至于几乎没有进行测试，这使得完整的功能测试几乎不可能。

我曾合作过的几家公司都处于这种情况，而这从未带来管理层所认为的时间或金钱节省。相反，缺乏测试实际上*浪费*了时间和金钱。
