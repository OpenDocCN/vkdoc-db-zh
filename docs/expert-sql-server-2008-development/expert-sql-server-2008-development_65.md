# 所有权链与权限提升

如果以上任一条件不成立，所有权链就会中断，Hugo 就必须通过其他方式获得授权才能从表中查询数据。如果执行上下文在存储过程内部发生变化，所有权链也会失效。例如，所有权链不适用于动态 SQL（有关动态 SQL 的更多信息，请参见第 8 章）。

## 跨数据库所有权链

当一个数据库中的存储过程请求访问另一个数据库中的对象时，虽然有可能维持所有权链，但这会变得相当复杂，安全性也更难维护。要设置 `cross-database ownership chaining`，拥有存储过程和所引用表的用户必须与一个服务器级别的登录名关联，并且每个数据库都必须使用 `ALTER DATABASE` 命令将 `DB_CHAINING` 属性设置为开启。该属性告知 SQL Server，两个数据库都可以参与跨数据库所有权链，无论是作为源还是目标——但无法控制链的方向，因此启用此选项可能会无意中打开安全漏洞。

我建议尽可能避免使用 `cross-database ownership chaining`，而是调用远程数据库中的存储过程。这样做将产生一个更安全、更灵活的解决方案。例如，如果数据库之间不依赖彼此进行身份验证，那么将数据库移动到不同的服务器上会容易得多。此外，通过适当使用架构，将对象拆分到多个数据库中已不像过去那样重要。如有可能，请考虑完全避免使用多个数据库。

## 无所有权链的权限提升

如果对象所有者与模块所有者不匹配，或者使用了动态 SQL，所有权链将不起作用。在这些情况下，你必须使用 SQL Server 提供的另外两种权限提升方法之一：使用 `EXECUTE AS` 子句扩展存储过程，或使用证书进行模块签名。

在存储过程中使用 `EXECUTE AS` 子句是一种简单有效的权限提升方法，但其灵活性远不及使用证书所能达到的程度。使用证书时，权限是累加的而非模拟的——证书提供的额外权限是扩展而非替换调用主体的权限。在以下章节中，我将更详细地讨论这些替代的权限提升方法。

### 存储过程与 EXECUTE AS

如本章前一节所述，`EXECUTE AS` 命令可以单独在 T-SQL 批处理中使用，以临时模拟其他用户。但是，`EXECUTE AS` 也可用于存储过程、函数和触发器。本节示例仅关注存储过程，但相同原理也适用于其他对象类型。

要使用 `EXECUTE AS` 更改整个存储过程的模拟上下文，请将其添加到 `CREATE PROCEDURE` 语句中，如下例所示：

```
CREATE PROCEDURE SelectSensitiveDataByImpersonation
WITH EXECUTE AS 'Louis'
AS
BEGIN
    SET NOCOUNT ON;
    SELECT *
    FROM dbo.SensitiveData;
END;
GO
```

当此存储过程由用户执行时，过程内的所有操作都将被视为是由 `Louis` 用户运行，而不是由调用用户运行（这是默认行为）。这包括任何动态 SQL 操作，或对 `Louis` 用户有权访问的表中的数据进行操作。存储过程执行完成后，上下文将自动还原回调用者的上下文。

请记住，使用 `EXECUTE AS` 不会中断所有权链，而是可以用来扩展它们并创建额外的灵活性。例如，考虑以下两个用户和相关表：

```
CREATE USER Kevin WITHOUT LOGIN;
GO
CREATE TABLE KevinsData ( SomeData int );
GO
```
