# 第 5 章：特权与授权

在本章的示例中，我假设用户使用其个人凭证连接到数据库的场景。

## 最小权限原则

在任何系统（无论是数据库还是其他类型）中锁定资源的关键，其本质非常简单：任何给定用户只应访问其所需的最少资源集合，并且仅在访问这些资源所需的时间内拥有访问权限。遗憾的是，在实践中，这更像是一个理想目标，而非数据安全的实际处方；许多系统不允许为用户分配的权限集被轻松地动态提升，而且从历史上看，微软 Windows 系列操作系统并未被设计为利用权限提升作为在运行时获取额外访问权限的手段。

许多多用户操作系统实现了当需要访问某用户拥有的资源时，能够`模拟`该用户的能力。模拟与重新认证略有不同；它不是注销并重新发送凭证（从而导致任何正在运行的进程被停止），而是允许一个进程临时`提升`其权限，承担被模拟主体所拥有的权利。在操作系统层面最常见的例子是 UNIX 的 `su` 命令，它允许用户临时采用另一个用户的身份，并在完成后轻松恢复。Windows 系统也能处理一定程度的模拟，例如由 `.NET` 的 `WindowsIdentity` 类提供的功能。

Windows 系统中的权限通常使用`访问控制列表（ACLs）`来提供。授予对资源的权限意味着将用户添加到列表中，之后该用户可以一次又一次地访问该资源，即使是在系统注销和登录之后。如果攻击者接管了系统中的一个账户，这种访问控制就不会提供额外的安全性。通过控制一个账户，攻击者自动拥有该账户有权访问的每个资源的完全访问权限。

通过使用模拟来控制访问，用户被要求在*每次需要访问时*动态地请求对资源的访问。此外，对资源的权利仅在模拟过程中保持。一旦用户`恢复`（即关闭模拟），额外的访问权限就不再被授予。实际上，这意味着如果一个账户被攻陷，攻击者还必须攻陷模拟上下文才能访问更安全的资源。

通过`最小权限`实现安全的理念涉及创建具有很少或没有权限的用户，并允许他们在需要更多访问权限时短暂提升其权限。这通常通过`代理`实现——这些是拥有资源访问权限但无法从外部进行身份验证的用户（或其他安全主体）。使用低权限的外部用户与高权限的代理用户相结合，提供了抵御攻击的缓冲，因为攻击者从外部可以直接攻陷的账户，本身没有任何直接关联的权限。访问更有价值的资源需要攻击者付出额外的努力，这为您提供了更多的机会在问题发生之前检测到它们。

## 在 SQL Server 中创建代理

SQL Server 2008 允许在服务器级别和数据库级别创建安全主体，这些主体可以通过代理使用。

- 在服务器级别，可以创建无法登录的代理`登录名`。
- 在数据库级别，可以创建与登录名无关的代理`用户`。

切换到这两种代理主体的执行上下文的唯一方法是通过模拟，这使得它们非常适合权限提升场景。

### 服务器级别代理



为了创建代理登录（可用于委派服务器级别的权限，如`BULK INSERT`或`ALTER DATABASE`），你必须首先在`master`数据库中创建一个证书。证书的更多细节将在第 6 章介绍，但目前你可以将证书视为一种无需密码即可验证主体身份的可信方式。以下语法可用于在`master`中创建证书。（请注意，在任何数据库中创建证书之前，都必须先创建一个主密钥。同样，请参阅第 6 章。）

```sql
USE master;
GO

CREATE CERTIFICATE Dinesh_Certificate
ENCRYPTION BY PASSWORD = 'stR0n_G paSSWoRdS, pLE@sE!'
WITH SUBJECT = 'Certificate for Dinesh';
GO
```

证书创建完成后，可以使用`CREATE LOGIN FROM CERTIFICATE`语法创建代理登录，如下所示：

```sql
CREATE LOGIN Dinesh
FROM CERTIFICATE Dinesh_Certificate;
GO
```

此登录可以像任何其他登录一样被授予权限。但是，要使用这些权限，登录必须映射到一个数据库用户。这是通过使用创建登录时相同的证书，使用`CREATE USER FOR CERTIFICATE`语法来创建用户实现的。有关如何使用代理登录获取服务器级别权限的更多信息，请参阅本章后面的“使用证书进行存储过程签名”部分。

### 数据库级别代理

可以通过向数据库添加一个未与服务器登录关联的用户来创建在数据库级别运行的代理主体。这使用`CREATE USER WITHOUT LOGIN`完成，如以下代码所示：

```sql
CREATE USER Bob
WITHOUT LOGIN;
GO
```

此用户与任何数据库用户一样，可以被分配所有权和其他权限。但是，无法登录服务器并以`Bob`身份进行身份验证。相反，你必须使用有效的服务器级别登录进行登录，并使用与你的登录关联的任何数据库用户对数据库进行身份验证。只有这样，你才能模拟`Bob`，承担该用户被分配的任何权限。本章后面的“使用`EXECUTE AS`进行基本模拟”一节将详细讨论这一点。

## 分层数据安全：洋葱模型

一般来说，攻击者为了访问有价值的资源必须穿透的层级越多，阻止其攻击的机会就越大。开发者应努力为任何敏感数据构建多层保护，以确保如果一个安全措施被攻破，其他障碍仍能阻止攻击者。

第一道防线是数据库服务器之外的所有内容，这些都属于身份验证范畴。一旦用户通过身份验证，SQL Server 的声明式权限系统就会启动，登录根据用户映射被授权访问一个或多个数据库。

从那里，每个用户被授权访问数据库中的特定资源。可以在此处添加另一层以增强安全性，即使用存储过程。通过仅通过存储过程分配权限，可以在权限提升发生的时间和原因上保持更大的控制权——但本章稍后将对此进行更多介绍。

当然，存储过程本身必须能够访问所需的任何表和列，如果需要，可以使用加密或行级安全方案进一步锁定这些资源。

图 5-1 展示了在定义 SQL Server 安全方案时应考虑的一些层级，以最大化敏感数据的保护。本章的剩余部分主要讨论在用户通过身份验证后，如何使用存储过程作为访问数据的主要层级来最好地控制对资源的访问。

存储过程层在数据访问方法和数据本身之间提供了一个理想的抽象层，允许通过参数或其他内联逻辑编程实现额外的安全性。
