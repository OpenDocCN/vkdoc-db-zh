### 不要走捷径

如果我们认为捷径可以让我们避免做一些觉得不必要的工作，那么想走捷径是人的天性。在编程方面，常常存在一些捷径，它们提供了一种方便、简洁的方法，用比其他更标准的方法更少的代码行数来完成特定任务。然而，这些快捷方法可能伴随着相关风险。最常见的是，快捷方法需要更少的代码，是因为它们依赖于某些假定的默认值，而不是过程中明确声明的值。因此，它们只能应用于那些默认值所施加的条件成立的情况。

通过依赖默认值，快捷方法可能会增加代码的僵化性，并引入外部依赖——默认值可能因服务器配置而异，或在不同版本的 `SQL Server` 之间发生变化。因此，走捷径会降低代码的可移植性，并引入一些未来可能失效的假设。

为了说明这一点，考虑一下当你在不显式声明合适数据长度的情况下将值 `CAST`（转换）为 `varchar` 数据类型时会发生什么：

```sql
SELECT CAST ('This example seems to work ok' AS varchar);
GO
```

该查询似乎正确运行，并产生以下输出：

```
This example seems to work ok
```

在一些开发者中，似乎存在一个常见的误解，即省略作为 `CAST` 操作目标的 `varchar` 类型的长度，会导致 `SQL Server` 动态分配一个足以容纳输入字符串所有字符的长度。然而，事实并非如此，如下面的代码清单所示：

```sql
SELECT CAST ('This demonstrates the problem of relying on default datatype length'
AS varchar);
GO
```

```
This demonstrates the problem
```

如果没有显式指定，在 `CAST` 为字符数据类型时，`SQL Server` 默认长度为 30 个字符。在第二个示例中，输入字符串被静默截断为 30 个字符，即使代码中没有明显的迹象表明这一点。如果这是有意为之，那么明确声明为 `varchar(30)` 以突出显示这是计划内的截断，会比简单地省略数据长度清晰得多。

有时会走的另一个捷径的例子是依赖数据类型之间的隐式 `CAST`。考虑下面的代码清单：

```sql
DECLARE
    @x int = 5,
    @y int = 9,
    @Rate decimal(18,2);

SET @Rate = 1.9 * @x / @y;
SELECT 1000 * @Rate;
GO
```

在此示例中，`@Rate` 是一个乘数因子，其值由两个参数 `@x` 和 `@y` 的比率乘以硬编码的比例因子 1.9 来确定。当应用于值 1000 时，如本例所示，结果如下：

现在假设管理层决定更改用于确定 `@Rate` 的计算方法，并将比例因子从 1.9 增加到 2。一个明显（但不正确）的解决方案是按如下方式修改代码：

```sql
DECLARE
    @x int = 5,
    @y int = 9,
    @Rate decimal(18,2);

SET @Rate = 2 * @x / @y;
SELECT 1000 * @Rate;
GO
```

这个变更并未按预期提高比率，实际上反而抵消了将任何比率应用于提供的值 1000 的效果。现在的问题是，用于确定 `@Rate` 的求和是一个纯粹的整数计算，即 `2 * 5 / 9`。在整数运算中，这等于 1。在前面的例子中，硬编码的值 1.9 导致 `@x` 和 `@y` 参数都隐式转换为 decimal 类型，因此求和是以十进制精度计算的。

单独来看，这个例子可能显得微不足道，但当深嵌在某些复杂代码的核心部分时，它可能成为意外行为和不必要的漏洞追踪的来源。为了避免这些复杂情况，最好始终明确声明计算中使用的任何参数的类型和精度，并避免它们之间的隐式 `CAST`。

使用快捷方式的另一个问题是它们可能会模糊开发者编写代码的意图。如果我们无法理解一行代码的目的是什么，就很难测试它是否达到了预期目的。考虑下面的代码清单：

```sql
DECLARE @Date datetime = '03/05/1979';
SELECT @Date + 365;
```

乍一看，这似乎相当无害：取一个特定日期并加上 365。但实际上这里使用了几个快捷方式，给这段代码的预期目的带来了歧义：

第一个快捷方式是从字符串值 `'03/05/1979'` 到 `datetime` 的隐式 `CAST`。我相信你知道，世界各地有多种呈现日期格式的方式，`03/05/1979` 是有歧义的。在英国，它表示 5 月 3 日，但对于美国读者来说，它表示 3 月 5 日。隐式转换的结果将取决于执行该函数的服务器的区域设置。

即使解决了 `dd/mm/yyyy` 或 `mm/dd/yyyy` 的顺序问题，输入值仍然存在歧义。选择的数据类型是 `datetime`，它存储日期和时间两部分，但赋给 `@Date` 的值没有指定时间，因此这段代码依赖于 `SQL Server` 的默认值午夜：`00:00:00`。然而，也许开发者的意图不是指定一个时间点，而是整个日历日。如果是这样，是否应该使用 `date` 数据类型来指定原始的 `@Date` 参数？还有 `SELECT` 查询的结果——它是否也应该是 `date` 类型？

最后，代码指定了整数 `365` 与 `datetime` 值的加法。当应用于日期值时，`+` 运算符会增加给定的天数，因此这似乎是使用 `DATEADD` 方法来增加 365 天的一个快捷方式。但是，


是*this*一个添加 1 年的捷径？如果是的话，这是另一个依赖于假设的捷径示例——在这种情况下，假设是该年有 365 天。

这些因素的结合意味着，这行简单代码的真实意图尚不明确：

```sql
SELECT DATEADD(DAY, 365, '1979-03-05');
```

这会导致以下结果：

```
1980-03-04 00:00:00.000
```

还是说，这段代码是以下内容的简写：

```sql
SELECT CAST(DATEADD(YEAR, 1, '1979-05-03') AS date);
```

这会导致一个相当不同的输出：

```
1980-05-03
```

> **注意**：关于时序数据相关问题的进一步讨论，请参阅第 11 章。

或许最广为人知的捷径方法示例是使用`SELECT *`来从表中检索所有列的数据，而不是按名称列出各个列。正如本章的第一个例子所示，这里的风险在于，未来对表结构的任何更改都将导致此查询返回的结果集结构`静默地`发生变化。往好了说，这可能导致检索到的数据列从未被使用，从而导致效率低下。往坏了说，这可能导致非常严重的错误（试想一下，如果结果中的数据列被发送到另一个按索引位置而非列名引用它们的函数时会发生什么，或者两个集合的列数和类型不匹配导致任何`UNION`查询失败的可能性）。还有很多其他原因应避免使用`SELECT *`，例如添加不必要的行到查询中会排除使用覆盖索引的可能性，这可能导致查询性能大幅下降。

### 测试

防御性实践非常强调在整个开发过程中测试和代码审查的重要性。为了防御可能在实际生产环境中发生的情况，应用程序应在它将在现实世界中经历的相同条件下进行测试。事实上，防御性编程建议应在极端条件下进行测试（**压力测试**）——如果你能构建一个健壮、高性能的应用程序来应对严峻压力，那么你就可以更确定它能应对正常需求。除了性能测试，还有功能测试和单元测试需要考虑，它们确保应用程序的每个部分都按照其契约按预期行为，并执行正确的功能。这些测试将在下一章中更详细地讨论。

在测试应用程序时，考虑测试所基于的样本数据非常重要。你不应该人为地清洗将用于测试代码的数据，或依赖人为生成的数据。如果应用程序需要在生产数据上运行，那么就应该在具有代表性的数据上进行测试，包括其缺陷。这样做将确保应用程序能够应对所有应用程序中典型的不完美数据——缺失或不完整的值、格式错误的字符串、`NULL`等等。随机抽样方法可用于确保测试数据代表整体数据集的公平样本，但防御性测试确保应用程序针对极端边缘情况进行测试也很重要，因为正是这些异常情况可能导致异常。

即使创建测试数据是为了确保对真实世界数据的统计学公平代表，并且经过精心选择以包含边缘情况，但仅在相对少量的测试数据上进行测试时，应用程序能保证多大程度的防御性，仍然存在固有问题。某些特殊情况只会在全规模环境中出现。性能影响是一个明显的例子：如果你只基于几千行数据进行性能测试，那么当应用程序在实际环境中面对数百万行数据而无法运行时，请不要惊讶（你会惊讶于我曾见过多少次应用程序仅基于在数据规模急剧缩减的情况下进行的性能测试就签署通过了）。你也不应简单地假设应用程序的性能会随着涉及的数据行数而可预测地扩展。通过精心的查询设计和良好调整的索引，一些应用程序可能在大型数据集上扩展良好。然而，其他应用程序的性能可能会呈指数级下降（例如在使用表之间的`CROSS JOIN`创建笛卡尔积时）。防御性测试不仅应考虑应用程序当前预期使用的数据量，还应考虑预期的未来增长因素。

测试时的另一个考虑因素是多个用户对系统的影响。有许多功能在单独测试时，被证明能以一致的、可重复的方式通过。然而，在存在并发性的情况下——即对数据库同一资源的多个请求——这些相同的测试可能会失败。为了演示这一点，以下代码清单创建了一个简单的表，包含两个整数列`x`和`y`，以及一个`rowversion`列`v`。

```sql
CREATE TABLE XandY (
    x int,
    y int,
    v rowversion);

INSERT INTO XandY (x, y) VALUES (0, 0);
GO
```

以下代码执行一个循环，从`XandY`表中读取当前值，将`x`的值增加 1，然后将新值写回表中。循环设置为运行 100,000 次，并且仅当`rowversion`列`v`自上次读取以来未更改时，循环计数器才会增加。

```sql
SET NOCOUNT ON;
DECLARE
    @x int,
    @y int,
    @v rowversion,
    @success int = 0;

WHILE @success < 100000
BEGIN
    -- Retrieve existing values
    SELECT
        @x = x,
        @y = y,
        @v = v
    FROM XandY

    -- Increase x by 1
    SET @x = @x + 1;

    SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
    BEGIN TRANSACTION
    IF EXISTS(SELECT 1 FROM XandY WHERE v = @v)
    BEGIN
        UPDATE XandY
        SET
            x = @x,
            y = @y
        WHERE v = @v;

        SET @success = @success + 1;
    END
    COMMIT;
END
GO
```

执行此代码后，正如您所预期的那样，`x`列的值增加到 100,000：

```
x         y         v
100000    0         0x00000000001EA0B9
```

现在，让我们尝试在并发情况下运行相同的查询。首先，让我们将表重置为初始值，如下所示：

```sql
UPDATE XandY SET x = 0;
GO
```

现在，在 SQL Server Management Studio 中打开一个新查询并输入以下代码：

```sql
SET NOCOUNT ON;
DECLARE
    @x int,
    @y int,
    @v rowversion,
    @success int = 0;

WHILE @success < 100000
BEGIN
    -- Retrieve existing values
    SELECT
        @x = x,
        @y = y,
        @v = v
    FROM XandY

    -- Increase y by 1
    SET @y = @y + 1;

    SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
    BEGIN TRANSACTION
    IF EXISTS(SELECT 1 FROM XandY WHERE v = @v)
    BEGIN
        UPDATE XandY
        SET
            x = @x,
            y = @y
        WHERE v = @v;

        SET @success = @success + 1;
    END
    COMMIT;
END
GO
```

这第二个查询在除了一点外的所有方面都与第一个查询相同：它不是将`@x`的值增加 1，而是将`@y`的值增加 1。然后，像之前一样，将两个值都写回表中。



因此，如果我们运行这两个查询，我们期望 `x` 和 `y` 的值都会是 100,000，对吗？为了找出答案，我们先执行第一个查询，它会更新 `x` 的值。在它仍在执行时，执行第二个脚本，它会更新 `y` 的值。几分钟后，当两个查询都完成后，在我的笔记本电脑上检查 `XandY` 表的内容，得到如下结果：

```
x y v

99899 99019 0x000000000021ACCC
```

尽管这些查询显然通过（在提交更新前测试 `@rowversion` 的值是否保持不变）包含了一定程度的并发处理能力，但在存在其他并发查询的环境中进行测试时，它们未能按预期方式运行。关于为何会发生这种情况的解释，以及处理此类情况的方法，将在第 9 章中阐述。

### 代码审查

测试通常是一个自动化过程，而代码审查则是由人主导的活动，涉及同行小组手动审查应用程序背后的代码。自动化测试和人工代码审查这两种活动是相辅相成的，能够发现代码改进的不同方面。自动化测试套件可以非常轻松地检查例程在给定数量的测试场景中是否产生了正确的输出，但要让它们明确断言一个例程是以最健壮或最高效的方式编码的、应用了正确的逻辑、或编码标准遵循了最佳实践，则非常困难。在这些情况下，代码审查是更有效的方法。

请考虑以下代码清单，它展示了一个用于测试给定电子邮件地址是否有效的 T-SQL 函数：

```sql
DECLARE @email_address varchar(255);

IF (

    CHARINDEX(' ',LTRIM(RTRIM(@email_address))) = 0

    AND LEFT(LTRIM(@email_address),1) <> '@'

    AND RIGHT(RTRIM(@email_address),1) <> '.'

    AND CHARINDEX('.',@email_address ,CHARINDEX('@',@email_address)) -

        CHARINDEX('@',@email_address ) > 1

    AND LEN(LTRIM(RTRIM(@email_address ))) -

        LEN(REPLACE(LTRIM(RTRIM(@email_address)),'@','')) = 1

    AND CHARINDEX('.',REVERSE(LTRIM(RTRIM(@email_address)))) >= 3

    AND (CHARINDEX('.@',@email_address ) = 0 AND CHARINDEX('..',@email_address ) = 0)

)

PRINT 'The supplied email address is valid';

ELSE

PRINT 'The supplied email address is not valid';
```

这段代码很可能通过功能测试，表明根据提供的一组测试电子邮件地址，该函数能正确识别所提供的电子邮件地址格式是否有效。

然而，在代码审查过程中，有经验的开发人员查看这段代码后可能会指出，使用 .NET 基础类库提供的正则表达式方法，例如下面这样，可以更好地实现为一个用户定义函数：

```sql
SELECT dbo.RegExMatch('\b[A-Z0-9._%+-]+@[A-Z0-9.-]+\.[A-Z]{2,4}\b', @email_address);
```

请注意，此示例假设您已注册了一个名为 `RegExMatch` 的函数，该函数实现了 .NET `System.Text.RegularExpressions.Regex` 类的 `Match` 方法。虽然两种方法都能达到相同的最终结果，但以这种方式重写代码创建了一个更高效、更易于维护的例程，并且还促进了可重用性，因为建议的 `RegExMatch` 函数可用于匹配其他场景中的正则表达式模式，例如检查电话号码是否有效。

具有挑战性且开放的代码审查对提高软件代码质量有显著效果，但它可能是一项成本高昂的工作，而且并非所有情况都需要对整个应用程序进行彻底的代码审查所需付出的努力。封装良好的代码的优势之一在于，可以从应用程序的其余部分中分离出最有可能从该活动中受益的模块，并进行单独审查。

### 验证所有输入

防御性编程建议你永远不要信任任何外部输入——不要对其类型（例如字母或数字）、长度、内容甚至其存在做出假设！这些规则不仅适用于从应用程序 UI 或网页发送的用户输入，也适用于应用程序所依赖的任何外部文件或网络资源。

良好的防御姿态是假设所有输入都是无效的并可能导致异常情况，除非被证明是安全的。有多种技术可用于确保输入是有效且安全的：

*   数据可以被“调整”。例如，不良字符可以被替换或转义。然而，准确识别需要处理的数据以及知道处理它的最佳方式存在一些困难。静默修改输入会影响数据完整性，通常不推荐，除非无法避免。
*   数据可以对照潜在危险输入的“黑名单”进行检查，如果发现包含已知的不良项目，则拒绝。例如，输入不应允许包含 SQL 关键字，如 `DELETE` 或 `DROP`，或包含非字母数字字符。
*   仅当输入完全由允许内容的“白名单”指定的内容组成时，才接受输入。从 UI 的角度来看，你可以认为这等同于允许用户仅从预定义的下拉列表中选择值，而不是在自由文本框中输入。可以说这是最安全的方法，但也是最僵化的方法，并且限制性太强，无法在许多实际应用中使用。

所有这些方法都容易出现缺陷。例如，假设你使用 `ISNUMERIC()` 函数来测试用户输入是否仅包含数值。你可能期望以下代码会拒绝输入：

```sql
DECLARE @Input varchar(32) = '10E2';

SELECT ISNUMERIC(@Input);
```

大多数异常情况是由于未预见到的但本质上是良性的环境导致的。然而，在处理用户输入时，你应始终意识到可能存在蓄意、恶意的攻击，这些攻击旨在利用未得到彻底防御的系统中暴露出的任何弱点。或许最广为人知的防御性编程技术涉及防止 SQL 注入攻击。也就是说，当用户故意尝试将恶意代码作为提供给应用程序的用户输入的一部分插入并执行时。

SQL 注入攻击通常利用实现不当的函数，这些函数基于未经验证的用户输入构建并执行动态 SQL。考虑以下示例：

```sql
CREATE PROCEDURE Hackable
    @Input varchar(32)
AS BEGIN
    DECLARE @sql varchar(256) = 'SELECT status FROM sys.sysusers WHERE name = ''' + @Input + '''';
    EXECUTE(@sql);
END
```

这段代码的预期目的相当直接——它返回参数 `@Input` 中提供的用户的状态。因此，可以按如下方式使用它来查找用户 John 的状态：

```sql
EXEC Hackable 'John';
GO
```

但是，如果用户输入的不是值 `John`，而是输入了 `'public'' or 1=1 --'`，如下所示？

```sql
EXEC Hackable @Input='public'' or 1=1 --';
GO
```

这将导致生成如下 SQL 语句：

```sql
SELECT status FROM sys.sysusers WHERE name = 'public' OR 1 = 1;
```

附加到查询末尾的条件 `OR 1 = 1` 将始终评估为真，因此效果将是使查询列出 `sys.sysusers` 表中的每一行。

尽管这是一个简单且众所周知的弱点，但它仍然惊人地普遍。防范此类明显的安全漏洞很容易实现，第 6 章将讨论各种相关技术。
