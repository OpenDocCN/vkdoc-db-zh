# 加密

加密是将数据编码的过程，其方式使得除了拥有适当机密知识（**密钥**）以再次解密数据的人之外，任何人都无法使用。SQL Server 2008 提供的加密功能是强大的特性，您应始终考虑将其作为整体安全策略的一部分。如果所有其他安全机制都失败，加密可以被证明是保护机密数据的非常有效的最后一道防线。

然而，不应轻率地考虑加密；这是一个复杂的主题，成功实施加密几乎总是需要在应用程序的架构和设计阶段进行一定程度的考虑。加密数据的物理行为相对简单，但如果没有适当的程序来安全管理对数据的访问以及解密它所需的密钥，任何实施加密的尝试都是徒劳的。未能正确实施加密实际上可能弊大于利，因为它会给您带来数据安全的错误感觉，而实际上数据正暴露给潜在的黑客。

此外，加密提供的额外保护伴随着相关的性能成本。

通常，数据库必须做更多的工作来加密写入的数据，并在需要该数据时再次解密。这意味着，特别是在大规模环境中，必须仔细关注以确保加密不会对应用程序性能产生不利影响。


在本章中，我的目的不是提供关于数据库加密的权威指南。相反，我将提供一个实用指南，介绍如何使用 SQL Server 2008 的主要加密功能，来创建一个安全、可扩展且能够处理机密数据的数据库应用程序。显然，这样的解决方案前提是其他安全方面，包括用户访问控制和网络保护，已经得到了充分的解决。我将特别关注前文段落中指出的两个主要问题——即，如何安全地存储和管理加密密钥的访问权限，以及如何设计一个既能保护敏感数据，又能最小化对性能负面影响的数据库应用程序架构。

## 你真的需要加密吗？

在着手解决如何为特定数据集实施加密之前，有必要先考虑是否真的需要加密。加密是一种保护数据的方法，因此关键问题是：你试图保护*什么*数据，以及试图防止*谁*（或什么）获取它。

### 什么应该被保护？

所有数据都很重要。如果它不重要，我们一开始就不会费心将它存储在数据库中。但这并不一定意味着所有数据都需要加密。

SQL Server 2008 中的大多数加密方法提供 `单元级加密`，它应用于单个数据项，或包含敏感信息的表列。相比之下，透明数据加密（SQL Server 2008 的一项新功能）则应用 `数据库级加密` 到整个数据库，它通过加密文件系统上的底层数据库文件来实现。这些方法不仅在作用范围上不同，而且对于任何使用加密数据的应用程序的设计都有显著不同的影响。

在决定加密策略之前，对数据进行分类以识别泄露风险以及泄露后果的严重性至关重要。以下是一些可能需要加密的数据类型示例：

*   许多组织定义了一个或多个级别的 `敏感数据`——即，如果落入错误之手，可能对业务造成负面影响的信息。此类信息可能包括定价信息、利润预测或即将进行的营销计划。如果竞争对手获取了此类信息，可能对未来的销售产生严重后果。
*   越来越多的政府正在通过法律和监管要求，规定必须实施某些最低安全标准来保护机密的客户数据，而这可能包括加密。未能满足这些标准可能导致公司面临巨额罚款，甚至被迫停止交易。
*   某些客户可能会在客户服务级别协议中规定所需的最低保护级别。在某些情况下，详细要求会明确规定哪些数据项必须加密、使用的加密算法类型、加密密钥的最小长度以及密钥必须轮换的时间表。
*   在许多情况下，对员工记录进行加密可能是有益的，特别是那些包含工资信息或其他个人信息的记录，以防止系统管理员或其他有权访问此类数据的用户进行随意窥探。

对业务需求进行透彻分析，以识别所有需要保护的数据元素，并确定应如何保护这些数据，对于确保任何提出的加密解决方案在满足安全需求的同时平衡性能要求至关重要。此外，与加密相关的应用程序设计和业务流程的变更通常会产生成本，因此能够根据从加密中获得的预期收益来证明该成本的合理性非常重要。
