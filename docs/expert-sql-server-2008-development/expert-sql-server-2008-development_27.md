# 第一章 — 数据库世界的软件开发方法论

## 异常是任何接口的重要组成部分

在思考隐含契约时，开发者经常未能考虑到的一种重要输出类型是，当事情出错时，给定方法可能抛出的异常。许多方法在某些情况下会抛出明确定义的异常，但如果这些异常没有得到充分记录，它们原本良好的意图就会变得相当浪费。通过确保正确记录异常，你不仅能让客户端捕获并处理你已预见的异常，还能帮助开发者理解哪些地方可能出错，并针对可能的问题进行防御性编码。通常，围绕潜在问题遵循代码路径几乎总是比不得不处理异常要好。

## 集成数据库与面向对象系统

一个似乎使数据库开发变得比其应有的难度大得多的主要问题，其实与开发本身无关，而是一个架构问题。面向对象框架和数据库系统通常不能很好地协同工作，主要是因为它们有一套不同的核心目标。面向对象系统旨在从行为的角度对业务实体进行建模——业务实体能做什么，其他实体能对它做什么或者能与它一起做什么？另一方面，数据库更关心实体之间的关系，而较少关心它们所参与的活动。

显然，我们有两个不兼容的范式来对业务实体建模。然而，两者都是几乎每个应用程序的必要组成部分，必须为了共同的目标——服务用户——而一起被利用。为此，数据库开发者需要知道什么属于哪里，以及何时将责任交还给他们的应用程序开发者同行，这一点很重要。不幸的是，如何为任何给定的业务流程的各个部分进行适当建模的问题，可能会迅速让人陷入一个灰色地带。你应该如何决定是在数据库中实现还是在应用程序中实现？

自古以来（或至少自互联网诞生以来），许多数据库论坛上的核心争论一直围绕着如何处理那始终存在的必需“逻辑”。可悲的是，尽管我们努力尝试，开发者们仍然没有想出如何在不需要实现业务需求的情况下开发应用程序。因此，争论仍在激烈进行。“业务逻辑”属于数据库？属于应用层？用户界面呢？较新的应用程序架构对这个古老的问题有何影响？

## 逻辑放置的简史

从前，计算机只是被称为“计算机”。它们日夜不停地为“哑”终端提供少量数据。那时，应用程序和数据之间没有太大区别，因此对于我们今天争论的架构问题，几乎没什么问题可问，也没什么答案可给。

但是，随着时间的推移，变革之风吹遍了世界各地装有空调的数据中心，以前被称为“计算机”的系统变成了“大型机”——1960 年代中期机架上的新计算机是“小型机”。比大型机更小更便宜的“小型机”迅速流行起来。它们相对于大型机较低的成本意味着，现在通过将应用程序运行在多台机器上来实现横向扩展在财政上成为可能。此外，这些机器足够便宜，甚至可以直接被最终用户用作之前无处不在的哑终端的替代品。在同一时期，我们也看到了第一批商业上可用的数据库系统，例如 `Adabas` 数据库管理系统（`DBMS`）。

小型机的出现标志着应用程序架构领域的多重变化。除了多服务器横向扩展的替代方案外，最终用户开始使用比终端更强大的机器这一事实意味着，在某些情况下，应用程序的部分工作可以卸载到用户界面（UI）层。工作负载现在可以进行分配以创建更具可扩展性的应用程序，而不再仅仅利用一台服务器的能力。

随着时间的推移，“微型计算机”（当今基于英特尔和 AMD 的系统的祖先）变得越来越强大，最终小型机消失了。然而，在小型机时代诞生的基于客户端/服务器的架构并没有消亡；应用程序开发者发现，将工作卸载到客户端可能比购买更大的服务器便宜得多。
