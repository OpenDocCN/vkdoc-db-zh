# 第三章 测试数据库例程

质量保证领域内存在多种测试方法，但总的来说，所有类型的软件测试都可以分为两类：

*   `黑盒测试` 指的是那些只对被测模块的输入输出做出假设，而不验证中间条件的测试。测试人员无需了解（也不要求了解）模块的内部工作原理——因此它们被包含在一个“黑盒”中。
*   `白盒测试` 则恰恰相反，它包括任何测试人员知晓并验证被测例程或函数内部实现的测试。白盒测试也被称为“开盒测试”，因为测试人员被允许查看模块内部以了解其运作方式，而不仅仅是检查其输入和输出。

在这两大分类之下，还有许多旨在针对应用程序不同特定领域的具体测试。黑盒测试的例子包括单元测试、安全测试以及基本的性能测试，如压力测试和耐久测试。随着测试阶段的推进，会识别出需要进一步测试的领域，所执行的测试类型也倾向于从黑盒转向白盒，以专注于特定的内部元素。

从数据库开发的视角来看，白盒测试的例子包括：验证模块内部工作的功能测试、执行数据验证的测试，以及当性能调优需要透彻了解数据访问方法时的测试案例。例如，在性能测试期间检索和分析查询计划，就是对存储过程进行白盒测试的一个例子。

### 单元测试与功能测试

专注于数据层的软件开发在测试方面有一个好处：需要熟悉的测试类型并不多。可以说，最重要的两类测试是那些验证应用程序行为是否符合预期，并返回正确结果的测试。这就是单元测试和功能测试的目的。

`单元测试` 是验证接口所公开契约的黑盒测试。例如，对存储过程的单元测试应该验证：给定一组特定的输入，存储过程返回的输出结果集是否符合被测存储过程接口所定义的正确集合。这里使用的“正确”一词需要仔细定义。它仅在存储过程契约所定义的范围内意味着“正确”；实际返回的数据本身并不重要。因此，只要结果代表了根据接口契约定义的、格式和数据类型正确的有效值，单元测试就应该通过。换句话说，单元测试测试的是接口按照其契约规定与外部世界进行通信的能力。

`功能测试` ，顾名思义，验证的是被测对象的功能。在测试术语中，“功能测试”一词的含义比“单元测试”模糊得多。它可以指代任何类型、在应用程序任何层级上，测试该部分应用是否正常工作的测试——换句话说，就是测试它是否执行了适当的操作序列以交付预期的正确最终结果。对于一个从数据库中选择数据的简单存储过程而言，功能测试要问的问题是：该存储过程是否返回了正确的数据？同样，我将仔细定义“正确”这个词。这一次，“正确”既包含为单元测试所做的那种验证（数据必须格式正确），也包含对返回值实际准确性的更深层次验证。进行这种验证所需的逻辑意味着，在数据库领域中，功能测试是一种白盒测试，与单元测试的黑盒性质形成对比。

让我们看一个例子来让这些想法更清晰一些。考虑下面这个可能用于银行应用程序的存储过程：

```sql
CREATE PROCEDURE GetAggregateTransactionHistory
    @CustomerId int
AS
BEGIN
    SET NOCOUNT ON;

    SELECT
        SUM
        (
            CASE TransactionType
                WHEN 'Deposit' THEN Amount
                ELSE 0
            END
        ) AS TotalDeposits,
        SUM
        (
            CASE TransactionType
                WHEN 'Withdrawal' THEN Amount
                ELSE 0
            END
        ) AS TotalWithdrawals
    FROM TransactionHistory
    WHERE
        CustomerId = @CustomerId;
END;
```

这个存储过程隐含的契约表明，给定 `@CustomerId` 参数输入一个客户 ID，将输出一个包含两列、零行或一行的列结果集（该契约并未对无效的客户 ID 或没有进行过任何交易的客户做出任何暗示）。

输出结果集中的列名将是 `TotalDeposits` 和 `TotalWithdrawals`，列的数据类型将与 `TransactionHistory` 表中 `Amount` 列的数据类型相同（我们假设它是 decimal 类型）。

### 如果客户不存在怎么办？

`GetAggregateTransactionHistory` 存储过程的输出，无论你传入的是一个碰巧没有交易的有效客户 ID，还是一个无效的客户 ID，都将是相同的。无论哪种情况，该过程都将不返回任何行。根据特定情况的需求，通过稍微改变规则来使接口更丰富可能更有意义：即仅当传入无效的客户 ID 时才不返回行。这样，调用方将能够识别无效数据，并向用户提供适当的错误消息，而不是暗示不存在的客户没有进行交易。

针对此存储过程的单元测试应该只做接口验证。应传入一个客户 ID，单元测试应检查输出结果集（或其缺失），以确保存在两列名称和数据类型正确的列，以及零行或一行。无需进行数据验证；例如，去核查汇总信息是否准确就超出了范围——那将是功能测试的工作。

我们之所以在单元测试和功能测试之间做出如此区分，是因为在测试纯粹的接口合规性时，我们希望将自己置于从更高层编程使用该接口的人的立场上。接口是否按文档工作，提供了适当级别的封装并以正确的格式返回数据？

系统中的每个接口都需要一个或多个此类测试（参见本章后面的“需要多少测试？”一节），因此它们需要保持专注和轻量。为每个接口编写完整的白盒测试可能不可行，而且在更高层级（例如通过用户界面本身）测试数据的有效性可能更简单。在 `GetAggregateTransactionHistory` 存储过程的例子中，编写一个功能测试本质上相当于重写整个存储过程——这绝不是开发人员时间的良好利用。

### 单元测试框架

使用单元测试框架可以简化单元测试，这些框架提供了结构化的编程接口，旨在帮助快速测试软件。这些框架通常利用 `调试断言` ，允许开发人员指定那些使测试为真或为假的条件。

调试断言是一种特殊的宏，只有在软件以调试模式编译时才会被启用。它接受一个表达式作为输入，如果表达式为假则抛出异常；否则返回真（或在某些语言中返回 void）。例如，以下断言将始终抛出异常：

```
Assert(1 == 0);
```
