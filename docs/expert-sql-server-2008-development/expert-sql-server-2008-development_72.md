# 加密功能

## 使用密码短语进行加密

有时，您可能希望在不处理创建和安全存储永久对称密钥相关问题的情况下加密数据。在这种情况下，`ENCRYPTBYPASSPHRASE`函数可能会很有用。

`ENCRYPTBYPASSPHRASE`根据提供的密码短语，使用 Triple DES 算法生成对称密钥，并使用该密钥加密提供的明文字符串。此方法的主要好处是它不需要创建或存储任何密钥或证书；当需要解密数据时，只需向`DECRYPTBYPASSPHRASE`方法提供相同的密码短语，该方法即可生成相同的对称密钥来解密密文。密钥本身在任何时候都不会被存储，仅在需要时临时生成。

要使用密码短语加密数据，请将密码短语后跟一个明文字符串作为参数提供给`ENCRYPTBYPASSPHRASE`函数，如下所示：

```sql
SELECT ENCRYPTBYPASSPHRASE('PassPhrase', 'My Other Secret Message');

GO
```

当我运行此代码清单时，我得到了如下所示的二进制值。但请记住，`ENCRYPTBYPASSPHRASE`使用非确定性对称加密，因此每次执行此查询时都会获得不同的结果。

`0x010000007A65B54B1797E637F3F018C4100468B115CB5B88BEA1A7C36432B0B93B8F616AC8D3BA7307D5005E`

要再次解密数据，请将相同的密码短语和加密后的文本传递到`DECRYPTBYPASSPHRASE`函数中。与 SQL Server 提供的所有非对称和对称加密方法一样，解密后的结果数据以`varbinary`数据类型返回，可能需要转换为适合您应用程序的格式。在本例中，将结果转换为`varchar`即可返回原始明文字符串：

```sql
SELECT CAST(DECRYPTBYPASSPHRASE('PassPhrase',
0x010000007A65B54B1797E637F3F018C4100468B115CB5B88BEA1A7C36432B0B93B8F616AC8D3BA7307D5005E) AS varchar(32));

GO
-- 结果: My Other Secret Message
```

如演示所示，`ENCRYPTBYPASSPHRASE`是一种非常简单的实现对称加密的方法，无需处理密钥管理相关问题。然而，它也并非没有缺点：

首先，由`ENCRYPTBYPASSPHRASE`生成的密文只能在使用用于加密它的密码短语时才能解密。这看似显而易见，但请记住 SQL Server 没有提供备份或重新创建密码短语的机制，因此，如果密码短语丢失，原始数据将永远无法恢复。这意味着您需要考虑如何安全地备份密码短语的策略；如果存储访问数据所需的唯一知识（密码短语）的地方是公开可访问的，那么加密数据就没有多大意义。

其次，`ENCRYPTBYPASSPHRASE`函数总是使用 Triple DES 算法加密数据。尽管这是一个广为人知且广泛使用的标准，但某些业务需求或客户规范可能要求更强的加密算法，例如基于 AES 的算法。虽然 SQL Server 2008 可以创建比特长度为 128、192 和 256 的 AES 对称密钥，但这些不能被`ENCRYPTBYPASSPHRASE`使用。

对称密钥的强度完全取决于生成它的密码短语，但 SQL Server 不会对提供给`ENCRYPTBYPASSPHRASE`的密码短语施加任何密码复杂性要求。这可能导致基于选择不当的密钥而产生的弱加密强度。

提供的明文只能是`varchar`或`nvarchar`类型（尽管通过先将任何其他数据类型进行`CAST`转换，可以比较容易地克服此限制）。

尽管存在这些弱点，`ENCRYPTBYPASSPHRASE`在某些情况下仍然是一个有用的选择，例如当需要加密来自`sysadmin`和`db_owner`角色用户的数据时。有关此主题的更多信息，请参阅题为“保护信息不被 DBA 访问”的侧栏。

## 保护信息不被 DBA 访问

最近企业 IT 部门外包或离岸外包的趋势，加上关于谁应被允许访问敏感数据的日益严格的法规，意味着限制对敏感数据的访问（即使是来自`sysadmin`和`db_owner`角色的用户）已成为一个日益普遍的需求。

DBA 拥有查看、插入、更新和删除任何数据库表中数据的权限，任何处于`sysadmin`角色的 DBA 都控制着 SQL Server 实例上的每个密钥和证书。那么，如何设计数据库架构来保护数据不被这些用户访问呢？任何依赖于自动密钥管理层次结构的解决方案都将不起作用；唯一的方法是依赖一个对这些用户保密的密码。这个密码可以与`ENCRYPTBYPASSPHRASE`函数一起使用，也可以用来保护一个证书或非对称密钥，而该证书或密钥又保护了加密数据的对称密钥。

即便如此，也需要注意风险；`sysadmin`仍然可以`DROP`任何密钥，并且您必须确保在提供密码时，采用安全的方式，使其在通过网络传递或通过数据库分析工具检测时无法被发现。

### 非对称密钥加密

非对称加密是使用一对相关的密钥执行的：**公钥**用于加密数据，相关的**私钥**用于解密生成的密文。非对称密钥对的公钥和私钥是相关的，但它们之间的数学关系足够复杂，以至于无法从已知的一个密钥推导出另一个密钥。图 6-5 说明了非对称加密模型。

除了非对称密钥对，非对称加密也可以使用**证书**实现。基于 X.509 标准的证书用于将公钥绑定到有权使用关联密钥加密数据的给定身份。证书可以由第三方认证机构（CA）颁发，也可以在 SQL Server 2008 内部创建自签名证书。自签名证书在上一章中用于演示为存储过程分配权限的一种方法。

证书加密和非对称密钥加密使用相同的算法，并且假设密钥长度相同，则提供完全相同的加密强度。对于 SQL Server 中处理非对称密钥加密的每种方法，都有一个等效的方法为证书加密提供相同的功能。

## 非对称密钥还是证书？

证书和非对称密钥加密都使用相同的、广泛使用的 RSA 算法。假设密钥长度相同，证书和非对称加密将提供相同的加密强度，并且两种类型可用的功能之间没有显著差异。那么，应该选择哪一个呢？

决策可能受到您选择在 SQL Server 内部生成密钥还是从外部来源生成密钥的影响。SQL Server 中的自签名证书密钥长度只能是 1024 位，而非对称密钥可以创建私钥长度为 512、1024 或 2048 位的更强密钥。

但是，非对称密钥和证书都可以从外部来源导入，在这种情况下，密钥长度最高可达 3456 位。

我推荐使用证书进行非对称加密，而不是非对称密钥对，因为证书允许在密钥旁边存储额外的元数据（例如过期日期），并且证书可以使用`BACKUP CERTIFICATE`命令轻松备份到 CER 文件，而非对称密钥对却奇怪地缺乏提供等效功能的方法。
