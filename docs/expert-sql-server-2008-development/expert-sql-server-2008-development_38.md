# 数据库编程中的防御性编程实践

## 示例：创建与修改存储过程

考虑以下存储过程 `Divide` 的代码清单，它描述了一个简单的用于将一个数字除以另一个数字的存储过程：

```sql
CREATE PROCEDURE Divide (
    @x decimal(18,2),
    @y decimal(18,2)
)
AS BEGIN
    SELECT @x / @y
END;
GO
```

根据上面的代码，如果例如 `@y` 的值为 `0`，就很容易引发异常。如果只是想阻止错误消息出现，可以在 `catch` 块中捕获（或“吞掉”）该异常，如下所示：

```sql
ALTER PROCEDURE Divide (
    @x decimal(18,2),
    @y decimal(18,2)
)
AS BEGIN
    BEGIN TRY
        SELECT @x / @y
    END TRY
    BEGIN CATCH
        /* Do Nothing */
    END CATCH
end;
GO
```

然而，重要的是要意识到前面的代码清单并非防御性的——它没有做任何事情来防止异常情况的发生，其唯一效果是允许系统继续运行，假装什么坏事都没发生。像这样的**异常隐藏**非常危险，并且几乎无法确保应用程序的正确运行。防御性的方法应该是，在尝试执行除法之前，显式检查该操作成功所需的所有条件是否满足。这意味着要断言诸如确保为 `@x` 和 `@y` 提供了值（即它们不是 `NULL`）、`@y` 不等于零、所提供的值位于 `decimal(18,2)` 数据类型可以存储的范围内等等。

以下代码清单为同一过程提供了一种简化的防御性方法：

```sql
ALTER PROCEDURE Divide (
    @x decimal(18,2),
    @y decimal(18,2)
)
AS BEGIN
    IF @x IS NULL OR @y IS NULL
    BEGIN
        PRINT '请为 @x 和 @y 提供值';
        RETURN;
    END

    IF @y = 0
    BEGIN
        PRINT '@y 不能等于 0';
        RETURN;
    END

    BEGIN TRY
        SELECT @x / @y
    END TRY
    BEGIN CATCH
        PRINT '发生了一个未处理的异常';
    END CATCH
END;
GO
```

对于前面的示例，每个断言都附带一条简单的 `PRINT` 语句，以通知过程执行所需的哪些条件失败了。在现实生活中，这些代码路径可能以多种方式处理此类断言——通常是记录错误、向用户报告消息，并在可能的情况下尝试继续系统操作。通过这样做，它们防止了与未预期异常相关的不可预测行为。

防御性编程可以与 **`快速失败`** 方法论形成对比，后者侧重于通过在异常发生时导致应用程序停止来立即识别遇到的任何错误。防御性方法并不主张敲响警钟和闪烁灯光，但这并不意味着它隐藏错误——它只是以更优雅的方式向最终用户报告错误，并在可能的情况下继续系统核心部分的运行。

## 为什么对数据库开发使用防御性方法？

如前所述，防御性编程并非唯一可应用于数据库开发的软件开发方法论。其他常见方法包括 TDD、XP 和快速失败开发。那么，为什么我选择在本章乃至本书中专注于防御性编程呢？我相信防御性编程最适合数据库开发，原因如下：

**数据库应用程序的预期寿命往往比其他软件应用程序更长。** 虽然认为数据库专业人员是软件开发领域中谨慎、一丝不苟的人可能是一个被过度使用的刻板印象，但事实是数据库开发往往比其他技术更缓慢、更谨慎。例如，Web 应用程序可能会为了利用当时流行的技术而每年修订和重新发布。相比之下，数据库开发往往是缓慢而稳定的，数据库应用程序可能保持多年有效而无需从技术角度更新。因此，更容易证明与防御性编程相关的更高前期开发成本是合理的。可靠性和抗缺陷的好处通常将享受更长的时间。

**用户（和管理层）对数据库应用程序中的缺陷容忍度更低。** 大多数最终用户已经习惯甚至预期桌面和 Web 软件中的缺陷。虽然这无疑会造成挫折感，但许多人习惯于在 Web 浏览器挂起或某些应用程序无法正常关闭时，通过按下 `ctrl+Alt+Delete` 来重置机器。然而，对个人桌面软件表现出的同样的容忍度通常不会扩展到企业数据库应用程序。最近一些广为人知的丑闻，其中几个政府和大型组织的系统中的缺陷被利用，进一步加剧了公众对任何可能对数据库完整性构成风险的事情的极端敏感性。

**数据库应用程序中存在的任何缺陷都可能比其他软件产生更严重的后果。** 可以说，人们更担心数据库缺陷而不是其他软件缺陷是绝对正确的。桌面应用程序中的意外错误可能导致文档或文件损坏，这很麻烦并可能导致不必要的返工。但数据库中的意外错误可能导致重要的个人、机密或敏感数据面临风险，这可能带来相当严重的后果。通常存储在数据库中的数据的性质要求采用一种谨慎、彻底的开发方法，例如防御性编程所提供的方法。

## 为长寿命而设计

消费软件应用程序的预期保质期越来越短，压缩的发布周期使得一个版本刚刚上市，前一个版本几乎就要过时。然而，情况并非必须如此。设计良好、经过防御性编程的应用程序可以持续运行多年。在我工作过的一个组织中，创建了一个战术性的短期管理信息数据存储，以便在组织系统因合并而进行整合期间，基本的业务报告功能可以继续。尽管只需要在合并后的立即期间使用，但这个（不幸命名为）短期管理信息数据库继续被使用长达十年之久，因为它比后来尝试的替代品更可靠、更健壮。

这是一个教训：在为数据库选择描述性名称时，要选择不会随时间推移而过时的名称！
