# 第 章 „ 面向数据库世界的软件开发方法论

可维护性还与可测试性在另一个关键方面相关联：一个例程越容易测试，就越容易修改。当存在能够快速验证新实现方法的测试时，破坏性变更就不再是大问题。

### 安全性

在一个身份盗窃几乎每晚都会上新闻、一台连接到互联网且未设防的计算机将在 30 秒内被入侵的时代，安全性被视为开发软件应用程序最重要的领域之一也就不足为奇了。然而，安全性也是最复杂的领域之一，而复杂性可能隐藏着经过训练的攻击者可以轻易利用的缺陷。
复杂的安全方案也会对特定软件的可测试性和可维护性产生巨大影响。从测试的角度来看，开发人员需要考虑给定的安全方案是否会创造出过多的变量，从而使测试变得不可行。例如，如果用户被分成不同的组，每个组拥有不同的安全权限，那么是否需要为每个用户组运行每组测试？在应用程序可以被视为“完全”测试之前，需要执行多少测试组合？
从维护的角度来看，安全方面的复杂性与任何其他类型实现的复杂性同样危险。一个给定的例程越复杂，维护起来就越困难（因此也越昂贵）。
在数据依赖型应用程序中，大部分安全责任通常会被推到数据层。数据层或数据库的安全责任通常包括诸如应用程序认证、数据查看授权和数据可用性等领域。将这些安全责任封装在数据库例程中，从整体应用程序可维护性的角度来看可能是有益的，但必须小心确保数据库例程不会变得如此不堪重负，以至于其可维护性、可测试性或性能受到影响。

## 为未来需求留有余地

在一个动态环境中，你面临不断变化的客户需求和额外的功能请求，很容易过多地关注明天的增强功能，而不是专注于今天的缺陷。浏览许多技术规范和数据字典时，常见到“为将来使用保留”这一短语。开发者倾向于相信，预先增加复杂性会通过减少后期开发工作量来带来优势。然而，这种猜测未来需求的方法常常适得其反，产生充满维护负担的软件。这些代码片段必须由开发团队携带并保持最新，以便编译应用程序，但它们通常在多年内完全未被使用。
甚至 SQL Server 本身也会遇到这些问题。例如，根据 `Books Online`，SQL Server 2008 中 `sp_stored_procedures` 存储过程返回的 `NUM_INPUT_PARAMS`、`NUM_OUTPUT_PARAMS` 和 `NUM_RESULT_SETS` 被保留供将来使用 (`http://msdn.microsoft.com/en-us/library/ms190504.aspx`)，正如它们在 SQL Server 2005 中一样 (`http://msdn.microsoft.com/en-us/library/ms190504(SQL.90).aspx`)。
在我参与过的一个有 15 年历史的应用程序中，最初的开发团队特别热衷于在代码库中预先填充为未来保留的功能。唉，经过几年、几轮裁员和大量人员流动后，原始团队成员已无一人留下。不得不处理这个 200 万行应用程序的剩余开发人员，不敢移除任何东西，生怕破坏某个早已被遗忘但仍被某个用户依赖的功能。至少可以这么说，这是一个惨淡的景象，很难想象这些年来为了保持所有冗余代码的更新浪费了多少时间。
