# CHAPTER 1 „ 面向数据库世界的软件开发方法论

### 接口即契约

可以说，接口是模块与其使用者之间表达的一种**契约**。该契约声明，如果使用者向接口指定了一组特定的参数，则将返回一组特定的值。**简单性**通常是这里的关键；避免定义那些根据输入改变返回值数量或类型的接口。例如，一个存储过程，如果用户传入特定参数就返回额外的列，这可能就是一个设计不佳接口的例子。

许多编程语言允许例程定义**显式契约**。这意味着输入参数是明确定义的，输出在编译时是已知的。不幸的是，SQL Server 中的 T-SQL 存储过程只定义输入，而过程本身可以动态改变其定义的输出。在这些情况下，开发人员有责任确保预期的输出有完善的文档记录，并且存在单元测试来验证它们（有关单元测试的信息，请参见第 3 章）。在本书中，我将通过文档和测试强制执行的契约称为**隐式契约**。

### 接口设计

如何衡量接口设计的成功与否是一个难题。一般来说，你应该尝试从维护的角度来看待它。如果六个月后，你因为性能或其他原因要完全重写该模块，你能保证所有的输入和输出都保持不变吗？

例如，考虑以下存储过程签名：

```
CREATE PROCEDURE GetAllEmployeeData
--用于排序的列，以逗号分隔
@OrderBy varchar(400) = NULL
```

假设这个存储过程完全按照其名称的含义行事——它返回数据库中每个员工的 `Employees` 表的所有数据。这个存储过程接受 `@OrderBy` 参数，根据注释，该参数定义为“用于排序的列”，并附加规定列应以逗号分隔。

这里的接口问题相当严重。首先，一个接口不仅要隐藏内部行为，还必须明确说明一组有效的输入参数将如何改变例程的输出。在这种情况下，该存储过程的使用者可能会认为，在内部，逗号分隔的列表将被简单地附加到一条动态 SQL 语句中。这是否意味着改变列表中列名的顺序会改变输出？另外，`ASC` 或 `DESC` 关键字是否可以接受？接口定义的契约不够具体，无法明确这一点。

其次，该存储过程的使用者必须拥有 `Employees` 表的列列表，才能知道可以在逗号分隔列表中传入的有效值。列列表应该硬编码在应用程序中，还是通过其他方式检索？而且，表的所有列是否都是有效的输入，这并不清楚。像 `Photo` 这样定义为 `varbinary(max)`、包含员工照片 JPEG 图像的列怎么办？允许使用者指定该列进行排序是否有意义？

从维护的角度来看，这类接口问题可能导致真正的问题。想象一下，如果三个不同的应用程序都在使用这个存储过程，并且都有自己的可排序列名硬编码列表，那么仅仅更改 `Employees` 表中一个列名需要付出多少努力。如果查询最初实现为动态 SQL，但后来需要更改为静态 SQL 以避免重编译成本，该怎么办？在那些假定可以使用 `ASC` 和 `DESC` 关键字的应用程序抛出运行时异常之前，我们是否有可能检测出它们是哪些应用程序？
