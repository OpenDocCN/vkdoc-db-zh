# 注释与代码清晰性

开发者试图实现的目标。使用注释描述*预期*行为，能更容易识别*意外*情况。一般来说，没有必要简单地注释内置函数的作用，但也有例外。例如，你能一眼看出下面代码预期的结果是什么吗？

```sql
DECLARE @y int = 2010, @c int, @n int, @k int, @i int, @j int, @l int, @m int,
@d int;

SET @c = (@y / 100);
SET @n = @y - 19 * (@y / 19);
SET @k = (@c - 17) / 25;
SET @i = @c - @c / 4 - ( @c - @k) / 3 + 19 * @n + 15;
SET @i = @i - 30 * (@i / 30);
SET @i = @i - (@i / 28) * (1 - (@i / 28) * (29 / (@i + 1)) * ((21 - @n) / 11));
SET @j = @y + @y / 4 + @i + 2 - @c + @c / 4;
SET @j = @j - 7 * (@j / 7);
SET @l = @i - @j;
SET @m = 3 + (@l + 40) / 44;
SET @d = @l + 28 - 31 * ( @m / 4 );

SELECT CAST(CONVERT(char(4),@y) + '-' + RIGHT('0' + CONVERT(varchar(2),@m),2) + '-'
+ RIGHT('0' + CONVERT(varchar(2),@d),2) AS DateTime);
```

我曾在生产应用中遇到过前面的函数，它被用来确定员工是否因在公共假期工作而有资格获得奖金。如果你还没弄明白，这个结果会返回任意给定年份（通过变量`@y`指定）的复活节星期日日期。代码确实完成了它的目的，但没有任何注释，我花了很长时间才弄清楚这个目的是什么！

在许多情况下，你可以通过使用**自文档化**的代码——选择命名良好的变量、列别名和表别名——来避免编写显式注释。考虑以下代码：

```sql
SELECT DATEPART(Y, '20090617');
```

在大多数编程语言中，日期格式函数中使用的字符`Y`表示与日期相关的年份。因此，你可能期望前面的代码返回所提供日期的完整年份`2009`，或者可能只是年份的最后一位数字`9`。要解释实际结果为`168`，代码本可以通过用`DAYOFYEAR`（它是其缩写）替换`Y`，从而轻松实现自文档化：

```sql
SELECT DATEPART(DAYOFYEAR, '20090617');
```

## 代码缩进与语句块

代码缩进和大量使用空白有助于识别逻辑代码块、循环和批处理，从而创建可理解、易于维护且未来引入错误可能性更低的代码。然而，这些实践对代码本身的执行显然没有直接影响。因此，至关重要的是代码的视觉布局要强化其逻辑行为，因为格式不佳的代码实际上可能具有误导性。考虑以下例子：

```sql
IF 1 = 1
    PRINT 'True';
ELSE
    PRINT 'False';
    PRINT 'Then Print this';
```

在这种情况下，最后一行代码的缩进使其看起来像是`ELSE`子句的一部分，但事实并非如此，无论测试结果如何，`Then Print this`都会被打印出来。

为了避免这种误导性情况，我总是建议大量使用由`BEGIN`和`END`标记的语句块，即使块只包含一个语句，如下所示：

```sql
IF 1 = 1
BEGIN
    PRINT 'True';
END
ELSE
BEGIN
    PRINT 'False';
END

PRINT 'Then Print This';
```

另一个可以轻松避免的误导性做法是未能使用括号来显式展示查询组件被解析的顺序。考虑以下代码清单：

```sql
DECLARE @Table TABLE (x int, y int);
INSERT INTO @Table VALUES (1,1), (1,2), (2,1), (2,2), (3,1), (3,2);

SELECT *
FROM @Table
WHERE
    x = 1 AND
    y = 1 OR y = 2;
GO
```

在这种情况下，和之前一样，代码缩进实际上损害了代码的真实逻辑，即选择所有`x=1 AND y=1`，或`y=2`的行。

## 如果其他方法都失败了……

防御性编程的一个基本特性是进行断言以确保异常情况不会发生。可以说，如果防御性编程的理想状态真的实现了，就没有必要实现异常处理代码，因为任何可能导致异常的情况在被允许发生之前都已被识别和处理。

不幸的是，显式测试所有异常场景在实际上是不可能的，在现实世界中，异常和错误处理仍然是任何软件应用程序中非常重要的一部分。有关 SQL Server 中异常和错误处理的详细讨论，请参阅第 4 章。

### 创建健康的开发环境

最好的应用程序并非由一两个程序员的个人才华创造，而是由开发团队协调、有效的协作创造。当编码成为一项共享、开放的活动时，成功的防御性开发最有可能发生。

协作编码的好处在于，你可以利用共享的技术知识和资源池，确保编码彻底而准确。不同的人能够从多个不同的批判性视角审视代码，这有助于识别可能被单个开发者忽视的任何假设。

如果开发者孤立工作，他们可能会引入依赖关系，这会给代码未来的可维护性带来风险。如果只有一个开发者了解某段特别复杂代码的细节，然后该开发者离职或无法联系上，将来你可能会遇到维护该代码的困难。事实上，开发者之间的个人竞争可能导致开发者*故意*给应用程序增加复杂性。编码者可能会试图确保只有他们自己理解某段复杂代码的工作原理，无论是作为炫耀其技术知识的一种方式，出于个人自豪感，还是作为制造对其依赖的一种手段——使他们自己变得不可或缺并确保未来的工作保障。所有这些都创造了一个不健康的开发环境，并可能对所产生代码的质量产生负面影响。

负责开发团队的管理者应努力营造一个持续专业发展的环境，其中共享学习和最佳实践是关键。软件开发是一个不断变化的领域——现在被认为是最佳实践的东西可能在几年内就过时了。为了确保应用程序保持前沿，应促进和鼓励开发者的个人培训和同行之间的知识共享。

实施防御性开发的成功与否也可能受到更广泛公司决策的影响，包括奖励制度。例如，公司可能实施一项奖励计划，为任何在生产应用中发现并解决错误的开发者支付个人奖金。虽然其初衷可能是提高软件质量，但效果可能恰恰相反——毕竟，如果防御性编码（在错误发生前预防）消除了开发者 later 个人获得修复错误奖励的机会，那么进行防御性编码的动力是什么？此类政策可能会鼓励竞争性的、个人主义的行为，开发者只为自己考虑，而不是基于项目的最佳利益采取行动。
