# 第二章：数据库编程最佳实践

防御性编程实践并没有一个明确的列表，但普遍认为在开发中采取防御姿态包括以下原则：

*   保持简单（或 `KISS`——保持简单，愚蠢）。应用程序的强大和有效并非源于其复杂性，而是源于其优雅的简洁。复杂性会隐藏缺陷，应在应用程序设计和编码实践中都予以避免。
*   “即使没坏，也要修复它。”防御性编程鼓励持续、主动地进行测试，并为应用程序做未来的防护，以应对未来可能发生的破坏性变更，而不是等待东西坏掉。
*   在所有的阶段和开发过程中都保持质疑、彻底和谨慎。应进行“如果……会怎样？”的分析，以识别在应用程序正常（和非正常）使用期间可能发生的异常场景。
*   应与不同的同行群体（包括其他开发人员或技术团队、顾问、最终用户和管理人员）进行广泛的代码审查和测试。这些不同的群体可能有不同的隐含假设，而这些假设可能不会被封闭的开发团队考虑到。
*   应尽可能避免假设。如果一个应用程序需要某个条件为真才能正确运行，则应有一个对此的显式断言，并且应插入相应的代码路径来根据结果进行检查并采取相应行动。
*   应用程序应由简短、高内聚、松耦合的模块构建而成。以这种方式良好封装的模块可以被彻底地独立测试，然后可以在整个应用程序中自信地复用。复用特定的代码模块，而不是复制功能，可以减少引入新缺陷的机会。

在本章的剩余部分，我将提供我认为是展示上述每项原则的最佳实践的简单示例，并且这些概念将在本书的后续章节中不断被重新审视。

## 对防御性编程的态度

采用防御性编程方法的关键优势本质上是双重的：

*   防御性应用程序通常健壮且稳定，需要较少的关键缺陷修复，并且更能抵御那些在其他情况下可能导致昂贵故障或崩溃的情况。因此，它们具有较长的预期寿命和相对较低的持续维护成本。
