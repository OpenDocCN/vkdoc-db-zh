# 第 7 章

## SQLCLR：架构与设计考量

当微软在 SQL Server 2005 中首次宣布 SQL Server 将托管 .NET 公共语言运行时（CLR）时，这在数据库界引起了很多兴奋。其中一部分兴奋是来自开发者的热情支持，他们设想了许多数据库场景可能从 .NET 基础类库提供的方法中受益。然而，也有相当多的紧张和抵制来自 DBAs，他们担心新技术带来的威胁，以及传言说不怀好意的开发者将能够创建大量 DBA 无法穿透的、编译好的进程内数据访问代码。

事实证明，`SQLCLR` 集成既没有许多人想象的那么可怕，也没有那么有用。那些希望将 `SQLCLR` 特性作为 `T-T-SQL` 完全替代品的人很快就被劝退了，因为编写 `CLR` 例程通常需要更多的代码，并且由于在 `CLR` 边界上封送数据的持续成本，性能和可靠性会受到影响。而对于那些本身不是 .NET 开发者的 DBA 来说，这个其实用途不多的功能涉及一个相当陡峭的学习曲线。
