# SQLCLR：架构与设计考量

在 .NET 的 `Int32` 类型中，类似 `checked { return x * -1; }` 这样的操作，如果 x 为 `-2147483648`，会因溢出而"环绕"，产生 `-2147483648`。在 SQL Server 中，这种行为永远不会发生——相反，会引发溢出异常。

为了在这两种类型范式之间提供一层抽象，.NET Framework 附带了一个名为 `System.Data.SqlTypes` 的命名空间。该命名空间包含一系列结构，这些结构将 SQL Server 类型和行为映射到 .NET 中。这些结构中的每一个都通过 `INullable` 接口实现可空性，该接口公开了一个 `IsNull` 属性，允许调用者确定给定类型的实例是否为 `NULL`。此外，这些类型遵循与 SQL Server 原生类型相同的范围、精度和运算符规则。

简单地说，正确使用 `SqlTypes` 类型是确保进出 SQLCLR 例程的数据由每个类型系统正确处理的最有效方法。我的建议是，只要可能，所有作为 SQLCLR 对象公开的方法都应使用 `SqlTypes` 类型作为输入和输出参数，而不是标准的 .NET 类型。这需要前期更多的开发工作，但它应该能在一定程度上使你的代码适应未来，并有助于避免类型兼容性问题。

## 封装代码以促进跨层重用

SQLCLR 集成的一个主要卖点，特别是对于使用 .NET Framework 进行应用程序开发的商店来说，是能够在有意义时轻松地在各层之间移动或共享代码。然而，实现这一目标并不那么容易。

### 问题

不幸的是，在 SQLCLR 环境中工作的一些设计必要性不能很好地转化到应用层，反之亦然。其中一个例子就是前一节描述的 `SqlTypes` 的使用；尽管建议在 SQLCLR 例程的所有接口中使用它们，但这个规定在应用层没有意义，因为 `SqlTypes` 不支持原生 .NET 类型支持的全部运算符和选项。在每种情况下都使用它们可能会使数据访问变得简单，但会剥夺你执行许多复杂数据操作任务的能力，因此与其说是有益的改变，不如说是一种阻碍。

重写代码或创建为不同层定制的多个版本根本不能促进可维护性。在最佳情况下，应用程序使用的任何给定逻辑片段都应该只在一个地方编码——无论有多少不同的组件使用该逻辑或其部署在何处。这是面向对象编程的核心设计目标之一，重要的是要记住，它也适用于在 SQL Server 内部重用的代码。

### 一个合理的解决方案

与其重写例程和类型以使它们与 `SqlTypes` 兼容并实现其他特定于数据库的逻辑，我建议你养成设计 **包装器方法** 和类的习惯。这些包装器应该将 `SqlTypes` 的输入和输出映射到原始代码实际使用的 .NET 类型，并通过程序集引用调用到底层例程。包装器也是实现特定于数据库的逻辑的好地方，这些逻辑在最初为应用层设计的例程中可能不存在。

除了代码本身可维护性的好处外，创建包装器还有其他几个优点。首先，单元测试不需要重写——在应用层工作的相同测试在数据层仍然适用（尽管你可能想为包装器例程编写辅助的单元测试）。其次——也许更重要的是——封装原始程序集有助于维护最低权限的编码模型并增强安全性，正如本章后面“使用代码访问安全权限”和“使用宿主保护权限”部分所讨论的那样。

## 一个简单的例子：电子邮件地址格式验证

网页表单要求提供电子邮件地址是很常见的，你无疑遇到过这样的表单，它们会告诉你输入的电子邮件地址是否不符合预期的标准格式。这种验证提供了一种比实际发送电子邮件并等待回复更快——但效果较差——的测试电子邮件地址的方法，并且如果某些内容明显不正确，它会立即给用户反馈。

除了在前端验证中使用此逻辑外，在数据库中实现相同的方法以驱动 `CHECK` 约束也是有意义的。这样，任何到达数据库的数据——无论它是否已经通过了应用程序中的检查——都将被再次检查正确性。

以下是一个简单的 .NET 方法，它使用正则表达式验证电子邮件地址的格式：

```
public static bool IsValidEmailAddress(string emailAddress)
{
    //Validate the e-mail address
    Regex r =
        new Regex(@"\w+([-+.]\w+)*@\w+([-.]\w+)*\.\w+([-.]\w+)*");
    return (r.IsMatch(emailAddress));
}
```

当然，这段代码可以在 SQL Server 和应用层中直接使用——在 SQL Server 中使用它只需要加载程序集并注册函数。但这有一些问题，最明显的是缺少正确的 `NULL` 处理。照现在这样，当传入 `NULL` 时，此方法将引发 `ArgumentException`。根据你的业务需求，更好的选择可能是返回 `NULL` 或 `false`。另一个潜在问题发生在需要数据库与应用层逻辑略有不同的方法中。对于电子邮件验证的情况，很难想象如何为不同的层增强逻辑，但对于其他方法，这样的修改会带来可维护性挑战。

解决方案是在 SQL Server 中编录包含此方法的程序集，但不直接将该方法作为 SQLCLR UDF 公开。而是创建一个使用 `SqlTypes` 并在内部调用初始方法的包装器方法。这意味着底层方法不需要修改就能创建一个与数据库正确交互的版本，并且相同的程序集可以部署在任何层。下面是一个示例，展示了在 `IsValidEmailAddress` 方法之上创建的包装器方法，以暴露一个正确支持 `NULL` 输入和输出的 SQLCLR UDF 版本。假设我在一个名为 `UtilityMethods` 的类中创建了内部方法，并且也包含了对 `UtilityMethods` 程序集所用命名空间的 `using` 语句。

```
[Microsoft.SqlServer.Server.SqlFunction]
public static SqlBoolean IsValidEmailAddress(
    SqlString emailAddress)
{
    // Return NULL on NULL input
    if (emailAddress.IsNull)
        return (SqlBoolean.Null);

    bool isValid = UtilityMethods.IsValidEmailAddress(emailAddress.Value);

    return (new SqlBoolean(isValid));
}
```

请注意，此技术不仅可用于将应用程序层中的程序集加载到 SQL Server 中，也可用于反向操作——将逻辑从数据层迁移出来。鉴于 SQLCLR 的性质，应始终考虑代码移动性的潜力，开发人员即使在创建专门用于数据库的代码时，也应考虑使用包装器来设计方法——这将最大限度地提高以后重用的潜力，当或如果相同的逻辑需要迁移到另一层，甚至如果逻辑需要在数据层内部重用多次。

在 SQLCLR 环境中工作时，跨程序集引用还有其他好处。通过正确利用引用，可以创建更健壮、更安全的 SQLCLR 解决方案。
