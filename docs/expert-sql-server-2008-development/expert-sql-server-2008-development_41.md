# 数据库编程的最佳实践

为了防止出现错误的风险，确保任何防御性代码都遵循最新标准是合理的。虽然无法保证代码始终保持弹性，但你应该养成一个习惯：重写任何依赖已弃用功能的旧代码，并且在新开发中不使用任何已弃用功能，以减少未来发生异常的可能性。

已弃用功能指的是那些虽然当前仍在使用，但已被替代品取代的功能。虽然它们可能仍然可用（以确保向后兼容性），但你不应该使用已知已弃用的功能来开发应用程序。考虑以下代码清单：

```sql
CREATE TABLE ExpertSqlServerDevelopment.dbo.Deprecated (
    EmployeeID int DEFAULT 0,
    Forename varchar(32) DEFAULT '',
    Surname varchar(32) DEFAULT '',
    Photo image NULL
);

CREATE INDEX ixDeprecated ON Deprecated(EmployeeID);

DROP INDEX Deprecated.ixDeprecated;

INSERT INTO ExpertSqlServerDevelopment.dbo.Deprecated (
    EmployeeID, Forename, Surname, Photo) VALUES
    (1, 'Bob', 'Smith', DEFAULT),
    (2, 'Benny', 'Jackson', DEFAULT)

SET ROWCOUNT 1;

SELECT 'Name' = ForeName + ' ' + Surname
FROM ExpertSqlServerDevelopment.dbo.Deprecated
ORDER BY ExpertSqlServerDevelopment.dbo.Deprecated.EmployeeID

SET ROWCOUNT 0;
```

这个查询在 SQL Server 2008 中按预期工作，但使用了许多应该避免的已弃用功能。幸运的是，发现已弃用功能的使用很容易——`sys.dm_os_performance_counters` 动态管理视图（DMV）会维护每个已弃用功能被使用时的计数，可以通过以下方式查询：

```sql
SELECT
    object_name,
    instance_name,
    cntr_value
FROM sys.dm_os_performance_counters
WHERE
    object_name = 'SQLServer:Deprecated Features'
    AND cntr_value > 0;
```

一个相关的，或许更严重的，对防御性应用程序的威胁是依赖未记录功能的代码。SQL Server 中存在许多此类功能——例如，以下代码清单演示了未记录的 `sp_MSForEachTable` 存储过程，它可用于对数据库中的每个表执行提供的查询。

```sql
EXEC sp_MSforeachtable "EXEC sp_spaceused '?'";
```

虽然可以肯定已弃用功能将在未来的某个时间点被移除，但这个时间尺度通常是已知的，并且通常有一个记录的升级路径，以确保任何先前由已弃用功能提供的功能将被替代方法所取代。相比之下，未记录功能可能在任何时间无警告地中断，并且可能没有明确的升级路径。我强烈建议你避免这种冒险（并且几乎总是不必要的）做法。

## 限制暴露面

如果防御性编程旨在确保应用程序能够应对异常事件的发生，那么一个基本的防御技术就是限制此类事件可能发生的数量。从逻辑上讲，异常只能发生在正在运行的功能中，因此不要安装超过必要的功能——通过减少应用程序的受攻击面，你可以限制对潜在攻击的暴露。当 `SAFE` 可以满足时，不要授予程序集 `EXTERNAL_ACCESS` 权限。除非它们增加价值或应用程序严格需要，否则不要启用诸如数据库邮件之类的功能。

所有用户都应经过身份验证，并且仅被授权访问所需的资源，且仅在需要的时间段内。应立即删除未使用的账户，并撤销不必要的权限。这样做可以减少系统被攻击破坏的机会，这将在第 5 章中更详细地讨论。

如果系统的安全性受到破坏，采用加密可能有助于限制造成的损害。不同的加密方法将在第 6 章中讨论。

## 践行良好编码规范

良好的编码规范——我指的是诸如清晰注释、一致布局和良好命名变量等实践——应被视为任何软件开发方法的重要组成部分，而不仅仅与防御性编程相关。我选择在这里包含它，部分是因为我认为它如此重要以至于怎么重申都不为过，但也因为防御性编程的本质比其他方法更强调这些领域，原因如下：

正如前面所述，防御性编程的目标是尽量减少由于未来不可预见的事件而导致错误的风险。这些未来的事件可能包括未来对代码的维护和增强。通过现在创建清晰、文档完善的代码，你可以增强其未来的可理解性，降低下次修改时意外引入错误的可能性。

此外，由于防御性编程旨在创建健壮、有弹性的应用程序，这些应用程序可能会在没有任何人工干预的情况下运行很长时间。当它们在几年后再次被审查时，负责的开发团队可能已经大不相同，或者最初的开发人员可能不再记得为什么采取某种方法。至关重要的是，这些信息被记录并在代码本身中清晰可见，以免引入可能损害应用程序稳定性的错误或新的假设。

布局良好的代码通常与经过深思熟虑的代码密不可分。通过采取简单的步骤，例如缩进代码块，你可以轻松识别循环内的步骤和循环外的步骤，防止粗心的错误。大多数 IDE 和代码编辑器提供布局功能，可以自动应用一致的制表符、空格、大小写等格式，这些设置通常可以自定义以匹配特定组织中的任何编码标准。

布局良好、注释有意义的代码将使彻底的代码审查更容易。如果需要修改代码，将更容易快速确定最佳方法。

最后，如果在代码的某一部分中发现错误，在一个编码良好的函数中追踪它要容易得多，从而以最小的干扰解决它。

基于这些原因，我相信践行良好的编码规范是防御性编程的关键部分。在以下各节中，我将就编码规范的一些具体方面发表一些看法。

#### 注释

每个人都知道注释是任何代码的重要组成部分，但我们很少有人能像应该的那样充分注释我们的代码（一个常见的理由是开发者更喜欢编写代码，而不是写关于代码的注释）。几乎每一个关于最佳编码实践的思想流派都指出，你应该在代码中自由使用注释，防御性编程也不例外。编写良好的注释更容易说明一个函数的目标是什么以及它为什么以某种方式编写，这意味着更容易发现任何可能破坏该代码的错误或假设。

好的注释应该给阅读代码的人提供额外的信息——而不仅仅是指出显而易见的东西或重述可能很容易在在线书籍中找到的信息。例如，以下注释没有帮助：

```sql
-- 将 x 设置为 5
SET @x = 5;
```
