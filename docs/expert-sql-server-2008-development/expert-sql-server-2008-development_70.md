# 第六章 加密

加密所能提供的最基本保护或许是针对**静态数据**面临的风险。在 SQL Server 中，`静态数据`指的是存储在文件系统上的底层数据库文件，包括事务日志和备份。如果未经授权的第三方能够访问这些文件，他们就有可能将数据库复制并还原到自己的系统上，从而可以从容地浏览您的数据。然而，如果数据库中的敏感数据以加密格式存储，攻击者还需要知道相关的密钥或密码才能解密被盗数据，否则这些数据对他们毫无价值。

除了数据库文件的物理盗窃外，黑客和外部威胁也对数据库安全构成持续风险。黑客通常知识渊博，并配备了各种工具来探测和利用您安全系统中的任何弱点。通过确保所有通过网络传输的数据都经过安全加密，您可以确保只有预期的接收方才能理解这些数据，从而降低数据被成功拦截并用于对付您的可能性。

许多组织也认识到内部来源对数据安全构成的威胁。心怀不满的 DBA 和试图试探其授权极限的开发人员可能会访问其工作不需要的数据。如果有合适的价码，此类信息很容易流出组织，落入不法分子之手。如果设计和部署得当，加密策略可以防止此类风险发生。

请记住，加密不能替代其他安全措施，例如适当的身份验证和授权程序。但是，如果实施正确，它可以提供额外的安全级别，这可能是您的数据暴露还是按预期保密的关键区别。

## SQL Server 2008 加密密钥层次结构

与几乎所有民用密码学一样，SQL Server 中的所有加密方法都基于标准化的、公开可用的加密算法。在许多方面，这是一件好事：这些算法由世界上一些顶尖的信息理论家开发，并且通常已被证明能够经受住集中、蓄意的攻击。然而，这种公开知识的使用对任何处理机密数据的数据库应用程序设计提出了一个有趣的假设，即`敌人知道系统`。

香农格言（即上述陈述的普遍称呼）是对荷兰密码学家奥古斯特·克尔克霍夫斯早期工作的重述。克尔克霍夫斯提出了一些原则，如果遵循这些原则，就能确保军事密码在部分系统事实被泄露的情况下仍能保持安全。这些原则最初于 1883 年发表在法国期刊`le Journal des Sciences Militaires`上，在现代密码学中仍然非常适用。

所有 SQL Server 加密方法都使用基于密钥或从中生成密钥的密码的算法。如果我们假设香农格言成立（即所使用的任何加密算法的细节都是公开知识），那么您的加密数据的安全性就*完全*依赖于保护算法用于解密和加密数据的密钥。因此，必须为存储密钥创建一个安全环境，这必须被视为任何加密策略的首要任务。事实上，安全密钥管理和分发问题是现代密码学中最重要的领域之一，因为未能正确保护加密密钥会危及您的整个安全策略。在研究 SQL Server 中可用的不同类型加密之前，首先有必要考虑如何管理这些加密方法所基于的密钥。

### 自动密钥管理层次结构

SQL Server 解决安全密钥管理问题的方法是实现一个加密密钥层次结构，每个密钥为低于它的密钥提供保护。`自动密钥管理`层次结构如图 6-1 所示。

自动密钥管理层次结构的主要特点将在以下标题下描述。

#### 对称密钥、非对称密钥和证书

在最低级别，数据项使用`对称密钥`、`非对称密钥对`的公钥或`证书`进行加密。这些密钥和证书存储在创建它们的用户数据库中。这三种方法各有优缺点，将在本章后面讨论。

创建每种类型密钥的语法略有不同，但遵循相同的基本模式：列出密钥名称、要使用的加密算法、将拥有该密钥的数据库用户名以及保护密钥的方法。例如，以下代码清单演示了使用 Rivest、Shamir 和 Adleman (RSA) 算法创建由数据库用户 Tom 拥有的 1,024 位非对称密钥所需的语法：

```sql
CREATE ASYMMETRIC KEY ExampleAsymKey
AUTHORIZATION Tom
WITH ALGORITHM = RSA_1024;
```

在此示例中，没有显式指定保护方法，因此密钥将受自动密钥层次结构保护。默认情况下，这意味着密钥和证书受其所在数据库的`数据库主密钥` (DMK) 保护。

#### 数据库主密钥

服务器上的每个用户数据库都可以有自己的 DMK，这是一个对称密钥，存储在`master`数据库和用户数据库中。DMK 受两种不同形式的加密保护：一种是使用基于服务器的`服务主密钥` (SMK) 的三重 DES（数据加密标准）算法进行的加密，另一种是使用密码进行的加密。

即使同时受 SMK *和*密码保护，默认情况下，SQL Server 在需要时可以通过仅使用 SMK 解密来自动打开 DMK，而无需提供密码。这意味着受 DMK 保护的任何依赖私钥的用户不需要被授予打开 DMK 的显式权限。用户只需要对各个依赖密钥或证书拥有权限，因为 SQL Server 将在必要时打开 DMK 以访问这些密钥。这对于保护数据免受`sysadmins`和`db_owners`的访问具有一些影响，将在后面讨论。

可以通过在要存储密钥的数据库中运行`CREATE MASTER KEY`语句来创建 DMK，如下所示：

```sql
CREATE MASTER KEY ENCRYPTION BY PASSWORD = '5Tr()ng_p455woRD_4_dA_DMK!!!';
```

**注意：** 每个数据库只能关联一个 DMK。如果您遵循了第 5 章中的示例，您可能已经为了从证书创建代理数据库用户而创建了一个 DMK。

#### 服务主密钥

SMK 是位于 SQL Server 加密层次结构顶部的对称密钥，用于加密服务器上存储的所有 DMK，并保护与链接服务器关联的登录名和凭据。

SMK 受操作系统的`数据保护 API` (DPAPI) 保护，使用 SQL Server 的服务账户凭据和计算机凭据。如果这些凭据中的任何一个失效（例如，如果服务账户被更改），SQL Server 将基于剩余的有效密钥重新创建失效的密钥。这确保了 SMK 能够抵御除最严重故障之外的所有问题。没有创建 SMK 的命令；它在首次需要时自动创建并存储在`master`数据库中。


SMK 提供了 SQL Server 2008 中自动密钥层次结构所包含的所有加密密钥的顶层控制。该 SMK 保护的层次结构为大多数情况提供了足够的安全性，并使密钥访问相对容易：使用 `GRANT` 命令将层次结构中各个密钥或密钥子树的使用权授予个人或组，授权用户可以根据需要打开和使用这些密钥。

### 替代加密管理结构

有些情况下，你可能希望使用替代方法来增强或替换默认的自动密钥管理，以保护你的加密密钥。SQL Server 2008 灵活的加密密钥管理结构提供了几种实现方式，我将在接下来的内容中讨论。

### 对称密钥分层与轮换

数据库内的对称密钥本身可以被**分层**以提供额外的层次级别。

例如，一个对称密钥可以被另一个对称密钥加密，而后者又由受 DMK 保护的证书进行保护。以这种方式对密钥进行分层可以提供额外的控制和安全性，并且更容易实现**密钥轮换**，即可以更改中间层密钥，而无需解密和重新加密所有底层数据。

在仅基于单个加密密钥的扁平模型中进行密钥轮换是非常繁琐的。

要理解原因，假设你有大量的数据被单个对称密钥加密。现在假设，由于新的客户端要求，该加密密钥必须每月轮换一次。轮换将需要使用新密钥解密和重新加密所有数据，这可能需要大量的工作，并且可能要求数据在用新密钥重新加密之前至少暂时进入未加密状态。在分层模型中，你可以非常轻松地轮换更高层的密钥，因为它们包含的唯一加密数据是下一层加密使用的对称密钥。

并非所有密钥都需要同时轮换；轮换层次结构中的任何密钥都会重新加密所有依赖的密钥，这使得任何暴力攻击破解你的加密层次结构并获取敏感数据的难度显著增加。但是，请记住，尽管正确处理的密钥轮换通常会提高安全性，但密钥轮换确实涉及一定程度的固有风险。例如，存储在磁带备份上的存档加密数据，将其留在物理安全的位置可能比为了用新密钥重新加密数据而取回磁带更安全。

在创建额外的密钥层时，你应该尝试确保更高层密钥的加密强度与其所保护的密钥相同或更高。用较弱的密钥保护一个强密钥不会为系统增加任何额外的安全性。

### 从自动加密层次结构中移除密钥

当 DMK 同时受 SMK 和密码保护时（如默认层次结构中那样），你无需显式提供密码即可打开 DMK；当具有适当权限的用户请求受 DMK 保护的密钥时，SQL Server 会自动使用 SMK 打开 DMK 以访问所需的依赖密钥。此行为的一个有趣特点是，拥有数据库 `GRANT` 权限的系统管理员或 `db_owner` 角色中的 DBA 因此有权查看层次结构中的*所有*依赖密钥。

在你希望限制这些角色中的个人访问加密数据的情况下，必须删除对 DMK 的 SMK 保护，这将强制改为使用密码进行 DMK 保护。以下 T-SQL 语句在用户具有 `CONTROL` 权限的数据库中运行时，将删除与该数据库关联的 DMK 的 SMK 保护：

```
ALTER MASTER KEY DROP ENCRYPTION BY SERVICE MASTER KEY;
```

当仅受密码保护时，DMK 会从默认的自动密钥管理层次结构中移除，并且每次需要时都必须由具有适当密码的用户显式打开：

```
OPEN MASTER KEY DECRYPTION BY PASSWORD = '5Tr()ng_p455woRD_4_dA_DMK!!!';

-- Perform some action on properties dependent on the DMK

CLOSE MASTER KEY;
```

单独的对称密钥、非对称密钥和证书也可以通过密码进行保护，以防止它们在每次需要时未提供适当密码的情况下被打开。

将密钥从自动密钥层次结构中移除的问题在于，随后就需要有一个保护这些密钥密码的系统。除非*此*系统设计有充分的安全措施，否则使用密码保护密钥的加密数据并不会比在自动密钥管理下加密时更安全（甚至可能更不安全）。

### 可扩展密钥管理

SQL Server 2008 支持通过可扩展密钥管理（EKM）方法管理加密密钥。EKM 使用外部硬件安全模块（HSM）在 SQL Server 之外保存对称或非对称密钥，并将这些密钥的管理委托给 HSM。

以下代码清单说明了在 SQL Server 中创建映射到存储在 EKM 设备上的密钥的非对称所需的语法：

```
CREATE ASYMMETRIC KEY EKMAsymKey
FROM PROVIDER EKM_Provider
WITH
ALGORITHM = RSA_1024,
CREATION_DISPOSITION = OPEN_EXISTING,
PROVIDER_KEY_NAME = 'EKMKey1';
GO
```

EKM 使用专用硬件提供完整的加密管理解决方案，这意味着 SQL Server 可以将加密过程留给 HSM，而专注于其他任务。实现 EKM 涉及使用供应商特定的库来配置 SQL Server 和 HSM 单元之间的连接，本章不予涵盖。

## 数据保护与加密方法

SQL Server 2008 提供了一系列加密方法，这些方法提供不同级别的保护，同时具有不同的相关性能和架构影响。我在此不试图详细描述每种方法的全部细节；对每种方法的详细描述感兴趣的读者应查阅专门讨论该主题的书籍，例如 *Expert SQL Server 2008 Encryption*，作者 Michael Coles（Apress，2009）。不过，我将简要概述以下方法：

- 哈希
- 对称密钥加密
- 非对称密钥加密
- 透明数据加密

### 哈希

哈希函数允许你对任何数据执行**单向**加密，并产生确定性的结果。
