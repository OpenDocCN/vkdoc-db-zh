# 软件开发方法论与数据库世界

20 世纪 90 年代末期，架构趋势发生了又一次范式转变——奇怪的是，它又回到了大型机和哑终端的世界。Web 服务器取代了大型机系统，成为集中的数据和 UI 系统，而浏览器则扮演了先前由终端承担的角色。本质上，这使应用程序架构经历了一个轮回，但有一个关键区别：现代基于 Web 的数据中心的特征是“**商用服务器**”**集群——廉价、标准化且易于更换的硬件，而不是单一的大型机。

云计算这一最新趋势，似乎将对传统的架构设计决策观点构成另一项严峻挑战。在基于云的模型中，应用程序使用共享的、虚拟化的服务器资源，这些资源通常由第三方通过互联网以服务的形式提供。

亚马逊、谷歌和微软等供应商已经提供了基于云的数据库服务，但在撰写本文时，这些服务仍处于非常初级的阶段。例如，当前的 SQL Server 数据服务的实现，在带宽和存储方面有严重限制，这意味着在大多数情况下，它无法替代专用数据中心。然而，向云端迁移的动力日益增强，未来几年它将如何影响数据架构决策，值得关注。

在思考这些问题时，需要记住的一个重要点是：单个数据库可能被多个应用程序共享，这些应用程序又可能暴露多个用户界面，如图 1-1 所示。

数据库开发人员必须努力确保数据被充分封装，以允许其在多个应用程序之间共享，同时确保不同应用程序的逻辑不会冲突，从而使整个数据库陷入不一致的状态。达到这种程度的封装需要仔细划分逻辑，尤其是数据验证规则。

规则和逻辑可以划分为三个基本组：

*   数据逻辑
*   业务逻辑
*   应用逻辑

设计应用程序时，重要的是理解这些划分，并考虑在应用层级中，任何给定的逻辑片段应放置在何处，以确保其可重用性。

### 数据逻辑

`Data logic`定义了数据库中的数据必须满足的条件，以保持一致、未损坏的状态。数据库开发人员无疑熟悉以主键和外键约束、检查约束、触发器等形式实现这些规则。数据规则并不规定*如何*操作数据或*何时*操作数据；相反，数据规则规定的是任何过程完成后数据必须处于的状态。

重要的是要记住，在大多数应用程序中，数据不仅仅是“数据”——相反，数据库中的数据是对实际业务的建模。因此，数据规则必须反映驱动业务本身的所有规则。例如，如果你正在设计一个支持银行应用程序的数据库，可能会遇到一项业务规则，规定某些类型的账户不允许透支。

为了对当前应用程序和所有可能的未来应用程序正确实施此规则，必须在数据本身的层面集中实施。如果数据保证是一致的，那么应用程序只需担心对数据*做什么*。

作为一般准则，你应尝试实施尽可能多的数据规则，以避免出现数据质量问题的可能性。数据库是数据的持有者，因此应充当数据是否有资格被持久化的最终仲裁者。任何对业务至关重要的验证规则，对数据也同样至关重要，反之亦然。在我参与众多数据库支持的应用程序的工作过程中，我从未见过数据规则过多的数据库；但我经常看到数据库因缺乏足够的规则而导致数据完整性问题。

#### 数据规则到底属于哪里？

许多面向对象的狂热者会争辩说，正确的解决方案根本不是数据库，而是一个接口总线，它充当数据库之上的外观，并控制所有进出数据库的通信。虽然这种方法在理论上可行，但存在几个问题。首先，这种方法完全忽略了数据库强制数据完整性的概念，并将数据库层变成了一个纯粹的存储容器，未能利用几乎所有现代数据库为此目的而内置的任何特性。此外，这样的接口层仍然必须与数据库通信，因此无论如何都必须在某个层面编写数据库代码。编写这样的接口层可能会消除一些数据库代码，但这只是推迟了与数据库打交道的必要性。最后，以我相当主观的观点来看，在许多情况下，应用层不如数据库稳定或持久。虽然应用程序和应用架构来来去去，但数据库似乎在企业中拥有极长的生命周期。同样的规则也适用于一个无所不包的接口总线。所有这些问题可能是一个重要原因，尽管我多年来听到架构师们争论这个问题，但我从未见过这样的系统被实现。

### 业务逻辑

`Business logic`这个术语在软件开发圈中通常被用作一个模糊的统称，指代应用程序所做的任何与 UI 无关、且涉及至少一个条件分支的事情。换句话说，这个术语被过度使用，没有真正的含义。

幸运的是，软件开发是一个不断变化的领域，我们不必坚持这种公认的缺乏定义的状态。就本文而言，`Business logic`被定义为任何规定如何或何时操作数据以改变其状态，但不规定如何持久化或验证数据的规则或过程。这方面的一个例子是将原始数据渲染成适合最终用户的报告所需的逻辑。我们假设已经应用了数据逻辑规则的原始数据，可以通过`Business logic`来确定适当的聚合和分析，以回答最终用户可能提出的问题。如果这些数据需要以其新形式持久化在数据库中，则必须再次应用数据规则；请记住，数据库应始终对任何给定数据是否允许拥有最终决定权。

那么，业务逻辑属于数据库吗？答案是明确的“可能”。作为一名数据库开发人员，你的主要关注点往往围绕着数据完整性和性能。抛开其他因素（如整体应用架构）不谈，这意味着在一般实践中，你应该尝试将`Business logic`放在能提供最佳性能的层中，或者放在最易于重用的层中。例如，如果许多应用程序共享相同的数据，并且每个应用程序都有类似的报告需求，那么设计存储过程来将数据渲染成正确的报告格式，可能比在每个应用程序中实现类似的报告更有意义。
