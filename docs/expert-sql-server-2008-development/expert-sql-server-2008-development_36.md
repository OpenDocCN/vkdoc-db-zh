# 数据库编程最佳实践

在许多情况下，防御性编程能够提升用户体验。通过主动预见并为异常情况做好准备，错误可以在其发生*之前*就被捕获，而无需事后处理。异常情况可以被隔离并处理，从而对用户体验产生最小的负面影响，避免引发整个系统故障。即使遇到极端的意外状况，系统仍然可以**优雅降级**，并按照既定文档描述的行为运作。

然而，如同任何思想流派一样，防御性编程也有其反对者。以下列举了对防御性编码常见的一些批评。针对每一点，我都尝试给出合理的回应。

## *防御性编码开发耗时更长*

相较于遵循其他软件实践开发的应用程序，遵循防御性方法论确实可能导致前期开发时间更长。防御性编程非常强调初始的需求收集和架构设计阶段，这些阶段可能比某些方法学更长、更复杂。编码本身也需要更长时间，因为可能需要添加额外的代码路径来处理检查并断言假设。

代码必须经过既具挑战性又彻底的广泛审查，然后还必须经过严格的测试。所有这些因素都导致防御性软件的总体开发和发布周期比其他方法更长。

防御性编程与所谓的`敏捷`开发实践之间存在尤为鲜明的对比，后者专注于在非常快速的开发和发布周期上发布频繁的迭代变更。然而，这并不一定意味着，在考虑应用程序的整个生命周期时，防御性编码的开发时间更长。在开发初期投入的额外细致和谨慎，通常会在项目的生命周期中得到回报，因为项目一旦上线，对代码修复部署的需求就会减少。

## *编写能够预见和处理所有可能场景的代码会使防御性应用程序变得臃肿。*

`代码膨胀`意味着应用程序包含了不必要的、低效的或浪费的代码。防御性代码防范的是*不太可能*发生但绝非不可能发生的事件。提前采取行动明确测试和处理异常情况，可以节省未来可能花费的大量追踪和调试时间。防御性应用程序可能包含比其他应用程序更多的总代码行数，但所有这些代码都应设计良好、目的明确。请注意，“防御性编程”这个标签有时会被误用：不加考虑或论证地在任何机会下添加不必要的检查，这*并非*防御性编程。这样的行为会导致代码既复杂又僵化。请记住，真正的防御性编程提倡简洁性、模块化和代码重用，这实际上有助于减少代码膨胀。

## *防御性编程隐藏了错误，导致它们得不到修复，而不是使其可见。*

这或许是对防御性实践最常见的误解，源于未能理解防御性应用程序中对待错误的基本态度。通过明确识别和检查异常场景，防御性编程实际上采取了一种非常主动的方式来识别错误。然而，当遇到可能导致异常情况的条件时，防御性应用程序被设计为优雅地失效——也就是说，在开发阶段，识别出可能导致异常的潜在场景，并创建相应的代码路径。
