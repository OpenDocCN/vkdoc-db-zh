# 第一部分 等待统计分析基础


## 1. 等待统计内部机制

SQL Server 等待统计是分析性能相关问题、优化 SQL Server 性能的重要工具。然而，许多数据库管理员或开发人员对其并不十分了解。我认为这与其相对复杂的性质、数量庞大的不同类型，以及许多等待统计缺乏文档说明有关。等待统计还直接与其发生的特定 SQL Server 实例绑定，这意味着即使硬件和数据库配置完全相同，也无法将服务器 A 的等待统计与服务器 B 的等待统计进行比较。从硬件固件级别到客户端计算机上 SQL Server Native Client 的配置，每一个配置选项都会影响等待统计！

因此，本书将首先回顾与等待统计相关的 SQL Server 内部机制。我们将讨论等待事件如何生成、如何查看等待事件，以及如何将其作为正确性能调优方法的一部分进行分析。然后在本书的第二部分，我们将探索和研究具体的等待事件。

本章将回顾等待统计在不同 SQL Server 版本中的历史。接下来，我们将审视 SQL 操作系统，即 SQLOS。SQLOS 的架构与等待统计以及整体性能故障排除密切相关。本章的其余部分将致力于讨论等待统计最重要的一个方面：**线程调度**。

在开始之前，我想提一下与讨论等待统计时所用术语相关的几点。在本书的介绍和前面的段落中，我提到了“等待统计”这个术语。“比较服务器 A 的等待统计与服务器 B 的等待统计”这句话在技术上是错误的，因为我们实际上是比较我们正在等待的资源的*特定等待类型*的*等待时间*（我们等待资源所花费的总时间）。从现在开始，当我使用“等待统计”这个术语时，我指的是等待统计的概念，我将在适当的地方使用正确的术语“等待时间”和“等待类型”。

### 等待统计简史

SQL Server 已经存在相当长一段时间了。SQL Server 的第一个版本可以追溯到 1989 年，当时是为 OS/2 平台发布的。直到 1995 年发布的 SQL Server 6.0，Microsoft 一直与 Sybase 合作开发 SQL Server。然而，在 1995 年，Microsoft 和 Sybase 分道扬镳。两家公司都继续活跃在数据库领域（SAP 于 2010 年收购了 Sybase）。2022 年，Microsoft 发布了 SQL Server 2022，而 SAP 则在 2014 年发布了 SAP Sybase ASE 16（但至今仍在维护），两者都是关系型企业级数据库系统。

在 SQL Server 6.0 到 SQL Server 2022 之间，许多事情发生了变化，很难将任何两个版本进行直接比较。但多年来唯一不变的是等待统计的概念。每个版本的 SQL Server 都会存储其内部进程的信息。尽管我们访问此类信息的方式多年来发生了变化，但等待统计仍然是内部日志记录过程的重要组成部分。

### SQLOS

计算机硬件世界日新月异。几乎每年，甚至每月，我们都会在处理器中集成更多核心，提升主板的内存容量，或引入如基于 PCI 的持久化闪存存储等全新的硬件概念。数据库管理系统（DBMSs）渴望利用新的硬件趋势。由于硬件变化迅速且需要在新硬件选项可用时立即加以利用，SQL Server 团队决定从 SQL Server 2005 开始改变 SQL Server 的平台层。

在 SQL Server 2005 之前，SQL Server 的平台层受到限制，许多操作由操作系统自身执行。这意味着 SQL Server 难以跟上快速变化的服务器硬件世界，因为为了利用更快的硬件或新的硬件特性而改变一个完整的操作系统，是一项耗时且复杂的操作。

SQLOS 通过引入调度器（schedulers）、任务（tasks）和工作线程（worker threads），改变了 SQL Server 访问处理器资源的方式。这使得 SQLOS 能更好地控制应如何由处理器完成工作。Windows 操作系统使用抢占式调度（preemptive scheduling）方法。这意味着 Windows 会给每个需要处理器时间的进程分配一个优先级和固定的时间片（或称量程 quantum）。这个进程优先级是根据许多变量计算得出的，如资源使用情况、预期运行时间、当前活动等。通过使用抢占式调度，Windows 操作系统可以选择在更高优先级的进程需要处理器时间时中断当前进程。这种调度方式可能对 SQL Server 生成的进程产生负面影响，因为这些进程很容易被更高优先级的进程（包括其他应用程序的进程）中断。因此，SQLOS 使用自己的（协作式 cooperative）非抢占式调度机制，确保 Windows 进程无法中断 SQLOS 进程。

**Nerd Note**
SQL Server 7 和 SQL Server 2000 也使用用户模式调度（UMS）进行非抢占式调度。SQLOS 将更多的系统组件更紧密地结合在一起，从而实现了更好的性能和可扩展性。

存在一些 SQLOS 无法使用非抢占式调度的例外情况，例如当 SQLOS 需要通过 Windows 操作系统访问资源时。我们将在本书后面的第 11 章“抢占式等待类型”中讨论这些例外情况。

#### 会话

会话（session）是客户端成功通过身份验证后与 SQL Server 建立的连接。我们可以通过查询 `sys.dm_exec_sessions` DMV 来访问会话信息，使用以下查询：

```sql
SELECT *
FROM sys.dm_exec_sessions;
```

在旧版本的 SQL Server 中，用户会话的 `session_id` 大于 50；小于 50 的保留给内部 SQL Server 进程。然而，在非常繁忙的服务器上，SQL Server 有可能需要使用大于 50 的 `session_id`。如果您只对用户启动的会话信息感兴趣，最好使用 `is_user_process` 列来过滤 `sys.dm_exec_sessions` DMV 的结果，而不是过滤 `session_id` 大于 50。以下查询将只返回用户会话，并过滤掉内部系统会话：

```sql
SELECT *
FROM sys.dm_exec_sessions
WHERE is_user_process = 1;
```

#### 工作线程

创建的每个任务都会分配一个工作线程，然后工作线程执行任务请求的操作。

极客备注

工作线程本身并不执行工作；而是向 Windows 操作系统请求一个线程来执行工作。为了简单起见，并且由于 Windows 线程在 SQLOS 之外运行，我已将此步骤从图 1-5 中省略了。如果您感兴趣，可以通过查询 `sys.dm_os_threads` 来访问有关 Windows 操作系统线程的信息。

当任务请求工作线程时，SQL Server 将首先查找一个空闲的工作线程分配给该任务。如果未找到空闲工作线程，并且工作线程数已达到最大值，则该请求将进入队列，直到某个工作线程完成其当前工作并变得可用。

SQL Server 可用于处理请求的工作线程数量是有限制的。此数量在启动期间由 SQL Server 自动计算和配置。我们可以使用以下公式自己计算最大工作线程数：

| 32 位系统且逻辑处理器数 <= 4 | 256 个工作线程 |
| --- | --- |
| 32 位系统且逻辑处理器数 > 4 | 256 + ((逻辑处理器数 – 4) * 8) |
| 64 位系统且逻辑处理器数 <= 4 | 512 个工作线程 |
| 64 位系统且逻辑处理器数 > 4 | 512 + ((逻辑处理器数 – 4) * 16) |

**示例**：对于一个有 16 个逻辑处理器的 64 位系统，我们使用前面的公式计算最大工作线程数：

```
512 + ((# of logical processors – 4) * 16) =
512 + ((16 – 4) * 16) =
512 + (12 * 16) =
512 + (192) = 704
```

这将分配最多 704 个工作线程。

工作线程数的默认值为 0，允许 SQL Server 在启动时使用前面的公式设置最大工作线程数。我们可以通过更改 SQL Server 属性中的 `最大工作线程数` 选项来修改默认值，如图 1-9 所示。

![](img/340881_3_En_1_Fig9_HTML.png)

`sys.dm_os_workers` DMV 大而复杂，许多列被 Microsoft 标记为“仅限内部使用”。然而，`task_address` 和 `scheduler_address` 这两列可用于连接我们迄今为止讨论的不同 DMV。

工作线程在暴露于处理器时会经历不同的阶段，我们在检查 `sys.dm_os_workers` DMV 中的 `state` 列时可以观察到：

*   `INIT` – 工作线程已由 SQLOS 初始化。
*   `RUNNING` – 工作线程当前正在处理器上执行工作。

#### 调度器

调度器组件的主要任务是——毫不意外地——在物理处理器上以任务的形式安排工作。当一个任务请求处理器时间时，正是调度器将工作线程分配给该任务，以便请求得到处理。调度器还负责确保工作线程相互协作，并在它们的时间片（或量子）到期时让出处理器。我们称之为协作式或非抢占式调度。工作线程需要在处理器时间到期时让出的原因是，调度器一次只允许一个工作线程在处理器上运行。如果工作线程无需让出，一个工作线程可能会在处理器上停留无限长的时间，从而阻塞该处理器的所有使用。

处理器和调度器之间是一一对应的关系。如果你的系统有两个处理器，每个处理器有四个核心，那么将会有八个调度器可供 SQLOS 用于处理用户请求，每个调度器映射到一个逻辑处理器。

我们可以通过查询 **sys.dm_os_schedulers** DMV 来获取有关调度器的信息：

```sql
SELECT *
FROM sys.dm_os_schedulers;
```

查询结果如图 1-11 所示。

![](img/340881_3_En_1_Fig11_HTML.png)

我们可以通过查看 `current_workers_count` 列来查看调度器关联的工作线程数。此数字还包括未执行任何工作的工作线程。`active_workers_count` 向我们显示特定调度器上处于活动状态的工作线程。这并不意味着它们正在处理器上运行，因为状态为“RUNNING”、“RUNNABLE”和“SUSPENDED”的工作线程也计入此数字。`work_queue_count` 也是一个有趣的列，因为它提供了有多少任务正在等待空闲工作线程的洞察。如果你在此列中看到很高的数字，可能意味着你遇到了 CPU 压力。

#### 综合运用

上面讨论的所有 SQL Server 调度部件都是相互连接的，因为每个连接、会话和查询请求都通过这些相同的组件进行处理。以下文本是一个查询请求如何被处理的示例：

*   用户通过应用程序连接到 SQL Server。
*   ↓
*   登录过程成功完成后，SQL Server 创建一个会话。
*   ↓
*   当用户向 SQL Server 发送查询时，会创建一个任务和一个请求来表示需要完成的工作单元。
*   ↓
*   调度器将分配工作线程给任务，以便任务能够完成。

要在 SQL Server 中查看此信息，我们需要连接一些讨论过的 DMV。清单 1-1 中的查询将向你展示如何组合不同的 DMV 来获取特定会话（本例中 ID 为 55 的会话）的调度信息。

```sql
SELECT
    r.session_id AS 'Session ID',
    r.command AS 'Type of Request',
    qt.[text] AS 'Query Text',
    t.task_address AS 'Task Address',
    t.task_state AS 'Task State',
    w.worker_address AS 'Worker Address',
    w.[state] AS 'Worker State',
    s.scheduler_address AS 'Scheduler Address',
    s.[status] AS 'Scheduler State'
FROM sys.dm_exec_requests r
CROSS APPLY sys.dm_exec_sql_text(r.sql_handle) qt
INNER JOIN sys.dm_os_tasks t
    ON r.task_address = t.task_address
INNER JOIN sys.dm_os_workers w
    ON t.worker_address = w.worker_address
INNER JOIN sys.dm_os_schedulers s
    ON w.scheduler_address = s.scheduler_address
WHERE r.session_id = 70;
```

## 等待统计

到目前为止，我们已经深入探讨了为 SQL Server 执行调度的不同组件以及它们是如何相互关联的。现在，我们将注意力转向本书的主要主题：等待统计。

在本章前面关于工作线程的部分，我描述了工作线程在调度程序上执行工作时所处的状态。当一个工作线程执行其工作时，它会在调度程序进程中经历三个不同的阶段（或队列）。根据工作线程所在的阶段（或队列），它会进入“运行中”、“可运行”或“已暂停”状态。图 1-13 展示了一个调度程序及其不同阶段的抽象视图。

![](img/340881_3_En_1_Fig13_HTML.jpg)

调度程序队列阶段的流程图有五个块：初始化、可运行、运行中、完成和等待中。

图 1-13

调度程序及其阶段和队列

当一个工作线程获得访问调度程序的权限时，它通常会在等待者列表中启动，并被分配为“已暂停”状态。等待者列表（在图 1-13 中显示为 Waiting）是一个无序列表，包含所有处于“已暂停”状态并等待资源可用的工作线程。这里的资源是指系统上的任何东西，例如数据页或锁请求。当一个工作线程在等待者列表中时，`SQLOS` 会记录它继续工作所需的资源类型（这就是等待类型），以及该特定资源变为可用之前它所花费的等待时间（这就是资源等待时间）。

当一个工作线程获得所需资源的访问权限时，它将移动到可运行队列。这是一个先进先出的列表，包含所有已获得资源访问权限并准备好在处理器上运行的工作线程。工作线程在可运行队列中花费的时间被记录为信号等待时间。

可运行队列顶部的工作线程将移动到“运行中”阶段，在此阶段它将获得处理器时间以执行其工作。它在处理器上花费的时间被记录为 CPU 时间。与此同时，可运行队列中的其他工作线程会向上移动，而已获得所需资源的工作线程则从等待者列表移入可运行队列。

极客笔记

可运行队列是一个先进先出的列表。如果不是这样，在繁忙的系统中，理论上一个任务可能永远无法从可运行状态转为运行中状态。然而，等待者列表不是先进先出的。

当工作线程处于“运行中”阶段时，可能会发生三种情况：

1.  工作线程需要额外的资源；在这种情况下，它从“运行中”阶段移动到等待者列表（即我们看到的“已暂停”状态）。
2.  工作线程耗尽了其时间配额（固定值为 4 毫秒），因此必须让出处理器；该工作线程被移动到可运行队列的底部。
3.  工作线程完成了其工作并离开调度程序。

工作线程总是在三个不同的阶段中移动。一个工作线程循环多次直到所有工作完成是很常见的。

了解请求在三个不同阶段之一所花费的不同时长，可以计算总的请求执行时间，以及请求为等待处理器时间或资源时间所花费的总时间。

用数学公式表示，总的请求执行时间如下所示：
`总请求执行时间 = 运行时间 + 可运行时间 + 暂停时间`

也可以写成如下形式：
`总请求执行时间 = CPU 时间 + 信号等待时间 + 资源等待时间`

## 2. 查询 SQL Server 等待统计信息

### 摘要

在本章中，我们回顾了不同版本 SQL Server 中等待统计信息的历史。随着 SQL Server 2005 中 SQLOS 的引入，SQL Server 处理请求的方式发生了很大变化，引入了调度器、工作线程和任务。所有这些不同部分的信息都存储在动态管理视图（DMV）或动态管理函数（DMF）中，它们易于查询，并返回大量关于 SQL Server 内部运行的信息。

通过使用这些 DMV，我们可以查看请求在 SQL Server 调度器处理过程中的进度，并了解它们是否在等待任何特定资源。请求所等待的资源以及它们等待这些资源所花费的时间被记录为等待统计信息，这也是本书的主要主题。

随着 SQL Server 2005 中动态管理视图（DMV）和动态管理函数（DMF）的引入，查看和分析等待统计信息变得更加容易和省力。在 SQL Server 2005 之前的版本中，用户只能通过 `DBCC SQLPERF('WAITSSTATS')` 命令来查看等待统计信息。如今，有多种 DMV 可以返回与等待统计信息相关的信息，在本章中，我们将详细查看四个最有用的 DMV：`sys.dm_os_wait_stats`、`sys.dm_os_waiting_tasks`、`sys.dm_exec_requests` 和 `sys.dm_exec_session_wait_stats`。

查看等待统计信息不仅限于 DMV。我们还可以使用性能监视器（Performance Monitor，或 Perfmon）来查看等待统计信息。

SQL Server 2008 引入了另一个查看等待统计信息的选项：扩展事件（Extended Events）。在 SQL Server 2008 中，扩展事件使用起来很困难（您必须用 T-SQL 编写整个扩展事件会话），但 Microsoft 在后续版本中极大地改进了扩展事件，使其更加用户友好且易于使用。

SQL Server 2016 SP1 引入了两种访问等待统计信息的新方法：通过一个名为 `sys.dm_exec_session_wait_stats` 的新 DMV，以及在执行计划中按查询为基础添加等待统计信息。

在 SQL Server 2017 中，Microsoft 通过将等待统计信息包含在查询存储（Query Store）中，进一步推进了记录等待统计信息的功能。查询存储是 SQL Server 2016 中引入的一项功能，它就像您查询工作负载的飞行记录器，记录查询语句、性能和资源利用情况。

在本章中，我们将查看上文提到的捕获等待统计信息的来源，从 DMV 开始。由于查询存储功能对故障排除和分析查询性能（包括等待统计信息）有重大影响，我们将在第 3 章“查询存储”中对其进行深入探讨。

#### sys.dm_os_wait_stats

`sys.dm_os_wait_stats` DMV 是关于等待统计信息最重要的 DMV 之一。这个 DMV 是 SQL Server 2005 之前使用的 `DBCC SQLPERF('WAITSTATS')` 命令的替代品。`DBCC SQLPERF('WAITSTATS')` 命令返回的所有信息都包含在 `sys.dm_os_wait_stats` DMV 中，甚至还多一些。

`sys.dm_os_wait_stats` DMV 显示自 SQL Server 实例启动以来，每个等待类型的总等待时间。它是累积性的，将等待时间添加到不同的等待类型中，导致总数不断增加。查询 `sys.dm_os_wait_stats` DMV 可以深入了解查询请求在等待什么。如果您想查看每个等待类型的总等待时间，这很有帮助，但很多时候您可能对特定时间段或查询的等待时间感兴趣。在这种情况下，可以使用以下命令重置 `sys.dm_os_wait_stats` DMV，而无需重启 SQL Server：

```
DBCC SQLPERF('sys.dm_os_wait_stats', CLEAR);
```

这将把等待统计信息重置回 0，这意味着您会丢失重置之前的所有信息。在第 4 章“建立可靠的基线”中，我们将研究一种不会完全重置 `sys.dm_os_wait_stats` 计数器的方法。

与 SQL Server 中的每个 DMV 一样，我们像查询表一样对 `sys.dm_os_wait_stats` DMV 执行查询，例如：

```
SELECT *
FROM sys.dm_os_wait_stats;
```

此查询的结果如图 2-1 所示。

![](img/340881_3_En_2_Fig1_HTML.png)

以下是 `sys.dm_os_wait_stats` DMV 中可用的列，以及每列所代表含义的描述：

*   `wait_type` – 等待类型的名称。`sys.dm_os_wait_stats` 会为特定 SQL Server 安装中可能的每个等待类型返回一行。
*   `waiting_tasks_count` – 工作线程必须等待该特定等待类型的次数计数。
*   `wait_time_ms` – 自 SQL Server 实例启动或上次手动重置以来，该特定等待类型的总等待时间（以毫秒为单位）。这是工作线程在“SUSPENDED”状态（挂起状态）下处于等待者列表（Waiter List）中所花费的时间。它还包括工作线程在“RUNNABLE”状态（可运行状态）下处于可运行队列（Runnable Queue）中，等待调度器授予其处理器时间所花费的时间。
*   `max_wait_time_ms` – 工作线程在该特定等待类型上等待的最长时间（以毫秒为单位）。
*   `signal_wait_time_ms` – 工作线程在可运行队列中等待可用处理器所花费的时间（以毫秒为单位）。

**极客注**

在处理大量查询、所有查询都请求处理器时间的系统中，信号等待时间（Signal wait time）是不可避免且正常的。信号等待时间也是检测 CPU 压力的一个重要指标。一般来说，根据您系统的硬件情况，看到信号等待时间指标超过总等待时间的 **15%** **可能**表明存在 CPU 压力，因为工作线程正在等待处理器变为可用状态。

您可能已经注意到 `sys.dm_os_wait_stats` DMV 没有返回资源等待时间（resource wait time）的列。如果我们想将资源等待时间作为附加列显示，我们需要自己计算这个值。

以下是`sys.dm_os_waiting_tasks` DMV 返回的列列表及其返回信息的描述：

*   `waiting_task_address` – 当前等待的任务地址。
*   `session_id` – 与特定任务关联的会话 ID。
*   `exec_context_id` – 执行上下文的 ID。只有当任务使用并行性（任务使用多个线程而不是单个线程）执行时，此值才会从默认值 0 更改。

*   `blocking_task_address` – 当前阻塞等待任务的任务地址。当任务未被其他任务阻塞时，此列将返回`NULL`。
*   `blocking_session_id` – 当前阻塞任务的会话 ID。与`blocking_task_address`一样，仅当此任务当前被另一个任务阻塞时才包含此信息。当没有阻塞或无法检索或识别阻塞任务的会话信息时，它将返回`NULL`。我们将在第 8 章“锁相关等待类型”讨论锁等待类型时解释阻塞和锁定。
*   `blocking_exec_context_id` – 另一个专门用于阻塞信息的列。在这种情况下，它将返回执行上下文的 ID。仅当使用并行性执行任务且其中一个线程负责阻塞时，才会返回非`NULL`的结果。然后，`blocking_exec_context_id`可用于识别哪个线程负责阻塞。
*   `resource_description` – 有关任务正在等待的资源的附加信息。此列的等待类型不多——最常见的是并行性以及锁或闩锁相关的等待类型。在分析锁或闩锁相关的等待类型时，这是一个非常有用的列；在这些情况下，我们可以精确指出我们正在等待其可用性的数据库对象（数据页、行、表等）。本书后面的一些示例（最著名的是第 8 章“锁相关等待类型”和第 9 章“闩锁相关等待类型”）将使用此列来收集有关我们正在等待的资源的额外信息。


##### 查询 `sys.dm_os_waiting_tasks`

由于 `sys.dm_os_waiting_tasks` DMV 返回了丰富的信息，根据您想要分析或排查的具体问题，有多种不同的使用方式。

##### 理解 `sys.dm_exec_requests`

与之前的 DMV 类似，我们按如下方式查询 `sys.dm_exec_requests`：

```sql
SELECT *

```

这将返回当前正在执行的每一个查询请求。图 2-6 显示了我的测试 SQL Server 上的部分结果。

![](img/340881_3_En_2_Fig6_HTML.png)

`sys.dm_exec_requests` 查询的输出截图。表格输出有 8 列和 13 行。最后 2 列中的 null 输出被突出显示。

图 2-6
`sys.dm_exec_requests`

`sys.dm_exec_requests` DMV 返回的列比前面讨论的 `sys.dm_os_wait_stats` 或 `sys.dm_os_waiting_tasks` DMV 更多。为了保持可读性，我将只描述常用于等待统计信息分析的列。以下是我的列表：

*   `session_id` – 此请求所属会话的 ID。
*   `start_time` – 请求到达的日期和时间。对于长时间运行的查询，这可能与您查询 DMV 的时间不同。
*   `command` – 请求正在执行的操作。最常见的命令与查询相关，如 `SELECT`、`INSERT`、`UPDATE` 和 `DELETE`，但根据请求的不同，还有更多命令。
*   `sql_handle` – 在请求中执行的批处理或存储过程的哈希值。并非所有请求都有句柄，通常只有当请求是由用户会话发起并且涉及查询时，您才会看到句柄。`sql_handle` 哈希被用作 `sys.dm_exec_sql_text` 动态管理函数 (DMF) 的输入，以检索查询文本。
*   `plan_handle` – 执行计划的哈希值。执行计划将向您展示 SQL Server 在执行查询时执行的操作，是查询执行信息的绝佳来源。我们使用 `plan_handle` 的方式与 `sql_handle` 相同，作为 `sys.dm_exec_query_plan` DMF 的输入，以返回查询的执行计划。
*   `wait_type` – 如果请求处于“SUSPENDED”或“RUNNABLE”状态，则为当前的等待类型。当请求当前正在运行（即未等待）时，此值将为 `NULL`。
*   `last_wait_type` – 如果请求在执行期间等待过，则为它遇到的上一个等待类型。
*   `total_elapsed_time` – 自请求到达以来的总时间（毫秒）。

此 DMV 中还有更多可用的列，并且都有各自的用途。完整的描述可在 Microsoft MSDN 页面 [`https://slrwnds.com/f6glon`](https://slrwnds.com/f6glon) 上找到，我鼓励您阅读该文章。`sys.dm_exec_requests` DMV 是您 DBA 工具包中的一个很棒的工具，也是除了分析等待统计信息之外，您将经常用于各种目的的 DMV 之一。



##### 查询 `sys.dm_exec_requests`

`sys.dm_exec_requests` 动态管理视图通过返回 `sql` 和 `plan` 句柄，提供了访问查询语句及其对应执行计划的途径。如果你对此信息感兴趣——大多数情况下你确实会感兴趣——必须将 `sql_handle` 和 `plan_handle` 传递给各自的动态管理函数，以便将哈希值转换为人类可以阅读和理解的形式。

`清单 2-2` 展示了一个针对 `sys.dm_exec_requests` 动态管理视图的查询，该查询同时检索查询语句和执行计划。为了保持此示例的结果集较小，我排除了当前的会话 ID，并忽略了小于 50 的会话 ID。

```sql
SELECT
r.session_id AS 'Session ID',
r.start_time AS 'Request Start',
r.[status] AS 'Current State',
r.[command] AS 'Request Command',
t.[text] AS 'Query',
p.query_plan AS 'Execution Plan'
FROM sys.dm_exec_requests r
OUTER APPLY sys.dm_exec_sql_text(r.sql_handle) AS t
OUTER APPLY sys.dm_exec_query_plan(r.plan_handle) p
WHERE r.session_id > 50
AND r.session_id <> @@SPID;
```

#### `sys.dm_exec_session_wait_stats`

与等待统计信息相关的动态管理视图中，较新的成员之一是 `sys.dm_exec_session_wait_stats` 动态管理视图。它在 SQL Server 2016 SP1 中引入，返回**每个会话级别**的等待统计信息。如果你还记得`第 1 章`“等待统计信息内部原理”中的内容，会话是用户或进程与 SQL Server 之间的一个活动连接。一个会话可以有多个请求，而这些请求又可以有多个任务来执行查询所需的操作。

`图 2-9` 展示了该动态管理视图的列，以及来自我测试系统的一些等待统计信息。

![](img/340881_3_En_2_Fig9_HTML.png)

`sys.dm_exec_session_wait_stats` 查询输出的截图。表格输出有 7 列和 14 行。它列出了会话 ID、等待类型、任务和计时。

`图 2-9` `sys.dm_exec_session_wait_stats`

前面的图看起来眼熟吗？可能确实如此，因为它实际上与 `sys.dm_os_wait_stats` 动态管理视图几乎完全相同，只是多了一个 `session_id` 列。

但不仅仅多了一个列。通过 `sys.dm_exec_session_wait_stats` 记录的等待统计信息是**某个特定会话在处于活动状态期间所执行的所有请求的累计值**。例如，如果你在一个会话内执行了十个不同的查询，该动态管理视图返回的等待时间是这十个查询的总等待时间。这意味着理解会话生命周期内发生了什么非常重要。也许该会话执行了大批量的查询，或者只执行了一个查询语句。在使用此动态管理视图分析会话等待统计信息之前，了解其工作负载是很重要的。

此外，会话 ID 在会话关闭后会被重用，这意味着你会看到新会话和旧会话混合在一起的等待统计信息。当会话 ID 被重用（或在连接池中被重置）时，该动态管理视图中特定会话的等待统计信息也会被重置。

考虑到以上信息，我们得出结论：当报告性能缓慢时，此动态管理视图并不能直接作为“首先查看的地方”。不过，它仍然有其用武之地，特别是当你能用涉及多个查询的特定操作重现某个特定的性能问题时。在这种情况下，你可以聚焦于特定的会话 ID 并重现问题，捕获查询执行期间发生的所有事件的等待统计信息。



### 结合 DMV 检测当前发生的等待

我们已经回顾了用于等待统计分析的重要的 DMV；现在，让我们看一个如何使用 DMV 找出 SQL Server 变慢原因的示例。收集这些信息不会立即解决您的所有问题，但它将提供一个线索，指明从何处开始寻找解决方案。

**技术备注**
等待和队列调优方法的核心思想是资源消耗，其中资源包括物理资源（`CPU`、`内存`、`磁盘 I/O`、`网络`）以及逻辑资源（`锁定/阻塞`）。因此，这里的诀窍在于，当涉及到调优资源消耗时，您只有两个选择：`用得更少`或`买得更多`。一旦理解了这一点，它就能指导您针对性能问题可能采取的行动和解决方案。

考虑以下场景：您是一家大型公司的数据库管理员（DBA），该公司使用单个数据库存储所有销售信息。该数据库运行在一个 SQL Server 实例上，每天有几百名用户查询该数据库。

通常，一切运行良好——用户能快速访问他们想要的信息，所有需要使用数据库的人也很满意。然而，今天对您这位 DBA 来说可不是个好日子。从上午 10 点开始电话就响个不停，一些用户聚集在您的办公室门口，眼神里充满愤怒——查询和插入销售信息的速度慢得令人难以置信。

既然本书是关于等待统计的，让我们看看如何分析与场景中性能问题相关的等待统计信息。

我们知道 `sys.dm_os_wait_stats` DMV 显示的是累积的等待统计信息，因此对于此场景帮助不大。一个好得多的起点是 `sys.dm_os_waiting_tasks` DMV，因为它会向我们显示当前所有正在等待的任务。

```sql
SELECT *
FROM sys.dm_os_waiting_tasks
ORDER BY session_id DESC;
```

## 使用 Perfmon 查看等待统计

分析等待统计所需获取额外指标的工具之一是性能监视器（Performance Monitor），或称 Perfmon。Perfmon 可在 Windows 操作系统中使用，包含操作系统相关的指标，包括与 SQL Server 相关的性能计数器。你可以通过从 Windows 运行对话框或命令行执行 `Perfmon` 命令来启动它。

## 使用扩展事件捕获等待统计信息

SQL Server 中的大多数等待统计信息都是累计记录的，并且由于许多内部进程也会生成等待统计信息，因此很难检测单个查询的影响。借助扩展事件，可以捕获查询遇到的确切等待时间以及查询正在等待的等待类型。这些信息将帮助我们分析对系统影响最大的查询，从而优化查询性能，将其影响降至最低。或者，如果愿意，我们也可以捕获在执行时遇到特定等待类型的查询。

扩展事件在 SQL Server 2008 中引入，用于替代 SQL Server Profiler 中的非安全事件。扩展事件比 Profiler 具有更强的可扩展性，并且随着 SQL Server 每个版本的发布，扩展事件所包含的事件数量也在持续增长。

**技术笔记**

当 SQL Server 2008 引入扩展事件（用于非安全事件）和 SQL Server 审核（用于安全事件）时，人们认为 SQL Server Profiler 的消亡已为期不远。在撰写本文的 2022 年，Profiler 依然健在，并且永远不会消亡。#TeamProfiler

扩展事件向来以难以使用而闻名，这在 SQL Server 2008 中首次引入它们时尤为如此。从 SQL Server 2012 开始，首次可以使用 SSMS 创建扩展事件会话，这使得使用扩展事件变得容易得多。

使用扩展事件时，有许多不同的与等待相关的事件可用。我们可以通过查询 `sys.dm_xe_map_values` DMV 来查看这些事件，该 DMV 保存了所有不同扩展事件的事件类型：

```sql
SELECT *
FROM sys.dm_xe_map_values
WHERE name = 'wait_types';
```

#### 极客笔记

我们在前面的脚本中通过设置 `metadatafile` 参数包含了元数据文件。如果您运行的是 `SQL Server 2012` 或更高版本，则不再需要此参数。


然后，我们在我们正在过滤的会话下执行与原始示例中相同的查询。让 `Extended Event` 会话运行一小段时间后，我们可以使用 `ALTER EVENT SESSION` 命令停止它：

停止 `Extended Event` 会话后，我们接下来将文件中的信息（作为 `XML`）导入到一个表中，以查看会话捕获的内容；我们使用 `sys.fn_xe_file_target_read_file` 函数来完成此操作。然后，我们解析 `XML` 信息，以更可读的格式返回结果。清单 2-4 中的查询可用于读取 `Extended Event` 文件，将其导入临时表，并将结果作为行返回。

```sql
-- Check if temp table is present
-- Drop if exist
IF OBJECT_ID('tempdb..#XE_Data') IS NOT NULL
DROP TABLE #XE_Data
-- Create temp table to hold raw XE data
CREATE TABLE #XE_Data
(
XE_Data XML
);
GO
-- Write contents of the XE file
-- into our table
INSERT INTO #XE_Data
(
XE_Data
)
SELECT  CAST (event_data AS XML)
FROM sys.fn_xe_file_target_read_file
(
'E:\Data\WaitStats_XE_0_132969501905230000.xel',   'E:\Data\WaitStats_XE_0_132969501905230000.xem',   null,   null   );
GO
-- Query information from our temp table
SELECT
XE_Data.value ('(/event/@timestamp)[1]', 'DATETIME') AS 'Date/Time',
XE_Data.value ('(/event/data[@name="opcode"]/text)[1]', 'VARCHAR(100)') AS 'Operation',
XE_Data.value ('(/event/data[@name="wait_type"]/text)[1]',
'VARCHAR(100)') AS 'Wait Type',
XE_Data.value ('(/event/data[@name="duration"]/value)[1]', 'BIGINT') AS 'Wait Time',
XE_Data.value ('(/event/data[@name="signal_duration"]/value)[1]',
'BIGINT') AS 'Signal Wait Time',
XE_Data.value ('(/event/action[@name="sql_text"]/value)[1]', 'VARCHAR(100)') AS 'Query'
FROM #XE_Data
ORDER BY 'Date/Time' ASC;
```

## 通过执行计划按查询分析等待统计信息

到目前为止，我们已经查看了由各种后台进程或我们自己执行的查询所产生的聚合等待时间。由于我是唯一在测试机器上执行查询的人，因此很容易将等待时间与特定查询关联起来。不幸的是，在繁忙的系统中，许多会话并发地不断执行各种查询，各类等待统计信息 DMV 很难用于分析特定查询的等待类型和等待时间。

值得庆幸的是，SQL Server 2016 SP1 的发布引入了在查询执行计划内部捕获的等待统计信息。这使得任何人都能看到查询在运行时遇到的等待类型和等待时间。尽管这看起来显而易见，但这也意味着，只有在查看**实际执行计划**而非**估计执行计划**时，才能获得按查询统计的等待信息。

### 技术说明

实际执行计划是查询执行期间实际使用的执行计划。SSMS 中有一个选项可以查看估计执行计划。使用该选项时，SQL Server 引擎会编译在查询执行期间*最可能被使用*的执行计划；但是，它并不实际执行查询本身。由于没有查询执行，因此在编译估计执行计划时也没有等待统计信息可供记录。

公开按查询统计的等待信息最简单的方法是启用 `包含实际执行计划` 选项（如图 2-27 所示），可通过点击 “查询 – 包含实际执行计划” 菜单项或使用快捷键组合 `CTRL_M` 来实现，然后再执行您的查询。

在本例中，我们的查询在执行时遇到了三种不同的等待类型：`PAGEIOLATCH_SH`、`MEMORY_ALLOCATION_EXT` 和 `ASYNC_NETWORK_IO`。对于每种等待类型，我们都能看到花费了多少等待时间以及等待该类型的次数。此信息在查看单个查询在执行期间遇到的等待统计信息时非常有用，并且或许能为调整查询性能提供思路。例如，如果您看到查询经常遇到与存储相关的等待统计信息，那么研究如何最小化该特定查询的存储访问以使其执行更快可能是值得的。

我想再次指出的一点是，按查询统计的等待信息只记录在实际执行计划中。我再次提及的原因是，实际执行计划只能通过**在查询执行前启用该选项**来获得。没有其他方法可以访问实际执行计划，即使通过查询存储也不行，我们将在下一章看到。事实上，存储在 SQL Server 计划缓存中的执行计划是估计执行计划，而不是实际计划。这意味着，如果您期望通过计划缓存检索按查询统计的等待信息，您将会失望。

值得庆幸的是，尽管查询存储功能不记录实际执行计划，但它（随着 SQL Server 2017 的发布）确实会记录等待统计信息和其他查询运行时信息，并与估计执行计划一同存储。将这些信息混合在一起意味着，通过查询存储，我们可以回溯历史，查看查询遇到了哪些等待统计信息！

### 本章小结

在本章中，我们回顾了访问等待统计信息的多种方法。我们深入探讨了关于等待统计信息的一些最重要的 DMV —— `sys.dm_os_wait_stats`、`sys.dm_os_waiting_tasks`、`sys.dm_exec_requests` 和 `sys.dm_exec_session_wait_stats` —— 描述了它们的功能和返回的数据，并为您提供了一些可用于查询这些 DMV 的示例。我们还经历了一个示例场景，其中我们组合使用了部分 DMV 来分析示例中是什么拖慢了 SQL Server。示例中展示的步骤是在性能问题发生时分析系统问题的绝佳方法。简要地，我们查看了性能监视器（Perfmon），以及如何从其内部获取等待统计信息。之后，我们仔细研究了扩展事件，以及如何使用扩展事件 GUI 或 T-SQL 来捕获特定查询或会话的等待相关信息。最后，我们探讨了执行计划中记录的等待统计信息。

## 3. 查询存储

随着 SQL Server 2016 的发布，微软引入了一种分析和排查查询性能的新方法：查询存储。查询存储通常被称为 SQL Server 的“飞行记录仪”，因为它提供了查询何时执行、执行性能如何以及执行期间使用的执行计划等方面的洞察。虽然查询存储最初并未公开等待统计信息，但 SQL Server 2017 的发布包含了这一备受期待的补充功能。

由于查询存储在查询性能分析和调优方面极具价值，我认为为了最大化此功能的效益，值得对其给予一些额外的关注。


### 什么是查询存储？

查询存储（Query Store）功能最早在 SQL Server 2016 中发布，其目标是以一种可访问的方式公开并持久化查询性能信息。在查询存储出现之前，查询性能分析是一项具有挑战性且耗时的过程，需要对 SQL Server 如何处理查询以及如何通过各种 DMV（动态管理视图）分析信息有透彻的了解。虽然查询执行的经验和知识仍然有帮助，但查询存储能帮助你保留必要的性能信息，而无需自己开发解决方案。

查询存储直接集成在 SQL Server 引擎内部，在查询执行时捕获并分析它们。这是与通过其他方法（如执行计划缓存挖掘）进行查询分析的一个主要区别，后者只能在后期阶段获取查询运行时信息。查询存储的另一个优势是它将查询运行时信息持久化到磁盘。这允许你查看查询运行时指标的历史记录，以及比较历史和当前的运行时统计数据。这与 DMV 不同，后者只在 SQL Server 运行期间记录信息，并在 SQL Server 重启时清除所有已记录的信息，这意味着所有计数器又从 0 开始。继续比较查询存储与 DMV，虽然所有 DMV 都在 SQL Server 实例级别记录信息，但查询存储允许你基于每个数据库来捕获查询运行时指标。这使得分析包含多个数据库的实例中特定数据库的性能变得相当容易和快捷。

### 启用查询存储

在 SQL Server 2022 中，新数据库默认启用查询存储（处于 `READ_WRITE` 模式）（新的 Azure SQL 数据库也默认启用）。对于更早版本的 SQL Server，用户必须手动启用查询存储才能开始收集查询执行信息。启用查询存储可以通过两种方式完成：使用 SQL Server Management Studio（`SSMS`）和使用 T-SQL 语句。

### 查询存储中如何处理等待统计信息

在开始访问查询存储中记录的等待统计信息之前，我们将先研究查询存储如何处理等待统计信息，因为这与本书第 1 章“等待统计信息内部原理”中描述的过程不同。第 1 章描述了 SQL Server 引擎跟踪查询在特定资源（即等待类型）上等待了多长时间。此信息是在细粒度级别记录的；例如，`PAGEIOLATCH_SH` 等待类型表示查询正在等待将数据页从磁盘读取到缓冲区缓存中，而 `PAGEIOLATCH_EX` 表示查询正在等待将数据页移动到缓冲区缓存中进行修改。查询存储不是在详细级别记录等待时间，而是使用更高级别的等待类型概览，以最大程度地减少性能和资源利用率开销。查询存储将各种等待类型分组到类别中，并在类别级别记录等待时间，而不是在单个等待类型级别。

这意味着无法在查询存储中搜索特定的等待类型，并且属于同一类别的等待类型之间无法区分。举个例子，`CMEMTHREAD` 和 `RESOURCE_SEMAPHORE` 等待类型（你将在第 6 章“I/O 相关等待类型”中了解更多）在查询存储中都被记录在 `Memory` 类别下，尽管这两种等待类型指示不同的情况。

```sql
INNER JOIN sys.query_store_plan qsp
ON qsws.plan_id = qsp.plan_id
INNER JOIN sys.query_store_query qsq
ON qsp.query_id = qsq.query_id
INNER JOIN sys.query_store_query_text qsqt
ON qsq.query_text_id = qsqt.query_text_id
ORDER BY qsws.total_query_wait_time_ms DESC;
```


在查询和查询结果中，有一个名为 `runtime_stats_interval_id` 的列。查询存储对运行时（和等待时间）指标进行分组的方式是基于时间间隔进行聚合。默认情况下，这些间隔是一小时的时间块，这意味着前面示例中为特定查询返回的等待统计信息是该间隔内一次或多次查询执行的聚合结果。虽然你可以通过设置查询存储属性中的 `Statistics Collection Interval` 来将间隔降低到更小的时间段，但这将对 SQL Server 实例的性能产生负面影响，因此在更改此设置时要小心。

我只展示了几个如何从查询存储内部检索等待统计信息相关指标的示例。凭借查询存储收集的所有信息，还有海量的其他信息可以与等待统计信息指标相结合。例如，按特定等待类别筛选，或检测生成不同执行计划的查询以及其对特定查询等待的影响。我建议所有使用 SQL Server 2016 或更高版本的用户启用查询存储，并探索它收集的指标。

**极客备注**

虽然查询存储仅在 SQL Server 2016 及更高版本中可用，但 William Durkin（Twitter 账号 @sql_williamd）和 Enrico van de Laar（@evdlaar）发布了一个名为 open Query Store 的项目，该项目为低于 2016 版本的 SQL Server 模拟查询存储数据收集。该项目完全开源且免费，可通过其 GitHub 页面 `https://slrwnds.com/n4mudf` 获取。

在本章中，我们探讨了 SQL Server 2016 中引入的一项新的查询性能和分析功能：查询存储。我们了解了查询存储的底层工作原理，以及如何访问查询存储内置报告中的查询等待统计信息。

## 4. 构建稳固的基准线

在第 2 章“查询 SQL Server 等待统计信息”中，我们花了大量时间描述和使用各种访问 `等待统计` 信息的方法。这些方法大多侧重于利用这些信息来检测当前的性能问题。虽然使用这些实时方法可以找到性能问题的确切原因，但这需要深入了解各种 `等待类型`，并且——最重要的是——对你特定的 SQL Server 系统的性能有经验。

如果你管理一个 SQL Server 实例，你会很快熟悉它在不同情况下的反应方式。但如果你管理数百个 SQL Server 实例，要熟悉它们的所有性能表现几乎是不可能的。由于 SQL Server `等待统计` 很大程度上取决于特定 SQL Server 实例的工作负载，没有两个 SQL Server 实例会针对相同的 `等待类型` 有相同的 `等待时间`。这使得检测潜在问题变得困难，因为我们不能简单地说“嘿，`CXPACKET` `等待类型` 有 20,000 毫秒的 `等待时间`，所以我们出问题了。”这完全取决于你的系统配置和工作负载。一个 SQL Server 实例每分钟可能有 20,000 毫秒（20 秒）花在 `CXPACKET` `等待类型` 上，却没有任何性能问题；而另一个实例只有 1,000 毫秒的 `等待时间`，用户却持续抱怨性能不佳。

如果我们想对 `等待统计` 或任何 `性能相关数据` 进行深入分析，我们需要一种收集 `性能相关指标` 并赋予其意义的方法。仅仅检测到 10,000 毫秒的等待资源没有任何意义，因为我们不知道它是否导致了性能问题。是的，也许你的用户正在抱怨性能糟糕，同时你注意到了这 10,000 毫秒的 `等待`，但无法确定这个 `等待` 是否明确导致了用户遇到的特定性能问题。有人可能会猜测，并简单地假设特定的资源 `等待` 就是性能问题的根源。作为一名前生产环境 DBA，我可以告诉你，DBA 不喜欢猜测是什么拖慢了我们的数据库服务器。我们想要确定性能问题的来源。这就是 `基准线` 能发挥作用的地方。

`基准线` 通过提供系统正常状态的定义，帮助赋予 `性能指标` 意义。没有 `基准线`，我们无从知晓系统或应用程序是运行在最佳状态还是极其缓慢。`基准线` 非常重要；事实上，它们如此重要，以至于我决定在本书中用一整章来讨论它们。没有稳固的 `基准线`，你的测量结果毫无意义！尽管本书侧重于 `等待统计`，但 `基准线` 对于你可以在系统上捕获的每一个 `性能相关指标` 都很有用，为你提供了一种宝贵的性能分析方法。

如果你阅读了上一章关于 `查询存储` 的内容，你可能会认为 `查询存储` 可以处理你所有的 `基准线`。虽然 `查询存储` 对于捕获和监控单个数据库内的查询性能很有用，但它并不提供整个 SQL Server 实例的性能概览。我个人认为 `查询存储` 是捕获和监控 `基准线` 常规流程的一个补充，而非替代。

### 什么是基准线？

如果我们查阅牛津词典中的“`基准线`”，会看到以下定义：“用于比较的最低点或起点。”这句话使用了关键词“起点”和“比较”，完美地抓住了 `基准线` 的本质。`基准线` 是起点，或者在我们的情况下，是用于与后续测量结果进行比较的度量值。理想情况下，我们在正常或标准情况下捕获 `基准线` 测量值。如果我们在以后的某个时间点再次执行相同的测量，就会将测量结果与 `基准线` 进行比较。如果在比较中我们的测量结果不同，则说明可能发生了变化。

即使你在分析性能时不使用 `基准线` 比较，无论你是否意识到，你仍然在使用 `基准线`。例如，如果你每月第一天收到工资，这是你正常的情况，或者在本章的语境下，是你的 `基准线`。如果由于某种原因，你在每月第一天没有收到工资，你会注意到与 `基准线` 的偏差。这可能成为调查你为何没有按时收到工资的原因。也许发薪日从 1 号改到了 5 号，或者在最坏的情况下，你工作的公司无法再支付你的工资。在任何一种情况下，我们都可以采取行动：要么接受变化，并由此创建一个新的 `基准线`；要么将情况恢复回 `基准线` 状态。图 4-1 展示了这个过程。

![](img/340881_3_En_4_Fig1_HTML.png)

*一个流程图连接了 `基准线`、`变更` 和 `新基准线`，指向接受 `新基准线` 或撤销 `变更` 以恢复至 `基准线`。*

定义和维护 `基准线` 是一个迭代过程。发生在你系统上的每一次 `变更` 都可能影响你的 `基准线`。应用程序的新版本可能会改变许多查询，或者你的公司可能授予新部门访问数据库的权限，从而增加了连接数。在这两个例子中，我们都会对 `基准线` 进行调整，因为正常情况已经发生了变化。

所有这些随着系统每次 `变更` 而调整和测量 `基准线` 的工作听起来很繁重，有时确实如此。但请相信我——拥有 `基准线` 的好处远远超过成本。`基准线` 帮助你比仅仅查看单次测量结果更快地发现问题，而在 `等待统计` 的情况下，它是找到你问题可靠、明确答案的唯一途径。让我们使用一个更技术性的例子，通过 DBA 简来说明这一点。

简维护着一个托管单一用户数据库的 SQL Server 实例。该数据库供公司每一位销售人员使用，记录公司与客户之间的每一笔财务交易。用户通过 `应用程序 X` 访问数据库。`应用程序 X` 运行的是 2.4 版本，非常稳定。性能良好，用户满意，资金持续流入。一切都很顺利，直到有一天一位顾问走进来，希望将 `应用程序 X` 升级到全新的 3.0 版本。升级到 3.0 版本轻而易举，没有出现任何问题，所有用户都喜欢新功能。

两天后电话响了。简的经理刚刚收到销售团队的消息，说 3.0 版本的性能非常糟糕，并要求立即解决。



所幸，Jane 知道基线的重要性，并在升级到 3.0 版本之前创建了一个。使用 2.4 版本的基线，Jane 将基线中的测量值与 3.0 版本中进行的测量值进行比较，并立即发现了 `锁等待时间` 测量值的巨大差异。由于其他测量值与 2.4 基线相比大致保持不变，Jane 将注意力集中在长时间运行的锁上，并识别出一个正在锁表的更新查询。Jane 回滚了该查询，情况恢复正常。随后，Jane 联系了应用程序供应商，并了解到此行为是软件中的一个错误导致的。

现在这个例子听起来可能有点牵强，但它是我几乎每天用来衡量变更影响或分析性能问题的方法的简化版本。如果没有 `锁等待时间` 的基线，Jane 就不会知道等待时间已经增加，因为 Jane 没有可比较的对象。Jane 可能会选择查看其他指标而不是 `锁等待时间`，从而浪费宝贵的时间和金钱。

这里的道理很简单：基线将帮助你更快地检测异常情况并解决性能问题！

### 基线类型与统计

我们根据收集的信息使用不同类型的基线。没有一种基线能满足我们所有的需求，尤其是在将基线用于性能故障排查目的时。例如，我们可以为每种等待类型创建一个基线，或者我们选择只为对系统影响最大的等待类型创建基线。我们也可以选择为特定日期或时间段（如工作时间）创建基线，并为非工作时间创建另一个基线。

在选择了想要创建基线的指标之后，我们接下来要决定如何计算基线。这些选择涉及一些数学计算，通常需要计算聚合值，如平均值。在许多情况下，基线由许多数据点的平均值组成，这取决于在每个时间范围内执行了多少次测量。如果你长时间收集测量值并根据这些测量值计算平均值，你创建的基线将比仅基于一天测量值的基线更可靠。但是，基于平均值创建基线也有缺点，因为平均值受偏斜数据的影响很大。在不深入探讨统计细节的情况下，偏斜数据意味着非常高或非常低的值会影响你的平均值。以一组学生和考试成绩为例。我们通过计算考试的平均成绩来评估小组的表现（学生评分在 1 到 10 之间，1 表示很差，10 表示优秀；考试需要达到 6 分或以上才算通过）。图 4-3 在图表中显示了考试结果。

![](img/340881_3_En_4_Fig3_HTML.png)

一个考试成绩的条形图，有学生 1 到 10 的条形。最高值是 10，对应学生 9 和 10。最低值是 3，对应学生 1。

**图 4-3** 考试成绩

如图 4-3 所示，只有四名学生的成绩高于通过考试所需的 6 分。小组中的其余成员成绩未达到要求。然而，如果我们看小组的平均表现，他们得了 6 分。我们可能会得出结论，小组的平均成绩为 6 分，表现足够好，但那样我们就会忽略只有十分之四的学生通过考试这一事实。在使用平均基线时，记住这一点很重要。如果你看到平均基线出现激增，这总是值得你调查的事情，因为它影响了你的基线。

有可用的统计方法来处理偏斜数据和平均值，其中一种是截尾（或截断）均值。这种方法移除序列中最高和最低测量值的 x 百分比，从而创建一个更稳定的平均值。回顾所有可用的统计方法超出了本书的范围。只需注意数据是如何收集的，是否使用了平均值，以及是否存在可能导致偏斜的异常值。

### 基线陷阱

希望上一节已经让你相信基线的重要性，但在你开始捕获每一个性能指标并将其转换为基线测量值之前，有一些陷阱需要避免。

#### 信息过载

尽管你可以自由地为环境中的每一个可能的指标创建基线，但这通常被认为是个坏主意。收集过多信息可能会在你寻找答案时蒙蔽你的双眼。每次发生性能问题时，将 100,000 个不同的指标与基线进行比较是巨大的时间浪费。这里的建议是保持指标数量较少，只包括对系统最重要的性能指标。例如，你可以包括与可用性组相关的性能指标，但如果你的系统不使用此功能，包含它们就没有用处。

#### 了解你的指标

选择性能指标的另一个重要方面是理解性能指标代表什么。没有这些知识，很难得出正确的结论。例如，如果你看到一个名为 `Locks` 的指标，你应该知道这个指标代表的是 `锁` 的总数还是每秒的 `锁` 数。

#### 关注测量值的重大变化

当将测量值与基线进行比较时，寻找任何大的增长或减少。特别是对于等待统计，等待时间的非常小的增长（1-2%）不值得担忧。如果你的某项等待时间测量值上升了 20%，那将是一个开始调查的好信号。

#### 使用固定间隔

在捕获等待统计信息时，你应该使用固定间隔。如果我们随机捕获等待时间，就不可能建立可靠的基线。自动捕获等待统计信息的一种方法是使用 `SQL Server Agent` 并将作业计划设置为固定间隔，例如每 15 分钟一次。



### 为等待统计分析构建基线

接下来，让我们开始工作，创建一个可用于等待统计分析的基线。正如我在本章开头提到的，基线对于使用等待统计信息分析性能问题至关重要。没有人的等待类型和等待时间与您的系统完全相同，因此您需要自己创建一个可供比较的基线。

在本节中，我将向您展示一种创建、维护和比较基线与测量值的方法。这只是一个示例；您可能会发现其他更适合您需求的方法。

我们将捕获 SQL Server 等待统计信息测量值，并将其存储在一个名为 `Baseline` 的数据库中，这样信息就不会丢失。由于等待统计信息是在 SQL Server 实例级别记录的，因此在每个 SQL Server 实例中创建一个单独的测量数据库是合理的。图 4-4 展示了我在 SSMS 中的 `Baseline` 数据库。

正如您在脚本中看到的，我们捕获了等待类型、等待时间、信号等待时间和等待任务数。我们还捕获了记录等待统计信息时的日期和时间。日期和时间被拆分到额外的列中，以便对数据进行分段，这样更容易基于特定的某天、小时、月份等构建特定的基线，而无需每次都转换 `datetime` 数据类型。

现在我们的表已经准备好了，是时候捕获一些等待统计信息并将它们插入到 `WaitStats` 表中。因为 `sys.dm_os_wait_stats` DMV 返回的是累积等待时间，所以我们必须计算两个捕获时刻之间等待时间的差异。如果我们只是直接从 `sys.dm_os_wait_stats` DMV 捕获信息，我们将总是获得不断增加的等待时间，使得比较变得困难。有两种方法可以捕获两次测量之间等待时间的变化，它们各有优缺点。

第一种方法，我称之为重置方法，它将从 `sys.dm_os_wait_stats` DMV 捕获等待统计信息，然后使用以下命令重置 DMV：

```
DBCC SQLPERF('sys.dm_os_wait_stats', CLEAR)
```

该方法的主要优点是使用简单，因为我们只需要捕获信息，然后再次重置它，并在下一次测量时启动相同的过程。不需要计算增量，因为在第一次测量之后，计数器被重置为 0。缺点是这个 `DBCC` 命令会重置 `sys.dm_os_wait_stats` DMV 内部的信息。这意味着我们丢失了 DMV 内部的累积信息，这是我们不想丢失的信息。图 4-5 说明了这种等待统计信息捕获方法。

> 第三种选择是获取指标而不清除或手动计算增量。您将能够在以后执行任何必要的计算。或者可能只在每次插入时存储非零指标。或者只存储自上次收集以来发生更改的指标。使用最适合您和您的系统的任何选项或方法。

还有更多方法可用于捕获等待统计信息，但我最常见到的是这两种，或它们的变体。您想使用哪种方法取决于您，因为最终两者都会返回相同的结果。

#### 重置捕获方法

重置捕获方法包含一个 T-SQL 脚本，用于从 `sys.dm_os_wait_stats` DMV 捕获信息，然后重置 DMV 内部的计数器。清单 4-3 显示了用于填充我们之前创建的 `WaitStats` 表的 T-SQL 脚本。

```
USE [Baseline]
GO
-- Insert Wait Stats into Baseline table
INSERT INTO WaitStats
SELECT
GETDATE() AS 'DateTime',
DATEPART(DAY,GETDATE()) AS 'Day',
DATEPART(MONTH,GETDATE()) AS 'Month',
DATEPART(YEAR,GETDATE()) AS 'Year',
DATEPART(HOUR, GETDATE()) AS 'Hour',
DATEPART(MINUTE, GETDATE()) AS 'Minute',
DATENAME(DW, GETDATE()) AS 'DayOfWeek',
wait_type AS 'WaitType',
wait_time_ms AS 'WaitTime',
waiting_tasks_count AS 'WaitingTasks',
signal_wait_time_ms AS 'SignalWaitTime'
FROM sys.dm_os_wait_stats;
-- Clear sys.dm_os_wait_stats
DBCC SQLPERF ('sys.dm_os_wait_stats',CLEAR)
GO
```

#### 增量捕获方法

增量捕获方法同样包含一个单独的 `T-SQL` 脚本，但它比重置捕获方法更为复杂。它使用一个临时表来存储第一次测量结果，等待 15 分钟后进行第二次测量，并计算差值。结果会被插入到 `WaitStats` 表中。清单 4-4 显示了如果您计划使用这种收集等待统计指标的方法时可以使用的 `T-SQL` 脚本。

```
USE [Baseline]
GO
-- 检查临时表是否已存在，如果存在则删除它。
IF EXISTS
(
SELECT *
FROM tempdb.dbo.sysobjects
WHERE ID = OBJECT_ID(N'tempdb..#ws_Capture')
)
DROP TABLE #ws_Capture;
-- 创建临时表以保存我们的第一次测量
CREATE TABLE #ws_Capture
(  wst_WaitType NVARCHAR(60),
wst_WaitTime BIGINT,
wst_WaitingTasks BIGINT,
wst_SignalWaitTime BIGINT   );
-- 将我们的第一次测量结果插入临时表
INSERT INTO #ws_Capture
SELECT wait_type,
wait_time_ms,
waiting_tasks_count,
signal_wait_time_ms
FROM sys.dm_os_wait_stats;
-- 等待下一次测量
-- 在此例中，我们将等待 15 分钟
WAITFOR DELAY '00:15:00'
-- 将第一次测量结果与新测量结果结合并计算差值
-- 将结果写入 WaitStats 表
INSERT INTO WaitStats
SELECT
GETDATE() AS 'DateTime',
DATEPART(DAY,GETDATE()) AS 'Day',
DATEPART(MONTH,GETDATE()) AS 'Month',
DATEPART(YEAR,GETDATE()) AS 'Year',
DATEPART(HOUR, GETDATE()) AS 'Hour',
DATEPART(MINUTE, GETDATE()) AS 'Minute',
DATENAME(DW, GETDATE()) AS 'DayOfWeek',
dm.wait_type AS 'WaitType',
dm.wait_time_ms - ws.wst_WaitTime AS 'WaitTime',
dm.waiting_tasks_count - ws.wst_WaitingTasks AS 'WaitingTasks',
dm.signal_wait_time_ms - ws.wst_SignalWaitTime AS 'SignalWaitTime'
FROM sys.dm_os_wait_stats dm
INNER JOIN #ws_Capture ws
ON dm.wait_type = ws.wst_WaitType;
-- 清理临时表
DROP TABLE #ws_Capture;
清单 4-4
增量捕获方法
```

现在简已经获取了今天 `PAGEIOLATCH_SH` 等待类型的等待时间，下一步是利用 `PAGEIOLATCH_SH` 等待类型的历史测量值创建一个基线，以便将今天的测量值与基线进行比较。简使用清单 4-6 所示的查询来构建基线。

```sql
-- 工作日上午 8 点到 9 点的基线
-- 不包括今天完成的测量值
SELECT
CONVERT(VARCHAR(5), ws_DateTime, 108) AS '时间',
AVG(ws_WaitTime) AS '基线'
FROM WaitStats
WHERE ws_WaitType = 'PAGEIOLATCH_SH'
AND ws_DayOfWeek IN ('Monday', 'Tuesday', 'Wednesday', 'Thursday','Friday')
AND (ws_Hour >= 8 AND ws_Hour < 9)
AND CONVERT(VARCHAR(5), ws_DateTime, 105) < CONVERT(VARCHAR(5), GETDATE(), 105)
GROUP BY CONVERT(VARCHAR(5), ws_DateTime, 108);

-- PAGEIOLATCH_SH 基线
```

相关性并不意味着因果关系。虽然更换磁盘似乎解决了问题，但现实是你的基础架构很复杂。有可能真正的根本原因是别的，比如在`VM`主机上发生内核拥塞，当时显示为`PAGIOLATCH_SH`，并且大约在磁盘被更换的同时也得到了解决。

```sql
SELECT *
INTO WaitStats_March
FROM WaitStats
WHERE ws_Month = 3;
```
这也让我们能够通过将不同的表连接在一起来比较不同时间段的特定等待时间。

#### 总结
在本章中，我们从理论和实践两个角度仔细研究了基线。基线对于你进行的任何类型的性能分析都极其重要。在等待统计的案例中，如果你想排查与`SQL Server`相关的性能问题，通常需要基线。由于等待统计对你的系统是唯一的，比较等待时间只有一种方法——那就是基线。

我为你提供了一些示例和`T-SQL`脚本，来创建你自己的等待统计基线表，这样你现在就可以开始捕获等待统计信息了。我们还通过一个例子，介绍了如何查询基线信息并将其与实际测量值进行比较，以排查与性能相关的事件。
