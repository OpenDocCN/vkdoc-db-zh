# 第六章 性能能力

## 文件与文件组

我在第 3 章中向你展示了如何创建数据库，并简要描述了数据库随附的默认文件：一个数据库文件（通常以 `.mdf` 为文件扩展名）和一个事务日志文件（通常以 `.ldf` 为文件扩展名）。

在许多情况下，使用单个数据库文件和单个事务日志文件足以满足 SQL Server 数据库的需求。然而，本章是关于调优的，因此我在本节提供了关于如何通过文件放置、文件组以及规划文件增长来最大化 I/O 性能的指导。

我还应该说明，如果你决定在一个 SQL Server 实例上托管多个数据库，你应该考虑将每个数据库的文件分散到多个磁盘上，或者使用配置为跨一系列驱动器条带化 I/O 的托管磁盘存储系统。我通常将系统数据库放在默认的数据库文件位置，除了 `tempdb` 数据库外，不太担心磁盘性能问题。

通常的做法是将 `tempdb` 数据库文件与其他系统数据库放在不同的磁盘上。我们的文档有一个很好的示例，说明如何将 `tempdb` 文件移动到其他路径，请参阅[移动系统数据库](https://docs.microsoft.com/sql/relational-databases/databases/move-system-databases#Examples)。

###### 分离数据和事务日志文件

我几乎总是建议使用单个事务日志文件。不存在性能




拥有多个事务日志文件的好处。如果你决定只使用一个数据库文件，我确实建议你将数据库文件和事务日志文件放置在不同的磁盘上。这是多年来针对 SQL Server 数据库的传统建议。该建议的主要原因之一在于数据库文件和事务日志文件的 I/O 使用模式特性。对数据库文件的写入被描述为 `随机` I/O。虽然 SQL Server 以连续页的方式写入数据库文件，但它可以写入文件中的任何位置。对事务日志的写入总是顺序的。SQL Server 在写入事务日志文件时总是追加日志块。此外，SQL Server 很可能通过检查点任务（例如 `RECOVERY WRITER`）并发地写入数据库文件，并通过 `LOG WRITER` 任务写入事务日志。因此，通过将数据库文件写入和事务日志写入分离到不同的磁盘上，你几乎总是能获得更好的 I/O 性能。
