# 谈谈多语言持久性

多语言持久性在过去几年中逐渐为人所知，当时微服务架构展现了其全部威力并成为主流。其理念是，每个服务都应为其执行的任务使用最佳的数据库技术（关系型、键值型、图、文档型等）。

虽然理论上这似乎完全正确，但这种方法内部存在一个巨大的挑战：存储在不同系统中的数据如何才能被集成并保持一致，从而可以信任其没有错误并用于决策？

不仅如此，如果使用不同的数据库来存储数据，那么所有这些不同的技术都需要足够了解才能恰当使用；否则，安全性、一致性和性能问题将迅速出现，成为日常噩梦。

但还有更多。如果某些时候需要收集数据然后重新分发到所有系统，该怎么办？集成是一个非常非常困难的问题。事实上，你不仅需要一个流程来维持数据集成的进行，在某些时候，还需要一个地方来存储数据的“黄金主版本”——一个商业分析系统可以从中获取数据以进行有意义的预测的地方，并且在当今，也是为机器学习算法提供可靠数据的地方。你将需要一个系统来管理你的 `master data`。

如你所见，每一步复杂性都在增加，而 `caching` 还未纳入考量！现在，如果确实需要复杂性，那也没关系。只要你在架构决策上有控制权，并且这种复杂性是必需的（例如你的业务像管理一个支持百万并发用户的在线文字处理器一样复杂），并且你已准备好处理它，那就没问题。但不幸的是，如果规划和管理不善，很容易陷入复杂架构带来的意外问题洪流中，问题日益复杂，吞噬 IT 预算的每一分钱。

多语言持久性和高度专业化是很好的选择，但并非总是适用于每个项目：有时，一个集成的解决方案，在一个地方提供所有选项，会好得多。Azure SQL 可以成为那个集成解决方案。
