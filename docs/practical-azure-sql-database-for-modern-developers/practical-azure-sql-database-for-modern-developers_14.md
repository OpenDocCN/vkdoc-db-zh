# 业务规则应置于何处？

业务规则：它们应当放入应用程序，而非数据库。这个答案简单明了。问题在于，业务规则的定义本身并不那么直截了当。

尽管对此类问题的深入探讨远超本书范围，但我们至少需要划清一条界限，以明确哪些处理应推给数据库，哪些不应。

业务规则通常作用于数据。除此之外，它们还负责处理、读取、修改和创建数据。可以说，所有 `data manipulation` 都应该被推到数据所在的数据库去执行。例如，如果创建一个新用户需要在 `User` 表中插入一行记录，那么该语句就应该在数据库内部执行，而不是通过将数据操作逻辑与业务逻辑以非常不优雅、难以维护的方式混杂在一起的混乱代码来实现。

一个更好的例子是当需要聚合数据时，也许是为了计算某个复杂的月末报告以监控业务关键指标。由于所谓的 `data gravity`，将复杂的计算移动到数据已存在的地方，比将数据移动到业务层要容易得多。数据有大小，因而有重量：移动它可能非常具有挑战性，并且必然需要大量努力。

这在分布式系统中尤其明显，其中每个不同的系统可能都想访问和修改相同的数据。如果在每个系统中都放置数据操作逻辑，就需要某种复杂的——真正复杂的——分布式协调。通过让数据库履行其职责——协调数据访问——这种复杂性完全可以避免。

划清这条界限后，就更容易实现清晰的关注点分离，为解决方案带来清晰性和简洁性，从而使其更易于创建，维护和演进的成本也更低。它将更敏捷。
