## 第 1 章 高可用性与灾难恢复概念

在当今运行关键任务应用程序的 24×7 环境中，企业严重依赖于其数据的可用性。尽管服务器及其软件通常可靠，但始终存在硬件故障或软件缺陷的风险，这些都可能导致服务器宕机。为减轻这些风险，关键业务应用程序通常依赖冗余硬件来提供容错能力。如果主系统发生故障，应用程序可以自动故障转移到冗余系统。这是高可用性 (HA) 的基本原理。

即使实施了 HA 技术，仍然存在导致应用程序不可用的小概率事件风险。这可能是由于重大事件造成的，例如自然灾害或恐怖行为导致的数据中心损毁，也可能是由于数据损坏或人为错误，导致应用程序数据丢失或损坏到无法修复的程度。

在这些情况下，一些应用程序可能依赖于恢复最新备份以尽可能多地恢复数据。然而，更关键的应用程序可能需要在辅助位置设置一台冗余服务器来保存数据的同步副本。这是灾难恢复 (DR) 的基本概念。本章讨论 HA 和 DR 背后的概念。

### 可用性级别

解决方案可供最终用户使用的时长称为 `可用性级别` 或 `正常运行时间`。为了提供真实的正常运行时间情况，公司应从用户的桌面测量解决方案的可用性。换句话说，即使您的 SQL Server 已连续运行超过一个月，用户仍可能因其他因素而经历其解决方案的中断。这些因素可能包括网络中断或应用程序服务器故障。

要计算其他级别的可用性，你可以使用`代码清单 1-1`中的脚本。运行此脚本前，请将`@Uptime`的值替换为你希望计算的正常运行时间级别。你还应将`@UptimeInterval`的值替换为相应的每周、每月或每年的运行时间间隔。

`代码清单 1-1. 计算可用性级别`

```sql
DECLARE @Uptime DECIMAL(5,3) ;
--指定要计算的正常运行时间级别
SET @Uptime = 99.9 ;

DECLARE @UptimeInterval VARCHAR(5) ;
--指定 WEEK, MONTH, 或 YEAR
SET @UptimeInterval = 'YEAR' ;

DECLARE @SecondsPerInterval FLOAT ;
--计算每个时间间隔的总秒数
SET @SecondsPerInterval =
(
    SELECT CASE
        WHEN @UptimeInterval = 'YEAR'
            THEN 60*60*24*365.243
        WHEN @UptimeInterval = 'MONTH'
            THEN 60*60*24*30.437
        WHEN @UptimeInterval = 'WEEK'
            THEN 60*60*24*7
    END
) ;

DECLARE @UptimeSeconds DECIMAL(12,4) ;
--计算允许的停机秒数
SET @UptimeSeconds = @SecondsPerInterval * (100-@Uptime) / 100 ;

--格式化结果
SELECT
    CONVERT(VARCHAR(12), FLOOR(@UptimeSeconds /60/60/24)) + ' 天, '
    + CONVERT(VARCHAR(12), FLOOR(@UptimeSeconds /60/60 % 24)) + ' 小时, '
    + CONVERT(VARCHAR(12), FLOOR(@UptimeSeconds /60 % 60)) + ' 分钟, '
    + CONVERT(VARCHAR(12), FLOOR(@UptimeSeconds % 60)) + ' 秒。' ;
```

### 服务级别协议与服务级别目标

当由第三方提供商负责管理服务器时，合同通常包含服务级别协议（`SLAs`）。这些`SLAs`定义了诸多参数，包括可接受的停机时间、故障发生时服务器可停机的最长时间，以及故障发生时可接受的数据丢失量。通常，如果未能满足这些`SLAs`，提供商将面临经济处罚。

如果服务器由内部管理，数据库管理员（`DBAs`）仍然有客户（customers）的概念。这些客户通常是应用程序的最终用户，主要联系人是业务负责人（business owner）。应用程序的业务负责人是组织内委托该应用程序并负责签署资金增强等事项的利益相关者。

在内部场景中，仍然可以定义`SLAs`，在这种情况下，如果未能满足这些`SLAs`，`IT 基础设施`或`平台`部门可能需要向业务团队承担费用结算（charge-back）责任。然而，在内部场景中，`IT`部门与业务团队协商确定服务级别目标（`SLOs`）要比制定`SLAs`常见得多。`SLOs`在本质上与`SLAs`非常相似，但采用`SLOs`意味着业务方不会在未能达成目标时对`IT`部门施加经济处罚。

### 恢复点目标与恢复时间目标

应用程序的恢复点目标（`RPO`）指明了在发生故障时可接受的数据丢失量。例如，对于一个支持报表应用程序的数据仓库，由于它可能每天只通过一个 `ETL`（提取、转换和加载）流程更新一次，而所有其他活动都是只读报表，因此这个周期可能较长，比如 24 小时。然而，对于高事务性系统，例如支持交易平台或 Web 应用程序的 `OLTP`（联机事务处理）数据库，`RPO` 将为零。`RPO` 为零意味着不可接受任何数据丢失。

应用程序的高可用性和灾难恢复可能有不同的 `RPO`。例如，出于成本或应用程序性能的考虑，站点内故障转移可能要求 `RPO` 为零。但是，如果同一应用程序故障转移到 `DR`（灾难恢复）数据中心，则可能可接受五到十分钟的数据丢失。这是因为实现站点内可用性和站点间恢复所用的技术存在差异。

应用程序的恢复时间目标（`RTO`）指定了从故障发生到恢复完成且用户可以重新连接之前，应用程序可以停机的最大时间。在计算可实现的应用程序 `RTO` 时，需要考虑许多方面。例如，集群从一个节点故障转移到另一个节点，并且 `SQL Server` 服务重新启动可能只需不到一分钟；然而，数据库恢复可能需要更长得多的时间。数据库恢复所需的时间取决于许多因素，包括数据库的大小、实例中的数据库数量以及故障转移发生时有多少事务正在处理中。这是因为所有未提交的事务都需要回滚。

就像 `RPO` 一样，根据是站点内还是站点间故障转移，存在不同的 `RTO` 也是很常见的。同样，这主要是由于技术差异造成的，但也考虑了如果主数据中心丢失，你需要花费多少时间在 `DR` 数据中心启动整个环境。

在发生数据损坏的情况下，应用程序的 `RPO` 和 `RTO` 也可能有所不同。根据损坏的性质以及已实施的 `HA/DR` 技术，数据损坏可能导致你需要从备份中还原数据库。

如果必须还原数据库，最坏的情况是可实现的还原点可能是上次备份的时间。这意味着你必须将特定 `RPO` 的硬性业务要求纳入备份策略。但是，如果只有部分数据库损坏，你或许可以从实时数据库中抢救一些数据，并且只从还原后的数据库中还原损坏的数据。

数据损坏也可能影响 `RTO`。最大的影响因素之一是备份是存储在本地服务器上，还是需要从磁带中检索。从磁带，甚至从异地位置检索备份文件，很可能会显著增加恢复过程的时间。

另一个影响因素是导致损坏的原因。如果是由有故障的 `IO` 子系统引起的，那么你可能需要留出时间让 Windows 管理员对卷运行磁盘检查命令（`CHKDSK`），并可能需要更多时间来更换磁盘。但是，如果损坏是由用户意外截断表或删除数据文件引起的，则无需担心此问题。

决定你应该努力实现多少个“9”的关键因素是停机成本。停机成本分为两类：有形成本和无形成本。有形成本通常比较容易计算。让我们以一个销售应用为例。在这种情况下，最明显的有形成本是收入损失，因为销售人员无法接单。无形成本则更难量化，但可能昂贵得多。例如，如果客户无法在你的公司下单，他们可能会在竞争对手的公司下单，并且再也不回来。其他无形成本可能包括员工士气低落导致的更高人员流失率，甚至公司声誉损失。由于无形成本的本质使其只能估算，行业的经验法则是将有形成本乘以三，并用这个数字来代表你的无形成本。

一旦你有了应用程序总停机成本的每小时数字，你就可以将这个数字按应用程序的预计生命周期进行扩展，并比较实施不同可用性级别的成本。例如，假设你计算出你的总停机成本是每小时`$2,000`，而你的应用程序的预计生命周期是三年。表 1-2 说明了你的应用程序的停机成本，比较了你为实施每个可用性级别所计算出的成本（已计入硬件、许可证、电力、布线、额外存储、额外支持设备（如新机架）、管理成本等）。这被称为解决方案的总拥有成本（`TCO`）。

### 总结

你的应用程序的可用性级别是以正常运行时间占总时间的百分比来衡量的...



应用程序面向用户可用。其可用性水平常以“9”的个数来描述。例如，99.9%的正常运行时间要求被称为“三个 9”的可用性。对正常运行时间的要求越高，实现解决方案的成本就越高。因此，您努力要达到的可用性水平应由服务级别协议（SLA）和停机成本来驱动。

恢复点目标是衡量在发生灾难时可接受丢失多少数据的指标。例如，如果您唯一的灾难恢复解决方案是备份，并且备份计划每小时执行一次，那么您可以实现一小时的恢复点目标。

恢复时间目标是衡量在发生故障后恢复解决方案所需时间的指标。例如，如果您设定的恢复时间目标是 30 分钟，那么您必须能够在半小时内恢复服务。

确定应用程序的停机成本非常重要，因为这是决定您可用性水平的主要驱动因素之一。停机成本包括有形成本和无形成本。有形成本可以计算，而无形成本需要估算。

冗余基础设施有助于维护您的应用程序和服务的可用性。冗余服务器将被分类为热、温或冷。热备服务器是与活动服务器保持同步并配置为允许自动故障转移的服务器。这适用于高可用性（HA）场景。温备服务器是与活动服务器保持同步，但未配置为自动故障转移的服务器。相反，必须由工程师手动执行故障转移。这适用于灾难恢复（DR）场景。冷备服务器不与活动服务器保持同步，因此无法自动故障转移。冷备服务器适用于所有数据为只读且从不修改的灾难恢复场景。
