### 约束推导
确定适用于给定派生 `relvar`（关系变量）或被给定派生关系满足的约束的过程。

#### 包含
通常指容器与其所容纳事物之间的关系；特指一个 `bag`（包）或 `set`（集）与其元素之间的关系。与 `inclusion`（包含）对比。

### 矛盾式
一种谓词，其所有可能的调用，无论代入其参数的参数是什么，都保证结果为 `FALSE`（假）。注：逻辑中的矛盾式与日常话语中的矛盾并不完全相同。粗略地说，在(a)日常话语中，矛盾是指某个命题 `p` 及其否定 `NOT(p)` 同时为真的情况；而在(b)相比之下，在逻辑中，它指任何“总是为 `FALSE`”的东西（例如，真值 `FALSE` 本身）。与 `tautology`（重言式）对比。

### 示例
令 `p1` 为谓词（实际上是一个命题）`2 + 2 = 5`；令 `p2` 为谓词 `x > x`，其中 `x` 表示任意整数；令 `p3` 为谓词 `(p) AND (NOT(p))`，其中 `p` 表示任意命题。

那么 `p1`、`p2` 和 `p3` 都是矛盾式。

### 逆否式
蕴含式 `IF (p) THEN (q)` 的逆否式是逻辑上等价的蕴含式 `IF (NOT(q)) THEN (NOT(p))`。

#### 受控冗余
如果冗余确实存在（并且用户知晓其存在），但“传播更新”以确保其永远不会导致任何不一致的任务是由 DBMS 而非用户管理的，则 `Redundancy`（冗余）是受控的。不受控的冗余可能是个问题，但受控的冗余则不应成为问题。通常，数据库不应包含任何不受控的冗余。

#### 正确的
参见 `correctness`（正确性）。

#### 正确性
（指数据库）真正反映现实世界中存在事态的属性（更多讨论请参见 `relvar predicate`（关系变量谓词）下的示例）。与 `consistency`（一致性）对比。

### 关联名称
SQL 术语，指元组演算范围变量或此类变量的名称。

#### 覆盖
（指一组 FD）如果 `s1` 和 `s2` 是两组 FD，那么当且仅当 `s1` 所蕴含的每个 FD 都被 `s2` 所蕴含时，`s2` 才是 `s1` 的一个覆盖。注：一些作者以更强的含义使用 `cover`（覆盖）一词，指与某个给定集合等价的 FD 集合（参见 `equivalence`（等价））。

### 叉积 / 笛卡尔积
`Cartesian product`（笛卡尔积），参见 *q.v.*。

### D_UNION

#### 数据
命题的编码表示，按照惯例假定为真命题。

### 数据定义操作符
一种操作符，它 (a) 定义某个数据库对象，如基本 `relvar`（关系变量）、视图、快照或约束，或者 (b) 删除（丢弃）或更新此类定义；换句话说，它是更新目录的操作符。注：删除定义实际上也会导致相应的对象被删除（至少对用户而言是如此），并且通常用此类术语描述。例如，**Tutorial D** 提供了一个名为 `DROP CONSTRAINT` 的操作符（而非“删除约束定义”）。

### 示例
参见 `relvars S`、`P` 和 `SP` 的定义。其他例子可以是：(a) 为这些 `relvar` 之一添加属性的操作，(b) 为这些 `relvar` 定义约束的操作，或 (c) 删除任何此类定义的操作。（严格来说，第一个例子——向某个 `relvar`（比如 `relvar S`）“添加属性”——的效果是删除原始 `relvar` 并引入一个同名的新 `relvar`，同时以某种方式保留原始 `relvar` 的信息内容。如何实现这种效果的细节超出了本词典的范围。）

#### 数据独立性
改变数据库的物理或逻辑设计而无需在用户感知数据库的方式上做出相应更改的能力。术语 `physical data independence`（物理数据独立性）和 `logical data independence`（逻辑数据独立性）用于指代这两种情况。两者都涉及拥有两组定义及其间的映射，使得 (a) 如果物理设计发生变化，通过改变从物理设计到逻辑设计的映射来保持 `physical data independence`；(b) 如果逻辑设计发生变化，通过定义从旧逻辑设计到新逻辑设计的映射来保持 `logical data independence`。

### 数据操作操作符
宽泛地说，指不是 `data definition operator`（数据定义操作符）的操作符。然而，这种区分并非严格不变；事实上，很难找到一个最终不以某种方式“操作”数据的操作符（除非它是一个只读操作符；有些作者可能认为只有更新操作符才是真正“操作”数据的）。这个术语最好避免使用。

#### 数据模型
1.  对构成用户与之交互的抽象机器的数据结构、数据操作符等所做的抽象、自包含的逻辑定义（与 `implementation`（实现）对比）。2. 某特定企业的持久数据的模型（即逻辑数据库设计）。

### 示例
对于第二个定义，任何逻辑数据库设计都足以作为例子。至于第一个定义，最明显的例子是 `relational model`（关系模型）。有一个很好的类比可以帮助解释这两个定义之间的区别：第一种意义上的 `data model` 就像一种编程语言，其结构可用于解决许多具体问题，但其本身与任何此类具体问题并无直接联系；第二种意义上的 `data model` 就像用该语言编写的特定程序——它利用模型（即该术语的第一种意义）提供的设施来解决某个特定问题。注：我们可以通过说模型是用户必须知道的，而实现是用户不需要知道的，来总结第一种意义上的 `data model` 与其实现之间的区别。

有时用作逻辑数据库设计过程同义词的术语。
