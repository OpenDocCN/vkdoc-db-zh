## 引言

本词典收录了 900 多个条目，涉及关系数据模型所包含或由其使用而产生的问题、术语和概念。许多条目不仅包含定义本身，还提供了说明性的示例（有时不止一个）。关于这些定义，我已尽力使其清晰、精确且准确；它们基于我个人对材料的最佳理解，这一理解是在我涉足该领域近 40 年的时间里逐渐打磨形成的。

我想强调一点，正如标题所示，这是一本关于关系的词典。为此，我特意省略了许多与关系问题只有边缘联系的术语和概念（例如，支撑性类型理论的几乎所有细节，尤其是类型继承细节）。在很大程度上，我还省略了数据库技术中普遍存在、并非关系数据库所特有的各种主题（例如，安全性问题、日志、恢复和并发控制等等）。此外，我还省略了某些 SQL 术语和概念——尽管 SQL 本应是一种关系语言，但这些概念实际上并非关系性的（例如，`外连接`、`UNION ALL` 和通过游标进行更新）。尽管如此，我必须补充说明，我特意收录了一些非关系术语，以便澄清一个与普遍看法相反的事实：这些术语所对应的概念确实不是关系性的（`索引` 就是一个典型的例子）。

我还必须解释，这是一本有态度的词典。我坚信，关系模型是数据库技术正确且恰当的基础，并且在可预见的未来仍将如此，下文中的许多定义都反映了这一信念。正如我在《数据库实战：面向实践者的关系理论》（*Database in Depth: Relational Theory for Practitioners*，O'Reilly Media Inc.，2005）一书中所言：

> [这是]我的观点，关系模型坚如磐石，“正确”，并将持久。一百年后，我完全预期数据库系统仍将以科德的关系模型为基础。为什么？因为该模型的基础——即集合论和谓词逻辑——其本身同样是坚如磐石。特别是谓词逻辑的要素可追溯至两千多年前，至少远至亚里士多德（公元前 384–322 年）。

## 贯穿全书的示例

用于阐释定义的示例大部分基于熟悉的——甚至可以说是陈旧的——供应商-零件数据库。我为再次搬出这个老例子而道歉，但我相信，在多种不同的出版物中使用相同的例子对学习有益无害。以下是关系变量定义（如果你不知道什么是关系变量，那么请查阅该术语的词典条目！）：
```
VAR S BASE RELATION
{ S# S#, SNAME NAME, STATUS INTEGER, CITY CHAR }
KEY { S# } ;

VAR P BASE RELATION
{ P# P#, PNAME NAME, COLOR COLOR,
WEIGHT WEIGHT, CITY CHAR }
KEY { P# } ;

VAR SP BASE RELATION
{ S# S#, P# P#, QTY QTY }
KEY { S#, P# } ;
```

语义如下：

关系变量 `S` 代表签约供应商。每个供应商有一个供应商编号 (`S#`)，该编号对该供应商是唯一的；一个名称 (`SNAME`)，不一定唯一；一个状态值 (`STATUS`)；以及一个所在地 (`CITY`)。属性 `S#`、`SNAME`、`STATUS` 和 `CITY` 的类型分别是 `S#`、`NAME`、`INTEGER` 和 `CHAR`。

关系变量 `P` 代表零件种类。每种零件有一个零件编号 (`P#`)，它是唯一的；一个名称 (`PNAME`)；一种颜色 (`COLOR`)；一个重量 (`WEIGHT`)；以及一个存放该种零件的地点 (`CITY`)。属性 `P#`、`PNAME`、`COLOR`、`WEIGHT` 和 `CITY` 的类型分别是 `P#`、`NAME`、`COLOR`、`WEIGHT` 和 `CHAR`。

关系变量 `SP` 代表发货记录（它显示哪个供应商供应或发运了哪些零件）。每条发货记录有一个供应商编号 (`S#`)、一个零件编号 (`P#`) 和一个数量 (`QTY`)；对于给定的供应商和给定的零件，在任何给定时间最多只有一条发货记录。属性 `S#`、`P#` 和 `QTY` 的类型分别是 `S#`、`P#` 和 `QTY`。

## 字母排序

为进行字母排序，我遵循了以下规则：
1.  标点符号（括号、连字符、下划线等）视为空白。
2.  大写字母排在小写字母之前。
3.  数字排在字母之前。
4.  空白排在所有其他字符之前。
