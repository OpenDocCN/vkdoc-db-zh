#### 闭合定律
集合 `s` 在运算 "+" 和 "`*`" 下是闭合的；也就是说，对于 `s` 中的所有 `x` 和 `y`，表达式 `x + y` 和 `x * y` 的结果都是 `s` 中的一个元素。

#### 交换律
对于 `s` 中的所有 `x` 和 `y`，`x + y = y + x` 且 `x * y = y * x`。

#### 结合律
对于 `s` 中的所有 `x`, `y` 和 `z`，`x + (y + z) = (x + y) + z` 且 `x * (y * z) = (x * y) * z`。

#### 单位元定律
在集合 `s` 中存在元素 0 和 1，使得对于 `s` 中的所有 `x`，`x + 0 = x` 且 `x * 1 = x`。元素 0 和 1 分别被称为加法单位元和乘法单位元。

#### 逆元定律
对于 `s` 中的所有 `x`，存在元素 `-x` 以及（除非 `x = 0`）`1 / x` 在 `s` 中，使得 `x + (-x) = 0` 且 `x * (1 / x) = 1`。元素 `-x` 和 `1 / x` 分别被称为 `x` 的加法逆元和乘法逆元（在每种情况下）。表达式 `x + (-y)` 和 `x * (1 / y)` 通常缩写为 `x - y` 和 `x / y`。

#### 分配律
（“`*`” 对于 “`+`” 的分配律）：对于 `s` 中的所有 `x`, `y` 和 `z`，`x * (y + z) = (x * y) + (x * z)`。

请注意，并非所有代数都遵循上述所有定律。例如，在集合代数中，“`+`” 和 “`*`” 运算符分别表示并集和交集，但逆元定律不适用于这些运算。

#### 左不可约的 FD
给定一个 FD 集合 `s`，当且仅当无法在不改变 `s` 的闭包的前提下从 `f` 的决定因素中移除任何属性时，`s` 中的 FD `f`（相对于 `s`）是左不可约的。

### 示例
令 `s` 为关系变量 `SP` 所满足的所有 FD 的集合。那么 FD `{S#,P#} → {QTY}` 和 `{S#,P#,QTY} → {QTY}` 都出现在 `s` 中。在这两个 FD 中，第一个是左不可约的，但第二个不是。

#### 左结合性
令 `Op` 为一个二元运算符，并假设 `Op` 以中缀风格表示。那么当且仅当对于所有 `x`, `y` 和 `z`，`x Op y Op z = (x Op y) Op z` 时，`Op` 是左结合的；类似地，当且仅当对于所有 `x`, `y` 和 `z`，`x Op y Op z = x Op (y Op z)` 时，它是右结合的。

*注意*：在数学和逻辑中，左结合性和右结合性都归结为通常理解的结合性（参见 `associative laws`）。然而，在计算中情况不一定如此。例如，考虑如果 `x`, `y` 和 `z` 是数字且 `Op` 是 “`+`” 时会发生什么。在这种情况下，表达式 `(x + y) + z` 和 `x + (y + z)` 很可能产生不同的结果；甚至可能一个导致溢出而另一个不会。

#### 多重赋值
一种允许同时执行多个独立赋值，且仅在所有独立赋值完全执行后才进行完整性检查的操作。注意，多重赋值隐含地涉及许多其他操作——例如，更新某个连接视图，或以引发补偿操作（参见 `compensating action`）的方式更新某个关系变量。

### 示例
以下“双删除”在逻辑上是一个多重赋值操作：
```sql
DELETE S WHERE S# = S#('S1') ,
DELETE SP WHERE S# = S#('S1') ;
```
注意第一个 `DELETE` 后的逗号分隔符，这在语法上表明整个语句尚未结束。

还需注意，如果两个独立的 `DELETE` 以相反的顺序指定，语义将完全保持不变。其扩展形式为：
```sql
S := S WHERE NOT ( S# = S#('S1') ) ,
SP := SP WHERE NOT ( S# = S#('S1') ) ;
```

#### 多值依赖
函数依赖概念的推广（每个函数依赖都是多值依赖，但并非所有多值依赖都是函数依赖）。

多值依赖是一个表达式——实际上是一个命题——形式为 *A* →→ *B*，其中 *A* 和 *B* 都是同一关系变量 *R* 的标题的子集，整个表达式读作“*B* 多值依赖于 *A*”，或“*A* 多值确定 *B*”，或更简单地，“*A* 双箭头 *B*”。

设 *C* 是同一关系变量 *R* 标题的子集，使得 *A*、*B* 和 *C* 的集合论并集等于该标题。设 *AB* 表示 *A* 和 *B* 的集合论并集，*AC* 同理。那么关系变量 *R* 满足多值依赖（MVD）*A* →→ *B* 和 *A* →→ *C*——等价地说，这些 MVD 在关系变量 *R* 中成立——当且仅当 *R* 满足连接依赖 ∗{ *AB,AC*}。

> **注意**：多值依赖也称为多值依赖性。观察 MVD *A* →→ *B* 中的 *A* 和 *B* 具体是属性集。然而在非正式场合，通常（尽管严格来说不正确）会说 *B* 中的属性本身多值依赖于 *A*，而不是说包含这些属性的集合 *B* 多值依赖于 *A*。同样，通常（尽管严格来说不正确）会说 *A* 中的属性本身多值确定 *B*，而不是说包含这些属性的集合 *A* 多值确定 *B*。注意，根据定义，如果关系变量 *R* 满足 MVD *A* →→ *B* 和 *A* →→ *C*，那么如果它包含元组 < *a,b1,c1*> 和 < *a,b2,c2*>，它也必然包含元组 < *a,b1,c2*> 和 < *a,b2,c1*>。进一步注意，同样直接从定义可知，关系变量 *R* 可以无损分解为其在 *AB* 和 *AC* 上的投影，当且仅当它满足 MVD *A* →→ *B* 和 *A* →→ *C*（这个定理是 Heath 定理的强化形式，是 Fagin 提出的众多定理之一）。最后注意，从形式化的角度看，MVD *A* →→ *B* 只是一个命题，根据定义，它要么被满足，要么不被满足（即求值为 `TRUE` 或 `FALSE`）；然而非正式地，通常将 *A* →→ *B* 定义为一个 MVD 仅当它实际被相关关系变量满足时。但该定义没有方式表达一个给定的关系变量不满足或违反某个 MVD（因为根据该定义，一个未被满足的 MVD 就不是 MVD！）。

#### 与非
在逻辑中，一个二元连接词（也称为谢弗竖线，通常写作垂直线“|”）；如果 *p* 和 *q* 是谓词，那么 ( *p*)|( *q*) 也是一个谓词。设 ( *ip*)|( *iq*) 是该谓词的一次调用（其中 *ip* 和 *iq* 分别是 *p* 和 *q* 的调用）。那么该调用 ( *ip*)|( *iq*) 求值为 `FALSE`，当且仅当 *ip* 和 *iq* 都求值为 `TRUE`。（换句话说，( *p*)|( *q*) 等价于 `NOT(( p) AND ( q))`。）

> **注意**：在表达式 ( *p*)|( *q*) 中包围 *p* 和 *q* 的括号在实践中可能不需要。另外，如上定义的 `NAND` 是一个逻辑运算符；然而，代数 `A`（参见 `A`）包含一个也叫 `NAND` 的运算符，根据定义它是一个代数运算符（它基本上是“连接的补”）。

#### 自然连接
1.  （二元情况）设关系 *r1* 和 *r2* 满足同名属性属于相同类型（换句话说，*r1* 和 *r2* 是可连接的，参见 `joinable`）。那么 *r1* 和 *r2* 的自然连接 *r1* `JOIN` *r2* 是一个关系，其标题是 *r1* 和 *r2* 标题的集合论并集，其主体是所有元组 *t* 的集合，其中 *t* 是来自 *r1* 的一个元组和来自 *r2* 的一个元组的集合论并集。
2.  （n 元情况）设关系 *r1, r2, . . ., rn*（*n* ≥ 0）满足同名属性属于相同类型（换句话说，*r1, r2, . . ., rn* 是可连接的，参见 `joinable`）。那么自然连接 `JOIN { r1,r2,. . .,rn}` 定义如下：如果 *n* = 0，结果是 `TABLE_DEE`；如果 *n* = 1，结果是 *r1*；否则，从集合 *r1, r2, . . ., rn* 中任意选择两个不同的关系，并用它们的（二元）自然连接替换它们，重复此过程，直到集合中只剩下一个关系 *r*，该关系即为最终结果。

### 示例
表达式 `S JOIN SP` 表示关系变量 `S` 和 `SP` 的当前值对应关系的自然连接。该连接是一个类型为 `RELATION {S# S#, SNAME NAME, STATUS INTEGER, CITY CHAR, P# P#, QTY QTY}` 的关系。此外，如果关系变量 `S` 和 `SP` 的当前值分别是 *s* 和 *sp*，则该关系的主体由所有形式为 < *s#,sn,st,sc,p#,q*> 的元组组成，其中元组 < *s#,sn,st,sc*> 出现在 *s* 中，且元组 < *s#,p#,q*> 出现在 *sp* 中。

#### 否定
如果 *p* 是一个谓词，其否定 `NOT( p)` 也是一个谓词。设 `NOT( ip)` 是该谓词的一次调用（其中 *ip* 是 *p* 的调用）。那么该调用 `NOT( ip)` 求值为 `TRUE`，当且仅当 *ip* 求值为 `FALSE`。

> **注意**：在表达式 `NOT( p)` 中包围 *p* 的括号在实践中可能不需要。



#### 失败即否定
这是“封闭世界假设”的一个密切相关概念；粗略地说，它意味着如果一个给定的元组本应（至少在原理上）出现在某个查询的结果中但实际没有出现，则该元组所代表的命题为假。

#### 负的

#### NF²
NF 平方；NFNF（“非第一范式”）的缩写。粗略地说，一个 NF² 关系变量是至少包含一个关系值属性的关系变量。

然而，该术语已被弃用，因为它基于对第一范式概念有缺陷的理解。此外，NF² 概念通常被认为包含了对传统关系运算符的某些扩展，这些扩展不仅仅是简写形式，因此没有被包含（或不需要）在关系模型中。

#### 零元
（*指一个运算符*）没有操作数；即，其定义不涉及任何参数。*注意*：一个运算符在语法上可能看起来是零元的，但由于其使用了可能被称为隐藏操作数的东西（例如系统时钟），它每次调用的效果可能并不限于相同。确实，这种“零元”运算符——当然，无论如何它们并非真正的零元——是常见的情况。

**示例**：许多语言提供一个运算符来生成随机（或者更确切地说，“伪随机”）数。这样的运算符有效地拥有一个隐藏操作数：即前一次调用返回的随机数。

#### 无损分解
用关系变量 *R* 的投影 *R1*、*R2*、……、*Rn* 来替换它，使得 (a) *R1*、*R2*、……、*Rn* 的连接保证等于 *R*，并且通常也使得 (b) *R1*、*R2*、……、*Rn* 中的每一个对于提供该保证都是必需的（即这些投影中没有一个在连接中是冗余的）。请注意，对于任何给定的关系变量 *R*，一个总是可用的“无损分解”是“替换” *R* 为相应的恒等投影（恒等投影，q.v.）。

**示例**：一个可能应用于供应商-零件数据库的无损分解是，用其在 {供应商号, 供应商名} 和 {供应商号, 状态, 城市} 上的投影来替换关系变量 S。然后可以通过将这两个投影重新连接在一起来重构关系变量 S。

（*指一个 FD、JD 或 MVD*）非平凡的。*参见* 平凡 FD; 平凡 JD; 平凡 MVD。

#### 或非
在逻辑中，一种二元连接词（也称为皮尔士箭头，通常写作向下的箭头 “↓”）；如果 *p* 和 *q* 是谓词，那么 ( *p*)↓( *q*) 也是一个谓词。令 ( *ip*)↓( *iq*) 为该谓词的一次调用（其中 *ip* 和 *iq* 分别是 *p* 和 *q* 的调用）。那么该调用 ( *ip*)↓( *iq*) 的求值结果为 TRUE 当且仅当 *ip* 和 *iq* 的求值结果均为 FALSE。（换句话说，( *p*)↓( *q*) 等价于 NOT(( *p*) OR ( *q*))。）*注意*：在表达式 ( *p*)↓( *q*) 中包围 *p* 和 *q* 的圆括号在实践中可能不需要。此外，按上述定义的“或非”是一个逻辑运算符；然而，代数 **A**（q.v.）包含一个名为“或非”的运算符——根据定义——它是一个代数运算符（本质上是“并集的补集”，不过这里的“并集”指的不是关系运算符中的那个名字，而是该运算符的一种广义形式）。

#### 范式
1. （*通用*）规范形式。2. （*指一个关系变量*）*参见* 第一范式；第二范式；等等。*注意*：最重要的关系范式是 BCNF 和 5NF（以及 6NF）；其他的主要是具有历史意义。

#### 规范化
使用无损分解的原则，用某个关系变量的某些投影来替换给定的关系变量，使得这些投影的连接保证等于原始关系变量。因此，请注意，投影是分解运算符，而连接是重组运算符，这与规范化过程（通常按后者理解）相关。规范化的目标是减少冗余（q.v.），从而消除可能发生的某些更新异常（q.v.）。

#### 规范化原则
一个不属于 5NF 的关系变量应被分解为一组 5NF 投影；原始关系变量应能通过将这些投影重新连接在一起来重构；分解过程应保持依赖关系；每个投影在重构过程中都应是必需的。

#### 规范化的
关系，因而也是关系变量的一种属性，根据该属性，所讨论的关系或关系变量中的每个元组对其每个属性都恰好包含一个（适当类型的）值。实际上，一个对其每个属性不包含恰好一个适当类型值的“元组”首先就不是一个元组；同样，一个包含这种“元组”的“关系”也不是关系，而其值是这种“关系”的“关系变量”也不是关系变量。换句话说，根据定义，所有关系，因而所有关系变量，都是规范化的。*注意*：关系变量尤其等价地被称为处于第一范式，1NF（q.v.）；也就是说，一个规范化的关系变量就是一个处于 1NF 的关系变量，这意味着它只是一个关系变量。然而，术语 *规范化的* 经常（尽管不准确地）被用来指代高于第一范式的某些范式（特别是 BCNF 或 5NF）。

#### 规范化的关系变量
一个关系变量。（根据定义，关系变量总是规范化的，其意义在于它们至少处于第一范式。）

#### 非

#### 一对多对应
严格来说，一种配对两个集合`s1`和`s2`（`s1`和`s2`不一定不同）的规则，使得`s1`的每个元素对应`s2`的至少一个元素，且`s2`的每个元素对应`s1`的恰好一个元素；等价地，指该配对本身。
然而，该术语常被宽松地使用，表示一种配对，其中：(a) `s1`的每个元素对应任意数量的`s2`元素（可能为零），且`s2`的每个元素对应`s1`的恰好一个元素；或 (b) `s1`的每个元素对应`s2`的至少一个元素，且`s2`的每个元素对应`s1`的至多一个元素；或 (c) `s1`的每个元素对应任意数量的`s2`元素（可能为零），且`s2`的每个元素对应`s1`的至多一个元素。除非预期含义明确，否则最好避免使用该术语。
**示例**（仅严格意义）：令`s1`为所有非负数的集合，`s2`为所有数的集合。那么，将非负数`x`与其平方根±√`x`配对，是从`s1`到`s2`的一个多对一对应。

#### 一一对应
严格来说，一种配对两个集合`s1`和`s2`（`s1`和`s2`不一定不同）的规则，使得`s1`的每个元素对应`s2`的恰好一个元素，且`s2`的每个元素对应`s1`的恰好一个元素（换句话说，是一个双射）；等价地，指该配对本身。
然而，该术语常被宽松地使用，表示一种配对，其中：(a) `s1`的每个元素对应至多一个`s2`元素，且`s2`的每个元素对应`s1`的恰好一个元素；或 (b) `s1`的每个元素对应恰好一个`s2`元素，且`s2`的每个元素对应`s1`的至多一个元素；或 (c) `s1`的每个元素对应至多一个`s2`元素，且`s2`的每个元素对应`s1`的至多一个元素。除非预期含义明确，否则最好避免使用该术语。
**示例**（仅严格意义）：令`s`为所有整数的集合。那么，将元素`x`与其后继`x`+1 配对，是从`s`到其自身的一一对应；将元素`x`与其前驱`x`-1 配对也是一一对应。

#### 开放世界假设
宽泛地说，假设数据库陈述或隐含的一切为真，而其他一切未知。
更精确地说，开放世界假设（OWA）——通常被拒绝，而倾向于封闭世界假设——指出：(a) 如果一个给定元组在给定时间出现在一个给定的关系变量中，则该元组所代表的命题在该时间为真；(b) 如果一个给定元组在该时间可以出现在该关系变量中但没有出现，则该元组所代表的命题在该时间可能为真也可能为假。换句话说，在任何给定时间，该关系变量包含且仅包含那些已知对应于真命题的元组——即，在该时间已知求值为 TRUE 的关系变量谓词的调用或完全实例化。
*注意*：所讨论的关系变量不必是基本关系变量，这就是为什么原始宽泛描述中出现“或隐含”这个短语。还要注意，该定义是用关系变量专门表述的，但一个精确类似的定义也适用于关系（参见`extension`，第三个定义）。最后注意，OWA 被强烈反对，因为它似乎直接导致需要支持三值逻辑。

#### 有序对
一种组合，通常记为<*x, y*>，由恰好两个元素*x*和*y*组成，其中*x*是第一元素，*y*是第二元素。有序对<*x1, y1*>和<*x2, y2*>相等当且仅当*x1* = *x2*且*y1* = *y2*（因此，一般来说<*x, y*> ≠ <*y, x*>）。形式上，<*x, y*>被定义为等于集合{{*x*}, {*x, y*}}；请注意，该集合中的一个元素确定了构成有序对的元素，另一个元素则确定了这些元素中哪一个排在第一位。*注*：这个定义可以轻松扩展到有序三元组、有序四元组，...，以及更一般地，对于任意*n* > 1 的有序*n*-元组。

#### 有序元组

#### 排序
为关系的属性施加从左到右的序列，特别是为元组施加从上到下的序列的过程，或该过程的结果，以便将相关关系中的数据转移出关系上下文，进入一个依赖此类序列的环境。请求这种排序（或称“定序”）的运算符具有重要的实用价值，但它们不是关系运算符，因为其结果不是关系。*注*：思考此类运算符的一个便捷方式是，它们将关系转换为表（因为表与关系不同，其列具有从左到右的顺序，行具有从上到下的顺序）。

*示例*：提供此功能的 SQL 运算符是`SELECT`（用于从左到右的列序）和`ORDER BY`（用于从上到下的行序）。

*注*：顺便值得指出，`ORDER BY`不是函数——也就是说，给定`ORDER BY`调用的结果通常是不确定的（例如，考虑`ORDER BY CITY`对图 1 所示的、作为`relvar S`当前值的关系的影响）。相比之下，关系代数的运算符确实都是函数。（顺便说一句，这些运算符的 SQL 模拟并不总是如此，因为 SQL 允许即使`v1`和`v2`不是相同值，`v1 = v2`形式的比较也返回`TRUE`。）`ORDER BY`在另一个方面也不同寻常。具体来说，它产生一个元组序列作为其结果，然而运算符“<”和“>”对于元组是明确定义不适用的（参见`元组比较`）。

#### 排序（数学）
参见`偏序`；`全序`。

#### 序数类型
一种定义了`全序`（参见*q.v.*）的类型。设`T`为这样一种类型。那么，通常与类型`T`相关联的运算符将包括：(a) 零元`FIRST`和`LAST`运算符，分别根据该全序返回类型`T`的第一个值和最后一个值；(b) 一元`NEXT`和`PRIOR`运算符，给定类型`T`的一个值`v`，分别返回在类型`T`中紧随`v`之后的值和紧接`v`之前的值，同样根据该全序。

*示例*：类型`INTEGER`是一个序数类型。`NEXT`和`PRIOR`运算符基本上分别就是`加一`和`减一`。`FIRST`和`LAST`运算符通常不直接支持，但返回最小和最大整数的选择器调用足以满足需求。

作为反例，假设系统还支持一个类型`POINT`，可能是用户定义的，表示二维空间中的几何点。那么类型`POINT`大概不会是一个序数类型，因为一个点以某种方式小于或大于另一个点的概念没有意义。

#### 正交分解
将某个给定的`关系变量`分解为限制条件，使得这些限制条件满足`正交设计原则`。

*示例*：假设我们用两个关系变量`LP`和`HP`替换`关系变量 P`，`LP`包含重量小于或等于 17.0 的零件的元组，`HP`包含重量大于 17.0 的零件的元组；那么该分解就是正交的。相比之下，假设`关系变量 HP`被定义为包含重量大于*或等于*17.0 的零件的元组；那么该分解就不是正交的，因为重量等于 17.0 的零件的元组在逻辑上应同时属于`LP`和`HP`。

#### 重载
为两个或多个不同的运算符使用相同的名称。这些运算符最好具有相似的语义。

*注*：重载也被更具体地称为重载多态；它也被称为特设多态。在继承上下文中，该术语有时也用来表示`包含多态`（参见*q.v.*）；但是，这种特定用法不被推荐，因为它基于对子类型和超类型本质的有缺陷的理解。进一步的细节超出了本词典的范围。

*示例*：`UNION`被重载，因为该名称（即名称）被用于表示关系并集和元组并集（而在 SQL 中它也被用于表示并集加上，尽管奇怪的是它本身不表示包并集）。同样，“=”被重载，因为它适用于每种类型的值（即，存在用于整数的“=”运算符，另一个用于供应商编号，另一个用于类型为`RELATION {S# S#, P# P#, QTY QTY}`的关系，等等）。

#### 覆盖

*示例*：假设存在一个名为`LOG`的运算符（可能是内置的），它返回自然对数。那么可能可以用一个返回以十为底的对数的运算符来覆盖它。

#### 偏序

设 `s` 是一个集合。那么，`s` 上的一个偏序是一个二元真值运算符，通常记为 "≤"，使得对于 `s` 中的所有 `x`、`y` 和 `z`：
- `x ≤ y` 或 `y ≤ x`（或两者，或可能两者都不）；
- `x ≤ x`（自反性）；
- 若 `x ≤ y` 且 `y ≤ z`，则 `x ≤ z`（传递性）；
- 若 `x ≤ y` 且 `y ≤ x`，则 `x = y`（反对称性）。

*对比***全序**。

### 示例

#### 前束范式
当且仅当所有量词都出现在开头时，一个谓词才处于前束范式（`PNF`）；更精确地说，当且仅当（a）它无量词或（b）它的形式是 `EXISTS x ( p)` 或 `FORALL x ( p)`，且其中的 `p` 本身也处于 `PNF` 时，它才处于 `PNF`。因此，一个 `PNF` 谓词的形式是
`Q1 x1 ( Q2 x2 ( ... ( Qn xn ( p ) ) ) )`，其中 (a) `n >= 0`，(b) 每个 `Qi`（`i` = 1,2, . . . , `n`）是 `EXISTS` 或 `FORALL`，以及 (c) 谓词 `p`——有时称为主式——是无量词的。

`示例`：考虑以下元组演算查询（“获取至少供应一个红色零件的供应商”）：
```
SX RANGES OVER { S } ;
SPX RANGES OVER { SP } ;
PX RANGES OVER { P } ;
SX WHERE
EXISTS PX ( PX.COLOR = 'Red' AND
EXISTS SPX ( SPX.S# = SX.S# AND
SPX.P# = PX.P# ) )
```
这里的 `WHERE` 子句中的谓词并不处于前束范式。

然而，下面是该查询的一个语义等价表述，其中谓词处于前束范式：
```
SX WHERE
EXISTS PX ( EXISTS SPX ( PX.COLOR = 'Red' AND
SPX.S# = SX.S# AND
SPX.P# = PX.P# ) )
```
前束范式在逻辑上并不比其他形式更正确，但稍加练习，它往往会成为最容易书写的形式。

#### 主域
设 `R` 为基础关系变量；设 `R` 有一个主键 `K`；设 `K` 是简单键（*参见* 简单键）；并且设 `K` 定义在域（即类型）`D` 上。那么，且仅当此时，`D` 才是一个主域。*注意*：由于 (a) 它明显违反了可替换性原则，并且 (b) 它依赖于一个从逻辑角度难以证明其合理性的概念——主键（*参见* 主键），主域概念被强烈反对。无论如何，这个术语（或许幸好）并不常用；我们在此提及它主要是出于历史原因。

`示例`：在供应商-零件数据库的情况下，唯一的主域是 `S#` 和 `P#`。

#### 主键
一个因某种原因被挑选出来进行特殊语法处理的候选键。虽然一个给定的关系变量可以有任意数量的候选键，但它最多只能有一个主键。然而，对于一个给定的关系变量，是否要选择某个候选键作为主键，以及如果选择，选哪一个，本质上是心理层面的问题，超出了关系模型本身的范围。*注意*：关系模型最初坚持基础关系变量（至少）应该始终有一个主键。它还坚持外键应专门引用主键。然而，这些规则从未有好的逻辑理由，而且无论如何，适用于基础关系变量而不适用于其他类型关系的规则本身就相当可疑（因为它们违反了可替换性原则）；因此，主键概念可以被舍弃而不会造成严重损失。我们在此提及它主要是出于历史原因。

#### 原语运算符
粗略地说，一个不是用其他运算符定义的运算符。更精确地说，设 `s` 是一个运算符集合。设 `Op` 是 `s` 中的一个运算符，它可以用 `s` 中的其他运算符来定义；将 `Op` 从 `s` 中移除，并重复此步骤直到无法再重复为止。剩下的就是一个相对于 `s` 的原语运算符集合。请注意，对于给定集合 `s`，其原语运算符集合不一定是唯一的。

#### 谨慎设计原则
在设计形式系统（包括数据库、`DBMS`、数据库语言和许多其他系统）时的一个指导原则。可以这样表述：当在选项 `A` 和 `B` 之间做设计选择时，如果 `A` 向上兼容 `B`，并且 `B` 的全部影响尚不明确，那么谨慎的决定是选择 `A`。选择 `A` 允许未来在需要时将设计“开放”给 `B`。相反，选择 `B` 则禁止未来将设计“收窄”回 `A`，即使这种收窄变得可取（即，如果一开始就明确 `B` 是个坏决定）。

`示例`：`SQL` 的设计者曾面临禁止重复行（*选项 A*）和允许重复行（*选项 B*）之间的选择。谨慎的决定本应是禁止它们（*选项 A*）；如果未来证明对此类支持有明确需求，他们本可以随后支持它们。不幸的是，设计者选择了允许它们（*选项 B*）。当然，这个决定后来被证明是非常糟糕的，但现在 `SQL` 已经无法以兼容的方式回到 *选项 A*。*注意*：正如这个例子所示，谨慎设计原则有助于避免这样一种情况：语言（或 `DBMS`，或数据库，或任何正在设计的东西）提供了某些选项，用户却必须被明确告知不要使用这些选项。

#### 不可区分者同一性原则
该原则指出，如果无法以任何方式区分两个对象，那么就不存在两个对象，而只有一个。或者等价地说：每个对象都有其唯一身份。*注意*：在关系模型中，这种唯一身份与其他所有东西一样表示——即，通过属性值（*参见* 信息原则）——并且由此产生了许多好处。还要注意，不可区分性与可替换性之间存在逻辑差异——两个对象可能是可替换的但又是可区分的（例如，想想两枚一分硬币）。最后请注意，此处的术语 `对象` 并非在其 `OO`（面向对象）意义上使用。

#### ****projection（投影）**
设 `r` 为一个关系，且 { `X`} 是 `r` 的标题的一个子集。
那么，`r` 在 { `X`} 上的投影，`r{` `X` `}`，是一个标题为 { `X`} 的关系，其体由所有满足以下条件的元组 `x` 组成：存在 `r` 中的某个元组 `t`，其 `X` 值为 `x`。*另见* tuple projection（元组投影）。

**示例*：表达式 `S{STATUS,CITY}` 表示对关系变量 `S` 当前值的关系进行的投影。该投影是一个类型为 `RELATION {STATUS INTEGER， CITY CHAR}` 的关系，包含所有形如 < `st，sc`> 的可能元组（且无其他元组），使得存在某个供应商编号 `s#` 和某个名称 `sn`，使得元组 < `s#,sn,st,sc`> 出现在关系变量 `S` 的当前值中。给定图 1 的样本值，该结果的基数为 4。*注*：为了用户友好性，`Tutorial D` 允许使用要移除的属性而非要保留的属性来表达投影；因此，例如，投影 `S{STATUS,CITY}` 可以等价地表达为 `S{ALL BUT S#,SNAME}`。类似的说明也适用于其他几个 `Tutorial D` 结构——`KEY`、`GROUP`、`SUMMARIZE` 等。

#### ****projection-join normal form（投影-连接范式）**
第五范式。其替代名称，也是原始名称，*投影-join normal form* 源于这样一个事实：5NF 是关于投影和连接（按这些操作符的经典理解）的“那个”范式。

#### ****proof（证明）**
( *逻辑学*) 通常指在某个逻辑系统中，由一系列语句组成的序列，这些语句共同确立某个语句是某些给定语句的逻辑推论；如果给定语句为真，则推论语句也为真。直接证明是一个序列，其中每个语句要么是公理，要么可以通过系统的推理规则从序列中前面的语句推导出来；最后一个语句是定理。公理是只有一个语句的直接证明的定理。间接证明，也称为 *reductio ad absurdum*（归谬法）证明，是一个语句序列，它通过显示其否定会导致矛盾来共同确立某个语句为定理。

#### ****proper inclusion（真包含）**
集合 `s1` 真包含集合 `s2`（“`s1` ⊃ `s2`”）当且仅当它是 `s2` 的真超集；集合 `s2` 被集合 `s1` 真包含（“`s2` ⊂ `s1`”）当且仅当它是 `s1` 的真子集。

#### ****proper subkey（真子键）**
一个不是键的子键（即，键的真子集）。

#### ****proper subset（真子集）**
集合 `s2` 是集合 `s1` 的真子集（“`s2` ⊂ `s1`”）当且仅当它是 `s1` 的子集且 `s1` 和 `s2` 不相同。

#### ****proper superkey（真超键）**
一个不是键的超键（即，不具有不可约性的超键）；宽泛地说，是键的真超集。

#### ****proper superset（真超集）**
集合 `s1` 是集合 `s2` 的真超集（“`s1` ⊃ `s2`”）当且仅当它是 `s2` 的超集且 `s1` 和 `s2` 不相同。

#### ****property（属性）**
属于另一事物的事物。*注*：人们经常建议，“感兴趣的属性”与基本关系变量中的属性之间应该存在一一对应关系。然而，考虑到“感兴趣的属性”这个术语没有精确的定义，这种建议很难成立。（当然，“属性”这个术语本身也是如此。）

#### ****proposition（命题）**
0 元谓词；一个没有参数（即，没有自由变量）的谓词；一个陈述性语句（逻辑意义上的，非编程语言意义上的）；因此，是一个明确地评估为真或假的东西。*注*：严格来说，如果 `S` 是一个陈述性语句，相应的命题并不是 `S` 本身——而是 `S` 所作的断言。例如，考虑语句“The sun is a star”和 *“Le soleil est une étoile*。” 显然，这里有两个不同的语句；然而，它们都表示同一个命题。不过，在非正式场合（有时在更正式的场合也是如此）通常忽略这种细微区别。

**示例*：
1.  太阳是一颗恒星。
2.  海王星是一颗恒星。
3.  所有政客都是腐败的。
4.  供应商 S1 签有合同，名为 Smith，状态为 20，且位于巴黎。
5.  存在一个城市 `c`，使得存在一个供应商编号 `s#`，使得供应商编号为 `s#` 的供应商位于城市 `c`。注意，这最后一个例子中有两个变量，`s#` 和 `c`；然而，这些变量是绑定的，而非自由的，整个例子仍然明确地评估为真或假（即至少有一个供应商位于至少一个城市的情况成立或不成立）。
6.  设 `p` 为任意谓词。如果 `p` 的每个参数都被量化或被某个参数替换，那么结果就是一个命题。例如，给定谓词“供应商编号为 `s#` 的供应商名为 `sn，` 状态为 `st，` 且位于城市 `sc，` ” 语句“存在一个城市 `sc`，使得存在一个供应商编号 `s#`，使得供应商编号为 `s#` 的供应商名为 Smith，状态为 20，且位于城市 `sc`”是一个命题。
7.  作为反例，表达式“`x` > 0 OR TRUE”不是一个命题（因为它涉及一个参数 `x`），即使它确实明确地评估为真。换句话说，虽然命题总是明确地评估为真或假，但并非所有明确地评估为真或假的东西都是命题。事实上，一个有用的非正式测试来检查某个语句 `S` 是否为命题如下：`S` 是一个命题当且仅当“`*S* 是真的吗？`”在自然语言中是一个结构良好的问题。

#### ****propositional calculus（命题演算）**
一个可靠、完备且可判定的正式系统，涉及命题和连接词，以及可以使用这些命题和连接词进行的推理。*对比* predicate calculus（谓词演算）。

#### ****propositional constant（命题常项）**
同命题。

#### ****propositional expression（命题表达式）**
表示命题的表达式；即，涉及命题常项、命题变量、连接词和括号的表达式。*注*：逻辑学教材不常用这个术语，通常更喜欢使用 *propositional form*（命题形式，参见）这个术语（如果他们确实使用任何术语来表达这个概念的话）。

**示例*：如果 `p` 和 `q` 是命题变量，那么 `p，` `q，` 合取式 ( `p`) AND ( `q`)， 析取式 ( `p`) OR ( `q`)，以及否定式 NOT( `p`) 都是命题表达式。

#### ****propositional form（命题形式）**
同命题表达式。

#### **propositional function（命题函数）**
同谓词。

#### **propositional logic（命题逻辑）**
同命题演算。
