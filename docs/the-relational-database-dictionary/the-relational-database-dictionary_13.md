# 关系数据库词典（扩展版）

#### 布尔代数

一个集合 `s` 以及其上的运算符 `<=`、`~`、`+` 和 `*` 构成一个布尔代数，当且仅当以下三个条件全部满足：(a) `0` 是 `+` 的恒等元；(b) `1` 是 `*` 的恒等元；(c) 对于 `s` 中的所有元素 `x`，都有 `0 <= x <= 1`。此时，集合 `s` 与运算符 `<=`、`~`、`+`、`*` 的组合就是一个布尔代数。

注意：尽管在此上下文中它们通常分别被称为加法和乘法，但必须明确理解 `+` 和 `*` 不一定是传统算术中同名的运算符。

### 示例（第二种定义）

设 `s` 为一个任意集合；设 `p` 为 `s` 的幂集；并设 `<=`、`~`、`+`、`*` 分别表示集合包含、集合补集、集合并集和集合交集。那么，刚刚定义的幂集 `p` 与运算符 `<=`、`~`、`+`、`*` 的组合就是一个布尔代数，其中空集和集合 `s` 本身分别充当所需的加法恒等元和乘法恒等元。（换句话说，熟悉的集合代数实际上就是一种布尔代数。）

#### 约束变量

在逻辑中，指一个变量——更准确地说，是在某个谓词内的一个变量引用的出现——它要么 (a) 出现在明确指定了该变量的量词的作用域内，要么 (b) 本身就是那个明确的规格说明。（此处“变量”一词是在逻辑意义上使用的，而非编程语言意义上的。）

对比：自由变量。

### 示例

令符号 `x` 和 `y` 表示整数。那么下面两个表达式都是谓词，并且 `x` 在每个谓词中都作为约束变量出现了两次：

```
EXISTS x ( x > 3 )
```

```
EXISTS x ( x > 3 ) AND y < 7
```

第一个谓词实际上是一个命题，其含义是“存在一个整数 `x` 使得 `x` 大于三”（这个命题的求值结果当然是 `TRUE`）。相比之下，第二个谓词不是一个命题，因为它除了两个约束变量外，还涉及一个自由变量（即 `y`）；因此，它没有真值。

转向一个数据库示例，下面是一个在供应商-零件数据库上的查询（“获取供应至少一种零件的供应商”），用元组演算（*参见*）表示：

```
S WHERE EXISTS SP ( SP.S# = S.S# )
```

关键字 `WHERE` 之后的布尔表达式是一个谓词，该谓词中对 `SP` 的引用是约束的（相比之下，对 `S` 的引用是自由的）。但请注意，在此特定示例中，符号 `S` 和 `SP` 不仅表示逻辑意义上的变量，也表示传统编程语言意义上的变量——但这只是因为我们采用了一定的技巧。下面是同一示例的扩展版本，应该有助于澄清问题：

```
SX RANGES OVER { S } ;
SPX RANGES OVER { SP } ;
SX WHERE EXISTS SPX ( SPX.S# = SX.S# )
```

这里，`SX` 和 `SPX` 被显式声明为逻辑意义上的变量，分别遍历（当前的）关系变量 `S` 和 `SP` 的值。

现在，对 `SPX` 的引用是约束的，而对 `SX` 的引用是自由的（在关键字 `WHERE` 之后的谓词中，两种情况下皆如此）。

实际上，在第一个版本的示例中发生的是，我们诉诸了一条语法规则，该规则允许使用关系变量名来表示一个隐式定义的、遍历（当前）同名关系变量值的范围变量。

注意：`SQL` 包含了一条完全类似的规则。

## Boyce/Codd 范式

关系变量 `R` 是 Boyce/Codd 范式（`BCNF`），当且仅当 `R` 所满足的每一个非平凡 `FD` 都被 `R` 的某个超键所蕴含；等价地，当且仅当对于 `R` 所满足的每一个非平凡 `FD A -> B`，`A` 都是 `R` 的一个超键。每一个 `BCNF` 关系变量都是 `3NF` 的。

注意：`BCNF` 是关于 `FD` 的“那个”范式。此外，尽管处于 `BCNF` 显然不排除也处于更高一级的范式（`4NF`），但术语 `BCNF` 常被宽松地用于指代处于 `BCNF` 但不处于 `4NF` 的关系变量。

### 示例

关于范式，展示一个反例通常比展示一个例子本身更具启发性。因此，假设关系变量 `SP` 有一个额外的属性 `SNAME`，表示相应供应商的名字；同时假设供应商名字是唯一的（即，任何时候都不会有两个供应商同名）。

这个修改后的 `SP` 版本有两个键，`{S#,P#}` 和 `{SNAME,P#}`，并且所有标题的子集——特别是 `{QTY}`——都（当然）函数依赖于这两个键。然而，该关系变量也满足 `FD` `{S#} -> {SNAME}` 和 `{SNAME} -> {S#}`；这些 `FD` 显然不是平凡的，也不是“从超键指出的箭头”，因此该关系变量不处于 `BCNF`（尽管它处于 `3NF`）。

#### 演算

1.  广义上，指形式计算系统（拉丁语 *calculus* 意为小石头，可能用于计数或其他计算形式）。2. 特指关系演算（如果上下文要求）。

#### 候选键

松散地说，是唯一标识符。更精确地说，设 `K` 是关系变量 `R` 标题的一个子集；那么 `K` 是 `R` 的一个候选键（简称键），当且仅当 (a) `R` 的任何可能值都不包含两个在 `K` 上值不同的不同元组（唯一性性质），同时 (b) `K` 的任何真子集都不能满足相同的条件（不可约性性质）。注意，每个关系变量，无论是基本的还是派生的，都至少有一个键。还需注意，根据定义，键是属性的集合（因此键值是元组）；但是，如果构成某个键 `K` 的属性集合只包含一个属性 `A`，那么通常（尽管严格来说不正确）会非正式地称该属性 `A` 本身为该键。

对比：子键；超键。

另见：键约束。

### 示例

在供应商-零件数据库中，`{S#}`、`{P#}` 和 `{S#,P#}` 分别是关系变量 `S`、`P` 和 `SP` 的唯一键。注意，`{SNAME}` 不是 `S` 的一个键，因为 `{SNAME}` 的值不一定唯一（尽管图 1 中显示的值恰好是唯一的）。还需注意，例如，`{S#,CITY}` 也不是 `S` 的一个键，因为尽管其值必须是唯一的，但它不是不可约的——我们可以移除 `CITY` 属性，剩下的部分仍然满足唯一性性质。（不可约性是期望的，因为没有它，系统将强制执行错误的完整性约束。在此例中，它不会强制执行供应商编号“全局”唯一的约束，而只会强制执行一个更弱的约束，即它们在每个城市内唯一。）


#### 规范形式
给定一个集合 *s1* 以及该集合元素间等价关系的定义，集合 *s1* 的子集 *s2* 是其规范形式集合，当且仅当 *s1* 中的每个元素 *x1* 都恰好与 *s2* 中的一个元素 *x2* 等价（且该元素 *x2* 即为元素 *x1* 的规范形式）。

许多适用于 *x1* 的“有趣”属性同样适用于 *x2*；因此，为了证明各种“有趣”的定理或结果，我们可以只研究小集合 *s2*，而非大集合 *s1*。

**示例**：令 *s1* 为非负整数集合 {0, 1, 2, ...}，并规定两个整数等价当且仅当它们除以五的余数相同。那么我们可以定义 *s2* 为集合 {0, 1, 2, 3, 4}。作为一个适用于此例的“有趣”定理，令 *x1*, *y1*, *z1* 为 *s1* 的任意三个元素，并令它们在 *s2* 中的规范形式分别为 *x2*, *y2*, *z2*；那么乘积 *y1* * *z1* 等价于 *x1*，当且仅当乘积 *y2* * *z2* 等价于 *x2*。

#### 基数
一个包（bag）或（特别是）集合中的元素数量；对于关系而言，指该关系主体（body）中元组的数量。也用于：(a) 关系变量（relvar），指该关系变量在给定时刻的值所对应关系的基数；(b) 关系或关系变量的某个属性，指在该关系或关系变量主体中出现的该属性不同值集合的基数——对于关系变量，是指在给定时刻的基数。

（当然，关系 *r* 的属性 *A* 的基数，等同于该关系在属性 *A* 上的投影 *r*{ *A*} 的基数；因此，此处的定义 (b) 严格来说是冗余的。）

**示例**：在图 1 中，(a) 当前作为关系变量 *SP* 值的关系的基数为 12（因此关系变量 *SP* 的基数当前也为 12）；(b) 该关系中属性 *S#* 的基数为 4（因此该属性在关系变量 *SP* 中的基数当前也为 4）。

#### 基数约束
1.  对给定关系变量基数的约束（关系变量约束的一个特例）；例如，一个约束表明在任何给定时刻供应商数量绝不能超过十个。
2.  令 *r* 为从集合 *s1* 到集合 *s2* 的一个关系，并令 *x1* 和 *x2* 分别为 *s1* 和 *s2* 的典型元素。在 E/R 建模及类似的设计方案中，可以为 *s1* 和 *s2* 分别指定以下基数约束：1, 0..1, 0.. *m*, 1.. *m*（也使用其他表示法）。为确定起见，假设所讨论的约束是针对集合 *s2* 指定的；那么该约束指示了在关系 *r* 中，每个 *x1* 对应多少个 *x2*。各种指定的含义如下：1 表示必须恰好有一个 *x2*；0..1 表示至多有一个 *x2*；0.. *m* 表示可以有任意数量的 *x2*，从零到某个未定义的上限 *m*；1.. *m* 表示可以有任意数量的 *x2*，从一到某个未定义的上限 *m*。*注意*：术语 *optional participation*（可选参与）和 *mandatory participation*（强制参与）有时分别用来指下界为 0 和下界为 1 的情况；然而，对于这些术语的含义并无普遍共识，最好避免使用。

#### 笛卡尔积
1. （*二元情况*）两个关系 *r1* 和 *r2* 的笛卡尔积，*r1* TIMES *r2*（其中 *r1* 和 *r2* 没有共同的属性名），是一个关系，其标题（heading）是 *r1* 和 *r2* 标题的集合理论并集，其主体（body）是所有元组 *t* 的集合，其中 *t* 是来自 *r1* 的一个元组和来自 *r2* 的一个元组的集合理论并集。
2. （*N 元情况*）*n* 个关系 *r1, r2, ..., rn*（*n* ≥ 0）的笛卡尔积，TIMES { *r1, r2, ..., rn*}（其中 *r1, r2, ..., rn* 中任意两者没有共同的属性名），是一个关系，其标题是 *r1, r2, ..., rn* 标题的集合理论并集，其主体是所有元组 *t* 的集合，其中 *t* 是来自 *r1* 的一个元组、来自 *r2* 的一个元组、...、以及来自 *rn* 的一个元组的集合理论并集。

*注意*：关系型笛卡尔积运算符在几个方面与同名的数学或集合理论运算符不同，因此有时被明确称为扩展的笛卡尔积。实际上，它是连接（join）的一个特例。

**示例**：令 *r1* 和 *r2* 分别为投影 *S*{*S#*} 和 *P*{*P#*}。那么笛卡尔积 *r1* TIMES *r2* 包含所有形如 < *s#, p#*> 的可能元组（其中 *s#* 是当前出现在关系变量 *S* 中的 *S#* 值，*p#* 是当前出现在关系变量 *P* 中的 *P#* 值），并且没有其他元组。（根据图 1 中的值，结果的基数为 30。）注意，表达式 (*S*{*S#*}) TIMES (*P*{*P#*}) 在语义上等价于表达式 (*S*{*S#*}) JOIN (*P*{*P#*}）。

## 笛卡尔积（集合论）
两个集合 *s1* 和 *s2* 的笛卡尔积是所有有序对元素的集合，其中对的第一个元素是 *s1* 的一个元素，第二个元素是 *s2* 的一个元素。*注意*：此定义显然可以扩展到任意数量的集合。

#### 级联
在某个请求更新所指定的目标之外，对某些额外目标重复执行该请求的更新，通常但不一定是为了避免发生某些完整性违规。*另见* 补偿操作（compensating action）。

#### 目录
在给定的数据库内，一组描述该数据库的关系变量（包括目录关系变量自身——即目录是自描述的）。此类关系变量有时被认为包含元数据（metadata）。

目录关系变量通常不是通过显式的赋值操作来更新，而是通过更用户友好的数据定义运算符（data definition operator）来更新（尽管这些运算符本质上只是某些关系赋值的简写形式）。

*参见* Principle of Cautious Design, The（谨慎设计原则）。

#### 单元格
有时用来指表中行与列交叉处的术语；不要将其与该单元格的内容混淆。请注意，“单元格”的概念在“表是关系的图片”（*参见* table）这一观点下有意义，但在“表就是关系”这一观点下则无意义，这就是为什么此定义是基于表而非关系来阐述的。诚然，我们可以非常不正式地用“元组与属性的交叉点”来考虑某个关系，但我们不能将这些交叉点视为与其内容有所区别。（从一个关系中拿走内容，则什么都不剩；正如 Lewis Carroll 可能观察到的那样，没有内容的关系就像没有猫的笑容。）

#### 子表
在 SQL 上下文中有时用来表示（SQL 中类似于）引用关系变量（referencing relvar）的术语，因不恰当而已弃用。
