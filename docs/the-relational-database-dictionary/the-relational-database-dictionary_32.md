# **一百五十一**

#### 关系数据库

遵守 `信息原理` 的数据库。在本词典中，除非有明确说明，我们假设所有数据库都是关系的。*注*：`SQL` 数据库最多只能被视为近似的关系数据库，因为 `SQL` 涉及许多对 `信息原理` 的偏离（包括但不限于 `表` 条目下指出的那些偏离）。

#### 关系数据库管理系统

管理关系数据库（且不管理其他数据库）的 `DBMS`；等价地说，是实现 `关系模型` 的 `DBMS`。*注*：`SQL DBMS` 最多只能被视为近似的关系系统，因为 `SQL` 涉及许多对关系模型的偏离（包括但不限于 `表` 条目下指出的那些偏离）。


#### 关系表达式

表示一个关系的表达式。关系字面量、关系选择器调用、`relcon` 与 `relvar` 引用以及关系代数运算符调用都是其特殊情形。

#### 关系包含

设关系 `r1` 和 `r2` 类型相同。那么 `r1` 包含 `r2`（“`r1` ⊇ `r2`”）当且仅当 `r1` 的主体是 `r2` 主体的超集，而 `r2` 包含于 `r1`（“`r2` ⊆ `r1`”）当且仅当 `r2` 的主体是 `r1` 主体的子集。关系 `r1` 等于关系 `r2`（“`r1` = `r2`”）当且仅当两者互相包含。注意，每个关系都包含自身，并且每个关系都包含其适用类型的空关系。

还需注意，术语“关系包含”通常（有点随意地）特指“⊆”运算符，而非“⊇”运算符。

#### 关系模型

关系数据库乃至整个关系技术所基于的正式理论或基础。

关系模型常被笼统地描述为具有三个方面：一个与关系本身相关的结构方面；一个与候选键和外键相关的完整性方面；一个与连接等运算符相关的操纵方面。

更精确地说，关系模型由以下组件构成：(a) 一个开放式的标量类型集合，特别包括 `BOOLEAN` 类型；(b) 一个关系类型生成器，以及对此生成器所生成类型的关系的既定解释；(c) 用于定义此类生成关系类型的关系变量的设施；(d) 一个关系赋值运算符；(e) 一个开放式的通用只读运算符（即关系代数或关系演算）集合，用于从关系推导出关系。尤其要注意部分 (e)；将关系模型仅仅视为结构而忽视其运算符是一种太过常见的错误，然而（正如 Codd 曾经所说）没有运算符的结构，就如同没有生理学的解剖学。此外，请注意这些运算符并不仅仅用于编写查询，正如许多人似乎认为的那样；相反，它们用于编写表达式，这些表达式服务于多种目的，包括但不限于查询。一个特别重要的目的是约束的制定（尽管在这种情况下，关系表达式只是某个布尔表达式的一个子表达式，通常尽管并非总是 `IS_EMPTY`（参见该词条）的调用）。*注*：为了实现物理数据独立性，关系模型刻意对与性能相关的一切（尤其是物理存储表示）保持沉默。

#### 关系运算符

一种运算符，它 (a) 接受关系或关系变量（或两者）作为操作数，并且 (b) 要么返回一个关系，要么更新至少一个关系变量。

#### 关系变量谓词

设 `R` 为一个关系变量（真实的或虚拟的）。那么 `R` 的关系变量谓词就是代表用户所理解的 `R` 的含义的那个谓词。如果 `R` 的度是 *n*，则该谓词将是 *n* 元的（它将为 `R` 的每个属性设有一个参数）。此外，在封闭世界假设下，在任何给定时间，`R` 的主体将包含且仅包含那些在该时间对谓词的调用求值为 `TRUE` 的元组。*对比* 关系变量约束（第二个定义）；后者是一种形式构造，而关系变量谓词必然是有些非形式化的。*注*：关系变量谓词有时被称为业务规则——尽管一些作者认为后一术语除了关系变量谓词本身外，还包括各种其他构造，特别是各种完整性约束的非形式化对应部分。（相反，另一些人认为这些附加构造是关系变量谓词的一部分。对此类问题尚无共识。）

*示例*：关系变量 `S` 的关系变量谓词是“供应商 `S#` 已签约，名为 `SNAME`，状态为 `STATUS`，且位于 `CITY`。”



至少，这个谓词是本词典其他地方假设（且确实陈述）的针对关系变量 `S` 的谓词。然而，更准确的说法是，该谓词其实是：“我们*知道*——或者（或许更好地说）我们*相信*——供应商 `S#` 已签订合同，其名为 `SNAME`，其状态为 `STATUS`，且位于 `CITY` 中。”关键在于，无法保证数据库真实地反映了现实世界中存在的事态（*参见* 正确性）；它所能做的只是反映用户告诉它的内容，而用户告诉它的内容又反过来反映了他们对现实世界的信念，而不一定是现实世界本身。但在非正式语境下，通常忽略这些考虑。

#### 重命名
令 `r` 为一个关系，`A` 为 `r` 的一个属性，并且令 `r` 没有名为 `B` 的属性。那么，重命名 `r RENAME ( A AS B )` 是一个关系，其 (a) 目标与 `r` 的目标相同，只是其中的属性 `A` 被重命名为 `B`；(b) 主体与 `r` 的主体相同，只是主体中所有对 `A` 的引用（更准确地说，是主体元组中对 `A` 的引用）都被替换为对 `B` 的引用。*另见* 元组重命名。

**示例**：表达式

```
P RENAME ( WEIGHT AS WT )
```

生成一个关系，该关系与关系变量 `P` 的当前值相同，只是属性 `WEIGHT` 被重命名为 `WT`。请注意，关系变量 `P` 本身在数据库中保持不变——`RENAME` 不像 SQL 中的 `ALTER TABLE`，它只是一个只读运算符，就像 `restrict` 一样，它接受某个关系作为输入并返回另一个关系作为输出。*注意*：出于用户友好性的考虑，**Tutorial D** 允许使用单次 `RENAME` 调用来表达两个或多个连续的重命名。因此，例如，表达式 `(P RENAME (WEIGHT AS WT)) RENAME (COLOR AS COL)` 可以缩写为 `P RENAME (WEIGHT AS WT, COLOR AS COL)`。类似的说明也适用于其他几种关系（和元组）运算——例如 `EXTEND`、`SUMMARIZE` 等。

#### 重复组
令某个表具有类型为 `T` 的列 `C`。那么，当且仅当出现在 `C` 中的值不是类型 `T` 的值，而是类型 `T` 值的集合（集合、列表、数组等等）时，`C` 是一个重复组列。重复组在关系模型中是被禁止的（这就是为什么此定义是用表和列而不是关系和属性来表述的）；实际上，具有重复组“属性”的“关系”是一个自相矛盾的概念。

#### 限制
令 `r` 为一个关系，`bx` 为作用于 `r` 的一个限制条件（因此特别要注意，`bx` 中的每个属性引用都标识了 `r` 的某个属性）。那么，根据 `bx` 对 `r` 进行的限制，即 `r WHERE bx`，是一个关系，其目标与 `r` 相同，其主体由 `r` 中所有满足 `bx` 求值结果为 TRUE 的元组组成。

*注意*：限制有时被称为选择，但不应与 SQL 的 `SELECT` 操作混淆。SQL 的 `SELECT` 操作——更具体地说，仅指 SQL `SELECT` 表达式中的 `SELECT` 部分——可以大致描述为 `summarize`、`extend`、`rename` 和“投影”（“投影”加引号是因为它通常不消除重复行，除非通过 `DISTINCT` 显式要求）的组合。

**示例**：以下表达式表示关系变量 `P` 当前值所对应关系的一个限制：

```
P WHERE WEIGHT < WEIGHT ( 17.5 )
```

#### 限制条件
一个布尔表达式，其中 (a) 所有属性引用都标识同一关系 `r` 的属性，且 (b) 没有关系变量引用。*参见* 限制。*注意*：真实语言中的 `WHERE` 子句通常允许的谓词比作用于相关关系的简单限制条件要通用得多。实际上，严格来说，限制条件甚至不允许包含任何连接词，因为连接词是逻辑运算符，而非代数运算符。然而，在实践中，以下恒等式使得真实语言最终能够支持连接词：

`r WHERE ( p AND q )`
≡ `( r WHERE p ) INTERSECT ( r WHERE q )`

`r WHERE ( p OR q )`
≡ `( r WHERE p ) UNION ( r WHERE q )`

`r WHERE ( NOT p )`
≡ `r MINUS ( r WHERE p )`

*示例*：关于谓词仅是相关关系的简单限制条件的 `WHERE` 子句示例，请参见限制。

这里是一个谓词更为通用的示例：

```
S WHERE P { P# } = ( ( SP RENAME ( S# AS SNO ) ) WHERE SNO = S# ) { P# }
```

这里的谓词不仅仅是关系变量 `S` 当前值所对应关系的简单限制条件，因为 (a) 它包含的属性引用并不标识该关系的属性，并且 (b) 它还包含两个关系变量引用（分别针对 `P` 和 `SP`）。然而，整个表达式可以看作是如下形式的简写：

```
WITH ( SP RENAME ( S# AS SNO ) ) AS t1 ,
( EXTEND S ADD ( ( P { P# } ) AS X ) ) AS t2 ,
( EXTEND t2 ADD ( ( ( t1 WHERE SNO = S# ) { P# } ) AS Y ) ) AS t3 :
t3 WHERE X = Y
```

在这个展开的表述中，`X` 和 `Y` 是 `t3` 的（关系值）属性，并且最后一行 `WHERE` 子句中的谓词确实是形式化定义的限制条件。

#### 标量
1. （*对于类型、属性、值或变量*）指没有用户可见组成部分的。该术语也常专门用作*标量值*的缩写。
2. （*对于运算符*）指返回标量结果的运算符。

*注*：关系模型要求有标量类型、属性和值；标量变量不是必需的，但在外部环境中几乎肯定需要它们，以支持例如从某个关系的某个元组中检索某个标量属性的值。类似的说明也适用于标量运算符。还需注意，不存在“绝对标量性”（或有时所称的“绝对原子性”）这样的东西——这个概念必然是相对的。例如，一个电话号码既可以被看作一个“原子”（即标量）值，也可以被看作一个由国家代码、地区代码和本地号码组成的元组值（涉及电话号码的数据库设计应当能够支持这两种看法）。再考虑 `TABLE_DUM` 的情况，它显然是一个关系，然而（像标量一样）没有用户可见的组成部分。

#### 第二范式
关系变量 `R` 处于第二范式（`2NF`），当且仅当 `R` 的每个非键属性 `A` 都满足集合 { `A` } 不可约地依赖于 `R` 的每个键；等价地说，当且仅当 `R` 所满足的每个非平凡函数依赖 `A → B`，要么 (a) `A` 是超键或不是子键，要么 (b) `B` 是子键。每个 `2NF` 关系变量都处于 `1NF`（实际上，每个关系变量都是）。*注*：虽然处于 `2NF` 显然并不排除也处于更高一级的范式（`3NF`），但 `2NF` 这个术语常被宽泛地用于指代处于 `2NF` 但不处于 `3NF` 的关系变量。此外，第二范式本身已不再非常重要（`BCNF`、`5NF` 和 `6NF` 是具有最实际意义的范式）；我们在此提及它主要是出于历史原因。

*示例*：正如在 Boyce/Codd 范式下所指出的，对于范式而言，展示一个反例通常比展示一个例子本身更具启发性。因此，假设关系变量 `SP` 有一个额外的属性 `CITY`，代表相应供应商所在的城市。这个修订版的 `SP` 满足函数依赖 `{S#} → {CITY}`，因此不处于 `2NF`（因为 `CITY` 是非键属性，然而 `{CITY}` 并不可约地依赖于键 `{S#,P#}`；等价地说，`{S#}` 不是超键，*是* 子键，而 `{CITY}` 不是子键）。

#### 二阶逻辑
谓词逻辑的一种形式，其中变量所取值的集合允许包含谓词。*对比* 一阶逻辑。

*示例*：考虑众所周知的数学归纳法原理，为简单起见，此处限于其应用于唯一参数为非负整数类型的一元谓词 `p`。该原理可以用（有些生硬的）英语表述如下：对于所有这样的谓词 `p`，如果 (a) `p(0)` 为真，并且 (b) 对于所有 `i`，如果 `p(i)` 为真则 `p(i+1)` 为真，那么 (c) 对于所有 `n`，`p(n)` 为真。用符号表示：
```
FORALL p ( ( p(0) AND FORALL i ( p(i) IMPLIES p(i+1) ) ) IMPLIES FORALL n ( p(n) ) )
```
在这个例子中，变量 `i` 和 `n` 的取值范围是非负整数，但变量 `p` 的取值范围是谓词（具体来说，是唯一参数为非负整数类型的一元谓词）。因此，整个表达式是二阶的。

#### SELECT 表达式
在 SQL 中，大多数查询被表述为表达式，依次包含 `SELECT` 子句（带可选的 `DISTINCT` 规范）、`FROM` 子句、`WHERE` 子句、`GROUP BY` 子句和 `HAVING` 子句（后三个是可选的）。这些表达式通常被宽泛地称为 `SELECT - FROM - WHERE - GROUP BY - HAVING` 表达式，或更简单地称为 `SELECT - FROM - WHERE` 表达式，或者更简单地称为 `SELECT` 表达式。不幸的是，在本词典性质的文献中，无法给出这个 SQL 结构的完整准确定义，部分原因是各子句之间的相互依赖关系；例如，`SELECT` 子句的语义取决于是否存在 `GROUP BY` 子句（以及其他因素）。但以下概念性算法大致描述了其整体语义：

（`FROM`）形成 `FROM` 子句中指定表的笛卡尔积。
（`WHERE`）丢弃该积中不满足 `WHERE` 子句中布尔表达式的行。
（`GROUP BY`）根据 `GROUP BY` 子句中指定列的值，将剩余的行划分成分组。
（`HAVING`）丢弃该划分中不满足 `HAVING` 子句中布尔表达式的组。



**( *SELECT*)** 从每个剩余的组中，通过应用 `SELECT` 和 `GROUP BY` 子句共同指定的组合（汇总、扩展、重命名和“投影”操作）来导出一行。

**( *DISTINCT*)** 丢弃冗余的重复行。

在任何更精确的解释中，都需要考虑以下因素：
- 表中列出的 SQL 表与其关系对应物之间的所有差异
- 各个子句都可以包含子查询（q.v.）这一事实
- 某些子查询可以是“关联的”这一事实
- 某些子查询可以是“横向的”这一事实
- 某些基本操作（包括相等性（`=`））没有完全定义（在某些情况下，实际上它们被明确允许为 SQL 标准所称的“可能非确定性”），意味着其结果是不可预测的

对此类事项的进一步解释超出了本词典的范围。

**注意**：令 `exp1` 和 `exp2` 为 `SELECT` 表达式。那么 SQL 允许以 `exp1` 和 `exp2` 为基础，形成各种并集、交集和差集。

#### selector
一种运算符（按定义是只读的），用于选择或指定给定类型的任意值；不要与 SQL 的 `SELECT` 操作混淆（关于后者的一个宽松描述，*参见* `SELECT` 表达式）。每种类型（包括元组和关系类型）都至少有一个关联的选择器。*注意*：最终，任何表达式要产生类型为 `T` 的值，唯一的方式是通过调用该类型 `T` 的某个选择器。事实上，选择器的概念本质上是熟悉概念“字面量”的推广；也就是说，所有字面量都是选择器调用，但并非所有选择器调用都是字面量（具体来说，一个选择器调用是字面量，当且仅当其所有参数本身都被指定为字面量）。

#### selector 示例
在 `INSERT` 的示例中出现了几个选择器调用；然而，那些示例都是字面量，因为它们的参数本身也被指定为字面量。与此相反，这里是一些不是字面量的选择器调用。令 `SX`、`PX` 和 `QX` 分别为 `CHAR`、`CHAR` 和 `INTEGER` 类型的变量。那么 (a) 表达式 `S#(SX)`、`P#(PX)` 和 `QTY(QX)` 分别是类型 `S#`、`P#` 和 `QTY` 的选择器调用；(b) 表达式 `TUPLE {S# S#(SX), P# P#(PX), QTY QTY(150)}` 是元组类型 `TUPLE {S# S#, P# P#, QTY QTY}` 的选择器调用；以及 (c) 表达式 `RELATION { ts }`（其中 `ts` 是刚展示的元组选择器调用）是关系类型 `RELATION {S# S#, P# P#, QTY QTY}` 的选择器调用。进一步的解释超出了本词典的范围。

#### semantic optimization 示例

```

P WHERE CITY = 'London'
```

此外，如果原始查询请求的是蓝色零件而不是红色零件，优化器可能能够确定结果为空，而根本无需实际执行查询。

#### set
一个对象的集合，称为元素，具有以下性质：对于任意对象 `x`，可以确定 `x` 是否出现在该集合中（*参见* set membership）。一个例子是集合 `{ y, x, z }`，它可以等价地写成 `{ x, y, z }`，因为集合的元素没有顺序（也不包含重复元素）。集合的每个子集本身也是一个集合。*另见* class（第一个定义）。*注意*：一个元素 `x` 和仅包含该元素 `x` 的集合 `{ x }` 之间存在逻辑差异（实际上是类型上的差异）。因此，关系语言需要提供 (a) 用于从基数为 1 的关系中提取单个元组的运算符，以及 (b) 用于从度为 1 的元组中提取单个属性值的运算符（*参见* attribute extractor；tuple extractor）。

还需注意，逆功能——实际上是从指定的属性值构建元组，以及从指定的元组值构建关系——是由适当的元组和关系选择器提供的（*参见* selector）。

#### set inclusion
集合 `s1` 包含集合 `s2`（`s1 ⊇ s2`）当且仅当它是 `s2` 的超集；集合 `s2` 被包含于集合 `s1`（`s2 ⊆ s1`）当且仅当它是 `s1` 的子集。集合 `s1` 等于集合 `s2`（`s1 = s2`）当且仅当每个集合都包含另一个。注意，每个集合都包含自身，并且每个集合都包含空集。还要注意，术语“集合包含”通常被（有点随意地）理解为特指 `⊆` 运算符，而不是 `⊇` 运算符。*注意*：术语“containment”有时用作此处“包含”含义的同义词，但这种用法不被推荐——最好说一个集合“包含”它的元素（*参见* containment）但“包含”它的子集（includes its subsets）。



#### 集合级
关系模型中的运算符都是**集合级**的，因为它们以完整的关系或关系变量（或两者）作为操作数，并要么产出完整的关系作为结果，要么更新完整的关系变量。（`关系级`会是一个更好的术语。）

*注意*：这个概念对更新运算符尤其意味着，在所有的更新完成之前，不得进行补偿操作和数据库完整性检查；也就是说，一个集合级更新不能被当作一系列单独的元组级更新来处理。

*对比*元组级。

#### 签名

#### 第六范式
关系变量 `R` 是**第六范式**，`6NF`，当且仅当它根本不能进行非平凡的非损失分解——即当且仅当它满足的连接依赖（JD）只有平凡的。因此，请注意，(a) `6NF` 是传统理解下的范式规范化方面的终极范式；(b) 每个 `6NF` 关系变量都在 `5NF` 中。观察发现，关系变量 `R` 在 `6NF` 中，当且仅当 (a) 它由一个键 `K` 和最多一个其他属性 `A`（如果 `K` 没有嵌入式依赖，参见该词条）或恰好一个其他属性 `A`（否则）构成，并且 (b) `K` 的任何真子集 { `B`} 都不满足函数依赖 { `A`} → { `B`} 或函数依赖 { `B`} → { `A`}。还观察到，该定义实际上可以简化为：关系变量 `R` 在 `6NF` 中，当且仅当它在 `5NF` 中，其度为 `n`，并且没有度小于 `n`-1 的键。

*注意*：`6NF` 的概念在时态数据库背景下给出了扩展定义，在那里它相对于 `5NF` 具有某些优势——这些优势在传统的（非时态）数据库中不一定适用。细节超出了本词典的范围。

*示例*：1. 关系变量 `SP` 在 `6NF` 中，因为它根本不能进行非平凡的非损失分解。（观察到 `SP` 肯定在 `5NF` 中；它的度是三；并且它没有度小于二的键。）相比之下，关系变量 `S` 和 `P` 不在 `6NF` 中，因为它们两者都可以非平凡地分解为几个二元投影。2. 设关系变量 `PLUS` 具有属性 `A`、`B` 和 `C`，类型均为 `INTEGER`，并设相应的关系变量谓词为 `A + B = C`。那么关系变量 `PLUS` 有三个不同的键：`{A,B}`、`{B,C}` 和 `{C,A}`。但 `PLUS` 在 `6NF` 中，因为它肯定在 `5NF` 中，度为三，并且没有度小于二的键。

#### 斯科伦常量
根据定义，表达式
```
EXISTS x ( p( x))
```
在逻辑上等价于表达式 `p( v)`，其中 `v` 是某个未知值；也就是说，原始表达式断言存在某个这样的 `v`，即使我们不知道它是什么。那个值 `v` 就是一个**斯科伦常量**。*另见*斯科伦函数。

#### 斯科伦函数
根据定义，表达式
```
FORALL y (EXISTS x ( q( y,x))
```
在逻辑上等价于表达式
```
FORALL y ( q( y,f( y)))
```
其中 `f` 是关于全称量化变量 `y` 的某个未知函数。那个函数 `f` 就是一个**斯科伦函数**。斯科伦常量（参见该词条）和斯科伦函数一起，为从任意表达式中系统地消除存在量词提供了基础，从而使该表达式更易于进行后续的形式化处理。进一步的细节超出了本词典的范围。



#### 小除法

多年来定义的众多关系除法运算符之一（参见 `除法`）。设关系 `r1`、`r2` 和 `r3` 满足：(a) `r1` 和 `r3` 中同名属性类型相同，`r3` 和 `r2` 中同名属性类型也相同（换句话说，`r1` 和 `r3` 是可连接的，`r3` 和 `r2` 也是可连接的）；(b) 集合 `{X}` 是 `r1` 和 `r3` 的公共属性；(c) 集合 `{Y}` 是 `r3` 和 `r2` 的公共属性；(d) 集合 `{X}` 和 `{Y}` 不相交。

那么，除法 `r1` `除以` `r2` `每` (`r3`)——其中 `r1` 是被除数，`r2` 是除数，`r3` 是“中介”——是一个与 `r1` 具有相同标题的关系，其主体定义如下：元组 `t` 出现在该主体中，当且仅当 `t` 出现在 `r1` 中，并且对于 `r2`{`Y`} 中的每一个元组 `<y>`，都存在一个元组 `<x,y>`（其中 `x` 等于 `t` 中的 `X` 值）出现在 `r3`{`X,Y`} 中。换句话说，表达式 `r1` `除以` `r2` `每` (`r3`) 在语义上等价于表达式 `r1` `不匹配` ((`r1`{`X`} `连接` `r2`{`Y`}) `不匹配` `r3`)。对比 `大除法`。

### 示例

除法 `S` `除以` `P` `每` (`SP`) 产生一个关系，其标题与关系变量 `S` 相同，主体由来自关系变量 `S` 的所有可能元组 `<s#,sn,st,sc>` 组成，这些元组满足供应商 `s#` 提供了关系变量 `P` 中提到的所有零件。（根据图 1 的示例值，结果只包含供应商 `S1` 的元组。）该表达式在语义上等价于下面这个表达式：

```
S 不匹配
( ( S { S# } 连接 P { P# } ) 不匹配 SP )
```

等价的元组演算表述是：

```
SX 范围属于 { S } ;
SPX 范围属于 { SP } ;
PX 范围属于 { P } ;
SX 满足 对所有 PX ( 存在 SPX
( SPX.S# = SX.S# AND SPX.P# = PX.P# ) )
```

等价的 `教程 D` 表述是：

```
S 满足 ( ‼SP ) { P# } = P { P# }
```

（参见 `影像关系`）。

#### 快照

一个真实的派生关系变量，而非虚拟的（对比 `视图`）。给定快照在给定时间的值，是在该时间之前某个时间对某个关系表达式（快照定义表达式，在定义快照本身时指定）进行求值的结果。快照根据需要“刷新”（意思是重新求值快照定义表达式，并将结果赋值为快照新的当前值），或者更常见的情况是，当某个规定的事件发生时（例如经过了一段特定的时间间隔）进行刷新。

注意：快照定义表达式必须至少提及一个关系变量，否则快照就不 specifically 是一个关系变量。

### 示例

以下语句定义了一个名为 `LSS` 的快照：

```
VAR LSS 快照 ( S 满足 城市 = 'London' )
每天刷新 ;
```

快照 `LSS` 在任何给定时间的值，等于快照定义表达式 `S 满足 城市 = ‘London’` 在最多 24 小时之前的值。

#### 存储过程

一个可能带参数的子程序；换句话说，就是一个运算符。注意：与术语 `封装的` 类似，*存储过程* 一词已逐渐意味着不幸地混合了模型和实现的考虑因素。从模型的角度来看，一个存储过程确实如前所述，只不过是一个运算符。然而，在实践中，存储过程具有许多属性，使得它们比单纯作为运算符要重要得多（尽管以下前两个属性可能普遍适用于运算符，至少如果所讨论的运算符是系统定义的）。首先，它们被单独编译，并且可以被不同的应用程序共享。其次，它们的编译代码通常物理地存储在数据本身物理存储的位置，具有明显的性能优势。第三，它们通常用于提供本应由 DBMS 提供但尚未提供的共享功能（完整性检查就是一个很好的例子）。另见 `触发过程`。

#### 概括

令关系 `r1` 和 `r2` 满足 `r2` 与 `r1` 的某个投影类型相同，并且令 `r2` 的属性为 `A1, A2, . . ., An`。

那么，概括 `SUMMARIZE r1 PER ( r2) ADD ( exp AS X)` 是一个关系，其 (a) 目标是 `r2` 的目标扩展了属性 `X`，以及 (b) 体由所有满足以下条件的元组 `t` 组成：`t` 是 `r2` 的一个元组，扩展了属性 `X` 的值 `x`。该值 `x` 是通过对 `r1` 中所有与元组 `t` 具有相同 `A1, A2, . . ., An` 属性值的元组计算表达式 `exp` 得到的。关系 `r2` 不能有名为 `X` 的属性，且 `exp` 不能引用 `X`。

> **注意**：表达式 `exp` 通常至少包含一个汇总（*summary*，q.v.）。另外，早期版本的 **Tutorial D** 显式支持 `SUMMARIZE` 运算符；相比之下，当前版本通过其对 `EXTEND` 运算符（q.v.）的支持，提供了等效的功能。下面的示例说明了两种风格。

### 示例

以下表达式表示对关系变量 `SP` 当前值的关系进行概括：

```
SUMMARIZE SP PER ( S { S# } ) ADD ( COUNT ( ) AS CT )
```

该概括是一个类型为 `RELATION {S# S#, CT INTEGER}` 的关系，为关系变量 `S`（而非 `SP`！—注意 `PER` 规范）中当前出现的每个不同 `S#` 值包含一个元组，且不包含其他元组。每个这样的元组包含一个供应商编号和该供应商编号在关系变量 `SP` 中当前出现的次数（表达式 `COUNT()` 是汇总的一个示例，q.v.）。例如，给定图 1 中的示例值，结果中供应商 `S2` 的元组具有 `S#` 值 `S2` 和 `CT` 值 `2`，而供应商 `S5` 的元组具有 `S#` 值 `S5` 和 `CT` 值 `0`。

相比之下，表达式

```
SUMMARIZE SP PER ( SP { S# } ) ADD ( COUNT ( ) AS CT )
```

——因为 `PER` 操作数是 `SUMMARIZE` 操作数的投影，实际上可以稍微简化为

```
SUMMARIZE SP BY { S# } ADD ( COUNT ( ) AS CT )
```

——产生一个类型为 `RELATION {S# S#, CT INTEGER}` 的关系，为关系变量 `SP` 中当前出现的每个不同 `S#` 值包含一个元组。例如，给定图 1 中的示例值，结果不包含供应商 `S5` 的元组。

以下是使用 `EXTEND` 代替 `SUMMARIZE` 的上述示例的类似物：

```
EXTEND S { S# } ADD ( COUNT ( ‼SP ) AS CT )
```

```
EXTEND SP { S# } ADD ( COUNT ( ‼SP ) AS CT )
```

这些示例中的表达式 `‼SP` 是表达式 `(SP MATCHING RELATION {TUPLE {S# S#}}){ALL BUT S#}` 的简写。

更多解释，*参见* 映像关系。

#### 汇总

一种构造，可以出现在 `SUMMARIZE` 表达式的 `ADD` 规范中（*参见* 概括）。请注意，汇总不是聚合运算符调用，尽管它们看起来可能很像。聚合运算符调用是一个表达式，它可以出现在任何可以出现适当类型字面量的地方。
