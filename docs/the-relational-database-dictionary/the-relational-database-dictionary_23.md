#### 必要性

令 `DM` 为数据模型（该术语的第一种含义，q.v.），并令 `DS` 为 `DM` 提供的一种数据结构。令 `dm` 为一个数据模型（该术语的第二种含义，使用 `DM` 的功能创建），并令 `dm` 包含 `DS` 的一个出现 `ds`。令 `db` 为一个符合 `dm` 的数据库。如果从 `db` 中移除与 `ds` 对应的数据会导致 `db` 丢失信息，则 `ds` 在 `dm` 中是必要的（并且，通俗地说，`DS` 在 `DM` 中是必要的）。例如，非关系系统通常提供多种本质上表示信息的方式，包括（例如）指针、记录排序、重复组等。*注意*：如果数据模型 `DM` 支持 `n` 种不同的数据结构——换句话说，如果 `DM` 提供了 `n` 种不同的方式（无论必要与否）来表示信息——那么 `DM` 也必须支持 `n` 种不同的运算符集合，这是不言而喻的。然而，如果 `n` > 1，那么在 `n` = 1 时不能做的有用事情，在 `n` > 1 时同样不能做（当然，`n` = 1 是最小值）。

*示例*：对于关系模型，我们确实有 `n` = 1；也就是说，关系模型只支持一种数据结构，即关系本身，并且该数据结构显然是必要的，因为如果将其移除，模型将无法表示任何信息。然而，由于关系模型实际上能够表示绝对任何数据，因此任何以某种形式支持关系以及某种额外数据结构 `DS` 的数据模型，都必须满足关系是无关紧要的，或者 `DS` 是无关紧要的。但如果关系是无关紧要的，那么 `DS` 必须在逻辑上等价于关系！——在这种情况下，可以说实际上是 `DS` 无关紧要，而不是关系。此外，一个不“以某种形式支持关系”的数据模型是极不可能的；即使 SQL 也可以被认为支持关系，如果避免使用各种 SQL 特性（如空值、匿名列、重复行等）。因此，例如，指针（对象 ID）、包、列表和数组都可以从所谓的对象模型中移除，而不会损失任何表示能力。事实上，它们未被移除的事实是“对象模型”未能正确区分模型和实现问题的表面证据。
