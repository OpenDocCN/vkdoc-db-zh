#### 实现依赖
该术语（尤其在 SQL 标准中）用于指代其语义可能因不同实现而异，且甚至无需为每个具体实现明确规定的特性。换言之，该术语实质上等同于“未定义”；实现方可以自由决定如何处理该特性，且该决策的结果无需记录（甚至可能在不同版本间变动）。例如，SQL 中若 `ORDER BY` 子句的规范未指定完整的排序规则，则该子句的全部影响即属此类。

#### 蕴含
若 `p` 和 `q` 是谓词，则蕴含式 `(p) IMPLIES (q)` 也是一个谓词。令 `(ip) IMPLIES (iq)` 为该谓词的一次调用（其中 `ip` 和 `iq` 分别是 `p` 和 `q` 的调用）。当且仅当 `ip` 计算结果为 `FALSE` 或 `iq` 计算结果为 `TRUE` 或两者同时成立时，该调用 `(ip) IMPLIES (iq)` 的计算结果为 `TRUE`。（换言之，`(p) IMPLIES (q)` 等价于 `(NOT(p)) OR (q)`。）*注*：在蕴含式 `(p) IMPLIES (q)` 中，`p` 和 `q` 分别称为前件与后件。实际应用中，包围它们的括号可能并非必需。

**示例**：对于未受过形式逻辑训练的人来说，蕴含关系往往难以理解。例如，考虑命题 `(2 + 2 = 4) IMPLIES (太阳是一颗恒星)`，或以更易理解的方式表达为：`若 (2 + 2 = 4) 则 (太阳是一颗恒星)`。该命题的计算结果为 `TRUE`，因为其前件与后件均为真；然而，太阳是否是一颗恒星显然与 `2+2 = 4` 是否成立无关。不过，以下观察可能有所帮助。

在 16 种二元连接词（*参见* 连接词）中，部分（但非全部）拥有诸如 `AND` 和 `OR` 等通用名称。但这些名称本质上只是一种助记手段——它们本身并无内在含义，之所以如此命名，仅仅是因为这些连接词的行为与自然语言中对应词项的行为相似（但未必完全相同）。特别是，在 16 种可用连接词中，名为 `IMPLIES` 的连接词，其行为最接近自然语言中通常理解的蕴含关系。但无人会或应该声称两者是同一事物。此外，这些连接词必然是形式化定义的；即，它们纯粹是根据其参数（对于 `IMPLIES` 而言即前件和后件）的真值而非含义来定义的，而自然语言中的对应词项显然不具备这一特性。

再举一例，命题 `若 (2 + 2 = 5) 则 (猫王依然在世)` 的计算结果也为 `TRUE`（这或许更违反直觉），因为其前件为假；然而，猫王是否在世显然与 `2+2 = 5` 是否成立无关。同样，部分理由（即该蕴含式计算结果为 `TRUE` 的事实）仅在于 `IMPLIES` 是形式化定义的。不过在此例中，或许有另一个论证更能令人接受。假设供应商-零件数据库需遵循约束：红色零件必须存储在伦敦（参见数据库约束示例中的约束 C2）：

#### 包含多态

#### **信息原理，The**
该原理指出，在关系数据库中唯一允许的变量类型是关系变量。等价地说，关系数据库只包含关系变量，别无他物。另一个等价的表述是：在任何给定时间，数据库的全部信息内容都以唯一的一种方式表示——即，作为关系。

*注：* 需要说明的是，这个原理的命名并不太准确。它更准确地应称为“统一表示原理”，甚至“表示统一性原理”，因为其关键点在于它意味着关系数据库中的所有信息都以相同的方式表示——即，作为关系。

#### **单射**
一种从集合`s1`到集合`s2`的映射或函数，使得`s2`中的每个元素最多是`s1`中一个元素的像。也称为非损失映射、单射或“一对一入”映射（尽管此处的“一对一”并非其严格意义上的用法，参见`一对一`）。

*示例*：从非负整数`x`到其平方`x²`的映射，是从所有非负整数的集合到其自身的一个单射。

#### **内连接**
与`连接`相同。限定词`内`用于将所讨论的连接与其外部对应物区分开来。而外连接——至少按通常理解——涉及空值和三值逻辑，因此本词典不作进一步讨论（尽管事实上可以定义一种完全不涉及空值的“可靠”的外连接形式）。

#### **INSERT**
宽松地讲，一个将给定元组集合插入到给定关系变量中的运算符。它是某个关系赋值的简写。

*示例*：INSERT 语句
```
INSERT SP
RELATION {
    TUPLE { S# S#('S3'), P# P#('P1'), QTY QTY(150) },
    TUPLE { S# S#('S5'), P# P#('P1'), QTY QTY(500) } } ;
```
是以下关系赋值的简写：
```
SP := ( SP ) UNION
    ( RELATION {
        TUPLE { S# S#('S3'), P# P#('P1'),
                QTY QTY(150) },
        TUPLE { S# S#('S5'), P# P#('P1'),
                QTY QTY(500) } } ) ;
```
在此示例中，表达式`S#(‘S3’)`和`S#(‘S5’)`是类型`S#`的选择器调用；表达式`P#(‘P1’)`是类型`P#`的选择器调用；表达式`QTY(150)`和`QTY(500)`是类型`QTY`的选择器调用。同样，两个形式为`TUPLE {. . .}`的表达式是元组类型`TUPLE {S# S#, P# P#, QTY QTY}`的选择器调用，唯一的形式为`RELATION {. . .}`的表达式是关系类型`RELATION {S# S#, P# P#, QTY QTY}`的选择器调用。

另外请注意，按此处定义的`INSERT`，尝试插入一个已在目标关系变量中存在的元组并不是错误。在展开式中将`UNION`替换为`D_UNION`可以解决这个问题（如果它算是个问题的话）。

#### **实例化**
宽松地讲，对谓词的一次调用，其中（根据定义）每个参数被某个实参替换。这种实例化的结果是一个命题。

*注：* 实际上，逻辑上的实例化概念比熟悉的编程语言中的函数调用概念更广义。具体来说，我们可以通过仅替换其`n`个参数中的`m`个（`m` ≤ `n`）来实例化一个`n`元谓词，从而得到一个`r`元谓词，其中`r` = `n`- `m`。如果`m` = `n`，则该实例化被称为完全的（并且术语“实例化”在无修饰的情况下通常意指完全实例化，除非上下文另有要求）；否则被称为部分的。

#### **完整性**
当且仅当数据库不违反任何已定义的完整性约束（即，当且仅当它是一致的）时，数据库才处于完整状态。完整性是正确性的必要但不充分条件。关系模型要求数据库始终处于完整状态，这里的“始终”实际上意味着在语句边界处（除非在`类型约束`下另有说明）。

#### **完整性约束**
一个已命名的布尔表达式，或某种等价于此类表达式的表达式，要求它始终（即，在语句边界处，除非在`类型约束`下另有说明）被满足（即，求值为`TRUE`）。有两种基本类型：数据库约束和类型约束，参见`数据库约束`和`类型约束`（但参见`属性约束`、`关系变量约束`、`元组约束`）。DBMS 应拒绝任何如果被接受就会导致某个完整性约束被违反（即，求值为`FALSE`）的更新。

*注：* 除类型约束外的约束专门约束更新，而根据定义更新专门应用于变量；因此，此类约束也专门应用于变量——即，任何受此类约束的东西必须是一个变量。

#### **智能键**
一种单属性键，除了其标识某个实体的主要目的外，还携带某种编码信息嵌入其中。对比`代理键`。

*示例*：假设从国内供应商购买的零件被分配 0-4999 范围内的零件编号，而从其他地方购买的零件被分配 5000-9999 范围内的零件编号。现在假设第 5001 种不同类型的零件是从国内供应商处购买的。显然，零件编号方案现在必须进行修订，任何以前依赖于国内购买零件编号小于 5000 这一事实的程序现在都将失败。正如这个例子所示，智能键应谨慎使用。（实际上，类似的评论也适用于在*任何*属性中编码信息，而不仅仅是键属性，但键属性似乎特别容易受到这种滥用。）

#### **交集**
1. （二元情况）两个关系`r1`和`r2`的交集，`r1 INTERSECT r2`，其中`r1`和`r2`是相同类型`T`，其结果是一个类型为`T`的关系，其主体是所有同时出现在`r1`和`r2`中的元组`t`的集合。



#### （N 元情况）
`n`元情况：`n`个关系`r1, r2, ..., rn`（`n` ≥ 0）的交集，`INTERSECT { r1, r2, ..., rn }`，其中`r1, r2, ..., rn`都是相同类型`T`，结果是一个类型为`T`的关系，其主体是所有出现在每一个`r1, r2, ..., rn`中的元组`t`的集合（除非`n` = 0，此时 (a) 需要某种未在此展示的语法机制来指定相关类型`T`，并且 (b) 结果是该类型的全关系（universal relation））。*注*：关系交集运算符在某些方面不同于同名的数学或集合论运算符。实际上，它是连接（join）的特例。

**示例**：表达式`(S{CITY}) INTERSECT (P{CITY})`表示关系`S`和`P`的当前值在`{CITY}`上的投影的交集。该交集是一个类型为`RELATION {CITY CHAR}`的关系。此外，如果关系变量`S`和`P`的当前值分别是`s`和`p`，那么该关系的主体由所有形如`< c >`、同时出现在`s{CITY}`和`p{CITY}`中的元组组成——这意味着`c`既是当前供应商城市，也是当前零件城市。注意，表达式`(S{CITY}) INTERSECT (P{CITY})`在语义上等价于表达式`(S{CITY}) JOIN (P{CITY})`。

参见 Laws of Algebra, The。

**示例**：

2.  （集合论）如果`s`是一个集合，那么`s`的补集的补集等于`s`。

2.  （关系变量的）第六范式。

#### irreducibly dependent
（FD 中依赖项的）设`A`和`B`是某个关系变量`R`的标题（heading）的子集。那么`B`不可约地依赖于`A`，当且仅当它函数依赖于`A`，并且不函数依赖于`A`的任何真子集。

#### irreducibly equivalent
（FD 集合的）设`s1`和`s2`是两个 FD 集合。那么`s1`与`s2`不可约地等价，当且仅当它与`s2`等价并且是不可约的。

### J

**JD**

连接依赖。

**由超键蕴含的 JD**

连接依赖 `⭐{A1, A2, . . ., An}` 由超键蕴含，当且仅当：(a) `A1, A2, . . ., An` 中的每一个都是相关关系变量 `R` 的超键；并且 (b) `R` 的键可以按某种顺序排列，使得对于每个这样的键，至少有一个 `A1, A2, . . ., An` 同时包含该键及其在该顺序下的后继键。

**示例**：设关系变量 `R` 仅有属性 `A`, `B`, `C`, `D`, `E`, `F`，并且仅有键 `{A}`, `{B}`, `{C}`。令 `ABD` 表示属性集 `{A, B, D}`，其他字母组合类似。那么以下 JD 中，第一个被 `R` 的超键所蕴含，但第二个和第三个不是：
```
⭐ { ABD, BCE, CF }
⭐ { ABD, CE, CF }
⭐ { ABD, BCE, DF }
```
注意，关系变量 `R` 必然满足第一个 JD，并且不满足第二个（后一个论断的有效性可能不明显，但事实上确实如此）。第三个 JD 可能满足，也可能不满足。如果满足，则关系变量 `R` 不在 5NF 中；反之，如果关系变量 `R` 在 5NF 中，则它不可能满足第三个 JD。

**可连接的**

1.  （二元情况）关系 `r1` 和 `r2` 是可连接的，当且仅当具有相同名称的属性属于相同的类型——等价地，当且仅当它们的属性集并集是一个合法的属性集。
2.  （多元情况）关系 `r1, r2, . . . , rn` (`n` > 0) 是可连接的，当且仅当对于所有的 `i`, `j`，关系 `ri` 和 `rj` 是可连接的 (`1 ≤ i ≤ n`, `1 ≤ j ≤ n`)。

#### 连接依赖

多值依赖概念的推广（每个多值依赖都是一个连接依赖，但有些连接依赖不是多值依赖）。连接依赖是一个表达式——实际上是一个命题——形式为 `⭐{A1, A2, . . ., An}`，其中 `A1, A2, . . ., An` 都是同一个关系变量 `R` 的属性集的子集，整个表达式读作“星 `A1, A2, . . ., An`”。关系变量 `R` 满足连接依赖（JD）`⭐{A1, A2, . . ., An}`——等价地，该 JD 在关系变量 `R` 中成立——当且仅当每一个是 `R` 合法值的关系都等于其在 `A1, A2, . . ., An` 上投影的连接。

> **注**：连接依赖也被称为连接依赖性。注意，除非 `A1, A2, . . ., An` 的并集等于 `R` 的属性集，否则关系变量 `R` 不可能满足 JD `⭐{A1, A2, . . ., An}`。进一步注意，从定义可以立即得出，关系变量 `R` 可以无损分解为其在 `A1, A2, . . ., An` 上的投影，当且仅当它满足 JD `⭐{A1, A2, . . ., An}`。最后注意，从形式角度看，JD `⭐{A1, A2, . . ., An}` 只是一个命题，根据定义，它可以被满足或不满足（即可以评估为 TRUE 或 FALSE）；然而，非正式地，通常只将 `⭐{A1, A2, . . ., An}` 定义为 JD，如果它确实被相关的关系变量满足。但这个定义使得无法说一个给定的关系变量不满足或违反某个 JD（因为根据该定义，一个不被满足的 JD 根本不是 JD！）。

**示例**：关系变量 `S` 满足 JD
```
⭐ { { S#, SNAME }, { S#, STATUS }, { S#, CITY } }
```
因为每一个是 `S` 合法值的关系都等于其在 `{S#, SNAME}`、`{S#, STATUS}` 和 `{S#, CITY}` 上投影的连接；即，关系变量 `S` 可以无损分解为这三个投影。（当然，没有要求必须实际执行这种分解——是否应该分解取决于这样做是否有任何好处。）

**键属性**

**键约束**

一种约束，表明给定关系变量的属性集的某个子集是该关系变量的一个键。在 **Tutorial D** 中，这样的约束通过在相关关系变量定义中使用 `KEY` 规范来定义（*参见*，例如，本词典引言中关系变量 `S`、`P` 和 `SP` 的定义）。但请注意，虽然系统肯定会强制执行这种约束所隐含的唯一性属性（*参见* 候选键），但它通常无法同时强制执行相应的不可约性属性。换句话说，在关系变量 `R` 的定义中指定 `KEY{K}` 意味着 `K` 肯定是一个超键，但不一定是关系变量 `R` 本身的一个键。

**示例**：假设我们在关系变量 `S` 的定义中指定 `KEY{S#, CITY}` 而不是 `KEY{S#}`。那么系统显然无法强制执行供应商编号本身（相对于供应商编号/城市组合）必须唯一的约束。（另一方面，如果我们同时指定 `{S#}` 和 `{S#, CITY}` 为键，系统至少应该能够识别前者是后者的真子集，并因此拒绝后一个规范。）
