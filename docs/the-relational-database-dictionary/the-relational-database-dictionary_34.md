#### `定理`
根据给定的推理规则从给定公理推导出的命题（因此，如果公理为真且推理规则有效，则该命题也为真）。在数据库上下文中，派生关系中的元组可以被看作定理，因为它们代表了从基关系中元组所代表的命题推导出来的命题。定理包含公理（参见该词条）作为一个退化的特例。*参见* 证明。

**示例**：给定图 1 的示例值，由表达式 `(S JOIN P){S#,P#}` 表示的关系包含元组 `<S1,P1>`。该元组可被视为一个定理；它代表（真实的）命题“供应商 S1 和零件 P1 位于同一城市”，这一事实可以从基关系中的元组所表示的公理推断出来。

#### `θ连接`
一种关系运算，等价于形如 `( r1 TIMES r2) WHERE A1 theta A2` 的表达式，其中 (a) `A1` 和 `A2` 分别是 `r1` 和 `r2` 的属性（具有相同类型 `T`），且 (b) `theta` 是任何对类型 `T` 的值有意义的比较运算符（例如，“=”、“>”）。

**示例**：以下表达式表示供应商和零件（按此顺序）在城市上的大于连接（即此处 theta 为 “>”）：`(( S RENAME ( CITY AS SC ) ) TIMES ( P RENAME ( CITY AS PC ) )) WHERE SC > PC`

这里我们假设 `CHAR` —— 属性 `CITY` 的类型 —— 是一个序数类型（对 `CHAR` 值的 “>” 可能意味着“按字母顺序更大”）。注意，我们可以在上述表达式中用 `JOIN` 替换 `TIMES` 而不改变其含义。同样，用 “<” 替换 “>” 将产生一个小于连接；用 “=” 替换则将产生一个等值连接。

#### 第三宣言
一个关于数据和数据库管理系统未来的正式提案；像 Codd 的原始论文一样，它可以被看作是 DBMS 设计及与之交互的语言接口的抽象蓝图。*参见* 本词典的引言。

#### `第三范式`
关系变量 `R` 处于第三范式，简称 3NF，当且仅当，对于 `R` 所满足的每一个非平凡函数依赖 `A → B`，`A` 是一个超键或者 `B` 是一个子键。每一个 3NF 的关系变量也处于 2NF。*注意*：文献中许多“3NF”的定义实际上是 BCNF 的定义；*读者须知*。

此外，尽管处于 3NF 显然并不排除同时处于下一个更高的范式（BCNF），但术语 `3NF` 常被宽泛地用来指代处于 3NF 但不处于 BCNF 的关系变量。无论如何，第三范式本身已不再那么重要（BCNF、5NF 和 6NF 是具有最大实际意义的范式）；我们在此提及它主要是出于历史原因。

**示例**：如“Boyce/Codd 范式”词条下所指出的，对于范式而言，展示一个反例通常比展示一个实例本身更具启发性。因此，假设关系变量 `S` 满足额外的函数依赖 `{CITY} → {STATUS}`；也就是说，给定供应商的状态是其所在位置的函数。由于 `{CITY}` 不是一个超键且 `{STATUS}` 不是一个子键，这个版本的关系变量 `S` 不处于 3NF（尽管它处于 2NF）。

#### `三值逻辑`
一种逻辑，除了通常的 `TRUE` 和 `FALSE` 外，还有一个“第三真值”，通常称为 `UNKNOWN`；缩写为 3VL。SQL 大致基于这种逻辑（相比之下，关系模型基于二值逻辑，参见该词条）。特别是，SQL 对空值（参见该词条）的支持是基于三值逻辑的，尽管事实上该支持在逻辑上存在缺陷；例如，SQL 将 `UNKNOWN` 和 null 视为等同，即使两者之间存在逻辑差异 —— `UNKNOWN` 是一个值，而 null 根本不是一个值，而是一个“标记”。（这只是 SQL 的 3VL 支持中众多逻辑错误之一。）*注意*：在 2VL 中的重言式在 3VL 中不一定是重言式；同样，在 2VL 中的矛盾式在 3VL 中也不一定是矛盾式。因此，在 2VL 中成立的定理在 3VL 中不一定成立，在 2VL 中有效的表达式转换在 3VL 中也不一定有效。

#### `事务`

#### `变迁约束`
一种数据库约束，用于限制给定数据库可以从一个状态有效变迁到另一个状态的方式。*对比* 状态约束。

**示例**（“任何供应商的状态都绝不允许降低”）：
```sql
CONSTRAINT TRC1 IS_EMPTY (
    ( ( S' { S#, STATUS } RENAME ( STATUS AS STATUS' ) )
      JOIN
      ( S { S#, STATUS } ) )
    WHERE STATUS' > STATUS ) ;
```
此公式依赖于一个约定，即带撇的关系变量名（如 `S'`）指的是在考虑当前更新之前该关系变量的状态。

#### `传递闭包`
令 `r` 为一个具有属性 `A` 和 `B` 的二元关系，两者类型均为 `T`。那么 `r` 的传递闭包，记作 `TCLOSE ( r )`，是一个关系 `r+`，定义如下：元组 `< a, b>` 出现在 `r+` 中，当且仅当它出现在 `r` 中，或者存在一个类型为 `T` 的值 `c`，使得元组 `< a, c>` 出现在 `r` 中且元组 `< c, b>` 出现在 `r+` 中。（注意这是一个递归定义；还要注意 `r+` 确实是一个传递关系，正如“传递闭包”这个名字所暗示的。*参见* 传递性，第二个定义。）如下面的伪代码算法所示，计算 `TCLOSE ( r )` 在概念上涉及迭代地构造某个中间结果（在前一次迭代中计算）与一个新的部分结果（在当前迭代中计算）的并集，直到该并集不再增长 —— 换句话说，直到它达到一个不动点或“定点”。

```
r+ := r ;
do until r+ ceases to grow ;
    r+ := WITH ( r+ RENAME ( B AS C ) ) AS t1 ,
               ( r RENAME ( A AS C ) ) AS t2 :
          r+ UNION ( ( t1 JOIN t2 ) { A, B } ) ;
end do ;
```

*另见* 递归查询。

#### 传递函数依赖
函数依赖 `A` → `B` 和 `B` → `C` 共同蕴含了传递函数依赖 `A` → `C`。（因此，如果关系变量 `R` 满足函数依赖 `A` → `B` 和 `B` → `C`，那么它也满足传递函数依赖 `A` → `C`。）

#### 传递性
1. （*对于二元逻辑运算符*）二元逻辑运算符 `Op`（为明确起见，我们假设其以中缀风格表示）是传递的，当且仅当，对于所有 `x`、`y` 和 `z`，如果 `x` `Op` `y` 和 `y` `Op` `z` 均为真，则 `x` `Op` `z` 也为真。
2. （*对于二元关系*）二元关系 `r` 是传递的，当且仅当，对于所有 `x`、`y` 和 `z`，如果元组 <`x`, `y`> 和 <`y`, `z`> 都出现在 `r` 中，那么元组 <`x`, `z`> 也出现。
3. （*对于函数依赖*）*参见* Armstrong 推理规则。

**示例**（*仅针对第一个定义*）：逻辑运算符 IMPLIES（蕴含）；“小于”运算符“<”。

#### 触发过程
严格来说，是一个动作（“触发动作”），在指定事件（“触发事件”）发生时执行，尽管该术语常被宽泛地使用，也包括触发事件本身。

触发过程可以被视为一个存储过程（*参见*），不同之处在于存储过程必须显式调用，而触发过程在触发事件发生时自动调用。

然而，除了这一区别，触发过程具有许多与存储过程相同的属性，并用于许多相同的目的。关系模型并未规定任何触发过程，但也并非必然禁止它们——尽管如果它们可能导致违反赋值原则（*参见*），则应被禁止，而它们在实践中非常可能导致违反。外键规则提供了一个具有重要实际意义的例子（其中，“过程”是以声明方式指定的）。*注*：触发事件与相应触发动作的组合通常简称为触发器。

#### 平凡分解
一种无损分解（*参见*），基于某种平凡依赖（即平凡的函数依赖、连接依赖或多值依赖，*参见*）执行。

*示例*：1. 供应商关系变量 `S` 可以平凡地分解为其在 `{S#,SNAME}` 和 `{S#,SNAME,STATUS,CITY}` 上的投影。
2. 任何关系变量都可以平凡地分解为其恒等投影。

#### 平凡函数依赖
一种不可能被违反的函数依赖。函数依赖 `A` → `B` 是平凡的，当且仅当 `A` 是 `B` 的超集。

#### 平凡连接依赖
一种不可能被违反的连接依赖。连接依赖 `5{A1, A2, ..., An}` 是平凡的，当且仅当 `A1, A2, ..., An` 中至少有一个是相关关系变量 `R` 的整个标题。

#### 平凡多值依赖
一种不可能被违反的多值依赖。多值依赖 `A` →→ `B` 是平凡的，当且仅当 `A` 是 `B` 的超集，或者 `A` 和 `B` 的集合并是相关关系变量 `R` 的标题（或两者兼有）。

#### 真值

#### 真值
在逻辑中，一个运算符（符号表示为“/.../”），给定一个逻辑表达式，返回该表达式的真值。例如，令符号 `x` 和 `y` 表示整数。那么表达式 `/x > y/` 在 `x` 所表示的整数大于 `y` 所表示的整数时返回真 (TRUE)，否则返回假 (FALSE)。注意 `/TRUE/` 和 `/FALSE/` 分别返回真 (TRUE) 和假 (FALSE)。还要注意，逻辑表达式 `p` 等价于 `q` 与 `/p/ = /q/` 含义相同；例如，表达式 `(海王星是行星)` 等价于 `(火星恰好有两个月球)` 与表达式 `/(海王星是行星)/ = /(火星恰好有两个月球)/` 含义相同。

#### 元组（数学）
给定集合 `s1, s2, ..., sn`（不一定互不相同），`t` 是这些集合上的一个 *n* 元组（简称元组），当且仅当它是一个元素的有序集合，其中第一个元素来自 `s1`，第二个来自 `s2`，依此类推。集合 `si` 是 `t` 的第 `i` 个域（`i` = 1, ..., `n`）。*注*：数学中的元组与其在关系模型中的对应物之间存在几个重要的逻辑差异。*参见* 关系（数学）；元组值。

#### 元组赋值
将一个元组值赋给一个（相同类型的）元组变量的操作。*参见* 赋值。

#### 元组演算
一种关系演算形式，在语义上等价于域演算（*参见*），其中范围变量的范围是关系，因此表示来自这些关系的元组。

*示例*：以下是一个元组演算形式化表达，用于查询“获取供应至少一个零件的供应商名称”（*参见* 域演算以了解其域演算类比）：
```
SX RANGES OVER { S } ;
SPX RANGES OVER { SP } ;
SX.SNAME WHERE EXISTS SPX ( SPX.S# = SX.S# )
```
用生硬的英语说：“获取供应商 `SX` 的名称，其中存在一个与 `SX` 具有相同供应商编号的发货记录 `SPX`。”

#### 元组比较
一种形如 (`exp1`) `theta` (`exp2`) 的布尔表达式，其中 `exp1` 和 `exp2` 是相同类型的元组表达式，`theta` 是“=”或“≠”（注意运算符“<”和“>”对于元组是明确定义为未定义的）。*注*：在实际操作中，比较式中包围 `exp1` 和 `exp2` 的括号可能不需要。

#### 元组分量
出现在所讨论元组中的一个<属性名, 属性值>对。注意，属性本身被定义为<属性名, 类型名>对。然而，其他形式体系也是可能的；例如，可以将元组分量定义为<属性名, 类型名, 属性值>三元组，而不是<属性, 属性值>对。（事实上，*《The Third Manifesto》* 正是这样做的。）当然，很容易看出这两种形式体系是同构的。

#### 元组相等

#### 元组表达式
表示一个元组的表达式。元组字面量、元组选择器调用、元组变量引用和只读元组运算符调用都是其特例。

#### 元组扩展
令 `t` 为一个元组。那么元组扩展 `EXTEND t ADD ( exp AS X)` 是一个与 `t` 相同的元组，只是它多了一个名为 `X` 的属性，其值由 `exp` 指定。元组 `t` 不能有一个名为 `X` 的属性，并且 `exp` 不能引用 `X`。

### 示例
令 `t` 为关系变量 `P` 中的某个元组。那么表达式
```
EXTEND t ADD ( WEIGHT * 454 AS GMWT )
```
产生一个与 `t` 相似的元组，只是它多了一个额外的属性 `GMWT`（“克重”），其值是同一元组中 `WEIGHT` 值的 454 倍。

#### 元组提取器
用于从指定基数为 1 的关系中提取单个元组的运算符。

### 示例
以下表达式从关系变量 `S` 的当前值中提取供应商 `S1` 的供应商元组：
```
TUPLE FROM ( S WHERE S# = S#('S1') )
```
如果 `TUPLE FROM` 的参数基数不恰好为 1，将发生运行时错误。

#### 元组解包

令 *s* 为一个元组，令 {*Y*} 为 *s* 的一个元组值属性，并令 {*X*} 为 *s* 除 *YT* 外的所有属性的集合。令 {*Y*} 具有属性 *Y1, Y2, . . ., Yn*；同时，令 {*X*} 不包含与 *Y1, Y2, . . ., Yn* 中任何属性同名的属性。那么，元组解包操作 `s` `UNWRAP` (*YT*) 的结果是另一个元组 *t*。*t* 的标题（heading）由 {*X*} 和 {*Y*} 的集合理论并集组成。令元组 *s* 的 *X* 值为 *x*，且 *YT* 值为一个具有 *Y1* 值 *y1,* *Y2* 值 *y2,* . . ., 以及 *Yn* 值 *yn* 的元组；那么，元组 *t* 将具有 *X* 值 *x,* *Y1* 值 *y1,* *Y2* 值 *y2,* . . ., 以及 *Yn* 值 *yn*。

### 示例

令 *t* 为来自关系变量 SP 的一个元组，并令 *tw* 为以下表达式的结果元组

```
t WRAP ( { P#, QTY } AS PQ_TUP )
```

（参见元组包装）。那么，表达式

```
tw UNWRAP ( PQ_TUP )
```

将产生 *t*。

#### 元组值

宽泛地说，一行（值）。更精确地，令 `TUPLE` {*H*} 为一个元组类型，并令 *t* 为一组 <<*A,T*>, *v*> 形式的配对，称为分量（component，参见该词条），通过为 {*H*} 中的每个属性 <*A,T*> 附加某个类型为 *T* 的值 *v*（称为 *t* 中属性 *A* 的属性值）而获得。那么，*t* 就是类型 `TUPLE` {*H*} 的一个元组值（简称元组），其标题为 {*H*}，且具有与 {*H*} 相同的度（degree）和属性。*注意*：关系模型中定义的元组在某些重要方面与同名的数学构造有所不同。特别是，数学中的元组通常没有命名属性；相反，它们的属性通过其从左到右的序数位置来标识。参见 元组（数学）。

#### 元组值属性

其类型为某个元组类型的属性。此类属性的值是指定类型的元组。*注意*：如果一个关系变量具有元组值属性，这一事实本身并不违反任何特定的范式级别（甚至第一范式也不违反）；然而，在数据库设计中，此类属性通常是不推荐的，至少在基本关系变量中如此，因为它们必然意味着某种结构上的不对称，从而在查询、约束和更新中也会引起不对称性（和复杂性）。*另见* 元组解包；元组包装。

#### 元组变量

宽泛地说，一行（变量）；更精确地说，是一个类型为某个元组类型的变量。令元组变量 *T* 的类型为 `TUPLE` {*H*}；那么，*T* 具有与该类型相同的标题（因此属性）和度。*注意*：关系模型本身并不要求元组变量，但为了支持例如从某个关系中提取某个元组等操作，外部环境很可能需要它们。

#### 元组包装

令 *t* 为一个元组，并令 *t* 的标题被划分为子集 {*X*} 和 {*Y*}。令 {*Y*} 的属性为 *Y1, Y2, . . ., Yn*；同时，令 {*X*} 不包含任何名为 *YT* 的属性。那么，元组包装操作 `t` `WRAP` ({*Y*} AS *YT*) 的结果是另一个元组 *s*。*s* 的标题由 {*X*} 扩展一个类型为 `TUPLE` {*Y*} 的属性 *YT* 组成。令元组 *t* 的 *X* 值为 *x,* *Y1* 值为 *y1,* *Y2* 值为 *y2,* . . ., 以及 *Yn* 值为 *yn*；那么，元组 *s* 将具有 *X* 值 *x*，以及 *YT* 值为一个具有 *Y1* 值 *y1,* *Y2* 值 *y2,* . . ., 以及 *Yn* 值 *yn* 的元组。

### 示例

令 *t* 为来自关系变量 SP 的供应商 S1 和零件 P1 对应的元组。那么，表达式

```
t WRAP ( { P#, QTY } AS PQ_TUP )
```

将产生一个类型为 `TUPLE {S# S#, PQ_TUP TUPLE {P# P#, QTY QTY}}` 的元组 *tw*，其 S# 值为 S1，PQ_TUP 值为一个 P# 值为 P1 且 QTY 值为 300 的元组。

#### 类型
值的命名集合；不应与所讨论值的内部或物理表示混淆，那是一个实现问题。每个值、每个变量、每个属性、每个只读运算符、每个参数以及每个表达式都属于某种类型。

类型可以是标量或非标量（具体来说，它们可以是元组类型或关系类型）；因此，关系的属性也可以是标量或非标量。类型也可以是系统定义的（即内置的）或用户定义的。它们也可以由生成器生成（*参见* 类型生成器）。

*注意*：类型不是值，也不是变量；特别是，关系值和关系变量不是类型。将类型等同于关系值或关系变量——这是文献中曾主张的观点——已被描述为“第一大谬误”。（关于第二个，*参见* 指针；第二大谬误。）

**示例**：这是一个类型定义示例：

```
TYPE POINT ...
{ ... CONSTRAINT ( X↑2 + Y↑2 ) ≤ 10000 } ;
```

这里的 `POINT` 是一个用户定义类型，表示二维空间中的几何点。它受一个类型约束，该约束实质上规定了，唯一有意义的点是位于以原点为中心、半径为 100 的圆内或圆上的点（符号“↑”表示乘方）。

*注意*：示例中故意省略了 `POINT` 值的（必要的）基于笛卡尔坐标 `X` 和 `Y` 的表示定义，因为详细讨论该定义会偏离主题太远。出于类似原因，也省略了其他无关细节。

#### 类型约束
对构成给定类型的一组值的定义。当为某个类型 *T* 调用某个选择器时，会检查该类型 *T* 的类型约束；换句话说，当且仅当调用某个选择器时所用的参数违反了适用的类型约束时，才会发生类型约束错误。*对比* 类型错误。

**示例**：*参见* 类型条目下的示例。

#### 类型错误
如果以非对应参数类型的参数调用某个运算符，则会发生错误。与完整性违规不同，此类错误应在编译时即可检测到（除非支持类型继承，在这种情况下，某些类型错误——绝非所有——可能直到运行时才能检测到）。

#### 类型生成器
在编译时而非运行时调用，并返回类型而非值的运算符。例如，传统的编程语言通常支持数组类型生成器，允许用户定义各种特定的数组类型。在关系模型中，元组和（尤其是）关系类型生成器是最重要的；它们允许用户定义各种特定的元组和关系类型。*参见* 关系类型；元组类型。

**示例**：考虑供应商关系变量定义：

```
VAR S BASE RELATION
{ S# S#, SNAME NAME, STATUS INTEGER, CITY CHAR }
KEY { S# } ;
```

此定义包含对 `RELATION` 类型生成器的调用（从语法上讲，是从关键字 `RELATION` 到关键字 `CHAR` 后的闭合大括号的所有内容，包括在内）。该调用返回一个特定的关系类型——即类型

```
RELATION
{ S# S#, SNAME NAME, STATUS INTEGER, CITY CHAR }
```

因此，这个类型实际上是一个生成的类型——正如所有关系类型以及所有元组类型也都是生成的一样。

#### 类型推断
确定给定表达式所表示值的类型的过程。请注意，此过程完全由定义表达式中所涉及的各种运算结果类型的规则所明确指定（实际上，表达式 *x* 所表示值的类型，正是 *x* 中涉及的最外层运算结果的类型）。

#### 类型继承
一种组织原则，根据该原则，一个类型可以被定义为一个或多个其他类型（称为超类型）的子类型。如果 *T2* 是超类型 *T1* 的（真）子类型，那么所有类型为 *T2* 的值也属于类型 *T1*，并且适用于类型 *T1* 值的只读运算符和类型约束会被类型 *T2* 的值继承。然而，类型 *T2* 的值会有其自身的只读运算符和类型约束，这些不适用于仅属于类型 *T1* 而不属于 *T2* 的值。

#### 去组
设 *s* 是一个关系，其具有一个类型为 `RELATION {Y}` 的属性 `YR`，并设 `{X}` 是 *s* 中除 `YR` 外所有属性的集合。设 `{Y}` 具有属性 `Y1`, `Y2`, ..., `Yn`；同时，设 `{X}` 不包含任何与 `Y1`, `Y2`, ..., `Yn` 中任何一个同名的属性。那么去组 `s UNGROUP ( YR)` 是另一个关系 *r*。*r* 的标题由 `{X}` 和 `{Y}` 的集合并组成。至于其主体，设 *z* 是一个关系，其标题由 `{X}` 扩展一个类型为 `TUPLE {Y}` 的属性 `YT` 组成，主体定义如下：对于 *s* 中的每个元组，*z* 包含一组元组，对于该 *s* 元组中 `YR` 值里的每个元组，包含一个（称为 *t*）；每个这样的元组 *t* 包含一个 `X` 值（称为 *x*）等于所讨论 *s* 元组中的 `X` 值，以及一个元组值（称为 *yt*）等于所讨论 *s* 元组中 `YR` 值里的某个元组。设 *z* 不包含其他元组。那么 *r* 是 `z UNWRAP ( YT)` 的结果。

**示例**：设 *spq* 是表达式 `SP GROUP ( { P#, QTY } AS PQ_REL )` 的结果关系（*参见* 分组）。那么表达式

```
spq UNGROUP ( PQ_REL )
```

表示对 *spq* 进行去组操作。该去组结果是一个类型为 `RELATION {S# S#, P# P#, QTY QTY}` 的关系；如果 *spq* 是从图 1 中作为关系变量 `SP` 的值所示的关系 *sp* 获得的，那么去组 *spq* 的结果就是 *sp* 本身。然而，假设 *spq* 额外包含一个元组，比如供应商 S5 的元组，其中 `PQ_REL` 值是一个空关系；那么前述 `UNGROUP` 的结果将不包含供应商 S5 的元组（实际上，结果仍然恰好是关系 *sp*）。因此，一般来说，对关系 *r* 进行去组操作，然后以看似逆向的方式再次对其进行分组，并不能保证回到 *r*（*对比* 分组）。

#### universal quantifier
令`p( x)`是一个带有参数`x`的谓词；那么`FORALL x ( p( x))`是一个谓词，它的意思是“对于所有可以替换参数`x`的参数`v`，`p( v`)为真。”在此例中，`FORALL x`是一个全称量词，`x`是一个全称量化约束变量，参见 q.v.。

*注*：一些作者单独将`FORALL`称为量词；文献对此并不一致。更重要的是，请注意，如果`v1, v2, . . ., vn`是上述示例中所有可能的参数，那么`FORALL x ( p( x))`等价于`( p( v1)) AND ( p( v2)) AND . . . AND ( p( vn)) AND TRUE`。特别要注意，当`n` = 0（即约束变量`x`的取值范围为空集）时，此表达式求值为`TRUE`。

另请注意，表达式`FORALL x ( p( x))`在逻辑上等价于表达式`NOT (EXISTS x (NOT ( p( x))))`。

*对比* existential quantifier。

### 示例
这是一个使用了全称量词和存在量词的元组演算查询（“获取供应所有零件的供应商”）：
```
SX RANGES OVER { S } ;
SPX RANGES OVER { SP } ;
PX RANGES OVER { P } ;
SX WHERE
FORALL PX ( EXISTS SPX ( SPX.S# = SX.S# AND
SPX.P# = PX.P# ) )
```
后一个表达式可以理解为“供应商 SX，其中对于所有零件 PX，存在一个装运 SPX，其供应商编号与 SX 相同，零件编号与 PX 相同。”

*参见* unwrapping。

#### unwrapping
令`s`是一个具有类型为`TUPLE { Y}`的属性`YT`的关系，并且`{ X}`是`s`中除`YT`外的所有属性的集合。令`{ Y}`具有属性`Y1, Y2, . . ., Yn`；同时，令`{ X}`不包含任何与`Y1, Y2, . . ., Yn`中任何一个同名的属性。那么解包操作`s UNWRAP ( YT)`会得到另一个关系`r`。`r`的标题由`{ X}`和`{ Y}`的集合论并集组成。`r`的主体包含一个元组，对应于`s`中的每个元组，没有其他元组。令`s`的元组`t`的`X`值为`x`，`YT`值为一个元组，其`Y1`值为`y1`，`Y2`值为`y2`，..., 以及`Yn`值为`yn`；那么`r`的对应元组具有`X`值`x`，`Y1`值`y1`，`Y2`值`y2`，..., 以及`Yn`值`yn`。*另见* tuple unwrapping。

### 示例
令`spw`是表达式`SP WRAP ( { P#, QTY } AS PQ_REL )`（*参见* wrapping）产生的关系。那么以下表达式表示对`spw`的一个解包：
```
spw UNWRAP ( PQ_REL )
```
该解包是一个类型为`RELATION {S# S#, P# P#, QTY QTY}`的关系。如果`spw`是通过包装图 1 中显示为关系变量 SP 值的关系`sp`得到的，那么解包`spw`的结果就是`sp`。

#### UPDATE
1. （*只读运算符*）*参见* what if。
2. （*更新运算符*）非常宽松地说，是一个在给定关系变量的给定元组集合中更新给定属性集合的运算符；稍宽松地说，是将给定关系变量中的给定元组集合替换为另一个这样的集合的运算符。它是某个关系赋值的简写。

### 示例（仅针对第二个定义）
`UPDATE`语句
```
UPDATE P WHERE CITY = 'London' :
{ WEIGHT := 2 * WEIGHT , CITY := 'Oslo' } ;
```
是以下关系赋值的简写：
```
P := WITH ( P WHERE CITY = 'London' ) AS t1 ,
( EXTEND t1 ADD
( 2 * WEIGHT AS NW ) ) AS t2 ,
( EXTEND t2 ADD
( 'Oslo' AS NC ) ) AS t3 ,
( t3 { P#, PNAME, COLOR, NW, NC } ) AS t4 ,
( t4 RENAME ( NW AS WEIGHT ) ) AS t5 ,
( t5 RENAME ( NC AS CITY ) AS t6 ,
( P MINUS t1 ) AS t7 :
t7 UNION t6 ;
```
在此例中，我们可能宽松地说，属性`WEIGHT`和`CITY`正在针对伦敦零件的元组进行更新；我们可能仍然宽松但稍少一点宽松地说，伦敦零件的元组正在被替换；但实际上发生的是，某个关系值被赋给了某个关系变量。

#### UPDATE rule
一种外键规则，参见 q.v.，指定当存在某个包含引用某个元组`t1`的外键值的元组`t2`并且——非常宽松地说（*参见* tuple level）——元组`t1`中对应的候选键被更新时，由 DBMS 采取的动作。

#### update
一种关系赋值（特别是`INSERT`、`DELETE`或`UPDATE`操作）。



### V

#### 值
一个“个体常量”（例如整数值`3`）。值可以具有任意复杂性（它们可以是标量或非标量；特别注意，元组和关系都是值）。值在时间或空间中没有位置；然而，它们可以通过某种编码在内存中表示，并且这些表示在时间和空间中确实有位置——实际上，同一个值的不同表示可以出现在时间和空间的任意多个不同位置，这意味着，粗略地说，同一个值可以同时或不同时地作为任意多个不同变量的当前值，和/或作为任意多个不同元组变量或关系变量的当前值中的任意多个属性值出现。

注意，根据定义，值不能被更新；因为如果它可以被更新，那么在那次更新之后，它就不再是那个值了。还要注意，每个值都属于某种类型（事实上，恰好属于一种类型，除非支持类型继承）。进一步注意，值不是类型，也不是变量。*对比* 外观。

#### 值集
在 E/R 建模中有时用来表示类型（type）的术语。

#### VAR
**Tutorial D** 中用于定义变量（特别是关系变量）的操作符。

#### 变量
1\.（*逻辑学*）参见 逻辑变量。
2\.（*编程语言*）值（`value`, q.v.）的表示的持有者。与值不同，变量（a）在时间或空间中具有位置，并且（b）可以被更新（即，变量的当前值可以被另一个值替换）。实际上，是变量就意味着可以被更新；等价地，是变量就意味着可以被赋值（而可以被赋值就意味着是变量）。注意，每个变量都被声明为某种类型。进一步注意，变量不是类型，也不是值。

#### 变量引用
1\.（*逻辑学*）参见 约束变量; 自由变量。
2\.（*编程语言*）在语法上，是一个变量名（除非在 伪变量引用 中注明），用于表示变量本身或该变量的值，具体取决于上下文。注意，如果此引用用于指定某个更新操作的目标——特别是，如果它出现在赋值语句的左侧——那么它肯定表示变量本身。另一方面，如果它表示变量的值，那么它可以被视为一个只读操作符（`read-only operator`）的调用——因此是表达式（`expression`, q.v.）的一种特殊情况——其中相关的只读操作符本质上是“返回指定变量的当前值”。

*注意*：如果变量引用`V`实际上表示`V`的值，那么（像所有表达式一样）它可以出现在任何适合类型的字面量可以出现的地方。另见 伪变量引用。

#### 垂直分解
分解为投影（projections）的非正式术语。

#### 视图
一种导出的关系变量（`relvar`），它是虚拟的，而非真实的（对比 `快照`）。

给定视图在给定时间的值，是在该时间计算某个关系表达式（视图定义表达式，在定义视图本身时指定）的结果。*注意*：视图定义表达式必须至少提及一个关系变量，否则它就不会是，具体来说，一个关系变量。同样注意，视图必须是可更新的，原因相同。

**示例**：以下语句定义了一个名为`LSV`的视图：
```
VAR LSV VIRTUAL ( S WHERE CITY = 'London' ) ;
```
在任何给定时间，作为视图`LSV`的值的关系，等于该时间的视图定义表达式`S WHERE CITY = 'London'`的值。

#### 视图物化
一种实现视图操作时技术上不太复杂的方法，根据该方法，（a）在操作被调用时计算定义视图的关系表达式，（b）由此物化出一个关系，然后（c）针对如此物化的关系执行相关操作。请注意，这种技术不能用于实现视图更新，而仅限于只读操作。对比 替换（第二个定义）。

另见 `物化视图`。

#### 视图更新
视图更新的理论或过程，视上下文而定。视图更新目前仍是一个有些争议的话题，但有人认为——与普遍看法相反——所有视图至少在理论上是可更新的。细节超出了本词典的范围，但需要注意（a）当然，视图更新可能因完整性约束违反而失败，就像基本关系变量更新一样，以及（b）有人认为某些人视为“不可能”的视图更新并非本质上不可能，而恰恰是因这种违反而失败。

#### 虚拟关系
给定虚拟关系变量（`virtual relvar`）在给定时间的值。

#### 虚拟关系变量
视图（`view`, q.v.）（对比 `真实关系变量`）。

### W

#### 合式公式
在逻辑学中，表示谓词（`predicate`）的形式表达式。

#### WFF
一个合式公式（`well-formed formula`）。缩写发音可以是“weff”、“wiff”或“woof”。参见 `闭合 WFF`; `开放 WFF`。

#### 假设分析
一种只读关系操作符，它返回如果对指定关系进行某些更改后将得到的关系（忽略这些更改实际上无法进行的事实，因为关系是值）。

**示例**：考虑以下表达式：
```
UPDATE S WHERE CITY = 'Paris' :
{ STATUS := 2 * STATUS , CITY := 'Nice' }
```



请注意，尽管它使用了关键字 `UPDATE`，但这个表达式确实是一个表达式而非语句（它没有终止的分号）；具体来说，它对关系变量 `S` 没有任何实际影响。它实际所做的是返回一个关系，该关系包含当前关系变量 `S` 值中，城市为巴黎的每个元组 `s` 所对应的一个元组 `t`——唯一的区别是，在元组 `t` 中，状态值是元组 `s` 中的两倍，并且城市变为了尼斯，而非巴黎。

#### WHERE 子句
一种形如 `WHERE bx`（其中 `bx` 是一个布尔表达式）的语法结构，广泛存在于 **Tutorial D**、SQL 及许多其他语言中。*参见* `DELETE`；限制；限制条件；`SELECT` 表达式；`UPDATE`；以及别处。

#### WITH
一种 **Tutorial D** 语法结构，用于为子表达式的结果引入名称。引入的名称随后可在整个表达式的后续部分（`WITH` 子句是其组成部分）中用来指代这些结果。（SQL 也包含一个 `WITH` 结构，其语义相似但不完全相同。）

**示例**：以下是一个实现查询“获取供应商编号对，记为 `S#` `x` 和 `S#` `y`，使得 `S#` `x` 和 `S#` `y` 各自供应完全相同的零件集合”的表述：

`WITH ( S RENAME ( S# AS SX ) ) { SX } AS tx,`
`( S RENAME ( S# AS SY ) ) { SY } AS ty :`
`( tx JOIN ty ) WHERE ( SP WHERE S# = SX ) { P# } =`
`( SP WHERE S# = SY ) { P# }`

#### WRAP
*参见* wrapping。

#### wrapping
设 `r` 是一个关系，并且 `r` 的标题被划分为子集 `{ X }` 和 `{ Y }`。设 `{ Y }` 的属性为 `Y1`、`Y2`、...、`Yn`；同时，设 `{ X }` 不包含任何名为 `YT` 的属性。那么，包装 `r WRAP ({ Y } AS YT)` 是另一个关系 `s`。`s` 的标题由 `{ X }` 扩展一个类型为 `TUPLE { Y }` 的属性 `YT` 构成。`s` 的主体包含 `r` 中每个元组所对应的一个元组，没有其他元组。设 `r` 的元组 `t` 具有 `X` 值 `x`、`Y1` 值 `y1`、`Y2` 值 `y2`、...、`Yn` 值 `yn`；那么 `s` 的对应元组将具有 `X` 值 `x` 和 `YT` 值（其值为一个元组，该元组包含 `Y1` 值 `y1`、`Y2` 值 `y2`、...、`Yn` 值 `yn`）。*另见* 元组包装。

**示例**：以下表达式表示关系变量 `SP` 当前值关系的一个包装：

`SP WRAP ( { P#, QTY } AS PQ_TUP )`

该包装是一个类型为 `RELATION { S# S#, PQ_TUP TUPLE { P# P#, QTY QTY } }` 的关系 `spw`；它包含关系变量 `SP` 当前出现的每个元组所对应的一个元组，没有其他元组。例如，给定图 1 中的示例值，供应商 `S1` 和零件 `P1` 对应的 `spw` 元组具有 `S#` 值 `S1` 和 `PQ_TUP` 值（其值为一个元组，该元组包含 `P#` 值 `P1` 和 `QTY` 值 300）。
