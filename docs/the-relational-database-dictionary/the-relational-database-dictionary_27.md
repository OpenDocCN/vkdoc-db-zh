#### EXISTS

对于前述示例，`EXISTS x ( p( x))` 等价于 `( p( v1)) OR ( p( v2)) OR . . . OR ( p( vn)) OR FALSE`。需特别注意，当 `n = 0`（即绑定变量 `x` 遍历一个空集）时，此表达式的求值结果为 `FALSE`。进一步观察可知，表达式 `EXISTS x ( p( x))` 在逻辑上等价于表达式 `NOT (FORALL x (NOT ( p( x))))`。

#### expression
在编程语言中，指只读的运算符调用；一种表示值的结构；因此，是计算或确定所讨论值的规则。每个表达式都具有某种类型——即其表示的值的类型。注意，常量和变量名（即变量引用，参见相关条目）都被视为只读的运算符调用，因此两者都构成合法的表达式。*对比* `statement`。

### 示例
`X+Y` 是一个表达式；实际上，它是“`+`”运算符的一个调用，它表示变量 `X` 和 `Y` 的当前值之和的值。相比之下，

```
Z := X + Y ;
```

是一条语句；它将右侧表达式 `X+Y` 所表示的值赋给左侧引用的变量 `Z`。*注意*：在以上两个例子中，`X` 和 `Y` 都是变量引用，因此它们本身也是表达式。

#### expression transformation
将给定表达式转换为另一个语义等价的表达式。该过程特别适用于关系表达式，在这种情况下有时被称为“查询重写”。

查询重写通常出于性能原因而进行；它可以由用户完成，但更重要的是由系统完成（*参见* `optimizer`）。*注意*：术语 `query rewrite` 在某些商业产品中也用于一个含义稍有限制的场景。*Caveat lector*（读者需留意）。

### 示例
关系表达式 `( r WHERE br) JOIN ( s WHERE bs)` 在语义上等价于关系表达式 `( r JOIN s) WHERE ( br) AND ( bs)`；因此，任何一个表达式都可以转换为另一个。将第二个转换为第一个从性能角度来看可能是有利的，因为第一个意味着先执行限制（`WHERE`）操作再执行连接（`JOIN`）操作；因此，连接的输入关系可能会更小，输出也会更小。事实上，这种转换可能决定了连接结果是能保留在内存中，还是不得不溢出到磁盘上。

#### extension
1. （*关系代数*）设 `r` 是一个关系。那么，扩展运算 `EXTEND r ADD ( exp AS X)` 是一个关系，其（a）头（heading）是 `r` 的头加上属性 `X`，且（b）体（body）由所有元组 `t` 组成，其中 `t` 是 `r` 的一个元组，其属性 `X` 的值是通过对 `r` 的该元组计算表达式 `exp` 而得出的。关系 `r` 不能有一个名为 `X` 的属性，并且 `exp` 不能引用 `X`。*另见* `tuple extension`。


### 2. （谓词）
令 `p` 为一个谓词；那么 `p` 的外延由 `p` 的所有完全实例化（即所有可以通过完全实例化从 `p` 推导出的命题）组成，这些命题的求值结果为 `TRUE`。

### 3. （关系）
承接上一定义，令 `r` 为一个关系。那么 `r` 的标题可以被视为表示了一个谓词（*参见* 关系谓词），而 `r` 的主体可以被视为表示了该谓词的外延。因此，术语 `外延` 有时也用于指代关系的主体。

*对比* 内涵。

### 4. （集合论）
*参见* 外延公理。

**示例**（仅针对第一个定义）：以下表达式表示了关系变量 `P` 的当前值所代表关系的一个外延：
```
EXTEND P ADD ( WEIGHT * 454 AS GMWT )
```
该外延是一个与关系变量 `P` 的当前值类似的关系，只是它多了一个额外的属性 `GMWT`（“克重”），该属性在任何给定元组中的值都是同一元组中 `WEIGHT` 值的 454 倍。请注意，关系变量 `P` 本身在数据库中保持不变——`EXTEND` 不像 `SQL` 中的 `ALTER TABLE`，它只是一个只读运算符（例如，与 `restrict` 类似），它以某个关系作为输入，并返回另一个关系作为输出。

更多示例见汇总运算。

#### 由超键隐含的 FD

#### FD 保持
将关系变量 `R` 分解为它的投影 `R1`， `R2`, . . ., `Rn`，使得 FD 被保持——即，`R` 满足的每一个 FD 都由 `R1`, `R2`, . . ., `Rn` 所满足的 FD 所隐含。这里的 `R1`, `R2`, . . ., `Rn` 被称为独立投影。*注*：非独立的投影被称为相互依赖的，而非依赖的。

*另见* 原子关系变量。

**示例**：假设关系变量 `S` 满足额外的 FD `{CITY}` → `{STATUS}`；即，给定供应商的状态是该供应商位置的函数。那么用其在 `{S#,SNAME,CITY}` 和 `{CITY,STATUS}` 上的投影替换 `S` 可以保持 FD，因为 `S` 满足的每一个 FD 要么被两个投影之一满足，要么被它们所满足的 FD 隐含。相比之下，假设 `S` 被替换为其在 `{S#,SNAME,CITY}` 和 `{S#,STATUS}` 上的投影。现在，FD `{CITY}` → `{STATUS}` 并未被这些投影满足的 FD 所隐含（即使分解是无损的）。一个实际的后果是，现在必须监控对两个投影中任何一个的更新（要么由 DBMS 监控，要么在当前实践中更可能由某个应用程序监控），以确保 FD `{CITY}` → `{STATUS}` 继续得到满足；例如，考虑将供应商 `S1` 从 `London` 移动到 `Paris` 所涉及的操作。换句话说，在后一种分解中，这两个投影不是独立的，而是相互依赖的。

考虑到当今商业产品中的技术水平，因此，通常更可取的做法是，在可能的情况下，以保持 FD 的方式进行分解——即，分解成独立投影。

#### 域（数学）
一种代数（*参见* algebra），其中运算符 “+” 和 “*” 具有实数加法和乘法的所有属性（交换律、结合律等）；等价地，一个遵循所有代数定律（*参见* The Laws of Algebra）的形式系统。

#### 第五范式
关系变量 `R` 是第五范式（`5NF`）的，当且仅当 `R` 满足的每个非平凡 JD 都由 `R` 的超键所隐含；等价地，当且仅当 `R` 满足的每个非平凡 JD `JD{ A1,A2,. . .,An }` 满足以下条件：(a) `A1`, `A2`, . . ., `An` 中的每一个都是 `R` 的超键，并且 (b) `R` 的键可以按某种顺序排列，使得该序列中每对相邻的键都包含在某个 `Ai` 中（1 ≤ `i` ≤ `n`）。每个 `5NF` 关系变量都是 `4NF` 的。*注*：`5NF` 是关于 JD 的“那个”范式。此外，虽然处于 `5NF` 显然并不排除也处于 `6NF`，但术语 `5NF` 常被宽泛地用于指代处于 `5NF` 但不处于 `6NF` 的关系变量。

观察到如果以下任一条件成立，则关系变量 `R` 肯定是 `5NF` 的：(a) 它处于 `3NF` 且没有复合键，或 (b) 它处于 `BCNF` 且至少有一个非键属性。

**示例**：如 Boyce/Codd 范式下所述，对于范式而言，展示一个反例通常比展示一个例子本身更具启发性。因此，考虑关系变量 `SPJ`，其属性为 `S#`（供应商编号）、`P#`（零件编号）和 `J#`（项目编号），谓词为“供应商 `S#` 向项目 `J#` 供应零件 `P#`。”令该关系变量是全键的（即，受键约束 `KEY{S#,P#,J#}` 约束）。并令它还受以下约束：如果 (a) 供应商 `Sx` 供应零件 `Py` 并且 (b) 零件 `Py` 被供应给项目 `Jz` 并且 (c) 项目 `Jz` 由供应商 `Sx` 供应，那么 (d) 供应商 `Sx` 向项目 `Jz` 供应零件 `Py`。那么 `SPJ` 等于其在 `{S#,P#}`、`{P#,J#}` 和 `{J#,S#}` 上的投影的连接——即它满足 JD `JD{{S#,P#},{P#,J#},{J#,S#}}`——因此它可以无损分解为这三个投影。由于该 JD 既不是平凡的，也未被唯一的超键（即整个标题）所隐含，关系变量 `SPJ` 不是 `5NF` 的，尽管它是 `4NF` 的。我们注意到关系变量 `SPJ` (a) 处于 `3NF` 但有复合键，并且 (b) 实际上处于 `BCNF` 但没有非键属性。

#### 文件
有时用来表示表（具有该词的任何可能含义），甚至关系或关系变量的术语。然而，所有此类用法都是不推荐的；最好将该术语保留给操作系统甚至物理层结构。

#### 第一个重大错误
将类型等同于关系或关系变量。*参见* 类型。

#### 第一范式
规范化的。根据定义，所有关系变量都是第一范式（`1NF`）的；也就是说，应用于关系变量时，术语 `1NF` 和 `规范化` 含义相同（*参见* 规范化以获取进一步解释）。由此可见，在像 `SQL` 这样的语言中，“表”可以被认为是处于 `1NF` 的，当且仅当它是某个关系变量的直接且忠实的表示，其中 *直接且忠实* 意味着该表中的每个行列交叉点（即每个单元格，*参见* cell）恰好包含适用类型的一个值，不多也不少。（该值可以任意复杂——它甚至可以是一个表——但重复一遍，必须恰好有一个这样的值，并且它必须是适用类型。）因此，特别地，如果表包含任何重复组（*参见* repeating group），则该表不处于第一范式。这一事实解释了通常对第一范式的非正式描述，即意味着 *没有重复组*。*注*：虽然处于 `1NF` 显然并不排除也处于 `2NF`，但术语 `1NF` 常被宽泛地用于指代仅处于 `1NF` 而不处于任何更高范式的关系变量。



#### 一阶逻辑

谓词逻辑的一种形式，其中变量所取值的集合不允许包含谓词。*对比* 二阶逻辑。*注*：命题逻辑（参见相关条目）有时可被视为一种“零阶”逻辑，因为它没有变量（因此其变量不取任何值）。

#### 扁平关系

“关系是扁平的”这一观点是一种普遍的误解（因此“扁平关系”这一说法强烈不建议使用）。*参见* 表。

#### 全称量词 (FORALL)

*参见* 全称量词。在文献中，`FORALL` 通常用一个倒置的 A 表示，即：∀。

#### 外键

设 `R1` 和 `R2` 为关系变量，它们不一定不同，并设 `K` 为 `R1` 的一个键。设 `FK` 是 `R2` 列表的一个子集，使得存在一个作用于 `R1` 上的、可能为空的属性重命名序列，该序列将 `K` 映射到 `K'`（假设），其中 `K'` 和 `FK` 包含完全相同的属性。此外，设 `R2` 和 `R1` 受到约束：在任何时候，`R2` 中的每个元组 `t2` 都有一个 `FK` 值，该值是当时 `R1` 中某个（必然是唯一的）元组 `t1` 的 `K'` 值。那么，`FK` 是一个外键，相关的约束是一个引用约束，而 `R2` 和 `R1` 分别是该约束的引用关系变量和对应的被引用关系变量。此外，`K`（而非 `K'`）有时被称为被引用键或目标键。*注*：引用和被引用的术语同样适用于元组；即前述讨论中的元组 `t2` 和 `t1` 分别是一个引用元组和对应的被引用元组。被引用元组有时也称为目标元组。

### 示例

在关系变量 `SP` 中，`{S#}` 和 `{P#}` 是分别对应于关系变量 `S` 和 `P` 中键 `{S#}` 和 `{P#}` 的外键。可以如下指定它们：

```
VAR SP BASE RELATION
{ S# S#, P# P#, QTY QTY }
KEY { S#, P# }
FOREIGN KEY { S# } REFERENCES S
FOREIGN KEY { P# } REFERENCES P ;
```

相比之下，下面是一个需要进行属性重命名的示例：

```
VAR EMP BASE RELATION
{ E# E#, ..., M# E#, ... }
KEY { E# }
FOREIGN KEY { M# } REFERENCES
( EMP { E# } ) RENAME ( E# AS M# ) ;
```

这里的属性 `M#` 表示由 `E#` 标识的员工的经理的员工编号（在此示例中，引用关系变量和被引用关系变量是同一个）。因此，例如，员工 `E3` 的 `EMP` 元组可能包含一个 `M#` 值 `E2`，这构成了对员工 `E2` 的 `EMP` 元组的引用。

请注意，没有要求被引用关系变量中对应于给定外键的键必须是主键，也没有要求引用关系变量和被引用关系变量必须是基本关系变量。

#### 第四范式（fourth normal form）
关系变量 `R` 是第四范式（`4NF`），当且仅当 `R` 所满足的每一个非平凡 MVD 都被 `R` 的某个超键所蕴含；等价地说，当且仅当对于 `R` 所满足的每一个非平凡 MVD `A →→ B`，`A` 都是 `R` 的一个超键（并且该 MVD `A →→ B` 实际上退化为 FD `A → B`）。每一个 `4NF` 的关系变量都处于 `BCNF`。*注意*：`4NF` 是针对 MVD 的“那个”范式。另外，虽然处于 `4NF` 显然并不妨碍同时也处于更高的范式（`5NF`），但术语 *4NF* 常被不严格地用来指代处于 `4NF` 但不处于 `5NF` 的关系变量。

无论如何，第四范式本身已不再非常重要（`BCNF`、`5NF` 和 `6NF` 是具有最大实际意义的范式）；我们在此提及它主要是出于历史原因。

**示例：** 如 Boyce/Codd 范式下所述，对于范式而言，展示一个反例通常比展示一个正例更具启发性。因此，考虑关系变量 `CTX`，其属性为 `C`（课程）、`T`（教师）和 `X`（教材），其谓词是“课程 `C` 可由教师 `T` 讲授，并使用教材 `X` 作为教科书。”假设该关系变量是全键的（即，受限于键约束 `KEY{C,T,X}`）。同时假设对于一门给定的课程，教师集合和教材集合是相互独立的。那么，`CTX` 等于其在 `{C,T}` 和 `{C,X}` 上的投影的连接——也就是说，它满足 MVD `{C} →→ {T}` 和 `{C} →→ {X}`——因此它可以无损分解为那两个投影。由于这些 MVD 既非平凡的，也未被唯一的超键（即整个标题）所蕴含，关系变量 `CTX` 不处于 `4NF`，尽管它处于 `BCNF`。

#### 自由变量（free variable）
在逻辑中，指一个变量——更准确地说，是某个谓词内出现的变量引用——它未被绑定；换句话说，是一个参数。（此处术语 *variable* 是在逻辑意义上使用，而非编程语言意义上的。）*对比* 绑定变量（bound variable）。

**示例：** 令符号 `x` 和 `y` 表示整数。那么下面两个表达式都是谓词，并且 `x` 在两者中都作为自由变量出现：

```
x < 7
EXISTS y ( y > 3 ) AND x < 7
```

第一个谓词不言自明。第二个稍微复杂一些，因为它涉及一个量化子表达式（其中 `y` 出现了两次，作为绑定变量）以及自由变量 `x`。

转向一个数据库示例，下面是一个在供应商-零件数据库上的查询（“获取至少供应一种零件的供应商”），用元组演算表示：

```
S WHERE EXISTS SP ( SP.S# = S.S# )
```

这里关键字 `WHERE` 后面的布尔表达式是一个谓词，其中对 `S` 的引用是自由的（相比之下，对 `SP` 的引用是绑定的）。然而，请注意，在这个特定例子中，符号 `S` 和 `SP` 不仅表示逻辑意义上的变量，也表示传统编程语言意义上的变量——但那是因为我们可以说是在某种程度上耍了个小花招。以下是同一个示例的扩展版本，应有助于澄清问题：

```
SX RANGES OVER { S } ;
SPX RANGES OVER { SP } ;
SX WHERE EXISTS SPX ( SPX.S# = SX.S# )
```

#### 函数（function）
1.  （*数学*）给定两个集合（不一定不同），一个规则——也称为映射（map 或 mapping）——将第一个集合（定义域）的每个元素与第二个集合（陪域）的恰好一个元素配对；等价地说，是构成该配对的序对 `<x, y>` 的集合。

2.  （*编程语言*）一个只读运算符（有时更具体地指由标识符如 `DIV` 表示的运算符，而非由像“/”这样的特殊符号表示）。然而，请注意，这个术语所表示的编程语言结构正是数学意义上的函数；因此，这里实际上只有一个概念，而非两个。还要注意，定义中并未要求定义域和陪域必须是标量集合；因此，一个只读运算符可以依据，比如说，三个参数来定义，在这种情况下，定义域将由一组三元组构成。类似的说明也适用于陪域。

#### 函数依赖（functional dependency）

同样，虽然严格来说并不正确，但人们常会这样描述属性间的依赖关系：仿佛`B`是函数依赖于`A`中的属性本身，而非依赖于包含这些属性的集合`A`。最后请注意，从形式化角度来看，函数依赖`A → B`只是一个命题，根据定义，它要么被满足，要么不被满足（即其值可为真或假）；然而在非正式语境中，通常只有当相关关系变量实际满足某个依赖时，才将`A → B`定义为函数依赖。但这种定义方式使得我们无法表述某个给定关系变量未能满足或违反了某个函数依赖（因为根据该定义，未被满足的依赖就不是函数依赖！）。

#### 大除（Great Divide）
多年来定义的众多关系除法算符之一（*参见* 除法）。设关系`r1, r2, r3`和`r4`满足：(a) `r1`和`r2`没有共同的属性名；(b) `r1`和`r3`中同名的属性属于同一类型，`r3`和`r4`中同名的属性属于同一类型，`r4`和`r2`中同名的属性也属于同一类型（换句话说，`r1`和`r3`可连接，`r3`和`r4`可连接，`r4`和`r2`也可连接）；(c) 集合`{ X}`、`{ Y}`和`{ Z}`分别是`r1`和`r3`、`r3`和`r4`、以及`r4`和`r2`的共同属性集；(d) 集合`{ X}`和`{ Y}`不相交，集合`{ Y}`和`{ Z}`也不相交。那么，除法`r1 DIVIDEBY r2 PER (r3,r4)`——其中`r1`是被除数，`r2`是除数，`r3`和`r4`是“中介”——是一个关系，其标题为`r1`和`r2`标题的集合并，其主体定义如下：当且仅当元组`t`出现在`r1 JOIN r2`中，并且对于`t`中`Z`值对应的`r4{Y,Z}`中的所有元组`<y,z>`，都存在一个`<x,y>`元组（其中`x`等于`t`中的`X`值）出现在`r3{X,Y}`中时，元组`t`才出现在该主体中。换句话说，表达式`r1 DIVIDEBY r2 PER (r3,r4)`在语义上等价于表达式`(r1 JOIN r2) NOT MATCHING ((r1{X} JOIN r4{Y,Z}) NOT MATCHING r3)`。*对比* 小除。

#### 希斯定理（Heath's theorem）

设 *A, B,* 和 *C* 是关系变量 *R* 的头部的子集，使得 *A, B,* 和 *C* 的集合理并集等于该头部。设 *AB* 表示 *A* 和 *B* 的集合理并集，*AC* 同理。如果 *R* 满足函数依赖 *A* → *B,* 那么 *R* 等于其在 *AB* 和 *AC* 上投影的连接。注意，这个定理的逆命题是错误的；也就是说，仅仅因为 *R* 等于其在 *AB* 和 *AC* 上投影的连接，并不意味着 *R* 满足函数依赖 *A* → *B*。（但是，如果我们将函数依赖 *A* → *B* 替换为多值依赖 *A* →→ *B,* 那么得到的陈述在两个方向上都是正确的；也就是说，*R* 等于其在 *AB* 和 *AC* 上投影的连接，*当且仅当*它满足多值依赖 *A* →→ *B*。*参见* multivalued dependency。）

#### 幂等
设 `Op` 为一个双目运算符，并假设为明确起见，`Op` 以中缀风格表示。那么，当且仅当对于所有 `x`，`x Op x = x` 时，`Op` 才是**幂等**的。

#### 恒等
1.  （**通用**）区分给定实体与其他所有实体的属性。
2.  （**运算符**）相等性。
3.  （**逻辑**）相等性；也指形式为 `(p) EQUIV (q)` 的重言式。
4.  （**比较**）一种布尔表达式，形式为 `(exp1) = (exp2)`，其中 `exp1` 和 `exp2` 是相同类型的表达式，并且保证无论涉及的任何变量的取值如何，其求值结果都为 TRUE。在实践中，包围 `exp1` 和 `exp2` 的括号可能不需要。
5.  （**恒等值**）设 `Op` 为一个可交换的双目运算符，并假设为明确起见，`Op` 以中缀风格表示。如果存在一个值 `i`，使得对于所有可能的值 `v`，`i Op v` 和 `v Op i` 都等于 `v`，那么 `i` 就是关于 `Op` 的**恒等值**（参见 `代数定律`）。**注**：恒等值也被称为恒等元、中性元、单位元，或者有时简称为单位。

#### 像关系
在 **Tutorial D** 中，指形式为 `‼r` 的表达式（其中 `r` 是一个关系表达式）所表示的值（一个关系）。符号 `‼` 读作 “bang bang” 或 “double bang”。这样的表达式可以出现在 (a) `WHERE` 子句中或 (b) `EXTEND` 调用中的 `ADD` 规范中（在每种情况下，只要关系表达式可以出现的地方都可以）。为明确起见，假设表达式 `‼r2` 出现在形式为 `r1 WHERE bx` 的关系表达式的布尔表达式 `bx` 中（这是情况 (a) 的一个特例；情况 (b) 和情况 (a) 的其他实例可类似定义）。此处的关系 `r1` 和 `r2` 必须满足同名属性属于同一类型（换句话说，`r1` 和 `r2` 必须是可连接的）；设它们的公共属性为 `A1, A2, . . . , An`（`n ≥ 0`）。那么表达式 `‼r2` 在语义上等价于表达式 `(r2) MATCHING RELATION {TUPLE {A1 A1, A2 A2, . . ., An An}}) {ALL BUT A1, A2, . . ., An}`，其中，对于元组选择器调用 `TUPLE {...}` 中的每一对形式为 `Ai Ai`（`i = 1, 2, . . ., n`）的项，第一个 `Ai` 是属性名，第二个是对 `r1` 中该名称属性的引用。

**示例**：表达式
```
S WHERE ( ‼SP ) { P# } = P { P# }
```
生成一个关系，其标题与关系变量 `S` 相同，其主体由来自关系变量 `S` 的所有可能的元组 `<s#,sn,st,sc>` 组成，这些元组对应的供应商 `s#` 供应关系变量 `P` 中提到的所有零件。（给定图 1 的示例值，结果只包含供应商 S1 的元组。）该表达式在语义上等价于这个表达式：
```
S WHERE ( ( SP MATCHING RELATION { TUPLE { S# S# } } ) { ALL BUT S# } ) { P# } = P { P# }
```
详细说明：设 `s` 和 `sp` 分别为关系变量 `S` 和 `SP` 的当前值。对于 `s` 中的一个给定元组 `t`，表达式（实际上是一个关系选择器调用）`RELATION {TUPLE {S# S#}}` 求值为一个只包含一个属性 `S#` 和一个元组的关系，并且该元组只包含来自 `t` 的 `S#` 值。因此，对应的像关系——即对应于元组 `t`，由表达式 `‼SP` 表示的关系——只包含 `SP` 中与该元组 `t` 匹配的那些元组，投影到 `{P#,QTY}`（`= {ALL BUT S#}`）上。因此，投影 `(‼SP){P#}` 求值为一个关系，记为 `ps`，该关系只有一个属性 `P#`，给出了与元组 `t` 对应的供应商所供应的所有零件的零件号。对于该供应商，布尔表达式 `(‼SP){P#} = P{P#}` 随后测试对应的关系 `ps`，看它是否等于 `P` 在 `{P#}` 上的投影。当且仅当与元组 `t` 对应的供应商供应了 `P` 中提到的所有零件时，该测试才会得到 TRUE。

下面是一个情况 (b) 的示例：
```
EXTEND S { S# } ADD ( COUNT ( ‼SP ) AS CT )
```
该表达式生成一个类型为 `RELATION {S# S#, CT INTEGER}` 的关系，为当前出现在关系变量 `S` 中的每个不同的 `S#` 值包含一个元组，且不包含其他元组。每个这样的元组包含一个供应商编号和一个计数，该计数表示该供应商编号当前在关系变量 `SP` 中出现的次数（表达式 `‼SP` 再次是表达式 `SP MATCHING RELATION {TUPLE {S# S#}}) {ALL BUT S#}` 的简写）。例如，给定图 1 中的示例值，结果中供应商 S2 的元组具有 `S#` 值 S2 和 `CT` 值二，供应商 S5 的元组具有 `S#` 值 S5 和 `CT` 值零。

#### 立即检查
在执行可能导致完整性约束被违反的更新时检查该约束。在关系模型中，所有约束检查都是**即时**的。*对比* `延迟检查`。

#### 即时约束
一种完整性约束，对其检查是即时的（参见 `立即检查`）。在关系模型中，所有约束都是**即时**的。*对比* `延迟约束`。

#### 实现
在真实机器上对构成某个给定数据模型（按该术语的第一个定义）的抽象机器的物理实现。为了物理数据独立性，模型及其**实现**应该严格分离；也就是说，模型应该对实现的任何方面都无话可说。



#### 实现定义
该术语（尤其在 SQL 标准中）用于指代其语义可能因不同实现而异，但至少必须为每个具体实现明确规定的特性。换言之，实现方可以自由决定如何处理该特性，但该决策的结果必须记录在案。例如，SQL 中的字符串最大长度即属于此类。
