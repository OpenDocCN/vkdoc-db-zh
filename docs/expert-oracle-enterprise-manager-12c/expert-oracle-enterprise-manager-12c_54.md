# Enterprise Manager 作业管理

## 作业操作说明

| 操作 | 描述 |
| --- | --- |
| 创建 | 创建一个操作系统或 SQL 脚本作业，以及其他类型的作业。 |
| 编辑 | 更新现有作业，更改计划时间、间隔、参数等。 |
| 类似创建 | 从现有作业创建一个新作业，可以选择编辑现有参数、计划等。（备份作业和某些其他维护作业可能不允许使用“类似创建”选项。） |
| 删除 | 从 EM 作业活动视图中移除单个计划/失败的作业，或移除该作业的所有实例。（任何库副本将被保留。） |
| 挂起 | 将作业“挂起”直到恢复，使现有计划得以继续。 |
| 恢复 | 恢复已挂起的作业计划。作业必须单独恢复。 |
| 失败时重试 | 如果作业属于允许的作业类型列表，则重试该作业。 |

## 作业库

Enterprise Manager 作业库允许提交作业到 EM 作业调度程序的存储库。此存储库包含管理员在备份和 EM 控制台处理之外创建的所有作业。该库仅保留已保存到其中的内容，并且必须与活动作业的更改保持同步。如果能有一个导出程序将当前作业的快照保存到库中，那将是非常有益的，但不幸的是目前并不存在。因此，务必将所有更改过的作业（包括简单的执行时间更改）从“作业活动”页面保存到库中。如果因任何原因必须参考库来重新创建作业，这将确保作业正确地重新提交到调度程序。

对于早期 10g 和 11g EM 作业的用户，没有将作业迁移到新 EM12c 环境的过程。此功能在新版 EM12c 中同样缺失，但 Oracle 计划在即将发布的版本中包含一个。此功能应具有多重好处，不仅适用于迁移，也作为升级的一部分、备份/恢复的次选方案以及复制的一部分。

作业库界面（如图 11-9 所示）与作业活动页面类似，并使用许多相似的选项，包括“编辑”和“访问”。

```sql
AND  mjes.start_time > sysdate-1
and mjes.retried is not null;
```

### 权限与责任

Enterprise Manager 在 12c 版本中扩展了对作业类型的支持。虽然 `操作系统命令`、`SQL 脚本` 和 `RMAN` 可能很常见，但现在 EM 作业环境支持数十种作业类型：



### 12.5.2 作业类型、作业事件和通知

Enterprise Manager 中存在多种作业类型，以下是一些常见的和新增的类型：

*   `Application Development Protocol (ADP)`: Oracle ADF 使用的作业协议。
*   `Agent Configuration and Control Operations`: 这是一系列 Enterprise Manager 计划作业类型中的前两种。这些作业是在 EM 控制台中作为代理配置或控制请求的一部分创建的。
*   `ASM Scripts`: 自动存储管理中的许多任务被安排为 EM 控制台中的作业。这是另一种由 EM 控制台创建的作业，与通过 EM Jobs 界面创建的作业相对。
*   `Clone Home`: 这是最受期待的新增功能之一，这种新的作业类型允许您轻松地将现有的 Oracle 主目录克隆到一个新的 Oracle 主目录。
*   `Database Configuration`: 另一种 EM 控制台作业类型，它可以在不要求您在目标服务器上执行任务的情况下更新数据库配置。
*   `Deployments`: EM12c 的一项新功能，此作业从 EM 控制台部署到目标，用于监控和配置。
*   `Discovery of Specific Oracle Components`: 作业也会被安排用于通过 EM 控制台在指定的 Oracle 目标上发现 Oracle 组件。这种自动发现通过从服务器动态提取配置和目标信息来节省时间和精力，确保任何目标的正确配置。
*   `Log Rotation`: 如果日志、跟踪文件和 Enterprise Manager 日志变得太大，可能会难以管理。这种作业类型可以在 EM Jobs 界面中创建，用于管理您定期访问和监控的许多数据库日志。
*   `OPatch Updates`: EM12c 的另一种新作业类型，自动化的 `OPatch` 更新确保您在 EM12c 环境中监控的每个目标上都安装了最新版本的 `OPatch` 实用程序。
*   `OS Commands`: 这种作业类型允许您将现有的 `OS` 脚本从 `cron` 调度程序或其他 `OS` 级调度程序迁移到 Enterprise Manager 控制台内的单一调度程序中。通过迁移到 EM Jobs 界面，在 EM 中断期间无需再通过第二个调度程序进行管理，所有操作都可以通过单一界面管理。
*   `RMAN`: 所有的 `Recovery Manager` 作业，虽然通过 EM 控制台中的 `RMAN` 界面创建，但它们会在 Job Activity 界面中显示为计划作业。这些作业类型无法保存到库中，并且在 Job Activity 界面中的编辑功能有限。
*   `Security`: 这种作业类型涉及从 EM 控制台的 `Setup` -> `Security` 部分访问的安全界面。这些作业包括针对管理员、角色、凭据和审计数据的作业。
*   `SQL Scripts`: 涉及 EM 作业的最常见作业类型，这允许您执行 `SQL`、存储过程以及任何其他将在 `SQL*Plus` 会话中执行的任务。
*   `Multitask Jobs`: 对于需要多个步骤来完成一个过程的作业，这是 EM12c 中的一项新功能。过去，您必须为复杂的、多步骤的过程编写脚本。现在，EM12c Jobs 界面使得创建具有多个步骤的作业变得简单，包括那些在每个步骤都有成功逻辑请求的作业。

定位 EM12c Jobs Service 支持的常见和新的作业类型非常简单。以下查询不仅显示了支持的作业类型，还显示了用于 `EMCLI` 执行的作业类型 ID。

```sql
select * from sysman.MGMT_JOB_TYPE_INFO
where job_type_category=0
and editable=1
and agent_bound=1
and job_type not like '%Obsolete%'
and job_type_default is not null;
```

图 11-23 显示了查询结果，其中包含了来自 `OMS` 存储库数据库中 `SYSMAN` 模式的作业类型以及每个作业类型的任何相关信息。这些数据清楚地显示了支持前端的作业类型 (`job_type_category=0`)、在作业创建后能够被编辑的作业类型（例如，备份作业在备份和恢复调度程序外部创建后无法被编辑），并且所有作业类型都可用于代理（`agent_bound` 与 `OMS` 级别的作业相对）。如图所示，所有过时的作业类型都被跳过，但仍在存储库中列出。

### 12.5.3 有效监控和作业状态

在 EM 作业系统中运行的任何给定作业的状态都极其重要，因为通知和警报都基于此状态。由于一个作业可能包含多个步骤，状态会随着每个步骤的执行而变化，并且可以触发事件来提醒管理员作业运行的不同状态。此作业状态可以通过作业事件进行监控、警报和管理。

作业事件返回报告状态并提醒责任方。您可以手动设置这些事件，因为作业事件默认是未启用的。这样做是为了避免系统因事件监控而过载。

作业事件如下：

*   `All`: 报告任何作业的所有状态。
*   `Scheduled`: 作业已计划运行。
*   `Action Required`: 需要管理员介入作业才能运行完成。这包括因计划中断而被跳过的作业。
*   `Suspended`: 作业已被暂停，因此无法执行其常规计划。
*   `Succeeded`: 作业已成功执行。
*   `Problems`: 一个或多个作业步骤中发生了故障。

可以通过 Job Events Generation Criteria 页面配置作业事件。您可以通过在 EM 控制台中选择 `Setup` -> `Incidents` -> `Job Events` 来访问此页面，如图 图 11-24 所示。随后将出现第 1 步页面，如图 图 11-25 所示。在这里，您可以决定默认情况下哪些事件会发送警报通知，并且可以按目标或组缩小范围。

一些管理员选择接收作业运行成功 (`Succeeded`) 的通知。根据我的专业经验，我建议不要配置作业来执行此通知。管理员在警报上花费的大部分时间并非由于监控不足，而是监控过度。如果 Enterprise Manager 被配置为包含电子邮件“成功”通知，您会习惯于接收它们，知道它们不需要响应，这可能导致错过关键问题通知。

对于那些希望关闭个别成功消息并简单地每天报告一次这些成功和作业失败的管理员，可以设置一个辅助作业来报告并消除不必要的个别通知。

where mj.job_id=mjes.job_id
and mjes.start_time >= sysdate-1
and mj.job_status!=0
and mj.expired=0;

### 从活动视图编辑作业
计划运行的作业以及已执行的作业都可以进行编辑。（取决于作业类型，例如备份，可能只能进行微小的更改。）之前已执行的作业随后可以作为新作业重新提交。当您更新一个新作业并做出更改时，这些更改也会提交给该作业的任何未来计划间隔。这是一个重要的区别，因为如果作业只需要一次性更改，那么更好的选择可能是对现有作业执行一次“类似创建”并安排单次执行，以确保常规执行时间不受影响。

### 停止正在运行的作业
点击“停止”按钮将停止该作业及其所有后续运行。作业将不再有未来的调度计划。

### 挂起作业
点击“挂起”按钮是将作业“搁置”以进行维护（这将导致目标暂时不可用）的正确方法。您可以在稍后时间重新启动该作业。

### 复制到库
“复制到库”选项将选定的作业复制到库中，以便以后访问、编辑、提交等。备份作业、数据泵作业和其他杂项作业无法保留在库中，如果您尝试保存它们，将会失败。这些作业可以通过 EM12c 的备份和恢复来保留，但目前，备份和数据泵作业没有库。

### 删除作业
删除作业时，可以选择删除该作业的特定运行，或者从作业系统中删除所有作业（包括历史记录）。请确保，如果您选择删除所有作业历史记录，不会违反业务所需的任何保留策略。删除后，作业将从调度程序中移除，必须重新提交。因此，如果您删除了所有历史记录且作业信息将来还需要使用，确保作业保留在库中就很重要。

### 管理管理器
EM12c 第 2 版增加了一个名为“管理管理器”(MTM)的作业系统页面。此页面使您能够执行类似于“作业活动”、“作业库”和“作业健康状况监视器”页面上的任务，但提供了更增强、更完整的作业环境视图。

MTM 显示以下内容：
*   EM 作业系统的标准健康状况
*   作业状态
*   诊断信息
*   代理管理
*   跟踪的上传间隔和通知

虽然直到 EM12c 第 2 版才可用，但“健康状况概览”页面现在结合了环境的整体负载和活动以及特定于目标的作业负载。

# 第 12 章 事件管理

作者：Pete Sharman

事件管理的目标是使管理员能够尽可能快速、高效地监控和解决其数据中心中可能正在发生的服务中断。我们的目标不是管理这些服务中断可能引发的众多离散事件，而是管理数量更少但更有意义的问题事件，并在这些问题事件的整个生命周期中根据业务优先级进行管理。

为此，EM12c 提供了一个名为“事件管理器”的集中式事件控制台，使管理员能够跟踪、诊断和解决这些问题事件，同时还提供功能帮助消除重复问题事件的根本原因。事件管理器还包含与 Oracle 专业知识对接的功能，通过相关的 My Oracle Support 知识库文章和文档，使管理员能够加速诊断和解决问题事件与故障的过程。最后，事件管理器还提供了对问题事件执行生命周期操作的能力，因此您可以将问题事件的所有权分配给特定用户、确认问题事件、设置问题事件的优先级、跟踪问题事件的状态、升级问题事件或将其抑制以便推迟到以后处理。您还可以针对问题事件发起通知，或通过服务台连接器创建服务台工单。

由于事件管理器是 EM12c 中的全新功能，本章将提供以下内容：

*   对新术语的解释，包括事件、问题事件和问题
*   用户界面介绍，以便您能够设置事件管理器
*   一些关于如何充分利用此新功能的建议指南

## 事件管理器术语

在深入探讨如何使用事件管理器功能之前，您需要了解一些新术语。让我们从了解事件和问题事件，以及它们之间的区别开始。

### 事件和问题事件

企业管理器仍然是管理和监控 Oracle 数据中心的主要工具，因此它也管理和监控从应用服务器到数据库再到主机及其操作系统的应用程序堆栈。当企业管理器检测到此基础架构中的任何问题时，它会引发事件。*事件* 可能是以下任何一种：

*   *指标警报*：这些警报（例如，CPU 利用率或表空间使用警报）表示您设置的关键阈值已被突破。
*   *作业事件*：这些事件可能是由于您使用作业系统时作业失败引起的。会引发一个事件来发出特定作业失败的信号。
*   *标准违规*：如果您正在使用合规性标准，并且任何被监控的目标违反了这些标准，则可能引发标准违规事件。
*   *可用性事件*：如果企业管理器检测到目标宕机，则可能引发可用性事件。
*   *其他事件*：也可能发生其他类型的事件，例如 SLA 警报或高可用性与连接器外部类。当然，用户可以定义自己的事件类型并引发事件。

所有这些事件都表明 Oracle 数据中心中发生了特定问题。在所有事件中，您真正希望能够确定哪些是重要的，并且能够从这些重要事件中关联出那些与相同潜在问题相关的事件。归根结底，作为管理员，您想要管理的是数量更少的重要问题事件。您可能有一个性能问题事件，它合并了多个性能事件；另一个与空间相关的问题事件；以及一个基于可用性问题的不同问题事件。因此，事件管理的目标不是管理数量众多的单个事件，而是管理数量更少、更有意义的问题事件。一个问题事件可以是一个重大事件（例如，目标宕机），也可以是一组都与同一问题相关的事件的组合（例如，空间耗尽可能被企业管理器检测为从数据库、主机和存储目标类型引发的独立事件）。

事件是企业管理器检测到并在您的 IT 基础设施中引发的重要事件。每个事件都有一组属性，表明事件的类型、其严重性（致命、严重等）、引发事件的对象或实体（通常是一个目标，但也可能是作业或其他对象）、与事件关联的消息、发生时间戳以及功能类别（如可用性、安全等）。

与这些事件相关联的是事件严重性。第一个是“致命”，这是企业管理器中一个新增的严重性级别，专门与目标可用性事件类型关联，表示目标已宕机。“严重”和“警告”事件的含义与先前版本中的含义相同。接下来，“建议”级别通常与合规性标准违规事件关联。“信息”级别仅用于表示事件已发生，但无需对此采取任何措施。

如前所述，一个问题事件可以包含一个或多个事件。让我们来看一个包含一个事件的问题事件的详细信息。图 12-1 显示了一个包含一个可用性事件的问题事件。

![9781430249382_Fig12-01.jpg](img/9781430249382_Fig12-01.jpg)

图 12-1。包含一个事件的问题事件

该事件发出信号表明数据库 DB1 已宕机，并包含一个指示事件引发时间的时间戳。由于这是一个目标可用性事件且数据库已宕机，因此严重性标记为“致命”。可以为该事件创建一个问题事件，因此该问题事件仅包含一个事件。为了管理和跟踪问题事件的解决情况，问题事件还具有其他属性，例如所有者（正在处理该问题事件的企业管理器用户）、状态、问题事件严重性（基于事件严重性）、优先级和一个注释字段。

### 问题

`问题`是事件的根本原因。在 `Enterprise Manager` 的术语中，问题特指与 `Automatic Diagnostic Repository (ADR)` 事件或 Oracle 软件事件相关的问题。每当 `Enterprise Manager` 检测到 `ADR` 事件被引发时，它会自动创建一个问题。一个 `ADR` 事件可被视为一个关键的 Oracle 软件问题，通常需要联系 Oracle 支持部门、提出服务请求，并可能收到针对该问题的修补程序。

每当引发一个 `ADR` 事件时，我们会为该 `ADR` 事件在 `Enterprise Manager` 中生成一个事件，同时也会自动生成一个问题对象。所有具有相同问题签名（即相同根本原因）的 `ADR` 事件将被链接到单个问题对象中（例如，具有相同参数的 `ORA-0600` 错误的 `ADR` 事件将被链接到同一个问题对象）。管理员可以在 `Incident Manager` 中管理问题，其方式与管理事件相同，因此您可以为问题分配负责人、跟踪解决进度等。此外，还有指向 `Support Workbench` 功能的上下文链接，允许管理员打包诊断材料、提出服务请求，并在用户界面中查看诊断活动的状态，例如 SR 编号以及最终的 Bug 编号（如果生成了的话）。

### 事件规则集

`事件规则集`是对通知规则的替代和增强。它们提供了一种自动化与事件、事件或问题相关的操作的方法。事件规则集的一些常见场景或用例包括：基于事件创建事件、发送电子邮件或寻呼等通知、创建帮助台工单，以及自动化事件工作流操作（例如，自动分配事件负责人或在事件打开一段时间后将其升级）。

`Enterprise Manager Cloud Control 12.1` 中有两种类型的事件规则集：企业规则集和私有规则集。`企业规则集`旨在用于实施您在管理事件和事件方面的操作实践。我们前面讨论的所有操作，例如创建事件、发送通知和创建工单，都在企业规则集类型中得到支持。由于这些操作会影响所有类型的事件和问题，因此创建这些规则集的用户需要具有 `Rule Set` 资源权限。一旦创建了企业类型规则集，它对所有 `Enterprise Manager` 用户都是可见的。当您创建规则集时，可以对该规则集进行进一步的协作开发，因此每个规则集基本上可以有多个作者（即多个 `Enterprise Manager` 用户）可以编辑该特定规则集。而 `私有规则集` 仅设计用于向您自己发送电子邮件通知。因此，创建私有规则集不需要特殊权限。

规则集被定义为应用于一组通用对象（如目标（包括组）或作业）的一个或多个规则的集合。在组内，您可以为规则集指定一组异构目标（即一组不同的目标类型）进行应用。规则集的理念是使您能够将与同一对象相关的不同规则逻辑地组合到一个可管理的单元中。这是我们在早期版本中针对通知规则多次听到的一项增强请求，当时管理员经常基于目标类型创建多个通知规则，最终这些规则都对同一个组进行操作。将这些规则逻辑地组合成一个单元，是从早期功能自然演进而来的，使得跟踪和管理这些多个规则更加容易。

特定规则集内的规则按特定顺序执行。默认情况下，规则按创建顺序执行，但您可以随时更改此顺序。规则集本身也可以按特定顺序执行。同样，默认情况下这是规则集的创建顺序，但也可以更改。

`Enterprise Manager Cloud Control 12.1` 附带提供的一些开箱即用的规则集，会自动为 Oracle 公司认为有意义的事件创建事件，并自动删除事件。您可以按原样使用这些规则集，但无法编辑它们。不过，您可以使用 `Create Like` 功能创建自己的版本，如果原始规则集不符合您的要求，可以将其禁用。

规则是规则集的组成部分。`规则`基本上是给 `EM12c` 的指令，指示在发生事件、事件或问题时如何自动执行操作。规则不会追溯操作，因此规则仅对在其创建之后发生的事件、事件或问题生效。

规则由两部分组成：条件和操作。`条件`指定规则所应用的事件、事件或问题。`操作`告诉 `EM12c` 您希望它对这些指定的事件、事件或问题执行哪些操作。每个操作又可以有一些附加条件。让我们看几个示例来了解其工作原理：

### 规则与通知的优先级

在某些情况下，你的`EM12c`站点可能处于高负载状态。在这些情况下，更重要的事件和事件会优先于其他事件进行处理。在确定处理优先级时会考虑两个因素。

第一个是目标的生命周期状态，优先级基于以下顺序：关键任务目标具有最高优先级，其次是生产、阶段和测试目标，然后开发目标的优先级最低。作为管理员，你的职责之一是为每个目标适当设置生命周期状态。此处显示的状态是产品附带的开箱即用状态。

第二个因素是事件或事件的类型。任何与可用性相关的事件或事件，例如目标宕机，始终设置为最高优先级。接下来，处理任何严重级别为关键或警告的事件或事件，最后，任何仅信息性的事件被视为最低优先级。


这种优先级处理仅在系统处于高负载时才会考虑。在系统正常负载下，事件和事故会按到达顺序处理。

### 向后兼容性

在向后兼容性方面，当您从早期版本迁移到`EM12c`时，通知功能提供了向后兼容性。如果您基于`PL/SQL`、操作系统脚本或`SNMP`陷阱创建过任何通知，这些通知方法将被迁移并在`EM12c`中继续工作。

不过，Oracle 确实鼓励您使用新的事件模型创建这些通知方法的新版本，以便利用我们一直在讨论的新特性。如果您在早期版本中创建过通知规则，Oracle 会在升级过程中将其作为一部分迁移为新版本中的事故规则集。

## 事件管理器指南

既然您已经了解了如何在`EM12c`版本中使用`事件管理器`功能，接下来让我们看看一些关于如何充分利用此功能的建议指南。不过，在此之前，最好先定义一些您可能希望达成的目标，以确保使用`事件管理器`实现可扩展、更智能的监控。从高层次来看，这些目标可定义如下：

### 满足您的监控要求。
这些要求当然因客户而异，但一般来说，这意味着必须发送适当的通知，并遵循事故管理操作流程。实现此目标的最简单方法是部署监控标准。

### 遵守安全实践。
在授予目标权限时，请确保遵循最小权限原则。换句话说，只授予执行所需监控所需的权限级别。

### 易于管理。
重要的是，您的基础设施不会成为管理难题，因此请充分利用管理组，以便能够“将多个作为一个”进行管理，同时只需最少的努力来监控新目标。

设置目标监控和目标权限角色已在第 7 章中介绍，因此这里只看看设置规则来管理事故的建议指南。

## 管理事件和事故

管理事件和事故的首要考虑之一是从源头控制事件。正如在性能管理中您希望确保只完成需要完成的工作一样，在`事件管理器`中，有意义的做法是只让那些您希望响应的事件和事故发生。不必要的事件会给您的系统带来不必要的负载。以下是您为实现此目的可能需要考虑的一些事项：

*   审查您的指标设置。这看似显而易见，但请确保只对您实际关心的指标设置阈值。
*   根据指标趋势调整指标阈值。您可以对监控系统执行的一项更重要的操作是，在一段时间内跟踪指标趋势（时长取决于您可能已有的历史数据量以及指标测量频率），以便您可以就哪些指标重要（这可以反馈到前一点）以及阈值应设置在什么水平做出明智的决定。
*   适当设置发生次数。例如，如果某些事件只发生一两次，您可能不需要被通知。您可以设置在收到通知前必须达到的指标发生次数。
*   使用修正操作自动清除指标警报。根据指标本身，您可以定义一个将自动执行的修正操作来解决问题。这将自动清除指标警报，因此您无需采取行动。您可能还需要考虑创建一个规则，如果事件已打开一定天数，则清除所有指标警报的事件。
*   为您可能不关心的指标禁用指标收集。一旦您拥有了第二点中讨论的历史信息，您就可以利用它来设置指标阈值，并禁用在您的特定环境中可能不重要的那些指标。
*   在监控模板中定义指标设置，并在监控集合中使用这些模板。这将使您能够充分利用管理组，当新目标添加到您的环境或需要调整阈值时，减少您需要做的工作量。

### 规则集建议

关于规则集的建议，让我们从一些通用建议开始。首先，正如刚刚讨论的，强烈建议将组作为规则集的目标。您希望将属于该组成员的所有相关规则放在同一个规则集中。然后，尽可能利用您可以控制规则集中规则执行顺序这一事实。

当您在`EM12c`中的规则集中创建规则时，您会注意到三种类型的规则。让我们看看关于如何使用这些类型的具体建议。对于操作事件的规则，其用例是为在`Enterprise Manager`中管理的警报或事件创建事故。如果您想在`Enterprise Manager`中使用工单连接器为这些事故创建工单，请使用事件规则来实现，即通过规则操作首先在事件上创建事故，然后为该事故创建工单。事件规则的另一个用例是使用事件连接器将事件简单地发送到第三方管理系统。最后一种情况是仅发送事件通知而不创建事故，这在早期版本中也是可以的。



# 规则

对于处理事件的规则，其主要用例之一是自动化与事件工作流相关的操作（例如，自动为事件分配负责人），设置事件的优先级或其升级级别，或者为事件发送通知。你还可以在另一个用例中，根据事件条件创建工单，例如，当事件升级到第 2 级时创建一个工单。

规则用于问题，以自动化管理工作问题的工作流，其方式与处理事件非常相似。例如，你可以自动为问题分配负责人，设置问题的优先级或升级级别，或者为问题发送通知。

事件规则集可轻松用于自动化和工作流管理。例如，你可能会执行以下操作：

*   自动分配事件以加快解决速度。创建事件时，自动分配事件负责人。或者，创建一个单独的规则，基于特定标准分配负责人。无论哪种情况，这都节省了手动将事件分配给特定负责人的需要。
*   自动化你的升级流程。你可能希望根据事件处于打开和未解决状态的时间长度来设置其升级级别。如果你没有像第一个要点中讨论的那样自动分配事件，你可能希望升级未分配的重要事件。
*   为升级的事件或问题发送通知。当事件或问题被升级时，你可以发送电子邮件通知、分配负责人或为事件或问题设置优先级。
*   自动分配优先级级别。与负责人分配一样，你可以在事件创建时自动分配一个优先级级别。然后，你可以在事件管理器控制台中按优先级排序，并首先处理高优先级的事件。

# 摘要

事件管理器为管理员提供了一个中心控制点，用于管理由企业经理检测到的基础设施内的事件、事故和问题。它使你能够快速有效地监控和解决服务中断，并提供了一种管理数量较少但有意义的事件，而不是大量离散事件的方法。通过与 My Oracle Support 的集成，它允许你访问集成的 Oracle 专业知识，以加速事件和问题的诊断与解决。

## Oracle Management Agent

`Oracle Management Agent`负责发送其安装主机的指标和待处理警报。为了提供高可用性，应为代理启用以下功能：

*   启用代理在操作系统启动时自动启动。这应在运行`root.sh`脚本后自动配置。
*   在冗余存储上安装代理。这包括配置了 RAID 的存储后端。
*   为代理看门狗进程配置重启参数。看门狗进程负责基于以下环境变量重启代理：
    ```
    EM_MAX_RETRIES  表示看门狗进程在 `EM_MAX_RETRY_WINDOW` 内尝试重启代理的最大次数。默认设置为 3。
    EM_RETRY_WINDOW  表示与 `EM_MAX_RETRIES` 一起使用的时间间隔，用于在尝试启动管理代理之前进行等待。默认设置为 600 秒。
    ```

安装 Enterprise Manager 代理后，应执行`<AGENT_HOME>/root.sh`脚本。这将在 Linux 和某些 Unix 操作系统上创建以下控制管理代理启动的脚本：

*   `/etc/init.d/gcstartup`
*   `/etc/init/gcstartup`
*   `/etc/rc.d/init.d/gcstartup`

如果这些文件不存在，请确保已执行`<AGENT_HOME>/root.sh`脚本。

代理的丢失或损坏将导致其关联目标的监控和指标数据上传丢失。同样，如果代理关闭，目标将无法与管理服务器通信，导致可管理性丧失。

### Management Repository



资料库是代理上传的监控数据的持久化存储。它存储所有监控目标的指标和配置数据。资料库丢失将导致管理服务器故障。建议启用数据库高可用性功能，例如用于数据文件存储（取决于冗余级别和后端 RAID 配置）的实时应用集群 (`RAC`) 或 `RAC One Node`、自动存储管理 (`ASM`)，以及用于资料库的数据卫士 (`Data Guard`)。此外，如果资料库配置为 `RAC` 数据库，则管理服务应配置为使用单客户端访问名称 (`SCAN`) 和非默认的 `RAC` 服务名。以下是一个为 `RAC` 数据库配置管理服务的示例：

```
(DESCRIPTION= (ADDRESS=(PROTOCOL=TCP)
(HOST=emrep-scan.example.com)(PORT=1521) )
(CONNECT_DATA=(SERVER=DEDICATED) (SERVICE_NAME = emrep)))"
-repos_user sysman
```

### 管理服务

管理服务 (`OMS`) 通过 Enterprise Manager 控制台提供用户界面，并处理来自代理的数据。`OMS` 丢失将导致 Enterprise Manager 完全中断：代理上传、作业、事件和通知都将停止运行。Oracle WebLogic Node Manager 和 Oracle 进程管理器与通知服务器 (`OPMN`) 将尝试在管理服务宕机时自动重启它。虽然这提供了一些好处，但如果主机宕机，它并不能保护 `OMS`。至少，如果可能，`OMS` 和资料库应安装在不同的主机上。多个 `OMS` 可以部署在服务器负载均衡器之后，以提供针对单台主机故障的保护。此外，您可以选择在共享文件系统上安装 `OMS`，这将在单台主机丢失时提供被动故障转移。我们将详细探讨保护 `OMS` 的每种选项。

### 级别 1 — 分离的 `OMS` 和资料库主机/无冗余

级别 1 配置由单个 `OMS` 和资料库组成，每个都安装在其自己的主机上。此配置提供的保护最少，因为任何主机的故障都将导致 Enterprise Manager 系统完全中断。应考虑 `OMS` 和资料库之间的距离，因为两者之间的高网络延迟会降低性能。建议除最小配置外的所有配置都采用此配置。

图 13-4 是级别 1 高可用性配置的示意图。代理直接上传到管理服务主机，而用户通过 Enterprise Manager 控制台或命令行 `EMCLI` 直接与物理 `OMS` 主机交互。如果管理服务或资料库数据库主机中的任何一个变得不可用，将发生完全中断，导致目标监控丢失。将每个组件保留在其自己的服务器上可以降低它们因资源开销而相互影响的可能性。例如，增加数据库参数 `sga_max_size` 和 `sga_target` 可能会降低 `OMS` 的性能，因为这样做会减少操作系统可用的内存量。此外，它为根据业务需求扩展环境奠定了基础。

### 级别 2 — 主动/被动 `OMS` 和数据卫士资料库

为了减少计划内或计划外中断期间的 `OMS` 停机时间，应在配置中引入一些冗余。级别 2 配置使用共享文件系统实现管理服务的主动/被动（或冷）故障转移集群解决方案。文件系统在两个或多个主机之间共享，且一次仅在一个主机上处于活动状态。

## 为 OMS 实施冷故障切换

为 OMS 实施冷故障切换方案后，您就受到了保护，免受单台主机故障的影响。然而，执行故障切换的时间可能从几分钟到几小时不等，具体取决于是手动完成还是通过 Oracle Clusterware 或其他方法自动完成。

存储库也需要受到保护，因为它现在是一个单点故障。如前所述，强烈建议设置一个本地 Data Guard，包含一个物理备用数据库。备用数据库应配置在与管理服务器和主数据库分离的主机上。但是，为了降低成本，也有可能在另一台 OMS 主机上创建主数据库和物理备用数据库。如果主存储库发生计划内或计划外的中断，物理备用数据库可以在剩余的主机上切换或故障转移到备用数据库。请注意，该主机现在将成为一个单点故障。

# 三级高可用性配置

在上一节中，我们确定了当 `OMS` 故障转移到另一台主机时，会出现一定的停机时间。某些环境无法容忍此类停机，因此需要提高可用性级别。幸运的是，这可以通过使用多个管理服务器并结合 `RAC` 数据库作为存储库来实现。`RAC` 为数据库提供了高可用性和可扩展性。你也可以考虑采用主动/被动 `RAC One Node`⁷ 配置。此外，还使用本地物理备用数据库来保护数据库以防数据库存储故障。管理服务位于服务器负载均衡器（`SLB`）之后。`SLB` 随后将来自企业管理器控制台和管理代理的流量定向到可用的 `OMS`。每个管理服务器可以安装在与 `RAC` 节点不同的主机上（参见图 13-16）。然而，你可能需要平衡使用独立服务器的成本与此类配置所需的保护级别。管理服务器和存储库数据库应彼此靠近以减少网络延迟。这可能要求管理服务器和 `RAC` 数据库实例共存于同一服务器上。

![9781430249382_Fig13-16.jpg](img/9781430249382_Fig13-16.jpg)

图 13-16 .  三级高可用性配置：多个 `OMS` 配置在服务器负载均衡器之后，并使用 `RAC` 数据库作为管理存储库

此级别的可用性还提供了根据业务需求进行扩展的能力。可以添加更多 `OMS` 服务器以进行横向扩展，同时可以向 `RAC` 数据库添加节点以扩展存储库。

## 配置步骤

设置三级高可用性配置所需的步骤如下：

1.  安装主 `OMS`。
2.  配置 `SLB`。
3.  添加存储库目标。
4.  配置软件库。

7.  将备用数据库转换为 `RAC`。

## 服务器负载均衡器配置

当配置了服务器负载均衡器时，`SLB` 会将代理和控制台流量定向到多个管理服务。`SLB` 配置应在安装第一个 `OMS` 之后进行。

### SLB 配置要求

`SLB` 的配置可能因设备制造商而异。然而，`SLB` 有几个要求，列举如下：

*   `虚拟服务器端口`：应配置四个端口——安全上传、代理注册、安全控制台和非安全控制台。如果 `OMS` 仅配置了安全上传和代理流量，则只需要安全上传和安全控制台虚拟服务器端口。
*   `持久性`：企业管理器控制台与 `OMS` 之间的 `HTTP` 和 `HTTPS` 流量需要持久性或“粘性”，以便发送到某个 `OMS` 的请求在同一会话期间不会切换到另一个。没有持久性可能导致用户因会话被不同的管理服务处理而不得不持续登录。
*   `应用程序监控`：检查管理服务的健康状况，以便请求不会被发送到不可用或性能下降的服务。

### 第 4 级—带备用站点的 MAA

前面描述的高可用性级别将针对计划外和计划停机提供保护。使用带有多个 `Oracle Management Services` 的 `SLB` 可以在单个 `OMS` 变得不可用时提供服务。使用 `Real Application Clusters` 结合本地 `Data Guard` 可为存储库提供最高级别的可用性，即使主机不可用也不会停机。如果执行了故障切换或切换操作，只会发生最短的停机时间。但是，如果主站点发生完全站点故障，整个 `Enterprise Manager Cloud Control` 环境将变得不可用，可能导致依赖于该基础设施的其他应用程序服务中断。

### 先决条件

在单独的站点配置备用 `Enterprise Manager Cloud Control` 系统可提供高可用性并防范站点故障。此配置包括位于 `SLB` 后面的多个主动/主动 `OMS` 服务器，以及位于主站点和另一个异地位置的 `RAC` 主数据库和 `Data Guard` 中的备用数据库。备用管理服务的先决条件如下：
- 按照前面所述，使用 `MAA` 配置来配置主站点。
- 使用类似的硬件和资源配置备用站点，以确保切换到备用站点后不会发生性能损失。
- 用于 `Software Library` 的共享存储应复制到备用站点。`Software Library` 的路径在两个位置也应相同。考虑使用基于磁盘的复制或基于软件的复制，例如 `rsync`⁸ 或类似工具。
- 还应在备用站点部署第二个 `SLB`，以确保在灾难恢复环境中的完全冗余。使用提供从主站点到备用站点的 `VIP` 故障转移的供应商解决方案。

满足这些先决条件后，按照 *`Oracle Enterprise Manager Cloud Control 12c Administrator’s Guide`* 中的说明安装和配置备用管理服务。

# 备用管理服务器安装后注意事项

## 备份

在高可用性 Enterprise Manager Cloud Control 配置中，`OMS`、仓库和代理这三个组件都单独配置以减少停机时间。尽管我们建议实施最高级别的可用性，但相关的成本和所需资源可能令人望而却步。此外，在任何 IT 框架中，随着时间的推移，组件故障很可能会发生。因此，任何高可用性架构如果不讨论备份和恢复策略都是不完整的。

备份 Enterprise Manager 生态系统需要一个涵盖仓库、`OMS` 和代理的解决方案。每个组件都根据其可用方法单独备份。管理仓库数据库存储由代理提供的指标和数据。应使用 `RMAN` 按照 Oracle 数据库的推荐备份策略进行备份。

## 恢复

执行每个 Enterprise Manager Cloud Control 组件的定期备份很重要。然而，更重要的是在需要时能够成功恢复这些组件。Enterprise Manager 的恢复可能需要恢复多个组件，包括存储库、管理服务和代理，具体取决于问题的性质。例如，在主机丢失的情况下恢复管理服务可能需要比将管理服务恢复到原始主机更多的步骤。你将在以下部分中研究这些场景。

## 存储库恢复

如果 `OMS` 关闭，Cloud Control 控制台将不可用。要恢复存储库数据库，应使用 `RMAN` 命令行实用程序。使用 `RMAN`，你可以通过完全恢复或时间点恢复来恢复数据库（请参阅 `Oracle Database Backup and Recovery Guide`）。对于后者，你可能需要重新同步与存储库不同步的代理。要使用 `EMCTL` 重新同步代理，请使用以下语法：

```
emctl resync repos (-full|-agentlist "agent names")
[-name "resync name"]
[-sysman_pwd "sysman password"]
```

此命令应在管理存储库恢复之后且在启动 `OMS` 之前执行。

在恢复存储库数据库后，使 Enterprise Manager Cloud Control 运行所需的步骤将根据执行的是完全恢复还是不完全恢复，以及是在同一主机还是不同主机上执行恢复而有所不同。表 13-3 总结了不同恢复场景所需的步骤。在每种情况下，都应在停止 `OMS` 后恢复存储库数据库。

每当在不同的主机上恢复存储库数据库时，都需要重新配置 `OMS`。这将新的数据库连接描述存储在 `OMS` 中。我们使用以下命令在 `OMS` 中重新配置存储库数据库：

```
emctl config oms -store_repos_details (-repos_host <host> -repos_port <port> -repos_sid
<sid> | -repos_conndesc <connect descriptor>) -repos_user <username> [-repos_pwd <pwd>]
```

按照以下步骤重新配置 `OMS`：

1.  停止所有 `OMS`。

    ```
    emctl stop oms
    ```

2.  在每个 `OMS` 上重新配置存储库连接字符串。

    ```
    emctl config oms -store_repos_details
    ```

    使用以下命令重新启动管理服务器和所有 `OMS`：

    ```
    emctl stop oms -all
    emctl start oms
    ```

在不同的主机上恢复存储库还需要将管理存储库数据库目标重新定位到运行在新主机上的不同代理。这仅在主机上已存在代理且未由其发现任何其他数据库的情况下才可完成。使用以下语法将目标重新定位到新主机：

```
emctl config repos [-sysman_pwd <sysman password>]
      [-agent <new agent>]    指定存储库目标的新目标代理
      [-host <new host>]      指定存储库目标的新主机名
      [-oh <new oracle home>] 指定存储库目标的新 OracleHome
      [-conn_desc [<jdbc connect descriptor>]]
                              如果指定了值，则使用该值更新连接描述符，
                              否则使用存储在 emoms.properties 中的值
      [-ignore_timeskew]      忽略代理上的时间偏差
```

还应使用此命令更新 `OMS` 和存储库目标的监控配置：



```
emctl config emrep [-sysman_pwd <sysman 密码>]
      [-agent <新代理>]    指定 emrep 目标的新目标代理
      [-conn_desc [<JDBC 连接描述符>]]
                              如果指定则使用该值更新连接描述符，
                              否则使用 emoms.properties 中存储的值
      [-ignore_timeskew]      忽略代理上的时间偏移
```

在数据库恢复并重新配置后，下一步是登录到 Enterprise Manager Cloud Control 控制台，验证所有操作是否已恢复。

### Oracle 管理服务恢复

恢复 OMS 需要恢复软件主目录（Fusion Middleware）和实例主目录（gc_inst）。软件主目录可以从文件系统备份恢复，或使用 Enterprise Manager Cloud Control 12c 安装程序在相同或不同主机上使用“仅安装软件”选项重新安装。OMS 主目录必须与正在恢复的 OMS 主目录位置相同。

OMS 中存在的所有插件都必须安装才能成功恢复。应以 SYSMAN 用户身份运行以下 SQL 查询：

```
SELECT epv.display_name, epv.plugin_id, epv.version, epv.rev_version,decode(su.aru_file,
null, 'Media/External', 'https://updates.oracle.com/Orion/Services/download/ '
||aru_file||'?aru='||aru_id||chr(38)||'patch_file='||aru_file) URL
FROM em_plugin_version epv, em_current_deployed_plugin ecp, em_su_entities su
WHERE epv.plugin_type NOT IN ('BUILT_IN_TARGET_TYPE', 'INSTALL_HOME')
AND ecp.dest_type='2'
AND epv.plugin_version_id = ecp.plugin_version_id
AND su.entity_id = epv.su_entity_id;
```

以下示例输出显示了插件、版本、修订版和 URL。如果通过 Self Update 下载，URL 将显示。否则，状态将显示为 Media/External。

```
SQL>

------------------------------  ------------------  -----------  ------------  ----------------
Oracle MOS (My Oracle Support)  oracle.sysman.mos   12.1.0.2.0              0  Media/External
Oracle Fusion Middleware        oracle.sysman.emas  12.1.0.3.0              0  Media/External
Oracle Database                 oracle.sysman.db    12.1.0.2.0       20120804  Media/External
Oracle Exadata                  oracle.sysman.xa    12.1.0.3.0              0  Media/External
```

如果列出了任何其他插件，应将它们下载到单个目录中；然后将扩展名从`.zip`重命名为`.opar`。如果未从文件系统备份恢复，请使用“仅安装软件”选项安装中间件和 OMS Oracle 主组件。软件重新安装或恢复后，下一步是安装其他插件（如果有）。通过指定`–PluginLocation`标志来选择下载插件的位置，执行位于`OMS_HOME/sysman/install`中的`PluginInstall.sh`脚本。在仅软件安装的情况下，之前应用的所有补丁都必须重新应用。

恢复软件主目录后，下一步是恢复或重新创建 OMS。这通过使用`omsca`实用程序并指定由`emctl exportconfig`命令生成的备份文件路径来完成。

```
omsca recover –as –ms -nostart –backup_file <exportconfig 文件>
```

恢复 OMS 所需的步骤可能因是否使用 SLB 或配置了多个管理服务以及是否在同一主机或不同主机上恢复而异。以下步骤是在没有 SLB 的同一主机上恢复单个 OMS 所必需的。OMS 实例将通过使用`emctl exportconfig`获取的 OMS 配置备份进行恢复。

1.  清理失败的主机。
2.  验证软件库是否可用。
3.  通过恢复文件系统备份或运行安装程序的“仅软件”选项来恢复软件主目录。（参见`Oracle Enterprise Manager Cloud Control 高级安装与配置指南第 2 版`）。
4.  在恢复模式下执行`omsca`，并指定先前获取的备份文件位置。
5.  如有必要，恢复代理。
6.  启动 OMS。
7.  验证 OMS 是否正常工作。

有关其他 OMS 恢复场景的详细信息，请参阅`Oracle Enterprise Manager Cloud Control 12c 管理员指南`。

### 管理代理恢复

管理代理的恢复需要重新安装它，最好从参考安装进行，或执行从先前备份的文件系统恢复。代理应使用现有补丁和自定义设置进行克隆。代理也应使用相同端口安装。重新安装后，应从 Enterprise Manager Cloud Control 中的“代理重新同步”页面执行重新同步。

![image](img/sq.jpg) `注意` 重新安装后，代理会被 OMS 阻止，以防止目标覆盖先前配置的数据。使用“重新同步代理”按钮重新同步并取消阻止代理。

在典型场景中，代理恢复通常遵循以下步骤：

1.  使用 Oracle Universal Installer (OUI) 清理现有代理 Oracle 主目录以清理库存。
2.  安装新代理或从参考安装克隆，使用相同端口和路径。也可以从先前备份恢复文件系统。
3.  从 Enterprise Manager Cloud Control 代理主页执行代理重新同步。
4.  如有必要，重新配置用户定义的指标。
5.  验证代理状态。

```
emctl status agent
emctl upload agent
```

## 切换和故障转移

当使用级别 2、3 或 4 高可用性配置时，管理服务和存储库都可以独立切换和故障转移。切换通常在计划维护期间进行，包括操作系统和软件打补丁，而故障转移通常在计划外维护期间发生，例如硬件或软件故障。

在级别 2 配置中，管理服务的切换和故障转移遵循相同的程序。如前所述，这需要将 OMS 虚拟主机名和 VIP 重新定位到另一主机，然后手动启动 OMS 或使用 Clusterware 自动化重新定位。在使用物理备用数据库用于 Enterprise Manager 存储库的级别 2、3 或 4 配置中，数据库必须切换（计划）或故障转移（未计划）。在级别 3 高可用性中，OMS 不需要任何操作，因为涉及多个管理服务。SLB 将监视管理服务，检测到一个服务失败时将流量路由到可用的 OMS。

要使用 Data Guard 切换管理存储库，请使用 DGMRL 命令行实用程序，因为 Enterprise Manager 控制将无法切换存储库。按照以下步骤切换管理存储库，而管理服务尚未切换：

## 总结

本章介绍了企业管理系统云控制（Enterprise Manager Cloud Control）的主要组件——`Oracle 管理服务`、`Oracle 管理存储库`、`Oracle 管理代理`以及`软件库`——这些组件都需要配置为高可用性。每个组件都需要通过不同的方法进行保护。

存储库数据库的高可用性需要使用数据库的高可用性特性，包括`ASM`、`RAC`和`数据守护`。每种特性都应在适当的时候使用 Oracle 推荐的最佳实践进行配置。`RAC`提供了可扩展性，并能在单个主机故障时实现无缝故障转移。`数据守护`则能在主机和存储发生故障时提供保护，在角色切换期间实现最短的停机时间（通常少于一分钟）。在最具可配置性的形态下，存储库数据库部署在`RAC`数据库上，并使用`数据守护`配置了一个本地的`RAC`物理备用库。一个具有相同配置的备用站点也可用于在主站点丢失时提供服务。

`Oracle 管理服务`可以使用多种技术来实现高可用性，每种技术在成本和复杂性上有所不同。最简单的技术涉及将`OMS`主机与存储库主机分离。您还了解了主动/被动或冷故障转移解决方案，其中多个主机在共享文件系统上共享单个`OMS`，但在任何给定时间点只有一个处于活动状态。这利用了`VIP`的概念，以便在集群中一个主机丢失时进行故障转移。需要手动进行故障转移，并且在`OMS`在另一台主机上重新启动时会出现一些停机时间。下一个层级是在负载均衡器后面设置多个管理服务。这可以在单个`OMS`丢失时实现无缝故障转移解决方案。在`MAA`配置中，备用站点的配置与主站点类似。这显然成本最高。然而，它不仅提供单站点可用性，还能抵御灾难。

管理代理通过在软件库中配置参考镜像来实现高可用性。这使得使用部署过程更容易恢复代理。

高可用性架构的一个关键方面是其备份和恢复策略。企业管理系统云控制中的每个组件都应在适当的时候遵循推荐的最佳实践，以确保可恢复性，满足业务的恢复点和恢复时间目标，从而将对业务的影响降至最低。

除了本章讨论的四个高可用性级别外，其他技术解决方案也可以提供不同级别的高可用性。这些包括但不限于虚拟化软件（如`Oracle VM Server`）、数据复制技术（包括`Oracle GoldenGate`和`Oracle Streams`）以及存储级复制解决方案。解决方案的选择取决于业务需求、资源和成本。

¹ `最大可用性架构 (MAA)` 是一套基于高可用性特性的 Oracle 推荐实践。这些建议基于产品开发验证和客户运行 Oracle 产品的经验。

² `快速恢复区` 在 11.2 版之前的 Oracle 数据库中被称为闪回恢复区。

³ `TZ`环境变量的格式取决于您的操作系统。有关设置`TZ`变量的更多信息，请参阅您的操作系统文档。

⁴ 回切的默认值设置为 0，这意味着`VIP`及其依赖资源在原始节点再次可用后不会自动回切。

⁵ `Oracle Restart`是 Oracle Database 11gR2 的单实例高可用性功能。它通过在发生故障时重新启动数据库实例、服务和监听器来提供高可用性。它还会在服务器启动时重新启动数据库组件。

⁶ `保护模式`指的是在主数据库故障时接受的潜在数据丢失。三种可用模式是：最大性能、最大可用性和最大保护。

⁷ `Oracle RAC One Node`是 Oracle Database 11g 第 2 版中提供的一项新选件。`Oracle RAC One Node`是一个在集群中的一个节点上运行的 Oracle `RAC`启用数据库的单实例。

⁸ `rsync`是一种软件，它能够在将文件和目录从一个系统复制到另一个系统的同时，通过使用差异算法最小化数据传输。
