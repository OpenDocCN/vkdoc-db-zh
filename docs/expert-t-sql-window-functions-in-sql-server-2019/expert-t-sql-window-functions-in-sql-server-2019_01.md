# 前言

`SQL` 开发于 20 世纪 70 年代，并自 1986 年起通过 `ANSI` 批准的委员会成为正式标准。时至 2019 年，`SQL` 已成为应用最广泛的声明性语言。在此过程中，窗口函数已成为该标准的重要组成部分。`ANSI` 并不制定标准，但在记录和维护标准方面扮演着重要角色。各个软件供应商自愿决定遵守标准，而像本书作者这样的工作者则致力于以实用的方式阐释 `SQL` 的使用。

在我自身的数据科学和高级分析职业生涯中，窗口函数在过去几年里一直是多个关键项目的重要组成部分。几年前，我曾基于本书的早期版本为用户小组制作过一个 YouTube 视频。自那时起，作为微软的职业架构师，我一直在为数据科学应用提供建议。在一个项目中，大约 20 个特征的输入并未产生足够的结果：通过使用窗口函数，我领导的一个团队（没错，直接编码）迅速将特征数量增加到了 1000 多个。不仅是数量的增长，准确率也得到了提高，从业务角度而言，该组织每年因此为其问题节省了数百万美元。在上个月，我遇到了一个无关的新项目，情况类似：一个时间序列类型的数据集，有机会从不到 20 个特征增长到更大的数量。

人们可能会想，自动化机器学习技术是否会自行完成此类组合，而我对此持怀疑态度。仅从窗口函数创建一个稳健的特征集不仅需要时间序列方面的考量，还需要基于对数据领域了解的聚类知识。即使自动化技术在此主题上取得巨大进展，我预计任何数据科学家在更典型的数据科学调查中（仅有少数特征和少量观测值时）仍需要掌握这些函数的基本知识。

`SQL` 是本地和云数据库技术的核心——而在数据科学领域，许多人使用 `Apache Spark`（`SQL Server 2019` 以及许多其他数据技术）。其在高级分析中的延伸应用，是窗口函数成为 `SQL` 语言专家级主题的又一个原因。主流应用范围从任何业务分析 `SQL` 查询，一直延伸到支持高级分析和机器学习算法。

多年来，能分别认识 Kathi Kellenberger 和 Clayton Groom 这两位备受尊敬的同行和专业人士，并见证他们如何通过多次演讲（他们通常自愿奉献自己的时间）以及本书的创建和现在的修订，各自成为技术用户社区的重要领导者，是我莫大的荣幸。在这次修订中，Ed Pollack 添加了关于棒球统计的材料，说明了并非所有时间序列都与金钱相关。仅拥有标准是不够的，还需要有专业的教练来解释这些函数如何描述业务分析的方法。本书提供了丰富的示例，共同为读者清晰地指引了一条通往 `SQL` 语言中数学最复杂但实际最有用的方面之一的路径。
