## 移动总量与移动平均值

到目前为止，查询已经变得越来越复杂。它们即将变得更加复杂。为了帮助保持其可理解性，你需要从概念上将它们视为在数据之上层层叠加的操作。早期的例子是单个查询，在一次遍历中聚合一组记录。为了引入连续的日期范围以消除交易数据中的空白，通过引入一个 CTE（公用表表达式）在主要聚合查询之前做一些预处理，从而增加了第二个操作。你已经在代码清单 10-4 中看到了这一点，但在本章中，我们将更进一步。

在本章的这一部分，你将通过将上一个示例中的聚合结果转化为一个 CTE，并在其上进行结果聚合，来添加第三个操作。在某些情况下，使用一个或两个操作也可以达到相同的结果，但对于需要嵌套窗口函数的情况，唯一的办法就是增加另一个层级。计算月度级别的``[Sales PY YTD]``（上年累计销售额）就是一个例子。为了创建``[Sales YTD]``（本年累计销售额）度量，你必须使用窗口函数的所有子句。已经没有其他方法可以让你将分区移回上一年了。通过先在第二个操作中计算``[Sales YTD]``，你就可以在第三个操作中使用窗口函数来计算``[Sales PY YTD]``。

同样，第二个优点是，任何更高阶操作的列计算都可以使用低阶操作中有意义的列名，从而使计算更容易理解，代码也更紧凑。同时，也必须权衡考虑性能和代码可读性。以下伪代码规划了剩余计算将在 T-SQL 中实现的逻辑：

```
[Current Sales] = 某一时间段（日、月、年）内 SalesAmount 的总和
[Prior Sales] = 上一时间段内 SalesAmount 的总和
[Growth] = [Current Sales] - [Prior Sales]
[Growth %] = [Growth] / [Prior Sales]
[3 MMT] = 过去三个月内 SalesAmount 的总和
[12 MMT] = 过去 12 个月内 SalesAmount 的总和
[3 MMA] = 过去三个月内月均销售额的平均值
[12 MMA] = 过去 12 个月内月均销售额的平均值
[3/12 ROC] = [3 MMT]/([3 MMT].LAG( 11 months))
[12/12 ROC] = [12 MMT] /([12 MMT].LAG( 11 months))
[3/12 RoC2]= ([3 MMT]/([3 MMT].LAG( 11 months))* 100) -100
[12/12 RoC2] = ([12 MMT] /([12 MMT].LAG( 11 months))* 100) -100
```

### 确保数据集完整

最后需要解决的一个问题是，确保任何移动总量或移动平均值计算所基于的数据集都包含正确数量的输入行。例如，一个三个月的移动平均值``[3 MMA]``必须包含三个完整月份的数据，否则计算结果就不正确。通过处理交易日期范围的空白，部分问题得到了解决，但并非全部。在数据集的开始，第一行没有前序行，因此计算该行的移动平均值是不正确的。数据集的第二行也只有一行前序行，为其计算平均值同样不正确。只有到达第三行时，计算三个月移动平均值的条件才满足。表 11-1 展示了在给定该产品上年无数据的情况下，应该如何计算三个月移动平均值：

表 11-1

消除跨时间段的平均计算中的不完整结果

| 月份 | 产品 | 销售额 | 年累计销售额 | 三月移动平均 |
| --- | --- | --- | --- | --- |

为了实现这一点，你可以简单地统计三个月窗口期内的行数（`COUNT`），而不是对结果进行平均，并使用这个计数值来决定何时执行计算。如果有三行数据，则执行计算；否则返回空值。下面的例子使用`CASE`语句来判断哪些行有两个前序行：

```
CASE WHEN COUNT(*)
OVER (ORDER BY p.CalendarYear, p.CalendarMonth
ROWS BETWEEN 2 PRECEDING AND CURRENT ROW) = 3
THEN AVG(SUM(f.SalesAmount))
OVER(ORDER BY p.CalendarYear, p.CalendarMonth
ROWS BETWEEN 2 PRECEDING AND current row)
ELSE null
END AS [Sales 3 MMA]
```

代码清单 11-1 中的查询包含了 3 个月和 12 个月滚动期的移动总量和移动平均值计算。这些计算在第二个 CTE 操作中实现，因此它们的结果可以在第三个操作的 select 语句中进行进一步处理。
