## 将数据库迁移到 Exadata

一个新安装和配置好的 Exadata 系统只有一个数据库，通常命名为 DBM。正如您可能预料的那样，它是一个 RAC 数据库，因此存在于所有可用的数据库服务器上。Exadata 可以运行多个数据库（无论是集群的还是非集群的），这正是您最初购买该系统的原因。如果您以前构建过 RAC 数据库，那么创建新数据库并不困难；如果您没有经验，我们将在本章提供说明和指导来帮助您。然而，您不会总是构建真正“新”的数据库，因为您的目标是将现有的应用程序数据库迁移到 Exadata。有多种方法可以完成此任务。其中一些方法，例如逻辑方法，需要一个空数据库来操作，而另一些方法（物理方法）则只需要一个 `init.ora` 文件和控制文件就可以开始整个过程。本章的目标是让将数据库迁移到 Exadata 的任务尽可能轻松。

## 物理迁移方法

在本节中，我们将讨论三种将数据库物理迁移到 Exadata 的选项：备份、可传输表空间和备用数据库。每种方法都有其优点和缺点。您选择哪种方法取决于必须使用的迁移策略。并非所有平台都是相同的。有些是“大端”（big-endian），即数据“字”的最高有效字节写入内存的最低地址，而另一些是“小端”（little-endian），即数据“字”的最低有效字节写入内存的最低地址。

## 第四阶段

恭喜，您已完成 Exadata 系统的存储重新配置。如果您在示例中使用了 `SizeArr` 设置，查看 `V$ASM_DISKGROUP` 时，您应该会看到类似以下的输出：

```
SYS> select name, free_mb, usable_file_mb
  2  from v$asm_diskgroup
  3  where name like '%MYEXA%';

NAME                              FREE_MB USABLE_FILE_MB
------------------------------ ---------- --------------
DATA_MYEXA1                      52375300       17051522
RECO_MYEXA1                      46920568       15638300

SYS>
```

值得关注的值是 `USABLE_FILE_MB` 列中的数值。根据本章前面的内容，对原始存储分配执行该查询的输出如下。

```
SQL> select name, free_mb, usable_file_mb
  2  from v$asm_diskgroup
  3  where name like '%MYEXA%';

NAME                              FREE_MB USABLE_FILE_MB
------------------------------  --------- --------------
DATA_MYEXA1                      81050256       26959176
RECO_MYEXA1                      20324596        6770138

SQL>
```

根据 Exadata 系统的配置方式，您可能还需要执行其他任务，包括重新安装任何正在运行的 OEM 代理，以及如果它未使用默认端口，则重新配置 SCAN 监听器。此类步骤是重新配置前准备阶段的一部分，应在开始任何存储重新分配之前予以记录。

## 注意事项

在 Exadata 上重新配置存储分配是可能的，但这是一个破坏性的过程，需要在执行前进行规划。

必须使用 `/opt/oracle.SupportTools/onecommand/deploy11203.sh` 脚本删除现有的存储配置。该脚本也将使用不同的参数来创建新的存储配置。

删除现有存储配置分三步执行：删除 GRID Infrastructure、删除单元磁盘，然后删除相关的用户帐户和组。由于在此过程中将删除 “oracle” 操作系统用户，因此有必要在 Exadata 系统中所有可用的数据库服务器上保留现有的操作系统用户主目录。这可以使用 `tar` 或 `cpio` 来创建这些位置中目录和文件的存档。

在 `/opt/oracle.SupportTools/onecommand` 下有一个名为 `onecommand.params` 的配置文件。必须编辑此文件以提供新的存储阵列大小。此文件中提供的大小是针对 +DATA 磁盘组的（+DATA 和 +RECO 磁盘组名称通常包含 Exadata 机器名称，例如，对于名为 MYEXA1 的 Exadata 机器，有 +DATA_MYEXA1 和 +RECO_MYEXA1）；恢复磁盘组的大小由此参数的值推导而来。

实际的重新配置再次使用 `/opt/oracle.SupportTools/onecommand/deploy11203.sh` 脚本进行，包含 26 个独立步骤。这些步骤重新建立用户帐户、连接性、软件安装、磁盘分区创建、网格磁盘和 ASM 磁盘组的创建、默认 DBM 数据库的创建，以及在各个主目录（homes）上应用数据库和安全补丁。此过程的一部分还确保进程间通信 (`IPC`) 使用 InfiniBand 而非标准网络，以避免性能问题。

您必须执行的下一步是恢复 “oracle” 操作系统用户主目录中除 `.ssh` 目录外的所有内容。由于重新配置过程已建立了无密码 `ssh` 连接，覆盖这些文件将使该功能失效。

最后，您必须以 “oracle” 用户身份连接（最好使用额外的会话），以验证您在重新配置前建立的所有功能是否按预期工作。

您可能需要执行的其他任务包括重新安装 OEM 代理以及，如果您将其更改为使用非默认端口，则重新配置 SCAN 监听器。

系统可能会提示您覆盖现有文件。除了 `.ssh` 目录外，您应该这样做，因为您正在恢复的文件是您在存储重新分配之前配置的文件。请记住，此过程为 “oracle” 重新建立了无密码 `ssh` 连接，使用重新配置前保存的内容替换 `.ssh` 目录内容可能会使无密码 `ssh` 失效，因为保存的 authorized_keys 文件中的签名与当前值不匹配。检查日志文件中是否有您可能错过的错误，因为详细的输出可能滚动太快而没被注意到，并纠正列出的错误。再次使用 `tar` 来恢复因现已纠正的错误而未能创建的文件或目录。如果需要第二次使用 `tar`，请在提示时*不要*替换现有文件。

如果您使用 `cpio` 来归档目录及其内容，以下命令将恢复归档的项目：

```
$ cpio -ivf ./.ssh < /tmp/oraclehome.cpio | tee oraclehome_restore.log
```

`cpio` 实用程序读取指定的存档；提取除 `.ssh` 目录外的文件和目录；并将它们放在当前位置。如 `tar` 示例中所述，您不想覆盖 `.ssh` 目录内容，因为重新配置过程已为 “oracle” 重新建立了无密码 `ssh` 连接，覆盖这些当前文件将使该配置失效。归档恢复完成后，您需要检查日志文件中的错误。假设没有错误，您应该恢复到在更改存储分配之前建立的 “oracle” 操作系统用户配置。如果您确实在恢复中发现了错误，请纠正导致这些错误的状况，然后再次使用 `cpio` 来恢复由于错误而未能创建的文件或目录。

打开另一个会话并以 “oracle” 身份登录，以查看所有恢复的设置是否按预期功能运行。如果您已配置了设置各种 ORACLE_HOME 环境的方法，则应测试该方法，以确保在 “oracle” 操作系统用户主目录的存档或这些归档文件和目录的恢复过程中没有遗漏任何内容。
